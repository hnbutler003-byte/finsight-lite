## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes
**Issue:** AI and media generation routes (chat, image, audio) have no rate limiting applied before registration. These are the most expensive endpoints — OpenAI calls — and are fully unprotected against abuse by school-aged users or bots.

**Fix:** Apply a dedicated AI rate limiter before these registrations: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter uses express-rate-limit with a window of 1 minute and max of 10 requests per IP. Add this before the registerChatRoutes/registerImageRoutes/registerAudioRoutes calls.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 40-75)
**Issue:** The `host` and `proto` values are taken directly from `req.headers['x-forwarded-host']` and `req.headers['x-forwarded-proto']` without validation. An attacker can send arbitrary Host headers, causing the Sitemap directive and all <loc> URLs to point to attacker-controlled domains, which can be used to poison search engine crawlers.

**Fix:** Replace dynamic host construction with a hardcoded environment variable: `const base = process.env.APP_BASE_URL || 'https://finsightlite.com';` Set APP_BASE_URL in Railway config. Remove all req.headers host/proto reading from these two handlers entirely.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored directly in the job payload. For a multi-page PDF or high-res image this can easily exceed 5–20 MB per job row, bloating the jobs table, slowing queue polling queries, and risking hitting DB row-size limits.

**Fix:** Upload the file to object storage before enqueuing and store only the storage path in the payload (e.g. { paperId, storagePath, ext, subject }). Inside the handler, stream or download the file by path. This keeps job rows small and makes payloads safe to log.

### `server/jobHandlers.ts`: runExtractPaper — for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted one-by-one inside a sequential for loop with an await on each iteration. For a 40-question paper this serialises 40 round-trips to the DB, multiplying latency and holding the worker busy far longer than necessary.

**Fix:** Collect valid questions into an array and perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, wrap all calls in Promise.all() after validating the array. Add a sanity cap (e.g. max 100 questions) before the insert to prevent runaway payloads.

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** ticker is declared notNull and unique by business logic but has no unique constraint or index in the schema. Duplicate tickers can be inserted, corrupting the simulation and causing incorrect portfolio valuation lookups.

**Fix:** Add `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` to the table definition. This enforces uniqueness at the DB level and doubles as the lookup index for ticker-based queries.

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** There is no unique constraint on (userId, stockId). A user can accumulate multiple rows for the same stock instead of a single holding being updated, causing incorrect quantity and average-price calculations in the simulation.

**Fix:** Add `(t) => ({ userStockUnique: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) })`. Upsert logic in the application layer should then use ON CONFLICT (user_id, stock_id) DO UPDATE to increment quantity and recalculate avgPurchasePrice.

### `server/index.ts`: app.use logging middleware / res.json patch
**Issue:** Every API response body is serialised with JSON.stringify and written to stdout unconditionally. For a financial literacy app with school-aged users this risks logging PII, session tokens, or grade data in plaintext. The patched res.json closure also keeps a reference to capturedJsonResponse alive for the lifetime of the request even when the response is large.

**Fix:** Add a response-body allowlist: only log bodies for non-2xx responses or log a fixed set of safe fields (e.g. { status, code } only). Gate the feature behind a DEBUG env flag: `if (process.env.LOG_RESPONSE_BODIES === 'true' && !path.startsWith('/api/auth'))`. Clear capturedJsonResponse after the finish handler fires to release the reference.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest / maybeRunAiUsagePurge
**Issue:** The interval handles are stored in variables (or returned) only after the async startup IIFE resolves. If process shutdown is signalled between server listen and interval assignment the handles are never cleared, leaking both intervals. Additionally neither scheduler awaits its own previous invocation, so if the DB is slow a second tick can overlap the first, causing duplicate job enqueues that bypass the conflict-guard timing window.

**Fix:** Declare `let digestInterval: NodeJS.Timeout | undefined` and `let purgeInterval: NodeJS.Timeout | undefined` in module scope before the startup IIFE. Assign immediately inside the IIFE and reference them in the SIGTERM/SIGINT handler with `clearInterval(digestInterval)`. Wrap each scheduler body with a running-flag guard: `let digestRunning = false; if (digestRunning) return; digestRunning = true; try { await maybeRunWeeklyDigest(); } finally { digestRunning = false; }`

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / route guard
**Issue:** The component relies on `useAdminAuth` for access control, but if that hook only redirects after the component mounts and the queries have already fired, sensitive admin data (user PII, school financials) is fetched before the auth check completes and is held in the React Query cache even if the redirect occurs.

**Fix:** Add `enabled: !!isAdmin` (from the auth hook) to every `useQuery` call so no network request is made until admin status is confirmed. On the server, ensure every /api/admin/* route validates the session token and role independently of the client guard.

## 🟡 Warnings (14)

- **`server/routes.ts` (Supabase init chain ~line 115-124):** The chained `.then()` Supabase init sequence means if any single step (e.g. seedStaticModules) throws, all subsequent steps are silently skipped with only a single catch log. Individual step failures are not distinguishable and critical seed steps like seedGameContent may never run.
  → *Run independent seed operations in parallel with individual error isolation: `await Promise.allSettled([seedFinancialAcademyLesson(), initStaticContentTables(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] Seed step', i, 'failed:', r.reason)));` Keep initSupabaseTables and trimOrgNamesInSupabase sequential before this block as they are prerequisite operations.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler ~line 100):** The filePath parameter from the URL is passed directly to objectStorage.searchPublicObject without path traversal sanitization. A request to `/public-objects/../../private/secrets` could potentially resolve outside the intended public directory depending on the ObjectStorageService implementation.
  → *Sanitize the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, '').replace(/^\//, '');` and validate it matches an allowlist pattern such as `if (!/^[a-zA-Z0-9/_.-]+$/.test(filePath)) return res.status(400).json({ error: 'Invalid path' });`*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model and max_completion_tokens)):** The OpenAI model name 'gpt-4o' and token limits (4096) are hardcoded in two separate API calls. Any model change or cost-control adjustment requires a code deploy rather than a config update, and different environments (dev/prod) cannot use different models.
  → *Read these from environment variables: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); Apply the same pattern to the vision call. Document the env vars in .env.example.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText sliced to 30000 chars for PDF path only):** The 30,000-character truncation guard is applied to the PDF branch but not to the vision branch. If the vision model returns an unexpectedly large response (or future code adds another branch), the raw string is passed directly to the second OpenAI call with no length cap, risking token overflows and unpredictable costs.
  → *Apply a single truncation after both branches resolve: extractedText = extractedText.slice(0, 30000); Place this line immediately before the null-check on extractedText.trim() so it covers all extraction paths consistently.*
- **`server/jobHandlers.ts` (admin-csv-export handler (type AdminRow and data assembly block)):** Based on the visible pattern, the CSV export handler likely fetches entire tables into memory as JS objects before serialising to CSV. For a growing Caribbean school platform with many users and transactions, this is an unbounded memory allocation on the job worker process with no streaming or pagination guard.
  → *Use a DB cursor or paginated query (e.g. limit/offset loop or drizzle streaming) and pipe rows directly into a CSV transform stream that writes to object storage in chunks. If the full dataset must be materialised, add a hard row-count check up front and abort with a descriptive error if it exceeds a configured limit (e.g. EXPORT_MAX_ROWS=100000).*
- **`shared/schema.ts` (categories table definition):** categories.userId uses varchar with no length parameter, but more critically there is no index on userId. Every query filtering user-specific categories (e.g. WHERE user_id = $1) will do a full table scan as the category list grows.
  → *Add an index: add a table-level index definition `(t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) })`. Also pin the varchar length to match the auth model, e.g. varchar('user_id', { length: 36 }) if users.id is a UUID.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** createdAt (and purchasedAt/executedAt) columns use .defaultNow() but no .notNull(). This means application code or a failed default can insert a NULL timestamp, making time-based queries and audit trails unreliable.
  → *Chain .notNull() after .defaultNow() on every audit timestamp column, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. Apply consistently across all tables including documentUploads and transactions.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate is the primary filter column for bill reminder queries (finding upcoming bills) but has no index. For a school-aged user base that may grow significantly, range queries like WHERE next_due_date <= NOW() AND user_id = $1 will degrade linearly.
  → *Add a composite index on (userId, nextDueDate): `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })`*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes the most recent Sunday using `new Date()` which resolves in the server's local timezone. On a UTC-based host (Replit containers run UTC) '7 PM Sunday' is UTC, but Caribbean users are UTC-4 or UTC-5, so the digest fires 4-5 hours early relative to the intended local time. The constant 19 is also hardcoded with no environment override.
  → *Accept a timezone parameter and use `Intl.DateTimeFormat` with `timeZone: process.env.DIGEST_TIMEZONE ?? 'America/Port_of_Spain'` to resolve the Sunday-at-19:00 wall-clock moment, then convert back to a UTC Date for comparison. Expose `DIGEST_HOUR` as an env var defaulting to 19.*
- **`server/index.ts` (startup IIFE — seedDatabase / probeDatabase / registerRoutes calls):** seedDatabase and probeDatabase are awaited sequentially before the HTTP server begins listening. If either operation is slow (cold DB connection, large seed) the process is unresponsive to health checks and load-balancer probes for the entire duration, risking premature container restarts. registerRoutes is also awaited in series after both complete.
  → *Start listening immediately after registerRoutes returns, then run seedDatabase in the background with its own try/catch: `httpServer.listen(port, host, () => log('serving')); seedDatabase().catch(err => { captureError(err); log('seed failed: ' + err.message); });` Keep probeDatabase before listen but add a timeout: wrap it in `Promise.race([probeDatabase(), new Promise((_, r) => setTimeout(() => r(new Error('DB probe timeout')), 5000))])` so a hung DB does not block startup indefinitely.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest — missing SIGTERM handler for job worker):** There is no visible SIGTERM or SIGINT handler that calls a stop method on startJobWorker, clears the scheduler intervals, or drains in-flight jobs before exit. On Replit or any container platform a rolling deploy sends SIGTERM; without a handler the process exits immediately, potentially interrupting a job mid-write and leaving the jobs table row in a 'running' state permanently.
  → *Add: `async function shutdown(signal: string) { log('shutdown: ' + signal); clearInterval(digestInterval); clearInterval(purgeInterval); await stopJobWorker(); httpServer.close(() => process.exit(0)); } process.once('SIGTERM', () => shutdown('SIGTERM')); process.once('SIGINT', () => shutdown('SIGINT'));` Implement stopJobWorker() to set a stop flag and await a Promise that resolves when the current job handler settles.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The /api/admin/help-chat POST sends the entire accumulated messages array on every turn with no length cap. A user who keeps chatting will send an ever-growing payload, inflating token cost and risking hitting API body-size or LLM context limits silently.
  → *Slice the history before sending: `const context = next.slice(-20);` then POST `{ messages: context }`. Add a hard cap (e.g., 40 messages) that trims the local state when exceeded so memory also stays bounded.*
- **`client/src/pages/AdminDashboard.tsx` (Messages scroll div ref callback):** `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a new anonymous function on every render of the messages list, causing the browser to detach and re-attach the ref node reference every render, which also triggers an unwanted scrollIntoView on every state update unrelated to new messages.
  → *Replace with a `useRef` + `useEffect`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` Then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (useMutation callbacks / table row handlers):** Mutation trigger functions (delete, edit, status-toggle) are defined as inline arrow functions directly in JSX map callbacks (e.g., `onClick={() => deleteMutation.mutate(row.id)}`). With large user/school lists this recreates a new function reference for every row on every parent re-render, causing all rows to re-render even when nothing changed.
  → *Wrap each handler with `useCallback` keyed on the row id, or extract the row into a memoised `<TableRow>` sub-component that receives stable callback props so React can bail out via `React.memo`.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six sequential awaits. If any registerXxxRoutes functions perform async I/O during registration (e.g. DB schema checks), this adds unnecessary serial latency. Refactor to `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` assuming routes don't depend on each other's registration order.

### Structured startup logging for Supabase init: ⚡ Medium Impact
The current init chain logs a single error string on failure. Adding timing and per-step success logs (e.g. `console.info('[Supabase] seedGameContent completed in ${ms}ms')`) would dramatically reduce debugging time during Railway deployments and make it clear which seed step is slow or failing in production.

### APP_BASE_URL environment config for canonical URL generation: 🔥 High Impact
Beyond fixing the host-header injection, centralizing the base URL in APP_BASE_URL enables consistent canonical URL generation across robots.txt, sitemap.xml, email templates, and OG tags without per-handler logic. Expected impact: eliminates an entire class of environment-specific URL bugs when deploying to staging vs production.

### Idempotency token for question deletion before re-insert: 🔥 High Impact
The current idempotency guard deletes all extracted questions and re-runs both expensive OpenAI calls on every retry. Store a content hash of the extracted text in the job result or on the exam_papers row; on retry, skip the OpenAI extraction calls if the hash matches and only redo the DB insert. This halves OpenAI costs on transient failures and makes retries significantly faster.

### Centralised OpenAI call wrapper with usage logging and timeout: 🔥 High Impact
Both openai.chat.completions.create calls in runExtractPaper lack an explicit timeout or AbortSignal. A stalled OpenAI request will hang the worker indefinitely. A thin wrapper that attaches AbortSignal.timeout(60_000), logs token usage to aiUsageEvents, and standardises error mapping would improve reliability across all handlers and give accurate cost tracking.

### Structured payload validation at handler entry: ⚡ Medium Impact
Job payloads are destructured and used directly without runtime validation (e.g. zod or a simple guard). A malformed enqueue call (wrong types, missing fields) will cause cryptic runtime errors deep inside a handler rather than a clear failure at the boundary. Adding per-handler zod schemas that parse job.payload at the top of each registerJobHandler callback would surface bad data immediately and protect all downstream logic.

### Centralise currency default via config constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least five tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions). Extract it to a shared constant DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' and reference it in each column default. This allows the app to support other Caribbean currencies (TTD, JMD) without a schema migration.

### Add partial index on documentUploads for pending processing jobs: ⚡ Medium Impact
Background workers that poll for unprocessed uploads query WHERE status = 'processing'. A partial index `CREATE INDEX idx_doc_uploads_processing ON document_uploads (user_id) WHERE status = 'processing'` would make this near-instant regardless of table size, and shrinks automatically as rows transition to completed/failed.

### Replace text columns for enums with Postgres native enums or check constraints: 📌 Low Impact
Columns like transactions.type, budgets.period, and billReminders.frequency use unconstrained text with Drizzle-level enums only. If rows are inserted outside the ORM (migrations, seed scripts, admin tools), invalid values can enter the database silently. Define these as pgEnum types or add DB-level CHECK constraints to enforce integrity at the storage layer.

### Parallel startup initialisation: ⚡ Medium Impact
registerJobHandlers and probeDatabase are independent of each other. Awaiting them with Promise.all would shave wall-clock startup time equal to the slower of the two minus the faster, which on a cold DB connection can be several hundred milliseconds. Apply the same pattern to any other independent async init steps discovered in the full file.

### Structured JSON logging: 🔥 High Impact
Replacing console.log/the custom log() function with a structured logger (pino at ~5 µs/log vs ~50 µs for util.format) would give consistent log levels, redaction rules for PII fields, and machine-parseable output for aggregation in Replit's log viewer or a future SIEM. This is especially important given the school-aged user base and likely future COPPA/data-protection obligations.

### Scheduler extraction into a dedicated module: ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, getMostRecentSundayAt7pm, and their interval setup into a schedulers.ts module with explicit start()/stop() exports would make unit-testing the overlap-guard and timezone logic straightforward without spinning up the full Express server, and would naturally enforce the interval-handle lifecycle fix.

### Memoize filtered/sorted table data: ⚡ Medium Impact
Search and filter operations over the users and schools arrays are likely recomputed on every keystroke re-render. Wrapping them in `useMemo` keyed on the raw data array and the filter string would cut redundant iterations and keep the UI responsive as data grows, with no architectural change required.

### Move chart data transforms into useMemo: ⚡ Medium Impact
Recharts data arrays built inline from raw query results (aggregations, date groupings) are reconstructed on every render. Memoising them avoids re-running O(n) reduce/map passes and prevents Recharts from treating new array references as data changes, which causes unnecessary animation resets.

### Introduce a shared ErrorBoundary and skeleton states: 🔥 High Impact
The dashboard renders many parallel queries but mixing loading booleans inline produces inconsistent UX when any single query fails. Wrapping each dashboard section in an ErrorBoundary with a consistent skeleton loader would prevent a single failed analytics call from leaving the whole page in a broken half-rendered state, important for school admin trust.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good patterns like parallel dynamic imports and try/catch on async endpoints. The main concerns are unhandled promise rejections on Supabase init swallowing individual step failures silently, missing rate limiting on AI/heavy endpoints that are exposed to school-aged users, and a host-header injection risk in robots.txt and sitemap.xml that could poison crawlers with attacker-controlled URLs.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and hardcoded model/token values that belong in environment config. The extract-paper handler is the highest-risk area due to payload size and per-question sequential awaits inside a loop.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with proper foreign keys and some indexes already defined, but has several reliability and performance gaps. Key concerns are missing NOT NULL constraints on audit timestamps, absent indexes on high-frequency filter columns, and undersized varchar types for user_id foreign keys that could cause silent truncation if the auth model uses UUIDs.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability and security concerns including a blocking synchronous-style database probe before the HTTP server is ready, unguarded interval schedulers that can leak if shutdown is called before they are assigned, and response-body logging that risks exposing sensitive financial or PII data in plaintext logs. Graceful shutdown handling for the job worker and scheduler intervals appears incomplete based on the visible code.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security gaps including missing auth guards on sensitive mutations, uncontrolled re-renders from inline JSX functions, and a scroll ref callback that creates a new function on every render. The chat panel sends the full growing message history to the backend on every turn without any size cap, creating a potential denial-of-service vector. Overall structure is reasonable but needs hardening before production use with school-aged users.
