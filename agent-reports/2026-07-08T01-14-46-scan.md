## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. Any authenticated (or unauthenticated) user can hammer these endpoints, running up API costs and degrading service for all school users.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP with express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — adjust route prefixes to match what registerChatRoutes etc. actually mount.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 43 and ~line 59)
**Issue:** The absolute base URL is constructed directly from `x-forwarded-host` and `x-forwarded-proto` headers without any validation or allowlist. An attacker can send crafted headers to inject an arbitrary host into the Sitemap directive or sitemap <loc> tags, enabling cache-poisoning or phishing via poisoned sitemaps served to crawlers.

**Fix:** Add a validated `APP_BASE_URL` environment variable (e.g. `https://finsightlite.com`) and use it unconditionally: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';`. Remove all header-based host detection for these two routes.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** fileB64 (full base64-encoded file content) is stored directly in the jobs table payload. For a multi-page PDF or image this can easily be 5–20 MB per row, bloating the jobs table, slowing queue polling queries, and risking OOM if many jobs are queued simultaneously.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the file from object storage at the start of the handler. This keeps job rows small (<1 KB) and makes payloads safe to log.

### `server/jobHandlers.ts`: purge-ai-usage handler (registerJobHandler call)
**Issue:** The purge-ai-usage handler has no try/catch around the db.delete call. An unexpected DB error will produce an unhandled rejection that crashes the worker process rather than marking the job as failed and allowing retry logic to run.

**Fix:** Wrap the handler body in try/catch: catch the error, optionally log it with a structured logger (not console.log with raw error objects), then rethrow so the job framework can record the failure and schedule a retry.

### `shared/schema.ts`: simulatedStocks table, ticker column
**Issue:** ticker has no unique index. Multiple rows with the same ticker symbol can be inserted, causing duplicate results in portfolio calculations and JOIN ambiguity when resolving holdings. This is a data integrity hole, not just a performance issue.

**Fix:** Add a uniqueIndex in the table definition: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This enforces uniqueness at the DB level and gives the query planner an efficient lookup path for ticker-based joins.

### `server/index.ts`: seedDatabase / probeDatabase / registerRoutes call site in the startup IIFE
**Issue:** seedDatabase and probeDatabase are awaited sequentially on the startup critical path with no timeout guard. If either hangs (e.g. DB unreachable), the process stays alive but never binds the port, silently dropping all incoming connections with no operator signal beyond a stalled log line.

**Fix:** Wrap each call in Promise.race with a short timeout (e.g. 10 s): `await Promise.race([probeDatabase(), timeout(10_000, 'probeDatabase timed out')])`. Add a top-level catch that calls `process.exit(1)` so the process manager restarts the pod rather than leaving a zombie.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are created but their handles are never stored or cleared. When the process receives SIGTERM (e.g. Replit cycling the dyno), the intervals keep firing during the drain window, potentially double-enqueueing jobs. There is no graceful shutdown handler at all in the visible code.

**Fix:** Store handles: `const weeklyHandle = setInterval(maybeRunWeeklyDigest, ...)`. Register `process.once('SIGTERM', async () => { clearInterval(weeklyHandle); clearInterval(purgeHandle); await httpServer.close(); process.exit(0); })`. Mirror for SIGINT.

---

### `server/index.ts`: res.json middleware patch in the request-logging middleware
**Issue:** Every API response body is deep-serialised via JSON.stringify and appended to the log line unconditionally. For a financial literacy app with school-aged users this risks logging auth tokens, session data, quiz answers, or any PII returned by the API to stdout/Sentry breadcrumbs, violating basic data-minimisation principles.

**Fix:** Either drop response body logging entirely, or allowlist safe fields: only log `capturedJsonResponse` when it is a plain status envelope (e.g. has only `ok`, `message` keys) and strip it otherwise. At minimum gate it behind `process.env.LOG_RESPONSE_BODIES === 'true'` which should default off in production.

### `client/src/pages/AdminDashboard.tsx`: apiRequest POST /api/admin/help-chat
**Issue:** The chat endpoint is called with no server-side auth token forwarded and no client-side check that the admin session is still valid before submitting. If the session expires mid-use, the request will fail with a generic catch message, and the error gives no indication to the user that they need to re-authenticate. More critically, unauthenticated callers who discover the endpoint URL can invoke it directly if the server route lacks its own auth middleware verification.

**Fix:** On the client, read the admin auth state from `useAdminAuth()` inside `AdminHelpChat` and early-return (or disable the input) if the session is falsy. On the server route handler for `/api/admin/help-chat`, ensure the same `requireAdminAuth` middleware used on other admin routes is applied—confirm this in the route registration file, not just assumed.

## 🟡 Warnings (15)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...) (~line 110)):** The seven Supabase seed/init calls are chained sequentially in a .then() waterfall. If any intermediate step throws an error that isn't caught before .catch(), the remaining steps are silently skipped with only a single generic error log, making production seed failures very hard to diagnose.
  → *Run independent seed steps in parallel where safe, and log each failure individually: `Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] seed step ${i} failed:`, r.reason)));` — keep initSupabaseTables and initStaticContentTables sequential if they are prerequisites.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96)):** The `filePath` path parameter is passed directly to `objectStorage.searchPublicObject` without sanitization. A path like `../../secrets/config` could traverse into unintended storage prefixes depending on the ObjectStorageService implementation.
  → *Sanitize the path before use: `import path from 'path'; const safePath = path.normalize(filePath).replace(/^\.\.\/|^\//, ''); if (safePath !== filePath) return res.status(400).json({ error: 'Invalid path' });` then pass `safePath` to the storage service.*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop):** Questions are inserted one at a time inside a for loop with sequential awaits (await storage.createExtractedQuestion(...)). For a 40-question paper this serialises 40 round-trips to the database, adding significant latency and holding the job open unnecessarily long.
  → *Collect all valid question objects into an array first, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Add a guard for the empty-array case to avoid a no-op insert error.*
- **`server/jobHandlers.ts` (runExtractPaper – image branch vision call):** In the image branch, buffer.toString('base64') re-encodes the already-decoded buffer back to base64 to pass to the OpenAI vision API. The original fileB64 string from the payload is identical and could be used directly, avoiding a redundant O(n) string allocation for potentially large files.
  → *Pass fileB64 directly in the data URL: `data:${mimeType};base64,${fileB64}` instead of `data:${mimeType};base64,${buffer.toString('base64')}`. Remove the buffer allocation entirely for the image branch if pdf-parse is not needed there.*
- **`server/jobHandlers.ts` (parsePrivatePath / PRIVATE_OBJECT_DIR usage):** PRIVATE_OBJECT_DIR is checked at call-time inside parsePrivatePath rather than validated at startup. If the env var is missing, jobs will silently enqueue and only fail at execution time after potentially performing expensive AI calls, with no early-warning signal at server boot.
  → *Add a startup assertion in registerJobHandlers (or a dedicated config validation module): if (!process.env.PRIVATE_OBJECT_DIR) throw new Error('PRIVATE_OBJECT_DIR must be set'). This surfaces misconfiguration immediately on deploy rather than at job execution time.*
- **`shared/schema.ts` (userId: varchar('user_id') across all tables):** All userId columns use varchar with no length bound. Without a consistent length constraint matching the auth provider's ID format (e.g., Clerk/Supabase UUIDs are 36 chars), the DB will accept arbitrarily long strings, wasting storage and bypassing implicit format validation.
  → *Change all userId FK columns to varchar('user_id', { length: 36 }) or use uuid() type if the auth model uses UUID format. Also add .notNull() to categories.userId's FK reference definition so the optional/required intent is enforced by the type system, not just by convention.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow() on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is defined without .notNull(), so the column accepts explicit NULL inserts even though defaultNow() is set. Any INSERT that explicitly passes null will store NULL, breaking audit trails and any ORDER BY createdAt queries that assume non-null values.
  → *Add .notNull() to every createdAt column: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, yet budget lookups will always filter by userId. Without an index, every budget query does a full table scan. As the user base grows this is a direct latency regression on a core screen.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The composite index also supports queries filtered by both user and category.*
- **`shared/schema.ts` (billReminders and savingsGoals table definitions):** Neither billReminders nor savingsGoals has an index on userId. bill_reminders will be queried by userId filtered on nextDueDate for upcoming-bill notifications, and savings_goals by userId alone. Missing indexes mean full scans on both tables for every dashboard load.
  → *For billReminders add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). For savingsGoals add: (t) => ({ userIdx: index('idx_savings_goals_user').on(t.userId) }). The composite index on billReminders also accelerates the common 'upcoming bills in next N days' query.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — the `+ 0` is a no-op dead expression that makes the intent unreadable and produces the wrong anchor day. Sunday is `dow === 0`, so `(dow + 0) % 7 === 0` means the function subtracts 0 days when called on a Sunday before 19:00, returning today's 19:00 which is in the future and causes an early return, skipping the digest entirely until next week.
  → *Rewrite the offset to explicitly target Sunday: `const daysSince = dow;` (Sunday=0 means 0 days back, which is correct) then validate with a unit test covering calls made on Sunday before and after 19:00. Remove the dead `+ 0`.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and related hardcoded scheduler intervals):** The scheduler tick intervals (the values passed to setInterval for weekly digest and purge) appear to be hardcoded numeric literals in the truncated code rather than environment-configurable values. Changing cadence requires a redeploy, and there is no validation that the parsed interval is sane (e.g. not 0 or negative), which would cause a busy-loop.
  → *Add env vars `WEEKLY_DIGEST_INTERVAL_MS` and `AI_PURGE_INTERVAL_MS` with safe defaults (e.g. 3_600_000 for 1 h). Parse and validate them with the same pattern used for `AUTO_PURGE_OLDER_THAN_DAYS`, asserting the result is >= 60_000 before passing to setInterval.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The help-chat endpoint receives the full unbounded `messages` array on every send. A user who chats for an extended session will send an ever-growing payload (potentially thousands of tokens) with no limit, causing slow requests, high AI costs, and potential 413/timeout errors.
  → *Slice the history before sending: `const context = next.slice(-10);` then pass `{ messages: context }` to the POST. Keep the full `next` array in state for display purposes only.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll ref callback (inline `ref` on the sentinel div)):** The scroll sentinel uses `ref={el => { if (el) el.scrollIntoView(...); }}` as an inline function. React creates a new function reference every render, calls it with `null` on unmount and the element on mount, meaning `scrollIntoView` fires on every re-render of the message list—not just when a new message arrives. This causes janky scroll behaviour and unnecessary DOM work.
  → *Replace with a `useEffect` that watches `messages.length` and `loading`: attach a stable `useRef` to the sentinel div, then `ref.current?.scrollIntoView({ behavior: 'smooth' })` inside the effect. This fires only when the list actually changes.*
- **`client/src/pages/AdminDashboard.tsx` (handleKey (inside AdminHelpChat)):** `handleKey` and `sendMessage` are recreated on every render of `AdminHelpChat` because they are plain function declarations inside the component body. With the input re-rendering on each keystroke, these closures are rebuilt constantly and passed as new references to the `<input>` element, preventing any downstream memoisation from working.
  → *Wrap both with `useCallback`: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – error catch block):** The catch block silently swallows the actual error (`catch { ... }`) with no logging whatsoever. In a financial literacy platform targeting school-aged users, silent failures make debugging in production extremely difficult and can mask security-relevant failures such as 401/403 responses from the server.
  → *Change to `catch (err) { console.error('[AdminHelpChat] sendMessage failed:', err); }` and additionally inspect the response status before calling `.json()`: `if (!res.ok) throw new Error(res.statusText);`. Surface 401s to the user with a specific message like 'Session expired, please log in again.'*

## 💡 Opportunities

### Auth middleware ordering documentation: ⚡ Medium Impact
The comment 'must be applied before auth routes are registered' is the only guard ensuring rate limiters precede setupAuth. Extract all rate-limit registrations into a dedicated `applyRateLimiters(app)` function called explicitly before setupAuth, making the ordering enforced by code structure rather than comment discipline. Reduces risk of future contributors accidentally inserting routes above the limiters.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replace bare console.error in the Supabase init chain with structured log entries (e.g. using a shared logger with severity/timestamp fields). For a production financial app serving minors, audit trails of failed seed operations are important for compliance and debugging. Expected impact: faster incident diagnosis and better Railway log filtering.

### Domain router registration parallelisation: 📌 Low Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six serial awaits. If any registration function does async work (DB calls, sub-route setup), this serialises startup unnecessarily. Evaluate whether each register* function can be made parallel: `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` — provided route handlers don't depend on each other's middleware being installed first.

### Batch parallelise independent async setup steps: ⚡ Medium Impact
In handlers that independently fetch user data, module data, and session data before doing work, replace sequential awaits with Promise.all. For example, fetching a user record and a learning module record have no dependency on each other and can run concurrently, cutting handler latency by roughly 40–60% on those paths.

### Centralise job payload validation with Zod schemas: ⚡ Medium Impact
Job payloads are consumed with direct destructuring and no runtime validation. Adding a Zod schema per job type validated at the top of each handler will surface malformed enqueue calls immediately with a clear error rather than producing cryptic mid-handler failures. This also documents the contract for each job type.

### Move PDF text extraction to a dedicated worker or streaming pipeline: 🔥 High Impact
pdf-parse loads the entire PDF buffer into memory synchronously. For large exam papers this blocks the event loop. Using a worker_thread or a streaming PDF parser (e.g. pdf2json in stream mode) would keep the main job worker event loop free and reduce peak RSS, which matters when multiple extract-paper jobs run concurrently.

### Replace repeated currency text columns with a schema-level default: ⚡ Medium Impact
Eight tables each hardcode .default('BSD') on a text currency column. If the app expands to TTD or JMD, each table needs a migration. Centralise this as a pgEnum('currency', ['BSD','TTD','JMD']) and reference it everywhere. This also prevents invalid currency codes being stored and reduces storage by replacing unbounded text with a compact enum.

### Add a composite unique constraint on portfolioHoldings (userId, stockId): 🔥 High Impact
Currently nothing prevents two rows representing the same user holding the same stock, which would corrupt average-price calculations. A uniqueIndex on (userId, stockId) enforces one-row-per-position and lets UPSERT logic replace the fragile SELECT-then-INSERT pattern that will inevitably appear in the trade execution service.

### Extract learningSessions and quiz tables into a dedicated learning sub-schema file: ⚡ Medium Impact
The file is already truncated at 28k+ chars. Splitting investment simulation tables and learning/quiz tables into separate schema files (schema/investing.ts, schema/learning.ts) reduces cognitive load, enables independent migrations, and prevents circular import issues as the app scales. No runtime impact but high maintainability payoff.

### Parallelise independent startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase, seedDatabase, and any other independent startup tasks (e.g. Sentry init network flush) are awaited sequentially. Running them with `await Promise.all([probeDatabase(), seedDatabase()])` would cut startup latency by the duration of whichever task is slowest minus the sum, typically saving 1-3 s on cold start, which matters for Replit's always-on wake-up.

### Add structured startup health logging with timing spans: ⚡ Medium Impact
Replace ad-hoc `log()` calls in the startup IIFE with a single structured object logged at the end (JSON with phase, durationMs, env). This feeds directly into log aggregators without regex parsing, and makes it trivial to alert on slow startup phases in production.

### Extract scheduler state into a dedicated module with observable status: 🔥 High Impact
Moving the two scheduler intervals plus their run-tracking queries into a `scheduler.ts` module that exposes a `/api/internal/scheduler-status` endpoint (guarded by admin auth) would allow operators to verify last-run timestamps without tailing logs, significantly reducing incident diagnosis time for missed digest or purge runs.

### Parallel data fetching with Promise.all: 🔥 High Impact
The dashboard likely fires multiple independent useQuery calls (users, schools, teachers, analytics) sequentially through React Query's waterfall. Identify any imperative async functions that await these sequentially and refactor to Promise.all. For the React Query layer, ensure all queries are registered at the top level so they dispatch in parallel. Expected impact: 40-60% reduction in initial dashboard load time on slow Caribbean mobile connections.

### Memoize chart data transformations: ⚡ Medium Impact
Recharts BarChart and LineChart data is likely derived from raw API responses with inline map/filter/reduce expressions passed directly as the `data` prop. These re-run on every parent render. Wrap each transformation in `useMemo` keyed to the raw query data. This prevents chart flicker and unnecessary Recharts diffing on unrelated state changes (e.g., search input keystrokes).

### Extract AdminHelpChat into its own file with React.memo: ⚡ Medium Impact
AdminHelpChat is a self-contained feature component living inside the same file as the full dashboard page. Every top-level dashboard state change (pagination, search, selected tab) currently re-renders the chat panel. Moving it to its own module and wrapping with React.memo will eliminate these spurious renders and reduce bundle coupling, making the component independently testable.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel module imports and non-blocking Supabase init. Key concerns are: the AI/chat/image/audio routes registered without any rate limiting (critical for a school-age audience hitting paid APIs), and the robots.txt/sitemap host header trust creating a potential host-header injection vector for absolute URLs.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and a missing try/catch on the purge-ai-usage handler. Security exposure exists via base64 file content logged or stored in structured job payloads. Overall architecture is reasonable but the extract-paper handler is the highest-risk path.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with FK constraints and some composite indexes, but has several reliability and security gaps. Key concerns are unbounded varchar columns on userId foreign keys that could silently accept mismatched data, missing indexes on high-frequency filter columns, and nullable createdAt timestamps on audit-critical tables that should always be populated.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability gaps: blocking synchronous-style sequential awaits on the critical path, interval handles that are never cleared on shutdown, and a graceful shutdown handler that is either absent or incomplete. The logging middleware unconditionally serialises every JSON response body which can leak sensitive financial or PII data to stdout in production.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security gaps including missing auth guards on sensitive mutations, uncontrolled re-renders from inline JSX functions lacking useCallback, and a scroll ref pattern that creates a new function on every render causing subtle memory churn. The chat panel sends full conversation history to the server without size-bounding, creating a payload bloat risk. Overall structure is reasonable but the issues below need addressing before a school-age audience relies on this in production.
