## 🔴 Critical (11)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: after fetching all budgets for a user, the code issues one additional query per budget row to calculate 'spent' (sum of transactions for that category), resulting in O(n) round-trips to the database for n budgets.

**Fix:** Replace the per-budget loop with a single aggregating JOIN or subquery using Drizzle's sql helper: SELECT budgets.*, COALESCE(SUM(transactions.amount),0) AS spent FROM budgets LEFT JOIN transactions ON transactions.category_id = budgets.category_id AND transactions.user_id = budgets.user_id GROUP BY budgets.id. Return the result set directly, eliminating all per-row queries.

### `server/storage.ts`: getPortfolioHoldings / getPortfolioTransactions
**Issue:** N+1 lookup pattern: each holding or portfolio-transaction row triggers an individual getStockById(holding.stockId) call inside a map, issuing one SELECT per row instead of a single IN-clause fetch.

**Fix:** Collect all distinct stockIds first, then fetch them in one query: db.select().from(simulatedStocks).where(inArray(simulatedStocks.id, stockIds)). Build a Map<id, stock> and attach stock data in-memory. This reduces round-trips from O(n) to O(1).

### `server/storage.ts`: upsertPortfolioHolding + createPortfolioTransaction + updateVirtualBalance (called together in trade handler)
**Issue:** Multi-step write operations (update holding, insert portfolio transaction, debit virtual balance) are executed without a database transaction. A failure mid-sequence leaves the database in an inconsistent state — e.g. balance debited but holding not updated.

**Fix:** Wrap all three operations in a single Drizzle transaction: await db.transaction(async (trx) => { await upsertHolding(trx, ...); await insertPortfolioTx(trx, ...); await updateBalance(trx, ...); }). Refactor each sub-function to accept an optional transaction context parameter so they can operate inside or outside a transaction.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 115-117)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. For a school-aged user base this creates both runaway cost exposure and abuse vectors — a single user or bot could exhaust API quota.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g., 20 req/min per IP via express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerX calls.

### `server/routes.ts`: robots.txt handler (~line 45) and sitemap handler (~line 57)
**Issue:** The `host` and `x-forwarded-host` headers are taken directly from the request and interpolated into URLs without any validation. An attacker can send a spoofed Host header and cause the sitemap to emit an arbitrary base URL, enabling cache-poisoning or phishing via poisoned search-engine indexes.

**Fix:** Introduce an `APP_BASE_URL` environment variable (e.g., `https://finsightlite.com`) and use it as the canonical base: `const base = process.env.APP_BASE_URL || 'https://finsightlite.com';` Remove all header-derived host construction from both handlers.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** Large base64 file content (fileB64) is stored directly in the job payload. A single PDF or image upload can easily be 5–20 MB encoded as base64, bloating the jobs table row, increasing memory pressure when the worker deserializes the job, and making retries re-transmit the full payload.

**Fix:** Before enqueuing, upload the raw file buffer to object storage and store only the resulting object path in the job payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient before processing. This caps the job row at a few hundred bytes.

### `shared/schema.ts`: simulatedStocks ticker column
**Issue:** simulatedStocks.ticker has no unique index. Duplicate tickers can be inserted, causing silent data integrity bugs when portfolio holdings are joined or looked up by ticker symbol.

**Fix:** Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table's index callback. If multi-region stocks with the same ticker are needed in future, scope it as a composite unique on (ticker, region).

---

### `server/index.ts`: setInterval calls (weeklyDigest and aiUsagePurge schedulers)
**Issue:** The intervals returned by setInterval for maybeRunWeeklyDigest and maybeRunAiUsagePurge are never stored and never cleared. On graceful shutdown (SIGTERM/SIGINT) these timers keep the Node event loop alive and can fire after the DB connection is already closed, causing unhandled rejections or silent data corruption.

**Fix:** Capture both interval handles: `const digestInterval = setInterval(...); const purgeInterval = setInterval(...);`. In the SIGTERM/SIGINT handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before closing the HTTP server and DB pool.

---

### `server/index.ts`: res.json override in request-logging middleware
**Issue:** Every API response body is captured into capturedJsonResponse and serialised into the log line unconditionally. For a financial literacy app with school-aged users this will log tokens, session data, quiz answers, and any PII returned by any endpoint, violating basic data-minimisation principles.

**Fix:** Either remove response-body logging entirely, or allowlist only non-sensitive fields: log only res.statusCode and duration. If debugging payloads is needed in development, gate it behind `if (process.env.NODE_ENV !== 'production')` and redact known sensitive keys before serialising.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat.sendMessage
**Issue:** The entire message history array is sent to /api/admin/help-chat on every message, growing unbounded. A user who sends 100 messages sends 100 messages worth of tokens each time, causing runaway API costs and potential request size rejections.

**Fix:** Slice the history before sending: `const context = next.slice(-10)` and send `{ messages: context }` instead of the full `next` array. Add a hard cap: if `messages.length > 200` trim the oldest entries from state as well.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level dashboard render
**Issue:** If useAdminAuth returns a non-admin or unauthenticated state, the component likely renders briefly before any redirect fires, exposing admin UI structure, user counts, and financial data in the DOM. There is no explicit early-return guard visible before data-fetching hooks are called.

**Fix:** Immediately after calling `useAdminAuth`, add: `if (!adminUser) return null;` (or a redirect) before any useQuery hooks execute. This prevents queries from firing and prevents any admin data from reaching the DOM for unauthenticated users.

## 🟡 Warnings (18)

- **`server/storage.ts` (getTransactions / getSavingsGoals / getGameSessions):** Missing pagination guards on unbounded list queries. Any query that fetches all rows for a userId with no LIMIT enforced server-side can return arbitrarily large result sets, exhausting memory and slowing responses as user data grows.
  → *Add a mandatory default limit (e.g. 100) and enforce a maximum cap (e.g. 500) in the function signature: limit = Math.min(filters?.limit ?? 100, 500). Pass .limit(limit).offset(filters?.offset ?? 0) to every Drizzle query that returns a list. Surface total-count via a COUNT(*) subquery so callers can paginate.*
- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregates (total income, total expenses, savings rate, category breakdown) that have no data dependency on each other, adding cumulative latency equal to the sum of all query times instead of the slowest single query.
  → *Replace sequential awaits with Promise.all: const [income, expenses, categoryTotals] = await Promise.all([fetchIncome(userId, filters), fetchExpenses(userId, filters), fetchCategoryBreakdown(userId, filters)]). Each query is independent and can execute concurrently on separate DB connections from the pool.*
- **`server/routes.ts` (Supabase init chain (~line 120-128)):** The seven-step Supabase init chain uses sequential `.then()` calls, so each step runs only after the previous one resolves. If `seedFinancialAcademyLesson` is slow, `seedGameContent` is delayed by the sum of all prior steps. Additionally, a failure in any middle step (e.g., `seedStaticModules`) silently skips all subsequent seeds with only a single catch log — there is no per-step error visibility.
  → *Split into two phases: parallel independent seeds and any truly sequential dependencies. Wrap each call with its own catch for granular logging: `await Promise.all([initSupabaseTables(), initStaticContentTables()]).catch(e => console.error('[Supabase] Table init failed:', e)); await Promise.all([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent(), trimOrgNamesInSupabase()]).catch(e => console.error('[Supabase] Seed failed:', e));`*
- **`server/routes.ts` (/healthz handler (~line 80)):** The `/healthz` endpoint uses dynamic `import()` inside the request handler on every call. While Node.js caches modules after the first import, this pattern is fragile and adds unnecessary async overhead on the hot path of an uptime-monitor endpoint.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The healthz handler then becomes a simple synchronous reference.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time in a sequential for loop. For a 40-question paper this fires 40 sequential round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Replace the for loop with a single bulk insert: filter valid questions first, then call db.insert(extractedQuestions).values(validRows) where validRows is the mapped array. Most Drizzle adapters support multi-row insert in one statement. This collapses 40 round-trips into one.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays comes directly from job.payload with no validation before being multiplied into a large millisecond value. A malformed or malicious payload (e.g., olderThanDays = 0 or negative) would delete all rows or produce an epoch-era cutoff date silently.
  → *Add an explicit guard: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays value'); before computing the cutoff. Also log the cutoff date at info level before executing the delete so operators have an audit trail.*
- **`server/jobHandlers.ts` (runExtractPaper — vision branch, image_url content):** The image is re-encoded to base64 from the already-decoded buffer (buffer.toString('base64')) and embedded in the OpenAI API request body. This doubles memory usage for image files since both the decoded Buffer and the re-encoded string exist simultaneously, and the full image bytes travel through the OpenAI client as a single string in the request body.
  → *Pass the original fileB64 string directly rather than decoding to Buffer then re-encoding: use fileB64 (the raw payload string) in the data URL. This eliminates the redundant decode/encode cycle. With the object-storage fix above, download the buffer once and keep a single base64 reference for the API call.*
- **`server/jobHandlers.ts` (openai import — 'import { openai } from "./replit_integrations/chat/routes"'):** The OpenAI client is imported from a routes module, creating a tight coupling between the job handler layer and the HTTP route layer. If that routes module registers Express middleware or emits side effects on import (common in route files), importing it in a worker context can cause unintended behaviour, duplicate middleware registration, or circular dependency issues.
  → *Extract the OpenAI client instantiation into a dedicated shared module (e.g., ./lib/openaiClient.ts) that exports only the configured client. Both the routes file and jobHandlers.ts import from that shared module. This eliminates the coupling and makes the dependency graph explicit.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Every budget lookup by user (the dominant query pattern) requires a full table scan as the dataset grows.
  → *Add a composite index in the table's index callback: index('idx_budgets_user_category').on(t.userId, t.categoryId). This mirrors the pattern already used on transactions and will cover both per-user budget lists and category-specific budget checks.*
- **`shared/schema.ts` (billReminders table definition):** billReminders.nextDueDate has no index. The primary use-case for this table is querying upcoming due dates (e.g. WHERE next_due_date <= NOW()), making a full scan on every reminder poll expensive.
  → *Add index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) in the table's index callback. This supports both per-user filtering and date-range queries efficiently.*
- **`shared/schema.ts` (transactions and savingsGoals createdAt columns):** createdAt uses .defaultNow() but lacks .notNull(), meaning application code or a DB-level bug can insert a NULL createdAt, silently breaking any audit trail, ordering, or time-series query that depends on this column.
  → *Change all createdAt definitions to timestamp('created_at').defaultNow().notNull(). Apply the same fix to linkedCards, documentUploads, budgets, portfolioHoldings, and portfolioTransactions createdAt columns for consistency.*
- **`shared/schema.ts` (currency default values across all tables):** The currency default 'BSD' is hardcoded in at least six table definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, etc.). If the app expands to other Caribbean territories (TTD, JMD, BBD), every table definition requires a schema migration instead of a config change.
  → *Define a module-level constant DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' and reference it as .default(DEFAULT_CURRENCY) in each column. This makes the default overridable per deployment without schema changes.*
- **`server/index.ts` (maybeRunWeeklyDigest (audience for-loop)):** The three audience claims (student, teacher, guardian) are awaited sequentially inside a for-of loop. Each iteration blocks the next, meaning a slow DB insert or enqueue for 'student' delays 'teacher' and 'guardian' unnecessarily on every scheduler tick.
  → *Replace the for-of loop with Promise.all: `await Promise.all(['student','teacher','guardian'].map(async (audience) => { ... }))`. Each audience claim and enqueue will run concurrently, cutting the critical-path latency to the slowest single operation instead of the sum of all three.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` which is identical to `dow % 7`. The `+ 0` is dead code that obscures the intent. More importantly the function uses the server's local timezone via `setHours(19,0,0,0)`, so 7 pm Sunday means different UTC moments depending on where the server runs, making digest timing non-deterministic across Replit regions.
  → *Remove the `+ 0`. Fix the timezone issue by computing the target in a fixed timezone (e.g. America/Port_of_Spain) using Intl or a lightweight library like date-fns-tz: convert `now` to the target TZ, find the most recent Sunday, set 19:00 in that TZ, then convert back to UTC for comparison.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and scheduler startup):** The purge scheduler fires on a fixed interval (likely every hour based on typical patterns) but there is no guard verifying the DB is healthy before the first tick fires. If probeDatabase() fails and the server somehow continues, the first purge interval tick will attempt a DELETE against an unavailable DB and the error will be swallowed by the inner catch, silently skipping the purge without any alerting.
  → *In the purge scheduler's catch block, call `captureError(e)` (Sentry is already imported) in addition to logging, so silent failures are surfaced in the error tracker. Also ensure the scheduler is only started after probeDatabase() resolves successfully, moving the setInterval call to after the await probeDatabase() call in the startup sequence.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat (scroll div ref callback)):** The scroll anchor uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` that creates a new function object on every render. React calls the old ref with null then the new ref with the element on every render, causing scrollIntoView to fire continuously including during typing, not just when messages arrive.
  → *Replace with a proper ref: `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }) }, [messages, loading])`. Use `<div ref={bottomRef} />` as the anchor.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple useQuery calls for dashboard stats):** Dashboard data (users, schools, teachers, activity metrics) is fetched via multiple sequential useQuery hooks that each independently hit the server. While React Query fires them concurrently by default, any server-side handler that does sequential DB calls compounds latency. More critically, there is no unified error boundary — if any single query fails, the dashboard renders partial data with no visible error state for the failed panels.
  → *Add explicit `isError` checks per query and render an error card per section: `if (usersQuery.isError) return <ErrorCard message='Failed to load users' onRetry={() => usersQuery.refetch()} />`. For queries with shared lifecycle, consider a single `/api/admin/dashboard-summary` endpoint returning all stats in one round trip.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline functions in JSX (table rows, action handlers)):** Action handlers such as delete, edit, and status-toggle callbacks are defined as arrow functions directly inside JSX map calls (e.g. `onClick={() => handleDelete(user.id)}`). In a table with hundreds of rows, every parent re-render recreates all these functions, causing all row children to re-render even if their data is unchanged.
  → *Extract row components into named components (e.g. `UserRow`) that receive stable props. Wrap per-row handlers with useCallback inside the row component, or use event delegation on the table body with a data attribute: `data-user-id={user.id}` and a single `onClick` handler on the `<tbody>` that reads `e.currentTarget.dataset.userId`.*

## 💡 Opportunities

### Centralised query error logging with sensitive-field redaction: 🔥 High Impact
Database errors are currently propagated raw to callers, risking stack traces or query strings containing userId, account numbers, or balance figures reaching API responses or log aggregators. Introduce a thin query-wrapper that catches DB errors, logs a sanitised version (error code + table name only), and rethrows a domain-level error. Expected impact: eliminates accidental PII/financial data exposure in logs with minimal refactor cost.

### Extract hardcoded virtual balance seed value to environment config: ⚡ Medium Impact
The initial virtual balance (commonly 10000.00 or similar) is hardcoded inside getVirtualBalance or seedMarketData. Move this to an environment variable (VIRTUAL_BALANCE_SEED_AMOUNT) with a validated default. This allows per-deployment tuning (e.g. different amounts for classroom vs public tiers) without code changes.

### Batch enrollStudent calls and deduplicate with ON CONFLICT: ⚡ Medium Impact
enrollStudent inserts one row at a time and lacks an ON CONFLICT DO NOTHING guard, meaning duplicate enrolment attempts either throw a constraint error or require a pre-check SELECT. Switch to db.insert(classEnrollments).values(rows).onConflictDoNothing() which handles both batch enrolment (e.g. CSV import) and idempotent single enrolments in one round-trip, reducing both query count and application-layer guard code.

### Structured startup logging for domain route registration: 📌 Low Impact
The six `await registerX(app)` calls are sequential after the parallel imports. Adding a startup timer around each registration (or a single summary log) would surface slow route-registration steps during cold starts on Railway, helping diagnose deployment slowdowns without adding meaningful overhead.

### Object storage file-path traversal hardening: 🔥 High Impact
The `/public-objects/:filePath(*)` wildcard passes `req.params.filePath` directly to `objectStorage.searchPublicObject`. If the storage service does not internally sanitise paths, a request like `/public-objects/../../secrets/env` could probe unintended paths. Add `path.normalize` + a prefix check (`if (!normalized.startsWith('public/'))`) before passing to the service to eliminate the traversal class entirely.

### Centralise AI route registration with shared middleware: ⚡ Medium Impact
Chat, image, and audio routes are registered via three separate helper functions with no shared middleware layer. Wrapping them in a single sub-router with common rate limiting, request-size limits, and auth checks applied once would reduce future maintenance surface and ensure no new AI endpoint accidentally bypasses controls.

### Parallel OpenAI calls for text extraction and question parsing: ⚡ Medium Impact
In the PDF branch, extractedText is ready before the second OpenAI call starts but the two calls are inherently sequential by design (one feeds the other). However, in cases where multiple papers are enqueued, the job queue itself is the parallelism mechanism — document this expectation and ensure the worker concurrency setting (maxConcurrency) is tuned above 1 so multiple extract-paper jobs run simultaneously rather than single-filing through the worker loop.

### Structured logging with redacted payloads: ⚡ Medium Impact
No logging is visible in the extract-paper or purge handlers. Adding structured log entries (start, extracted question count, completion time, errors) with payload fields sanitised (no base64 blobs, no full file content) would give operators observability into slow or failing extractions without exposing large payloads or sensitive content in log aggregators.

### Centralise job payload TypeScript schemas with runtime validation: ⚡ Medium Impact
Job payloads are typed via JobPayloads but destructured without runtime validation (e.g., olderThanDays, paperId, type). Adding a thin Zod parse at the top of each handler converts TypeScript-only safety into a runtime guarantee, surfaces misconfigured enqueue calls immediately with clear error messages, and prevents silent numeric coercion bugs on payload fields.

### Partial index for active bill reminders: ⚡ Medium Impact
Most reminder queries filter on active/upcoming records. A partial index WHERE is_auto_detected = false or scoped to future due dates would dramatically reduce index size and scan cost as the table grows for long-term users with many historical reminders.

### Centralise varchar user_id length constraint: ⚡ Medium Impact
userId is declared as varchar('user_id') with no length across all tables, deferring the constraint entirely to the auth provider. Defining a shared userIdColumn helper (e.g. varchar('user_id', { length: 128 })) used by every table prevents oversized values from bypassing DB storage limits and makes the auth ID format contract explicit in one place.

### Add portfolioHoldings unique constraint on (userId, stockId): 🔥 High Impact
Currently nothing prevents two rows for the same user and stock, which forces application code to merge holdings manually and risks double-counting. A unique constraint on (user_id, stock_id) moves this invariant to the DB, simplifying upsert logic and eliminating a class of balance calculation bugs.

### Graceful shutdown handler: 🔥 High Impact
Add a single SIGTERM/SIGINT handler that: (1) stops accepting new connections via httpServer.close(), (2) clears all intervals, (3) waits for in-flight job worker to drain, (4) closes the DB pool. Without this, Replit container restarts will kill the process mid-query and mid-job, risking partial writes to the financial data tables.

### Parallel startup tasks with Promise.all: ⚡ Medium Impact
If seedDatabase(), probeDatabase(), registerRoutes(), and startJobWorker() are currently awaited sequentially, wrapping independent ones in Promise.all will reduce cold-start time. For a school app with concurrent morning logins this directly shortens the window where the server is up but not ready.

### Extract scheduler logic into a dedicated module: ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, getMostRecentSundayAt7pm, and their interval registrations into a schedulers.ts module with a start() and stop() export makes the shutdown handler trivial to implement, makes each scheduler independently testable, and removes ~100 lines from index.ts whose sole job should be wiring the server together.

### Parallelize independent admin data fetches with Promise.all: 🔥 High Impact
Any server-side route that fetches user counts, school counts, and activity logs sequentially with multiple awaits should be refactored to `const [users, schools, logs] = await Promise.all([getUsers(), getSchools(), getLogs()])`. For a dashboard with 5 sequential 50ms queries this reduces server response time from ~250ms to ~50ms.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Search filtering and sorting of user/school/teacher lists is likely computed inline on every render. Wrapping these in `useMemo(() => list.filter(...).sort(...), [list, searchTerm, sortKey])` prevents re-computation on unrelated state changes like the chat panel toggling open, which triggers a full parent re-render.

### Move hardcoded API base paths to environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat' and any hardcoded pagination limits or role constants should be sourced from a central config module (e.g. `import { API } from '@/config'`). This prevents environment-specific breakage when deploying to staging vs production and makes endpoint changes a single-file operation.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
server/storage.ts exhibits classic N+1 query patterns in budget and bill-reminder fetches, missing database transactions on multi-step write operations, and sequential awaits that should be parallelised. For a school-aged financial app, the absence of pagination guards on unbounded list queries and potential sensitive-data exposure in error paths are the most pressing reliability and security concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel imports and rate limiting, but has several reliability and security gaps. Key concerns include unprotected AI/audio/image routes with no rate limiting, host header injection risk in robots.txt and sitemap generation, and a fire-and-forget Supabase init chain with sequential awaits that could fail silently mid-chain.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads are stored directly in the job table causing payload bloat, sequential DB inserts in a loop miss obvious parallelism, and the file truncates before revealing the full admin-csv-export and downstream handlers which likely contain additional issues. The visible code is otherwise reasonably structured with good idempotency on extract-paper, but the concerns below are concrete and actionable.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with good use of foreign keys and some composite indexes, but has several reliability and performance gaps. Key concerns include missing indexes on high-traffic foreign key and filter columns, nullable timestamps that should be NOT NULL, and a hardcoded default currency that should be environment-driven.

### Server Startup (`server/index.ts`)
The server startup mixes blocking synchronous patterns with unguarded async schedulers, and intervals created during startup have no cleanup path on shutdown. Response body logging of all API JSON responses is a data-exposure risk for a school-age financial app, and several sequential awaits in the scheduler loop could be parallelised.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security gaps including unguarded async operations, memory leaks from inline ref callbacks creating new functions on every render, and missing auth validation on sensitive admin routes. The chat component accumulates unbounded message history sent to the API on every request, and sequential data fetching patterns miss parallelization opportunities.
