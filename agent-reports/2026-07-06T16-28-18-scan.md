## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 110-112)
**Issue:** AI-heavy endpoints (chat, image, audio) are registered with no rate limiting middleware applied before them. On a school-aged user app these can be trivially abused to exhaust OpenAI quota or cause runaway costs.

**Fix:** Create an `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 48) and sitemap.xml handler (~line 61)
**Issue:** Both handlers build a base URL from `req.headers['x-forwarded-host']` and `req.headers['x-forwarded-proto']` without any validation or allowlist. An attacker can send spoofed Host headers to inject arbitrary URLs into the Sitemap directive and sitemap <loc> tags, enabling cache poisoning or phishing via crawlers.

**Fix:** Add a hardcoded fallback from environment config: `const base = process.env.CANONICAL_BASE_URL || `${proto}://${host}`;` and set CANONICAL_BASE_URL in production. Optionally validate `host` against an allowlist regex before use.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** fileB64 (full base64 file content) is stored in the job payload in the jobs table. For a multi-page PDF or high-res image this can be several MB per job row, bloating the DB, blowing through row-size limits, and leaking exam content to anyone with DB read access.

**Fix:** Upload the file to object storage before enqueuing, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside the handler with objectStorageClient. Delete the staging object after extraction completes or permanently fails.

### `server/jobHandlers.ts`: uploadPrivateObject / parsePrivatePath
**Issue:** relPath is constructed by string concatenation with root from PRIVATE_OBJECT_DIR without path-traversal sanitisation. A crafted relPath containing '../' sequences could escape the intended bucket prefix and overwrite arbitrary objects in object storage.

**Fix:** After building full, resolve it with path.posix.normalize and assert it still starts with the normalised root prefix: if (!normalizedFull.startsWith(normalizedRoot)) throw new Error('Path traversal detected'). Use path.posix throughout instead of manual string splits.

### `server/index.ts`: capturedJsonResponse logging block (~line 72)
**Issue:** Every API response body is serialized and written to stdout via `JSON.stringify(capturedJsonResponse)`. For a financial literacy app with school-aged users this will log session tokens, user PII, quiz answers, and any JWT/auth payloads in plaintext, violating COPPA/data-minimization principles and creating a credential-leak vector in log aggregators.

**Fix:** Remove the `capturedJsonResponse` capture entirely, or gate it behind `process.env.NODE_ENV === 'development'` and add a blocklist: `const SENSITIVE_PATHS = ['/api/auth', '/api/user']; if (!SENSITIVE_PATHS.some(p => path.startsWith(p))) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`. Never log response bodies in production.

### `server/index.ts`: setInterval calls in startSchedulers() or equivalent (~line 120-160)
**Issue:** The `setInterval` handles for the weekly-digest scheduler and AI-usage purge scheduler are not stored in any variable, so there is no way to call `clearInterval()` on them during graceful shutdown. When the process receives SIGTERM (e.g., Replit container restart), in-flight DB operations from the scheduler tick will be aborted mid-transaction, risking partial writes to `weeklyDigestRuns` or `aiUsagePurgeRuns` without the rollback path executing.

**Fix:** Store every interval handle: `const digestInterval = setInterval(maybeRunWeeklyDigest, 60_000);`. In your SIGTERM handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before `httpServer.close()`. This ensures no new ticks fire while existing async work drains.

---

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** The entire message history array is sent to /api/admin/help-chat on every request with no size cap. For a school-aged user platform, this means unbounded growth of the payload and potential exposure of prior conversation content (including any sensitive admin data mentioned) in every subsequent request body.

**Fix:** Slice the history before sending: `messages: next.slice(-10)` to keep only the last 10 messages. Also ensure the /api/admin/help-chat route enforces admin-only authentication server-side, as the client-side useAdminAuth hook alone is insufficient.

## 🟡 Warnings (12)

- **`server/routes.ts` (initSupabaseTables chain (~line 115-123)):** The seven Supabase init/seed functions are chained sequentially with .then(), so each waits for the previous to finish. Most of these (seedStaticModules, seedRegionalContent, seedGameContent) are independent and could run concurrently. This unnecessarily extends cold-start time.
  → *After the mandatory ordering (initSupabaseTables → trimOrgNamesInSupabase → seedFinancialAcademyLesson → initStaticContentTables), run the remaining seeds in parallel: `initSupabaseTables().then(() => trimOrgNamesInSupabase()).then(() => seedFinancialAcademyLesson()).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));`*
- **`server/routes.ts` (/healthz handler (~line 82)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every request. Dynamic imports are cached after the first call, but the initial cold hit adds latency and the pattern obscures the true dependency. More critically, this endpoint is unauthenticated and reveals internal DB availability state to anyone who probes it.
  → *Move the db and sql imports to the top of the file as static imports. Optionally restrict /healthz to internal network IPs or protect it with a secret token header (`req.headers['x-health-token'] === process.env.HEALTH_TOKEN`) to avoid leaking infrastructure state.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96)):** The `filePath` parameter accepts wildcard paths and is passed directly to `objectStorage.searchPublicObject` with no sanitization. A path like `../../private/config` could traverse into unintended storage keys depending on the ObjectStorageService implementation.
  → *Sanitize the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, '').replace(/^[\/]+/, '');` and reject requests where the normalized path is empty or starts with a dot. Confirm ObjectStorageService does not perform filesystem operations before relying solely on the SDK.*
- **`server/jobHandlers.ts` (for loop inside runExtractPaper (~line where questions are inserted)):** Extracted questions are inserted one-at-a-time with sequential awaits inside a for loop. For a 40-question paper this is 40 serial round-trips to the DB, each adding latency and holding the job worker busy. If any single insert throws, partial data is left behind without cleanup.
  → *Collect all valid question objects into an array and call a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Wrap in a transaction so partial failure rolls back atomically instead of relying on the delete-on-retry idempotency patch.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The entire handler body has no try/catch. An unexpected DB error (connection timeout, constraint violation) will propagate as an unhandled rejection inside the job runner, and olderThanDays is used directly in arithmetic with no validation—passing 0 or a negative number deletes all records or nothing silently.
  → *Wrap the handler body in try/catch and rethrow with context. Add a guard: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'). Log deletedRows at info level for audit.*
- **`server/jobHandlers.ts` (openai import: "import { openai } from './replit_integrations/chat/routes'"):** The OpenAI client is imported from a route module (chat/routes), coupling a job handler to HTTP-layer internals. If that module initialises Express middleware or registers routes as a side effect on import, importing it here triggers those side effects in the job worker context, causing unpredictable behaviour and making the dependency invisible to static analysis.
  → *Extract the OpenAI client instantiation into a dedicated shared module (e.g. server/openaiClient.ts) that exports only the configured client. Both the chat routes and job handlers import from there. This eliminates the side-effect risk and makes the dependency explicit.*
- **`server/index.ts` (maybeRunWeeklyDigest() -> rollback await (~line 105)):** Inside the catch block that rolls back a failed digest claim, the `await db.delete(...)` call is not wrapped in its own try/catch. If the rollback DB call itself throws (network blip, pool timeout), the exception propagates out of the inner catch, bypasses the outer catch's log, and becomes an unhandled promise rejection, silently leaving a phantom claim row that blocks future digest runs for that audience/week.
  → *Wrap the rollback in its own try/catch: `try { await db.delete(weeklyDigestRuns).where(...); } catch (rbErr) { captureError(rbErr); log(`CRITICAL: rollback failed for ${audience}/${weekStart}: ${(rbErr as Error).message}`); }`. Also call `captureError(e)` on the original enqueue failure so Sentry captures it.*
- **`server/index.ts` (startup sequence: seedDatabase / probeDatabase / registerRoutes (~line 180-210)):** `seedDatabase()` and `probeDatabase()` are awaited sequentially before the HTTP server binds, and `seedDatabase` likely performs multiple round-trip queries. On a cold Replit container with a remote Neon/Postgres DB this can add 2-8 seconds of blocking startup time during which the process is alive but serving nothing, and a SIGTERM will leave no cleanup path because the shutdown handler isn't registered until after the awaits complete.
  → *Register the SIGTERM/SIGINT handlers as the very first async step, before any await. Then run independent startup tasks in parallel where safe: `await Promise.all([probeDatabase(), seedDatabase()])`. If seeding must follow a proven connection, do `await probeDatabase(); await seedDatabase();` but register signal handlers first.*
- **`server/index.ts` (getMostRecentSundayAt7pm() ~line 85):** The digest window calculation uses `new Date()` which resolves to the server's local timezone. On a UTC-based host (Replit, Render, Railway all default to UTC) `setHours(19, 0, 0, 0)` sets 7 PM UTC, not 7 PM Caribbean time (UTC-4/UTC-5). Caribbean teachers and guardians will receive the digest at 2-3 PM local time on Sunday rather than the expected evening slot, and the window comparison will misfire for users in AST/EST.
  → *Pin the schedule to a specific IANA timezone using the `Intl` API or the `date-fns-tz` library: `const target = toZonedTime(now, process.env.DIGEST_TIMEZONE ?? 'America/Port_of_Spain'); target.setHours(19,0,0,0);`. Expose `DIGEST_TIMEZONE` as an env var defaulting to `'America/Port_of_Spain'`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > Messages scroll div):** The scroll-to-bottom ref uses an inline callback `ref={el => { if (el) el.scrollIntoView(...); }}` directly in JSX. This creates a new function instance on every render, and more critically, calling scrollIntoView inside a render-phase ref callback is a side-effect that violates React's rendering contract and can cause layout thrashing on every re-render of the messages list.
  → *Replace with a proper useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function):** The catch block silently swallows the actual error and only posts a generic user-facing string. For an admin dashboard, losing the error detail makes debugging production incidents very hard. There is also no network-timeout handling, so if the AI backend hangs the UI spinner locks indefinitely.
  → *Log the caught error: `console.error('[AdminHelpChat] sendMessage failed:', err)`. Add an AbortController with a timeout (e.g. 30 s): `const controller = new AbortController(); const timer = setTimeout(() => controller.abort(), 30000);` pass `{ signal: controller.signal }` to apiRequest and clear the timer in finally.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > handleKey):** handleKey is defined as an inline arrow function inside the component body with no useCallback, so a new function reference is created on every render. Because it closes over sendMessage (which itself closes over messages, input, loading), this causes unnecessary re-renders of the controlled input element on every keystroke.
  → *Wrap both sendMessage and handleKey in useCallback: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`.*

## 💡 Opportunities

### Centralized route registration error boundary: ⚡ Medium Impact
The six `await register*Routes(app)` calls have no individual try/catch. If one throws, the entire server fails to start with no indication of which domain router caused it. Wrap each in a labeled try/catch that logs the failing module name and rethrows, giving actionable cold-start diagnostics.

### Duplicate health check endpoints: 📌 Low Impact
Both `/api/health` and `/healthz` exist and serve overlapping purposes. The first returns no DB check, the second does. Consolidate into a single `/api/health` endpoint with an optional `?deep=1` query param for the DB ping, and document which is used by Railway vs. uptime monitors to avoid confusion and maintenance drift.

### ObjectStorageService singleton instantiation: ⚡ Medium Impact
The `objectStorage` instance is created at module load time (`const objectStorage = new ObjectStorageService()`) outside registerRoutes, meaning any constructor-side effects (connections, credential reads) happen before auth and Sentry are configured. Move instantiation inside registerRoutes or lazy-initialize it on first request to ensure observability is in place before side effects occur.

### Parallel AI calls in extract-paper: ⚡ Medium Impact
The text-extraction step (PDF parse or vision API) and any pre-flight DB reads run sequentially before the second OpenAI call. Where independent setup work exists (e.g. fetching paper metadata while waiting for the vision response), wrapping with Promise.all would cut end-to-end job latency noticeably for image-based papers that already take 5-10 s on the vision API.

### Job payload schema validation at enqueue time: 🔥 High Impact
There is no runtime schema validation (e.g. Zod) on job payloads when they are enqueued. Handlers currently trust payload shape implicitly, meaning a mis-typed caller silently enqueues a job that will always fail on first run. Adding Zod schemas keyed to each job type in a shared JobPayloads map and validating at registerJobHandler entry would catch this at enqueue time and eliminate an entire class of hard-to-debug silent failures.

### Centralised job-level error telemetry: ⚡ Medium Impact
Only the ai-health-check handler wraps errors with captureError/Sentry. All other handlers (purge-ai-usage, extract-paper, perf-scan) surface failures only through the job runner's generic mechanism. Adding a thin middleware wrapper in registerJobHandler that automatically calls captureError on any thrown error—tagging job type, jobId, and attempt number—would give uniform observability across all jobs without modifying each handler.

### Structured JSON logging with log-level filtering: 🔥 High Impact
Replace all `console.log` / `log()` string concatenation with a structured logger (pino or winston) emitting JSON lines with fields: `{level, ts, source, method, path, statusCode, durationMs}`. This eliminates the PII-in-logs risk by making it trivial to redact fields by key, and enables log-level env var (`LOG_LEVEL=warn`) to silence debug noise in production without code changes. Expected impact: eliminates the response-body leak finding and cuts log ingestion cost by ~60% in production.

### Centralized graceful shutdown coordinator: 🔥 High Impact
Extract a `shutdown(signal)` async function registered for SIGTERM, SIGINT, and `uncaughtException` that: (1) stops all intervals, (2) calls `httpServer.close()` with a 10 s timeout, (3) drains the job worker, (4) closes the DB pool. Currently shutdown logic is scattered or absent, so Replit container restarts will kill in-flight DB transactions. A single coordinator with `Promise.race([drain(), sleep(10_000)])` gives deterministic shutdown under 10 seconds.

### Parallelize independent startup I/O with Promise.all: ⚡ Medium Impact
The current sequential `await probeDatabase()` → `await seedDatabase()` → `await registerRoutes()` pattern adds cumulative cold-start latency. `probeDatabase` and any config-fetch operations that don't depend on each other can run concurrently. On a Neon serverless DB with a 300-500 ms wake-up time, parallelizing just probe + any remote config fetch saves 300-500 ms of startup time, which matters for Replit's request-triggered wake behavior.

### Parallelize independent admin data queries with Promise.all: 🔥 High Impact
If the dashboard fires multiple useQuery calls sequentially (users, schools, teachers, analytics), any that are independent should be declared at the same level so React Query runs them concurrently. Confirm none use waterfall patterns like using data from query A as the enabled condition for query B unless truly dependent. Concurrent fetching can cut initial load time by 40-60% on slow Caribbean mobile connections.

### Externalize hardcoded API paths and AI model config to environment variables: ⚡ Medium Impact
Strings like '/api/admin/help-chat' and any AI model names or temperature settings embedded in the server handler should be sourced from process.env (e.g. ADMIN_HELP_CHAT_PATH, OPENAI_MODEL). This enables staging/production parity and avoids code changes for config-only updates.

### Add a virtualized list for large admin data tables: ⚡ Medium Impact
Admin tables listing all users, schools, or transactions for a growing Caribbean school network can reach thousands of rows. Rendering them as plain mapped JSX causes significant DOM bloat and janky scrolling. Introduce @tanstack/react-virtual or react-window for any table expected to exceed 100 rows, reducing DOM nodes by ~95% and improving scroll frame rate.
