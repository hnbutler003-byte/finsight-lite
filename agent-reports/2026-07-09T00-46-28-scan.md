## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (lines ~107-109)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. In a school-age user app these endpoints can be trivially abused to run up API costs or trigger model abuse, and there is no per-user or per-IP throttle applied before or after registration.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 43) and sitemap handler (~line 57)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a spoofed Host header to inject an arbitrary URL into the Sitemap directive and loc elements, enabling cache-poisoning or phishing via search-engine-cached robots.txt/sitemap.

**Fix:** Add a validated APP_BASE_URL environment variable (e.g. process.env.APP_BASE_URL) and use it as the canonical base. Fall back to host-header construction only in development, and validate the header against a regex or allowlist: const base = process.env.APP_BASE_URL ?? buildAndValidateBase(req);

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both schedulers are started with setInterval but their handles are never stored or cleared. On SIGTERM/SIGINT the intervals keep firing during shutdown, potentially enqueuing jobs or writing to a closing database connection. In test environments that restart the module, each restart leaks an additional interval.

**Fix:** Capture handles: `const digestInterval = setInterval(maybeRunWeeklyDigest, ...)` and `const purgeInterval = setInterval(maybeRunAiUsagePurge, ...)`. In the graceful shutdown handler add `clearInterval(digestInterval); clearInterval(purgeInterval);` before draining connections and calling `process.exit(0)`.

### `server/index.ts`: res.json middleware / logLine construction in app.use finish handler
**Issue:** Every API response body is captured via monkey-patching res.json and then fully serialized into the log line with JSON.stringify(capturedJsonResponse). For endpoints returning user quiz results, balances, or AI-generated content this logs complete PII and financial data to stdout/Sentry in plaintext with no size cap or field filtering.

**Fix:** Either remove response body logging entirely for production, or apply an allowlist of safe fields and a hard size cap: `const safe = JSON.stringify(capturedJsonResponse); if (safe.length < 512) logLine += ' :: ' + safe;` and never log responses from auth, user-profile, or any route under /api/users. Set NODE_ENV guard so body logging only runs in development.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat.sendMessage
**Issue:** The full message history array is sent to /api/admin/help-chat on every message with no size cap. A user can accumulate hundreds of messages and send a multi-megabyte payload, causing server timeout or OOM on the API route, and potentially leaking prior admin session context if the component is reused across different admin users without a reset.

**Fix:** Slice the history to the last N turns before sending: `messages: next.slice(-10)`. Also call `setMessages([])` when the chat panel is closed (`setOpen(false)`) so stale context is not carried across sessions.

---

### `client/src/pages/AdminDashboard.tsx`: Mutation handlers (deleteUser, deleteSchool, etc.)
**Issue:** Destructive mutations (delete user, delete school, promote to admin) rely solely on the `useAdminAuth` hook guard at render time but do not re-verify the caller's role inside the mutation function before firing the API request. If the hook's cached auth state is stale (e.g. session expired but token not yet cleared), the request is sent anyway, depending entirely on server enforcement that may not exist on all routes.

**Fix:** Add a pre-flight check inside each mutationFn: `if (!adminUser?.role || adminUser.role !== 'admin') throw new Error('Unauthorized');` before calling `apiRequest(...)`. Also ensure the API routes validate the session server-side and return 401/403 explicitly, which the mutation's `onError` handler should surface to the UI.

## 🟡 Warnings (12)

- **`server/routes.ts` (Supabase init chain (~lines 112-120)):** The seven Supabase seed/init functions are chained sequentially with .then() instead of being parallelised. Each function almost certainly hits independent tables, so they stall startup unnecessarily. Additionally, the single .catch() at the end only logs; a seed failure leaves the app running in a partially initialised state with no alerting.
  → *Group truly independent operations with Promise.all and add Sentry capture on failure: Promise.all([initSupabaseTables(), initStaticContentTables()]).then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => { console.error('[Supabase] Init error:', e); Sentry.captureException(e); });*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96)):** filePath is taken directly from req.params.filePath and passed to objectStorage.searchPublicObject without path traversal sanitisation. A request to /public-objects/../../private/secrets could resolve outside the intended public prefix depending on the ObjectStorageService implementation.
  → *Sanitise before use: const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); and assert it does not start with '..' or '/' before passing to the storage service. Reject with 400 if the normalised path escapes the expected prefix.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index, but budget lookups are almost certainly filtered by userId and joined on categoryId. A full table scan on budgets will degrade as user count grows.
  → *Add a composite index in the budgets table factory function: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a single-column index on userId alone if you ever query budgets without a categoryId filter.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt uses .defaultNow() but no .notNull(), so the column is nullable in the DB schema. Any raw SQL insert that omits the column will store NULL, breaking ORDER BY createdAt DESC queries silently.
  → *Chain .notNull() after .defaultNow() on every createdAt column: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Drizzle supports this and it generates a NOT NULL constraint in the migration.*
- **`shared/schema.ts` (userId: varchar('user_id') across all tables):** varchar with no length argument defaults to unlimited in Postgres but is inconsistently typed. More critically, there is no shared length constraint, so a provider returning a 36-char UUID and one returning a shorter ID will both silently pass, making cross-table joins fragile and preventing a useful DB-level check constraint.
  → *Define a reusable column alias: const userIdCol = () => varchar('user_id', { length: 36 }). Use it everywhere userId appears. If Clerk UIDs are longer (e.g. 'user_' + 26 chars = 31), set length: 32 or 64 accordingly — pick one and enforce it consistently.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text, not timestamp. Storing a date as text means Postgres cannot enforce validity, range queries (find stocks updated before date X) require casting, and sorting is lexicographic not chronological.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any insertion logic that currently serializes a date string to pass a JS Date object instead.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on (userId, nextDueDate), yet the primary use case — fetching upcoming bills for a user — will filter on exactly these two columns. Without an index this is a full scan of all reminders.
  → *Add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) as the third argument to pgTable for billReminders.*
- **`server/index.ts` (seedDatabase / probeDatabase / startJobWorker calls in startup IIFE):** Startup awaits seedDatabase, probeDatabase, registerRoutes, and startJobWorker sequentially before httpServer.listen(). seedDatabase and probeDatabase have no data dependency on each other and together can add hundreds of milliseconds of blocking cold-start time before the server accepts any connections.
  → *Run independent startup tasks in parallel: `await Promise.all([probeDatabase(), seedDatabase()])` then proceed with route registration and job worker init. If seedDatabase depends on probeDatabase succeeding, chain only those two: `await probeDatabase(); await Promise.all([seedDatabase(), startJobWorker()])`. This keeps the critical path short.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — the `+ 0` is a no-op left from a likely edit that removed a timezone offset constant. More importantly, `new Date()` and `d.setHours(19, 0, 0, 0)` use the server's local timezone, so on a UTC-hosted Replit instance 7 PM local is actually midnight UTC, causing the digest to fire at the wrong time for Caribbean (UTC-4/UTC-5) users.
  → *Remove the dead `+ 0`. Replace local-time arithmetic with explicit UTC-to-Caribbean offset: define `DIGEST_HOUR_UTC = 23` (19:00 AST = 23:00 UTC) as an env-configurable constant, then set `d.setUTCHours(DIGEST_HOUR_UTC, 0, 0, 0)` and compute day-of-week from `d.getUTCDay()`. Store the constant in environment config so it can be adjusted without a deploy.*
- **`server/index.ts` (maybeRunWeeklyDigest and maybeRunAiUsagePurge async interval callbacks):** Both async functions are passed directly to setInterval. setInterval does not await the returned Promise, so if one invocation takes longer than the interval period (e.g. slow DB insert during claim) a second invocation starts concurrently. Two simultaneous purge runs can race on the same records even with the conflict guard.
  → *Wrap each callback in a mutex guard using a module-level boolean flag: `let digestRunning = false; setInterval(async () => { if (digestRunning) return; digestRunning = true; try { await maybeRunWeeklyDigest(); } finally { digestRunning = false; } }, interval)`. Apply the same pattern to the purge scheduler.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat JSX — `<div ref={el => { if (el) el.scrollIntoView(...) }} />`):** The inline ref callback is a new function on every render. Because it is not a stable ref, React calls it with null then the element on every render cycle, firing scrollIntoView on every keystroke, state update, and parent re-render — not just when a new message arrives. This causes janky scroll behavior and unnecessary DOM thrashing.
  → *Replace with a stable `useRef` + `useEffect` that only triggers when `messages.length` or `loading` changes: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length, loading]);` then `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard data fetching (useQuery calls for users, schools, teachers, analytics)):** Multiple independent useQuery calls are defined sequentially, each with its own staleTime/network round-trip. Where queries have no interdependency (e.g. user count, school list, analytics summary), they fire serially during SSR hydration and on refetch, adding cumulative latency visible to the admin on every dashboard load.
  → *Consolidate independent queries into a single `/api/admin/dashboard-summary` endpoint that uses `Promise.all` server-side, or use `useQueries` from TanStack Query to fan them out in parallel on the client: `const results = useQueries({ queries: [usersQueryOptions, schoolsQueryOptions, analyticsQueryOptions] })`. This eliminates waterfall round-trips.*

## 💡 Opportunities

### Domain router registration parallelisation: ⚡ Medium Impact
The six domain router register functions (registerAuthDomainRoutes through registerEmailRoutes) are awaited sequentially after being imported in parallel. If any of these perform async setup (DB schema checks, cache warming), running them with Promise.all would cut cold-start latency proportionally to the slowest one rather than their sum.

### Centralised error-handling middleware: 🔥 High Impact
There is no global Express error handler registered in this file. Unhandled errors thrown inside route handlers will produce default Express HTML error responses, leaking stack traces to users in non-production envs. Add app.use((err, req, res, next) => ...) as the final middleware with Sentry capture and sanitised JSON responses.

### Sitemap caching: 📌 Low Impact
The sitemap and robots.txt are regenerated on every request with string joins. For a public-facing SEO asset that never changes at runtime, generate the strings once at startup and cache them in module scope. This eliminates per-request allocation and lets you set a long Cache-Control max-age header, reducing crawler load.

### Centralise currency default to environment config: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, portfolioHoldings). Extract it to a single constant DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' in a shared config file and reference it in every .default() call. This prevents drift if the app expands to TTD or JMD and avoids a multi-file migration to change the default.

### Add unique index on simulatedStocks.ticker: 🔥 High Impact
ticker should be unique across all simulated instruments. Without a uniqueIndex, duplicate tickers can be inserted, which will cause ambiguous JOIN results in portfolio value calculations. Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table factory. This also enables efficient lookup-by-ticker in the investment simulation routes.

### Replace portfolioHoldings with upsert-friendly unique constraint: 🔥 High Impact
portfolioHoldings currently has no unique constraint on (userId, stockId), so buy operations that use INSERT instead of upsert will create duplicate holding rows per stock per user. Add a uniqueIndex on (userId, stockId) and switch buy logic to an INSERT ... ON CONFLICT DO UPDATE to atomically update avgPurchasePrice and quantity, eliminating the need for a read-before-write and the race condition it creates.

### Structured JSON logging instead of formatted console strings: 🔥 High Impact
Replacing the custom log() formatter and ad-hoc logLine concatenation with a structured logger (pino is zero-dep and fast) gives log ingestion pipelines (Sentry, Datadog) parseable fields like method, path, statusCode, durationMs without regex parsing. Expected impact: faster incident triage and elimination of the current PII-in-string risk.

### Centralised graceful shutdown coordinator: 🔥 High Impact
Currently shutdown logic is scattered across SIGTERM/SIGINT handlers added in multiple places. Extract a single ShutdownManager that registers handlers once, accepts drain callbacks (close httpServer, clearInterval schedulers, drain job worker queue, close DB pool) in registration order, and enforces a hard timeout (e.g. 10 s) before forced exit. This prevents the intervals-still-firing and half-closed-DB issues described in findings.

### Environment config validation at startup with a schema: ⚡ Medium Impact
SESSION_SECRET is checked manually but AUTO_PURGE_OLDER_THAN_DAYS, digest schedule hours, job concurrency limits, and DB URL are parsed inline throughout the file. Consolidate into a single zod or env-var schema validated synchronously before initSentry() is called. Invalid config fails fast with a clear message listing all missing vars rather than silently falling back to hardcoded defaults that may be wrong for production.

### Memoize filtered and sorted table data: ⚡ Medium Impact
User/school table rows are filtered and sorted on every render using inline array operations inside the JSX. Wrapping these with `useMemo` keyed on the raw data array and search/filter state would prevent O(n) re-computation on unrelated state changes (e.g. dialog open/close, pagination clicks), which matters once user counts reach the hundreds typical of a school district deployment.

### Add suspense boundaries and skeleton loaders per section: 🔥 High Impact
Currently the entire dashboard waits for all queries before rendering. Wrapping each card section (Users, Schools, Analytics) in its own `<Suspense fallback={<SkeletonCard />}>` with per-query `suspense: true` would let the fastest data paint immediately, reducing perceived load time for school-aged users on mobile Caribbean networks with higher latency.

### Debounce search input to reduce redundant query calls: ⚡ Medium Impact
If the search input triggers a query param or client-side filter recompute on every keystroke, it fires up to ~30 filter passes per second during fast typing. Adding a `useDebounce` hook (300 ms) on the search value before it feeds into the filter or query key would eliminate redundant work and, if search is server-side, prevent request flooding.
