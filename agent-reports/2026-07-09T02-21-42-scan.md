## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied beforehand. A student or bot can hammer these endpoints, causing runaway OpenAI spend and degraded service for all Caribbean school users.

**Fix:** Create a dedicated aiLimiter in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Confirm the exact path prefixes from each register function first.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 38 and ~line 57)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host with no validation or allowlist. An attacker can send a crafted Host header causing the Sitemap directive and canonical URLs to point to an attacker-controlled domain, enabling cache poisoning or SEO manipulation targeting minors.

**Fix:** Add a hardened helper: const ALLOWED_HOST = process.env.PUBLIC_HOST; const host = ALLOWED_HOST || req.headers['x-forwarded-host'] || req.headers.host || 'localhost'; Require PUBLIC_HOST to be set in production (assert it on startup). This eliminates the injection vector entirely.

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** There is no unique constraint on (userId, stockId). A user can accumulate duplicate holding rows for the same stock, requiring the application to aggregate them at query time and risking double-counting of positions — a correctness bug for a financial literacy tool.

**Fix:** Add a unique index to the portfolioHoldings table: (t) => ({ userStockUnique: uniqueIndex('idx_holdings_user_stock_unique').on(t.userId, t.stockId) }). Update the buy logic to upsert (update quantity/avgPurchasePrice) rather than insert a new row.

### `server/index.ts`: seedDatabase / probeDatabase calls in startup IIFE
**Issue:** probeDatabase and seedDatabase are awaited sequentially before httpServer.listen() is called, meaning the process accepts no traffic and cannot respond to health checks until both complete. On a slow or flaky DB connection this can cause the process to be killed by the platform before it ever binds a port.

**Fix:** Start listening first, then run probeDatabase and seedDatabase. Wrap each in its own try/catch so a seed failure does not prevent the server from serving traffic. If DB connectivity is truly required before serving, add a /healthz route that returns 503 until ready rather than delaying listen().

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are not captured in variables, so the SIGTERM / SIGINT graceful-shutdown handler cannot call clearInterval() on them. The intervals will fire during the drain window after shutdown is initiated, potentially enqueuing duplicate jobs or performing DB writes against a closing connection pool.

**Fix:** Assign each setInterval to a module-level const (e.g. const weeklyDigestTimer = setInterval(...)), then call clearInterval(weeklyDigestTimer) and clearInterval(purgeTimer) inside the shutdown handler before awaiting httpServer.close().

---

### `server/index.ts`: res.json monkey-patch in request logging middleware
**Issue:** capturedJsonResponse is set to the full response body and then unconditionally serialised into the log line for every /api route. For a financial literacy app with school-aged users this can log session tokens, user PII, quiz answers, or AI-generated content in plaintext to stdout, violating basic data-minimisation principles and potentially FERPA/COPPA requirements.

**Fix:** Replace the full-body capture with a size-bounded, field-filtered log. At minimum, only log the body when NODE_ENV !== 'production', or define an allowlist of safe top-level keys (e.g. { ok, error, count }) and log only those. Remove the res.json override entirely in production and rely on structured APM (Sentry) for response introspection.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / dashboard data useQuery calls
**Issue:** Multiple useQuery calls for users, schools, teachers, and analytics appear to fire sequentially in component body without verifying the admin session is valid first. If useAdminAuth returns a non-admin or unauthenticated user, all these queries still execute and their results are rendered, meaning a race condition can briefly expose admin data before the redirect fires.

**Fix:** Add `enabled: !!isAdmin` (from useAdminAuth) to every sensitive useQuery options object. This ensures queries are held until authentication is confirmed, eliminating the race and avoiding wasted network requests on unauthenticated loads.

## 🟡 Warnings (12)

- **`server/routes.ts` (Supabase init chain (~line 112)):** The seven Supabase seed/init functions are chained sequentially with .then(). Each waits for the previous to fully complete before starting, making cold-start initialization needlessly slow. Some of these (e.g. seedStaticModules, seedRegionalContent, seedGameContent) are likely independent and can run concurrently.
  → *Group truly independent operations into Promise.all calls: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()])).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Verify foreign-key dependencies between seeds before parallelising.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 99)):** filePath comes directly from req.params.filePath and is passed to objectStorage.searchPublicObject and downloadObject without any path traversal sanitisation. A request to /public-objects/../../secrets/config could potentially reach unintended storage paths depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' }); This prevents traversal regardless of what ObjectStorageService does internally.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Every budget lookup by user (the most common query pattern) requires a full table scan. As user count grows this becomes a bottleneck.
  → *Add a composite index in the budgets table definition: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a standalone userIdx for queries that filter only by userId.*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text instead of timestamp. Storing dates as strings makes range queries (e.g. find stocks updated before a date) incorrect or require casting, breaks index efficiency, and risks locale-dependent format bugs.
  → *Change the column definition to: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed/migration scripts that write string values to this column.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt, documentUploads.createdAt (all audit timestamps)):** All createdAt columns use .defaultNow() but are missing .notNull(). This means application code or a failed default can INSERT a NULL audit timestamp, making audit trails unreliable — a compliance risk for a financial app.
  → *Append .notNull() to every createdAt and executedAt/purchasedAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default still applies; notNull merely rejects explicit NULLs.*
- **`shared/schema.ts` (categories.userId, linkedCards, portfolioHoldings, portfolioTransactions — all userId columns):** userId is defined as varchar('user_id') with no length argument. PostgreSQL will accept unbounded varchar, but without a consistent max length (e.g. the 36 chars of a UUID) there is no database-level guard against truncated or oversized IDs being stored, which can cause silent FK mismatches.
  → *Specify an explicit length matching your auth provider's ID format: varchar('user_id', { length: 36 }) for UUIDs or { length: 255 } if using opaque tokens. Apply consistently across every userId column in the schema.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function uses new Date() and local-system wall-clock arithmetic with no timezone specification. On a server whose TZ env var is UTC (the Replit default), 7 pm local time is 7 pm UTC, which will differ from the intended Caribbean local time (UTC-4 or UTC-5). Digests will fire 4-5 hours earlier than expected and the weekStart key will occasionally be the wrong date near midnight boundaries.
  → *Pin the schedule to a specific IANA timezone. Use a library like date-fns-tz or Temporal to compute 'most recent Sunday 19:00 America/Port_of_Spain'. Alternatively, store the intended fire time as a UTC constant derived from TZ_OFFSET env var and validate it in the startup guard alongside SESSION_SECRET.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and scheduler):** The purge scheduler fires on a fixed interval (visible in truncated code) and deletes AI usage records older than N days, but there is no guard preventing the purge from running while a previous purge job is still executing. Concurrent purge runs against the same date range can cause duplicated DB work or, depending on the ORM transaction isolation, partial deletes and inconsistent aiUsagePurgeRuns rows.
  → *Add a module-level boolean flag (let purgeRunning = false). At the top of maybeRunAiUsagePurge set it to true and return early if already true; reset it in the finally block. This is the same advisory-lock pattern already used for the weekly digest claim row but applied in-process to prevent overlapping executions within a single dyno.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - sendMessage):** The /api/admin/help-chat call sends the full accumulated message history on every turn with no size cap. A user or attacker can cause the array to grow unboundedly, inflating request payloads and potentially leaking conversation context across sessions if messages state is not reset on close.
  → *Slice the history before sending: `messages: next.slice(-20)` to cap at 20 turns. Also call `setMessages([])` inside the `setOpen` toggle when transitioning to closed so stale conversation data is not retained in memory between sessions.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - messages.map render, scroll div ref):** The scroll-anchor div uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` which creates a brand-new function object on every render. React treats this as a changed ref each render, firing the callback repeatedly and causing excessive scrollIntoView calls during typing or loading state changes.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats/filter derived values):** User list filtering, sorting, and aggregate stat calculations (total users, active schools, leaderboard ranks) are performed as plain expressions or inline .filter/.sort/.reduce chains directly in the render body. On each parent re-render or any unrelated state change these O(n) operations re-run against the full dataset.
  → *Wrap each derived collection in useMemo with the relevant source array as dependency, e.g. `const filteredUsers = useMemo(() => users.filter(u => u.name.includes(search)), [users, search])`. Apply the same pattern to aggregate stats and sorted leaderboard arrays.*
- **`client/src/pages/AdminDashboard.tsx` (apiRequest calls inside mutation handlers / useEffect):** Multiple async operations inside useMutation handlers and useEffect bodies (e.g. fetching school details then teacher list then analytics sequentially) use consecutive awaits where the requests are independent. This adds cumulative latency equal to the sum of all round-trips rather than the slowest one.
  → *Replace sequential independent awaits with `const [schools, teachers, analytics] = await Promise.all([apiRequest(...), apiRequest(...), apiRequest(...)])`. Only keep sequential awaits where a later call genuinely depends on the result of an earlier one.*

## 💡 Opportunities

### Startup environment validation: 🔥 High Impact
Add a validateEnv() function called before registerRoutes that asserts required env vars (PUBLIC_HOST, SUPABASE_URL, OPENAI_API_KEY, SESSION_SECRET, etc.) are present and non-empty, throwing on startup rather than failing silently at runtime. This prevents misconfigured deployments from serving real school users with broken functionality.

### Structured logging for Supabase init errors: ⚡ Medium Impact
The catch on the Supabase init chain logs with console.error but swallows the error silently in production. Pipe this through the existing Sentry integration (captureException) so seed failures are visible in the error dashboard. Missed seed failures mean students could hit missing lesson content with no alert to the team.

### Domain router registration parallelisation: 📌 Low Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six sequential awaits. If any registerXRoutes function does async work (DB calls, dynamic imports), this adds latency. Evaluate whether each register function can be called concurrently: await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]); guarded by ensuring none of them depend on a prior router's side effects.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, etc.). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' and reference it in each .default() call. A future multi-currency expansion then requires one change, not six, and eliminates the risk of typos creating inconsistent defaults.

### Add covering index on billReminders for due-date polling: ⚡ Medium Impact
Bill reminder notifications are driven by querying nextDueDate for a given userId. Without an index on (userId, nextDueDate) this query scans all reminders. Adding index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) will make the scheduler query O(log n) and is especially important if reminders are polled on a cron interval.

### Replace serial PKs with UUID PKs for user-facing tables: 🔥 High Impact
serial integer PKs on transactions, savingsGoals, and portfolioHoldings expose sequential IDs to the client, allowing users to enumerate other users' records by incrementing IDs — a significant IDOR risk for an app serving minors. Switching to uuid().defaultRandom().primaryKey() eliminates enumeration and also makes shard-safe IDs available if the DB is ever partitioned.

### Parallel startup I/O with Promise.all: 🔥 High Impact
probeDatabase, seedDatabase, registerRoutes, and startJobWorker are likely awaited sequentially. Running independent initialisation steps in parallel with Promise.all (e.g. [probeDatabase(), registerJobHandlers()]) would cut cold-start time proportionally to the slowest individual step rather than their sum, reducing platform health-check timeout risk.

### Structured logging with log levels: ⚡ Medium Impact
All logging currently uses console.log via the log() helper with no level filtering. Introducing a lightweight structured logger (pino is zero-dependency and ~5x faster than console.log) with level control via LOG_LEVEL env var would allow debug verbosity in development, suppress noisy lines in production, and produce machine-parseable JSON for log aggregators without the PII risk of the current full-body dumps.

### Scheduler extraction into a dedicated worker module: ⚡ Medium Impact
Weekly digest and purge scheduling logic lives inline in index.ts alongside HTTP server setup. Extracting schedulers into a schedulers.ts module that exports start() and stop() functions returning the interval handles would make graceful shutdown trivial, enable unit testing of scheduler logic in isolation, and keep index.ts focused solely on server wiring.

### Centralize admin route auth guard: 🔥 High Impact
Rather than relying on each query's enabled flag and a post-render redirect in useEffect, wrap the entire AdminDashboard route in a dedicated RequireAdmin component that checks the session synchronously and renders null or a redirect before any children mount. This eliminates the auth race at the architectural level and removes duplicated guard logic across query calls.

### Paginate or virtualize large admin tables: 🔥 High Impact
User and school tables appear to load and render the full dataset client-side. As the Caribbean school network grows this will cause slow initial renders and large memory usage. Moving to server-side pagination (adding page/limit query params to the API) or using a virtual list (TanStack Virtual) would keep render time constant regardless of record count.

### Extract AdminHelpChat into its own lazy-loaded module: ⚡ Medium Impact
AdminHelpChat pulls in its own state, the Loader2/Bot/Send icons, and chat rendering logic inside the already large AdminDashboard bundle. Splitting it to a separate file and loading it with React.lazy/Suspense would reduce the initial parse cost of the dashboard page, improving time-to-interactive for admins who rarely use the chat.
