## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 113-115)
**Issue:** No rate limiting is applied to AI chat, image generation, or audio routes before they are registered. These are the most expensive endpoints (OpenAI API costs, latency) and are completely unthrottled, enabling abuse and runaway billing.

**Fix:** Add dedicated AI rate limiters before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter is a stricter limiter (e.g. 10 req/min per IP) defined in rateLimiter.ts, applied before the registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 40) and sitemap.xml handler (~line 57)
**Issue:** Host header is taken directly from `req.headers['x-forwarded-host'] || req.headers.host` without validation and interpolated into output sent to browsers and crawlers. A forged Host header can produce URLs pointing to attacker-controlled domains in the sitemap and robots.txt, enabling cache poisoning or SEO manipulation.

**Fix:** Resolve the base URL from a single trusted environment variable: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and remove all dynamic host/proto detection from these handlers entirely. Set APP_BASE_URL in your deployment config.

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** Four sequential awaits (getUserXp, transactions count, gameSessions count, lessonRows, badgeRows) fire one after another per student inside the weekly-digest fan-out loop. For a class of 30 students this is 150+ sequential round-trips before any email is enqueued.

**Fix:** Wrap the independent queries inside studentWeekStats with Promise.all: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...).from(transactions)..., db.select(...).from(gameSessions)..., db.select(...).from(userLearningProgress)..., db.select(...).from(userBadges)...]). This cuts per-student latency to one parallel round-trip.

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** Inside bootstrapStudentContacts there is a per-student await db.select().from(emailContacts) inside a for loop, producing a classic N+1 query. For an org with 500 students this fires 500 individual SELECT statements synchronously.

**Fix:** Collect all student IDs first, then run a single batch query: const existingContacts = await db.select().from(emailContacts).where(and(eq(emailContacts.userKind, 'student'), inArray(emailContacts.userId, studentIds))). Build a Set of existing IDs and check membership in the loop instead of querying per student. The inArray import is already present in the file.

### `server/index.ts`: setInterval calls in maybeRunWeeklyDigest / maybeRunAiUsagePurge schedulers
**Issue:** The setInterval handles are not stored and never cleared. On Replit hot-reload or any in-process restart the old intervals keep firing alongside new ones, causing duplicate job enqueues, double-billing AI usage deletes, and a steady memory leak of closed-over DB references.

**Fix:** Assign each interval to a module-level variable: `const digestInterval = setInterval(maybeRunWeeklyDigest, INTERVAL_MS);`. In the graceful-shutdown handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before `httpServer.close()`.

### `server/index.ts`: res.json middleware / logLine construction
**Issue:** Every API response body is serialised with JSON.stringify and written to stdout unconditionally. For a financial literacy app with school-aged users this will log quiz scores, progress records, and any PII returned by the API in plaintext, violating basic data-minimisation principles and potentially COPPA/local data-protection rules.

**Fix:** Remove `capturedJsonResponse` capture entirely, or gate it behind `process.env.NODE_ENV !== 'production'`. At minimum redact the body and log only `res.statusCode` and `duration` in production: `if (process.env.NODE_ENV !== 'production') logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`;`

---

### `server/index.ts`: graceful shutdown / SIGTERM handler
**Issue:** Based on the visible code there is no SIGTERM/SIGINT handler. When Replit stops or redeploys the container, in-flight HTTP requests are dropped immediately, the job worker is not drained, and DB transactions can be left open.

**Fix:** Add `process.on('SIGTERM', shutdown); process.on('SIGINT', shutdown);` where `shutdown` calls `httpServer.close(cb)`, clears all intervals, signals the job worker to stop accepting new work, and awaits any in-flight job promise before `process.exit(0)`. Give it a 10-second timeout before forced exit.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / admin action mutations
**Issue:** Destructured admin auth state is used for UI gating but mutation callbacks (delete user, edit content, etc.) do not re-verify auth state before firing the API call. If the admin session expires mid-session, the UI flag may still show authenticated while mutations fire unauthenticated requests, leaking the fact that those endpoints exist and potentially receiving partial data in error responses that gets rendered.

**Fix:** In each mutation's onMutate or at the top of the mutationFn, check `if (!isAdminAuthenticated) { throw new Error('Session expired'); }` and redirect to login. Also ensure the server returns 401 and the mutation's onError handler redirects rather than showing a toast with response detail.

## 🟡 Warnings (16)

- **`server/routes.ts` (Supabase init chain (~line 118-127)):** The sequential `.then()` chain means a failure in any early step (e.g. `trimOrgNamesInSupabase`) silently skips all subsequent seeding steps, and only a single aggregated error is logged with no indication of which step failed. Individual step failures are swallowed.
  → *Run independent seed steps with `Promise.allSettled` and log each rejection individually: `const steps = [trimOrgNamesInSupabase, seedFinancialAcademyLesson, initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent]; await initSupabaseTables(); const results = await Promise.allSettled(steps.map(fn => fn())); results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] Step', steps[i].name, 'failed:', r.reason));`*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The `filePath` parameter from the URL is passed directly to `objectStorage.searchPublicObject(filePath)` without sanitization. A path like `../../secrets/config` could traverse outside the intended public-objects scope depending on the storage implementation.
  → *Sanitize the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Add `import path from 'path'` at the top of the file.*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export')):** The entire CSV is built as a single string in memory via Array.map and string concatenation before being passed to Buffer.from(). For large exports (thousands of rows with wide columns) this can exhaust the Node.js heap in a single job worker, and the full CSV string is also stored as the job result payload in the job table.
  → *Stream rows into a Transform stream or use a chunked approach: write headers once, then process rows in batches of 500 using a for loop appending to a writable stream piped directly to the object storage upload. Use objectStorageClient resumable upload or a pass-through stream. Remove the objectPath from the returned payload if it is stored in the jobs table; store only the fileName and rowCount, and retrieve the path from a separate metadata store.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage')):** The purge-ai-usage handler has no try/catch. If the DELETE fails (DB timeout, connection drop), the job framework receives an unhandled rejection. Depending on the job runner's behavior this may mark the job as succeeded or crash the worker process rather than triggering a retry.
  → *Wrap the handler body in try/catch: try { const result = await db.delete(...); return { deletedRows, cutoffDate }; } catch (err) { throw new Error(`purge-ai-usage failed: ${err instanceof Error ? err.message : String(err)}`); }. Rethrowing ensures the job runner marks it as failed and can retry with backoff.*
- **`server/jobHandlers.ts` (bootstrapStudentContacts / weekly-digest fan-out):** When orgId is falsy, the handler runs db.select().from(users) with no WHERE clause, fetching every user in the database into memory before iterating. In a multi-tenant deployment this is an unbounded full-table scan that will grow linearly with the user base and has no pagination or limit guard.
  → *Add a mandatory audience/orgId guard at the top of the weekly-digest handler and throw if both are absent: if (!orgId && audience !== 'all-confirmed-opt-in') throw new Error('weekly-digest requires orgId or explicit all-opt-in audience flag'). For the all-users path, process in cursor-paginated batches of 200 using a keyset on users.id with a WHERE id > lastSeenId LIMIT 200 loop rather than loading the full table.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets.userId despite it being the primary filter column for all budget queries. Every budget lookup for a user will do a full table scan as the table grows.
  → *Add an index in the budgets table definition: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to enforce one budget per category per period at the DB level.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt is defined as .defaultNow() without .notNull(), so the column is nullable in the DB schema. Any code filtering or ordering by createdAt must handle NULLs, and ORM-generated types will show it as Date | null, leading to defensive null checks throughout the codebase or silent bugs.
  → *Add .notNull() to every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default ensures inserts never need to supply the value, so notNull() is safe here.*
- **`shared/schema.ts` (userId varchar columns on all tables (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** userId is declared as varchar() with no length argument, which in PostgreSQL defaults to unlimited length. If users.id in the auth model is a fixed-length type (e.g. UUID char(36) or text), the mismatch can prevent index usage on joins and may cause silent type coercion issues. No length constraint also means no DB-level guard against malformed IDs.
  → *Align the type exactly with users.id. If users.id is a UUID, use uuid('user_id') from drizzle-orm/pg-core on all referencing columns. If it is text, change varchar() to text() for consistency. Run a migration to alter column types to match.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text() instead of a date or timestamp. Storing a date as a string means DB-level date comparisons, ordering, and range queries will use lexicographic sorting which silently returns wrong results for non-ISO formats and cannot use date indexes efficiently.
  → *Change lastPriceUpdateDate: text('last_price_update_date') to lastPriceUpdateDate: timestamp('last_price_update_date'). Update any application code that writes a string to pass a JS Date object instead. Add an index if this column is used in WHERE or ORDER BY for price-refresh scheduling.*
- **`shared/schema.ts` (savingsGoals and billReminders tables — missing userId indexes):** savingsGoals and billReminders have no indexes on userId. Both tables will be queried per-user on every dashboard load. As user count grows, these become sequential scans. billReminders.nextDueDate is also a likely ORDER BY and WHERE target with no index.
  → *Add index definitions: for savingsGoals add index('idx_savings_goals_user_id').on(t.userId); for billReminders add index('idx_bill_reminders_user_id').on(t.userId) and index('idx_bill_reminders_due_date').on(t.userId, t.nextDueDate) to support 'upcoming bills for user' queries efficiently.*
- **`server/index.ts` (main() or top-level startup block):** seedDatabase() and probeDatabase() are awaited sequentially before the HTTP server binds. If either takes several seconds (cold DB connection, large seed), the process appears hung and health checks fail. These are independent operations that can run concurrently.
  → *Replace sequential `await probeDatabase(); await seedDatabase();` with `await Promise.all([probeDatabase(), seedDatabase()]);` so both resolve in parallel, cutting startup latency by whichever is slower.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** `d.setHours(19, 0, 0, 0)` operates in the local timezone of the server process, which on Replit is UTC. If the intended audience is Caribbean (UTC-4/UTC-5), 7 PM local is 11 PM–midnight UTC, so the computed target will be consistently wrong by 4-5 hours, causing digests to fire at the wrong time or be skipped.
  → *Accept a timezone string from config (e.g. `DIGEST_TIMEZONE=America/Port_of_Spain`) and compute the target using `Intl.DateTimeFormat` or the `luxon`/`date-fns-tz` library to derive the correct UTC instant for Sunday 19:00 in that zone.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a new function on every render. React calls the old ref with null and the new ref with the element on every render, causing scrollIntoView to fire unnecessarily and producing a memory-pressure pattern in a chat that can grow long.
  → *Replace with a useRef<HTMLDivElement>(null) assigned to the div, then call scrollIntoView inside a useEffect that depends on [messages, loading]. This fires scroll only when content actually changes and eliminates the per-render function allocation.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – apiRequest POST /api/admin/help-chat):** The help-chat endpoint receives the full growing messages array on every send, meaning message history sent over the wire grows unboundedly. For school-aged users on Caribbean mobile connections this causes increasing payload sizes and latency per turn, and there is no maximum history cap client-side.
  → *Slice the history before sending: `messages: next.slice(-20)` (keep last 20 messages). Add a MAX_HISTORY constant set to 20 so it is easy to tune. This keeps payloads bounded without losing useful context.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple top-level useQuery calls for dashboard stats, users, schools, etc.):** Several useQuery hooks with independent keys fire sequentially during component mount because they share no coordination. While React Query does batch some internally, dependent UI sections (stats cards, charts, tables) each trigger their own network round-trips without any prefetch or parallel Promise.all strategy, adding 300-800ms of cumulative latency on slow connections.
  → *Use `useQueries` from @tanstack/react-query to declare all independent admin data queries in a single array. This allows React Query to fire them in parallel. Alternatively, create a single `/api/admin/dashboard-summary` endpoint that returns all bootstrap data in one request and cache it under one key.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort expressions in JSX render body):** User list filtering (search text match, role filter, school filter) and sorting are computed directly in the render return without useMemo. On every keystroke or parent re-render the filter runs over the full users array. With hundreds of school users this causes visible input lag in the search box.
  → *Extract the filtered and sorted users into a useMemo: `const filteredUsers = useMemo(() => users.filter(...).sort(...), [users, searchTerm, roleFilter, schoolFilter, sortField, sortDir])`. This ensures the O(n) work only runs when its specific dependencies change, not on every render.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six `await registerXRoutes(app)` calls after the parallel imports are sequential. If any route registration does async work (DB schema checks, cache warming), they block each other unnecessarily. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` to cut cold-start time proportionally to the slowest registrar rather than their sum.

### Structured startup logging for Supabase init steps: ⚡ Medium Impact
Currently init failures only appear as console.error strings. Replacing with a structured logger (e.g. pino) with step name, duration, and success/failure fields would make Railway/Sentry log aggregation actionable and allow alerting on seed failures that currently go unnoticed in production.

### Dynamic import caching for /healthz DB modules: 📌 Low Impact
Every request to `/healthz` calls `await import('./db')` and `await import('drizzle-orm')` dynamically. While Node caches modules after first load, the dynamic import syntax prevents bundlers from tree-shaking and adds unnecessary async overhead on each health check. Import `db` and `sql` statically at the top of the file alongside other imports.

### Idempotency key on bootstrapStudentContacts upsert: 🔥 High Impact
The current insert into emailContacts inside the fan-out loop has no ON CONFLICT clause visible in the truncated code. If the weekly-digest job is retried after partial completion, duplicate contact rows or duplicate enqueued emails can result. Adding ON CONFLICT (user_kind, user_id) DO NOTHING (or DO UPDATE SET updated_at = now()) to the insert and using a job-scoped deduplication key on enqueueJob calls (e.g. key: `weekly-digest-${weekStart}-${userId}`) would make the entire fan-out idempotent at no extra cost.

### Extract hardcoded 7-day window into environment config: ⚡ Medium Impact
The value 7 * 24 * 60 * 60 * 1000 for the digest lookback period is hardcoded in weekly-digest and duplicated alongside olderThanDays arithmetic in purge-ai-usage. Moving these to DIGEST_LOOKBACK_DAYS and AI_USAGE_RETENTION_DAYS env vars with validated numeric parsing at startup would let ops adjust retention without a deploy and eliminates the risk of the two usages drifting.

### Centralise job handler error boundary wrapper: ⚡ Medium Impact
Each handler currently has inconsistent or absent error handling. Wrapping registerJobHandler with a higher-order function withErrorBoundary(name, handler) that catches all thrown errors, logs them with the job ID and payload type (excluding PII), and rethrows would enforce a consistent error contract across all handlers without modifying each one individually. This also provides a single place to add OpenTelemetry span recording per job.

### Enforce single active budget per category via unique constraint: 🔥 High Impact
Add a uniqueIndex on budgets(userId, categoryId, period). Currently nothing at the DB level prevents duplicate budgets for the same category and period, which would cause incorrect budget vs. spend calculations silently. This constraint replaces application-layer duplicate checks and guarantees data integrity.

### Replace repeated currency text columns with a shared enum type: ⚡ Medium Impact
The string 'BSD' default appears in at least six tables with no DB-level constraint on valid values. Define a PostgreSQL enum or a currencies reference table and reference it via FK. This prevents invalid currency codes from being inserted and makes currency-aware reporting queries far simpler.

### Add a composite index on portfolioHoldings(userId, stockId): ⚡ Medium Impact
Portfolio value calculations join holdings to stocks filtered by userId. Without an index on (userId, stockId), every portfolio page load scans the full holdings table. A unique composite index on (userId, stockId) also enforces one holding row per stock per user, preventing quantity drift from duplicate inserts.

### Centralise all magic-number intervals into environment config: ⚡ Medium Impact
Scheduler polling intervals (weekly digest check cadence, purge check cadence) appear to be hardcoded millisecond literals. Moving them to `DIGEST_POLL_INTERVAL_MS` and `PURGE_POLL_INTERVAL_MS` env vars with documented defaults lets ops tune frequency without a redeploy and makes the values self-documenting in a `.env.example` file.

### Replace polling schedulers with a single unified tick loop: ⚡ Medium Impact
Two independent setInterval calls each wake up and hit the database independently. Consolidating them into one `setInterval(runScheduledTasks, POLL_MS)` that calls both async functions with `Promise.allSettled` halves DB round-trips on each tick, simplifies the single interval handle to clean up, and makes it easier to add future scheduled tasks.

### Add structured logging with log levels instead of raw console.log: 🔥 High Impact
All output goes through the custom `log()` helper which writes unstructured strings to stdout. Replacing this with a lightweight structured logger (pino is zero-dependency and extremely fast) enables log-level filtering (suppress debug in production), machine-parseable JSON for log aggregators, and automatic redaction of fields like `userId` or `email` via pino's `redact` option — important for a platform handling minors' data.

### Split AdminDashboard into route-level sub-pages: 🔥 High Impact
The single file exceeds 1,400 lines and mixes chat, user management, content management, analytics, and school management. Splitting into lazy-loaded route components (React.lazy + Suspense) per admin section would reduce initial JS parse time by an estimated 60-70% for any given admin view and make auth checks per-section explicit.

### Add optimistic updates to destructive mutations: ⚡ Medium Impact
Delete and status-toggle mutations currently wait for server confirmation before updating the UI, causing a frozen table row for 500-1500ms on Caribbean mobile latency. Implementing React Query's onMutate optimistic update pattern with rollback on onError would make the UI feel instant and is straightforward given the existing queryClient reference.

### Replace hardcoded API base paths with environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/admin/users', etc. are scattered inline throughout the file. Centralizing them in a typed API_ROUTES constant object (or a dedicated api-routes.ts) eliminates the risk of silent breakage when endpoints change and makes it possible to point at a staging base URL via VITE_API_BASE without touching component code.

### API Routes (`server/routes.ts`)
The routes file is generally well-structured with good patterns like parallel module imports and rate limiting, but has several reliability and security gaps. The most critical issues are unguarded promise chains in the Supabase init sequence that mask individual step failures, missing rate limiting on AI/chat/image/audio routes which are expensive endpoints, and host header injection risk in the robots.txt and sitemap generators.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including sequential database queries inside fan-out loops that will degrade under load, missing try/catch around async operations in job handlers that will cause silent failures, and large CSV payloads being constructed in memory before upload. The weekly-digest handler is the most problematic area with N+1 query patterns and no error boundaries per recipient.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable createdAt timestamps that should be NOT NULL, and an untyped varchar for userId that risks silent schema mismatches across all tables.

### Server Startup (`server/index.ts`)
The server startup sequence contains multiple blocking synchronous-style serial awaits that delay readiness, and the interval-based schedulers lack cleanup handles, causing memory leaks and zombie timers on hot reloads. Security is reasonable but response body logging exposes full JSON payloads (including potentially sensitive financial data) in plaintext logs, which is inappropriate for a school-aged user app.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx is a large monolithic component with several reliability and security concerns including unguarded async operations, missing auth verification on sensitive actions, and a memory leak from an inline ref callback creating a new function on every render. The chat scroll ref creates a new function instance on every render cycle, and sequential data fetches that could be parallelized add unnecessary latency for admin users.
