## 🔴 Critical (9)

### `server/storage.ts` — getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the associated category and a separate aggregation query computes spent amount, resulting in 2N+1 database round-trips for N budgets.

**Fix:** Rewrite using a single JOIN query: SELECT budgets.*, categories.*, SUM(transactions.amount) as spent FROM budgets LEFT JOIN categories ON budgets.categoryId = categories.id LEFT JOIN transactions ON transactions.categoryId = budgets.categoryId AND transactions.userId = budgets.userId GROUP BY budgets.id. This collapses all queries into one round-trip.

### `server/storage.ts` — getLeaderboard
**Issue:** The leaderboard query constructs a dynamic SQL fragment using the subject and period filter values without parameterization evidence visible in the truncated code. If these strings are interpolated into a raw sql`` template tag rather than passed as bound parameters, it is a SQL injection vector accessible to any authenticated student.

**Fix:** Replace any string interpolation inside sql`` with Drizzle bound parameters: sql`WHERE subject = ${filters.subject}` — the tagged template in Drizzle automatically parameterizes interpolated values. Audit every sql`` call in this file to confirm no JavaScript string concatenation is used inside the template literal.

### `server/storage.ts` — enrollStudent / removeEnrollment
**Issue:** enrollStudent inserts a ClassEnrollment row and likely updates an enrollment count or triggers downstream state, while removeEnrollment deletes the row. Neither operation appears wrapped in a database transaction, so a crash between steps leaves the database in a partially updated state (e.g., enrollment record exists but count is wrong, or vice versa).

**Fix:** Wrap both functions in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Perform all related reads and writes using the tx handle inside the callback so the entire operation is atomic and rolls back automatically on error.

### `server/routes.ts` — registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 110)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These call OpenAI APIs which are costly and latency-sensitive. A single unauthenticated user can spam these endpoints, causing runaway API spend and degraded service for all school-aged users.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` placed immediately before the three registerXRoutes calls.

### `server/routes.ts` — robots.txt handler (~line 47) and sitemap.xml handler (~line 62)
**Issue:** Both handlers build an absolute base URL by trusting `x-forwarded-proto` and `x-forwarded-host` headers directly from the request without any validation. An attacker can set these headers to an arbitrary domain, causing the sitemap and robots.txt to advertise a malicious URL — a classic host-header injection.

**Fix:** Read the canonical origin from an environment variable: `const base = process.env.CANONICAL_ORIGIN ?? 'https://finsightlite.com';` and ignore the forwarded headers entirely for URL construction in these two handlers.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full base64-encoded file content (fileB64) is stored directly in the job payload, meaning every exam upload bloats the jobs table with potentially megabytes of binary data per row. This creates DB bloat, slow job queue queries, and potential OOM when workers load many jobs into memory simultaneously.

**Fix:** Before enqueuing, upload the file buffer to object storage and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). In runExtractPaper, download the buffer from object storage using objectStorageClient at the start of processing, then delete the temp object on success or terminal failure.

### `server/index.ts` — setInterval calls inside maybeRunWeeklyDigest and maybeRunAiUsagePurge scheduler setup
**Issue:** The setInterval handles are never stored and never cleared. On SIGTERM/SIGINT the intervals keep the event loop alive, preventing clean shutdown, and if the module is ever re-evaluated in tests or hot-reload scenarios the intervals accumulate, causing duplicate job enqueues and a memory leak.

**Fix:** Store each handle: `const weeklyHandle = setInterval(maybeRunWeeklyDigest, INTERVAL_MS)`. Register a single graceful-shutdown function that calls `clearInterval(weeklyHandle)` and `clearInterval(purgeHandle)` before `httpServer.close()` in both SIGTERM and SIGINT handlers.

### `server/index.ts` — res.json middleware patch in the request-logging middleware (~line 62-80)
**Issue:** Every JSON response body is captured into capturedJsonResponse and serialized unconditionally into the log line. For a school-aged Caribbean financial literacy app this will log sensitive fields (balances, quiz scores, session tokens returned in API responses) to stdout in plain text, violating data-minimisation expectations and likely CARICOM/local privacy obligations.

**Fix:** Remove body capture from the logging middleware entirely. Log only method, path, status code, and duration. If response-body debugging is needed, gate it strictly on `process.env.LOG_RESPONSE_BODY === 'true'` and add a denylist of paths (e.g., /api/auth, /api/user) that are never logged regardless of the flag.

### `client/src/pages/AdminDashboard.tsx` — apiRequest POST /api/admin/help-chat
**Issue:** There is no client-side auth token or role assertion before calling /api/admin/help-chat. The component uses useAdminAuth at the page level but AdminHelpChat is a standalone component that does not verify admin status before sending requests. If the chat panel is ever extracted or rendered outside the auth-guarded route, it will make unauthenticated requests.

**Fix:** Accept an `isAdmin: boolean` prop from the parent (which already has useAdminAuth), and guard the sendMessage call: `if (!isAdmin) return;`. Alternatively call useAdminAuth inside AdminHelpChat and return null if not authenticated.

## 🟡 Warnings (17)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for total income, total expenses, category breakdown, recent transactions, and savings goals are executed one after another. These are independent queries with no data dependency between them, adding unnecessary latency proportional to the number of queries times average DB round-trip time.
  → *Wrap all independent queries in a single Promise.all call: const [income, expenses, categories, recentTx, goals] = await Promise.all([incomeQuery, expensesQuery, categoriesQuery, recentTxQuery, goalsQuery]). This reduces total latency to the duration of the slowest single query.*
- **`server/storage.ts` (getTransactions):** The limit filter parameter has no enforced maximum cap. A caller can pass limit: 1000000 or omit limit entirely, causing the query to return unbounded rows into memory. For school-aged users with potentially high transaction counts this is a memory and latency risk.
  → *Add a guard at the top of the function: const safeLimit = Math.min(filters?.limit ?? 50, 200). Apply safeLimit to the query's .limit() clause unconditionally. Never allow callers to bypass pagination by omitting the parameter.*
- **`server/routes.ts` (Supabase init chain (~line 118)):** The seven-step Supabase init chain is fully sequential (each .then() waits for the previous). If any step after the first fails, the .catch() fires but there is no indication of which step failed, and all subsequent steps are skipped silently. This can leave the DB in a partially seeded state with no actionable log.
  → *Run independent seed steps in parallel and log step identity on failure: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] Seed step ${i} failed:`, r.reason)));`. Keep initSupabaseTables and initStaticContentTables sequential only if they are true prerequisites.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint performs dynamic imports of `./db` and `drizzle-orm` on every invocation. Dynamic imports are cached after first load, but the pattern is misleading and adds unnecessary indirection. More critically, this endpoint is completely unauthenticated and reveals internal DB availability state, which is an information-disclosure risk for an app serving minors.
  → *Import db and sql statically at the top of the file. Add a check that restricts /healthz responses to internal callers only, e.g. verify a secret header `Authorization: Bearer $HEALTH_SECRET` set by the Railway health-check config, returning 401 otherwise.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with individual awaits. For a paper with 60 questions this means 60 serial round-trips to the DB, each adding latency and holding the job worker busy far longer than necessary.
  → *Collect all valid question objects into an array first, then issue a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, replace the loop with await Promise.all(validQuestions.map((q, i) => storage.createExtractedQuestion({...}))) to run inserts concurrently.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The openai client is imported from a routes file, coupling a job handler to HTTP layer internals. If that routes module initialises middleware, registers route handlers, or reads request-scoped state at import time, importing it in a background worker context can cause side effects or subtle failures. It also exposes the API key configuration path indirectly.
  → *Extract the openai client instantiation into a dedicated shared module (e.g., server/lib/openaiClient.ts) that exports only the configured OpenAI instance. Both the routes file and jobHandlers.ts should import from that shared module.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays comes directly from job.payload with no validation. A malformed or malicious payload (e.g., olderThanDays: 0 or a negative number) would compute a cutoff in the future and delete all AI usage records, or behave unexpectedly. There is also no try/catch, so a DB error will surface as an unhandled rejection rather than a structured job failure.
  → *Add input validation at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('Invalid olderThanDays'). Wrap the delete in a try/catch that logs the error and rethrows so the job framework records it as failed with a meaningful message.*
- **`shared/schema.ts` (categories table definition):** userId in categories uses bare varchar() with no length, but more critically has no index. Every lookup of user-specific categories (which happens on every transaction form load) will do a full table scan as the dataset grows.
  → *Add a named index: (t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) }). Also set varchar('user_id', { length: 255 }) to match the users.id type explicitly and prevent silent truncation.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these tables have indexes on userId, yet all will be queried exclusively by userId in every dashboard and feature screen. For a school-aged user base that could scale quickly, these become full sequential scans immediately.
  → *Add index definitions to each table's third argument. Example for budgets: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Do the same for all six tables with a consistent naming convention idx_{table}_user_id.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker has no unique constraint and no index. Duplicate tickers can be inserted silently, and any lookup by ticker symbol (e.g. searching for 'BTC') will full-scan the table. This is a data integrity hole in the investment simulation feature.
  → *Replace text('ticker').notNull() with text('ticker').notNull().unique() and add it to a uniqueIndex: (t) => ({ tickerIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }).*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt — and all other createdAt columns):** All createdAt columns use .defaultNow() but omit .notNull(). This means the column is nullable in the database type, so any INSERT that explicitly passes null will succeed and audit trail rows will have no timestamp, breaking time-based reporting and sorting.
  → *Chain .notNull() after .defaultNow() on every createdAt column: timestamp('created_at').defaultNow().notNull(). Also apply to purchasedAt and executedAt in portfolio tables.*
- **`shared/schema.ts` (billReminders — nextDueDate and frequency columns):** billReminders has no index on nextDueDate or userId+nextDueDate. The primary use case for this table is querying upcoming bills for a user sorted by due date — without this index every reminder check (likely a scheduled job) scans the full table.
  → *Add a composite index: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This covers both the user filter and the date ordering in a single index scan.*
- **`server/index.ts` (startJobWorker / registerJobHandlers / seedDatabase / probeDatabase call site in startup IIFE):** Sequential top-level awaits block the HTTP server from binding during startup. probeDatabase, seedDatabase, registerJobHandlers, and startJobWorker are awaited one-after-another even though several are independent, adding cumulative cold-start latency before the server accepts connections.
  → *Group independent startup tasks with Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])` before the tasks that depend on them (e.g., registerJobHandlers then startJobWorker). Bind the HTTP server to its port immediately after express setup and before the async startup work, using a 503 health-check response until readiness is confirmed.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / maybeRunAiUsagePurge):** The purge interval and the scheduler fire rate (the setInterval period) are not exposed as environment config, hardcoding both the 180-day retention window (partially addressed) and the polling cadence. If compliance requirements change (e.g., a regulator mandates 90-day retention), the polling cadence buried in the source must also change, and there is no startup log confirming the effective values being used.
  → *Add `AI_PURGE_INTERVAL_HOURS` env var (default 6) alongside the existing `AI_PURGE_OLDER_THAN_DAYS`. At startup log both effective values: `log('[startup] AI usage purge: retaining ${AUTO_PURGE_OLDER_THAN_DAYS} days, checking every ${INTERVAL_HOURS}h')`. Apply the same pattern to the weekly-digest interval.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The full conversation history (next: ChatMessage[]) is sent to /api/admin/help-chat on every message. If the conversation grows long this exposes potentially sensitive admin context in request payloads and logs, and there is no server-side truncation guard visible on the client. Additionally, the catch block swallows the error object entirely with a bare `catch {}`, losing stack traces for debugging.
  → *Slice the messages array to the last N turns before sending: `{ messages: next.slice(-10) }`. Change `catch {}` to `catch (err) { console.error('[AdminHelpChat] sendMessage failed:', err); }` so errors are observable without exposing them to the UI.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is recreated on every render and fires scrollIntoView every time any state changes (input typing, loading toggle), not just when a new message arrives. This causes jank during typing.
  → *Replace with a stable ref and a useEffect that only triggers on messages length change: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]); ... <div ref={bottomRef} />`*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — messages state):** Chat messages accumulate indefinitely in component state with no upper bound. For a long admin session this array can grow large, causing slow renders of the message list (every item re-renders on each new message) and increasing payload size on every send.
  → *Cap stored messages: after each assistant reply, trim state to the last 50 messages: `setMessages(prev => [...prev, reply].slice(-50));`. For the message list, add a stable key derived from role+index and consider virtualising with react-window if the list exceeds ~100 items.*

## 💡 Opportunities

### Centralized pagination middleware — 🔥 High Impact
Extract a getPaginationParams(query) helper that parses and validates page/limit from request query strings with enforced maximums, and apply it uniformly to all list-returning storage methods. This eliminates the current per-function inconsistency and prevents any future unbounded query from reaching the database, with near-zero added latency.

### Replace category hydration loops with JOIN queries — 🔥 High Impact
getBillReminders, getPortfolioHoldings, and getPortfolioTransactions each appear to fetch a list then loop to attach related entities. Replacing these with Drizzle leftJoin queries eliminates the N+1 pattern across all three methods, potentially reducing page-load DB time by 60-80% for users with moderate data volumes.

### Extract hardcoded virtual balance seed amount to environment config — ⚡ Medium Impact
The initial virtual balance value (likely 10000 or similar) used in getVirtualBalance when no record exists should be read from an environment variable VIRTUAL_BALANCE_SEED_AMOUNT with a documented default. This allows the product team to adjust starting balances for different school cohorts or sponsor campaigns without a code deployment.

### Centralised AI endpoint auth guard — 🔥 High Impact
Add a shared middleware that verifies session authentication before all /api/chat, /api/image, and /api/audio routes. Currently auth enforcement is delegated to each registered sub-router, risking inconsistency. A single `requireAuth` middleware applied at the prefix level guarantees no AI route is reachable by unauthenticated school users and reduces per-router boilerplate.

### Environment-driven canonical origin config — ⚡ Medium Impact
Introduce a validated CANONICAL_ORIGIN environment variable (fail fast on startup if absent in production) and use it in robots.txt, sitemap, and any future absolute-URL generation. This eliminates the host-header injection surface and makes the app portable across staging and production Railway environments without code changes.

### Structured startup logging for route registration — 📌 Low Impact
Replace silent success of the parallel domain-router imports with a structured log after each registerX call completes (e.g. `logger.info('Routes registered', { domain: 'ai', durationMs })`). During cold-starts on Railway this gives immediate visibility into which domain is slow to initialise, reducing mean-time-to-diagnose for latency regressions.

### Parallel admin CSV data fetches — 🔥 High Impact
The admin-csv-export handler likely fetches multiple tables sequentially (the truncated code pattern suggests this). Any independent DB queries (e.g., fetching users, transactions, and sessions for a report) should be wrapped in Promise.all. For a report touching 4 tables each taking ~200ms, this alone cuts export time from ~800ms to ~200ms.

### Idempotency token on extracted question bulk insert — ⚡ Medium Impact
The current idempotency strategy deletes all questions for a paper before re-extracting, which means a retry always burns two OpenAI API calls. Adding a status check at the start of runExtractPaper — if the paper is already 'completed', return the cached result immediately — would make retries free for the common case where the DB write succeeded but the job acknowledgement failed.

### Structured logging with sanitised payloads — ⚡ Medium Impact
Any logging inside these handlers that references job.payload risks writing base64 file content or full extracted text to log aggregators, causing excessive log volume and potential data exposure for a school-aged user app. Introduce a sanitisePayload utility that redacts known large or sensitive fields (fileB64, content, extractedText) before any log call, and apply it consistently across all job handlers.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it in each .default() call. When the app expands to TTD or JMD this becomes a one-line change instead of a six-location search.

### Add a composite unique constraint on portfolioHoldings (userId, stockId) — 🔥 High Impact
Currently a user can have multiple rows for the same stock, forcing application code to aggregate them. Adding a unique constraint on (userId, stockId) enforces one holding row per user per asset at the DB level, simplifies upsert logic, and prevents double-counting bugs in portfolio value calculations.

### Replace text() with varchar(255) for all FK reference columns — ⚡ Medium Impact
userId is declared as varchar() with no length in every table, while the referenced users.id type may have a defined length. Mismatched types can prevent index usage on joins in PostgreSQL and cause unexpected behavior. Standardise to varchar('user_id', { length: 255 }) across all tables to match the auth model explicitly.

### Centralised graceful-shutdown registry — 🔥 High Impact
Introduce a ShutdownManager that collects teardown callbacks (clearInterval, httpServer.close, db pool drain) registered from each module. A single SIGTERM/SIGINT handler iterates them with a 10-second timeout and process.exit(0/1) based on outcome. This eliminates the current scattered signal handling and ensures intervals, the HTTP server, and the DB connection all drain in a deterministic order, reducing the risk of mid-request data corruption on deploy.

### Structured JSON logging with field-level PII filtering — 🔥 High Impact
Replace console.log/log() calls with a structured logger (pino is zero-dependency and extremely fast). Configure a redact array covering known sensitive field names (e.g., 'password', 'token', 'balance', 'score'). This eliminates the manual capturedJsonResponse leak, gives log-level control per environment, and produces machine-parseable output ready for a Caribbean-hosted log aggregator without a post-processing pipeline.

### Startup readiness probe decoupled from HTTP bind — ⚡ Medium Impact
Bind the HTTP server immediately on startup and return 503 from a /healthz route until async initialisation (seed, job worker, DB probe) completes. Set a boolean `isReady` flag after Promise.all resolves. This lets container orchestration (Replit, Railway, Fly.io) detect the process is alive within milliseconds while still protecting routes from requests before the DB is ready, removing the current window where the port is unbound and the platform may restart the dyno.

### Parallel data fetching with Promise.all — 🔥 High Impact
The main dashboard likely fires multiple useQuery calls sequentially (users, schools, teachers, audit log). Where these are independent, consolidate into a single aggregated endpoint or ensure React Query's parallel queries are used rather than chained awaits inside useEffect. This would reduce dashboard initial load time by the sum of sequential round-trip latencies.

### Extract AdminHelpChat into its own file with React.memo — ⚡ Medium Impact
AdminHelpChat lives inside the same module as the large AdminDashboard. Every state change in AdminDashboard (search input, pagination, modal open) will re-evaluate the entire module scope. Moving AdminHelpChat to its own file and wrapping with React.memo prevents it from re-rendering on unrelated parent state changes.

### Environment-config for API base URL and chat endpoint path — ⚡ Medium Impact
The string '/api/admin/help-chat' is hardcoded inline. In a multi-environment Caribbean deployment (staging, production, disaster-recovery) this should come from an env var (e.g. VITE_API_BASE_URL). Centralise all API path constants in a single lib/api-routes.ts file to prevent silent breakage during environment promotion.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic reliability and performance issues: N+1 query patterns in budget/bill-reminder lookups, missing Promise.all parallelization in dashboard aggregation, and absence of pagination guards on unbounded list queries. For a school-aged financial app handling real user data, the lack of transactions around multi-step writes and missing input validation on raw SQL expressions are the most critical concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good use of Promise.all for parallel imports and proper error handling in most async paths. Key concerns are an unguarded Supabase init chain that swallows individual step failures silently, missing rate limiting on the AI/chat/image/audio endpoints which are the heaviest and most abuse-prone routes, and a host-spoofing vulnerability in the robots.txt and sitemap handlers.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads are stored in the job table (memory and DB pressure), sequential DB inserts inside a loop should be batched, and the openai import from a routes file is an abstraction leak that risks exposing sensitive config. Error handling is mostly present but a few async paths lack boundaries.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing NOT NULL constraints on audit timestamps, unindexed foreign keys on high-query tables, and an undersized varchar for userId that could cause silent truncation across all tables.

### Server Startup (`server/index.ts`)
The server startup sequence contains blocking synchronous-style sequential awaits that delay readiness, and the interval-based schedulers lack cleanup references causing memory leaks and preventing graceful shutdown. The response-logging middleware serializes every JSON response body unconditionally, creating a PII/data-exposure risk for a school-aged financial app and a potential memory pressure issue under load.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx is a large monolithic component with several reliability and security gaps. Key concerns include unguarded async operations sending sensitive chat history to the server, inline scroll refs creating potential memory leaks, and missing useCallback on frequently-recreated handlers passed into child components causing unnecessary re-renders.
