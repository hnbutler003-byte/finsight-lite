## 🔴 Critical (6)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied to AI/chat/image/audio routes before they are registered. These are the highest-cost endpoints (OpenAI API calls) and are completely unthrottled, exposing the app to runaway API costs and denial-of-wallet attacks from school-aged users or bad actors.

**Fix:** Apply a dedicated aiLimiter (e.g. 20 req/15min per IP) immediately before registering these routes: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); — create the limiter in rateLimiter.ts alongside authLimiter.

### `server/routes.ts` — /public-objects/:filePath(*) handler
**Issue:** The filePath param is passed directly to objectStorage.searchPublicObject without any sanitization. A request like /public-objects/../secrets/env could traverse into unintended storage paths depending on the ObjectStorageService implementation, constituting a path traversal vulnerability.

**Fix:** Sanitize filePath before use: const safePath = path.normalize(filePath).replace(/^\.\.\/|^\//, ''); then validate it does not start with '..' or '/'. Reject with 400 if the normalized path differs from the original or contains traversal sequences.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full base64 file content is stored in the job payload row. For any non-trivial exam paper this can easily be 1–10 MB per row, bloating the jobs table, causing slow job queue polls, and leaking binary data into DB backups and logs.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the buffer from object storage at the start of the handler using objectStorageClient, then delete the staging object on success or terminal failure. The job row payload stays under 1 KB.

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** The olderThanDays value comes directly from job.payload with no validation before being used in arithmetic to construct a Date cutoff passed to the database. A payload with olderThanDays: 0 or a negative number would delete all records or produce an invalid date, causing irreversible data loss.

**Fix:** Add a guard at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); This keeps the job in a retryable-error state rather than silently destroying data.

### `client/src/pages/AdminDashboard.tsx` — ObservabilityCard — sentry.projectUrl link
**Issue:** sentry.projectUrl is rendered directly as an href from the API response without any validation. A compromised or misconfigured backend could return a javascript: URI, creating a XSS vector in the admin panel — the most sensitive surface in this app.

**Fix:** Validate the URL before rendering: const safeProjectUrl = useMemo(() => { try { const u = new URL(sentry?.projectUrl ?? ''); return ['https:','http:'].includes(u.protocol) ? u.href : null; } catch { return null; } }, [sentry?.projectUrl]); then only render the anchor when safeProjectUrl is non-null.

### `client/src/pages/AdminDashboard.tsx` — TABS constant — static array at module scope
**Issue:** TABS references Lucide icon components (LayoutDashboard, Globe, etc.) directly in a module-level array literal. This is fine for the icons, but the array is re-used as a render-time source of truth with no useMemo guard at the call site. More critically, the 'dbviewer' and 'perf' tabs appear to be always present regardless of auth role; if useAdminAuth returns a non-superadmin role, these sensitive tabs are still rendered and presumably query sensitive endpoints.

**Fix:** Filter TABS based on the authenticated admin's role before rendering the sidebar: const visibleTabs = useMemo(() => adminUser?.role === 'superadmin' ? TABS : TABS.filter(t => !['dbviewer','perf','jobs'].includes(t.id)), [adminUser?.role]); This prevents lower-privilege admins from even seeing — and clicking into — privileged views.

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...) chain):** The sequential .then() chain means a failure in any intermediate step (e.g. seedFinancialAcademyLesson) silently skips all subsequent seeds and only logs one error with no indication of which step failed. Additionally, steps that are independent (e.g. seedRegionalContent, seedGameContent) run sequentially when they could be parallelized.
  → *Wrap each step individually so failures are isolated and identifiable: run them in a self-invoking async IIFE with individual try/catch per step, or group independent seeds with Promise.allSettled([seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r,i) => r.status==='rejected' && console.error(`Seed step ${i} failed:`, r.reason))).*
- **`server/routes.ts` (/healthz route, dynamic import inside handler):** Every call to /healthz triggers dynamic import('./db') and import('drizzle-orm'). While Node caches modules after first load, the dynamic import machinery still executes on each request and adds unnecessary overhead to what should be a sub-millisecond health check endpoint.
  → *Move the db and sql imports to the top of the file as static imports alongside the other imports. The /healthz handler then references them directly with no runtime import cost.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one inside a sequential for-await loop. For a 40-question paper this issues 40 serial round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Replace the loop with a single bulk insert: filter valid questions into an array first, then call db.insert(extractedQuestions).values(validRows) in one statement. Wrap it in a transaction with the updateExamPaper call so both succeed or fail atomically.*
- **`server/jobHandlers.ts` (runExtractPaper — JSON.parse on extractionResponse):** JSON.parse is called on the raw OpenAI response string with no try/catch. If the model returns malformed JSON (which happens, even with response_format: json_object on certain error paths), the job throws an untyped SyntaxError that bypasses the structured error handling and leaves the paper stuck in 'processing' if it is not the final attempt.
  → *Wrap the JSON.parse in a try/catch. On parse failure, throw a typed Error with a descriptive message so the outer catch can set status to 'failed' on the terminal attempt and the retry logic can re-attempt on earlier attempts.*
- **`server/jobHandlers.ts` (runExtractPaper — model: 'gpt-4o' and max_completion_tokens: 4096):** The OpenAI model name and token limits are hardcoded in two separate completions.create calls. Changing the model for cost or capability reasons requires a code deploy, and the 30000-char text slice limit is also a magic number with no explanation.
  → *Add OPENAI_EXTRACTION_MODEL (default 'gpt-4o'), OPENAI_EXTRACTION_MAX_TOKENS (default 4096), and EXTRACTION_TEXT_CHAR_LIMIT (default 30000) to environment config and reference process.env values with fallbacks. Document the 30000 limit rationale in a comment (GPT-4o context ceiling with room for system prompt).*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, yet every budget query will filter by userId. Full table scans will occur as user data grows.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). If a user should only have one budget per category per period, make it a uniqueIndex on (userId, categoryId, period) instead, which also enforces business logic.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on userId or nextDueDate. Queries for upcoming bills due for a user (WHERE user_id = ? ORDER BY next_due_date) will do a sequential scan and an in-memory sort.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the table definition.*
- **`shared/schema.ts` (transactions and budgets createdAt columns):** createdAt uses .defaultNow() but has no .notNull(). A null createdAt on a financial transaction is a data integrity violation and will silently break any ORDER BY or reporting query that assumes the column is populated.
  → *Change all audit timestamp columns to timestamp('created_at').defaultNow().notNull(). Apply the same fix to linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (categories table definition — userId column):** categories.userId is declared as varchar without a length argument, which maps to TEXT in Postgres and prevents index-only scans. More critically, there is no index on userId even though category lookups are always scoped to a user or to system categories (userId IS NULL).
  → *Match the varchar length to the users.id type (e.g., varchar('user_id', { length: 255 })). Add (t) => ({ userIdx: index('idx_categories_user').on(t.userId) }) so partial scans for per-user categories are fast.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker has no unique constraint and no index. Duplicate tickers can be inserted silently, and any lookup by ticker symbol (the natural key for stock data) will do a full table scan.
  → *Add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to the table definition. This enforces uniqueness at the DB level and provides an efficient index for ticker-based lookups.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo, and the queryFn closes over the params object rather than a stable string. This means any parent re-render rebuilds params and potentially triggers a cache miss in React Query even when the filter values have not changed.
  → *Wrap the params construction in useMemo keyed on [actorType, action, from, to], then derive a stable paramsString from it. Pass paramsString into queryKey instead of the raw filter values, and reference paramsString inside queryFn: const paramsString = useMemo(() => { const p = new URLSearchParams(); ... return p.toString(); }, [actorType, action, from, to]);*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set("limit", "200")):** The page size limit of 200 is hardcoded as a magic string literal. For a school-age financial app with growing audit trails this will silently truncate results and is impossible to tune without a code change.
  → *Extract to a named constant at module scope: const AUDIT_LOG_PAGE_LIMIT = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 200); then use params.set('limit', String(AUDIT_LOG_PAGE_LIMIT)). This allows environment-level tuning without redeployment.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — queryFn):** The queryFn has no error handling. If the /api/admin/observability endpoint returns a non-OK response, r.json() on an error body (e.g. HTML 502 page) will throw an unhandled rejection that React Query will silently swallow because there is no onError handler and no error UI rendered in ObservabilityCard.
  → *Add an explicit check in the queryFn: queryFn: async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error(`Observability fetch failed: ${r.status}`); return r.json(); } and destructure isError from useQuery to render a fallback Badge with 'Unavailable' state.*

- **`server/routes.ts` (/api/health and /healthz — duplicate health check routes):** Two health check endpoints exist (/api/health and /healthz) with different behaviors: /api/health returns a trivial JSON response with no DB check, while /healthz actually validates the database connection. ## 💡 Opportunities

### Parallel domain route registration — ⚡ Medium Impact
The six domain routers are imported in parallel with Promise.all (good) but then registered sequentially with six serial awaits. If any registerX function performs async I/O during registration (DB checks, cache warming), these block each other. Collect them and run await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) — eliminates cumulative cold-start latency proportional to the number of routes that do async work at registration time.

### Structured logging instead of console.error for Supabase init failures — ⚡ Medium Impact
Supabase init errors are caught and logged with console.error, losing structured context (which seed failed, timestamp, correlation ID). Routing these through the existing Sentry integration or a structured logger (pino/winston) would surface failures in dashboards and enable alerting — critical for a production app where silent seed failures leave students with missing lesson content.

### Sitemap generation should be cached — 📌 Low Impact
The sitemap is regenerated on every request including string interpolation and array mapping. For a static URL list that never changes at runtime, generate the XML once at startup into a module-level constant and serve that string directly. Eliminates per-request allocation with zero functional trade-off.

### Parallel object-storage existence check and header set in streamPrivateObjectToResponse — 📌 Low Impact
The function currently awaits file.exists() before setting any headers, which is correct, but the Content-Type and Content-Disposition headers are set synchronously after the async check. More impactfully, callers that need metadata (size) alongside existence could combine those into a single getMetadata call, eliminating a second round-trip for any future callers that add Content-Length support. Adding Content-Length now would also prevent Nginx/proxies from chunking small exports unnecessarily.

### Dead-letter / alerting hook on terminal job failures — ⚡ Medium Impact
The extract-paper handler syncs status to 'failed' on terminal attempts, but no alert or dead-letter event is emitted. Adding an enqueueJob('notify-admin', { jobType, jobId, error }) call inside the terminal-failure catch block (or a generic postFailure hook in the job runner) would give operations visibility into extraction failures affecting real students without requiring log scraping.

### Transaction wrapping for extract-paper idempotency delete + bulk insert + status update — 🔥 High Impact
Currently the delete of old extracted questions, the question inserts, and the updateExamPaper are three separate database operations with no transaction boundary. A crash between delete and insert leaves the paper with zero questions and status still 'processing'. Wrapping all three in a single db.transaction() call makes the idempotency guarantee airtight and removes a class of partial-failure states that are currently invisible to the UI.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a const DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it in each .default() call. This prevents silent drift if the app expands to TTD or JMD and makes a future config-driven change a one-line edit.

### Add a partial index for active/pending document uploads — ⚡ Medium Impact
documentUploads rows in 'processing' status are the ones polled most frequently, but the table has no indexes at all. A partial index CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing' will make status-polling queries near-instant and avoids scanning completed/failed rows that dominate the table over time.

### Replace serial PKs with uuid on user-facing tables — 📌 Low Impact
Using serial (sequential integer) PKs on transactions, savingsGoals, and portfolioTransactions exposes record counts and insert rates to any authenticated user who can read their own IDs — a meaningful information leak for a financial app. Switching to gen_random_uuid() UUIDs eliminates enumeration risk with no query-performance penalty given the existing FK indexes.

### Split AdminDashboard into lazy-loaded tab modules — 🔥 High Impact
The file is 126 KB+ with every tab panel in one bundle. Code-splitting each tab with React.lazy/Suspense would cut the initial parse time significantly for Caribbean mobile users on slower connections. Each tab (AuditLogPanel, ObservabilityCard, etc.) is already a discrete component — wrapping them in dynamic imports is low-risk and would reduce the main chunk by an estimated 60-70%.

### Consolidate overview stat queries with Promise.all — ⚡ Medium Impact
The dashboard overview tab likely fires multiple independent useQuery calls sequentially (organizations count, schools count, students count, etc.). Replacing them with a single /api/admin/stats endpoint that runs all DB aggregations in parallel server-side, or at minimum batching client-side fetches with Promise.all inside a single queryFn, would reduce waterfall latency on dashboard load by the number of round trips saved.

### Add staleTime and refetchOnWindowFocus:false to audit and observability queries — ⚡ Medium Impact
Audit log and observability data fetched with default React Query settings will re-fetch every time an admin switches browser tabs. For a 200-row audit query this is unnecessary load on the DB. Setting staleTime: 60_000 and refetchOnWindowFocus: false on these queries would eliminate spurious refetches with no UX regression since admins explicitly have a Refresh button.
