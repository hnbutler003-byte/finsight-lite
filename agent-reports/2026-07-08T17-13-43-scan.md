## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. Any authenticated or unauthenticated caller can hammer these endpoints, incurring unbounded OpenAI API costs and enabling denial-of-wallet attacks — especially dangerous for a school-aged user base.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP using express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions.

### `server/routes.ts`: robots.txt handler (~line 43) and sitemap.xml handler (~line 58)
**Issue:** Both handlers construct the base URL from `req.headers['x-forwarded-host']` and `req.headers['x-forwarded-proto']` without any validation or allowlist. An attacker can inject an arbitrary host header, causing the Sitemap directive and canonical URLs to point to a malicious domain — a host-header injection / cache-poisoning vector.

**Fix:** Read the trusted base URL from environment config: `const base = process.env.APP_BASE_URL || 'https://finsightlite.com';` Remove all dynamic header-derived URL construction from these two handlers. Set APP_BASE_URL in Railway/Replit env vars.

### `server/jobHandlers.ts`: registerJobHandler callbacks (all handlers)
**Issue:** No handler wraps its body in try/catch. If any handler throws (e.g. object storage unavailable, DB connection drop), the promise rejects without a caught error boundary. Depending on the job runner implementation this can produce an unhandled rejection that crashes the Node process or silently marks the job as failed with no structured error payload.

**Fix:** Either enforce a wrapper inside registerJobHandler itself that catches and normalises errors, or wrap each handler body: try { ... return result; } catch (err) { throw new JobError(err instanceof Error ? err.message : String(err), { jobId: job.id, kind: job.kind }); }. At minimum add a global process.on('unhandledRejection') guard that logs and does not exit for job worker processes.

### `server/jobHandlers.ts`: bootstrapStudentContacts (inside weekly-digest handler)
**Issue:** When orgId is absent the handler does db.select().from(users) with no column projection, fetching every column of every user row including password hashes, tokens, or any other sensitive fields added to the users table in future. These full rows are iterated in memory and the email/username fields are extracted.

**Fix:** Project only the required columns: db.select({ id: users.id, email: users.email, username: users.username }).from(users). This prevents accidental inclusion of sensitive columns and reduces payload size over the wire from the DB.

### `server/index.ts`: bootstrap / main IIFE (startup sequence)
**Issue:** seedDatabase(), probeDatabase(), registerRoutes(), and startJobWorker() are awaited sequentially on the startup critical path with no timeout guard. If any one hangs (e.g. DB unreachable), the process stalls silently — no request is ever served and no exit occurs.

**Fix:** Wrap the entire bootstrap in a Promise.race against a 30-second timeout that calls process.exit(1) with a descriptive message. Also run independent init steps in parallel: await Promise.all([probeDatabase(), seedDatabase()]) before the steps that depend on them, then registerRoutes() and startJobWorker() together where possible.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are started but their IDs are never stored and never cleared. On SIGTERM/SIGINT the intervals keep the event loop alive, delaying or preventing clean shutdown, and if the module is ever re-evaluated in tests the intervals stack up, causing duplicate job enqueues.

**Fix:** Store both IDs: const weeklyDigestInterval = setInterval(...) and const purgeInterval = setInterval(...). In the graceful shutdown handler call clearInterval(weeklyDigestInterval) and clearInterval(purgeInterval) before closing the HTTP server and DB pool.

---

### `server/index.ts`: res.json override inside the request-logging middleware
**Issue:** The middleware monkey-patches res.json and stores the full parsed response body in capturedJsonResponse for every API request, then JSON.stringifies it again into the log line. For endpoints that return user financial records, AI outputs, or PII this will print sensitive data to stdout/log aggregators in plaintext — a compliance risk for an app serving school-aged users under Caribbean data-protection rules.

**Fix:** Remove capturedJsonResponse capture entirely, or limit logging to status code and duration. If response body inspection is genuinely needed for debugging, log only the top-level keys (Object.keys(body)) and only when NODE_ENV !== 'production', never the values.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook call / top-level of AdminDashboard component
**Issue:** The file imports and calls `useAdminAuth` but based on the visible code the component renders its full JSX tree before any auth-redirect logic could navigate away. If `useAdminAuth` only sets state or fires a side-effect, the dashboard data queries (useQuery calls) will still execute and their results will be briefly visible to unauthenticated users before the redirect completes.

**Fix:** After calling `useAdminAuth`, check its returned auth state synchronously: `const { isAdmin, isLoading } = useAdminAuth(); if (isLoading) return <Loader2 />; if (!isAdmin) { setLocation('/admin/login'); return null; }` — this prevents any child queries from mounting and any data from rendering for unauthorized users.

## 🟡 Warnings (10)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...) (~line 108)):** The Supabase initialisation chain runs all seven steps sequentially in a single promise chain. If any intermediate step (e.g. seedFinancialAcademyLesson) throws, the `.catch` swallows the error and all subsequent steps (seedStaticModules, seedRegionalContent, seedGameContent) are silently skipped with no per-step visibility.
  → *Wrap each step individually so failures are logged but don't abort the chain: replace the `.then` chain with an async IIFE that awaits each call inside its own try/catch block, logging `[Supabase] Step X failed: e.message` for each. This ensures all steps are attempted and failures are individually observable.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 95)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject()` without sanitisation. A path traversal payload like `../../private/keys` could resolve to unintended storage paths depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and reject any path containing `..` segments: `if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });`*
- **`server/jobHandlers.ts` (studentWeekStats):** Five sequential awaits (getUserXp, txCount query, gameCount query, lessonRows query, badgeRows query) run one-after-another per user inside studentWeekStats. For the weekly-digest fan-out this compounds per-student, adding ~4x avoidable latency per contact.
  → *Wrap the four independent DB calls in a single Promise.all: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...txCount), db.select(...gameCount), db.select(...lessonRows), db.select(...badgeRows)]); This cuts per-student latency to roughly the slowest single query.*
- **`server/jobHandlers.ts` (admin-csv-export handler):** The entire CSV is built as a single string in memory (csv = [cols.join(','), ...rows].join('\n')) then converted to a Buffer before upload. For large exports (thousands of students/transactions) this allocates the full dataset twice in the V8 heap simultaneously and blocks the event loop during string concatenation.
  → *Stream the CSV directly to object storage using a PassThrough stream: create a PassThrough, pipe it into objectStorageClient.bucket(b).file(o).createWriteStream({contentType}), write the header row, then write each data row incrementally with stream.write(rowString + '\n') inside the loop. Await the finish event. This keeps memory usage proportional to one row at a time.*
- **`server/jobHandlers.ts` (weekly-digest handler — bootstrapStudentContacts inner loop):** Inside bootstrapStudentContacts there is a per-student await db.select().from(emailContacts).where(...) executed sequentially inside a for loop. For an org with 500 students this fires 500 serialised round-trips to check for existing contacts before inserting, creating an O(n) sequential DB fan-out.
  → *Batch the existence check: before the loop, fetch all existing emailContacts for the relevant userKind and orgId in one query into a Set<string> keyed by userId. Replace the per-iteration SELECT with a Set.has(s.id) lookup, then collect missing entries and bulk-insert them with db.insert(emailContacts).values(batch) after the loop.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function uses new Date() and setHours(19,0,0,0) which operates in the server's local timezone. On a cloud host (typically UTC) '7 pm' is UTC 19:00, not Caribbean local time (UTC-4/UTC-5), so the digest fires 4-5 hours early and the weekStart key computed from toISOString() can fall on the wrong calendar day near midnight.
  → *Replace setHours with explicit UTC arithmetic using setUTCHours and offset by the desired Caribbean timezone (e.g. UTC-4: setUTCHours(23, 0, 0, 0) for 7 pm AST). Alternatively use a named timezone with the Temporal API or the 'luxon' library already common in TS projects. Also source the timezone offset from an env var (DIGEST_TIMEZONE_OFFSET_HOURS) so it is configurable without a code change.*
- **`server/index.ts` (graceful shutdown handler (SIGTERM/SIGINT block)):** There is no graceful shutdown handler visible in the file. The HTTP server and database pool are never explicitly closed, so in-flight requests are dropped, DB connections leak, and the job worker may lose work mid-execution when the container restarts.
  → *Add: process.on('SIGTERM', shutdown) and process.on('SIGINT', shutdown) where shutdown = async () => { clearInterval(weeklyDigestInterval); clearInterval(purgeInterval); await new Promise(res => httpServer.close(res)); await db.$client.end(); process.exit(0); }. Give the shutdown a hard 10-second deadline via setTimeout(() => process.exit(1), 10_000).unref().*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > Messages div > div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a new function object on every render of the messages list, causing the browser to fire scrollIntoView every single time any state changes (including the input field changing on each keystroke), not just when a new message arrives.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`. This restricts scrolling to actual message changes only.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function in AdminHelpChat):** The sendMessage function calls `res.json()` without checking `res.ok`, so a 4xx or 5xx HTTP response from `/api/admin/help-chat` is silently treated as a successful reply. If the server returns an error JSON with no `reply` field the catch block is never hit and the user sees no feedback.
  → *After `const res = await apiRequest(...)` add `if (!res.ok) throw new Error(res.statusText);` before calling `res.json()`. This ensures non-2xx responses fall into the catch block which already appends a user-visible error message.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage in AdminHelpChat — messages state passed to API):** The full conversation history (`messages: next`) is sent to `/api/admin/help-chat` on every message with no length cap. A user can accumulate an unbounded array that grows the request payload indefinitely, risking payload size errors and unnecessarily high token costs on the backend LLM call.
  → *Slice the history before sending: `messages: next.slice(-20)` to cap at the last 20 messages. Add a visible indicator when history is trimmed (e.g. a faded 'Earlier messages not included' separator) so the admin understands context limits.*

## 💡 Opportunities

### Parallel Supabase seed execution: ⚡ Medium Impact
The seven Supabase init/seed steps are chained sequentially via `.then`. Steps that are not interdependent (e.g. seedRegionalContent, seedGameContent, seedStaticModules) could run concurrently with Promise.allSettled, reducing server cold-start time measurably — especially relevant on Railway where startup time affects first-request latency.

### Centralised route registration error boundary: ⚡ Medium Impact
Each `await register*Routes(app)` call can throw independently with no per-call error handling. Wrapping the block in a try/catch with specific logging per domain (auth, students, ai, orgs) would make startup failures debuggable in production logs rather than surfacing as a generic unhandled rejection crashing the process.

### Dynamic import caching for /healthz: 📌 Low Impact
`/healthz` re-executes `await import('./db')` and `await import('drizzle-orm')` on every request. Although Node caches modules after first load, the async import() call still adds overhead on a hot path. Hoist these imports to the top of the file as static imports to eliminate the per-request dynamic resolution cost.

### Idempotency key on weekly-digest fan-out: 🔥 High Impact
If the weekly-digest job is retried (worker crash after partial enqueue), duplicate send-email jobs are enqueued for already-processed contacts. Add a deterministic idempotency key composed of weekStart + userId to each enqueueJob call, and have the job table UPSERT on conflict ignore. This prevents duplicate emails on retry with zero extra queries.

### Centralise job error serialisation: ⚡ Medium Impact
Each handler currently returns ad-hoc result shapes and throws raw errors. Introduce a thin JobResult<T> type and a withJobErrorBoundary(handler) higher-order wrapper inside registerJobHandler that catches, logs with structured fields (jobId, kind, durationMs), and re-throws a normalised JobError. This eliminates the need for per-handler try/catch and makes job failure metrics uniform across all handlers.

### Move large export payloads out of job row: ⚡ Medium Impact
The admin-csv-export handler stores objectPath and rowCount in the job result column, which is fine, but upstream callers may be tempted to store the raw CSV in the payload column when enqueuing. Enforce a payload size guard (e.g. JSON.stringify(payload).length > 64KB throws at enqueue time) to prevent the job table becoming a de-facto blob store, which degrades queue poll query performance as the table grows.

### Parallel startup initialization: ⚡ Medium Impact
probeDatabase() and seedDatabase() are independent and can run concurrently via Promise.all. On a cold start against a remote DB each can take 1-3 seconds; parallelizing them cuts startup latency by roughly half, reducing the window where the process is alive but not yet serving health checks.

### Externalize scheduler intervals to environment config: ⚡ Medium Impact
The weekly digest and purge check intervals are hardcoded as numeric literals (e.g. every 5 minutes). Expose them as DIGEST_CHECK_INTERVAL_MS and PURGE_CHECK_INTERVAL_MS env vars with safe defaults. This lets ops teams adjust polling frequency in production without a redeploy and makes integration tests fast by setting a 100 ms interval.

### Replace res.json monkey-patch with a structured request-id logger: 🔥 High Impact
The current approach of overriding res.json per-request allocates a closure and a reference on every request and risks capturing sensitive payloads. Replace it with a correlation-ID middleware (uuid v4 attached to res.locals.requestId) and log only method, path, status, duration, and requestId. This eliminates the memory allocation per request and removes the PII logging risk in one change.

### Parallel admin dashboard data fetching: 🔥 High Impact
The dashboard likely fires multiple sequential useQuery calls (users, schools, teachers, analytics) that are independent. Ensuring all queries have their own query keys and are declared at the top level (not conditionally) lets React Query issue them in parallel. For any server-side aggregation still using sequential awaits, replace with Promise.all. Expected impact: 40-60% reduction in initial dashboard load time.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Admin tables with search/filter/sort state recalculate derived lists on every render including unrelated state changes (e.g. chat open/close toggling). Wrapping filter+sort pipelines in useMemo(()=>..., [rawData, searchTerm, sortField]) ensures O(n) work only runs when inputs change, keeping the UI responsive with hundreds of rows.

### Extract AdminHelpChat into its own file: ⚡ Medium Impact
AdminHelpChat is a self-contained stateful component with its own types, state, and network calls embedded at the top of a file that is already over 147,000 characters. Splitting it into client/src/components/AdminHelpChat.tsx enables independent testing, reduces the parse/compile unit for the dashboard, and makes the auth guard at the top of AdminDashboard easier to reason about.
