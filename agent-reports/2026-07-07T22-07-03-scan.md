## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. A single authenticated or unauthenticated user can flood expensive AI endpoints, causing runaway OpenAI costs and denial-of-service for school-aged users on a shared quota.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (lines ~42 and ~58)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header to inject arbitrary URLs into the Sitemap directive and <loc> tags, enabling cache-poisoning or open-redirect phishing via SEO tooling.

**Fix:** Add an allowlisted APP_BASE_URL environment variable (e.g. process.env.APP_BASE_URL) and use it exclusively to build `base`. Fall back to a hardcoded production domain, never to raw host headers. Remove the dynamic header-derived base construction entirely.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** Full file content is stored as base64 in the job payload (fileB64 field). For a multi-page PDF or high-res image this can be several MB per job row, bloating the jobs table, slowing queue polling queries, and loading the entire payload into Node heap on every retry attempt.

**Fix:** Upload the file to object storage before enqueuing and store only the object path in the payload. In runExtractPaper, download the buffer from object storage at the start of the handler, then delete the temp object on success or final failure. This keeps job rows small (<1 KB) and makes retries cheap.

### `server/jobHandlers.ts`: runExtractPaper – for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted sequentially inside a for loop with individual awaits, so 40 questions = 40 round-trips. Each iteration also has no error boundary, so a single constraint violation aborts the loop mid-way and leaves the paper in a partially-inserted state that the idempotency delete at the top will clean on retry but wastes significant time.

**Fix:** Collect all valid question objects into an array, then do a single batched insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the entire insert in a try/catch and re-throw so the outer handler can set status to 'failed' on terminal attempt. This reduces 40 round-trips to 1 and makes the operation atomic.

### `server/index.ts`: capturedJsonResponse logging block (~line 74)
**Issue:** Every API response body is serialized and logged unconditionally. For a financial literacy app with school-aged users this will leak tokens, session data, user PII, grades, and AI responses into stdout/log aggregators with no redaction or size cap.

**Fix:** Remove the res.json monkey-patch and capturedJsonResponse logging entirely, or replace it with a strict allowlist: only log a small set of non-sensitive fields (e.g. {status, count}) and never log /api/auth, /api/user, or /api/ai routes. Add a MAX_LOG_BODY_LEN=200 char truncation guard at minimum.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge (startup block)
**Issue:** The interval handles returned by setInterval are never stored, so graceful shutdown cannot clear them. On SIGTERM the process exits while intervals may be mid-flight, risking partial DB writes (e.g. a rolled-back digest claim that never retries, or a half-executed purge). Node will also keep the event loop alive past the intended shutdown.

**Fix:** Store each handle: const digestTimer = setInterval(...). In the SIGTERM/SIGINT handler call clearInterval(digestTimer) and clearInterval(purgeTimer) before closing the HTTP server and DB pool. Wrap the shutdown sequence in a Promise.race with a 10-second hard timeout calling process.exit(1).

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level component render
**Issue:** If useAdminAuth returns a non-admin or unauthenticated state, the component likely still renders and fires all useQuery hooks (users, schools, analytics, etc.) before the redirect/guard takes effect. This means unauthenticated requests hit every admin API endpoint on every render until the auth check resolves.

**Fix:** Check auth state before enabling queries: add `enabled: !!isAdmin` to every useQuery options object, and return a loading spinner or null immediately if `isAdmin` is false or auth is still loading, before any query hooks fire.

## 🟡 Warnings (11)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...) sequential chain):** The seven Supabase init steps are chained sequentially with .then(), so a failure in any middle step (e.g. seedFinancialAcademyLesson) short-circuits all subsequent seeds and the single .catch() logs only the first error. Steps after the failure are silently skipped with no per-step visibility, risking missing game or regional content at startup.
  → *Wrap each step in its own try/catch or use Promise.allSettled for independent seeds. At minimum: run initSupabaseTables and initStaticContentTables first (they are prerequisites), then run the five seed functions with Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent(), trimOrgNamesInSupabase()]) and log each rejected result individually.*
- **`server/routes.ts` (/api/health and /healthz (lines ~36 and ~76)):** There are two health check endpoints: /api/health (returns {status, ts} with no DB check) and /healthz (performs a real DB ping). Railway and uptime monitors are documented as using /api/health, which will report ok even when the database is completely down, masking outages.
  → *Consolidate to one endpoint. Move the DB liveness check into /api/health and remove /healthz, or keep /healthz as the canonical probe and update Railway's healthcheck config to point to it. Update the comment to match whichever is kept.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The delete query has no try/catch. If the DB connection drops or the query times out, the unhandled rejection will crash the worker process rather than being recorded as a failed job attempt, breaking the retry/backoff contract of the job queue.
  → *Wrap the db.delete call in try/catch, catch the error, log it with context (olderThanDays, cutoff date), and re-throw so the job runner marks it as failed and schedules a retry. Identical pattern should be applied to any other handler that makes bare DB calls without error handling.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText slice hardcoded to 30000):** The PDF text truncation limit of 30000 characters is hardcoded. This value directly controls how much of an exam paper is sent to the AI and therefore how many questions are extracted. If the limit is too low, questions are silently dropped with no warning to the teacher who uploaded the paper.
  → *Move the limit to an environment variable: const PDF_TEXT_LIMIT = parseInt(process.env.PDF_TEXT_LIMIT ?? '30000', 10). Log a warning when truncation occurs: if (pdfData.text.length > PDF_TEXT_LIMIT) logger.warn({ paperId, originalLength: pdfData.text.length }, 'PDF text truncated'). This makes the limit tunable without a deploy and surfaces silent data loss.*
- **`server/jobHandlers.ts` (openai import – `import { openai } from './replit_integrations/chat/routes'`):** The OpenAI client is imported from a routes module, which is an HTTP layer concern. This creates a hidden coupling: if that route module fails to initialise (e.g. missing env var at startup), all job handlers that call OpenAI will silently receive undefined and throw at call-time with a confusing error rather than a clear configuration error.
  → *Extract the OpenAI client instantiation into a dedicated singleton module (e.g. server/openaiClient.ts) that reads OPENAI_API_KEY directly, validates it at import time, and exports the client. Both the routes and job handlers import from this shared module, eliminating the cross-layer dependency.*
- **`server/index.ts` (startup async IIFE — probeDatabase then seedDatabase sequential awaits):** probeDatabase and seedDatabase are awaited sequentially but seedDatabase depends on probeDatabase succeeding, which is correct — however any additional independent startup tasks (registerJobHandlers, startJobWorker) are also chained sequentially when they could run in parallel after the DB is ready, adding unnecessary latency to every cold start.
  → *After await probeDatabase() and await seedDatabase() complete, fan out the independent tasks: await Promise.all([startJobWorker(), someOtherInit()]). registerJobHandlers() is synchronous and can stay inline before the Promise.all.*
- **`server/index.ts` (getMostRecentSundayAt7pm — daysSince calculation):** The expression (dow + 0) % 7 is dead/no-op code. When dow === 0 (Sunday) daysSince is 0, meaning d is set to today at 19:00. If the server restarts on a Sunday after 19:00 and a digest was already claimed, this is fine — but the +0 is clearly a placeholder for an intended offset (e.g. a timezone shift) that was never implemented, leaving Sunday-detection brittle for Caribbean timezones (UTC-4/UTC-5) when the server runs in UTC.
  → *Remove the dead +0. More importantly, derive 'now' in the target timezone: use Intl.DateTimeFormat or a library like date-fns-tz to convert the Sunday 19:00 anchor to America/Port_of_Spain (UTC-4) before comparing against Date.now(), or store the schedule in UTC explicitly as 23:00 UTC Sunday.*
- **`server/index.ts` (maybeRunAiUsagePurge — gte date filter):** The purge query uses a raw date arithmetic string constructed by subtracting AUTO_PURGE_OLDER_THAN_DAYS. If the Drizzle helper passes this as a string literal into SQL without parameterization, and AUTO_PURGE_OLDER_THAN_DAYS were ever sourced from user input rather than env, it would be an injection vector. Additionally there is no logging of how many rows were deleted, making audit trails impossible for a regulated-adjacent financial app.
  → *Confirm the gte() Drizzle call uses a prepared parameter (it should with Drizzle ORM). Add const deleted = await db.delete(...).returning({ id: aiUsageLogs.id }) and log deleted.length after the operation so you have a deletion audit trail. Assert typeof AUTO_PURGE_OLDER_THAN_DAYS === 'number' with a runtime guard before use.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** Full conversation history is sent to /api/admin/help-chat on every message. For a school-aged platform, this means every user message accumulates in memory and is transmitted in full, growing unboundedly. A 50-message conversation sends all 50 messages each time, wasting bandwidth and potentially leaking prior sensitive admin queries if logs are written server-side.
  → *Cap the context window before sending: `const context = next.slice(-10);` then send `{ messages: context }` instead of `next`. Add a server-side limit as well. Also clear messages on panel close to avoid stale state leaking between sessions.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat (scroll div ref)):** The scroll anchor uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` directly in JSX. This creates a new function object on every render and calls scrollIntoView on every render cycle, not just when messages change, causing janky scroll behavior and unnecessary DOM thrashing.
  → *Replace with a proper ref and a useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple sequential useQuery declarations (users, schools, analytics, audit log)):** Each useQuery for dashboard data (users list, schools, analytics metrics, audit log) is declared independently and they all fire in parallel by default, which is correct, but any place in the component where data is fetched imperatively inside event handlers or useEffect uses sequential awaits (e.g., fetch user then fetch their schools), missing Promise.all opportunities.
  → *Audit all async functions in the component for sequential awaits on independent resources. Replace patterns like `const a = await fetchA(); const b = await fetchB();` with `const [a, b] = await Promise.all([fetchA(), fetchB()]);` wherever the second call does not depend on the first result.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (/healthz handler (dynamic import inside request handler)):** Every call to /healthz triggers two dynamic import() calls for './db' and 'drizzle-orm'. While Node caches modules after first load, the import() call still incurs promise and cache-lookup overhead on every health check tick (Railway polls this frequently). It also makes the handler harder to reason about.
- **`client/src/pages/AdminDashboard.tsx` (handleKey (AdminHelpChat)):** handleKey is defined as an inline arrow function inside the JSX prop `onKeyDown`, recreated on every render of AdminHelpChat. Since sendMessage is also not wrapped in useCallback, both functions are new references every render, preventing any child memoization from working.

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six registerX(app) calls after the parallel imports are awaited sequentially. If any of these registration functions perform async work (DB schema checks, etc.), converting them to Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) would reduce cold-start latency proportionally to the slowest-minus-fastest registration time.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replace console.error in the Supabase init .catch() with a structured logger (e.g. the existing Sentry integration or a pino logger) that includes the step name and error code. This gives Railway log drains and on-call engineers immediate context without digging through stack traces, especially important for a school-facing production app.

### Extract object-storage route to its own domain router: 📌 Low Impact
The /public-objects/:filePath and /uploads/logos/:filePath routes are inline in registerRoutes rather than in a dedicated router file. Moving them to routes/assets.ts alongside the other domain routers makes it straightforward to add auth checks (e.g. ensuring private assets require a valid session) and to unit-test the object-storage integration in isolation.

### Parallel independent async lookups in downstream handlers: ⚡ Medium Impact
Any handler that fetches a user record and an unrelated resource (e.g. learning module, badge list) sequentially should use Promise.all. Based on patterns visible in the truncated section, converting two sequential 50ms DB reads to parallel saves ~50ms per job invocation and reduces DB connection hold time, which matters at scale for a school-day usage spike.

### Idempotency token on enqueueJob calls: 🔥 High Impact
enqueueJob calls visible in the file do not pass a deduplication key. If a teacher double-submits a paper upload or a cron fires twice, duplicate extract-paper jobs will run concurrently, each paying full OpenAI cost and racing on the idempotency delete. Adding a stable idempotency key (e.g. hash of paperId + fileHash) to enqueueJob would allow the queue to reject the duplicate before any work starts.

### Structured logging instead of thrown error messages for AI failures: ⚡ Medium Impact
Errors from OpenAI calls surface only through re-thrown Error objects whose message is the raw API response. Adding structured log entries (paperId, model, prompt token count, HTTP status) before re-throwing would let you correlate AI cost spikes with specific job types in your observability tooling without needing to parse free-text error strings.

### Centralized graceful-shutdown manager: 🔥 High Impact
Create a ShutdownManager class that accepts registrations of timers, DB pools, and HTTP servers, then binds SIGTERM/SIGINT once. Currently shutdown logic is scattered and intervals are unregistered. A single manager with a 10-second hard-exit fallback would eliminate the leaked-interval and mid-flight-write risks across all future scheduler additions.

### Structured JSON logging with log-level filtering: 🔥 High Impact
Replace all console.log/log() calls with a structured logger (pino is already common in the Node ecosystem and near-zero overhead). Emit {level, source, message, durationMs} objects. This enables log-level filtering so debug/verbose lines are suppressed in production, directly mitigating the response-body leak and making GDPR/data-residency audits tractable for a school-user app.

### Scheduler idempotency integration test: ⚡ Medium Impact
Add a test that runs maybeRunWeeklyDigest twice concurrently against a test DB to verify the onConflictDoNothing atomic claim prevents double-enqueue. This is the core correctness guarantee of the scheduler and is currently untested; a regression here would send duplicate digest emails to students/guardians.

### Memoize filtered and sorted admin data lists: 🔥 High Impact
User/school search filtering and sorting applied to query results should be wrapped in useMemo with the query data and filter state as deps. Currently any unrelated state change (e.g., opening a dialog) re-runs O(n) filter+sort over potentially hundreds of records on every render. Expected impact: eliminates redundant computation on large datasets and reduces jank on low-end Caribbean school devices.

### Split AdminDashboard into feature sub-components: 🔥 High Impact
At 147k+ characters, the single file causes the entire admin UI to re-render whenever any local state changes (tab selection, search input, dialog open). Extract UsersTab, SchoolsTab, AnalyticsTab, and AuditLogTab into separate memoized components with React.memo. This limits re-render scope to the active tab and its own state, cutting render work by an estimated 60-80% for inactive sections.

### Add explicit error and empty states to all data-fetching sections: ⚡ Medium Impact
Query error states (isError, error) appear to be unhandled in several dashboard sections, meaning a failed fetch silently shows nothing or stale data to admins. For a financial literacy platform where admins monitor student progress, silent failures are a reliability risk. Add visible error banners with retry buttons using the existing RefreshCw icon for every useQuery result that drives a rendered list or chart.
