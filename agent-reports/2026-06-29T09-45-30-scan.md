## 🔴 Critical (10)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate SELECT per budget row to calculate 'spent' amount, plus another SELECT per row to join category data. For a class with 30 students each viewing budgets, this becomes hundreds of round-trips.

**Fix:** Replace the per-budget loops with a single JOIN query using Drizzle's leftJoin(categories, eq(budgets.categoryId, categories.id)) and a subquery or groupBy aggregation for spent amounts: SELECT budgets.*, categories.*, COALESCE(SUM(transactions.amount),0) as spent FROM budgets LEFT JOIN categories ON ... LEFT JOIN transactions ON transactions.categoryId = budgets.categoryId AND transactions.userId = budgets.userId GROUP BY budgets.id. One query replaces N+2.

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Stock trade execution spans three separate async writes (upsert holding, insert portfolio transaction, update virtual balance) with no wrapping database transaction. A failure between steps leaves the database in a corrupt state — e.g., balance debited but holding not updated — with no rollback path.

**Fix:** Wrap all three operations in a single Drizzle transaction: await db.transaction(async (tx) => { await tx.insert(portfolioHoldings)...; await tx.insert(portfolioTransactions)...; await tx.update(userVirtualBalance)...; }). Any thrown error will automatically rollback all three writes.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. For a school-aged user base this is a critical cost and abuse vector — a single user or bot could exhaust API quota or generate inappropriate content at scale.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before calling registerChatRoutes etc. Also add per-authenticated-user limits if sessions are available at that point.

### `server/routes.ts`: robots.txt handler (~line 38) and sitemap.xml handler (~line 53)
**Issue:** Both handlers construct a base URL by directly interpolating `req.headers['x-forwarded-host']` and `req.headers.host` into the response without any validation. A malicious request with a crafted Host header will reflect attacker-controlled URLs in sitemap.xml and robots.txt, enabling cache-poisoning or phishing via search engine indexing.

**Fix:** Define `APP_BASE_URL` as a required environment variable (e.g. `https://finsightlite.com`) and use it as the canonical base. Fall back to the header-derived value only in local dev after validating it matches an allowlist: `const base = process.env.APP_BASE_URL ?? derivedBase;`

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter / extract-paper handler)
**Issue:** The raw base64-encoded file content is stored directly in the job payload, which lives in the job table. For a multi-page PDF or high-res image this can easily be 5–20 MB per row, causing table bloat, slow job-queue queries, and potential OOM when workers deserialize many jobs concurrently.

**Fix:** Before enqueuing, upload the file buffer to private object storage and store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of the handler. This keeps job rows to tens of bytes and makes retries cheaper.

### `server/jobHandlers.ts`: runExtractPaper – Vision API call (~line where image_url is constructed)
**Issue:** buffer.toString('base64') is called on a Buffer that was itself decoded from fileB64 via Buffer.from(fileB64, 'base64'). If fileB64 is already a valid base64 string this round-trip is a no-op and wastes CPU, but if the stored payload was ever re-encoded (e.g. JSON stringify of the job row) the image data sent to OpenAI will be corrupted and the Vision call will silently return garbage text.

**Fix:** Pass the original fileB64 string directly to the data URI: use `data:${mimeType};base64,${fileB64}` instead of re-encoding the decoded buffer. Remove the intermediate Buffer.from/toString('base64') round-trip for the vision branch.

### `server/index.ts`: seedDatabase / probeDatabase / startJobWorker calls in top-level startup block
**Issue:** probeDatabase, seedDatabase, and startJobWorker are awaited sequentially in the startup IIFE, meaning the HTTP server is not listening until all three finish. If any one hangs (e.g. DB unreachable), the process stalls silently with no timeout and no health-check endpoint available to a load balancer.

**Fix:** Start httpServer.listen() immediately after registerRoutes(), then run probeDatabase/seedDatabase/startJobWorker in a background async function with a per-operation AbortSignal or Promise.race timeout (e.g. 10 s). Log and exit if the DB probe fails, but let the server respond 503 from a /healthz route during init so orchestrators don't kill it prematurely.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are never stored and never cleared. On SIGTERM/SIGINT the intervals keep firing, the process cannot exit cleanly, and if the module is hot-reloaded (Replit dev mode) new intervals are added on top of old ones, causing duplicate job enqueues and a memory leak.

**Fix:** Store both handles: const weeklyHandle = setInterval(...). Register a single graceful-shutdown function that calls clearInterval(weeklyHandle), clearInterval(purgeHandle), then httpServer.close(cb) and drains the job worker before process.exit(0). Attach it to both process.on('SIGTERM') and process.on('SIGINT').

### `server/index.ts`: res.json monkey-patch in the request-logging middleware (~line 68)
**Issue:** capturedJsonResponse holds the full JSON body of every API response and is stringified unconditionally into the log line. For a Caribbean school app this will print student grades, quiz answers, guardian contact details, and any AI-generated content verbatim into stdout/Sentry breadcrumbs with no redaction or size cap.

**Fix:** Remove the full-body capture entirely, or limit logging to safe metadata: status code, response size (res.get('content-length')), and duration. If body logging is needed for debugging, gate it on a LOG_RESPONSE_BODIES=true env var that is off by default, and strip keys like 'email', 'token', 'password', 'answer' before serialising.

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** The help-chat endpoint is called with the full growing message array on every send, and there is no server-side auth token attached to the /api/admin/help-chat request beyond whatever cookie the browser sends. If the route lacks its own admin middleware check, any authenticated non-admin user who discovers the endpoint can query the admin AI assistant.

**Fix:** Verify /api/admin/help-chat uses requireAdminAuth middleware server-side. On the client, confirm apiRequest forwards the session cookie and add an explicit check: if (!isAdmin) return; at the top of sendMessage so the call is never made from a non-admin context. Also cap the messages array sent to the last N turns (e.g. slice(-10)) to avoid unbounded payload growth.

## 🟡 Warnings (17)

- **`server/storage.ts` (getLeaderboard):** getLeaderboard accepts an optional limit parameter but the interface definition shows no default cap and no maximum guard. Without a hard upper bound, a caller can request an unlimited result set from gameSessions joined with users, causing a full table scan returned into Node.js heap memory.
  → *Add a server-enforced cap: const safeLimit = Math.min(filters?.limit ?? 50, 200); and apply .limit(safeLimit) to the query unconditionally. Never trust the caller to supply a reasonable limit for leaderboard-style queries.*
- **`server/storage.ts` (getDashboardStats):** getDashboardStats fires sequential awaits for total income, total expenses, savings goals, budgets, and recent transactions — each a separate round-trip — when all five are fully independent queries that can run in parallel.
  → *Replace sequential awaits with: const [income, expenses, goals, budgetRows, recentTx] = await Promise.all([db.select(...), db.select(...), db.select(...), db.select(...), db.select(...)]); This will reduce dashboard load time by roughly (N-1) * avg_query_latency, typically cutting 4 serial round-trips to 1 parallel batch.*
- **`server/storage.ts` (getTransactions):** getTransactions accepts an optional limit filter but applies no default limit and no maximum cap. On a student account with years of imported transactions, this returns the entire transactions table for that userId into memory. The JOIN with categories for each row compounds the allocation.
  → *Set a mandatory default: const pageLimit = Math.min(filters?.limit ?? 100, 500); and always apply .limit(pageLimit) to the query. Add an offset parameter to the interface for proper pagination. Also verify the (userId, createdAt) composite index exists on the transactions table to avoid full-partition scans.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 108)):** The sequential `.then()` chain means a failure in any intermediate step (e.g. `seedFinancialAcademyLesson`) causes all subsequent seeds to be silently skipped with only a single catch-level error log. In production this leads to partially seeded state that is hard to diagnose.
  → *Run independent seeds in parallel and surface individual failures: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r,i) => r.status==='rejected' && console.error(`[Supabase] Seed ${i} failed:`, r.reason)));` Keep `initSupabaseTables` and `initStaticContentTables` sequential only if they are true prerequisites.*
- **`server/routes.ts` (/healthz handler (~line 75)):** The `/healthz` endpoint performs dynamic `import('./db')` and `import('drizzle-orm')` on every health check invocation. Uptime monitors and Railway call this endpoint every 10-30 seconds, causing repeated module resolution overhead and obscuring actual DB latency in the timing.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of routes.ts as static imports (they are already used transitively). The health handler then just calls `await db.execute(sql\`select 1\`)` directly with no dynamic import cost.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96)):** The `filePath` parameter accepts arbitrary path segments including `../` sequences. If `objectStorage.searchPublicObject` or `downloadObject` passes this value to a filesystem or storage SDK without sanitisation, it is a path traversal vulnerability exposing private objects.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).end();` Add `import path from 'path'` at the top. Confirm objectStorage methods also do not join this value onto a sensitive base path.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Each valid question triggers an individual awaited DB insert inside a sequential for-loop. For a 40-question exam paper this is 40 serial round-trips. If any single insert fails mid-loop, previously inserted questions are orphaned (the earlier delete-for-idempotency only runs at the top of the next attempt, so a partial set is briefly visible to readers between retries).
  → *Collect all valid question objects into an array and perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the delete and bulk insert in a single DB transaction so partial states are never visible: await db.transaction(async (tx) => { await tx.delete(...); await tx.insert(...).values(...); }).*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export') and any multi-step handlers with sequential awaits):** Based on the visible pattern and the truncated admin-csv-export handler, independent data fetches (e.g. fetching users, then transactions, then badges for a report) are likely sequential awaits. The truncated section also obscures whether there are missing try/catch blocks around the CSV serialization and uploadPrivateObject call, which would leave the job stuck with no status update on failure.
  → *Run independent queries with Promise.all: const [users, txns, badges] = await Promise.all([fetchUsers(), fetchTxns(), fetchBadges()]). Wrap the entire handler body in try/catch; on catch, update the export job record to a 'failed' status (mirroring the pattern used in extract-paper) so the UI reflects the failure rather than hanging.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet every budget query will filter by userId. With growing user data this becomes a full table scan on every budget load.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a uniqueIndex on (userId, categoryId, period) to enforce one budget per category per period at the DB level.*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, portfolioTransactions table definitions):** None of these tables define indexes on userId or their primary query columns (e.g. nextDueDate on billReminders, executedAt on portfolioTransactions). Every user-scoped query will do a sequential scan.
  → *Add index definitions: billReminders needs index on (userId, nextDueDate), savingsGoals on (userId), portfolioHoldings on (userId, stockId), portfolioTransactions on (userId, executedAt). Use the same (t) => ({}) third-argument pattern already used on transactions.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders — all createdAt columns):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). Drizzle will infer it as nullable in TypeScript, meaning application code must handle null for an audit field that should never be null.
  → *Append .notNull() to every createdAt column: timestamp('created_at').defaultNow().notNull(). Same fix applies to purchasedAt on portfolioHoldings and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker is declared as text('ticker').notNull() with no unique constraint. Multiple rows with the same ticker symbol would cause silent data inconsistency when portfolio holdings reference a stock by ticker for display or price lookup.
  → *Add a uniqueIndex on ticker: (t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This is a one-line addition in the table's third argument and prevents duplicate seed data from corrupting portfolio calculations.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes 'most recent Sunday' using (dow + 0) % 7, where the +0 is a no-op left from a copy-paste. More critically, it operates on local server time with no timezone specification. On a Replit dyno (UTC) this fires at 19:00 UTC; Caribbean schools are UTC-4 to UTC-5, so the digest runs at 2-3 pm local time, not 7 pm as intended.
  → *Replace the wall-clock arithmetic with an explicit timezone. Accept a TZ env var (e.g. DIGEST_TIMEZONE=America/Port_of_Spain) and use Intl.DateTimeFormat or the 'date-fns-tz' library to compute the target instant in that zone. Remove the dead +0.*
- **`server/index.ts` (sequential awaits in startup (probeDatabase, seedDatabase, registerJobHandlers, startJobWorker)):** registerJobHandlers is purely synchronous and startJobWorker does not depend on seedDatabase's result, yet all four calls are awaited in series. On a cold start with a slow DB this adds unnecessary latency before the server becomes ready.
  → *Await probeDatabase first (it must pass before anything else). Then run Promise.all([seedDatabase(), startJobWorker()]) since seeding and worker startup are independent. Call registerJobHandlers() (sync) before the Promise.all as it has no async dependencies.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll ref callback):** The auto-scroll sentinel uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` with no cleanup. React calls this callback with null when the element unmounts, but because the conditional `if (el)` guards against that, it silently swallows the cleanup path. More importantly, this pattern re-runs scrollIntoView on every render, not just when messages change, causing jarring scroll resets when unrelated parent state updates.
  → *Replace with a proper useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and render `<div ref={bottomRef} />`. This scopes the scroll to actual message changes only.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats data fetching (multiple useQuery calls at top of component)):** Multiple independent useQuery hooks (users, schools, lessons, transactions, etc.) are declared sequentially, which is correct for parallel fetching, but there is no unified loading or error state rendered at the dashboard level. If any query fails, the section silently renders empty/undefined data with no user feedback, which is especially problematic for a financial admin tool.
  → *Collect isLoading and isError from each query: `const isLoading = usersQuery.isLoading || schoolsQuery.isLoading || ...`. Render a single skeleton or error banner when any critical query is in a bad state rather than letting each widget render with undefined data silently.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline mutation callbacks passed directly as JSX props (e.g. onClick handlers for delete/edit actions inside mapped lists)):** Inline arrow functions like `onClick={() => handleDeleteUser(user.id)}` are recreated for every item on every parent render. In a list of hundreds of users/schools this creates significant GC pressure and forces all list-item children to re-render even when unrelated state (e.g. search input) changes.
  → *Wrap handler factories with useCallback at the component level, or better yet extract list-item rows into their own memoised sub-components (e.g. `const UserRow = memo(({ user, onDelete, onEdit }) => ...)`) and pass stable useCallback-wrapped handlers from the parent. This also makes the file easier to maintain given its size.*

## 🔵 Info / Tech Debt (3)

- **`server/jobHandlers.ts` (runExtractPaper – model name and token limits):** The OpenAI model name 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two separate API calls. Changing the model for cost or capability reasons requires a code deploy, and the token cap silently truncates responses for long papers without any warning logged or surfaced to the caller.
- **`shared/schema.ts` (userId varchar columns across all tables):** All userId columns are declared as varchar('user_id') with no length argument. In PostgreSQL this defaults to unlimited length, which is fine functionally, but it loses the DB-level enforcement of expected ID length (e.g. Clerk user IDs are fixed-length prefixed strings). More critically, without .notNull() on the categories.userId optional FK, a misconfigured insert could silently write a null userId on what should be a user-owned category.
- **`client/src/pages/AdminDashboard.tsx` (Hardcoded API base paths and pagination defaults (e.g. page size constants inline in JSX)):** Values such as page sizes, the help-chat API path string '/api/admin/help-chat', and any chart colour hex strings are hardcoded inline throughout the file. For a school-facing Caribbean deployment these may need to vary per environment or white-label config, and hunting through 147 k of JSX to update them is error-prone.

## 💡 Opportunities

### Batch category resolution with an in-memory Map: 🔥 High Impact
Multiple methods (getBillReminders, getTransactions, getBudgets) independently re-fetch the same categories rows per request. Fetch all categories for a userId once per request and resolve via a Map<number, Category> lookup. Eliminates redundant round-trips across the request lifecycle and is a low-risk, high-frequency win for the student dashboard path.

### Move hardcoded virtual balance seed amount to environment config: ⚡ Medium Impact
The initial virtual balance granted to new users (visible in seedMarketData / getVirtualBalance fallback) appears hardcoded (e.g., 10000.00). This should be INITIAL_VIRTUAL_BALANCE_CENTS in environment config so it can be adjusted per school sponsor or curriculum without a code deploy.

### Add structured error logging with userId scrubbing before log emission: ⚡ Medium Impact
Async operations in storage.ts lack try/catch with structured logging. When errors are added, ensure userId is the only identifier logged — never log transaction amounts, card numbers, or goal descriptions, as this is a minor-user app with heightened data sensitivity under Caribbean data protection frameworks. Use a log wrapper that allowlists safe fields before emitting.

### Centralised route-level auth guard middleware: 🔥 High Impact
Currently each domain router (students, orgs, lessons, ai) must individually enforce authentication. A single `app.use('/api/(students|ai|lessons|orgs)', requireAuth)` middleware applied in routes.ts before domain registration would prevent any auth bypass from an accidentally unguarded handler, and is easier to audit for a school-aged user base handling minors' data.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replace bare `console.error` in the Supabase init chain with structured log entries including step name, duration, and error code. This makes Railway log drain and Sentry alerts actionable without SSH access, reducing MTTR for seed failures at school year start when content must be available.

### Cache-Control header on /public-objects assets: ⚡ Medium Impact
The 30-day CDN TTL is passed to downloadObject but there is no `Cache-Control` header set at the Express layer. Browsers and Railway's edge will not cache these logo/video assets, causing repeated origin hits. Add `res.setHeader('Cache-Control', 'public, max-age=2592000, immutable')` before calling downloadObject so browser and any intermediate proxy also caches, reducing latency for students on low-bandwidth Caribbean connections.

### Move file bytes out of job table via pre-upload pattern: 🔥 High Impact
Adopting an upload-then-enqueue pattern (store file in object storage first, enqueue only the path) reduces average job row size from megabytes to ~200 bytes, dramatically improves job-queue scan performance, and makes the worker stateless enough to scale horizontally without memory pressure from large deserialized payloads.

### Transactional bulk insert for extracted questions: 🔥 High Impact
Replacing the sequential per-question insert loop with a single transactional bulk insert eliminates N-1 unnecessary round-trips, ensures atomicity so readers never see a partial question set, and removes the window where a crash between delete and re-insert leaves the paper in a broken state across retry attempts.

### Centralise job failure status sync into job runner middleware: ⚡ Medium Impact
The pattern of catching errors and calling storage.update*({status:'failed'}) is duplicated across handlers. Extracting this into a wrapper in the job runner (e.g. withJobErrorSync(entityId, updateFn, handler)) removes boilerplate, ensures consistent failure reporting across all handlers including admin-csv-export, and makes it impossible to accidentally omit the status update when adding new handlers.

### Composite unique constraint on portfolioHoldings: 🔥 High Impact
Add a unique constraint on (userId, stockId) in portfolioHoldings so the application can use an upsert (INSERT ... ON CONFLICT DO UPDATE) to increment quantity and recalculate avgPurchasePrice atomically in the DB rather than doing a read-modify-write in application code. This eliminates a race condition on concurrent buy orders for the same user and stock.

### Extract hardcoded currency default 'BSD' to a shared constant: ⚡ Medium Impact
The string 'BSD' appears as a default in at least 7 separate column definitions. Define a single DEFAULT_CURRENCY = 'BSD' constant in a shared config file and reference it, so a future multi-currency expansion or correction only requires one change and cannot produce mismatched defaults across tables.

### Add partial index on billReminders for upcoming due dates: ⚡ Medium Impact
The most common query against billReminders will be fetching reminders due within the next N days per user. A partial index such as CREATE INDEX idx_bill_reminders_upcoming ON bill_reminders (user_id, next_due_date) WHERE next_due_date > NOW() would be significantly smaller and faster than a full index, and directly maps to the notification/reminder scheduler query pattern.

### Centralised graceful-shutdown registry: 🔥 High Impact
Introduce a ShutdownManager class with a register(fn: () => Promise<void>) method. All modules (intervals, DB pool, job worker, HTTP server) self-register their teardown. A single SIGTERM handler calls them in reverse-registration order with a 10 s hard-kill timeout via Promise.race. This eliminates the current pattern where teardown logic would have to be duplicated across SIGTERM and SIGINT handlers.

### Structured logging with PII field filtering: 🔥 High Impact
Replace the ad-hoc console.log/log() calls with a structured logger (pino is zero-dependency and fast). Define a static redact list (['email','password','token','answer','sessionId']) applied at the logger level. This gives consistent JSON logs consumable by Sentry/Datadog and prevents accidental PII exposure without per-callsite discipline.

### Scheduler extracted to a dedicated module with unit-testable clock injection: ⚡ Medium Impact
Move maybeRunWeeklyDigest, maybeRunAiUsagePurge, and getMostRecentSundayAt7pm into a scheduler.ts module that accepts a clock function (defaulting to () => new Date()). This makes the Sunday-at-7pm logic and the idempotency guard unit-testable without real DB or timers, catching the timezone bug and the (dow+0) dead code in CI rather than production.

### Split AdminDashboard into lazy-loaded tab pages: 🔥 High Impact
At 147k+ characters this is a single-file mega-component. Code-splitting each admin tab (Users, Schools, Lessons, Analytics, Audit Log) into separate React.lazy() pages would cut initial JS parse time by an estimated 60-70%, which matters on lower-end devices common in Caribbean school labs. It also makes the codebase testable in isolation.

### Memoize filtered and sorted list derivations with useMemo: ⚡ Medium Impact
Search filtering and sorting of users, schools, and lessons is almost certainly recomputed inline on every render. Wrapping these with useMemo keyed on the raw data array and search/sort state would eliminate redundant O(n) passes on every keystroke in unrelated inputs, directly improving perceived responsiveness of the search UX.

### Introduce a shared useAdminQuery wrapper with standardised error handling: ⚡ Medium Impact
Every useQuery call in this file independently handles (or silently ignores) errors. A thin wrapper hook that enforces a consistent onError toast, retryDelay, and staleTime policy across all admin queries would reduce boilerplate, guarantee the finance-app SLA of always showing actionable error messages, and make adding request-level logging or auth expiry redirection trivial from one place.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step write operations, and sequential awaits where Promise.all would suffice. For a school-aged financial app handling real user data, the absence of pagination guards on unbounded queries and missing try/catch on critical async paths are the most pressing reliability concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel dynamic imports and non-blocking Supabase init. Key concerns are: AI/chat/image/audio routes registered without any rate limiting, host header injection risk in robots.txt and sitemap, and the Supabase init chain silently swallowing individual step failures.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads are stored directly in the job table (a classic bloat vector), sequential DB inserts in a hot loop where bulk insert would be faster and safer, and the base64 image string is double-encoded before being sent to the Vision API. Security posture is generally reasonable but a few hardcoded model names and missing auth delegation in job context warrant attention.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with FK constraints and some indexes already in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable createdAt timestamps that should be NOT NULL, and varchar user_id columns without a length constraint that could cause silent truncation issues.

### Server Startup (`server/index.ts`)
The server startup sequence performs multiple blocking async operations (DB probe, seed, job worker init) synchronously before the HTTP server is ready, and intervals started during startup are never cleared on shutdown, creating memory leaks and zombie timers. Response JSON is logged in full on every API response, which will leak sensitive financial and student PII into logs in a school-facing app.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx is a large, densely packed admin page with several reliability and security gaps. The most critical issues are missing auth guards on sensitive mutations, memory leaks from unclean scroll refs, and sequential awaits on independent async operations. Performance is also hurt by large inline functions recreated on every render.
