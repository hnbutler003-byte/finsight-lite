## 🔴 Critical (5)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (called together in trade routes)
**Issue:** Multi-step trade operations (upsert holding, create portfolio transaction, update virtual balance) are not wrapped in a database transaction. A failure between steps leaves the database in an inconsistent state — e.g., balance debited but holding not recorded, or holding updated but no audit trail written.

**Fix:** Wrap the three operations in a single `db.transaction(async (tx) => { ... })` call inside a new `executeTrade(userId, ...)` storage method. Pass the transaction client `tx` to each sub-operation so all succeed or all roll back atomically.

---

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: the method fetches all holdings for a user then issues a separate `getStockById` DB call per holding to attach the stock object. For a portfolio with 20 stocks this is 21 round-trips.

**Fix:** Replace the per-holding lookup with a single JOIN or `inArray` query: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. Return the joined rows directly as `PortfolioHolding & { stock: SimulatedStock }`.

---

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 (full base64-encoded PDF/image) is stored directly in the job payload, bloating the jobs table with potentially megabytes per row. This degrades DB performance for all job queue queries and risks hitting row-size limits.

**Fix:** Upload the file to object storage before enqueuing and store only the object path in the payload. In runExtractPaper, download the buffer from object storage at the start of the job, then delete the temp object on completion or terminal failure.

---

### `shared/schema.ts` — budgets table definition
**Issue:** budgets table has no index on userId, yet every budget query will filter by userId. A full table scan runs on every dashboard load for every user.

**Fix:** Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also enforces the business rule of one budget per user per category.

---

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel — useQuery queryFn
**Issue:** The queryFn calls apiRequest and chains .then(r => r.json()) with no catch or error boundary. If the server returns a non-OK response (e.g. 401 session expiry, 500), r.json() will either throw or return an error payload that is silently spread into `rows`, potentially rendering malformed data in the audit table shown to admins.

**Fix:** Add explicit status checking in the queryFn: `async () => { const r = await apiRequest('GET', url); if (!r.ok) throw new Error(await r.text()); return r.json(); }`. Also destructure `error` from useQuery and render an error state in the panel when it is set.

## 🟡 Warnings (12)

- **`server/storage.ts` (getDashboardStats):** Dashboard stat aggregation issues sequential `await` calls for total income, total expenses, transaction list, budget list, and savings goals. Each is an independent DB query, so they block unnecessarily and multiply latency for every page load by every student.
  → *Replace sequential awaits with `const [income, expenses, txns, budgets, goals] = await Promise.all([...])` grouping all independent queries. This reduces wall-clock latency to the slowest single query instead of the sum of all queries.*
- **`server/storage.ts` (getLeaderboard):** The leaderboard query accepts an optional `limit` parameter but applies no maximum cap or default guard. A missing or manipulated limit (e.g., limit=0 or an extremely large value passed from the client) can either return the entire game_sessions table or produce unexpected results, causing memory pressure on the Node process.
  → *Apply a bounded default: `const safeLimit = Math.min(filters?.limit ?? 20, 100)`. Always pass `safeLimit` to `.limit()` in the Drizzle query. Reject or clamp values above 100 at the route validation layer (Zod schema) before they reach storage.*
- **`server/storage.ts` (getLeaderboard (userName, avatar fields assembled from joined user row)):** The leaderboard response includes `userName` and `avatar` fields sourced from the users table and logged or returned directly. If the user row contains a real email address used as a display name, it is exposed to all leaderboard viewers — a COPPA/privacy concern for school-aged users.
  → *Select only `users.displayName` (or a dedicated `screenName` column) and `users.avatarUrl` in the leaderboard query. Never select or forward `users.email`, `users.id` (if it is an OAuth subject), or any PII. Add a server-side projection list explicitly naming safe columns rather than spreading the whole user object.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time in a sequential for-loop with individual awaits. For a paper with 40 questions this is 40 serial round-trips to the DB, each blocking the next.
  → *Replace the loop with a single bulk insert: filter valid questions into an array first, then call db.insert(extractedQuestions).values(validRows) once. Also wrap the delete + bulk insert in a single transaction so a crash between them cannot leave the paper with zero questions.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** openai client is imported from a routes module, which is an HTTP-layer file. This creates a hidden coupling: any refactor of chat/routes (middleware, exports, side-effects on import) silently breaks all AI job handlers. It also likely exposes other route-level exports to this module.
  → *Move the openai client instantiation to a dedicated shared module (e.g. server/lib/openai.ts) and import from there in both the routes file and jobHandlers.ts.*
- **`server/jobHandlers.ts` (runExtractPaper — delete then re-insert idempotency block):** The idempotency guard deletes existing extractedQuestions rows and then reinserts them in separate operations outside a transaction. If the process crashes after the delete but before inserts complete, the paper is left with zero questions and status still 'processing', with no recovery path on the next retry because the delete already ran cleanly.
  → *Wrap await db.delete(extractedQuestions)... and the subsequent bulk insert in a db.transaction() call so both operations are atomic. On retry the transaction either fully replays or rolls back, leaving the table consistent.*
- **`shared/schema.ts` (varchar('user_id') on all tables):** All userId columns are declared as varchar with no length argument. In PostgreSQL, unbounded varchar is equivalent to text, but this is misleading and the absence of a length means no DB-level guard against oversized values being stored. If the auth provider (e.g. Clerk/Supabase) uses fixed-length IDs, the constraint should be explicit.
  → *Change all userId columns to varchar('user_id', { length: 256 }) — or use text('user_id') consistently if no length bound is intended. Pick one and apply it uniformly across categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** portfolioHoldings has no index on userId or stockId, and portfolioTransactions has no index on userId or executedAt. Portfolio summary queries (group by stock, filter by user, order by date) will full-scan both tables as user data grows.
  → *Add to portfolioHoldings: (t) => ({ userIdx: index('idx_portfolio_holdings_user').on(t.userId), userStockUniq: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). Add to portfolioTransactions: (t) => ({ userDateIdx: index('idx_portfolio_tx_user_date').on(t.userId, t.executedAt) }).*
- **`shared/schema.ts` (createdAt columns across all tables):** createdAt is defined as timestamp('created_at').defaultNow() with no .notNull(). defaultNow() does not imply NOT NULL in Drizzle-ORM; an explicit insert of null will succeed and silently store NULL, breaking any ORDER BY createdAt or audit queries.
  → *Append .notNull() to every createdAt column, e.g. timestamp('created_at').defaultNow().notNull(). Apply to: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on nextDueDate or userId. The primary use-case query is 'fetch upcoming bills for a user ordered by nextDueDate', which requires both columns. Without an index this is a sequential scan ordered in memory.
  → *Add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) as the third argument to pgTable for billReminders.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery in ObservabilityCard has no error or loading state. If the /api/admin/observability endpoint fails, the component renders silently with undefined data, producing a broken UI with no user feedback.
  → *Destructure `isLoading` and `error` from useQuery. Render a skeleton or spinner when isLoading is true, and an error message (e.g. <Alert variant='destructive'>) when error is truthy. Guard all `data?.x` accesses with a loaded check.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params construction inside render body):** A new URLSearchParams object is constructed on every render of AuditLogPanel outside of any memoization. This is a derived value from state that recalculates unconditionally, and the resulting string is passed into the queryFn closure, creating a stale-closure risk and unnecessary object allocations on every render cycle.
  → *Wrap the params construction in useMemo: `const paramString = useMemo(() => { const p = new URLSearchParams(); ... return p.toString(); }, [actorType, action, from, to]);` and use `paramString` in both the queryKey array and the queryFn URL.*

## 🔵 Info / Tech Debt (3)

- **`server/jobHandlers.ts` (extractedText = pdfData.text.slice(0, 30000)):** The magic number 30000 (character truncation limit for PDF text) is hardcoded with no explanation or configurability. If the OpenAI context window or pricing tier changes, this requires a code change rather than a config update.
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set("limit", "200")):** The audit log query limit of 200 is hardcoded as a magic string literal. In a financial literacy app with audit requirements, this cap will silently truncate audit history, and changing it requires a code deploy rather than a config change.
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — refetch Button onClick):** The onClick handler `() => refetch()` is an inline arrow function created on every render. In a component that re-renders on each filter state change (actorType, action, from, to), this produces a new function reference on every keystroke, preventing React from bailing out of child re-renders for the Button component.

## 💡 Opportunities

### Centralise hardcoded virtual balance seed value — ⚡ Medium Impact
The initial virtual balance (e.g., 10000.00) granted to new users on first `getVirtualBalance` call appears hardcoded in the method body. Move it to an environment variable `VIRTUAL_BALANCE_SEED_USD` with a validated default. This lets the product team adjust the starting amount for classroom challenges without a code deploy.

### Add missing pagination to getAllQuestions — 🔥 High Impact
`getAllQuestions` returns every extracted question row matching a subject filter with no LIMIT/OFFSET or cursor. As exam papers accumulate across all schools this will grow unbounded and serialize large payloads to the client. Introduce a `limit` + `offset` parameter (max 100 per page) and return a `{ data, total }` envelope so the UI can paginate the question bank.

### Extract shared enrollment+student join into a reusable query helper — ⚡ Medium Impact
`getEnrollmentsByClass` and `getStudentClasses` both manually join enrollments to users/classes with nearly identical Drizzle query structures. Extracting a typed `enrollmentWithRelations(filter)` query builder eliminates duplication and ensures both code paths stay in sync when the schema changes, reducing future N+1 regression risk.

### Parallel AI calls for two-step extraction — ⚡ Medium Impact
The PDF path runs text extraction then a second OpenAI call sequentially. While the second call depends on the first, the vision path could pre-validate the image buffer (size, MIME) concurrently with any setup work. More impactfully, if multiple papers are enqueued, worker concurrency config should be reviewed so the job runner processes them in parallel rather than serially — check the registerJobHandler concurrency option.

### Stream object storage exports instead of buffering in memory — 🔥 High Impact
The admin-csv-export handler (truncated but implied by uploadPrivateObject usage) likely builds the full CSV in memory before uploading. For large datasets this can exhaust Node.js heap. Pipe a CSV transform stream directly into objectStorageClient write stream to cap memory use at a single chunk size regardless of row count.

### Centralise job payload schemas with Zod — ⚡ Medium Impact
Job payloads are accessed via destructuring (e.g. const { paperId, fileB64, ext, subject } = job.payload) with no runtime validation. A malformed enqueue call produces a confusing runtime crash deep inside the handler. Add a Zod schema per job type in a shared jobSchemas.ts and parse at the top of each handler to get early, descriptive errors and inferred types.

### Composite unique constraint on portfolioHoldings — 🔥 High Impact
Adding uniqueIndex on (userId, stockId) in portfolioHoldings enforces the business invariant that a user has exactly one holding record per stock, preventing duplicate rows from race conditions on concurrent buy operations. This removes the need for application-level deduplication logic.

### Extract repeated currency default into a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least five separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions). Define const DEFAULT_CURRENCY = 'BSD' once at the top of the file and reference it via .default(DEFAULT_CURRENCY), so a currency change requires one edit instead of five with risk of inconsistency.

### Add partial index for active document upload processing — ⚡ Medium Impact
Polling queries for in-progress document uploads filter on status = 'processing'. A partial index — index('idx_doc_uploads_processing').on(t.userId).where(sql`status = 'processing'`) — will be tiny and fast compared to a full index, materially improving upload status polling latency under load without storage overhead.

### Consolidate admin bootstrap queries with Promise.all — 🔥 High Impact
If the main AdminDashboard component fires multiple sequential useQuery calls on mount (overview stats, organizations, schools, etc.), sequential fetches add cumulative latency. Group independent initial data fetches using Promise.all inside a single prefetch or use TanStack Query's parallel query pattern to cut initial load time by the number of sequential round-trips.

### Virtualize the AuditLogPanel table — ⚡ Medium Impact
With a 200-row hard cap and potential for growth, rendering all rows as DOM nodes causes layout thrashing on filter changes. Replacing the table body with a windowed list (e.g. @tanstack/react-virtual) would keep DOM node count constant and eliminate jank on slower Chromebooks common in Caribbean school settings.

### Extract tab panel components to prevent full-dashboard re-renders — ⚡ Medium Impact
With 13 tabs defined in TABS, if all panel components are rendered in the same component tree as the active-tab state, every tab switch re-renders the entire dashboard. Wrapping each panel in React.memo and lazy-loading inactive panels with React.lazy/Suspense would isolate renders and reduce JS parse cost on initial load.
