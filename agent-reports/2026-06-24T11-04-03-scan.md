## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied before registering AI/chat/image/audio routes. These call OpenAI APIs and are the most expensive and abuse-prone endpoints in the app. Any unauthenticated or authenticated user can hammer them with no throttle.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Confirm the route prefixes match what each register function mounts.

### `server/routes.ts`: robots.txt handler (~line 40) and sitemap.xml handler (~line 57)
**Issue:** Both handlers construct the base URL directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. A spoofed Host header produces a poisoned Sitemap directive and robots.txt, which could be cached by crawlers or used for cache-poisoning attacks.

**Fix:** Read the canonical base URL from a single env var (e.g. BASE_URL=https://finsightlite.com) set at deploy time. Fall back to a hardcoded production domain, never to raw host headers. Remove the x-forwarded-host/host derivation entirely from both handlers.

### `server/jobHandlers.ts`: runExtractPaper / enqueueJob call-site for extract-paper
**Issue:** fileB64 (full base64-encoded exam paper) is stored in the job payload row in the database. A single image or PDF can easily be 5-20 MB encoded, multiplied across retries and concurrent uploads this will bloat the jobs table, slow queue polling queries, and risk hitting row-size limits.

**Fix:** Upload the file to object storage before enqueuing the job, store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), and have runExtractPaper download the buffer from object storage at execution time using objectStorageClient. This keeps job rows small and makes retries cheaper.

### `server/jobHandlers.ts`: streamPrivateObjectToResponse – objectPath parameter
**Issue:** The objectPath passed to this function is constructed from user-influenced data (e.g. a job result stored in the DB). There is no validation that the resolved path stays within the expected bucket prefix, making it possible for a crafted path containing '..' segments to escape the intended bucket and read arbitrary objects from object storage.

**Fix:** After splitting the path into bucket and object, assert that bucket === process.env.EXPECTED_EXPORT_BUCKET (or a whitelist) and that the object string does not contain '..' segments. Throw a 403-equivalent error if either check fails before calling file.exists().

### `shared/schema.ts`: budgets table definition (~line 36)
**Issue:** budgets has no index on userId, and no unique constraint on (userId, categoryId, period). This means a user can accumulate duplicate active budgets for the same category and period through concurrent inserts or retries, corrupting budget aggregation queries without any database-level guard.

**Fix:** Add to the budgets pgTable third argument: `(t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId), uniqueBudget: uniqueIndex('uq_budgets_user_category_period').on(t.userId, t.categoryId, t.period) })`. Handle the resulting unique violation in application code with an upsert.

### `shared/schema.ts`: portfolioHoldings table definition (~line 78)
**Issue:** portfolioHoldings has no index on userId or stockId, and no unique constraint on (userId, stockId). Every portfolio value calculation will full-scan this table per user. Without a unique constraint, duplicate holding rows for the same stock accumulate silently, causing double-counted positions in a financial simulation used by minors — a correctness failure with educational trust implications.

**Fix:** Add third argument: `(t) => ({ userStockUniq: uniqueIndex('uq_portfolio_holdings_user_stock').on(t.userId, t.stockId), userIdx: index('idx_portfolio_holdings_user_id').on(t.userId) })`. In buy logic, use an INSERT ... ON CONFLICT (userId, stockId) DO UPDATE to increment quantity and recalculate avgPurchasePrice.

### `shared/schema.ts`: simulatedStocks table, ticker column (~line 90)
**Issue:** simulatedStocks.ticker has no unique index. Ticker symbols are the natural business key for stocks; duplicate tickers cause incorrect price lookups and wrong portfolio valuations when joining portfolioHoldings to simulatedStocks by ticker in application queries. This is a data integrity failure in a financial simulation.

**Fix:** Add third argument to simulatedStocks: `(t) => ({ tickerUniq: uniqueIndex('uq_simulated_stocks_ticker').on(t.ticker) })`. If multi-currency or multi-region tickers are needed later, make the unique constraint on (ticker, region) instead.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat - sendMessage
**Issue:** The full `messages` array is sent to /api/admin/help-chat on every message with no length or token cap. A user (or script) can accumulate thousands of messages and send arbitrarily large payloads, causing server-side LLM cost spikes and potential DoS. There is also no server-side auth check visible on this endpoint path for the admin role.

**Fix:** Slice the history before sending: `messages: next.slice(-20)` to cap context at the last 20 turns. Add a `maxMessages` guard: if `messages.length > 100` show a 'Start new conversation' prompt and reset state. Ensure /api/admin/help-chat validates the admin session server-side before forwarding to the LLM.

## 🟡 Warnings (11)

- **`server/routes.ts` (initSupabaseTables().then(...).catch(...) (~line 110)):** The seven-step sequential Supabase init chain catches errors only at the outermost level. If seedFinancialAcademyLesson() throws, the remaining seeds (initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent) are silently skipped with no per-step visibility into which step failed.
  → *Wrap each seed in its own try/catch with a step-specific log, or restructure as Promise.allSettled([initSupabaseTables(), ...]) after separating steps that have true dependencies from those that are independent. At minimum log the step name: .then(() => seedFinancialAcademyLesson().catch(e => console.error('[Supabase] seedFinancialAcademyLesson failed:', e))).*
- **`server/routes.ts` (/api/health (~line 28) and /healthz (~line 75)):** There are two separate health check endpoints (/api/health and /healthz) with different behaviors: /api/health returns ok without checking the DB, /healthz hits the DB. Railway and uptime monitors will get inconsistent signals depending on which endpoint is configured. The simpler one is dead weight.
  → *Remove /api/health and standardise on /healthz which performs a real DB liveness check. Update Railway and any uptime monitor configs to use /healthz. If a lightweight ping without DB is needed for a load balancer, document the distinction explicitly.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** filePath is taken directly from req.params.filePath and passed to objectStorage.searchPublicObject without sanitisation. A path like ../../private/config could traverse to unintended storage keys depending on the ObjectStorageService implementation.
  → *Sanitise filePath before use: const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); and verify it does not start with '..' or contain null bytes. Alternatively, restrict with a regex allowlist such as /^[\w\-./]+$/ and return 400 for non-matching values.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one-by-one with sequential awaits inside a for loop. For a 40-question paper this is 40 serial round-trips to the database, adding latency proportional to question count and holding the job worker busy unnecessarily.
  → *Collect all valid question objects into an array first, then insert them in a single batched call: await db.insert(extractedQuestions).values(questionsToInsert). If storage.createExtractedQuestion must be used, at minimum use Promise.all over the filtered array, though a single bulk insert is strongly preferred.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls, model and max_completion_tokens):** The OpenAI model name 'gpt-4o' and token limits (4096) are hardcoded in two separate places. Changing the model for cost or capability reasons requires a code deploy, and the values differ subtly between the vision call and extraction call with no explanation.
  → *Read these from environment variables: OPENAI_EXTRACT_MODEL (default 'gpt-4o'), OPENAI_EXTRACT_MAX_TOKENS (default 4096). Reference them as constants at the top of the handler so both calls share a single source of truth and ops can tune without a redeploy.*
- **`server/jobHandlers.ts` (admin-csv-export handler – data variable declaration):** The variable 'data' is declared with 'let data' and no type or initialiser before the branching logic. If 'type' does not match any known branch, 'data' will be undefined when passed to the CSV serialiser, causing a silent runtime error or an empty export with no indication of the bad input.
  → *Initialise with 'let data: AdminRow[] = []' and add an explicit else branch that throws an Error with the unrecognised type value, so the job fails fast and visibly rather than producing a silently empty file.*
- **`shared/schema.ts` (categories table definition (~line 12)):** categories.userId uses varchar with no length specified, and the same pattern appears on every table. PostgreSQL varchar without length is functionally equivalent to text, but the inconsistency with the referenced users.id type risks silent join mismatches if users.id is ever given a length constraint. More critically, there is no index on categories.userId, so any query filtering user-specific categories (which will be the dominant query pattern) performs a full table scan.
  → *Add an explicit index: in the pgTable call add a third argument `(t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) })`. Also standardise all FK columns to match the exact type of users.id — if users.id is text, change all varchar('user_id') to text('user_id') for type consistency.*
- **`shared/schema.ts` (transactions, budgets, documentUploads, savingsGoals, billReminders — createdAt columns (~lines 33, 42, 63, 72, 85)):** createdAt on every table uses .defaultNow() but omits .notNull(). This means application code can explicitly insert NULL for createdAt, bypassing the default and producing rows with no audit timestamp. For a financial app this breaks chronological ordering, audit trails, and any time-series reporting.
  → *Chain .notNull() on every createdAt: `createdAt: timestamp('created_at').defaultNow().notNull()`. Similarly, transactions.documentUploadId and transactions.categoryId are nullable FKs which is intentional, but transactions.createdAt being nullable is not — fix that one with highest priority as it affects the core ledger.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - scroll div ref callback):** `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a brand-new inline function on every render of the messages list. React will call the old teardown and new setup on every state change, causing repeated scrollIntoView calls mid-typing (e.g., while `loading` toggles) and unnecessary layout thrashing.
  → *Replace with a stable ref: `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading])`. Attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - sendMessage async flow):** `await res.json()` is called sequentially after `await apiRequest(...)` completes, which is correct, but `apiRequest` itself likely wraps fetch without a timeout. If the LLM backend is slow or unresponsive, `loading` stays true indefinitely with no escape hatch for the user.
  → *Wrap the fetch in an AbortController with a timeout: `const controller = new AbortController(); const timer = setTimeout(() => controller.abort(), 30000);` Pass the signal to apiRequest, and in the catch block detect `error.name === 'AbortError'` to show 'Request timed out' instead of the generic error message. Clear the timer in finally.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard data fetching (useQuery calls for stats, users, schools)):** Multiple useQuery hooks that each trigger independent API calls appear to be defined sequentially with no coordination. If any downstream sections of the dashboard (analytics charts, user tables) derive their data from these queries without memoization, every query cache update will re-run expensive filter/sort/aggregate logic inline in JSX on each render.
  → *Wrap derived computations (filtered user lists, chart data transformations, aggregate counts) in `useMemo` with the raw query data as dependencies. For independent bootstrap queries (stats, schools, parishes) that do not depend on each other, confirm they are not accidentally serialized by conditional `enabled` flags chained on each other.*

## 🔵 Info / Tech Debt (1)

- **`client/src/pages/AdminDashboard.tsx` (Large inline mutation handlers and filter functions inside JSX return):** Inline arrow functions passed as `onClick` handlers to table rows, action buttons (delete user, edit school, assign badge), and filter selects are re-created on every render. ## 💡 Opportunities

### Parallel Supabase seed execution for independent steps: ⚡ Medium Impact
seedStaticModules, seedRegionalContent, and seedGameContent have no interdependencies and currently run sequentially. Wrapping them in Promise.allSettled would cut startup time proportionally to the slowest individual seed rather than their sum, meaningfully reducing cold-start latency on Railway.

### Centralised BASE_URL environment config: 🔥 High Impact
Extracting a single BASE_URL env var consumed by robots.txt, sitemap.xml, and any absolute URL construction eliminates the host-header derivation logic in two places, makes the app trivially portable across staging/production environments, and removes the security surface identified above.

### Auth-check middleware on AI routes for school-aged user protection: 🔥 High Impact
Even with rate limiting, AI chat/image/audio endpoints should verify an authenticated session before proxying to OpenAI, especially given the school-aged audience and COPPA/data-minimisation obligations. Adding a requireAuth middleware before registerChatRoutes/registerImageRoutes/registerAudioRoutes ensures unauthenticated callers are rejected before any AI cost is incurred.

### Batch idempotency delete + status reset in one transaction: 🔥 High Impact
runExtractPaper currently issues a DELETE on extractedQuestions and later an UPDATE on exam_papers as separate statements with no transaction wrapping. Wrapping both the initial cleanup and the final status update in a db.transaction() call would eliminate the window where a crash leaves the paper in an inconsistent state (questions deleted but status still 'processing'), reducing the need for the ad-hoc status sync in the catch block.

### Centralise OpenAI client import away from route module: ⚡ Medium Impact
openai is imported from './replit_integrations/chat/routes', a route file, rather than a dedicated client module. This creates a tight coupling between the job handler and HTTP routing code, risks circular imports, and makes it harder to swap or mock the client in tests. Extract the OpenAI client initialisation into its own module (e.g. './openaiClient.ts') and import from there.

### Add per-handler timeout enforcement: 🔥 High Impact
Long-running handlers like extract-paper make two sequential OpenAI calls with no timeout set on the HTTP requests. If OpenAI is slow, a job worker thread can be held for minutes, starving other jobs. Wrap each openai call with a Promise.race against an AbortController-based timeout (e.g. 60 s) so the job fails fast and is retried rather than silently blocking the worker pool.

### Centralise currency default via app-level config constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the currency default in at least five separate tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions). Extract `DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD'` into a shared config and reference it in schema column defaults. This eliminates a painful multi-file change if the app ever expands to TTD or JMD markets.

### Replace serial PKs with UUID or ULID on user-facing tables: 🔥 High Impact
Using serial integer PKs on transactions, savingsGoals, and portfolioHoldings exposes sequential IDs in API responses, allowing users to enumerate other users' records if any auth check ever fails. Switching to gen_random_uuid() (pgcrypto) or ULID removes the enumeration vector and is especially important given the school-aged user base subject to stricter data protection norms in Caribbean jurisdictions.

### Add partial index on documentUploads for pending processing queue: ⚡ Medium Impact
Document processing jobs almost certainly query WHERE status = 'processing' repeatedly. A partial index `index('idx_document_uploads_processing').on(t.userId).where(sql`status = 'processing'`)` would make the processing queue lookup O(active jobs) instead of O(all uploads), which degrades significantly as historical completed uploads accumulate.

### Parallel bootstrap data fetching: 🔥 High Impact
If the dashboard fetches stats, user list, and school list in sequence (either via chained enabled flags or server-side sequential awaits in the API route), switching to Promise.all / independent parallel useQuery calls with no interdependency would cut initial dashboard load time by roughly the sum of the slowest N-1 requests, which matters on high-latency Caribbean mobile connections.

### Virtualized user/school tables: 🔥 High Impact
Rendering full unpaginated or large-page user and school tables as DOM nodes is the single largest render bottleneck on low-RAM Android devices common in school settings. Replacing the table body with react-virtual or TanStack Virtual would keep DOM node count constant regardless of dataset size, eliminating jank when admins load large districts.

### Chat message persistence across panel open/close: ⚡ Medium Impact
Currently closing the AdminHelpChat panel (setOpen false) unmounts the component and loses all message history. Lifting `messages` state into a parent context or Zustand slice would preserve conversation across open/close toggles, significantly improving admin UX during multi-step workflows without any network cost.
