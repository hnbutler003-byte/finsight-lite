## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI/image/audio routes are registered with no rate limiting middleware applied before them. For a school-aged user base these are the most expensive endpoints (OpenAI API cost, latency) and are trivially abusable without per-IP or per-user throttling.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` placed immediately before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 42 and ~line 58)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. A forged Host header will produce a poisoned sitemap/robots.txt response, which can be cached by proxies and mislead crawlers or be used for cache-poisoning attacks.

**Fix:** Resolve the base URL once at startup from an environment variable: `const BASE_URL = process.env.PUBLIC_URL || 'https://finsightlite.com';` and use that constant in both handlers instead of reading from request headers.

---

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no indexes. Every budget lookup by user (the primary query pattern) requires a full table scan. As users accumulate budgets this degrades linearly.

**Fix:** Add a composite index in the table's index callback: index('idx_budgets_user_category').on(t.userId, t.categoryId). Also add index('idx_budgets_user').on(t.userId) if single-user lookups are common without a category filter.

---

### `shared/schema.ts`: simulatedStocks table — lastPriceUpdateDate column
**Issue:** lastPriceUpdateDate is typed as text rather than timestamp. Storing dates as text prevents index range scans, breaks ordering, and allows malformed values to persist silently in a financial simulation context.

**Fix:** Change the column type to timestamp('last_price_update_date'). If timezone awareness is needed use timestamp('last_price_update_date', { withTimezone: true }). Update any seed/migration scripts accordingly.

---

### `server/index.ts`: maybeRunWeeklyDigest / startup block
**Issue:** Scheduler intervals (setInterval for weekly digest, AI usage purge, etc.) are started but their handles are never stored or cleared, so process.on('SIGTERM'/'SIGINT') graceful shutdown never cancels them. On Replit hot-reload this creates duplicate intervals each restart, firing the same jobs multiple times per tick.

**Fix:** Store every interval handle: `const digestInterval = setInterval(maybeRunWeeklyDigest, ...)`. In your graceful shutdown handler call `clearInterval(digestInterval)` (and likewise for every other interval) before `httpServer.close()`. Add `process.on('SIGTERM', shutdown)` and `process.on('SIGINT', shutdown)` with a single async shutdown() function that awaits httpServer.close() in a Promise wrapper.

---

### `server/index.ts`: res.json monkey-patch / log middleware
**Issue:** Every API response body is captured in `capturedJsonResponse` and unconditionally serialised into the log line via `JSON.stringify(capturedJsonResponse)`. For a financial literacy app with school-aged users this will dump tokens, session data, quiz scores, and any PII returned by the API into stdout/log sinks with no truncation or redaction.

**Fix:** Either remove response-body logging entirely for production (`if (process.env.NODE_ENV !== 'production')`), or whitelist only safe scalar fields (statusCode, a non-sensitive `type` field) and hard-cap the serialised string: `logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 120)`.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level dashboard mutations
**Issue:** Destructuring from useAdminAuth is used for UI gating but mutation functions (delete user, update school, etc.) do not re-verify admin status server-side within the client call path. If the hook's session expires mid-session, mutations still fire with a stale token and the only guard is the server. More critically, there is no client-side abort: the UI buttons stay enabled after logout if the component is still mounted.

**Fix:** Add a guard at the top of each mutation's `onMutate` or `mutationFn`: check `isAdmin` from useAdminAuth and early-return with a toast if false. Also call `queryClient.clear()` and redirect on auth errors (401 responses) caught in a global apiRequest interceptor.

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 115)):** The seven Supabase init steps run sequentially in a .then chain. A failure in any middle step (e.g. seedFinancialAcademyLesson) short-circuits all subsequent steps silently — the single .catch only logs a single error with no indication of which step failed or which steps were skipped. This can leave the DB in a partially seeded state on every restart.
  → *Run independent seed operations in parallel with named error handling: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] seed step ${i} failed:`, r.reason)));` Keep initSupabaseTables and trimOrgNamesInSupabase sequential only if they are true prerequisites.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 102)):** The filePath parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without sanitisation. A path traversal payload like `../../private/keys` could expose non-public objects depending on how the storage service resolves paths.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Add `import path from 'path';` at the top of the file.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every request. Dynamic imports are cached after first load, but the pattern is fragile and misleading — if the module cache is invalidated the import cost reappears. More importantly, this endpoint is unauthenticated and reveals DB availability state to the public internet.
  → *Move the db and sql imports to the top of the file as static imports. Restrict /healthz to internal callers only by checking a shared secret header: `if (req.headers['x-healthz-secret'] !== process.env.HEALTHZ_SECRET) return res.status(403).end();` or bind the route only to localhost via a separate internal Express instance.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Neither portfolioHoldings nor portfolioTransactions has an index on userId or (userId, stockId). Portfolio value calculations and transaction history queries will scan the entire table per user.
  → *Add to portfolioHoldings: index('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId). Add to portfolioTransactions: index('idx_portfolio_tx_user').on(t.userId) and index('idx_portfolio_tx_user_stock').on(t.userId, t.stockId).*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, linkedCards, etc.)):** userId is declared as varchar with no length bound. PostgreSQL stores unbounded varchar identically to text, but the missing length constraint means IDs of any length are accepted. If the auth provider (e.g. Clerk/Supabase) issues fixed-length IDs, oversized values from malformed requests will silently persist.
  → *Specify a length matching your auth provider's ID format, e.g. varchar('user_id', { length: 36 }) for UUID-style IDs or varchar('user_id', { length: 32 }) for Clerk. Apply consistently to every userId column in the schema.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt, documentUploads.createdAt (all audit timestamp columns)):** All createdAt columns use .defaultNow() but are not marked .notNull(). If the default is somehow bypassed (bulk insert, raw SQL migration, ORM version edge case), NULL audit timestamps are stored and timestamp-based reporting silently breaks.
  → *Chain .notNull() after .defaultNow() on every createdAt column: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same to purchasedAt and executedAt in portfolio tables.*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes calls)):** startup awaits appear to be sequential (probeDatabase, then seedDatabase, then registerRoutes, then startJobWorker, then registerJobHandlers, then scheduler setup). probeDatabase and seedDatabase have no dependency on each other and together block the event loop during startup, increasing cold-start latency directly impacting first-request readiness.
  → *Run independent startup steps in parallel with Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])`. Only then proceed to registerRoutes() and job worker setup which may depend on DB being ready. Add explicit try/catch around the Promise.all and call process.exit(1) on failure to avoid silently serving requests against an uninitialised DB.*
- **`server/index.ts` (maybeRunWeeklyDigest (inner enqueueJob catch block)):** The rollback of the `weeklyDigestRuns` claim on enqueue failure is itself not wrapped in try/catch. If the DELETE also fails (transient DB error), the exception propagates out of the outer try/catch, is swallowed by the logger, and the stale claim row is never removed, permanently preventing that audience's digest from running for that week.
  → *Wrap the rollback DELETE in its own try/catch: `try { await db.delete(...) } catch (rollbackErr) { captureError(rollbackErr); log('claim rollback also failed: ' + rollbackErr.message); }` so the stale-claim scenario is at least surfaced to Sentry and doesn't silently break future runs.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — the `+ 0` is dead/no-op code and the comment that was presumably meant to explain a UTC offset adjustment is absent. More critically, the function uses the server's local timezone via `setHours(19,0,0,0)` rather than a configured Caribbean timezone (e.g. America/Port_of_Spain, UTC-4). In a cloud environment the host timezone is almost always UTC, so 7 PM local fires at 7 PM UTC, not 7 PM AST.
  → *Remove the `+ 0`. Use a timezone-aware library (date-fns-tz or Temporal) to compute the target time in the correct Caribbean timezone: `toZonedTime(now, 'America/Port_of_Spain')`, derive the most recent Sunday 19:00 in that zone, then convert back to UTC for comparison. Expose the timezone as `DIGEST_TIMEZONE` env var defaulting to `'America/Port_of_Spain'`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > sendMessage):** The full message history array `next` is sent to /api/admin/help-chat on every message with no size limit. A long session will send an ever-growing payload, inflating bandwidth, LLM token costs, and eventually hitting request-size limits or causing OOM on the server.
  → *Slice the history before sending: `const payload = next.slice(-20)` (keep last 20 messages) and pass `payload` instead of `next` to apiRequest. Add a constant MAX_CHAT_HISTORY = 20 at module level so the limit is easy to adjust.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > JSX messages.map > scroll div ref):** The scroll-into-view ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new inline function on every render. Because React treats a changed ref callback as unmount+remount, this fires scrollIntoView on every re-render (e.g., while the user types), not just when a new message arrives, causing jarring scroll jumps.
  → *Replace with a `useRef<HTMLDivElement>(null)` assigned to the div, then add a `useEffect(() => { scrollRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages])` so scrolling only triggers when the messages array actually changes.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple top-level useQuery calls (users, schools, lessons, analytics)):** Each admin sub-section issues its own useQuery that fetches independently on mount. Several of these (e.g., analytics + user stats) are fired simultaneously but use sequential awaits inside server route handlers, and the client re-fetches all of them whenever the query key changes even when navigating between tabs that have already loaded data. There is no `staleTime` set, so every tab switch triggers a refetch.
  → *Add `staleTime: 5 * 60 * 1000` (5 minutes) to all admin useQuery calls that return reference data (schools, lessons). For analytics data that must be fresh, set `staleTime: 60_000`. This eliminates redundant network calls on tab switches and is a one-line change per query.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage (AdminHelpChat) and all useMutation mutationFn callbacks):** apiRequest calls inside sendMessage and throughout mutation functions have no explicit timeout. A hung request will leave `loading` or the mutation in a pending state indefinitely, freezing the send button or action button for the user with no recovery path.
  → *Wrap apiRequest with an AbortController timeout: create `const controller = new AbortController(); setTimeout(() => controller.abort(), 15000);` and pass `signal: controller.signal` to fetch. Catch `AbortError` specifically and show a 'Request timed out' toast. Clean up the timeout in a finally block.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import block run sequentially. If any of these perform async setup (DB calls, table checks), they add cumulative cold-start latency. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` provided they do not depend on each other's side effects, which the current structure suggests they don't.

### Centralise BASE_URL as startup-time config: ⚡ Medium Impact
Beyond fixing the header-injection issue, resolving BASE_URL once at module load from `process.env.PUBLIC_URL` and exporting it from a config module would eliminate the repeated proto/host header parsing across robots.txt, sitemap, and any future SEO endpoints, reducing duplication and making environment promotion (staging → prod) a single env-var change.

### Add structured startup logging for route registration: 📌 Low Impact
Currently there is no observability into which domain routers registered successfully or how long each took. Adding a simple perf.now() wrapper around each register call and logging `[routes] registerAiRoutes completed in Xms` would make Railway deployment logs and Sentry traces immediately useful when a cold-start hangs or a domain router throws.

### Partial index on active bill reminders: ⚡ Medium Impact
billReminders is queried by upcoming nextDueDate for all users. A partial index filtered to non-past dates — index('idx_bill_reminders_due').on(t.userId, t.nextDueDate) — combined with a WHERE nextDueDate >= NOW() clause would dramatically reduce index size and scan cost as historical reminders accumulate.

### Centralise default currency as an environment constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, linkedCards). Extract to a single DEFAULT_CURRENCY env var read once at startup. A future currency expansion or sandbox/production environment split then requires one config change instead of a schema migration.

### Add a unique constraint on simulatedStocks.ticker: 🔥 High Impact
ticker is the natural business key for a simulated stock but has no uniqueness guarantee. Duplicate tickers would corrupt portfolio value calculations and mislead school-aged users about their simulated returns. Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) to enforce this at the database level.

### Centralised graceful shutdown orchestrator: 🔥 High Impact
Introduce a single ShutdownManager that registers all cleanup callbacks (clearInterval, httpServer.close, job worker drain, DB pool end) and is invoked by both SIGTERM and SIGINT. This prevents the current scattered, order-dependent teardown and ensures Kubernetes/Replit deployment rolling restarts don't leave orphaned intervals or in-flight jobs half-executed.

### Structured JSON logging with log-level filtering: 🔥 High Impact
Replace ad-hoc console.log string concatenation with a structured logger (pino is zero-dependency and extremely fast). Emit JSON lines with fields: level, source, durationMs, method, path, statusCode — never the response body. This enables log-level env control (LOG_LEVEL=info in prod) and makes Sentry/Datadog ingestion trivial without the current PII risk.

### Extract scheduler config to environment variables: ⚡ Medium Impact
Interval durations for weekly digest polling and AI purge checks are currently hardcoded magic numbers in setInterval calls. Expose them as DIGEST_POLL_INTERVAL_MS and PURGE_POLL_INTERVAL_MS env vars with safe defaults. This allows tuning polling frequency in production without a code deploy and makes the scheduler easier to disable in test environments.

### Memoize filtered and sorted admin table data: ⚡ Medium Impact
User/school/lesson list filtering and sorting are computed inline during render without useMemo. With hundreds of records these run on every keystroke in unrelated inputs. Wrapping them in useMemo(fn, [rawData, searchTerm, sortKey]) would eliminate redundant computation and is a <5 line change per table.

### Replace hardcoded API base paths with environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/users', '/api/schools' are scattered as literals throughout the file. Consolidating them into a typed API_ROUTES constant (or a shared api-routes.ts module) prevents typos, makes base-URL changes a single-point edit, and enables easier mocking in tests.

### Extract heavy sub-sections into lazily loaded components: 🔥 High Impact
The full analytics charts, user management table, and lesson editor all render in the same bundle. Using React.lazy + Suspense for each admin tab would cut initial parse/render time significantly for the most common admin entry point (the dashboard overview), with a fallback skeleton already compatible with the existing Card components.
