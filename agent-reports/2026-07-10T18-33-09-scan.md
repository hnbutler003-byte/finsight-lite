## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then fires a separate SELECT SUM(amount) query per budget row to compute 'spent', resulting in N+1 database round-trips for a user with N budgets.

**Fix:** Replace the per-budget spent query with a single JOIN or subquery: SELECT b.*, COALESCE(SUM(t.amount), 0) as spent FROM budgets b LEFT JOIN transactions t ON t.category_id = b.category_id AND t.user_id = b.user_id AND t.type = 'expense' WHERE b.user_id = ? GROUP BY b.id. Execute once and map results.

### `server/storage.ts`: getPortfolioHoldings
**Issue:** N+1 query pattern: getPortfolioHoldings fetches all holdings then calls getStockById(holding.stockId) in a loop inside a map, issuing one SELECT per holding instead of a single IN query.

**Fix:** After fetching holdings, collect all unique stockIds, run one query: db.select().from(simulatedStocks).where(inArray(simulatedStocks.id, stockIds)), build a Map<id, stock>, then join in-memory when constructing the return array.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI chat, image, and audio routes are registered with no rate limiting middleware. In a school-aged user app this means a single authenticated session can flood OpenAI API calls, generating runaway costs or triggering OpenAI rate limit bans that affect all users.

**Fix:** Create a dedicated `aiLimiter` (e.g. express-rate-limit, 20 req/min per IP or session) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Adjust window/max based on expected classroom concurrency.

### `server/routes.ts`: robots.txt handler (~line 42) and sitemap.xml handler (~line 58)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. A request with a crafted Host header will produce a sitemap or robots.txt pointing to an attacker-controlled domain, enabling cache-poisoning or SEO injection attacks.

**Fix:** Read the canonical origin from an environment variable (`process.env.APP_BASE_URL`) set at deploy time and fall back to it unconditionally: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';`. Remove all header-derived base URL logic from these two handlers.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** Day-of-week rollback arithmetic is wrong. `const daysSince = (dow + 0) % 7` always returns 0 on Sunday (dow=0), so the function never rolls back to the previous Sunday — it returns today at 19:00, meaning the digest fires every interval tick on Sundays instead of once.

**Fix:** Replace with `const daysSince = dow === 0 ? 0 : dow;` and ensure the guard `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` is reached. The real fix: `const daysSince = (dow + 7) % 7;` is still 0 for Sunday. To get the *most recent past* Sunday use `const daysSince = dow; if (daysSince === 0 && now < 19:00 today) add 7` — or simply: subtract `dow` days, set hours to 19, then if still in the future subtract 7 more days. The existing guard handles the future case but the initial offset is always 0, making every Sunday re-trigger.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are never stored and never cleared. There is no graceful shutdown handler (SIGTERM/SIGINT) that calls clearInterval on them. In a Replit or containerised environment that restarts the process, the intervals will fire during shutdown, potentially enqueuing duplicate jobs or writing to a half-closed DB connection.

**Fix:** Store both handles: `const weeklyTimer = setInterval(...); const purgeTimer = setInterval(...);`. Then register: `process.on('SIGTERM', () => { clearInterval(weeklyTimer); clearInterval(purgeTimer); httpServer.close(() => process.exit(0)); });` and mirror for SIGINT. This ensures no new jobs are enqueued after the shutdown signal arrives.

### `server/index.ts`: res.on('finish') logging middleware
**Issue:** Every API response body is fully serialized via `JSON.stringify(capturedJsonResponse)` and written to stdout unconditionally. For a financial literacy app with school-aged users this will routinely log auth tokens, session data, quiz scores, and any PII returned by the API — violating reasonable data-minimisation expectations and COPPA/data-protection norms for minors.

**Fix:** Either remove response-body logging entirely, or allowlist only non-sensitive fields: replace the logLine append with a safe summary such as `logLine += ` :: [${Array.isArray(capturedJsonResponse) ? capturedJsonResponse.length + ' items' : typeof capturedJsonResponse}]`;`. Never log full response bodies in production; gate it behind `if (process.env.LOG_RESPONSE_BODIES === 'true')` and ensure that env var is never set in production.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / top of component
**Issue:** No early return or redirect guard after useAdminAuth. If the hook returns a non-admin or unauthenticated user, the full dashboard still renders and all admin queries still fire, exposing sensitive data to unauthorized users.

**Fix:** Immediately after calling useAdminAuth, add: `if (!admin) { setLocation('/admin/login'); return null; }` before any query hooks or JSX so no data fetches or UI renders for unauthenticated sessions.

---

### `client/src/pages/AdminDashboard.tsx`: console.log or console.error calls within mutation onError / catch blocks
**Issue:** Error handlers that log full API error objects to the console can expose stack traces, user IDs, SQL error messages, or internal route structures in production browser devtools, which is a security and compliance risk for a financial app serving minors.

**Fix:** Remove all console.log/error calls from production paths. Use a structured logger that is disabled in production (e.g., check `import.meta.env.MODE !== 'production'` before logging) or send errors to a server-side logging service instead of the browser console.

## 🟡 Warnings (15)

- **`server/storage.ts` (getClassProgressSummary):** Missing pagination guard: getClassProgressSummary accepts optional limit/offset but callers that omit limit get an unbounded result set. For a class with hundreds of students this pulls all rows and all their game sessions into Node.js memory.
  → *Add a hard server-side cap: const safeLimit = Math.min(opts?.limit ?? 50, 200). Apply .limit(safeLimit).offset(opts?.offset ?? 0) unconditionally on the query so no code path can trigger a full-table scan into memory.*
- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices updates every stock row with a new random price in a loop of sequential awaits with no wrapping database transaction. A process crash mid-loop leaves prices partially updated, corrupting portfolio valuation for all users.
  → *Wrap all price updates in a single db.transaction(async (tx) => { for (const stock of stocks) { await tx.update(...) } }). If any update fails the entire price tick rolls back atomically.*
- **`server/storage.ts` (getDashboardStats):** getDashboardStats fires multiple independent aggregate queries (total income, total expenses, category breakdown, recent transactions) with sequential awaits. Each waits for the previous to finish despite having no data dependency on each other.
  → *Replace sequential awaits with const [income, expenses, breakdown, recent] = await Promise.all([incomeQuery, expenseQuery, breakdownQuery, recentQuery]). This cuts wall-clock latency for the dashboard load to the duration of the slowest single query.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** The sequential `.then()` chain means a failure in any intermediate step (e.g. `seedFinancialAcademyLesson`) skips all subsequent seeds but the single `.catch` at the end only logs 'Init error' with no indication of which step failed. Debugging seeding regressions in production will be blind.
  → *Wrap each step individually or use an async IIFE with a named step pattern: `(async () => { const steps: [string, ()=>Promise<any>][] = [['initTables', initSupabaseTables], ['trimOrgs', trimOrgNamesInSupabase], ...]; for (const [name, fn] of steps) { try { await fn(); } catch(e) { console.error(`[Supabase] Step '${name}' failed:`, e); } } })();`. This ensures later seeds still run after an earlier failure.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** There is no authentication or authorization check on this endpoint. Any unauthenticated request can enumerate and download objects by guessing file paths. For a school-aged user platform this may expose uploaded content (e.g. org logos, lesson assets) without access control.
  → *Determine which objects are truly public (logos used on the login page) versus private (student assets). For private assets, add a session/auth guard middleware before this handler. For public assets, consider serving them via a CDN with signed URLs generated server-side rather than proxying through the app server, which also removes the memory/streaming cost.*
- **`shared/schema.ts` (categories table definition):** categories.userId is varchar with no length limit and no index, but it is used as a filter to distinguish user-owned vs system categories. Every query like WHERE user_id = ? OR user_id IS NULL will do a full table scan as the table grows.
  → *Add `index('idx_categories_user_id').on(t.userId)` in a table-level index callback. Also size the varchar to match the users.id column (e.g., varchar('user_id', { length: 36 }) if using UUIDs) so Postgres can use more efficient storage and statistics.*
- **`shared/schema.ts` (budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** createdAt columns across all these tables use .defaultNow() but omit .notNull(), meaning a deliberate NULL insert bypasses the default and leaves an unindexable, semantically invalid audit timestamp.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g., `createdAt: timestamp('created_at').defaultNow().notNull()`. This matches the intent already applied to transactions.createdAt and makes the constraint explicit at the DB level.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** stockId is a foreign key referenced in every portfolio read and trade query but has no index. JOIN and WHERE clauses on stock_id will perform sequential scans on these tables as user portfolios grow.
  → *Add a table-level index callback to both tables: `(t) => ({ stockIdx: index('idx_portfolio_holdings_stock_id').on(t.stockId) })` and equivalently for portfolioTransactions. For portfolioHoldings also consider a unique index on (userId, stockId) to enforce one holding row per user per stock at the DB level.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text instead of timestamp. Storing a date as free-form text prevents range queries (e.g., stale price detection), breaks sorting, and wastes storage compared to a 8-byte timestamptz.
  → *Change the column type to `timestamp('last_price_update_date')`. Update any seed/migration code that writes string dates to pass a JS Date object instead. This enables efficient WHERE last_price_update_date < NOW() - INTERVAL '1 day' queries for price-staleness checks.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** nextDueDate has no index despite being the primary filter for due/overdue bill queries (WHERE next_due_date <= NOW() AND user_id = ?). Without an index this becomes a full table scan for every reminder poll across all users.
  → *Add a composite index in the table callback: `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })`. This covers both per-user reminder lists and any background job that sweeps for globally overdue bills.*
- **`server/index.ts` (seedDatabase and probeDatabase awaits at startup (inside async IIFE or top-level)):** If seedDatabase and probeDatabase are awaited sequentially they add their combined latency to startup time before the HTTP server begins accepting connections. More critically, if either throws without a surrounding try/catch the unhandled rejection will crash the process without a useful shutdown sequence.
  → *Run them in parallel where safe: `await Promise.all([probeDatabase(), seedDatabase()]);`. Wrap the entire startup block in try/catch: `try { await Promise.all([...]); } catch (err) { captureError(err); process.exit(1); }`. This halves startup DB latency and ensures errors are reported to Sentry before exit.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / hardcoded scheduler intervals):** The scheduler tick intervals (the values passed to setInterval for weekly digest and purge) appear to be hardcoded numeric literals in the truncated code. Changing the polling frequency requires a code deploy rather than a config change, and there is no minimum-sanity guard against a misconfigured value of 0 or negative that would create a hot-loop.
  → *Extract intervals to environment variables: `const DIGEST_POLL_MS = Math.max(60_000, parseInt(process.env.DIGEST_POLL_MS ?? '3600000', 10));` and similarly for the purge interval. Add the same `Number.isFinite` guard already used for AUTO_PURGE_OLDER_THAN_DAYS. This lets ops teams tune polling without a redeploy and prevents a misconfigured value from hammering the database.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback `ref={el => { if (el) el.scrollIntoView(...) }}`):** An anonymous arrow function is passed as a ref callback on every render. This creates a new function instance each render, causes React to unmount/remount the ref target repeatedly, and scrolls the chat to bottom on every parent state change, not just when messages are added.
  → *Replace with a useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and attach `ref={bottomRef}` to the div.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function in AdminHelpChat):** `sendMessage` is defined as a plain async function inside the component without useCallback, so it is recreated on every render. It is passed to both the button onClick and handleKey, meaning both handlers get new references every render, defeating React's reconciliation for those elements.
  → *Wrap sendMessage and handleKey with useCallback: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e) => { ... }, [sendMessage]);`*
- **`client/src/pages/AdminDashboard.tsx` (Multiple useQuery calls at dashboard top-level (users, schools, lessons, analytics, etc.)):** All admin stat queries are issued independently with no shared loading or error boundary. If any single query fails, the dashboard silently shows stale/empty data with no user-visible error, causing confusion for admins who may act on incomplete information.
  → *Add explicit `isError` and `isLoading` handling per query or use a combined loading state: `const isLoading = usersQuery.isLoading || schoolsQuery.isLoading || ...;` and render a single error banner when any `isError` is true, including the specific resource that failed.*

## 🔵 Info / Tech Debt (1)

- **`server/routes.ts` (duplicate health check endpoints: /api/health (~line 34) and /healthz (~line 72)):** Two health check endpoints exist with different semantics: `/api/health` returns immediately with no DB check, `/healthz` performs a live DB query. ## 💡 Opportunities

### Centralised pagination helper: 🔥 High Impact
Introduce a single helper function applyPagination(query, opts: {limit?: number, offset?: number}, maxLimit = 200) that enforces the cap and applies .limit().offset(). Apply it to getTransactions, getGameSessions, getLeaderboard, and getClassProgressSummary to eliminate repeated ad-hoc limit handling and prevent future unbounded queries.

### Sensitive field redaction in logs: ⚡ Medium Impact
Several insert/update paths log full argument objects (e.g. InsertLinkedCard, InsertDocumentUpload) which may contain card metadata or document identifiers. Add a redact() utility that strips fields matching /card|account|token|hash|key/i before any logger call, reducing PII exposure risk for a student-facing app under Caribbean data protection obligations.

### Seed functions run on every startup without idempotency checks: ⚡ Medium Impact
seedMarketData and seedLearningModules appear to INSERT rows without first checking whether data already exists, risking duplicate rows on each server restart. Add a SELECT COUNT(*) guard or use INSERT ... ON CONFLICT DO NOTHING so seeds are safe to run repeatedly without side effects.

### Parallel domain route registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import are executed sequentially. If any registration function performs async I/O (DB schema checks, cache warm-up), this adds to cold-start time. Refactor to `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` assuming routes don't depend on each other's side effects, which could cut cold-start by the sum of their individual async waits.

### Object storage proxy memory pressure: 🔥 High Impact
The `/public-objects` handler streams files through the Node process on every request. For static assets like logos this bypasses any CDN caching and ties up event-loop I/O under classroom load spikes. Uploading assets to object storage with public read ACLs and redirecting clients directly to the storage URL (308) would eliminate the proxy hop entirely, the same pattern already used for `/uploads/logos`.

### Dynamic import caching in /healthz: 📌 Low Impact
`import('./db')` and `import('drizzle-orm')` inside the `/healthz` handler are re-evaluated on every health poll (typically every 30 s from Railway). While Node module caching means the modules are not re-executed, the dynamic import machinery adds minor overhead. Move these imports to the top of the file as static imports to eliminate the overhead and make the dependency graph explicit for bundlers.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract `export const DEFAULT_CURRENCY = 'BSD'` from an env-aware config file and reference it in each column default. This means a regional expansion or test-environment override requires one change instead of six schema edits.

### Add unique constraint on simulatedStocks.ticker: ⚡ Medium Impact
ticker is used as a human-facing identifier and almost certainly queried by value, but has no unique constraint and no index. A duplicate ticker inserted by a seed script or admin tool would silently create two rows and corrupt portfolio lookups. Add `uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker)` to guarantee uniqueness and accelerate ticker-based lookups without a separate indexed column.

### Replace open-ended text columns with sized varchar where domain length is known: 📌 Low Impact
Columns like linkedCards.lastFour (always 4 chars), transactions.currency (3-char ISO code), and categories.color (7-char hex) are typed as unbounded text. Using varchar with explicit limits (e.g., varchar('last_four', {length:4})) pushes validation to the DB layer, reduces per-row storage in TOAST, and makes intent clear to future developers without requiring application-level length checks.

### Centralise graceful shutdown into a single registered handler: 🔥 High Impact
Currently shutdown concerns (intervals, httpServer, DB pool) are spread across the file or absent entirely. Create one `shutdown(signal)` async function that clears intervals, closes the httpServer, drains the job worker, and closes the DB pool in dependency order, then register it for both SIGTERM and SIGINT. This eliminates the risk of partial shutdown leaving zombie jobs or open handles that prevent the process from exiting, which Replit autoscaling environments will surface as slow restarts.

### Replace full response-body capture with structured request/response metadata logging: ⚡ Medium Impact
The current res.json monkey-patch allocates a copy of every response object in memory until the finish event fires. For large list responses (quiz history, usage reports) this creates unnecessary GC pressure. Replacing it with logging only method, path, status, duration, and a response-size header eliminates the allocation and the PII risk simultaneously, with no loss of operational observability.

### Add a startup readiness probe separate from the public HTTP listener: ⚡ Medium Impact
The server currently begins accepting public traffic on the main port before seedDatabase and probeDatabase complete, because Express binds in listen() which may be called before the async init finishes (depending on structure in truncated code). Introduce a separate internal healthcheck endpoint that returns 503 until init completes, or delay httpServer.listen() until after Promise.all resolves. This prevents the app from serving 500s during the cold-start window, which is especially impactful in Replit's wake-from-sleep flow.

### Parallelize independent admin data fetches: 🔥 High Impact
Any place where two or more awaits fetch unrelated resources sequentially (e.g., fetching users then schools) should be replaced with Promise.all([fetchUsers(), fetchSchools()]). This reduces total load time by the sum of the slower requests and is especially impactful on slower Caribbean mobile networks.

### Extract tab sections into separate memoized components: 🔥 High Impact
Each dashboard tab (Users, Schools, Lessons, Analytics) is rendered inline in a single giant component. Moving each to its own file with React.memo prevents all tabs from re-rendering when unrelated state (e.g., chat open/close) changes, directly reducing wasted renders.

### Move hardcoded pagination size and API base path to environment config: ⚡ Medium Impact
Hardcoded values like page sizes (e.g., 10 per page) and any relative API prefixes (/api/admin/*) should be sourced from import.meta.env variables. This enables staging vs production configuration without code changes and makes limits tunable without a redeploy.
