## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 110-112)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied beforehand. For a school-aged user base this exposes the app to runaway API costs and abuse — a single user could spam requests and exhaust OpenAI quota.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Verify the path prefixes match what each register function actually mounts.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~lines 43, 58)
**Issue:** The `host` value is read directly from `req.headers['x-forwarded-host'] || req.headers.host` and interpolated unescaped into HTTP response bodies. A crafted Host header can inject arbitrary content into robots.txt (plain text) or break XML structure in sitemap.xml, enabling cache-poisoning or response-splitting attacks.

**Fix:** Validate the constructed base URL against an allowlist from environment config: `const ALLOWED_HOST = process.env.PUBLIC_HOST; const base = ALLOWED_HOST ? `https://${ALLOWED_HOST}` : `${proto}://${host}`;` and reject or fall back if the header value doesn't match a safe hostname regex (`/^[a-zA-Z0-9.-]+(:\d+)?$/`).

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Raw file content stored as base64 in the job payload (fileB64). For a multi-page PDF this can be several MB per job row, bloating the jobs table, slowing queue polling queries, and exposing file bytes to anything that reads job rows (logs, admin UIs, DB backups).

**Fix:** Upload the file to private object storage at enqueue time and store only the resulting object path in the payload. Inside runExtractPaper, download the file by path, process it, then delete the temp object. This keeps job rows small (<1 KB) and removes raw file bytes from the DB.

### `shared/schema.ts`: simulatedStocks table, ticker column
**Issue:** The ticker column has no unique constraint and no index. If seed data is re-run or an admin inserts a duplicate ticker, queries that look up a stock by ticker (e.g. searching 'BISX:BPF') will return multiple rows silently, corrupting portfolio calculations. This is a data-integrity gap in a finance context.

**Fix:** Add a unique index: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This gives both the uniqueness guarantee and query performance for ticker lookups.

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** The full messages array is sent to /api/admin/help-chat on every request and grows unbounded in memory and payload size. A 2-hour session could accumulate hundreds of messages sent repeatedly, and there is no server-side validation that the caller is authenticated as admin before the AI processes arbitrary input.

**Fix:** Slice the context window before sending: `const context = next.slice(-10)` and pass `context` instead of `next`. Add an Authorization header check on the /api/admin/help-chat route server-side that verifies the admin session token before forwarding to the AI provider.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level component render
**Issue:** If useAdminAuth returns a loading state or an unauthenticated state, the component likely still renders query hooks and potentially exposes admin UI fragments before the auth check resolves, because there is no early return before the useQuery calls that fetch user lists, school data, and audit logs.

**Fix:** Immediately after calling useAdminAuth, add: `if (authLoading) return <FullPageSpinner />; if (!isAdmin) { setLocation('/admin/login'); return null; }` Ensure this appears before any useQuery declarations so data is never fetched for unauthenticated sessions.

## 🟡 Warnings (14)

- **`server/routes.ts` (Supabase init chain (~lines 116-124)):** The seven Supabase init/seed functions are chained sequentially with `.then()`. If any one step takes 2-3 seconds (e.g. seedRegionalContent seeding many rows), server cold-start is delayed unnecessarily. Steps that are independent (e.g. seedStaticModules, seedRegionalContent, seedGameContent) do not need to block each other.
  → *After the mandatory ordering constraint (initSupabaseTables must precede trimOrgNamesInSupabase and initStaticContentTables must precede seeding), run independent seeds in parallel: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables().then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]))])).catch(e => console.error('[Supabase] Init error:', e));`*
- **`server/routes.ts` (/api/health and /healthz (~lines 33, 73)):** There are two overlapping health-check endpoints (`/api/health` returns `{status:'ok'}` with no DB check; `/healthz` does a real DB probe). Railway and uptime monitors likely hit `/api/health`, which will report `ok` even when the database is down, hiding outages.
  → *Consolidate to one endpoint. Move the DB liveness check into `/api/health` and remove or 308-redirect `/healthz` to it. If a shallow ping is needed separately, explicitly document why and name it `/api/health/shallow` vs `/api/health/live` to avoid confusion.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without any sanitisation. A path like `../../private/keys` or URL-encoded traversal sequences could resolve to unintended storage keys depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, '').replace(/^\//, '');` and validate it matches an expected pattern such as `/^[\w\-./]+$/`. Reject with 400 if it fails. Also confirm ObjectStorageService itself does not perform filesystem path resolution.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one inside a for loop with sequential awaits. For a 40-question paper this is 40 round-trips to the DB, each blocking the next. A crash mid-loop leaves a partial set; the idempotency delete at the top of a retry then destroys the partial set, which is correct but the retry still pays the sequential cost again.
  → *Build the full insert array first, then execute a single db.insert(extractedQuestions).values(allRows) call. Wrap it and the updateExamPaper call in a single transaction so partial inserts are impossible and no idempotency delete is needed on retry.*
- **`server/jobHandlers.ts` (runExtractPaper — model: "gpt-4o" (two occurrences)):** The OpenAI model name is hardcoded to "gpt-4o" in two separate chat.completions.create calls. Changing the model for cost control or testing requires a code deploy rather than an environment variable change.
  → *Read the model from an env var with a safe default: const EXTRACTION_MODEL = process.env.OPENAI_EXTRACTION_MODEL ?? "gpt-4o"; and use EXTRACTION_MODEL in both calls. Do the same for max_completion_tokens if budget control is needed.*
- **`server/jobHandlers.ts` (admin-csv-export handler (type AdminRow = Record<string, unknown>; let dat...)):** Based on the truncated code pattern, the export handler likely fetches entire DB tables into memory (let dat = await db.select()...) and serialises them as a single in-memory string before uploading. For large schools this risks OOM and multi-second event-loop blocks during CSV serialisation.
  → *Stream rows in pages (e.g. LIMIT 1000 with cursor-based pagination), pipe each page through a CSV transform stream directly into the object storage upload stream. Node's stream.pipeline with object storage's createWriteStream eliminates the full in-memory accumulation.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload without validation. A malformed or malicious job payload with olderThanDays = 0 would delete all AI usage records; a negative value would delete nothing but succeed silently, masking a misconfiguration.
  → *Add a guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays value: ' + olderThanDays); This causes the job to fail visibly rather than silently corrupt data.*
- **`shared/schema.ts` (categories table definition):** userId in categories is varchar with no length, and the column has no index. Every query filtering system vs user-owned categories (WHERE user_id = ? OR user_id IS NULL) will do a full table scan as the category list grows. The nullable FK also means Drizzle won't enforce referential integrity for the optional owner relationship at the DB level consistently across engines.
  → *Add an explicit index: index('idx_categories_user_id').on(t.userId) in the table's index callback. Match the varchar length to the users.id column (e.g. varchar('user_id', { length: 36 }) if using UUIDs). This is the same fix needed on every userId column across all tables.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions tables):** These tables have no indexes on userId even though every application query will filter by userId. portfolioHoldings also has no index on stockId, which is joined against simulatedStocks on every portfolio view. As user counts grow this will cause full table scans on every dashboard load.
  → *Add index callbacks to each table. For budgets: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). For portfolioHoldings: (t) => ({ userIdx: index('idx_holdings_user_id').on(t.userId), stockIdx: index('idx_holdings_stock_id').on(t.stockId) }). Apply the same pattern to linkedCards, savingsGoals, billReminders, and portfolioTransactions.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt is defined with .defaultNow() but without .notNull(). This means the column is nullable in the DB schema and Drizzle's inferred TypeScript type is Date | null, forcing null-checks throughout the application. A missing timestamp is almost never a valid business state for an audit column.
  → *Change every createdAt definition to timestamp('created_at').defaultNow().notNull(). Same applies to executedAt in portfolioTransactions and purchasedAt in portfolioHoldings.*
- **`shared/schema.ts` (billReminders table, nextDueDate column and portfolioTransactions executedAt column):** billReminders.nextDueDate has no index despite being the primary column for cron-style due-date queries (WHERE next_due_date <= NOW()). Similarly portfolioTransactions.executedAt has no index despite being used for time-series portfolio history queries. Both will degrade significantly under load.
  → *For billReminders add: (t) => ({ dueDateIdx: index('idx_bill_reminders_due_date').on(t.nextDueDate), userIdx: index('idx_bill_reminders_user_id').on(t.userId) }). For portfolioTransactions add executedAt to the index alongside userId: index('idx_portfolio_tx_user_executed').on(t.userId, t.executedAt).*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scrollIntoView ref callback):** The ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is an inline arrow function recreated on every render. React treats a new function reference as an unmount+remount of the ref, so scrollIntoView fires on every parent state change (including while the user is typing), not just when a new message arrives.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]); // then <div ref={bottomRef} />`*
- **`client/src/pages/AdminDashboard.tsx` (Chart data transformation / dashboard stats derivation):** Bar and Line chart data arrays are most likely constructed with inline map/reduce expressions directly in JSX or in the render body without useMemo. On any state change (search filter, pagination, dialog open/close) these O(n) transformations rerun unnecessarily across potentially thousands of student/school records.
  → *Wrap each derived dataset in useMemo keyed to its source query data: `const chartData = useMemo(() => rawUsers.map(u => ({ name: u.school, score: u.avgScore })), [rawUsers]);` Apply this to every recharts data prop that derives from a query result.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple independent useQuery calls for dashboard bootstrap data):** Dashboard likely fires separate sequential-equivalent useQuery calls for users, schools, teachers, and analytics on mount. While React Query fires them concurrently, each has its own loading state and the UI has no unified loading/error boundary, meaning partial data renders silently if any single query fails.
  → *Either combine into a single /api/admin/dashboard-summary endpoint to reduce round trips, or use `useQueries` and derive a unified `isLoading = results.some(r => r.isLoading)` and `isError = results.some(r => r.isError)` flag. Render a single error card with retry when any query fails rather than silently showing stale or empty charts.*

## 💡 Opportunities

### Environment-driven base URL instead of header inference: 🔥 High Impact
Set `PUBLIC_HOST` (and optionally `PUBLIC_PROTO`) as environment variables and use them as the authoritative source for base URL construction in robots.txt, sitemap.xml, and any future canonical link generation. This eliminates the header-injection surface, removes per-request string construction overhead, and makes deployments to staging vs production deterministic without code changes.

### Domain router registration parallelisation: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel imports are still sequential. If any registration function performs async I/O (DB schema checks, route pre-warming), they stall each other. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` to cut cold-start time proportionally to the slowest individual registration.

### Structured logging for init errors instead of console.error: ⚡ Medium Impact
The Supabase init `.catch(e => console.error('[Supabase] Init error:', e))` swallows the full error object in a format that won't be parsed by Railway or Sentry log pipelines. Replace with a structured logger call that includes `error.message`, `error.stack`, and a `context: 'supabase-init'` field. This makes post-deploy failures immediately actionable without digging through raw logs.

### Batch parallel object-storage existence checks in streamPrivateObjectToResponse: 📌 Low Impact
file.exists() is awaited before piping begins. If multiple concurrent export downloads hit this, each pays a separate round-trip. Pre-signing URLs or caching a short-lived exists result per object key would reduce latency by one network hop per download.

### Centralise AI call error handling with a retryable wrapper: ⚡ Medium Impact
Both openai.chat.completions.create calls in runExtractPaper are unwrapped — a transient 429 or 503 from OpenAI surfaces as an untyped job failure. A thin wrapper that detects retryable HTTP status codes and throws a typed RetryableError (which the job runner already understands) would improve retry signal quality and avoid burning retry attempts on non-retryable errors.

### Move PDF/image extraction into a dedicated worker pool: 🔥 High Impact
pdf-parse runs synchronous CPU-intensive parsing on the main Node thread. Under concurrent extract-paper jobs this will block the event loop. Offloading to a worker_threads pool (even a pool of one) isolates CPU work and keeps the job queue responsive for lightweight handlers like purge-ai-usage.

### Centralise userId column definition: 🔥 High Impact
Define a shared userIdCol helper (e.g. const userIdCol = (colName = 'user_id') => varchar(colName, { length: 36 }).notNull().references(() => users.id)) and reuse it across all tables. This eliminates the current inconsistency where length is unspecified and notNull is missing in some places, and makes future auth ID format changes (e.g. switching from UUID to CUID) a single-line change.

### Replace repeated currency text column with a shared enum type: ⚡ Medium Impact
Six tables each define currency as text('currency').default('BSD').notNull() with no constraint on valid values. A pgEnum('currency_code', ['BSD', 'USD', 'TTD', 'JMD']) column type would enforce valid ISO codes at the DB level, reduce storage from variable-length text to a small enum OID, and make adding a new supported currency a migration rather than a search-and-replace.

### Add partial index for active bill reminders due-date polling: ⚡ Medium Impact
The cron job that fetches upcoming bills likely queries WHERE next_due_date <= NOW() AND status != 'dismissed' (or similar). A standard index on nextDueDate will include all historical rows. A partial index (WHERE is_auto_detected = false OR next_due_date >= CURRENT_DATE) would keep the working set small and cut index scan time for the most frequent query pattern on this table.

### Paginated admin tables with server-side filtering: 🔥 High Impact
If user/school lists are filtered and sliced client-side from a full dataset query, moving filtering to query params (?search=&page=&limit=) and fetching only the current page eliminates transferring thousands of records on every dashboard load. Expected impact: 60-80% reduction in initial data transfer for institutions with large user bases.

### Memoize all table row callback handlers with useCallback: ⚡ Medium Impact
Edit, delete, and view handlers defined inline in table row JSX (e.g., onClick={() => openEditDialog(user)}) cause every visible row to re-render when any single row's state changes. Extracting to useCallback or a row sub-component with React.memo breaks this cascade and is especially valuable for the paginated user and school tables.

### Add Content-Security-Policy and sanitize AI chat output: ⚡ Medium Impact
AI assistant replies rendered via whitespace-preserving text nodes are safe now, but if the rendering is ever upgraded to support markdown (dangerouslySetInnerHTML or a markdown renderer), XSS becomes possible since reply content is not sanitised. Introduce DOMPurify sanitization and a strict CSP header at the server layer before any rich rendering is added.
