## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 113-115)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. Any authenticated or unauthenticated caller can hammer expensive AI endpoints, causing runaway OpenAI spend and potential DoS for a school-aged user base.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP using express-rate-limit) and apply it before each register call: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 42) and sitemap.xml handler (~line 57)
**Issue:** The `host` and `x-forwarded-host` headers are taken directly from the request and interpolated into URLs sent to clients without validation. An attacker can spoof the Host header to inject arbitrary URLs into the Sitemap directive or sitemap <loc> tags, enabling cache-poisoning or phishing via SEO.

**Fix:** Add a SITE_BASE_URL environment variable (e.g. `https://finsightlite.com`) and use it unconditionally: `const base = process.env.SITE_BASE_URL ?? 'https://finsightlite.com';`. Remove all header-derived base URL logic from both handlers.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** fileB64 (the full base64-encoded exam paper) is stored directly in the job payload, meaning every retry duplicates this large blob in the jobs table. A single 5MB PDF becomes ~6.7MB of base64, and with retries this bloats the DB and risks hitting row size limits.

**Fix:** Upload the file to object storage before enqueueing the job, then pass only the storage path in the payload (e.g., { paperId, objectPath: '/private/papers/123.pdf', ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start. Remove fileB64 from JobPayloads['extract-paper'] entirely.

---

### `server/jobHandlers.ts`: for loop inside runExtractPaper (~line 100-110)
**Issue:** Extracted questions are inserted one-by-one with sequential awaits inside a for loop. For a 40-question paper this is 40 serial round-trips. If any single insert fails mid-loop, partial data is left and the idempotency delete at the top of a retry wipes valid prior work but the root cause (e.g. a malformed question) will re-fail at the same index.

**Fix:** Collect all valid question objects into an array first, then do a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Add a try/catch around the bulk insert. This reduces 40 round-trips to 1 and makes the operation atomic.

---

### `server/index.ts`: seedDatabase / probeDatabase calls in startup
**Issue:** Blocking sequential awaits on seedDatabase and probeDatabase during startup delay the HTTP server from accepting connections. If either call hangs (e.g., DB unreachable), the server never starts and no timeout kills the process.

**Fix:** Wrap both in Promise.race with a 10-second timeout: `await Promise.race([Promise.all([probeDatabase(), seedDatabase()]), new Promise((_, r) => setTimeout(() => r(new Error('startup timeout')), 10000))])`. Also run them in parallel with Promise.all since they are independent.

---

### `server/index.ts`: setInterval for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The two setInterval calls are assigned to variables but those handles are never passed to clearInterval in any shutdown handler. On SIGTERM/SIGINT the intervals keep firing during drain, potentially enqueuing duplicate jobs mid-shutdown. The handles also leak if the module is reloaded in test environments.

**Fix:** Capture both handles (`const digestInterval = setInterval(...)`) and clear them in the graceful shutdown handler before closing the HTTP server: `clearInterval(digestInterval); clearInterval(purgeInterval);`. Register the shutdown handler with `process.on('SIGTERM', shutdown)` and `process.on('SIGINT', shutdown)`.

---

### `server/index.ts`: res.json logging middleware (~line 68)
**Issue:** Every API JSON response body is serialized and appended to the log line unconditionally. For a financial literacy app with school-aged users this will log scores, progress data, and any PII returned by API endpoints in plaintext server logs with no scrubbing or size cap.

**Fix:** Either remove response body logging entirely, or add an allowlist of safe paths and a size guard: `if (capturedJsonResponse && path.startsWith('/api') && !SENSITIVE_PATHS.has(path)) { logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 200)}`; }`

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage
**Issue:** The full `messages` array is sent to `/api/admin/help-chat` on every turn with no size cap. A user can accumulate thousands of tokens of history, causing unbounded cost and potential denial-of-service against the OpenAI quota. There is also no server-side rate limit enforced from the client side.

**Fix:** Slice the history before sending: `const context = next.slice(-20);` (keep last 20 messages) and pass `context` instead of `next`. Add a visible message count warning in the UI when approaching the cap.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage, missing auth check
**Issue:** `/api/admin/help-chat` is called without verifying the admin session client-side first. If the `useAdminAuth` hook redirects asynchronously but the chat widget is already mounted, a non-admin user can fire requests before the redirect completes.

**Fix:** Consume `useAdminAuth` inside `AdminHelpChat` (or pass `isAdmin` as a prop) and guard the send: `if (!isAdmin) return;` at the top of `sendMessage`. The server must also enforce the session, but the client guard prevents the visible UX race.

## 🟡 Warnings (10)

- **`server/routes.ts` (Supabase init chain (~line 118-126)):** Seven async Supabase operations are chained sequentially with .then(), meaning each waits for the previous to complete. On a cold start this adds cumulative latency before the app is fully ready, and a failure in any middle step (e.g. seedFinancialAcademyLesson) silently skips all subsequent seeds with only a single catch log.
  → *Group independent operations with Promise.all where order does not matter, and log each failure individually so skipped steps are visible: `await Promise.all([seedFinancialAcademyLesson(), initStaticContentTables()]).catch(e => console.error('[Supabase] seed error:', e));` Structure the chain into explicit ordered phases with per-phase error logging.*
- **`server/routes.ts` (/healthz handler (~line 78)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every call. Dynamic imports are cached after the first resolution, but the overhead of going through the module resolution path on each health-check request (which uptime monitors hit every 30s) is unnecessary and adds latency.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The health check handler then closes over the already-resolved module references.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The DB delete and the rowCount extraction have no try/catch. An unexpected DB error (connection drop, lock timeout) will crash the job worker with an unhandled rejection and no contextual logging, making it hard to diagnose in production.
  → *Wrap the entire handler body in try/catch. On catch, log { job: 'purge-ai-usage', olderThanDays, cutoff, err } and rethrow so the job framework can schedule a retry. Also validate that olderThanDays is a positive integer before constructing the cutoff date to prevent a bad payload from deleting all rows.*
- **`server/jobHandlers.ts` (parsePrivatePath / PRIVATE_OBJECT_DIR usage):** PRIVATE_OBJECT_DIR is checked at call-time inside parsePrivatePath, not at startup. If the env var is missing, errors surface only when a job runs, not when the server boots, making misconfigured deployments hard to catch before they affect students.
  → *Add a startup assertion in registerJobHandlers (or a dedicated config module): if (!process.env.PRIVATE_OBJECT_DIR) throw new Error('PRIVATE_OBJECT_DIR must be set'). This fails fast at boot rather than silently at job execution time.*
- **`server/jobHandlers.ts` (openai import at top of file):** openai is imported from ./replit_integrations/chat/routes, a routes module. Importing a configured client from a routes file creates a hidden coupling: if that routes module has side effects (registering Express routes, initialising middleware), those side effects fire whenever jobHandlers.ts is loaded, even in non-HTTP contexts like test runners or CLI scripts.
  → *Extract the OpenAI client instantiation into its own module (e.g., ./lib/openaiClient.ts) that exports only the configured client. Both the routes module and jobHandlers.ts import from that shared module. This eliminates the side-effect risk and makes the dependency explicit.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince = (dow + 0) % 7` — the `+ 0` is dead/placeholder arithmetic that produces the wrong result for Sunday (dow=0 yields daysSince=0, so it never subtracts days and always returns today, not the most recent Sunday). The subsequent guard `if (d.getTime() > now.getTime())` patches Sunday only if the current time is before 7 pm, but midnight-to-7 pm on Sunday will silently skip the digest.
  → *Replace with `const daysSince = dow === 0 ? 0 : dow;` and verify the full truth table. For Sunday before 7 pm, daysSince should be 0 and the `> now` guard subtracts 7 days correctly — but document this explicitly and add a unit test covering Sunday 6:59 pm vs 7:01 pm.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and scheduler):** The purge scheduler interval value (how often it checks) appears to be hardcoded in the setInterval call rather than sourced from environment config. For a production app the check frequency should be configurable without a redeploy, and the current pattern makes it invisible during code review because the truncated file hides the literal.
  → *Add `PURGE_CHECK_INTERVAL_MS=process.env.PURGE_CHECK_INTERVAL_MS ?? '3600000'` to environment config and use it: `setInterval(maybeRunAiUsagePurge, parseInt(process.env.PURGE_CHECK_INTERVAL_MS ?? '3600000', 10))`. Document the default in .env.example.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages scroll ref callback):** The inline `ref={el => { if (el) el.scrollIntoView(...) }}` creates a new function on every render and calls scrollIntoView on every re-render (including while typing), not just when a new message arrives.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – handleKey):** `handleKey` is defined as a plain arrow function inside the component body without `useCallback`, so it is recreated on every render and passed as a new reference to the `onKeyDown` prop, causing the input element to detach/reattach the handler unnecessarily.
  → *Wrap with useCallback: `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);` and likewise wrap `sendMessage` in `useCallback` with `[messages, input, loading]` deps.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – error handler in catch block):** The catch block silently swallows the error object entirely (`catch {`). No logging occurs, making it impossible to diagnose API failures (auth errors, 429 rate limits, network issues) in production.
  → *Change to `catch (err) { console.error('[AdminHelpChat] sendMessage failed:', err); setMessages(prev => [...prev, { role: 'assistant', content: 'Sorry, something went wrong. Please try again.' }]); }`. Avoid logging message content to prevent leaking user input to console in production builds.*

## 💡 Opportunities

### Parallel domain-route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six awaited calls. If any registerXRoutes function does async work (DB calls, sub-imports), this is again serial. Audit each registrar; those that are synchronous or purely app.use() calls should be called without await to eliminate unnecessary microtask overhead at startup.

### Structured startup logging with phase timing: ⚡ Medium Impact
Adding console.time/timeEnd (or a pino child logger) around the Supabase init chain and route registration phases would expose exactly where cold-start time is being spent. For a Railway-deployed app serving school users who may hit the app fresh each session, reducing cold-start from e.g. 4s to 1.5s meaningfully improves first-load experience.

### Object storage filePath traversal hardening: 🔥 High Impact
The `/public-objects/:filePath(*)` wildcard passes the raw path segment to `objectStorage.searchPublicObject()` without sanitisation. Depending on the storage SDK, a crafted path like `../../private/secret` could access unintended objects. Validate filePath against an allowlist of extensions and strip any `..` segments before passing to the storage layer.

### Parallel AI calls in extract-paper: 🔥 High Impact
The text extraction step (pdf-parse or vision) and the subsequent GPT-4o extraction prompt are fully sequential. For image uploads the vision call already returns structured-enough text that a second LLM pass could be designed as a single prompt, eliminating one ~3-5s OpenAI round-trip per job and halving P50 latency for MoneyLab uploads.

### Idempotency token on enqueueJob for extract-paper: 🔥 High Impact
If a teacher double-submits a paper upload, two extract-paper jobs race against each other and the idempotency delete at the top of runExtractPaper will cause them to wipe each other's questions. Adding a unique idempotency key (e.g., paperId) to enqueueJob so the second enqueue is a no-op would prevent duplicate processing entirely.

### Structured logging with job context: ⚡ Medium Impact
Most handlers throw raw errors with no job metadata attached. Adding a thin logging wrapper that annotates every thrown error with { jobType, jobId, attempt } before rethrowing would drastically reduce mean-time-to-diagnosis for failures in the Caribbean school deployment where support bandwidth is limited.

### Graceful shutdown handler: 🔥 High Impact
Add a single `shutdown` async function triggered by SIGTERM and SIGINT that clears both intervals, stops accepting new connections via `httpServer.close()`, waits for in-flight requests to drain with a 15-second hard kill fallback, and flushes Sentry. This prevents duplicate job enqueues and ensures the process exits cleanly on container restarts, which is critical for a hosted Replit deployment.

### Parallel startup I/O with bounded timeout: ⚡ Medium Impact
probeDatabase, seedDatabase, registerRoutes, and startJobWorker are likely run sequentially today. Running independent ones with Promise.all and a startup deadline would reduce cold-start latency visibly for school users on slower Caribbean connections and surface hung DB connections immediately rather than silently.

### Response body log scrubbing middleware: 🔥 High Impact
Replace the open-ended JSON response logger with a structured logger that records only method, path, status, and duration — never body content. For COPPA/FERPA-adjacent compliance with school-aged users, response bodies containing scores or user identifiers must not appear in application logs. This is a one-line removal with high compliance impact.

### Parallel data fetching with Promise.all: 🔥 High Impact
The dashboard likely fires multiple sequential useQuery calls (users, schools, analytics, etc.) that are independent. Consolidating them into a single aggregated endpoint or ensuring they are initiated simultaneously (they already are if using separate useQuery hooks, but any server-side sequential awaits should use Promise.all) would reduce dashboard time-to-interactive by the sum of serial round-trip times.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Search/filter logic applied to large user or school lists inside the render body should be wrapped in useMemo with the list and filter string as dependencies. This prevents O(n) re-filtering on every keystroke-unrelated re-render, which matters when lists reach hundreds of rows.

### Extract chat widget into a lazy-loaded module: ⚡ Medium Impact
AdminHelpChat along with its Lucide icons and state is bundled into the main AdminDashboard chunk. Moving it to a dynamic import (`const AdminHelpChat = lazy(() => import('./AdminHelpChat'))`) would reduce initial parse cost for admins who never open the chat, shrinking the main chunk noticeably given the icon imports.
