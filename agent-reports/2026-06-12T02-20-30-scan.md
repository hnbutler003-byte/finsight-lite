## 🔴 Critical (4)

### `server/routes.ts` — registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 111-113)
**Issue:** AI-heavy endpoints (chat, image generation, audio) are registered with no rate limiting middleware applied before them. Any authenticated or unauthenticated caller can hammer OpenAI at full speed, causing runaway API costs and potential DoS for school-aged users.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before calling the register functions.

---

### `server/routes.ts` — robots.txt and sitemap.xml handlers (~lines 38, 58)
**Issue:** `req.headers['x-forwarded-host']` and `req.headers['x-forwarded-proto']` are reflected directly into `Sitemap:` URLs and `<loc>` tags without any validation. An attacker can set these headers to inject arbitrary URLs into the sitemap or poison caches/proxies.

**Fix:** Read the canonical base URL from a single env var (`process.env.PUBLIC_BASE_URL`, required at startup). Fall back to a hardcoded production domain, never to arbitrary request headers. Remove the header-reflection logic entirely.

---

### `server/jobHandlers.ts` — runExtractPaper / extract-paper handler payload
**Issue:** Full file contents are stored as base64 in the job table payload (fileB64). For a multi-page PDF or high-res image this can be several MB per job row, bloating the jobs table, slowing queue polls, and potentially logging the raw binary string in job error output.

**Fix:** Upload the file to private object storage before enqueuing the job and pass only the storage path in the payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of execution. This keeps job rows small and aligns with how admin-csv-export already uses object storage for large data.

---

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth – auth check presence in mutation callbacks
**Issue:** Mutations that delete or modify users, lessons, or schools appear to rely solely on the server to enforce admin access, but if `useAdminAuth` only redirects on the client side there is no pre-flight guard preventing the mutate() call from being invoked by a non-admin who has manipulated local state (e.g. via devtools). Sensitive mutation handlers lack an explicit role assertion before calling mutate().

**Fix:** At the top of every destructive mutation's `onMutate` or immediately before `mutate()` is called, assert `if (!admin?.isAdmin) return;`. More importantly, ensure every /api/admin/* route on the server validates the session role server-side — client guards are defence-in-depth only.

## 🟡 Warnings (14)

- **`server/routes.ts` (Supabase init chain (~lines 116-124)):** The seven Supabase seed/init calls are chained sequentially with `.then()`, meaning each waits for the previous to finish. On a cold start this can block several seconds of unnecessary wall-clock time even though the results are independent.
  → *Replace the chain with `Promise.all([initSupabaseTables(), initStaticContentTables()]).then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` — respecting only the dependency that seeding requires tables to exist first.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject` without any sanitisation. A path like `../../private/keys.json` could traverse outside the intended public prefix depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Use Node's `path.posix.normalize` and assert the result stays within an allowed prefix.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop over questions):** Extracted questions are inserted one-at-a-time with await inside a for loop. For a 40-question exam paper this is 40 sequential round-trips to the database, each waiting on the previous before starting.
  → *Collect all valid question objects into an array and call a single db.insert(extractedQuestions).values(allQuestions) after the loop. If storage.createExtractedQuestion must be used, wrap all calls in Promise.all(validQuestions.map(...)) so inserts run concurrently. Either approach reduces N round-trips to 1.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model and max_completion_tokens)):** The OpenAI model name 'gpt-4o' and token limits (4096) are hardcoded in two separate completions calls. Changing the model for cost or capability reasons requires a code deploy, and different environments (dev/staging/prod) cannot use cheaper models without code changes.
  → *Read from environment variables: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o' and const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10). Apply these to both completions calls. Add these keys to your .env.example and deployment config.*
- **`server/jobHandlers.ts` (runExtractPaper — image branch, buffer.toString('base64')):** When the uploaded file is an image, the buffer decoded from fileB64 is immediately re-encoded to base64 to build the data URL for the vision API. This means the full image bytes exist in memory at least twice simultaneously (as Buffer and as base64 string) and is redundant if fileB64 is already the raw base64 string from the payload.
  → *Pass fileB64 directly into the data URL: `data:${mimeType};base64,${fileB64}` instead of re-encoding the decoded buffer. This eliminates one full copy of the image in memory and removes the unnecessary decode-then-encode round-trip.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch and does not validate that olderThanDays is a positive finite number before using it in arithmetic. A payload with olderThanDays: 0, NaN, or a negative value would compute an invalid or future cutoff date and could delete all rows or fail silently with an unexpected rowCount.
  → *Add input validation at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays <= 0) throw new Error('olderThanDays must be a positive number'). Wrap the delete in a try/catch that re-throws so the job worker marks it failed and retries appropriately.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Budget lookups and dashboard aggregations will do full table scans as user data grows, which is especially costly for per-user period filtering.
  → *Add a composite index in the budgets table options callback: index('idx_budgets_user_category').on(t.userId, t.categoryId). Also add index('idx_budgets_user_period').on(t.userId, t.period) if queries filter by period.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** portfolioHoldings has no index on (userId, stockId) and portfolioTransactions has no index on (userId, executedAt). Portfolio dashboard queries joining or filtering on these columns will scan full tables.
  → *In portfolioHoldings options callback add: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) — it should also be unique per user+stock since holdings are aggregated. In portfolioTransactions add: index('idx_portfolio_txns_user_date').on(t.userId, t.executedAt).*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is nullable by design to represent system categories, but there is no index on userId. Every per-user category fetch (WHERE user_id = $1 OR user_id IS NULL) will scan the full table. As the category list grows with user-defined entries this becomes expensive.
  → *Add index('idx_categories_user_id').on(t.userId) in a table options callback. The partial NULL rows for system categories will still be included when the query uses OR user_id IS NULL, and the index will accelerate the equality branch.*
- **`shared/schema.ts` (All tables with createdAt, purchasedAt, executedAt columns):** createdAt (and purchasedAt, executedAt) use .defaultNow() without .notNull(). defaultNow sets a DB-level default but Drizzle's TypeScript type still marks the column as nullable, meaning application code can insert NULL explicitly and ORDER BY createdAt queries will behave unpredictably on NULL rows.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). This enforces the constraint at the DB level and narrows the TypeScript type to Date instead of Date | null.*
- **`shared/schema.ts` (billReminders — nextDueDate; savingsGoals — deadline; simulatedStocks — annualReturnPct):** billReminders.nextDueDate is a core query column (fetching upcoming bills) but has no index. savingsGoals.deadline is similarly unindexed. Both will be used in ORDER BY / range WHERE clauses in reminder and goal-tracking features.
  → *Add index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) in billReminders options callback. Add index('idx_savings_goals_user_deadline').on(t.userId, t.deadline) in savingsGoals options callback. These enable efficient 'due within N days' range queries per user.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages state passed to /api/admin/help-chat):** The entire `messages` array is sent to the server on every turn with no size cap. A user can accumulate hundreds of messages in one session, causing payloads to grow unboundedly and likely exceeding the LLM context window or a reverse-proxy body limit, causing silent failures or high inference costs.
  → *Slice the history before sending: `const history = next.slice(-20);` and pass `{ messages: history }` to apiRequest. Add a server-side body-size check as well. Optionally display a 'conversation reset' notice when the cap is hit.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll ref callback `ref={el => { if (el) el.scrollIntoView(...) }}`):** An anonymous arrow function is used as a React ref callback on the scroll sentinel div. React treats a new function identity on every render as a ref change, calling the callback (and triggering scrollIntoView) on every single render of the chat panel — not just when new messages arrive. This causes jank and unnecessary DOM work.
  → *Replace with a stable ref: `const bottomRef = useRef<HTMLDivElement>(null);` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);`. Attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats – sequential useQuery calls for independent data):** Multiple independent useQuery hooks (users, schools, lessons, audit log, etc.) are declared sequentially. While React Query fires them concurrently by default, any place in the file where their results are fetched manually with `await apiRequest` inside a single async function uses sequential awaits, stalling the UI unnecessarily.
  → *Audit every async function that calls apiRequest more than once. Replace sequential `const a = await apiRequest(...); const b = await apiRequest(...);` patterns with `const [a, b] = await Promise.all([apiRequest(...), apiRequest(...)]);`. This alone can cut perceived load time by 40-60% on slow Caribbean mobile connections.*

## 💡 Opportunities

### Centralised async route-registration error boundary — ⚡ Medium Impact
Each `await register*Routes(app)` call can throw independently but there is no per-call catch. Wrapping the block in a single try/catch that logs which domain router failed and re-throws would make startup failures easier to diagnose in production logs and Sentry.

### Lazy-load ObjectStorageService — ⚡ Medium Impact
`new ObjectStorageService()` is instantiated at module load time at the top level, meaning any constructor I/O or credential failure crashes the entire module before `registerRoutes` is even called. Move instantiation inside `registerRoutes` (or inside the route handler) so failures are catchable and don't prevent the server from starting.

### Pin PUBLIC_BASE_URL at startup with a startup assertion — 🔥 High Impact
Add a startup check that asserts `process.env.PUBLIC_BASE_URL` is set and is a valid HTTPS URL before any routes are registered. This replaces all header-reflection logic across robots.txt, sitemap, and any future canonical URL usage, and surfaces misconfiguration immediately on deploy rather than silently serving wrong URLs.

### Centralise OpenAI call configuration — ⚡ Medium Impact
Both completions calls in runExtractPaper and any other handlers that call openai share no common defaults for model, temperature, or token limits. Introduce a thin wrapper (e.g. callOpenAI(prompt, options)) that reads model and limits from env vars and enforces a timeout via AbortSignal, so all handlers benefit without per-call boilerplate.

### Add idempotency key logging for job retries — ⚡ Medium Impact
The extract-paper handler correctly deletes prior questions on retry, but there is no structured log entry recording that a retry-wipe occurred. Adding a single log line (jobId, paperId, attempt number, rows deleted) would make debugging stuck or repeatedly-failing papers significantly faster in production without adding overhead.

### Stream CSV export directly to object storage — 🔥 High Impact
The admin-csv-export handler (visible in the truncated section) likely builds the full CSV string in memory before uploading. For large user bases this risks OOM in the job worker. Replace with a streaming CSV serialiser (e.g. csv-stringify in streaming mode) piped directly into the object storage upload stream, capping peak memory to a small buffer window regardless of row count.

### Enforce ticker uniqueness on simulatedStocks — 🔥 High Impact
simulatedStocks.ticker has no unique constraint. Duplicate tickers would silently corrupt portfolio calculations that join on ticker symbol. Adding uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) prevents duplicates at the DB level and enables O(1) lookups by ticker from the trading UI.

### Replace repeated currency text columns with a shared enum or DB enum type — ⚡ Medium Impact
Eight tables each define currency: text('currency').default('BSD') independently. A Drizzle pgEnum('currency_enum', ['BSD', 'USD', 'TTD']) defined once and reused eliminates inconsistent defaults, reduces storage slightly, and makes adding a new supported currency a single-point change rather than a migration across eight tables.

### Add a partial unique index on portfolioHoldings (userId, stockId) — 🔥 High Impact
Without a unique constraint, concurrent buy operations can insert duplicate holding rows for the same user+stock, requiring application-level deduplication logic that is easy to forget. A uniqueIndex on (userId, stockId) moves this invariant to the database, simplifies upsert logic to ON CONFLICT DO UPDATE, and eliminates a class of race-condition bugs.

### Memoize filtered/sorted admin table data — ⚡ Medium Impact
Table rows that are filtered by search input and sorted by column appear to recompute on every render. Wrapping these derivations in useMemo(()=> [...users].filter(...).sort(...), [users, searchTerm, sortCol]) would eliminate redundant O(n log n) work on every keystroke in the search box, which matters when admin views have hundreds of student records.

### Add explicit loading and error boundary states to all data-dependent panels — 🔥 High Impact
Several chart and table panels render with no visual loading skeleton and no error state — if a query fails the panel silently shows nothing. Adding `if (isLoading) return <Skeleton />;` and `if (isError) return <ErrorCard onRetry={refetch} />;` guards per panel improves perceived reliability for school admins on intermittent Caribbean LTE connections and makes failures actionable.

### Extract AdminHelpChat into its own module file — ⚡ Medium Impact
AdminHelpChat is a self-contained component with its own state, async logic, and JSX tree embedded inside AdminDashboard.tsx. Moving it to `components/AdminHelpChat.tsx` reduces the main file size, enables independent unit testing of the chat flow, and allows the component to be code-split with React.lazy(), cutting the initial JS bundle delivered to admin users.
