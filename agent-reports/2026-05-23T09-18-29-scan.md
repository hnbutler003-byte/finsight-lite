## 🔴 Critical (10)

### `server/routes.ts` — app.get("/public-objects/:filePath(*)")
**Issue:** Path traversal vulnerability: req.params.filePath is passed directly to objectStorage.searchPublicObject without sanitization. An attacker could request paths like '../../private/secrets' to potentially access unintended stored objects.

**Fix:** Normalize and validate the path before use: `const filePath = req.params.filePath; if (filePath.includes('..') || path.isAbsolute(filePath)) return res.status(400).json({ error: 'Invalid path' }); const safe = path.normalize(filePath).replace(/^\//, '');` then pass `safe` to objectStorage.

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied to AI and image/audio generation routes before they are registered. These endpoints invoke costly external APIs (OpenAI) and have no visible per-IP or per-user throttle in this file, making them trivially abusable for cost amplification attacks.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter / job.payload)
**Issue:** Full file contents are stored as a base64 string inside the job payload row. A single PDF or image can easily be 5–20 MB encoded, meaning every retry reads and writes that blob to the jobs table, bloating the DB and spiking memory on the worker node each attempt.

**Fix:** Upload the file to private object storage before enqueuing, store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), then let runExtractPaper download it with objectStorageClient at execution time. Add a cleanup step on terminal failure or success to delete the temp object.

---

### `server/jobHandlers.ts` — runExtractPaper — for loop over questions with sequential awaits
**Issue:** Each extracted question is inserted one at a time with await storage.createExtractedQuestion(...) inside a for loop. For a 50-question paper this is 50 sequential round-trips to the DB, easily taking 5–15 seconds and holding the worker busy unnecessarily.

**Fix:** Batch-insert all valid questions in a single DB call: filter the questions array first, then call db.insert(extractedQuestions).values(validRows) once. If storage.createExtractedQuestion must be used, collect promises and await Promise.all(promises) — but a single bulk insert is strongly preferred for atomicity and speed.

### `server/jobHandlers.ts` — admin-csv-export handler (type AdminRow / dat variable)
**Issue:** The truncated code shows a pattern where all rows for a CSV export are fetched into memory at once (let dat = await db.select()...). For large tables (transactions, users) this can exhaust Node heap. There is also no visible try/catch around the export handler, so an OOM or DB error will leave the job permanently stuck in 'processing' with no status update.

**Fix:** Wrap the entire handler body in try/catch and update job or export record status to 'failed' on error. For large tables, stream results using a cursor or paginate with LIMIT/OFFSET, writing chunks to the object storage upload stream incrementally rather than buffering the full dataset.

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique constraint. Nothing prevents two rows with ticker 'BAHA' being inserted, which will silently corrupt portfolio P&L calculations that join on ticker.

**Fix:** Change the column definition to: ticker: text('ticker').notNull().unique() and add a uniqueIndex in the table callback: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker).

### `server/index.ts` — maybeRunWeeklyDigest and maybeRunAiUsagePurge setInterval calls
**Issue:** Interval handles returned by setInterval are never stored and therefore never cleared. On SIGTERM/SIGINT the event loop stays alive because these intervals keep firing, preventing graceful shutdown and causing duplicate job enqueues during rolling restarts.

**Fix:** Store each handle: `const digestInterval = setInterval(maybeRunWeeklyDigest, ...)`. In the shutdown handler call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` before `httpServer.close()`. Gate the intervals behind the same `isShuttingDown` flag used elsewhere.

---

### `server/index.ts` — res.json monkey-patch in the request logging middleware
**Issue:** Every API response body is fully JSON-serialized and appended to the log line with no truncation and no field redaction. For a financial literacy app with school-aged users this will log quiz answers, progress data, session tokens returned in response bodies, and any PII — a compliance and privacy violation.

**Fix:** Cap the serialized snippet: `logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 200)`. Additionally, strip known sensitive keys before logging: drop fields named `token`, `password`, `secret`, `sessionId` from `capturedJsonResponse` before serializing.

---

### `server/index.ts` — startup await sequence (seedDatabase, probeDatabase, registerRoutes, etc.)
**Issue:** The startup sequence calls `await probeDatabase()` and `await seedDatabase()` sequentially before the HTTP server begins listening. If either operation hangs (e.g. cold DB connection), the process blocks indefinitely with no timeout, making the service appear down to the load balancer with no diagnostic output.

**Fix:** Wrap the entire startup block in `Promise.race([startupLogic(), new Promise((_, r) => setTimeout(() => r(new Error('Startup timeout after 30 s')), 30_000))])`. Log and `process.exit(1)` on timeout. Also add a startup-progress log after each major step so the hang point is identifiable.

---

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth hook / top of component
**Issue:** If useAdminAuth performs an async session check but the component renders its full query tree (all useQuery calls across every tab) before the auth check resolves, all admin API requests fire unauthenticated. Any backend route that relies solely on client-side auth gating is also a security gap.

**Fix:** Gate all tab-level useQuery calls with `enabled: isAuthenticated` (the boolean returned by useAdminAuth). At the top of AdminDashboard, early-return a loading spinner until auth is confirmed: `if (authLoading) return <Loader2 className='animate-spin'/>;` and `if (!isAdmin) { setLocation('/'); return null; }` before any query hooks execute.

## 🟡 Warnings (14)

- **`server/routes.ts` (initSupabaseTables().then(...).catch(...)):** The sequential .then() chain means each seed step blocks the next, and if any intermediate step throws synchronously before returning a promise (e.g. a missing import), the .catch() at the end will not capture it, leaving an unhandled rejection. Additionally, independent seed steps (seedStaticModules, seedRegionalContent, seedGameContent) run sequentially when they could be parallelized.
  → *Wrap the entire chain in an async IIFE and use Promise.all for independent steps: `(async () => { try { await initSupabaseTables(); await trimOrgNamesInSupabase(); await seedFinancialAcademyLesson(); await initStaticContentTables(); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]); } catch (e) { console.error('[Supabase] Init error:', e); } })();`*
- **`server/routes.ts` (app.get("/healthz")):** The health check dynamically imports 'db' and 'drizzle-orm' on every request using await import() inside the handler. These are not cached by the module system across calls in all bundler configurations, and at minimum adds resolution overhead on every health probe, which in Kubernetes-style environments can be called every 10 seconds.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The health check handler then references the already-loaded module singletons.*
- **`server/routes.ts` (const host = req.headers["x-forwarded-host"] || req.headers.host || "localhost"):** The sitemap.xml handler trusts the X-Forwarded-Host and Host request headers without validation to construct absolute URLs. A malicious request with a spoofed Host header will produce a sitemap containing attacker-controlled URLs, which can be used for cache poisoning or SEO manipulation.
  → *Read the canonical base URL from an environment variable set at deploy time: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and remove the dynamic header-based construction entirely.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete before re-extraction):** The retry-safe delete (db.delete(extractedQuestions).where(eq(...paperId))) runs unconditionally on every attempt, including the very first one. This is correct but also silently deletes questions if a job is manually re-triggered after a paper was already successfully processed and questions were reviewed, with no guard on paper status.
  → *Before deleting, fetch the current exam paper status. If status === 'completed', short-circuit and return the existing result immediately (true idempotency). Only proceed with deletion and re-extraction when status is 'processing' or 'failed'.*
- **`server/jobHandlers.ts` (openai import — `import { openai } from './replit_integrations/chat/routes'`):** The OpenAI client is imported from a routes module, coupling job execution infrastructure to an HTTP layer module. This creates a hidden dependency that makes the routes module initialize on worker startup, potentially registering Express middleware or side-effects in a context where no HTTP server is running.
  → *Extract the OpenAI client instantiation into a dedicated singleton module (e.g. server/openaiClient.ts) that reads OPENAI_API_KEY from environment config directly. Both the routes file and jobHandlers should import from that shared module.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on (userId, categoryId) despite being the primary query pattern for budget lookups. Full table scans will occur as user count grows.
  → *Add an index in the table's second argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also consider a uniqueIndex on (userId, categoryId, period) to prevent duplicate budget rows per period.*
- **`shared/schema.ts` (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders — createdAt columns):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). Drizzle will type this as Date | null, meaning application code must null-check everywhere and any INSERT that explicitly passes null will succeed, corrupting audit trails.
  → *Append .notNull() to every createdAt definition, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Same applies to portfolioHoldings.purchasedAt and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (billReminders table — nextDueDate / portfolioHoldings — purchasedAt indexes):** billReminders queries will almost always filter by userId + nextDueDate to surface upcoming bills, and portfolioHoldings will be queried by userId + stockId. Neither has an index, so both become sequential scans under load.
  → *Add to billReminders callback: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). Add to portfolioHoldings callback: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) — unique is correct here since a holding is one row per user+stock that gets quantity-updated on buy/sell.*
- **`shared/schema.ts` (categories table — userId / isDefault columns):** categories.userId uses varchar without a length, and there is no index on (userId) for filtering user-owned vs system categories. Every category dropdown render that filters WHERE user_id = $1 OR user_id IS NULL will scan the full table.
  → *Match the length used in the auth users table, e.g. varchar('user_id', { length: 255 }), and add an index: index('idx_categories_user').on(t.userId) in the table callback. Also add isDefault: boolean('is_default').default(false).notNull() to prevent null boolean comparisons.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince` as `(dow + 0) % 7` — the `+ 0` is dead/placeholder code that does nothing, and the logic rolls back to Sunday using only the day-of-week offset without accounting for the timezone of the Caribbean target audience. `new Date()` uses server UTC, so 7 PM ET/AST will fire at the wrong wall-clock time for users.
  → *Remove `+ 0`. Introduce a `TZ` env var (e.g. `America/Port_of_Spain`) and compute the target using `Intl.DateTimeFormat` with that timezone, or use a library like `date-fns-tz`. Add `DIGEST_TIMEZONE` to environment config and document it.*
- **`server/index.ts` (maybeRunWeeklyDigest — audience loop):** The three audience digest jobs (`student`, `teacher`, `guardian`) are enqueued sequentially with `await` inside a `for...of` loop. Each iteration hits the database twice (insert + optional delete). Under a slow DB this adds unnecessary latency to the scheduler tick and holds the async context longer than needed.
  → *Replace the `for...of` loop with `await Promise.all(['student','teacher','guardian'].map(audience => claimAndEnqueue(audience)))` where `claimAndEnqueue` contains the existing insert/enqueue/rollback logic. Ensure each iteration has its own try/catch so one failure does not skip the remaining audiences.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel / queryFn):** The queryFn arrow function inside useQuery is reconstructed on every render because `params` (a new URLSearchParams instance) is built inline outside of useMemo. This also means the queryKey array diverges from what the URL actually sends, making cache hits unreliable.
  → *Move params construction into a useMemo keyed on [actorType, action, from, to], then use the memoized string in both the queryKey and the queryFn: `const paramStr = useMemo(() => { const p = new URLSearchParams(); ... return p.toString(); }, [actorType, action, from, to]);` and set `queryKey: ['/api/admin/audit-log', paramStr]`.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard / queryFn):** The queryFn has no error handling. If the fetch rejects or the server returns a non-OK status, r.json() may throw an unhandled promise rejection that TanStack Query will swallow silently, leaving the component stuck with no visible error state rendered.
  → *Wrap the queryFn: `async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error(`Observability fetch failed: ${r.status}`); return r.json(); }` and destructure `isError, error` from useQuery to render a fallback `<Badge className='bg-red-600'>Unavailable</Badge>` inside the card.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant / tab panel rendering):** Each tab panel likely issues its own independent useQuery calls sequentially on mount. For the overview/dashboard tab in particular, multiple stat queries (organizations, schools, teachers, students, sponsors, challenges) firing one-by-one cause a waterfall that delays time-to-interactive by the sum of all round trips.
  → *Consolidate dashboard aggregate stats into a single `/api/admin/stats` endpoint returning all counts in one response, or use `Promise.all` inside a single aggregated queryFn: `queryFn: () => Promise.all([apiRequest(...).then(r=>r.json()), ...])`. This converts an O(n) waterfall into a single parallel round trip.*

## 💡 Opportunities

### Parallel domain route registration — ⚡ Medium Impact
The six registerX(app) calls after the Promise.all import are awaited sequentially. If each does async DB work during registration (e.g. building route-level caches), running them in parallel with Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) could meaningfully reduce cold-start latency. Only safe if the routers do not depend on each other's side effects at registration time.

### Structured logging instead of console.error — ⚡ Medium Impact
All error paths use console.error with raw Error objects. Replacing with a structured logger (e.g. pino) that emits JSON with severity, route, and traceId fields would make errors correlatable in Sentry and any log aggregator without code changes in each catch block.

### Extract object-storage serving into dedicated middleware — 📌 Low Impact
The /public-objects handler duplicates concerns (path lookup, streaming, cache-header setting) that will be needed if more storage buckets are added. Extracting an objectStorageMiddleware factory that accepts a bucket prefix and cache TTL would reduce duplication and make the path-traversal fix a single point of maintenance.

### Parallel OpenAI calls where extraction has two sequential steps — ⚡ Medium Impact
In runExtractPaper the vision/PDF text extraction and the subsequent GPT-4o question-extraction call are inherently sequential, but the paper status update (storage.updateExamPaper) and bulk question insert at the end are also sequential. Awaiting both concurrently with Promise.all([bulkInsert, updatePaper]) would cut the final write phase latency roughly in half with no correctness risk.

### Environment-config guard at startup rather than at call time — ⚡ Medium Impact
PRIVATE_OBJECT_DIR is validated only when parsePrivatePath is first called at runtime inside a job. A missing env var will cause jobs to fail silently after enqueueing. Add a startup assertion in registerJobHandlers (or a dedicated validateEnv() call at server boot) that checks all required vars — PRIVATE_OBJECT_DIR, OPENAI_API_KEY, etc. — and throws immediately on startup so misconfiguration is caught in deployment, not in production job execution.

### Structured logging with redaction for job payloads — 🔥 High Impact
Job payloads currently risk leaking base64 file contents, user PII, or API responses into logs if any logging middleware serialises the job object. Introduce a sanitizePayload utility that replaces keys matching /b64|buffer|content|token/i with '[REDACTED]' before any log statement, and apply it consistently across all handler entry points. This is especially important for a school-aged-user platform under potential COPPA/FERPA-adjacent obligations.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract to a module-level constant DEFAULT_CURRENCY = 'BSD' imported from a config file so a currency change or multi-currency expansion requires one edit, not six schema migrations.

### Add a composite unique constraint to prevent duplicate budget periods — 🔥 High Impact
Without a unique constraint on (userId, categoryId, period), the application layer is solely responsible for preventing a user from having two monthly budgets for Groceries. A DB-level uniqueIndex('idx_budgets_unique_period').on(t.userId, t.categoryId, t.period) makes this invariant enforceable and removes the need for defensive SELECT-before-INSERT logic in the service layer.

### Replace serial primary keys with uuid on user-facing tables — 🔥 High Impact
Serial integer IDs on transactions, savingsGoals, and portfolioTransactions expose enumerable resource IDs in API URLs (e.g. /transactions/1, /transactions/2), enabling horizontal enumeration attacks — a real risk for a student finance app. Switching to uuid('id').primaryKey().defaultRandom() removes ordinality leakage and aligns with the varchar user_id already used from the auth model.

### Centralised graceful-shutdown registry — 🔥 High Impact
Instead of ad-hoc shutdown logic scattered across the file, introduce a `ShutdownRegistry` that collects cleanup callbacks (clearInterval, httpServer.close, db pool drain, job worker stop) and runs them in order on SIGTERM/SIGINT. This makes it impossible to accidentally add a new resource without also registering its teardown, and gives a single place to enforce a shutdown timeout deadline.

### Environment config validation at startup with a schema — ⚡ Medium Impact
Currently env vars are checked ad-hoc (SESSION_SECRET) or silently defaulted (AI_PURGE_OLDER_THAN_DAYS). Consolidate all env var reads into a single `validateEnv()` call using `zod` at the top of `index.ts`, failing fast with a structured error listing every missing or invalid variable. This prevents partial misconfiguration from reaching production silently.

### Structured JSON logging instead of console.log string concatenation — ⚡ Medium Impact
The `log()` helper concatenates strings and writes to stdout. Replacing it with a structured logger (e.g. `pino`) emitting JSON lines enables log aggregation, severity filtering, and automatic field redaction rules — directly addressing the PII-in-logs risk and making log search in production viable without grep heuristics.

### Split AdminDashboard into per-tab route chunks — 🔥 High Impact
At 126k+ characters this single file bundles every tab's logic, queries, and UI together. Splitting each tab (AuditLogPanel, DBViewer, PerfAgent, etc.) into its own lazily-loaded route component via React.lazy/Suspense would reduce the initial JS parse cost by ~70-80% for users who only visit the overview tab, and make each panel independently testable.

### Add staleTime and refetchInterval config to admin queries — ⚡ Medium Impact
Without explicit staleTime, TanStack Query refetches every admin query on every window focus event. For slow aggregations (audit log, observability) this creates unnecessary server load from admin browser tab switches. Setting `staleTime: 30_000` and `refetchOnWindowFocus: false` on these queries would cut redundant requests by an estimated 60% during normal admin sessions.

### Replace inline Button onClick handlers with useCallback — ⚡ Medium Impact
Large inline arrow functions passed to onClick props (e.g., the Refresh button in AuditLogPanel, delete/edit handlers in table rows) are recreated every render, invalidating React.memo on child Button components. Wrapping these in useCallback would stabilize references and prevent cascading re-renders across all table rows when parent filter state changes.

### API Routes (`server/routes.ts`)
The route registration code is generally well-structured with good use of Promise.all for parallel imports and proper try/catch on the health check. The main concerns are an unhandled-rejection risk in the Supabase init chain, missing rate limiting on AI/heavy endpoints registered via registerChatRoutes/registerImageRoutes/registerAudioRoutes, and a path traversal vulnerability in the public object storage endpoint.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads stored directly in the job table create memory and DB pressure, sequential question inserts in a loop are extremely slow, and the file truncation hides unknown handler registrations that may compound these problems. Security and idempotency patterns are partially correct but incomplete.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns include missing indexes on high-cardinality filter columns, unguarded nullable timestamps that default to NULL instead of NOW(), and a missing unique constraint on ticker that could cause duplicate simulated stock entries.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability gaps: blocking synchronous-style sequential awaits that could run in parallel, interval handles that are never cleared on shutdown preventing clean process exit, and response JSON bodies fully serialized into logs which risks leaking sensitive financial or PII data for every API response. The overall structure is reasonable but these issues compound under load or during deployments.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx shows several reliability and performance concerns: async operations lack try/catch, multiple independent queries fire sequentially when they could be parallelized, and inline functions in JSX cause unnecessary re-renders on every parent update. The file is extremely large (126k+ chars) suggesting it violates single-responsibility and will be costly to maintain and test.
