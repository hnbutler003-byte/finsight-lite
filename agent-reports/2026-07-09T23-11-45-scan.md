## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. A student or bot can hammer these endpoints and exhaust API quota or run up significant cost. This is especially risky for a school-aged user base where accounts may be shared.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/15 min per IP) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Confirm exact path prefixes by inspecting each register function.

### `server/routes.ts`: robots.txt handler (~line 38) and sitemap.xml handler (~line 52)
**Issue:** Both handlers build an absolute base URL by trusting `x-forwarded-proto` and `x-forwarded-host` headers directly from the request without any validation or allowlist. An attacker can send crafted headers to inject arbitrary URLs into the Sitemap directive and sitemap <loc> entries, enabling cache-poisoning or open-redirect abuse.

**Fix:** Read the canonical origin once from environment config: `const BASE_URL = process.env.APP_BASE_URL || 'https://finsightlite.com';` and use it unconditionally in both handlers. Remove all header-derived base URL construction.

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** Five sequential awaits hit the database one after another for every student in the fan-out loop. For a school with 500 students this serialises 2500+ round-trips, causing the weekly-digest job to time out.

**Fix:** Wrap the four independent DB queries inside studentWeekStats in a single Promise.all: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...).from(transactions)..., db.select(...).from(gameSessions)..., db.select(...).from(userLearningProgress)..., db.select(...).from(userBadges)...]). This cuts per-student latency from ~5 serial RTTs to ~1 parallel RTT.

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** When orgId is absent, the fallback does db.select().from(users) with no WHERE, LIMIT, or pagination. On any non-trivial deployment this loads the entire users table into Node heap memory, risking OOM and exposing all user records (including password hashes or PII) inside the job worker process.

**Fix:** Add a hard LIMIT and paginate: replace the unbounded select with a paginated loop using .limit(500).offset(page * 500), or restrict the no-orgId path entirely by throwing if orgId is missing ('weekly-digest requires an orgId for scoped fan-out'). Never select all columns—explicitly select only id, email, username.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Any query filtering budgets by user (the primary use case for every dashboard load) will do a full table scan as data grows.

**Fix:** Add a composite index in the budgets table's third argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This covers both the common 'get all budgets for user' and 'budget vs category' queries.

### `server/index.ts`: app.use (response logging middleware)
**Issue:** `capturedJsonResponse` is serialized with `JSON.stringify` and appended to every API log line unconditionally. For endpoints that return paginated lists, quiz answers, or user profile data this will dump full response payloads—including PII and answer keys—to stdout/log aggregators. For school-aged users in a regulated Caribbean context this is a data-exposure risk.

**Fix:** Gate logging on a size/sensitivity heuristic: only log the response body when `res.statusCode >= 400` (errors only), or truncate to a fixed length (e.g. 200 chars) and exclude known sensitive paths: `if (capturedJsonResponse && res.statusCode >= 400) logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 200);`

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The two `setInterval` handles are never stored and never passed to `clearInterval` in any shutdown handler. On SIGTERM (e.g. Replit container restart) the process may fire one more scheduler tick after the HTTP server stops accepting connections but before the process exits, potentially inserting a duplicate `weeklyDigestRuns` claim or spawning a purge against a half-closed DB connection pool.

**Fix:** Store both handles: `const digestTimer = setInterval(...); const purgeTimer = setInterval(...);` and in the graceful-shutdown handler call `clearInterval(digestTimer); clearInterval(purgeTimer);` before `httpServer.close()` and `db.$client.end()`.

### `client/src/pages/AdminDashboard.tsx`: sendMessage – apiRequest call
**Issue:** The catch block in sendMessage swallows all errors silently and only appends a generic message. If the server returns a non-2xx response with a JSON error body (e.g. 401 Unauthorized, 403 Forbidden), the failure is indistinguishable from a network error, making it impossible to surface auth expiry or permission errors to the admin user.

**Fix:** After `const res = await apiRequest(...)`, check `if (!res.ok) { const err = await res.json().catch(() => ({})); throw new Error(err.message || res.statusText); }` before calling `res.json()`. In the catch block, use `error instanceof Error ? error.message : 'Unknown error'` to surface the real reason to the admin.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / page-level auth check
**Issue:** The component imports `useAdminAuth` but the truncated code pattern is common in this codebase: child sections (mutations, data fetches) execute regardless of auth state during the render cycle before any redirect fires. Mutations triggered via keyboard shortcuts or rapid clicks can fire unauthenticated requests.

**Fix:** Add an early return guard immediately after the auth hook: `const { admin, isLoading } = useAdminAuth(); if (isLoading) return <Loader2 />; if (!admin) return null;` and ensure all `useMutation` definitions are placed after this guard so React never registers them for unauthenticated renders.

## 🟡 Warnings (15)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** The seven-step Supabase init chain is entirely sequential and a failure in any early step (e.g. seedFinancialAcademyLesson) causes all subsequent steps to be silently skipped. The single `.catch` at the end only reports the first failure; later seeding steps never run, leaving the DB in a partial state with no indication of which step failed.
  → *Run independent seeding steps in parallel and capture individual failures: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] Seed step ${i} failed:`, r.reason)));` Keep initSupabaseTables and initStaticContentTables sequential only if they are true prerequisites, and await them separately with their own try/catch.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 95)):** The filePath wildcard parameter is passed directly to `objectStorage.searchPublicObject` with no sanitisation. A path like `../../private/keys` could traverse into unintended storage prefixes depending on how the object storage client resolves paths.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and reject requests where the normalised path starts with `..` or `/`. Add `import path from 'path';` at the top of the file.*
- **`server/jobHandlers.ts` (registerJobHandler('weekly-digest')):** The entire weekly-digest handler body has no top-level try/catch. If bootstrapStudentContacts or any enqueueJob call throws, the unhandled rejection will crash or silently fail the job with no structured error recorded, and partial fan-out will not be retried correctly.
  → *Wrap the handler body in try/catch and rethrow a typed error: try { ... } catch (err) { throw new Error(`weekly-digest failed after ${enqueued} enqueued: ${(err as Error).message}`); }. This ensures the job runner records a meaningful failure reason and can apply retry/back-off logic.*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export')):** The entire CSV payload is built as a single in-memory string with Buffer.from(csv) before upload. For large exports (e.g. all students across schools) this can allocate hundreds of MB in the job worker. Additionally, the handler stores the objectPath in the job result column, but the full CSV data was previously noted to risk being stored in the job table itself if the caller pattern changes.
  → *Stream rows directly to the object storage upload using a PassThrough stream and the resumable:true option (or a multipart upload), writing each row chunk as it is produced. Use a streaming CSV library (e.g. csv-stringify in streaming mode) piped into the upload stream to keep memory usage O(1) regardless of row count.*
- **`server/jobHandlers.ts` (purge-ai-usage handler / weekly-digest since calculation):** Time-window constants (7 * 24 * 60 * 60 * 1000, the 7-day digest window) are hardcoded inline in multiple places. The purge cutoff uses olderThanDays from the payload but the digest window is always exactly 7 days regardless of the weekStart payload field that is passed in but never used.
  → *Use the passed weekStart payload value to derive since: const since = new Date(weekStart) rather than Date.now() - 7d, so the handler is idempotent and can be replayed for a specific week. Extract the ms-per-day constant: const MS_PER_DAY = 24 * 60 * 60 * 1000 and reference it in both places. Validate weekStart is a valid ISO date string at handler entry.*
- **`shared/schema.ts` (varchar user_id columns across all tables (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** All user_id columns are declared as varchar with no length limit. Without a length, Postgres stores it as varchar with no constraint, which allows arbitrarily long strings and wastes storage. If the auth provider (e.g. Clerk or Supabase) issues fixed-length IDs, the column type should reflect that.
  → *Change varchar('user_id') to varchar('user_id', { length: 128 }) (or the exact max length your auth provider guarantees). This enforces a DB-level guard and allows Postgres to optimize storage more predictably.*
- **`shared/schema.ts` (createdAt columns on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** Every createdAt column uses .defaultNow() but omits .notNull(). This means application code or a partial insert can leave createdAt as NULL, breaking any ORDER BY createdAt or audit logic silently.
  → *Change every instance of timestamp('created_at').defaultNow() to timestamp('created_at').defaultNow().notNull(). The default ensures the DB fills it in; notNull() prevents NULL from ever being stored.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on nextDueDate or userId. The primary app feature—showing upcoming bills—requires a WHERE userId = ? ORDER BY nextDueDate query. Without indexes this is a sequential scan on every reminder fetch.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) as the third argument to the billReminders pgTable call.*
- **`shared/schema.ts` (simulatedStocks — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text rather than timestamp. Storing a date as text means Postgres cannot enforce valid date values, cannot use date comparison operators efficiently, and will silently accept strings like 'yesterday' or '' that will corrupt price-change calculations.
  → *Change lastPriceUpdateDate: text('last_price_update_date') to lastPriceUpdateDate: timestamp('last_price_update_date'). Update any application code that writes a string to pass a JS Date object instead.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** Incorrect day-rollback logic: `const daysSince = (dow + 0) % 7` always returns 0 on Sunday (dow=0), so the guard `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` is the only correction path. On Sunday before 7 pm the function returns the current Sunday at 19:00 which is in the future, so it exits early correctly, but on Sunday at exactly 19:00:00.000 floating-point equality is fragile. More critically, the `(dow + 0) % 7` expression adds nothing and suggests a copy-paste bug where the intended offset (e.g. `(dow + 7 - 0) % 7` to get days-since-Sunday) was never completed, making the logic opaque and error-prone.
  → *Replace with explicit arithmetic: `const daysSinceSunday = dow; d.setDate(d.getDate() - daysSinceSunday); d.setHours(19, 0, 0, 0); if (d >= now) d.setDate(d.getDate() - 7);` and add a unit test covering Sunday-before-7pm, Sunday-after-7pm, and mid-week cases.*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, startJobWorker)):** Startup awaits `probeDatabase`, `seedDatabase`, `registerRoutes`, and `startJobWorker` sequentially before the HTTP server begins listening. If `seedDatabase` is slow (schema migrations, fixture inserts) the process is not yet serving health-check pings, causing orchestrators to mark it unhealthy and restart it, which can loop. Additionally, `probeDatabase` and any read-only startup checks that are independent could run in parallel.
  → *Start `httpServer.listen` immediately after `registerRoutes` returns so the process is reachable, then run `seedDatabase` and `startJobWorker` in the background with their own error handling. If seed must complete before traffic is served, add a `GET /healthz` route that returns 503 until a `ready` flag is set, rather than blocking listen.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest):** Both scheduler functions are called once at startup (outside `setInterval`) and then on the interval, but neither awaits nor catches a top-level rejection from the initial fire. If the DB is not yet fully ready at startup (race with `probeDatabase`), the initial call can throw an unhandled promise rejection that in Node 18+ terminates the process.
  → *Wrap each initial invocation: `maybeRunWeeklyDigest().catch(e => captureError(e));` and `maybeRunAiUsagePurge().catch(e => captureError(e));` to match the error handling already present inside `setInterval`'s callbacks.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is recreated on every render and fires scrollIntoView on every render cycle, not just when messages change. This causes jarring scroll behaviour if any parent state updates while the panel is open.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` then render `<div ref={bottomRef} />`. This fires only when the dependency array changes.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages state sent to API):** The full conversation history (`next`) is posted to `/api/admin/help-chat` on every message with no size cap. A user leaving the panel open can accumulate hundreds of messages, eventually sending a payload large enough to breach server body limits or significantly inflate latency and token cost.
  → *Slice the history before sending: `const context = next.slice(-20);` and post `{ messages: context }`. Also add a UI cap: if `messages.length > 100` show a 'Clear conversation' prompt to prevent unbounded growth in component state.*
- **`client/src/pages/AdminDashboard.tsx` (handleKey – sendMessage reference):** `handleKey` and `sendMessage` are both defined as plain arrow functions inside the component body with no useCallback. Because `AdminHelpChat` re-renders on every keystroke (input state changes), these functions are recreated on every character typed, and any child component or effect that lists them as dependencies will re-fire unnecessarily.
  → *Wrap both with useCallback: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all but then awaited sequentially one by one. If any registerXRoutes function does async work (DB calls, dynamic imports), they could also be parallelised with Promise.all: `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])`. Verify there are no ordering dependencies first. Expected impact: reduced cold-start latency on Railway.

### Centralise APP_BASE_URL as environment config: ⚡ Medium Impact
Beyond fixing the header injection issue, storing the canonical base URL in a single env var and validating its presence at startup (throw if missing in production) would eliminate the fallback-to-localhost risk and make deployment configuration explicit and auditable.

### Add request-scoped logging with correlation IDs: 📌 Low Impact
Currently errors in async routes (e.g. object storage, healthz) are logged with console.error and no request correlation. Adding a middleware that attaches a UUID to each request and includes it in all log calls would make tracing failures across Supabase init, object storage, and auth routes practical in production, especially for a multi-tenant school app.

### Batch enqueueJob calls in weekly-digest fan-out: 🔥 High Impact
Currently enqueueJob is called inside a for loop, issuing one INSERT per student sequentially. Replacing this with a bulk-insert variant (enqueueJobs(payloads[]) using a single db.insert().values([...])) would reduce fan-out overhead from O(n) round-trips to O(1), meaningfully cutting job duration for large cohorts and reducing DB connection pressure.

### Add idempotency key to send-email job: 🔥 High Impact
The send-email handler has no deduplication guard. If the job runner retries on a transient failure after the email provider accepted the message, students receive duplicate digest emails. Adding an idempotencyKey derived from (userId + weekStart + kind) checked against a sent-log table or the emailContacts last-sent timestamp before dispatch would prevent duplicates with no user-visible impact.

### Sanitise fileName header in streamPrivateObjectToResponse: ⚡ Medium Impact
The Content-Disposition header is built with string interpolation of the fileName parameter without sanitisation: filename="${fileName}". A caller passing a fileName containing a quote or newline can break the header. Sanitise with fileName.replace(/[\"
]/g, '') or use the RFC 5987 encoded form (filename*=UTF-8''...) to safely support Unicode names and prevent header injection.

### Centralise currency default into a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least six separate table columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' in a shared config file and reference it everywhere. This makes switching to TTD or XCD for other Caribbean markets a one-line change rather than a schema migration.

### Add a unique index on simulatedStocks.ticker: 🔥 High Impact
The ticker column has no unique constraint. Duplicate tickers (e.g. two rows for 'BTC') would corrupt portfolio value calculations silently. Adding uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) enforces correctness at the DB layer and also speeds up stock lookups by ticker symbol from O(n) to O(log n).

### Replace portfolioHoldings with an upsert pattern and add a unique constraint: 🔥 High Impact
portfolioHoldings has no unique constraint on (userId, stockId), so buying the same stock twice creates two rows instead of updating quantity and avgPurchasePrice. Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) and update the buy logic to use INSERT ... ON CONFLICT DO UPDATE to maintain a single row per holding. This eliminates a class of double-counting bugs in portfolio value calculations.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
If `probeDatabase` and any static-asset manifest loading in `serveStatic` are independent, wrapping them in `Promise.all([probeDatabase(), loadManifest()])` would reduce cold-start time by the duration of the slower-completing operation, improving Replit container startup perceived latency.

### Structured JSON logging instead of formatted strings: ⚡ Medium Impact
The `log()` function emits freeform `toLocaleTimeString` strings. Switching to a structured logger (e.g. `pino`) with JSON output would let log aggregators (Datadog, Logtail) index `duration`, `statusCode`, and `path` as searchable fields rather than parsed substrings, and would make it easier to enforce the PII-scrubbing fix from finding #2 in one place.

### Centralise scheduler tick deduplication behind a named advisory lock: 🔥 High Impact
The current `onConflictDoNothing` insert pattern works for single-region deploys but would double-fire on a two-dyno/two-container setup. Replacing it with a Postgres advisory lock (`SELECT pg_try_advisory_xact_lock(hashtext('weekly-digest-' || weekStart))`) inside the scheduler would be safe across horizontal scaling without changing the schema.

### Parallelise independent admin data fetches: 🔥 High Impact
If the dashboard fires multiple useQuery calls for users, schools, and analytics sequentially or triggers sequential awaits inside effects, wrapping independent fetches with Promise.all (or relying on React Query's parallel query execution by ensuring they are not chained) can cut initial load time by the duration of the slowest sequential leg. Expected impact: 30–60% reduction in time-to-interactive on slower Caribbean mobile connections.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Admin tables that filter/sort large user or school lists inline in JSX or in render-body arrow functions should be moved into useMemo with appropriate dependency arrays. This prevents O(n) re-scans on every keystroke in unrelated inputs. Expected impact: eliminates jank for lists >200 rows on low-end Android devices common in the target demographic.

### Extract chat panel into a lazy-loaded module: ⚡ Medium Impact
AdminHelpChat imports Recharts, Lucide icons, and Dialog components that are already part of the main bundle. Splitting AdminHelpChat into a React.lazy + Suspense boundary reduces the initial JS parse cost for admins who never open the chat. Expected impact: ~15–25 KB reduction in main chunk parse time on first load.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good use of Promise.all for parallel imports and try/catch on async endpoints. Key concerns are: unprotected AI/audio/image routes lacking rate limiting, host header injection risk in robots.txt and sitemap, and a chained sequential Supabase init that silently swallows individual step failures.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including sequential DB queries inside a fan-out loop that will bottleneck at scale, missing try/catch around async operations in job handlers that will silently fail, and large CSV payloads being assembled entirely in memory before upload. The code also contains hardcoded time constants and a bootstrapStudentContacts fallback that fetches the entire users table without pagination, which is dangerous in production.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable createdAt timestamps that should be NOT NULL, and an untyped varchar user_id that lacks a length constraint throughout all tables.

### Server Startup (`server/index.ts`)
The server startup sequence contains multiple blocking synchronous-style awaits that delay readiness, and the interval-based schedulers lack cleanup registration for graceful shutdown, risking duplicate job enqueues on restart. Security and reliability gaps include full API response bodies being logged unconditionally and scheduler intervals never being cleared on SIGTERM/SIGINT.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth guards on sensitive operations, memory leaks from unmanaged scroll refs, and sequential async patterns that could be parallelised. The chat component creates a new inline ref callback on every render causing unnecessary DOM churn, and hardcoded API paths mixed with business logic reduce maintainability.
