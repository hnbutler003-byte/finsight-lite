## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 115-117)
**Issue:** AI-heavy routes (chat, image, audio) are registered with no rate limiting middleware applied before them. For a school-aged user app calling OpenAI, a single user or bot can exhaust API quota and run up costs with no throttle.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls.

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap.xml handler (~line 63)
**Issue:** Both handlers build a base URL directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation. An attacker can send a crafted Host header and inject arbitrary URLs into the Sitemap directive or sitemap <loc> tags, a classic host-header injection.

**Fix:** Add an allowlist check against a `ALLOWED_HOSTS` env var or a single `APP_BASE_URL` env var: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and use that instead of deriving the base from request headers entirely.

---

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Any query filtering budgets by user (the primary access pattern) will do a full table scan as the table grows.

**Fix:** Add a composite index in the budgets table factory function: `(t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })`. Also add a standalone index on userId alone if queries filter without categoryId.

---

### `server/index.ts`: setInterval calls in maybeRunWeeklyDigest / maybeRunAiUsagePurge scheduler setup
**Issue:** Interval handles returned by setInterval for the weekly digest and AI purge schedulers are never stored, so they cannot be cleared during graceful shutdown. On SIGTERM the process will attempt to close the HTTP server but the intervals keep the event loop alive and the process either hangs or is force-killed, risking mid-flight job corruption.

**Fix:** Capture every setInterval return value: `const digestInterval = setInterval(maybeRunWeeklyDigest, ...)`. Add a shutdown handler that calls `clearInterval(digestInterval)` and any other interval handles before or alongside `httpServer.close()`. Store all handles in a module-level array and iterate it in the shutdown routine.

---

### `server/index.ts`: res.json monkey-patch in the request logging middleware (capturedJsonResponse block)
**Issue:** Every API response body is serialised with JSON.stringify and written to stdout unconditionally. For a school-aged financial-literacy app this will expose quiz scores, session tokens, user PII, and any AI-generated content in plain server logs. There is no truncation, no field redaction, and no log-level gate.

**Fix:** Either remove response-body logging entirely from this middleware, or introduce an allowlist of safe, non-sensitive fields and a hard character cap (e.g. 200 chars). At minimum gate it behind `process.env.NODE_ENV !== 'production'` so production logs are never polluted with response payloads.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat - POST /api/admin/help-chat request
**Issue:** The help-chat endpoint is called with apiRequest but the response is consumed with `res.json()` without checking `res.ok` first. A non-2xx response (e.g. 429 rate limit, 500 server error) will silently produce a JSON parse of an error body or throw an unhandled promise rejection that falls through to the generic catch message, masking the real error class.

**Fix:** After `const res = await apiRequest(...)`, add `if (!res.ok) throw new Error(`Chat API error: ${res.status}`);` before calling `res.json()`. In the catch block, inspect the error message to provide rate-limit-specific guidance ('Too many requests, please wait') versus generic failure.

## 🟡 Warnings (11)

- **`server/routes.ts` (Supabase init chain (~line 120-128)):** The seven Supabase init/seed functions are chained sequentially with `.then()`. Each waits for the previous to complete before starting, unnecessarily extending startup time. Some are independent (e.g. seedRegionalContent and seedGameContent do not depend on seedStaticModules).
  → *After the mandatory ordering dependencies (initSupabaseTables → trimOrgNamesInSupabase → seedFinancialAcademyLesson → initStaticContentTables), run the remaining seeds in parallel: `initStaticContentTables().then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]))`. Wrap the whole chain in a named async IIFE so the `.catch` is not swallowing typed errors silently.*
- **`server/routes.ts` (/healthz handler (~line 85)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every call. Dynamic imports are cached by Node after the first load, but the repeated `await import()` syntax adds unnecessary microtask overhead on every health-check tick from Railway or uptime monitors hitting this frequently.
  → *Hoist the imports to the top of the file as static imports: `import { db } from './db'; import { sql } from 'drizzle-orm';` so the health check handler body contains only the `await db.execute(sql\`select 1\`)` call.*
- **`shared/schema.ts` (varchar('user_id') on all tables (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** varchar('user_id') is declared without a length argument, which in PostgreSQL defaults to unbounded varchar (equivalent to text). This wastes type precision and bypasses implicit length validation. If the auth provider issues fixed-length IDs (e.g., Clerk 32-char IDs), unbounded varchar silently accepts corrupt or truncated values.
  → *Change all `varchar('user_id')` declarations to `varchar('user_id', { length: 36 })` (or the exact length your auth provider guarantees, e.g. 32 for Clerk). This enforces a DB-level constraint and signals intent to future developers.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as `text` instead of `timestamp`. Storing a date as text means no DB-level ordering, range queries, or timezone handling. Any ORDER BY or date comparison on this column will sort lexicographically, producing wrong results for non-ISO strings.
  → *Change `text('last_price_update_date')` to `timestamp('last_price_update_date')`. Update any application code that writes a string to pass a Date object instead.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** portfolioHoldings.stockId and portfolioTransactions.stockId have no index. Portfolio views will join or filter on stockId frequently; without an index these are full scans. Also, portfolioHoldings has no unique constraint on (userId, stockId), so duplicate holding rows for the same user+stock can silently accumulate, corrupting quantity/average-price calculations.
  → *Add `(t) => ({ userStockIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) })` to portfolioHoldings. Add `(t) => ({ stockIdx: index('idx_portfolio_tx_stock').on(t.stockId), userIdx: index('idx_portfolio_tx_user').on(t.userId) })` to portfolioTransactions.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The day-of-week arithmetic `(dow + 0) % 7` is a no-op: adding zero before modulo means Sunday (dow=0) computes daysSince=0, so the function returns today at 19:00 rather than the previous Sunday, causing the digest to fire every Sunday interval tick instead of exactly once per week window. The dead `+ 0` also signals unfinished intent.
  → *Replace `(dow + 0) % 7` with `dow === 0 ? 0 : dow` or simply `dow` to correctly compute days-since-last-Sunday. Add a unit test for a Wednesday input (dow=3, expect 3 days back) and a Sunday input (dow=0, expect 0 days back with the >now guard pushing it to -7).*
- **`server/index.ts` (startup sequence: seedDatabase, probeDatabase, registerRoutes, startJobWorker calls):** seedDatabase and probeDatabase are awaited sequentially before the server starts listening. If either takes several seconds (e.g. slow DB connection on cold start) the process is unresponsive to health checks and Replit's process monitor may restart it. Additionally startJobWorker is awaited after the HTTP server is already accepting traffic, meaning requests can arrive before the job worker is ready.
  → *Run independent startup tasks in parallel: `await Promise.all([probeDatabase(), seedDatabase()])`. Start the HTTP server listening only after all startup tasks resolve. Guard the listen call in a try/catch that calls process.exit(1) on failure so a misconfigured DB doesn't leave the process in a zombie-listening state.*
- **`server/index.ts` (maybeRunWeeklyDigest and maybeRunAiUsagePurge async functions passed to setInterval):** setInterval does not await the async callbacks; if a previous invocation is still running when the next tick fires, both run concurrently. The DB-level conflict guard protects against duplicate inserts but concurrent executions will still issue redundant DB round-trips and log noise, and an unhandled rejection inside the interval callback will crash Node 18+ or silently swallow errors on older runtimes.
  → *Wrap the interval body with an in-flight guard: `let running = false; setInterval(async () => { if (running) return; running = true; try { await maybeRunWeeklyDigest(); } finally { running = false; } }, interval)`. This is a simple mutex that prevents overlapping executions without an external lock.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - sendMessage):** The catch block in sendMessage silently swallows the error with no logging. In a financial literacy platform serving school-aged users, silent failures make debugging production incidents extremely difficult and hide potential auth/network errors.
  → *Add `console.error('AdminHelpChat sendMessage failed:', err)` inside the catch block (with err typed as unknown), and surface a more specific message when the response status indicates auth failure (401/403) so admins know to re-authenticate rather than just retry.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - messages scroll div ref):** The inline `ref={el => { if (el) el.scrollIntoView(...) }}` creates a brand-new function on every render of the message list. Every time any state changes (input typing, loading toggle), React tears down and recreates this ref callback, calling scrollIntoView spuriously on intermediate renders.
  → *Extract to a `useCallback`-memoized ref callback: `const scrollRef = useCallback((el: HTMLDivElement | null) => { el?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and use `ref={scrollRef}`. This fires only when the element mounts or message count changes.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - POST /api/admin/help-chat):** The help-chat API call sends the full accumulated messages array on every turn with no size cap. After a long session this payload grows unbounded, risking hitting server body-size limits and sending excessive tokens to the LLM backend, increasing cost and latency.
  → *Slice the context window before sending: `messages: next.slice(-20)` (keep last 20 messages). Add a visible message count indicator so admins know the history is windowed. Optionally add a 'Clear chat' button that resets state.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (/api/health (~line 35) and /healthz (~line 85)):** There are two health check endpoints with different paths and different behavior — `/api/health` returns no DB check, `/healthz` does a live DB ping. This is duplicate/dead surface area that creates confusion for ops tooling and exposes an unauthenticated DB-probing endpoint unnecessarily.
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** All audit timestamp columns use `.defaultNow()` but are not marked `.notNull()`. Drizzle will infer them as nullable in TypeScript, meaning application code can legally insert a null createdAt, and query results type these as `Date | null` forcing unnecessary null checks everywhere downstream.
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - handleKey):** handleKey is defined as a plain inline arrow function inside the component body, recreated on every render. It is passed to the onKeyDown prop of the input, causing the input element to receive a new function reference every keystroke (since typing triggers re-render via setInput), which defeats React's reconciliation optimisation for that prop.

## 💡 Opportunities

### Centralized route registration error boundary: 🔥 High Impact
Each `await registerXRoutes(app)` call can throw but there is no try/catch around them. A failure in registerOrgRoutes for example silently crashes startup. Wrap the six registration calls in a single try/catch that logs which module failed and rethrows, giving a clear startup error instead of an opaque rejection.

### APP_BASE_URL environment config for canonical URL generation: 🔥 High Impact
Introducing a single `APP_BASE_URL` env var used by robots.txt, sitemap, and any future canonical link generation eliminates the host-header injection surface permanently and makes the app trivially deployable to staging vs production domains without code changes.

### Object storage route auth guard: ⚡ Medium Impact
The `/public-objects/:filePath(*)` route serves files with no auth check. If any non-public files (e.g. user-uploaded content stored under a predictable path) end up in the same bucket namespace, they are world-readable. Add a path allowlist or prefix check (e.g. only allow paths starting with `logos/`, `videos/`) to constrain what can be retrieved anonymously.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate table columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract `export const DEFAULT_CURRENCY = 'BSD'` into a shared config and reference it everywhere. When the app expands to TTD or JMD support, there is currently no single place to audit or change the default, risking inconsistent data.

### Add composite index on billReminders (userId, nextDueDate): 🔥 High Impact
The bill reminders feature will routinely query upcoming bills for a user sorted by due date — the exact pattern of a range scan on (userId, nextDueDate). Without this index the query degrades to a full table scan. Adding `index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate)` will make due-date lookups O(log n) and directly improve dashboard load time.

### Replace open-ended text enum columns with a dedicated PostgreSQL enum type: ⚡ Medium Impact
Columns like transactions.type, budgets.period, billReminders.frequency, and simulatedStocks.riskLevel use Drizzle's inline text enum, which stores as plain varchar with only a CHECK constraint. Defining these as `pgEnum` types gives you a reusable DB-level type, cleaner Drizzle inference, and prevents CHECK constraint bypass via raw SQL. It also self-documents valid domain values in pg_type, which aids DBA auditing in a financial context.

### Centralised graceful-shutdown registry: 🔥 High Impact
Introduce a ShutdownManager class that accepts cleanup callbacks (clearInterval, httpServer.close, db pool drain, job worker stop) and registers a single SIGTERM/SIGINT handler. Currently shutdown logic is scattered and incomplete. A registry ensures deterministic teardown order and a configurable force-exit timeout (e.g. 10 s), which prevents Replit container restarts from leaving orphaned DB connections.

### Structured JSON logging with redaction: 🔥 High Impact
Replace ad-hoc console.log/log() calls with a structured logger (pino is zero-dependency and extremely fast). Define a redact list covering password, token, sessionId, and any AI response fields. This eliminates the PII-in-logs risk, enables log-level filtering per environment, and produces machine-parseable output suitable for future log aggregation without refactoring.

### Extract scheduler configuration to environment variables: ⚡ Medium Impact
The digest and purge scheduler intervals appear to be hardcoded constants in the truncated code. Expose them as DIGEST_INTERVAL_MS and PURGE_INTERVAL_MS env vars with the current hardcoded values as defaults. This allows tuning cadence in staging without a redeploy, and makes it immediately obvious during an incident what polling frequency is active.

### Virtualise long admin list renders: 🔥 High Impact
Admin dashboards for schools/users/transactions likely render large flat lists. Introducing @tanstack/react-virtual for any paginated table with >50 rows would cut initial paint time and memory use significantly, especially on lower-end Caribbean school devices.

### Batch independent dashboard queries with Promise.all: 🔥 High Impact
If the dashboard fires sequential useQuery calls for stats, users, schools, and transactions on mount, replacing them with a single aggregated endpoint or using Promise.all in a custom hook reduces waterfall latency. Expected impact: 40-60% reduction in dashboard time-to-interactive on slow connections.

### Move hardcoded API path strings to a central constants file: ⚡ Medium Impact
Strings like '/api/admin/help-chat' scattered inline across the component make endpoint changes error-prone and untestable. Centralising into an `API_ROUTES` const object enables find-all refactoring, easier mocking in tests, and a single place to prefix with an env-configured base URL.
