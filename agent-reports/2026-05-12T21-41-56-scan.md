## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: fetches all holdings then issues a separate DB query per holding to resolve the associated SimulatedStock, causing O(n) round-trips to the database.

**Fix:** Replace the per-holding stock lookup with a single JOIN using Drizzle's relational query API or a manual join: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. This collapses n+1 queries into 1.

### `server/storage.ts` — getLeaderboard
**Issue:** No pagination guard on leaderboard query. The `limit` parameter is optional with no enforced maximum, meaning a missing or malicious caller can trigger a full-table scan and return unbounded rows into memory.

**Fix:** Enforce a hard cap: `const safeLimit = Math.min(filters?.limit ?? 20, 100);` and always pass `safeLimit` to `.limit()`. Return HTTP 400 if a caller requests more than the cap at the route layer.

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Stock buy/sell flow calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three separate awaits outside any database transaction. A crash between any two steps leaves the portfolio in an inconsistent state (e.g., balance deducted but holding not updated).

**Fix:** Wrap all three operations in a single Drizzle transaction: `await db.transaction(async (tx) => { await upsertHolding(tx, ...); await insertPortfolioTx(tx, ...); await updateBalance(tx, ...); })`. Refactor each helper to accept an optional `tx` parameter that defaults to `db`.

### `server/jobHandlers.ts` — extract-paper handler / runExtractPaper
**Issue:** Full base64-encoded file content (fileB64) is stored directly in the jobs table payload. For a multi-page PDF or image this can easily exceed 5–10 MB per row, bloating the jobs table, blowing past any row-size limits, and exposing raw file bytes in any job introspection log or admin UI that prints job.payload.

**Fix:** Upload the file to object storage before enqueuing the job. Store only the object storage path in the payload (e.g., { paperId, objectPath: '/uploads/papers/123.pdf', ext, subject }). In the handler, stream the file from object storage using the existing objectStorageClient helpers. Delete the staged object after successful extraction.

---

### `server/jobHandlers.ts` — streamPrivateObjectToResponse
**Issue:** The function accepts objectPath and constructs bucket/object by splitting on '/'. There is no validation that the resolved path stays within the expected private bucket namespace, making it possible for a caller to pass a crafted path like '../../other-bucket/sensitive-file' and read arbitrary objects from object storage.

**Fix:** After splitting, assert that parts[0] matches the expected bucket name from PRIVATE_OBJECT_DIR environment config: const expectedBucket = parsePrivatePath('_').bucket; if (bucket !== expectedBucket) throw new Error('Invalid object path'); This pins all reads to the configured private bucket.

### `shared/schema.ts` — budgets table definition
**Issue:** budgets.userId has no index. Any query filtering budgets by user (the dominant access pattern) will do a full table scan as the table grows.

**Fix:** Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also add a composite index on (userId, categoryId) since budget lookups almost always filter on both.

---

### `shared/schema.ts` — billReminders table definition
**Issue:** billReminders has no index on nextDueDate or userId. Reminder-due queries (WHERE userId = ? AND nextDueDate <= ?) will full-scan. For a reminders feature that likely runs on a scheduler, this is a repeated performance hit.

**Fix:** Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the table definition.

---

### `client/src/pages/AdminDashboard.tsx` — apiRequest calls inside queryFn lambdas
**Issue:** Every queryFn uses `.then(r => r.json())` without any error handling. If the server returns a non-2xx status (e.g., 401 session expiry, 500 crash), r.json() may still resolve with an error body or throw a parse error, and TanStack Query will surface a generic error rather than an actionable message. For a financial-data admin panel this is a silent data integrity risk.

**Fix:** Create a typed helper: `async function adminFetch<T>(method: string, url: string): Promise<T> { const r = await apiRequest(method, url); if (!r.ok) throw new Error(await r.text()); return r.json(); }` and use it in all queryFn definitions. This ensures TanStack Query's error boundary receives real error messages and the UI can display them.

## 🟡 Warnings (11)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregation queries (total income, total expenses, savings goals count, budgets). Each query is independent and the combined latency is additive.
  → *Run all independent stat queries concurrently: `const [incomeResult, expenseResult, goalsResult, budgetsResult] = await Promise.all([db.select(...), db.select(...), db.select(...), db.select(...)]);`. Expected latency reduction: from sum of all query times to the slowest single query.*
- **`server/storage.ts` (getAllQuestions):** getAllQuestions fetches every ExtractedQuestion (and optionally joins ExamPaper) with no limit, offset, or maximum row guard. As exam papers accumulate this will load the full questions table into Node.js heap on every call.
  → *Add mandatory pagination parameters `(filters?: { subject?: string; limit?: number; offset?: number })` with `const safeLimit = Math.min(filters?.limit ?? 50, 200);` and apply `.limit(safeLimit).offset(filters?.offset ?? 0)` to the query. Update callers to paginate.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one-at-a-time inside a for loop with sequential awaits. For a 40-question exam paper this issues 40 serial round-trips to the database, making extraction latency proportional to question count and holding the DB connection for far longer than necessary.
  → *Replace the for loop with a single bulk insert: filter valid questions first, map them to insert rows, then call db.insert(extractedQuestions).values(rows) once. If storage.createExtractedQuestion must be used for its side-effects, use Promise.all on a mapped array with a concurrency cap (e.g., p-limit at 5).*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation before being used in arithmetic. A missing or non-numeric value produces NaN, making the cutoff date invalid and causing the DELETE to silently wipe all rows (WHERE createdAt < NaN is implementation-defined) or fail unpredictably.
  → *Add explicit validation at the top of the handler: const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days <= 0) throw new Error('olderThanDays must be a positive number'); Use the validated value for the cutoff calculation.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete before re-extraction):** The idempotency delete (db.delete(extractedQuestions).where(eq(...paperId))) runs unconditionally on every attempt, including the first. If a concurrent request has already completed the job and the job table hasn't been updated yet, a retry triggered by a transient network error will delete the successfully extracted questions before re-running the full expensive OpenAI pipeline.
  → *Check the current paper status before deleting: fetch the paper record first; if status is already 'completed', return early with the existing result. Only delete existing questions if status is 'processing', confirming this is a genuine retry of an incomplete run.*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** All userId columns are declared as varchar with no length argument. PostgreSQL varchar without length is equivalent to text, so the constraint provides no protection. More critically, if users.id has a defined length (e.g., varchar(36) for UUIDs), these FK columns silently diverge, risking join inefficiency and future migration pain.
  → *Define a shared constant (e.g., export const USER_ID_TYPE = varchar('user_id', { length: 36 })) matching the length in users.id, and use it consistently. Alternatively switch all to text if users.id is text, keeping types uniform.*
- **`shared/schema.ts` (createdAt columns on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is declared with .defaultNow() but without .notNull(). This means application code can insert a null createdAt, breaking any ORDER BY createdAt queries silently and violating the implicit invariant that audit timestamps always exist.
  → *Chain .notNull() after .defaultNow() on every createdAt column: timestamp('created_at').defaultNow().notNull(). Same applies to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table definition):** simulatedStocks.ticker has no unique index. Nothing prevents duplicate tickers being inserted, which would corrupt portfolio valuation lookups that likely key on ticker. There is also no index on ticker despite it being a natural lookup key.
  → *Replace the plain column definition with a uniqueIndex: add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to the table definition.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object and its string representation are reconstructed on every render inside the component body, not memoized. This also means the queryKey array contains primitive values but the queryFn closes over a stale `params` reference built outside useMemo, so cache invalidation and the actual fetch can diverge if React batches state updates.
  → *Wrap params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]);` then use `params.toString()` in both queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery in ObservabilityCard has no error state, no loading state, and no staleTime/refetchInterval config. If the /api/admin/observability endpoint fails, the component silently renders with undefined data and every badge will show 'Disabled'/'Not configured' with no indication to the admin that the data fetch failed.
  → *Destructure `isLoading` and `isError` from useQuery, render a loading skeleton when isLoading is true, and render an error badge (e.g., 'Unavailable') with a retry button when isError is true. Add `staleTime: 60_000` to avoid hammering the endpoint on every tab switch.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and tab-switching handlers):** Inline arrow functions passed as onClick handlers to tab buttons (e.g., `onClick={() => setTab(t.id)}`) are recreated on every parent render. With 13 tabs each mounting icon components and badges, this causes all tab buttons to re-render on any state change in AdminDashboard, including search input keystrokes.
  → *Wrap the handler in useCallback: `const handleTabClick = useCallback((id: string) => setTab(id), []);` and pass `() => handleTabClick(t.id)` — or better, memoize one callback per tab id using a Map ref pattern to avoid the inline wrapper entirely. Move TABS outside the component (it already is) to avoid re-allocation.*

## 💡 Opportunities

### Centralise query join patterns with Drizzle relational queries — 🔥 High Impact
getBillReminders, getPortfolioTransactions, and getEnrollmentsByClass all resolve foreign-key relations with per-row queries or manual joins. Defining Drizzle `relations()` in the schema and using `db.query.<table>.findMany({ with: { relation: true } })` would eliminate all N+1 patterns in one refactor pass and make future relation queries one-liners.

### Add a storage-layer result cache for market stock data — ⚡ Medium Impact
getMarketStocks is called on every portfolio page load and the underlying simulated prices change infrequently. A 30-second in-process cache (Map + timestamp) or a Redis cache would eliminate redundant DB reads under concurrent student users without changing the interface contract.

### Extract hardcoded virtual balance seed value to environment config — ⚡ Medium Impact
The initial virtual balance amount (visible in getVirtualBalance upsert logic) appears to be a hardcoded numeric literal. This should be `process.env.VIRTUAL_BALANCE_INITIAL ?? 10000` so educators and sponsors can configure starting balances per deployment without a code change.

### Parallelize independent DB lookups in admin-csv-export — 🔥 High Impact
The truncated admin-csv-export handler likely fetches multiple tables sequentially (users, transactions, sessions, etc.). Wrapping independent queries in Promise.all would cut export job duration proportionally to the number of sequential queries — likely a 3–5× speedup for a full export.

### Centralise job payload validation with Zod schemas — ⚡ Medium Impact
Each handler currently trusts job.payload fields without runtime type-checking (e.g., olderThanDays, paperId, type). Defining a Zod schema per job type and parsing payload at handler entry would catch malformed jobs immediately, produce clear error messages, and eliminate a whole class of silent bad-input bugs across all handlers.

### Cap extracted text sent to GPT-4o at a validated token estimate — ⚡ Medium Impact
extractedText is sliced to 30,000 characters but this is a byte slice, not a token slice. A dense 30,000-char document can exceed 8,000 tokens and cause unexpected API errors or inflated costs. Use tiktoken (cl100k_base) to count tokens and truncate to a safe limit (e.g., 6,000 tokens) before sending, then store the truncation threshold as an environment config value.

### Centralise currency default as an environment-driven config constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the currency default in at least six separate table columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract DEFAULT_CURRENCY from an env var or a single config constant and reference it via .default(DEFAULT_CURRENCY). When the app expands to other Caribbean currencies (TTD, JMD, etc.) this change becomes a single-line edit instead of a migration across six columns.

### Add a partial index on documentUploads for in-flight processing rows — ⚡ Medium Impact
Polling queries for documents still in 'processing' status will scan all uploads. A partial index WHERE status = 'processing' would make these queries O(active jobs) instead of O(all uploads). In Drizzle this requires a raw SQL index via pgTable's third callback using sql`...` until Drizzle supports partial indexes natively.

### Promote learningModules.slug unique constraint to a uniqueIndex for query performance — 📌 Low Impact
The slug column uses .unique() which creates a unique constraint backed by a btree index, but it is not explicitly named or documented. Replacing it with uniqueIndex('idx_learning_modules_slug').on(t.slug) makes the index visible in migration diffs, allows DBA-level monitoring by name, and ensures slug-based route lookups (WHERE slug = ?) use the index rather than relying on constraint inference.

### Split monolith into lazy-loaded tab modules — 🔥 High Impact
All 13 tab panels appear to be defined in a single file exceeding 126 KB of source. Using React.lazy + Suspense per tab (e.g., `const AuditLogPanel = lazy(() => import('./tabs/AuditLogPanel'))`) would reduce initial parse/compile time by ~80% for non-admin users who land on a specific tab, and eliminate the current problem of all tab queries mounting simultaneously.

### Centralise admin query error handling with a QueryCache onError callback — 🔥 High Impact
Rather than adding try/catch to every queryFn individually, configure a dedicated QueryClient for the admin scope with `queryCache: new QueryCache({ onError: (err) => toast({ title: 'Admin API error', description: err.message, variant: 'destructive' }) })`. This gives consistent user-visible error feedback across all 13+ queries with a single change point.

### Add refetchInterval and staleTime defaults to all admin queries — ⚡ Medium Impact
Dashboard, jobs, and perf-agent panels benefit from live data, but audit log, schools, and user lists do not need real-time updates. Setting `staleTime: 30_000` on list queries and `refetchInterval: 10_000` only on jobs/perf panels would reduce unnecessary network traffic by an estimated 60-70% during a typical admin session while keeping operational panels fresh.
