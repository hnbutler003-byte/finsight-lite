## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. Any authenticated or unauthenticated caller can hammer these endpoints, causing runaway OpenAI spend and DoS risk — especially critical for a school-aged user base where abuse is likely.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls.

### `server/routes.ts`: robots.txt handler (~line 38) and sitemap.xml handler (~line 52)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header causing the robots.txt Sitemap directive and sitemap <loc> tags to point to an arbitrary domain — a host header injection that can be weaponised for cache poisoning or phishing via SEO scrapers.

**Fix:** Add a `CANONICAL_BASE_URL` environment variable (e.g. `https://finsightlite.com`) and use `process.env.CANONICAL_BASE_URL` as the primary source for `base`, falling back to the header-derived value only in local dev behind an explicit `NODE_ENV === 'development'` guard.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no composite or individual index. Any dashboard query fetching budgets per user will result in a full table scan as the table grows, which will degrade for active users with many budget records.

**Fix:** Add an index in the table's second argument: `(t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })`. Also add a uniqueIndex on (userId, categoryId, period) to prevent duplicate budget entries for the same category and period.

---

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** There is no unique constraint on (userId, stockId) in portfolioHoldings. Without it, a race condition or retry in the buy flow can insert duplicate holding rows for the same user and stock, causing double-counted quantities and incorrect portfolio valuations.

**Fix:** Add `(t) => ({ userStockUniq: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) })` as the table's second argument. Application upsert logic should then use `ON CONFLICT (user_id, stock_id) DO UPDATE SET quantity = ...` to atomically update quantity rather than inserting a new row.

### `server/index.ts`: probeDatabase / startup await sequence
**Issue:** probeDatabase, seedDatabase, and other async startup steps are awaited sequentially on the critical path before the server binds. If probeDatabase is doing a real DB round-trip synchronously or each step takes 200–500 ms, total startup latency compounds. More critically, the interval handles returned by setInterval for maybeRunWeeklyDigest and maybeRunAiUsagePurge are never stored, so there is no way to call clearInterval during graceful shutdown, causing those callbacks to fire against a torn-down DB connection pool.

**Fix:** Store every setInterval return value: `const weeklyHandle = setInterval(maybeRunWeeklyDigest, ...)`. In the SIGTERM/SIGINT handler call `clearInterval(weeklyHandle)` and `clearInterval(purgeHandle)` before closing the HTTP server and DB pool. Also run independent startup steps (seedDatabase, registerJobHandlers, probeDatabase) with Promise.all where they have no data dependency on each other.

---

### `server/index.ts`: res.json monkey-patch in request logger middleware
**Issue:** Every JSON response body is captured into capturedJsonResponse and then serialized in full via JSON.stringify into the log line with no size cap, field allowlist, or redaction. For a financial literacy app with school-aged users this will log session tokens, quiz scores, personally identifiable student data, and any AI-generated content verbatim to whatever log sink is attached, violating basic data minimisation and likely COPPA/FERPA obligations.

**Fix:** Replace the full-body log with a size-bounded, field-redacted summary. At minimum: only log for non-2xx responses or explicitly opted-in debug mode (`process.env.LOG_RESPONSE_BODY === 'true'`), and when you do log, strip known sensitive keys (`token`, `password`, `email`, `studentId`) and truncate the serialised string to 200 chars: `logLine += ' :: ' + JSON.stringify(redact(capturedJsonResponse)).slice(0, 200)`.

## 🟡 Warnings (8)

- **`server/routes.ts` (await registerAuthDomainRoutes ... await registerEmailRoutes (~lines 117-122)):** The six domain routers are imported in parallel with Promise.all (good) but then registered with six sequential awaits. If any registration function does async work (DB probes, middleware setup), this serialises what could run concurrently, adding unnecessary cold-start latency.
  → *After the parallel import destructuring, call all six register functions concurrently: `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), registerAiRoutes(app), registerOrgRoutes(app), registerLessonRoutes(app), registerEmailRoutes(app)]);` — safe as long as each router only calls `app.use/get/post` (which is synchronous in Express).*
- **`server/routes.ts` (initSupabaseTables().then(...).catch(...) (~line 105)):** The seven Supabase seed/init steps are chained sequentially in a fire-and-forget promise. A failure in any middle step (e.g. seedFinancialAcademyLesson) silently skips all subsequent steps and only logs a single generic error, making it impossible to tell which step failed or to retry individual steps.
  → *Wrap each step in its own try/catch with a labelled log, or run independent steps in parallel: `await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])` where there are no inter-dependencies. At minimum log the failing function name: `.catch(e => console.error('[Supabase] Init error in chain:', e.message))`.*
- **`server/routes.ts` (/healthz handler (~line 77)):** The /healthz endpoint performs a live DB query (`select 1`) on every call with no rate limiting or auth. Uptime monitors calling this every 30 seconds is fine, but the endpoint is publicly reachable and could be used to generate steady DB connection churn or fingerprint infrastructure. Additionally, dynamic import of `./db` and `drizzle-orm` inside the handler on every request is wasteful.
  → *Move the `import('./db')` and `import('drizzle-orm')` to the top of the file as static imports. Add a simple IP-based rate limiter (e.g. 10 req/min) or restrict to internal networks via a check on `req.ip`. Consider caching the last successful check result for 5 seconds to avoid per-request DB hits.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt, linkedCards.createdAt, documentUploads.createdAt, billReminders.billReminders.createdAt):** All createdAt columns use `.defaultNow()` but are missing `.notNull()`. Drizzle will type these as `Date | null` in TypeScript, meaning application code must defensively null-check audit timestamps that should never be null, and NULL values can slip in via raw SQL or migrations.
  → *Append `.notNull()` to every `.defaultNow()` call across all tables, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. This enforces the invariant at the DB level and removes spurious nullability from the generated types.*
- **`shared/schema.ts` (userId: varchar('user_id') — all tables):** All varchar user_id columns are declared without a length argument, which defaults to unlimited length in PostgreSQL. This wastes index space, skips implicit input length validation, and deviates from the fixed-length IDs (e.g. 21-char nanoid or UUID) that auth providers typically issue.
  → *Determine the exact ID length your auth provider emits (e.g. 36 for UUID, 21 for nanoid) and declare `varchar('user_id', { length: 36 })` consistently across every table. This shrinks index pages and catches oversized values at the DB boundary.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** `lastPriceUpdateDate` is typed as `text` instead of `timestamp`. Storing a date as plain text means the DB cannot enforce valid date formats, range queries (`WHERE lastPriceUpdateDate > ...`) will not use index range scans, and ordering will be lexicographic rather than chronological.
  → *Change the column to `timestamp('last_price_update_date')` and migrate existing data with `ALTER TABLE simulated_stocks ALTER COLUMN last_price_update_date TYPE timestamp USING last_price_update_date::timestamp`. Update any application code that writes a string to pass a `Date` object instead.*
- **`server/index.ts` (maybeRunWeeklyDigest – delete-on-enqueue-failure compensation):** When enqueueJob throws, the code deletes the weeklyDigestRuns row to 'roll back the claim'. This is a non-atomic two-phase pattern: if the process crashes or the DELETE itself fails between the INSERT and the DELETE, the row stays claimed but no job was ever enqueued, silently dropping that week's digest with no alerting and no retry beyond the next interval tick which will see the existing row and skip.
  → *Add a `status` column (`'claimed' | 'enqueued' | 'failed'`) to weeklyDigestRuns. Set it to 'enqueued' after a successful enqueueJob call. On failure, UPDATE status to 'failed' rather than deleting. Change the scheduler's conflict guard to `WHERE status = 'enqueued'` so failed rows are automatically retried on the next interval. Add a captureError/Sentry call on the catch branch so ops are alerted immediately.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes Sunday at 19:00 in the server's local timezone (whatever `new Date()` and `setHours` resolve to), not a fixed Caribbean timezone such as America/Port_of_Spain (UTC-4). On a UTC-based cloud host the digest fires at 19:00 UTC = 15:00 local time, four hours early, which will produce wrong 'last week' window boundaries for the weekStart key and confuse deduplication.
  → *Use the Intl API or a lightweight library to get the current time in 'America/Port_of_Spain'. Derive daysSince and the 19:00 anchor in that zone: `const localNow = new Date(new Date().toLocaleString('en-US', { timeZone: process.env.DIGEST_TIMEZONE ?? 'America/Port_of_Spain' }))`, then perform all date arithmetic on localNow. Expose DIGEST_TIMEZONE as an environment variable so it is configurable without a code change.*

## 🔵 Info / Tech Debt (1)

- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / hardcoded scheduler intervals):** The scheduler tick intervals (e.g. the setInterval periods for weekly digest and purge checks) appear hardcoded in milliseconds inside the startup block rather than derived from environment config. ## 💡 Opportunities

### Centralise base-URL resolution into a shared utility: ⚡ Medium Impact
The proto+host derivation logic is duplicated identically in robots.txt and sitemap.xml handlers. Extract it into a `getCanonicalBaseUrl(req)` helper in a shared `utils/url.ts` that reads from `CANONICAL_BASE_URL` env first, reducing duplication and fixing the host injection in one place.

### Add explicit auth middleware guard before AI route registration: 🔥 High Impact
Beyond rate limiting, chat/image/audio routes should verify the caller is an authenticated student or teacher before hitting OpenAI. Adding a reusable `requireAuth` middleware applied at the router level (not per-handler) ensures no future route added inside those modules is accidentally public, which matters given the school-aged user context.

### Replace fire-and-forget Supabase init with structured startup sequencing: ⚡ Medium Impact
Making the Supabase init blocking (with a timeout) during server startup ensures the app never serves requests against an unseeded DB. Use `Promise.race([initChain(), timeout(15_000)])` and log a startup warning if seeding is slow rather than silently backgrounding it. This prevents subtle bugs where early requests arrive before seed data exists.

### Centralise default currency as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, portfolioHoldings). Extract `const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD'` into a shared config file and reference it in all `.default()` calls. This makes currency changes a one-line config edit instead of a schema-wide search-and-replace.

### Add partial index on transactions for unprocessed document uploads: ⚡ Medium Impact
Queries that check `documentUploadId IS NOT NULL` to reconcile auto-synced transactions will scan the full transactions index. A partial index `CREATE INDEX idx_transactions_document ON transactions(document_upload_id) WHERE document_upload_id IS NOT NULL` would make reconciliation queries sub-millisecond even at high transaction volumes, which matters for the document processing status polling flow.

### Enforce referential integrity on billReminders.categoryId: 📌 Low Impact
billReminders.categoryId references categories but has no NOT NULL constraint, which is intentional, but there is also no ON DELETE SET NULL or ON DELETE RESTRICT clause declared. If a category is deleted, Postgres will reject the delete due to the FK but the error will surface as an unhandled DB exception in the application layer. Explicitly declare `.references(() => categories.id, { onDelete: 'set null' })` to make the intent clear, keep the column nullable consistently, and prevent surprise errors during category cleanup operations.

### Parallel startup initialisation with Promise.all: 🔥 High Impact
probeDatabase, seedDatabase, and registerJobHandlers have no stated data dependency on each other yet are awaited sequentially. Running them with Promise.all would cut startup latency by roughly (N-1) * slowest-independent-step, likely saving 500–1500 ms on cold starts against a remote DB, which matters for Replit's container wake-up time and health-check timeouts.

### Structured logging with pino or winston: 🔥 High Impact
All logging currently uses console.log with a hand-rolled time prefix and string concatenation. Replacing this with pino (near-zero overhead, JSON output) enables log-level filtering, redaction rules via `redact` option, and direct ingestion into structured log aggregators without parsing. This directly mitigates the PII-in-logs finding and gives ops queryable, level-gated output with no behaviour change to existing call sites since log() is a single exported function.

### Centralised graceful shutdown coordinator: ⚡ Medium Impact
SIGTERM and SIGINT handling is currently either absent or scattered. A single shutdown coordinator that (1) stops accepting new connections, (2) clears all intervals, (3) waits for in-flight requests via server.close(), (4) drains the job worker, and (5) closes the DB pool in order would prevent the interval-leak and torn-pool issues found above and is a one-time ~40-line addition that protects all future async resources added to the process.
