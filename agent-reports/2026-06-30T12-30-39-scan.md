## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI-heavy routes (chat, image, audio) are registered with no rate limiting middleware applied before them. For a school-aged user base these endpoints call expensive external APIs (OpenAI) and are trivially abusable, leading to runaway API costs or denial-of-service against the AI tier.

**Fix:** Create an aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls.

### `server/routes.ts`: robots.txt handler (~line 38) and sitemap.xml handler (~line 53)
**Issue:** The base URL is constructed from raw, unvalidated request headers (`x-forwarded-host`, `host`). An attacker can send a crafted Host header causing the Sitemap directive or sitemap <loc> values to point to an arbitrary domain, enabling cache-poisoning or open-redirect abuse.

**Fix:** Read the canonical origin from a single env var: `const base = process.env.CANONICAL_ORIGIN ?? 'https://finsightlite.com';` and ignore request headers entirely for URL construction in these two handlers.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter / extract-paper handler)
**Issue:** The full base64-encoded file (potentially MBs) is stored in the job payload in the jobs table. This bloats the DB, slows job queue polling queries, and risks OOM when workers load many jobs into memory simultaneously.

**Fix:** Upload the file to object storage at enqueue time and store only the object path in the job payload. In runExtractPaper, download the file from object storage at the start of the handler using the path, then delete it after successful extraction. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** simulatedStocks.ticker has no unique index. If a seed or sync job inserts duplicate tickers, portfolio queries that join on ticker string will return duplicate rows, silently inflating portfolio valuations—a critical correctness bug for a financial app.

**Fix:** Add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) as the table's second argument. This enforces uniqueness at the DB level regardless of application-layer guards.

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** There is no unique constraint on (userId, stockId) in portfolioHoldings. Nothing prevents multiple holding rows for the same user and stock, which would cause double-counting of shares and incorrect P&L calculations without additional deduplication logic in every query.

**Fix:** Add (t) => ({ userStockUniqueIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) to the table definition. Upsert logic (INSERT ... ON CONFLICT DO UPDATE) can then safely increment quantity and recalculate avgPurchasePrice atomically.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** The day-of-week rollback logic uses `(dow + 0) % 7` which is a no-op offset. Sunday (dow=0) yields daysSince=0, so setDate subtracts 0, leaving `d` as today. The subsequent `if (d > now)` guard then subtracts 7 days, meaning on Sunday before 7 pm the function returns the *previous* Sunday, and on Sunday after 7 pm it returns today — but on any other day it returns the current week's Sunday with no rollback, potentially returning a future date that bypasses the early-return guard and enqueues for the wrong week.

**Fix:** Replace `(dow + 0) % 7` with `dow` (days since last Sunday = dow itself, since Sunday is 0). Verify: Monday dow=1 → subtract 1 day → last Sunday. Wednesday dow=3 → subtract 3 → last Sunday. Add a unit test covering each day of the week and the before/after 7 pm boundary.

### `server/index.ts`: res.json middleware / log(logLine)
**Issue:** The request-logging middleware captures the full serialized JSON response body (`capturedJsonResponse`) and logs it unconditionally for every `/api` route. For FinSight Lite this will log quiz results, financial literacy scores, user profile data, and any AI-generated content verbatim to stdout/Sentry breadcrumbs, violating data-minimisation principles and potentially exposing PII for school-aged users.

**Fix:** Gate body logging behind a `NODE_ENV !== 'production'` check, or replace full-body logging with a whitelist of safe scalar fields (e.g., `{ status, count, page }`). At minimum, strip any field matching /email|name|token|secret|password/i before logging.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level component render
**Issue:** If useAdminAuth returns a falsy/unauthenticated state there is no early return guarding the rest of the dashboard before data-fetching hooks run. All useQuery calls for users, schools, analytics etc. will fire regardless of auth state, leaking admin data queries to unauthenticated renders and wasting server resources.

**Fix:** Immediately after the useAdminAuth call add: `if (!adminAuth?.isAuthenticated) { redirect('/admin/login'); return null; }` before any other hook calls, or structure the component so queries are conditionally enabled: `enabled: !!adminAuth?.isAuthenticated`.

## 🟡 Warnings (15)

- **`server/routes.ts` (Supabase init chain (~line 110)):** The seven-step sequential `.then()` chain means a failure in any middle step (e.g. `seedFinancialAcademyLesson`) aborts all subsequent seeding silently — only a single `.catch` at the end logs the error. The app starts with partially initialised tables and no alerting beyond a console.error.
  → *Run independent seed operations in parallel and wrap each in its own try/catch so one failure does not cascade: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.filter(r => r.status === 'rejected').forEach(r => logger.error('[Supabase] seed error', r.reason)));` Keep `initSupabaseTables` and `initStaticContentTables` sequential only if they are true prerequisites.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100)):** The filePath parameter is passed directly to `objectStorage.searchPublicObject` with no path traversal sanitisation. A request to `/public-objects/../../private/keys` could expose non-public storage objects depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and reject any path containing `..`: `if (req.params.filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });`*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one in a sequential await loop. For a paper with 50 questions this creates 50 serial round-trips to the DB, dramatically increasing handler runtime and holding the DB connection longer than necessary.
  → *Collect all valid question objects into an array first, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Add a guard for empty array before inserting. This reduces N round-trips to 1.*
- **`server/jobHandlers.ts` (purge-ai-usage handler (registerJobHandler call)):** The purge-ai-usage handler has no try/catch. If the DELETE query fails (e.g., DB timeout, connection drop) the raw DB error propagates with no structured context, making it hard to distinguish transient from permanent failures and potentially leaking DB internals into job error logs.
  → *Wrap the db.delete call in try/catch, catch the error, log a structured message with { job: 'purge-ai-usage', cutoffDate, olderThanDays }, then rethrow so the job runner can apply its retry policy. Mirror the pattern already used in the ai-health-check handler.*
- **`server/jobHandlers.ts` (ai-health-check handler – capture(err, { job, triggeredBy })):** The Sentry capture passes job.payload.triggeredBy directly into the capture context. If triggeredBy ever contains a user-supplied string (e.g., a user ID or email injected at enqueue time) it will be sent to Sentry, potentially violating COPPA/data-minimisation obligations for a school-aged user base.
  → *Allowlist the triggeredBy value before passing to Sentry: only forward it if it matches a known internal enum ('job' | 'admin' | 'cron'). For any other value, replace with 'user-initiated' to avoid leaking PII into the error tracker.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Any query filtering budgets by user (the dominant access pattern) will do a full table scan as the table grows.
  → *Add an index in the table's second argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). A composite index on (userId, categoryId) covers both the list-by-user and the per-category lookup.*
- **`shared/schema.ts` (linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** All userId columns across these six tables are declared as varchar with no length argument. PostgreSQL will accept any length, but Drizzle emits varchar without a precision, meaning there is no enforcement that the value matches the length used in the users table. A mismatch silently stores truncated or oversized IDs, breaking FK integrity at the application layer.
  → *Audit the users.id column length (e.g., UUID = 36 chars, Clerk ID ~32 chars) and declare every userId as varchar('user_id', { length: 36 }) (or whichever length matches) consistently across all tables. This makes the DB enforce the contract the FK implies.*
- **`shared/schema.ts` (createdAt columns across all tables):** All createdAt columns use .defaultNow() but omit .notNull(). This means an explicit INSERT of NULL is accepted by Postgres, resulting in rows with no creation timestamp, which breaks any audit trail or time-based reporting.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default handles normal inserts; notNull prevents explicit null overrides.*
- **`server/index.ts` (setInterval (maybeRunWeeklyDigest / maybeRunAiUsagePurge)):** Both `setInterval` handles are stored locally but the truncated code shows no reference to them being passed into the graceful-shutdown handler. If `SIGTERM` fires, the HTTP server closes and the job worker stops, but the intervals keep the event loop alive (or fire one final time mid-shutdown) and their in-flight `await db...` calls run against a potentially closed DB pool, causing unhandled rejections.
  → *Assign both intervals to module-level `let digestInterval: NodeJS.Timeout` and `let purgeInterval: NodeJS.Timeout` variables. In the `shutdown()` function (or SIGTERM handler), call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` *before* closing the HTTP server and DB pool so no new DB calls are initiated after teardown begins.*
- **`server/index.ts` (seedDatabase / probeDatabase (startup await chain)):** `seedDatabase()` and `probeDatabase()` are awaited sequentially during startup inside what appears to be a top-level async IIFE. If `seedDatabase` performs schema migrations or bulk inserts it can block the event loop for seconds before the HTTP server binds, meaning health-check endpoints and the job worker are unavailable during that window with no timeout guard.
  → *Add a `Promise.race([seedDatabase(), timeout(30_000)])` wrapper that rejects with a clear error if seeding exceeds 30 s. Run `probeDatabase()` in parallel with non-dependent startup work via `Promise.all([probeDatabase(), registerRoutes(app)])` where safe. Consider moving heavy seed logic to a separate one-shot migration script rather than running it on every boot.*
- **`server/index.ts` (maybeRunAiUsagePurge / AUTO_PURGE_OLDER_THAN_DAYS):** The purge scheduler deletes AI usage rows older than `AUTO_PURGE_OLDER_THAN_DAYS` days but the delete query is not shown to have a row limit, meaning a first-run purge on a large table could issue a DELETE with hundreds of thousands of matching rows in a single transaction, locking the table and causing read timeouts for active student users.
  → *Add a `LIMIT` clause (e.g., 1000 rows per run) and loop until affected rows < limit, or use a cursor-based chunked delete with a small sleep between batches (`await sleep(50)`). Log the total deleted count at INFO level so ops can monitor catch-up runs.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new function on every render. React calls the old ref with null and the new ref with the element on every render cycle, causing redundant scrollIntoView calls and a minor but real perf leak in a chat that can accumulate many messages.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – AdminHelpChat):** The full `messages` array is sent to `/api/admin/help-chat` on every send with no length or token cap. For a school-age platform session that stays open all day this payload grows unbounded, eventually causing slow requests or backend rejections. There is also no maximum retry or timeout on the fetch.
  → *Slice to the last N messages before sending: `messages: next.slice(-20)`. Add an AbortController with a timeout: `const controller = new AbortController(); setTimeout(() => controller.abort(), 15000);` and pass `signal: controller.signal` to apiRequest.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline JSX handler functions throughout render):** Handler functions such as form submits, row delete confirmations, and filter onChange callbacks are defined as arrow functions directly inside JSX props (e.g. `onClick={() => { setSelectedUser(u); openModal(); }}`). Because the parent dashboard component holds many state slices, any state change re-renders the whole tree and recreates every one of these handlers, causing all child Button/Select/Dialog components to receive new prop references and re-render unnecessarily.
  → *Extract each handler with useCallback at the component level: `const handleDeleteUser = useCallback((id: number) => { ... }, [mutate, queryClient]);`. For child rows/cards that receive callbacks, wrap in React.memo so they only re-render when their own props change.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple sequential useQuery calls for dashboard stats):** Individual useQuery hooks for users, schools, teachers, analytics, and audit logs are defined sequentially. While React Query fires them in parallel by default within a single render, any query that depends on another (e.g. per-school analytics triggered by a selected school) uses a separate awaited apiRequest inside a useEffect rather than a dependent query, making those sequential and adding 200-400 ms of avoidable latency.
  → *Replace dependent useEffect+apiRequest patterns with a dependent useQuery using the `enabled` option: `useQuery({ queryKey: ['school-analytics', selectedSchoolId], queryFn: () => apiRequest(...), enabled: !!selectedSchoolId })`. This keeps all fetching in the React Query lifecycle with caching and deduplication.*

## 💡 Opportunities

### Parallel domain-route registration: ⚡ Medium Impact
The six `await registerXRoutes(app)` calls after the parallel import are still sequential. If any of these do async work at registration time (DB lookups, schema checks) they add cumulative cold-start latency. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` — safe as long as routes don't depend on each other's middleware ordering, which Express handles by insertion order not registration-call order.

### Structured logging for init and error paths: ⚡ Medium Impact
All error paths use bare `console.error`. Replacing with a structured logger (pino or winston) that emits JSON with a `severity` field would let Railway/Sentry log drain correctly classify startup failures versus request errors, and would surface the Supabase seed failures as alerts rather than buried console output.

### Object-storage CDN caching header extraction: 🔥 High Impact
The `/public-objects` handler sets a 30-day cache via `downloadObject` but there is no `Cache-Control` or `ETag` header set at the Express layer, so Railway's edge and any school-network proxy will re-request assets on every page load. Adding `res.setHeader('Cache-Control', 'public, max-age=2592000, immutable')` before streaming would reduce per-student bandwidth and API egress cost materially for a Caribbean low-bandwidth context.

### Parallel independent async initialisations inside handlers: ⚡ Medium Impact
Several handlers perform multiple independent DB reads or storage lookups sequentially (e.g., fetching user record then fetching related module in separate awaits). Wrapping independent fetches in Promise.all would cut per-job latency by the duration of the slower concurrent operation, meaningful under Caribbean mobile network conditions.

### Idempotency token on extracted-question wipe: 🔥 High Impact
The current idempotency strategy deletes all questions for a paperId at the start of every retry. If two worker instances pick up the same job simultaneously (e.g., after a queue visibility timeout), both will delete-then-insert concurrently causing duplicate or partial rows. Add a DB-level unique constraint on (paperId, order) and use INSERT ... ON CONFLICT DO NOTHING instead of a pre-delete wipe to make retries truly safe.

### Structured job result schema validation: ⚡ Medium Impact
JSON.parse on the OpenAI extraction response has no schema validation; a malformed or truncated model response silently produces an empty questions array and marks the job successful. Introducing a lightweight Zod schema for the expected response shape would catch model regressions immediately and allow the job to fail-and-retry rather than silently store nothing.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, etc.). Extract const DEFAULT_CURRENCY = 'BSD' into a shared config and reference it in each .default() call. When the app expands to TTD or JMD, one change propagates everywhere instead of requiring a grep-and-replace across the schema.

### Add index on billReminders.nextDueDate for reminder scheduling queries: ⚡ Medium Impact
The bill reminder feature will almost certainly query WHERE nextDueDate <= NOW() AND userId = ? to surface upcoming bills. Without an index on (userId, nextDueDate) this is a full table scan per user poll. Adding a composite index here eliminates the scan and directly supports any background job that pages through due reminders.

### Replace text type with timestamp for simulatedStocks.lastPriceUpdateDate: ⚡ Medium Impact
lastPriceUpdateDate is declared as text, meaning date comparisons and ORDER BY on this column require casting, are not index-friendly, and are locale-dependent. Changing it to timestamp('last_price_update_date') enables efficient range queries (e.g., find stocks not updated in 24 hours) and removes the risk of unparseable date strings entering the column.

### Parallel startup I/O with Promise.all: 🔥 High Impact
registerRoutes, startJobWorker, registerJobHandlers, and probeDatabase are likely independent and can be awaited concurrently. Running them sequentially adds their cumulative latency to cold-start time. Grouping independent calls under Promise.all could cut startup time by 40–70% on a cold Replit dyno where each DB round-trip takes ~200 ms.

### Structured logging with log levels: ⚡ Medium Impact
All observability currently goes through a single `log()` function that writes plain strings to stdout. Replacing it with a structured logger (pino is zero-dependency and very fast) adds log levels, JSON output for aggregators, and automatic redaction rules. This is especially important for a school-aged-user app where accidental PII in logs is a compliance risk.

### Scheduler deduplication via DB advisory locks: 📌 Low Impact
The weekly-digest scheduler uses an insert-on-conflict pattern for deduplication, which is correct, but each replica still queries and attempts an insert every interval tick. Using a PostgreSQL advisory lock (`SELECT pg_try_advisory_lock(hashtext('weekly-digest'))`) before the insert would short-circuit earlier and eliminate redundant DB writes when multiple dynos are running, reducing DB load proportionally to replica count.

### Memoize filtered and sorted table data: 🔥 High Impact
User/school table rows are likely filtered and sorted inside render using .filter().sort() on every keystroke of the search input. Wrapping these derivations in useMemo keyed on the raw data array and search string would eliminate O(n log n) work on each render, noticeably improving responsiveness for admins managing hundreds of students.

### Replace hardcoded API base paths with environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat' and any hardcoded pagination limits (page size = 10, 20, etc.) should come from a central config or environment variable. This prevents silent breakage when the API is versioned or deployed behind a prefix, and makes the values discoverable and changeable without a code change.

### Add React Query error and loading boundary components: ⚡ Medium Impact
The dashboard renders charts and tables before confirming query success. Adding explicit `if (isLoading) return <SkeletonDashboard />` and `if (isError) return <ErrorBanner retry={refetch} />` guards per section would prevent partial/broken UI states that could confuse school admins and surface misleading empty charts to teachers.
