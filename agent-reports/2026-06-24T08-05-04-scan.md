## 🔴 Critical (4)

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes
**Issue:** AI and image/audio generation routes are registered with no rate limiting middleware. For a school-aged user base on a financial literacy app, unbounded OpenAI calls are a direct cost and abuse vector — a single user could exhaust API quotas or trigger large bills.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g., 20 req/15 min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three `register*Routes` calls.

---

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored in the job table payload. A single PDF or image can be several megabytes encoded as base64, bloating the jobs table, slowing queue queries, and leaking file content to anything that reads job rows (logs, admin UIs, backups).

**Fix:** Upload the file to object storage before enqueuing the job, store only the object storage path in the payload (e.g., { paperId, objectPath, ext, subject }), then download the buffer at the start of runExtractPaper. Remove fileB64 from JobPayloads['extract-paper'] entirely.

---

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId has no index. Every query filtering budgets by user (the primary access pattern) will perform a full table scan. Same issue exists on linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions — none have a userId index defined in their table config callbacks.

**Fix:** Add an index callback to each affected table, e.g. for budgets: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). For portfolioHoldings also add a composite index on (userId, stockId) since holdings lookups will join both. Repeat the pattern for all tables missing userId indexes.

---

### `shared/schema.ts`: simulatedStocks table — ticker column
**Issue:** simulatedStocks.ticker has no unique index and no NOT NULL constraint beyond .notNull(). Ticker is a natural unique key used in lookups and joins from portfolioHoldings/portfolioTransactions. Without a unique index, duplicate tickers can be inserted silently, corrupting simulation data for school-aged users.

**Fix:** Replace the plain column definition with: ticker: text('ticker').notNull(), and add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) in the table config callback.

## 🟡 Warnings (14)

- **`server/routes.ts` (~line 97-105 (Supabase init chain)):** The seven Supabase seed functions are chained sequentially with `.then()`, meaning each waits for the prior to complete before starting. Some of these (e.g., seedStaticModules, seedRegionalContent, seedGameContent) are likely independent and add unnecessary cold-start latency.
  → *After the required init steps, run independent seeds in parallel: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await seedFinancialAcademyLesson(); await initStaticContentTables(); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]);` — wrap the whole block in try/catch instead of relying on a trailing .catch().*
- **`server/routes.ts` (~line 14 and line 80 (/api/health vs /healthz)):** Two health check endpoints exist: `/api/health` (returns immediately, no DB check) and `/healthz` (hits the DB). Railway and uptime monitors will silently diverge — one could report healthy while the DB is down. The duplicate also wastes maintenance surface.
  → *Remove `/api/health` and standardise on `/healthz` for all monitors. If a cheap liveness probe is needed separately from a DB readiness probe, rename them `/healthz/live` and `/healthz/ready` with clear docs.*
- **`server/routes.ts` (~line 52-55 (robots.txt and sitemap.xml host construction)):** `x-forwarded-host` and `host` headers are used directly to construct absolute URLs without sanitisation. A spoofed Host header causes the sitemap URL in robots.txt to point to an attacker-controlled domain, which search engines will follow.
  → *Add a whitelist check: `const ALLOWED_HOST = process.env.PUBLIC_HOST; const host = ALLOWED_HOST || req.headers['x-forwarded-host'] || req.headers.host || 'localhost';` and set PUBLIC_HOST in environment config for production. Reject or fall back if the resolved host doesn't match.*
- **`server/routes.ts` (~line 87 (public-objects route)):** The `/public-objects/:filePath(*)` route has no authentication check. Any unauthenticated request can enumerate and download any file in object storage by guessing paths, including potentially sensitive org logos or uploaded documents.
  → *Determine which objects are truly public (logos used on landing pages) vs. org/user-scoped. For scoped assets, add a session check middleware before the handler. For genuinely public assets, keep the route open but add a prefix convention (e.g., `/public-objects/public/...`) and reject requests outside that prefix without a valid session.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 40-question paper this is 40 serial round-trips to the database, dramatically increasing latency and holding the job worker busy unnecessarily.
  → *Collect all valid question objects into an array, then execute a single batch insert: await db.insert(extractedQuestions).values(validQuestions). Wrap it in a transaction with the updateExamPaper call so partial inserts don't survive a crash mid-batch.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** PDF text is silently truncated to 30,000 characters with no log, no warning stored on the paper record, and no indication to the user. For multi-page exam papers this will silently drop questions from the second half of the document, producing a misleadingly 'completed' result with fewer questions than the paper contains.
  → *After slicing, check if pdfData.text.length > 30000 and if so log a warning and set a truncated: true flag on the exam paper row (add the column or use an existing metadata field). Surface this in the job return value and in the UI so teachers know the extraction is partial. Also move 30000 to an env var or named constant MAX_PDF_CHARS.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete then re-insert):** The idempotency strategy deletes all previously extracted questions at the top of every attempt, then re-inserts them. If the job crashes after deletion but before the final updateExamPaper, the paper status stays 'processing' with zero questions in the DB. A subsequent retry correctly re-extracts, but any read between crash and retry returns inconsistent state (completed paper, no questions).
  → *Wrap the delete, batch insert, and status update in a single database transaction so they are atomic. If the transaction rolls back on failure, the previous questions remain intact and the UI remains consistent until the next attempt succeeds.*
- **`server/jobHandlers.ts` (parsePrivatePath — PRIVATE_OBJECT_DIR usage):** PRIVATE_OBJECT_DIR is checked at call-time inside an async job handler. If the env var is missing the job will fail at runtime after potentially doing expensive AI work, with no early startup validation. There is also no check that the constructed path stays within the intended bucket prefix, creating a path-traversal risk if relPath is ever derived from user input.
  → *Validate PRIVATE_OBJECT_DIR at server startup (throw if missing). In parsePrivatePath, assert that the resolved object path starts with the expected prefix after normalization and reject paths containing '..' segments before splitting, returning a 400-equivalent error to the job rather than allowing traversal.*
- **`shared/schema.ts` (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — createdAt columns):** createdAt columns across all tables use .defaultNow() but omit .notNull(). This allows NULL to be inserted explicitly (e.g. via a raw query or ORM escape), breaking any ORDER BY createdAt DESC queries and audit trails silently.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Same applies to purchasedAt on portfolioHoldings and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on nextDueDate or (userId, nextDueDate). The primary background job use-case for bill reminders is querying rows WHERE next_due_date <= NOW() across all users, or WHERE user_id = ? ORDER BY next_due_date. Without this index, reminder queries degrade linearly with table growth.
  → *Add a composite index in the table config callback: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This covers both the per-user and the global scheduler query patterns.*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is nullable (correct for system categories) but has no index, and there is no partial index or constraint distinguishing system categories (userId IS NULL) from user-owned ones. Queries like WHERE user_id = ? OR user_id IS NULL to fetch a user's full category list will scan the entire table.
  → *Add an index on userId: (t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) }). For a PostgreSQL partial index on system categories, add a raw SQL migration: CREATE INDEX idx_categories_system ON categories (id) WHERE user_id IS NULL; This avoids mixing system and user rows in the same full-scan path.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The full `messages` array is sent to /api/admin/help-chat on every turn with no size limit. For a school-aged user session left open, this array grows unboundedly, eventually producing multi-MB POST bodies and likely exceeding server payload limits or OpenAI token caps with no user feedback.
  → *Slice the history before sending: `const context = next.slice(-20);` and send `{ messages: context }` instead of `{ messages: next }`. Add a visible message counter or auto-clear warning when messages.length > 18.*
- **`client/src/pages/AdminDashboard.tsx` (messages.map render / scroll ref callback):** The scroll-to-bottom anchor uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}`. This anonymous function is recreated on every render, so scrollIntoView fires on every keystroke in the input (because `input` state changes cause re-render), not just when a new message arrives.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage / apiRequest catch block):** The catch block in sendMessage swallows the error entirely (bare `catch {}`). Network failures, 401s from an expired admin session, and 500s are all silently collapsed into the same generic chat bubble. A 401 should redirect to login; other errors should at minimum be logged for observability.
  → *Change to `catch (err) { const status = (err as any)?.status; if (status === 401) { setLocation('/admin/login'); return; } console.error('[AdminHelpChat] sendMessage failed', err); setMessages(prev => [...prev, { role: 'assistant', content: 'Sorry, something went wrong. Please try again.' }]); }`*

While minor in isolation, this component re-renders on every character typed (input state), so the listener is replaced on every keystroke — unnecessary allocation in a chat loop.

## 💡 Opportunities

### Centralise route module registration error handling: ⚡ Medium Impact
Each `await register*Routes(app)` call can throw independently but none are wrapped in try/catch. A single failing domain router (e.g., registerOrgRoutes) crashes the entire server startup silently past the top-level caller. Wrap all six calls in a single try/catch with a descriptive startup error and process.exit(1) to make failures observable in Railway logs.

### Move hardcoded sitemap URL list to a shared route manifest: 📌 Low Impact
The sitemap URLs are hardcoded in routes.ts, meaning any new page added to the React app won't appear in the sitemap unless this file is also manually updated. Extract to a shared `sitePages.ts` constant imported by both the sitemap handler and any future static generation tooling, reducing drift risk.

### Add request-scoped timeout to /healthz DB probe: ⚡ Medium Impact
The /healthz DB check has no timeout — a hung DB connection will hold the health check open indefinitely, causing Railway's health probe to wait until its own timeout. Wrap the db.execute call with a Promise.race against a 3-second timeout and return 503 on expiry to ensure fast failure signalling.

### Parallel independent DB lookups in admin-csv-export: 🔥 High Impact
The truncated code strongly suggests admin-csv-export fetches multiple tables sequentially. Replacing sequential awaits with Promise.all for independent queries (e.g., users, transactions, sessions) would reduce export job duration proportionally to the number of queries, likely cutting wall-clock time by 50-70% for typical export sizes.

### Job payload schema validation at enqueue time: ⚡ Medium Impact
Adding Zod schemas for each JobPayload variant and validating at enqueueJob call sites would catch missing or malformed fields (e.g., missing paperId, negative olderThanDays) before a job is persisted, preventing wasted worker cycles and confusing runtime errors deep inside handlers.

### Centralised OpenAI error handling with token-usage logging: ⚡ Medium Impact
Both openai.chat.completions.create calls in runExtractPaper are unwrapped — a rate-limit or context-length error surfaces as an untyped throw. Wrapping in a shared helper that catches OpenAI API errors, logs prompt_tokens/completion_tokens from the response, and maps rate-limit errors to a retryable job error would improve observability and prevent silent cost overruns on a school budget.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, portfolioHoldings). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it in each .default() call. When the app expands to TTD or JMD, only one line changes instead of six, eliminating the risk of inconsistent defaults.

### Add composite index on portfolioTransactions (userId, stockId, executedAt): 🔥 High Impact
Portfolio performance calculations require aggregating transactions per user per stock ordered by time. A three-column composite index on (user_id, stock_id, executed_at) would allow index-only scans for these aggregations. Without it, every portfolio summary page triggers a sequential scan joined to sorting, which becomes expensive as simulation trading volume grows.

### Enforce learningModules.slug uniqueIndex at the Drizzle level consistently: 📌 Low Impact
slug is marked .unique() inline on the column rather than via the table config uniqueIndex() helper used elsewhere. Inline unique and uniqueIndex() both produce a constraint, but mixing conventions makes it harder to audit indexes in one place. Standardise to the config callback pattern using uniqueIndex('idx_learning_modules_slug').on(t.slug) so all indexes are visible in a single table definition block.

### Parallelize independent admin data fetches with Promise.all: 🔥 High Impact
If the dashboard fires multiple useQuery calls for users, schools, leaderboards, and audit logs sequentially (or the underlying loaders do sequential awaits), consolidate independent fetches into Promise.all on the server loader or use React Query's `useQueries` in parallel. Expected impact: 40-60% reduction in dashboard Time-to-Interactive on slow Caribbean mobile connections.

### Memoize filtered/sorted table datasets with useMemo: ⚡ Medium Impact
Admin tables that filter users or schools by search input likely recompute on every render. Wrapping the filter+sort pipeline in useMemo keyed on the raw data and search string prevents re-computation on unrelated state changes (e.g., chat open/close toggling). For 1000+ student records this is measurably expensive.

### Add a global error boundary around the dashboard: ⚡ Medium Impact
A runtime error in any sub-component (chart data shape mismatch, null user record) currently crashes the entire admin view with no recovery path. Wrapping AdminDashboard in a React ErrorBoundary with a 'Reload' CTA ensures partial failures don't lock admins out of the platform entirely.
