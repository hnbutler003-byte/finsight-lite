## 🔴 Critical (13)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then iterates each one to compute 'spent' via a separate aggregation query per budget row, resulting in N+1 database round-trips for a user with N budgets.

**Fix:** Replace the per-budget loop with a single JOIN or subquery using Drizzle's sql aggregation: SELECT budgets.*, COALESCE(SUM(transactions.amount),0) AS spent FROM budgets LEFT JOIN transactions ON transactions.category_id = budgets.category_id AND transactions.user_id = budgets.user_id GROUP BY budgets.id. Return the result set in one query.

### `server/storage.ts`: getBillReminders
**Issue:** N+1 query pattern: getBillReminders fetches reminders then resolves each reminder's category with a separate SELECT, producing one extra query per reminder row. For a student with 20 reminders this is 21 queries.

**Fix:** Collect all distinct categoryIds from the reminders array, fetch them in a single WHERE category_id IN (...) query using Drizzle's inArray(), then build a Map<id, Category> and attach categories in memory before returning.

### `server/storage.ts`: completeModule / updateUserXp / addUserBadge (called together in handlers)
**Issue:** Multi-step writes that update learning progress, XP, and badges are issued as separate awaits with no wrapping database transaction. A process crash or network error after step 1 leaves the database in a partially updated state — e.g. module marked complete but XP not credited.

**Fix:** Wrap all related mutations in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Pass the tx object to each sub-operation so all writes commit atomically or roll back together.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~lines 41-80)
**Issue:** Host header injection: `req.headers['x-forwarded-host']` and `req.headers.host` are used directly to construct absolute URLs without any validation. An attacker can send a crafted Host header to poison sitemap/robots URLs returned to crawlers or users.

**Fix:** Resolve the base URL once from a trusted environment variable: `const BASE_URL = process.env.BASE_URL` (e.g. 'https://finsightlite.com'). Use that constant in both handlers instead of reconstructing from request headers. Throw or fall back to a safe default if the env var is missing at startup.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~lines 102-104)
**Issue:** The OpenAI-backed chat, image, and audio routes have no rate limiting applied before they are registered, despite auth endpoints having both `authLimiter` and `strictAuthLimiter`. For a school-aged audience these are the most expensive endpoints and are trivially abusable without a per-IP or per-user limit.

**Fix:** Before the three `register*Routes` calls, add `app.use('/api/chat', aiLimiter)`, `app.use('/api/image', aiLimiter)`, `app.use('/api/audio', aiLimiter)` where `aiLimiter` is a rate limiter (e.g. express-rate-limit with max 20 req/min per IP). Export it from `./rateLimiter` alongside the existing limiters.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler payload
**Issue:** Full file contents are stored as base64 in the job payload column (fileB64). For a multi-page PDF or high-res image this can easily be 5–20 MB per job row, bloating the jobs table, slowing queue polling queries, and risking OOM when the worker deserializes many jobs concurrently.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object path in the payload. In runExtractPaper, download the buffer from object storage at the start of execution and delete the object on terminal failure or success. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts`: purge-ai-usage handler (registerJobHandler call)
**Issue:** The purge-ai-usage handler has no try/catch. If the DELETE throws (e.g., DB connection drop), the unhandled rejection will crash the worker process rather than being caught and recorded as a failed job, because the job runner may not wrap individual handlers.

**Fix:** Wrap the handler body in try/catch: catch the error, optionally log it with a structured logger, then re-throw so the job framework can record the failure and schedule a retry. Also validate that olderThanDays is a positive finite number before computing the cutoff to prevent accidental full-table deletes.

### `shared/schema.ts`: simulatedStocks table, ticker column
**Issue:** ticker has no unique constraint. Duplicate tickers (e.g., two 'BISX01' rows) would silently corrupt portfolio calculations and leaderboard P&L comparisons without any DB-level guard.

**Fix:** Replace the plain column with a uniqueIndex in the table callback: `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })`.

### `shared/schema.ts`: portfolioHoldings table
**Issue:** There is no unique constraint on (userId, stockId). A race condition or retry in the buy flow can insert duplicate holding rows for the same user and stock, causing double-counted quantities and wrong P&L without any DB rejection.

**Fix:** Add a unique index: `(t) => ({ userStockUniqueIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) })` and handle upsert logic (INSERT ... ON CONFLICT DO UPDATE) in the application layer.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The intervals created for the weekly digest and AI usage purge schedulers are not stored in variables, making it impossible to call clearInterval() during graceful shutdown. The process will keep these timers alive and they can fire during or after shutdown, causing database writes against a closing connection.

**Fix:** Capture each interval: `const weeklyDigestTimer = setInterval(...)` and `const purgeTimer = setInterval(...)`. In the SIGTERM/SIGINT handler, call `clearInterval(weeklyDigestTimer)` and `clearInterval(purgeTimer)` before closing the HTTP server and database pool.

---

### `server/index.ts`: res.json monkey-patch in request logging middleware
**Issue:** Every API response body is captured into capturedJsonResponse and serialized in full via JSON.stringify into the log line with no length cap, content filtering, or environment guard. For a school-aged user financial app this risks logging session tokens, PII, quiz answers, or AI-generated content verbatim in production logs.

**Fix:** Add a production guard and truncation: in the finish handler, only append the body in development (`if (process.env.NODE_ENV !== 'production' && capturedJsonResponse)`), or at minimum truncate the stringified output to a safe length (e.g., 200 chars) and strip known sensitive keys like `token`, `password`, `secret` before logging.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage
**Issue:** The entire `messages` array is sent to `/api/admin/help-chat` on every request with no length cap. A school-aged user (or a malicious actor) can accumulate thousands of messages in a session, sending an unbounded payload to the LLM endpoint and incurring runaway API costs.

**Fix:** Slice history before sending: `const history = next.slice(-20);` and send `{ messages: history }`. Also enforce a max on the state side: after push, if `next.length > 100` trim the oldest entries so the in-memory array cannot grow unbounded.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage fetch
**Issue:** The response from `/api/admin/help-chat` is consumed with `res.json()` without checking `res.ok`. A 401, 403, or 500 from the server will silently parse an error JSON body as if it were a valid reply, potentially displaying raw server error messages (stack traces, DB details) to admin users.

**Fix:** Add a status check immediately after the fetch: `if (!res.ok) throw new Error(await res.text());` before calling `res.json()`. The existing catch block will then display the user-friendly fallback message instead of leaking server internals.

## 🟡 Warnings (14)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues multiple independent aggregation queries (total income, total expenses, category breakdown, recent transactions, savings goals) sequentially with await, adding latency unnecessarily. Each query is independent.
  → *Replace sequential awaits with Promise.all: const [income, expenses, categoryTotals, recentTxns, goals] = await Promise.all([queryIncome, queryExpenses, queryCategoryTotals, queryRecentTxns, queryGoals]); This reduces wall-clock time from sum-of-latencies to max-of-latencies.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard accepts an optional limit parameter but there is no upper bound guard, allowing a caller to pass limit=1000000 and force a full table scan and massive payload. For a school-facing app with student data this is both a DoS and data-exposure risk.
  → *Add a server-side cap: const safeLimit = Math.min(filters?.limit ?? 20, 100); and use safeLimit in the query. Document the maximum in the IStorage interface signature.*
- **`server/routes.ts` (Supabase init chain (~lines 107-115)):** The sequential `.then()` chain runs seven Supabase operations one-by-one. The `.catch()` only fires for the first rejection that surfaces; if `seedFinancialAcademyLesson` throws, the remaining seeds never run and the error swallows silently after logging. There is also no structured error context per step, making incident diagnosis very hard.
  → *Run independent seed operations with `Promise.allSettled` so each step is attempted regardless of others, and log per-step failures with the step name: `Promise.allSettled([initSupabaseTables(), initStaticContentTables()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] Step', i, r.reason)))`. Split truly sequential dependencies into a smaller chained group only where ordering is required.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~lines 92-100)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject` without any sanitisation. A path traversal payload like `../../private/keys` could reach unintended storage paths depending on how the ObjectStorageService resolves keys.
  → *Sanitise the path before use: `const safePath = path.normalize(filePath).replace(/^\.\.\/|^\//, '')`. Reject requests where the normalised path differs suspiciously from the input or starts with `..`. Import Node's `path` module at the top of the file.*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop (for loop over questions)):** Each extracted question is inserted with a sequential await inside a for loop. For a 60-question paper this serialises 60 round-trips to the DB, adding hundreds of milliseconds of unnecessary latency and holding the job worker slot much longer than needed.
  → *Collect all valid question objects into an array, then execute a single batch insert: await storage.createExtractedQuestions(validQuestions) (or db.insert(extractedQuestions).values(validQuestions) directly). If storage layer does not support batch insert, use Promise.all with a concurrency cap of ~10.*
- **`server/jobHandlers.ts` (runExtractPaper – model name and token limits):** The OpenAI model 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two separate completions.create calls. Changing the model for cost or capability reasons requires a code deploy, and different environments (dev/staging/prod) cannot be configured independently.
  → *Read model and token limits from environment variables: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); Apply to both calls. Document these vars in .env.example.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, so fetching a user's budgets requires a full table scan. As user count grows this will degrade significantly since every budget lookup filters by userId.
  → *Add an index in the table's index callback: `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })`. Also consider a unique index on (userId, categoryId, period) to prevent duplicate budgets per category per period.*
- **`shared/schema.ts` (savingsGoals, billReminders, budgets, linkedCards, documentUploads — createdAt columns):** createdAt is defined as `.defaultNow()` but without `.notNull()`. Drizzle will infer the column as nullable, meaning application code or a raw INSERT omitting createdAt yields a NULL audit timestamp, breaking any time-based sorting or reporting.
  → *Append `.notNull()` to every `createdAt` definition that uses `.defaultNow()`, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. The transactions table already sets this pattern correctly elsewhere.*
- **`shared/schema.ts` (billReminders table, nextDueDate column):** There is no index on (userId, nextDueDate), yet the primary query pattern for bill reminders is 'fetch upcoming bills for a user sorted by due date'. Without this index every reminder dashboard load scans the entire table.
  → *Add `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })` to the billReminders table definition.*
- **`server/index.ts` (seedDatabase / probeDatabase calls in startup IIFE):** seedDatabase and probeDatabase are awaited sequentially during startup. If these are independent operations, each one blocks the next unnecessarily, extending the time before the HTTP server is ready to accept connections.
  → *Replace sequential awaits with Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])`. Ensure neither function has a side-effect dependency on the other before doing so.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The digest schedule time (Sunday 7 PM) and the interval period (currently inferred from the truncated code) are hardcoded. The timezone is also implicit — new Date() uses server local time, which on cloud/Replit infrastructure is UTC, silently shifting the intended Caribbean local-time trigger.
  → *Add env vars `DIGEST_DAY_OF_WEEK` (default 0), `DIGEST_HOUR_LOCAL` (default 19), and `DIGEST_TIMEZONE` (default 'America/Port_of_Spain'). Use a library like `date-fns-tz` or the `Intl` API to compute the target instant in the configured timezone rather than relying on server wall-clock hour.*
- **`server/index.ts` (maybeRunAiUsagePurge and maybeRunWeeklyDigest async schedulers):** Both scheduler functions are invoked by setInterval but the interval callback does not await the async function and does not catch top-level rejections from the invocation itself. If the async function throws before its internal try/catch (e.g., during argument construction), the rejection is unhandled and will crash Node.js 15+ or silently disappear on older versions.
  → *Wrap each interval callback to catch the outer promise: `setInterval(() => { maybeRunWeeklyDigest().catch(e => captureError(e)); }, intervalMs)`. This ensures any unexpected throw is routed to Sentry rather than becoming an unhandled rejection.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new function on every render and calls scrollIntoView on every re-render (including while the user is typing), not just when a new message arrives. This causes jarring scroll-jacking mid-type.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`. This scopes scrolling to message-list changes only.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – handleKey):** `handleKey` and `sendMessage` are defined as plain functions inside the component body with no useCallback. Because `AdminHelpChat` likely re-renders when `input` state changes (on every keystroke), both functions are recreated on every keystroke and passed as new references to the `<input>` and `<button>`, preventing memoization of those elements.
  → *Wrap both with useCallback: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e) => { ... }, [sendMessage]);`. This stabilises references across input keystrokes.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (/api/health and /healthz (~lines 35-37 and 84-91)):** There are two health check endpoints (`/api/health` and `/healthz`) with different behaviour: the first returns immediately with no DB check, the second does a live DB query. This creates confusion for Railway and uptime monitors about which to trust, and the shallow `/api/health` gives a false-positive 200 when the DB is down.
- **`server/jobHandlers.ts` (runExtractPaper – vision path base64 re-encoding):** The image buffer is decoded from fileB64 into a Buffer and then immediately re-encoded back to base64 for the OpenAI vision URL: buffer.toString('base64'). This is a no-op round-trip that wastes CPU and doubles peak memory for the image data in the worker heap.
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – open state toggle):** When the chat panel is closed (`open = false`) the `messages` state is retained in memory indefinitely for the lifetime of the parent page. For an admin session that stays open all day this accumulates chat history and holds closure references to all prior message strings, constituting a soft memory leak with no way for the user to reset.

## 💡 Opportunities

### Centralise pagination guards across all list queries: 🔥 High Impact
getTransactions, getGameSessions, getPortfolioTransactions, and several teacher-dashboard list methods accept optional limit/offset but apply no default or maximum. Adding a shared clampPagination(limit, offset, maxLimit=200) utility and calling it at the top of every list method prevents runaway queries and makes limits auditable in one place.

### Extract hardcoded virtual balance seed value to environment config: ⚡ Medium Impact
The initial virtual balance (commonly 10000.00 or similar) is hardcoded in getVirtualBalance / updateVirtualBalance. Moving it to an environment variable (VIRTUAL_BALANCE_SEED) lets ops adjust the educational starting balance without a code deploy and makes it obvious it is a configurable constant rather than business logic.

### Add structured logging sanitisation before any user-data log statements: 🔥 High Impact
Several storage methods log full user objects or transaction rows during error handling. For a app serving minors under Caribbean data-protection obligations (and GDPR-adjacent frameworks), logs should strip PII fields (name, email, cardNumber) via a redact() helper before writing to stdout, preventing accidental exposure in log aggregation services.

### Parallel domain-route registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel `Promise.all` import are still sequential. If any route registrar does async work (DB introspection, schema validation), they add latency in series. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` assuming they don't share mutable middleware state. Expected impact: reduced cold-start time proportional to the slowest individual registrar.

### Structured startup logging with per-step timing: ⚡ Medium Impact
The Supabase init chain and route registration emit no timing or success logs on the happy path. Adding `console.info('[boot] supabase seeds complete in Xms')` and similar markers per phase would dramatically reduce MTTR for deployment issues in a production Caribbean school environment where on-call context is limited.

### BASE_URL environment config centralisation: 🔥 High Impact
Beyond fixing the header injection, centralising BASE_URL in a validated config module (checked at process startup) would eliminate three separate inline reconstructions of proto+host across robots.txt, sitemap, and any future handlers. A single `config.ts` that throws on missing required env vars at boot is the standard pattern and prevents silent misconfiguration in Railway deployments.

### Idempotency token on extracted-question batch: 🔥 High Impact
The current idempotency strategy deletes all questions for the paper and re-inserts on every retry. Under high concurrency two worker instances retrying the same job could race on the delete+insert, causing partial data. Adding a DB-level unique constraint on (paperId, order) and using INSERT ... ON CONFLICT DO NOTHING would make the operation safe without the delete step, removing the race window entirely.

### Structured logging instead of throw-only error propagation: ⚡ Medium Impact
Several catch blocks silently swallow errors (catch { /* ignore */ }) or only re-throw without logging context. Integrating a structured logger (e.g., pino) with job ID, attempt number, and paperId at each catch site would dramatically reduce mean time to diagnose failures in production, especially for school admins uploading papers.

### Job payload schema validation at enqueue time: ⚡ Medium Impact
Payloads are typed via TypeScript generics but are not validated at runtime before being persisted to the job table. Adding a Zod schema per job type and parsing the payload inside each registerJobHandler at the top of the handler would catch malformed jobs (e.g., missing paperId, NaN olderThanDays) immediately on dequeue rather than mid-execution, preventing partial side-effects.

### Centralise default currency as a DB-level enum: ⚡ Medium Impact
The string literal 'BSD' is hardcoded as a default in at least six separate columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Introducing a Postgres enum type for currency via Drizzle's `pgEnum` removes the risk of silent typos like 'BDS' and makes adding a second supported currency (TTD, JMD) a single-location change.

### Add composite index for leaderboard queries on portfolioTransactions: 🔥 High Impact
Leaderboard features typically aggregate portfolio performance per user ordered by total return. Adding `index('idx_portfolio_tx_user_executed').on(t.userId, t.executedAt)` on portfolioTransactions would turn those aggregation scans into efficient index range scans, directly improving leaderboard load time as transaction volume grows.

### Replace serial PKs with UUIDs for user-facing tables: ⚡ Medium Impact
Using sequential integer IDs (serial) for tables like savingsGoals and billReminders exposes record counts and makes IDs guessable — a concern for a school-aged audience where IDOR enumeration is trivial. Switching to `uuid` PKs with `gen_random_uuid()` defaults eliminates enumeration risk with no application logic change beyond the schema migration.

### Centralized graceful shutdown coordinator: 🔥 High Impact
Currently SIGTERM/SIGINT handling is likely ad-hoc. Introduce a single shutdown() function that: (1) clears all interval handles, (2) stops accepting new HTTP connections via server.close(), (3) drains in-flight job worker, (4) closes the DB pool. This eliminates the race between active timers and a closing DB connection and is required for zero-error Kubernetes/Replit restarts.

### Replace per-request res.json monkey-patch with structured logging middleware: ⚡ Medium Impact
The current approach of overriding res.json on every request object adds overhead and is fragile (breaks if any downstream middleware also wraps res.json). Replace with a response-interceptor pattern using a write stream hook or adopt a structured logger (pino with pino-http) that captures status codes and safe metadata without touching the response body, reducing both CPU overhead and log-leak risk.

### Extract scheduler interval durations to named constants with env overrides: 📌 Low Impact
Magic interval millisecond values scattered through the startup IIFE make it hard to tune check frequency under load or in tests. Define named constants (e.g., WEEKLY_DIGEST_CHECK_INTERVAL_MS, PURGE_CHECK_INTERVAL_MS) parsed from env vars with safe defaults, and document the intended Caribbean-timezone semantics alongside them. This also enables faster CI test cycles by setting short intervals in test env.

### Split monolithic dashboard into route-level chunks: 🔥 High Impact
At 147k+ characters this is almost certainly one file containing many tab/section components. Splitting into separate files with React.lazy() + Suspense would reduce initial bundle parse time and scope re-renders to the active tab, meaningfully improving TTI for admin users on slower Caribbean mobile connections.

### Centralise API base URL in environment config: ⚡ Medium Impact
Endpoint strings like '/api/admin/help-chat' appear hardcoded inline. Moving these to a typed constants file (or VITE_API_BASE env var) prevents silent breakage when the API is deployed behind a path prefix and makes endpoint auditing trivial during security reviews.

### Add React Query for help-chat to deduplicate in-flight requests: ⚡ Medium Impact
The manual fetch + loading state in sendMessage can be replaced with a useMutation from @tanstack/react-query (already imported). This gives automatic deduplication, retry config, and integrates with the existing QueryClient's error boundary, removing ~20 lines of hand-rolled state management.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits several reliability and security concerns including N+1 query patterns in budget and bill reminder lookups, missing database transactions around multi-step writes, and sequential awaits that should be parallelised. The file is large and handles sensitive financial data for minors, making these gaps higher risk than in a typical app.

### API Routes (`server/routes.ts`)
The routes file is generally well-structured with good patterns like parallel module loading and rate limiting, but has three meaningful reliability and security gaps. The most critical issues are unhandled promise rejections from fire-and-forget Supabase init, a host header injection vector in robots.txt/sitemap, and missing rate limiting on the AI/chat/image/audio routes which are the heaviest endpoints in a school-aged user context.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and security gaps including large base64 payloads stored directly in the job table, sequential DB inserts that should be batched, missing try/catch on the AI usage purge handler, and a hardcoded model name that should be environment-configurable. The extract-paper handler is particularly risky as it stores full file contents in the job payload, creating memory and storage pressure at scale.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and basic indexes present, but has several reliability and performance gaps. Key concerns are missing indexes on high-cardinality filter columns, nullable createdAt timestamps that should be guaranteed, and a missing unique constraint on the stock ticker that could allow duplicate simulation assets.

### Server Startup (`server/index.ts`)
The server startup sequence contains several reliability and security gaps: blocking synchronous-style sequential awaits delay startup, intervals created during initialization lack cleanup handles preventing graceful shutdown, and full JSON response bodies are logged unconditionally which risks leaking sensitive financial or PII data in production logs. Overall structure is sound but the scheduler and shutdown path need hardening.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including a scroll ref pattern that creates a new function on every render, missing auth checks on the help-chat endpoint consumer, and unguarded async flows. The file is extremely large (147k+ chars truncated) suggesting significant component splitting opportunities that would also resolve re-render performance issues.
