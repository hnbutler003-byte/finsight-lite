## 🔴 Critical (6)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These hit OpenAI APIs which are expensive and slow. A malicious or runaway client can exhaust API quotas or cause cost spikes with no throttle in place.

**Fix:** Add dedicated rate limiters before these registrations, e.g.: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); where aiLimiter is a new rate-limit config (e.g. 20 req/min per IP) defined in rateLimiter.ts alongside the existing auth limiters.

### `server/routes.ts` — /public-objects/:filePath(*) handler
**Issue:** The filePath parameter is passed directly to objectStorage.searchPublicObject() without sanitisation. A request to /public-objects/../../secrets/env could traverse outside the intended public prefix depending on how ObjectStorageService resolves paths, leaking private objects.

**Fix:** Sanitise the path immediately after extraction: const filePath = req.params.filePath.replace(/(\.\.\/|\.\.\\/)/g, ''); and additionally verify the resolved key starts with the expected prefix (e.g. 'public/') before passing to the storage service. Throw a 400 if validation fails.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full file contents are stored as a base64 string inside the jobs table payload. A single uploaded PDF or image can easily be several MB, multiplied by retry attempts. This bloats the job table, slows queue polling queries, and risks hitting row-size limits depending on the DB engine.

**Fix:** Upload the file to private object storage immediately on receipt (before enqueueing), store only the resulting object path in the job payload, then download it inside runExtractPaper with objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

---

### `server/jobHandlers.ts` — admin-csv-export handler (type AdminRow = Record<string, unknown>)
**Issue:** The admin CSV export job has no visible authorization re-check inside the handler itself. Job payloads are enqueued by an HTTP route, but if the job queue is ever triggered by an internal scheduler or replayed, the handler will execute without verifying the requesting user still has admin privileges, potentially exporting sensitive user/transaction data to an attacker-controlled path.

**Fix:** Add an explicit privilege check at the top of the handler: look up the requestingUserId stored in the payload, query the users table to confirm the role is still 'admin', and throw an Error('Unauthorized') if not. This makes the job idempotent-safe against privilege revocation between enqueue and execution.

---

### `shared/schema.ts` — portfolioHoldings table definition
**Issue:** No unique constraint on (userId, stockId) in portfolioHoldings. Nothing prevents two rows for the same user and stock, which would double-count holdings in portfolio value calculations and corrupt P&L logic without any DB-level protection.

**Fix:** Add (t) => ({ userStockUnique: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) so duplicate holdings are rejected at the database layer, not just handled defensively in application code.

### `client/src/pages/AdminDashboard.tsx` — TABS array — id: 'dbviewer'
**Issue:** A 'DB Viewer' tab is exposed in the admin UI. If the corresponding API endpoint does not enforce server-side admin role verification independently of the client-side useAdminAuth hook, any user who bypasses the React routing layer can query raw database tables. Client-side auth checks are not a security boundary.

**Fix:** Verify that /api/admin/db-viewer (or equivalent) validates the session token and admin role server-side on every request, returning 403 for non-admins regardless of how the request originated. Add an integration test that calls the endpoint without admin credentials and asserts a 401/403 response. On the client, this tab should also only render if the admin auth check has resolved successfully.

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...) chain):** The sequential .then() chain means a failure in any intermediate step (e.g. seedFinancialAcademyLesson) rejects the whole chain and skips all subsequent seeds, with only a single catch log. Steps are also run sequentially when most are independent, adding unnecessary startup latency.
  → *Replace with Promise.allSettled for independent seeds so one failure does not block others, and log each rejection individually: const results = await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]); results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] Seed step failed:', r.reason)); Keep initSupabaseTables and initStaticContentTables as sequential prerequisites if they create tables the seeds depend on.*
- **`server/routes.ts` (/api/health and /healthz):** Two separate health check endpoints exist (/api/health and /healthz) with different behaviours — /api/health returns immediately with no DB check while /healthz does a real DB ping. This creates confusion: Railway or uptime monitors pointed at /api/health will report healthy even when the database is down.
  → *Consolidate to a single endpoint (keep /healthz with the DB check, remove /api/health or make it redirect to /healthz). If a lightweight liveness probe is needed separately from a readiness probe, document the distinction explicitly and ensure the load balancer/Railway config uses the DB-checking endpoint for readiness.*
- **`server/routes.ts` (sitemap.xml handler):** req.headers['x-forwarded-proto'] and req.headers['x-forwarded-host'] are used without validation to construct absolute URLs in the sitemap. An attacker can send spoofed host headers causing the sitemap to emit arbitrary URLs (host header injection), which can poison SEO crawlers or be used for cache poisoning.
  → *Resolve the base URL from a trusted environment variable instead: const base = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com'; Remove the dynamic header-based construction entirely. Set PUBLIC_BASE_URL in Railway/deployment config.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop with sequential await storage.createExtractedQuestion):** Each extracted question is inserted one at a time inside a for loop. For a 40-question paper this means 40 sequential round-trips to the database, adding hundreds of milliseconds of latency and holding the job worker busy unnecessarily.
  → *Replace the for loop with a single bulk insert: filter valid questions first, then call db.insert(extractedQuestions).values(validRows) where validRows is the mapped array. This collapses 40 round-trips into one and is naturally atomic, removing the need for the pre-attempt delete if wrapped in a transaction.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (both vision and extraction)):** max_completion_tokens is hardcoded to 4096 in both OpenAI calls. This is a tuning parameter that differs between environments (dev quota limits vs production) and should not require a code change to adjust.
  → *Read from environment: const maxTokens = parseInt(process.env.OPENAI_MAX_COMPLETION_TOKENS ?? '4096', 10) and use maxTokens in both create() calls. Add OPENAI_MAX_COMPLETION_TOKENS to the documented env var list.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** The PDF text is silently truncated to 30,000 characters with no log or warning. For a long exam paper this can cause questions near the end to be silently dropped, and the caller receives a questionCount that looks normal but is incomplete. There is also no guard on the vision path where extracted text could similarly be very large.
  → *After slicing, if pdfData.text.length > 30000 emit a structured warning log (include paperId and original length). Consider storing the truncation flag on the exam_papers row so the UI can surface a 'paper may be incomplete' notice. Also apply a consistent cap to the vision extractedText path.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets.userId even though every budget query will filter by userId. At scale this becomes a full table scan per user dashboard load.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also add a unique index on (userId, categoryId, period) to enforce one budget per category per period at the DB level instead of relying on application logic.*
- **`shared/schema.ts` (varchar user_id columns across all tables (transactions, budgets, linkedCards, etc.)):** varchar('user_id') with no length argument maps to unlimited varchar in Postgres, which is functionally text with no length enforcement. If the auth provider issues fixed-length IDs (e.g., Clerk's 32-char user_xxxxx), an unbounded varchar wastes storage statistics and skips a cheap sanity constraint.
  → *Change to varchar('user_id', { length: 64 }) across all tables to match your auth provider's maximum ID length. This also makes index pages smaller and improves query planner estimates.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt (and all other createdAt columns)):** createdAt uses .defaultNow() but is not marked .notNull(). A row inserted with an explicit NULL for createdAt will store NULL, making time-based auditing and sorting unreliable — especially critical for a financial app with regulatory audit trail needs.
  → *Append .notNull() to every createdAt column: createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (billReminders table definition):** No index on billReminders.nextDueDate or (userId, nextDueDate). A cron job or dashboard query fetching upcoming due bills for all users will scan the entire table, and this column is the primary filter for reminder notifications.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the table definition. If you query overdue bills globally for notifications, also add a standalone index on nextDueDate alone.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo, causing a new object reference each render. Because this object is used to build the fetch URL inline rather than being part of the queryKey, stale param combinations can be silently used if React batches state updates, and the construction work is repeated unnecessarily.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return p; }, [actorType, action, from, to]);` This ensures the URL is stable between renders and keeps it consistent with the queryKey dependencies.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** The useQuery call has no isLoading or isError handling. If the /api/admin/observability endpoint is slow or fails, the card silently renders with undefined data, and optional chaining (data?.sentry) suppresses errors without any user feedback. For an admin observability card this is particularly misleading — a failed fetch looks identical to a healthy 'all disabled' state.
  → *Destructure `isLoading` and `isError` from useQuery and render a skeleton or error badge: `if (isLoading) return <Card>...</Card> with a Loader2 spinner; if (isError) return a Card with an AlertTriangle and 'Failed to load observability status' message. This prevents silent false-negatives on the health status display.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** `fmtDate` accepts `any` and calls `new Date(d)` with no validation. Passing an already-invalid date string (e.g. a malformed ISO string from the API) silently produces 'Invalid Date' rendered to the UI. There is no try/catch, and the function is used across multiple entity tables where bad data from the DB would produce visible corruption.
  → *Add a guard: `const dt = new Date(d); if (isNaN(dt.getTime())) return '—';` before calling toLocaleDateString. Also type the parameter as `string | number | Date | null | undefined` rather than `any` so TypeScript can catch misuse at call sites.*

## 💡 Opportunities

### Centralise AI endpoint rate limiting as reusable middleware — 🔥 High Impact
Define aiLimiter in rateLimiter.ts (e.g. 20 requests/minute per IP with a clear error message referencing the free-tier limit) and apply it once via app.use('/api/ai', aiLimiter) rather than per-registration. This makes future AI routes automatically protected and keeps the policy in one place, reducing the chance of a new route being added without rate limiting.

### Move Supabase init outside registerRoutes into server startup — ⚡ Medium Impact
Running DB seeding inside registerRoutes couples HTTP route setup with data initialisation. If seeding is slow it delays the server becoming ready. Moving it to the top-level server startup script (e.g. index.ts) before the HTTP server begins listening allows proper sequencing, cleaner error handling, and makes registerRoutes pure (only HTTP concerns).

### Add structured logging with request IDs to error handlers — ⚡ Medium Impact
Current error paths use console.error with unstructured strings. Adding a correlation request ID (e.g. via crypto.randomUUID() set on res.locals in a middleware) and emitting structured JSON logs (or forwarding to the existing Sentry setup) would make production debugging dramatically faster, especially for the object storage and DB-health errors which currently lose context.

### Parallelize independent DB reads in reporting/export handlers — ⚡ Medium Impact
Any handler that fetches multiple independent tables sequentially (e.g., users then transactions then badges) should use Promise.all(). Based on the truncated export handler pattern this is likely present. Converting three sequential awaits to Promise.all cuts handler wall-clock time by roughly 60-70% for average DB latency of ~20ms each.

### Introduce a job payload size validation gate at enqueue time — 🔥 High Impact
Add a middleware step inside enqueueJob that checks JSON.stringify(payload).length against a configurable MAX_JOB_PAYLOAD_BYTES (e.g. 64 KB) and throws synchronously before the row is inserted. This catches the fileB64 class of bug at the source for all future job types, not just extract-paper.

### Wrap bulk question insert and paper status update in a single transaction — ⚡ Medium Impact
Currently the bulk insert and the subsequent updateExamPaper are two separate DB operations. A crash between them leaves the paper in 'processing' status with questions already written. Wrapping both in db.transaction() makes the handler fully atomic and eliminates the pre-attempt delete workaround for idempotency.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least 6 separate table columns (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, portfolioHoldings). Extract const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' and reference it in each column definition. When FinSight expands to TTD or JMD markets, one env change covers the entire schema rather than a multi-file migration.

### Add partial index on transactions for recent-record queries — 🔥 High Impact
Dashboard and reporting queries almost always filter to the last 30-90 days. A partial index such as CREATE INDEX idx_transactions_recent ON transactions(user_id, date) WHERE date > NOW() - INTERVAL '90 days' would be dramatically smaller and faster than the full composite index for the common case. Drizzle supports sql`` expressions in index definitions to achieve this.

### Replace text columns with enums for period and frequency fields — ⚡ Medium Impact
budgets.period and billReminders.frequency are constrained by application-level enums but stored as plain text, so a direct DB insert or migration script can silently store invalid values. Promote these to Postgres native enums using pgEnum() from drizzle-orm/pg-core. This halves storage per row, enables DB-level constraint enforcement, and makes pg_dump output self-documenting for auditors.

### Parallel data fetching on Dashboard overview tab — 🔥 High Impact
The overview tab likely fires multiple sequential useQuery calls (users, schools, challenges, sponsors) that are independent of each other. Ensure all overview queries are initiated in the same render cycle so React Query fires them in parallel. If any are inside conditional renders or sequential effects, restructure to unconditional hooks with enabled flags. This can cut initial dashboard load time by 2-4x depending on API latency.

### Virtualize large admin tables — 🔥 High Impact
Rendering 200 audit log rows plus full entity tables for students, teachers, and classes as plain DOM nodes creates significant layout/paint cost. Integrate TanStack Virtual (already likely available via TanStack Query) to render only visible rows. Expected impact: 60-80% reduction in time-to-interactive for tables with >50 rows, and elimination of scroll jank on lower-end school devices.

### Extract tab panels into separate lazy-loaded components — ⚡ Medium Impact
With 13 tabs in a single 126k+ character file, the entire component tree — including Recharts, all form state, and all query subscriptions — is parsed and initialized on first load regardless of which tab is active. Split each tab panel into its own file and use React.lazy + Suspense. This reduces the initial JS parse budget and means query subscriptions only mount when their tab is visited.
