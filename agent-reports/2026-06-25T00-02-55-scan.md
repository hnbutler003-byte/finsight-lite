## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate COUNT/SUM query per budget row to calculate 'spent', resulting in 1+N database round-trips for N budgets.

**Fix:** Replace the per-budget loop with a single LEFT JOIN query using Drizzle's aggregation: join transactions on categoryId and dateRange in one query, grouping by budgetId to compute spent in SQL. Example: db.select({ ...budgets, spent: sql<number>`coalesce(sum(${transactions.amount}),0)` }).from(budgets).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).groupBy(budgets.id).

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional limit but the default is unbounded (no server-side cap), so a call with no limit can return every game session row joined to users, potentially scanning the entire table.

**Fix:** Add a hardcoded maximum: const safeLimit = Math.min(filters?.limit ?? 25, 100); and apply .limit(safeLimit) unconditionally in the query. Never trust the caller to supply a limit for a leaderboard endpoint.

### `server/storage.ts`: completeModule
**Issue:** Missing database transaction on multi-step write: completeModule inserts a userLearningProgress row and then calls updateUserXp, but these two writes are not wrapped in a db.transaction(), so a crash between them leaves XP unawarded with a completed-module record already committed.

**Fix:** Wrap both operations in db.transaction(async (tx) => { await tx.insert(userLearningProgress)...; await tx.update(userXp)...; }) so they commit or roll back atomically.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI/image/audio routes registered with no rate limiting middleware applied before them. These call external paid APIs (OpenAI) and are the most expensive endpoints in the app. Any authenticated or unauthenticated user can hammer them, causing runaway API costs and denial-of-service for other users.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Verify the exact path prefixes those helpers mount on.

### `server/routes.ts`: robots.txt handler (~line 40) and sitemap.xml handler (~line 56)
**Issue:** The base URL is constructed directly from user-controlled request headers (x-forwarded-host, host) without any validation or allowlist. An attacker can send a crafted Host header and the response will reflect an arbitrary domain in the Sitemap directive and all <loc> tags — a classic Host Header Injection usable for cache poisoning or phishing via cached sitemaps.

**Fix:** Read the canonical origin from an environment variable: const base = process.env.SITE_URL || 'https://finsightlite.com'; Remove all header-derived base URL logic from both handlers. SITE_URL should be set in Railway/Replit env config.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Large base64 file content (fileB64) is stored directly in the job payload, bloating the jobs table. A PDF or image could easily be several MB encoded as base64, causing table bloat, slow job queries, and memory pressure when the worker deserializes the row.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object storage path in the payload (e.g., { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. This keeps job rows small and makes retries cheaper.

### `shared/schema.ts`: simulatedStocks table — ticker column
**Issue:** ticker has no unique constraint. Duplicate tickers (e.g. two rows with ticker 'BTC') would produce ambiguous join results in portfolioHoldings and portfolioTransactions and silently corrupt portfolio value calculations.

**Fix:** Replace the plain column definition with ticker: text('ticker').notNull() and add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) in the table's index callback.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / top-level dashboard query setup
**Issue:** The component imports useAdminAuth but the truncated body almost certainly fires useQuery calls (for users, schools, analytics) regardless of whether the auth check has resolved. If useAdminAuth returns a loading state and the queries run concurrently, unauthenticated requests reach the server and the UI may flash partial data before the redirect fires.

**Fix:** Gate all useQuery calls with `enabled: !!adminUser` (where adminUser comes from useAdminAuth). Add a top-level early return that renders a spinner while auth is resolving, before any query hooks are called — or move queries inside a child component that is only mounted post-auth.

## 🟡 Warnings (14)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits that could be parallel: getDashboardStats issues getTransactions, getBudgets, and getSavingsGoals sequentially with individual awaits, adding latency equal to the sum of all three query times on every dashboard load.
  → *Replace sequential awaits with const [txns, budgets, goals] = await Promise.all([getTransactions(userId, filters), getBudgets(userId), getSavingsGoals(userId)]); These queries are independent and can safely run concurrently.*
- **`server/storage.ts` (getTransactions):** Missing pagination guard: getTransactions accepts an optional limit but no maximum cap or default, meaning callers can omit it and retrieve the user's full transaction history in one query, which grows unboundedly over time.
  → *Apply a default and ceiling: const resolvedLimit = Math.min(filters?.limit ?? 50, 500); and always call .limit(resolvedLimit) in the query. Callers needing full exports should use a dedicated paginated export endpoint.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97)):** The filePath parameter from the URL is passed directly into objectStorage.searchPublicObject() without any sanitisation. A path traversal sequence like ../../private/secrets could resolve to objects outside the intended public prefix depending on how searchPublicObject is implemented.
  → *Sanitise filePath before use: const safePath = path.normalize(filePath).replace(/^\.\.\/|^\//, ''); then assert safePath === filePath or throw a 400. Alternatively, strip leading slashes and reject any path containing '..' before passing to the storage layer.*
- **`server/routes.ts` (Supabase init chain (~line 118-126)):** The seven Supabase init/seed functions are chained sequentially with .then(), so they execute one after another. If any step takes 500 ms (common for cold Supabase connections), total startup delay is 3-4 seconds of unnecessary serialisation. More critically, there is no visibility into which individual step failed because the single .catch() swallows the step name.
  → *Group independent operations with Promise.all and log the failing step name: Promise.all([initSupabaseTables(), initStaticContentTables()]).then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error at step:', e?.message, e));*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop (for loop over questions)):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this is 40 serial round-trips to the database, each waiting on the previous. This dramatically increases job duration and holds the DB connection longer than necessary.
  → *Collect all valid questions into an array and issue a single batched insert: await db.insert(extractedQuestions).values(validQuestions). Add the idempotency delete before this block as it already exists, so retry safety is maintained.*
- **`server/jobHandlers.ts` (runExtractPaper – openai model and token constants):** The OpenAI model name 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two separate completions calls. If the model is deprecated or token limits need adjustment for cost control, every call site must be edited manually and redeployed.
  → *Read these from environment variables with safe defaults: const AI_MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o'; const AI_MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10); Apply these constants to all openai.chat.completions.create calls in this file.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch and no validation that olderThanDays is a positive finite number. A malformed payload with olderThanDays = 0 or a negative value would compute a future cutoff date and silently delete no rows or — depending on DB behaviour — all rows. There is also no minimum floor guard against wiping recent data.
  → *Add input validation at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); Then wrap the delete in try/catch and return a structured error result rather than letting the worker catch an unhandled rejection with no context.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Any query filtering budgets by user (the most common access pattern) will do a full table scan as the table grows.
  → *Add a composite index in the pgTable third argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). A single-column index on userId alone would also serve range queries.*
- **`shared/schema.ts` (billReminders table definition):** billReminders.nextDueDate has no index. Reminder-due queries (e.g. WHERE next_due_date <= NOW()) will full-scan the table, and this column is almost certainly used in scheduled jobs or dashboard lookups.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) so queries filtering by user and upcoming due date use the index.*
- **`shared/schema.ts` (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — createdAt columns):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). Drizzle will not emit NOT NULL in the DDL, so the column is nullable in the database even though the application always expects a value. A direct INSERT bypassing Drizzle could store NULL and break ORDER BY createdAt DESC queries silently.
  → *Append .notNull() to every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default still applies so existing insert code requires no changes.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions — userId + stockId):** Neither table has an index on (userId, stockId). Portfolio value calculations join or filter on both columns for every authenticated request. Without an index, each lookup scans the full table.
  → *Add (t) => ({ userStockIdx: index('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) to portfolioHoldings, and an equivalent index on portfolioTransactions. For portfolioHoldings consider a unique index on (userId, stockId) to enforce one holding row per user per stock at the DB level.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - sendMessage function):** The /api/admin/help-chat POST sends the full conversation history (next: ChatMessage[]) to the server on every message. With no server-side token limit or client-side trim, a long session will eventually send thousands of tokens, blow past model context limits, and silently fail or return garbage. There is also no max-length guard on the input field.
  → *Slice the messages array to the last N turns before sending: `const payload = next.slice(-20)`. Add `maxLength={500}` to the input element. On the server, enforce a hard token cap before forwarding to the LLM.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - scroll anchor ref callback (inline `ref={el => { if (el) el.scrollIntoView(...) }}`)):** The inline ref callback is a new function reference on every render. React will call it with null then the element on every re-render (including each keystroke in the input and each loading state toggle), causing repeated scrollIntoView calls that fight user scroll position and produce jank.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading])`. Assign `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function in AdminHelpChat):** res.json() is called without checking res.ok first. A 401, 429, or 500 from the server returns a JSON error body that gets treated as a valid reply, so the assistant may echo a raw server error object or an auth error message to the user as if it were a legitimate assistant response.
  → *After `const res = await apiRequest(...)`, add `if (!res.ok) throw new Error(res.statusText)` before calling `res.json()`. The existing catch block will then surface the friendly fallback message instead of leaking server internals.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint dynamically imports './db' and 'drizzle-orm' on every request. Dynamic imports are cached by Node after the first call, but the overhead of import() resolution on a hot path used by Railway health checks and uptime monitors every 30 seconds adds unnecessary async work and makes cold-path errors harder to surface at startup.
- **`server/jobHandlers.ts` (runExtractPaper – vision branch base64 re-encoding):** In the image/vision branch, buffer is created from fileB64 via Buffer.from(fileB64, 'base64'), then immediately re-encoded back to base64 with buffer.toString('base64') to build the data URL. This is a redundant decode-then-encode cycle that wastes CPU and doubles peak memory usage for large images.
- **`client/src/pages/AdminDashboard.tsx` (ChatMessage messages state — messages.map((m, i) => ... key={i})):** Using array index as React key for chat messages means React cannot correctly reconcile when messages are removed or reordered (e.g. if error messages are spliced in). If the same error fallback is appended twice, both get key collisions and only one renders.

## 💡 Opportunities

### Batch category hydration for bill reminders: 🔥 High Impact
getBillReminders appears to fetch reminders then look up each category individually. Collect all distinct categoryIds from the result set, fetch them in one inArray query, then do an in-memory map join. This reduces database round-trips from 1+N to exactly 2 regardless of reminder count.

### Centralise pagination defaults as named constants: ⚡ Medium Impact
Default and maximum page sizes are currently implicit or absent across at least five list methods. Defining DEFAULT_PAGE_SIZE=50 and MAX_PAGE_SIZE=500 in a shared config module and referencing them everywhere eliminates the risk of one method silently returning thousands of rows while another caps at 25.

### Add query-level result limits to leaderboard and getAllQuestions before auth layer grows: ⚡ Medium Impact
getAllQuestions with a subject filter performs a full table scan with no row cap. With a growing Caribbean school user base, adding .limit() now alongside a cursor-based pagination token prevents this from becoming a production incident later without requiring an API version bump.

### Centralised auth guard middleware: 🔥 High Impact
registerChatRoutes, registerImageRoutes, and registerAudioRoutes likely contain per-handler auth checks. Apply a single requireAuth middleware at app.use('/api/chat'), app.use('/api/image'), app.use('/api/audio') before registration so no handler can accidentally be left unauthenticated. Given this is a school-aged user app, enforcing authentication at the router level also prevents anonymous data leakage.

### Structured startup logging with step timing: ⚡ Medium Impact
Wrapping the Supabase init block and the Promise.all domain router imports with console.time / console.timeEnd (or a structured logger) would expose which seed step or which domain router is slow on cold start. On Railway free tier, cold starts are common; this data would let you defer non-critical seeds (e.g. seedGameContent) to a background job instead of blocking route registration.

### Cache-Control header on /public-objects responses: ⚡ Medium Impact
The objectStorage.downloadObject call already accepts a TTL (2592000 s = 30 days) but it is unclear if it sets Cache-Control on the HTTP response or only controls internal storage. Adding res.setHeader('Cache-Control', 'public, max-age=2592000, immutable') before downloadObject ensures CDN and browser caching, reducing object storage egress costs which matter on a Caribbean educational app with constrained budgets.

### Parallel OpenAI calls where extraction is independent: ⚡ Medium Impact
If future handlers need to call OpenAI for classification and extraction independently (e.g., subject detection + question extraction on separate pre-split text chunks), wrapping them in Promise.all would halve latency. Even now, structuring the two sequential completions calls as a pipeline with explicit dependency documentation will make parallelisation easier when the model supports streaming batch.

### Job payload schema validation at handler entry: 🔥 High Impact
No handler validates its payload shape before using destructured fields. Adding a lightweight Zod schema parse at the top of each registerJobHandler callback would catch malformed enqueue calls at job start rather than mid-execution, producing clearer error messages and preventing partial side-effects (e.g., partial DB deletes before a crash on undefined field access).

### Centralised OpenAI client with usage logging: ⚡ Medium Impact
openai is imported from a chat routes file, coupling job handlers to HTTP route internals. Extracting a shared openai.ts singleton that also records token usage to aiUsageEvents after every completion call would give accurate cost tracking across all job types without duplicating logging logic in each handler.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a column default in at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards area). Extract const DEFAULT_CURRENCY = 'BSD' in a shared config and reference it. When the app adds multi-currency support, there is currently one place to change per table rather than one place total.

### Add a partial index on documentUploads.status for processing rows: ⚡ Medium Impact
Background jobs polling for documents WHERE status = 'processing' will scan all historical completed/failed rows. A partial index — CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing' — keeps the working set tiny and speeds up the polling query significantly as upload volume grows.

### Replace serial PKs with uuid on user-facing tables: 🔥 High Impact
Serial integer PKs on transactions, savingsGoals, and similar tables expose record counts and enable enumeration attacks (GET /transactions/1, /2, …). Switching to uuid (pgTable with uuid defaultRandom()) eliminates enumeration risk without requiring application-layer obfuscation, which is important given the school-aged user base and the sensitivity of financial data.

### Parallel admin data fetching with Promise.all: 🔥 High Impact
If the dashboard fetches users, schools, analytics, and audit logs in sequential useQuery calls that each await independently, converting them to run in parallel (or ensuring all useQuery calls are at the same render level with no serial dependency) will cut initial load time proportionally to the number of endpoints — likely 2-4x faster cold load.

### Memoize expensive chart data transformations: ⚡ Medium Impact
BarChart and LineChart data arrays are almost certainly derived inline from raw query results. Wrapping those derivations in useMemo keyed on the query data will prevent Recharts from re-rendering on every parent state change (e.g. search input keystrokes), which is especially important on low-end devices common in Caribbean school labs.

### Extract AdminHelpChat into its own file with an error boundary: ⚡ Medium Impact
AdminHelpChat is a self-contained feature with its own state and async logic. Moving it to a dedicated file and wrapping it in a React ErrorBoundary means a crash in the chat widget cannot unmount the entire AdminDashboard, improving overall reliability for admins mid-task.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits several reliability and performance issues including N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that block unnecessarily. For a school-aged financial app, the absence of pagination guards on unbounded list queries and missing try/catch on async operations are the most operationally dangerous concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel imports and rate limiting on auth routes. Key concerns are unprotected AI/chat/audio/image routes with no rate limiting, a trust-all host header pattern enabling potential Host header injection in robots.txt and sitemap, and the Supabase init chain running sequentially when it could be parallelised.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several meaningful reliability and performance gaps: large base64 payloads are stored directly in the job table, sequential DB inserts inside a loop should be batched, and hardcoded model names and token limits should be environment-configurable. Error handling is partially present but inconsistent across handlers, and sensitive data risks exist in log-adjacent code paths.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-cardinality filter columns, nullable createdAt timestamps that should always be set, and missing NOT NULL constraints on columns that are semantically required.
