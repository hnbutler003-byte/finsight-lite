## 🔴 Critical (9)

### `server/storage.ts` — getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the category and a separate aggregation query computes `spent` from transactions. For a class of 30 students each with 10 budgets, this is 600+ queries per dashboard load.

**Fix:** Rewrite as a single query using Drizzle's `leftJoin` on categories and a correlated subquery or `groupBy` aggregate for spent amounts: `db.select({ ...budgets, categoryName: categories.name, spent: sql<number>`COALESCE(SUM(t.amount),0)` }).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).where(eq(budgets.userId, userId)).groupBy(budgets.id, categories.id)`

### `server/storage.ts` — updateVirtualBalance / createPortfolioTransaction / upsertPortfolioHolding
**Issue:** Stock buy/sell flow updates virtual balance, creates a portfolio transaction, and upserts a holding in three separate awaits with no wrapping database transaction. A process crash or network drop between any two steps leaves the user's financial state permanently inconsistent — e.g., balance debited but holding never recorded.

**Fix:** Wrap all three operations in `db.transaction(async (tx) => { ... })` passing the `tx` client to each sub-operation. Drizzle's transaction callback rolls back all changes automatically on throw, ensuring atomicity for every trade.

### `server/routes.ts` — AI tutor/chat route handlers (registerChatRoutes and inline AI endpoints)
**Issue:** AI endpoints calling OpenAI lack per-user/per-org rate limiting beyond the quota reservation system. If quotaErrorMessage or reserveQuota throws or returns unexpectedly, the OpenAI call may still proceed, and there is no HTTP-level rate limiter middleware (like the authLimiter applied to auth routes) guarding these expensive endpoints against burst abuse from school accounts.

**Fix:** Create a dedicated aiLimiter using express-rate-limit (e.g. 20 req/min per IP or session) and apply it with app.use('/api/ai', aiLimiter) and app.use('/api/chat', aiLimiter) before registerChatRoutes and registerAudioRoutes. Additionally wrap every reserveQuota call in a try/catch that explicitly returns 429 on quota failure rather than relying on the caller to handle it.

### `server/routes.ts` — initSupabaseTables, initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent calls at startup
**Issue:** These async initialization calls at server startup lack individual try/catch blocks. If any single seed function rejects (e.g. Supabase unreachable), the unhandled rejection will crash the entire server process or silently swallow the error depending on Node version, taking down the app for all students.

**Fix:** Wrap each init/seed call individually: try { await initSupabaseTables(); } catch (e) { console.error('Supabase init failed, continuing:', e); } — use non-fatal error handling with Sentry capture (captureException) so the server can still serve cached/static content while alerting on the infrastructure failure.

### `server/routes.ts` — videoUpload multer config (limits: fileSize 500MB) and XLSX/PapaCSV parse handlers
**Issue:** The 500MB video upload limit with multer.memoryStorage() means a single malicious or accidental large upload buffers the entire file in Node.js heap, which will OOM-kill the server process — taking down all concurrent student sessions. Similarly, XLSX.read() and Papa.parse() on user-supplied files are synchronous blocking operations on the event loop.

**Fix:** Switch videoUpload to multer.diskStorage() writing to a temp directory, then stream directly to object storage using a pipe rather than buffering. For XLSX/CSV parsing, either move parsing into a worker thread (worker_threads) or enqueue it as a background job using your existing enqueueJob infrastructure so the event loop is never blocked during a lesson or quiz session.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** The full base64-encoded file is stored in the job payload in the jobs table. A single uploaded PDF or image can easily be 5–20 MB encoded, meaning every retry duplicates that payload in the DB row and WAL, bloating storage and slowing job queue queries.

**Fix:** Upload the file to object storage at upload time and store only the resulting object path in the job payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient before processing. This keeps job rows small and makes the pattern consistent with the admin-csv-export handler which already uses object storage correctly.

---

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** The purge handler has no guard against a caller passing olderThanDays: 0 or a negative value, which would produce a cutoff in the future and delete all rows in the aiUsageEvents table — a silent, unrecoverable data loss event.

**Fix:** Add an explicit guard at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'). Also enforce a minimum floor in the job payload schema (e.g. z.number().int().min(1)) so invalid payloads are rejected at enqueue time rather than at execution time.

### `shared/schema.ts` — budgets table definition
**Issue:** budgets has no index on userId, meaning any per-user budget query (the most common access pattern) requires a full table scan. As user count grows this will degrade dashboard load times significantly.

**Fix:** Add a composite index on (userId, categoryId) inside budgets table options: `(t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })`. This covers both per-user listing and the unique budget-per-period constraint you likely want to enforce.

### `client/src/pages/AdminDashboard.tsx` — TABS constant / tab-level useQuery calls (pattern across tab panels)
**Issue:** Based on the TABS array (13 tabs) and the pattern visible in ObservabilityCard and AuditLogPanel, each tab panel likely mounts its own useQuery unconditionally. If all panels are mounted simultaneously (e.g. hidden via CSS rather than conditional rendering), every admin page-load fires 13+ parallel API requests, many to sensitive endpoints, regardless of which tab is active.

**Fix:** Conditionally render tab panel components (e.g. {activeTab === 'audit' && <AuditLogPanel />}) so queries only execute when the tab is visible. Alternatively pass enabled: activeTab === tabId to each useQuery. This also prevents stale background refetches on a 30-second interval hitting admin-only endpoints unnecessarily.

## 🟡 Warnings (15)

- **`server/storage.ts` (getDashboardStats):** Multiple independent aggregate queries (total income, total expenses, savings rate, budget utilisation, recent transactions) are executed with sequential `await` calls. Each round-trip to the DB adds latency that compounds, making dashboard loads slow under any real network conditions.
  → *Collect all independent query promises into an array and resolve with `const [income, expenses, budgetRows, recentTx] = await Promise.all([incomeQuery, expenseQuery, budgetQuery, recentQuery])`. Only queries that depend on a prior result should remain sequential.*
- **`server/storage.ts` (getLeaderboard):** Leaderboard query joins gameSessions with users and sorts by aggregate score without a `limit` guard being enforced at the SQL level when the optional `filters.limit` parameter is undefined. Omitting the parameter causes a full-table scan and returns every session row, which will OOM the Node process as the game scales.
  → *Apply a hard default at the query level: `const cap = Math.min(filters?.limit ?? 50, 100);` and always append `.limit(cap)` to the Drizzle query, never relying on the caller to pass a sane value.*
- **`server/storage.ts` (getLeaderboard / getUserXp (userId string logged on error)):** Error catch blocks log the full `userId` string alongside stack traces. For a school-age user base this constitutes a PII exposure in server logs — student identifiers can be correlated with financial behaviour and game scores.
  → *Replace `console.error('failed for user', userId, err)` patterns with a structured logger that hashes or truncates the identifier: `logger.error({ userRef: hashId(userId), msg: 'leaderboard fetch failed', err: err.message })`. Never log raw user identifiers at error level.*
- **`server/routes.ts` (getOrgUsageToday and getOrgUsageThisMonth calls (aiUsage quota check block)):** getOrgUsageToday and getOrgUsageThisMonth are called sequentially with two separate awaits before every AI request. These are independent DB reads and each adds full round-trip latency on the hot path for every student AI interaction.
  → *Replace the two sequential awaits with const [usageToday, usageMonth] = await Promise.all([getOrgUsageToday(orgId), getOrgUsageThisMonth(orgId)]); This halves the DB wait time on the quota check path, which is critical for perceived responsiveness for school-aged users on Caribbean network conditions.*
- **`server/routes.ts` (Audit log and error logging blocks throughout route handlers):** Several audit() and console.error() calls log the full error object or req.body, which in an educational finance app can expose student PII (names, financial figures, join codes) and internal stack traces to server logs that may be aggregated by third-party log services.
  → *Create a sanitizeForLog(obj) helper that strips known sensitive keys (password, idToken, token, dob, financialData) before any console.error or audit() call. For errors, log only e.message and e.code rather than the full Error object. Apply this consistently — especially in the Google auth and file-upload handlers where request bodies contain tokens.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with an await on each iteration. For a 40-question paper this is 40 round-trips to the DB, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy far longer than needed.
  → *Collect all valid question objects into an array first, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used for business logic reasons, replace the loop with await Promise.all(validQuestions.map((q, i) => storage.createExtractedQuestion({...}))). Either way, wrap in a transaction so a partial failure does not leave orphaned rows that the idempotency delete at the top was meant to prevent.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The OpenAI client is imported from a route module rather than a dedicated singleton. This tightly couples job handlers to the HTTP layer, risks circular-dependency module loading errors, and means any middleware, request-scoped state, or side effects in that routes file execute at import time inside the worker context.
  → *Move the OpenAI client instantiation to a shared module such as server/lib/openai.ts that exports a single configured instance (new OpenAI({ apiKey: process.env.OPENAI_API_KEY })). Import from there in both the route file and jobHandlers.ts. This also removes the risk of accidentally logging or exposing route-layer internals through the job context.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — userId columns):** All userId foreign key columns are typed as `varchar` with no length specified. Drizzle maps this to `varchar` without a length limit in Postgres, which defaults to unlimited — equivalent to `text`. If the auth provider (e.g. Clerk, Supabase) emits fixed-length IDs (e.g. 36-char UUIDs), the missing length means the index on these columns is larger than necessary and the column type gives no database-level validation against malformed IDs.
  → *Change all `varchar('user_id')` declarations to `varchar('user_id', { length: 36 })` (or 255 if using Clerk's longer IDs). This tightens storage, improves B-tree index efficiency, and acts as a schema-level guard against malformed foreign key values.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** `createdAt` uses `.defaultNow()` but is not marked `.notNull()`. Drizzle will type these as `Date | null` in the inferred TypeScript types, forcing null-checks everywhere in application code and allowing explicit NULL inserts that break audit trails.
  → *Append `.notNull()` to every `createdAt` column: `timestamp('created_at').defaultNow().notNull()`. The default ensures no insert needs to supply the value, and notNull guarantees the column is never NULL at the DB level.*
- **`shared/schema.ts` (simulatedStocks table definition):** `ticker` column has no unique index. Duplicate tickers in a simulated market would corrupt portfolio P&L calculations — two stocks with ticker 'BISX' would make stock lookups by ticker non-deterministic.
  → *Add `(t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` to the simulatedStocks table options. Also add an index on `riskLevel` if you plan to filter/sort the stock picker UI by risk, which is a near-certain product requirement for a financial literacy app.*
- **`shared/schema.ts` (billReminders table definition):** `nextDueDate` has no index. The core use-case for bill reminders is querying upcoming due dates (e.g. 'bills due in the next 7 days' across all users for notification jobs, or per-user dashboard). Without an index this becomes a sequential scan on every reminder check.
  → *Add `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })` to billReminders table options. This covers both the per-user dashboard query and any background notification job that filters by nextDueDate range.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo, and the queryFn closes over the params object reference rather than the stable string. This means the query key and the actual URL fetched can diverge if React batches state updates mid-render.
  → *Memoize the params string: const paramString = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p.toString(); }, [actorType, action, from, to]); Then use paramString in both queryKey and queryFn URL so they are always in sync.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no isLoading or isError handling. When the /api/admin/observability fetch is in-flight or fails, data is undefined and every optional-chain access silently renders empty/wrong UI with no feedback to the admin user.
  → *Destructure isLoading and isError from useQuery. Render a skeleton or spinner when isLoading is true, and a visible error badge (e.g. 'Failed to load observability data') when isError is true, before attempting to render sentry/alertEmail fields.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** fmtDate accepts type 'any' and passes it directly to new Date(). If the value is null-ish but truthy (e.g. empty string ''), new Date('') produces an Invalid Date that toLocaleDateString silently renders as 'Invalid Date', corrupting table cells throughout the dashboard.
  → *Narrow the guard: if (!d || (typeof d === 'string' && d.trim() === '')) return '—'; Additionally type the parameter as string | number | Date | null | undefined to get compile-time safety.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — hardcoded limit):** The audit log query hardcodes limit=200 inline in the component. For a school-aged financial app with growing audit data this will silently return a truncated result set with no pagination indicator, and for large tenants will transfer a large payload on every manual refresh.
  → *Move the limit to a named constant (AUDIT_LOG_PAGE_SIZE = 200) or an environment config value. Add a visible 'Showing up to N records' note in the UI and a server-side cursor/page token so admins can page through older entries rather than receiving a silent truncation.*

## 💡 Opportunities

### Centralised pagination guard middleware — 🔥 High Impact
At least six list-returning methods (getTransactions, getAllQuestions, getGameSessions, getLeaderboard, getEnrollmentsByClass, getPortfolioTransactions) accept optional limit/offset but apply no maximum cap. Introduce a shared `safePaginate(limit?: number, offset?: number, max = 200)` helper used by every list query to prevent unbounded scans without repeating the logic.

### Extract hardcoded virtual balance seed value to environment config — ⚡ Medium Impact
The initial virtual balance (likely 10000 or similar) granted to new users on first login appears hardcoded in `getVirtualBalance`. This value affects every student's investment simulation experience and should be `VIRTUAL_BALANCE_SEED_USD` in `.env` so it can be tuned per deployment or sponsor without a code redeploy.

### Replace per-entity category hydration with a shared category map — ⚡ Medium Impact
Both `getBillReminders` and `getBudgets` independently query the categories table to hydrate category names. A single `Map<number, Category>` built once per request (or cached with a short TTL) and passed into both resolvers eliminates the duplicate round-trips and makes the pattern reusable as more entities reference categories.

### Decompose routes.ts into domain controllers — 🔥 High Impact
At 200k+ characters, routes.ts is unmaintainable and makes targeted security audits nearly impossible. Splitting into controllers/ai.ts, controllers/lessons.ts, controllers/auth.ts, controllers/org.ts would reduce blast radius of bugs, enable per-domain rate limiting middleware stacks, and make it feasible to write unit tests against individual route groups — directly improving reliability for student-facing features.

### Centralize async route wrapper with error boundary — 🔥 High Impact
Wrapping every async Express handler in a shared asyncRoute(fn) higher-order function that catches rejections and calls next(err) would eliminate the inconsistent try/catch coverage seen throughout the file. This single change prevents all unhandled promise rejections from crashing the server and routes errors to Express's error middleware for consistent JSON responses — critical for school users who should never see raw 500 stack traces.

### Extract hardcoded file size limits and MIME patterns to config — ⚡ Medium Impact
File size limits (5MB logo, 500MB video, 10MB exam) and MIME type allowlists are scattered as magic numbers across four multer configs. Moving these to a validated config object (e.g. UPLOAD_LIMITS.video.maxBytes from environment or a config file) makes it safe to tune limits per deployment environment (school network vs. admin portal) without touching route logic, and enables runtime validation that limits are sane before the server accepts traffic.

### Parallel AI calls in extract-paper — ⚡ Medium Impact
The PDF text extraction step and the subsequent GPT extraction call are inherently sequential, but for image uploads the vision call and any metadata fetches are independent. More impactfully, if multiple papers are queued, introducing a concurrency limit (e.g. p-limit with concurrency 3) in the worker dispatch layer would cut total wall-clock processing time proportionally without adding API risk.

### Job payload schema validation at enqueue boundary — 🔥 High Impact
No handler validates the shape of job.payload before destructuring it. Adding Zod schemas per job type and calling schema.parse(job.payload) at the top of each handler would surface malformed payloads immediately with a descriptive error rather than a cryptic undefined-access crash mid-handler, and would serve as living documentation of each job's contract.

### Idempotency token for extract-paper retries — ⚡ Medium Impact
The current idempotency strategy (delete then re-insert) is correct but creates a window where a concurrent retry could race through the delete and both attempts re-insert questions simultaneously. Wrapping the delete + insert block in a serializable DB transaction, or using a distributed lock keyed on paperId, would close this race without changing the external behaviour.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract `const DEFAULT_CURRENCY = 'BSD'` at the top of the file and reference it via `.default(DEFAULT_CURRENCY)`. When the app expands to TTD or JMD, one line changes instead of six, eliminating the risk of a missed update introducing a currency mismatch bug.

### Add a unique constraint on budgets (userId, categoryId, period) — 🔥 High Impact
Nothing currently prevents a user from having two active monthly budgets for the same category, which would cause double-counting in budget vs. spending calculations. A `uniqueIndex('idx_budgets_user_category_period').on(t.userId, t.categoryId, t.period)` enforces one budget per category per period at the database level, removing the need for application-layer deduplication logic and preventing a class of data integrity bugs.

### Replace serial PKs with UUIDs on user-facing tables — 🔥 High Impact
Using `serial` (auto-increment integer) PKs on transactions, savingsGoals, and portfolioTransactions exposes sequential IDs in URLs and API responses, allowing users to enumerate other users' resources if any auth check is ever missed. Replacing with `uuid().primaryKey().defaultRandom()` eliminates enumeration risk and is consistent with the varchar UUID-style userId foreign keys already in use. This is especially important for a school-aged user base under stricter data protection obligations.

### Split AdminDashboard into per-tab lazy-loaded modules — 🔥 High Impact
At 126k+ characters this is a single-file mega-component. Use React.lazy + Suspense to split each tab panel (AuditLogPanel, ObservabilityCard, DB Viewer, etc.) into separate chunks. Expected impact: initial JS bundle for the admin route drops significantly; each tab's code is only downloaded when first accessed.

### Centralise admin API error handling with a shared query wrapper — 🔥 High Impact
ObservabilityCard and likely all other panels repeat the same .then(r => r.json()) pattern with no catch or response.ok check. A thin adminApiRequest wrapper that throws on non-2xx responses and a shared onError handler in QueryClient defaults would surface auth failures (401/403) consistently across all admin queries rather than silently returning undefined data.

### Add staleTime to read-heavy admin queries — ⚡ Medium Impact
Queries like /api/admin/observability and /api/admin/audit-log default to staleTime=0, so every tab focus triggers a background refetch. Setting staleTime to 60_000 (1 min) for observability and 30_000 for audit log would cut redundant network traffic on an admin session without meaningful staleness risk.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits where parallel execution is safe. Given this is a financial app for minors, the missing transaction guards on balance mutations are the most critical concern.

### API Routes (`server/routes.ts`)
routes.ts shows several reliability and security gaps that are particularly concerning for a school-aged user base: AI/heavy endpoints lack rate limiting, sequential awaits exist where Promise.all would suffice, and sensitive data appears exposed in logs. The file is extremely large (200k+ chars) which itself is an architectural smell suggesting route handlers need decomposition into dedicated controllers.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads are stored directly in the job table (a classic bloat vector), question insertion is sequential in a hot loop instead of batched, and the openai import originates from a route file which is an unusual coupling that risks circular dependencies and credential exposure. Error handling is mostly present but inconsistent across handlers.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with good use of foreign keys and composite indexes on transactions, but has several reliability and performance gaps: missing indexes on high-cardinality filter columns, inconsistent NOT NULL enforcement on audit timestamps, and untyped varchar foreign keys that risk silent data corruption. No security or injection issues are present in the schema layer itself, but schema gaps will propagate to runtime bugs.
