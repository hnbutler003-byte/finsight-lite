## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 113)
**Issue:** No rate limiting is applied to AI/chat, image, or audio routes before they are registered. These call OpenAI APIs and are expensive; a single user or bot can exhaust API quota or run up costs.

**Fix:** Import or create an aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and add `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls.

### `server/routes.ts`: robots.txt handler (~line 50) and sitemap.xml handler (~line 63)
**Issue:** The `host` value is taken directly from `req.headers.host` or `req.headers['x-forwarded-host']` with no validation and interpolated into the Sitemap URL and sitemap <loc> elements. A crafted Host header lets an attacker inject arbitrary URLs into both responses (Host Header Injection / cache poisoning).

**Fix:** Add an allowlist check: define `const ALLOWED_HOSTS = (process.env.ALLOWED_HOSTS || '').split(',');` and validate `host` against it before interpolation. Fall back to `process.env.APP_BASE_URL` as the canonical source so the value is never user-controlled in production.

### `server/jobHandlers.ts`: runExtractPaper (job.payload.fileB64)
**Issue:** Large base64-encoded file content is stored directly in the job table payload. A 5MB image becomes ~6.7MB of base64 in the DB row, bloating the jobs table, slowing queue polling queries, and risking OOM if many jobs are loaded into memory simultaneously.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload (e.g., { paperId, objectPath: '/uploads/papers/123.pdf', ext, subject }). In the handler, stream/download the file from object storage instead of decoding a base64 string from the job row.

### `shared/schema.ts`: simulatedStocks table, ticker column
**Issue:** ticker has no unique constraint. Duplicate tickers (e.g. two 'BTC' rows) would silently corrupt portfolio value calculations and JOIN results without any DB-level guard.

**Fix:** Replace the plain text column with a uniqueIndex: add (t) => ({ tickerUniq: uniqueIndex('idx_stocks_ticker_unique').on(t.ticker) }) as the table's second argument.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** The day-of-week rollback logic uses `(dow + 0) % 7` which is a no-op offset, causing Sunday (dow=0) to compute daysSince=0, so `d` stays on the current day rather than rolling back to the most recent Sunday. The guard `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` only partially compensates and will still fire on the wrong day for mid-week calls.

**Fix:** Use `const daysSince = dow === 0 ? 0 : dow;` and set the date then unconditionally check: after setting hours to 19:00, if `d > now` subtract 7 days. Alternatively replace the whole function with: `const d = new Date(now); d.setDate(d.getDate() - d.getDay()); d.setHours(19,0,0,0); if (d > now) d.setDate(d.getDate()-7); return d;`

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The return values of both `setInterval` calls are never stored, so there is no way to call `clearInterval` during graceful shutdown. If the process receives SIGTERM the intervals are orphaned, in-flight scheduler ticks can fire after the DB connection is closed, and any test harness that imports this module leaks timers indefinitely.

**Fix:** Store each handle: `const weeklyDigestInterval = setInterval(...); const purgeInterval = setInterval(...);`. In the SIGTERM/SIGINT handler call `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before closing the HTTP server and DB pool. Add these to the existing shutdown sequence alongside `httpServer.close()`.

### `server/index.ts`: res.json middleware / logLine construction
**Issue:** `capturedJsonResponse` is serialized in full via `JSON.stringify` and written to stdout for every API response. For a financial literacy app with school-aged users this risks logging PII (user profiles, grades, session tokens in response bodies) and adds non-trivial CPU cost on large paginated responses.

**Fix:** Remove response body logging entirely, or gate it behind an explicit `LOG_RESPONSE_BODIES=true` env flag that defaults to off in production. At minimum apply a size guard: `if (capturedJsonResponse && process.env.NODE_ENV !== 'production') { logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 200)}`; }`

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** No payload size limit on /api/admin/help-chat: the full `messages` array (unbounded growth) is sent on every request. A user leaving the chat open can accumulate thousands of tokens and exhaust API quota or cause backend OOM.

**Fix:** Slice the history before sending: `const history = next.slice(-20);` and send `{ messages: history }`. Add a max-message guard: if `messages.length > 100` trim from the front before appending. Also enforce a max character limit on `text` (e.g. 1000 chars) before the API call.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level dashboard render
**Issue:** If `useAdminAuth` returns a loading state before the auth check resolves, the dashboard body (queries, mutations, charts) still renders and fires API requests. Any auth-gated query that fires before the token is confirmed leaks admin data fetching to unauthenticated sessions during the loading window.

**Fix:** Destructure `{ isAdmin, isLoading }` from `useAdminAuth` and return a loading spinner early: `if (isLoading) return <Loader2 />;` and `if (!isAdmin) { setLocation('/admin/login'); return null; }` before any query hooks are called or JSX is returned.

## 🟡 Warnings (15)

- **`server/routes.ts` (await registerAuthDomainRoutes ... registerEmailRoutes (~lines 138-143)):** The six domain router registrations are called sequentially with individual awaits after the modules were loaded in parallel. If any single registerX throws (e.g. a DB call inside registration), the error propagates uncaught out of registerRoutes, crashing the process with no structured error message.
  → *Wrap the block in try/catch: `try { await registerAuthDomainRoutes(app); ... } catch (err) { console.error('[routes] Domain router registration failed:', err); throw err; }` so the failure is logged with context before the process exits.*
- **`server/routes.ts` (Supabase init chain (~line 120)):** The Supabase seeding functions run sequentially in a .then() chain. Each seed step blocks the next; if the chain is long and DB is slow this delays availability of seeded data. More critically, each individual step has no timeout, so a hung DB call silently stalls the entire chain with no alerting.
  → *Group independent seeds with Promise.all where order does not matter: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await Promise.all([seedFinancialAcademyLesson(), initStaticContentTables()]).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]));` Wrap the whole block in a helper with a `Promise.race` timeout of 30 s.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time in a sequential for-loop with individual awaits. For a 50-question exam paper this means 50 serial round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Collect all valid question objects into an array first, then issue a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Add a guard so the insert is skipped if the array is empty.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete then re-insert):** The idempotency strategy deletes previously inserted questions at the top of every attempt, including the first. This means a transient DB failure between the delete and the first insert leaves paperId with zero questions and no record of what was lost. It also silently discards questions from a partially-successful prior attempt before confirming the new extraction will succeed.
  → *Wrap the delete + all inserts + updateExamPaper in a single database transaction so either all succeed or none are committed: await db.transaction(async (tx) => { await tx.delete(...); await tx.insert(...).values(validQuestions); await tx.update(...); }). This makes retries safe without data loss windows.*
- **`server/jobHandlers.ts` (openai import — 'import { openai } from "./replit_integrations/chat/routes"'):** The OpenAI client is imported from a chat routes module, coupling an unrelated HTTP route file to the job handler. If that routes module initialises Express middleware or registers routes as a side-effect on import, every worker process that loads jobHandlers.ts will trigger those side-effects.
  → *Extract the OpenAI client instantiation into a dedicated shared module (e.g., ./lib/openaiClient.ts) that exports only the configured client. Both the chat routes and job handlers import from that single source.*
- **`server/jobHandlers.ts` (extractedText truncation — pdfData.text.slice(0, 30000)):** The PDF text is silently truncated to 30,000 characters with no logging and no update to the paper record to indicate truncation occurred. For longer exams this causes questions in the latter half of the paper to be silently dropped, and teachers have no visibility into why their question count is lower than expected.
  → *After slicing, check if pdfData.text.length > 30000 and if so set a flag. Either store truncated: true on the exam_papers row, or log a structured warning with paperId and original length. Consider chunking and making multiple extraction calls for long documents rather than silently dropping content.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, meaning every budget lookup for a user (e.g. dashboard load) does a full table scan. This will degrade as user count grows.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Consider a composite index on (userId, categoryId) since budget queries almost always filter by both.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Neither portfolioHoldings nor portfolioTransactions has an index on userId or stockId. Portfolio dashboard queries joining these two tables against a user will require sequential scans on both.
  → *Add indexes: for portfolioHoldings use (t) => ({ userStockIdx: index('idx_holdings_user_stock').on(t.userId, t.stockId) }); for portfolioTransactions use (t) => ({ userIdx: index('idx_ptx_user').on(t.userId), executedAtIdx: index('idx_ptx_executed').on(t.userId, t.executedAt) }).*
- **`shared/schema.ts` (transactions table, createdAt column; budgets createdAt; savingsGoals createdAt; documentUploads createdAt):** createdAt columns across multiple tables use .defaultNow() but omit .notNull(), meaning the column is nullable in the DB schema. A null createdAt breaks audit trails and any ORDER BY createdAt DESC queries silently misorder rows with NULLs.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same fix to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions.*
- **`shared/schema.ts` (billReminders table, nextDueDate column; savingsGoals deadline column):** billReminders.nextDueDate is NOT NULL (correct) but there is no index on it, making the scheduled job or cron query that fetches upcoming due bills (WHERE next_due_date <= NOW()) a full table scan. savingsGoals.deadline is nullable with no index, making deadline-proximity queries equally inefficient.
  → *Add to billReminders: (t) => ({ dueDateIdx: index('idx_reminders_due').on(t.userId, t.nextDueDate) }). For savingsGoals add: (t) => ({ deadlineIdx: index('idx_goals_deadline').on(t.userId, t.deadline) }).*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, startJobWorker)):** The startup awaits operations sequentially that are partially independent (seed, probe, register routes, start job worker). More critically, `seedDatabase()` runs before the HTTP server is listening, so a slow or failing seed (e.g. large fixture data or a momentary DB blip) blocks the port from opening and causes load-balancer health checks to time out and restart the container in a crash loop.
  → *Separate readiness from liveness: start listening on the port immediately after `registerRoutes`, then run `probeDatabase`, `seedDatabase`, and `startJobWorker` in the background. Gate API traffic that requires seeded data with a readiness flag, or use a `/healthz` route that returns 503 until `seedDatabase` resolves. Also run `probeDatabase` and `seedDatabase` with `Promise.all` where they are independent.*
- **`server/index.ts` (maybeRunAiUsagePurge / AUTO_PURGE_OLDER_THAN_DAYS):** The purge scheduler deletes AI usage rows older than N days but the delete query and its error path are truncated in the visible code. If the delete is unbounded (no LIMIT), a large table causes a long-running transaction that locks rows, spikes DB CPU, and may time out—taking down the scheduler silently because the outer catch only logs the message without re-enqueueing or alerting.
  → *Add a LIMIT to the delete (e.g. batch 1000 rows per tick) and loop until 0 rows deleted. Capture the error with `captureError(e)` so Sentry records it, not just a console log. Expose a metric or structured log entry with `rowsDeleted` so the purge is observable.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` is recreated on every render and fires scrollIntoView on every keystroke/state change, not just when a new message arrives. This causes jarring scroll-jacking while the user is typing.
  → *Replace with a `useRef` + `useEffect`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` Then use `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (handleKey (AdminHelpChat)):** `handleKey` and `sendMessage` are plain function declarations inside the component body, recreated on every render. They are passed as props to the input's `onKeyDown` and the send button's `onClick`, causing those elements to re-render every time any state (e.g. `loading`, `input`) changes.
  → *Wrap both with `useCallback`: `const sendMessage = useCallback(async () => { ... }, [input, loading, messages]);` and `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`.*
- **`client/src/pages/AdminDashboard.tsx` (apiRequest POST /api/admin/help-chat):** The catch block swallows the actual error silently (`catch {` with no parameter) and only shows a generic message. For a school-targeted platform, network errors vs. auth errors vs. rate-limit errors should be distinguished so admins know if their session expired vs. a transient failure.
  → *Change to `catch (err) {` and inspect: `const msg = err instanceof Response && err.status === 401 ? 'Session expired – please log in again.' : err instanceof Response && err.status === 429 ? 'Rate limit reached, please wait.' : 'Something went wrong.';` then use `msg` in the fallback assistant message. Log `console.error` only in non-production.*

## 💡 Opportunities

### Centralise base-URL resolution: 🔥 High Impact
Extract `proto + host` construction into a single `getBaseUrl(req)` utility that reads `process.env.APP_BASE_URL` first and falls back to validated headers. Used in robots.txt, sitemap, and any future redirect logic, this eliminates the duplicated header-reading code and the injection surface in one place.

### Add per-user AI quota enforcement: 🔥 High Impact
IP-based rate limiting alone is insufficient for a school app where many students share a NAT IP. Add a per-authenticated-user token-bucket check (store counts in Redis or Supabase) on AI endpoints so individual students cannot exhaust the shared OpenAI budget during class sessions.

### Lazy-load heavy domain routers: ⚡ Medium Impact
All six domain routers are eagerly imported at startup via Promise.all, increasing cold-start memory pressure. Routes that are rarely hit (e.g. email, lessons) could be loaded on first request using express-lazy-router or a simple import-on-demand wrapper, cutting baseline RSS by the weight of their transitive dependencies.

### Centralised job payload size validation: 🔥 High Impact
Add a payload size check in the enqueueJob function (or a job middleware layer) that rejects any payload exceeding a configurable limit (e.g., MAX_JOB_PAYLOAD_BYTES from env). This prevents the base64 antipattern from recurring for future job types and surfaces the problem at enqueue time with a clear error rather than silent DB bloat.

### Structured logging with paperId/jobId correlation: ⚡ Medium Impact
All catch blocks currently either swallow errors silently or rethrow bare Error objects. Adding a structured logger call (e.g., logger.error({ jobId: job.id, paperId, err })) at each failure boundary would make it possible to correlate job failures with specific records in production without manually joining job table rows to application logs.

### Extract magic numbers into named environment config constants: ⚡ Medium Impact
Values like 30000 (PDF char limit), 4096 (max_completion_tokens), and 0.3 (temperature) are hardcoded inline. Moving them to environment variables (PDF_EXTRACT_CHAR_LIMIT, AI_MAX_TOKENS, AI_TEMPERATURE) allows tuning in production without a code deploy and makes the intent explicit for future maintainers.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, portfolioHoldings). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' and reference it in each .default() call. Changing the default currency for a new market then requires one line edit instead of hunting six locations.

### Replace text columns for period and frequency with enums: ⚡ Medium Impact
budgets.period is a plain text column with no enum constraint, unlike frequency on billReminders which does use an enum. An invalid value like 'anually' (typo) is accepted silently. Add { enum: ['weekly', 'monthly', 'quarterly', 'yearly'] } to budgets.period to match the pattern already used elsewhere and prevent bad data at the DB level.

### Add partial index on documentUploads for processing status: 📌 Low Impact
Polling queries that check for documents still in 'processing' status will scan all uploads. A partial index WHERE status = 'processing' would make the worker queue lookup O(active jobs) instead of O(all uploads). In Drizzle this requires a raw sql index expression until partial indexes are natively supported, but the performance gain is significant once upload volume grows.

### Structured JSON logging instead of formatted strings: 🔥 High Impact
Replacing `console.log` string interpolation with a structured logger (pino or winston JSON transport) would make log lines machine-parseable, allow log-level filtering (suppressing debug noise in production), and avoid accidental PII leakage through uncontrolled string concatenation. Expected impact: faster log ingestion, easier alerting, and a clear paper trail for compliance with student data regulations.

### Centralised graceful-shutdown coordinator: 🔥 High Impact
Currently shutdown logic is spread across ad-hoc SIGTERM handlers. Introducing a single shutdown manager that sequentially: stops accepting new requests (server.close), drains in-flight HTTP, clears intervals, waits for the job worker to finish its current job, then closes the DB pool—with a hard timeout—prevents torn writes and connection-leak warnings on every deploy. Particularly important for a school app that deploys during off-hours.

### Extract scheduler logic into a dedicated module: ⚡ Medium Impact
Moving `maybeRunWeeklyDigest`, `maybeRunAiUsagePurge`, and their intervals out of `server/index.ts` into `server/scheduler.ts` with an exported `startSchedulers(): SchedulerHandle` / `stopSchedulers(handle)` pair would make the startup file easier to reason about, allow unit-testing scheduler logic in isolation with fake timers, and make the interval cleanup pattern impossible to forget.

### Parallel data fetching with Promise.all: 🔥 High Impact
The dashboard likely fires multiple independent useQuery calls (users, schools, teachers, analytics) sequentially in waterfall. Consolidate into a single aggregated endpoint or ensure all queries have distinct keys and fire simultaneously. This can cut initial dashboard load time by 40-60% on slow Caribbean mobile connections.

### Memoize filtered/sorted table data: ⚡ Medium Impact
Any user/school/teacher list that is filtered or sorted by search input should be wrapped in `useMemo(() => data.filter(...).sort(...), [data, searchTerm, sortField])`. Without this, filtering re-runs on every parent re-render including unrelated state changes like the chat toggle, causing visible lag with >200 row datasets.

### Extract chart components to prevent full-dashboard re-renders: ⚡ Medium Impact
Recharts BarChart/LineChart components defined inline in the dashboard body re-mount whenever any dashboard state (e.g. active tab, search input) changes. Extract each chart into a named component wrapped with React.memo and stable prop references via useMemo for chart data arrays to avoid expensive SVG reconciliation on every keystroke.

### API Routes (`server/routes.ts`)
The route file is generally well-structured with good use of Promise.all for parallel imports and proper try/catch on async handlers. Key concerns are: unhandled promise rejections from unguarded async route registrations, a missing rate limit on AI/chat endpoints which are heavy and costly, and a subtle host-header injection risk in robots.txt and sitemap generation.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads are stored directly in the job table (memory/DB pressure), sequential question inserts in a loop should be batched, and the file truncates before revealing the full admin-csv-export and other handlers which likely contain additional gaps. Core error handling for the extract-paper path is reasonable, but idempotency and payload size are the dominant concerns.

### Database Schema (`shared/schema.ts`)
The schema is reasonably well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-cardinality filter columns, nullable timestamps that should be NOT NULL, and absent indexes on foreign keys used in joins across investment and budget tables.

### Server Startup (`server/index.ts`)
The server startup sequence contains a blocking synchronous-style chain of async operations that delay readiness, intervals are created without any cleanup reference stored for graceful shutdown, and the logging middleware serializes full JSON response bodies to stdout on every API call which is a data-exposure and performance risk in production. Core reliability and security gaps exist despite reasonable guard rails around SESSION_SECRET and database probing.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth guards on sensitive mutations, unprotected async operations, and inline anonymous functions causing unnecessary re-renders. The chat component creates a new scroll ref callback on every render and the help-chat endpoint sends full message history without size limits, creating a potential abuse vector in a school-aged user context.
