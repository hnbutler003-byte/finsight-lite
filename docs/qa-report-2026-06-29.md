### Content upload (admin and org-admin lesson endpoints)

The admin lesson endpoint (`POST /api/admin/organizations/:id/lessons`) uses **snake_case** field names (`video_url`, `grade_level`). The org-admin endpoint (`POST /api/org-admin/lessons`) uses **camelCase** (`videoUrl`, `gradeLevel`). Both endpoints apply identical YouTube URL validation via the shared `ytUrlOpt` Zod schema.

Tested:
- Valid YouTube URL (`youtu.be/dQw4w9WgXcQ`): accepted, lesson created. PASS.
- Invalid URL (vimeo.com): org-admin endpoint returned HTTP 400 `"Please enter a valid YouTube link."` PASS.
- Admin endpoint: same validation applies when `video_url` field is supplied (snake_case). PASS.

### File upload error paths

- Invalid MIME type (`application/octet-stream`): returned HTTP 400. PASS.
- Oversize video (multer `LIMIT_FILE_SIZE`): would return HTTP 413 per route handler lines 155-158. The MIME check fires first in the 1 MB test (400), which is correct behaviour -- size is only checked after MIME passes.

### HOLD items

**API field-name inconsistency (HOLD -- design decision required)**

The admin lesson endpoint uses snake_case field names (`video_url`, `grade_level`, `content_sections`) while the org-admin endpoint uses camelCase (`videoUrl`, `gradeLevel`, `contentSections`). Both work correctly. Normalising them requires a coordinated front-end + back-end change and should not be done speculatively. Flag for a future API consistency pass.
