## 🔴 Critical (5)

### `server/routes.ts` — registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 93)
**Issue:** AI and image generation endpoints are registered with no rate limiting middleware, making them trivially abusable by school-aged users or bots. These are the most expensive endpoints (OpenAI cost and latency) and have no protection.

**Fix:** Create a dedicated aiLimiter (e.g. 10 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions.

### `server/routes.ts` — sitemap handler — const host = req.headers[...] (~line 38)
**Issue:** The sitemap uses req.headers.host and x-forwarded-host directly to construct absolute URLs without validation. A request with a crafted Host header produces a sitemap with attacker-controlled URLs, which can be used for cache poisoning or phishing via sitemaps.

**Fix:** Resolve the base URL from an environment variable: const base = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com'; and remove the dynamic host construction entirely.

---

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 — a full base64-encoded PDF or image — is stored directly in the job payload row. For a multi-page PDF this easily exceeds 5–20 MB per row, bloating the jobs table, slowing queue polls, and risking row-size limits depending on the DB engine.

**Fix:** Before enqueuing, upload the file to private object storage and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of the job. This keeps job rows small (<1 KB) and makes retries cheaper.

---

### `shared/schema.ts` — portfolioHoldings and portfolioTransactions table definitions
**Issue:** portfolioHoldings has no unique constraint on (userId, stockId), so a user can accumulate duplicate holding rows for the same stock instead of a single aggregated position. This will silently corrupt portfolio value calculations.

**Fix:** Add a unique index to portfolioHoldings: (t) => ({ uniqueUserStock: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). Buy/sell logic should then upsert on this constraint rather than blindly inserting. Also add index('idx_portfolio_tx_user').on(t.userId, t.executedAt) to portfolioTransactions for history queries.

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth hook usage / top-level component
**Issue:** The component imports useAdminAuth but the truncated code does not show a hard return/redirect guard before data-fetching queries are initiated. If useAdminAuth is async (session check via network), all useQuery calls for sensitive admin data (audit logs, observability, DB viewer) will fire immediately on mount before auth is confirmed, leaking admin API calls to unauthenticated users if the server-side guard is misconfigured.

**Fix:** Immediately after calling useAdminAuth, add a guard: `if (!isAdmin) { redirect('/login'); return null; }` before any JSX or query hooks. If hooks cannot be called conditionally, use `enabled: isAdmin` on every sensitive useQuery call to prevent fetching until auth is confirmed.

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables() chain (~line 100)):** The sequential .then() chain means a failure in any middle step (e.g. seedFinancialAcademyLesson) causes all subsequent steps to be skipped silently. The single .catch() at the end only logs the first rejection and gives no indication of which step failed or what was skipped.
  → *Run independent seed steps with Promise.allSettled so each step is attempted regardless of others, and log per-step rejections: Promise.allSettled([initSupabaseTables(), seedFinancialAcademyLesson(), initStaticContentTables(), ...]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] Step ${i} failed:`, r.reason)));*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 80)):** filePath is taken directly from the URL and passed to objectStorage.searchPublicObject without sanitisation. A path like ../../private/config could traverse into unintended storage prefixes depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|^\//, ''); and reject any path that starts with '..' or is absolute. Import Node's path module at the top of the file.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with an await on each iteration. For a 40-question paper this means 40 serial round-trips to the database, adding latency and holding the job worker busy unnecessarily.
  → *Replace the loop with a single bulk insert: filter valid questions into an array first, then call db.insert(extractedQuestions).values(validQuestions) in one statement. If storage.createExtractedQuestion must be used, wrap calls in Promise.all with a concurrency cap (e.g., p-limit at 5) rather than sequential awaits.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation. A malformed or malicious payload (e.g., olderThanDays: 0 or a negative number) would compute a future cutoff date and delete all rows, or a string value would produce a NaN cutoff silently deleting nothing — both are data-integrity risks.
  → *Add an explicit guard at the top of the handler: const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days < 1) throw new Error('Invalid olderThanDays: ' + job.payload.olderThanDays); This causes the job to fail loudly rather than silently misbehave.*
- **`server/jobHandlers.ts` (admin-csv-export handler):** The job handler performs no authorization check on job.payload.requestedBy or equivalent before querying and exporting potentially sensitive user/transaction data. Any code path that can enqueue an 'admin-csv-export' job (including a bug or SSRF in another handler) will exfiltrate data without a gate.
  → *At the start of the handler, resolve the requesting user from payload and assert their role: const requester = await db.select().from(users).where(eq(users.id, job.payload.requestedById)).then(r => r[0]); if (requester?.role !== 'admin') throw new Error('Unauthorized export request'); This is defense-in-depth even if the enqueue call site already checks roles.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups are always user-scoped. Every budget query will do a full table scan as the dataset grows.
  → *Add a composite index in the budgets table definition: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a unique constraint on (userId, categoryId, period) to prevent duplicate budget entries per user per category per period.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders — all userId columns):** userId is declared as varchar with no length specified, defaulting to unlimited length in Postgres. This is both inefficient (no length optimisation) and inconsistent — user IDs from auth providers like Clerk or Supabase are fixed-length strings (e.g. 32–36 chars). An unbounded varchar wastes storage and slows index scans.
  → *Change all userId columns to varchar(36) (UUID) or varchar(32) to match your auth provider's ID format, e.g. userId: varchar('user_id', { length: 36 }).notNull().references(() => users.id). Confirm the exact length with your auth model in ./models/auth.*
- **`shared/schema.ts` (transactions createdAt, budgets createdAt, savingsGoals createdAt, billReminders createdAt, linkedCards createdAt, documentUploads createdAt):** All createdAt audit columns use .defaultNow() but lack .notNull(). This means application code or a future migration could insert rows with a NULL createdAt, breaking audit trails and time-based queries in a financial context.
  → *Append .notNull() to every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default guarantees the value is always set at insertion, so notNull() adds safety with zero operational cost.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on nextDueDate or userId. The most common query for this feature — fetching upcoming bills for a user sorted by due date — will full-scan the table.
  → *Add a composite index: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This turns the upcoming-bills query into an index range scan.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo. Since it feeds into the queryKey array indirectly through primitive values but the params object itself is rebuilt inline, any parent re-render causes unnecessary object churn. More critically, the queryFn closes over `params` which is a new object reference each render, making the closure stale-prone if React batches renders unexpectedly.
  → *Wrap params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]);` and use `params.toString()` inside queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no isLoading, isError, or error state handling. If the /api/admin/observability endpoint fails or is slow, the component silently renders with all undefined values and shows misleading 'Disabled'/'Not configured' badges rather than a loading or error state. In an admin context, a failed health check should be surfaced explicitly.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when isLoading is true, and an explicit error alert (e.g., red badge saying 'Unavailable') when isError is true, so admins are not misled by false 'Disabled' statuses caused by fetch failures.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** `fmtDate(d: any)` calls `new Date(d)` without validation. If `d` is an invalid date string, `new Date(d)` returns `Invalid Date` and `toLocaleDateString` returns the string 'Invalid Date' silently. This will surface as garbled dates in audit logs and student/teacher tables visible to admins.
  → *Add validation: `const dt = new Date(d); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString(...)`. Also type the parameter as `string | number | Date | null | undefined` instead of `any` to catch misuse at compile time.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log fetch limit of 200 is hardcoded as a magic string. For a school-aged user platform that may grow, 200 rows returned in a single unpaginated fetch with no server-side cursor will cause increasing memory pressure and slow renders. There is also no pagination UI, meaning admins cannot see records beyond the 200-row cap.
  → *Move the limit to a named constant `const AUDIT_LOG_PAGE_SIZE = 200` at module level (or pull from env config). Implement cursor-based or offset pagination: add `page` state, pass `offset = page * AUDIT_LOG_PAGE_SIZE` to the query, and render previous/next controls. Include the limit/offset in the queryKey so React Query caches pages independently.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (duplicate health check endpoints (~lines 28 and 47)):** Two health check routes exist: GET /api/health (simple) and GET /healthz (DB ping). Having two is not inherently wrong, but /api/health is documented as the Railway/uptime monitor endpoint yet does no DB check, while /healthz does a DB check but is not consistently referenced. This creates confusion about which to use and means Railway could report 'ok' when the DB is down.
- **`server/jobHandlers.ts` (runExtractPaper — image branch):** For image uploads, buffer.toString('base64') is called to re-encode the buffer that was just decoded from fileB64. This is a redundant round-trip (base64 → Buffer → base64) and doubles peak memory usage for large images. It also means the original base64 string from the payload and the re-encoded one must match exactly, which is only guaranteed if the original had no line breaks or padding differences.

## 💡 Opportunities

### Parallel domain-router registration — ⚡ Medium Impact
The six registerXxxRoutes calls after the parallel import are still awaited sequentially. If any of these do async work (DB migrations, cache warm-up), they add to cold-start time unnecessarily. Wrap them in Promise.all: await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]). Assuming route registration is not order-dependent, this could cut cold-start time proportional to the slowest individual registration.

### Structured logging for Supabase init errors — ⚡ Medium Impact
The current console.error for Supabase init failures loses structured context in log aggregators (Datadog, Logtail). Replace with a structured logger call that includes step name, error code, and timestamp. This makes it possible to set alerts on seeding failures in production, which is critical for a school app that depends on lesson content being available.

### Object storage cache-control via environment config — 📌 Low Impact
The TTL passed to downloadObject (60*60*24*30 — 30 days) is hardcoded. For a school app, lesson assets or logos can change without notice. Move this to OBJECT_STORAGE_CACHE_TTL_SECONDS in environment config so it can be adjusted per environment (shorter in staging, longer in prod) without a code deploy.

### Batch DB writes in extract-paper with a single transaction — 🔥 High Impact
Wrap the bulk question insert and the updateExamPaper call inside a single db.transaction(). This guarantees atomicity — if the status update fails, questions are not left orphaned — and removes the need for the pre-retry delete hack that currently provides partial idempotency.

### Extract magic literals into environment config — ⚡ Medium Impact
Values like the 30000-character PDF text slice cap, max_completion_tokens: 4096, and model: 'gpt-4o' are hardcoded throughout. Moving them to named environment variables (PDF_TEXT_CHAR_LIMIT, OPENAI_MAX_TOKENS, OPENAI_MODEL) allows tuning per environment without deploys and makes cost/quality tradeoffs explicit.

### Add structured logging with sanitization before any AI or export job — ⚡ Medium Impact
Current error paths propagate raw error messages that may include file content snippets or user data into job failure logs. Introducing a thin logging wrapper that redacts payload.fileB64 and truncates content fields before logging would prevent accidental PII or exam-content exposure in log aggregators, which matters given the school-aged user base.

### Currency enum column type — ⚡ Medium Impact
currency is a free-text column on transactions, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions with only a runtime default of 'BSD'. Replacing it with text('currency', { enum: ['BSD', 'USD', 'TTD', 'JMD'] }).notNull() enforces valid values at the DB level, eliminates an entire class of silent data bugs, and signals to future developers which currencies the app intentionally supports.

### Centralise currency and period defaults — ⚡ Medium Impact
The string literals 'BSD' and 'monthly' are hardcoded across five or more table definitions. Extract them as exported constants (e.g. export const DEFAULT_CURRENCY = 'BSD') in a shared constants file and reference them in the schema. This ensures a single change point if the app expands to multi-currency support and prevents typo-introduced inconsistencies across tables.

### Add partial index for active document processing — 📌 Low Impact
documentUploads is queried for status='processing' rows to drive polling or retry logic. Add a partial index: index('idx_doc_uploads_processing').on(t.userId, t.createdAt).where(sql`status = 'processing'`). This index stays tiny in production (only unresolved uploads qualify) and makes processing-queue queries near-instant regardless of total upload history size.

### Split monolithic file into feature-scoped modules — 🔥 High Impact
At ~126K characters, this single file contains at minimum 13 tab panels, multiple query hooks, and chart components. Split each tab (AuditLogPanel, ObservabilityCard, BackgroundJobsPanel, etc.) into separate files under pages/admin/. This will cut initial parse/compile time, enable React.lazy() per-tab code splitting, and make isolated unit testing practical. Expected impact: 30-50% reduction in TTI for the admin route.

### Centralise admin API query definitions with useCallback-wrapped queryFns — ⚡ Medium Impact
Multiple inline arrow functions are passed directly as queryFn values (e.g., `() => apiRequest('GET', ...).then(r => r.json())`). Extract these into a dedicated adminApi service module and wrap repeated patterns in a typed factory. This eliminates duplicate fetch logic, makes error handling consistent, and removes the risk of stale closures over local variables like `params`.

### Add React Error Boundary around each admin tab panel — 🔥 High Impact
A runtime error in any single tab (e.g., a bad date in audit log, a recharts crash on malformed chart data) will unmount the entire AdminDashboard. Wrap each panel component in an ErrorBoundary with a fallback UI. Given this is a financial literacy platform with school admins who are not developers, a full dashboard crash is a high-severity UX and trust issue.
