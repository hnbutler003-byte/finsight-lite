## 🔴 Critical (9)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI/image/audio routes are registered with no rate limiting middleware. For a school-aged user base these endpoints call paid external APIs (OpenAI) and are trivially abusable, risking cost exhaustion and degraded availability.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/min per IP via express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Adjust window/max to match your OpenAI tier limits.

### `server/routes.ts` — /public-objects/:filePath(*) handler (~line 87)
**Issue:** The `filePath` parameter is passed directly from user input into `objectStorage.searchPublicObject(filePath)` with no sanitisation. A path traversal payload such as `../../private/keys` could reach unintended storage keys depending on the ObjectStorageService implementation.

**Fix:** Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and additionally validate it matches an allowlist pattern such as `/^[\w\-./]+$/`. Import `path` from Node core at the top of the file.

### `server/jobHandlers.ts` — extract-paper handler / job.payload
**Issue:** Full base64-encoded file content (fileB64) is stored directly in the jobs table payload. A single PDF or image could be several MB, bloating the jobs table, slowing queue polls, and risking OOM on workers that load many jobs into memory simultaneously.

**Fix:** Before enqueuing, upload the file to private object storage and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient at job execution time, then delete the temp object on success or terminal failure.

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** The purge-ai-usage handler has no try/catch and no validation that olderThanDays is a positive number. A payload with olderThanDays: 0 or a negative value would compute a future cutoff and delete all rows, causing irreversible data loss with no error surfaced.

**Fix:** Add input validation at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'). Wrap the delete in a try/catch that logs the error with structured context (jobId, cutoffDate) before rethrowing so the job framework records the failure correctly.

### `shared/schema.ts` — budgets table definition
**Issue:** budgets.userId has no index. Every budget lookup by user (the dominant query pattern) will do a full table scan as user data grows.

**Fix:** Add an index in the budgets table's index factory: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also add a unique index on (userId, categoryId, period) to enforce one budget per category per period per user at the DB level.

### `shared/schema.ts` — billReminders table definition
**Issue:** billReminders.nextDueDate and billReminders.userId have no indexes. The primary query for reminders is 'find upcoming bills for user X where nextDueDate <= threshold', which will full-scan without these.

**Fix:** Add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders table definition.

### `server/index.ts` — res.json middleware / res.on('finish') handler
**Issue:** Every API response body is captured in `capturedJsonResponse` and then serialized with `JSON.stringify` into the log line unconditionally. For a financial literacy app with school-aged users this will log session tokens, quiz answers, user PII, and any sensitive payload returned by AI endpoints — a serious data-exposure risk in log aggregators.

**Fix:** Add an allowlist of safe, non-sensitive paths (e.g. health-check, public content endpoints) and only log response bodies for those. For all other `/api` routes log only status code and duration: `if (capturedJsonResponse && SAFE_LOG_PATHS.has(path)) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`

### `server/index.ts` — startJobWorker / maybeRunWeeklyDigest / maybeRunAiUsagePurge interval registrations
**Issue:** The `setInterval` calls for the digest scheduler and AI purge scheduler (visible in the truncated section) are fired-and-forgotten — their return values are not stored. On SIGTERM or SIGINT the process exits without calling `clearInterval`, which in a containerised environment can cause the scheduler to fire mid-operation during a rolling restart, corrupting the atomic-claim pattern used in `maybeRunWeeklyDigest`.

**Fix:** Store every interval handle: `const digestInterval = setInterval(maybeRunWeeklyDigest, INTERVAL_MS);`. Register a single `gracefulShutdown` function on `process.on('SIGTERM')` and `process.on('SIGINT')` that calls `clearInterval` on each handle, then `httpServer.close()`, then closes the DB pool before `process.exit(0)`.

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth hook usage / top-level component
**Issue:** useAdminAuth is imported and presumably called, but the truncated code does not show an early return guard before any useQuery/useMutation hooks fire. If auth state is async (loading), queries will execute with no credentials and hit the server unauthenticated before the redirect fires, leaking admin endpoint invocations.

**Fix:** After calling `const { isAdmin, isLoading } = useAdminAuth()`, add `if (isLoading) return <LoadingSpinner />;` and `if (!isAdmin) { setLocation('/admin/login'); return null; }` before any query hooks or JSX that renders sensitive data.

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...)):** The sequential `.then()` seed chain means a failure in any intermediate step (e.g. `seedFinancialAcademyLesson`) silently skips all subsequent steps and only logs a single generic error with no indication of which step failed. Individual promise rejections before the catch are also unhandled if the chain is broken.
  → *Replace with an explicit async IIFE with per-step try/catch so each step is logged and subsequent steps still run: `(async () => { for (const [name, fn] of [['initTables', initSupabaseTables], ...]) { try { await fn(); } catch (e) { console.error(`[Supabase] ${name} failed:`, e); } } })();`*
- **`server/routes.ts` (/healthz handler (~line 63)):** `import('./db')` and `import('drizzle-orm')` are dynamic imports inside the route handler, called on every /healthz request. While Node caches modules after first load, the dynamic import machinery still runs on every call and the pattern is misleading — it implies the module may not be loaded, creating confusion for future maintainers and a minor overhead spike on the very first probe.
  → *Hoist the db and sql imports to the top of the file as static imports alongside the other imports. The /healthz handler then just calls `await db.execute(sql\`select 1\`)` directly with no dynamic import overhead.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one in a sequential for-loop with individual awaits. For a 40-question paper this is 40 serial round-trips to the database, adding significant latency and holding the job worker busy unnecessarily.
  → *Collect valid questions into an array, then issue a single bulk insert: await db.insert(extractedQuestions).values(validQuestions) where validQuestions is the mapped array. If storage.createExtractedQuestion must be used, replace the for-loop with Promise.all(validQuestions.map(q => storage.createExtractedQuestion(q))). Add a guard: if validQuestions.length === 0 throw before attempting insert.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText slice hardcoded to 30000):** The PDF text truncation limit of 30000 characters is hardcoded. This is both a magic number that should be configurable and a potential correctness issue — it silently drops content from longer papers with no warning returned to the caller or stored on the paper record.
  → *Introduce an env var MAX_EXTRACT_TEXT_CHARS with a sensible default (e.g., 30000). After slicing, if the original text exceeded the limit, log a structured warning and store a truncated: true flag in the updateExamPaper call so the UI can surface a warning to the teacher.*
- **`server/jobHandlers.ts` (openai import — 'import { openai } from "./replit_integrations/chat/routes"'):** The OpenAI client is imported from a routes module, creating a hidden coupling between job infrastructure and HTTP route internals. If that routes module has side effects (e.g., registering Express routes, initialising middleware), importing it in a job worker context will trigger those side effects in a non-HTTP process.
  → *Extract the OpenAI client initialisation into a dedicated shared module such as server/lib/openaiClient.ts that exports only the configured client. Both the routes file and jobHandlers import from that single source. This eliminates side-effect coupling and makes the dependency explicit.*
- **`shared/schema.ts` (documentUploads table definition):** documentUploads.status has no index. Polling queries like 'find all processing uploads' (used by background job workers) will full-scan. Also, createdAt is nullable (no .notNull()) which breaks time-based ordering guarantees.
  → *Add (t) => ({ statusIdx: index('idx_document_uploads_status').on(t.status), userIdx: index('idx_document_uploads_user').on(t.userId) }) and change createdAt to timestamp('created_at').defaultNow().notNull() on documentUploads, budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions.*
- **`shared/schema.ts` (categories table definition — userId column):** categories.userId is nullable by design (system categories), but there is no index on userId. User-specific category lookups (WHERE user_id = $1 OR user_id IS NULL) will full-scan as the category list grows per user.
  → *Add (t) => ({ userIdx: index('idx_categories_user').on(t.userId) }) to the categories table. Also add .notNull() to categories.isDefault since a missing boolean default is logically equivalent to false and should be explicit.*
- **`shared/schema.ts` (varchar('user_id') — all tables):** All user_id foreign key columns are declared as varchar with no length parameter. Without an explicit length, Postgres stores it as varchar with no limit (equivalent to text), but the intent is to match the auth provider's ID format (e.g., Clerk uses 32-char strings). A mismatch causes silent over-allocation and prevents future length-enforcement migrations.
  → *Declare all user_id columns as varchar('user_id', { length: 64 }) (or match the exact length from your auth provider). This enforces correctness at the DB layer and makes the contract with the auth model explicit.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** Day-of-week arithmetic is wrong. `(dow + 0) % 7` always returns Sunday=0, Monday=1…Saturday=6 unchanged, so on any day other than Sunday the function walks back to the most recent Sunday correctly only by accident when dow happens to equal daysSince. The real bug: on Sunday before 7 pm, `d` is set to today at 19:00 which is in the future, then correctly shifted back 7 days — but on Saturday (dow=6) daysSince=6, so it walks back 6 days to the previous Sunday, which is correct. However the variable name `daysSince` and the `(dow + 0) % 7` expression are misleading and the logic will silently break if anyone adjusts the target day-of-week.
  → *Replace with an explicit, readable calculation: `const daysToLastSunday = dow; d.setDate(d.getDate() - daysToLastSunday);` and add a unit test covering Sunday-before-7pm, Sunday-after-7pm, and mid-week cases to lock in the correct behaviour.*
- **`server/index.ts` (startup sequence (seedDatabase / probeDatabase / registerRoutes calls)):** Based on the visible code, `seedDatabase`, `probeDatabase`, and route/job registration are awaited sequentially in the startup IIFE. `seedDatabase` and `probeDatabase` have no data dependency on each other and together likely account for the majority of startup latency — a real concern on Replit's cold-start environment where school users may hit a 502 during the gap.
  → *Run independent startup tasks in parallel: `const [_, dbOk] = await Promise.all([seedDatabase(), probeDatabase()]);` then proceed with route registration which depends on DB readiness. Add a startup timeout guard: if the Promise.all does not resolve within 15 s, log and `process.exit(1)` to avoid a silent hang.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The messages array sent to /api/admin/help-chat grows unbounded with every turn. A long conversation will eventually send a multi-MB payload on each keystroke-Enter, and there is no client-side cap or server-side size validation visible in this file.
  → *Slice the history before sending: `messages: next.slice(-20)` (keep last 20 messages). Add a guard: `if (next.length > 100) setMessages(next.slice(-50))` to prevent the local state itself from ballooning.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — scroll div ref callback):** `ref={el => { if (el) el.scrollIntoView(...); }}` is an inline function recreated on every render. In React, an inline ref callback fires on every render cycle (unmount then remount), causing scrollIntoView to fire even when no new message arrived, producing janky scroll behavior and wasted layout recalculations.
  → *Replace with a stable useCallback ref or a useEffect that depends on messages.length: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage — catch block):** The catch block silently swallows the actual error object. For an admin-facing tool this hides network failures, 401s (session expiry), and 500s, making debugging extremely difficult. A 401 should redirect to login rather than showing a generic chat error.
  → *Type the catch: `catch (err: unknown)`. Check `err instanceof Response && err.status === 401` to call `setLocation('/admin/login')`. Log other errors with `console.error('[AdminHelpChat]', err)` so they surface in monitoring without exposing them to the end user.*

## 💡 Opportunities

### Parallel Supabase seed execution — ⚡ Medium Impact
The seven Supabase init/seed functions are currently sequential. Steps that are independent (e.g. seedStaticModules, seedRegionalContent, seedGameContent) could run concurrently with Promise.allSettled, reducing server cold-start time by the sum of their individual latencies minus the longest one.

### Structured startup logging for route registration — 📌 Low Impact
Domain router registrations (`registerStudentRoutes`, `registerAiRoutes`, etc.) are awaited silently. Adding a timed log per registration (start/end ms) would make it easy to pinpoint which domain router is slowing cold starts in Railway deployment logs.

### Sitemap generation moved to build-time or cached response — 📌 Low Impact
The sitemap is rebuilt from a static array on every request including header reads. Since the URL list never changes at runtime, generate it once at module load into a `const sitemapXml` string and serve that constant, eliminating repeated string interpolation and header parsing per request.

### Idempotency token on bulk question delete-then-reinsert — 🔥 High Impact
The current idempotency strategy (delete all questions, reinsert on retry) creates a window where questions are absent if the job crashes between delete and insert. Wrapping the delete + insert in a database transaction (db.transaction(async tx => { ... })) eliminates this gap and ensures readers always see either the old set or the new set, never zero rows.

### Structured logging instead of thrown Error strings for job outcomes — ⚡ Medium Impact
Errors thrown with plain string messages lose context (paperId, attempt number, model used, token counts) that is critical for debugging AI extraction failures in production. Adopting a lightweight structured logger (e.g., pino) and attaching job metadata to every log call would cut mean-time-to-diagnosis significantly and is especially important for a student-facing app where silent failures affect learning outcomes.

### Cap and stream large CSV exports rather than building in memory — ⚡ Medium Impact
The admin-csv-export handler (visible in the truncated portion) likely accumulates all rows into a JS array before serialising. For schools with thousands of transactions or sessions this will cause GC pressure and potential OOM on constrained Replit instances. Switching to a streaming CSV approach (piping db cursor rows through a transform stream directly to object storage) would keep memory flat regardless of row count.

### Composite unique constraint on simulatedStocks.ticker — ⚡ Medium Impact
ticker is used as a display key and likely queried directly, but has no unique constraint or index. Adding uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) prevents duplicate tickers being seeded and makes ticker-based lookups O(log n) instead of O(n).

### Centralise currency default into a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract const DEFAULT_CURRENCY = 'BSD' into a shared config and reference it, so a currency change or multi-currency expansion requires one edit.

### Add DB-level check constraints for numeric ranges — 🔥 High Impact
amount, targetAmount, currentAmount, quantity, and pricePerUnit have no lower-bound check constraints — a bug or injection could persist negative balances or zero-quantity holdings silently. Adding check constraints (amount > 0, quantity > 0) in Drizzle's extras field catches these at the persistence layer before application logic runs, critical for a financial app.

### Structured JSON logging with redaction — 🔥 High Impact
Replace the freeform `console.log` string concatenation with a structured logger (pino is already common in the Node ecosystem and near-zero overhead). Define a redact list covering fields like `password`, `token`, `sessionId`, `answer` so sensitive keys are stripped automatically before any log ships to an aggregator. Expected impact: eliminates the PII-in-logs risk systematically and makes log querying in production significantly faster.

### Readiness and liveness HTTP endpoints — ⚡ Medium Impact
Add `GET /healthz` (liveness, always 200) and `GET /readyz` (readiness, checks DB probe result cached from startup) so Replit's load balancer and any future Kubernetes/Cloud Run deployment can distinguish a cold-starting instance from a crashed one. This prevents the scheduler intervals from receiving traffic before the DB is ready and avoids 502s for students during deploys.

### Atomic scheduler lock via advisory DB lock instead of insert-on-conflict — ⚡ Medium Impact
The current `onConflictDoNothing` claim pattern requires a delete-on-failure rollback path that can leave orphaned rows if the process crashes between insert and enqueue. Replace with a PostgreSQL advisory lock (`SELECT pg_try_advisory_xact_lock(hashtext('weekly-digest-' || weekStart || audience))`) inside a transaction so the lock is automatically released on crash with no rollback code needed. Reduces the risk of duplicate digest emails being sent to school users.

### Parallel admin data fetching with Promise.all — 🔥 High Impact
If the dashboard fires multiple independent useQuery calls sequentially (users, schools, metrics, audit log), they waterfall in React Query's default behavior when keys are derived from shared state. Consolidate into a single bootstrapping query using Promise.all on the server side or use `useQueries` (parallel hook) on the client to cut initial load time by the number of sequential round trips minus one.

### Memoize filtered/sorted table data — ⚡ Medium Impact
Admin tables with search, filter, and sort state almost certainly recompute their derived arrays on every keystroke render. Wrapping those computations in useMemo keyed on the raw data + filter/sort state will prevent O(n) array operations during unrelated re-renders (e.g., the chat panel opening/closing triggers a full parent re-render).

### Extract AdminHelpChat into its own file with React.memo — ⚡ Medium Impact
AdminHelpChat is a fully self-contained stateful widget but lives inside the same module as a large dashboard. Moving it to its own file and wrapping the export in React.memo ensures the chat panel never re-renders due to dashboard-level state changes (pagination, search), removing the largest unnecessary re-render source in the component tree.

### API Routes (`server/routes.ts`)

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads are stored in the job table (memory/DB pressure), question inserts are sequential in a loop instead of batched, and the file truncates before showing all handlers so additional issues may exist beyond what's visible. Core error boundaries are present but idempotency and parallelism gaps remain.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some composite indexes already in place. Key gaps are missing indexes on high-frequency filter columns (userId alone, nextDueDate, status), varchar(user_id) columns lacking explicit length constraints that mirror the auth model, and several timestamp/numeric columns missing NOT NULL that will silently store NULLs and break aggregation queries in a financial app.

### Server Startup (`server/index.ts`)
The server startup sequence contains blocking synchronous operations and sequential awaits that delay readiness, scheduler intervals are started without being stored for cleanup on shutdown, and the response-logging middleware serializes full JSON response bodies to logs on every API call which risks leaking sensitive financial or PII data for school-aged users. Graceful shutdown handling is absent, meaning in-flight jobs and DB connections are not cleanly terminated on SIGTERM.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth guards on sensitive mutations, unprotected async operations, and inline scroll refs that create new function instances on every render. The chat component leaks no resources but sends full message history to the backend on every turn without size-capping, creating a potential payload-growth DoS vector. Overall structure is reasonable but a few critical gaps need attention before production use.
