## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row a separate SELECT is fired to sum transactions (spent calculation) and another to fetch the category, producing 2N+1 database round-trips per call.

**Fix:** Replace the per-budget loop with a single query using Drizzle's leftJoin on categories and a subquery/groupBy aggregate for spent amounts: `db.select({ ...budgets, categoryName: categories.name, spent: sql<number>`COALESCE(SUM(transactions.amount),0)` }).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).where(eq(budgets.userId, userId)).groupBy(budgets.id, categories.id)`

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (buy/sell flow in routes calling storage)
**Issue:** Portfolio buy and sell operations call upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as independent awaits with no wrapping database transaction. A process crash or concurrent request between any two steps leaves balance and holdings permanently out of sync.

**Fix:** Wrap all three operations in a single `db.transaction(async (tx) => { ... })` block inside a new storage method e.g. `executeTrade(userId, stockId, quantity, price, side)`. Pass the transaction client `tx` to each sub-operation so they share the same atomic unit. Roll back automatically on any thrown error.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 103-105)
**Issue:** No rate limiting applied to AI/image/audio endpoints before they are registered. These are the most expensive endpoints in the app (OpenAI calls, image generation) and are completely unprotected against abuse, which is especially risky for a school-aged user base where accounts may be shared or compromised.

**Fix:** Apply a dedicated AI rate limiter before registering these routes: const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, standardHeaders: true }); app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Import the limiter from your existing rateLimiter module.

### `server/routes.ts`: robots.txt handler (~line 46) and sitemap.xml handler (~line 60)
**Issue:** Both handlers construct absolute URLs by trusting x-forwarded-host and host headers directly without validation. An attacker can send a crafted Host header causing the app to emit a Sitemap or canonical URL pointing to a malicious domain, which CDN or proxy caches may then serve to real users (host header injection / cache poisoning).

**Fix:** Validate the host against an allowlist from environment config: const ALLOWED_HOST = process.env.APP_HOST || 'finsightlite.com'; const host = ALLOWED_HOST; — ignore x-forwarded-host and req.headers.host entirely for constructing public-facing URLs. The protocol can similarly be locked via APP_PROTOCOL env var.

### `server/jobHandlers.ts`: runExtractPaper (paperId, fileB64, ext, subject)
**Issue:** Large base64-encoded file content (fileB64) is stored directly in the job payload column. For a multi-page PDF or high-res image this can easily be several megabytes per row, bloating the jobs table, slowing queue polls that SELECT * FROM jobs, and duplicating the file in memory on every retry attempt.

**Fix:** Upload the file to private object storage before enqueuing the job, store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), and have runExtractPaper download the buffer via objectStorageClient at runtime. This keeps job rows small and makes retries cheap.

### `shared/schema.ts`: simulatedStocks pgTable / ticker column
**Issue:** ticker has no unique index. Duplicate tickers can be inserted freely, which will corrupt portfolio calculations when code looks up a stock by ticker symbol. This is a data integrity gap with no FK or constraint to catch it.

**Fix:** Add a uniqueIndex inside the table callback: tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker). If multi-currency stocks with the same ticker are ever needed, scope it to uniqueIndex on (t.ticker, t.currency) instead.

### `shared/schema.ts`: portfolioHoldings / portfolioTransactions pgTable definitions
**Issue:** portfolioHoldings has no unique constraint on (userId, stockId), so a user can have multiple holding rows for the same stock. This forces application code to aggregate at query time and makes buy/sell logic error-prone. portfolioTransactions has no index on (userId, executedAt), making per-user transaction history queries a full scan.

**Fix:** Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) to portfolioHoldings to enforce one row per user per stock. Add index('idx_portfolio_txns_user_date').on(t.userId, t.executedAt) to portfolioTransactions for efficient history retrieval.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook call / top of AdminDashboard component
**Issue:** The component imports and calls useAdminAuth but the truncated code does not show a guard that returns null or redirects before any query hooks are invoked. If useAdminAuth returns an unauthenticated state after a brief token expiry, all child useQuery hooks still fire and their results (user lists, financial data) are rendered or logged before the redirect completes, exposing sensitive data in the React render tree and network tab.

**Fix:** Immediately after calling useAdminAuth, add: `if (!isAdmin) { setLocation('/admin/login'); return null; }` before any useQuery declarations. Ensure this check runs synchronously so no queries are registered for unauthenticated sessions.

## 🟡 Warnings (15)

- **`server/storage.ts` (getBillReminders):** N+1 pattern: each BillReminder row triggers a separate category lookup inside a loop, producing N+1 queries. For a class of 30 students each with 10 reminders this is 300 unnecessary round-trips per page load.
  → *Collect all distinct categoryIds from the fetched reminders, do one `db.select().from(categories).where(inArray(categories.id, categoryIds))`, build a Map<id, Category>, then attach categories in-process. Alternatively use a single leftJoin query from the start.*
- **`server/storage.ts` (getDashboardStats):** Four to six independent aggregate queries (total income, total expenses, savings rate, budget count, recent transactions, goal progress) are fired sequentially with individual awaits, adding latency proportional to query count × round-trip time.
  → *Wrap all independent aggregate SELECTs in a single `Promise.all([...])` call. They share no data dependencies so they can execute concurrently: `const [incomeRow, expenseRow, budgetRow, goalsRow] = await Promise.all([incomeQuery, expenseQuery, budgetQuery, goalsQuery])`. Expected latency reduction: from sum of RTTs to max of RTTs.*
- **`server/storage.ts` (getLeaderboard):** No pagination guard on the leaderboard query. The `limit` parameter in the filter is optional and likely defaults to undefined, meaning a single call can return every game session row joined with user data — unbounded result set that will grow with every student game played.
  → *Apply a hard server-side cap: `const safeLimit = Math.min(filters?.limit ?? 50, 100)` and always append `.limit(safeLimit)` to the Drizzle query. Document the max in the interface signature as `limit?: number /* max 100 */` so callers have a contract.*
- **`server/routes.ts` (Supabase init chain (~line 108-116)):** The seven-step Supabase init chain runs sequentially (each .then() waits for the previous), which unnecessarily serialises independent operations like seedRegionalContent and seedGameContent. A failure in seedFinancialAcademyLesson silently skips all subsequent seeds with only a console.error, leaving the DB in a partial state with no alerting.
  → *Group independent seeds into Promise.all batches and add structured error reporting: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => { console.error('[Supabase] Init error:', e); captureException(e); }); Import captureException from your existing Sentry module.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 93)):** The filePath parameter is passed directly to objectStorage.searchPublicObject without any path sanitisation. A request like /public-objects/../../private/secret could traverse outside the intended public directory depending on how ObjectStorageService resolves paths.
  → *Sanitise the path before use: import path from 'path'; const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); if (safePath !== filePath || filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' }); then pass safePath to searchPublicObject.*
- **`server/jobHandlers.ts` (for loop inside runExtractPaper (~line with 'for (let i = 0; i < questions.length')):** Extracted questions are inserted one-by-one with sequential awaits inside a for loop. For a 50-question paper this is 50 serial round-trips to the database, each blocked on the previous, dramatically increasing total job runtime and holding the DB connection longer than necessary.
  → *Collect all valid question objects into an array and call a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, at minimum batch with Promise.all on chunks of 10 to parallelise without overwhelming the pool.*
- **`server/jobHandlers.ts` (max_completion_tokens: 4096 (both openai.chat.completions.create calls)):** The OpenAI max_completion_tokens value is hardcoded to 4096 in two places. Changing the limit for production tuning or cost control requires a code deploy rather than a config change, and the same magic number is duplicated, creating a maintenance inconsistency risk.
  → *Add OPENAI_MAX_COMPLETION_TOKENS to environment config with a default fallback: const maxTokens = parseInt(process.env.OPENAI_MAX_COMPLETION_TOKENS ?? '4096', 10). Reference maxTokens in both completions calls. Document the env var in .env.example.*
- **`server/jobHandlers.ts` (runExtractPaper — await db.delete(extractedQuestions) at top of function):** The idempotency delete runs unconditionally on every attempt, including the first. On attempt 1 there is nothing to delete, but the call still executes a DELETE with a WHERE clause on paperId before any AI work begins. More critically, if the paper status was already set to 'completed' by a race-condition concurrent worker, this delete will silently wipe valid data without checking the current paper status first.
  → *Before deleting, fetch the current paper status: const paper = await storage.getExamPaper(paperId). If paper.status === 'completed' return early with the existing question count (true idempotency). Only delete existing questions if status is 'processing' or 'failed' and job.attempts > 1.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — file.createReadStream().on('error')):** The read stream error handler only fires if headers have not been sent, but there is no guarantee the 'error' event will be emitted synchronously before res.setHeader and pipe execute. If the stream emits an error after partial data has been piped, the response is left in a torn state (partial file sent, no error communicated to the client) and the error is silently swallowed without any logging.
  → *Attach the error handler before calling pipe and always log the error: stream.on('error', (err) => { console.error('[streamPrivateObject] stream error:', err); if (!res.headersSent) res.status(500).json({ message: 'Export stream failed' }); else res.destroy(err); }).pipe(res). Also add res.on('close', () => stream.destroy()) to avoid leaking the upstream read stream if the client disconnects.*
- **`shared/schema.ts` (userId: varchar('user_id')):** All user_id columns use bare varchar() with no length argument. In Drizzle/Postgres this maps to varchar without a limit, which is fine, but the intent is ambiguous and inconsistent with explicit-length best practice. More critically, if the auth model defines users.id as a fixed-length char or uuid type, the implicit cast may suppress index usage on joins and FK lookups across every table.
  → *Change every userId column to use uuid('user_id') type (or at minimum varchar(36)) to match the users.id primary key type exactly. Example: userId: uuid('user_id').notNull().references(() => users.id). This ensures the FK join uses the same operator class and the index is always hit.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow()):** createdAt is missing .notNull() on every table (categories, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions). defaultNow() supplies a default but does not prevent an explicit NULL insert, leaving audit timestamps unreliable for any query that filters or orders by createdAt.
  → *Add .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same fix to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (budgets pgTable definition):** budgets has no index on userId even though every budget query will filter by userId. With many users this becomes a full table scan. Similarly, billReminders has no index on (userId, nextDueDate) despite reminder-due queries needing both columns.
  → *Add compound indexes: on budgets add index('idx_budgets_user').on(t.userId) inside the table callback. On billReminders add index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). This directly mirrors the pattern already used on transactions.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > sendMessage):** The chat sends the full conversation history (all messages) to /api/admin/help-chat on every request. If a session grows to 50+ messages this payload becomes large, but more critically there is no maximum message count guard, meaning a user can cause unbounded server-side token consumption and cost on the LLM backend.
  → *Slice the messages array before sending: `messages: next.slice(-10)` to keep only the last 10 turns. Add a hard cap: if `next.length > 100` show a 'Start new conversation' prompt and reset state instead of sending.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > JSX > div ref callback):** The scroll-to-bottom ref is written as an inline arrow `ref={el => { if (el) el.scrollIntoView(...) }}`. React creates a new function reference every render and calls the old ref with null then the new ref with the element on every re-render, causing a scrollIntoView call on every state change including loading toggling — not just when a new message arrives.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > sendMessage):** sendMessage is defined as a plain async function inside the component body without useCallback. It is passed to the onKeyDown handler and the send button's onClick. Every render of AdminHelpChat creates new function references for both handlers, causing the input and button to re-render unnecessarily and also making it impossible to correctly debounce or memoize the handler downstream.
  → *Wrap sendMessage with useCallback: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and similarly wrap handleKey. This stabilises references across renders.*

## 💡 Opportunities

### Centralised query-result caching for market stock data: ⚡ Medium Impact
getMarketStocks is called on every portfolio page render but stock prices are only updated on a schedule. Adding a short-lived in-process cache (e.g. node-cache with 30 s TTL) would eliminate repeated identical SELECTs under classroom concurrent load without requiring Redis infrastructure.

### Typed storage error boundary with structured logging: 🔥 High Impact
Most async storage methods lack try/catch, so database errors propagate as unhandled rejections with raw Postgres messages (which can include table names and column values) reaching the HTTP response. Wrapping each public method with a shared `withStorageError(fn)` helper that catches, logs a sanitised message, and rethrows a domain error prevents data-model leakage and enables consistent error monitoring.

### Batch badge and XP grant via single transaction method: 🔥 High Impact
updateUserXp and addUserBadge are called sequentially in several reward flows with no atomicity. Introducing a single `grantReward(userId, xpDelta, badgesToAdd)` storage method that wraps both in db.transaction() eliminates the partial-grant race condition and reduces the call-site boilerplate duplicated across learning, game-session, and challenge completion paths.

### Parallel domain route registration: ⚡ Medium Impact
The six domain route register functions are called sequentially with await after being imported in parallel. If any registerX function performs async I/O (DB migrations, schema checks), this adds unnecessary latency. Wrap them in Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) — safe as long as no router depends on another being registered first, which is typical for Express sub-routers.

### Structured startup logging with timing: ⚡ Medium Impact
Adding timestamped structured logs around each init phase (Supabase seeds, route registration) would give observability into cold-start duration on Railway restarts. Currently a slow seed is invisible until a user hits a missing-data bug. A simple Date.now() diff logged to stdout with a [startup] prefix costs nothing and dramatically reduces debug time.

### Object storage client singleton guard: 📌 Low Impact
ObjectStorageService is instantiated at module load time as a module-level const. If the constructor opens a connection or registers event listeners, repeated hot-reloads in development will leak resources. Move instantiation inside a lazy getter or confirm the constructor is side-effect-free; if it holds connections, implement a destroy() call on process SIGTERM.

### Centralise OpenAI client import to avoid cross-module coupling: ⚡ Medium Impact
openai is imported from ./replit_integrations/chat/routes, meaning a job handler depends on an HTTP route module's side-effects to initialise the AI client. Extract the openai client instantiation into a shared ./lib/openai.ts singleton. This removes the hidden coupling, makes the client independently testable, and prevents the route module from being accidentally tree-shaken or refactored in a way that breaks job workers.

### Add structured job-level logging with paperId/jobId correlation: 🔥 High Impact
Currently failures surface only via thrown errors with no contextual metadata. Wrapping each handler's body with a lightweight logger that emits { jobType, jobId, paperId, attemptNumber, durationMs } at start and end would allow Datadog/CloudWatch to alert on P99 extraction latency, retry rates per job type, and terminal failures — high value for a production financial-literacy platform with SLA expectations.

### Move PDF parsing off the job worker thread with a dedicated queue: 🔥 High Impact
pdf-parse is a CPU-bound, synchronous-under-the-hood operation that blocks the Node.js event loop during text extraction. For large exam papers this stalls all other jobs sharing the same worker process. Offload PDF text extraction to a separate child_process.execFile call wrapping a small pdf-parse script, or promote it to its own job type so it can be scaled independently from the AI extraction step.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' and reference it in all .default() calls. When the app expands to TTD or XCD, one line changes instead of six scattered edits with risk of inconsistency.

### Replace text() with pgEnum for shared enum types: ⚡ Medium Impact
type ('income'|'expense'), status ('processing'|'completed'|'failed'), and riskLevel ('low'|'medium'|'high') are each defined inline as text enums on individual columns. Defining them once with Drizzle's pgEnum() lets Postgres enforce the constraint at the DB level, enables reuse across tables, and generates correct TypeScript union types automatically rather than duplicating them.

### Add partial index on documentUploads for unfinished processing jobs: 🔥 High Impact
Background workers that poll for documents in 'processing' status will scan the entire documentUploads table. A partial index such as CREATE INDEX idx_doc_uploads_pending ON document_uploads(created_at) WHERE status = 'processing' limits the scan to only in-flight rows, which is critical when completed rows dominate the table at scale.

### Parallel data fetching with Promise.all on dashboard load: 🔥 High Impact
The dashboard almost certainly fires multiple independent useQuery calls sequentially during mount (users, schools, quiz stats, leaderboard etc.). Consolidate related independent fetches into a single Promise.all inside one query fn or use React Query's useQueries to fan them out in parallel. This can cut initial load time by 40-60% on a slow Caribbean mobile connection where each round-trip adds 200-400ms.

### Virtualise large data tables with TanStack Virtual: 🔥 High Impact
An admin dashboard listing all students across all schools can easily reach 500-2000 rows. Rendering all rows in the DOM causes layout thrashing and slow paint times on low-end Android devices common in the target demographic. Adding @tanstack/react-virtual with a fixed row height will keep the DOM node count under 30 regardless of dataset size.

### Extract AdminHelpChat into its own lazy-loaded chunk: ⚡ Medium Impact
The chat component and its dependencies (Bot, MessageCircle, Send icons plus all chat state logic) are bundled into the main AdminDashboard chunk even though they are only needed when the user clicks Help. Wrapping AdminHelpChat in React.lazy + Suspense will shave the initial JS parse cost, improving Time-to-Interactive on the dashboard for admins who never open the chat.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
server/storage.ts shows systemic N+1 query patterns in budget and bill-reminder fetches, missing database transactions on multi-step writes (portfolio buy/sell, XP + badge grants), and sequential awaits that should be parallelised. For a school-aged financial app these patterns risk data inconsistency on concurrent requests and will degrade noticeably under classroom-scale load.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good patterns like parallel module loading and rate limiting on auth endpoints. Key concerns are: unhandled promise rejections on the Supabase init chain that could silently corrupt app state, missing rate limiting on AI/heavy endpoints registered via registerChatRoutes/registerImageRoutes/registerAudioRoutes, and a host header injection risk in robots.txt and sitemap.xml that could lead to cache poisoning.

jobHandlers.ts has several reliability and performance issues including large base64 payloads stored directly in the job table, sequential DB inserts that should be batched, and a hardcoded token limit that belongs in environment config. The extract-paper handler is the most problematic area, combining all three concerns with an additional memory-pressure risk from decoding large base64 buffers inside the job worker.

The schema is generally well-structured with proper foreign keys and some indexes already defined, but has several reliability and performance gaps. Key concerns are missing NOT NULL constraints on audit columns, absent indexes on high-cardinality lookup columns, and an undersized varchar for user_id that will silently truncate UUIDs in some Postgres configurations.
