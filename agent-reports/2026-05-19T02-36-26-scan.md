## 🔴 Critical (7)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (investment trade flow)
**Issue:** The buy/sell flow calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three separate awaits with no wrapping database transaction. A process crash or DB error between any two steps leaves the virtual balance debited but no holding recorded, or a holding updated with no audit trail — silent data corruption for student portfolios.

**Fix:** Wrap all three operations in a single db.transaction(async (tx) => { ... }) block using Drizzle's transaction API, passing the transaction client (tx) to each operation. Roll back automatically on any thrown error. Expose an atomic executeTrade(userId, stockId, quantity, price, type) method on the storage class that encapsulates this transaction.

### `server/storage.ts` — getLeaderboard
**Issue:** getLeaderboard fetches game sessions then performs per-user lookups (user name, avatar) in a loop — a classic N+1 pattern. With 30 students per class across many schools this will fire dozens of sequential DB round-trips per leaderboard render.

**Fix:** Rewrite as a single aggregated query using Drizzle's join: JOIN gameSessions with users on userId, GROUP BY userId, SELECT userId, users.name, users.avatar, SUM(score) as totalScore, COUNT(*) as gamesPlayed, ORDER BY totalScore DESC, LIMIT (filters.limit ?? 10). Apply the subject/period filters as WHERE clauses in the same query. Eliminate the post-fetch loop entirely.

### `server/routes.ts` — /api/auth/google handler
**Issue:** No rate limiting on the Google OAuth and AI endpoints. POST /api/auth/google and any downstream AI quota routes can be hammered freely, enabling credential-stuffing on the auth endpoint and runaway OpenAI spend on AI endpoints — both catastrophic for a school app with tight budgets.

**Fix:** Add express-rate-limit middleware scoped to these route groups before they are registered. Example: `import rateLimit from 'express-rate-limit'; const authLimiter = rateLimit({ windowMs: 15*60*1000, max: 20, standardHeaders: true }); app.post('/api/auth/google', authLimiter, handler);`. For AI endpoints use a stricter limit (e.g., max: 10 per minute per IP) and additionally gate on the existing quota system.

### `server/routes.ts` — escapeHtml import usage in email helpers and any catch blocks that log error objects
**Issue:** The import list includes escapeHtml from ./email, indicating user-supplied strings are rendered into email HTML. If any catch block or audit log call passes a raw Error object or req.body field directly into a log or email template without sanitisation, it can expose internal stack traces or reflect user input into outbound email — a XSS-equivalent risk for HTML email clients and an information-disclosure risk in logs.

**Fix:** Audit every `catch (e)` block: log only `e.message` (never the full object or stack) via a structured logger that redacts keys matching /password|token|secret/i. In email templates always pass user-originating strings through escapeHtml before interpolation, and add a lint rule (eslint-plugin-no-unsanitized) to enforce this statically.

### `server/jobHandlers.ts` — runExtractPaper (caller passes fileB64 from job.payload)
**Issue:** Raw base64 file content is stored directly in the job payload column. For a PDF or image this can be hundreds of kilobytes to several megabytes per row, bloating the jobs table, slowing queue polling queries, and potentially hitting column size limits.

**Fix:** Upload the file to object storage before enqueuing the job. Store only the object storage path in the payload (e.g. { paperId, objectPath, ext, subject }). In runExtractPaper, download the file from object storage using the path, then process it. This keeps job rows small and makes the queue table scannable.

### `shared/schema.ts` — simulatedStocks table definition
**Issue:** The ticker column has no unique constraint. Duplicate tickers (e.g. two rows with ticker='BAHA') would silently corrupt portfolio lookups and price calculations without any DB-level rejection.

**Fix:** Add a unique index in the table callback: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). If multi-region tickers are needed in future, make the unique index composite on (ticker, region).

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel — queryFn
**Issue:** The queryFn calls apiRequest and chains .then(r => r.json()) with no try/catch and no check on r.ok. If the server returns a 401, 403, or 500, r.json() will resolve with an error body object and rows will silently be set to [] (the default), hiding auth failures from the admin user entirely.

**Fix:** Add a response status check inside the queryFn: queryFn: async () => { const r = await apiRequest('GET', `/api/admin/audit-log?${params.toString()}`); if (!r.ok) throw new Error(`Audit log fetch failed: ${r.status}`); return r.json(); } — then handle the isError state in the UI to show an explicit failure message.

## 🟡 Warnings (17)

- **`server/storage.ts` (getTransactions (and getAllQuestions, getGameSessions)):** getTransactions applies filters.limit only when the caller passes it; when omitted there is no default cap. getAllQuestions and getGameSessions have no limit at all. As student transaction history and question banks grow, these queries will return unbounded result sets, causing memory pressure on the Node process and multi-second response times.
  → *Add a MAX_PAGE_SIZE = 500 constant (from env: parseInt(process.env.MAX_PAGE_SIZE ?? '500')). In getTransactions default limit to Math.min(filters?.limit ?? 100, MAX_PAGE_SIZE). Add .limit(MAX_PAGE_SIZE) to getAllQuestions and getGameSessions. Add an offset/cursor parameter to all three for proper pagination support.*
- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues multiple independent DB queries sequentially (total income, total expenses, category breakdown, recent transactions, savings goals, bill reminders). Each awaits the previous result despite having no data dependency on it, adding cumulative round-trip latency on every dashboard load.
  → *Group all independent queries into a single Promise.all([...]) call: const [incomeResult, expenseResult, categoryResult, recentTxns, goals, reminders] = await Promise.all([query1, query2, query3, query4, query5, query6]). This reduces wall-clock time to the slowest single query instead of the sum of all queries.*
- **`server/storage.ts` (getClassProgressSummary / getEnrollmentsByClass):** getClassProgressSummary calls getEnrollmentsByClass to get student IDs, then iterates each student to fetch their XP and learning progress individually with no try/catch. A single bad student record throws and kills the entire summary for the teacher. Additionally, sensitive fields (passwordHash visible on Teacher objects returned from getTeacherByEmail) are never stripped before the object leaves the storage layer.
  → *1) Wrap the per-student fetch loop in Promise.all() with individual .catch(()=>null) guards so one bad record doesn't abort the batch. 2) Add a sanitizeTeacher helper that deletes passwordHash before any Teacher object is returned from getTeacherByEmail or getTeacherById. 3) Add a top-level try/catch in getClassProgressSummary that logs the error and returns a safe empty summary rather than propagating a 500.*
- **`server/routes.ts` (/api/auth/google handler — resolvedClass branch with three sequential awaits):** getOrgEnvironmentById, getOrganization, and later getOrgEnvironmentByJoinCode are all awaited sequentially inside the classCode resolution block. Each is an independent network round-trip to Supabase, adding 100-300 ms of unnecessary latency on every student login.
  → *Where the results are independent, use Promise.all. For example: `const [classEnv, resolvedOrgEnv] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), getOrgEnvironmentByJoinCode(code)]);`. Then derive classOrg only after classEnv resolves: `const classOrg = classEnv ? await getOrganization(classEnv.org_id) : null;`.*
- **`server/routes.ts` (objectStorage instantiation and multer videoUpload config):** The 500 MB video upload limit and object storage configuration are hardcoded constants with no environment-variable override. In production the limit may need to differ per deployment tier, and hardcoding it makes A/B tuning or emergency reduction impossible without a code deploy.
  → *Replace hardcoded limits with env vars: `limits: { fileSize: Number(process.env.VIDEO_UPLOAD_MAX_BYTES ?? 500 * 1024 * 1024) }`. Do the same for logo (LOGO_UPLOAD_MAX_BYTES) and exam (EXAM_UPLOAD_MAX_BYTES) multer instances. Validate these vars at startup with zod so a misconfigured deploy fails fast.*
- **`server/routes.ts` (/healthz handler):** The health-check endpoint imports db and drizzle dynamically on every request (`await import('./db')`). Dynamic imports are not free — Node caches them after the first call but the import machinery still runs a module-registry lookup on every hit. More critically, the endpoint is unauthenticated and publicly reachable, leaking the fact that a database is present and its liveness state to unauthenticated callers.
  → *Move the db and sql imports to the top of the file (static imports) so there is zero per-request overhead. Add a simple shared secret check for external monitoring: `if (req.headers['x-health-token'] !== process.env.HEALTH_TOKEN) return res.status(404).end();` — this keeps uptime monitors working while hiding the endpoint from scanners.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a paper with 60 questions this means 60 serial round-trips to the database before the job can complete, multiplying latency and holding the DB connection for the entire duration.
  → *Collect all valid question objects into an array first, then do a single batch insert: await db.insert(extractedQuestions).values(validQuestions). Add the same idempotency delete before the batch insert as already exists. This reduces 60 round-trips to 1.*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export') — type AdminRow, let dat):** The CSV export handler has no idempotency guard. If the job is retried after a partial write (e.g. the upload succeeded but the job record was not marked complete), a second run will re-query, re-build, and overwrite the object storage file silently. More critically, the truncated code shows `let dat` suggesting data assembly may span multiple sequential awaits (fetching users, then transactions, then modules separately) that could run in parallel.
  → *At the start of the handler, check whether the export object already exists in object storage (file.exists()) and return early with the existing path if found — use a deterministic object path keyed on job.id or a content hash. For the data fetching, wrap independent queries in Promise.all: const [userData, txData, moduleData] = await Promise.all([db.select()...users, db.select()...transactions, db.select()...learningModules]).*
- **`server/jobHandlers.ts` (runExtractPaper — image_url content block):** When processing an image, the buffer is decoded from base64 (Buffer.from(fileB64, 'base64')) and then immediately re-encoded to base64 (buffer.toString('base64')) to embed in the OpenAI request. If fileB64 is already the raw base64 string from the job payload, this is a redundant decode/re-encode cycle and doubles peak memory usage for the image data.
  → *Pass fileB64 directly as the base64 string in the data URL: `data:${mimeType};base64,${fileB64}`. Only decode to a Buffer for the pdf-parse path where a Buffer is required. This halves the memory allocated for image payloads during vision extraction.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId has no index despite being the primary filter column for all budget queries. Every budget lookup by user triggers a full table scan.
  → *Add an index in the budgets table's second argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also add a composite uniqueIndex on (userId, categoryId, period) to enforce one budget per user per category per period at the DB level.*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, documentUploads table definitions):** None of these tables define any indexes on userId, despite every application query filtering by userId. At even modest scale (thousands of rows per user) this causes full table scans on every fetch.
  → *Add index callbacks to each table: billReminders — index('idx_bill_reminders_user_id').on(t.userId) and index('idx_bill_reminders_next_due').on(t.userId, t.nextDueDate); savingsGoals — index on userId; portfolioHoldings — index on (userId, stockId); documentUploads — index on (userId, status).*
- **`shared/schema.ts` (varchar('user_id') on all tables):** All userId columns are declared as varchar with no length argument, which in PostgreSQL resolves to unbounded character varying — identical to text. This provides no storage or validation benefit over text and misleads readers about expected input size.
  → *Specify a length matching your auth provider's ID format, e.g. varchar('user_id', { length: 36 }) for UUID-based IDs or { length: 128 } for Clerk/Firebase-style IDs. This adds a DB-level guard against malformed foreign keys and documents intent.*
- **`shared/schema.ts` (createdAt columns across all tables; purchasedAt, executedAt in portfolio tables):** All createdAt/purchasedAt/executedAt columns use .defaultNow() without .notNull(), making them nullable in the Drizzle type system. An INSERT that explicitly passes null will succeed, corrupting audit and ordering queries that rely on these timestamps.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt and purchasedAt in portfolioTransactions and portfolioHoldings.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object and query string are reconstructed on every render outside of useMemo, causing the queryKey to be stale relative to actual params being sent. The params object is built before the useQuery call but not memoized, so if parent re-renders occur the params object reference changes but queryKey may not, leading to query/params desync.
  → *Wrap both the params construction and the queryKey in a single useMemo: const { params, queryKey } = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return { params: p, queryKey: ['/api/admin/audit-log', actorType, action, from, to] }; }, [actorType, action, from, to]); Then use params.toString() inside the queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery for /api/admin/observability has no isLoading or isError handling. If the request fails or is pending, the component silently renders with undefined data, and badge class expressions like sentry?.serverEnabled will render 'Disabled' misleadingly even during load — a user might act on false negative status.
  → *Destructure isLoading and isError from useQuery. Return a skeleton or spinner when isLoading is true, and a visible error state (e.g., an alert card) when isError is true. At minimum: if (isLoading) return <Loader2 className='animate-spin' />; if (isError) return <div className='text-red-400'>Failed to load observability data</div>;*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log page size limit is hardcoded as the string '200' directly in the component. For a school-facing financial app this creates two problems: it cannot be tuned per environment (dev vs prod), and '200' rows fetched unconditionally on every filter change puts unnecessary load on the DB with no pagination.
  → *Move the limit to an environment variable: const AUDIT_LOG_LIMIT = import.meta.env.VITE_AUDIT_LOG_LIMIT ?? '50'; and add cursor-based or offset pagination controls to the panel rather than fetching a fixed large batch.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** fmtDate accepts type any and calls new Date(d) without validation. Passing an invalid date string (e.g., malformed API data) will render 'Invalid Date' silently in the UI. For a financial literacy app displaying transaction or enrollment dates to school admins, silent invalid dates erode trust.
  → *Add a validity check: function fmtDate(d: unknown): string { if (!d) return '—'; const dt = new Date(d as string); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); } Also type the parameter as string | number | null | undefined instead of any.*

## 💡 Opportunities

### Centralised pagination middleware — 🔥 High Impact
Introduce a PaginationParams type { limit: number; offset: number } and a parsePagination(query) helper validated against MAX_PAGE_SIZE. Apply it consistently to every list method in IStorage. This removes the current ad-hoc per-method limit logic, prevents regressions when new list endpoints are added, and gives the API a consistent cursor contract for the React Query layer.

### Storage-layer audit logging for financial writes — ⚡ Medium Impact
Wrap createTransaction, updateVirtualBalance, and executeTrade (once atomic) with a lightweight audit log insert (userId, action, delta, timestamp) inside the same DB transaction. For a regulated-adjacent financial literacy app this creates an immutable paper trail with zero extra round-trips and satisfies future compliance requirements without retrofitting.

### Environment-driven seed guards — ⚡ Medium Impact
seedMarketData and seedLearningModules currently have no guard against running in production — a misconfigured startup script or migration runner could duplicate seed data. Add an early return if process.env.NODE_ENV === 'production' and a DB-side idempotency check (INSERT ... WHERE NOT EXISTS or ON CONFLICT DO NOTHING) so seeds are safe to call on every deploy.

### Centralise multer instance configuration — ⚡ Medium Impact
Four separate multer instances (upload, logoUpload, videoUpload, examUpload) duplicate fileFilter and limits logic. Extract a factory function `makeUpload({ maxBytes, mimePattern, extensions })` that reads limits from env vars and returns a configured instance. This removes ~60 lines of duplication and ensures limit changes propagate everywhere from one place.

### Batch Supabase lookups in enrolment and lesson-fetch paths — 🔥 High Impact
Patterns like getStudentOrgIds followed by per-org getLessonsByOrg in a loop result in N+1 Supabase queries. Replacing these with a single joined query or a Promise.all over the org IDs — then deduplicating in application memory — would cut lesson-list latency by 60-80% for students enrolled in multiple environments.

### Move AI quota cache to a shared Redis layer — 🔥 High Impact
getCachedExplanation and setCachedExplanation appear to use in-process memory (hashTutorQuestion suggests a Map). Under multiple Node workers or a restart this cache is cold, causing redundant OpenAI calls. Replacing the in-process Map with a Redis GETEX/SETEX call (TTL 24 h) would reduce repeat-question API spend by an estimated 30-50% in classroom settings where many students ask similar questions.

### Deterministic object-storage paths for re-entrant exports — 🔥 High Impact
Key all exported CSV object paths on a stable identifier (e.g. export-type + ISO date truncated to the day, or job.id) so that any retry of a completed export can detect the existing file and skip regeneration entirely. This makes the export handler fully idempotent at zero extra cost and avoids redundant DB scans on retries.

### Structured logging instead of thrown Error strings — ⚡ Medium Impact
Errors thrown in job handlers currently carry only a message string, losing paperId, attempt count, and subject context. Wrapping errors with context (e.g. Object.assign(err, { paperId, attempt: job.attempts })) before rethrowing gives the job runner and any log aggregator structured fields to alert and group on without adding a logging library dependency.

### Extract paper text-length cap to environment config — 📌 Low Impact
The slice(0, 30000) limit on extracted PDF text is hardcoded. Moving it to an environment variable (PDF_TEXT_CHAR_LIMIT) lets the limit be tuned per deployment without a code deploy, and makes it visible as an operational parameter rather than a magic number buried in business logic.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, portfolioHoldings). Extract it as a const DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it in each .default() call. This means a currency change requires one edit instead of six, eliminating the risk of inconsistent defaults across tables.

### Add partial index on documentUploads for processing status — ⚡ Medium Impact
Background jobs polling for unprocessed uploads will repeatedly scan all documentUploads rows. A partial index — WHERE status = 'processing' — would make these polling queries O(unprocessed) instead of O(all uploads), which matters significantly as completed records accumulate over months.

### Add check constraint on numeric amount columns — 🔥 High Impact
transactions.amount, budgets.amount, savingsGoals.targetAmount, and billReminders.amount have no lower-bound constraint. A bug or malicious input could insert zero or negative amounts that pass FK and type validation but produce nonsensical financial data. Add a Drizzle sql-based check constraint amount > 0 on each monetary column to enforce this invariant at the database layer rather than relying solely on application-level Zod validation.

### Consolidate per-tab data fetching with parallel Promise.all — 🔥 High Impact
Each tab panel likely fires its own independent useQuery on mount. The overview/dashboard tab in particular probably needs counts from organizations, schools, teachers, and students sequentially or in isolated hooks. Wrapping these in a single /api/admin/summary endpoint fetched with one query (or using Promise.all in a single queryFn) would cut dashboard load time from 4 sequential round-trips to 1, which is meaningful on Caribbean mobile networks with higher latency.

### Extract tab panels into code-split lazy components — 🔥 High Impact
AdminDashboard appears to import all tab panels (including Recharts, VideoField, DB viewer) in a single bundle. Using React.lazy + Suspense per tab would reduce initial bundle size significantly — Recharts alone is ~150KB gzipped. School admin users on slow connections would see the dashboard shell faster.

### Add React.memo to ObservabilityCard and AuditLogPanel — ⚡ Medium Impact
Both components are defined inside or at module level but will re-render whenever AdminDashboard's active tab state changes, even when they are not visible. Wrapping them in React.memo prevents unnecessary reconciliation and query-refetch-triggered re-renders from propagating into hidden panels.

For a school-aged financial app these issues carry real risk — a failed mid-write leaves corrupted data, and unbounded queries on a growing student dataset will cause latency spikes.

No runtime code issues are visible in this file, but schema-level gaps will cause query performance and data integrity problems at scale.
