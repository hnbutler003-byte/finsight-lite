## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. Any unauthenticated or authenticated user can hammer these endpoints, incurring unbounded OpenAI API costs and potential DoS for a school-aged user base.

**Fix:** Create a dedicated `aiLimiter` (e.g. 10 req/min per IP via express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 44) and sitemap.xml handler (~line 58)
**Issue:** The `host` and `proto` values are taken directly from `x-forwarded-host` and `x-forwarded-proto` request headers without any validation or allowlist. A spoofed Host header will cause the app to emit a crafted absolute URL in both robots.txt and sitemap.xml, enabling cache-poisoning or open-redirect attacks.

**Fix:** Read the canonical base URL from a validated environment variable: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and remove all header-derived URL construction from these handlers. Set APP_BASE_URL in Railway config.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** setInterval return values are discarded, so the interval handles cannot be cleared during graceful shutdown. If the server receives SIGTERM, both schedulers keep firing against a potentially closed DB connection, causing unhandled errors and preventing the process from exiting cleanly.

**Fix:** Store handles: `const digestInterval = setInterval(...)`. In the SIGTERM/SIGINT handler call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` before closing the HTTP server and DB pool.

---

### `server/index.ts`: res.json monkey-patch in request logging middleware
**Issue:** capturedJsonResponse holds the full serialized response body and is unconditionally appended to the log line for every /api route. For a financial literacy app serving school-aged users this will log quiz answers, scores, AI responses, and potentially session-adjacent data to stdout, violating data-minimisation expectations and creating a PII exposure risk in any log aggregator.

**Fix:** Remove body logging entirely in production: `if (process.env.NODE_ENV !== 'production' && capturedJsonResponse) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`. In production log only method, path, status code, and duration.

---

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** The POST to /api/admin/help-chat sends the full message history as a plain array with no size cap. A user can accumulate thousands of messages and send an unbounded payload to the server, causing OOM or abuse of any upstream LLM API with no rate-limit or max-turn guard on the client.

**Fix:** Slice the history before sending: `messages: next.slice(-20)` (keep last 20 turns). Also add a character limit check: `if (text.length > 1000) return;` before building `next`. Add a visible message-count warning in the UI when approaching the limit.

---

### `client/src/pages/AdminDashboard.tsx`: apiRequest / mutation handlers
**Issue:** Mutation handlers that perform destructive actions (delete user, delete school, promote to admin) call `apiRequest` inside `useMutation` without any secondary auth confirmation beyond a UI dialog. If the admin session cookie is stolen via XSS, an attacker can silently execute deletions. Additionally, sensitive data (user emails, IDs) appears to be logged via `console.log` or included in unredacted toast messages visible in browser DevTools.

**Fix:** On the server, enforce re-verification of admin role on every mutating endpoint (do not rely solely on session). On the client, remove any `console.log` statements containing user PII. For irreversible actions (delete, promote) require the admin to type a confirmation string (e.g. the resource name) in the dialog before enabling the confirm button, reducing accidental or automated abuse.

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...) (~line 110)):** The Supabase init is a sequential `.then()` chain with a single catch at the end. If `seedFinancialAcademyLesson` rejects, the subsequent `initStaticContentTables`, `seedStaticModules`, `seedRegionalContent`, and `seedGameContent` steps are all silently skipped, leaving the DB in a partial state with no per-step error visibility.
  → *Run independent seeds in parallel where possible and log per-step failures: wrap each call in its own try/catch inside an async IIFE, or use `Promise.allSettled([seedFinancialAcademyLesson(), initStaticContentTables(), ...]).then(results => results.forEach((r,i) => r.status==='rejected' && console.error(`[Supabase] step ${i} failed:`, r.reason)));`*
- **`server/routes.ts` (/healthz handler (~line 80)):** The `/healthz` endpoint dynamically imports `./db` and `drizzle-orm` on every request. Dynamic imports are cached after the first load, but the pattern is misleading and adds unnecessary async overhead on hot-path health checks used by Railway's load balancer polling at high frequency.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports, then use them directly inside the route handler.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups will always filter by userId. At scale this becomes a full table scan for every dashboard load.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also prevents duplicate budget entries per user/category.*
- **`shared/schema.ts` (varchar('user_id') on all tables):** All userId columns are declared as varchar with no length argument, which in PostgreSQL defaults to unlimited length. This wastes storage and skips DB-level enforcement of the auth provider's ID format (e.g. Clerk uses 32-char IDs). Inconsistency across tables also risks silent truncation bugs if a length is added later to only some columns.
  → *Determine the max length of the auth provider's user ID (e.g. 32 for Clerk) and declare all userId columns as varchar('user_id', { length: 32 }). Apply consistently across every table: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Neither portfolioHoldings nor portfolioTransactions has any index on userId or stockId. Portfolio transaction history queries (ORDER BY executedAt, WHERE userId = ?) will perform sequential scans as data grows, which is unacceptable for a per-user financial history feature.
  → *Add indexes: portfolioHoldings gets index('idx_holdings_user').on(t.userId) and uniqueIndex('idx_holdings_user_stock').on(t.userId, t.stockId). portfolioTransactions gets index('idx_ptxn_user_executed').on(t.userId, t.executedAt) to support paginated history queries.*
- **`shared/schema.ts` (transactions createdAt, budgets createdAt, documentUploads createdAt, savingsGoals createdAt):** createdAt columns across multiple tables use .defaultNow() but omit .notNull(). This means application code or a raw INSERT can explicitly pass NULL for createdAt, producing rows with no audit timestamp — breaking any reporting or ordering logic that assumes createdAt is always populated.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.*
- **`shared/schema.ts` (simulatedStocks lastPriceUpdateDate column):** lastPriceUpdateDate is stored as text rather than timestamp. Sorting or range-filtering stocks by last update date (e.g. to detect stale prices) requires full table scans with string comparison, and the text type provides no DB-level validation that the value is actually a valid date.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed data or price-update service that writes a string to instead pass a JS Date object or ISO timestamp that Drizzle will handle correctly.*
- **`server/index.ts` (seedDatabase / probeDatabase calls in startup IIFE):** seedDatabase and probeDatabase are awaited sequentially during startup when they are likely independent operations, adding unnecessary latency before the HTTP server begins accepting connections. If either rejects without a top-level catch the process crashes with an unhandled rejection rather than a clean error message.
  → *Wrap both in Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])`. Wrap the entire startup IIFE in try/catch that calls `captureError` then `process.exit(1)` so failures surface cleanly instead of as unhandled rejections.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` where the `+ 0` is a no-op, suggesting copy-paste error. More critically, the date arithmetic uses the server's local timezone, so on a UTC host '7 PM Sunday' fires at a different wall-clock time than intended for Caribbean (UTC-4/UTC-5) users, meaning digests may be sent at 11 PM or midnight local time.
  → *Replace the local-time arithmetic with an explicit UTC-offset calculation: accept a timezone string (e.g. `America/Port_of_Spain`) from `process.env.DIGEST_TIMEZONE`, use `Intl.DateTimeFormat` or the `luxon` library already available to compute the target instant, then compare against `Date.now()`.*
- **`server/index.ts` (maybeRunAiUsagePurge / AUTO_PURGE_OLDER_THAN_DAYS interval):** The purge scheduler runs on a fixed interval (visible in truncated code) but has no concurrency guard. If a purge job takes longer than the interval period a second invocation starts before the first finishes, potentially issuing concurrent DELETE statements that race each other, inflate DB load, and could double-delete rows if the ORM batches deletes with LIMIT.
  → *Add a module-level boolean flag `let purgeRunning = false`. At the top of `maybeRunAiUsagePurge` return early if `purgeRunning` is true; set it to true before the DB call and reset it in a finally block after the call completes.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The auto-scroll sentinel uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` directly in JSX. This creates a new function on every render, fires scrollIntoView on every re-render (including while the user is reading), and is never cleaned up. On a slow device or long conversation this causes continuous layout thrashing.
  → *Replace with a stable `useRef` + `useEffect`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then `<div ref={bottomRef} />`. This scrolls only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – handleKey):** `handleKey` and `sendMessage` are redefined as plain arrow functions on every render of AdminHelpChat. Because they close over `messages`, `input`, and `loading`, they prevent React from batching state updates correctly and cause the input element to receive a new `onKeyDown` reference on every keystroke, forcing unnecessary DOM patching.
  → *Wrap both in `useCallback`: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e) => { ... }, [sendMessage]);`. This stabilises references and prevents unnecessary re-renders of the input and button.*
- **`client/src/pages/AdminDashboard.tsx` (useQuery calls across the dashboard):** Multiple `useQuery` hooks fetch independent resources (users, schools, teachers, analytics) sequentially at component mount with no `staleTime` set. Without `staleTime`, every navigation to this page or parent re-render refetches all endpoints simultaneously, hammering the API and causing cascading loading spinners visible to the admin.
  → *Add `staleTime: 60_000` (1 minute) to each dashboard query that is not real-time critical, e.g. `useQuery({ queryKey: ['admin-users'], queryFn: ..., staleTime: 60_000 })`. For analytics charts use `staleTime: 300_000`. This eliminates redundant network calls on revisit while keeping data reasonably fresh.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import are executed sequentially. If any of these registration functions do async work (DB calls, schema checks), they block each other. Refactor to `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` to reduce server cold-start time, which directly impacts first-request latency for Caribbean users on slower connections.

### Object storage filePath traversal guard: 🔥 High Impact
The `/public-objects/:filePath(*)` handler passes `req.params.filePath` directly to `objectStorage.searchPublicObject` with no sanitisation. Add a check `if (filePath.includes('..') || path.isAbsolute(filePath)) return res.status(400).json({error:'Invalid path'});` before the storage call to prevent path traversal attempts against the object storage bucket.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replace `console.error` in the Supabase init chain with a structured logger (e.g. pino) that includes timestamps, step names, and error codes. This makes Railway log aggregation searchable and allows alerting on seed failures before they impact students accessing lesson content in production.

### Composite unique constraint on portfolioHoldings: 🔥 High Impact
Adding uniqueIndex on (userId, stockId) in portfolioHoldings prevents duplicate holding rows for the same user+stock, which would silently corrupt quantity aggregations. This is a data-integrity guarantee that application-layer upsert logic alone cannot reliably provide under concurrent requests.

### Centralize currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table columns. If the app needs to support TTD or JMD for other Caribbean markets, every column default must be updated individually and a migration run. Extract a DEFAULT_CURRENCY constant from an environment variable or shared config and reference it in all column definitions.

### Add billReminders index on userId and nextDueDate: ⚡ Medium Impact
The most common bill reminder query will be 'fetch upcoming bills for user X ordered by nextDueDate'. Without a composite index on (userId, nextDueDate) this is a full scan plus sort. Adding index('idx_reminders_user_due').on(t.userId, t.nextDueDate) allows the DB to satisfy the query with an index range scan.

### Formalise graceful shutdown lifecycle: 🔥 High Impact
Consolidate SIGTERM and SIGINT handling into a single shutdown function that: clears all intervals, stops accepting new HTTP connections via httpServer.close(), drains in-flight requests with a 10-second timeout, then closes the DB pool. This prevents zombie intervals and torn transactions on Replit container restarts, which are frequent.

### Extract scheduler state into a dedicated module: ⚡ Medium Impact
Move maybeRunWeeklyDigest, maybeRunAiUsagePurge, their intervals, and their concurrency guards into a `server/schedulers.ts` module that exports start() and stop() functions. This makes the startup sequence in index.ts linear and readable, and makes the schedulers independently testable without spinning up the full Express app.

### Add structured logging with log-level filtering: ⚡ Medium Impact
Replace the hand-rolled `log()` function with a structured logger (pino is already common in the Node ecosystem and has near-zero overhead). Set LOG_LEVEL from an env var so debug/body logging is off by default in production. This eliminates the PII-in-logs risk at the infrastructure level rather than relying on per-callsite if-checks.

### Memoize filtered and sorted table data: 🔥 High Impact
User/school table rows are likely filtered and sorted on every render using inline `.filter().sort()` chains in JSX or render body. Wrapping these in `useMemo` keyed on the raw data + search string + sort column will prevent O(n log n) work on every keystroke in the search box, noticeably improving responsiveness with 500+ users.

### Parallel-fetch independent admin bootstrap queries with Promise.all: ⚡ Medium Impact
If any dashboard section awaits multiple independent resources sequentially (e.g. fetching schools then fetching teachers in separate serial awaits inside a single async function), converting them to `Promise.all([fetchSchools(), fetchTeachers()])` would halve the initial load time by using the available network concurrency.

### Extract chart components to prevent full dashboard re-renders: ⚡ Medium Impact
Recharts `BarChart` and `LineChart` are rendered inline in the main dashboard component. Any state change (search input, pagination, dialog open) re-renders and re-paints all charts. Extracting them into `React.memo`-wrapped child components accepting only their data prop will isolate re-renders to the chart subtree only when analytics data actually changes.
