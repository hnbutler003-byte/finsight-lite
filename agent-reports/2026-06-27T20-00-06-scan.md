## 🔴 Critical (5)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the associated category and another aggregates spent transactions. With 10 budgets this fires 21 queries per dashboard load.

**Fix:** Rewrite as a single query using Drizzle's leftJoin on categories and a subquery or aggregated join for spent amounts: db.select({ ...budgets, categoryName: categories.name, spent: sql<number>`COALESCE(SUM(t.amount),0)` }).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).groupBy(budgets.id, categories.id).where(eq(budgets.userId, userId))

---

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Buy/sell stock flow calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three separate awaits with no wrapping database transaction. A failure between any two steps leaves balances and holdings in an inconsistent state.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { ... }) passing the tx client to each operation so the entire trade is atomic and rolls back on any error.

---

### `server/storage.ts`: getTransactions
**Issue:** No upper bound on the limit filter and no default page size. Passing limit=undefined returns every transaction row for the user, which can be thousands of rows for active users, causing unbounded memory growth per request.

**Fix:** Add a hard cap: const safeLimit = Math.min(filters?.limit ?? 50, 200); and apply .limit(safeLimit) unconditionally. Add an offset parameter for cursor-based pagination.

---

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** The full file contents are stored as base64 in the job payload (fileB64). For a PDF or image this can easily be several MB per job row, bloating the jobs table, slowing queue polls, and risking row-size limits in Postgres.

**Fix:** Upload the file to private object storage at enqueue time and store only the object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage before processing. This keeps job rows small and decouples file lifetime from job retention.

---

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** No rate limiting or debounce on the admin help chat send action. A user (or script) can spam POST /api/admin/help-chat repeatedly because the only guard is the local `loading` boolean, which resets after each response. On a school platform this is a direct cost and abuse vector against an LLM endpoint.

**Fix:** Add a cooldown ref: `const lastSent = useRef(0); if (Date.now() - lastSent.current < 1000) return; lastSent.current = Date.now();` at the top of sendMessage. Also enforce a server-side rate limit (e.g., express-rate-limit) on POST /api/admin/help-chat keyed to the admin session.

## 🟡 Warnings (14)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits fire getTransactions, getBudgets, getSavingsGoals, and getBillReminders one after another. These are independent queries with no data dependency between them, adding ~3x avoidable round-trip latency on every dashboard load.
  → *Replace sequential awaits with: const [txns, budgets, goals, reminders] = await Promise.all([getTransactions(userId, filters), getBudgets(userId), getSavingsGoals(userId), getBillReminders(userId)]); then compute aggregates from the resolved arrays.*
- **`server/storage.ts` (seedMarketData / seedLearningModules):** Stock prices, module content, and XP thresholds are hardcoded literals inside seed functions rather than sourced from environment config or a seed data file. Changes require code edits and redeployment, and values like initial virtual balance (likely hardcoded to a fixed number) cannot be tuned per school or sponsor without code changes.
  → *Extract seed constants into a dedicated seeds/marketData.ts and seeds/learningModules.ts file. Read any numeric tuning values (starting balance, XP per level) from process.env with documented defaults, e.g. const STARTING_BALANCE = Number(process.env.VIRTUAL_STARTING_BALANCE ?? 10000);*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question exam paper this fires 40 round-trips to the database serially, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Collect all valid question objects into an array and perform a single batch insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, replace the loop with await Promise.all(validQuestions.map((q, i) => storage.createExtractedQuestion({...}))) only if the underlying function is safe to run concurrently; otherwise prefer the single bulk insert.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (model: 'gpt-4o')):** The OpenAI model name 'gpt-4o' is hardcoded in two separate completions calls. If the model needs to change (cost control, deprecation, A/B testing) it requires a code deploy rather than a config change, and the two call sites can drift out of sync.
  → *Add OPENAI_MODEL to environment config (e.g., process.env.OPENAI_MODEL ?? 'gpt-4o') and reference it via a single imported constant. Apply to both the vision extraction call and the JSON extraction call.*
- **`server/jobHandlers.ts` (import { openai } from './replit_integrations/chat/routes'):** The openai client is imported from a route file rather than a dedicated singleton module. Route files carry Express middleware side-effects and their own imports; importing from them in job handlers creates a hidden coupling that can cause circular dependency issues, double-initialization, or break if the route file is refactored.
  → *Extract the openai client instantiation into its own module (e.g., server/lib/openaiClient.ts) and import from there in both the route file and jobHandlers.ts.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch. An unexpected DB error (connection drop, lock timeout) will bubble up as an unhandled rejection. Unlike extract-paper, there is no outer error boundary, so the job may silently fail without updating any status or emitting a useful log.
  → *Wrap the delete operation in try/catch, log the error with enough context (olderThanDays, cutoff date), and rethrow so the job framework can record the failure and schedule a retry.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no composite index, yet budget lookups will always filter by userId. At scale this becomes a full table scan per user dashboard load.
  → *Add a composite index in the budgets table factory: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Mirror the pattern already used on transactions.*
- **`shared/schema.ts` (billReminders table definition):** billReminders.nextDueDate has no index. The primary query pattern for reminders is 'fetch upcoming bills for user X due before date Y', which requires filtering on both userId and nextDueDate. Without an index this is a sequential scan.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders table definition.*
- **`shared/schema.ts` (simulatedStocks table definition — ticker column):** ticker has no unique index. Portfolio queries join on ticker for price lookups, and duplicate tickers would silently corrupt simulation results. There is also no index, so any ticker-based lookup is a full scan.
  → *Replace the plain text column with: ticker: text('ticker').notNull(), and add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to enforce uniqueness and enable fast lookups.*
- **`shared/schema.ts` (transactions, budgets, savingsGoals, billReminders, documentUploads — createdAt columns):** createdAt is defined as .defaultNow() without .notNull(). Drizzle will emit the column as nullable in Postgres, meaning application code or direct DB inserts can store NULL, breaking any ORDER BY createdAt DESC queries and audit logic silently.
  → *Change every createdAt definition to: createdAt: timestamp('created_at').defaultNow().notNull(). Same fix applies to purchasedAt on portfolioHoldings and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is varchar without a length specifier, matching however wide users.id is, but more critically it has no index. Every per-user category fetch (sidebar, transaction form) will full-scan the categories table filtering on userId IS NULL OR userId = $1.
  → *Add an index: (t) => ({ userIdIdx: index('idx_categories_user_id').on(t.userId) }). Also consider adding .notNull() with a sentinel system-user ID rather than using NULL to represent system categories, which simplifies query logic and avoids IS NULL scans.*
- **`client/src/pages/AdminDashboard.tsx` (scroll ref callback inside messages list: `ref={el => { if (el) el.scrollIntoView(...) }}`):** An inline arrow function is used as a React ref callback on the scroll sentinel div. React calls ref callbacks on every render, so scrollIntoView fires on every render of the message list — including unrelated parent re-renders — causing jarring scroll jumps when the admin interacts with other parts of the dashboard.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and set `ref={bottomRef}` on the div. This fires scroll only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — messages state):** Chat history is stored only in component state and is wiped every time the chat panel is closed (`open` toggle unmounts nothing but a future refactor easily could, and currently closing/reopening resets UX context). More critically, there is no maximum message history cap, so a long session accumulates unbounded message objects in memory and sends the full growing array to the API on every turn, increasing payload size and LLM token cost linearly.
  → *Cap history sent to the API: `const context = next.slice(-20);` and pass `context` instead of `next` to apiRequest. Optionally persist messages to sessionStorage so they survive panel toggles: `useEffect(() => sessionStorage.setItem('adminChat', JSON.stringify(messages)), [messages]);`*
- **`client/src/pages/AdminDashboard.tsx` (apiRequest calls throughout dashboard (mutations, queries)):** Multiple async operations in mutation handlers use sequential awaits where requests are independent, and none of the top-level async event handlers (outside of sendMessage) have try/catch blocks visible in the truncated code. For a financial literacy platform, a silent failure on a delete or update mutation leaves the UI in a stale inconsistent state with no user feedback.
  → *Wrap every mutationFn body in try/catch and call `toast({ title: 'Error', description: err.message, variant: 'destructive' })` in the catch. For any place where two independent fetches are awaited sequentially, replace with `const [a, b] = await Promise.all([fetchA(), fetchB()])` to cut latency.*

## 💡 Opportunities

### Batch category resolution for bill reminders: 🔥 High Impact
getBillReminders likely fetches categories in a loop similar to getBudgets. Collect all distinct categoryIds from the reminder rows first, run a single inArray(categories.id, ids) query, then build a Map for O(1) lookup when assembling the response. Eliminates N category queries per call.

### Introduce a query result cache for market stock data: ⚡ Medium Impact
getMarketStocks is called on every portfolio page render but simulated prices change on a fixed schedule. Caching the result in a module-level Map with a 30-second TTL (cleared by the price-update interval) would cut redundant DB reads significantly for concurrent student users in a classroom setting.

### Centralise userId ownership guard into a reusable helper: ⚡ Medium Impact
Every mutating method (deleteTransaction, deleteBudget, deleteSavingsGoal, etc.) repeats the same pattern of fetching a row then throwing if row.userId !== userId. Extract this into a typed assertOwnership<T>(row: T | undefined, userId: string, entity: string): asserts row is T helper to eliminate ~15 duplicated ownership-check blocks and ensure consistent 403 vs 404 error semantics.

### Idempotency token for question deletion before re-extraction: 🔥 High Impact
The current idempotency approach deletes all questions for a paperId on every retry before re-extracting. If two workers race (duplicate job delivery), the second delete can wipe questions inserted by the first. Add a DB-level unique constraint on (paperId, order) and use INSERT ... ON CONFLICT DO NOTHING for true idempotent upserts, eliminating the delete-and-reinsert race.

### Stream PDF text extraction instead of slicing to 30,000 chars: ⚡ Medium Impact
extractedText is hard-truncated at 30,000 characters with .slice(0, 30000). For long papers this silently drops questions at the end. Replace with page-by-page extraction and send multiple extraction calls, or use a token-aware chunking strategy, so no questions are silently lost.

### Centralise job payload type validation with Zod at handler entry: ⚡ Medium Impact
Each handler destructures job.payload fields (paperId, fileB64, olderThanDays, type) without runtime validation. A malformed enqueue call would produce a confusing runtime error deep inside the handler. Defining a Zod schema per job type and parsing at the top of each handler would give early, clear errors and double as documentation of the contract.

### Enforce varchar length on all user_id foreign key columns: ⚡ Medium Impact
All userId columns are bare varchar() with no length, inheriting unlimited storage. If users.id is a UUID (36 chars) or CUID, declare varchar('user_id', { length: 36 }) everywhere. This tightens the data contract, prevents oversized values from slipping in via raw SQL, and slightly improves index efficiency on btree scans.

### Add partial index for active/pending document uploads: ⚡ Medium Impact
documentUploads will accumulate rows over time; most queries only care about status = 'processing' rows (polling for completion). A partial index — CREATE INDEX idx_doc_uploads_pending ON document_uploads(user_id) WHERE status = 'processing' — would make these polling queries near-instant without indexing the full table.

### Centralise currency default into a shared Postgres enum or config constant: 📌 Low Impact
The string literal 'BSD' is hardcoded as .default('BSD') in at least six tables (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, linkedCards). If the app expands to TTD or JMD, each table must be altered separately and the risk of inconsistency is high. Extract a DEFAULT_CURRENCY constant in a shared config and reference it, or use a Postgres enum type for currency to get DB-level validation.

### Centralise hardcoded API path strings into environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat' and other endpoint paths appear inline throughout the file. Extracting them into a typed constants module (e.g., API_ROUTES.adminHelpChat) makes base-URL changes and environment-specific overrides (staging vs. prod) a single-line change and eliminates typo-driven 404s that are hard to trace.

### Extract AdminHelpChat into its own file with React.memo: ⚡ Medium Impact
AdminHelpChat is a self-contained feature component embedded in a 3000+ line dashboard file. Moving it to components/AdminHelpChat.tsx and wrapping with React.memo prevents it from re-rendering on any AdminDashboard state change (tab switches, search input, pagination), which will be measurable given the message list rendering cost.

### Add query deduplication and staleTime to repeated useQuery calls: 🔥 High Impact
Multiple useQuery hooks across dashboard tabs likely share overlapping data (users, schools, courses) and refetch independently on mount. Setting a shared staleTime (e.g., 30000ms) and using consistent queryKeys prevents redundant network calls when the admin switches tabs quickly, reducing server load on what is a Caribbean-hosted backend with constrained bandwidth.
