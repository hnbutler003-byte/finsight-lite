## 🔴 Critical (11)

### `server/storage.ts` — getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then loops to calculate 'spent' for each budget with a separate aggregation query per budget row. For a class of 30 students each with 10 budgets, this is 300+ sequential DB round-trips per dashboard load.

**Fix:** Replace the per-budget loop with a single JOIN or subquery using Drizzle's sql`` aggregation: SELECT budgets.*, COALESCE(SUM(transactions.amount),0) as spent FROM budgets LEFT JOIN transactions ON transactions.category_id = budgets.category_id AND transactions.user_id = budgets.user_id AND transactions.type = 'expense' GROUP BY budgets.id WHERE budgets.user_id = userId. Return the result set directly without a secondary loop.

### `server/storage.ts` — getLeaderboard
**Issue:** getLeaderboard performs an in-application JOIN by fetching game sessions and then calling getUser() individually for each distinct userId to resolve display names and avatars. This is a classic N+1 that scales linearly with active users and has no pagination guard — the filters.limit is applied after all users are fetched and resolved in memory.

**Fix:** Rewrite as a single aggregation query joining gameSessions to users: SELECT users.id, users.name, users.avatar, SUM(game_sessions.score) as totalScore, COUNT(*) as gamesPlayed FROM game_sessions JOIN users ON users.id = game_sessions.user_id WHERE ... GROUP BY users.id ORDER BY totalScore DESC LIMIT :limit. Apply the limit inside SQL, not post-fetch.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (buy/sell trade handler)
**Issue:** The trade execution path calls upsertPortfolioHolding and updateVirtualBalance as independent operations with no wrapping database transaction. A process crash or network error between the two writes leaves the portfolio holding updated but the virtual balance unchanged (or vice versa), corrupting the user's financial state permanently.

**Fix:** Wrap both writes in a single Drizzle transaction: await db.transaction(async (tx) => { await upsertPortfolioHolding(...) using tx; await updateVirtualBalance(...) using tx; }). Pass the transaction client (tx) down into each helper or inline the logic within the transaction block.

### `server/routes.ts` — /public-objects/:filePath(*)
**Issue:** Path traversal vulnerability: req.params.filePath is passed directly to objectStorage.searchPublicObject without sanitization. A request to /public-objects/../../private/secrets could escape the intended directory depending on how ObjectStorageService resolves paths.

**Fix:** Sanitize the path before use: const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, ''); or use Node's path.normalize and assert the result does not start with '..' before passing to the storage service.

### `server/routes.ts` — registerChatRoutes, registerImageRoutes, registerAudioRoutes
**Issue:** AI and image/audio routes are registered with no rate limiting middleware applied before them. On a school-age user app these are the most expensive endpoints and are wide open to abuse — a single user can exhaust OpenAI quota or cause runaway costs.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the register calls.

### `server/jobHandlers.ts` — runExtractPaper function signature / extract-paper handler
**Issue:** fileB64 (full base64-encoded file contents) is stored directly in the job payload, meaning every exam upload bloats the jobs table with potentially megabytes of binary data per row. This will degrade DB performance over time and makes job table queries slow.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of the handler. This keeps job rows small and separates concerns.

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique index, so duplicate ticker symbols (e.g. two rows with ticker='BISX') can be inserted silently. Any lookup by ticker will return non-deterministic results and portfolio calculations will be wrong.

**Fix:** Add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to the simulatedStocks pgTable call.

### `server/index.ts` — res.json monkey-patch in request logger middleware
**Issue:** Every API response body is captured into capturedJsonResponse and then serialized in full via JSON.stringify into the log line. For endpoints returning student PII, financial records, or AI responses this leaks sensitive data to stdout/log aggregators. There is no size guard or field allowlist.

**Fix:** Replace the full-body log with a fixed-length truncated excerpt and exclude any route whose path matches /api/user, /api/auth, or similar sensitive prefixes. E.g.: `const preview = JSON.stringify(capturedJsonResponse).slice(0, 120);` and skip logging entirely for auth routes.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are assigned but the references are never stored, so they cannot be cleared in a shutdown handler. When the process receives SIGTERM (e.g. Replit cycling the dyno), the intervals keep the event loop alive and any in-flight DB writes inside the callbacks can be torn mid-transaction, risking duplicate or partial digest/purge records.

**Fix:** Store both interval IDs: `const digestInterval = setInterval(...); const purgeInterval = setInterval(...);` then in a SIGTERM/SIGINT handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before draining the job worker and closing the HTTP server.

---

### `client/src/pages/AdminDashboard.tsx` — sendMessage
**Issue:** The full `messages` array is sent to `/api/admin/help-chat` on every request with no length cap. A user can accumulate an unbounded conversation that inflates request payload size and LLM token costs, and there is no server-side rate limit referenced client-side.

**Fix:** Slice the history before sending: `const history = next.slice(-20);` and send `{ messages: history }`. Add a visible message count warning (e.g., after 15 messages) and disable input when limit is reached. Pair with server-side rate limiting per admin session.

---

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth / top-level dashboard render
**Issue:** If `useAdminAuth` returns a loading state before the auth status is confirmed, the dashboard content (including admin data queries) begins executing and components render with potentially undefined auth context. There is no early return for the auth-loading state, risking brief exposure of admin UI and firing unauthorized API calls.

**Fix:** Add an explicit loading guard at the top of the component: `const { admin, isLoading } = useAdminAuth(); if (isLoading) return <FullPageSpinner />; if (!admin) return <Redirect to='/admin/login' />;` before any query hooks or JSX that depends on admin identity.

## 🟡 Warnings (19)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues at least 4 independent async queries (total income, total expenses, savings goals, bill reminders) with sequential awaits. Each adds a full DB round-trip latency to the critical path that renders the student dashboard.
  → *Replace sequential awaits with Promise.all: const [incomeResult, expenseResult, goalsResult, remindersResult] = await Promise.all([db.select(...income...), db.select(...expenses...), db.select(...goals...), db.select(...reminders...)]). All four queries are read-only with no data dependency on each other, making this a safe and direct parallelisation.*
- **`server/storage.ts` (getBillReminders):** getBillReminders fetches all reminders for a user then resolves each reminder's category with a separate query inside a .map() loop, creating an N+1 pattern. Additionally there is no pagination guard — a user with hundreds of past reminders loads all rows into memory.
  → *Fetch reminders with a LEFT JOIN to categories in one query: db.select().from(billReminders).leftJoin(categories, eq(billReminders.categoryId, categories.id)).where(eq(billReminders.userId, userId)).limit(100). Add a default limit (e.g. 100) and expose an offset parameter for pagination. Remove the post-fetch category resolution loop.*
- **`server/routes.ts` (initSupabaseTables().then(...).catch(...)):** The Supabase init chain runs sequentially via .then() chaining — each seed step waits for the previous, which is fine for ordering, but if seedFinancialAcademyLesson() rejects, initStaticContentTables/seedStaticModules/seedRegionalContent/seedGameContent never run and the single .catch() swallows the error silently with only a console.error. Downstream features that depend on seeded data will silently fail.
  → *Run independent seed steps with individual error handling, or at minimum log which step failed: wrap each step in its own .catch(e => console.error('[Supabase] Step X failed:', e)) so a single failure does not abort the entire chain.*
- **`server/routes.ts` (/healthz):** The /healthz endpoint performs a dynamic import of db and drizzle-orm on every call. Import resolution and module caching aside, this is unnecessary overhead on an endpoint that uptime monitors may hit every 30 seconds.
  → *Move the db and sql imports to the top of the file as static imports alongside the other imports. The modules are already used transitively; eagerly importing them also surfaces startup failures earlier.*
- **`server/routes.ts` (app.get('/sitemap.xml')):** The x-forwarded-proto and x-forwarded-host headers are used without validation to construct URLs embedded in the sitemap XML response. A spoofed Host header can inject arbitrary content into the XML (host header injection), potentially poisoning cached sitemaps.
  → *Resolve the base URL from a trusted environment variable: const base = process.env.APP_BASE_URL || 'https://finsightlite.com'; and remove reliance on request headers entirely for this static sitemap.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with await on each iteration. For a paper with 50 questions this means 50 serial round-trips to the DB, multiplying latency and holding the job worker busy unnecessarily.
  → *Collect all valid question objects into an array first, then execute a single batch insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used for business logic reasons, wrap calls with Promise.all on a chunked array (e.g., chunks of 10) to parallelise without overwhelming the connection pool.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** openai client is imported from a route file rather than a dedicated singleton module. Route files typically register Express handlers and may have side effects on import. This creates hidden coupling: any refactor of the chat routes risks breaking job handlers, and the import may trigger route registration in worker contexts.
  → *Extract the OpenAI client instantiation into a dedicated shared module (e.g., server/lib/openaiClient.ts) that exports the configured client. Both the chat route and jobHandlers.ts should import from that module.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload without validation. A misconfigured or maliciously crafted payload with olderThanDays: 0 would delete the entire aiUsageEvents table in production.
  → *Add an explicit guard before the delete: if (typeof olderThanDays !== 'number' || olderThanDays < 7) throw new Error('olderThanDays must be a number >= 7'). Also enforce a minimum via a zod/schema parse on the payload at job enqueue time so bad values are rejected before they reach the handler.*
- **`server/jobHandlers.ts` (runExtractPaper — vision branch image_url content):** The raw base64 image string (potentially several MB) is re-encoded inline into the OpenAI API request body as a data URI. Combined with the payload already holding fileB64, this data is duplicated in memory within the same execution, doubling peak heap usage per job and risking OOM in constrained environments.
  → *After fixing finding #1 (store only objectPath in payload), download the buffer once, pass it to the vision call directly, and discard the buffer reference after the API call returns. Never re-derive base64 from a Buffer that was itself decoded from base64 — use the original download stream or a single Buffer reference throughout.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on (userId, categoryId) despite being the most likely query pattern — fetching a user's budget for a specific category will do a full table scan as the dataset grows.
  → *Add a composite index in the table's second argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget rows per period.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders is queried for upcoming due dates but has no index on (userId, nextDueDate). Any 'show upcoming bills' query will scan the entire table.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders pgTable call.*
- **`shared/schema.ts` (All varchar('user_id') column definitions):** All userId foreign key columns use bare varchar() with no length argument, which defaults to unlimited in Postgres. This is inconsistent with the referenced users.id column and prevents the DB from enforcing a known maximum, risking silent acceptance of malformed IDs (e.g. UUIDs vs short IDs).
  → *Determine the exact format of users.id (e.g. UUID = 36 chars, CUID = 25 chars) and change every instance to varchar('user_id', { length: 36 }) to match. This enforces a contract at the DB layer and improves index efficiency.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt (and other createdAt columns)):** createdAt columns use .defaultNow() but are not marked .notNull(). A row inserted with an explicit NULL override would store NULL, breaking any ordering or auditing query that assumes createdAt is always populated.
  → *Chain .notNull() after .defaultNow() on every createdAt (and executedAt, purchasedAt) column: timestamp('created_at').defaultNow().notNull(). This is a one-line change per column.*
- **`server/index.ts` (startup sequence — probeDatabase and seedDatabase awaits):** probeDatabase() and seedDatabase() are awaited sequentially before the HTTP server begins listening. If the DB probe is slow (cold Replit container, network hiccup) the process sits idle and any health-check that fires during that window will time out. Additionally, if seedDatabase throws, the process exits without ever attempting graceful shutdown of resources already initialised above it.
  → *Run independent startup tasks in parallel where safe: `await Promise.all([probeDatabase(), registerRoutes(app, httpServer)]);` then seed only if probe succeeds. Wrap the entire startup block in try/catch and call a shared `shutdown()` helper on failure so Sentry can flush before exit.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes the target time using `new Date()` local system time with no timezone qualification. On a Replit server the system timezone is UTC, so 7 pm UTC may not match the intended Caribbean local time (UTC-4 or UTC-5). This causes the weekly digest to fire 4-5 hours off schedule for all Caribbean users.
  → *Accept an explicit IANA timezone via env var (e.g. `DIGEST_TIMEZONE=America/Port_of_Spain`) and use the Intl API or the `date-fns-tz` package to resolve Sunday 19:00 in that zone before converting to UTC for comparison. Document the env var in your README and Replit Secrets.*
- **`server/index.ts` (maybeRunAiUsagePurge — inner enqueueJob catch block):** The rollback delete inside the catch uses a bare `await db.delete(...)` with no surrounding try/catch. If the rollback itself fails (transient DB error), the exception propagates out of the inner catch, bypasses the outer try/catch log, and becomes an unhandled promise rejection that can crash the process on Node 18+ where unhandled rejections are fatal by default.
  → *Wrap the rollback in its own try/catch: `try { await db.delete(...) } catch (rollbackErr) { captureError(rollbackErr); log('purge rollback also failed: ...'); }` — matching the pattern already used in maybeRunWeeklyDigest but actually applied to the purge scheduler too.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — scroll ref inline callback):** `<div ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }} />` creates a new function object on every render. React calls the old cleanup ref with null and the new one with the element on every render cycle, causing repeated scrollIntoView calls even when no new message was added.
  → *Replace with a `useRef` + `useEffect`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — handleKey):** `handleKey` and `sendMessage` are redefined on every render of `AdminHelpChat` and passed as props to the input element, causing the input to receive new function references every render even when `messages`, `input`, and `loading` have not changed.
  → *Wrap both in `useCallback`: `const sendMessage = useCallback(async () => { ... }, [input, loading, messages]);` and `const handleKey = useCallback((e) => { ... }, [sendMessage]);`. This prevents unnecessary re-renders of the input element.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard data fetching — multiple independent useQuery calls):** Multiple `useQuery` calls for stats, users, schools, and audit logs fire sequentially in React's render cycle but each waits for its own network round-trip independently. While React Query does parallelize these, any dependent sequential `await` chains inside `queryFn` callbacks (e.g., fetch user count then fetch school count) double the latency unnecessarily.
  → *Audit each `queryFn` for sequential awaits and replace with `Promise.all`: `const [users, schools] = await Promise.all([fetchUsers(), fetchSchools()]);`. For independent queries that must stay separate, confirm they all have distinct `queryKey` arrays so React Query caches them individually and does not re-fetch on unrelated state changes.*

## 💡 Opportunities

### Centralised pagination guard utility — 🔥 High Impact
At least 6 list-returning methods (getTransactions, getGameSessions, getExamPapers, getLeaderboard, getAllQuestions, getBillReminders) accept an optional limit but apply it inconsistently or not at all. A shared helper enforceLimit(requested?: number, max = 200): number that clamps input and defaults to a safe value would prevent unbounded memory growth across all list endpoints without duplicating logic.

### Replace manual category resolution with Drizzle relational queries — 🔥 High Impact
Multiple methods (getBudgets, getBillReminders, getTransactions) manually stitch related rows in JavaScript after fetching. Migrating these to Drizzle's with: { category: true } relational API or explicit LEFT JOINs eliminates the N+1 patterns structurally and makes future query additions safe by default rather than requiring per-developer discipline.

### Extract hardcoded virtual balance seed value to environment config — ⚡ Medium Impact
The initial virtual balance granted to new users (likely 10000 or similar) appears hardcoded inside getVirtualBalance or seedMarketData. For a sponsored school deployment this figure needs to differ per sponsor tier. Moving it to VIRTUAL_BALANCE_SEED_AMOUNT in environment config allows ops to tune it per deployment without a code change or redeployment.

### Parallel domain route registration — ⚡ Medium Impact
The six domain route register calls (registerAuthDomainRoutes through registerEmailRoutes) are awaited sequentially after being imported in parallel. If any register function does async work (DB checks, schema validation), running them sequentially adds unnecessary startup latency. Evaluate whether they can be wrapped in a single Promise.all([registerAuthDomainRoutes(app), ...]) — this could cut cold-start time by the sum of their individual async costs.

### Structured logging for Supabase init errors — ⚡ Medium Impact
The catch handler uses console.error with a plain string prefix. Replacing it with a structured logger (e.g. pino or winston) that emits JSON including step name, error code, and stack trace would make Railway log aggregation and alerting on seed failures actionable rather than buried in plain text output.

### Sitemap caching with Cache-Control header — 📌 Low Impact
The sitemap is regenerated on every request despite being entirely static. Adding res.setHeader('Cache-Control', 'public, max-age=86400') and optionally caching the generated XML string in memory after first render would eliminate repeated string concatenation work on a Googlebot-crawled endpoint.

### Parallel DB + storage operations in admin-csv-export — ⚡ Medium Impact
Based on the visible pattern, the export handler likely fetches multiple tables sequentially. Wrapping independent table fetches in Promise.all would reduce export job wall-clock time proportionally to the number of parallel queries, with no added complexity.

### Centralised payload schema validation at job registration — 🔥 High Impact
Adding zod schemas for each job payload type, parsed at the top of each registerJobHandler callback, would catch malformed payloads before any DB or API side-effects occur and produce structured error messages instead of runtime TypeErrors mid-handler. This would harden all handlers uniformly with ~20 lines of shared infrastructure.

### Job-level idempotency key for extract-paper retries — ⚡ Medium Impact
The current idempotency approach (delete then re-insert on every attempt) risks a race if two workers pick up the same job concurrently. Storing a processingToken on the exam_papers row and using an atomic compare-and-set before starting work would make retries safe under concurrent worker scenarios.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a .default() in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, linkedCards). Extract const DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it everywhere. When the app expands to TTD or JMD support, this is a single-line change instead of a grep-and-replace across the schema.

### Add a partial index for active bill reminders — ⚡ Medium Impact
Queries for upcoming bills will almost always filter on active/non-deleted rows and future dates. A partial index such as CREATE INDEX … WHERE next_due_date >= NOW() would be dramatically smaller than a full index and speed up the most common dashboard query. Express this in Drizzle using .where(sql`next_due_date >= now()`) inside the index definition.

### Replace text() with pgEnum for repeated enum columns — 📌 Low Impact
Enum values like type ('income'|'expense'), status ('processing'|'completed'|'failed'), and riskLevel ('low'|'medium'|'high') are defined inline on multiple tables using text() with an enum option. Drizzle's pgEnum creates a real Postgres TYPE, giving DB-level enforcement, smaller storage (int OID vs variable-length string), and a single source of truth. Declare e.g. const transactionTypeEnum = pgEnum('transaction_type', ['income','expense']) and reuse it across categories and transactions.

### Unified graceful shutdown handler — 🔥 High Impact
Add a single `shutdown(signal)` async function that clears intervals, calls `httpServer.close()`, drains the job worker queue, flushes Sentry, then exits. Register it for SIGTERM, SIGINT, and `uncaughtException`. This prevents data corruption on Replit dyno restarts and makes the shutdown path testable. Expected impact: eliminates race conditions on deploy/restart cycles.

### Structured logging with log-level filtering — ⚡ Medium Impact
Replace ad-hoc console.log/log() calls with a structured logger (pino is zero-dep and fast). Set LOG_LEVEL=info in production to suppress verbose scheduler noise, and log objects instead of interpolated strings so fields like userId, audience, and weekStart are queryable in your log aggregator without regex parsing.

### Extract scheduler logic into a dedicated module — ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, and their interval setup into a schedulers.ts module that exports start() and stop() functions would make server/index.ts a thin composition root, allow the schedulers to be unit-tested in isolation with fake timers, and make the interval references naturally scoped and clearable.

### Memoize filtered and sorted table data — 🔥 High Impact
User/school list filtering and sorting applied inline on every render should be wrapped in `useMemo` keyed on the raw data and filter/sort state. For lists of 500+ students this avoids O(n log n) work on every keystroke in unrelated fields, measurably reducing input lag.

### Externalize hardcoded API base paths and magic numbers to environment config — ⚡ Medium Impact
Strings like `/api/admin/help-chat` and numeric limits (page sizes, max retries) found inline throughout the file should be moved to `import.meta.env.VITE_API_BASE` and a shared `constants.ts`. This prevents misconfiguration across staging/production deployments and makes limits auditable in one place.

### Add error boundaries around chart and table sections — ⚡ Medium Impact
Recharts components throw on malformed data (e.g., non-numeric YAxis values from bad API responses). Wrapping `<BarChart>` and `<LineChart>` sections in a React `ErrorBoundary` with a graceful fallback prevents a single bad data point from crashing the entire admin dashboard for all users.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts contains several critical reliability and security issues including N+1 query patterns in budget and bill-reminder fetches, missing database transactions on coupled writes, and sequential awaits that should run in parallel. The codebase is functional but has patterns that will degrade under load and expose data integrity risks in a financial context.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel imports and rate limiting. Key concerns are unprotected AI/heavy endpoints, an unhandled promise rejection risk on Supabase init chain, and a path traversal vulnerability on the public object storage endpoint.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads stored in the job table risk bloating the DB, sequential DB inserts in a loop should be batched, and the openai import from a route file is an architectural smell that exposes uncontrolled coupling. Error handling is partially present but inconsistent across handlers.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with appropriate foreign keys and some indexes already defined, but has several reliability and performance gaps. Key concerns are missing indexes on high-cardinality filter columns, nullable timestamps that should be NOT NULL, and untyped varchar columns that risk silent data truncation.

### Server Startup (`server/index.ts`)
The server startup sequence contains several reliability and security concerns: async scheduler intervals are never cleared on shutdown, the full JSON response body is logged on every API call creating both a memory and data-exposure risk, and sequential DB probes and seed operations block startup when they could run in parallel. Graceful shutdown handling is absent, meaning in-flight jobs and open DB connections are not drained before process exit.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx exhibits several reliability and security concerns including missing auth guards on sensitive operations, unparallelized async data fetches that increase load time, and a scroll ref callback that creates a new function on every render causing unnecessary DOM thrashing. The chat panel also sends full conversation history to the server without any length cap, creating a potential abuse vector. Overall structure is reasonable but the issues below need addressing before production use.
