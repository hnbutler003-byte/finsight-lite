## 🔴 Critical (7)

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 42, ~line 59)
**Issue:** Host header injection: `req.headers['x-forwarded-host']` and `req.headers.host` are used unsanitised to construct absolute URLs written into robots.txt and sitemap.xml. An attacker can send a crafted Host header to poison cached responses or redirect crawlers to a malicious domain.

**Fix:** Resolve the base URL once at startup from a trusted environment variable (e.g., `SITE_URL=https://finsightlite.com`) and use that constant instead of deriving it from request headers. Never trust `x-forwarded-host` or `host` for security-sensitive URL construction.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** fileB64 (full base64-encoded PDF or image) is stored directly in the job payload in the DB table. For a multi-page PDF this can easily be 5-20 MB per row, bloating the jobs table, slowing queue polling queries, and risking OOM when multiple jobs are loaded into the worker process simultaneously.

**Fix:** Before enqueuing, upload the file to object storage and store only the resulting storage path in the job payload (e.g., { paperId, storagePath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of the handler, then delete the temp object on success or terminal failure. This keeps job rows small (<1 KB) and leverages already-present objectStorageClient infrastructure.

### `server/jobHandlers.ts`: runExtractPaper – for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a 50-question paper this is 50 serial round-trips to the database, adding latency proportional to network RTT times question count and increasing the window for a partial-insert failure mid-loop.

**Fix:** Replace the for loop with a single bulk insert using Drizzle's db.insert(extractedQuestions).values([...]) API. Build the values array in a synchronous map/filter first, then issue one await. Wrap the bulk insert and the subsequent updateExamPaper call in a db.transaction() so both succeed or both roll back atomically, eliminating the partial-insert idempotency problem the delete-on-retry is trying to paper over.

### `shared/schema.ts`: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — all tables
**Issue:** userId is declared as varchar with no length argument, which defaults to unlimited length in Postgres. All FK references to users.id should match the exact type (length-bounded varchar or uuid). Mismatched types silently prevent index usage on joins and increase storage overhead.

**Fix:** If users.id is a UUID, change every userId column to use uuid('user_id') type from drizzle-orm/pg-core instead of varchar. If it is a fixed-length string, specify the length explicitly, e.g. varchar('user_id', { length: 36 }). Ensure users.id in ./models/auth uses the same type.

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** simulatedStocks.ticker has no unique index. Duplicate tickers can be inserted, which would corrupt portfolio calculations (wrong stock matched on joins) and break any UI lookup by ticker symbol.

**Fix:** Add a uniqueIndex in the table's third argument: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). uniqueIndex is already imported in the file.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage, apiRequest call
**Issue:** The full `messages` array (entire conversation history) is sent to `/api/admin/help-chat` on every turn with no length cap. For a school platform this means the payload grows unbounded, increasing API cost, latency, and risk of leaking earlier sensitive admin conversation content if the response is logged server-side.

**Fix:** Slice the history before sending: `const history = next.slice(-10);` (keep last 10 messages) and send `{ messages: history }`. Add a corresponding server-side limit as well. Document the truncation policy so audit logs are consistent.

### `client/src/pages/AdminDashboard.tsx`: Top of component – useQuery calls before auth check
**Issue:** Multiple useQuery hooks appear to fire unconditionally before confirming the admin session via useAdminAuth. If the auth hook has any async resolution delay, queries will execute (and potentially return data to the component) before the user is confirmed as an admin, creating a window where unauthorized users could receive admin API responses.

**Fix:** Add `enabled: !!adminUser` to every useQuery options object, where `adminUser` comes from `useAdminAuth()`. Render a loading spinner or redirect immediately when `adminUser` is null/undefined rather than allowing queries to proceed: `if (!adminUser) return <Navigate to='/admin/login' />;` placed before any query-dependent JSX.

## 🟡 Warnings (11)

- **`server/routes.ts` (initSupabaseTables chain (~line 118)):** The sequential `.then()` chain means a failure in any intermediate step (e.g., `trimOrgNamesInSupabase`) skips all subsequent seeds silently — the single `.catch` only logs the first rejection. Seed state becomes unpredictably partial on every failed deploy.
  → *Wrap each call individually so failures are isolated: run them with `Promise.allSettled([initSupabaseTables(), ...])` or use a sequential runner that logs and continues on error rather than short-circuiting the whole chain.*
- **`server/routes.ts` (registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 115)):** OpenAI-backed chat, image, and audio routes have no rate limiting applied in this file. For a school-aged Caribbean audience on a budget API plan, a single misbehaving client or a simple loop can exhaust quotas and generate runaway costs with no circuit breaker.
  → *Apply a dedicated AI rate limiter before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where `aiLimiter` uses a tight window (e.g., 10 req/min per IP) defined in your existing `rateLimiter` module.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 103)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject` without any path sanitisation. A request to `/public-objects/../../private/secret` could traverse outside the intended public directory depending on the storage implementation.
  → *Sanitise the path before use: `const safe = path.normalize(filePath).replace(/^\.\.\//, ''); if (safe !== filePath || filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Add this check at the top of the handler.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The OpenAI client is imported from a route module rather than a dedicated singleton. Route modules typically register Express routes as a side effect on import, meaning importing this file inside a job worker process will silently register HTTP route handlers (or crash if the Express app isn't initialised), and couples the job handler to the full HTTP layer unnecessarily.
  → *Extract the openai client instantiation into its own module (e.g., server/lib/openaiClient.ts) that exports only the configured OpenAI instance. Both the route file and jobHandlers.ts import from there. This is a one-file refactor with no behaviour change but eliminates the side-effect import risk.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation before being used in arithmetic to compute a cutoff date. A payload with olderThanDays: 0, negative, or NaN would either delete all records or produce an invalid Date silently passed to the DB query.
  → *Add an explicit guard at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('Invalid olderThanDays: ' + olderThanDays); This causes the job to fail fast and retry rather than silently corrupting data. Also define a minimum floor (e.g., 7 days) as a named constant sourced from environment config.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index despite being the primary filter columns for budget lookups. Every budget query for a user will do a full table scan as the table grows.
  → *Add an index in the table's third argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a unique index on (userId, categoryId, period) to enforce one budget per category per period at the DB level.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** All audit/creation timestamps use .defaultNow() but are nullable (no .notNull()). A bug or direct DB insert can leave these NULL, breaking any ORDER BY or audit query that assumes they are always set.
  → *Append .notNull() after .defaultNow() on every audit timestamp column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). This is a one-line change per column and requires a migration.*
- **`shared/schema.ts` (billReminders table definition):** billReminders.nextDueDate is the primary column used to query upcoming bills (WHERE next_due_date <= now()) but has no index. For school-aged users checking reminders daily, this will scan the full table on every dashboard load.
  → *Add an index in the table's third argument: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). A partial index filtering on active reminders would be even better if a soft-delete/active flag is added.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and fires scrollIntoView every time any state changes (including the loading spinner toggling), not just when a new message arrives. This causes janky scroll behavior and wastes layout cycles.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and render `<div ref={bottomRef} />`. This fires scroll only when messages or loading actually change.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage / handleKey):** sendMessage and handleKey are plain functions recreated on every render. Because AdminHelpChat likely renders inside a parent that re-renders on tab switches or query invalidation, every re-render allocates new function references. handleKey is also passed to the input's onKeyDown, breaking referential equality for any memoized child.
  → *Wrap both in useCallback: `const sendMessage = useCallback(async () => { ... }, [input, loading, messages]);` and `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`*
- **`client/src/pages/AdminDashboard.tsx` (Multiple sequential useQuery / fetch calls in dashboard sections):** Stats, users, schools, and audit data appear to be fetched in sequential waterfall queries (each dependent on the previous resolving) rather than in parallel. On a Caribbean mobile network this adds hundreds of ms of avoidable latency per page load for school-aged users on slower connections.
  → *Ensure all independent queries have no cross-dependencies and are declared at the same component level so React Query fires them in parallel automatically. For any imperative fetches inside event handlers or useEffects, replace sequential `await fetch1(); await fetch2();` with `const [r1, r2] = await Promise.all([fetch1(), fetch2()]);`.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel imports are still sequential. If any route registrar does async work (DB schema checks, cache warming), they block each other unnecessarily. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` to cut cold-start time proportionally to the slowest registrar rather than their sum.

### Centralised base-URL config module: ⚡ Medium Impact
proto/host derivation logic is duplicated verbatim in both robots.txt and sitemap handlers. Extract a single `getBaseUrl()` utility that reads from `process.env.SITE_URL` with a fallback, eliminating duplication and the host-injection surface in one change.

### Structured startup logging for Supabase seed results: 📌 Low Impact
The current `.catch(e => console.error(...))` gives no visibility into which seed step succeeded or failed. Replacing it with per-step structured logs (step name, duration, success/failure) would make Railway deploy logs actionable and reduce debugging time when seeds fail in production.

### Parallel admin CSV export queries: ⚡ Medium Impact
The admin-csv-export handler (visible in the truncated section) likely fetches multiple DB tables sequentially. Wrapping independent queries in Promise.all would cut export job duration proportionally to the number of parallel queries, improving perceived responsiveness for admins and reducing job slot occupancy.

### Centralised job payload Zod schemas: 🔥 High Impact
Job payloads are currently consumed with direct destructuring and no runtime validation. Adding a Zod schema per job type (co-located with the JobPayloads type) and parsing payload at handler entry would catch malformed enqueue calls at job start rather than mid-execution, dramatically reducing hard-to-diagnose partial-execution failures in production.

### Object storage streaming instead of buffer loading for PDF handling: ⚡ Medium Impact
Once fileB64 is removed from payloads and replaced with storage paths, pdf-parse and vision calls should stream or chunk large files rather than loading the full Buffer into worker memory. For school exam papers up to tens of MB this prevents heap spikes when several extract-paper jobs run concurrently in the same worker process.

### Enforce one active budget per user-category-period at DB level: 🔥 High Impact
Add a unique constraint on (userId, categoryId, period) in the budgets table. Without it, application bugs can insert duplicate budgets for the same period, silently doubling budget totals in the UI. The DB constraint is the only reliable guard.

### Replace repeated currency text columns with a shared enum or domain: ⚡ Medium Impact
The default currency 'BSD' is hardcoded as a string default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). A Postgres enum type or a single app-level constant (CURRENCY_DEFAULT = process.env.DEFAULT_CURRENCY ?? 'BSD') centralises the value so a future multi-currency rollout requires one change, not six migrations.

### Add portfolioHoldings unique constraint on (userId, stockId): ⚡ Medium Impact
Currently multiple rows can exist for the same user and stock, forcing the application to aggregate quantity and avgPurchasePrice in code. A unique constraint plus an ON CONFLICT DO UPDATE (upsert) strategy in the insert logic would keep one canonical row per holding, simplify queries, and eliminate a class of double-buy bugs.

### Memoize filtered and sorted table data: 🔥 High Impact
User/school list filtering and sorting applied on every render should be wrapped in useMemo keyed on the raw data and filter state. On a dashboard with hundreds of student records this prevents O(n) recomputation on every keystroke in unrelated inputs, directly reducing jank on low-end devices common in the Caribbean school context.

### Extract chart sections into memoized child components: ⚡ Medium Impact
Recharts BarChart and LineChart trees are large and expensive to reconcile. Moving each chart into its own React.memo component with explicit props prevents full chart re-renders when unrelated state (e.g., the help chat open state) changes in the parent. Expected impact: eliminates the most expensive reconciliation work on every chat open/close toggle.

### Move hardcoded API base paths to environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat' and any hardcoded pagination limits or role constants scattered through the file should be centralised in a typed config module (e.g., src/config/api.ts) referencing import.meta.env variables. This enables staging/production environment switching without code changes and makes security audits easier for a regulated financial-literacy product.
