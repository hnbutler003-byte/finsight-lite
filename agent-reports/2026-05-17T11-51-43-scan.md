## 🔴 Critical (7)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: the function fetches all holdings then issues a separate getStockById query per holding inside a loop or map, producing O(n) database round-trips per user.

**Fix:** Use a single JOIN query: db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId)). This collapses N+1 into 1 query.

---

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** A stock trade involves three separate writes (upsert holding, insert portfolio transaction, update virtual balance) with no wrapping database transaction. A crash or timeout between any two steps leaves the portfolio in an inconsistent state — e.g. balance debited but holding not updated.

**Fix:** Wrap all three operations in db.transaction(async (trx) => { ... }) using Drizzle's transaction API. Pass the trx client to each sub-operation so they share the same atomic unit and auto-rollback on failure.

---

### `server/storage.ts` — getLeaderboard
**Issue:** Leaderboard query joins or aggregates across gameSessions and users with no enforced limit default, and the result includes raw userId values logged or returned without scrubbing. For a school-aged user base this exposes PII (student identifiers) in API responses and potentially in server logs.

**Fix:** Enforce a hard default limit of 50 (const safeLimit = Math.min(filters?.limit ?? 50, 100)). Remove userId from the returned object or replace it with an opaque rank index. Never log the raw leaderboard payload at info/debug level.

---

### `server/routes.ts` — AI/tutor endpoints (any route calling openai or enqueueJob for AI tasks)
**Issue:** No rate limiting middleware is applied to AI-backed endpoints. A single student account or a scripted attack can exhaust OpenAI quota and inflate costs without bound. quotaErrorMessage and reserveQuota exist but are applied inside handler logic, not at the middleware layer, so malformed requests that fail schema validation still hit the quota check path.

**Fix:** Add express-rate-limit (or a Redis-backed equivalent) as middleware specifically on AI route groups before handler logic: `const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.use('/api/tutor', aiLimiter);`. Set max to a value derived from an environment variable (AI_RPM_LIMIT) so it can be tuned per deployment.

---

### `server/routes.ts` — multer fileFilter callbacks (logoUpload, videoUpload, upload, examUpload)
**Issue:** All four multer fileFilter callbacks validate only file extension or MIME type as reported by the client, which is trivially spoofed. A student could upload a file with mimetype 'image/png' that is actually executable content, bypassing the filter entirely.

**Fix:** After multer accepts the file into memoryStorage, add a secondary validation step that inspects the actual magic bytes of req.file.buffer using the 'file-type' package (e.g. `const type = await fileTypeFromBuffer(req.file.buffer); if (!allowedMimeTypes.includes(type?.mime)) return res.status(400).json({message: 'Invalid file content'});`). This must happen in the route handler before the buffer is passed to storage or processing logic.

---

### `server/jobHandlers.ts` — runExtractPaper / job.payload.fileB64
**Issue:** Full file contents (base64-encoded PDF or image) are stored directly in the job payload, meaning every row in the jobs table carries potentially megabytes of binary data. This bloats the jobs table, slows queue polling queries, and risks hitting row-size limits in Postgres.

**Fix:** Upload the file to private object storage before enqueuing and store only the resulting storage path in the payload (e.g. { paperId, storagePath, ext, subject }). Inside runExtractPaper, download the file from object storage at the start of processing. This keeps job rows small and makes retries cheaper.

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** The ticker column has no unique index. Duplicate tickers (e.g., two rows with ticker 'BTC') would silently coexist, causing non-deterministic results in portfolio joins and lookup queries that assume tickers are unique identifiers.

**Fix:** Replace the plain column declaration with `ticker: text('ticker').notNull().unique()` or add `tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker)` in the index callback.

## 🟡 Warnings (13)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for income, expenses, savings, and budget queries run one after another instead of in parallel. With 4+ independent DB round-trips, this adds ~40-120ms of unnecessary latency per dashboard load.
  → *Replace sequential awaits with Promise.all: const [incomeResult, expenseResult, savingsResult, budgetResult] = await Promise.all([db.select(...), db.select(...), db.select(...), db.select(...)]). Each query is independent and safe to parallelize.*
- **`server/storage.ts` (getTransactions):** The limit filter has no enforced maximum cap. A caller passing limit=999999 or omitting it entirely will load every transaction row for a user into memory, causing unbounded memory growth in classroom scenarios with many students sharing a session.
  → *Add a server-side cap: const safeLimit = Math.min(filters?.limit ?? 50, 200). Never allow the ORM query to omit a limit clause entirely — default to 50 and document the max.*
- **`server/routes.ts` (/api/auth/google handler):** Sequential awaits on independent lookups inside Google auth inflate latency and block the event loop unnecessarily. getOrgEnvironmentById and getOrganization are called sequentially even though the class lookup is already complete.
  → *Where both results are needed but neither depends on the other, replace sequential awaits with Promise.all: `const [classEnv, otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), someOtherCall()])`. Apply the same pattern to any other independent lookups in this handler.*
- **`server/routes.ts` (/healthz handler):** The health check dynamically imports './db' and drizzle-orm on every request using await import(). Node caches dynamic imports after the first resolution, but the async import machinery still runs on every call, adding unnecessary overhead and obscuring startup errors that should surface at boot time.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The health check handler then becomes a simple `await db.execute(sql\`select 1\`)` with no dynamic import overhead.*
- **`server/routes.ts` (e?.message exposure in catch blocks (e.g. /healthz and inferred AI handlers)):** Raw error messages from database and OpenAI clients are forwarded directly to API responses (e.g. `error: e?.message`). These messages can contain connection strings, table names, or upstream API error details that reveal internal infrastructure to clients.
  → *Create a small helper: `function safeError(e: unknown, fallback: string) { console.error(e); return fallback; }`. Return only the fallback string in responses; log the full error server-side. For the healthz route specifically, return `{ ok: false, error: 'db unavailable' }` without the raw message.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one inside a sequential for loop. For a 40-question paper this fires 40 serial round-trips to the database. If the process crashes mid-loop the idempotency delete at the top of a retry wipes partial progress correctly, but each retry still pays the full sequential cost and holds an implicit partial state window.
  → *Collect all valid question objects into an array first, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the bulk insert and the subsequent updateExamPaper call in a single transaction so both succeed or both roll back atomically, eliminating the partial-completion window entirely.*
- **`server/jobHandlers.ts` (openai.chat.completions.create calls — model: "gpt-4o"):** The OpenAI model name is hardcoded as the string literal "gpt-4o" in at least two places (vision extraction and question parsing). Changing models for cost or capability reasons requires a code deploy rather than a config change, and different environments cannot use cheaper models for testing.
  → *Add an env var such as OPENAI_EXTRACTION_MODEL with a fallback: const extractionModel = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'. Reference this constant in both completions.create calls. Document the var in .env.example.*
- **`server/jobHandlers.ts` (registerJobHandler "admin-csv-export" — rows fetched into memory):** Based on the visible pattern (type AdminRow = Record<string, unknown>; let dat...) the export handler appears to fetch an unbounded result set into a single in-memory array before serialising to CSV. For a growing Caribbean school deployment with many transactions or users this will exhaust Node's heap and crash the worker process.
  → *Use a database cursor or paginated query (e.g. fetch 500 rows at a time with .limit(500).offset(page * 500) in a loop, or use Drizzle's streaming API if available) and pipe chunks directly to the object storage upload stream. Never hold the full result set in memory.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — fileName header):** The Content-Disposition header interpolates the fileName parameter directly: filename="${fileName}". If fileName contains a double-quote or non-ASCII characters this breaks the header and could allow header injection. The caller controls this value.
  → *Sanitise the filename before setting the header: const safeName = fileName.replace(/["\\]/g, '_'); res.setHeader('Content-Disposition', `attachment; filename="${safeName}"`). For full RFC 6266 compliance also emit the filename* parameter with percent-encoding for non-ASCII names.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index, but budget lookups will always filter by userId and often by categoryId. At even modest scale (thousands of users), every budget fetch becomes a full table scan.
  → *Add a composite index in the table's index callback: `userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId)`. Mirror the same pattern already used on transactions.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on nextDueDate or (userId, nextDueDate). Any query fetching upcoming bills for a user (e.g., 'due in next 7 days') will scan every row in the table.
  → *Add `userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate)` in the table's index callback. This is critical for the reminder-polling feature common in billing UIs.*
- **`shared/schema.ts` (userId: varchar('user_id') — all tables):** All userId foreign key columns are declared as unbounded `varchar` with no length limit. This means the DB will accept arbitrarily long strings, wastes index space, and makes column sizing unpredictable. For a Clerk/Auth UUID or CUID the max length is known and fixed.
  → *Replace `varchar('user_id')` with `varchar('user_id', { length: 128 })` (or 36 for UUID) consistently across all tables. This tightens storage, improves index efficiency, and adds an implicit validation layer at the DB level.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt (and other createdAt columns)):** All createdAt columns use `.defaultNow()` but are not marked `.notNull()`. If the default is ever skipped programmatically (e.g., via a raw insert or migration script), the column silently accepts NULL, breaking any ORDER BY createdAt DESC queries and audit trails.
  → *Chain `.notNull()` after `.defaultNow()` on every createdAt timestamp: `timestamp('created_at').defaultNow().notNull()`. This is the pattern already correctly applied to `currency` and `status` fields elsewhere in the schema.*

## 💡 Opportunities

### Centralise pagination defaults as environment config — ⚡ Medium Impact
Hardcoded limit values (50, 200, 100) appear in multiple query functions. Extract them to a single config object loaded from environment variables (DEFAULT_PAGE_SIZE, MAX_PAGE_SIZE). This lets ops teams tune limits for low-memory Caribbean deployment targets without code changes.

### Add a query-layer caching tier for market stock data — 🔥 High Impact
getMarketStocks is called on every portfolio page load but the underlying simulated data changes infrequently. A 30-second in-process cache (Map + timestamp) would eliminate redundant DB reads across concurrent classroom users hitting the investment simulator simultaneously.

### Extract multi-step trade logic into a dedicated service class — 🔥 High Impact
The holding upsert, portfolio transaction insert, and balance update are currently scattered across storage methods and called sequentially in route handlers. Moving them into a single TradeService.executeTrade() method that owns the DB transaction boundary would make the atomicity guarantee explicit, testable, and reusable across buy/sell flows.

### Extract route groups into dedicated router modules — 🔥 High Impact
The single registerRoutes function is handling auth, AI, file upload, leaderboard, lessons, and email routes in one file estimated at 3000+ lines. Splitting into Express Router modules (e.g. routers/ai.ts, routers/lessons.ts, routers/auth.ts) would make rate limiting, middleware application, and testing scoped and explicit, and eliminate the risk of accidentally skipping auth middleware on a new route added in the wrong place.

### Centralise environment configuration with validation at startup — ⚡ Medium Impact
Values like file size limits (5MB, 500MB, 10MB) and AI quota thresholds appear as hardcoded literals scattered through the file. Moving these into a validated config module (using zod.parse on process.env at startup) means misconfiguration fails fast at boot rather than silently using wrong values in production, and makes environment-specific tuning straightforward.

### Batch repeated per-request Supabase lookups with a request-scoped cache — ⚡ Medium Impact
The Google auth handler and likely other handlers call getOrganization, getOrgEnvironmentById, and similar functions multiple times within a single request for the same IDs. A simple Map-based per-request memoisation wrapper around these functions would eliminate redundant round-trips to Supabase, reducing p99 latency on auth-heavy flows for school-scale concurrent logins.

### Parallel OpenAI calls where extraction steps are independent — ⚡ Medium Impact
If future handlers need multiple independent AI completions (e.g. subject detection and difficulty scoring as separate calls), they should be issued with Promise.all rather than sequentially. Applying this pattern now as a team convention would prevent sequential-await debt from accumulating across new handlers.

### Centralised job payload size guard at enqueue time — 🔥 High Impact
Add a size check inside enqueueJob that throws if JSON.stringify(payload).length exceeds a configured threshold (e.g. 256 KB). This creates a hard boundary that catches oversized payloads (like embedded base64 files) at the call site before they reach the database, making the fileB64 anti-pattern immediately visible to developers rather than silently degrading at scale.

### Structured logging with redaction for job handler errors — ⚡ Medium Impact
Job handler errors currently propagate raw to the worker logger. For school-aged user data (COPPA/Caribbean privacy considerations) and AI response content, add a logging wrapper that strips any field matching /b64|base64|content|token/i before writing to the log sink. This prevents accidental PII or file content exposure in log aggregators with minimal overhead.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, linkedCards). Extract `export const DEFAULT_CURRENCY = 'BSD'` into a shared config file and reference it everywhere. When the app expands to TTD or JMD markets, this becomes a single-line change instead of a schema migration hunt.

### Add partial index for active/pending document uploads — ⚡ Medium Impact
Queries polling for documents in 'processing' status will scan all documentUploads rows as the table grows. Add a partial index `CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id, created_at) WHERE status = 'processing'`. In Drizzle this requires a raw SQL index or a future `where` clause on index — document this as a pending migration. Expected impact: near-constant time polling queries regardless of historical upload volume.

### Replace sequential per-table relation definitions with co-located relation blocks — 📌 Low Impact
If relations are defined in separate blocks far from their tables (common in large schema files), circular reference issues and missed relation declarations become hard to spot during code review. Co-locating each table's `relations()` call directly below its `pgTable` definition makes it trivially auditable that every foreign key has a corresponding relation declared, reducing the risk of missing back-references that break Drizzle query joins.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
server/storage.ts exhibits several reliability and security risks including missing database transactions on multi-step writes, sequential awaits that block unnecessarily, and N+1 query patterns in dashboard aggregation. The codebase also contains hardcoded configuration values and insufficient pagination guards that could cause memory pressure at scale with school-aged user cohorts.

### API Routes (`server/routes.ts`)
server/routes.ts is a large, monolithic route file with several reliability and security gaps. The most critical issues are missing rate limiting on AI endpoints (exposing the app to cost exhaustion), sequential awaits in auth paths that could be parallelised, and sensitive data exposure risks in error responses. Overall structure is reasonable but the file needs decomposition and hardened error handling.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps including large base64 payloads stored directly in the job table, sequential DB inserts that should be batched, and a hardcoded OpenAI model string that should be environment-configurable. The extract-paper handler's idempotency delete-then-reinsert pattern is correct in intent but the sequential per-question inserts create both a performance bottleneck and a wider window for partial failure on retry.

### Database Schema (`shared/schema.ts`)
The schema is well-structured with foreign keys and composite indexes in place, but has several reliability and security gaps. Key concerns are missing indexes on high-frequency filter columns, overly permissive varchar sizing on user_id columns, and missing NOT NULL constraints on audit-critical timestamp fields that could silently produce incomplete records.
