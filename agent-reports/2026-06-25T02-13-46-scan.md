## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row, a separate query fetches the associated category and another aggregates spent amount, meaning 10 budgets = 21 queries.

**Fix:** Rewrite using a single Drizzle query with leftJoin on categories and a subquery or groupBy aggregation for spent totals. Example: db.select({ ...budgets, categoryName: categories.name, spent: sql<number>`COALESCE(SUM(${transactions.amount}),0)` }).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).groupBy(budgets.id, categories.id).where(eq(budgets.userId, userId))

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (trade execution flow)
**Issue:** Multi-step trade execution (upsert holding, insert portfolio transaction, update virtual balance) runs as three independent DB calls with no wrapping transaction. A failure after step 1 leaves holdings and balance in an inconsistent state.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { ... }) using Drizzle's transaction helper so any failure rolls back all three writes atomically.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 104-106)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. These are expensive endpoints that can be abused to drive up API costs or DoS the service, especially critical for a school-aged user base where accounts may be shared.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/15min per IP) in rateLimiter.ts and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the registerChat/Image/Audio calls.

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap.xml handler (~line 62)
**Issue:** Both handlers build absolute URLs directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header to inject arbitrary URLs into the Sitemap directive and robots.txt, enabling cache poisoning or open redirect abuse.

**Fix:** Add a trusted-host check: `const ALLOWED_HOST = process.env.PUBLIC_HOST; const host = ALLOWED_HOST || req.headers['x-forwarded-host'] || req.headers.host || 'localhost';` and set PUBLIC_HOST in your Railway environment. Reject or fall back to the env value if the header doesn't match.

### `server/jobHandlers.ts`: extract-paper registerJobHandler / job.payload
**Issue:** fileB64 (the entire exam file as a base64 string) is stored directly in the job payload, meaning every retry stores a potentially multi-MB blob in the jobs table. For a PDF this can easily exceed 10-20 MB per row, causing DB bloat, slow job queue queries, and OOM risk when many jobs are loaded into memory simultaneously.

**Fix:** Upload the file to object storage at enqueue time and store only the resulting storage path in the job payload (e.g. { paperId, storagePath, ext, subject }). Inside the handler, stream/download the file from object storage using the path, then decode it locally. This keeps job rows small (<1 KB) and the binary data in purpose-built storage.

### `server/jobHandlers.ts`: runExtractPaper – for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted sequentially in a for loop with individual awaits. For a 50-question paper this means 50 serial round-trips to the database, each waiting for the previous to complete. This is the slowest possible insertion strategy and holds the job worker thread idle for the cumulative latency of all queries.

**Fix:** Replace the for loop with a single bulk insert: filter valid questions first, then call db.insert(extractedQuestions).values(validQuestions.map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0, 4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i + 1 }))). One round-trip replaces N, dropping insertion time from O(N*latency) to O(latency).

### `shared/schema.ts`: simulatedStocks table — ticker column
**Issue:** ticker is defined as text().notNull() with no unique constraint. Nothing prevents two rows with the same ticker symbol (e.g. duplicate seed runs), which would corrupt portfolio value calculations and confuse the learning modules that reference stocks by ticker.

**Fix:** Replace the plain notNull() with a uniqueIndex: import uniqueIndex from drizzle-orm/pg-core and add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) as the table's second argument.

---

### `shared/schema.ts`: portfolioHoldings table — userId + stockId
**Issue:** There is no unique constraint on (userId, stockId) in portfolioHoldings. A user can accumulate multiple rows for the same stock from concurrent buy requests, causing double-counting of holdings and incorrect average purchase price calculations in the simulation engine.

**Fix:** Add a composite unique index: (t) => ({ userStockUnique: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). Upsert logic in the buy handler should then use ON CONFLICT (user_id, stock_id) DO UPDATE to merge quantity and recalculate avgPurchasePrice atomically.

## 🟡 Warnings (11)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits fetch total income, total expenses, savings goals, and recent transactions one after another, adding latency proportional to the number of independent DB round-trips.
  → *Replace sequential awaits with Promise.all: const [incomeResult, expenseResult, goalsResult, recentTx] = await Promise.all([db.select(...), db.select(...), db.select(...), db.select(...)]) — all four queries are independent and safe to run concurrently.*
- **`server/storage.ts` (getTransactions):** No maximum limit guard: callers can omit the limit filter and the query returns every transaction row for a user with no cap, risking out-of-memory conditions on accounts with large history.
  → *Apply a server-side ceiling: const safeLimit = Math.min(filters?.limit ?? 100, 500); and always pass .limit(safeLimit) to the Drizzle query regardless of what the caller supplies.*
- **`server/storage.ts` (getLeaderboard):** Leaderboard query joins user display names and scores then logs or returns raw userId values without redacting them; for school-aged users this exposes internal user identifiers to any authenticated caller with no role check.
  → *Return only opaque display names and avatars — strip raw userId from the response type and the SELECT clause. Add a role/auth check at the storage call site (or within the method) to ensure only teachers or admins can retrieve the full leaderboard.*
- **`server/routes.ts` (Supabase init chain (~line 109-117)):** The sequential `.then()` chain means a failure in `seedFinancialAcademyLesson()` silently skips `initStaticContentTables()`, `seedStaticModules()`, `seedRegionalContent()`, and `seedGameContent()`. The single `.catch()` only logs the first failure; subsequent seed steps that depend on prior ones may leave the DB in a partially initialised state with no per-step visibility.
  → *Run independent init steps in parallel where possible and log per-step failures: wrap each call individually with `.catch(e => console.error('[Supabase] Step X failed:', e))` or use `Promise.allSettled([initSupabaseTables(), ...]).then(results => results.forEach((r,i) => r.status==='rejected' && console.error(...)))`.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 92-100)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without path traversal sanitisation. A request to `/public-objects/../../private/secrets` could resolve outside the intended public prefix depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Use Node's `path` module and validate the resolved path stays within the public prefix.*
- **`server/jobHandlers.ts` (runExtractPaper – model: 'gpt-4o' (two occurrences)):** The OpenAI model name is hardcoded as 'gpt-4o' in two separate chat completion calls. In a school-facing financial literacy app, switching models for cost control or capability reasons requires a code deploy rather than a config change, and both call sites could drift out of sync independently.
  → *Add OPENAI_MODEL to environment config (e.g. process.env.OPENAI_MODEL ?? 'gpt-4o') and reference it in both completions calls. Validate its presence at startup alongside other required env vars so misconfiguration fails fast rather than silently using undefined.*
- **`server/jobHandlers.ts` (runExtractPaper – idempotency delete at top of function):** The handler unconditionally deletes all previously extracted questions for the paperId at the start of every attempt, including retries triggered by transient network errors after a successful extraction. If the paper status was already set to 'completed' by a prior attempt that only failed in the job acknowledgement step, this wipes valid data and re-runs two expensive OpenAI calls unnecessarily.
  → *Add an early-exit idempotency check before the delete: fetch the current paper record and if status === 'completed' return the existing questionCount immediately without calling OpenAI or touching extractedQuestions. Only proceed with the delete-and-reprocess path when status is 'processing' or 'failed'.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse – objectPath parameter):** The objectPath passed in is split directly into bucket and object path components with no validation or sanitisation. A caller supplying a path containing '..' segments could traverse outside the intended bucket prefix, potentially reading arbitrary objects from object storage. Given this serves school-aged users' uploaded files, path traversal is a meaningful risk.
  → *After splitting the path into parts, assert that no part equals '..' and that the resolved bucket matches the expected value from PRIVATE_OBJECT_DIR: const allowedBucket = process.env.PRIVATE_OBJECT_DIR?.split('/').filter(Boolean)[0]; if (bucket !== allowedBucket || parts.some(p => p === '..')) { res.status(400).json({ message: 'Invalid path' }); return; }*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, meaning every budget lookup for a user (e.g. checking budget vs. spend on dashboard load) performs a full table scan. For a school-aged user base this will degrade as transaction volume grows.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The compound index also eliminates the need for a separate single-column index.*
- **`shared/schema.ts` (transactions — createdAt / documentUploads — createdAt / portfolioTransactions — executedAt):** Multiple timestamp audit columns use .defaultNow() without .notNull(). defaultNow() only sets a default at the application layer via Drizzle; if a raw INSERT bypasses the ORM the column accepts NULL, breaking any ORDER BY createdAt DESC or time-series aggregations silently.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: createdAt: timestamp('created_at').defaultNow().notNull(). Apply to createdAt on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (billReminders — nextDueDate / savingsGoals — deadline indexes):** billReminders queries will almost always filter by userId and nextDueDate together (e.g. 'show me bills due this week'). There is no composite index on these columns, so reminder lookups require scanning all reminders for the user. Same problem applies to savingsGoals filtered by userId.
  → *Add index definitions to both tables: billReminders — (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }); savingsGoals — (t) => ({ userIdx: index('idx_savings_goals_user').on(t.userId) }).*

## 🔵 Info / Tech Debt (1)

- **`server/routes.ts` (duplicate health check endpoints (~line 30 and ~line 72)):** Two health check routes exist: `/api/health` (returns immediately) and `/healthz` (queries the DB). ## 💡 Opportunities

### Centralised pagination helper: 🔥 High Impact
Multiple list methods (getTransactions, getGameSessions, getPortfolioTransactions, getAllQuestions) each re-implement offset/limit logic independently. Extract a single paginate(query, page, pageSize, maxPageSize) utility that enforces the server-side ceiling and returns { data, total, page, pageSize }. This eliminates the missing-guard risk class across all list endpoints and makes adding cursor-based pagination later trivial.

### Environment-based configuration for hardcoded virtual balance seed: ⚡ Medium Impact
The initial virtual balance amount (commonly 10000 or similar) appears hardcoded in seedMarketData or getVirtualBalance rather than read from an environment variable. Move it to VIRTUAL_BALANCE_SEED_AMOUNT in .env so it can be tuned per deployment (school vs. sponsor demo) without a code change.

### Batch category resolution for bill reminders: ⚡ Medium Impact
getBillReminders exhibits the same N+1 pattern as getBudgets — one category lookup per reminder. Collect all distinct categoryIds from the reminder rows first, fetch them in a single inArray(categories.id, ids) query, then map them into a lookup object for O(1) attachment. Eliminates proportional DB round-trips with zero schema changes.

### Parallelise sequential domain route registrations: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import block run sequentially. If any route registration does async I/O (e.g. fetching config from DB), they block each other unnecessarily. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` to reduce cold-start time on Railway deploys.

### Centralise host/proto resolution into a reusable utility: 📌 Low Impact
The identical 3-line host-resolution block (`proto`, `host`, `base`) is duplicated verbatim in both robots.txt and sitemap.xml handlers. Extract to `getPublicBaseUrl(req): string` in a `utils/request.ts` helper. This also makes the future allowlist fix (finding #2) a single-point change.

### Add structured startup logging for Supabase init steps: ⚡ Medium Impact
Currently init success produces no logs, making it impossible to tell in Railway's log stream whether seeding ran on a given deploy. Add timestamped console.info entries per completed step. This is especially valuable for diagnosing issues in production where students report missing lesson content.

### Parallel OpenAI + storage calls in admin-csv-export: ⚡ Medium Impact
Based on the visible pattern in the file, admin export handlers likely fetch multiple data types sequentially. Wrapping independent db queries in Promise.all would reduce total job duration proportionally to the number of independent fetches, with no correctness risk since the queries are read-only and unrelated.

### Centralised job payload size guard at enqueue time: 🔥 High Impact
Add a payload size check inside enqueueJob (e.g. reject payloads exceeding 64 KB) so the fileB64 anti-pattern and any future large-payload mistakes are caught at the point of enqueueing rather than discovered through DB bloat in production. One guard protects all current and future job types.

### Structured error metadata on terminal job failures: ⚡ Medium Impact
The extract-paper handler catches terminal failures and sets status 'failed' but discards the error message. Storing a truncated error reason on the exam_papers record (e.g. failureReason column) would allow the teacher UI to surface 'No text could be extracted' vs 'OpenAI timeout' vs 'PDF parse error', reducing support burden for a school-facing product.

### Centralise currency default into a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') across at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract a DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' constant in a config module and reference it in each column definition. This allows the platform to support TTD or JMD for other Caribbean schools without a schema migration.

### Add partial index on documentUploads.status for processing rows: ⚡ Medium Impact
Background workers poll for documents with status='processing'. A partial index — index('idx_doc_uploads_processing').on(t.userId, t.createdAt).where(sql`status = 'processing'`) — would be tiny in size (only unresolved uploads) and reduce worker poll latency significantly compared to scanning the full table filtered at runtime.

### Add NOT NULL to categories.icon and categories.color with sentinel defaults: 📌 Low Impact
icon and color are nullable, forcing every consumer to null-check before rendering UI components. Defaulting to a known fallback (e.g. icon: text('icon').notNull().default('circle'), color: text('color').notNull().default('#6B7280')) eliminates null-guard branches in the frontend and prevents accidental rendering of unstyled category chips for school-aged users.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts contains several reliability and performance gaps including N+1 query patterns in budget/bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that should run in parallel. For a school-aged financial app, the absence of pagination guards and potential sensitive-data exposure in error paths are the most pressing concerns.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good patterns like parallel module loading and rate limiting on auth endpoints. The most significant concerns are unprotected AI/chat/audio/image routes with no rate limiting, a Supabase init chain that silently swallows errors for critical seeding steps, and host header injection risk in robots.txt and sitemap.xml that could enable cache poisoning attacks.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads are stored in the job table (a classic OOM/DB bloat vector), sequential DB inserts inside a loop where bulk insert would be far faster, and a hardcoded OpenAI model string that should be environment-configurable. Error handling is mostly present but idempotency on the extract-paper handler deletes previous work unconditionally rather than checking completion state first.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and correctness gaps. Key issues include missing indexes on high-frequency query columns, nullable timestamps that should be NOT NULL, and a missing unique constraint on the ticker column that could allow duplicate simulated stocks.
