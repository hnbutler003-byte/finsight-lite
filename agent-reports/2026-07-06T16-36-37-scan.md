## 🔴 Critical (10)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate aggregation query per budget row to compute `spent`, resulting in 1+N round trips to the database for N budgets.

**Fix:** Replace the per-budget loop with a single LEFT JOIN query that aggregates transaction amounts grouped by categoryId within the date range, then merge the result set in JavaScript. Use Drizzle's `sql` helper: SELECT b.*, COALESCE(SUM(t.amount),0) AS spent FROM budgets b LEFT JOIN transactions t ON t.category_id = b.category_id AND t.user_id = b.user_id WHERE b.user_id = :userId GROUP BY b.id.

### `server/storage.ts`: getTransactions / getDashboardStats
**Issue:** Missing pagination guard: getTransactions accepts an optional `limit` but applies no default maximum, allowing an unbounded SELECT on the transactions table. getDashboardStats similarly issues a full-table scan per user with no row cap, which will degrade and OOM under real school cohort load.

**Fix:** Add a hard server-side cap: `const safeLimit = Math.min(filters?.limit ?? 100, 500);` and always apply `.limit(safeLimit)` in the Drizzle query. For getDashboardStats use a date-bounded subquery or materialised summary rather than scanning all rows.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered without any rate limiting middleware applied at this level. If the downstream route files also omit rate limiting, these endpoints are fully open to abuse — especially critical for a school-aged user base where API cost and prompt-injection risk are high.

**Fix:** Apply a dedicated aiLimiter (e.g. 20 req/min per IP) before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls. Confirm the downstream files do not already apply this before closing the finding.

### `server/routes.ts`: robots.txt handler (~line 37) and sitemap.xml handler (~line 52)
**Issue:** The `base` URL is constructed directly from `x-forwarded-host` and `x-forwarded-proto` headers without any validation. A spoofed Host header produces an arbitrary URL in the Sitemap directive and sitemap <loc> elements, which could be used to poison search-engine indexes or craft misleading content. In a school-facing app this is a reputational and safety risk.

**Fix:** Validate against an allowlist: `const ALLOWED_HOSTS = new Set((process.env.ALLOWED_HOSTS || '').split(',')); const host = ALLOWED_HOSTS.has(rawHost) ? rawHost : process.env.DEFAULT_HOST;` Pull DEFAULT_HOST and ALLOWED_HOSTS from environment config rather than trusting the request.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored in the job payload, bloating the jobs table with potentially megabytes per row. This degrades DB performance, increases backup size, and risks exposing file bytes in job logs or error traces.

**Fix:** Upload the file to object storage at enqueue time and store only the resulting object path in the job payload. In runExtractPaper, download the file from object storage using that path instead of decoding fileB64. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts`: purge-ai-usage handler
**Issue:** The db.delete call has no try/catch. An unhandled DB error will crash the worker process rather than being recorded as a failed job, preventing retries and breaking observability.

**Fix:** Wrap the entire handler body in try/catch, capture the error with the existing sentry captureError utility, then rethrow so the job framework can record failure and schedule a retry: try { ... } catch (err) { capture(err, { job: 'purge-ai-usage' }); throw err; }

### `server/index.ts`: startSchedulers / setInterval calls
**Issue:** setInterval handles for the weekly digest and AI usage purge schedulers are never stored, so they cannot be cleared during graceful shutdown. On SIGTERM the process exits with live timers that may fire mid-shutdown, and in test/reload scenarios each call to startSchedulers leaks additional intervals.

**Fix:** Store every setInterval return value in a module-level array (e.g. `const timerHandles: NodeJS.Timeout[] = []`). In the SIGTERM/SIGINT handler call `timerHandles.forEach(clearInterval)` before closing the HTTP server and draining the job worker.

---

### `server/index.ts`: request logging middleware — res.on('finish') handler
**Issue:** Every API response body is captured via monkey-patching res.json and then serialized with JSON.stringify into the log line unconditionally. For a Caribbean school-aged-user financial app this will log quiz answers, user scores, session tokens, and any PII returned by the API, violating data-minimisation principles and creating a compliance risk.

**Fix:** Remove `capturedJsonResponse` capture entirely, or gate it behind `process.env.NODE_ENV === 'development'` and add an explicit allowlist of safe fields (e.g. only log `{ status, error }` keys). Never log response bodies in production.

### `client/src/pages/AdminDashboard.tsx`: sendMessage – apiRequest POST /api/admin/help-chat
**Issue:** The full accumulated messages array is sent to /api/admin/help-chat on every message. As the conversation grows this payload becomes arbitrarily large. Combined with no server-side session binding, a non-admin user who discovers the endpoint can POST arbitrary message histories with no client-enforced cap, potentially running up LLM costs or probing the system prompt.

**Fix:** Enforce a rolling window before sending: `const context = next.slice(-10)` and send `context` instead of `next`. On the server route add an explicit admin session check before forwarding to the LLM provider, independent of any client-side guard.

---

### `client/src/pages/AdminDashboard.tsx`: Top-level AdminDashboard component – useAdminAuth hook usage
**Issue:** The component imports `useAdminAuth` but the truncated code suggests the admin-protected sections render immediately during the auth loading state before the hook resolves. If `useAdminAuth` returns `{ isAdmin, isLoading }` and the dashboard renders data-fetching `useQuery` calls unconditionally, those queries fire before authentication is confirmed, leaking admin query keys into unauthenticated React Query caches and making unnecessary network requests.

**Fix:** Gate all `useQuery` calls and dashboard JSX behind auth resolution: `if (isLoading) return <FullPageSpinner />; if (!isAdmin) { setLocation('/'); return null; }` placed immediately after the hook call, before any query hooks. This ensures queries only register once admin identity is confirmed.

## 🟡 Warnings (18)

- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices updates every stock row sequentially in a loop with individual awaited UPDATE statements and no wrapping database transaction, so a mid-loop failure leaves prices partially updated — a correctness hazard for the investment simulation feature used by students.
  → *Wrap all price updates in a single `db.transaction(async (tx) => { for (const stock of stocks) { await tx.update(...) } })` call. If the loop is large, batch with a CASE-WHEN single UPDATE or use `db.insert().onConflictDoUpdate()` with the full new-price array.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard fetches game sessions for all users then calls `getUser(userId)` inside a map to resolve display names — a classic N+1 producing N user lookups after the initial sessions query. User data may also expose PII (email, internal IDs) in the returned leaderboard payload.
  → *Rewrite as a single JOIN: SELECT gs.user_id, u.name, u.avatar, SUM(gs.score) AS totalScore, COUNT(*) AS gamesPlayed FROM game_sessions gs JOIN users u ON u.id = gs.user_id GROUP BY gs.user_id ORDER BY totalScore DESC LIMIT :limit. Project only the display-safe columns (name, avatar) — never return email or internal user fields.*
- **`server/storage.ts` (completeModule / addUserBadge):** completeModule awards XP and then conditionally calls addUserBadge in separate awaited statements with no transaction boundary. A crash between the two writes leaves XP credited but badge unissued (or vice-versa), corrupting gamification state for student accounts.
  → *Wrap the XP update and badge insert in `db.transaction(async (tx) => { await tx.update(userXp)...; await tx.insert(userBadges)...; })` so both succeed or both roll back atomically.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 112)):** The seven Supabase seed/init operations are chained sequentially with .then(), meaning each waits for the previous to complete. Total cold-start blocking time is the sum of all operations. While fire-and-forget, this delays data availability unnecessarily and makes the chain fragile — a failure in seedFinancialAcademyLesson silently skips all subsequent seeds.
  → *Run independent seeds in parallel: `Promise.all([initStaticContentTables(), seedRegionalContent(), seedGameContent()]).catch(...)`. Keep initSupabaseTables first as a prerequisite, then fan out: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables().then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]))])).catch(e => console.error('[Supabase] Init error:', e));`*
- **`server/routes.ts` (/healthz handler (~line 78)):** The /healthz endpoint performs a live DB query (`select 1`) on every call with no rate limiting or caching. Uptime monitors hitting this every 30 seconds is fine, but the endpoint is publicly accessible and could be used to generate sustained DB load. Additionally, dynamic import inside a hot request path (`await import('./db')`) bypasses module caching on some runtimes and adds latency.
  → *Move the `db` and `sql` imports to the top of the file (static imports). Add a lightweight in-memory cache: only re-probe the DB if the last successful check was >10 s ago, returning the cached result otherwise. Apply the existing `authLimiter` or a dedicated `healthLimiter` to this route.*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop):** Each extracted question is inserted with a separate awaited DB call inside a for loop, causing N sequential round-trips (potentially 50+ for a full exam paper). This is slow and holds the worker busy for seconds unnecessarily.
  → *Collect all valid question objects into an array, then call a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, wrap calls in Promise.all with a concurrency cap (e.g. p-limit at 5).*
- **`server/jobHandlers.ts` (runExtractPaper – model string 'gpt-4o'):** The OpenAI model name is hardcoded in two separate chat completion calls. Changing the model for cost or capability reasons requires a code deploy rather than a config change, and different environments cannot use different models.
  → *Add OPENAI_MODEL to environment config (e.g. process.env.OPENAI_MODEL ?? 'gpt-4o') and reference that constant in both openai.chat.completions.create calls. Validate the env var is present at startup alongside other required vars.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The function accepts an arbitrary objectPath string and constructs a bucket/object reference from it without validating that the resolved path stays within the expected private directory. A caller passing a crafted path like '../../public/sensitive' could read unintended buckets.
  → *After splitting the path, assert that parts[0] matches the expected bucket name from PRIVATE_OBJECT_DIR. Alternatively, reuse parsePrivatePath here so both upload and download go through the same validated path logic: const { bucket, object } = parsePrivatePath(objectPath).*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups will always filter by userId. At scale this becomes a full table scan for every dashboard load.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget entries per category per period.*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, portfolioTransactions table definitions):** None of these tables define indexes on userId, yet all will be queried by userId in every route. portfolioTransactions also lacks an index on (userId, executedAt) needed for time-ordered history queries.
  → *Add index definitions to each table: billReminders and savingsGoals get index('idx_bill_reminders_user').on(t.userId) and index('idx_savings_goals_user').on(t.userId). portfolioHoldings gets a uniqueIndex on (userId, stockId). portfolioTransactions gets index('idx_portfolio_tx_user_date').on(t.userId, t.executedAt).*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text rather than timestamp. Storing dates as text prevents range queries, ORDER BY date correctness, and timezone handling. In a financial simulation context this will produce wrong sorting and broken date comparisons.
  → *Change the column definition to: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed data or insertion logic that currently writes ISO strings to pass Date objects instead.*
- **`shared/schema.ts` (transactions createdAt, budgets createdAt, linkedCards createdAt, documentUploads createdAt, savingsGoals createdAt, billReminders createdAt):** All createdAt columns use .defaultNow() without .notNull(), meaning the column is nullable in the database type system even though a default is always set. An explicit INSERT with createdAt: null would succeed and break audit trails — critical for a financial app.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for portfolioHoldings.purchasedAt and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (categories userId, linkedCards table):** categories.userId is varchar without a length constraint, and linkedCards has no unique constraint on (userId, lastFour, bankName), allowing duplicate card registrations. Unbounded varchar also bypasses any application-level ID length assumptions and wastes storage.
  → *Set a consistent length on all user_id varchar columns matching your auth provider's ID length, e.g. varchar('user_id', { length: 128 }). Add uniqueIndex('idx_linked_cards_user_card').on(t.userId, t.lastFour, t.bankName) to linkedCards to prevent duplicate card entries.*
- **`server/index.ts` (main() async startup function — sequential awaits for probeDatabase, seedDatabase, registerRoutes):** probeDatabase, seedDatabase, and registerRoutes are awaited sequentially before httpServer.listen() is called. If any step is slow (e.g. cold DB connection) the process appears unresponsive to the process manager and no requests are served. seedDatabase and probeDatabase have no dependency on each other and can run in parallel.
  → *Run `await Promise.all([probeDatabase(), seedDatabase()])` before registerRoutes (which does depend on a live DB). Also start httpServer.listen() immediately and return 503 on health checks until the DB probe succeeds, so the process manager sees the port open quickly.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — the `+ 0` is dead/no-op arithmetic that suggests a timezone offset was intended (e.g. converting UTC to Caribbean local time, UTC-4) but was never implemented. Sunday at 19:00 is evaluated in server local time/UTC, so the digest fires at the wrong wall-clock time for Caribbean users.
  → *Replace the no-op offset with the correct UTC offset for the target timezone (e.g. `(dow + 7 - 0) % 7` is still wrong; use a proper offset: compute the current time in `America/Port_of_Spain` using `Intl.DateTimeFormat` or the `luxon`/`date-fns-tz` library before deriving the most-recent Sunday 19:00 local time). Remove the dead `+ 0`.*
- **`server/index.ts` (maybeRunAiUsagePurge — db.delete inside scheduler):** The AI usage purge runs a DELETE with a date-range filter inside a bare setInterval with no concurrency guard. If the interval fires again before the previous async DELETE resolves (e.g. under DB slowness), multiple overlapping deletes execute simultaneously. There is also no upper-bound on rows deleted per run, risking a long-running transaction that locks the table.
  → *Add a module-level `let purgeRunning = false` flag; set it true at the start of the async work and false in a finally block, returning early if already true. Add a `LIMIT` clause (e.g. 1000 rows) and loop until no rows remain, yielding between batches with a short await, to avoid long table locks.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scrollIntoView ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` creates a brand-new function on every render of the messages list. React calls the old ref with null and the new ref with the element on every render cycle, causing spurious scroll jumps and unnecessary DOM thrashing on every keystroke or state change.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`. This fires scroll only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage catch block):** The catch block silently swallows all errors including network failures, non-2xx HTTP responses, and JSON parse errors, appending only a generic string. There is no distinction between a 401 (session expired), 429 (rate limit), or a 500, so admins cannot act on the real cause.
  → *Inspect the response before calling `.json()`: `if (!res.ok) throw new Error(res.status === 429 ? 'Rate limit reached, wait a moment.' : 'Server error ' + res.status);` Catch typed errors and set a descriptive message. Also add `finally { setLoading(false); }` is already present – but ensure the error message references `err instanceof Error ? err.message : 'Unknown error'`.*

- **`server/routes.ts` (registerAuthDomainRoutes / registerStudentRoutes / ... awaited sequentially (~line 127)):** The six domain router registration calls are awaited one at a time after the parallel import. If any of these async registration functions performs I/O (DB schema checks, cache warm-up, etc.), the total startup time is their sum. The parallel import gain is partially wasted.

## 💡 Opportunities

### Parallel independent fetches in getDashboardStats: 🔥 High Impact
getDashboardStats sequentially awaits income total, expense total, savings goals count, and budget count — all independent queries. Running them with Promise.all([incomeQ, expenseQ, goalsQ, budgetsQ]) would cut dashboard load time by ~3x on typical DB latency (4 serial round-trips become 1 parallel batch).

### Centralised pagination middleware: ⚡ Medium Impact
At least six list-returning methods (getTransactions, getExamPapers, getGameSessions, getAllQuestions, getPortfolioTransactions, getLeaderboard) each have bespoke or missing limit handling. Introducing a shared `paginate(query, {limit, offset})` utility with an enforced 500-row cap eliminates the repeated pattern and prevents future regressions.

### Environment-config extraction for hardcoded simulation constants: 📌 Low Impact
simulateMarketPrices contains hardcoded volatility multipliers and price-floor values (e.g. 0.02 drift, $0.01 floor) embedded in arithmetic expressions. Moving these to `process.env.MARKET_VOLATILITY` / a config object allows tuning without redeployment and makes the educational simulation easier to adjust for different Caribbean curriculum contexts.

### Centralise base-URL resolution into a single middleware: ⚡ Medium Impact
The proto/host resolution logic is duplicated identically in robots.txt and sitemap.xml handlers. Extract into a `resolveBaseUrl(req)` utility that validates against an env allowlist (fixing the security finding simultaneously). This removes ~10 lines of duplication and ensures any future routes needing absolute URLs use the validated version.

### Replace fire-and-forget Supabase init with a tracked startup health gate: 🔥 High Impact
Currently seed failures are only logged; the app serves traffic with potentially missing seed data (missing lesson content, game data) silently. Introduce a lightweight `startupReady` promise that resolves after critical seeds complete and expose its status on /healthz. This prevents Railway from routing traffic before the app is truly ready and gives operators visibility into seed failures.

### Consolidate duplicate /api/health and /healthz endpoints: 📌 Low Impact
Two health endpoints exist (/api/health returns a trivial JSON blob, /healthz does a DB probe) with overlapping purposes. Merge into a single /healthz that returns both the timestamp and DB status, and add a lightweight in-memory TTL cache on the DB check. This reduces confusion for ops tooling and eliminates the redundant endpoint.

### Idempotency token on question bulk insert: 🔥 High Impact
The current idempotency guard deletes then re-inserts all questions on retry. A safer pattern is to insert with ON CONFLICT DO NOTHING keyed on (paperId, order), removing the delete and eliminating the window where a crash leaves zero questions for a paper that had previously completed extraction. Expected impact: eliminates data-loss race condition on retries.

### Structured logging instead of implicit error propagation: ⚡ Medium Impact
Several catch blocks silently swallow errors (catch { /* ignore */ }) with no log output. Adding a lightweight structured log call (e.g. logger.warn({ paperId, err }, 'status sync failed')) at each swallowed error site would surface intermittent failures in production monitoring without changing retry behavior.

### Extract AI prompt strings to versioned constants file: ⚡ Medium Impact
System prompts are defined inline inside async functions, making A/B testing, auditing, and prompt versioning impractical. Moving them to a dedicated prompts.ts module with named exports enables prompt diffing in git, simplifies unit testing of extraction logic, and allows runtime prompt overrides via feature flags.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate table columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' and reference it in each column definition. This prevents silent divergence if the app ever needs to support TTD or JMD alongside BSD.

### Add a composite unique constraint on portfolioHoldings (userId, stockId): 🔥 High Impact
Without this constraint, concurrent buy operations can insert two rows for the same user and stock instead of updating quantity, corrupting portfolio value calculations. A uniqueIndex forces the application to use an upsert pattern and guarantees one holding row per asset per user.

### Replace serial primary keys with uuid for user-facing IDs: 🔥 High Impact
Using serial integer PKs exposes sequential IDs in API responses, allowing users to enumerate other users' transactions, goals, and documents by incrementing IDs — a significant IDOR risk for an app serving school-aged users. Switching to gen_random_uuid() as the PK type eliminates enumeration and aligns with the varchar user_id already used on the auth model.

### Structured JSON logging instead of console.log strings: 🔥 High Impact
Replacing the freeform `log()` function with a structured logger (e.g. `pino`) emitting JSON lines would allow log aggregators to filter by severity, source, and userId without regex parsing. This is especially valuable for a school app where per-student audit trails may be legally required under Caribbean data-protection frameworks.

### Centralised graceful-shutdown orchestrator: 🔥 High Impact
Collecting all shutdown concerns (HTTP server close, job worker drain, interval cleanup, DB pool end, Sentry flush) into a single ordered shutdown() function triggered by both SIGTERM and SIGINT would prevent the current partial-shutdown risk where some resources are cleaned up and others are not, and would make it straightforward to add future resources.

### Extract scheduler state into a dedicated SchedulerRegistry: ⚡ Medium Impact
Moving weekly-digest and purge scheduling logic into a small SchedulerRegistry class with start(), stop(), and status() methods would eliminate the leaked-interval and overlapping-run bugs as a class invariant, and make the schedulers independently testable without spinning up the full Express server.

### Parallel data fetching with Promise.all: 🔥 High Impact
If the dashboard fires multiple independent useQuery calls for users, schools, analytics, and audit logs sequentially or in waterfall due to component structure, refactor to ensure all queries are registered at the same render level so React Query dispatches them in parallel. Expected impact: reduced dashboard initial load time by the sum of all sequential round-trips minus the longest single request.

### Externalize API base URL and LLM endpoint to environment config: ⚡ Medium Impact
Hardcoded paths like `/api/admin/help-chat` should be composed from `import.meta.env.VITE_API_BASE_URL` so staging and production environments can be targeted without code changes. This also prevents accidental production API calls during local development.

### Virtualize long admin lists to prevent DOM bloat: ⚡ Medium Impact
Admin tables listing all users, schools, or audit events likely render every row into the DOM simultaneously. Introducing @tanstack/react-virtual for these lists will cap DOM nodes to the visible viewport, reducing memory usage and time-to-interactive for datasets over ~200 rows, which is realistic for a school district deployment.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits multiple N+1 query patterns in budget and bill-reminder fetches, missing database transactions on multi-step writes, and sequential awaits where parallel execution is safe. For a school-aged fintech app, the absence of pagination guards on unbounded list queries and lack of try/catch on async DB operations are the most pressing reliability and security risks.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good patterns like parallel module loading and try/catch on most async ops. Key concerns are: the Supabase init chain runs sequentially instead of in parallel, host/proto headers are trusted without validation creating a potential open-redirect/injection vector in robots.txt and sitemap, and AI/chat routes lack visible rate limiting at the registration point.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps including large base64 payloads stored directly in the job table, sequential DB inserts that should be batched, and missing try/catch around the AI usage purge handler. Security exposure exists via base64 file data potentially appearing in job logs, and hardcoded model names should be environment-configurable.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-cardinality filter columns, nullable timestamps that should be NOT NULL, and unbounded varchar columns that risk data integrity issues for a financial application serving minors.

### Server Startup (`server/index.ts`)
The startup sequence blocks the event loop with sequential awaits for DB probe, seeding, and route registration before the server begins listening, creating unnecessary delay. Interval-based schedulers are started without storing their handles, making graceful shutdown impossible and causing memory/timer leaks. The response-body logging middleware unconditionally serializes and logs every API JSON response body, which risks exposing sensitive financial or PII data in logs for a school-aged user app.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including a scrollIntoView callback that creates a new function on every render causing memory pressure, missing auth guard checks before rendering sensitive admin UI, and unprotected async operations. The chat component rebuilds message history on every keystroke due to inline state updates, and hardcoded API paths should be environment-driven.
