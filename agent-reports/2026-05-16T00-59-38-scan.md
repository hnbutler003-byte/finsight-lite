## 🔴 Critical (10)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: the function fetches all holdings for a user then issues a separate getStockById query per holding inside a loop, producing 1+N database round-trips for N holdings.

**Fix:** Replace the per-holding lookup with a single JOIN query using Drizzle's with() or a manual inner join: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. This collapses N+1 into a single query regardless of portfolio size.

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** N+1 query pattern: enrollments are fetched first, then a separate user lookup is fired for each enrollment's studentId inside a map/loop, producing 1+N queries per class load. This will degrade noticeably with classes of 30+ students.

**Fix:** Use a single JOIN: `db.select().from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId))`. Alternatively collect all studentIds and use `inArray(users.id, ids)` with a follow-up Map lookup to assemble the result in one extra query instead of N.

### `server/storage.ts` — getTransactions / getDashboardStats
**Issue:** Unbounded queries with no pagination guard: getTransactions accepts an optional limit but does not enforce a maximum cap, and getDashboardStats aggregates across an unbounded date range. A single user with years of transactions can cause multi-MB result sets and OOM pressure on the Node process.

**Fix:** Enforce a hard server-side cap: `const safeLimit = Math.min(filters?.limit ?? 100, 500)` and always apply `.limit(safeLimit)` in getTransactions. For getDashboardStats, require or default startDate to 'now minus 90 days' if omitted, and add a date-range guard: if the range exceeds 366 days throw a 400-level error before hitting the DB.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (trade execution path)
**Issue:** Multi-step write operations (update balance, upsert holding, insert portfolio transaction) are executed as independent statements with no wrapping database transaction. A process crash or network error between steps leaves the database in an inconsistent state — e.g., balance debited but holding never recorded.

**Fix:** Wrap the entire trade sequence in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the `tx` client to each sub-operation so all writes commit atomically or roll back together. This is non-negotiable for any code that moves virtual money.

### `server/routes.ts` — AI/tutor endpoints using reserveQuota and openai
**Issue:** No rate limiting middleware is applied to AI endpoints before the quota reservation logic. A single authenticated student (or a compromised session) can hammer the endpoint in a tight loop; reserveQuota is an async DB write that will be called on every request before it can reject, creating a thundering-herd DB write storm that could exhaust the connection pool.

**Fix:** Add express-rate-limit before the route handler: `import rateLimit from 'express-rate-limit'; const aiLimiter = rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.post('/api/tutor/ask', isAuthenticated, aiLimiter, handler)`. The keyGenerator must use userId, not IP, since school networks share a single IP.

### `server/routes.ts` — catch blocks throughout async route handlers
**Issue:** Multiple catch blocks pass e.message or the full error object directly into res.json({ error: e.message }). For Supabase/Drizzle errors this can expose table names, column names, constraint names, and query fragments — a meaningful information disclosure for a financial app serving minors.

**Fix:** Create a central error sanitizer: `function safeError(e: unknown) { const msg = e instanceof Error ? e.message : 'Unexpected error'; const isSafe = /quota|not found|unauthorized|forbidden|validation/i.test(msg); return isSafe ? msg : 'An internal error occurred'; }`. Use it in every catch: `res.status(500).json({ error: safeError(e) })`. Log the raw error server-side via your Sentry integration instead.

### `server/routes.ts` — videoUpload multer config / video upload route handler
**Issue:** 500 MB files are accepted into multer.memoryStorage(). A single upload of a large video file loads the entire file into Node.js heap. Two concurrent uploads of 500 MB each can OOM a typical 1 GB Replit container, taking down the server for all school users.

**Fix:** Switch videoUpload to disk storage with a temp directory: `multer({ storage: multer.diskStorage({ destination: os.tmpdir(), filename: (_req, file, cb) => cb(null, crypto.randomUUID() + path.extname(file.originalname)) }) })`. Then stream the file to object storage with a pipeline rather than reading the buffer. Clean up the temp file in a finally block.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored in the job payload. For a multi-page PDF or high-res image this can easily be 5–20 MB per job row, bloating the jobs table, slowing queue polling queries, and loading the entire payload into Node heap on every retry attempt.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload. In runExtractPaper, download the file from object storage using the path, process it, and delete the temp object on success. This keeps job rows small (< 1 KB) and makes retries cheap.

### `shared/schema.ts` — simulatedStocks table definition (~line 70)
**Issue:** The ticker column has no unique index. If seed scripts or admin tooling insert a duplicate ticker symbol (e.g. 'BISX'), portfolio queries joining on ticker will return duplicate rows and inflate holdings calculations — a silent data correctness bug in a financial context.

**Fix:** Add a unique index in the table config: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also consider adding a NOT NULL constraint to annualReturnPct or defaulting it to 0.00 so return calculations never silently operate on null.

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel — queryFn arrow function
**Issue:** The queryFn promise chain has no error handling. If apiRequest throws (network error, 401, 500), TanStack Query will catch it internally but the component never surfaces an actionable error message — the table just stays empty or shows stale data, which is dangerous for an audit log admins rely on for compliance.

**Fix:** Add explicit error UI: destructure error from useQuery and render an alert when it is non-null, e.g. <div className='text-red-400'>Failed to load audit log: {error.message}. <button onClick={refetch}>Retry</button></div>. Also add a .catch inside the queryFn to log the raw response body before re-throwing so engineers see details in Sentry.

## 🟡 Warnings (13)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits on independent aggregation queries (total income, total expenses, category breakdown, recent transactions) that have no data dependency on each other, adding cumulative latency equal to the sum of all query times instead of the slowest one.
  → *Replace sequential awaits with `const [income, expenses, breakdown, recent] = await Promise.all([incomeQuery, expensesQuery, breakdownQuery, recentQuery])`. Ensure all four queries reference only read operations so there is no write-ordering concern. Expected latency reduction: 3x–4x on a typical dashboard load with a remote DB.*
- **`server/routes.ts` (/api/auth/google handler):** Sequential awaits in Google auth class-code resolution make 3-4 serial DB/Supabase round-trips: getClassByCode → getOrgEnvironmentById → getOrganization, each waiting on the previous. Under load this multiplies latency and holds the event loop longer than necessary for a school-hours traffic spike.
  → *After resolving resolvedClass, fan out independent lookups with Promise.all: const [classEnv, otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), someOtherIndependentCall()]). The org domain check depends on classEnv, so only that chain must stay sequential; sibling lookups for resolvedOrgEnv can run concurrently with the class branch.*
- **`server/routes.ts` (import * as XLSX from 'xlsx'):** xlsx (SheetJS community edition) is imported at module load time and used synchronously via XLSX.read() on uploaded buffers. XLSX.read() is a blocking CPU operation — parsing a large spreadsheet can block the Node.js event loop for hundreds of milliseconds, stalling all concurrent HTTP requests including active student quiz sessions.
  → *Offload XLSX parsing to a worker thread using Node's `worker_threads` module or enqueue it as a background job via your existing `enqueueJob` infrastructure. Pass the buffer via `workerData` and return the parsed rows through a MessageChannel. This keeps the event loop free during class import operations.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this fires 40 serial round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy far longer than needed.
  → *Replace the for loop with a single batched insert: collect valid questions into an array, then call db.insert(extractedQuestions).values(validQuestions) once. Add a transaction wrapper around the batch insert and the subsequent updateExamPaper call so both succeed or both roll back atomically, which also tightens idempotency on retry.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (both vision and extraction)):** max_completion_tokens is hardcoded to 4096 in both OpenAI calls. If the model or quota changes, or if you need to tune limits per environment (dev vs prod), this requires a code deploy rather than a config change.
  → *Read the limit from environment config: const maxTokens = parseInt(process.env.OPENAI_MAX_COMPLETION_TOKENS ?? '4096', 10); and use maxTokens in both calls. Add a validation guard at startup to warn if the value is outside a sane range (512–16384).*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call. If the database is unavailable or the query times out, the unhandled rejection will surface as an uncaught exception in the worker process rather than being recorded as a failed job, potentially crashing the worker.
  → *Wrap the db.delete call in try/catch and rethrow a typed error so the job framework can record it as a failure and schedule a retry: try { const result = await db.delete(...); ... } catch (err) { throw new Error(`purge-ai-usage failed: ${(err as Error).message}`); }*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — res.setHeader Content-Disposition):** The fileName parameter is interpolated directly into the Content-Disposition header without sanitization: filename="${fileName}". A fileName containing a double-quote or newline character can break the header or enable header injection, exposing admin export endpoints to manipulation.
  → *Sanitize the filename before setting the header: const safeFileName = fileName.replace(/["\\\r\n]/g, '_'); res.setHeader('Content-Disposition', `attachment; filename="${safeFileName}"`); Alternatively use the RFC 5987 encoded form: filename*=UTF-8''${encodeURIComponent(fileName)}*
- **`shared/schema.ts` (budgets table definition (~line 36)):** budgets table has no index on userId even though budget lookups will always filter by user. Every budget query for a user triggers a full table scan as the table grows.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The compound index also prevents duplicate budget entries per category per user if you add a uniqueIndex instead.*
- **`shared/schema.ts` (varchar user_id columns across all tables (transactions ~line 22, budgets ~line 37, etc.)):** All userId columns are declared as varchar() with no length argument. In PostgreSQL, bare varchar without length is equivalent to text, so no truncation guard exists. More importantly, if the auth provider (e.g. Clerk) uses IDs up to 32 chars, a future schema migration that adds a length could silently corrupt existing data, and ORM type inference loses specificity.
  → *Declare all userId columns with an explicit length matching your auth provider's ID format, e.g. varchar('user_id', { length: 64 }).notNull().references(() => users.id). Audit the auth model in ./models/auth to confirm the exact max length and apply consistently across all 8+ tables.*
- **`shared/schema.ts` (createdAt columns across all tables (e.g. transactions ~line 33, budgets ~line 41)):** All createdAt columns use .defaultNow() but omit .notNull(). This means application code or a raw INSERT that explicitly passes NULL will store a null audit timestamp, breaking any analytics or sorting query that relies on createdAt being present.
  → *Chain .notNull() after .defaultNow() on every createdAt column: createdAt: timestamp('created_at').defaultNow().notNull(). Same fix applies to purchasedAt on portfolioHoldings and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (billReminders table definition (~line 58)):** nextDueDate has no index, but bill reminder queries will nearly always filter or order by this column (e.g. 'show reminders due this week'). Without an index, these queries scan the entire table per user.
  → *Add a compound index on (userId, nextDueDate): (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This supports both user-scoped lookups and date-range filtering in a single index scan.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery result has no isLoading or error handling. If the /api/admin/observability request fails or is slow, the component silently renders with undefined data and all badge conditions evaluate falsy with no user feedback.
  → *Destructure isLoading and error from useQuery. Return a skeleton or spinner when isLoading is true, and an error alert (e.g. 'Failed to load observability status') when error is truthy. This prevents silent failures for admin users diagnosing production issues.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params construction (above useQuery call)):** URLSearchParams is constructed inline on every render, not memoized. Because params is rebuilt unconditionally, its toString() value is used in the queryFn closure but the queryKey uses the raw filter primitives, creating a subtle stale-closure risk if params ever diverges from the key, and wastes cycles on every keystroke in the filter inputs.
  → *Wrap params construction in useMemo: const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); /* ... */ p.set('limit', '200'); return p; }, [actorType, action, from, to]); Then use params.toString() in both queryKey and queryFn to guarantee they stay in sync.*

## 💡 Opportunities

### Centralised query result caching for market data and learning modules — 🔥 High Impact
getMarketStocks and getLearningModules are read-heavy, rarely-mutated queries called on every page load for every user. Wrapping them with a simple in-process TTL cache (node-cache or a Map with a 60-second expiry) would eliminate repeated DB round-trips for static data, meaningfully reducing DB load at scale without adding infrastructure.

### Extract hardcoded virtual balance seed value to environment config — ⚡ Medium Impact
The initial virtual balance (likely hardcoded as a numeric literal in getVirtualBalance's upsert path) is a game-balance decision that product will want to tune without a deployment. Move it to an environment variable VIRTUAL_BALANCE_SEED_AMOUNT with a validated fallback, making it configurable per environment.

### Add structured error logging with context instead of raw console.error — ⚡ Medium Impact
Async operations that do have try/catch blocks log raw Error objects via console.error, which in production emits unstructured text with no userId, operation name, or request trace. Replacing with a structured logger (pino or winston) that includes {userId, operation, errorCode} as JSON fields would make incidents in this school-user context traceable without exposing PII in plaintext logs.

### Batch Supabase org/environment lookups with a request-scoped cache — 🔥 High Impact
The Google auth flow and several other handlers call getOrganization(), getOrgEnvironmentById(), and getOrgQuotaSettings() independently within the same request lifecycle. Introducing a lightweight per-request Map cache (keyed by org_id) and fetching org + quota settings in one Promise.all at the start of each handler would eliminate 2-3 redundant round-trips per authenticated request, meaningfully reducing p95 latency during school-hours peak.

### Extract AI quota enforcement into a reusable middleware — 🔥 High Impact
reserveQuota, getCachedExplanation, hashTutorQuestion, and finalizeUsage/releaseReservation appear to be duplicated across multiple AI route handlers. Consolidating them into a single Express middleware (app.use('/api/ai', quotaMiddleware)) would eliminate the duplication risk, ensure consistent behavior if quota logic changes, and make each route handler 30-40 lines shorter.

### Replace hardcoded file-size and MIME-type constants with environment config — ⚡ Medium Impact
Values like 500 * 1024 * 1024 (video limit), 5 * 1024 * 1024 (logo/upload limit), and 10 * 1024 * 1024 (exam limit) are scattered across four multer configs. Moving these to environment variables (VIDEO_UPLOAD_MAX_BYTES, LOGO_UPLOAD_MAX_BYTES, etc.) allows ops tuning without code deploys, and lets staging environments use smaller limits to catch OOM issues before they hit production.

### Batch object-storage existence checks in admin-csv-export — 🔥 High Impact
If the truncated admin-csv-export handler fetches multiple records and checks object storage per row sequentially, replacing those with Promise.all batches (capped at ~10 concurrent with a concurrency limiter) could cut export job duration by 60–80% for large datasets.

### Centralise OpenAI call error handling with retry telemetry — ⚡ Medium Impact
Both OpenAI calls in runExtractPaper lack specific handling for rate-limit (429) vs. transient (5xx) errors. A shared wrapper that inspects the error code, logs structured telemetry to aiUsageEvents, and surfaces a retryable vs. permanent failure distinction would improve observability and prevent wasted retries on permanent quota exhaustion.

### Extract PDF/image text extraction into a dedicated pre-processing job — ⚡ Medium Impact
Splitting extract-paper into two jobs — one for file-to-text extraction and one for text-to-questions via OpenAI — would allow independent retries at each stage, avoid repeating expensive OCR/vision calls on extraction-API failures, and make each job's payload and timeout budget smaller and easier to reason about.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, linkedCards). Extract it as const DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it in each .default() call. When the app expands to TTD or JMD markets, one line changes instead of six, eliminating the risk of a missed update causing currency mismatch in reports.

### Add a partial index on documentUploads status for processing-queue queries — ⚡ Medium Impact
Background jobs that poll for unprocessed uploads will query WHERE status = 'processing'. A standard index on status has low cardinality (3 values) and poor selectivity. A PostgreSQL partial index — index('idx_doc_uploads_processing').on(t.userId).where(sql`status = 'processing'`) — will be tiny and extremely fast, directly improving upload pipeline polling latency as document volume grows.

### Replace portfolioHoldings quantity integer with numeric for fractional share support — 📌 Low Impact
quantity on portfolioHoldings and portfolioTransactions is typed as integer, preventing fractional share simulation (e.g. 0.5 units of a bond). Caribbean school users learning about index funds or ETFs will encounter artificially rounded positions. Changing to numeric('quantity', { precision: 12, scale: 6 }) matches the precision of pricePerUnit and avgPurchasePrice, keeps financial math consistent, and avoids a breaking migration later when fractional investing is added.

### Consolidate overview-tab parallel data fetching with Promise.all — 🔥 High Impact
The overview/dashboard tab likely fires multiple independent useQuery calls (organizations, schools, teachers, students counts) sequentially across component mounts. Wrapping independent fetches in a single server-side summary endpoint or using Promise.all in a single queryFn would halve round-trip latency on slow Caribbean mobile connections, which is the primary user demographic.

### Memoize large inline JSX table-row renderers with useCallback — ⚡ Medium Impact
List panels for students, teachers, and classes almost certainly map over arrays with inline arrow functions creating new references on every parent render. Extracting row renderers into useCallback-wrapped functions (or separate memoized RowComponent children) will prevent full list re-renders when unrelated state like the search input changes, reducing jank on low-end devices common in school settings.

### Add auth-guard early return at component top level — ⚡ Medium Impact
If useAdminAuth is called but its redirect side-effect races with the component rendering (e.g. due to a cached but expired session), child queries with admin-only endpoints may fire before the redirect completes, leaking a 401 error burst. Adding an explicit if (!adminUser) return <Redirect to='/admin/login' /> immediately after the hook call eliminates this race and removes the redundant auth checks that are likely duplicated inside individual panels.

Error handling is generally present but incomplete in a few critical paths.
