## 🔴 Critical (5)

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 103-105)
**Issue:** AI and media generation routes have no rate limiting applied before registration. These are expensive OpenAI calls that can be abused by school-aged users or external actors, causing runaway API costs.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three register calls.

### `server/routes.ts`: robots.txt handler (~line 43) and sitemap handler (~line 56)
**Issue:** Host and proto are read directly from user-controlled request headers (x-forwarded-host, host) without validation and interpolated into response output. A crafted Host header can redirect crawlers or poison sitemap URLs — host header injection.

**Fix:** Read the canonical base URL from a single environment variable: `const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com';` and remove all header-derived base URL logic from both handlers.

---

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler payload)
**Issue:** fileB64 (full base64-encoded PDF or image) is stored directly in the job payload, bloating the jobs table with potentially megabytes of binary data per row. This degrades queue query performance and risks hitting row-size limits.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload. Inside runExtractPaper, download the file from object storage at execution time. Remove fileB64 from JobPayloads['extract-paper'] entirely.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat - sendMessage
**Issue:** The help chat endpoint at /api/admin/help-chat sends the full message history on every request with no truncation. A user could craft a very long conversation to send enormous payloads, and there is no client-side length cap on `text` before appending to `next` and posting. This is a denial-of-service and cost-amplification risk against your LLM billing.

**Fix:** Add a hard cap before posting: `if (text.length > 500) return toast({ title: 'Message too long' });` and truncate the history sent to the last N messages, e.g. `messages: next.slice(-10)`. Also verify the /api/admin/help-chat route enforces admin session server-side before calling the LLM.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level component mount
**Issue:** The dashboard renders its full JSX tree and fires all useQuery calls before the auth check from useAdminAuth resolves. If useAdminAuth returns an async session check, all admin data queries execute unauthenticated for the duration of that check, leaking query traffic and potentially receiving unauthorized data if server auth is misconfigured.

**Fix:** Gate all queries with `enabled: !!adminUser` using the resolved user from useAdminAuth. Add an early return that renders a loading spinner while `isLoadingAuth` is true, before any child components or queries are instantiated.

## 🟡 Warnings (9)

- **`server/routes.ts` (Supabase init chain (~line 108-116)):** Seven async Supabase operations are chained sequentially with .then(), so each waits for the previous to complete. Total startup side-effect time is the sum of all operations. If any middle step fails, downstream seeds never run even if they are independent.
  → *Split into two phases: truly dependent ops first (initSupabaseTables → trimOrgNamesInSupabase), then run independent seeds in parallel: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await Promise.all([seedFinancialAcademyLesson(), initStaticContentTables().then(seedStaticModules), seedRegionalContent(), seedGameContent()]);` wrapped in a single catch.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 91)):** filePath is taken directly from req.params and passed to objectStorage.searchPublicObject with no path sanitisation. A request to /public-objects/../../private/keys could traverse to non-public storage paths depending on the ObjectStorageService implementation.
  → *Sanitise before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });`*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one with sequential awaits inside a for loop. For a 50-question paper this serialises 50 round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Collect all valid question objects into an array, then call a single db.insert(extractedQuestions).values(allValidQuestions) batch insert after the loop. Add a createExtractedQuestions bulk method to storage if needed.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch. If the DELETE fails (e.g. DB connection drop), the unhandled rejection will crash the worker process rather than being recorded as a failed job with retry semantics.
  → *Wrap the entire handler body in try/catch, catch the error, optionally call captureError from sentry, then re-throw so the job framework can mark the job as failed and schedule a retry.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000) and max_completion_tokens: 4096):** Both the 30,000 character truncation limit and the 4,096 token cap are hardcoded magic numbers. Changing them requires a code deploy, and there is no indication in logs which limit was hit, making silent truncation of exam content invisible in production.
  → *Move both values to environment variables: PDF_EXTRACT_CHAR_LIMIT (default 30000) and OPENAI_MAX_COMPLETION_TOKENS (default 4096). Log a warning when the char limit is actually hit: if (pdfData.text.length > limit) logger.warn({ paperId, originalLength: pdfData.text.length }, 'PDF text truncated').*
- **`server/jobHandlers.ts` (openai import – `import { openai } from './replit_integrations/chat/routes'`):** The OpenAI client is imported from a route module, creating a hidden coupling between job infrastructure and HTTP route internals. If that route module is refactored or conditionally loaded, all job handlers silently lose AI capability with no startup-time error.
  → *Extract the OpenAI client instantiation into a dedicated singleton module (e.g. server/openaiClient.ts) that reads OPENAI_API_KEY directly and throws at import time if missing. Both routes and job handlers import from that shared module.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - scroll div ref):** The auto-scroll sentinel uses `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` as an inline callback ref. This creates a new function object on every render of the messages list, and scrollIntoView fires on every render cycle, not just when messages are added. This causes jank and unnecessary DOM thrashing.
  → *Replace with a stable useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple top-level useQuery calls (stats, users, schools, etc.)):** Each admin data section (stats, users, schools, teachers, lessons) appears to have independent useQuery hooks with no `enabled` coordination. On first load this fires 5+ parallel requests before the admin session is confirmed, and any tab-switch interaction refetches all queries simultaneously with default staleTime, creating redundant network load.
  → *Set `staleTime: 5 * 60 * 1000` (5 minutes) on stable reference queries (schools, lessons). Add `enabled: !!adminUser && activeTab === 'users'` style guards so only the active tab's data is fetched. This reduces cold-load requests by ~60%.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort expressions in JSX (user list, lesson list rendering)):** Filtering and sorting of user and lesson arrays appear to be computed inline inside the render return using .filter().sort() chains directly on query data. These re-run on every parent re-render including unrelated state changes like `input` keystroke in the help chat.
  → *Wrap each derived list in useMemo with the appropriate dependencies, e.g. `const filteredUsers = useMemo(() => users?.filter(u => u.name.includes(search)).sort(...), [users, search]);`. This is especially important for large school datasets common in a district-wide deployment.*

## 💡 Opportunities

### Centralise base URL as environment config: 🔥 High Impact
Introduce a single PUBLIC_BASE_URL env var consumed by robots.txt, sitemap, email templates, and any future absolute-URL generation. Eliminates header-injection surface and removes duplicated proto/host assembly logic currently in two route handlers.

### Lazy-load ObjectStorageService: ⚡ Medium Impact
objectStorage is instantiated at module load time (top-level `new ObjectStorageService()`), so any constructor I/O or credential failure crashes the entire module import. Move instantiation inside the route handler or into a lazy singleton so startup failures are isolated and recoverable.

### Add request-scoped timeout middleware for AI routes: ⚡ Medium Impact
OpenAI calls have no server-side timeout enforced at the route layer. A stalled upstream response holds a Node.js socket open indefinitely. Add a per-request AbortController with a 30s timeout passed into registerChatRoutes/registerImageRoutes to bound worst-case connection lifetime and free resources.

### Parallel independent async lookups in multi-step job handlers: 🔥 High Impact
Any job handler that fetches user data, module data, and badge data in separate sequential awaits (visible in truncated code patterns) should wrap independent fetches in Promise.all. Expected impact: 40–60% latency reduction on handlers with 3+ independent DB reads.

### Structured logging for job lifecycle events: ⚡ Medium Impact
Adding structured log entries (job name, jobId, attempt number, duration) at job start and completion would make it possible to set alerts on stuck or slow jobs without full APM tooling. Especially valuable for the extract-paper handler which calls OpenAI twice sequentially.

### Idempotency key on enqueueJob for extract-paper: ⚡ Medium Impact
If a user double-submits an exam paper upload, two extract-paper jobs will run concurrently for the same paperId. The delete-before-insert idempotency guard only helps within a single run. Adding a unique idempotency key (e.g. paperId) to the job queue's enqueue call would prevent duplicate concurrent jobs at the queue level.

### Parallelize independent admin bootstrap queries: 🔥 High Impact
If any initialization logic awaits sequential API calls (e.g. fetch config then fetch schools then fetch stats), refactor to Promise.all or rely on React Query's parallel useQuery hooks with proper enabled flags. Expected impact: reduce time-to-interactive for the dashboard by the sum of sequential latencies, typically 300-800ms on Caribbean mobile connections.

### Extract tab panels into separate lazy-loaded components: 🔥 High Impact
The single large AdminDashboard component likely contains all tab content in one bundle. Splitting Users, Schools, Lessons, and Analytics panels into React.lazy components would reduce initial parse time by an estimated 40-60% and isolate re-render scope to the active tab only.

### Add chat message persistence with sessionStorage: ⚡ Medium Impact
AdminHelpChat resets messages to [] on every open/close toggle via the `open` state. Persisting messages to sessionStorage and rehydrating on open would prevent admins losing context mid-session, a common frustration in multi-step support workflows, with minimal implementation cost.
