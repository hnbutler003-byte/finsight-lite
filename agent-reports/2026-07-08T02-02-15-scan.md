## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query is issued to sum transactions for that budget's category. With 20 budgets this is 21 round-trips to the database.

**Fix:** Replace the per-budget spent lookup with a single aggregated query using GROUP BY categoryId joined to the budgets result set, then merge in JavaScript. E.g., `db.select({ categoryId: transactions.categoryId, spent: sql<number>`sum(amount)` }).from(transactions).where(and(eq(transactions.userId, userId), inArray(transactions.categoryId, categoryIds))).groupBy(transactions.categoryId)` then build a Map<categoryId, spent> and attach in one pass.

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional limit but if the caller omits it there is no default cap applied to the query, meaning the full gameSessions + users join can be returned unbounded. On a school deployment with thousands of sessions this will cause memory and latency spikes.

**Fix:** Add a hard default and maximum: `const safeLimit = Math.min(filters?.limit ?? 50, 200);` and always pass `.limit(safeLimit)` to the Drizzle query regardless of caller input.

### `server/storage.ts`: enrollStudent / createClass
**Issue:** Multi-step writes that must be atomic are executed without a database transaction. For example, createClass generates a join code and inserts the class in separate statements; if the process crashes between steps the row is left in an inconsistent state. Similarly enrollStudent checks for existing enrollment then inserts without a transaction, creating a race-condition duplicate enrollment window.

**Fix:** Wrap each multi-step write in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API so all statements share the same connection and roll back together on failure.

### `server/routes.ts`: /public-objects/:filePath(*) handler (~line 95)
**Issue:** The public object-storage endpoint has no authentication or authorization check. Any unauthenticated caller can enumerate and download arbitrary files by guessing or iterating filePaths. For a school-aged user app this could expose student-related assets or org logos tied to private accounts.

**Fix:** Add a middleware guard that either (a) validates the filePath against an allowlist of truly public prefixes (e.g. only 'logos/' is public) and rejects everything else with 403, or (b) requires a signed URL / session token for non-public prefixes. At minimum add `if (!/^logos\//.test(filePath)) return res.status(403).json({ error: 'Forbidden' });` before the storage call.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 126)
**Issue:** The OpenAI-backed chat, image, and audio route registrations have no rate limiting applied at this layer. The authLimiter is only applied to auth paths. Heavy AI endpoints called by school-aged users with no per-user or per-IP rate limit will result in runaway OpenAI spend if a single client loops requests or if the app is scraped.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 requests per minute per IP using express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three register calls. Pair with per-user tracking if sessions are available.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored in the job payload row in the database. For a typical exam paper PDF or image this can be 1–10 MB per job row, causing significant DB bloat, slow job queue queries, and high memory allocation when the worker deserializes the payload.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download it inside runExtractPaper with objectStorageClient. Delete the temp object after successful extraction. This keeps job rows under 1 KB.

### `server/index.ts`: seedDatabase / probeDatabase / startJobWorker startup block
**Issue:** Startup awaits seedDatabase, probeDatabase, registerRoutes, and startJobWorker sequentially. probeDatabase and seedDatabase are independent of route registration and could overlap, but more critically none of these calls have a timeout guard. A hung DB probe will stall the HTTP server from ever binding, with no watchdog to kill the process.

**Fix:** Wrap the entire async startup IIFE in a Promise.race against a hard timeout (e.g. 30 s via a setTimeout that calls process.exit(1) with an error log). Parallelise independent steps: await Promise.all([probeDatabase(), seedDatabase()]) before registerRoutes so DB work and schema checks run concurrently.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are created with setInterval but their return values are never stored. The SIGTERM / SIGINT graceful-shutdown handler therefore cannot call clearInterval on them. On a rolling deploy the process receives SIGTERM, the shutdown handler closes the HTTP server and DB, but both intervals keep the event loop alive and the process hangs until the container runtime force-kills it.

**Fix:** Store both handles: const digestTimer = setInterval(...); const purgeTimer = setInterval(...); then inside the shutdown handler call clearInterval(digestTimer) and clearInterval(purgeTimer) before closing the HTTP server.

---

### `client/src/pages/AdminDashboard.tsx`: apiRequest calls in mutation handlers (e.g. delete, create teacher/school)
**Issue:** Mutation functions passed to useMutation call `apiRequest` but the response is never checked for non-2xx HTTP status before calling `.json()`. If the server returns a 403 or 500, `res.json()` may still resolve with an error body and the mutation `onSuccess` fires incorrectly, silently swallowing the error and leaving the UI in an inconsistent state.

**Fix:** In each mutationFn, check `if (!res.ok) { const err = await res.json().catch(() => ({})); throw new Error(err.message ?? res.statusText); }` before consuming the body. TanStack Query will then route to `onError` correctly. Centralise this check in `apiRequest` itself so all callers get the guard for free.

## 🟡 Warnings (15)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregations: total income, total expenses, savings goal sum, and transaction list are each awaited one after another. These queries are independent and together add their latencies serially.
  → *Run all independent aggregation queries concurrently: `const [incomeResult, expenseResult, goalsResult, recentTxns] = await Promise.all([incomeQuery, expenseQuery, goalsQuery, txnQuery]);` then combine results. Expected latency reduction is roughly (N-1) * avg_query_time.*
- **`server/storage.ts` (simulateMarketPrices):** Hardcoded price volatility constants and market simulation intervals appear directly in the function body rather than being drawn from environment config. This makes tuning for a school demo environment versus production impossible without a code change, and any interval set here is never cleared, creating a memory/timer leak if the module is re-initialised (e.g., during hot-reload in dev).
  → *Move volatility factor and interval duration to environment variables (MARKET_VOLATILITY_FACTOR, MARKET_TICK_INTERVAL_MS) with safe defaults. Store the return value of setInterval in a module-level variable and export a cleanup function that calls clearInterval, then call it in any shutdown/teardown hook.*
- **`server/routes.ts` (Supabase init chain (~line 130)):** The seven Supabase init/seed functions are chained sequentially with .then(), so each waits for the previous to finish. If any step takes 2-3 seconds the total cold-start delay compounds to 15+ seconds of background work, and a failure in step 2 silently skips steps 3-7 with only a single catch log giving no per-step visibility.
  → *Replace the sequential chain with parallel execution where steps are independent, and add per-step error logging: `Promise.all([initSupabaseTables(), initStaticContentTables()]).then(([]) => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` Adjust grouping based on actual dependency order.*
- **`server/routes.ts` (/healthz handler (~line 82)):** The /healthz route calls `await import('./db')` and `await import('drizzle-orm')` on every request. Dynamic imports are cached by Node after the first call but the async resolution overhead and the pattern misleads future maintainers into thinking a fresh import happens each time. More importantly, there is no timeout on the `db.execute` call, so a hung DB connection will cause the health check to hang indefinitely, breaking Railway's liveness probe.
  → *Move the `db` and `sql` imports to the top of the file (static imports). Add a query timeout: wrap the execute in a `Promise.race([db.execute(sql`select 1`), new Promise((_, reject) => setTimeout(() => reject(new Error('DB timeout')), 3000))])` so the probe always responds within a bounded time.*
- **`server/routes.ts` (robots.txt and sitemap.xml handlers (~lines 46, 64)):** Both handlers derive the base URL from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or sanitization. A request with a forged Host header will produce a robots.txt or sitemap pointing at an attacker-controlled domain, which can be used for SEO poisoning or phishing if the response gets cached by crawlers.
  → *Add a SITE_BASE_URL environment variable (e.g. `https://finsightlite.com`) and use it as the authoritative base. Fall back to header-derived value only in development: `const base = process.env.SITE_BASE_URL || `${proto}://${host}`;` This eliminates the Host header injection vector in production.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop. For a 50-question paper this is 50 serial round-trips to the database, adding latency proportional to network RTT × question count and holding the worker busy far longer than necessary.
  → *Collect all valid question objects into an array, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, batch with Promise.all(validQuestions.map(q => storage.createExtractedQuestion(q))) and add a concurrency limiter (p-limit, limit 5) to avoid overwhelming the connection pool.*
- **`server/jobHandlers.ts` (openai.chat.completions.create calls – model: 'gpt-4o', max_completion_tokens: 4096):** The OpenAI model name and token limits are hardcoded strings. Changing the model for cost or capability reasons requires a code deploy, and different environments (dev/staging/prod) cannot be configured independently.
  → *Read from environment variables with fallbacks: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); Apply the same pattern to the vision call and any other hardcoded model references in the file.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge job has no try/catch. If the DELETE statement fails (e.g. DB timeout, lock contention) the unhandled rejection propagates with no structured error context logged, making it hard to diagnose in production. The result cast via 'unknown' also silently swallows the actual rowCount if the ORM changes its return shape.
  → *Wrap the delete in try/catch and log the error with structured context before rethrowing. Replace the unsafe cast with a typed query: const [{ count }] = await db.select({ count: sql<number>'count(*)' }).from(aiUsageEvents).where(lt(aiUsageEvents.createdAt, cutoff)); run the delete; return { deletedRows: count }. Or use Drizzle's returning() if supported by the dialect.*
- **`server/index.ts` (res.json middleware patch (capturedJsonResponse)):** The logging middleware monkey-patches res.json and stores the full parsed response body in capturedJsonResponse for every request, then JSON.stringify's it again on finish. For large paginated API responses (quiz history, leaderboard) this doubles memory for the response body and re-serialises it on every request. Worse, any response that contains PII or tokens (session data, user emails) is printed to stdout in plaintext.
  → *Limit response logging to non-200 responses or only log body length/type for 2xx. At minimum add a size guard: only capture and log the body when it is a non-2xx status or the serialised length is under a safe threshold (e.g. 512 bytes). Strip or redact known sensitive keys (token, password, secret) before logging.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The Sunday-at-19:00 calculation uses new Date() local server time with no timezone specification. On a UTC-based host (Replit containers run UTC) 'Sunday at 7 pm' fires at 19:00 UTC, but Caribbean schools expect AST (UTC-4), meaning the digest would enqueue at 3 pm local time. The offset is hardcoded nowhere and there is no environment variable to configure the target timezone or hour.
  → *Add an env var DIGEST_TIMEZONE (default 'America/Port_of_Spain') and DIGEST_HOUR (default 19). Use the Intl API or a minimal helper to resolve 'most recent Sunday at DIGEST_HOUR in DIGEST_TIMEZONE' to a UTC timestamp before comparing with Date.now().*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest — unguarded concurrent executions):** Both scheduler functions are async but setInterval does not await them. If a DB or network call inside one invocation takes longer than the interval period (10 minutes for digest check), a second invocation starts concurrently. The DB-level onConflictDoNothing guard protects double-enqueue for the digest, but the purge function has no equivalent guard, so two overlapping purge runs can delete overlapping sets of rows and emit duplicate log lines.
  → *Add a module-level boolean lock (let purgeRunning = false; let digestRunning = false;) at the top of each async scheduler. At the start of the function, return immediately if the flag is true; set it to true on entry and reset it in a finally block on exit.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a new function on every render. React will call the old teardown and new setup on every messages state change, causing redundant scrollIntoView calls and allocation churn.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading])`. Pass `ref={bottomRef}` to the div. This fires exactly when content changes and creates zero closures per render.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function inside AdminHelpChat):** `sendMessage` is recreated on every render and passed as an onClick handler and referenced inside onKeyDown. More critically, if the component unmounts while the fetch is in-flight (user closes chat), `setMessages` and `setLoading` are called on an unmounted component, which will produce React state-update-on-unmounted-component warnings and potential subtle bugs.
  → *Add an AbortController tied to a useEffect cleanup: `const abortRef = useRef<AbortController | null>(null); useEffect(() => () => abortRef.current?.abort(), []);`. Pass `signal: abortRef.current.signal` to apiRequest. In the catch block, guard with `if (err.name === 'AbortError') return;`. Also wrap sendMessage in useCallback with [messages, input, loading] deps.*
- **`client/src/pages/AdminDashboard.tsx` (Main dashboard data filtering and sorting (useMemo usage audit)):** Despite the import of useMemo, large derived datasets (filtered user lists, sorted leaderboard rows, aggregated chart data) appear to be computed inline in the render body based on query results and search/filter state. For a dashboard with hundreds of students and schools these recompute on every keystroke or unrelated state change, blocking the main thread.
  → *Wrap each derived collection in useMemo: e.g. `const filteredUsers = useMemo(() => users?.filter(u => u.name.toLowerCase().includes(search.toLowerCase())), [users, search]);`. Apply the same pattern to chart aggregation and leaderboard sorting. This limits recomputation to when the actual dependencies change.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple independent useQuery calls for dashboard bootstrap data):** The dashboard fires several sequential-in-effect useQuery calls (users, schools, teachers, leaderboard, audit log) each with their own loading state but no coordinated loading UI. The page renders partially populated sections with stale/undefined data while sibling queries are still in-flight, creating a jarring flash of empty tables and potential null-dereference errors in derived computations.
  → *Either (1) consolidate into a single `/api/admin/dashboard-summary` endpoint returning all bootstrap data in one round-trip, or (2) derive a combined `isLoading` flag: `const isLoading = usersQ.isLoading || schoolsQ.isLoading || teachersQ.isLoading;` and render a single skeleton/spinner gate before any data-dependent JSX. Add explicit `isError` checks with user-visible error messages for each query.*

## 🔵 Info / Tech Debt (1)

- **`server/jobHandlers.ts` (runExtractPaper – image branch: buffer.toString('base64')):** The PDF/image buffer decoded from fileB64 is re-encoded to base64 to embed inline in the OpenAI vision API request. This means the payload is base64-decoded once (Buffer.from(fileB64)) and then immediately re-encoded (buffer.toString('base64')), which is a no-op transformation that wastes CPU and doubles peak memory for the buffer. ## 💡 Opportunities

### Centralise pagination defaults: 🔥 High Impact
Introduce a shared constant (e.g., DEFAULT_PAGE_SIZE = 50, MAX_PAGE_SIZE = 200) imported by every list-returning method. Currently each method either hard-codes a limit or omits one entirely, making behaviour inconsistent and risky as data grows in school deployments.

### Extract category join into a reusable query helper: ⚡ Medium Impact
Both getBudgets and getBillReminders independently query and join the categories table. Extracting a `getCategoriesByIds(ids: number[]): Promise<Map<number, Category>>` helper eliminates duplicate code and makes it trivial to batch the lookup, reducing round-trips from 2 duplicated patterns to 1 shared utility.

### Add structured error logging with sanitisation: ⚡ Medium Impact
Caught errors in async storage methods are either swallowed or logged with console.error including raw error objects that may contain query strings with user data. Replacing with a structured logger that strips SQL and PII before output would improve debuggability without exposing sensitive data in logs, which matters given the app's school-aged user base.

### Parallel domain-router registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then awaited sequentially. If any registerXRoutes function does async work (DB calls, schema checks), total registration time is additive. Audit each register function; those that do no async work should be made synchronous. Those that do should be invoked in a second Promise.all where order permits, cutting cold-start time proportionally.

### Structured startup logging with per-phase timing: ⚡ Medium Impact
Currently Supabase init errors surface only as a single console.error. Adding structured logs with phase names and elapsed time (e.g. using a lightweight wrapper) would make Railway deployment logs actionable and allow alerting on slow seed steps without waiting for user-facing failures.

### Centralise host/proto extraction into a reusable helper: 📌 Low Impact
The pattern `req.headers['x-forwarded-proto'] || req.protocol` is duplicated verbatim in both robots.txt and sitemap.xml handlers. Extract to a `getBaseUrl(req)` utility in a shared helpers file. When the SITE_BASE_URL fix (finding #4) is applied, there will be a single place to update, reducing drift risk.

### Idempotent bulk-upsert for extracted questions: ⚡ Medium Impact
Replace the delete-then-insert idempotency pattern with an INSERT … ON CONFLICT DO UPDATE (upsert) keyed on (paperId, questionText hash). This eliminates the window between the DELETE and the first INSERT where a concurrent retry could see zero questions, and removes the need to wipe partial progress on every retry, reducing unnecessary write amplification.

### Stream PDF parsing to avoid full-buffer memory spike: ⚡ Medium Impact
pdf-parse loads the entire PDF buffer into memory synchronously. For large papers this can spike worker RSS by 50–200 MB. Switching to a streaming parser (pdfjs-dist with page-by-page extraction) or enforcing an upload size limit (e.g. 10 MB) checked before enqueuing would cap peak memory and improve worker density on shared infrastructure.

### Centralise OpenAI client import to eliminate cross-module coupling: 📌 Low Impact
Importing openai from a route file (./replit_integrations/chat/routes) couples jobHandlers.ts to HTTP layer internals. If that route module is refactored or lazy-loaded, this import silently breaks. Extract the OpenAI client initialisation into a dedicated ./lib/openaiClient.ts singleton and import from there across all consumers to enforce clean dependency boundaries.

### Parallel Promise.all for independent startup I/O: ⚡ Medium Impact
probeDatabase and seedDatabase do not depend on each other. Running them with Promise.all instead of sequentially would cut startup latency by roughly the duration of the slower of the two operations, which matters for cold-start time on Replit's always-on tier and for health-check probe timeouts during deploys.

### Structured JSON logging instead of console.log string concatenation: ⚡ Medium Impact
Replacing the custom log() helper and the middleware's logLine string with a structured logger (pino is zero-dep and ~5x faster than console.log) would make log lines machine-parseable for any future log aggregator, eliminate the PII risk from full response body serialisation, and reduce per-request logging overhead under load.

### Extract scheduler registration into a dedicated module with lifecycle hooks: ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, and their setInterval wiring into a schedulers.ts module that exports start() and stop() functions would make the timers testable in isolation, make the shutdown handler trivially correct (call schedulers.stop()), and remove ~80 lines of scheduling logic from index.ts, which should only own server wiring.

### Paginate queries server-side instead of client-side slicing: 🔥 High Impact
Current pagination slices a fully-fetched array client-side (e.g. `users.slice(page*10, page*10+10)`). For a growing Caribbean school network this means downloading all records on every mount. Move limit/offset to the API and use TanStack Query's `keepPreviousData` option for smooth page transitions. Expected impact: 60-80% reduction in initial payload for large deployments.

### Extract tab panels into separate lazy-loaded route components: 🔥 High Impact
All dashboard sections (Users, Schools, Teachers, Leaderboard, Audit Log, AI Chat) appear to live in one 1000+ line component, loading all Recharts, VideoField, and dialog logic upfront. Code-split each panel with React.lazy/Suspense. This reduces the initial JS bundle by an estimated 30-40% and improves TTI for admin users on lower-bandwidth Caribbean connections.

### Add rate-limiting and input sanitisation to /api/admin/help-chat: ⚡ Medium Impact
The help-chat endpoint receives free-form user messages forwarded directly to an LLM. Without per-session rate limiting and a maximum message-history length cap, a malicious admin account could trigger runaway LLM API costs or attempt prompt injection. Enforce a server-side limit (e.g. 20 req/min per session, max 20 messages in context) and strip any system-role injection patterns before forwarding to the model.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts is a large monolithic storage layer with several reliability and performance gaps. The most critical issues are N+1 query patterns in budget/bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits where Promise.all would reduce latency. Security posture is adequate at the storage layer but hardcoded fallback values and missing pagination guards add risk at scale.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good use of Promise.all for parallel imports and proper error handling in most async paths. Key concerns are: the Supabase init chain runs sequentially when steps could parallelize, the public object-storage endpoint lacks authentication allowing any caller to enumerate and download assets, and the /healthz route performs a dynamic import on every request which is wasteful and slightly blocking.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads stored in the job table create DB bloat and memory pressure, sequential DB inserts in a loop should be batched, and hardcoded model identifiers/token limits should be environment-configurable. Security is acceptable but a few async paths lack error boundaries.

### Server Startup (`server/index.ts`)
The server startup sequence contains blocking synchronous-style serial awaits that delay readiness, intervals that are registered without cleanup references making graceful shutdown incomplete, and the response-logging middleware unconditionally serialises every JSON response body which is a memory and security risk in production. Core reliability scaffolding (job worker, digest scheduler, purge scheduler) is present but the shutdown handler does not cancel the two setInterval timers, leaving the process unable to exit cleanly.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx shows several performance and reliability gaps: expensive list filtering/sorting runs without memoization on every render, async operations lack proper error boundaries, and a scroll-into-view ref callback creates a new function on every render cycle. The chat component is functional but has subtle memory and UX concerns. Security posture is partially addressed by useAdminAuth but warrants scrutiny on data exposure.
