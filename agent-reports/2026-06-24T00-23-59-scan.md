## 🔴 Critical (4)

### `server/routes.ts`: 98-102 (registerChatRoutes, registerImageRoutes, registerAudioRoutes)
**Issue:** registerChatRoutes, registerImageRoutes, and registerAudioRoutes are called without await and without any rate limiting middleware applied before them. These functions register OpenAI/AI endpoints that are expensive and prone to abuse, especially critical for a school-aged user base where prompt injection or API cost exhaustion is a real risk.

**Fix:** Apply a dedicated aiLimiter (e.g., 20 req/min per IP) via app.use('/api/chat', aiLimiter) and app.use('/api/image', aiLimiter) before calling the register functions. If these functions return Promises, add await. Check each registration function's signature and wrap in try/catch or add .catch() if they can reject.

### `server/routes.ts`: 86-93 (/public-objects/:filePath(*) handler)
**Issue:** The /public-objects/:filePath(*) endpoint has no authentication check and no path-traversal sanitization. A wildcard filePath param passed directly to objectStorage.searchPublicObject() could allow unauthenticated access to any stored object, including potentially sensitive org logos or user-uploaded content, and may allow path traversal depending on ObjectStorageService implementation.

**Fix:** Add path sanitization: const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); and reject requests where filePath includes '..'. For non-public assets, add an auth middleware guard. Consider restricting the allowed prefix to a known-safe list (e.g., only 'logos/', 'videos/').

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** ticker column has no unique constraint. Duplicate tickers would silently corrupt portfolio lookups and any JOIN between portfolioHoldings and simulatedStocks that resolves via ticker string.

**Fix:** Replace the plain text() column with: ticker: text('ticker').notNull(), and add a uniqueIndex in the table options: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_stocks_ticker_unique').on(t.ticker) }).

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage at top of AdminDashboard
**Issue:** If useAdminAuth only checks client-side state (JWT in localStorage/memory) and the dashboard renders data-fetching useQuery hooks unconditionally before auth is confirmed, unauthenticated users or users with expired tokens will still fire admin API calls. A brief render before redirect leaks admin endpoint paths and triggers 401 noise.

**Fix:** Gate all useQuery hooks behind the confirmed auth state: check `const { isAuthenticated, isLoading } = useAdminAuth()` and return a loading spinner early if `isLoading`, or redirect immediately if `!isAuthenticated`. Move all useQuery declarations below that guard or use the `enabled: isAuthenticated` option on every query.

## 🟡 Warnings (10)

- **`server/routes.ts` (109-116 (Supabase init chain)):** Seven Supabase seed/init functions are chained sequentially with .then(), meaning each waits for the previous to complete. Total cold-start delay is the sum of all operations. Also, if seedFinancialAcademyLesson() succeeds but initStaticContentTables() was already seeded, failure aborts the remaining seeds silently beyond the single .catch().
  → *Group independent operations with Promise.all: run initSupabaseTables and initStaticContentTables in parallel, then their dependent seeds in parallel: Promise.all([initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()])), initStaticContentTables().then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]))]).catch(e => console.error('[Supabase] Init error:', e));*
- **`server/routes.ts` (57-67 (/healthz endpoint)):** The /healthz endpoint uses dynamic import() inside a request handler on every call, which is resolved from cache after the first call but still incurs module cache lookup overhead and obscures the import graph. More critically, it exposes a database-probing endpoint with no rate limiting, enabling an attacker to hammer the DB probe or fingerprint DB availability.
  → *Move the db and sql imports to the top of the file as static imports. Add the existing authLimiter or a dedicated health limiter: app.use('/healthz', basicLimiter) to prevent abuse. The /api/health duplicate (line 35) should be removed or consolidated.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets.userId even though every budget query will filter by user. At scale this causes a full table scan per user dashboard load.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budgets per category per period.*
- **`shared/schema.ts` (transactions table — createdAt / portfolioTransactions — executedAt / documentUploads — createdAt):** Multiple audit timestamp columns use .defaultNow() without .notNull(). This allows NULL to be explicitly inserted, breaking ORDER BY createdAt queries and audit trails silently.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply consistently to createdAt on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (billReminders and savingsGoals table definitions):** No index on userId for billReminders or savingsGoals. The due-date reminder job and goal progress screen both query by userId alone, causing full table scans as user count grows.
  → *Add table-level indexes: for billReminders: (t) => ({ userDueIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }); for savingsGoals: (t) => ({ userIdx: index('idx_savings_goals_user').on(t.userId) }).*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is a bare varchar() with no length specified and no index, yet it is used to distinguish system vs user-owned categories. Without an index, filtering user categories requires a full scan of the categories table on every transaction form load.
  → *Set an explicit length matching users.id (e.g. varchar('user_id', { length: 36 })) and add an index: (t) => ({ userIdx: index('idx_categories_user').on(t.userId) }). This also aligns the column type with the FK target to avoid implicit casts in the query planner.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` creates a new function on every render and calls scrollIntoView on every re-render (including while the user is typing), not just when a new message arrives. This causes jarring scroll-jacking mid-input.
  → *Replace with a stable useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function in AdminHelpChat):** The POST to `/api/admin/help-chat` sends the full growing messages array on every turn with no size cap. For a school-aged platform with long sessions this will eventually exceed server payload limits and leak the entire conversation history in each request body, which may surface in server logs.
  → *Slice the context window before sending: `const context = next.slice(-10);` and send `{ messages: context }` instead of the full array. Add a max message count guard (e.g., 100 messages) to prevent unbounded state growth: `if (messages.length > 100) setMessages(prev => prev.slice(-50));`*
- **`client/src/pages/AdminDashboard.tsx` (Multiple independent useQuery calls for dashboard bootstrap data):** Separate useQuery calls for users, schools, teachers, lessons, and analytics each fire sequentially during component mount with no coordination, causing a waterfall of 5+ round-trips that all need to resolve before the dashboard is useful. This directly degrades perceived performance on Caribbean mobile networks.
  → *Consolidate into a single `/api/admin/dashboard-summary` endpoint that returns all bootstrap data in one response, or use `Promise.all` inside a single custom query function: `queryFn: () => Promise.all([fetchUsers(), fetchSchools(), fetchLessons()]).then(([users, schools, lessons]) => ({ users, schools, lessons }))`. This reduces latency by the number of sequential round-trips minus one.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort functions passed directly in JSX (search/filter handlers)):** Filtering and sorting of user/lesson lists is performed with inline arrow functions referencing search state, causing the full filter pass to re-execute on every parent render regardless of whether the relevant state changed. With hundreds of school users this is a measurable CPU cost on low-end devices.
  → *Wrap derived filtered lists in useMemo: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(search.toLowerCase())), [users, search]);`. Apply the same pattern to every sorted/filtered dataset. Remove the inline expressions from JSX entirely.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six registerXxxRoutes() calls after the Promise.all import are still invoked sequentially with individual awaits. If any of these internally do async setup (DB checks, cache warming), they could be parallelized with Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]). Expected impact: reduced server startup time proportional to the slowest sequential init.

### Centralized request logging with sensitive-field redaction: 🔥 High Impact
There is no request logging middleware visible in this file. Adding a structured logger (e.g., pino-http) with explicit redaction of Authorization headers, session cookies, and any PII fields in request bodies would improve observability and ensure compliance with COPPA/FERPA requirements relevant to school-aged users. Impact: faster debugging, reduced accidental PII exposure in logs.

### robots.txt and sitemap.xml host header injection hardening: ⚡ Medium Impact
Both /robots.txt and /sitemap.xml construct absolute URLs from req.headers['x-forwarded-host'] and req.headers.host without validation. A malicious request with a crafted Host header could inject arbitrary URLs into the sitemap or robots Sitemap directive. Validate the host against an allowlist from APP_BASE_URL env var and fall back to that value if the header doesn't match.

### Centralise default currency as an enum or config constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default across at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a shared DEFAULT_CURRENCY constant and a pgEnum so changing the default or adding multi-currency support requires a single change, not a migration per table.

### Add a composite unique index on portfolioHoldings (userId, stockId): 🔥 High Impact
Without this constraint, concurrent buy orders can create duplicate holding rows for the same user and stock, forcing the application layer to aggregate them manually and making position calculations error-prone. A unique index enforces the invariant at the DB level and enables ON CONFLICT DO UPDATE upsert semantics.

### Replace serial PKs with uuid on user-facing tables: ⚡ Medium Impact
Using serial (sequential integer) PKs on transactions, savingsGoals, and documentUploads exposes row counts and insertion rates via enumerable IDs in API responses — a security concern for a school-aged user app. Switching to gen_random_uuid() as the default also simplifies future sharding and eliminates the join-time type mismatch between integer FKs and varchar user IDs.

### Skeleton loading states for dashboard panels: 🔥 High Impact
Currently, dashboard cards likely render empty or partially populated while queries are in-flight. Adding Skeleton placeholder components (matching the card grid layout) during `isLoading` states will eliminate layout shift and communicate progress to admin users, which is especially important on slower Caribbean connections.

### Debounced search input to cut redundant filter work: ⚡ Medium Impact
Filtering large user or lesson lists on every keystroke is wasteful. Introducing a 250ms debounce on the search input (via a custom useDebounce hook) before updating the filter state would reduce useMemo re-computations by ~80% during typical typing, directly reducing jank on low-spec school devices.

### Extract AdminHelpChat into a lazy-loaded chunk: ⚡ Medium Impact
AdminHelpChat imports Lucide icons, maintains its own state, and is only shown on demand. Wrapping it in React.lazy/Suspense removes it from the main AdminDashboard bundle, reducing initial parse time which matters on school tablets with limited CPU and memory.
