## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query is issued to sum transactions for 'spent' calculation. With 10 budgets this fires 11 queries; with 50 it fires 51.

**Fix:** Replace the per-budget transaction sum with a single GROUP BY query joining budgets to transactions: SELECT budget_id, SUM(amount) as spent FROM transactions WHERE user_id = ? AND budget_id IN (...) GROUP BY budget_id. Then merge results in JS with a Map lookup in O(1) per budget.

### `server/storage.ts`: enrollStudent / createClass
**Issue:** Multi-step writes (e.g. creating a class then inserting the teacher-class link, or enrolling a student and updating counters) are not wrapped in a database transaction. A crash between steps leaves the database in a partial/corrupt state.

**Fix:** Wrap all multi-step writes in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Every INSERT/UPDATE that must succeed atomically must share the same tx handle so a failure rolls back all steps.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app) — lines 110-112
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These call OpenAI APIs which have per-token costs and latency. A school-aged user or a script could hammer these endpoints causing runaway spend or denial of service for other users.

**Fix:** Add a dedicated aiLimiter (e.g. 20 req/15min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Confirm the exact path prefixes used inside those register functions first.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler — lines 40 and 58
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header, causing the Sitemap directive and sitemap <loc> values to point to an arbitrary domain — a classic Host Header Injection that can poison caches or be used in phishing.

**Fix:** Add a validated APP_BASE_URL environment variable (e.g. 'https://finsightlite.com') and use that as the base instead of deriving it from request headers. Fall back to header-derived value only in local dev with an explicit NODE_ENV guard. Remove the dynamic host derivation in production entirely.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter / extract-paper handler)
**Issue:** The full base64-encoded file is stored in the job payload in the jobs table. For a multi-page PDF or high-res image this can easily be 5-20 MB per row, bloating the jobs table, slowing queue queries, and risking exposure of exam content in DB dumps or logs.

**Fix:** Upload the file to object storage before enqueuing the job, store only the storage path in the payload (e.g., { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by setInterval are never stored or cleared. The SIGTERM/SIGINT graceful-shutdown handler closes the HTTP server and DB but leaves both intervals running, meaning in-flight scheduler ticks can fire after the DB connection is torn down, causing unhandled promise rejections and potential duplicate job enqueues during rolling restarts.

**Fix:** Capture both handles: const digestInterval = setInterval(...); const purgeInterval = setInterval(...); then inside the shutdown handler call clearInterval(digestInterval) and clearInterval(purgeInterval) before closing the HTTP server. Also await a short drain (e.g., 500 ms) to let any in-progress tick finish before closing the DB pool.

---

### `server/index.ts`: res.json middleware / logLine construction (~line 68-78)
**Issue:** Every API response body is captured via res.json monkey-patching and serialized in full into the application log (JSON.stringify(capturedJsonResponse)). For a financial literacy app with school-aged users this will log account balances, quiz scores, session tokens, or any PII returned by the API, violating reasonable data-minimisation expectations and potentially FERPA/COPPA obligations.

**Fix:** Either remove response body logging entirely in production, or allowlist safe fields. At minimum gate it: if (process.env.NODE_ENV !== 'production' && capturedJsonResponse) { logLine += ' :: ' + JSON.stringify(capturedJsonResponse); }. For production, log only the status code and duration, never the body.

### `client/src/pages/AdminDashboard.tsx`: AdminDashboard – top of component render
**Issue:** useAdminAuth is called but the component continues to render full admin UI — queries, mutations, charts — before confirming the auth check has resolved. If the hook returns isLoading briefly before redirecting, sensitive admin data is fetched and partially rendered for an unauthenticated user.

**Fix:** Add an early return immediately after the auth hook: `const { admin, isLoading } = useAdminAuth(); if (isLoading) return <FullPageSpinner />; if (!admin) return null;` This ensures all useQuery calls below only execute for confirmed admins, and aligns with the hook's redirect logic.

## 🟡 Warnings (14)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits are used to fetch income totals, expense totals, recent transactions, and category breakdowns one after another. Each is an independent query with no data dependency on the previous result.
  → *Replace sequential awaits with Promise.all: const [income, expenses, recent, breakdown] = await Promise.all([queryIncome, queryExpenses, queryRecent, queryBreakdown]). Expected latency reduction of 60-75% on typical DB round-trip times.*
- **`server/storage.ts` (getLeaderboard):** No pagination guard on the leaderboard query. The 'limit' filter is optional and appears to have no enforced maximum, meaning a caller can omit it and receive every game session row joined to users — unbounded result set in a school context with many students.
  → *Add a hard cap: const safeLimit = Math.min(filters?.limit ?? 25, 100). Pass safeLimit to the query's .limit() clause. Never allow limit to be undefined or caller-controlled beyond this cap.*
- **`server/storage.ts` (simulateMarketPrices):** Market price simulation appears to run on a setInterval or similar timer with no stored reference, making it impossible to clear the interval on server shutdown or test teardown, causing a memory/handle leak.
  → *Store the interval reference: export const marketSimInterval = setInterval(simulateMarketPrices, INTERVAL_MS). Call clearInterval(marketSimInterval) in the server shutdown handler (process.on('SIGTERM', ...)) and in test afterAll hooks. Move INTERVAL_MS to an environment variable with a sensible default.*
- **`server/routes.ts` (Supabase init chain — lines 115-123):** The sequential .then() chain means a failure in seedFinancialAcademyLesson() will skip initStaticContentTables(), seedStaticModules(), seedRegionalContent(), and seedGameContent() silently — the .catch only logs the first error and subsequent seeds never run. This can leave the DB in a partially seeded state with no per-step visibility.
  → *Run all seed functions with individual try/catch blocks or use Promise.allSettled to let each step run independently and log per-step success/failure: Promise.allSettled([initSupabaseTables(), trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] Step', i, 'failed:', r.reason)));*
- **`server/routes.ts` (/healthz handler — lines 77-86):** The /healthz endpoint dynamically imports ./db and drizzle-orm on every request using await import() inside the handler. Dynamic imports are cached after first load, but on the very first cold call this is unnecessarily slow and adds latency to an endpoint that Railway and uptime monitors hit frequently. More importantly, the endpoint is unauthenticated and reveals DB availability state publicly.
  → *Move the db and sql imports to the top of the file as static imports. Optionally gate /healthz behind an internal secret header (e.g. X-Health-Token matching a HEALTH_SECRET env var) so external actors cannot probe DB availability.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 60-question paper this means 60 sequential round-trips to the database, adding hundreds of milliseconds of latency and holding the job worker busy unnecessarily.
  → *Collect valid questions into an array and issue a single batch insert: await db.insert(extractedQuestions).values(validQuestions) where validQuestions is built from the filtered questions array with order assigned inline. Wrap in a try/catch so a constraint error doesn't leave the paper in 'processing' state.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The db.delete call has no try/catch. If the database is unavailable or the delete times out the job throws an unhandled rejection that bypasses any structured error capture, and the job framework may retry a destructive delete operation without idempotency guarantees.
  → *Wrap the delete in try/catch, capture the error with the existing sentry captureError import, and re-throw. Add a WHERE id NOT IN (SELECT id … LIMIT N) guard or use a cursor-based delete if aiUsageEvents can be large, to avoid a single massive delete transaction.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model: 'gpt-4o', max_completion_tokens: 4096)):** The OpenAI model name and token limits are hardcoded strings. If the deployment needs to switch models (e.g., for cost or compliance reasons in a Caribbean school context) every reference must be hunted down manually, and there is no way to override them per environment without a code change.
  → *Read from environment config: const AI_MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o'; const AI_MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10); and use these constants in every completions.create call. Add these keys to the documented .env.example.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — file.createReadStream().pipe(res)):** The read stream is piped to the response but there is no 'close' or 'finish' handler to destroy the stream if the client disconnects mid-download. On a low-bandwidth Caribbean mobile connection this is common, leaving the GCS/object-storage read stream open and consuming memory until the server process notices the broken pipe via the error handler.
  → *Use the 'pipeline' utility from node:stream/promises or attach a res.on('close', () => readStream.destroy()) listener before calling .pipe(res). Store the stream in a local variable so the close handler has a reference: const readStream = file.createReadStream(); res.on('close', () => readStream.destroy()); readStream.on('error', ...).pipe(res);*
- **`server/index.ts` (seedDatabase / probeDatabase / startJobWorker calls in async main()):** seedDatabase and probeDatabase are awaited sequentially before the HTTP server starts listening. If either takes more than a few seconds (e.g., cold DB connection), the process appears hung with no health-check endpoint available, causing container orchestrators to restart the pod unnecessarily.
  → *Start listening immediately after registerRoutes(), then run probeDatabase and seedDatabase in parallel with Promise.all in the background. Return a 503 from a /healthz route until the probe succeeds. Example: server.listen(port, host, () => log('listening')); Promise.all([probeDatabase(), seedDatabase()]).then(() => appReady = true).catch(err => { captureError(err); process.exit(1); });*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and maybeRunAiUsagePurge):** The purge threshold is computed once at module load from AI_PURGE_OLDER_THAN_DAYS, which is good, but the scheduler fires on a fixed interval (likely every hour based on typical patterns in the truncated code) with no jitter. In a multi-replica deployment all replicas will attempt the atomic insert claim simultaneously on the same clock tick, creating a thundering-herd on the weeklyDigestRuns/aiUsagePurgeRuns table every interval boundary.
  → *Add startup jitter: setTimeout(() => { maybeRunAiUsagePurge(); purgeInterval = setInterval(maybeRunAiUsagePurge, INTERVAL_MS); }, Math.random() * INTERVAL_MS); This spreads replica attempts across the full interval window while preserving the atomic-claim correctness already in place.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` creates a new function on every render, causing scrollIntoView to fire on every state update in the component — including while typing — not just when new messages arrive.
  → *Replace with a useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then use `ref={bottomRef}` on the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage):** The full `messages` array is sent to `/api/admin/help-chat` on every turn with no length or token cap. A user leaving the chat open for an extended session will send an unbounded payload, and there is no server-side truncation guard visible in this file.
  → *Slice the history before sending: `const history = next.slice(-20);` and pass `history` instead of `next` to apiRequest. Document the 20-turn limit and enforce a matching cap server-side.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages.map key):** Chat messages use array index `key={i}` as the React reconciliation key. If a message is inserted, removed, or the array is updated mid-stream, React will diff against the wrong nodes, causing incorrect content to appear in the wrong bubbles.
  → *Assign a stable unique id when each message is created: `const next: ChatMessage[] = [...messages, { id: crypto.randomUUID(), role: 'user', content: text }]` and update the ChatMessage type accordingly. Use `key={m.id}` in the map.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (Duplicate health check routes — lines 31 and 77):** There are two health check routes: GET /api/health (line 31, no DB check) and GET /healthz (line 77, with DB check). Having two overlapping health endpoints with different response contracts creates confusion for infrastructure config and wastes maintenance surface. Railway and uptime monitors should point to one canonical endpoint.
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes (dow + 0) % 7 which is a no-op addition of zero, making daysSince equal to the raw getDay() value (0=Sunday). This means on Sunday before 7 pm the function returns today's date but with 19:00 set, and the guard 'if (d.getTime() > now.getTime()) return' correctly skips it. However, on Saturday (dow=6) daysSince=6, rolling back 6 days lands on Sunday of the current week, which is correct, but the intent-obscuring dead expression '(dow + 0)' suggests a copy-paste error where a constant offset (e.g., to shift the scheduled day) was removed incompletely, making future maintenance error-prone.
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage catch block):** The catch block silently swallows all errors with a generic user-facing string. Network errors, 401s from an expired admin session, and 500s from the AI provider are all treated identically, making debugging in production impossible.

## 💡 Opportunities

### Centralised query error boundary: 🔥 High Impact
No consistent try/catch or error-mapping layer wraps the async DB calls. Adding a thin wrapper (e.g. a withDb helper that catches Drizzle/pg errors and rethrows typed AppError instances) would prevent raw Postgres error messages — which can expose table names and column details — from bubbling to API responses. Eliminates a class of accidental data leakage relevant to school-age user data.

### Extract hardcoded seed/config values to environment variables: ⚡ Medium Impact
Values like default virtual balance amounts, stock seed prices, and XP thresholds appear to be hardcoded literals. Moving these to environment config (process.env with zod validation at startup) allows tuning per deployment (e.g. different Caribbean island currencies or school grade levels) without a code change or redeploy risk.

### Batch category hydration for bill reminders: ⚡ Medium Impact
getBillReminders hydrates a category for each reminder individually, mirroring the same N+1 pattern as getBudgets. Collecting all distinct categoryIds from the reminder rows and issuing one inArray query, then joining in memory, would reduce query count from N+1 to 2 regardless of reminder list size.

### Parallel domain-route registration: ⚡ Medium Impact
The six registerXxxRoutes calls after the parallel import are still awaited sequentially. If any of these do async setup work (DB migrations, schema checks), they add to startup time. Wrap them in Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) assuming they do not depend on each other's side effects, cutting cold-start time proportionally.

### Centralised request logging with redaction: 🔥 High Impact
No request logging middleware is visible in this file. Adding a logger (e.g. pino-http) before route registration with a redact config for Authorization headers and cookie values would provide audit trails critical for a financial app serving minors, while preventing accidental credential leakage in logs.

### Object storage client singleton: 📌 Low Impact
ObjectStorageService is instantiated at module scope as a top-level const. If the constructor opens connections or allocates resources, verify it is safe to share across all requests and that it handles reconnection. Consider exporting a singleton from object_storage.ts itself to prevent accidental multiple instantiations if this module is imported elsewhere.

### Parallel independent awaits in multi-step job handlers: ⚡ Medium Impact
Several handlers (visible in the truncated section based on typical patterns: fetching user, module, and teacher records) likely use sequential awaits for independent lookups. Wrapping independent fetches in Promise.all would reduce job latency by the sum of the slowest-minus-fastest query times, meaningful for report-generation jobs that query 3-5 tables.

### Centralised job payload size guard at enqueue time: 🔥 High Impact
Add a payload size check inside the enqueueJob function (e.g., reject if JSON.stringify(payload).length > 64 KB) to catch oversized payloads before they reach the DB. This prevents a class of accidental large-payload bugs like the fileB64 issue without requiring every call-site to remember the constraint.

### Structured logging instead of implicit throw for extraction failures: ⚡ Medium Impact
Currently extraction errors are thrown and caught at the handler boundary with no structured context. Emitting a structured log event (paperId, attempt number, error code, character count of extractedText) before re-throwing would allow a dashboard or alert to surface chronic failures by subject or file type, useful for monitoring teacher uploads in a school deployment.

### Parallel startup I/O with readiness flag: 🔥 High Impact
probeDatabase(), seedDatabase(), registerJobHandlers(), and startJobWorker() are all independent and can run concurrently with Promise.all. On a cold Replit container with a remote Postgres instance this could cut startup latency by 2-4 seconds, making the app available to students faster and reducing orchestrator timeout risk.

### Structured JSON logging: ⚡ Medium Impact
The current log() function emits freeform strings. Replacing it with a structured logger (e.g., pino) emitting JSON with fields like {level, source, method, path, statusCode, durationMs} would make logs machine-parseable for alerting, eliminate the PII-in-body risk by design, and reduce string concatenation overhead on every request.

### Extract scheduler tick logic into testable pure functions: ⚡ Medium Impact
getMostRecentSundayAt7pm and the claim-insert-enqueue pattern inside maybeRunWeeklyDigest are currently untestable without a live DB. Extracting the schedule-decision logic into pure functions and the DB interaction behind an injectable interface would allow unit tests to cover edge cases (DST transitions, leap years, replica races) without spinning up Postgres.

### Memoize filtered/sorted admin table data: 🔥 High Impact
Any search filtering or sorting applied to user/school lists inside the component should be wrapped in useMemo keyed on the raw query data and filter state. Without this, every keystroke in a search box re-runs the full filter over potentially thousands of rows on the main thread, blocking input responsiveness.

### Parallelize independent admin bootstrap queries: 🔥 High Impact
If the dashboard fires multiple useQuery calls for users, schools, and analytics stats sequentially or in dependent chains, refactor independent ones to execute concurrently using Promise.all inside a single custom hook. This can cut initial dashboard load time by 40-60% on Caribbean mobile connections where latency per round-trip is high.

### Extract AdminHelpChat into its own file with lazy import: ⚡ Medium Impact
AdminHelpChat along with its Recharts and Lucide imports contributes to the main AdminDashboard bundle. Moving it to a separate file and importing it with React.lazy + Suspense defers its JS parse cost until the chat is first opened, improving initial TTI for admins who never use the chat feature.
