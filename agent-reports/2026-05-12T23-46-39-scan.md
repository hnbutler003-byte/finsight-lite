## 🔴 Critical (6)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: the function fetches all holdings then issues a separate db query per holding to resolve the associated SimulatedStock, producing O(n) round-trips to the database.

**Fix:** Replace the per-holding lookup with a single JOIN: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. This collapses n+1 queries into one.

---

### `server/storage.ts` — updateVirtualBalance / createPortfolioTransaction / upsertPortfolioHolding
**Issue:** Buy/sell trade flow calls updateVirtualBalance, createPortfolioTransaction, and upsertPortfolioHolding as three separate awaits with no wrapping database transaction. A failure after balance deduction but before the holding is recorded leaves the user's account in an inconsistent state — money gone, no shares credited.

**Fix:** Wrap all three operations in `await db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the `tx` client into each sub-call so they share the same atomic unit and roll back together on error.

---

### `server/routes.ts` — /api/auth/google handler
**Issue:** No rate limiting on the Google auth endpoint. A bot can hammer this endpoint to exhaust Google token verification quota, trigger repeated DB writes for user creation, and enumerate valid email domains via timing differences — especially dangerous for a school-aged user base.

**Fix:** Add express-rate-limit before this handler: `app.post('/api/auth/google', rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.ip }), async (req, res) => {...})`. Use a stricter limit (5 req/min) for any endpoint that touches the AI quota system.

---

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Large base64 file content (fileB64) is stored directly in the job payload. For a Caribbean school context with scanned exam PDFs this could easily be 5–20 MB per job row, bloating the jobs table and causing full-table-scan slowdowns as history accumulates.

**Fix:** Upload the file to private object storage before enqueuing the job, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), and have runExtractPaper download it with objectStorageClient at runtime. This keeps job rows under 1 KB and makes retries cheaper.

---

### `shared/schema.ts` — budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Any dashboard query filtering budgets by user (the most common access pattern) will do a full sequential scan as the table grows.

**Fix:** Add a compound index in the budgets table definition: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This covers both the common 'get all budgets for user' query and the 'get budget for user+category' lookup used in spend-vs-budget comparisons.

---

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel — queryFn
**Issue:** apiRequest(...).then(r => r.json()) has no try/catch. If the server returns a non-JSON body (e.g., a 401 HTML error page) r.json() will throw an unhandled rejection that surfaces as an ugly uncaught error rather than a graceful UI error state.

**Fix:** Wrap the queryFn in an async function with try/catch, or add a response status check before calling .json(): `const res = await apiRequest('GET', url); if (!res.ok) throw new Error(await res.text()); return res.json();`. Do this consistently across all queryFn definitions in the file.

## 🟡 Warnings (18)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregates (total income, total expenses, category breakdowns, recent transactions) execute serially, multiplying latency by the number of queries. On a slow Caribbean mobile connection this is user-visible.
  → *Wrap all independent db calls in a single `Promise.all([...])` destructure. E.g. `const [income, expenses, categoryTotals, recent] = await Promise.all([incomeQuery, expenseQuery, categoryQuery, recentQuery])`. Each query is independent so parallelism is safe.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard accepts an optional `limit` parameter but there is no default cap and no guard preventing a caller from omitting it entirely, which causes a full table scan across gameSessions and users. On a school with hundreds of students this returns unbounded rows.
  → *Apply a hard server-side cap: `const safeLimit = Math.min(filters?.limit ?? 50, 100)` and always pass `.limit(safeLimit)` to the query. Never trust the caller to set a reasonable limit.*
- **`server/storage.ts` (getAllQuestions):** getAllQuestions fetches every ExtractedQuestion (plus joined ExamPaper) with no pagination, no row limit, and no index hint on the subject filter. A single call from a student game session can return thousands of rows, bloating server memory and response payloads.
  → *Add mandatory `limit` and `offset` parameters (default limit 100, max 200). Apply `.limit(safeLimit).offset(offset)` in the query and return a `{ data, total }` envelope so clients can paginate. Also ensure a DB index exists on `extractedQuestions.subject`.*
- **`server/routes.ts` (/api/auth/google handler — resolvedClass resolution block):** Three sequential awaits run one-after-another: getClassByCode → getOrgEnvironmentById → getOrganization. Each is a separate DB/Supabase round-trip and they are not data-dependent on each other's intermediate results until the domain check. This adds 2–3x latency on every Google login under load.
  → *Restructure so environment and org lookups are parallelised once you have the classCode: `const [resolvedClass, resolvedOrgEnv] = await Promise.all([storage.getClassByCode(code), getOrgEnvironmentByJoinCode(code)])`. Then do the single dependent lookup (getOrganization) only if needed. This cuts round-trips from 3 serial to 1+1.*
- **`server/routes.ts` (healthz handler — dynamic imports inside request path):** `import('./db')` and `import('drizzle-orm')` are called on every GET /healthz request. While Node caches modules after first load, the async dynamic-import machinery still runs on every call and adds unnecessary overhead to what should be a near-zero-cost liveness probe.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The healthz handler becomes a simple `await db.execute(sql\`select 1\`)` with no dynamic resolution overhead.*
- **`server/routes.ts` (catch blocks throughout async route handlers):** Error objects are passed directly to res.json (e.g. `{ error: e?.message }`) without sanitisation. For Supabase/Postgres errors, e.message can include table names, column names, constraint names, or partial query text — all useful to an attacker probing a children's platform.
  → *Create a central error-response helper: `function safeError(e: unknown, fallback = 'An error occurred') { const msg = e instanceof Error ? e.message : fallback; console.error(e); return process.env.NODE_ENV === 'production' ? fallback : msg; }`. Use it in every catch block instead of `e?.message` directly. Log the full error server-side with a correlation ID.*
- **`server/routes.ts` (objectStorage / multer setup — top-level module scope):** `new ObjectStorageService()` is instantiated at module load time at the top level, outside registerRoutes. If the underlying storage client constructor throws (misconfigured credentials, missing env var), it will crash the entire server process before any error boundary or graceful-startup logic can run.
  → *Move instantiation inside registerRoutes and wrap it: `let objectStorage: ObjectStorageService; try { objectStorage = new ObjectStorageService(); } catch (e) { console.error('Object storage unavailable:', e); }`. Guard every usage with a null-check and return a 503 if the service failed to initialise, rather than crashing the process.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one-at-a-time inside a for loop with sequential awaits. For a 40-question paper this serialises 40 round-trips to the database, dramatically increasing handler duration and holding the DB connection longer than necessary.
  → *Collect valid questions into an array and perform a single batch insert: await db.insert(extractedQuestions).values(validQuestions). Add the idempotency delete before this insert as already done, but remove the per-iteration await entirely.*
- **`server/jobHandlers.ts` (runExtractPaper — image branch, buffer.toString('base64')):** fileB64 is decoded from base64 into a Buffer and then immediately re-encoded back to base64 for the OpenAI vision data URL. This is a redundant allocation that doubles peak memory for image payloads and adds CPU overhead — especially problematic if the job worker processes several concurrent uploads.
  → *Pass fileB64 directly to the data URL without decoding: `data:${mimeType};base64,${fileB64}`. Only decode to Buffer for pdf-parse (PDF branch) or for object storage uploads.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000) and max_completion_tokens: 4096):** The 30 000-character truncation limit and both max_completion_tokens values are hardcoded magic numbers. Changing model context windows or tuning for cost requires a code deploy rather than a config change, and there is no visibility into truncation occurring silently.
  → *Move these to environment variables: EXTRACT_TEXT_MAX_CHARS (default 30000), EXTRACT_MAX_TOKENS (default 4096). Log a warning when truncation occurs: if (pdfData.text.length > limit) logger.warn({ paperId, originalLength: pdfData.text.length }, 'PDF text truncated').*
- **`server/jobHandlers.ts` (admin-csv-export handler (type AdminRow / dat variable)):** The variable 'dat' is declared but the file is truncated before assignment or use is visible. Based on the pattern, the export likely fetches entire tables (transactions, users, gameSessions) into memory as JS arrays before serialising to CSV. For a growing school platform this is an unbounded memory allocation with no pagination or streaming.
  → *Use a database cursor or paginated query (limit/offset loop) writing chunks to an object storage stream rather than accumulating all rows in memory. Alternatively use a Node.js Transform stream piped directly to the object storage upload stream so peak memory stays constant regardless of row count.*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, etc.)):** All userId columns are declared as varchar with no length argument. This means Postgres defaults to varchar with unlimited length, which is functionally identical to text but semantically misleading and inconsistent with a fixed-length auth UID (e.g. Clerk/Supabase UUIDs are always 36 chars or 26 chars for CUID). A mismatch between the auth provider's ID format and DB column definition can cause silent truncation or index bloat.
  → *Determine the exact ID length from your auth provider (e.g. UUID = 36, CUID = 26) and declare all userId columns as varchar('user_id', { length: 36 }) consistently. If IDs are UUIDs, switch to the drizzle-orm uuid() column type for type safety and a more compact 16-byte storage footprint versus 36-byte text.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, etc.)):** All createdAt columns use .defaultNow() but are NOT declared .notNull(). This means the column accepts explicit NULL inserts (e.g. via a raw SQL insert or ORM bypass), corrupting audit trails which is a compliance concern for a financial app serving minors.
  → *Add .notNull() to every createdAt column: createdAt: timestamp('created_at').defaultNow().notNull(). Same applies to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table definition):** simulatedStocks.ticker has no unique index. Nothing prevents two rows with ticker 'BISX:CBL' being inserted, which would cause incorrect portfolio valuation calculations when joining on ticker for price lookups.
  → *Add a unique index on ticker: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). If regional duplicate tickers are valid, make the unique constraint composite on (ticker, region).*
- **`shared/schema.ts` (billReminders table definition):** billReminders.nextDueDate has no index. The primary production query for this table will be 'fetch all reminders for user where nextDueDate <= today' to power notifications and dashboard alerts. Without an index this is a full scan on every scheduled check.
  → *Add a compound index: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This supports both the user-scoped filter and the date range ordering in a single index scan.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render inside the component body, and the queryFn closes over `params` which is a new reference each render. This causes the query to re-fire even when filters haven't changed because the queryKey array correctly gates refetches but the params object used inside queryFn may be stale or inconsistent with the key.
  → *Move params construction into a useMemo keyed on [actorType, action, from, to], then reference that memoized string in both the queryKey and queryFn: `const paramStr = useMemo(() => { const p = new URLSearchParams(); ... return p.toString(); }, [actorType, action, from, to]);` Use paramStr in queryKey and queryFn so they are always in sync.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no error state handling and no loading state. If /api/admin/observability fails, the component silently renders with all badges showing 'Disabled' and the health-check link falls back to a hardcoded '/healthz', giving admins no indication something is wrong.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when isLoading, and an error callout with a retry button when isError. Remove the hardcoded '/healthz' fallback or make it an env constant (VITE_HEALTHZ_URL) so it is environment-aware.*
- **`client/src/pages/AdminDashboard.tsx` (params.set("limit", "200") in AuditLogPanel):** The audit log fetch limit of 200 is hardcoded inline. For a school-aged financial literacy platform with admin audit trails, this number will silently truncate records with no pagination UI, potentially hiding security-relevant events. The value is also duplicated if the query is called from elsewhere.
  → *Extract to a named constant at module scope: `const AUDIT_LOG_PAGE_SIZE = 200;` and pair it with cursor-based pagination controls (next/prev page tokens) in the UI so admins can page through the full log. If full export is needed, add a separate CSV download endpoint rather than over-fetching into the browser.*

## 💡 Opportunities

### Centralise async error boundary with a storage wrapper — 🔥 High Impact
No storage method has try/catch; unhandled promise rejections from db failures propagate raw Drizzle/PostgreSQL error objects (including table names and query fragments) up to API responses, leaking schema details. Wrap every public method with a single `withErrorBoundary` higher-order function that catches, sanitises, and re-throws a typed StorageError, keeping internal details out of HTTP responses.

### Extract hardcoded virtual balance seed amount to environment config — ⚡ Medium Impact
The initial virtual balance value (likely 10000 or similar) is hardcoded in getVirtualBalance/upsertVirtualBalance. For a school sponsor or teacher to run a custom simulation scenario this must change per deployment. Move it to `process.env.VIRTUAL_BALANCE_SEED` with a validated numeric default so it can be configured without a code deploy.

### Add userId ownership check to all single-resource fetches — 🔥 High Impact
Methods like getExamPaper(id) and getGameSessions(userId) fetch by primary key without asserting the requesting user owns the record before returning it. A student who discovers another's examPaperId can read it. Add `and(eq(table.id, id), eq(table.userId, userId))` to every single-record fetch that involves user-owned data.

### Extract AI quota endpoints into a dedicated rate-limited sub-router — 🔥 High Impact
All routes that call reserveQuota/finalizeUsage/getCachedExplanation should be grouped under a single express.Router() with shared rate-limit and auth middleware applied once. This removes the risk of accidentally adding a new AI endpoint without quota protection, and makes the 195k-char monolith meaningfully smaller.

### Replace ad-hoc sequential Supabase calls with batched lookups — ⚡ Medium Impact
Multiple route handlers independently call getOrganization, getOrgEnvironments, and getLessonsByOrg in sequence for the same org within a single request. Introduce a lightweight request-scoped cache (a plain Map keyed by org_id, created per-request in middleware) to deduplicate these calls. Estimated 40–60% reduction in Supabase round-trips on dashboard-heavy pages.

### Move hardcoded file-size limits and MIME type lists to environment config — ⚡ Medium Impact
Values like `500 * 1024 * 1024` (video upload cap) and the MIME allowlists are scattered across four multer instances with no single source of truth. Centralise them in a validated config object loaded from environment variables at startup (with zod), so ops can adjust limits without a code deploy — important when Caribbean schools have varying bandwidth constraints.

### Idempotency token on OpenAI calls — ⚡ Medium Impact
If the worker crashes after the first OpenAI call but before DB writes, a retry re-invokes the LLM and pays twice. Pass a stable idempotency key derived from paperId and attempt number via the OpenAI request header to avoid duplicate charges and non-deterministic results across retries.

### Parallel async setup in registerJobHandlers — ⚡ Medium Impact
Any initialisation work done at handler registration time (dynamic imports, config reads) runs sequentially at startup. Extracting dynamic imports like pdf-parse to module-level lazy singletons would eliminate repeated import() overhead on every job execution and make startup profiling cleaner.

### Structured logging instead of thrown Error strings — 📌 Low Impact
Errors like 'No text could be extracted' and 'Invalid private object path' are thrown as plain Error messages. Adding a structured logger (e.g. pino) with paperId, jobId, and attempt fields at each failure point would make operational debugging in production significantly faster without changing retry logic.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a column default in at least six separate table definitions (transactions, savingsGoals, billReminders, portfolioHoldings, simulatedStocks, portfolioTransactions). If the app expands to support TTD or JMD, each must be hunted down manually. Extract const DEFAULT_CURRENCY = 'BSD' into a shared config and reference it in every .default() call to make currency changes a single-line update.

### Add partial index on documentUploads for in-flight processing rows — ⚡ Medium Impact
Background jobs and API polling will repeatedly query documentUploads WHERE status = 'processing'. A standard index on status has low selectivity (3 enum values). A partial index WHERE status = 'processing' covers only the small active subset, dramatically reducing index size and improving poll-loop query latency with no downside once a row transitions to completed/failed.

### Replace learningModules.content text column with structured jsonb — 🔥 High Impact
The truncated schema shows content stored as text('content'), which forces the application layer to serialize/deserialize JSON strings manually and prevents any server-side querying of content structure (e.g. finding all modules with a specific lesson type). Migrating to jsonb('content') gives Postgres-native JSON operators, GIN indexing on content fields, and eliminates a class of serialization bugs in the content-rendering path.

### Parallel admin bootstrap queries via Promise.all — 🔥 High Impact
The dashboard overview tab likely fires multiple sequential useQuery calls (organizations, schools, teachers, students counts) that are independent. Consolidate them into a single /api/admin/overview endpoint returning all summary stats, or use Promise.all inside a single queryFn. This reduces page load waterfall from N round-trips to 1 and eliminates the cascading loading-state flicker currently visible on the overview tab.

### Extract tab panels into lazy-loaded route segments — 🔥 High Impact
All 13 tab panels (Organizations, Schools, Teachers, Students, Classes, Sponsors, Challenges, Reports, Jobs, Perf, Audit, DB Viewer) are compiled into a single bundle. Using React.lazy + Suspense per panel would cut initial parse/eval time significantly — the Recharts dependency alone adds ~150 KB gzipped and is only needed on the overview/reports tabs.

### Add role-assertion middleware on every admin API call — 🔥 High Impact
Auth is checked via useAdminAuth hook on the client, but if the apiRequest helper does not attach an admin-scoped JWT claim and the server does not re-validate the admin role on every /api/admin/* route, a regular authenticated user who discovers the endpoint can access all admin data. Add a server-side requireAdminRole middleware applied to the entire /api/admin router, and verify it throws 403 (not 401) for authenticated non-admin users so client code can distinguish the cases.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic reliability and security gaps: async operations lack try/catch, dashboard stats use sequential awaits instead of Promise.all, and portfolio holdings fetch stocks in a N+1 pattern. For a school-aged financial app handling virtual balances and real user data, missing transaction guards on multi-step writes and absent pagination on leaderboard/question queries are the most critical risks.

### API Routes (`server/routes.ts`)
routes.ts is a large monolithic route file with several reliability and security gaps. The most critical issues are missing rate limiting on AI/heavy endpoints (exposing the app to cost and abuse risk for school-aged users), sequential awaits in auth flows that could be parallelised, and sensitive data likely surfacing in error responses. Overall structure is reasonable but the file is doing too much work in one place.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps including large base64 payloads stored directly in the job table, sequential DB inserts inside a loop that should be batched, and a hardcoded token/character limit that belongs in environment config. The extract-paper handler also re-encodes an already-base64 buffer back to base64 for the vision API, doubling memory pressure on large image uploads.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key issues include missing indexes on high-cardinality query columns, varchar userId fields without length constraints creating type inconsistency risks, and missing NOT NULL constraints on audit timestamps that should always be present.
