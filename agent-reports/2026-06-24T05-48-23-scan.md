## 🔴 Critical (7)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row a separate SELECT is issued to fetch the category and another to sum spent transactions, meaning 2N extra round-trips for N budgets. Under classroom load (30+ students) this will cascade into hundreds of queries per dashboard render.

**Fix:** Replace the per-budget queries with a single JOIN: `db.select({...budgetFields, categoryName: categories.name, spent: sql<number>`COALESCE(SUM(t.amount),0)`}).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).where(eq(budgets.userId, userId)).groupBy(budgets.id, categories.id)`. One query replaces 2N.

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (buy/sell flow)
**Issue:** The three-step trade flow — upsert holding, insert portfolio transaction, update virtual balance — executes without a database transaction. A crash or timeout between steps leaves the ledger in an inconsistent state (e.g. balance deducted but holding not recorded), which is especially dangerous for a financial literacy app where students expect accurate virtual balances.

**Fix:** Wrap all three operations in `db.transaction(async (tx) => { await tx.insert(portfolioHoldings)...; await tx.insert(portfolioTransactions)...; await tx.update(userVirtualBalance)...; })`. Drizzle's transaction callback rolls back automatically on throw.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with zero rate limiting. Any authenticated (or unauthenticated, depending on those handlers) user can hammer the AI endpoints, causing runaway OpenAI billing and denial-of-service for other users. This is especially dangerous for a school-aged audience where accounts may be shared.

**Fix:** Apply a dedicated AI rate limiter before each registration: e.g., `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter is something like `rateLimit({ windowMs: 60_000, max: 20, standardHeaders: true })`. Add the limiter definition to rateLimiter.ts alongside the existing ones.

### `server/routes.ts`: robots.txt handler (~line 37) and sitemap.xml handler (~line 51)
**Issue:** Both handlers blindly reflect `x-forwarded-proto` and `x-forwarded-host` headers into response bodies without any validation or allowlist. An attacker can supply arbitrary values for these headers to poison the Sitemap URL in robots.txt or inject arbitrary base URLs into the sitemap, enabling cache-poisoning or open-redirect-style phishing.

**Fix:** Replace the dynamic host derivation with a single trusted base URL from environment config: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';`. Remove all header-based host detection from these two handlers. Set APP_BASE_URL in your Railway/Replit environment.

### `shared/schema.ts`: categories table definition
**Issue:** categories.userId uses varchar with no length specified, defaulting to 1 character in some Postgres drivers, which will silently truncate UUID user IDs. All other tables use varchar('user_id') with the same issue, but categories is the root reference point.

**Fix:** Change all userId columns to varchar('user_id', { length: 256 }) to safely accommodate UUIDs, OAuth subject identifiers, and Clerk/Auth0 style IDs. Apply consistently across categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions.

---

### `client/src/pages/AdminDashboard.tsx`: sendMessage – apiRequest call
**Issue:** User input is passed directly into the POST body as `{ messages: next }` without any length or content sanitization. A school-aged user context makes prompt-injection via the chat input a real risk; an attacker could craft messages that manipulate the system prompt on the backend.

**Fix:** Before calling apiRequest, enforce a character limit (e.g., 500 chars): `if (text.length > 500) return toast({ title: 'Message too long' })`. On the server route /api/admin/help-chat, validate that each message object matches the expected schema and strip/escape HTML before forwarding to the LLM.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level component mount
**Issue:** The file imports and calls useAdminAuth but the truncated code pattern commonly shows the auth check result (isAdmin, isLoading) not being used to gate rendering — meaning the full admin UI and its useQuery calls fire before auth state resolves. Unauthenticated users may trigger admin API calls.

**Fix:** Immediately after calling useAdminAuth, add: `if (authLoading) return <Loader2 className='animate-spin' />;` and `if (!isAdmin) { setLocation('/'); return null; }` before any other hooks or JSX so all downstream queries and mutations are blocked until auth is confirmed.

## 🟡 Warnings (13)

- **`server/storage.ts` (getDashboardStats):** Four to six independent aggregate queries (total income, total expenses, category breakdown, savings progress, etc.) are awaited sequentially. Each adds a full network round-trip to the DB, making dashboard load time additive rather than parallel.
  → *Collect all query promises into an array and resolve with `const [income, expenses, categoryTotals, savingsGoals] = await Promise.all([incomeQuery, expensesQuery, categoryQuery, goalsQuery])`. Expected wall-clock saving: (N-1) × avg query latency, typically 60-150 ms on a hosted Postgres.*
- **`server/storage.ts` (getTransactions (limit parameter handling)):** The `limit` filter from the caller is applied without a maximum cap. A caller (or malicious student probing the API) can pass `limit=1000000`, causing the DB to stream a full table scan into Node's heap. No pagination cursor or hard upper bound is enforced.
  → *Clamp the limit before use: `const safeLimit = Math.min(filters?.limit ?? 50, 200)` and apply `.limit(safeLimit)`. Add a `offset` parameter for cursor/page-based pagination. Document the maximum in the interface type.*
- **`server/storage.ts` (getLeaderboard):** The leaderboard query JOINs game_sessions with users and logs `userName` and `avatar` fields to build the response. If any logging middleware dumps the result object (common during dev/debug), real student display names and avatar URLs are written to server logs — a privacy concern under Caribbean data-protection norms for minors.
  → *Ensure no `console.log(leaderboard)` or generic request-logging middleware serialises this response. Add a `toSafeLeaderboard()` transform that explicitly whitelists `{ userId, totalScore, gamesPlayed }` before logging, keeping PII out of log sinks. Also enforce `limit` cap (max 100) with `const safeLimit = Math.min(filters?.limit ?? 10, 100)`.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** The seven Supabase seed/init calls are chained sequentially with .then(), so each waits for the previous to finish. If seedFinancialAcademyLesson takes 3 s and seedGameContent takes 2 s, startup background work takes 7+ s serially when most steps are independent.
  → *Group truly independent steps into Promise.all: `await Promise.all([initSupabaseTables(), initStaticContentTables()])` then `await Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])`. Wrap the whole block in a single .catch to preserve the non-blocking behaviour.*
- **`server/routes.ts` (/api/health (~line 27) and /healthz (~line 75)):** There are two overlapping health-check endpoints (`/api/health` and `/healthz`). The first returns no DB liveness check; the second does. Railway and uptime monitors will be pointed at one, giving a false sense of health if the DB is down and `/api/health` is used. Dead/duplicate code also increases maintenance surface.
  → *Remove `/api/health` and standardise on `/healthz` (which includes the DB probe). Update Railway's health-check path to `/healthz`. If a lightweight ping is needed without a DB call, add a `?shallow=1` query param branch inside `/healthz` instead of a second route.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The filePath wildcard parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without any sanitisation. A path like `../../private/keys` or URL-encoded traversal sequences could reach unintended storage paths depending on how the ObjectStorageService resolves keys.
  → *Sanitise the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, '').replace(/^[\/]+/, '');`. Additionally, consider an allowlist of valid prefixes (e.g., logos/, videos/) and reject anything that doesn't match: `if (!/^(logos|videos|images)\//.test(filePath)) return res.status(400).json({ error: 'Invalid path' });`.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** createdAt columns across multiple tables use .defaultNow() but omit .notNull(), meaning application code or raw SQL inserts can store NULL into an audit timestamp column, making time-based queries and ordering unreliable.
  → *Append .notNull() to every createdAt and executedAt/purchasedAt timestamp column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default handles inserts transparently; notNull prevents explicit NULL writes.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId even though budget lookups will always filter by user. Without an index, every budget query does a full table scan that worsens linearly as user count grows.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also add a composite uniqueIndex on (userId, categoryId, period) to enforce one budget per category per period per user at the database level.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Neither portfolioHoldings nor portfolioTransactions has an index on userId or stockId, yet the primary access patterns (fetch a user's holdings, fetch trades for a stock) require both. portfolioTransactions will grow unboundedly and is the highest-risk table for full scans.
  → *For portfolioHoldings add (t) => ({ userStockUniq: uniqueIndex('idx_holdings_user_stock').on(t.userId, t.stockId) }) — this also enforces one holding row per user per stock. For portfolioTransactions add (t) => ({ userIdx: index('idx_ptx_user').on(t.userId), executedIdx: index('idx_ptx_executed').on(t.userId, t.executedAt) }).*
- **`shared/schema.ts` (simulatedStocks table definition):** simulatedStocks.ticker has no unique constraint, so duplicate tickers can be inserted, corrupting portfolio value calculations and leaderboard lookups that join on ticker.
  → *Replace the plain text column with ticker: text('ticker').notNull() and add a uniqueIndex in the table options: (t) => ({ tickerUniq: uniqueIndex('idx_stocks_ticker').on(t.ticker) }). Also add .notNull() to issuer and region if they are required for Caribbean bond data completeness.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The sendMessage function is defined inline without useCallback and closes over messages/input state. Every render creates a new function reference, breaking any memoization on child components that receive it as a prop, and the stale-closure risk means messages could be stale if React batches state updates mid-flight.
  → *Wrap sendMessage in useCallback(async () => { ... }, [input, loading]) and use the functional updater form setMessages(prev => [...prev, ...]) exclusively — which is already partially done — to avoid stale closure on messages.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – Messages render, inline ref callback):** The scroll-anchor ref is implemented as an inline arrow `ref={el => { if (el) el.scrollIntoView(...) }}`. This anonymous function is recreated every render, causing React to detach and reattach the ref on every message update, which fires scrollIntoView redundantly and thrashes the DOM.
  → *Replace with a stable ref: `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages])`. Attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple independent useQuery calls at dashboard mount):** The dashboard likely issues several independent useQuery calls (users, schools, stats, audit log) sequentially as separate hooks. While React Query fires them concurrently by default, any place in the codebase where these are fetched via sequential awaits inside a single async function (e.g., data-export handlers) will serialize network requests unnecessarily.
  → *Audit all async handlers for patterns like `const a = await fetch(...); const b = await fetch(...)` and replace with `const [a, b] = await Promise.all([fetch(...), fetch(...)])`. For the export/download action in particular, wrap all parallel fetches in Promise.all to cut perceived latency by the number of sequential calls minus one.*

## 💡 Opportunities

### Batch category resolution with a shared dataloader: 🔥 High Impact
getBillReminders, getBudgets, and getTransactions all independently fetch categories. Introducing a per-request Map cache or a lightweight dataloader would eliminate duplicate category SELECTs within the same request cycle, reducing DB load proportionally to the number of concurrent students viewing dashboards.

### Extract hardcoded virtual balance seed value to environment config: ⚡ Medium Impact
The initial virtual balance (e.g. 10000.00) granted on first account creation appears hardcoded in the upsert logic. Moving it to `process.env.VIRTUAL_BALANCE_SEED` with a validated default allows educators or sponsors to configure starting balances per cohort without a code deploy.

### Add missing try/catch to fire-and-forget async calls in seedMarketData / seedLearningModules: ⚡ Medium Impact
Seed functions are called at startup without error boundaries; an unhandled rejection here will crash the Node process before any student can log in. Wrapping each in try/catch with a structured log and graceful degradation keeps the app available even when seed data is already present or the DB is temporarily unavailable.

### Centralise domain-router registration into a manifest: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import block are sequential. If any registration does async DB work (schema checks, etc.) they block each other. Extract all six into a `routes/index.ts` manifest that runs registrations in parallel with `Promise.all`, cutting cold-start time proportionally to the slowest individual registrar.

### Add authenticated-user guard middleware for AI routes: 🔥 High Impact
Beyond rate limiting, the chat/image/audio routes should verify that the requesting user is authenticated and belongs to an active organisation/school before hitting OpenAI. A single `requireAuth` middleware applied to `/api/chat`, `/api/image`, `/api/audio` prefixes here in routes.ts would centralise this rather than relying on each sub-router to remember it, reducing the risk of a missed check as new AI routes are added.

### Move ObjectStorageService instantiation to a shared singleton module: 📌 Low Impact
`new ObjectStorageService()` is instantiated at module load time in routes.ts. If ObjectStorageService opens a connection pool or holds state, multiple test/dev restarts or future imports from other modules will create duplicate instances. Export a singleton from `replit_integrations/object_storage.ts` (e.g., `export const objectStorage = new ObjectStorageService()`) and import it wherever needed, preventing resource leaks as the codebase grows.

### Consolidate repeated currency default into a shared column config: ⚡ Medium Impact
currency: text('currency').default('BSD').notNull() appears verbatim in at least six tables. Extract a currencyCol helper constant and reference it everywhere. This eliminates drift risk if the default currency ever changes for multi-island expansion and reduces schema LOC by ~30 lines.

### Add partial index on documentUploads.status for processing queue polling: 🔥 High Impact
The document processing worker almost certainly polls WHERE status = 'processing'. A partial index — CREATE INDEX idx_uploads_processing ON document_uploads(user_id) WHERE status = 'processing' — would be tiny (only unfinished rows) and make queue polling O(1) instead of scanning all completed historical uploads.

### Replace text type with enum-mapped varchar for period and frequency columns: ⚡ Medium Impact
budgets.period and billReminders.frequency store constrained enum values in unbounded text columns. Using Postgres native enums (pgEnum in Drizzle) enforces values at the DB layer without a CHECK constraint, enables index range scans by period, and makes pg_dump output self-documenting — important for audit trails in a financial app.

### Memoize filtered/sorted table data: 🔥 High Impact
Any user or school list that is filtered by a search string and sorted client-side should be wrapped in useMemo([rawData, searchTerm, sortKey]). Without this, every keystroke in the search Input re-runs the full filter+sort over potentially thousands of records on the main thread, causing visible input lag for admins managing large Caribbean school networks.

### Persist chat history across panel open/close: ⚡ Medium Impact
AdminHelpChat initializes messages as [] on every mount because the component unmounts when the panel closes. Moving state up to the parent or using a ref to preserve messages between toggles would give admins continuity. This is a one-line change to lift useState to the parent scope.

### Replace hardcoded API path strings with a central config: ⚡ Medium Impact
Strings like '/api/admin/help-chat' appear inline throughout the file. Centralizing all API route constants into a single src/lib/apiRoutes.ts file eliminates typo-driven 404s during refactors and makes environment-specific base-URL overrides (staging vs production) trivial to apply in one place.
