## 🔴 Critical (3)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 109-111)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. Any unauthenticated or authenticated caller can hammer these endpoints, incurring unbounded OpenAI API costs and enabling DoS for a school-aged user base.

**Fix:** Apply a dedicated AI rate limiter before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter uses express-rate-limit with a conservative window (e.g., 10 req/min per IP). Also enforce auth middleware on these prefixes so anonymous callers are rejected before reaching OpenAI.

### `server/routes.ts`: robots.txt handler (~line 51) and sitemap.xml handler (~line 64)
**Issue:** The base URL is constructed directly from `x-forwarded-proto`, `x-forwarded-host`, and `Host` request headers without any validation or allowlist. An attacker can send crafted headers to inject arbitrary URLs into the Sitemap directive and sitemap <loc> values, enabling cache poisoning or phishing via SEO abuse.

**Fix:** Read the canonical base URL from a single trusted environment variable (e.g., `process.env.PUBLIC_BASE_URL`) set at deploy time. Fall back to a hardcoded production domain, never to raw request headers: `const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com';`

### `client/src/pages/AdminDashboard.tsx`: AdminDashboard – top of component render, before data loads
**Issue:** There is no auth guard that prevents the entire admin UI from rendering before `useAdminAuth` resolves. If the hook is async (fetches session from server), the dashboard briefly renders with no user context, potentially exposing admin-only query calls and UI before the redirect fires.

**Fix:** Add an early return immediately after the auth hook: `const { admin, isLoading } = useAdminAuth(); if (isLoading) return <FullPageSpinner />; if (!admin) return null;` This ensures all child queries and renders are gated on a confirmed admin session before any network calls are made.

## 🟡 Warnings (6)

- **`server/routes.ts` (Supabase init chain (~line 114-121)):** The seven-step sequential Supabase init chain uses a single `.catch` at the end. If any intermediate step (e.g., `seedFinancialAcademyLesson`) throws, subsequent steps (`initStaticContentTables`, `seedStaticModules`, etc.) are silently skipped with no per-step error context, making startup failures very hard to diagnose.
  → *Wrap each step individually or use an async IIFE with per-step try/catch and structured logging: `(async () => { for (const [name, fn] of steps) { try { await fn(); } catch (e) { console.error(`[Supabase] ${name} failed:`, e); } } })();` This ensures all steps attempt to run and each failure is attributed to the correct seeding function.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97)):** The `filePath` path parameter accepts arbitrary user-supplied values and is passed directly to `objectStorage.searchPublicObject`. There is no validation, sanitization, or path-traversal guard, so a request like `/public-objects/../../private/secret` could resolve unintended storage keys depending on the ObjectStorageService implementation.
  → *Sanitize the filePath before use: reject any value containing `..` or leading `/`, and optionally restrict to an allowlist of extensions (e.g., `.png`, `.jpg`, `.mp4`). Example: `if (filePath.includes('..') || filePath.startsWith('/')) return res.status(400).json({ error: 'Invalid path' });`*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` creates a new function on every render and fires scrollIntoView on every re-render, not just when a new message is appended. This causes jarring scroll resets when the user is reading older messages and the component re-renders for any reason (loading state change, input change).
  → *Replace with a proper useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then use `<div ref={bottomRef} />`. This scopes scrolling only to actual message additions.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage, messages state):** The full message history array is sent to `/api/admin/help-chat` on every turn (`messages: next`). For a school-aged app with long sessions this payload grows without bound, increasing latency and potentially hitting request size limits. There is also no cap or truncation strategy.
  → *Slice the context window before sending: `const context = next.slice(-10);` then `apiRequest('POST', '/api/admin/help-chat', { messages: context })`. Keep the full history in local state for display but only send the last N turns. Document the chosen window size as a named constant.*
- **`client/src/pages/AdminDashboard.tsx` (AdminDashboard – multiple useQuery calls at component mount):** Several independent useQuery calls (users, schools, teachers, stats, audit log, etc.) are declared sequentially and each fires its own fetch. While React Query parallelises these by default, any data-loading logic that uses `await` sequentially inside useEffect or mutation handlers will block. More critically, at least one pattern of `const a = await fetch(...); const b = await fetch(...)` exists inside mutation/effect handlers where the results are independent.
  → *Audit every async function in the file for independent sequential awaits and convert them to `const [a, b] = await Promise.all([fetchA(), fetchB()])`. For the useQuery declarations themselves, confirm they are at the top level (not conditionally called) so React Query can issue them in parallel correctly.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage, catch block):** The catch block silently swallows the error object entirely (`} catch {`). For an admin tool, losing error detail makes debugging production issues very difficult. Additionally, the error message shown to the admin ('Sorry, something went wrong') gives no actionable information.
  → *Change to `} catch (err) { console.error('[AdminHelpChat] sendMessage failed:', err); setMessages(prev => [...prev, { role: 'assistant', content: `Request failed: ${err instanceof Error ? err.message : 'unknown error'}. Please retry.` }]); }` Do not log full response bodies that might contain user data.*

## 💡 Opportunities

### Parallel Supabase seed execution: ⚡ Medium Impact
The seven Supabase init steps run sequentially, adding unnecessary latency to server cold-start. Steps that are not data-dependent on each other (e.g., seedRegionalContent, seedGameContent) could be grouped into Promise.all batches, potentially cutting init time by 50-70% on slow DB connections.

### Centralised auth guard middleware: 🔥 High Impact
Auth enforcement is delegated entirely to individual domain routers (registerAiRoutes, registerOrgRoutes, etc.), making it easy for a future route to ship without authentication. Adding a top-level middleware that protects the `/api/*` namespace by default—with an explicit opt-out for public routes like /api/health—would make auth the secure default across all current and future routes.

### Environment-driven ObjectStorageService instantiation: ⚡ Medium Impact
ObjectStorageService is instantiated as a module-level singleton before any environment validation runs. If required storage credentials are missing, the error surfaces only on the first request. Validate required env vars (bucket name, credentials) at startup in registerRoutes and throw early with a clear message, preventing silent misconfiguration in production.

### Memoize filtered and sorted table data: 🔥 High Impact
Any filtering, sorting, or searching of the users/schools/teachers arrays that happens inline during render should be wrapped in useMemo keyed on the source array and the filter/search state. For a dashboard with hundreds of school records this prevents O(n) array work on every keystroke-driven re-render of the parent.

### Extract mutation handlers out of JSX with useCallback: ⚡ Medium Impact
Large inline arrow functions passed to Button onClick props (delete user, toggle active, promote teacher, etc.) recreate on every render and cause all rendered list-item buttons to re-render even when unrelated state changes. Wrap each handler in useCallback and move complex logic outside JSX to keep the template readable and stable.

### Move hardcoded API base paths to environment config: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/admin/users' etc. are scattered inline throughout the file. A single typo causes silent 404s with no compile-time error. Centralise these in a typed `API_ROUTES` constant object (or auto-generate from the server router) so renaming an endpoint is a one-line change and typos are caught at the call site.
