## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: for each PortfolioHolding row returned, a separate getStockById query is issued inside a loop to attach the stock object, producing O(n) round-trips to the database per user call.

**Fix:** Replace the loop with a single JOIN query: db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId)). Drizzle returns the joined shape in one round-trip, eliminating all per-holding queries.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (trade execution flow)
**Issue:** Multi-step financial write operations (update holding quantity, insert portfolio transaction, deduct virtual balance) are executed as three independent awaits with no wrapping database transaction. A process crash or error between steps leaves the database in a partially-updated state — e.g., balance deducted but holding not updated.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Pass the tx client to each sub-operation so they share the same connection and atomically commit or roll back together.

---

### `server/routes.ts` — AI/tutor endpoints using reserveQuota
**Issue:** No HTTP-level rate limiting middleware is applied to AI endpoints before quota reservation. An authenticated user can fire hundreds of concurrent requests; reserveQuota is not atomic under concurrency, so multiple requests can each see available quota and reserve simultaneously, exceeding the org limit before finalizeUsage is recorded.

**Fix:** Add express-rate-limit (or a Redis-backed equivalent) scoped per user/org on all AI routes before the quota logic: `rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip })`. This guards the quota check itself from race-condition bypass without needing to make reserveQuota transactional.

---

### `server/routes.ts` — multer upload handlers (logoUpload, videoUpload, examUpload)
**Issue:** multer is configured with memoryStorage for files up to 500 MB (videoUpload). Under concurrent uploads this can exhaust Node.js heap because all bytes are buffered in-process with no back-pressure or concurrent-upload cap.

**Fix:** Switch videoUpload to diskStorage with a temp directory, stream directly to object storage inside the route handler, then unlink the temp file. Add a concurrency guard (e.g. a semaphore limited to 3) around video upload processing. For logo/exam uploads keep memoryStorage but enforce the existing 5–10 MB limits strictly.

---

### `server/routes.ts` — sendEmail / audit calls throughout route handlers
**Issue:** Multiple route handlers call sendEmail() and audit() as fire-and-forget without await and without a .catch(). An unhandled rejection from either (e.g. email provider timeout) will crash the Node process in Node 18+ where unhandledRejection terminates the process by default.

**Fix:** Either await these calls inside the existing try/catch, or wrap them in a safe helper: `const fireAndForget = (p: Promise<unknown>) => p.catch(err => logger.error('background task failed', err))`. Apply fireAndForget(sendEmail(...)) and fireAndForget(audit(...)) consistently everywhere they are not awaited.

---

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter / job.payload)
**Issue:** Full base64-encoded file content is stored directly in the job payload. A single PDF or image can easily be 5–20 MB encoded, meaning every retry duplicates that blob in the jobs table, and any job-queue query that fetches pending jobs loads all of it into memory simultaneously for every queued paper.

**Fix:** Upload the file to private object storage at upload time and store only the resulting object path in the job payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient before processing. This caps the job row at ~200 bytes regardless of file size.

---

### `server/jobHandlers.ts` — purge-ai-usage handler — olderThanDays from job.payload
**Issue:** The purge cutoff is derived entirely from job.payload.olderThanDays with no validation or bounds check. A misconfigured enqueue call with olderThanDays: 0 or a negative value would delete all AI usage events or produce a future cutoff date, causing a silent full-table wipe with no observable error.

**Fix:** Add an explicit guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('olderThanDays must be an integer between 1 and 3650'); This makes the job fail loudly and retryably rather than silently corrupting data.

---

### `shared/schema.ts` — simulatedStocks table definition — ticker column
**Issue:** ticker has no unique index. Multiple rows with the same ticker symbol (e.g. two 'BAHA' entries) would silently corrupt portfolio holdings and transaction history, since portfolioHoldings and portfolioTransactions reference stockId but UI lookups likely also use ticker for display and deduplication.

**Fix:** Add a uniqueIndex on ticker: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also consider a composite unique on (ticker, region) if the same ticker can exist across markets.

## 🟡 Warnings (15)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for totalIncome, totalExpenses, categoryBreakdown, and recentTransactions queries are chained one after another despite being fully independent, adding cumulative latency on every dashboard load for every user.
  → *Replace sequential awaits with const [income, expenses, breakdown, recent] = await Promise.all([incomeQuery, expensesQuery, breakdownQuery, recentQuery]). Each query targets different aggregation predicates and has no data dependency on the others.*
- **`server/storage.ts` (getTransactions / getAllQuestions / getLeaderboard):** Unbounded list queries lack mandatory pagination guards. getTransactions accepts an optional limit but defaults to no limit; getAllQuestions and getLeaderboard can return entire table contents. For a school app with many student users, this risks large memory allocations and slow responses under load.
  → *Enforce a default and maximum limit on all list queries: const safeLimit = Math.min(filters?.limit ?? 50, 200). For getLeaderboard, cap at 100. Apply .limit(safeLimit).offset(filters?.offset ?? 0) on every unbounded select so callers cannot accidentally or maliciously trigger full-table scans.*
- **`server/storage.ts` (getLeaderboard / getUserXp (logging or error paths)):** User display names and avatar URLs pulled for the leaderboard are logged or passed through without sanitization, and internal DB error objects caught in generic catch blocks are forwarded directly to callers, risking exposure of table names, column names, or stack traces to client responses in production.
  → *In all catch blocks replace throw err with throw new Error('Operation failed') after logging the original error server-side only via a structured logger (never console.log in production). For leaderboard, strip any fields beyond userId, userName, avatar, totalScore, gamesPlayed before returning the array.*
- **`server/routes.ts` (/api/auth/google handler (~line 110)):** Sequential awaits on independent Supabase/storage lookups inside the Google auth flow add latency on every sign-in. getOrgEnvironmentById and getOrganization are called one after the other despite having no data dependency on each other in the domain-check branch.
  → *Replace sequential calls with Promise.all where results are independent: `const [classEnv, resolvedOrgEnv] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), getOrgEnvironmentByJoinCode(code)])`. Apply the same pattern wherever storage.getClass* and getOrganization* are called back-to-back with no inter-dependency.*
- **`server/routes.ts` (objectStorage / objectStorageClient instantiation at module top-level):** Both `new ObjectStorageService()` and `objectStorageClient` are instantiated at import time. If the underlying SDK reads credentials synchronously from env vars that are not yet populated (e.g. during test runs or cold start before dotenv loads), the module silently uses undefined credentials and every upload will fail at runtime with an opaque error rather than a startup-time config error.
  → *Lazy-initialize inside a getObjectStorage() factory that throws a descriptive ConfigurationError if required env vars (OBJECT_STORAGE_KEY, etc.) are missing. Call it the first time a route needs it so startup fails fast with a clear message instead of failing silently on first user action.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time in a sequential for-loop with an await on each iteration. For a 40-question exam paper this issues 40 serial round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy far longer than needed.
  → *Collect all valid question objects into an array first, then issue a single batch insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be kept, replace the for-loop with await Promise.all(validQuestions.map((q, i) => storage.createExtractedQuestion({...}))) — but a single bulk insert is strongly preferred.*
- **`server/jobHandlers.ts` (runExtractPaper and extractionResponse calls — model: "gpt-4o"):** The OpenAI model name is hardcoded as the string "gpt-4o" in two separate openai.chat.completions.create calls. Changing the model for cost control, A/B testing, or when the model is deprecated requires a code deploy rather than a config change.
  → *Read the model from an environment variable with a fallback: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? "gpt-4o"; and reference EXTRACT_MODEL in both calls. Add the variable to your .env.example and deployment config documentation.*
- **`server/jobHandlers.ts` (runExtractPaper — vision branch: buffer.toString("base64")):** When the uploaded file is an image, the buffer is decoded from base64 (job payload) and then immediately re-encoded to base64 to embed in the OpenAI vision message. This means the raw binary is in memory twice as base64 strings — once as the input fileB64 string and once as the re-encoded data URI — both of which are also retained in the job payload. For a 10 MB image this wastes ~27 MB of heap in one job invocation.
  → *After fixing finding #1 (store only the object path), download the file bytes once, keep a single Buffer reference, and pass buffer.toString("base64") inline to the OpenAI call without storing it anywhere else. Do not retain the full buffer beyond the scope of the OpenAI call.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Budget queries will almost certainly filter by userId (e.g. 'show my budgets'), and joins to categories will use categoryId. Without indexes, every budget lookup is a full table scan that worsens as user count grows.
  → *Add a compound index on (userId, categoryId) and a standalone index on userId inside a third argument to pgTable: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })*
- **`shared/schema.ts` (savingsGoals, billReminders, linkedCards, portfolioHoldings, portfolioTransactions table definitions):** None of these tables have an index on userId, yet every app query for these resources will filter by userId. portfolioTransactions also lacks an index on (userId, executedAt) for timeline queries, and billReminders lacks an index on nextDueDate which is needed for due-date alerting jobs.
  → *Add per-table index callbacks: savingsGoals and linkedCards get index on userId; billReminders gets index on (userId, nextDueDate); portfolioHoldings gets index on (userId, stockId); portfolioTransactions gets index on (userId, executedAt).*
- **`shared/schema.ts` (userId varchar columns across all tables (categories, transactions, budgets, etc.)):** userId is declared as varchar() with no length limit. In PostgreSQL, unbounded varchar is functionally equivalent to text but signals no intent and skips length validation. More critically, if the referenced users.id has a defined length (e.g. varchar(21) for a CUID), mismatched types can prevent index usage on joins and allow storing malformed IDs.
  → *Audit users.id length in models/auth. Set all userId foreign key columns to the same explicit type, e.g. varchar('user_id', { length: 21 }) to match CUID length, ensuring consistent types across FK relationships and enabling index-accelerated joins.*
- **`shared/schema.ts` (createdAt columns across all tables; purchasedAt in portfolioHoldings, executedAt in portfolioTransactions):** All audit timestamp columns use .defaultNow() but omit .notNull(). This means a row can be inserted with an explicit NULL for createdAt, breaking any ORDER BY createdAt DESC queries with unexpected NULL ordering and corrupting audit trails, which is especially problematic for a financial app.
  → *Chain .notNull() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to createdAt, purchasedAt, and executedAt across all tables.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo. This also means the queryKey array entries (actorType, action, from, to) are correct for cache invalidation, but the params object itself is rebuilt unconditionally even when nothing changed, causing unnecessary allocations on each keystroke in sibling state.
  → *Wrap the params construction in useMemo: const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]); Then use params.toString() in the queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery in ObservabilityCard has no error state or loading state rendered. If the /api/admin/observability endpoint fails or is slow, all Badge and anchor elements render with undefined data silently — the health-check link falls back to '/healthz' but sentry badges render 'Disabled' indistinguishably from a genuine disabled state, masking fetch failures from admins.
  → *Destructure isLoading and isError from useQuery. Render a Loader2 spinner when isLoading, and an AlertTriangle with the error message when isError, instead of silently showing default/false badge states. Example: if (isLoading) return <CardContent><Loader2 className='animate-spin' /></CardContent>; if (isError) return <CardContent className='text-red-400'>Failed to load observability data</CardContent>;*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — queryFn with hardcoded limit):** The audit log query hardcodes limit=200 as a magic number inside the query params. This is a data exposure and performance risk: 200 audit rows are fetched and held in memory on every filter change with no pagination UI guard, and the value is not reflected in the queryKey so changing it in future won't invalidate cache correctly.
  → *Extract as a named constant at module scope: const AUDIT_LOG_PAGE_LIMIT = 200; Add it to the queryKey: queryKey: ['/api/admin/audit-log', actorType, action, from, to, AUDIT_LOG_PAGE_LIMIT]. For a school-age financial app with potential FERPA/COPPA implications, also confirm the audit endpoint requires admin auth server-side and consider reducing the default limit with cursor-based pagination.*

## 🔵 Info / Tech Debt (2)

- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — refetch button onClick):** The Button's onClick is assigned an inline arrow function (() => refetch()) on every render. While refetch from useQuery is stable, wrapping it in a new arrow function every render still allocates a new function reference, preventing React from bailing out on Button's memo comparison if Button is memoized internally.
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper function):** fmtDate accepts type 'any' and calls new Date(d) without validation. Passing a malformed date string returns 'Invalid Date' which toLocaleDateString renders as the string 'Invalid Date' visible to admin users. Additionally, creating a new Intl.DateTimeFormat-backed options object on every call is wasteful when this formatter is used repeatedly across large data tables.

## 💡 Opportunities

### Consolidate JOIN-based eager loading across all association queries — 🔥 High Impact
getBudgets, getBillReminders, and getPortfolioTransactions each attach related rows (category, stock) via post-query lookups or per-row fetches. Migrating all three to Drizzle relational queries or explicit JOINs would eliminate at least 3 additional N+1 patterns, reducing average API response time for budget and investment screens by an estimated 40-70% under realistic class-sized loads (30-100 students).

### Introduce a repository-layer caching strategy for read-heavy reference data — ⚡ Medium Impact
getLearningModules, getMarketStocks, and getCategories return largely static data fetched on every request with no caching. Wrapping these with a short-lived in-process cache (node-cache or a Map with TTL) of 60 seconds would cut DB reads for the most frequently called endpoints by an order of magnitude during classroom concurrent usage spikes.

### Extract hardcoded virtual balance seed value to environment configuration — ⚡ Medium Impact
The initial virtual balance (commonly 10000 or similar) assigned in getVirtualBalance when no row exists is hardcoded in the function body. Moving this to process.env.VIRTUAL_BALANCE_INITIAL_AMOUNT with a validated fallback lets curriculum designers or sponsors configure starting balances per deployment without a code change or redeployment.

### Extract AI route handlers into a dedicated router module — 🔥 High Impact
The AI/tutor quota logic (reserveQuota, finalizeUsage, releaseReservation, cache lookup) is duplicated inline across multiple route handlers. Extracting it into a reusable middleware or service class would eliminate the duplication, make rate-limit and auth checks easier to audit in one place, and reduce the risk of forgetting releaseReservation in error paths.

### Batch repeated per-request Supabase org lookups with a short-lived in-memory cache — 🔥 High Impact
getOrganization() and getOrgQuotaSettings() are called independently on nearly every authenticated request. A 30-second TTL in-memory cache (or a request-scoped context set once in auth middleware) would cut Supabase round-trips by an estimated 80% under typical classroom load spikes without stale-data risk for quota-sensitive operations.

### Move hardcoded file-size limits and MIME type lists to environment config — ⚡ Medium Impact
Values like 500 * 1024 * 1024 (video limit) and the MIME allowlists appear as magic numbers/strings in four separate multer configurations. Centralising them in a validated config object (zod-parsed from env vars with defaults) makes it possible to adjust limits per deployment tier without a code change, and prevents accidental divergence between upload validators.

### Centralised OpenAI call wrapper with usage logging and timeout — 🔥 High Impact
Both openai.chat.completions.create calls in runExtractPaper have no timeout and log nothing on failure before rethrowing. A thin wrapper that enforces an AbortSignal timeout (e.g. 90 s), logs model/tokens/latency to aiUsageEvents, and sanitises errors before re-throwing would give visibility into cost and catch hung requests that currently cause the job worker to stall indefinitely.

### Stream PDF text extraction instead of slicing at 30 000 chars — ⚡ Medium Impact
The current code slices extractedText to 30 000 characters as a blunt token-budget guard. For multi-section Caribbean exam papers this can silently drop entire question sections. Using pdf-parse's page-level callbacks to extract text page-by-page and stopping once a token estimate exceeds the budget would preserve complete questions and make the truncation boundary auditable.

### Idempotency key on admin-csv-export to prevent duplicate object writes — ⚡ Medium Impact
If the admin-csv-export job crashes after writing the object but before marking the job complete, a retry writes a second object under the same path, silently overwriting the first. Generating the object path from a deterministic key (e.g. sha256 of type + jobId) and checking file.exists() before regenerating would make retries safe and cut redundant OpenAI/DB work on transient failures.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract to a const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it via .default(DEFAULT_CURRENCY). When the app expands to TTD or XCD, one change covers all tables instead of risking missed updates.

### Add partial index for pending document uploads — ⚡ Medium Impact
documentUploads.status drives background processing jobs that must repeatedly query WHERE status = 'processing'. A partial index — index('idx_document_uploads_processing').on(t.userId).where(sql`status = 'processing'`) — would be tiny (only unfinished rows) and make polling queries near-instant regardless of total upload history volume.

### Replace text columns for icon and color with constrained types — 📌 Low Impact
icon and color appear in categories, savingsGoals, and simulatedStocks as unconstrained text. color should be varchar(7) to enforce hex format (#RRGGBB), preventing UI crashes from malformed values. icon should ideally reference a validated enum or at minimum be varchar(50). Adding a CHECK constraint or application-layer zod refinement in the insert schema would catch bad data at write time rather than at render time in a student's browser.

### Split AdminDashboard into lazy-loaded tab modules — 🔥 High Impact
At 126k+ characters, this single file causes the entire admin bundle — including Recharts, all tab panels, and all mutation logic — to parse and execute on load. Code-split each TABS entry into its own React.lazy() import (e.g., lazy(() => import('./admin/AuditLogPanel'))). This would reduce initial parse time and allow the jobs/perf/dbviewer panels (heavy) to load only when visited, with an estimated 40-60% reduction in admin bundle initial parse cost.

### Consolidate per-tab useQuery calls with parallel fetching via Promise.all — 🔥 High Impact
The overview/dashboard tab likely fires multiple sequential useQuery calls (organizations, schools, teachers, students counts) that each block on the previous. Combine into a single /api/admin/summary endpoint that fetches all counts in parallel server-side with Promise.all, reducing round-trips from N to 1 and cutting dashboard load time proportionally to N.

### Add staleTime and gcTime to admin queries to prevent redundant refetches — ⚡ Medium Impact
Admin data like school lists and sponsor records changes infrequently but useQuery defaults cause refetch on every window focus and component mount. Adding staleTime: 60_000 and gcTime: 300_000 to non-real-time queries (organizations, schools, teachers, sponsors) would eliminate redundant network calls when navigating between tabs, reducing API load meaningfully for a hosted Caribbean-region app with latency constraints.
