## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 114-116)
**Issue:** No rate limiting is applied to AI routes (chat, image, audio) before they are registered. These hit OpenAI APIs and are the most expensive endpoints in the app, making them trivial to abuse for cost exhaustion or DoS, especially critical for a school-aged user base.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); — place these lines immediately before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~lines 42 and 60)
**Issue:** The base URL is constructed directly from Host and X-Forwarded-Host/Proto request headers without any validation or allowlist. An attacker can send a crafted Host header and inject an arbitrary URL into the Sitemap directive and all <loc> tags, enabling cache-poisoning or open-redirect abuse.

**Fix:** Read the canonical base URL from a single env var (e.g. process.env.PUBLIC_BASE_URL) and fall back to a safe default. Remove all header-derived host/proto logic: const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com'; This is a one-line config change that eliminates the attack surface entirely.

---

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** Large base64 file content (fileB64) is stored directly in the job payload in the jobs table. For a multi-page PDF or high-res image this can easily be several MB per row, bloating the DB, slowing job queue queries, and causing memory pressure when the worker deserialises the payload.

**Fix:** Upload the file to object storage before enqueuing the job. Store only the object storage path in the payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of the handler. This keeps job rows small and makes the upload independently retryable.

---

### `server/jobHandlers.ts`: for loop inside runExtractPaper (~line with 'for (let i = 0; i < questions.length')
**Issue:** Extracted questions are inserted one at a time with sequential awaits inside a for loop. For a 40-question paper this is 40 serial round-trips to the database. A crash mid-loop leaves a partial set, and the idempotency delete at the top of the function only helps on a full retry, not a partial write within the same attempt.

**Fix:** Collect all valid question objects into an array, then do a single db.insert(extractedQuestions).values(allValidQuestions) call after the loop. This is atomic (all-or-nothing within one statement), eliminates N-1 round-trips, and removes the partial-write window entirely.

---

### `shared/schema.ts`: budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, documentUploads
**Issue:** userId is the primary filter column on every child table but has no index defined. Only transactions has a userId composite index. Every dashboard query loading budgets, goals, or holdings for a user will trigger a full table scan as data grows.

**Fix:** Add an index on userId for each table missing one, e.g. in budgets: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). For portfolioHoldings also add a unique composite index on (userId, stockId) since duplicate holdings rows are likely a bug.

---

### `shared/schema.ts`: simulatedStocks (ticker)
**Issue:** ticker has no unique constraint. Multiple rows with the same ticker symbol (e.g. two 'BTC' rows) will corrupt portfolio calculations silently — quantity lookups and price joins will return duplicate rows and double-count holdings.

**Fix:** Add a uniqueIndex to the table definition: (t) => ({ tickerUnique: uniqueIndex('idx_simulated_stocks_ticker_unique').on(t.ticker) }). If multi-region duplicate tickers are intentional, make the unique constraint composite on (ticker, region).

## 🟡 Warnings (7)

- **`server/routes.ts` (Supabase init chain (~lines 119-128)):** The sequential .then() chain means a failure in any intermediate step (e.g. seedFinancialAcademyLesson) causes all subsequent steps to be silently skipped, with only one generic error logged. In production this leads to partially initialised DB state with no visibility into which step failed.
  → *Run independent seed steps with Promise.allSettled so each result is individually inspectable: const results = await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]); results.forEach((r, i) => { if (r.status === 'rejected') console.error('[Supabase] Seed step', i, 'failed:', r.reason); }); Keep initSupabaseTables and trimOrgNamesInSupabase sequential before this block as they are prerequisites.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The filePath wildcard parameter is passed directly to objectStorage.searchPublicObject with no path traversal sanitisation. A request like /public-objects/../../private/keys could potentially escape the intended storage prefix depending on the ObjectStorageService implementation.
  → *Sanitise before use: const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); and assert it does not start with '..' or '/'. Add import path from 'path' at the top of the file.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch. If the DELETE throws (e.g. DB connection lost), the unhandled rejection will propagate to the job runner. Depending on the runner implementation this may retry the delete with the same cutoff, or mark the job failed with no status recorded. More importantly, the pattern is inconsistent with every other handler in the file which wraps DB calls.
  → *Wrap the db.delete call in try/catch, log the error with context (olderThanDays, cutoff date), and rethrow so the job runner can apply its retry/backoff policy. Also validate that olderThanDays is a positive integer before computing the cutoff to prevent a malformed payload deleting all rows.*
- **`server/jobHandlers.ts` (runExtractPaper – vision branch (~image_url construction)):** The raw base64 payload is re-encoded and sent to OpenAI with no size check. A 20 MB image will produce a ~27 MB base64 string embedded in a JSON request body, which will either hit OpenAI's request size limit and throw an unhandled error, or succeed but consume excessive tokens. There is no validation that the buffer is within acceptable bounds before the API call.
  → *Add a guard before the vision branch: if (buffer.length > 5 * 1024 * 1024) throw new Error('Image exceeds 5 MB limit for vision extraction'); Expose the limit as an env var (MAX_VISION_UPLOAD_BYTES) so it can be tuned without a deploy. Surface this as a 'failed' paper status via the existing terminal-attempt catch block.*
- **`shared/schema.ts` (categories (userId), transactions (categoryId, documentUploadId), billReminders (categoryId), portfolioHoldings (purchasedAt), portfolioTransactions (executedAt), documentUploads (createdAt), savingsGoals (createdAt)):** Multiple timestamp and FK columns use .defaultNow() or .default() without .notNull(), making them nullable at the DB level. createdAt being nullable means ORDER BY createdAt DESC returns unpredictable results for rows inserted without the default firing (e.g. bulk inserts bypassing Drizzle).
  → *Append .notNull() to all createdAt, purchasedAt, and executedAt columns. For nullable FKs that are intentionally optional (categoryId, documentUploadId) this is acceptable, but document the intent with a comment so future devs don't add .notNull() incorrectly.*
- **`shared/schema.ts` (categories (userId), transactions (userId), budgets (userId), linkedCards (userId) — all varchar('user_id') declarations):** varchar('user_id') has no length argument, which in PostgreSQL defaults to unbounded (equivalent to text), but the intent is clearly to match the auth provider's fixed-length user ID. If the auth model uses a fixed length (e.g. 36 for UUID, 28 for Firebase UID), silent mismatches in joins across tables are possible when different ORM versions serialize differently.
  → *Determine the auth provider's user ID max length (e.g. Firebase UIDs are 28 chars, UUIDs are 36). Change all declarations to varchar('user_id', { length: 36 }) (or appropriate length) and ensure the users.id column in models/auth uses the same length to guarantee join compatibility.*
- **`shared/schema.ts` (simulatedStocks (lastPriceUpdateDate)):** lastPriceUpdateDate is typed as text instead of timestamp. Storing dates as text breaks chronological sorting, range queries, and timezone handling. Any query like WHERE lastPriceUpdateDate > '2024-01-01' will do lexicographic not date comparison, silently returning wrong results.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). If the source data is a date-only value with no time component, use date type from drizzle-orm/pg-core instead.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (duplicate health check endpoints (~lines 33 and 71)):** Two health check routes exist: GET /api/health (line 33, no DB check) and GET /healthz (line 71, with DB check). Railway and uptime monitors should target one authoritative endpoint. The dead /api/health route gives a false positive when the DB is down.
- **`server/jobHandlers.ts` (parsePrivatePath / PRIVATE_OBJECT_DIR usage):** PRIVATE_OBJECT_DIR is checked at call-time inside parsePrivatePath, meaning a missing env var causes a runtime throw deep inside an async job rather than a startup failure. Other env-dependent values (appBaseUrl from email, openai key) follow the same late-failure pattern. For a school-facing production app this means misconfigured deployments fail silently on first use.

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six await registerXRoutes(app) calls after the parallel import block are executed sequentially. If any route registrar does async I/O at registration time (e.g. fetching config), this adds latency. Refactor to pass app into each import Promise and register inside the module, or use Promise.all on registration too. Expected impact: reduced cold-start time by the sum of any async work done in individual registrars.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replacing console.error with a structured logger (e.g. pino) that emits JSON including step name, duration, and error code would make Railway log aggregation and alerting actionable. Currently a single string makes it impossible to set up metric-based alerts on seed failures without log parsing.

### Cache-Control headers on public object endpoint: 📌 Low Impact
The /public-objects/:filePath route passes a 30-day TTL to downloadObject but does not set Cache-Control or ETag headers at the Express layer. Add res.setHeader('Cache-Control', 'public, max-age=2592000, immutable') before streaming so CDN and browser caches are correctly instructed, reducing origin load for logo/video assets served to students.

### Parallel independent DB lookups in report-generation handlers: 🔥 High Impact
Handlers that fetch user XP, badges, and learning progress for report generation likely do so sequentially. Wrapping independent queries in Promise.all would reduce report generation latency proportionally to the number of independent queries, typically 2-4x for a 3-query sequence on a 50 ms DB connection.

### Job payload schema validation at handler entry: ⚡ Medium Impact
No handler validates its payload shape before using it. Adding zod schemas per job type (e.g. z.object({ paperId: z.number(), ext: z.enum(['.pdf','.png','.jpg']) })) parsed at the top of each handler would catch malformed enqueue calls immediately with a clear error rather than a confusing mid-handler TypeError, and prevent prototype-pollution style payload abuse.

### Centralised AI call wrapper with token usage logging: ⚡ Medium Impact
openai.chat.completions.create is called in at least two places inside this file with no logging of prompt_tokens or completion_tokens from the response. Wrapping all OpenAI calls in a thin helper that records usage to aiUsageEvents would give visibility into per-job AI cost, which is directly relevant for a cost-constrained school app and would enable the existing purge-ai-usage job to be more targeted.

### Partition or archive transactions by date: 🔥 High Impact
For school-aged users accumulating daily transactions, the transactions table will become the largest table. Adding a createdAt-based index and documenting a soft-archive strategy (e.g. is_archived boolean with partial index WHERE is_archived = false) now will avoid costly table rewrites later when the table reaches millions of rows.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least 6 tables. If the app expands to TTD or JMD markets, each table needs a separate migration. Extract a DEFAULT_CURRENCY constant in a shared config and reference it in schema definitions to make currency changes a one-line update.

### Add composite index on portfolioTransactions (userId, stockId, executedAt): 🔥 High Impact
Portfolio performance history queries will always filter by userId and stockId then order by executedAt. Without this composite index, each trade history load requires a full userId scan followed by an in-memory sort. This index directly accelerates the most data-intensive user-facing feature.

### API Routes (`server/routes.ts`)

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several meaningful reliability and security gaps. The most serious are: large base64 file payloads stored directly in the job table (memory and DB pressure), sequential DB inserts inside a loop that should be batched, and missing try/catch on the AI usage purge handler which can silently corrupt retry state. Security is also a concern with base64 image data echoed back into OpenAI requests without size validation.

The investment simulation tables are entirely unindexed despite being join-heavy.
