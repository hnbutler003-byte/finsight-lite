## 🔴 Critical (6)

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 44-80)
**Issue:** Host header injection: `req.headers['x-forwarded-host']` and `req.headers.host` are attacker-controlled in environments without a trusted reverse proxy. A crafted Host header produces a poisoned Sitemap URL and base URL in the sitemap, which can be used for cache poisoning or phishing.

**Fix:** Read the canonical base URL from a single environment variable (e.g. `process.env.APP_BASE_URL`) set at deploy time. Remove all runtime header-derived base URL logic. Example: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';`

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 113-115)
**Issue:** No rate limiting is applied before registering AI/image/audio routes. These endpoints call OpenAI APIs which are expensive and abuse-prone, especially for a school-aged audience. Any authenticated (or unauthenticated) user can spam them without restriction.

**Fix:** Apply a dedicated AI rate limiter before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where `aiLimiter` is an express-rate-limit instance with a conservative window (e.g. 10 req / 60s per IP). Add this to the existing rateLimiter module.

### `shared/schema.ts`: simulatedStocks table, lastPriceUpdateDate column
**Issue:** lastPriceUpdateDate is typed as text instead of timestamp. Storing dates as text prevents range queries, ordering, and timezone-aware comparisons. Any WHERE lastPriceUpdateDate > X query will do lexicographic string comparison, silently returning wrong results.

**Fix:** Change to: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any insertion code that writes ISO strings to pass Date objects instead.

---

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** The day-of-week rollback logic uses `(dow + 0) % 7` which is a no-op addition of zero, meaning Sunday (dow=0) produces daysSince=0 and the date is never rolled back, so the function returns the current Sunday at 19:00 even when today is not Sunday. This causes the weekly digest to never correctly identify the most recent past Sunday when called on a non-Sunday day where dow=0 at midnight.

**Fix:** Replace `(dow + 0) % 7` with `dow === 0 ? 7 : dow` so Sunday is treated as 7 days back from itself, giving the correct most-recent-past Sunday: `const daysSince = dow === 0 ? 7 : dow;`

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The two `setInterval` handles are never stored and are not cleared in the graceful shutdown handler. When the process receives SIGTERM/SIGINT the intervals keep firing during the drain window, potentially enqueuing duplicate jobs or running DB writes against a closing connection pool.

**Fix:** Capture both handles: `const weeklyDigestInterval = setInterval(...)` and `const purgeInterval = setInterval(...)`. In the shutdown handler, call `clearInterval(weeklyDigestInterval)` and `clearInterval(purgeInterval)` before awaiting `httpServer.close()`.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / top-level dashboard render
**Issue:** If useAdminAuth returns a non-admin or unauthenticated state, there is no early return or redirect before the heavy useQuery calls and full component tree are evaluated. All admin data queries fire regardless of auth state, leaking request volume and potentially receiving and rendering sensitive data before the redirect resolves.

**Fix:** Immediately after calling useAdminAuth, add: if (!isAdmin || isLoading) return isLoading ? <Loader2 /> : <Redirect to='/login' />; before any useQuery hooks. This ensures queries only mount when auth is confirmed and eliminates the window where unauthenticated renders occur.

## 🟡 Warnings (12)

- **`server/routes.ts` (Supabase init chain (~line 118-126)):** The seven sequential `.then()` Supabase seed calls are fire-and-forget with only a terminal `.catch`. If any intermediate step (e.g. `seedFinancialAcademyLesson`) throws, subsequent seeds are silently skipped and the app starts with incomplete data, with no way to distinguish which step failed from the single error log.
  → *Wrap each call individually and log per-step failures, or run independent seeds in parallel: `Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]).catch(e => console.error('[Supabase] Seed error:', e))`. For truly sequential dependencies, log the failing step name explicitly inside a named async IIFE so operators know exactly where startup data is missing.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100-109)):** No authentication or authorization check on the public-objects endpoint. While named 'public', it proxies arbitrary paths from object storage. A path traversal or enumeration attack could expose unintended files (e.g. internal org assets addressed by guessable names).
  → *Add an allowlist of permitted path prefixes (e.g. `['logos/', 'videos/']`) and reject requests that don't match: `if (!ALLOWED_PREFIXES.some(p => filePath.startsWith(p))) return res.status(403).json({ error: 'Forbidden' });`. Additionally sanitize `filePath` with `path.normalize` and reject paths containing `..`.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet every budget query will filter by userId. At scale this becomes a full table scan per user dashboard load.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget rows for the same category and period.*
- **`shared/schema.ts` (savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these tables have indexes on userId, but all will be queried exclusively by userId in the app. savingsGoals and billReminders also query on deadline/nextDueDate for reminder scheduling, with no index on those columns.
  → *Add composite indexes: for savingsGoals add index on (userId, deadline); for billReminders add index on (userId, nextDueDate); for portfolioHoldings and portfolioTransactions add index on (userId, stockId). Pattern: (t) => ({ userIdx: index('idx_savings_goals_user').on(t.userId, t.deadline) })*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** All audit/timestamp columns use .defaultNow() without .notNull(). Drizzle will type these as nullable (Date | null), meaning application code must null-check every audit timestamp and ORDER BY createdAt queries can behave unexpectedly with NULLs sorting first or last depending on the DB.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). This costs nothing at the DB level since the default is always applied.*
- **`shared/schema.ts` (categories.userId, transactions.userId, budgets.userId (and all other userId columns)):** All userId columns are declared as varchar() with no length argument. In PostgreSQL, bare varchar without length is equivalent to text, which is fine, but it signals no intent and loses the self-documenting constraint. More critically, if the auth provider (e.g. Clerk) uses UUIDs (36 chars) or Supabase-style IDs, there is no schema-level enforcement and a misconfigured auth integration could insert truncated IDs silently.
  → *Determine the exact ID format from your auth provider (e.g. Clerk UIDs are 32-char prefixed strings, UUIDs are 36 chars) and set explicit length: varchar('user_id', { length: 36 }) or switch to text() deliberately. Add a CHECK constraint or use uuid type if IDs are UUIDs.*
- **`server/index.ts` (res.json middleware / logLine construction in res.on('finish')):** Every API response body is captured via monkey-patching `res.json` and then serialized with `JSON.stringify(capturedJsonResponse)` unconditionally on every request. For endpoints returning large datasets (quiz results, digest payloads) this doubles serialization cost and buffers the entire response object in the closure until the response finishes. For a school-age financial app this also risks logging tokens, grades, or guardian PII.
  → *Guard the body capture behind a flag or size limit, e.g. only log if `process.env.NODE_ENV !== 'production'`, or truncate: `logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 200)}`;`. For production, log only status code and duration.*
- **`server/index.ts` (seedDatabase and probeDatabase calls at startup (inside async IIFE or top-level await)):** Both `seedDatabase()` and `probeDatabase()` are awaited sequentially at startup before the HTTP server begins listening. These are independent operations and each involves DB round-trips; running them sequentially adds their combined latency to startup time, delaying readiness probes in containerised or Replit deployments.
  → *Run them in parallel: `await Promise.all([probeDatabase(), seedDatabase()])`. Ensure neither function has side-effects that depend on the other completing first; if seed depends on schema existence verified by probe, chain only those two steps and parallelise everything else (e.g., `registerJobHandlers`, `startJobWorker`).*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and related scheduler):** The purge threshold is read once at module load into a module-level constant, which is fine, but the scheduler that uses it calls `maybeRunAiUsagePurge` on a fixed interval without any concurrency guard. If a previous purge job is still running (slow DB delete of old rows) when the interval fires again, a second overlapping purge will start, causing redundant large deletes and potential lock contention on the ai_usage table.
  → *Add a boolean lock: `let purgeRunning = false;`. At the top of `maybeRunAiUsagePurge`, return early if `purgeRunning` is true; set it to `true` on entry and reset to `false` in a `finally` block. Apply the same pattern to `maybeRunWeeklyDigest` if enqueue latency can exceed the check interval.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat / sendMessage):** No AbortController or component-mounted guard on the fetch inside sendMessage. If the chat panel is closed while a request is in flight, setMessages and setLoading fire on an unmounted component, causing a state-update-on-unmounted-component warning and potential stale-closure bugs.
  → *Add a useEffect cleanup with an AbortController: const controller = new AbortController(); pass { signal: controller.signal } to apiRequest, and in the useEffect cleanup call controller.abort(). Gate setMessages/setLoading calls on a mounted ref (const mounted = useRef(true); useEffect(() => () => { mounted.current = false; }, [])).*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat messages list render / inline ref callback):** The scroll-to-bottom anchor uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` directly in JSX. This callback is recreated on every render, calling scrollIntoView on every state change (including loading toggle), causing janky scroll behaviour and unnecessary DOM thrashing.
  → *Replace with a stable useRef + useEffect: const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]); then render <div ref={bottomRef} /> with no inline logic.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple useQuery declarations across dashboard sections):** Several useQuery calls for related admin data (users, schools, teachers, analytics) appear to be declared unconditionally and in sequence without staleTime configuration, meaning every tab switch or parent re-render triggers background refetches for all data sets simultaneously, including tabs not currently visible.
  → *Add staleTime: 60_000 (or appropriate TTL) to each useQuery options object. For queries only needed in specific tabs, add enabled: activeTab === 'users' (etc.) so inactive tab queries are suspended. This can reduce network calls by 60-80% during normal navigation.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (registerRoutes function — duplicate health check endpoints (~line 31 and ~line 83)):** Two health check endpoints exist: `/api/health` (returns immediately, no DB check) and `/healthz` (performs a DB query). Railway and uptime monitors will be pointed at one or the other, creating confusion. The `/api/health` endpoint gives a false positive if the DB is down.
- **`client/src/pages/AdminDashboard.tsx` (Large inline handler functions in JSX (Trash2/Pencil onClick handlers)):** Delete and edit action handlers for table rows are defined as inline arrow functions inside the JSX map callback, e.g. onClick={() => handleDelete(user.id)}. In a table rendering hundreds of school-aged user records, this creates O(n) new function instances on every render, preventing React from bailing out of row re-renders via memo.

## 💡 Opportunities

### Parallel Supabase seed execution: ⚡ Medium Impact
The seven Supabase init calls run sequentially via chained `.then()`, meaning startup data hydration takes 7× the average call latency. Group independent seeds (modules, regional content, game content) into a single Promise.all after the prerequisite table-init steps complete. Expected impact: reduce cold-start data-init time by 60-70%.

### Centralized APP_BASE_URL config: 🔥 High Impact
Both robots.txt and sitemap.xml independently reconstruct the base URL from request headers using duplicated logic. Extract this into a single `getBaseUrl(req)` utility or, better, a deploy-time `APP_BASE_URL` env var consumed once at startup. This eliminates the host injection risk and removes ~12 lines of duplicated code.

### Domain route registration parallelization: 📌 Low Impact
The six domain routers are imported in parallel via Promise.all (good), but then registered sequentially with six serial awaits. If any registrar does async work (DB schema checks, etc.), this adds latency. Refactor to `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` assuming routes don't depend on each other's registration order.

### Unique constraint on simulatedStocks ticker column: 🔥 High Impact
The ticker column has no unique index, so duplicate tickers can be inserted, causing portfolio calculations to join against multiple rows and silently double-count positions. Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table definition. This also enables safe upsert-on-conflict patterns for price update jobs.

### Centralise currency default to an environment constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least 6 separate table columns (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, portfolioHoldings). If the app expands to TTD or JMD markets, each column default must be hunted down individually. Extract to a shared constant DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' and reference it in every .default() call.

### Add partial index for unread/pending document processing queries: ⚡ Medium Impact
documentUploads will be polled frequently for status = 'processing' rows (to drive upload progress UI). A full index on (userId, status) is wasteful since 'completed' rows dominate over time. Add a partial index: index('idx_document_uploads_processing').on(t.userId).where(sql`status = 'processing'`). This keeps the index small and fast as the table grows.

### Structured JSON logging instead of console.log string interpolation: ⚡ Medium Impact
Replace the `log()` helper's `console.log` string with `console.log(JSON.stringify({time, source, message}))`. This makes logs parseable by Replit's log drain or any future SIEM without regex, and eliminates the risk of newline injection from user-controlled message content appearing in log lines.

### Centralise scheduler boot into a single startSchedulers() function: ⚡ Medium Impact
Extract both `setInterval` registrations and their initial eager calls into a single `startSchedulers(): NodeJS.Timeout[]` function that returns all handles. The shutdown handler then receives the array and clears all of them in one loop. This removes the current scattered interval setup and makes adding future schedulers safe by construction.

### Add a /healthz liveness probe endpoint before other middleware: 📌 Low Impact
Register a `GET /healthz` route that returns 200 immediately, before session, Sentry, and body-parser middleware. This lets Replit's health checker and any load balancer confirm the process is alive without the overhead of the full middleware stack, and decouples liveness from DB connectivity (use a separate /readyz for DB checks).

### Split into feature-scoped sub-pages: 🔥 High Impact
The file is over 147k characters, indicating Users, Schools, Teachers, Analytics, and Chat are all co-located. Extract each tab into its own lazy-loaded page component (React.lazy + Suspense). This reduces initial bundle parse time, enables per-section code splitting, and makes each section independently testable. Expected impact: 30-50% reduction in initial JS execution time on low-end Caribbean school devices.

### Centralise hardcoded API path strings into a shared constants module: ⚡ Medium Impact
Paths like '/api/admin/help-chat', '/api/admin/users', etc. appear as string literals throughout the file. Move them to a single client/src/lib/apiRoutes.ts constants object. This prevents typo-driven 404s, makes base-URL migration trivial, and is required before adding request signing or per-environment routing.

### Add React.memo to pure stat card and chart components: ⚡ Medium Impact
BarChart and summary Card components that receive only primitive props re-render on every parent state change (e.g. search input keystrokes). Wrapping them in React.memo with explicit prop comparison will eliminate recharts re-layout calculations on unrelated state updates, visibly reducing jank during search filtering on slow devices.
