## 🔴 Critical (13)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query is fired to sum transactions for that budget's category. With 20 budgets this is 21 DB round-trips.

**Fix:** Replace the per-budget spent lookup with a single aggregated query using GROUP BY categoryId joined to the budgets result set, then merge in JS. Example: `db.select({ categoryId: transactions.categoryId, spent: sql<number>`sum(amount)` }).from(transactions).where(eq(transactions.userId, userId)).groupBy(transactions.categoryId)` then Map the results and attach to each budget.

### `server/storage.ts`: getLeaderboard
**Issue:** Leaderboard query joins user display names and avatars and returns them without any caller-supplied auth scope, meaning any authenticated user can enumerate all userIds, userNames, and avatars for the entire platform. For a school-aged user base this is a COPPA/child-privacy risk.

**Fix:** Add a required `classId` or `schoolId` scope parameter so the leaderboard only returns peers within the student's own class or school. Apply a `where(eq(gameSessions.classId, classId))` filter before aggregation, and strip raw userId from the public payload, replacing it with an opaque rank-scoped token.

### `server/storage.ts`: getTransactions / getDashboardStats
**Issue:** getTransactions has an optional `limit` filter but no enforced maximum, and getDashboardStats calls getTransactions without any limit. On an account with thousands of rows this loads the full table into Node process memory.

**Fix:** Enforce a hard cap: `const safeLimit = Math.min(filters?.limit ?? 200, 500)` and always pass it to the Drizzle query via `.limit(safeLimit)`. For getDashboardStats, use a dedicated aggregation SQL query (SUM/COUNT with WHERE) instead of fetching all transaction rows into memory.

### `server/storage.ts`: upsertPortfolioHolding / updateVirtualBalance (buy/sell flow)
**Issue:** The buy/sell path calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three separate awaits with no wrapping database transaction. A process crash between any two steps leaves the portfolio holding updated but the virtual balance unchanged, or vice versa — corrupting the simulated financial state.

**Fix:** Wrap all three operations in a single Drizzle transaction: `await db.transaction(async (tx) => { await tx.insert/update portfolioHoldings...; await tx.insert portfolioTransactions...; await tx.update userVirtualBalance... })`. All three tables must be mutated atomically.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 107)
**Issue:** AI and image/audio routes are registered with no rate limiting. For a school-aged user base these are the most expensive endpoints (OpenAI API costs, latency) and are trivially abusable without auth or rate guards.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/15 min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 44) and sitemap.xml handler (~line 57)
**Issue:** The `host` header is taken directly from `req.headers.host` without validation and interpolated into response bodies. A forged Host header causes the Sitemap URL and all sitemap <loc> values to reflect attacker-controlled content, enabling cache-poisoning and open-redirect attacks.

**Fix:** Add a validated SITE_BASE_URL environment variable: `const base = process.env.SITE_BASE_URL ?? \`${proto}://${host}\`;` and reject requests where the derived host does not match an allowlist. At minimum: `const ALLOWED_HOSTS = new Set((process.env.ALLOWED_HOSTS ?? '').split(',').filter(Boolean));` and return 400 if the host is not in the set.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter / enqueueJob call site)
**Issue:** The full base64-encoded file is stored in the job payload in the DB job table. A 5MB PDF becomes ~6.7MB of base64 JSON in a single row, repeated across retries. This bloats the jobs table, slows queue polling queries, and risks exceeding row/column size limits on most databases.

**Fix:** Upload the file to object storage before enqueuing, store only the object storage path in the payload (e.g. { paperId, objectPath, ext, subject }), then download it inside the handler with objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts`: runExtractPaper – for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted one at a time inside a sequential for loop with individual awaits. For a 40-question paper this fires 40 serial round-trips to the DB, dramatically increasing handler latency and holding the job slot longer than necessary.

**Fix:** Replace the loop with a single batched insert: filter valid questions first, then call db.insert(extractedQuestions).values(validQuestions.map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0,4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i+1 }))); This reduces N round-trips to 1.

### `server/index.ts`: startSchedulers / setInterval calls
**Issue:** setInterval handles for the weekly digest and AI purge schedulers are never stored and never cleared. On SIGTERM/SIGINT the intervals keep firing against a closing DB connection, causing unhandled rejections and preventing the process from exiting cleanly.

**Fix:** Store each handle: `const digestTimer = setInterval(maybeRunWeeklyDigest, 60_000)`. In the graceful-shutdown handler call `clearInterval(digestTimer)` and `clearInterval(purgeTimer)` before closing the HTTP server and DB pool. Register shutdown on both SIGTERM and SIGINT.

### `server/index.ts`: main() startup sequence — probeDatabase, seedDatabase, registerRoutes, startJobWorker calls
**Issue:** probeDatabase, seedDatabase, and startJobWorker are awaited sequentially before httpServer.listen() is called. A slow seed or DB probe (network hiccup) blocks the port from opening, causing load-balancer health checks to fail and Replit to kill the dyno before it is ever marked healthy.

**Fix:** Bind the HTTP server first (`await new Promise(r => httpServer.listen(port, r))`), then run probeDatabase + seedDatabase in parallel via `await Promise.all([probeDatabase(), seedDatabase()])`, then startJobWorker. Return 503 from a /healthz route until the DB probe succeeds.

---

### `server/index.ts`: res.json logging middleware (~line 68)
**Issue:** Every API response body is captured via monkey-patching res.json and serialized in full to stdout. For FinSight Lite this will log student quiz scores, guardian contact details, AI usage records, and session-adjacent data — a compliance and data-exposure risk, and a CPU/memory cost for large payloads.

**Fix:** Remove `capturedJsonResponse` capture entirely, or gate it behind `process.env.LOG_RESPONSE_BODIES === 'true'` and truncate to 200 chars max: `JSON.stringify(capturedJsonResponse).slice(0, 200)`. Never enable in production.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat / sendMessage
**Issue:** The full `messages` array (including all prior turns) is sent to `/api/admin/help-chat` on every request. For a school-aged platform this leaks the entire session conversation to the server on each call, grows the payload unboundedly, and if logs are verbose exposes all user inputs. There is also no max-length guard, so a long session will eventually blow past any server body-size limit.

**Fix:** Send only the last N turns (e.g. 10) to bound payload and limit exposure: `messages: next.slice(-10)`. Add a server-side guard too. If full history is needed for context, summarise older turns server-side rather than transmitting raw.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat / sendMessage (missing auth check)
**Issue:** The help-chat POST to `/api/admin/help-chat` is fired directly from the component without verifying that the admin session is still valid before the call. If the session has expired the request will likely fail with a 401 and only a generic 'something went wrong' message surfaces, with no redirect to login.

**Fix:** Check the result of `useAdminAuth` (which is already imported) before calling `sendMessage`. If `!isAdmin` is true, either disable the chat UI entirely or redirect to login. On the server route, ensure the admin middleware returns a structured 401 JSON so the client can detect and redirect rather than swallowing the error.

## 🟡 Warnings (15)

- **`server/storage.ts` (seedMarketData / seedLearningModules):** Both seed functions use hardcoded stock tickers, prices, module titles, and XP values baked directly into the source file. Any change requires a code deploy, and the values differ from what should be environment- or config-driven for a multi-school/multi-sponsor deployment.
  → *Move seed data to JSON config files (e.g. `config/market-seed.json`, `config/modules-seed.json`) loaded at runtime with `JSON.parse(fs.readFileSync(...))`. For environment-specific overrides (e.g. TTD vs USD default currency), read from `process.env.DEFAULT_CURRENCY ?? 'USD'`. This also makes sponsor-specific content injectable without redeploy.*
- **`server/routes.ts` (Supabase init chain (~line 111)):** The seven Supabase init steps are chained sequentially with `.then()`. A failure in `seedFinancialAcademyLesson` causes all subsequent steps (static modules, regional content, game content) to be silently skipped, leaving the DB in a partially seeded state with no per-step visibility.
  → *Replace the chain with `Promise.allSettled` where steps are independent, or use an explicit async IIFE with individual try/catch per step so each failure is logged and subsequent steps still run: `(async () => { for (const [name, fn] of steps) { try { await fn(); } catch (e) { console.error(`[Supabase] ${name} failed:`, e); } } })();`*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97)):** There is no authentication check on this endpoint. Any unauthenticated user can enumerate and download arbitrary objects from storage by guessing or brute-forcing filePaths. For an app storing school-aged user assets this is a privacy/COPPA risk.
  → *For non-public assets, require a valid session before serving: add a middleware guard that checks `req.isAuthenticated()` (or equivalent session check) and returns 401 otherwise. Truly public assets (marketing logos) should be served from a CDN with a fixed path prefix, not a catch-all proxy.*
- **`server/jobHandlers.ts` (purge-ai-usage handler – olderThanDays from job.payload):** The purge cutoff age is passed in the job payload rather than read from environment config, meaning whoever enqueues the job controls how far back records are deleted. A misconfigured or malicious enqueue call could purge all AI usage history. There is also no validation that olderThanDays is a positive integer.
  → *Read the retention period from an environment variable (e.g. AI_USAGE_RETENTION_DAYS, defaulting to 90) inside the handler and ignore the payload field. Add a guard: if (!Number.isInteger(days) || days < 1) throw new Error('Invalid retention period'). Remove olderThanDays from the job payload type.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** The magic number 30000 and the token cap 4096 are hardcoded literals with no named constant or environment override. If the OpenAI model or context window changes, these must be hunted down by grep rather than updated in config.
  → *Define named constants at module top or read from env: const MAX_EXTRACTED_CHARS = parseInt(process.env.MAX_EXTRACTED_CHARS ?? '30000', 10); const MAX_COMPLETION_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10); Reference these in both openai.chat.completions.create calls and the slice.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, but queries filtering a user's budgets will do full table scans as the dataset grows. Unlike transactions, no composite index is defined.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). If a user can only have one budget per category per period, also add a uniqueIndex on (userId, categoryId, period) to enforce that constraint at the DB level.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker has no unique index. Duplicate tickers can be inserted silently, causing incorrect portfolio valuations and confusing JOIN results when looking up stock prices by ticker symbol.
  → *Replace the plain text column with: ticker: text('ticker').notNull(), and add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table config. Also add .notNull() if not already present on all display-critical fields like name and description.*
- **`shared/schema.ts` (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders — createdAt columns):** createdAt uses .defaultNow() but no .notNull(), meaning application code or a direct INSERT omitting that column produces a NULL timestamp. Audit queries ordered by createdAt will silently misorder rows with NULL values.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Same applies to portfolioHoldings.purchasedAt and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (userId columns across all tables (varchar with no length argument)):** All userId foreign key columns are declared as varchar('user_id') with no length parameter. PostgreSQL defaults this to varchar with no limit (equivalent to text), losing the intent to match a fixed-length auth ID (e.g. UUID = 36 chars or CUID = 25 chars). This allows arbitrarily long strings to be stored without DB-level rejection.
  → *Determine the exact length of the auth provider's user ID (e.g. 36 for UUID, 25 for CUID) and declare it explicitly: varchar('user_id', { length: 36 }). Apply consistently to every table. This enforces the contract at the DB layer and enables more efficient index storage.*
- **`shared/schema.ts` (billReminders and portfolioHoldings tables):** billReminders has no index on userId or nextDueDate, despite being the primary columns for the most common query pattern: 'fetch upcoming bills for this user ordered by due date'. portfolioHoldings similarly lacks an index on userId and stockId for portfolio lookups.
  → *For billReminders add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). For portfolioHoldings add: (t) => ({ userStockIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) — the unique constraint also prevents duplicate holding rows per user per stock.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** `d.getDay()` returns the day in the **local** timezone of the server process, but `d.setHours(19,0,0,0)` also uses local time. On a UTC Replit dyno the 'most recent Sunday 7 pm' is computed in UTC, so the digest fires at 7 pm UTC regardless of the Caribbean timezone (AST = UTC-4), meaning it actually runs at 3 pm local time and the idempotency key `weekStart` is date-stamped in UTC, silently breaking weekly deduplication across the DST boundary.
  → *Compute the target in a fixed offset: convert `now` to AST by subtracting 4 hours (`new Date(now.getTime() - 4*3600*1000)`), derive the Sunday date string from that, then reconstruct the UTC trigger time as `new Date(Date.UTC(y, m, d, 23, 0, 0))` (7 pm AST = 11 pm UTC). Store `weekStart` as the AST date string.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest interval setup):** Both scheduler functions are invoked immediately at startup via a bare call (before the DB is confirmed ready) AND via setInterval. If the DB probe has not completed yet, the first immediate invocation will throw and the error is silently swallowed by the outer try/catch, potentially leaving a partial weeklyDigestRuns insert with no corresponding job.
  → *Do not call the scheduler functions immediately at startup. Let the first interval tick be the earliest invocation, scheduled only after `await probeDatabase()` confirms the DB is reachable. If an on-boot run is required, await it explicitly after the DB is ready and before starting the interval.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat JSX / scroll div ref callback):** `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is an inline arrow function created on every render. It fires `scrollIntoView` every time any state changes in the parent (e.g. while `loading` toggles), not just when a new message arrives, causing janky scroll interruptions mid-typing.
  → *Replace with a `useRef` + `useEffect` that only triggers on `messages.length` change: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]); ... <div ref={bottomRef} />`*
- **`client/src/pages/AdminDashboard.tsx` (useEffect blocks fetching dashboard data (sequential awaits pattern)):** Multiple `useQuery` hooks are used throughout the dashboard but any `useEffect` blocks that perform sequential `await apiRequest(...)` calls (e.g. fetching users then schools then courses one after another) add unnecessary latency. For a real-time admin dashboard this compounds load time on every tab switch.
  → *Replace sequential awaits with `Promise.all`: `const [users, schools, courses] = await Promise.all([apiRequest(...), apiRequest(...), apiRequest(...)]);`. If these are already separate `useQuery` calls with no data dependency between them, ensure they are not nested inside a single `useEffect` that gates each on the prior result.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat messages state / setMessages pattern):** `setMessages(next)` is called with a snapshot variable built from `messages` state before the async call, and then `setMessages(prev => [...prev, ...])` is used in the callback. This is inconsistent: if React batches an intermediate render between the two set-calls the optimistic `next` snapshot may be stale, causing the user message to appear duplicated or out of order.
  → *Use the functional updater form consistently throughout: `setMessages(prev => [...prev, { role: 'user', content: text }]);` and capture the value to send via `const payload = [...messages, { role: 'user', content: text }]` before any state update, passing `payload` to the API call. This eliminates the stale-closure risk.*

## 💡 Opportunities

### Parallelize independent async lookups with Promise.all: 🔥 High Impact
getDashboardStats sequentially awaits transactions, budgets, and savings goals that have no data dependency on each other. Replacing with `const [txns, budgets, goals] = await Promise.all([...])` would cut dashboard load latency by roughly 60-70% on typical DB round-trip times of 20-40ms each.

### Introduce a repository-layer pagination contract: 🔥 High Impact
All list methods (getTransactions, getGameSessions, getExamPapers, getEnrollmentsByClass) return unbounded arrays. Adding a standard `{ page: number; pageSize: number }` parameter with a shared `applyPagination(query, opts)` helper enforces limits uniformly and prevents memory spikes as the school user base grows.

### Add structured error wrapping to all storage methods: ⚡ Medium Impact
No storage method wraps DB calls in try/catch, so raw Drizzle/postgres error messages (including table names, column names, and query fragments) propagate unfiltered to API responses. Wrapping each method with a shared `withStorageError(fn, context)` utility that logs internally and throws a sanitized StorageError prevents leaking schema details to clients.

### Parallel Supabase seed execution for independent steps: ⚡ Medium Impact
Steps like seedStaticModules, seedRegionalContent, and seedGameContent are likely independent. Running them with Promise.allSettled after initStaticContentTables completes could cut server cold-start time by 2-4 seconds depending on Supabase latency.

### Structured startup logging for route registration: ⚡ Medium Impact
Currently there is no logging when domain routers are successfully registered. Adding a startup summary log (route group name + duration) would make cold-start traces in Railway/Sentry far easier to diagnose when a specific module import hangs or fails silently.

### Move ObjectStorageService instantiation inside request scope or use singleton module: 📌 Low Impact
The `objectStorage` instance is created at module load time in the outer scope. If ObjectStorageService opens connections or allocates resources on construction, a failed import will silently crash startup. Move to a lazy singleton with explicit error handling so startup failures surface clearly.

### Batch object-storage existence checks in streamPrivateObjectToResponse: ⚡ Medium Impact
If multiple files are ever checked in a request cycle, each triggers a separate file.exists() RPC. Introducing a lightweight in-process LRU cache (e.g. lru-cache with a 60-second TTL) for recently confirmed paths would eliminate redundant storage RPCs on repeated download requests for the same export file, reducing p99 latency on the export-download endpoint.

### Idempotency token for OpenAI calls inside extract-paper: 🔥 High Impact
If the handler crashes after the first OpenAI call but before the second, a retry re-runs both API calls, doubling cost and latency. Caching the first call's result (e.g. storing extractedText in object storage keyed by paperId + attempt) and checking for it on entry would make retries skip the expensive vision/PDF call and proceed directly to question extraction, saving ~50% of AI spend on failed attempts.

### Move admin-csv-export row fetching to a streaming cursor: 🔥 High Impact
Loading all rows for a CSV export into memory at once (implied by the truncated admin-csv-export handler pattern) risks OOM for large teacher/student datasets. Replacing the bulk fetch with a Drizzle cursor or paginated loop writing directly to a stream that pipes into object storage upload would cap memory use to a single page of rows regardless of dataset size.

### Centralise currency default into a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least 6 separate table columns (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, portfolioHoldings). Extract to a single exported constant DEFAULT_CURRENCY = 'BSD' in a config file and reference it in each .default() call. This prevents silent drift if the default ever needs to change and makes it searchable.

### Add a composite unique constraint on budgets (userId, categoryId, period): ⚡ Medium Impact
Without this constraint, application bugs or race conditions can insert duplicate budget rows for the same user, category, and period. A DB-level unique constraint catches this regardless of app logic and removes the need for defensive deduplication in query results.

### Use pgEnum for repeated text enum types: ⚡ Medium Impact
The type enum ['income','expense'], risk_level, frequency, and status enums are defined inline on multiple columns. Extracting them to pgEnum declarations (e.g. export const transactionTypeEnum = pgEnum('transaction_type', ['income','expense'])) enforces the enum at the Postgres type level, enables Drizzle to generate proper enum migrations, and makes the values reusable across tables without duplication.

### Parallel startup I/O with structured readiness gating: 🔥 High Impact
probeDatabase and seedDatabase have no data dependency on each other. Running them with Promise.all shaves their combined latency down to max(probe, seed) instead of probe+seed. On a cold Replit container with a remote Neon DB this is typically 1-3 seconds saved before the port opens, directly reducing health-check failure risk.

### Centralized interval registry for clean shutdown: ⚡ Medium Impact
Introduce a small `timerRegistry: NodeJS.Timeout[]` array. Push every setInterval handle into it at creation. The single SIGTERM handler iterates the array calling clearInterval on each, then drains the job worker queue before closing the HTTP server. This eliminates the timer-leak class of bugs for any future schedulers added to the file.

### Replace full response-body logging with structured request telemetry: ⚡ Medium Impact
Instead of serializing response bodies, emit a structured log object with method, path, statusCode, durationMs, and requestId only. This removes the data-exposure risk, cuts per-request CPU for JSON.stringify on large payloads, and produces logs that are trivially parseable by Replit's log drain or a future observability tool.

### Memoize chart data transforms: ⚡ Medium Impact
Recharts BarChart and LineChart data passed as props appears to be derived inline from query results on every render. Wrapping these derivations in `useMemo` keyed on the raw query data will prevent expensive array transforms during unrelated re-renders (e.g. search input changes), which is noticeable on lower-end devices common in Caribbean school settings.

### Centralise hardcoded API base paths: ⚡ Medium Impact
String literals like `/api/admin/help-chat` and similar admin route prefixes are scattered across the file. Moving these to a typed constants module (or reading from `import.meta.env.VITE_API_BASE`) makes environment-specific deployments (staging vs production) safe and eliminates silent routing bugs when the prefix changes.

### Add loading and empty states to all admin data tables: 🔥 High Impact
If `useQuery` is still fetching or returns an empty array, tables silently render with zero rows. For admin users managing schools and teachers this looks like a data loss bug. Adding explicit skeleton rows during loading and a clear empty-state message with a CTA (e.g. 'No teachers yet — add one') dramatically reduces support requests from school administrators.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits several critical reliability and security issues including N+1 query patterns in budget and leaderboard lookups, missing database transactions on multi-step writes, and sequential awaits that should be parallelized. For a school-aged Caribbean fintech app, the leaderboard exposing userId/userName without auth scoping and the lack of pagination guards on unbounded list queries are the most pressing concerns.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good patterns like parallel dynamic imports and rate limiting on auth endpoints. Key concerns are: unprotected AI/heavy routes without rate limiting, a host-header injection risk in robots.txt and sitemap generation, and a chained sequential Supabase init that silently swallows partial failures.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance concerns including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and hardcoded numeric literals that should be environment config. The extract-paper handler is the most problematic due to payload size and per-row insert loops, which will degrade under load and risk hitting job table size limits.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes present, but has several reliability and performance gaps. Key concerns include missing indexes on high-cardinality filter columns, nullable timestamps that should be NOT NULL, and a varchar user_id without a length constraint that could silently accept oversized values.

### Server Startup (`server/index.ts`)
The startup sequence performs multiple blocking async operations (DB probe, seed, job worker init) sequentially before the HTTP server binds, which delays readiness unnecessarily. Interval-based schedulers are started without storing their handles, making clean shutdown impossible and creating memory/timer leaks. The response-body logging middleware captures and serializes full API response bodies including potentially sensitive financial/user data to stdout on every request.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and performance gaps including unguarded async operations, memory leaks from missing cleanup, and inline ref callbacks that create new function instances on every render. The chat component sends the entire message history on every request which leaks conversation context unnecessarily, and there are scroll ref patterns that create new functions each render cycle.
