## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the matching category and another aggregates spent transactions. With 20 budgets this is 41 round-trips per dashboard load.

**Fix:** Replace the per-budget loops with a single JOIN query using Drizzle's leftJoin(categories, eq(budgets.categoryId, categories.id)) and aggregate spent via a subquery or sql`SUM` in the same statement. Return the joined result directly.

---

### `server/storage.ts`: getLeaderboard
**Issue:** getLeaderboard pulls all game sessions into Node memory and sorts/slices in JS. With growing school usage this will OOM the process; there is also no limit guard defaulting to a safe cap, so a caller omitting limit returns every row.

**Fix:** Push aggregation to the database: use db.select({ userId, totalScore: sql`SUM(score)`, gamesPlayed: sql`COUNT(*)` }).from(gameSessions).groupBy(gameSessions.userId).orderBy(desc(sql`SUM(score)`)).limit(filters?.limit ?? 50). Add a hard server-side cap of 100 regardless of caller input.

---

### `server/storage.ts`: createTransaction / updateVirtualBalance
**Issue:** Buy/sell flows call createPortfolioTransaction, upsertPortfolioHolding, and updateVirtualBalance as three separate awaits with no wrapping db.transaction(). A process crash between any two steps leaves balances and holdings permanently inconsistent.

**Fix:** Wrap all three operations inside a single db.transaction(async (tx) => { ... }) block using the Drizzle transaction API. Each sub-call must accept and use the tx client so all writes commit or roll back atomically.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~lines 45, 63)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or sanitisation. An attacker can send a crafted Host header causing the Sitemap directive and sitemap <loc> elements to point to an arbitrary domain (host-header injection), which can poison crawlers or be used in cache-poisoning attacks.

**Fix:** Add an allowlist check against a ALLOWED_HOST env var: `const allowedHost = process.env.PUBLIC_HOST || 'finsightlite.com'; const host = (req.headers['x-forwarded-host'] || req.headers.host || allowedHost).toString().split(',')[0].trim(); const safeHost = host === allowedHost ? host : allowedHost;` Use safeHost to build base URL.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 116)
**Issue:** AI/image/audio routes are registered with no rate limiting middleware applied before them in this file. Unlike the auth routes above which explicitly apply authLimiter, these compute-heavy and costly OpenAI-backed endpoints are left unprotected against abuse, which is a significant risk for a school-aged user app with API cost exposure.

**Fix:** Before the three register calls, add per-route limiters: `const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, standardHeaders: true }); app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` Import rateLimit from express-rate-limit alongside existing rateLimiter imports.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Full file content stored as base64 in the job payload. A single PDF or image can easily be several MB encoded, bloating the jobs table with every enqueue and retry, and making the row expensive to read on each attempt.

**Fix:** Upload the file to object storage before enqueuing the job, store only the storage path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer at the start of runExtractPaper. This keeps job rows small and makes retries cheap.

---

### `server/jobHandlers.ts`: streamPrivateObjectToResponse
**Issue:** The function accepts an arbitrary objectPath string and immediately splits it into bucket/object without validating that the resolved path stays within the expected private directory. A caller with a manipulated path (e.g. containing '../') could reference objects outside the intended bucket.

**Fix:** After splitting, verify that parts[0] matches the expected bucket from PRIVATE_OBJECT_DIR and that the joined object path does not contain '..' segments. Throw a 403-equivalent error if validation fails, before calling objectStorageClient.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – messages state / sendMessage
**Issue:** The full `messages` array is sent to /api/admin/help-chat on every turn with no size cap. For a long session this payload grows unboundedly, increasing latency and leaking the entire conversation to the server log on every request. There is also no auth token attached, so any unauthenticated client can hit this endpoint from the browser.

**Fix:** Slice the history before sending: `messages: next.slice(-20)`. Add the admin session token as an Authorization header inside apiRequest or via a dedicated admin API wrapper. On the server, verify the admin JWT before processing the chat.

## 🟡 Warnings (11)

- **`server/storage.ts` (getTransactions):** Missing pagination guard: when filters.limit is undefined the query returns every transaction for the user with no upper bound. A student with years of seeded demo data will cause multi-MB result sets on every dashboard render.
  → *Apply a hard default: const safeLimit = Math.min(filters?.limit ?? 100, 500); and always append .limit(safeLimit) to the query. Expose a cursor-based or offset parameter in TransactionFilters so callers can page explicitly.*
- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues at least four sequential awaits (total income, total expenses, category breakdown, savings goals) that are fully independent. Each adds one network round-trip to the DB sequentially.
  → *Wrap the independent queries in a single Promise.all: const [incomeResult, expenseResult, categoryResult, goalsResult] = await Promise.all([incomeQuery, expenseQuery, categoryQuery, goalsQuery]); This cuts DB wait time to the slowest single query instead of the sum of all four.*
- **`server/routes.ts` (/healthz handler (~line 76)):** Dynamic imports of ./db and drizzle-orm inside the /healthz handler execute on every health check request. These modules are re-resolved from cache each call but the overhead accumulates under frequent polling from Railway/uptime monitors, and any module-load error surfaces as an unhandled shape rather than a clean 503.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file alongside other static imports. The health check body becomes simply `await db.execute(sql\`select 1\`)` with no dynamic import cost per request.*
- **`server/routes.ts` (Supabase init chain (~line 120)):** The seven Supabase initialisation calls are chained sequentially with .then(), meaning each step must complete before the next begins. If any single step takes 500ms, the total blocking time is 3.5s+ during startup. Additionally, if an intermediate step fails, all subsequent seeds are skipped silently because only one .catch() exists at the end.
  → *Group independent operations with Promise.all where order does not matter: `await initSupabaseTables(); await Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()]); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]);` Wrap the whole block in try/catch and log each individual rejection.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 101)):** The filePath parameter is passed directly to objectStorage.searchPublicObject() without path traversal sanitisation. A request to /public-objects/../private/secrets could resolve outside the intended public bucket prefix depending on the ObjectStorageService implementation.
  → *Normalise and validate the path before use: `import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Then pass the sanitised filePath to objectStorage.*
- **`server/jobHandlers.ts` (for loop inside runExtractPaper (~line 110-120)):** Each extracted question is inserted with a sequential await inside a for-loop. For a 40-question paper this serialises 40 round-trips to the database, multiplying latency linearly.
  → *Collect valid questions into an array and replace the loop with a single batch insert: await storage.createExtractedQuestions(validQuestions) using Drizzle's .values([...]) bulk insert. Add the batch method to storage if it does not exist.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The entire handler body has no try/catch. A DB error will produce an unhandled rejection that may silently fail depending on the job runner's error handling, leaving no structured error record or alerting.
  → *Wrap the delete call in try/catch, capture the error via the existing captureError/Sentry utility (imported similarly to ai-health-check), then re-throw so the job runner marks the job as failed and schedules a retry.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls):** Model name 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two separate API calls. Changing the model or limits for cost or policy reasons requires a code deploy rather than a config change.
  → *Read from environment variables with fallbacks: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); Apply consistently to both calls in runExtractPaper.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – inline ref callback on scroll div):** `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is an anonymous function created on every render. It fires scrollIntoView every time the component re-renders (e.g., while the user is typing in the input), not just when a new message arrives, causing jarring scroll jumps mid-input.
  → *Replace with a `useRef` + `useEffect` that depends on `messages.length`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` Then use `ref={bottomRef}` on the div.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – missing error detail discrimination):** The catch block swallows all errors and always appends the same generic assistant message. Network errors, 401 Unauthorized, 429 Rate Limit, and 500 Server Error are all indistinguishable to the user and to future debugging. For a school platform this makes moderation or quota issues invisible.
  → *Check `res.ok` before calling `res.json()` and surface the HTTP status: `if (!res.ok) throw new Error(res.status.toString());`. In the catch handler inspect the error message to show contextual feedback, e.g. '429' → 'Too many requests, please wait a moment.'*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – handleKey defined inline in JSX / missing useCallback):** `handleKey` is redefined on every render of AdminHelpChat and passed as `onKeyDown` to a controlled input. Combined with `sendMessage` also being recreated each render, this causes the input element to receive a new function reference every keystroke, bypassing React's bailout optimisations.
  → *Wrap both with useCallback: `const sendMessage = useCallback(async () => { … }, [messages, input, loading]);` and `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`*

## 🔵 Info / Tech Debt (1)

- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – hardcoded endpoint '/api/admin/help-chat'):** The API base path is hardcoded as a string literal. ## 💡 Opportunities

### Centralised pagination middleware: 🔥 High Impact
Add a validatePagination(limit, offset) helper that enforces min/max bounds and is called at the top of every list-returning method. Eliminates the recurring pattern of unguarded limits and makes auditing easy as the user base scales across Caribbean schools.

### Extract investment trade into an atomic service layer: 🔥 High Impact
Move buy/sell logic (balance check → holding upsert → transaction record → balance debit) into a dedicated InvestmentService class that always uses a db.transaction(). This removes the risk of partial writes reappearing as new storage methods are added and makes unit-testing the trade flow straightforward.

### Replace in-process seedMarketData interval with a database job flag: ⚡ Medium Impact
simulateMarketPrices is likely called on a setInterval that is never cleared if the module is hot-reloaded in development, leaking the interval. Store the last-simulated timestamp in the DB and use a single scheduled job (pg_cron or a startup check) to decide whether to run, eliminating the leak and making simulation stateless across restarts.

### Consolidate duplicate health-check endpoints: ⚡ Medium Impact
Two health endpoints exist: /api/health (returns status ok, no DB check) and /healthz (DB ping). This creates confusion about which one Railway and uptime monitors should use. Consolidate into a single /api/health endpoint that performs the DB ping and returns structured JSON with a db field. Remove /healthz or make it a permanent redirect. Eliminates ambiguity and ensures monitors actually verify DB connectivity.

### Extract object storage instance to shared module: ⚡ Medium Impact
ObjectStorageService is instantiated at module level in routes.ts as a file-scoped singleton. If other route modules also instantiate it independently, multiple connection pools are created. Move the singleton to a dedicated server/services/objectStorage.ts and import it wherever needed. Reduces memory footprint and centralises configuration.

### Add startup readiness gate before serving traffic: 📌 Low Impact
Domain route registration and Supabase seeding happen after the HTTP server is already returned and listening. Requests arriving during the ~500ms window before registerStudentRoutes etc. complete will hit unregistered routes and receive 404s. Return the server only after all await registerXRoutes() calls complete (they already do), but also await the Supabase init chain rather than fire-and-forget it, or expose a readiness endpoint that returns 503 until seeding finishes.

### Parallel independent DB reads in job handlers: ⚡ Medium Impact
Any handler that fetches multiple independent records (e.g. user + module + progress) with sequential awaits should use Promise.all. This is not visible in the truncated portion but is a systemic pattern to audit; each parallelised fetch reduces handler latency proportionally to the number of queries.

### Centralised job error wrapper: 🔥 High Impact
Only ai-health-check has a try/catch with Sentry capture; other handlers rely on the job runner to catch errors with no structured context. Introduce a wrapHandler(name, fn) utility that wraps every handler with try/catch, calls captureError with job name and attempt count, then re-throws. Eliminates repeated boilerplate and guarantees uniform observability.

### PDF text extraction memory cap: ⚡ Medium Impact
buffer is decoded from base64 into a Node.js Buffer for every retry attempt. For large PDFs this allocates significant memory per worker thread. Pre-extracting text before enqueue (tied to the object storage fix) eliminates this allocation entirely at retry time and reduces peak worker memory.

### Virtualise long admin list renders: 🔥 High Impact
Admin dashboards commonly render full user/school/teacher tables without virtualisation. Using @tanstack/react-virtual for any list exceeding ~50 rows would cut DOM node count and initial paint time significantly, important on lower-end Caribbean school hardware.

### Parallelise independent dashboard data fetches: 🔥 High Impact
If the dashboard fires sequential useQuery calls for users, schools, and analytics, convert them to run concurrently with Promise.all or ensure all queries are registered at the same render cycle so React Query batches them. This can halve dashboard load time on high-latency Caribbean mobile connections.

### Add React Error Boundary around AdminHelpChat: ⚡ Medium Impact
An unhandled render error inside the chat panel (e.g. malformed assistant markdown) will crash the entire AdminDashboard. Wrapping AdminHelpChat in an ErrorBoundary with a graceful fallback protects the rest of the admin UI and improves reliability for school administrators.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits where Promise.all would reduce latency. Given this is a school-aged financial app, the absence of pagination guards on unbounded queries and reliance on in-memory leaderboard aggregation are the most critical reliability and security concerns.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good patterns like parallel dynamic imports and rate limiting on auth routes. Key concerns are: the Supabase init chain runs sequentially where it could be parallelised, the /healthz endpoint uses dynamic imports on every call creating unnecessary overhead, and the host header in robots.txt/sitemap is used without sanitisation creating a potential host-header injection vector.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues: large base64 payloads stored directly in the job table risk bloating the DB, sequential question inserts inside a loop should be batched, and the purge-ai-usage handler has no try/catch leaving errors untracked. Security is a concern with fileB64 payloads potentially logging sensitive binary data, and hardcoded model/token values should be environment-configurable.
