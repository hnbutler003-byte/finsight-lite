## 🔴 Critical (6)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app) (~line 115)
**Issue:** AI and image/audio routes are registered with no rate limiting applied before them. For a school-aged user base these are the most expensive endpoints (OpenAI cost, latency, abuse vector) yet they have no throttle, while lower-risk auth endpoints do. A single user or bot can exhaust OpenAI quota.

**Fix:** Add dedicated rate limiters before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); where aiLimiter is a stricter limiter (e.g. 20 req/15 min per IP) defined in rateLimiter.ts alongside existing limiters.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full base64-encoded file content (fileB64) is stored as part of the job payload in the job table. For a PDF or image this can easily be hundreds of kilobytes to several megabytes per row, bloating the jobs table, slowing queue polls, and leaking exam content into general-purpose DB backups.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), and have the handler download the buffer from object storage at execution time using the existing objectStorageClient helpers. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `shared/schema.ts` — budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions tables
**Issue:** userId is declared as varchar() with no length argument, which defaults to unlimited in Drizzle/Postgres but signals no intent. More critically, none of these tables have an index on userId alone or as the leading column, meaning every user-scoped query (e.g. fetching a user's budgets or savings goals) will perform a full sequential scan as data grows.

**Fix:** Add explicit indexes: e.g. in budgets table options callback add `userIdx: index('idx_budgets_user').on(t.userId)`, and repeat for linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings (userId+stockId composite), and portfolioTransactions (userId+executedAt). Also pin varchar to varchar('user_id', { length: 128 }) to match your auth provider's ID length.

### `server/index.ts` — maybeRunWeeklyDigest / maybeRunAiUsagePurge scheduler registration (post-listen block)
**Issue:** setInterval handles for the weekly digest and AI purge schedulers are never stored or cleared. On SIGTERM/SIGINT the intervals keep the event loop alive, preventing clean shutdown and potentially firing mid-drain, corrupting in-flight DB operations.

**Fix:** Store both interval handles: `const weeklyDigestInterval = setInterval(maybeRunWeeklyDigest, ...); const purgeInterval = setInterval(maybeRunAiUsagePurge, ...);`. In the graceful shutdown handler call `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before closing the HTTP server and DB pool.

---

### `server/index.ts` — res.json monkey-patch inside request logging middleware
**Issue:** Every API response body is captured into `capturedJsonResponse` and then serialised wholesale with `JSON.stringify` into the log line. For FinSight Lite this will routinely log auth tokens, student scores, guardian PII, and AI responses — a compliance and security violation for a children's financial app.

**Fix:** Remove the `res.json` override entirely, or replace it with a strict allowlist: only log `{ status, error }` fields for non-2xx responses, never log response bodies on 2xx. At minimum add a check: `if (res.statusCode >= 400) logLine += ` :: ${JSON.stringify({ error: capturedJsonResponse?.error })}`;`

### `client/src/pages/AdminDashboard.tsx` — useMutation callbacks / admin action handlers
**Issue:** Mutation handlers (delete user, promote teacher, reset password, etc.) do not appear to re-verify admin status on the client before dispatching. If the admin session expires mid-session, the UI still renders action buttons and fires mutations; the only gate is the server. More critically, success/error toasts reference raw server error messages which may contain stack traces or internal field names — a security information-disclosure risk for a school-facing app.

**Fix:** Before each mutate() call, assert `if (!adminUser) { toast({ title: 'Session expired' }); redirect('/admin/login'); return; }`. Sanitise error display: catch the error object and show only `error.message` if it is a known AppError type, otherwise show a generic 'Action failed' string. Never forward raw server responses to toast.

## 🟡 Warnings (18)

- **`server/routes.ts` (initSupabaseTables() chain (~line 118)):** Supabase init runs as a fire-and-forget promise chain with only a terminal .catch(). If any intermediate step (e.g. seedGameContent) throws synchronously or returns a rejected promise outside the chain, it becomes an unhandled rejection. More critically, each seed step runs sequentially in a .then() chain when most are independent, blocking total init time unnecessarily.
  → *Group independent seeds with Promise.all and wrap the entire block in an async IIFE with its own try/catch: (async () => { try { await initSupabaseTables(); await trimOrgNamesInSupabase(); await Promise.all([seedFinancialAcademyLesson(), initStaticContentTables()]); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]); } catch (e) { console.error('[Supabase] Init error:', e); } })();*
- **`server/routes.ts` (app.get('/api/health') and app.get('/healthz') (~lines 36 and 80)):** There are two health check endpoints: /api/health (returns immediately, no DB check) and /healthz (performs a real DB liveness probe). Railway and uptime monitors should use /healthz, but /api/health exists and is documented first, likely causing monitors to be pointed at the wrong endpoint and missing real DB outages.
  → *Remove /api/health entirely. Rename /healthz to /api/health to keep the /api/ prefix convention consistent and update any Railway/monitor config to use this single endpoint. If a fast no-DB ping is genuinely needed for a load balancer, document it as /api/ping explicitly.*
- **`server/routes.ts` (robots.txt and sitemap.xml handlers (~lines 44 and 60)):** Both handlers derive base URL from req.headers['x-forwarded-host'] and req.headers.host with no sanitisation. A request with a crafted Host header will produce a sitemap or robots.txt pointing to an attacker-controlled domain (host header injection), which can poison crawlers or CDN caches.
  → *Read the canonical base URL from an environment variable: const base = process.env.PUBLIC_BASE_URL || `${proto}://${host}`; Set PUBLIC_BASE_URL=https://finsightlite.com in production. This eliminates host header injection and makes the value auditable.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 40-question exam paper this means 40 serial round-trips to the database, adding significant latency and holding the job worker busy unnecessarily.
  → *Collect all valid question objects into an array first, then execute a single batched insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion contains non-DB side-effects that must be preserved, wrap them in Promise.all() instead of sequential awaits.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (both vision and extraction)):** The OpenAI model name 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two places. If the organisation needs to swap models per environment (cost control, regional availability, compliance) or tune token limits, a code change and redeploy is required.
  → *Add OPENAI_EXTRACTION_MODEL (default 'gpt-4o') and OPENAI_EXTRACTION_MAX_TOKENS (default 4096) to environment config and reference them via process.env. Apply the same pattern to the vision call. Validate these vars at startup alongside PRIVATE_OBJECT_DIR.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call. If the delete fails (DB timeout, connection drop) the job will throw an unhandled rejection. More importantly, olderThanDays comes directly from job.payload with no validation — a malformed or malicious payload (e.g. olderThanDays: 0 or negative) would compute a future cutoff and silently delete nothing, or a very large value would purge data that should be retained.
  → *Add input validation at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('Invalid olderThanDays'). Wrap the db.delete in try/catch and re-throw so the job framework records the failure and can retry.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete then re-insert pattern):** The idempotency strategy deletes previously inserted questions at the start of every retry attempt and then re-inserts them. If the process crashes between the delete and the completion of the re-insert (e.g. after deleting but mid-insert), the paper is left with zero questions and status still 'processing' until the next retry. On the final attempt the outer catch sets status to 'failed', but any intermediate crash window leaves the data in an inconsistent state visible to users.
  → *Wrap the delete and all subsequent inserts plus the status update inside a single DB transaction (db.transaction(async (tx) => { ... })). This guarantees the state is either fully rolled back or fully committed, eliminating the inconsistency window on crash.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions ~line 34, budgets ~line 44, etc.)):** All createdAt timestamps use .defaultNow() but omit .notNull(). This means application code or a raw INSERT omitting the column produces a NULL createdAt, breaking any ORDER BY createdAt DESC queries and audit trails silently.
  → *Chain .notNull() after .defaultNow() on every createdAt column: `createdAt: timestamp('created_at').defaultNow().notNull()`. Apply the same to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table, ticker column (~line 85)):** ticker has no unique index and no NOT NULL length cap via varchar. Duplicate tickers would silently corrupt portfolio value calculations since holdings join on stockId but display logic likely looks up by ticker string. A duplicate BISX ticker would cause wrong price resolution.
  → *Change ticker to `varchar('ticker', { length: 10 }).notNull()` and add a uniqueIndex: `tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker)` in the table options callback.*
- **`shared/schema.ts` (categories table, userId column (~line 16)):** userId on categories is nullable (intentionally for system categories), but there is no partial index or unique constraint preventing a user from creating duplicate category names. A user could accumulate duplicate 'Food' expense categories, causing ambiguous categoryId foreign key targets in transactions.
  → *Add a conditional unique index on (userId, name, type) for user-owned categories: `userCategoryNameIdx: uniqueIndex('idx_categories_user_name_type').on(t.userId, t.name, t.type)`. For system categories (userId IS NULL), handle deduplication at the seed/migration layer with a separate unique index on (name, type) WHERE user_id IS NULL using a raw SQL migration.*
- **`shared/schema.ts` (billReminders table, nextDueDate column (~line 75)):** nextDueDate has no index despite being the primary filter for reminder queries (e.g. 'find all bills due in the next 7 days' across all users). Without an index this query scans the entire table, and for a school-aged multi-user app this will degrade as the user base grows.
  → *Add a composite index on (userId, nextDueDate) in the billReminders table options: `userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate)`. This covers both per-user reminder fetches and any scheduled job that sweeps upcoming due dates.*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, startJobWorker calls before httpServer.listen)):** Database probe, seed, route registration, and job worker startup are awaited sequentially in the main startup block. `seedDatabase` and `probeDatabase` have no dependency on each other and together can add hundreds of milliseconds of blocking startup latency before the port opens.
  → *Run independent startup steps in parallel: `await Promise.all([probeDatabase(), seedDatabase()]);` then proceed with `registerRoutes`, `startJobWorker`, and `registerJobHandlers` which may depend on DB being ready. This cuts startup time proportional to the slower of the two independent operations.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince` as `(dow + 0) % 7` — the `+ 0` is a no-op dead expression that obscures intent. More critically, the time is set in the server's local timezone via `setHours(19, 0, 0, 0)`, which will silently drift for deploys across timezone boundaries (e.g. Replit servers vs Caribbean users), causing digest jobs to fire at wrong times.
  → *Remove the dead `+ 0`. Replace `setHours` with an explicit UTC calculation: construct the target as `Date.UTC(year, month, day, 23, 0, 0, 0)` (UTC 23:00 = UTC-4 19:00 for AST) so the schedule is timezone-stable regardless of server locale.*
- **`server/index.ts` (graceful shutdown handler (SIGTERM/SIGINT block)):** The shutdown sequence closes `httpServer` but there is no evidence of the job worker queue or the Drizzle/pg connection pool being explicitly closed. If the job worker holds a long-polling DB connection or the pool has idle clients, the process will hang until the OS kills it, risking mid-flight job corruption.
  → *Export a `stopJobWorker()` function from `./jobs` that drains the queue and resolves when idle. In the shutdown handler: `await Promise.all([stopJobWorker(), db.$client.end()])` (adjust to your pg pool's `.end()` API) before calling `process.exit(0)`, and add a hard timeout fallback: `setTimeout(() => process.exit(1), 10_000)`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — messages scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new function on every render and calls scrollIntoView on every re-render (e.g. while typing), not just when a new message arrives. This causes janky scroll behaviour and wastes layout cycles.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function inside AdminHelpChat):** sendMessage is redefined on every render and is passed to both the button onClick and onKeyDown handler with no useCallback. For a chat component that re-renders on every keystroke (controlled input), this means fresh function allocations on every character typed and both handlers always reference a stale closure risk.
  → *Wrap sendMessage with useCallback and declare its deps: `const sendMessage = useCallback(async () => { ... }, [input, messages, loading]);`. Also memoize handleKey the same way. This eliminates per-keystroke reallocations and stale-closure risk.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard data fetching (multiple useQuery calls)):** Multiple useQuery hooks fire independently for users, schools, teachers, quizzes, and audit logs. Each query is declared sequentially and TanStack Query will issue them as separate round trips with no explicit parallelism config. If any intermediate query suspends or errors, later ones still proceed independently but the loading states are tracked separately with no unified error boundary — missing error UI leaves the page partially blank with no feedback.
  → *Introduce a combined loading/error banner: derive `const isLoading = usersQuery.isLoading || schoolsQuery.isLoading || ...` and render a top-level skeleton or alert when any is true or in error. For truly parallel fetching, verify no query depends on another's data; if so, use `enabled` flags explicitly. Consider useQueries([...]) to co-locate them and share a single loading state.*
- **`client/src/pages/AdminDashboard.tsx` (Hardcoded API base paths and pagination constants):** Strings like '/api/admin/help-chat', page size values (e.g. 10 per page), and likely base URLs are inlined throughout the file. In a Caribbean multi-school deployment these will need to differ per environment, and magic numbers make pagination behaviour invisible to reviewers.
  → *Extract to a config module: `export const API_BASE = import.meta.env.VITE_API_BASE ?? '';` and `export const ADMIN_PAGE_SIZE = Number(import.meta.env.VITE_ADMIN_PAGE_SIZE ?? 10);`. Reference these constants in all query paths and pagination logic. This also prevents accidental production/staging endpoint mix-ups.*

## 💡 Opportunities

### Centralise AI rate-limit config in environment variables — 🔥 High Impact
AI endpoint limits (requests per window, window duration) are hardcoded in rateLimiter.ts. Expose them as AI_RATE_LIMIT_MAX and AI_RATE_LIMIT_WINDOW_MS env vars so they can be tightened in production or relaxed in staging without a code deploy. Especially important for a school-aged audience where usage spikes around school hours.

### Add structured startup logging for route and Supabase init phases — ⚡ Medium Impact
Currently init failures surface only via console.error with no timing or phase metadata. Adding structured logs (phase, durationMs, success/failure) for each Supabase seed step and each route module registration would reduce mean-time-to-diagnose on cold-start failures in Railway deployments.

### Cache robots.txt and sitemap.xml responses at the HTTP layer — 📌 Low Impact
Both handlers recompute and re-send the full response on every request. Adding res.setHeader('Cache-Control', 'public, max-age=86400') reduces unnecessary compute and allows CDN/Railway edge to cache them. Since the base URL is now env-derived (per finding 4), the output is stable across requests.

### Centralised OpenAI call wrapper with usage logging and timeout — 🔥 High Impact
Both openai.chat.completions.create calls in runExtractPaper have no timeout or circuit-breaker. A hung OpenAI request will block the worker indefinitely. Introduce a thin wrapper that enforces a configurable AbortSignal timeout (e.g. 60 s), logs token usage to aiUsageEvents atomically, and surfaces a typed error for the retry logic. This prevents worker starvation and gives accurate cost visibility.

### Move job payload size validation to enqueueJob — 🔥 High Impact
Adding a payload size guard (e.g. reject if JSON.stringify(payload).length > 64 KB) in the shared enqueueJob function would catch oversized payloads (like base64 files) at enqueue time with a clear error, before they pollute the job table. This is a single-line change with broad protective impact across all job types.

### Replace sequential admin CSV query fan-out with parallel execution — ⚡ Medium Impact
The truncated admin-csv-export handler likely fetches multiple tables sequentially. If those queries are independent they should run via Promise.all(). For a dataset spanning users, transactions, and sessions this could reduce export job duration by 50-70% under normal DB load.

### Centralize currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, etc.). Extract `export const DEFAULT_CURRENCY = 'BSD'` to a config file and reference it in each column definition. When the app expands to support TTD or JMD, there is currently no single place to update or validate the default.

### Replace text() with varchar() for bounded string columns — ⚡ Medium Impact
Columns like lastFour, fileType, frequency, riskLevel, and ticker use unbounded text() despite having known maximum lengths. Switching to varchar with explicit limits (e.g. lastFour: varchar length 4, ticker: length 10) enforces data integrity at the DB layer without application validation, and gives the query planner better statistics for index selectivity.

### Add updatedAt tracking to mutable tables — 🔥 High Impact
budgets, savingsGoals, and billReminders are all user-editable but have no updatedAt column, making it impossible to implement optimistic concurrency control or sync conflict resolution later. Adding `updatedAt: timestamp('updated_at').defaultNow().notNull()` now and updating it via a Postgres trigger or Drizzle middleware avoids a costly migration once sync features are built.

### Structured logging with redaction middleware — 🔥 High Impact
Replace ad-hoc `console.log` string concatenation with a structured logger (e.g. `pino`) configured with a redact array covering `['password', 'token', 'sessionId', 'email', '*.secret']`. This gives machine-parseable logs for Sentry/Datadog ingestion and eliminates the entire class of accidental PII leakage in one place rather than per-callsite fixes.

### Extract scheduler into a dedicated worker module — ⚡ Medium Impact
Moving `maybeRunWeeklyDigest`, `maybeRunAiUsagePurge`, and their interval registration into a `./scheduler.ts` module that exports `startSchedulers()` and `stopSchedulers()` would make lifecycle management testable in isolation, eliminate the interval-leak risk structurally, and keep `index.ts` under 80 lines focused solely on wiring.

### Add startup health-check timeout with early SIGTERM — ⚡ Medium Impact
Wrap the entire async startup block in a `Promise.race` against a `setTimeout` of 30 seconds that calls `process.exit(2)`. Currently if `probeDatabase` or `seedDatabase` hangs (e.g. cold DB on Replit free tier), the process never binds the port and the orchestrator never gets a signal to restart it, causing silent unavailability.

### Split AdminDashboard into feature sub-pages — 🔥 High Impact
At 131k+ characters this is a god-component. Extracting UsersPanel, SchoolsPanel, QuizzesPanel, and AuditLogPanel as separate lazy-loaded route segments would reduce initial bundle size by ~60%, enable per-panel error boundaries, and make each section independently testable — critical for a school app where quiz management and user admin have different risk profiles.

### Add React.memo to pure table-row and chart sub-components — ⚡ Medium Impact
BarChart and LineChart wrappers, along with any user/school row renderers, are re-created on every parent state change (e.g. search input). Wrapping them in React.memo and memoising their data props with useMemo would eliminate chart re-renders on unrelated state updates, noticeably improving responsiveness on low-end Caribbean school devices.

### Centralise admin API calls into a typed service layer — ⚡ Medium Impact
Raw apiRequest('POST', '/api/admin/...') calls are scattered inline. Extracting an adminApi service (adminApi.deleteUser(id), adminApi.helpChat(messages)) gives a single place to attach auth headers, enforce CSRF tokens, and mock in tests — reducing the surface area for auth-bypass bugs and duplicated fetch logic.

### API Routes (`server/routes.ts`)

jobHandlers.ts has several reliability and performance gaps: large base64 payloads are stored directly in the job table, sequential DB inserts inside a loop should be batched, and hardcoded model names and token limits should be environment-configurable. Security posture is adequate for the visible scope, but the file warrants attention before production load.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some composite indexes, but has several reliability and security gaps. Key concerns are missing NOT NULL constraints on audit timestamps, absent indexes on high-cardinality filter columns like userId in budget/savings/bill tables, and an undersized varchar for userId that could silently truncate Clerk/Auth0 UUIDs.

### Server Startup (`server/index.ts`)
The server startup sequence contains blocking synchronous-style serial awaits that delay readiness, interval schedulers that are never cleared on shutdown preventing clean process exit, and response JSON logging that risks leaking sensitive financial or PII data into logs for all API responses. Overall structure is reasonable but reliability and security gaps need addressing before production use with school-aged users.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx is a large, monolithic component with several reliability and security concerns. The most critical issues are missing auth guard enforcement on sensitive mutations, a memory leak from an unref'd scroll div callback on every render, and sequential awaits where parallel fetching would reduce load time. Overall structure is reasonable but the file's size suggests it needs decomposition.
