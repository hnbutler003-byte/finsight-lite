## 🔴 Critical (11)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row a separate SELECT is issued to resolve the category and a separate aggregate query is fired to compute 'spent', meaning 20 budgets = 41 round-trips to the database on every dashboard load.

**Fix:** Rewrite with a single JOIN query using Drizzle's leftJoin(categories, eq(budgets.categoryId, categories.id)) and a subquery or window function for the spent aggregate, returning all data in one round-trip. If the ORM makes the aggregate awkward, use sql`SELECT ..., COALESCE(SUM(t.amount),0) AS spent FROM budgets LEFT JOIN ... LEFT JOIN transactions t ...` with a GROUP BY.

### `server/storage.ts`: getLeaderboard
**Issue:** getLeaderboard performs an unbounded full-table scan on gameSessions with no default LIMIT guard. The caller-supplied limit is optional and defaults to nothing, so a single unauthenticated or misconfigured call can pull every game session row into Node.js memory and serialize it.

**Fix:** Add a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 50, 200); and apply .limit(safeLimit) unconditionally before the query executes. Never trust caller-supplied limits without a ceiling.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 110)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. A single user or bot can exhaust OpenAI quota and drive up costs, which is especially dangerous for a school-aged user base.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before calling registerChatRoutes, registerImageRoutes, registerAudioRoutes.

### `server/routes.ts`: robots.txt handler (~line 44) and sitemap.xml handler (~line 57)
**Issue:** The base URL is constructed directly from x-forwarded-host and host request headers without any validation or allowlist. An attacker can send a forged Host header, causing the sitemap and robots.txt to emit arbitrary URLs (host header injection / cache poisoning).

**Fix:** Read the canonical base URL from an environment variable: const base = process.env.CANONICAL_URL ?? 'https://finsightlite.com'; Ignore the request headers entirely for URL construction in these two handlers.

### `server/jobHandlers.ts`: runExtractPaper (job.payload.fileB64)
**Issue:** The full base64-encoded exam paper file is stored directly in the job payload, which means every job row in the database carries potentially several megabytes of binary data. This bloats the jobs table, slows queue polling queries, and risks exceeding row size limits.

**Fix:** Upload the file to object storage before enqueuing the job and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, fetch the buffer from object storage using that path. This keeps job rows small and the file in the correct storage tier.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets table has no index on userId, yet budget queries will always filter by userId. Without an index, every budget lookup performs a full table scan, which degrades as user count grows.

**Fix:** Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also prevents duplicate budget rows per user/category combination, which is a latent data integrity bug.

### `shared/schema.ts`: portfolioHoldings and portfolioTransactions table definitions
**Issue:** Neither portfolioHoldings nor portfolioTransactions has an index on userId or (userId, stockId). Portfolio value calculations and trade history queries will fan out to full table scans for every authenticated request in the investment simulation feature.

**Fix:** Add to portfolioHoldings: (t) => ({ userStockIdx: uniqueIndex('idx_holdings_user_stock').on(t.userId, t.stockId) }). Add to portfolioTransactions: (t) => ({ userIdx: index('idx_ptx_user').on(t.userId), userStockIdx: index('idx_ptx_user_stock').on(t.userId, t.stockId) }). The unique index on holdings also enforces one holding row per user/stock, preventing phantom duplicate holdings.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** Day-of-week calculation is wrong. `(dow + 0) % 7` returns 0 when dow is already 0 (Sunday), so on Sunday itself `d.setDate(d.getDate() - 0)` leaves the date unchanged, but then the guard `if (d.getTime() > now.getTime())` subtracts 7 days, meaning the digest never fires on the correct Sunday at or after 7 pm — it always schedules for the *previous* Sunday.

**Fix:** Replace the daysSince calculation with `const daysSince = dow;` (no modulo needed since getDay() already returns 0–6). Then the guard condition correctly handles the case where the computed 7 pm is still in the future. Add a unit test covering Sunday before and after 19:00.

### `server/index.ts`: res.json monkey-patch in request logging middleware
**Issue:** Every API response body is captured into `capturedJsonResponse` and then serialised in full with `JSON.stringify` and written to stdout. For a financial literacy app with school-aged users this will log tokens, session data, grade records, or any PII that appears in a JSON response — a clear data-exposure risk and a potential PCI/FERPA compliance issue.

**Fix:** Remove the `res.json` override entirely, or replace it with a size-gated, field-redacted summary: log only `res.statusCode` and `duration`. If response inspection is needed for debugging, gate it behind `process.env.LOG_RESPONSE_BODIES === 'true'` and add an explicit denylist of sensitive field names to strip before logging.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by `setInterval` are never stored, so they cannot be cleared during graceful shutdown. If the process receives SIGTERM while an interval callback is mid-await (e.g. inside a DB transaction), Node will exit with the async operation orphaned, potentially leaving the `weeklyDigestRuns` claim row inserted but no job enqueued, causing that week's digest to be permanently skipped.

**Fix:** Store both handles: `const weeklyDigestInterval = setInterval(...); const purgeInterval = setInterval(...);`. In the SIGTERM/SIGINT handler call `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before awaiting `httpServer.close()`. Also add an `unref()` call on each interval so they don't keep the process alive on their own: `weeklyDigestInterval.unref();`.

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** The help-chat endpoint `/api/admin/help-chat` is called without any admin auth token or session header. If the server relies solely on cookie-based session and the endpoint lacks its own admin role check, any authenticated non-admin user who discovers the endpoint can query the AI assistant with full admin context in the request body.

**Fix:** On the server, add an `isAdmin` middleware guard to the `/api/admin/help-chat` route identical to other `/api/admin/*` routes. On the client, verify `useAdminAuth` is checked before rendering `AdminHelpChat` at all — gate the floating button render on `adminUser != null`.

## 🟡 Warnings (16)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats fires multiple independent database queries sequentially (transactions, budgets, savings goals, etc.) with individual awaits, adding all round-trip latencies in series. On a Caribbean cloud deployment with 30–50 ms DB latency, five sequential queries add 150–250 ms of avoidable wait per page load.
  → *Wrap all independent queries in a single Promise.all: const [txRows, budgetRows, goalRows] = await Promise.all([db.select(...), db.select(...), db.select(...)]); Each query is independent so they can safely run concurrently.*
- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices updates every stock price with a separate UPDATE statement in a for-loop with no wrapping database transaction. If the process crashes mid-loop the market is left in a partially updated state, and concurrent reads see inconsistent mid-update prices.
  → *Wrap the entire price update loop inside await db.transaction(async (tx) => { for (const stock of stocks) { await tx.update(...) } }). This guarantees atomicity: either all prices update or none do.*
- **`server/storage.ts` (getAllQuestions):** getAllQuestions maps over extracted questions and performs an individual db.select() for the parent ExamPaper per question inside a Promise.all, which is still an N+1 pattern (just parallelised). 500 questions = 501 concurrent DB connections, which will exhaust the connection pool.
  → *Collect all unique paperIds from the question rows first, then do one IN query: const papers = await db.select().from(examPapers).where(inArray(examPapers.id, uniqueIds)). Build a Map<id, ExamPaper> and join in JavaScript. Zero extra DB round-trips regardless of result size.*
- **`server/routes.ts` (Supabase init chain (~line 114)):** The chained .then() promises run sequentially and a failure in any middle step (e.g. seedFinancialAcademyLesson) silently skips all subsequent steps (seedStaticModules, seedRegionalContent, seedGameContent). The single .catch at the end only reports the first failure, leaving the DB in a partially seeded state with no visibility.
  → *Run independent seed steps in parallel where safe, and log per-step failures individually: const steps = [trimOrgNamesInSupabase, seedFinancialAcademyLesson, seedStaticModules, seedRegionalContent, seedGameContent]; await initSupabaseTables(); for (const step of steps) { await step().catch(e => console.error(`[Supabase] ${step.name} failed:`, e)); } This ensures all steps are attempted and failures are attributable.*
- **`server/routes.ts` (/healthz handler (~line 77)):** The /healthz endpoint dynamically imports db and drizzle-orm on every request. Dynamic imports are cached after the first load, but the repeated await import() calls on a hot health-check path (hit every 30s by Railway) add unnecessary async overhead and obscure the module dependency graph.
  → *Move the db and sql imports to the top of the file as static imports alongside other imports. The health check handler then simply calls await db.execute(sql`select 1`) with no dynamic import overhead.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one inside a sequential for loop with individual awaits. For a 40-question paper this means 40 serial round-trips to the database. If the process crashes mid-loop the idempotency delete at the top of a retry will clean up, but the latency cost on every successful run is unnecessary.
  → *Collect all valid question objects into an array, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Guard with a transaction so the delete and bulk insert are atomic, eliminating the crash-recovery window entirely.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage', ...)):** The purge-ai-usage handler has no try/catch. An unexpected DB error (connection drop, lock timeout) will produce an unhandled promise rejection that may not be surfaced through the job worker's normal failure path, leaving the job silently stuck.
  → *Wrap the db.delete call in a try/catch, log the error with context (olderThanDays, cutoff), and rethrow so the job framework can record the failure and schedule a retry.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** The PDF text is silently truncated to 30,000 characters with no warning recorded on the job or the exam paper record. For long papers this means questions in the second half are silently dropped and the teacher has no visibility into the truncation.
  → *After slicing, check if pdfData.text.length > 30000 and if so update the exam paper record with a truncation warning field (or append a note to the job result). Also make 30000 a named constant or env var (PDF_TEXT_MAX_CHARS) so it is discoverable.*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, etc.)):** All userId columns are declared as varchar with no length parameter. PostgreSQL will accept unbounded varchar, but omitting the length forfeits an important validation boundary and may differ from the actual Clerk/auth user ID length, risking silent truncation or mismatched FK joins if the auth provider changes ID format.
  → *Declare userId as varchar('user_id', { length: 256 }) on every table to match Clerk's maximum user ID length and make the constraint explicit. This is a one-line change per table and ensures the DB enforces the same contract as the auth layer.*
- **`shared/schema.ts` (transactions createdAt, budgets createdAt, savingsGoals createdAt, documentUploads createdAt, billReminders createdAt):** All createdAt columns use .defaultNow() without .notNull(). This means application code can explicitly insert NULL for createdAt, producing rows with no audit timestamp in a financial ledger. For a regulated-adjacent app handling minors' financial data, audit trails must be guaranteed at the DB layer.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Also apply to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions for the same reason.*
- **`shared/schema.ts` (simulatedStocks lastPriceUpdateDate column):** lastPriceUpdateDate is stored as text rather than timestamp. Sorting or filtering stocks by price update recency requires a full cast on every query, and there is no format validation at the DB layer, so malformed date strings can be inserted silently.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed or insert logic that writes string dates to pass a JS Date object instead. Drizzle will handle the serialization correctly.*
- **`server/index.ts` (startServer async IIFE — seedDatabase, probeDatabase, registerRoutes, startJobWorker calls):** `seedDatabase()` and `probeDatabase()` are awaited sequentially before the HTTP server binds. If `seedDatabase` takes several seconds (cold DB, large seed), the process is alive but not serving health-check requests, causing orchestrators (e.g. Replit, Railway) to mark it as unhealthy and restart it in a loop. Additionally, `registerRoutes` and `startJobWorker` are also awaited in sequence even though the job worker does not depend on route registration.
  → *Bind the HTTP server first (`httpServer.listen(port)`), then run `await Promise.all([probeDatabase(), seedDatabase()])` in parallel for the two independent operations. Start the job worker concurrently with seeding: `await Promise.all([seedDatabase(), startJobWorker()])`. Register a `/-/health` route before any async work so orchestrators get a 200 immediately.*
- **`server/index.ts` (maybeRunAiUsagePurge — AUTO_PURGE_OLDER_THAN_DAYS cutoff query):** The cutoff date is constructed by string interpolation or direct JS Date arithmetic and passed into a Drizzle `gte` filter, but if the underlying column is a `timestamp` type and the value is constructed as a plain `Date` object from `new Date(Date.now() - days*86400000)`, DST transitions can silently shift the boundary by one hour, purging one extra hour of records or missing one hour — a data-integrity issue in a financial record context.
  → *Construct the cutoff using UTC arithmetic exclusively: `const cutoff = new Date(); cutoff.setUTCDate(cutoff.getUTCDate() - AUTO_PURGE_OLDER_THAN_DAYS); cutoff.setUTCHours(0,0,0,0);` and pass that to Drizzle. Log the ISO string of the computed cutoff at INFO level before the delete so every purge run is auditable.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat):** The scroll-to-bottom ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is an inline function recreated and fired on every render, not just when a new message arrives. This causes unwanted scroll jumps when unrelated state (e.g. `loading`) changes and creates a new function object every render.
  → *Replace with a `useRef` + `useEffect` that only fires when `messages.length` or `loading` changes: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length, loading]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (useEffect / data fetching at dashboard mount):** Multiple independent `useQuery` calls that have no data dependency on each other (e.g. users list, schools list, analytics) are initiated sequentially via individual query hooks but any imperative `await apiRequest` calls inside `useEffect` blocks are written as sequential awaits, blocking total load time unnecessarily.
  → *Audit every `useEffect` that contains two or more `await apiRequest(...)` calls with no interdependency and replace with `const [a, b] = await Promise.all([apiRequest(...), apiRequest(...)])`. This can halve perceived load time on slow Caribbean mobile connections.*
- **`client/src/pages/AdminDashboard.tsx` (large inline filter/sort expressions in JSX):** Filtering and sorting of users, schools, or lesson lists appears to be computed directly inside the render return (e.g. `.filter(...).sort(...)` inline in JSX map calls). These run on every render including unrelated state changes like toggling a dialog or typing in the chat input.
  → *Lift each filter+sort expression into a `useMemo` with the source array and search/filter state as dependencies: `const filteredUsers = useMemo(() => users.filter(u => u.name.includes(search)).sort(...), [users, search]);`. For school-aged user volumes this is a meaningful saving per keystroke.*

## 💡 Opportunities

### Centralise pagination defaults as named constants: 🔥 High Impact
Default and maximum page sizes (currently scattered or absent) should be declared as MODULE_MAX_PAGE_SIZE = 200 and DEFAULT_PAGE_SIZE = 50 constants at the top of storage.ts and applied uniformly to every list query. This prevents future developers from accidentally omitting limits on new list methods.

### Add a query-layer result cache for market stock data: ⚡ Medium Impact
getMarketStocks is called on every portfolio page render but stock prices only change when simulateMarketPrices runs. Caching the result in a module-level Map with a TTL matching the simulation interval (e.g. 60 s) eliminates repeated full-table reads and is safe because the cache is explicitly invalidated at the end of simulateMarketPrices.

### Extract sensitive field redaction into a shared sanitiser: ⚡ Medium Impact
User objects returned from getUser and upsertUser include fields like passwordHash that risk appearing in logs or API responses if a handler accidentally serialises the whole object. A single stripSensitive(user: User) helper that deletes passwordHash before returning would eliminate this class of accidental exposure across all storage methods.

### Parallel domain route registration: ⚡ Medium Impact
The six registerXxxRoutes calls after the parallel imports are still awaited sequentially. If any of these async registrations do their own I/O (DB schema checks, etc.), wrapping them in Promise.all would reduce cold-start latency. Expected impact: 50-200ms reduction in server startup time depending on each router's init cost.

### Structured startup logging for Supabase seed steps: ⚡ Medium Impact
Replace console.error with a structured logger (e.g. pino) that emits JSON with step name, duration, and error details. This enables Railway log drains and alerting to catch partial seed failures in production, which is critical for a school app where missing lesson content would silently degrade the student experience.

### Cache-Control header on /public-objects route: 📌 Low Impact
The downloadObject call passes a 30-day TTL but there is no Cache-Control header set at the Express layer for cache misses or error responses. Adding res.setHeader('Cache-Control', 'public, max-age=2592000, immutable') for successful serves and 'no-store' for 404s would allow CDN/browser caching and reduce object storage egress costs for static assets like logos served to many students.

### Parallel object-storage existence check and stream setup: 🔥 High Impact
In streamPrivateObjectToResponse the file.exists() check and any metadata fetch are sequential. For high-traffic export endpoints, pre-signing a short-lived URL and redirecting the client to object storage directly would offload streaming from the Node process entirely, freeing the event loop and eliminating memory pressure from piped streams.

### Atomic idempotency via DB transaction in extract-paper: ⚡ Medium Impact
Wrapping the delete-then-bulk-insert in a single serializable transaction means a mid-job crash leaves the paper in its previous state rather than a partially wiped one, removing the need for status-sync logic in the catch block and simplifying retry semantics.

### Structured job result schema validation: ⚡ Medium Impact
The JSON parsed from the OpenAI response is cast with 'as' but never validated at runtime. Adding a lightweight schema check (zod.safeParse) before accessing parsed.questions would catch model hallucinations or format drift early and produce a clear error message rather than a downstream null-access failure.

### Composite unique constraint on budgets prevents silent duplicate budgets: 🔥 High Impact
Adding a uniqueIndex on (userId, categoryId, period) in the budgets table ensures that application bugs or retry logic cannot create two active budgets for the same category and period. Without it, budget-vs-spending calculations silently double-count the limit, which is a correctness issue for young users learning to budget.

### Replace repeated currency text columns with a shared enum or DB check constraint: ⚡ Medium Impact
The literal string 'BSD' is hardcoded as a default in at least six separate columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Centralise this as a pgEnum('currency', ['BSD', 'USD', 'TTD']) and reference it everywhere. This eliminates invalid currency strings entering the DB and makes adding a new supported currency a single-file change.

### Add billReminders index on (userId, nextDueDate) for reminder scheduling queries: ⚡ Medium Impact
A background job or dashboard widget that fetches upcoming bills for a user will always filter by userId and order by nextDueDate. Without a compound index the query scans all reminders. Adding index('idx_reminders_user_due').on(t.userId, t.nextDueDate) in the table definition keeps this query O(log n) regardless of reminder volume.

### Structured JSON logging instead of console.log strings: 🔥 High Impact
The `log()` helper emits free-form strings. Replacing it with a structured logger (pino or winston with JSON output) would let log aggregators (Datadog, Logtail) index `duration`, `statusCode`, `path`, and `source` as queryable fields rather than requiring regex parsing. Expected impact: 50-70% reduction in time spent debugging production incidents.

### Centralise all magic numbers and intervals as validated env config: ⚡ Medium Impact
Scheduler intervals (weekly digest check cadence, purge check cadence) appear to be hardcoded literals in the setInterval calls. Exposing them as `WEEKLY_DIGEST_CHECK_INTERVAL_MS` and `PURGE_CHECK_INTERVAL_MS` env vars with validation (same pattern as `AUTO_PURGE_OLDER_THAN_DAYS`) allows tuning in production without redeployment and makes load-testing easier by setting short intervals in staging.

### Add a readiness vs liveness split on the health endpoint: ⚡ Medium Impact
A single health endpoint that runs a DB probe on every call will add latency to health checks and can cause thundering-herd DB connections when the orchestrator polls aggressively. Split into `/-/live` (returns 200 immediately, no I/O) and `/-/ready` (probes DB, checks job worker state). This prevents the orchestrator from restarting a healthy process simply because a transient DB probe timed out.

### Unified loading and error boundary for admin data: 🔥 High Impact
Multiple query hooks each handle their own loading/error state with scattered conditional renders. Wrapping the dashboard in a single React Error Boundary and a shared skeleton loader tied to `isLoading` from a combined query status would eliminate duplicated loading UI and ensure error states are never silently swallowed — critical for school admins who may misread a blank table as 'no data'.

### Memoize child panel components with React.memo: ⚡ Medium Impact
Sub-components like the chat panel, user table rows, and chart wrappers are likely re-rendered whenever top-level AdminDashboard state (search text, active tab) changes. Wrapping stable child components in `React.memo` and passing `useCallback`-wrapped handlers as props would eliminate these cascading renders, reducing main-thread work on low-end Android devices common in the Caribbean school market.

### Paginate or virtualise large admin tables: 🔥 High Impact
Rendering all users or all lessons into a DOM table at once causes a large initial paint and slow interaction. Implementing server-side pagination (already partially present via ChevronLeft/ChevronRight icons) with a fixed page size of 25 and removing client-side full-array filtering would dramatically reduce memory use and time-to-interactive on low-RAM tablets.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that should run in parallel. For a school-aged Caribbean fintech app these reliability gaps risk data corruption on concurrent writes and measurable latency on every dashboard load.

### API Routes (`server/routes.ts`)
The route registration code is generally well-structured with good patterns like parallel module imports and rate limiting. Key concerns are unprotected AI/chat/image/audio routes with no rate limiting, a fire-and-forget Supabase init chain that swallows individual step errors, and host header injection risk in robots.txt and sitemap generation.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, missing try/catch on the AI usage purge handler, and a hardcoded token limit that belongs in environment config. The extract-paper handler's idempotency delete-then-reinsert pattern is correct in intent but the sequential per-question inserts in a loop will be slow and risk partial state on crash.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with FK constraints and some indexes already in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable timestamps that should be NOT NULL, and unbounded varchar types that risk data integrity issues for a financial application serving minors.

### Server Startup (`server/index.ts`)
The server startup sequence contains several reliability and security gaps: blocking synchronous-style sequential awaits on the critical path, intervals created without cleanup handles stored for graceful shutdown, and response body logging that can leak sensitive financial or PII data in plain logs. The weekly digest scheduler logic also has a subtle day-of-week bug that will cause it to fire on the wrong day.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth guards on sensitive operations, memory leaks from unregistered cleanup, and expensive inline computations running on every render. The chat component leaks a scroll ref callback on every render and sequential data fetches miss parallelization opportunities.
