## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate SELECT for spent amount per budget row in a loop, and another per-row fetch for the category. For a class with 30 students each viewing their budget page simultaneously this fans out to 60+ queries per user.

**Fix:** Replace the loop with a single JOIN query using Drizzle: join budgets with categories and use a subquery or aggregation (sum of transactions grouped by categoryId and date range) to compute spent in one round-trip. Example: db.select({ ...budgets, categoryName: categories.name, spent: sql<number>`coalesce(sum(${transactions.amount}),0)` }).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).where(eq(budgets.userId, userId)).groupBy(budgets.id, categories.id)

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Stock trade execution performs upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three independent awaits with no wrapping database transaction. A failure between any two steps leaves the virtual balance debited but no holding recorded, or a holding updated with no audit trail — corrupting student portfolio state silently.

**Fix:** Wrap all three operations inside db.transaction(async (tx) => { ... }) passing the tx client to each operation. Drizzle's transaction callback rolls back automatically on throw. Define each sub-operation to accept an optional tx parameter so they can be called both standalone and within a transaction.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting. A single user or bot can hammer these endpoints, running up API costs and degrading service for all school-aged users on the platform.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls.

### `server/routes.ts`: /public-objects/:filePath(*) handler (~line 95)
**Issue:** The filePath wildcard parameter is passed directly to objectStorage.searchPublicObject without sanitisation. A request to /public-objects/../../private/config could traverse into unintended storage paths depending on how searchPublicObject resolves keys.

**Fix:** Sanitise the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, '').replace(/^[/\\]+/, '');` and reject requests where the sanitised path differs from the original or is empty, returning 400.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** fileB64 (full base64-encoded PDF or image) is stored directly in the job payload, which lives in the jobs table. A single 5MB PDF becomes ~6.7MB of base64 in the DB row, multiplied across retries and concurrent jobs. This bloats the job table, slows queue polling queries, and risks hitting row-size limits.

**Fix:** Upload the file to object storage before enqueuing the job. Store only the storage path in the payload (e.g. { paperId, objectPath, ext, subject }). In the handler, stream the object from storage into the buffer. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts`: runExtractPaper – question insert loop (for loop over questions[])
**Issue:** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this fires 40 serial round-trips to the DB. A single network hiccup mid-loop leaves a partial set with no transaction wrapping, so the idempotency delete at the top of a retry wipes good data but the partial state was already visible to readers.

**Fix:** Collect all valid question records into an array first, then do a single db.insert(extractedQuestions).values(validQuestions) inside a db.transaction() block. This is one round-trip, atomic, and naturally idempotent with the existing delete guard.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** Day-of-week rollback logic is subtly wrong: `(dow + 0) % 7` on Sunday (dow=0) returns 0, so no days are subtracted and the target is set to 19:00 today. If it is currently before 19:00 on a Sunday the guard `if (d.getTime() > now.getTime())` moves it back a full week, causing the digest to fire a week late every Sunday until after 7 pm.

**Fix:** Replace the daysSince calculation with `const daysSince = dow === 0 ? 0 : 7 - (7 - dow) % 7` — or more clearly: compute the most recent Sunday as `d.setDate(d.getDate() - dow)` (dow=0 stays, any other day steps back). Then unconditionally check whether 19:00 that Sunday is still in the future and subtract 7 days if so. Add a unit test covering Sunday before and after 19:00.

### `server/index.ts`: setInterval (weekly digest and purge schedulers)
**Issue:** Both `setInterval` calls store no reference and are never cleared. The graceful-shutdown SIGTERM/SIGINT handlers close the HTTP server and DB but cannot cancel these intervals, so in-flight scheduler ticks can fire after the DB connection is closed, causing unhandled promise rejections and potentially duplicate job inserts on restart.

**Fix:** Capture both interval IDs: `const digestInterval = setInterval(...); const purgeInterval = setInterval(...);`. In the shutdown handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before closing the DB, and await a short drain (e.g. 500 ms) so any already-started tick can finish before the DB pool tears down.

---

### `server/index.ts`: res.json middleware / log(logLine)
**Issue:** Every API response body is JSON-serialised and appended to the server log unconditionally (`logLine += JSON.stringify(capturedJsonResponse)`). For a financial literacy app with school-aged users this will write quiz scores, user profile data, session tokens, or any AI-generated content verbatim to stdout/log aggregators, violating reasonable data-minimisation expectations.

**Fix:** Gate full-body logging behind a development-only flag: `if (process.env.NODE_ENV !== 'production' && capturedJsonResponse) { logLine += ' :: ' + JSON.stringify(capturedJsonResponse); }`. In production log only the status code and duration. If response inspection is needed for debugging, log only a fixed allowlist of non-sensitive top-level keys.

## 🟡 Warnings (10)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues at least four sequential awaits (total income, total expenses, category breakdown, recent transactions) that are fully independent. On a slow Caribbean mobile connection each round-trip compounds latency unnecessarily.
  → *Replace sequential awaits with Promise.all: const [income, expenses, breakdown, recent] = await Promise.all([queryIncome(userId, filters), queryExpenses(userId, filters), queryCategoryBreakdown(userId, filters), queryRecentTransactions(userId, filters)]). Expected p95 latency reduction of 40-60% for dashboard load.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard accepts an optional limit parameter but the interface definition shows no server-side cap, meaning a caller can omit limit and trigger a full table scan across all game sessions joined to users. With school-wide leaderboards this is an unbounded read.
  → *Apply a hard server-side maximum: const safeLimit = Math.min(filters?.limit ?? 50, 100). Add .limit(safeLimit) to the Drizzle query unconditionally. Also add a composite index on (gameSessions.userId, gameSessions.subject, gameSessions.createdAt) to support the period and subject filter paths.*
- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices appears to be called on an interval (based on the method name and typical usage patterns for simulated market data) with no evidence of cleanup or singleton guard. If the module is re-imported during hot reload in development, multiple intervals stack up, causing duplicate price mutations and potential memory leak.
  → *Store the interval reference in a module-level variable: let marketInterval: NodeJS.Timeout | null = null. Expose a stopMarketSimulation() function that calls clearInterval(marketInterval). In the start path, guard with if (marketInterval) return before calling setInterval. Register process.on('SIGTERM', stopMarketSimulation) for production cleanup.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 120)):** Seven Supabase init/seed operations are chained sequentially with .then(), each waiting for the previous to complete. This unnecessarily extends server cold-start time and means a failure in seedFinancialAcademyLesson silently skips all subsequent seeds with only a single catch log.
  → *Run independent operations in parallel and log individual failures: `await Promise.allSettled([initSupabaseTables(), initStaticContentTables()]).then(() => Promise.allSettled([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] seed error', i, r.reason)));`*
- **`server/routes.ts` (robots.txt and sitemap.xml handlers (~lines 45-85)):** Both handlers trust x-forwarded-proto and x-forwarded-host headers without validation and fall back to 'localhost'. On a misconfigured or spoofed proxy, an attacker could inject an arbitrary host into the Sitemap directive, causing search engines to index a malicious URL.
  → *Add a SERVER_BASE_URL environment variable (e.g. `https://finsightlite.com`) and use it as the canonical base: `const base = process.env.SERVER_BASE_URL || \`${req.protocol}://${req.headers.host}\`;` This removes reliance on potentially spoofed forwarded headers.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage', ...)):** The purge-ai-usage handler has no try/catch. An unexpected DB error (connection drop, lock timeout) will surface as an unhandled rejection in the job runner with no structured error context, and the job will silently fail or crash the worker process depending on the runner implementation.
  → *Wrap the handler body in try/catch, import captureError from sentry (same pattern as ai-health-check), log the error with { job: 'purge-ai-usage', olderThanDays, cutoffDate }, and rethrow so the job runner can schedule a retry.*
- **`server/jobHandlers.ts` (openai.chat.completions.create calls – max_completion_tokens: 4096):** Token limits are hardcoded to 4096 in two separate OpenAI calls. This is a magic number with no environment override, making it impossible to tune per-environment (e.g. reduce cost in staging) without a code deploy. It also couples model capacity assumptions to source code.
  → *Add MAX_COMPLETION_TOKENS to environment config (e.g. process.env.OPENAI_MAX_TOKENS ?? '4096') and read it once at module load: const MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10). Reference MAX_TOKENS in both completions calls.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** PDF text is hard-truncated at 30,000 characters with no warning logged and no update to the exam paper record indicating truncation occurred. Teachers uploading long papers will silently get incomplete question extraction with no feedback, and the stored questions will appear complete.
  → *After slicing, check if pdfData.text.length > 30000 and if so: (1) log a structured warning with paperId and original length, (2) call storage.updateExamPaper(paperId, { truncated: true }) if the schema supports it, or include a truncationWarning field in the job return value so the caller can surface this in the UI.*
- **`server/index.ts` (seedDatabase / probeDatabase awaits at startup):** `seedDatabase()` and `probeDatabase()` are awaited sequentially before the HTTP server binds. If either call is slow (cold DB, network latency) the process hangs silently with no timeout, delaying readiness. There is also no try/catch around the combined startup block, so a thrown error produces an unhandled rejection rather than a clean `process.exit(1)` with a diagnostic message.
  → *Wrap the entire async startup IIFE in try/catch that calls `captureError(e); process.exit(1)` on failure. Run independent startup tasks in parallel with `await Promise.all([probeDatabase(), seedDatabase()])`. Add a `Promise.race` timeout (e.g. 15 s) around the DB calls so a hung connection fails fast with a clear error message.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / scheduler hardcoded interval):** The purge and digest scheduler intervals are hardcoded as numeric literals (e.g. every 60 minutes for digest, every N minutes for purge) with no environment override. In staging or CI environments the full interval makes automated testing of the scheduler path impractical, and a misconfigured production deploy cannot be corrected without a code change.
  → *Follow the same pattern already used for `AUTO_PURGE_OLDER_THAN_DAYS`: read intervals from env vars (`DIGEST_CHECK_INTERVAL_MS`, `PURGE_CHECK_INTERVAL_MS`) with safe integer parsing and sensible defaults. Document the vars in `.env.example`.*

## 🔵 Info / Tech Debt (1)

- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint dynamically imports db and drizzle-orm on every request. While infrequent in normal operation, uptime monitors hitting this endpoint repeatedly will trigger repeated module resolution lookups. Additionally, this endpoint is unauthenticated and leaks the fact that a DB is in use and its error messages to the public internet.

## 💡 Opportunities

### Batch category resolution with dataloader pattern: 🔥 High Impact
getBillReminders and getTransactions both resolve category names per-row. Introduce a lightweight in-request cache (a Map keyed by categoryId populated with a single SELECT ... WHERE id IN (...)) to eliminate repeated lookups. For a student with 50 transactions this collapses 50 queries to 1.

### Paginate getTransactions and getGameSessions at storage layer: 🔥 High Impact
Both methods load unbounded result sets into memory. Adding mandatory offset/limit parameters with a default page size of 50 and returning a { data, total } shape will prevent OOM risk as student transaction history grows and enables cursor-based infinite scroll in the UI.

### Extract hardcoded virtual balance seed amount to environment config: ⚡ Medium Impact
The initial virtual balance granted to new users (likely 10000 or similar) is hardcoded in getVirtualBalance or seedMarketData. Moving this to VIRTUAL_BALANCE_SEED_AMOUNT in environment config allows sponsors or schools to customise starting balances without a code deploy.

### Centralised AI route rate limiting middleware: 🔥 High Impact
Extract a reusable aiRateLimiter middleware configured via environment variables (AI_RATE_LIMIT_WINDOW_MS, AI_RATE_LIMIT_MAX) and apply it as a router-level middleware in each AI route file. This gives per-route tuning and avoids the current all-or-nothing gap, reducing runaway OpenAI spend risk.

### Static import all domain routers at module load: ⚡ Medium Impact
The Promise.all dynamic import block defers route registration until registerRoutes runs, adding latency on cold starts and making tree-shaking impossible. Converting these to static top-level imports allows Node.js to resolve and compile all modules before the first request and simplifies the async flow.

### Consolidate duplicate /api/health and /healthz endpoints: 📌 Low Impact
Two health check routes exist (/api/health returning a trivial JSON and /healthz performing a DB ping). Merge them into a single /api/health endpoint that performs the DB ping, guarded by a short cache (e.g. 5s) so uptime monitors do not hammer the database. Remove the duplicate to reduce confusion and maintenance surface.

### Parallel object-storage existence check and header setup in streamPrivateObjectToResponse: 📌 Low Impact
Currently file.exists() is awaited before setting headers and piping. For the happy path this adds one extra RTT. The exists() call exists solely for a clean 404; consider using the stream's error event alone (already handled) and removing the exists() pre-check, saving one storage API call per export download.

### Structured job payload validation at handler entry: ⚡ Medium Impact
No handler validates its payload shape before use. A malformed job (e.g. missing olderThanDays, or NaN after parseInt) causes cryptic runtime errors deep in the handler. Adding a lightweight Zod schema parse at the top of each handler would give immediate, loggable validation errors and prevent partial side-effects from bad payloads.

### Idempotency key on purge-ai-usage to prevent overlapping runs: ⚡ Medium Impact
If the scheduler fires purge-ai-usage while a previous run is still in-flight (slow DB delete on large tables), two concurrent deletes with the same cutoff will race. Adding a job-level idempotency key derived from floor(Date.now() / intervalMs) and checking for an already-running job before enqueuing would eliminate duplicate purge runs with no code complexity beyond what the jobs system likely already supports.

### Structured JSON logging: 🔥 High Impact
Replace the ad-hoc `log()` string concatenation with a structured logger (pino is already common in the Node ecosystem and has near-zero overhead). Each log entry becomes a JSON object with fields like `method`, `path`, `statusCode`, `durationMs`, `userId` (if available). This makes log aggregation and PII scrubbing via field allowlists trivial, and removes the risk of sensitive values being interpolated into free-form strings.

### Scheduler extraction into a dedicated module: ⚡ Medium Impact
Moving `maybeRunWeeklyDigest`, `maybeRunAiUsagePurge`, and their intervals into a `server/scheduler.ts` module that exports `startSchedulers()`/`stopSchedulers()` would make the startup file easier to read, enable unit-testing each scheduler function in isolation with a mocked DB, and make the interval handle lifecycle explicit and testable.

### Startup readiness probe endpoint: ⚡ Medium Impact
Register a `GET /healthz` route before any other middleware that returns 200 only after `probeDatabase()` succeeds and the job worker is running. Kubernetes/Replit health checks can then distinguish a starting process from a broken one, preventing traffic from being routed to an instance whose DB connection has silently dropped.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that should be parallelised. For a school-aged financial app handling real virtual balances and portfolio data, these reliability gaps are particularly risky.

### API Routes (`server/routes.ts`)
The route file is generally well-structured with good separation of concerns and existing rate limiting on auth endpoints. Key concerns are: the Supabase init chain runs sequentially when it could be parallelised, the AI/chat/image/audio routes lack rate limiting despite being expensive OpenAI calls, and the /public-objects endpoint has a path traversal risk via unsanitised filePath.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads stored in the job table create memory/DB pressure, sequential DB inserts in a loop should be batched, and the purge-ai-usage handler lacks a try/catch leaving failures silent. Security posture is reasonable but a hardcoded token limit and missing idempotency on the purge job warrant attention.
