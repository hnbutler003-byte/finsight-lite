## 🔴 Critical (4)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. In a school-aged user app these endpoints are the most expensive and abuse-prone, and a single user or bot can exhaust API quotas or trigger runaway costs.

**Fix:** Create a dedicated aiLimiter (e.g. 10 req/min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 40, ~line 57)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can supply an arbitrary Host header and cause the generated Sitemap or canonical URLs to point to a malicious domain (host-header injection).

**Fix:** Add a validated SITE_BASE_URL environment variable and use it as the canonical base: const base = process.env.SITE_BASE_URL || `${proto}://${host}`; with a startup assertion that SITE_BASE_URL is set in production (NODE_ENV === 'production'). Never trust raw host headers for URLs embedded in served content.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler payload)
**Issue:** Full file content is stored as base64 in the job payload (`fileB64`). For a multi-page PDF or high-res image this can be several megabytes per job row, bloating the jobs table, slowing queue polling queries, and risking OOM if many jobs are loaded simultaneously.

**Fix:** Upload the file to object storage before enqueuing and store only the object path in the payload. In runExtractPaper, download the buffer from object storage at the start of the handler, then delete the temporary object when done. This keeps job rows small (<1 KB) and moves binary data to the appropriate store.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – no authentication header / admin guard
**Issue:** The help-chat widget calls `apiRequest('POST', '/api/admin/help-chat', ...)` but AdminHelpChat is a standalone component with no check against `useAdminAuth`. If the widget is ever rendered conditionally by a parent that has a bug, or if the route is reached without a valid admin session, the LLM endpoint is invoked without credentials, potentially exposing system-prompt content or internal platform data to unauthenticated users.

**Fix:** Move the AdminHelpChat render inside the authenticated branch of AdminDashboard (after the `useAdminAuth` guard resolves), or add `const { admin } = useAdminAuth(); if (!admin) return null;` at the top of AdminHelpChat. The server endpoint must also validate the session independently.

## 🟡 Warnings (9)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 105)):** Seven async Supabase operations are chained sequentially with .then(). A failure in any single step (e.g. seedFinancialAcademyLesson) silently skips all subsequent steps because the rejection is only caught at the very end, making it impossible to distinguish which step failed or retry individual steps.
  → *Replace the chain with individual try/catch blocks or use Promise.allSettled for independent seeds, and log each failure with its step name: await Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()].map(p => p.catch(e => console.error('[Supabase seed]', e.message)))). Keep initSupabaseTables and trimOrgNamesInSupabase sequential if they have ordering dependencies, but wrap each in its own labeled catch.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint dynamically imports 'db' and 'drizzle-orm' on every request. While Node caches modules after the first import, the dynamic import() call itself adds unnecessary async overhead on what is meant to be a fast liveness probe, and exposes DB internals at an unauthenticated endpoint with no IP restriction.
  → *Import db and sql statically at the top of the file alongside other imports. Optionally gate /healthz behind an internal network check or a secret header (req.headers['x-health-token'] === process.env.HEALTH_SECRET) to prevent external probing of DB availability.*
- **`server/jobHandlers.ts` (for loop inside runExtractPaper (~line with `for (let i = 0; i < questions.length; i++)`)):** Each extracted question is inserted with a sequential `await storage.createExtractedQuestion(...)` inside a for-loop. For a 50-question paper this is 50 serial round-trips to the database, adding latency proportional to network RTT × question count and holding the job open unnecessarily long.
  → *Collect all valid question objects into an array, then do a single batched insert: `await db.insert(extractedQuestions).values(validQuestions)`. This reduces N round-trips to 1 and is naturally atomic, which also strengthens the idempotency guarantee.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The `purge-ai-usage` handler has no try/catch. An unexpected DB error (connection drop, constraint violation) will produce an unhandled rejection that surfaces as an opaque job failure with no contextual metadata captured to Sentry, unlike the ai-health-check handler which does capture errors.
  → *Wrap the delete call in try/catch, call `captureError(err, { job: 'purge-ai-usage', olderThanDays })` in the catch block, then rethrow so the job framework still marks the job as failed and can retry. Keep the same pattern already used in ai-health-check.*
- **`server/jobHandlers.ts` (openai.chat.completions.create calls inside runExtractPaper (max_completion_tokens: 4096)):** `max_completion_tokens: 4096` and the model name `gpt-4o` are hardcoded. In a school-facing financial literacy app these directly control cost and output size; changing them requires a code deploy rather than a config change, and they cannot be tuned per environment (dev vs prod).
  → *Add `OPENAI_EXTRACT_MODEL` (default `gpt-4o`) and `OPENAI_EXTRACT_MAX_TOKENS` (default `4096`) to environment config and read them via `process.env`. Validate at startup with a config guard so misconfiguration fails fast rather than silently using a wrong model.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The help-chat POST sends the full accumulated messages array to /api/admin/help-chat on every call with no size cap or rate-limit guard on the client side. A user (or script) can inflate the payload indefinitely, and because the catch block silently swallows the error object, server-side failures (401, 429, 500) are all collapsed into the same generic string, making debugging impossible.
  → *Add a MAX_MESSAGES constant (e.g. 50) and slice next before sending: `const trimmed = next.slice(-MAX_MESSAGES)`. Re-throw or log `err` in the catch: `catch (err) { console.error('[help-chat]', err); setMessages(...) }`. Add a debounce or disable the send button for 1 s after each request.*
- **`client/src/pages/AdminDashboard.tsx` (scroll ref callback inside message list render):** The auto-scroll is implemented as an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` on a dummy div. This creates a new function reference on every render and calls scrollIntoView on every re-render of the chat panel (e.g. while the user is typing in the input), not just when a new message arrives.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (useQuery calls at top of AdminDashboard component):** Multiple independent useQuery hooks (users, schools, teachers, analytics, audit log, etc.) are declared sequentially but each issues its own fetch on mount with no coordination. Where two or more datasets are needed before the page is usable, this produces a waterfall of network requests rather than parallel fetches, adding latency proportional to the number of queries.
  → *For queries that must all succeed before rendering the main UI, combine them with `Promise.all` inside a single queryFn or use `useQueries` from @tanstack/react-query: `const results = useQueries({ queries: [usersQuery, schoolsQuery, teachersQuery] })`. This lets the browser pipeline all requests simultaneously.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline handler functions in JSX (onClick, onChange throughout)):** Throughout the JSX, complex state-update and mutation-trigger functions are defined inline (e.g. table row delete handlers, filter onChange callbacks, pagination onClick). These recreate new function objects on every render, causing every child Button/Select/Input to re-render even when unrelated state changes (e.g. the search input updating triggers recreation of the delete handler for every row).
  → *Extract repeated or complex handlers with useCallback: `const handleDeleteUser = useCallback((id: number) => { deleteUserMutation.mutate(id); }, [deleteUserMutation]);`. Apply the same pattern to pagination and filter handlers. Memoize expensive derived lists (filtered/sorted table data) with useMemo.*

## 💡 Opportunities

### Parallel domain-route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six individual awaits. If any registerXRoutes function does async work (e.g. DB schema checks), they could be parallelised with Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]). This can reduce cold-start time proportionally to the slowest sequential initialiser.

### Centralised error-logging middleware: ⚡ Medium Impact
Error handling is scattered across individual route try/catch blocks with inconsistent console.error calls. Adding an Express error-handling middleware at the bottom of registerRoutes (app.use((err, req, res, next) => ...)) with Sentry capture would centralise error reporting, ensure no unhandled errors silently return 500s, and reduce duplicated catch blocks across route files.

### Environment-validated configuration module: 📌 Low Impact
Values like cache TTL (60*60*24*30 in the object storage handler) and the fallback 'localhost' host are hardcoded inline. Introducing a validated config module using zod or envalid at startup (throwing on missing required vars) would make misconfiguration visible at boot rather than at runtime, and centralise all tunable values for the school deployment context.

### Parallel object-storage existence check and stream setup: 📌 Low Impact
In `streamPrivateObjectToResponse`, `file.exists()` is awaited before streaming begins. If the storage SDK supports it, combining the exists check with a conditional read (or catching a 404 error on createReadStream) removes one extra round-trip per export download, reducing perceived latency for teachers exporting class data.

### Centralised OpenAI error handling and retry classification: 🔥 High Impact
Both OpenAI calls in runExtractPaper can throw rate-limit (429) or server (5xx) errors. Neither is caught or classified, so all failures look identical to the retry scheduler. Adding a thin wrapper that maps 429 to a retryable error with exponential backoff delay and 400 to a non-retryable terminal failure would reduce wasted retries and protect API quota.

### Job payload schema validation at handler entry: ⚡ Medium Impact
Handlers destructure payload fields (e.g. `paperId`, `olderThanDays`) without validating types or presence. A malformed enqueue call produces a confusing mid-handler crash. Adding a Zod schema parse at the top of each handler gives early, readable errors and prevents partial side-effects from running against undefined inputs.

### Centralised loading and error boundary for dashboard data: 🔥 High Impact
Each query currently renders its own ad-hoc loading spinner or silently shows stale data. Wrapping the dashboard in a React Error Boundary and consolidating isLoading/isError states across the parallel useQueries result array would give consistent UX and prevent partial renders where some panels show data while others are still fetching. Expected impact: eliminates several classes of blank-panel bugs reported by school admins.

### Environment-config extraction for hardcoded API strings: ⚡ Medium Impact
Endpoint paths like '/api/admin/help-chat' and any pagination defaults (page size constants) appear to be hardcoded inline. Moving these to a central `config.ts` fed by `import.meta.env` variables means staging and production can differ without code changes and values are easy to audit. Expected impact: reduces misconfiguration deployments and simplifies future endpoint versioning.

### Virtualise long admin tables to prevent DOM bloat: ⚡ Medium Impact
User, school, and teacher tables render every row into the DOM when dataset sizes grow (Caribbean school networks can have thousands of student records). Replacing the current map-to-tr approach with @tanstack/react-virtual or react-window limits DOM nodes to the visible viewport, cutting initial render time and memory usage significantly for admins managing large districts.
