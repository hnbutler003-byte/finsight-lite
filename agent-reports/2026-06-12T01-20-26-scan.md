## 🔴 Critical (7)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting applied to OpenAI-backed chat, image, and audio routes before they are registered. Any authenticated (or unauthenticated, depending on those route handlers) user can hammer these endpoints, incurring unbounded OpenAI API costs and degrading service for all school-aged users.

**Fix:** Create a dedicated `aiLimiter` (e.g. express-rate-limit, 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — adjust path prefixes to match what those register functions actually mount.

### `server/routes.ts` — robots.txt handler (~line 43) and sitemap.xml handler (~line 56)
**Issue:** The `host` and `x-forwarded-host` headers are taken directly from the request and interpolated into output without validation. A spoofed Host header causes the sitemap URL and robots.txt Sitemap directive to reflect attacker-controlled values, enabling cache-poisoning or SEO manipulation.

**Fix:** Introduce an allowlist: read a single `BASE_URL` env var at startup (e.g. `const BASE_URL = process.env.BASE_URL ?? 'https://finsightlite.com'`) and use that instead of deriving the base from request headers. Remove the `x-forwarded-host`/`host` header logic entirely from these two handlers.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** The full file contents (fileB64) are stored in the job payload in the jobs table. For a multi-page PDF or high-res image this can easily exceed several MB per row, causing database bloat, slow job queue queries, and potential OOM when the worker deserialises the payload.

**Fix:** Upload the file to private object storage at enqueue time and store only the object path in the job payload. In runExtractPaper, stream or download the file from object storage using the path instead of decoding a base64 string from the payload.

### `shared/schema.ts` — portfolioHoldings table
**Issue:** There is no unique constraint on (userId, stockId). A user should have exactly one holding row per stock, with quantity updated on buys/sells. Without this constraint, application bugs or race conditions can insert duplicate holding rows, causing double-counted portfolio values — a correctness defect in a financial simulation.

**Fix:** Add a unique index to the portfolioHoldings table definition: (t) => ({ userStockUniq: uniqueIndex('uq_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). Application upsert logic should use INSERT ... ON CONFLICT DO UPDATE.

---

### `server/index.ts` — maybeRunWeeklyDigest / maybeRunAiUsagePurge scheduler setup
**Issue:** setInterval handles for the weekly digest and AI purge schedulers are never stored, so they cannot be cleared during graceful shutdown. The intervals will fire during teardown, potentially writing to a closing DB connection and causing unhandled rejections that mask the real shutdown error.

**Fix:** Capture both handles: `const digestTimer = setInterval(maybeRunWeeklyDigest, ...)` and `const purgeTimer = setInterval(maybeRunAiUsagePurge, ...)`. In the SIGTERM/SIGINT handler call `clearInterval(digestTimer)` and `clearInterval(purgeTimer)` before draining the HTTP server, so no new scheduler ticks fire after shutdown begins.

---

### `server/index.ts` — res.json middleware / logLine construction
**Issue:** The request-logging middleware captures and serialises the full JSON response body of every API response into the log line with no redaction. For a financial literacy app with school-aged users this will log auth tokens, session data, quiz scores tied to user IDs, and any PII returned by any endpoint — a compliance and data-protection violation.

**Fix:** Either remove response-body logging entirely, or maintain an allowlist of safe paths (e.g. `/api/health`) and field-level redact sensitive keys before logging: strip any key matching /token|secret|password|email|dob|userId/i from capturedJsonResponse before calling JSON.stringify. Set a hard length cap (e.g. 200 chars) on the serialised output.

---

### `client/src/pages/AdminDashboard.tsx` — AdminHelpChat — missing admin auth check
**Issue:** AdminHelpChat calls `/api/admin/help-chat` directly from the client. If the component is ever rendered outside the admin auth gate (e.g., during a routing misconfiguration or future refactor), there is no client-side guard preventing the request. The API endpoint bears sole responsibility, which is insufficient defense-in-depth for a FinSight admin surface.

**Fix:** Consume `useAdminAuth()` inside AdminHelpChat and gate the sendMessage call: `const { isAdmin } = useAdminAuth(); if (!isAdmin) return;` at the top of sendMessage. Also ensure the floating button is not rendered at all when isAdmin is false.

## 🟡 Warnings (17)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 117)):** The seven-step sequential seed chain only has a single `.catch` at the end. If any intermediate step rejects, subsequent steps are skipped silently — for example if `seedFinancialAcademyLesson` fails, `initStaticContentTables` and all game/regional content never runs. The `.catch` logs the first error but gives no indication of which steps were skipped.
  → *Run independent seed steps in parallel where possible and wrap each in its own error boundary: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] seed step ${i} failed:`, r.reason)));` Keep `initSupabaseTables` and `initStaticContentTables` sequential only if they create tables the seeds depend on.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 104)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject` without sanitisation. A path traversal payload like `../../private/config` could potentially reach unintended storage paths depending on how `searchPublicObject` constructs its key.
  → *Normalise and validate the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Also enforce an allowlist of valid prefixes (e.g. `logos/`, `videos/`) if the storage bucket holds anything sensitive.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted sequentially inside a for loop with individual awaits. For a 40-question exam this means 40 serial round-trips to the database, multiplying latency and holding the job worker busy unnecessarily.
  → *Collect all valid question objects into an array first, then issue a single batch insert: await db.insert(extractedQuestions).values(validQuestions). Add a guard for the empty-array case to avoid a no-op insert error.*
- **`server/jobHandlers.ts` (runExtractPaper — image branch, visionResponse call):** The code does Buffer.from(fileB64, 'base64') to get a buffer, then immediately calls buffer.toString('base64') to embed it in the vision API request. This is a redundant decode-then-re-encode round-trip that wastes CPU and doubles peak memory for the image data.
  → *Use the original fileB64 string directly in the data URL: `data:${mimeType};base64,${fileB64}`. Drop the intermediate Buffer creation for the image branch, or create the buffer only once and reuse it consistently.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls):** The OpenAI model name ('gpt-4o') and max_completion_tokens (4096) are hardcoded in two separate API calls. Changing the model for cost or capability reasons requires a code deploy rather than a config change, and the two calls could silently drift to different models.
  → *Add OPENAI_EXTRACTION_MODEL and OPENAI_EXTRACTION_MAX_TOKENS to environment config and read them once at module load: const EXTRACTION_MODEL = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'. Reference the constants in both openai calls to keep them in sync.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge handler accepts olderThanDays directly from the job payload with no validation or minimum bound. A misconfigured or malicious enqueue call with olderThanDays=0 would delete all AI usage records; a negative value would delete nothing but silently succeed, masking the misconfiguration.
  → *Validate the payload before use: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'). Consider also enforcing a minimum of 7 or 30 days to prevent accidental mass deletion.*
- **`shared/schema.ts` (categories table, userId column):** userId is declared as varchar with no length argument. Drizzle maps bare varchar to varchar(255) in Postgres, but if the users.id type is a UUID (36 chars) or a CUID/KSUID this is fine by accident. More critically, categories has no index on userId, so every query filtering user-specific vs system categories (WHERE user_id = $1 OR user_id IS NULL) does a full table scan.
  → *Add an explicit length matching the users.id type (e.g. varchar('user_id', { length: 36 }) for UUIDs) and add: index('idx_categories_user_id').on(t.userId) in the table's index callback.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — createdAt columns):** createdAt uses .defaultNow() but is NOT marked .notNull(). Postgres will accept an explicit NULL insert, bypassing the default and leaving audit timestamps as NULL. For a financial app targeting minors, audit trails must be complete.
  → *Chain .notNull() on every createdAt (and executedAt, purchasedAt) that uses defaultNow(), e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The transactions table already does this correctly — apply the same pattern everywhere.*
- **`shared/schema.ts` (billReminders table, nextDueDate column):** billReminders has no index on nextDueDate or userId. The primary query pattern for reminders is 'fetch upcoming bills for user ordered by due date', which requires both columns. Without a composite index this is a sequential scan on every dashboard load.
  → *Add an index callback to billReminders: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })*
- **`shared/schema.ts` (simulatedStocks table, ticker column):** ticker has no unique index. Duplicate ticker symbols (e.g. two rows for 'BTC') would silently exist and cause ambiguous JOIN results in portfolioHoldings and portfolioTransactions queries. There is also no index on ticker despite it being the natural lookup key.
  → *Replace the plain column with a unique constraint and add the index: ticker: text('ticker').notNull(), then in the index callback: (t) => ({ tickerUniq: uniqueIndex('uq_simulated_stocks_ticker').on(t.ticker) })*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, startJobWorker)):** The startup sequence awaits seedDatabase, probeDatabase, registerRoutes, and startJobWorker sequentially even though several of these are independent. On a cold Replit container this adds cumulative latency before the HTTP server binds, increasing the window where the process is unkillable and health checks fail.
  → *Run independent startup tasks in parallel: `await Promise.all([probeDatabase(), seedDatabase()])` then proceed to `registerRoutes` and `startJobWorker`. Ensure seedDatabase is idempotent (it likely already is) and that probeDatabase failure still aborts startup by letting the rejection propagate out of Promise.all.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince = (dow + 0) % 7` — adding zero is a no-op left from an editing artifact, and more critically the time zone is implicitly the server's local time (Replit containers run UTC). 7 pm server-local is not 7 pm Caribbean time (UTC-4/UTC-5), so the digest fires ~4-5 hours early for the target audience.
  → *Replace the bare `new Date()` and `setHours(19,0,0,0)` logic with an explicit UTC offset. Introduce an env var `DIGEST_TZ_OFFSET_HOURS` defaulting to `-4` (AST). Compute the cutoff as `target.setUTCHours(19 - offsetHours, 0, 0, 0)` so the schedule is stable regardless of where the server runs. Also remove the `+ 0` dead expression.*
- **`server/index.ts` (maybeRunWeeklyDigest — audience loop):** The three audience variants (student, teacher, guardian) are processed with sequential awaits inside a for-of loop. Each iteration does a DB insert plus a job enqueue round-trip before the next starts. Under normal conditions this is ~3× slower than necessary and blocks the scheduler callback for the full duration.
  → *Refactor to `await Promise.all(['student','teacher','guardian'].map(async (audience) => { ... }))` so all three claim-and-enqueue operations run concurrently. The onConflictDoNothing insert is already atomic per row so there is no correctness risk from parallelism.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — scroll ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a brand-new function on every render of the messages list. In React, an inline ref callback fires twice per render cycle in StrictMode (called with null then the element), causing redundant scrollIntoView calls and wasted work on every message update.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`. This fires only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function in AdminHelpChat):** The full conversation history (`next`) is sent to `/api/admin/help-chat` on every message. There is no cap on message history length, so a long session will send an ever-growing payload, increasing latency and cost. Combined with no request timeout or abort signal, a slow network leaves `loading` stuck at true with no escape hatch.
  → *1) Slice history before sending: `const trimmed = next.slice(-20)` and send `trimmed`. 2) Use an AbortController with a timeout: `const controller = new AbortController(); const timer = setTimeout(() => controller.abort(), 15000);` and pass `{ signal: controller.signal }` to apiRequest, clearing the timer in finally.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — handleKey and sendMessage defined inline per render):** `sendMessage` and `handleKey` are recreated on every render of AdminHelpChat because they are plain async/arrow functions inside the component body with no useCallback. The input element's onKeyDown and the send button's onClick receive a new function reference every render, preventing any memoization downstream and causing unnecessary child re-renders.
  → *Wrap both in useCallback with correct dependency arrays: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat — error response data exposed in catch-all):** The catch block silently swallows the error and returns a generic string, but the `data` object from a failed JSON parse (e.g., a 500 HTML response) will throw a second unhandled exception inside the try block at `data.reply`, which the outer catch will then catch — masking the real error entirely and making debugging impossible without additional instrumentation.
  → *Check response status before parsing: `if (!res.ok) throw new Error('Server error ' + res.status);` immediately after the await. Then parse JSON only on success. Log the caught error to a structured logger (not console.log with PII) in the catch block: `logger.error('help-chat failed', { status: err.message })`.*

## 💡 Opportunities

### Centralise BASE_URL as startup-time config — 🔥 High Impact
Multiple places (robots.txt, sitemap, potentially emails) reconstruct the app's base URL from request headers. Reading `BASE_URL` once from the environment at module load and exporting it as a shared constant eliminates the header-injection risk, removes duplicated logic, and makes URL construction testable without HTTP.

### Register domain routers in parallel — ⚡ Medium Impact
The six domain routers are imported in parallel via `Promise.all` but then `await`ed sequentially one-by-one. If any `register*` function does async I/O (DB calls, file reads), they block each other unnecessarily. Run them concurrently: `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` — this can meaningfully reduce cold-start latency on Railway.

### Add request-scoped correlation IDs before Sentry middleware — ⚡ Medium Impact
Sentry context is attached after auth but there is no correlation ID injected into `res.locals` or response headers. Adding a short UUID (`crypto.randomUUID()`) as `X-Request-Id` at the top middleware stack and forwarding it to Sentry's `extra` would make cross-service log tracing practical — important for debugging AI call failures tied to specific student sessions.

### Centralise object-storage path helpers — 📌 Low Impact
parsePrivatePath and uploadPrivateObject are defined at module top-level and are reused across handlers, but streamPrivateObjectToResponse duplicates the bucket/object split logic inline. Extract a single parsePath(absOrRel) utility used by all three functions to eliminate the drift risk and reduce tested surface area.

### Move file ingestion out of the job payload entirely — 🔥 High Impact
Beyond fixing the immediate base64-in-payload issue, consider making file upload to object storage a synchronous pre-step in the API route before enqueueing. The job then receives only a stable object path, making the payload tiny, enabling safe retries without re-uploading, and decoupling file storage from job queue schema.

### Add structured logging with sanitisation before any log statements — ⚡ Medium Impact
The truncated file likely contains logging of job payloads. Because payloads may contain base64 file data or user-identifiable fields, ensure any logger strips or truncates payload keys before emission. A thin wrapper that omits known sensitive keys (fileB64, content) would prevent accidental data exposure in log aggregation services.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, etc.). Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it via sql`${DEFAULT_CURRENCY}` or a shared config import. This means a currency change requires one edit instead of six, eliminating drift bugs.

### Add partial index for active/pending document uploads — ⚡ Medium Impact
documentUploads is queried by status='processing' to find in-flight uploads for polling. A partial index CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing' would be tiny in practice and make those polling queries near-instant. Drizzle supports this via .where(sql`status = 'processing'`) in the index definition.

### Replace serial PKs with UUID or CUID2 for user-facing tables — 📌 Low Impact
Sequential integer PKs on transactions, savingsGoals, and portfolioTransactions expose row counts and insertion rates to any user who inspects API responses — a minor but real information leak for a school-aged audience where account enumeration should be prevented. Switching to gen_random_uuid() or CUID2 eliminates enumeration risk and aligns PK type with the users.id foreign key references.

### Structured JSON logging with log levels — 🔥 High Impact
Replace ad-hoc console.log/log() calls with a structured logger (pino is already common in the Node ecosystem and near-zero overhead). Emit JSON lines with level, timestamp, requestId, and userId fields. This makes log-based alerting in Sentry/Datadog trivial and eliminates the current free-form string concatenation that enables accidental PII leakage.

### Formalise graceful shutdown into a single coordinator — 🔥 High Impact
Consolidate SIGTERM and SIGINT handling into one `shutdownCoordinator` function that: (1) stops accepting new connections via httpServer.close(), (2) clears all interval timers, (3) drains the job worker, (4) closes the DB pool, with a hard timeout (e.g. 10 s) that forces process.exit(1). Currently these concerns are likely scattered or absent, making Replit container restarts risk in-flight DB writes.

### Extract scheduler timing into environment config — ⚡ Medium Impact
The digest interval period and purge interval period are hardcoded magic numbers in setInterval calls. Move them to env vars (DIGEST_INTERVAL_MS, PURGE_INTERVAL_MS) with safe defaults. This allows tuning cadence in staging vs production without a deploy and makes the values visible in the runtime config audit trail.

### Decompose AdminDashboard into feature-sliced sub-pages — 🔥 High Impact
At 131k+ characters, this single file almost certainly contains multiple full page sections (users, schools, analytics, audit log, etc.) bundled together. Splitting into lazy-loaded route segments with React.lazy/Suspense would cut initial bundle size and make each section independently testable, reducing regression risk for this school-aged user platform.

### Centralise admin API base URL via environment config — ⚡ Medium Impact
Hardcoded path strings like '/api/admin/help-chat' scattered across a 131k file are a maintenance hazard. Introduce a typed API_ROUTES constant object populated from import.meta.env.VITE_API_BASE (defaulting to '') so all paths are refactorable in one place and staging/production endpoints differ without code changes.

### Add React Error Boundary around AdminHelpChat — ⚡ Medium Impact
A chat widget rendering in a fixed overlay has no isolation from the rest of the dashboard. An unhandled render error in AdminHelpChat currently propagates and unmounts the entire AdminDashboard. Wrapping it in a lightweight ErrorBoundary that catches and shows a fallback prevents the chat from taking down the whole admin UI.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps. ### Server Startup (`server/index.ts`)

Core reliability and security gaps exist despite reasonable guard rails like SESSION_SECRET validation and Sentry integration.
