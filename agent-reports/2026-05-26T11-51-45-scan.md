## 🔴 Critical (6)

### `server/routes.ts` — /public-objects/:filePath(*) handler (~line 78)
**Issue:** Path traversal vulnerability: req.params.filePath is passed directly to objectStorage.searchPublicObject() without sanitization. A request to /public-objects/../../private/secrets could escape the intended directory depending on how searchPublicObject resolves paths.

**Fix:** Add sanitization before use: const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); then verify the resolved path still starts with the expected prefix (e.g. 'logos/', 'videos/'). Reject with 400 if it does not.

### `server/routes.ts` — registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 96)
**Issue:** No rate limiting is applied to the AI/image/audio routes before they are registered. These endpoints call OpenAI APIs which are expensive and latency-sensitive; a single abusive user or bot can exhaust quota or cause cost spikes without any throttling.

**Fix:** Apply a dedicated AI rate limiter before registering these routes: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); where aiLimiter is a stricter windowMs/max config (e.g. 10 requests per minute per IP) defined in your existing rateLimiter module.

---

### `server/jobHandlers.ts` — runExtractPaper / job.payload.fileB64
**Issue:** Full base64-encoded file content is stored in the job table payload. For a multi-page PDF or high-res image this can easily be 5–20 MB per row, bloating the jobs table, slowing queue polling queries, and risking hitting DB row-size limits.

**Fix:** Upload the file to object storage before enqueuing the job and store only the storage path in the payload (e.g., { paperId, objectPath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of the handler using objectStorageClient. Delete the staged object after successful extraction or on final failure.

---

### `shared/schema.ts` — All tables with createdAt, purchasedAt, executedAt columns
**Issue:** Audit timestamp columns (createdAt, purchasedAt, executedAt) use .defaultNow() without .notNull(). This means application code can explicitly insert NULL, corrupting time-series reporting and audit trails — critical for a financial app serving minors.

**Fix:** Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to all tables: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.

---

### `client/src/pages/AdminDashboard.tsx` — AdminHelpChat.sendMessage
**Issue:** The chat endpoint sends the full conversation history (next: ChatMessage[]) to /api/admin/help-chat with no size cap. A user who chats extensively will send an ever-growing payload, and if the server echoes or logs messages, sensitive admin context leaks into server logs. There is also no maximum message count guard client-side.

**Fix:** Slice the history before sending: `const trimmed = next.slice(-20)` and pass trimmed to apiRequest. On the server, ensure /api/admin/help-chat strips any sensitive fields before logging and enforces a token/message limit. Add `console.log` scrubbing middleware so message content never appears in production logs.

---

### `client/src/pages/AdminDashboard.tsx` — Top-level component / route entry point
**Issue:** useAdminAuth is imported and called, but there is no visible early-return or redirect guard before the dashboard JSX renders and before useQuery calls fire. If the hook is async or the token check is delayed, all admin queries (users, schools, analytics) execute and their results render momentarily for unauthenticated users before a redirect occurs.

**Fix:** Immediately after calling useAdminAuth, add: `const { isAdmin, isLoading } = useAdminAuth(); if (isLoading) return <Loader2 className="animate-spin" />; if (!isAdmin) { setLocation('/admin/login'); return null; }` — before any useQuery hooks to prevent data exposure and wasted network calls.

## 🟡 Warnings (14)

- **`server/routes.ts` (Supabase init chain (~line 100)):** The seven Supabase seed/init functions are chained sequentially with .then() when most are independent. Beyond being slow, if any step throws a non-promise error (synchronous throw inside the function before returning a promise), the .catch() at the end will NOT capture it, producing a silent failure.
  → *Wrap the entire block and run independent steps in parallel: (async () => { try { await initSupabaseTables(); await trimOrgNamesInSupabase(); await Promise.all([seedFinancialAcademyLesson(), initStaticContentTables()]); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]); } catch (e) { console.error('[Supabase] Init error:', e); } })();*
- **`server/routes.ts` (/api/health and /healthz (~lines 38 and 55)):** There are two overlapping health check endpoints (/api/health and /healthz). /api/health returns a trivial {status:'ok'} with no real system check, while /healthz does a real DB probe. Railway and uptime monitors will get a false positive 'ok' if pointed at /api/health while the DB is down.
  → *Remove /api/health or consolidate: redirect /api/health to /healthz, or promote /healthz to /api/health and delete the shallow duplicate. Document which endpoint Railway/monitors should use in a comment.*
- **`server/routes.ts` (proto/host construction in /sitemap.xml (~line 45)):** The host value used to construct sitemap URLs is taken directly from req.headers['x-forwarded-host'] or req.headers.host with no validation. A forged Host header will produce sitemap entries pointing to an attacker-controlled domain, which harms SEO and could be used in cache-poisoning attacks.
  → *Use a hardcoded base URL from environment config: const base = process.env.APP_BASE_URL || 'https://finsightlite.com'; Remove the dynamic host/proto construction entirely for this endpoint.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one at a time inside a sequential for-loop. For a 40-question paper this is 40 serial round-trips to the database, each waiting for the previous to complete. This dramatically increases handler latency and holds the DB connection longer than necessary.
  → *Replace the loop with a single bulk insert: filter valid questions into an array of insert records first, then call db.insert(extractedQuestions).values(records) once. Wrap in a transaction with the updateExamPaper call so both succeed or both roll back, eliminating the partial-insert race that the delete-on-retry guard is working around.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays comes directly from job.payload with no validation. A malformed or malicious payload (e.g., olderThanDays = 0 or a negative number) would delete the entire aiUsageEvents table or produce an unexpected cutoff date with no error surfaced to the operator.
  → *Add a guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays value: ' + olderThanDays); This causes the job to fail visibly rather than silently destroying data.*
- **`server/jobHandlers.ts` (runExtractPaper — visionResponse / extractionResponse OpenAI calls):** The two OpenAI API calls (vision text extraction and JSON extraction) are made sequentially when they are actually dependent, but the error path does not distinguish between a transient API timeout and a permanent content error. A network timeout on the second call leaves the paper in 'processing' status until maxAttempts is exhausted, with no backoff signal to the queue.
  → *Wrap each openai.chat.completions.create call in its own try/catch. On a recognized transient error (HTTP 429, 503, or a network timeout), re-throw a typed RetryableError so the job queue can apply exponential backoff. On a permanent content error (empty choices array, JSON parse failure), call storage.updateExamPaper(paperId, { status: 'failed' }) immediately and throw a non-retryable error to avoid burning remaining attempts.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — res.setHeader Content-Disposition):** fileName is interpolated directly into the Content-Disposition header value without sanitization. If fileName contains a double-quote or newline character (possible if derived from user-supplied data elsewhere in the truncated code), this enables header injection.
  → *Sanitize fileName before use: const safeFileName = fileName.replace(/["
]/g, '_'); then use safeFileName in the header. For full RFC 6266 compliance, also encode non-ASCII characters using encodeURIComponent and emit both filename and filename* parameters.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups by user will be the dominant query pattern. A full table scan occurs on every dashboard load for every user.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to enforce one budget per category per period at the DB level.*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, portfolioTransactions table definitions):** None of these four tables define indexes on userId, despite being queried per-user constantly. portfolioTransactions also lacks an index on stockId used in JOIN conditions.
  → *Add index definitions to each table: billReminders on (userId, nextDueDate) to support due-date filtering; savingsGoals on (userId); portfolioHoldings on (userId, stockId) as a composite; portfolioTransactions on (userId) and separately on (stockId).*
- **`shared/schema.ts` (userId columns across all tables (varchar without length)):** userId is declared as varchar() with no length limit on every table. In PostgreSQL, unbounded varchar is functionally equivalent to text but signals no intent and bypasses length validation. More critically, there is no shared type constant, so if the auth provider changes ID length, each table must be updated manually and inconsistencies will cause FK violations silently.
  → *Define a single constant: const userIdCol = () => varchar('user_id', { length: 128 }) and reuse it in every table. Pick 128 to accommodate UUIDs, Clerk IDs, and Auth0 sub claims. This centralises any future length change to one line.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** The ticker column has no unique index. Duplicate tickers inserted via seed scripts or admin tooling will cause incorrect portfolio valuation calculations when JOINing by ticker string, and there is no DB-level guard against it.
  → *Add a unique index: (t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This also enables efficient O(log n) lookup by ticker for price-update jobs.*
- **`client/src/pages/AdminDashboard.tsx` (useEffect / scroll ref inside AdminHelpChat messages list):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new function on every render. Because it is not a stable ref, React will call it with null then the element on every messages update, firing scrollIntoView redundantly and preventing garbage collection of detached DOM nodes when the chat panel unmounts while a message is mid-render.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading])`. Assign `ref={bottomRef}` to the sentinel div. This is stable, runs only when content changes, and cleans up correctly on unmount.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple parallel useQuery calls for dashboard stats):** Several independent useQuery calls (users, schools, lessons, analytics, audit log, etc.) are declared sequentially. While React Query fires them concurrently, each has its own loading/error state that is not surfaced individually. When any one fails, the dashboard silently shows partial data with no per-section error boundary or retry UI, leaving school-aged-user admin staff unaware of stale data.
  → *Wrap each data section card with its own error/loading guard: `if (usersQuery.isError) return <ErrorCard retry={usersQuery.refetch} />`. Alternatively, use `useQueries` to manage them as a group and derive a combined `isAnyError` flag for a top-level alert banner, while still showing successfully loaded sections.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort functions in JSX (table rendering sections)):** Filtering and sorting of users, schools, and lessons are computed with inline expressions or functions defined inside the JSX return, regenerated on every render. In a dashboard with hundreds of rows this causes measurable jank, especially on low-end Caribbean school devices where FinSight Lite is targeted.
  → *Extract each filtered/sorted list into a useMemo: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(search.toLowerCase())).sort(...), [users, search, sortField, sortDir])`. Apply the same pattern for schools and lessons. This limits recomputation to when the relevant dependencies change.*

## 💡 Opportunities

### Centralise route registration error handling — 🔥 High Impact
Each registerXxxRoutes(app) call is awaited individually with no try/catch. A failure in one (e.g. registerAiRoutes) silently skips remaining registrations and the server starts in a partially configured state. Wrap the block in a single try/catch that either rethrows (crashing startup cleanly) or logs clearly which domain failed.

### Move Supabase seeding out of the request-path startup — ⚡ Medium Impact
Running seven seed operations at server start adds latency to cold boots and pollutes route registration with infrastructure concerns. Extract to a separate cli script or a one-time migration job triggered by an env flag (RUN_SEED=true), keeping registerRoutes lean and focused solely on route wiring.

### Add auth middleware guard to /public-objects route — ⚡ Medium Impact
Even after path sanitization, school-aged user data (profile images, documents) served from object storage should verify the requester is authenticated or that the asset is genuinely public. Add a lightweight isPublicAsset(filePath) check and gate non-public paths behind session auth to prevent enumeration of other users' files.

### Batch DB writes inside extract-paper with a transaction — 🔥 High Impact
Combining the bulk question insert and the updateExamPaper call into a single DB transaction eliminates the window where questions exist but the paper status is still 'processing'. This removes the need for the delete-on-retry idempotency guard entirely, simplifying the handler and reducing total DB round-trips from O(n+1) to O(1).

### Move file decoding out of the job queue into a pre-upload step — 🔥 High Impact
Storing file content in the job payload ties queue performance to file size. Shifting to a pattern where the API route uploads to object storage and enqueues only a path keeps the jobs table lean, allows queue polling to remain fast as volume grows, and makes payload serialization/deserialization significantly cheaper across all retries.

### Centralize OpenAI call error classification — ⚡ Medium Impact
Multiple handlers in this file call OpenAI independently with no shared retry/error-classification logic. Extracting a callOpenAI(params) wrapper that catches 429/503 and throws a RetryableError, and catches parse failures as PermanentError, would make all handlers behave consistently under API instability without duplicating logic.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Define const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' in a config module and reference it in each .default() call. This makes currency changes for expansion to other Caribbean markets a single-line config change rather than a six-table migration.

### Add partial index on documentUploads for processing status — ⚡ Medium Impact
Background workers polling for documents in 'processing' status will scan the full documentUploads table as it grows. Add a partial index: index('idx_document_uploads_processing').on(t.status).where(sql`status = 'processing'`). For a school-aged user base with high upload volume during term time, this reduces worker poll query cost by orders of magnitude.

### Replace learningModules content text column with structured jsonb — 🔥 High Impact
Storing lesson content as raw text forces all parsing and validation into application code and prevents DB-level querying of content structure (e.g. finding all modules with video embeds). Migrating to jsonb with a validated JSON Schema allows content queries, partial updates of individual content blocks, and GIN indexing for full-text search across lesson content — directly valuable for the financial literacy feature set.

### Split dashboard into lazy-loaded tab modules — 🔥 High Impact
The file is 131k+ characters covering users, schools, lessons, analytics, audit logs, and chat. Splitting each tab into a React.lazy() module reduces the initial JS parse time by an estimated 60-70%, which is significant for school networks with limited bandwidth in the Caribbean.

### Centralise hardcoded API path strings into an env-driven config — ⚡ Medium Impact
Paths like '/api/admin/help-chat', '/api/admin/users', etc. appear as string literals throughout the file. Moving them to a single apiRoutes constant (populated from import.meta.env) makes environment-specific overrides (staging vs production) safe and eliminates typo-class bugs across all admin mutations.

### Add useCallback to all mutation handler functions passed as props — ⚡ Medium Impact
Handlers like onDelete, onEdit, onSave passed into table row components are recreated every render, causing every row to re-render when any parent state changes. Wrapping them in useCallback with correct dependency arrays, combined with React.memo on row components, will eliminate O(n) re-renders on search input changes.
