## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the category and a separate aggregation query calculates spent amount. With 20 budgets this fires 41 queries per dashboard load.

**Fix:** Rewrite using a single JOIN query with Drizzle: join budgets to categories on categoryId, and use a LEFT JOIN with SUM(transactions.amount) grouped by budgetId and filtered by the current period. Return all data in one round-trip. Example: db.select({...budgetFields, categoryName: categories.name, spent: sql<number>`COALESCE(SUM(${transactions.amount}),0)`}).from(budgets).leftJoin(categories,...).leftJoin(transactions,...).groupBy(budgets.id).where(eq(budgets.userId, userId))

### `server/storage.ts`: getTransactions / getDashboardStats
**Issue:** getTransactions has no enforced maximum limit. An unbounded query with no pagination guard can return tens of thousands of rows into server memory, and getDashboardStats calls it without a limit when computing totals, compounding the risk.

**Fix:** Add a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 100, 500); and apply .limit(safeLimit) unconditionally. For getDashboardStats, replace the full getTransactions call with a single aggregation SQL query (SUM/COUNT grouped by type) instead of fetching all rows and reducing in JS.

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Buy/sell flow calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three separate awaits with no wrapping database transaction. A process crash between any two steps leaves the portfolio holding updated but the ledger record or balance missing, corrupting the user's virtual account.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { await tx.insert/update holding; await tx.insert portfolioTransaction; await tx.update virtualBalance; }). Drizzle's transaction callback rolls back all three on any thrown error, ensuring atomicity.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 108-110)
**Issue:** AI and media generation routes (chat, image, audio) are registered with no rate limiting middleware. These call expensive third-party APIs (OpenAI) and are trivially abusable by school-aged users or bots, leading to runaway costs or denial-of-service.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 42) and sitemap.xml handler (~line 57)
**Issue:** The `host` and `proto` values are read directly from user-controlled request headers (`x-forwarded-host`, `x-forwarded-proto`, `host`) with no validation and interpolated into response bodies. An attacker can inject an arbitrary base URL into the Sitemap directive and all sitemap <loc> values, enabling cache-poisoning or phishing via Google's sitemap index.

**Fix:** Add a trusted-host allowlist using an environment variable: `const ALLOWED_HOST = process.env.SITE_HOST || 'finsightlite.com'; const host = ALLOWED_HOST;` and ignore the header-derived value entirely for the base URL, or at minimum validate the derived host against the allowlist before use.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter / job.payload)
**Issue:** The full file contents are stored as a base64 string inside the job payload row. For a multi-page PDF this can easily be 5-20 MB per job row, bloating the jobs table, slowing queue polling queries, and leaking raw exam paper content to anything that reads the jobs table (logs, admin UIs, DB backups with weaker ACLs).

**Fix:** Upload the file to object storage immediately on ingest (before enqueuing), store only the resulting object path in the job payload, then download the file at the start of runExtractPaper. Remove fileB64 and ext from JobPayloads['extract-paper'] and replace with objectPath: string. This also removes the idempotency concern around re-uploading on retry.

---

### `shared/schema.ts`: simulatedStocks table, ticker column
**Issue:** ticker has no unique constraint. Two rows with ticker 'BTC' can be inserted silently, causing portfolio holdings and transactions to reference different stock rows for the same instrument, corrupting simulation P&L calculations.

**Fix:** Replace text('ticker').notNull() with text('ticker').notNull() and add a uniqueIndex in the table callback: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }).

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / admin mutation endpoints
**Issue:** The component relies on `useAdminAuth` for client-side route guarding, but if any `useMutation` calls (add teacher, delete user, etc.) do not include the admin session token in headers AND if the server routes lack their own auth middleware, a user who bypasses the React guard can call admin API endpoints directly. Client-side-only auth on a financial literacy platform with school-age user data is a critical gap.

**Fix:** Verify every `/api/admin/*` Express route has an `isAdminAuthenticated` middleware guard server-side. In `apiRequest`, ensure the admin session cookie or Bearer token is always forwarded. Add an integration test that hits each admin mutation endpoint without credentials and asserts a 401/403 response.

## 🟡 Warnings (17)

- **`server/storage.ts` (getLeaderboard):** getLeaderboard fetches game sessions then performs a separate getUser (or equivalent) lookup per unique userId to resolve display names and avatars, creating an N+1 pattern. With no limit guard this scans the full gameSessions table.
  → *Add .limit(filters?.limit ?? 50) capped at 100 as a guard. Replace the per-user lookup with a single JOIN to the users table inside the aggregation query: db.select({ userId, userName: users.firstName, avatar: users.profileImageUrl, totalScore: sql`SUM(score)`, gamesPlayed: sql`COUNT(*)` }).from(gameSessions).innerJoin(users, eq(gameSessions.userId, users.id)).groupBy(gameSessions.userId).orderBy(desc(sql`SUM(score)`)).limit(safeLimit)*
- **`server/storage.ts` (seedMarketData / seedLearningModules):** Seed functions insert rows with no conflict guard and are apparently called at startup, meaning every server restart either throws a unique-constraint error (swallowed silently) or duplicates seed data if constraints are missing. No transaction wraps the bulk inserts.
  → *Use Drizzle's onConflictDoNothing() on each insert: db.insert(simulatedStocks).values(stockData).onConflictDoNothing(). Wrap the entire seed block in db.transaction() so a partial seed failure rolls back cleanly. Gate execution with an environment check: if (process.env.NODE_ENV === 'production' && process.env.SKIP_SEED === 'true') return;*
- **`server/routes.ts` (Supabase init chain (~line 113-121)):** The sequential `.then()` chain means a failure in any intermediate step (e.g. `seedFinancialAcademyLesson`) silently skips all subsequent steps and only logs a single generic error. In production this leaves DB tables partially initialised with no per-step diagnostic.
  → *Run independent seed operations with `Promise.allSettled` and log each rejection individually: `const results = await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]); results.forEach((r,i) => r.status==='rejected' && console.error('[Supabase] seed step', i, r.reason));` Keep `initSupabaseTables` and `initStaticContentTables` sequential if they are true prerequisites, but separate their errors.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96)):** There is no authentication or authorization check before serving objects from storage. Any unauthenticated request can enumerate and download any file by guessing paths, including potentially sensitive org logos or user-uploaded content.
  → *If assets are meant to be public, document that explicitly and restrict allowed path prefixes (e.g. only `logos/` and `videos/`). If any assets are private, add a session/auth middleware guard before this route. At minimum, validate `filePath` against a regex whitelist to prevent path traversal: `if (!/^[\w\-./]+$/.test(filePath)) return res.status(400).end();`*
- **`server/routes.ts` (/healthz handler (~line 77)):** The `/healthz` endpoint performs a live DB query (`select 1`) on every call with no rate limiting. Uptime monitors calling this every 30 seconds is fine, but the endpoint is publicly reachable and could be used to flood the DB connection pool.
  → *Apply a tight rate limiter (e.g. 10 req/min per IP) or restrict the route to internal/loopback IPs: `if (!['127.0.0.1','::1'].includes(req.ip) && req.headers['x-internal-token'] !== process.env.HEALTH_TOKEN) return res.status(403).end();` Alternatively cache the result for 5 seconds to avoid redundant DB hits.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop. For a 60-question paper this fires 60 round-trips to the database serially, adding significant latency and holding the job open longer than necessary. There is no transaction wrapping these inserts, so a crash mid-loop leaves partial data that the idempotency delete at the top of a retry will silently discard without logging.
  → *Collect all valid question objects into an array, then execute a single db.insert(extractedQuestions).values(allQuestions) call wrapped in a db.transaction(). This reduces N round-trips to 1 and makes the write atomic, eliminating the need for the pre-delete idempotency hack.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (model and max_completion_tokens)):** The OpenAI model name 'gpt-4o' and token limits (4096) are hardcoded in two separate call sites. If the model needs to be swapped for cost or availability reasons (common in a cost-sensitive educational app), every call site must be found and changed manually, with no way to change it at runtime without a redeploy.
  → *Add OPENAI_VISION_MODEL, OPENAI_EXTRACTION_MODEL, and OPENAI_MAX_TOKENS to environment config with documented defaults. Read them at module load time: const EXTRACTION_MODEL = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'. Apply consistently to all openai call sites in this file.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse – createReadStream error handler):** The readable stream's error event handler calls res.status(500).json() only if headers have not been sent, but if headers were sent mid-stream the error is silently swallowed with no logging. For a school-aged user this results in a truncated download with no feedback, and ops has no visibility into the failure.
  → *In the else branch of the !res.headersSent check, add: console.error('[streamPrivateObject] mid-stream error for', objectPath, err). Also call res.destroy(err) to cleanly terminate the response socket rather than leaving it open, which can cause the client to hang.*
- **`server/jobHandlers.ts` (purge-ai-usage handler – olderThanDays from job.payload):** olderThanDays is taken directly from the job payload with no validation. A payload of 0 or a negative number would compute a future cutoff date and delete all rows, or a payload of 1 could purge data that compliance or audit requirements mandate retaining. There is no minimum floor enforced.
  → *Add a guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 30) throw new Error('olderThanDays must be a number >= 30'). Define the minimum retention period as an env var PURGE_AI_USAGE_MIN_DAYS defaulting to 30, and validate against it rather than a magic number.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, so any query fetching a user's budgets (the most common read pattern) requires a full table scan. As transaction volume grows this will be the first table to cause latency spikes.
  → *Add an index in the table's index callback: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget rows per category per period.*
- **`shared/schema.ts` (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders — createdAt columns):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). The DB default fires on INSERT but application-level Drizzle inserts that explicitly pass null will store NULL, breaking any ORDER BY createdAt or audit queries silently.
  → *Append .notNull() to every createdAt definition, e.g. timestamp('created_at').defaultNow().notNull(). Also apply the same fix to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions tables):** Neither table has an index on userId or stockId. Portfolio page queries joining holdings to transactions for a single user will perform sequential scans on both tables, which is unacceptable for a real-time simulation dashboard.
  → *Add index callbacks to both tables: portfolioHoldings gets index('idx_portfolio_holdings_user').on(t.userId) and index('idx_portfolio_holdings_stock').on(t.stockId); portfolioTransactions gets index('idx_portfolio_txn_user_executed').on(t.userId, t.executedAt) for time-ordered history queries.*
- **`shared/schema.ts` (categories table, userId column):** userId on categories is a bare varchar with no length specified and no index. Queries that filter categories by userId (e.g. fetching a user's custom categories merged with system categories) will scan the full table. Additionally the lack of a length cap on varchar is inconsistent with users.id and may mask data truncation bugs.
  → *Change to varchar('user_id', { length: 36 }) to match UUID length of users.id, and add (t) => ({ userIdx: index('idx_categories_user').on(t.userId) }) so per-user category lookups use an index scan.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and fires scrollIntoView on every message list re-render including the loading spinner toggle, causing janky scroll behavior and unnecessary DOM thrashing.
  → *Replace with a `useRef` + `useEffect`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – apiRequest POST /api/admin/help-chat):** The full conversation history (`messages: next`) is sent to the server on every message with no length cap. For a school-aged app session that stays open, this array grows unbounded, eventually sending kilobytes of chat history per request and risking hitting API token/payload limits silently.
  → *Slice the history before sending: `messages: next.slice(-20)` (keep last 20 turns). Add a visible message count cap (e.g. 100) and reset or warn the user. Also validate `data.reply` is a string before setting state to avoid XSS via injected HTML if the field is later rendered with dangerouslySetInnerHTML.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage):** The catch block swallows the actual error with no logging, making production debugging impossible. The error object is completely discarded (`} catch {`), so transient auth failures, 403s from the admin endpoint, and network errors are all silently collapsed into the same user message.
  → *Change to `catch (err)` and log: `console.error('[AdminHelpChat] sendMessage failed:', err)`. Additionally inspect the response status before calling `res.json()` — if `!res.ok` throw with status so auth errors surface distinctly (e.g. redirect to login on 401).*
- **`client/src/pages/AdminDashboard.tsx` (Large inline JSX functions / event handlers throughout dashboard):** Based on visible patterns, chart data transformations, filter functions, and table row renderers are defined as inline arrow functions inside JSX or inside the component body without useCallback/useMemo. On a dashboard with potentially hundreds of student/school rows, these recompute and cause child re-renders on every keystroke in search inputs.
  → *Wrap expensive derivations (filtered user lists, chart data aggregations) in `useMemo` with appropriate deps arrays. Wrap mutation callbacks passed as props (delete handlers, edit openers) in `useCallback`. For the search input specifically, add a 300ms debounce using `useRef` + `setTimeout` cleanup to avoid filtering on every keystroke.*

## 💡 Opportunities

### Parallel async fetches in dashboard aggregation: 🔥 High Impact
getDashboardStats currently awaits transactions, then budgets, then savings goals sequentially. Converting these to Promise.all([getTransactions(...), getBudgets(...), getSavingsGoals(...)]) would reduce dashboard response time by roughly the sum of the two slowest individual query times, estimated 40-80ms on typical Caribbean network latencies to a hosted DB.

### Centralised pagination defaults via config: ⚡ Medium Impact
Default limits (100 transactions, 50 leaderboard rows, etc.) are scattered as magic numbers throughout storage methods. Extract them to a single server/config.ts object (PAGINATION = { transactions: 100, leaderboard: 50, maxAllowed: 500 }) so they can be tuned via environment variables without hunting through query files. This also prevents a future developer from unknowingly removing the only guard.

### Audit log writes outside transaction scope: 🔥 High Impact
deletionLog inserts that record GDPR-style deletions appear to be written after the delete completes but outside any transaction. If the audit insert fails, the record is deleted with no trace. Wrap delete + audit-log insert in a single db.transaction() to guarantee both succeed or both roll back, which is a compliance requirement for a platform handling minors' data.

### Centralise base-URL resolution into a single helper: ⚡ Medium Impact
The host/proto header logic is duplicated verbatim in both robots.txt and sitemap.xml handlers. Extract a `getBaseUrl(req)` utility in a shared module that validates against SITE_HOST env var. Eliminates the duplication and ensures any future security fix is applied in one place.

### Lazy-load heavy domain routers to reduce cold-start blocking: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all but their registration (`await registerXRoutes`) is still sequential and synchronous in effect. If any router's module-level code does I/O, it stalls subsequent registrations. Wrap each registration in a try/catch so one broken router does not abort the server entirely, and log per-router timing to identify slow modules.

### Replace hardcoded fallback 'localhost' with SITE_HOST env var: 📌 Low Impact
Both robots.txt and sitemap.xml fall back to `'localhost'` when headers are absent (e.g. during Railway health probes). This can result in `localhost` being emitted in production sitemaps submitted to Google. Set `SITE_HOST=finsightlite.com` in environment config and use it as the authoritative fallback, eliminating the localhost leak entirely.

### Centralise OpenAI call instrumentation: 🔥 High Impact
Every openai.chat.completions.create call in this file independently handles errors and has no latency or token-usage logging. Wrapping calls in a thin helper that records model, prompt tokens, completion tokens, latency, and paperId to aiUsageEvents would give cost visibility per job type with no repeated boilerplate, and would directly feed the existing purge-ai-usage job with meaningful data.

### Replace per-question sequential inserts with bulk insert transaction: 🔥 High Impact
Beyond fixing the critical finding, adopting a bulk-insert pattern throughout jobHandlers.ts (admin CSV export rows, badge grants, XP updates) would reduce average job execution time measurably under load and lower DB connection pressure, which matters on a single-instance Caribbean school deployment likely running on constrained infrastructure.

### Add job payload schema validation at handler entry: ⚡ Medium Impact
No handler validates the shape of job.payload before destructuring it. Using a lightweight Zod schema per job type at the top of each handler would catch malformed enqueue calls at runtime with a clear error rather than a confusing undefined-property crash deep in business logic, and would serve as living documentation of each job's contract.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a column default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a module-level constant DEFAULT_CURRENCY = 'BSD' and reference it in each .default() call. When the app expands to TTD or JMD support, one change propagates everywhere and migration scripts become trivially auditable.

### Add composite unique constraint on portfolioHoldings (userId, stockId): 🔥 High Impact
Without this constraint, application bugs or race conditions on simultaneous buy orders can create two separate holding rows for the same user and stock, causing quantity and avgPurchasePrice calculations to double-count. A unique index forces the application to use an upsert pattern, which is the correct model for a holdings ledger.

### Replace text columns used as enums with pgEnum for type reuse: ⚡ Medium Impact
type, period, frequency, status, riskLevel and others are defined as inline text enums on each table independently. Drizzle's pgEnum creates a reusable Postgres ENUM type, eliminating the risk of mismatched allowed values between tables (e.g. transaction.type vs category.type both use income/expense but are defined separately) and producing cleaner migration diffs.

### Parallel data fetching with Promise.all: 🔥 High Impact
If the dashboard fires multiple independent useQuery calls sequentially (users, schools, analytics, leaderboard), they waterfall by default when used with sequential awaits in any loader or prefetch logic. Consolidate into a single `/api/admin/dashboard-summary` endpoint or use `Promise.all` in a prefetch to cut initial load time by up to 60% on slow Caribbean mobile connections.

### Virtualized list rendering for large student/school tables: 🔥 High Impact
Rendering hundreds of table rows in DOM simultaneously causes layout thrashing and slow paint on low-end devices common in school environments. Integrate `@tanstack/react-virtual` for the user and school tables to render only visible rows, reducing DOM nodes by ~90% and making scroll buttery smooth.

### Extract AdminHelpChat into a lazy-loaded chunk: ⚡ Medium Impact
The chat component including its state, Lucide icons, and API logic is bundled into the main AdminDashboard chunk even though it is hidden by default. Wrapping it in `React.lazy` + `Suspense` would shave it from the initial JS parse cost, improving Time-to-Interactive for admins who never open the chat.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits classic N+1 query patterns in budget and bill-reminder fetchers, missing database transactions on multi-step writes, and sequential awaits where Promise.all would reduce latency. For a school-aged financial app, the absence of pagination guards on unbounded queries and potential exposure of sensitive data in logs are the most pressing reliability and security concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with rate limiting, parallel imports, and error handling in most async paths. Key concerns are: unprotected AI/heavy routes lacking rate limits, potential host header injection in robots.txt and sitemap, and a fire-and-forget Supabase init chain that swallows individual step failures silently.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads are embedded directly in job rows (a known anti-pattern for job queues), sequential DB inserts in a loop should be batched, and the file includes a hardcoded model name and token limit that should be environment-configurable. Security is generally adequate but sensitive file content transits the job table in plaintext base64.

### Database Schema (`shared/schema.ts`)
The schema is well-structured with proper foreign keys and some composite indexes, but has several reliability and performance gaps. Key concerns include missing NOT NULL constraints on audit timestamps, absent indexes on high-cardinality foreign key and filter columns, and no unique constraint enforcement on ticker symbols that would cause silent duplicates in the investment simulation.
