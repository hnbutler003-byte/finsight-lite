## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 115-117)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. On a school-aged user platform these are the most expensive endpoints per request, and any authenticated (or unauthenticated) user can hammer them, causing runaway OpenAI spend and denial-of-service for other users.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` placed immediately before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap handler (~line 60)
**Issue:** Both handlers build absolute URLs directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header and inject an arbitrary domain into the Sitemap directive and all sitemap <loc> entries, enabling cache-poisoning or phishing via shared CDN caches.

**Fix:** Add a validated base URL from environment config: `const BASE_URL = process.env.BASE_URL;` (required at startup). Use it as the source of truth and ignore forwarded headers entirely for URL construction: `const base = BASE_URL || 'https://finsightlite.com';`. Remove the dynamic header-derived `proto`/`host` logic from both handlers.

---

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored in the job payload row in the database. For a multi-page PDF or high-res image this can easily exceed 5-20 MB per job row, bloating the jobs table, slowing queue polls, and risking OOM when the worker deserializes the payload.

**Fix:** Upload the file to object storage before enqueuing, store only the object storage path in the job payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. Delete the temp object after successful extraction.

---

### `server/index.ts`: startJobWorker / seedDatabase / probeDatabase call site in startup IIFE
**Issue:** Startup awaits probeDatabase, seedDatabase, and startJobWorker sequentially. If any of these hangs (e.g. DB unreachable with no timeout), the HTTP server never starts listening and there is no top-level timeout to abort and exit, leaving the process alive but useless.

**Fix:** Wrap the entire startup IIFE in Promise.race against a hard timeout (e.g. 30 s via a rejected promise) that calls process.exit(1). Additionally, probeDatabase and seedDatabase are independent and can be replaced with `await Promise.all([probeDatabase(), seedDatabase()])` to run concurrently, saving wall-clock startup time.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler setInterval calls return handles that are never stored, so they cannot be cleared during graceful shutdown. When SIGTERM is received, in-flight scheduler ticks may still be executing DB writes or enqueuing jobs while the process is tearing down, causing partial or duplicate records.

**Fix:** Store each handle: `const weeklyHandle = setInterval(...)`. In the SIGTERM/SIGINT handler call `clearInterval(weeklyHandle)` and `clearInterval(purgeHandle)` before draining and closing the HTTP server. Also await a short settling delay after clearing to let any in-progress tick complete.

---

### `server/index.ts`: res.json middleware / logLine construction in app.use finish handler
**Issue:** Every API response body is captured in `capturedJsonResponse` and unconditionally serialized with JSON.stringify into the console log. For FinSight Lite this risks logging auth tokens, session data, student PII, or financial records in plaintext to stdout/log aggregators with no redaction or size cap.

**Fix:** Gate response body logging behind a `NODE_ENV !== 'production'` check, or implement an allowlist of safe routes (e.g. only log bodies for non-sensitive endpoints). At minimum add a byte-length guard: only log if `JSON.stringify(capturedJsonResponse).length < 200`. Never log routes under `/api/auth`, `/api/user`, or any route returning user financial data.

## 🟡 Warnings (12)

- **`server/routes.ts` (Supabase init chain (~line 120-128)):** The seven Supabase init steps are chained sequentially with `.then()`. A failure in any middle step (e.g. `seedFinancialAcademyLesson`) skips all subsequent seeds silently—only one `.catch` exists at the end, so `seedStaticModules`, `seedRegionalContent`, and `seedGameContent` never run if an earlier step rejects. Students may hit missing game or regional content at runtime.
  → *Run independent seeds in parallel with individual error isolation using Promise.allSettled: `const steps = [trimOrgNamesInSupabase, seedFinancialAcademyLesson, initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent]; await initSupabaseTables(); const results = await Promise.allSettled(steps.map(fn => fn())); results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] seed step', i, 'failed:', r.reason));`*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 102)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject()` without path traversal sanitisation. A request to `/public-objects/../../private/keys` could resolve to unintended storage paths depending on the ObjectStorageService implementation.
  → *Normalise and validate the path before use: `import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Place this before the searchPublicObject call.*
- **`server/routes.ts` (/healthz handler (~line 79)):** The /healthz endpoint performs a live DB query (`select 1`) on every call and is fully unauthenticated with no rate limiting. Uptime monitors typically poll every 30s, but this endpoint is also publicly reachable—an attacker can use it to exhaust DB connection pool slots with concurrent requests.
  → *Add the existing `authLimiter` (or a dedicated lax limiter, e.g. 60 req/min) to this route: `app.get('/healthz', healthLimiter, async (_req, res) => {...})`. Alternatively cache the last DB check result for 5 seconds to avoid redundant queries from legitimate monitors.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this means 40 round-trips to the database executed one-at-a-time, adding hundreds of milliseconds of latency and holding the job worker busy unnecessarily.
  → *Collect valid questions into an array and perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions) where validQuestions is built from the filtered loop. Add the order field during the map step. This reduces 40 round-trips to 1.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The db.delete call has no try/catch. If the DELETE statement fails (deadlock, connection drop) the job will throw an unhandled rejection that bypasses any structured error logging or retry logic already present in other handlers, and the job framework may silently swallow it depending on implementation.
  → *Wrap the db.delete in try/catch, log the error with context (olderThanDays, cutoff), and rethrow so the job framework can record the failure and schedule a retry: try { const result = await db.delete(...); ... } catch (err) { capture(err, { job: 'purge-ai-usage', cutoff }); throw err; }*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (max_completion_tokens: 4096)):** The token limit 4096 and model name 'gpt-4o' are hardcoded in two separate places. Changing model or limits for cost control or compliance requires a code deploy rather than a config change, which is a reliability and operational risk for a school app with tight budget constraints.
  → *Add OPENAI_EXTRACTION_MODEL (default 'gpt-4o') and OPENAI_EXTRACTION_MAX_TOKENS (default 4096) to environment config and reference them as process.env.OPENAI_EXTRACTION_MODEL and parseInt(process.env.OPENAI_EXTRACTION_MAX_TOKENS ?? '4096', 10) in both completions calls.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function uses `new Date()` and `d.setHours(19, 0, 0, 0)` which applies the server's local timezone. On a UTC-based host (Replit containers run UTC) '7pm' means 19:00 UTC, not 7pm Caribbean time (UTC-4/UTC-5). Weekly digests will fire 4-5 hours early for all Caribbean users.
  → *Accept an explicit IANA timezone via env var (e.g. `DIGEST_TIMEZONE=America/Port_of_Spain`) and compute the target time using `Intl.DateTimeFormat` or the `date-fns-tz` / `luxon` library to convert 19:00 local Caribbean time to a UTC timestamp before comparison.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest async scheduler callbacks):** Both scheduler callbacks are `async` functions passed directly to `setInterval`. If the async body throws an unhandled rejection (e.g. DB connection drop mid-tick), Node.js will emit an unhandledRejection. The outer try/catch only covers synchronous throws from the interval setup, not async ones that reject after the tick starts.
  → *The async functions already have internal try/catch blocks, which is correct, but verify that every await inside each function (including the rollback delete in the catch block of maybeRunWeeklyDigest) is also wrapped, because the rollback `await db.delete(...)` itself can throw and is currently unguarded, bubbling out of the catch block.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a brand-new function on every render of the messages list. React will call the old ref with null and the new ref with the element on every render, causing a scrollIntoView call on every keystroke and state update, not just when a new message arrives.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`. This fires only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – apiRequest POST /api/admin/help-chat):** The help-chat endpoint is called with the full accumulated messages array that grows unboundedly. There is no trim or cap on the history sent to the backend, so a long session will send an ever-growing payload, increasing latency and potentially hitting LLM token limits silently. The error catch block swallows the actual error, making debugging impossible in production.
  → *Cap history before sending: `const history = next.slice(-20);` and pass `history` instead of `next`. In the catch block log the error: `catch (err) { console.error('[AdminHelpChat] sendMessage failed', err); ... }`. Consider also surfacing a typed error state rather than injecting an assistant message.*
- **`client/src/pages/AdminDashboard.tsx` (useMutation callbacks for delete/update operations):** Mutation callbacks (onSuccess, onError) that call toast() and queryClient.invalidateQueries() are defined as large inline arrow functions directly inside JSX or component body without useCallback. Because the parent AdminDashboard re-renders frequently (search input state, pagination state, tab state), every child component that receives these callbacks as props will also re-render unnecessarily on every keystroke in the search box.
  → *Extract mutation definitions to the top of the component and wrap stable callbacks with useCallback where they are passed as props. For mutations declared with useMutation, define onSuccess/onError at the useMutation call site rather than in mutate() call-site overrides so they are stable references.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats queries – multiple independent useQuery calls):** The dashboard fetches users, schools, teachers, lessons, and quiz stats with separate useQuery calls that fire sequentially during the same render cycle. While React Query does batch these, each has its own loading state that is checked individually, causing multiple loading spinners or partial renders and 5 separate HTTP round-trips where a single aggregated endpoint or Promise.all would suffice.
  → *Either (a) introduce a single `/api/admin/stats` endpoint that returns all dashboard metrics in one response, or (b) if separate endpoints must remain, use `useQueries` from React Query to declare all queries in one call and derive a unified `isLoading = queries.some(q => q.isLoading)` flag so the UI has a single consistent loading state.*

## 🔵 Info / Tech Debt (2)

- **`server/jobHandlers.ts` (runExtractPaper – image branch: buffer.toString('base64')):** When the uploaded file is an image, the base64 string is decoded to a Buffer and then immediately re-encoded to base64 to build the data URI. This means the full image bytes exist in memory at least twice simultaneously (buffer + re-encoded string), doubling peak RSS for the worker process on large images.
- **`client/src/pages/AdminDashboard.tsx` (Hardcoded pagination page size and API base assumptions):** Page size values (e.g. 10, 20 per page) and any hardcoded `/api/` path prefixes appear to be inline magic numbers/strings with no reference to environment configuration. In a school-facing Caribbean deployment these limits may need to differ per environment or tenant, and hardcoded paths break if the app is served under a sub-path.

## 💡 Opportunities

### Parallel domain route registration: 📌 Low Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six serial awaits. If any registerXRoutes function does async I/O at registration time (e.g. DB schema checks), this adds latency. Refactor each to return synchronously where possible, or accept that registration is a one-time cold-start cost and add a total startup timer log to catch regressions.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replacing bare console.error in the Supabase init chain with structured log entries (including step name, duration, and error code) would make Railway log aggregation and incident response significantly faster. Given this is a school platform with scheduled lesson content, silent seed failures directly impact student experience.

### Centralise BASE_URL and host derivation: 🔥 High Impact
Three separate handlers (robots.txt, sitemap, health) each independently derive or hardcode host information. Extract a single `getBaseUrl(req)` utility that reads from `process.env.BASE_URL` with a validated fallback, applied consistently. This eliminates the Host injection surface area in one place rather than patching each handler individually.

### Idempotency guard via DB unique constraint instead of pre-delete: 🔥 High Impact
The current retry strategy deletes all previously inserted questions for a paperId before re-extracting. A failed mid-insert retry loses partial valid data unnecessarily. Adding a unique constraint on (paperId, order) and using INSERT ... ON CONFLICT DO UPDATE (upsert) via Drizzle's onConflictDoUpdate would make each retry truly idempotent without data loss and remove the blind delete.

### Stream PDF parsing to avoid full-buffer memory spike: ⚡ Medium Impact
pdf-parse loads the entire PDF buffer into memory synchronously. For schools uploading multi-year past-paper PDFs (10-30 MB), this spikes worker RSS significantly. Replacing with pdf2json or pdfjs-dist with streaming support and enforcing a file-size limit (e.g. 15 MB checked before enqueue) would cap memory usage per job and provide a user-friendly error earlier.

### Centralise job error capture into a middleware wrapper: ⚡ Medium Impact
Error capture with Sentry is only applied in the ai-health-check handler; all other handlers lack it. Wrapping registerJobHandler with a thin decorator that auto-catches, calls capture(err, { job: name }), and rethrows would give uniform observability across all job types with one change instead of retrofitting each handler individually.

### Structured JSON logging with log levels: 🔥 High Impact
Replace all `console.log` / `log()` calls with a structured logger (e.g. `pino`) emitting JSON with `level`, `source`, `requestId`, and explicit field allowlists. This eliminates the PII-in-logs risk by design, enables log-level filtering in production, and integrates cleanly with Sentry breadcrumbs. Expected impact: eliminates the entire class of accidental data exposure in logs.

### Centralized graceful shutdown manager: 🔥 High Impact
Introduce a single ShutdownManager that registers SIGTERM/SIGINT once and coordinates ordered teardown: (1) stop accepting new requests, (2) clear scheduler intervals, (3) wait for in-flight jobs, (4) close DB pool, (5) exit. Currently shutdown logic is scattered or absent, risking duplicate signal handlers and partial teardown. A 10 s hard-kill timeout via `setTimeout(process.exit, 10_000)` ensures the process never hangs.

### Parallel startup dependency initialization: ⚡ Medium Impact
Replace the sequential `await probeDatabase(); await seedDatabase(); await registerRoutes(); await startJobWorker()` chain with a dependency-aware parallel strategy: run probeDatabase first, then fan out seedDatabase and registerJobHandlers concurrently via Promise.all, then start the job worker and begin listening. This can reduce cold-start time by 40-60% when DB probe and seed are independent network calls.

### Split AdminDashboard into feature sub-pages: 🔥 High Impact
At ~3000+ lines a single file causes the entire admin shell to re-render when any local state changes (search, pagination, tab selection). Splitting UsersTab, LessonsTab, SchoolsTab into separate lazy-loaded route components would scope re-renders and reduce initial bundle size for admin sessions, with an estimated 30-50% reduction in unnecessary renders during typical navigation.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Any client-side filtering of users/lessons/schools by the search input should be wrapped in useMemo with the raw query data and search term as dependencies. Without this, the filter runs on every render including unrelated state updates like toggling a dialog, which is measurably slow once datasets exceed a few hundred rows — realistic for a school district deployment.

### Add request-scoped admin auth verification on mutation routes: 🔥 High Impact
Client-side useAdminAuth guards the UI but if mutation API routes do not independently verify the admin session token server-side (e.g. via middleware on every /api/admin/* route), a logged-out or role-escalated user could call them directly. Audit every useMutation endpoint to confirm server middleware rejects non-admin tokens with 401/403 before any DB write occurs.
