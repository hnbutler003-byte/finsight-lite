## 🔴 Critical (9)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. A single user or bot can hammer these endpoints, running up API costs and degrading service for all school-aged users.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — adjust path prefixes to match what registerChatRoutes etc. actually mount.

### `server/routes.ts` — /public-objects/:filePath(*)
**Issue:** The wildcard filePath parameter is passed directly to objectStorage.searchPublicObject() without sanitisation. A request to /public-objects/../../private/secret could traverse into unintended storage paths depending on the ObjectStorageService implementation.

**Fix:** Sanitise before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` — import path from 'node:path' at the top of the file.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** The full base64-encoded exam file is stored in the job payload in the jobs table. A single PDF or image can easily be 5–20 MB encoded, meaning every retry duplicates that payload in the DB row, bloating the table and slowing job queue queries.

**Fix:** Upload the file to object storage before enqueuing, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside the handler with objectStorageClient. This keeps job rows small and makes retries cheap.

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique index. Duplicate tickers would silently corrupt portfolio calculations (e.g. JOIN on stockId returning multiple rows). There is also no index on ticker despite it being the natural lookup key from the UI.

**Fix:** Change the table definition to include: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This enforces uniqueness at the DB level and provides an index for ticker-based lookups.

### `server/index.ts` — getMostRecentSundayAt7pm
**Issue:** The day-of-week calculation `(dow + 0) % 7` is a no-op — it never adjusts for Sunday (dow=0) vs Saturday (dow=6), so `daysSince` equals `dow` directly. For a Monday (dow=1) this returns the previous Monday, not the previous Sunday. The digest will fire on the wrong week boundary and the deduplication key `weekStart` will be wrong, causing missed or duplicate sends.

**Fix:** Replace `(dow + 0) % 7` with `(dow + 7 - 0) % 7` — but the real fix is explicit: `const daysSince = dow === 0 ? 0 : dow;` is still wrong. Use `const daysSince = dow;` and verify with unit tests for each day of week, OR use `const daysSince = (dow % 7);` with a Sunday=0 check: if dow===0 set daysSince=0 and only subtract if d > now, else subtract 7. Recommend adding a dedicated unit test for all 7 weekdays.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both `setInterval` return values are discarded — the interval handles are never stored. The graceful shutdown handler (SIGTERM/SIGINT) cannot clear these intervals, so in-flight scheduler ticks can still fire after the server begins draining, potentially enqueuing jobs against a closing DB connection and causing unhandled rejections on shutdown.

**Fix:** Capture both handles: `const weeklyDigestTimer = setInterval(maybeRunWeeklyDigest, ...)` and `const purgeTimer = setInterval(maybeRunAiUsagePurge, ...)`. In the SIGTERM/SIGINT handler, call `clearInterval(weeklyDigestTimer)` and `clearInterval(purgeTimer)` before awaiting server close and DB teardown.

### `server/index.ts` — res.on('finish') logging block inside the request logger middleware
**Issue:** The logger unconditionally serialises the full `capturedJsonResponse` body into the log line for every API response. For a school-aged Caribbean user base this will expose session tokens, student PII, quiz answers, and any AI response content in plaintext logs. There is no truncation, no field redaction, and no log-level gating.

**Fix:** Either remove body logging entirely, or allowlist safe scalar fields: `const safe = { id: capturedJsonResponse?.id, status: capturedJsonResponse?.status }; logLine += ` :: ${JSON.stringify(safe)}`;`. At minimum, cap length: `const snippet = raw.slice(0, 120); logLine += snippet.length < raw.length ? snippet + '…' : snippet;`. Add a `LOG_RESPONSE_BODIES=true` env guard so it is opt-in and never on in production.

### `client/src/pages/AdminDashboard.tsx` — ObservabilityCard – sentry.projectUrl link
**Issue:** The Sentry project URL is rendered directly from API response data as an `href` without any validation. A compromised or misconfigured API response could inject an arbitrary URL (including javascript: URIs) into an anchor tag rendered by an authenticated admin.

**Fix:** Validate the URL origin before rendering: `const safeSentryUrl = useMemo(() => { try { const u = new URL(sentry?.projectUrl ?? ''); return ['https:'].includes(u.protocol) && u.hostname.endsWith('.sentry.io') ? u.href : null; } catch { return null; } }, [sentry?.projectUrl]);` and only render the link when `safeSentryUrl` is non-null.

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel – audit log table rendering
**Issue:** The audit log rows fetched from the server are logged or rendered without redacting sensitive fields. Audit entries for financial or student data in a school-aged Caribbean app may include PII (student IDs, email addresses, IP addresses). Rendering these raw in an unprotected table and potentially in browser devtools/console violates data minimisation principles.

**Fix:** On the server route, project only the fields the UI actually needs (actor, action, timestamp, resourceType, resourceId) and explicitly exclude or hash any PII before sending. On the client, do not pass raw row objects to console.log in any debug paths. Add a server-side role check confirming the requester has `role === 'superadmin'` before the audit log endpoint responds, not just relying on the client-side `useAdminAuth` hook.

## 🟡 Warnings (14)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain):** The seven Supabase seed steps are chained sequentially with .then(), so each waits for the previous to fully complete. On cold start this unnecessarily delays seeding by the sum of all round-trip times. Additionally there is no per-step error isolation — a failure in seedFinancialAcademyLesson silently skips all subsequent seeds.
  → *Run independent steps in parallel where safe: `await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])`. Wrap each step individually so one failure does not abort others: `const safe = (fn) => fn().catch(e => console.error(`[Supabase] ${fn.name} failed:`, e));`*
- **`server/routes.ts` (/healthz async handler):** The /healthz route dynamically imports db and drizzle-orm on every request. While it has a try/catch, repeated dynamic imports on a busy health-check poller add unnecessary overhead and mask the real module resolution cost in latency metrics.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to static top-level imports. The existing try/catch is correct and should be kept.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one-by-one inside a sequential for-loop with individual awaits. For a 40-question exam paper this is 40 serial round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Collect valid questions into an array and do a single batched insert: await db.insert(extractedQuestions).values(validQuestions). Add the idempotent delete before this, which is already present, so retries remain safe.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The OpenAI client is imported from a route module rather than a dedicated singleton. This creates an implicit dependency on the route module's side-effects (express router registration, middleware setup) just to get a client instance, and will break silently if that module is refactored or lazily loaded.
  → *Extract the OpenAI client instantiation into its own module (e.g. server/openaiClient.ts) that exports a single configured instance. Both the route module and jobHandlers import from there. This eliminates the coupling and makes the dependency explicit.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call. If the database is temporarily unavailable the job worker will receive an unhandled rejection. Additionally, olderThanDays comes directly from job.payload with no validation — a malformed payload (e.g. NaN or negative value) produces a nonsensical or destructive cutoff date silently.
  → *Wrap the handler body in try/catch and rethrow so the job framework records failure correctly. Add a guard: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('Invalid olderThanDays: ' + olderThanDays); before computing the cutoff.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups will always filter by userId. At scale this causes a full table scan per user dashboard load.
  → *Add a compound index in the budgets table's third argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a single-column index on userId alone if queries ever fetch all budgets without a category filter.*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, portfolioTransactions, linkedCards table definitions):** All five tables query-critical userId columns lack indexes. billReminders.nextDueDate is also filtered for due-date alerting but has no index. Each of these will full-scan as user data grows.
  → *Add index definitions for each table, e.g. for billReminders: (t) => ({ userIdx: index('idx_bill_reminders_user').on(t.userId), dueDateIdx: index('idx_bill_reminders_due').on(t.nextDueDate) }). Apply the same pattern to the remaining tables targeting their userId columns.*
- **`shared/schema.ts` (varchar user_id columns across all tables (categories, transactions, budgets, linkedCards, etc.)):** All varchar('user_id') columns are declared without a length argument. In PostgreSQL, varchar without length is equivalent to text, so the column type provides no enforcement. If the auth provider issues fixed-length IDs (e.g. Clerk's 32-char user_xxxxx), the schema silently accepts arbitrarily long strings, masking upstream bugs or injection attempts.
  → *Replace varchar('user_id') with varchar('user_id', { length: 64 }) (or the exact max length your auth provider guarantees). This adds a cheap DB-level guard and documents the contract explicitly.*
- **`shared/schema.ts` (createdAt columns on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is defined as .defaultNow() without .notNull(). defaultNow() only fires on INSERT; it does not prevent a NULL being written if a caller explicitly passes null. Audit timestamps being nullable breaks any ORDER BY createdAt DESC query and violates expected invariants for a financial record.
  → *Append .notNull() to every createdAt column: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`server/index.ts` (async startup IIFE / top-level await sequence (probeDatabase, seedDatabase, registerRoutes, startJobWorker)):** The startup sequence awaits `probeDatabase`, `seedDatabase`, `registerRoutes`, and `startJobWorker` sequentially inside what appears to be a single async block. `seedDatabase` and `registerJobHandlers`/`startJobWorker` have no dependency on each other and could run in parallel, but more critically none of these awaits is wrapped in a top-level try/catch — an unhandled rejection here will crash the process with no structured log message and Sentry will not capture it because `initSentry()` may not have finished instrumenting the process rejection handler yet.
  → *Wrap the entire startup block in `try { ... } catch (err) { captureError(err); log(`fatal startup error: ${err.message}`); process.exit(1); }`. Then parallelise independent steps: `await Promise.all([seedDatabase(), registerJobHandlers()])` before the steps that depend on them.*
- **`server/index.ts` (capturedJsonResponse variable inside request logger closure):** The `res.json` monkey-patch assigns the entire response body to a closure-scoped variable that is held in memory until the `finish` event fires. For streaming or chunked responses that never emit `finish`, or for requests that error before responding, this reference is never released. On a school server handling concurrent class logins this is a per-request memory leak.
  → *Null out the reference after logging: at the end of the `res.on('finish', ...)` callback add `capturedJsonResponse = undefined;`. Also guard the monkey-patch to skip non-API paths early: `if (!req.path.startsWith('/api')) return next();` before patching `res.json`.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object and its string are reconstructed on every render inside the component body without memoization. Because the queryKey array also changes reference every render when from/to/action are empty strings, this causes redundant refetches and garbage-collects the previous query cache entry unnecessarily.
  → *Wrap the params construction in useMemo: `const paramString = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', String(AUDIT_LOG_LIMIT)); return p.toString(); }, [actorType, action, from, to]);` and use `paramString` in both the queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel – params.set("limit", "200")):** The audit log page size is hardcoded to 200 rows. This bypasses any server-side guard and could return a very large payload in production, causing memory pressure on the client and slow renders for a table with 200 rows of audit data.
  → *Move the limit to an env-driven constant: `const AUDIT_LOG_LIMIT = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 50);` and apply it via `params.set('limit', String(AUDIT_LOG_LIMIT))`. Also add server-side enforcement capping the value at a safe maximum (e.g. 200) regardless of what the client sends.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard – useQuery for /api/admin/observability):** The query has no isLoading or isError handling. If the observability endpoint is slow or returns an error, the card silently renders empty/undefined values. For a school-aged app with admin operators who depend on this data, a failed health-check status is invisible.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when `isLoading`, and a visible error badge (e.g. `<Badge className='bg-red-600'>Unavailable</Badge>`) for each row when `isError`, so operators know the status could not be fetched rather than assuming it is disabled.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (app.get('/api/health') and app.get('/healthz')):** Two health check endpoints exist: /api/health returns a trivial JSON response with no DB check, and /healthz performs a real DB probe. Railway and uptime monitors should use /healthz, but /api/health will misleadingly report ok even when the database is down.
- **`server/jobHandlers.ts` (runExtractPaper — vision branch, buffer.toString("base64")):** The image buffer, already decoded from base64, is re-encoded to base64 to build the data URL. If fileB64 is kept in the payload (current design) this means the full image bytes are in memory twice simultaneously — once as Buffer, once as a new base64 string — doubling peak memory per worker for image jobs.

## 💡 Opportunities

### Parallel domain route registration — ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then awaited sequentially. If any of these async register functions perform IO (DB schema checks, etc.), they could also be run in parallel: `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])`. Expected impact: reduced cold-start time proportional to the slowest individual route registration.

### Structured logging for object storage errors — ⚡ Medium Impact
console.error in the /public-objects handler logs the raw error object with no request context. Replace with a structured logger (e.g. pino) that includes filePath, request id, and user id (if authenticated) so storage errors are traceable in production without exposing stack traces to the client.

### Sitemap caching — 📌 Low Impact
The sitemap XML is regenerated on every request by re-mapping the static urls array and doing string interpolation. Add a Cache-Control header (`res.setHeader('Cache-Control', 'public, max-age=86400')`) and optionally cache the rendered string in memory after first build. This removes pointless CPU work on every crawler hit.

### Parallel AI calls for text extraction and question parsing — ⚡ Medium Impact
In the PDF branch, text extraction is synchronous (pdf-parse is CPU-bound) but the subsequent OpenAI extraction call is independent of anything else. If future flow adds post-processing steps (e.g. subject tagging, difficulty calibration) those can be Promise.all'd with DB writes, cutting overall job duration by 30–50% for multi-step handlers.

### Centralised payload schema validation with Zod at handler entry — 🔥 High Impact
Every handler destructures job.payload with no runtime type checking. Adding a per-handler Zod schema parsed at the top of each handler would catch malformed enqueue calls immediately, produce actionable error messages in job failure logs, and prevent subtle bugs like the olderThanDays NaN case. Impact is reliability across all job types.

### Object storage streaming instead of full-buffer download for exports — ⚡ Medium Impact
The admin-csv-export handler (visible in the truncated section) likely builds full in-memory buffers before uploading. Switching to streaming pipelines (Transform stream → object storage write stream) would cap memory per export job at kilobytes regardless of export size, which matters as the user base grows.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards implied). Extract a DEFAULT_CURRENCY = 'BSD' constant in a shared config file and reference it. This eliminates the risk of a typo diverging the default across tables and makes a future multi-currency expansion a one-line change.

### Add a partial index for active bill reminders — ⚡ Medium Impact
Queries that power the reminders dashboard will filter WHERE next_due_date >= NOW() almost exclusively. A partial index — index('idx_bill_reminders_upcoming').on(t.nextDueDate).where(sql`next_due_date >= now()`) — would be far smaller and faster than a full index, reducing both query time and index maintenance overhead as historical rows accumulate.

### Replace serial primary keys with UUIDs for user-facing tables — 🔥 High Impact
Using serial (sequential integer) PKs on transactions, savingsGoals, and portfolioHoldings exposes enumerable IDs in API responses (e.g. /transactions/1042 reveals approximate record count and enables IDOR probing). Switching to gen_random_uuid() default UUIDs eliminates this information leakage and aligns with the existing varchar user_id pattern already sourced from the auth provider.

### Parallel startup I/O with Promise.all — ⚡ Medium Impact
Running probeDatabase, seedDatabase, and registerJobHandlers sequentially adds their latencies in series at cold start. If seedDatabase and registerJobHandlers are independent of each other (only both depending on probeDatabase completing first), batching them with `await Promise.all([seedDatabase(), registerJobHandlers()])` after the probe would cut startup time by roughly the duration of the shorter of the two operations — likely 200-800 ms on Replit's shared DB tier.

### Centralised graceful-shutdown registry — 🔥 High Impact
Currently shutdown logic is scattered: interval handles are untracked, the HTTP server close and DB teardown order is implicit. Introduce a small `onShutdown(fn)` registry — an array of async callbacks called in reverse-registration order inside the SIGTERM handler. Each subsystem (intervals, job worker, DB pool) registers its own teardown. This eliminates the current risk of DB connections closing before in-flight jobs finish and makes adding future resources (Redis, WebSocket server) safe by default.

### Extract scheduler logic into a dedicated module with testable pure functions — 🔥 High Impact
getMostRecentSundayAt7pm and the audit-window calculations live inline in index.ts making them untestable without booting the server. Moving them into a `schedulerUtils.ts` module and exporting the pure date functions enables unit tests for all 7 weekday inputs — which would have caught the existing day-of-week bug before production. This is especially important for a digest that targets Caribbean school calendars where the week boundary matters for teacher reports.

### Split AdminDashboard into per-tab lazy-loaded modules — 🔥 High Impact
The file is reported at 126k+ characters, meaning all tab panels (DB Viewer, Perf Agent, Background Jobs, etc.) are bundled and parsed on initial load even though only one tab is active. Converting each tab panel to a React.lazy + Suspense import would reduce initial JS parse time significantly — likely cutting first-meaningful-paint by 30–50% on low-end Caribbean school devices.

### Deduplicate per-entity CRUD query patterns with a shared hook — ⚡ Medium Impact
Each entity section (organizations, schools, teachers, students) almost certainly duplicates the same useQuery/useMutation/useQueryClient invalidation pattern. Extracting a generic `useAdminEntity(entityName)` hook would eliminate hundreds of lines of near-identical code, reducing the risk of inconsistent error handling or stale cache behaviour in one section vs another.

### Add staleTime and gcTime to all admin queries — ⚡ Medium Impact
Without staleTime, every tab switch or parent re-render triggers a background refetch on every query. Admin data for schools/teachers/students changes infrequently. Setting `staleTime: 60_000` and `gcTime: 300_000` globally for admin queries would halve network traffic to the API and eliminate unnecessary loading flickers between tab transitions.

### API Routes (`server/routes.ts`)
The route file is generally well-structured with good patterns like parallel module loading and rate limiting on auth endpoints. Key concerns are: the Supabase init chain runs sequentially via chained .then() calls when steps could be parallelised, AI/chat/image/audio routes lack rate limiting before registration, and the /public-objects/:filePath route has a path traversal risk with no sanitisation.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps that matter for a school-facing financial app: large base64 payloads are stored directly in the job table, sequential DB inserts inside a loop should be batched, and the file is importing the OpenAI client from a route module which creates a tight coupling and potential singleton leak. Error handling is partially present but inconsistent across handlers.

### Database Schema (`shared/schema.ts`)
The schema is reasonably well-structured with foreign keys and some indexes in place, but has several reliability and security gaps. Key concerns are missing indexes on high-frequency query columns, inconsistent NOT NULL discipline on audit timestamps, and varchar user_id columns lacking length constraints that could silently accept oversized values.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability and security gaps: intervals are created without any cleanup reference stored for graceful shutdown, the weekly digest scheduler uses a flawed day-of-week calculation that will fire on wrong days, and full JSON response bodies are logged unconditionally which risks leaking PII or tokens for a school-aged user base. The overall structure is reasonable but these specific issues need addressing before production.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx is a large monolithic component with several reliability and performance gaps. Key concerns include missing error states on data-fetching queries, inline query param construction that rebuilds on every render without memoization, and hardcoded API limits that should be environment-driven config. Security posture around auth checks and sensitive data exposure in the audit log also needs attention.
