## 🔴 Critical (6)

### `server/routes.ts` — registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 97-99)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied beforehand. A single user or bot can hammer these endpoints, incurring unbounded OpenAI cost and potentially triggering 429s that affect all users.

**Fix:** Create a dedicated `aiLimiter` (e.g. express-rate-limit, 20 req/min per IP) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Mirror the pattern already used for auth routes above.

### `server/routes.ts` — /public-objects/:filePath(*) handler (~line 83)
**Issue:** The wildcard `filePath` parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without sanitisation. A request to `/public-objects/../../private/secrets` could traverse outside the intended public prefix depending on how the storage service resolves paths.

**Fix:** Normalise and validate the path before use: `const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); if (safePath !== filePath || filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Add `import path from 'path';` at the top.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full base64-encoded file content is stored in the job payload (job.payload.fileB64). For a PDF or image this can be hundreds of KB to several MB per row, bloating the jobs table, slowing queue polls, and leaking file content into any job-log aggregator or error reporter that serialises the payload.

**Fix:** Upload the file to private object storage before enqueuing, store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download it at the start of runExtractPaper with objectStorageClient. This keeps job rows small (<1 KB) and is already consistent with the uploadPrivateObject helper in this file.

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** olderThanDays is taken directly from job.payload without validation. A payload of { olderThanDays: 0 } or a negative number would delete all rows in aiUsageEvents, and { olderThanDays: NaN } would produce an invalid Date silently passed to the WHERE clause.

**Fix:** Add a guard at the top of the handler: const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days < 1) throw new Error('olderThanDays must be a positive integer'); This makes the job fail-fast and retryable rather than silently destroying data.

### `shared/schema.ts` — portfolioHoldings table — quantity column
**Issue:** quantity is typed as integer, which allows negative values at the database level. A bug or race condition in a sell-order handler could decrement quantity below zero, creating phantom negative holdings with no DB-level guard.

**Fix:** Add a check constraint: quantity: integer('quantity').notNull() and in the table callback add a check expression check(sql`quantity >= 0`). Apply the same constraint to portfolioTransactions.quantity. Also consider switching to numeric for fractional share support if that is a future requirement.

### `client/src/pages/AdminDashboard.tsx` — ObservabilityCard — sentry.projectUrl anchor
**Issue:** The Sentry project URL is rendered directly from the API response into an anchor href without any validation. A compromised or misconfigured backend could return a javascript: URI or a phishing URL, which the browser will follow when an admin clicks it.

**Fix:** Validate the URL before rendering: `const safeSentryUrl = sentry?.projectUrl?.startsWith('https://sentry.io/') ? sentry.projectUrl : null;` and only render the anchor when `safeSentryUrl` is non-null. Apply the same guard to `data?.healthz?.url`.

## 🟡 Warnings (13)

- **`server/routes.ts` (Supabase init chain (~line 102-110)):** The seven Supabase init/seed functions are chained sequentially with `.then()`. If each takes 500ms the server blocks useful startup logging for 3.5+ seconds, and a failure in any step (e.g. `seedStaticModules`) silently skips all subsequent seeds with only a single catch log, making it hard to diagnose which step failed.
  → *Run independent seeds in parallel and log per-step errors: `Promise.allSettled([initSupabaseTables(), initStaticContentTables()]).then(results => { /* check each result.status */ }).then(() => Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent(), trimOrgNamesInSupabase()])).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] seed step ${i} failed:`, r.reason)));`*
- **`server/routes.ts` (sitemap handler (~line 39)):** The `host` header is reflected directly into the sitemap XML `<loc>` tags without validation. A request with `Host: evil.com` produces XML containing attacker-controlled URLs, which could be indexed by crawlers or used in cache-poisoning attacks.
  → *Replace the dynamic host derivation with a hardcoded environment variable: `const base = process.env.SITE_BASE_URL ?? 'https://finsightlite.com';` and remove the `req.headers.host` fallback entirely. Add `SITE_BASE_URL` to your `.env` and deployment config.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop over questions):** Extracted questions are inserted one-at-a-time inside a sequential for-loop with individual awaited storage.createExtractedQuestion calls. For a 40-question paper this serialises 40 round-trips to the database, magnifying latency and holding the job worker occupied far longer than necessary.
  → *Replace the loop with a single bulk insert: collect valid questions into an array first, then call db.insert(extractedQuestions).values(rows) once. If storage.createExtractedQuestion must be reused for business logic, wrap all calls in Promise.all() with a concurrency cap (e.g. p-limit at 5) so they run in parallel.*
- **`server/jobHandlers.ts` (runExtractPaper — vision branch image_url content):** When the uploaded file is an image, buffer.toString('base64') is called a second time to build the data-URI for the OpenAI vision API. The original fileB64 from the payload is already base64; calling Buffer.from(fileB64, 'base64').toString('base64') round-trips unnecessarily and doubles peak memory use for large images.
  → *Use fileB64 directly in the data-URI: `data:${mimeType};base64,${fileB64}`. Only decode to a Buffer once (for pdf-parse), and keep a single decoded reference if both paths ever need it.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The function pipes a read stream directly to res but does not set any Content-Length header and does not handle the case where res is already destroyed or closed before the stream finishes. A client disconnect after headers are sent will leave the read stream open until GC, a slow leak under concurrent export requests.
  → *Attach a 'close' listener on res that destroys the read stream: const stream = file.createReadStream(); res.on('close', () => stream.destroy()); stream.on('error', ...).pipe(res); This ensures the upstream GCS read stream is torn down immediately on client disconnect.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index, but budget lookups will always filter by userId and frequently join on categoryId. At scale this becomes a full table scan per user dashboard load.
  → *Add a composite index in the budgets table definition: index('idx_budgets_user_category').on(t.userId, t.categoryId). Mirror this pattern for billReminders (userId + nextDueDate) and savingsGoals (userId) which have the same gap.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — userId columns):** userId is declared as varchar with no length argument. PostgreSQL will accept unbounded varchar, but without a length cap a malformed or malicious JWT sub claim could insert arbitrarily long strings, wasting index space and breaking any downstream fixed-width assumptions.
  → *Change all userId columns to varchar('user_id', { length: 128 }) to match typical OIDC/Clerk subject claim lengths. Apply consistently across every table that references users.id, and ensure users.id itself carries the same constraint.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker has no unique index. If a seed script or admin route inserts a duplicate ticker symbol, portfolio lookups that join on ticker will silently return multiple rows and corrupt position calculations.
  → *Replace the plain text declaration with: ticker: text('ticker').notNull() and add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table's index callback. Also add notNull() to issuer and region or document explicitly that nulls are valid business states.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt, and all other createdAt columns):** createdAt uses .defaultNow() but is not marked .notNull(). defaultNow() only fires on INSERT via Drizzle; a raw SQL insert or migration script that omits the column will write NULL, making time-series analytics and audit queries unreliable without IS NOT NULL guards everywhere.
  → *Add .notNull() after .defaultNow() on every createdAt (and executedAt, purchasedAt) column. Example: createdAt: timestamp('created_at').defaultNow().notNull(). This is a non-breaking change if no existing rows have NULL in those columns.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo. Because the object reference changes each render, any downstream effect or child component receiving it as a prop will also re-render unnecessarily, and the queryKey array already captures the individual filter values correctly — the params object is redundant noise in the render path.
  → *Wrap the URLSearchParams construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return p; }, [actorType, action, from, to]);` This prevents the object from being recreated on every parent-triggered re-render.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set("limit", "200")):** The audit log fetch limit of 200 is hardcoded as a magic string. For a school-aged financial literacy app with potentially growing audit trails, this will silently truncate results and the value cannot be adjusted per environment (e.g., lower in dev, paginated in prod) without a code change.
  → *Define `const AUDIT_LOG_LIMIT = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 200);` in a shared config file and reference it here. Also add a visible UI indicator when the result set is exactly at the limit so admins know data may be truncated.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — useQuery for /api/admin/observability):** The query has no error state handled and no loading skeleton. If the endpoint fails (network error, 401, 500), all Badge and anchor elements render with undefined data silently — the href falls back to '/healthz' and badges show 'Disabled' indistinguishably from a legitimate disabled state. For an admin security surface this is a reliability and UX gap.
  → *Destructure `isLoading, isError` from useQuery. Render a skeleton or spinner when `isLoading`, and an explicit red error banner when `isError` so admins can distinguish a fetch failure from a genuinely unconfigured integration. Example: `if (isError) return <Alert variant='destructive'>Failed to load observability config</Alert>;`*
- **`client/src/pages/AdminDashboard.tsx` (TABS array and component body — top level):** The TABS constant is defined at module scope but the entire dashboard (13 tabs × their data-fetching logic) appears to live in one 126k-character file. This causes every tab's queries and state to be instantiated regardless of which tab is active, leading to redundant API calls on mount and making the bundle for this route extremely large.
  → *Split each tab into a lazily-loaded component: `const AuditLogPanel = React.lazy(() => import('./tabs/AuditLogPanel'));` wrapped in `<Suspense>`. Only mount the active tab's component so queries only fire when that tab is selected. This alone can eliminate 10–12 unnecessary API calls on initial admin load.*

## 💡 Opportunities

### Consolidate domain router registration with Promise.all — ⚡ Medium Impact
The six `await registerXRoutes(app)` calls at lines 112-117 are sequential. If any of them do async I/O during setup (DB schema checks, cache warming), running them with `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` could meaningfully reduce cold-start time with zero behaviour change.

### Add structured startup logging with per-route timing — ⚡ Medium Impact
There is no timing or confirmation logging when routes are registered. Adding `console.info('[routes] registered X in Yms')` around each domain router registration would make production incident debugging significantly faster, especially when a single router's async init hangs silently.

### Extract object-storage serving into a dedicated middleware module — 📌 Low Impact
The `/public-objects` handler mixes path validation, storage lookup, and response streaming in one inline async function inside registerRoutes. Extracting it to `middleware/servePublicObject.ts` would allow independent unit testing of the path sanitisation logic and reuse across future storage-backed routes.

### Centralise job payload schemas with Zod — 🔥 High Impact
Every handler currently accesses job.payload fields without runtime validation. Adding a Zod schema per job type (parsed at handler entry) would catch malformed payloads at the boundary, produce actionable error messages in job logs, and remove the need for ad-hoc guards like the olderThanDays fix above. Expected impact: eliminates a whole class of silent data-corruption bugs across all handlers.

### Move fileB64 out of job payload before the queue exists — 🔥 High Impact
Beyond the critical finding above, adopting a pattern where enqueueJob for extract-paper always receives an objectPath (never raw bytes) would let you add payload-size enforcement in the enqueueJob function itself — e.g. throw if JSON.stringify(payload).length > 64 KB. This guards all future job types, not just extract-paper.

### Add structured logging with redacted payload fields — ⚡ Medium Impact
Job handlers currently rely on whatever the job runner logs by default, which likely serialises the full payload. Introducing a thin logging wrapper that redacts known sensitive keys (fileB64, any *Token, *Secret fields) before emitting would prevent accidental credential or file-data exposure in log aggregators — important given the app's school-age user base and Caribbean data-privacy obligations.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, etc.). Extract DEFAULT_CURRENCY = 'BSD' into a shared config and reference it via sql`${DEFAULT_CURRENCY}`. This means a currency change or multi-currency rollout requires one edit, not a grep-and-replace migration hunt.

### Add partial index on documentUploads for processing-status polling — ⚡ Medium Impact
Background jobs that poll for unprocessed uploads will repeatedly scan documentUploads filtering on status = 'processing'. A partial index — index('idx_document_uploads_processing').on(t.userId, t.createdAt).where(sql`status = 'processing'`) — will make these scans near-instant and shrink as rows complete, keeping index maintenance cost low.

### Replace learningModules.content text column with structured jsonb — 🔥 High Impact
Storing rich educational content as a flat text blob prevents server-side filtering by content block type, language, or media references without pulling the full payload. Migrating to jsonb with a defined schema (blocks array) enables partial updates, GIN-indexed full-text search, and cheaper API responses when only metadata is needed — directly beneficial for low-bandwidth Caribbean mobile users.

### Parallel data fetching on dashboard overview — 🔥 High Impact
If the Overview tab fires multiple sequential useQuery calls that are independent (e.g., user counts, school counts, sponsor totals), they should be co-located or use Promise.all inside a single queryFn. TanStack Query fires them in parallel by default only if they are defined at the same render level — verify none are inside conditional branches or nested components that stagger them. Expected impact: reduce overview load time by the sum of all sequential round-trips.

### Extract pagination constants and query limits to shared config — ⚡ Medium Impact
Multiple hardcoded values (limit: 200 in audit log, likely page sizes elsewhere in the 126k file) should be centralised in a `config/admin.ts` file. This makes it trivial to tune limits per environment and ensures consistency across tabs without grep-and-replace on a massive file.

### Add React.memo to pure display sub-components — ⚡ Medium Impact
ObservabilityCard and similar stateless card components will re-render every time the parent AdminDashboard re-renders (e.g., on tab change or any top-level state update). Wrapping them in React.memo prevents wasted reconciliation work. Given the number of tabs and cards visible simultaneously on the overview, this can meaningfully reduce frame budget usage on low-end school devices common in the Caribbean market.
