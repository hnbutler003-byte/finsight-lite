## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI-heavy routes (chat, image, audio) are registered with no rate limiting middleware applied before them. For a school-aged user base these endpoints hit paid external APIs and are trivially abusable, causing runaway cost or denial of service.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 37) and sitemap.xml handler (~line 52)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist check. An attacker can send a crafted Host header, injecting an arbitrary URL into the Sitemap directive and all sitemap <loc> entries — a classic Host Header Injection.

**Fix:** Add a server-side allowlist check: `const ALLOWED_HOSTS = (process.env.ALLOWED_HOSTS || '').split(','); const host = ALLOWED_HOSTS.includes(rawHost) ? rawHost : ALLOWED_HOSTS[0];` Or, simpler and more reliable: read the canonical base URL from a single `SITE_URL` environment variable (`process.env.SITE_URL`) and ignore request headers entirely for these two routes.

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** 5 sequential awaits inside studentWeekStats (getUserXp, txCount query, gameCount query, lessonRows query, badgeRows query) are called serially. This function is then called once per student inside a for-loop in the fan-out handler, making total DB round-trips O(5*N) with no parallelism. For a class of 30 students this is 150 sequential queries.

**Fix:** Refactor studentWeekStats to use Promise.all for all 5 independent queries: const [xp, txResult, gameResult, lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...txCount), db.select(...gameCount), db.select(...lessonRows), db.select(...badgeRows)]). Then in the fan-out loop use a concurrency-limited batch (e.g. p-limit with limit of 10) instead of a bare for-loop.

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** Inside bootstrapStudentContacts, a db.select() on emailContacts is executed sequentially inside a for-loop per student (one SELECT per student to check existence, then a conditional INSERT). This is an N+1 query pattern and will deadlock or timeout under load. It also has no idempotency guard at the batch level — if the job retries mid-loop, duplicate contact rows may be inserted for students already processed.

**Fix:** Collect all student IDs first, then do a single bulk SELECT WHERE userId IN (...) to get existing contacts. Diff the result set in memory to find missing contacts, then do a single bulk INSERT with onConflictDoNothing() for the missing ones. Wrap the entire operation in a transaction to ensure retry-safe idempotency.

### `server/index.ts`: startupSequence / main IIFE (bottom of file)
**Issue:** Blocking sequential awaits during startup: probeDatabase(), seedDatabase(), maybeRunWeeklyDigest(), maybeRunAiUsagePurge(), and registerRoutes() all run sequentially before httpServer.listen(). A slow DB probe or seed stalls the entire process, and any unhandled rejection before the top-level catch will crash without the HTTP server ever becoming ready.

**Fix:** Wrap the independent startup steps in Promise.all where safe (e.g., probeDatabase and seedDatabase can often overlap). More importantly, call httpServer.listen() first so the process is ready for health checks, then run background init steps asynchronously. Add a top-level process.on('unhandledRejection') handler that logs via captureError and exits cleanly rather than letting Node default-crash.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are created with setInterval() but the returned timer handles are never stored. This makes it impossible to call clearInterval() during shutdown, so the intervals keep firing after the HTTP server closes, preventing the Node process from exiting cleanly and leaking resources in test/restart scenarios.

**Fix:** Store the handles: const weeklyDigestTimer = setInterval(...) and const aiPurgeTimer = setInterval(...). In the SIGTERM/SIGINT graceful-shutdown handler, call clearInterval(weeklyDigestTimer) and clearInterval(aiPurgeTimer) before closing the HTTP server and DB pool.

### `server/index.ts`: res.on('finish') inside the request-logging middleware
**Issue:** capturedJsonResponse is set to the full parsed response body and then unconditionally serialized into every API log line via JSON.stringify(capturedJsonResponse). For a Caribbean school-aged-user app this will log session tokens, student grades, guardian contact info, and any AI-generated content in plaintext to stdout/log aggregators, violating basic data-minimization principles.

**Fix:** Remove body capture from production logging entirely, or at minimum allowlist only non-sensitive fields (e.g., log only { status, count } shape). Delete the res.json monkey-patch and the capturedJsonResponse branch from the log line. If response-body inspection is needed for debugging, gate it behind a LOG_RESPONSE_BODIES=true env flag that defaults to false.

### `client/src/pages/AdminDashboard.tsx`: sendMessage
**Issue:** The help-chat endpoint receives the full accumulated message array (next) with no size cap. A user or attacker can drive unbounded token usage and potentially extract system prompt context by sending hundreds of messages in a session.

**Fix:** Slice the history before sending: const payload = next.slice(-20) and send { messages: payload }. Also enforce a max session message count (e.g. 40 total) and reset or warn when exceeded.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / page-level auth check
**Issue:** If useAdminAuth returns a loading state before the session is confirmed, the dashboard content is likely rendered momentarily before the redirect fires. For a financial literacy platform with school-aged users this risks flashing restricted data and bypassing role enforcement on slow connections.

**Fix:** After calling useAdminAuth, add an explicit early return: if (authLoading) return <FullPageSpinner />; if (!isAdmin) { setLocation('/'); return null; } before rendering any dashboard JSX. Ensure the check happens before any useQuery calls that fetch admin data.

## 🟡 Warnings (14)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 115)):** Seven sequential Supabase operations are chained with .then(), running one after another. Steps like seedStaticModules, seedRegionalContent, and seedGameContent are likely independent and can easily add 3-10 seconds to cold-start time unnecessarily. Additionally the chain has no per-step error visibility — a failure in seedFinancialAcademyLesson silently skips all subsequent seeds.
  → *Split into two phases: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await initStaticContentTables();` (order-dependent), then `await Promise.all([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])` (independent), each wrapped with individual catch blocks for granular error logging. Keep the outer non-blocking pattern.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100)):** The filePath parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without sanitisation. A path like `../../private/secrets` could traverse outside the intended public storage prefix depending on the ObjectStorageService implementation, leaking private objects.
  → *Sanitise before use: `import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and reject requests where `filePath.startsWith('..')` with a 400. Also confirm ObjectStorageService prefixes all lookups with a hardcoded `public/` bucket prefix server-side.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The olderThanDays value comes directly from job.payload with no validation. A malformed or malicious job payload (e.g. olderThanDays = 0 or negative) would delete all ai_usage_events rows or produce an invalid date, causing silent data loss. There is also no try/catch, so an unhandled DB error will crash the job worker process rather than marking the job as failed.
  → *Add input validation: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays: ' + olderThanDays). Wrap the delete in try/catch and rethrow a typed error so the job framework can record failure and retry. Move the default retention period to an environment variable PURGE_AI_USAGE_RETENTION_DAYS with a safe fallback of 90.*
- **`server/jobHandlers.ts` (weekly-digest handler — since/periodEnd constants):** The 7-day window (7 * 24 * 60 * 60 * 1000) is hardcoded inline. The weekStart payload field is accepted but then ignored entirely — periodEnd is set to new Date() (wall-clock time) instead of using weekStart to derive a deterministic window. This means re-running or retrying the job produces a different dataset, breaking idempotency and making historical digest reruns impossible.
  → *Derive the window from the payload: const periodEnd = weekStart ? new Date(new Date(weekStart).getTime() + 7*24*60*60*1000) : new Date(); const since = new Date(periodEnd.getTime() - 7*24*60*60*1000). Extract the 7-day constant to an env var DIGEST_WINDOW_DAYS. This ensures any retry or backfill of a specific week produces identical results.*
- **`server/jobHandlers.ts` (admin-csv-export handler):** Sensitive PII (student names, emails, financial data) is fetched into memory as a full JS array before streaming to object storage. For large schools this payload can easily exceed 100MB in the Node.js heap. Additionally, the entire CSV string is built via string concatenation in memory. The raw data array is also what gets returned as the job result — if the job framework stores job results in the jobs table (common pattern), this writes the full dataset into the DB row.
  → *Use a streaming CSV approach: pipe the DB cursor through a csv-stringify Transform stream directly into the object storage upload stream using objectStorageClient resumable upload or a multipart approach, never materialising the full array. Return only metadata in the job result: { type, rowCount, objectPath, fileName } — never the data rows themselves. Cap exports with a hard LIMIT and return an error if rowCount would exceed e.g. 50000 rows.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index despite being the primary WHERE columns for every budget lookup. A user with many budgets will cause sequential scans on every dashboard load.
  → *Add a compound index in the budgets table factory: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This mirrors the pattern already used on transactions.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text, not timestamp. Sorting or filtering by this column will use lexicographic ordering, silently producing wrong results (e.g. '2024-09-01' < '2024-10-01' works, but '2024-9-01' breaks). There is also no index on ticker despite it being the natural lookup key.
  → *Change the column type to timestamp('last_price_update_date') and add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table index factory. Update any seed/migration scripts that write string dates.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt, portfolioTransactions.executedAt (and other audit timestamps)):** All audit/created-at timestamps use defaultNow() but are not marked NOT NULL. A failed default or an explicit null insert will silently store NULL, breaking any ORDER BY createdAt queries and audit trails — critical for a financial app.
  → *Add .notNull() to every defaultNow() timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply consistently across all tables including portfolioTransactions.executedAt and portfolioHoldings.purchasedAt.*
- **`shared/schema.ts` (All tables — userId column declarations):** userId is declared as varchar('user_id') with no length parameter. Without a length cap, PostgreSQL uses unlimited varchar (equivalent to text) and cannot enforce a consistent key format. If the auth provider issues fixed-length IDs (e.g. Clerk's 32-char IDs), unconstrained varchar silently accepts malformed values and wastes index space.
  → *Declare a consistent length matching your auth provider: varchar('user_id', { length: 64 }). Apply uniformly to every table. This makes the constraint self-documenting and keeps B-tree index pages denser.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes 'most recent Sunday' using (dow + 0) % 7, where the +0 is a no-op. When dow === 0 (Sunday) the formula returns 0, which is correct, but the comment and arithmetic suggest a day-offset was intended and was deleted, leaving a latent bug: on a Sunday before 7 pm the scheduler will target today's 7 pm, find it is in the future, subtract 7 days, and correctly fall back — but this logic is fragile and untested. Additionally the 7 pm time is hardcoded with no timezone context; new Date() uses server local time, which on a cloud host is UTC, silently shifting the digest to midnight Caribbean time.
  → *Replace the hardcoded hour with a WEEKLY_DIGEST_HOUR_UTC env var defaulting to 23 (7 pm UTC-4 Caribbean). Document the timezone assumption explicitly. Simplify the Sunday-finding logic and add a unit test covering the Sunday-before and Sunday-after boundary.*
- **`server/index.ts` (startJobWorker() call during startup):** startJobWorker() is called without await and its return value (likely a promise or a handle) is discarded. If the worker throws during initialization the error is silently swallowed. The worker is also not stopped in the graceful-shutdown handler, meaning in-flight jobs may be interrupted mid-execution when the process receives SIGTERM.
  → *Await startJobWorker() (or handle its returned promise with .catch(captureError)). Expose a stopJobWorker() function from jobs.ts that drains the current job and resolves when idle, then call it in the SIGTERM handler before closing the HTTP server.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` is recreated on every render and calls scrollIntoView unconditionally whenever any state changes (input typing, loading toggle), causing jarring scroll jumps while the user is typing.
  → *Replace with a stable useRef + useEffect: const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]); then render <div ref={bottomRef} />.*
- **`client/src/pages/AdminDashboard.tsx` (handleKey (AdminHelpChat)):** handleKey is defined as a plain function inside the component body with no useCallback, and sendMessage is also recreated every render. Both are passed as props/handlers to input elements, causing the input to unmount/remount its event binding on every keystroke-triggered state update.
  → *Wrap sendMessage with useCallback(async () => { ... }, [messages, input, loading]) and handleKey with useCallback((e) => { ... }, [sendMessage]). This stabilises the references and eliminates the remount cycle.*
- **`client/src/pages/AdminDashboard.tsx` (apiRequest POST /api/admin/help-chat):** The catch block silently swallows the error object with a bare `catch { }`, so network failures, 401s, and 500s from the admin chat endpoint are all indistinguishable. This makes debugging production issues impossible and hides auth expiry from the user.
  → *Change to catch(err) { console.error('[AdminHelpChat] sendMessage failed:', err); ... } and inspect the response status before calling res.json() — if res.ok is false, throw new Error(res.statusText) so auth failures surface correctly and can redirect to login.*

## 💡 Opportunities

### Centralise base-URL resolution into a shared utility: ⚡ Medium Impact
Both robots.txt and sitemap.xml duplicate the same 3-line proto/host/base derivation logic. Extract to `getCanonicalBaseUrl(req)` in a utils module backed by a `SITE_URL` env var with request-header fallback. This eliminates the duplication and makes the host-header fix a single-point change.

### Add structured startup logging for Supabase seed results: ⚡ Medium Impact
Currently seed failures only log a generic '[Supabase] Init error'. Wrapping each seed call in its own try/catch with a named log (e.g. '[Supabase] seedGameContent failed: ...') lets ops teams identify exactly which seed broke without redeploying with extra debug output. Pairs well with the Promise.all parallelisation fix.

### Cache /healthz DB probe with a short TTL: 📌 Low Impact
Uptime monitors typically hit /healthz every 30 seconds. Each hit issues a real `SELECT 1` query to the database. Adding a 10-second in-memory result cache (storing last result timestamp + status) would reduce unnecessary DB round-trips by ~70% under normal monitoring load with no meaningful loss of accuracy.

### Centralised job payload validation with Zod schemas: 🔥 High Impact
Every handler currently trusts job.payload fields without type-checking at runtime. Adding a Zod schema per job type validated at the top of each handler would catch malformed payloads before any DB work begins, produce actionable error messages in job failure logs, and eliminate the entire class of silent data-corruption bugs seen in purge-ai-usage and weekly-digest. Expected impact: eliminates ~80% of the defensive null-checks currently scattered through handlers.

### Concurrency-limited fan-out with p-limit in digest handler: 🔥 High Impact
Replacing the bare async for-loop in weekly-digest with p-limit(10) would allow 10 student digest emails to be prepared and enqueued concurrently instead of serially, while preventing the handler from spawning unbounded parallel DB connections. For a 300-student org this would reduce fan-out time from ~O(300 * 5 query RTTs) to ~O(30 * 5 query RTTs) — roughly a 10x wall-clock improvement with no infrastructure changes.

### Structured log sanitisation to prevent PII leakage: ⚡ Medium Impact
Several handlers log or return objects that may contain email addresses, usernames, and financial transaction counts. Adding a scrubFields wrapper (email, userId, username keys redacted to first 3 chars + ***) around all console.log and job return values would protect Caribbean minor users' PII in compliance with regional data protection expectations, with near-zero performance cost.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a column default in at least 6 tables (transactions, savingsGoals, billReminders, portfolioHoldings, simulatedStocks, portfolioTransactions). Extract a DEFAULT_CURRENCY = 'BSD' constant in a shared config file and reference it in each .default() call. When the app expands to TTD or JMD support, one change propagates everywhere instead of requiring a grep-and-replace across migrations.

### Add a partial index for unprocessed document uploads: ⚡ Medium Impact
documentUploads is queried by status='processing' to find pending jobs. A partial index — index('idx_doc_uploads_processing').on(t.userId).where(sql`status = 'processing'`) — covers only the small hot subset of rows, making the polling query near-instant and shrinking index size as completed rows are excluded.

### Replace serial primary keys with UUIDs for user-facing records: 🔥 High Impact
Using serial (auto-increment integers) for transactions, savingsGoals, and portfolioHoldings exposes sequential IDs in API URLs, allowing enumeration attacks (GET /transactions/1, /2, /3). Replace with pgTable using uuid defaulting to gen_random_uuid(). This is especially important for a school-aged user base where data privacy obligations are heightened.

### Centralized graceful-shutdown orchestrator: 🔥 High Impact
Add a single shutdown(signal) async function that sequentially: clears both scheduler intervals, calls stopJobWorker(), closes the httpServer with a timeout (e.g., 10 s), then closes the DB pool. Register it for both SIGTERM and SIGINT. This eliminates the current state where three independent resources (HTTP server, intervals, job worker) have no coordinated teardown, which causes hanging processes in Replit restarts and CI.

### Structured JSON logging with PII field filtering: 🔥 High Impact
Replace the ad-hoc console.log string interpolation with a structured logger (pino is zero-dependency and fast). Define a redact list covering fields like password, token, sessionId, email, guardianContact. This gives log levels, request-id correlation, and automatic PII redaction — directly addressing the response-body leak and making logs usable in the Caribbean school context without manual scrubbing.

### Parallel startup probes with readiness gate: ⚡ Medium Impact
probeDatabase() and any other external-service checks (e.g., email provider ping) should run in parallel via Promise.all with a shared timeout (use Promise.race with a 5 s rejection). Return a /healthz route immediately on listen, and only mark the process ready (e.g., respond 200 vs 503) after all probes pass. This reduces startup latency and gives load balancers a reliable readiness signal instead of the current all-or-nothing sequential block.

### Parallel data fetching with Promise.all: 🔥 High Impact
The dashboard likely fires multiple sequential useQuery calls (users, schools, analytics, etc.) that are independent of each other. Audit the query dependency chain and ensure none are waterfalled. Where custom fetch logic exists with sequential awaits, replace with Promise.all or Promise.allSettled to cut initial load time by the sum of individual round-trips.

### Memoise filtered/sorted table data: ⚡ Medium Impact
Search and filter operations on user/school lists (likely driven by useState for search term and sort column) are recomputed on every render including unrelated state changes like the chat panel toggling. Wrap these derivations in useMemo([rawData, searchTerm, sortCol]) to avoid O(n) array work on every keystroke in the chat input.

### Extract chart components to prevent full-dashboard re-renders: ⚡ Medium Impact
Recharts BarChart and LineChart trees are expensive to reconcile. If they live inline in the main AdminDashboard render function, every state change (tab switch, modal open, chat input) triggers a full reconcile of the SVG tree. Extract each chart into a React.memo-wrapped component accepting only its data slice as props so Recharts only re-renders when analytics data actually changes.

### API Routes (`server/routes.ts`)
The routes.ts file is generally well-structured with good use of Promise.all for parallel imports and proper error handling in most async paths. Key concerns are: unprotected AI/chat/image/audio route registration without rate limiting, a host-header injection risk in robots.txt and sitemap generation, and the sequential Supabase init chain running serially when steps are independent.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including sequential database queries inside fan-out loops that will cause severe latency at scale, missing try/catch blocks around critical async operations, and hardcoded timing constants that belong in environment config. The weekly-digest handler is the most problematic area, executing 5+ sequential DB round-trips per user inside a for-loop with no concurrency control.

### Database Schema (`shared/schema.ts`)
The schema is broadly well-structured with foreign keys and composite indexes already in place, but has several reliability and data-integrity gaps. Key concerns are missing indexes on high-frequency filter columns, nullable timestamps that should be NOT NULL, and an untyped varchar user_id that risks silent data corruption across all tables.

### Server Startup (`server/index.ts`)
The server startup performs multiple sequential blocking async operations (DB probe, seed, scheduler checks) before the HTTP server begins listening, which delays readiness and risks unhandled rejections crashing the process. Interval-based schedulers are started but never stored or cleared, creating memory leaks and preventing graceful shutdown. The response-body logging middleware unconditionally serializes every JSON API response body into the log line, which will expose PII and sensitive financial data for school-aged users.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security gaps including missing auth guards on admin routes, uncontrolled re-renders from large inline functions, and a scroll ref pattern that creates a new callback on every render. The chat component sends the full message history to the API on every turn without size limits, creating a potential data exposure and DoS vector. Overall structure is reasonable but needs targeted hardening before production use with school-aged users.
