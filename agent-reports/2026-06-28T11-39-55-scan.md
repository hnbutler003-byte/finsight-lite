## 🔴 Critical (11)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the matching category and a separate aggregation query calculates spent amount, resulting in 2N+1 database round-trips for a single getBudgets call.

**Fix:** Replace the per-budget loop with a single JOIN query using Drizzle's leftJoin to fetch budgets with their categories in one shot, and use a grouped SUM subquery or a single aggregation query with inArray(transactions.categoryId, budgetCategoryIds) to compute all spent values at once, then merge in-memory.

---

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional limit but if the caller omits it the query returns all game session rows aggregated across all users with no upper bound, which can become a full-table scan as the student user base grows.

**Fix:** Add a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 50, 100); and apply .limit(safeLimit) unconditionally in the Drizzle query so no caller can accidentally request unbounded results.

---

### `server/storage.ts`: enrollStudent / createClass / completeModule
**Issue:** Multi-step write operations (e.g., enrollStudent inserts an enrollment then updates a counter, createClass inserts a class then generates a join code and updates it, completeModule inserts progress then awards XP) are not wrapped in a database transaction, leaving the schema in a partially-written state if any step throws.

**Fix:** Wrap each multi-step write in db.transaction(async (tx) => { ... }) using Drizzle's transaction helper, passing tx as the query executor for all inserts and updates within the operation so they commit or roll back atomically.

---

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting applied before them. These are the most expensive endpoints in the app and are callable by any authenticated (or unauthenticated) user, creating a vector for cost exhaustion and abuse.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerX calls. Confirm the actual path prefixes from each register function.

### `server/routes.ts`: robots.txt handler (~line 39) and sitemap.xml handler (~line 55)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header to inject arbitrary URLs into robots.txt and sitemap responses, which can mislead crawlers or be used in cache-poisoning attacks.

**Fix:** Resolve the base URL once at startup from an environment variable: `const BASE_URL = process.env.BASE_URL || 'https://finsightlite.com';` and use that constant in both handlers instead of deriving it from request headers.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored directly in the job payload row. A single-page PDF or image can easily exceed 1–5 MB, and multi-page PDFs can be far larger. This bloats the jobs table, slows job queue polling queries, and risks hitting any row-size limits on the underlying database.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object storage path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts`: runExtractPaper – for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a 60-question paper this means 60 serial round-trips to the database, each adding network latency. There is also no transaction wrapping these inserts, so a crash midway leaves partial data that the idempotency delete at the top of a retry must clean up.

**Fix:** Replace the for loop with a single db.insert(extractedQuestions).values(validQuestions) bulk insert wrapped in a transaction together with the storage.updateExamPaper call. Build the values array with a .filter/.map first, then insert once. This reduces 60 round-trips to 1 and makes the write atomic so partial-insert retries are no longer needed.

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** ticker column has no unique constraint. Duplicate tickers would corrupt portfolio calculations silently since holdings reference stockId but display logic likely joins on ticker for deduplication.

**Fix:** Add a uniqueIndex on ticker: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) in the table's third argument.

---

### `server/index.ts`: startupSequence or equivalent top-level async IIFE
**Issue:** Blocking sequential awaits at startup: probeDatabase(), seedDatabase(), registerRoutes(), and startJobWorker() are awaited one after another. probeDatabase and seedDatabase have no data dependency on each other relative to route registration, so slow DB or seed operations block the HTTP server from binding and serving health-check routes, causing orchestrators (e.g. Replit, Docker) to mark the process unhealthy and restart it.

**Fix:** Bind the HTTP server and register routes immediately. Run probeDatabase() first alone (it gates everything), then run seedDatabase() and startJobWorker() in parallel with Promise.all([seedDatabase(), startJobWorker()]) before starting the digest/purge schedulers. This cuts startup latency and lets /healthz respond immediately.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Interval handles are not stored in variables and are never cleared. There is no SIGTERM/SIGINT handler that calls clearInterval() on these handles or closes the httpServer. On rolling restarts (common in Replit and containerized deploys) the process receives SIGTERM, the intervals keep firing during the drain window, potentially double-enqueueing jobs, and the HTTP server never closes, forcing a SIGKILL after the timeout.

**Fix:** Store each interval: const digestInterval = setInterval(...). Register a single shutdown handler: process.once('SIGTERM', async () => { clearInterval(digestInterval); clearInterval(purgeInterval); await new Promise(r => httpServer.close(r)); process.exit(0); }). Mirror for SIGINT.

---

### `server/index.ts`: res.json monkey-patch inside request logging middleware
**Issue:** Every API response body is captured into capturedJsonResponse and then JSON.stringify'd unconditionally in the 'finish' handler for logging. For endpoints that return paginated lists or AI-generated content this serializes potentially large objects on every request. Worse, for auth endpoints this will log session tokens, user PII, or JWT payloads to stdout in plaintext, violating basic security hygiene for a school-aged-user app.

**Fix:** Remove the res.json monkey-patch entirely for production. At minimum, add a blocklist: skip capture when path matches /api/auth/* or /api/session*, and cap the logged body to 200 chars: logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 200)}`. Long-term, log only status code and duration; use structured APM (Sentry breadcrumbs already wired in) for response body sampling.

## 🟡 Warnings (13)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregations: total income, total expenses, category breakdown, and recent transactions are each awaited one after another despite having no data dependency on each other, adding unnecessary latency on every dashboard load.
  → *Replace the sequential awaits with a single Promise.all: const [incomeResult, expenseResult, categoryRows, recentTxns] = await Promise.all([incomeQuery, expenseQuery, categoryQuery, recentQuery]); Each query is independent and safe to run concurrently.*
- **`server/storage.ts` (getTransactions / getExamPapers / getGameSessions):** Unbounded list queries: getTransactions respects an optional limit param but getExamPapers and getGameSessions have no limit or pagination parameters at all, meaning a student who uploads many papers or plays many games will cause increasingly large result sets to be loaded fully into Node.js heap on every request.
  → *Add limit and offset (or cursor) parameters to getExamPapers and getGameSessions signatures, enforce a default limit of 50 and a max of 200 server-side, and update callers to pass pagination tokens. Add the same hard cap to getTransactions when no limit is supplied by the caller.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 108)):** The seven-step sequential seed chain runs operations that are likely independent (seedStaticModules, seedRegionalContent, seedGameContent) one after another. A failure in step 2 silently skips all subsequent steps, and the single .catch only logs a generic message with no indication of which step failed.
  → *Run independent seed operations in parallel and capture individual errors: `await Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] seed[${i}] failed:`, r.reason)));` Keep initSupabaseTables → trimOrgNamesInSupabase → seedFinancialAcademyLesson → initStaticContentTables sequential if they have ordering dependencies.*
- **`server/routes.ts` (/healthz handler (~line 81)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every invocation. Dynamic imports are cached by Node after the first call, but the import() expression itself still incurs promise overhead and hides the real module graph from bundlers. More critically, uptime monitors may hit this every 30 seconds, making the pattern misleading.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. The db connection is already established at startup; the dynamic import adds latency and obscures dependencies.*
- **`server/jobHandlers.ts` (runExtractPaper – model: 'gpt-4o'):** The OpenAI model name is hardcoded as the string literal 'gpt-4o' in two separate openai.chat.completions.create calls. Changing the model for cost control, capacity, or a new release requires a code deploy rather than a config change, and the two call sites can drift out of sync.
  → *Add OPENAI_MODEL to your environment config (e.g. process.env.OPENAI_MODEL ?? 'gpt-4o') and reference that constant in both calls. Validate its presence at startup alongside other required env vars so a misconfiguration fails fast rather than silently falling back.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call. If the database is temporarily unavailable the unhandled rejection will propagate to the job runner as an uncaught error. Additionally, olderThanDays is read directly from job.payload with no validation; a payload of { olderThanDays: 0 } would delete the entire table.
  → *Wrap the handler body in try/catch and re-throw with a descriptive message. Add a guard: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); before computing the cutoff date.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The function accepts an arbitrary objectPath string and splits it into bucket and object without validating that the resolved path stays within the expected private root. A caller passing a crafted path such as '../../other-bucket/sensitive-file' could read objects outside the intended storage boundary.
  → *After splitting, assert that parts[0] matches the expected bucket name from PRIVATE_OBJECT_DIR (i.e. the first segment of the configured root). Throw a 403 if it does not match. Alternatively, reuse parsePrivatePath (which already enforces PRIVATE_OBJECT_DIR) and apply it here instead of duplicating the split logic.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups will always filter by userId. At scale this becomes a full table scan per user dashboard load.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget entries per period.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Both portfolio tables lack indexes on userId and stockId. Portfolio queries will always filter by userId, and join or lookup by stockId. Without indexes these are full scans that will degrade with any meaningful data volume.
  → *Add (t) => ({ userStockIdx: index('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) to portfolioHoldings, and (t) => ({ userIdx: index('idx_portfolio_txns_user').on(t.userId), executedAtIdx: index('idx_portfolio_txns_executed').on(t.executedAt) }) to portfolioTransactions.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** All createdAt columns use .defaultNow() without .notNull(). Drizzle will not enforce NOT NULL at the DB level unless explicitly stated, meaning an explicit null insert bypasses the default and stores NULL, breaking any ORDER BY createdAt or audit queries.
  → *Change every createdAt definition to timestamp('created_at').defaultNow().notNull(). Same applies to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on nextDueDate or userId. The primary use case for this table is querying upcoming bills by user and due date (e.g. reminders due in the next 7 days), which without an index will scan the entire table.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) so range queries on nextDueDate per user use the index efficiently.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The calculation uses `new Date()` (local server time) with no timezone specification. On Replit or any UTC-based host, 'Sunday at 7 pm' resolves to UTC 19:00, which may be Sunday morning in the Caribbean (UTC-4). Digests will fire at the wrong local time for the target audience, and the weekStart ISO string will be off by a day for users in UTC-4 through UTC-5.
  → *Accept an IANA timezone via env var (DIGEST_TIMEZONE=America/Port_of_Spain). Use the Intl API or the 'luxon' library (already likely in the dep tree for a FinSight app) to compute 'most recent Sunday 19:00' in that timezone before converting to UTC for comparison and ISO string generation.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and related scheduler):** The purge scheduler runs on a fixed hardcoded interval (visible in the truncated code) but AUTO_PURGE_OLDER_THAN_DAYS is the only value read from env. The purge interval cadence itself (e.g. every 6 hours) appears to be a magic number in the setInterval call. If the interval is shorter than the DB deletion query execution time under load, overlapping purge runs can cause lock contention on the ai_usage table.
  → *Extract the poll interval to AI_PURGE_INTERVAL_MS env var with a safe default of 3_600_000 (1 hour). Add a running guard: let purgeRunning = false; inside the interval callback, return early if purgeRunning is true, otherwise set it to true in a try block and false in finally. This is idempotent and prevents overlap without distributed locking.*

## 💡 Opportunities

### Batch category hydration with a shared in-memory map: 🔥 High Impact
getCategories is called repeatedly inside loops across getBudgets and getBillReminders. Fetching all categories for a user once and building a Map<id, Category> lookup would eliminate redundant queries and cut per-request DB calls significantly for users with many budgets or reminders.

### Move hardcoded virtual balance seed amount to environment config: ⚡ Medium Impact
The initial virtual balance (likely hardcoded as a numeric literal such as 10000) given to new users in getVirtualBalance should be read from an environment variable like VIRTUAL_BALANCE_SEED_TTD so it can be adjusted per deployment or sponsor agreement without a code change.

### Centralise auth ownership checks into a reusable guard: ⚡ Medium Impact
Ownership checks (e.g., and(eq(table.id, id), eq(table.userId, userId))) are copy-pasted across deleteTransaction, deleteBudget, deleteSavingsGoal, deleteExamPaper, and others. Extracting a typed assertOwnership(result, userId, entityName) helper would eliminate duplication and ensure consistent 403 vs 404 error semantics across all resource mutations.

### Centralise BASE_URL as startup-time config: 🔥 High Impact
Beyond fixing the header injection, resolving BASE_URL once from process.env at module load time and exporting it from a config module (e.g. server/config.ts) eliminates the repeated header-parsing logic in robots.txt and sitemap handlers and makes the value available to other modules (e.g. email templates, OAuth callbacks) without duplication.

### Add per-user AI quota enforcement at the route layer: 🔥 High Impact
IP-based rate limiting alone is insufficient for a school app where many students may share a NAT IP. Adding a lightweight per-userId counter in Redis or an in-memory LRU (e.g. 50 AI calls/day/user) in the aiLimiter middleware would prevent a single student account from exhausting OpenAI credits and aligns with the app's school-aged audience context.

### Validate and sanitise filePath in /public-objects/:filePath(*): ⚡ Medium Impact
The wildcard filePath param is passed directly to objectStorage.searchPublicObject without path-traversal validation. Adding a check that resolves the path and confirms it stays within the expected prefix (e.g. no `../` sequences) before the storage call prevents potential path traversal into unintended storage buckets.

### Parallel OpenAI calls for vision + extraction pipeline: ⚡ Medium Impact
In the image branch of runExtractPaper the vision transcription and the subsequent extraction are inherently sequential, but for PDF the text extraction (cpu-bound pdf-parse) and any metadata fetches could be parallelised with Promise.all if the handler is ever extended. More immediately, if the admin-csv-export handler (truncated) issues multiple independent db queries sequentially, wrapping them in Promise.all would reduce export latency proportionally to the number of queries.

### Structured logging instead of thrown Error strings for job diagnostics: ⚡ Medium Impact
Error messages like 'No text could be extracted from the uploaded file' are thrown as plain Errors with no structured metadata (paperId, ext, attempt number). Adding a lightweight structured logger (e.g. pino) and attaching job context to every log/error in handlers would make production triage significantly faster and is low-effort to retrofit.

### Idempotency token on extract-paper to prevent duplicate AI spend: 🔥 High Impact
The current idempotency guard deletes previously inserted questions and re-runs both OpenAI calls on every retry, doubling (or tripling) API cost on transient failures. Persisting the raw extractedText to object storage after the first successful extraction and checking for its existence at the top of runExtractPaper would skip the expensive vision/PDF-parse + first GPT call on retries, cutting redundant spend to zero.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions across transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and simulatedStocks. Extract a DEFAULT_CURRENCY constant from an environment variable or a single config import. This prevents a silent mismatch if the app ever needs to support TTD or JMD alongside BSD.

### Add a partial index for active document uploads: ⚡ Medium Impact
Polling or dashboard queries for in-progress document uploads filter on status = 'processing'. A partial index (CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing') would make these status-check queries near-instant and avoid scanning completed/failed rows, which will dominate the table over time.

### Replace text type with varchar on low-cardinality fixed-length columns: 📌 Low Impact
Columns like currency (always 3 chars), lastFour (always 4 chars), and ticker (typically 3-5 chars) are declared as text, which has no length enforcement. Using varchar(3), varchar(4), varchar(10) respectively adds a DB-level guard against bad data and marginally improves storage alignment. For a school-aged user app where input validation may be inconsistent, this is a meaningful safety net.

### Parallel job-handler registration and route setup: ⚡ Medium Impact
registerJobHandlers() and serveStatic() have no interdependency but are called sequentially. If either does any async I/O (e.g. reading handler manifests from disk), wrapping them in Promise.all cuts startup time proportionally. Even if currently synchronous, the pattern makes future async additions safe by default.

### Structured JSON logging with log-level filtering: 🔥 High Impact
All logging goes through the plain console.log wrapper `log()`. Replacing it with a lightweight structured logger (pino, already zero-dep and extremely fast) gives log-level filtering (suppress debug in prod), JSON output for log aggregators, and automatic redaction of fields like password and token. For a regulated school-data context this directly reduces PII-in-logs risk without touching business logic.

### Extract scheduler logic into a dedicated module: ⚡ Medium Impact
maybeRunWeeklyDigest, maybeRunAiUsagePurge, and their setInterval wiring live in index.ts alongside HTTP server bootstrap. Moving them to server/schedulers.ts with an exported start()/stop() pair makes unit-testing the deduplication logic (the onConflictDoNothing claim pattern) straightforward without spinning up an HTTP server, and makes the shutdown handler a single stop() call.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits several reliability and performance concerns including N+1 query patterns in budget and bill-reminder lookups, missing database transactions around multi-step write operations, and sequential awaits that should be parallelized. For a school-aged financial app handling student data, the absence of pagination guards on unbounded list queries and potential exposure of sensitive data in logs are the most pressing concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel module loading and non-blocking Supabase init. Key concerns are missing rate limiting on AI/heavy endpoints registered via registerChatRoutes/registerImageRoutes/registerAudioRoutes, host header injection risk in robots.txt and sitemap construction, and the sequential Supabase seed chain losing all error context after the first failure.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps including large base64 payloads stored in the job table, sequential question inserts that should be batched, and a hardcoded OpenAI model string that should be environment-configurable. The extract-paper handler is the highest-risk area due to payload size and the per-row insert loop under a for-await.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-cardinality query columns, nullable timestamps that should be NOT NULL, and missing NOT NULL constraints on createdAt fields that will silently produce nulls under certain conditions.

### Server Startup (`server/index.ts`)
The server startup sequence has multiple blocking synchronous-style sequential awaits that delay readiness, and scheduled intervals are created without cleanup references, causing memory leaks and preventing graceful shutdown. The response logging middleware unconditionally serializes every JSON response body to a string on every API request, which is a significant CPU and memory concern for a school-facing app handling potentially sensitive user data.
