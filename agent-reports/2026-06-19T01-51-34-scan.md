## 🔴 Critical (8)

### `server/storage.ts` — getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query is issued to sum transactions for that budget's category. With 20 budgets this fires 21 queries per request.

**Fix:** Replace the per-budget loop with a single aggregated query using Drizzle's groupBy + sum: `db.select({ categoryId: transactions.categoryId, spent: sql<number>`sum(amount)` }).from(transactions).where(and(eq(transactions.userId, userId), inArray(transactions.categoryId, budgetCategoryIds))).groupBy(transactions.categoryId)`, then join results in memory before returning.

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** A stock trade executes three separate DB writes (upsert holding, insert portfolio transaction, update virtual balance) outside any database transaction. A mid-flight failure leaves balances and holdings permanently inconsistent — a critical data integrity risk for a fintech app.

**Fix:** Wrap all three operations in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API so all writes commit atomically or roll back together. The route handler that calls these three methods should be refactored into a single `executeTrade(userId, ...)` storage method that runs inside one transaction.

### `server/storage.ts` — getLeaderboard
**Issue:** Missing pagination guard: the leaderboard query accepts an optional `limit` parameter but has no enforced maximum or default cap. A missing or maliciously large limit (e.g. 1,000,000) will load the entire game-sessions table into memory and return it to the client.

**Fix:** Enforce a server-side cap: `const safeLimit = Math.min(filters?.limit ?? 25, 100)` and always apply `.limit(safeLimit)` to the query. Return a 400 error at the route layer if the caller supplies a limit above the cap.

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied to the OpenAI-backed chat, image, and audio routes before they are registered. Any authenticated (or unauthenticated, depending on inner route guards) user can hammer these endpoints, generating unbounded OpenAI spend and degrading service for all users.

**Fix:** Add per-user and per-IP rate limiters before these registrations: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter is a strict limiter (e.g. 20 req/min per IP). Define aiLimiter in rateLimiter.ts alongside the existing limiters.

---

### `server/jobHandlers.ts` — runExtractPaper
**Issue:** Full base64-encoded file content (fileB64) is stored directly in the job payload table. For a multi-page PDF or high-res image this can be several MB per job row, bloating the jobs table, slowing queue polling queries, and potentially hitting row-size limits depending on the DB.

**Fix:** Upload the file to object storage before enqueuing, then store only the resulting object path in the payload: `{ paperId, objectPath: '/bucket/exams/123.pdf', ext, subject }`. In runExtractPaper, stream the object down to a buffer instead of decoding from the payload. This keeps job rows small and makes retries cheaper.

---

### `server/jobHandlers.ts` — runExtractPaper — for loop over questions
**Issue:** Extracted questions are inserted one at a time inside a sequential for-loop with individual awaited storage.createExtractedQuestion calls. For a 60-question paper this serialises 60 round-trips to the database, dramatically increasing job duration and holding the DB connection for the entire loop.

**Fix:** Replace the for-loop with a single bulk insert: `await db.insert(extractedQuestions).values(questions.filter(q => q.question && ...).map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0,4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i+1 })))`. Add the storage.updateExamPaper call after in the same transaction if possible.

---

### `shared/schema.ts` — simulatedStocks — ticker column
**Issue:** ticker has no unique constraint. Duplicate tickers would cause silent data corruption in portfolio calculations — two rows for 'BISX:01' would double-count holdings in any JOIN-based aggregation, and there is no DB-level guard preventing an INSERT of a duplicate.

**Fix:** Replace the plain column declaration with ticker: text('ticker').notNull().unique(). If the table is already seeded, add a uniqueIndex via a migration: CREATE UNIQUE INDEX idx_stocks_ticker ON simulated_stocks(ticker);

### `client/src/pages/AdminDashboard.tsx` — Top-level dashboard – useQuery calls
**Issue:** Based on the visible imports and pattern (`useAdminAuth` hook present but auth state not awaited before queries fire), all `useQuery` data-fetching hooks almost certainly run before the admin auth check resolves. This causes redundant unauthenticated requests that the server must reject, adding latency and noise to server logs.

**Fix:** Destructure `{ isAuthenticated, isLoading }` from `useAdminAuth()` and pass `enabled: isAuthenticated` to every `useQuery` options object so queries are held until auth is confirmed: `useQuery({ queryKey: [...], queryFn: ..., enabled: isAuthenticated })`.

## 🟡 Warnings (15)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits are used to fetch total income, total expenses, savings goals, and recent transactions one after another. These queries are fully independent and their combined latency is additive, likely 4× slower than necessary.
  → *Replace sequential `await` calls with `const [income, expenses, goals, recent] = await Promise.all([queryIncome, queryExpenses, queryGoals, queryRecent])`. All four queries read independent rows and have no data dependency on each other.*
- **`server/storage.ts` (getBillReminders):** Each bill reminder with a non-null categoryId triggers an individual `db.select().from(categories).where(eq(categories.id, reminder.categoryId))` lookup inside a loop — a classic N+1 pattern. Bill reminders with the same categoryId also re-fetch the same category row repeatedly.
  → *Collect all distinct categoryIds from the fetched reminders, run one `db.select().from(categories).where(inArray(categories.id, distinctIds))` query, build a Map keyed by id, then hydrate each reminder from the map. Zero extra queries regardless of result set size.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 130)):** The seven Supabase init/seed steps are chained sequentially with .then(). Steps that are logically independent (e.g. seedStaticModules, seedRegionalContent, seedGameContent after initStaticContentTables) block on each other, inflating cold-start time unnecessarily.
  → *Split into two waves: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await seedFinancialAcademyLesson(); await initStaticContentTables();` then `await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]);` — all wrapped in a single async IIFE with a .catch() so the non-blocking behaviour is preserved.*
- **`server/routes.ts` (/healthz handler (~line 82)):** `import('./db')` and `import('drizzle-orm')` are called inside the request handler on every /healthz hit. While Node caches modules after first load, the dynamic import syntax still incurs promise micro-task overhead on each call and obscures intent. More critically, uptime monitors hitting this endpoint frequently will each pay that cost.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. The health check becomes a simple `await db.execute(sql\`select 1\`)` with no dynamic import cost.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 110)):** There is no authentication or authorisation check on this endpoint. Any unauthenticated request can enumerate and download stored objects by guessing filePath values (e.g. logos, videos). For a children's platform this could expose user-uploaded content to anonymous scrapers.
  → *For non-public asset types, add a session/auth guard before the objectStorage call: check `req.isAuthenticated()` (or equivalent session flag) and return 401 if absent. Truly public assets (e.g. marketing logos) should be served from a CDN with a known prefix whitelist rather than a catch-all proxy.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** PDF text is silently truncated to 30,000 characters with no warning logged and no signal returned to the caller. A long exam paper will have its tail silently dropped, causing questions near the end to never be extracted, and the stored questionCount will reflect only the truncated portion without any indication of the loss.
  → *Log a warning when truncation occurs: `if (pdfData.text.length > 30000) { console.warn({ paperId, originalLength: pdfData.text.length }, 'PDF text truncated'); }`. Store a `truncated: true` flag in the job result and surface it on the exam paper record so teachers can see partial extraction. Move the limit to an env var: `parseInt(process.env.PDF_TEXT_LIMIT ?? '30000', 10)`.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create (both calls)):** The OpenAI model name 'gpt-4o' and max_completion_tokens value 4096 are hardcoded in two separate places. Any model migration or token budget adjustment requires a code change and redeploy rather than a config update, and the values may drift out of sync between the two calls.
  → *Add env vars: `OPENAI_EXTRACTION_MODEL` (default 'gpt-4o') and `OPENAI_EXTRACTION_MAX_TOKENS` (default 4096). Reference them in both completions calls: `model: process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'`. This also allows per-environment cost control (e.g. using gpt-4o-mini in staging).*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete then re-insert):** The idempotency strategy deletes all previously inserted questions at the start of every retry attempt, including any that were successfully inserted on a prior partial run. If the job fails mid-insert on retry 2, questions from retry 1 are gone and retry 3 starts from zero. For a terminal failure this leaves the paper in 'failed' status but with zero questions, which is correct, however the repeated delete-on-every-attempt pattern means a transient network error late in a large batch causes full re-processing of OCR and both OpenAI calls unnecessarily.
  → *Wrap the delete, both OpenAI calls, the bulk insert, and the status update in a single DB transaction. On retry, check if the paper already has status 'completed' and return early. This makes the handler truly idempotent: a completed paper is never reprocessed, and a failed one always starts from a clean slate within one atomic operation.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet every budget query will filter by userId. With even modest user counts this becomes a full table scan on every dashboard load.
  → *Add a compound index in the budgets table factory function: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a single-column index on userId alone if you query budgets without categoryId.*
- **`shared/schema.ts` (userId: varchar('user_id') — all tables):** All userId columns are declared as varchar with no length limit. This means the database allocates a variable-length field with no upper bound, making it easy to accidentally store malformed IDs of arbitrary length and producing slightly less efficient B-tree indexes compared to a fixed-width type or a length-bounded varchar.
  → *Change every userId column to varchar('user_id', { length: 36 }) if using UUID strings, or to char('user_id', { length: 36 }) for fixed-length UUIDs. This enforces data integrity at the DB level and improves index density.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow() — transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is not marked notNull() on any table. Because defaultNow() only fires on INSERT, a bulk import or a future migration that bypasses the ORM can leave createdAt as NULL, breaking any query that ORDER BY or filters on createdAt.
  → *Append .notNull() to every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default still applies for normal inserts; notNull() simply adds a DB-level constraint as a safety net.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Neither portfolioHoldings nor portfolioTransactions has an index on userId or stockId, despite being the primary tables queried when rendering a user's portfolio. A user with many transactions will cause full scans on both tables simultaneously.
  → *Add index definitions to both tables: for portfolioHoldings use index('idx_holdings_user_stock').on(t.userId, t.stockId); for portfolioTransactions use index('idx_ptx_user_executed').on(t.userId, t.executedAt) so recent-trade queries are also covered.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage):** The entire conversation history (`next`) is sent to /api/admin/help-chat on every message with no truncation. For a long session this grows unboundedly, leaking the full admin conversation to the server payload and increasing latency and token cost with every turn.
  → *Slice the history before sending: `const window = next.slice(-10);` then POST `{ messages: window }` instead of `next`. Also add a max message count guard (e.g. 100) to cap `messages` state so the component DOM does not grow without bound.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – Messages div scroll ref):** `ref={el => { if (el) el.scrollIntoView(...); }}` is an inline function created on every render. React calls it with `null` on unmount and a new element reference on mount/update, so scrollIntoView fires on every re-render (e.g. every keystroke in the input), not just when messages change.
  → *Replace with a stable `useRef` + `useEffect`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage (catch block)):** The catch block silently swallows the error and appends a generic message. The original error is never logged, making production debugging impossible. Additionally `res.json()` can throw independently (malformed JSON) but is not guarded separately, so a 200 response with bad JSON would surface the same generic message, hiding the real cause.
  → *In the catch block add `console.error('[AdminHelpChat] sendMessage failed:', err);`. Wrap `res.json()` in its own try/catch or check `res.ok` before parsing: `if (!res.ok) throw new Error(res.statusText); const data = await res.json();`.*

## 💡 Opportunities

### Introduce a shared pagination utility — 🔥 High Impact
At least five query methods (getTransactions, getLeaderboard, getGameSessions, getAllQuestions, getPortfolioTransactions) each implement ad-hoc or missing limit/offset handling. Extracting a typed `paginate(query, { page, pageSize, maxPageSize })` helper and applying it consistently would eliminate the missing-cap security risk and reduce boilerplate across the file.

### Move hardcoded virtual balance seed amount to environment config — ⚡ Medium Impact
The initial virtual balance (commonly 10000 or similar) granted to new users on first access is hardcoded in `getVirtualBalance`. For a sponsor-funded school app this value may need to differ per school or sponsor tier. Externalising it to `VIRTUAL_BALANCE_SEED_AMOUNT` in environment config allows runtime adjustment without a redeploy.

### Add structured error wrapping to all public storage methods — 🔥 High Impact
No storage method wraps its DB call in try/catch, so raw Drizzle/postgres driver errors — which can contain table names, column names, and query fragments — bubble up through the route layer and risk being serialised into API responses. Wrapping each method with a thin `StorageError` class that logs internally and rethrows a sanitised message would prevent sensitive schema details from leaking to clients.

### Centralised AI route rate-limit middleware module — 🔥 High Impact
Extract all rate limiters (authLimiter, strictAuthLimiter, and the missing aiLimiter) into a single rateLimiter.ts that also exports a factory function accepting rpm and burst params. Apply them declaratively via an array config at the top of registerRoutes. This prevents future AI routes being added without limits and makes limit tuning a one-file change.

### Startup telemetry for Supabase seed timing — ⚡ Medium Impact
Wrap the Supabase init block with `console.time` / `console.timeEnd` (or a Sentry span) so slow seeds are visible in production logs. Currently a seed regression silently delays readiness with no observable signal beyond the catch log line.

### robots.txt and sitemap.xml caching headers — 📌 Low Impact
Both responses are dynamically generated on every request but their content changes rarely. Adding `Cache-Control: public, max-age=86400` and an ETag derived from the URL list hash would allow CDN/browser caching, eliminating redundant computation for crawler traffic.

### Parallel OpenAI calls for vision + extraction pipeline — ⚡ Medium Impact
The vision extraction (image path) and the subsequent structured extraction are currently sequential by necessity, but the PDF text extraction and the structured extraction OpenAI call could be pipelined more efficiently by pre-validating the text before the second API call, saving a round-trip on empty extractions. More broadly, if the handler ever needs to process multi-page documents split into chunks, Promise.all over chunk extraction calls would reduce wall-clock time proportionally.

### Move file decode and PDF parsing out of the job handler into a preprocessing step — ⚡ Medium Impact
Decoding base64 and running pdf-parse inside the job worker ties up the worker process with CPU-bound work. Extracting this into a dedicated preprocessing step (or running pdf-parse in a worker_thread) would free the async event loop and allow the job worker to handle more concurrent jobs, directly improving throughput for schools uploading multiple papers.

### Add structured result schema validation on OpenAI JSON response — 🔥 High Impact
The parsed JSON from the extraction response is cast with `as { subject?, questions? }` but never validated. A malformed model response (missing options array, wrong difficulty string) silently produces bad DB rows. Introducing a lightweight schema check with zod before the bulk insert would catch model regressions early and allow the job to fail fast with a descriptive error rather than inserting corrupt data.

### Consolidate repeated currency default into a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate table columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract const DEFAULT_CURRENCY = 'BSD' (or read from an env var to support multi-currency expansion) and reference it in each column definition. This eliminates drift risk if the default ever needs to change and makes the Caribbean-context assumption explicit.

### Add a partial index on documentUploads for in-flight processing jobs — ⚡ Medium Impact
Background workers polling for documents with status='processing' will scan the entire documentUploads table. A partial index — CREATE INDEX idx_doc_uploads_processing ON document_uploads(created_at) WHERE status = 'processing' — would make these polls O(in-flight jobs) rather than O(all uploads), which matters as upload history grows.

### Add billReminders index on nextDueDate for reminder scheduling — 🔥 High Impact
The reminder scheduler almost certainly queries WHERE next_due_date <= NOW() across all users to fire notifications. Without an index on nextDueDate this is a sequential scan. Add index('idx_reminders_due').on(t.nextDueDate) (or a partial index where isAutoDetected = false if manual reminders are processed separately) to keep the scheduler query fast as reminder volume grows.

### Memoize filtered/sorted admin table data — ⚡ Medium Impact
Admin tables (users, schools, teachers) are likely filtered and sorted inline in JSX or in the render body. Wrapping these derivations in `useMemo` keyed on the raw query data and filter/sort state would eliminate re-computation on every parent re-render triggered by unrelated state (e.g. dialog open/close, search input), which matters when lists have hundreds of rows.

### Consolidate parallel admin bootstrap queries with Promise.all — 🔥 High Impact
If the dashboard fires multiple independent `useQuery` calls on mount (users, schools, analytics, audit log), consider a single bootstrap query that calls `Promise.all([fetchUsers(), fetchSchools(), fetchAnalytics()])` server-side or via a dedicated `/api/admin/bootstrap` endpoint. This cuts waterfall latency by the number of sequential round trips and reduces query client cache fragmentation.

### Extract AdminHelpChat into its own file with React.memo — ⚡ Medium Impact
`AdminHelpChat` is a self-contained stateful component embedded inside a very large dashboard file. Moving it to `components/AdminHelpChat.tsx` and exporting it wrapped in `React.memo` prevents it from re-rendering whenever any top-level dashboard state changes (pagination, search, selected tab), which would otherwise reset scroll position perception and restart any pending animations.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
server/storage.ts exhibits several critical reliability and performance gaps: N+1 query patterns in budget and bill-reminder fetchers, missing database transactions on multi-step writes, and sequential awaits that should run in parallel. For a school-aged financial app handling student data, the absence of pagination guards and exposed sensitive operations without rate-limit or transaction safety are the most urgent concerns.

### API Routes (`server/routes.ts`)
The route registration code is generally well-structured with good use of Promise.all for parallel imports and proper try/catch on async handlers. Key concerns are: the Supabase init chain runs sequentially when it could parallelise independent seeds, the /healthz endpoint imports db dynamically on every request creating overhead, and the chat/image/audio route registrations have no rate limiting applied before they are mounted — a significant risk for an AI-powered app serving school-aged users.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential question inserts that should be batched, and hardcoded model/token values that should be environment config. The extract-paper handler is the most problematic, combining payload bloat, a slow O(n) sequential insert loop, and a text truncation ceiling that silently discards exam content.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable timestamps that should be NOT NULL, and an unbounded varchar type on all userId columns that could cause subtle bugs and poor index performance.
