## 🔴 Critical (4)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI-backed routes (chat, image, audio) are registered with no rate limiting middleware. For a school-aged audience these endpoints hit paid external APIs and a single user or script can exhaust quotas or cause runaway costs.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerX calls.

### `server/routes.ts` — app.get('/public-objects/:filePath(*)')
**Issue:** The filePath parameter is passed directly to objectStorage.searchPublicObject without sanitisation, enabling path traversal attacks (e.g. ../../private/secrets). A malicious user could probe internal object-storage paths.

**Fix:** Sanitise the input before use: const safePath = path.normalize(filePath).replace(/^\.\.\/|^\//, ''); then reject any path containing '..' — if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });

### `server/jobHandlers.ts` — runExtractPaper
**Issue:** fileB64 (base64-encoded PDF or image) is stored directly in the job payload. A single-page PDF can easily exceed 500 KB base64-encoded; multi-page exams can hit several MB. This bloats the jobs table, slows queue polling queries, and risks hitting any payload size limit on the jobs table column.

**Fix:** Upload the file to private object storage before enqueuing, then store only the resulting object path in the payload. In runExtractPaper, download the file from object storage at the start of the handler and delete it on terminal success or failure. Remove fileB64/ext from JobPayloads['extract-paper'] and replace with objectPath.

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique constraint. Duplicate tickers would corrupt portfolio value calculations since holdings join to stocks by stockId but display and seed scripts likely key off ticker strings.

**Fix:** Add a unique index: (t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). If multi-region/multi-currency stocks can share a ticker, scope the unique index to (ticker, region) instead.

## 🟡 Warnings (10)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 100)):** The seven Supabase seed/init steps are chained sequentially with .then(). If seedStaticModules takes 2 s and seedRegionalContent takes 2 s they add up linearly at startup, delaying readiness unnecessarily. Steps that are independent should run in parallel.
  → *Group independent steps: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Verify FK/ordering constraints before parallelising each pair.*
- **`server/routes.ts` (app.get('/healthz')):** The /healthz endpoint performs dynamic imports of ./db and drizzle-orm on every invocation. Dynamic import() is cached by Node after the first call but the overhead and unpredictability under high-frequency health checks (Railway polls every few seconds) is unnecessary and can mask module-load errors silently.
  → *Import db and sql statically at the top of the file alongside the other imports. This also makes startup errors visible immediately rather than on first health check.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop with sequential awaits):** Each extracted question is inserted with an individual awaited storage.createExtractedQuestion call inside a for loop. For a 40-question exam paper this serialises 40 round-trips to the database, significantly increasing handler duration and holding the job worker busy far longer than necessary.
  → *Replace the for loop with a single batched insert: filter valid questions into an array first, then call db.insert(extractedQuestions).values(validRows) once. If storage.createExtractedQuestion must be used for its side-effects, use Promise.all with a concurrency cap (e.g. p-limit at 5) rather than pure serial execution.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls):** max_completion_tokens is hardcoded to 4096 in both OpenAI calls. If the model or quota changes this value cannot be tuned without a code deploy, and the value is duplicated in two places making it easy to update one but not the other.
  → *Add OPENAI_MAX_COMPLETION_TOKENS to environment config with a default fallback: const maxTokens = Number(process.env.OPENAI_MAX_COMPLETION_TOKENS ?? 4096). Use this constant for both completions.create calls.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch. A database error (e.g. lock timeout, connection drop) will propagate as an unhandled rejection through the job runner. Depending on how registerJobHandler catches errors, this may silently mark the job as failed with no contextual information logged, making it hard to diagnose recurring purge failures.
  → *Wrap the db.delete call in try/catch, log the error with context (olderThanDays, cutoff date), and rethrow so the job runner can record the failure and schedule a retry. Apply the same pattern to admin-csv-export and any other single-expression handlers.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The streaming path sets response headers and pipes the read stream but does not attach a cleanup handler for premature client disconnects. If the client closes the connection mid-stream the read stream is not explicitly destroyed, leaking the underlying socket/file descriptor until GC.
  → *Listen for the res 'close' event and call readStream.destroy() if the stream has not already ended: const stream = file.createReadStream(); res.on('close', () => stream.destroy()); stream.on('error', ...).pipe(res);*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups are always user-scoped. Every budget query will perform a full table scan as the dataset grows.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget entries per category per period.*
- **`shared/schema.ts` (varchar('user_id') on all tables):** userId columns are declared as plain varchar() with no length argument, which defaults to unlimited in Postgres. Foreign key references to users.id will silently accept any length string, risking inconsistent joins if the auth provider changes ID format.
  → *Audit the users.id column type in ./models/auth and match it exactly, e.g. varchar('user_id', { length: 36 }) for UUID-based auth or text() if the auth model uses text. Apply the same explicit length uniformly across all tables.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, etc.)):** createdAt is defined as .defaultNow() but without .notNull(), meaning the column accepts explicit NULL inserts, which breaks audit trails and any ORDER BY createdAt queries that assume non-null values.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Same fix applies to purchasedAt on portfolioHoldings and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate is a primary query filter for surfacing upcoming bills, but has no index. For a school-aged user base with recurring reminders this will degrade as rows accumulate.
  → *Add a composite index on (userId, nextDueDate) in the table definition: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This covers the common query pattern 'fetch this user's bills due in the next N days'.*

## 💡 Opportunities

### Structured startup error reporting for Supabase init — ⚡ Medium Impact
The catch at the end of the Supabase chain only logs a generic error. Adding per-step labelling (e.g. wrapping each step in a named async function with its own console.error tag) would cut mean-time-to-diagnosis when a seed step fails silently in production, which is particularly important during school-term deployments.

### Centralise domain-router registration into a manifest — ⚡ Medium Impact
The six parallel imports + six sequential await register calls will grow as the app scales. Defining a ROUTE_MANIFEST array of {path, importer, register} tuples and looping over it would eliminate the copy-paste pattern, make adding/removing routers a one-line change, and guarantee consistent ordering without cognitive overhead.

### Add authenticated object-storage endpoint for private assets — 🔥 High Impact
Currently all /public-objects assets are unauthenticated. As the app grows to include student progress reports or teacher materials, a separate /private-objects/:filePath route with session/JWT middleware would prevent accidental exposure of sensitive educational data without requiring a full CDN overhaul.

### Idempotent question deletion scope is too broad — ⚡ Medium Impact
The retry guard deletes ALL extractedQuestions for paperId before re-running. If another concurrent job or process legitimately wrote questions for the same paper between attempts, they are silently wiped. Use a transaction that checks paper status is still 'processing' before deleting, or add a jobRunId column to extractedQuestions and only delete rows from the current job's previous attempt.

### Centralise OpenAI model name into environment config — ⚡ Medium Impact
The string 'gpt-4o' appears at least twice in this file. Extracting it to OPENAI_MODEL env var lets operators switch models (e.g. to gpt-4o-mini for cost reduction on Caribbean school budgets) without a code change or redeploy, and eliminates drift when one call site is updated.

### Add structured logging to all job handlers — 🔥 High Impact
None of the handlers emit structured log entries (job id, attempt number, duration, outcome). Without this, diagnosing slow or repeatedly-failing jobs in production requires correlating raw error stack traces. Adding a thin wrapper in registerJobHandler that logs {jobType, jobId, attempt, durationMs, outcome} at completion would give operators actionable observability with minimal code change.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a .default() on at least six separate columns across transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, and portfolioHoldings. Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' in a config file and reference it in each column definition. This eliminates drift risk when the app expands to TTD or JMD.

### Add partial index for active/pending document uploads — ⚡ Medium Impact
documentUploads queries for status='processing' rows (to poll or resume) will scan all uploads. A partial index — WHERE status = 'processing' — on (userId, createdAt) would make polling queries constant-time. In Drizzle this requires a raw sql`` expression in the index definition.

### Replace text() with enum PG type for repeated status/period columns — ⚡ Medium Impact
Columns like budgets.period, documentUploads.status, and billReminders.frequency use text() with a Drizzle-side enum list, but the constraint is not enforced at the database level — only at the ORM layer. Using pgEnum() creates a native Postgres enum, enforcing values at the DB level, reducing storage size, and making invalid states impossible even from raw SQL tooling or migrations run outside the app.
