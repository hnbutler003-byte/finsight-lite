## 🔴 Critical (10)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row a separate query fetches the associated category and another aggregates spent amount, meaning 10 budgets = 21 database round-trips.

**Fix:** Rewrite using a single JOIN query with Drizzle: db.select({ ...budgets, categoryName: categories.name, spent: sql<number>`coalesce(sum(${transactions.amount}),0)` }).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).where(eq(budgets.userId, userId)).groupBy(budgets.id, categories.id). This collapses all fetches into one round-trip.

### `server/storage.ts`: upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Stock trade execution performs upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three independent awaits with no wrapping database transaction. A crash between any two steps leaves balances and holdings permanently inconsistent.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { ... }) passing the tx client to each operation. Drizzle's transaction callback automatically rolls back on throw, ensuring atomicity. Add a try/catch at the call site to surface rollback errors to the HTTP layer.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 117-119)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. An unauthenticated or authenticated user can flood these endpoints, incurring unbounded OpenAI API costs and degrading service for all school users.

**Fix:** Create an `aiLimiter` (e.g. 20 req/min per IP via express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — adjust paths to match the actual prefixes used inside each register function.

### `server/routes.ts`: robots.txt handler (~line 50) and sitemap handler (~line 65)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header to inject an arbitrary domain into the Sitemap directive or sitemap `<loc>` URLs, enabling cache-poisoning or phishing via poisoned CDN/proxy responses.

**Fix:** Resolve the base URL from a single trusted environment variable at startup: `const BASE_URL = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com';` and use that constant in both handlers, removing all header-derived host construction.

### `server/jobHandlers.ts`: runExtractPaper / job.payload.fileB64
**Issue:** Full file content (fileB64) is stored directly in the job payload row. A single uploaded PDF or image can easily exceed 5–10 MB base64-encoded, bloating the jobs table, slowing queue polling queries, and copying that data on every retry attempt.

**Fix:** Upload the file to object storage before enqueuing, then store only the resulting object path (e.g. `privatePath`) in the payload. In runExtractPaper, download the buffer from object storage using that path instead of decoding from payload. Delete the object after successful extraction or on terminal failure.

### `server/jobHandlers.ts`: purge-ai-usage handler – job.payload.olderThanDays
**Issue:** There is no validation or cap on `olderThanDays` before it is used to compute a cutoff date and drive a bulk DELETE. A misconfigured or maliciously crafted job payload (e.g. `olderThanDays: 0`) would delete the entire aiUsageEvents table in one unbounded statement with no row-limit safeguard.

**Fix:** Add a guard at the top of the handler: `if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('Invalid olderThanDays value');`. Additionally consider adding `LIMIT` via a subquery or batching deletes (e.g. delete 5 000 rows at a time in a loop) to avoid long table locks in production.

### `shared/schema.ts`: portfolioHoldings and portfolioTransactions table definitions
**Issue:** portfolioHoldings has no unique constraint on (userId, stockId), allowing duplicate holding rows per user per stock. This means buy logic that should update an existing holding can instead silently insert a second row, corrupting portfolio value calculations. portfolioTransactions also has no index on userId+executedAt, making trade history queries slow.

**Fix:** Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) to portfolioHoldings to enforce one row per user per stock. Add index('idx_portfolio_transactions_user_executed').on(t.userId, t.executedAt) to portfolioTransactions for efficient trade history pagination.

### `server/index.ts`: setInterval (scheduler setup, post-registerRoutes block)
**Issue:** The two `setInterval` calls for `maybeRunWeeklyDigest` and `maybeRunAiUsagePurge` are stored in variables but those variables are never passed to `clearInterval` inside the SIGTERM/SIGINT graceful-shutdown handler. On a rolling restart the old process keeps firing both intervals until forcibly killed, which can race with the new process and double-enqueue jobs despite the DB-level deduplication guard.

**Fix:** Capture both interval handles (`const weeklyDigestInterval = setInterval(...)`) and inside the shutdown handler call `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before awaiting `httpServer.close()`. Also add a short `setTimeout` fallback (e.g. 10 s) to force-exit if `httpServer.close` stalls.

---

### `server/index.ts`: res.json monkey-patch in request logger middleware
**Issue:** Every API response body is captured into `capturedJsonResponse` and then serialised in full via `JSON.stringify(capturedJsonResponse)` and written to the console log. For a financial literacy app with school-aged users this will routinely log session tokens, quiz scores tied to named students, teacher PII, and any AI-generated content — all in plaintext server logs with no truncation or field redaction.

**Fix:** Either remove the response-body capture entirely and log only method/path/status/duration, or whitelist a small set of non-sensitive scalar fields (e.g. `{ status, count }`) and cap the serialised output at 120 characters: `logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 120);`. Never log full response bodies in production.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth – top of AdminDashboard component
**Issue:** The file imports and calls useAdminAuth but the truncated code shows no early return guard before data-fetching useQuery calls. If useAdminAuth returns a non-admin or unauthenticated state, all admin queries (users, schools, audit logs) still fire because the hooks execute unconditionally, leaking existence of admin endpoints and wasting server resources.

**Fix:** Immediately after the useAdminAuth call, add: `if (!isAdmin) { redirect('/'); return null; }` before any useQuery hooks. On the server, ensure every /api/admin/* route independently verifies the session role rather than relying solely on client-side gating.

## 🟡 Warnings (19)

- **`server/storage.ts` (getTransactions):** No maximum limit guard: callers can omit the limit filter entirely, causing a full table scan returned to the client. For a school app with many student transactions this will exhaust memory and response buffers.
  → *Apply a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 100, 500); and always append .limit(safeLimit) to the query regardless of caller input. Expose a pagination cursor or offset parameter for legitimate large fetches.*
- **`server/storage.ts` (getDashboardStats):** Three or more aggregate queries (total income, total expenses, category breakdown, savings goals) are awaited sequentially inside getDashboardStats, adding latency equal to the sum of all round-trip times on every dashboard load.
  → *Fan them out with const [income, expenses, goals] = await Promise.all([incomeQuery, expensesQuery, goalsQuery]). Each query is independent and read-only so parallel execution is safe and will cut dashboard latency by roughly (N-1)/N where N is the number of queries.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard fetches game sessions then performs per-user lookups for userName and avatar in a loop (N+1), and the default limit is either missing or not enforced, allowing unbounded result sets on a public-facing leaderboard endpoint accessible to students.
  → *Join gameSessions with users in a single aggregation query: db.select({ userId: gameSessions.userId, userName: users.displayName, avatar: users.avatarUrl, totalScore: sql<number>`sum(${gameSessions.score})`, gamesPlayed: sql<number>`count(*)` }).from(gameSessions).innerJoin(users, eq(gameSessions.userId, users.id)).groupBy(gameSessions.userId, users.displayName, users.avatarUrl).orderBy(desc(sql`sum(${gameSessions.score})`)).limit(Math.min(filters?.limit ?? 20, 100)).*
- **`server/routes.ts` (Supabase init chain (~line 122-130)):** The seven Supabase init/seed functions are chained sequentially with `.then()`. Each awaits the previous before starting, meaning cold-start time is the sum of all their latencies. If any intermediate step is slow (e.g. `seedRegionalContent`), all subsequent seeds are blocked unnecessarily.
  → *Where steps are independent, run them in parallel: `Promise.all([initStaticContentTables(), seedFinancialAcademyLesson()]).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` — keep `initSupabaseTables` and `trimOrgNamesInSupabase` first if they are true prerequisites, then fan out the rest.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 105)):** There is no authentication or authorization check on this endpoint. Any unauthenticated request can fetch arbitrary objects by guessing paths. For a platform with school-aged users this could expose student-related assets (e.g. uploaded profile content) stored under predictable paths.
  → *Add a middleware guard before the handler that checks if the file path is under a truly public prefix (e.g. `logos/`, `videos/`) and reject everything else unless the request carries a valid session. Example: `if (!/^(logos|videos)\//.test(filePath)) return res.status(403).json({ error: 'Forbidden' });`*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one-by-one inside a sequential for loop with individual awaits. For a 40-question paper this is 40 round-trips. If any single insert fails mid-loop, a retry re-deletes and re-extracts everything, wasting two expensive OpenAI calls.
  → *Collect all valid question objects into an array, then do a single `db.insert(extractedQuestions).values(allQuestions)` batch insert inside a transaction that also calls `storage.updateExamPaper`. This reduces N round-trips to 1 and makes the write atomic so retries are clean.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create (both calls), hardcoded model and token values):** `model: 'gpt-4o'` and `max_completion_tokens: 4096` are hardcoded in two separate places. Changing the model or adjusting limits for cost control requires a code deploy rather than a config change, and the two call-sites can silently drift apart.
  → *Add `OPENAI_EXTRACTION_MODEL` (default `gpt-4o`) and `OPENAI_EXTRACTION_MAX_TOKENS` (default `4096`) to environment config and read them at module load time: `const MODEL = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'`. Reference these constants in both completions calls.*
- **`server/jobHandlers.ts` (runExtractPaper – image branch, buffer.toString('base64')):** When the input is an image, the code decodes fileB64 from base64 into a Buffer and then immediately re-encodes it back to base64 for the vision API: `buffer.toString('base64')`. This is a redundant allocation that doubles peak memory for large images.
  → *Pass `fileB64` directly to the image_url content instead of re-encoding the buffer: `url: \`data:${mimeType};base64,${fileB64}\``. This eliminates the intermediate Buffer allocation for the vision path (the Buffer is still needed for pdf-parse but not for images).*
- **`shared/schema.ts` (categories table definition):** No index on categories.userId, which is used in every per-user category lookup. With growing users this causes a full table scan on every categories fetch, and the column also lacks NOT NULL — system categories use null userId but user-owned categories will silently accept null on insert without a check constraint to enforce intent.
  → *Add a partial index: index('idx_categories_user_id').on(t.userId).where(sql`user_id IS NOT NULL`). This covers user-owned category queries without indexing the null system rows. Also add a DB-level check constraint or application-level validation ensuring userId is non-null when isDefault is false.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets.userId. Every budget query for a user (balance checks, dashboard loads) will full-scan the budgets table. This is a hot path — the dashboard likely fetches budgets on every page load.
  → *Add a composite index on (userId, categoryId) inside the table's index callback: index('idx_budgets_user_category').on(t.userId, t.categoryId). This covers both user budget listings and the common userId+categoryId join used to check budget utilisation per category.*
- **`shared/schema.ts` (transactions.createdAt / budgets.createdAt / documentUploads.createdAt / savingsGoals.createdAt):** All createdAt columns use .defaultNow() but omit .notNull(). Drizzle will not add NOT NULL to the DDL without explicit declaration, meaning application code or a raw INSERT can write NULL into createdAt, breaking any ORDER BY createdAt or audit logic silently.
  → *Append .notNull() to every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same fix to purchasedAt on portfolioHoldings and executedAt on portfolioTransactions.*
- **`shared/schema.ts` (billReminders table definition):** No index on billReminders.nextDueDate or billReminders.userId. The bill-reminder notification job must query all reminders due today across all users, making userId+nextDueDate a near-certain query pattern. Without an index this scan grows linearly with total reminder rows.
  → *Add a composite index: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). If a background job queries across all users for due reminders (e.g. a cron), also add a standalone index on nextDueDate alone for that access pattern.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** daysSince is always 0 when dow===0 (Sunday) due to `(dow + 0) % 7`, but more critically the guard `if (d.getTime() > now.getTime())` only fires when the computed Sunday-7pm is in the future on a Sunday before 7pm, so on any other day of the week the date arithmetic is wrong: `(dow + 0) % 7` should be `dow` to get days-since-Sunday, but Monday gives 1, Tuesday 2, etc., which is correct — however Sunday gives 0, so `d.setDate(d.getDate() - 0)` leaves today, and if it is Sunday before 7pm the future-guard correctly rolls back 7 days. The real bug is the redundant `+ 0` is a leftover placeholder that signals this was edited, and the function is never unit-tested, making silent scheduling drift likely.
  → *Replace `(dow + 0) % 7` with just `dow` and add a unit test covering Sunday-before-7pm, Sunday-after-7pm, and mid-week cases. Extract the constant `19` (hour) to an env var `WEEKLY_DIGEST_HOUR_UTC` so it can be adjusted without a deploy.*
- **`server/index.ts` (seedDatabase / probeDatabase calls inside async IIFE at startup):** `seedDatabase()` and `probeDatabase()` are awaited sequentially before `httpServer.listen` is called, which is correct ordering, but neither call has an individual timeout. If the database is slow to accept connections (common on Replit cold-start), the process hangs indefinitely with no log output and no liveness signal, causing the platform to SIGKILL it. Additionally the two calls are sequential when they could run in parallel since `probeDatabase` is a read and `seedDatabase` is idempotent.
  → *Wrap both in `Promise.all([probeDatabase(), seedDatabase()])` and wrap the whole block in a `Promise.race` against a `new Promise((_, reject) => setTimeout(() => reject(new Error('DB init timeout')), 15000))`. Log a clear message and call `process.exit(1)` on timeout.*
- **`server/index.ts` (maybeRunAiUsagePurge (async scheduler callback)):** The interval callback is an `async` function passed directly to `setInterval`. If the async body throws an unhandled rejection (e.g. DB is temporarily unavailable), Node.js will emit an `unhandledRejection` event. The outer `try/catch` inside `maybeRunAiUsagePurge` catches most paths, but if the DB insert for the dedup-claim row itself throws before the inner try, the catch only logs — it does not re-throw, which is correct. However the sibling `enqueueJob` await inside `maybeRunWeeklyDigest` for each audience iteration is inside a nested try, meaning a failure on audience #2 still marks audience #1 as claimed but un-enqueued with no retry signal beyond the rollback. Confirm the rollback delete itself is also wrapped in a try/catch because a failure there silently swallows the original error.
  → *Wrap the rollback `await db.delete(...)` call in its own try/catch that logs `'rollback also failed: ...'` and calls `captureError(rollbackErr)` so Sentry captures it. Add `captureError(e)` (not just `log`) to the outer scheduler catch blocks so transient DB failures surface in Sentry rather than only in stdout.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The help-chat POST sends the full accumulated messages array to /api/admin/help-chat on every turn, growing unboundedly. A 50-turn conversation sends 50 messages every request, and there is no max-length guard, meaning a malicious admin user can craft arbitrarily large payloads or exhaust token budgets.
  → *Slice the messages array before sending: `messages: next.slice(-10)` to keep only the last 10 turns. Add a message count cap (e.g. 100) with a UI warning, and enforce a max payload size check server-side.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The auto-scroll element uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` directly in JSX. This creates a new function reference on every render, causing React to call it with null then the element on every re-render, producing redundant scrollIntoView calls and potential jank during loading states.
  → *Replace with a stable useCallback ref: `const scrollRef = useCallback((el: HTMLDivElement | null) => { el?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and attach `ref={scrollRef}` to the div.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple useQuery calls across dashboard tabs):** Based on the visible imports (Users, Schools, GraduationCap, Trophy, Coins, BookOpen sections) each tab section likely has its own useQuery that fires when the component mounts regardless of which tab is active. This causes redundant parallel fetches of admin data the user may never view, increasing server load and response time for the initial page load.
  → *Add `enabled: activeTab === 'users'` (etc.) to each tab-specific useQuery so data is only fetched when the relevant tab is first visited. Combine with `staleTime: 60_000` to avoid re-fetching on every tab switch.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage input):** User input is sent directly to the AI endpoint with no sanitization or length limit on the client. A school-aged context makes prompt injection attacks more likely (students discovering the admin panel), and there is no check preventing empty-after-trim edge cases from reaching the API if the loading guard races.
  → *Add `if (text.length > 500) return;` before building the next messages array, display a character counter in the UI, and on the server validate and truncate the last user message to 500 chars before forwarding to the LLM provider.*

## 💡 Opportunities

### Centralised pagination helper: 🔥 High Impact
Extract a reusable paginate(query, { page, pageSize, maxPageSize }) helper that enforces the hard cap and returns { data, total, page }. Applying it consistently across getTransactions, getGameSessions, getAllQuestions, and getLeaderboard eliminates per-method guard duplication and prevents future regressions when new list endpoints are added.

### Query result caching for read-heavy static data: ⚡ Medium Impact
getLearningModules and getMarketStocks return data that changes infrequently but are called on every dashboard render. Introducing a short-lived in-process cache (node-cache or a Map with TTL) of 60 seconds for these two methods would eliminate redundant DB round-trips under classroom load where many students hit the app simultaneously.

### Structured logging with sensitive-field redaction: ⚡ Medium Impact
Console.log calls throughout storage methods risk leaking userId, card details, or score data in plaintext logs. Replacing them with a structured logger (pino) configured with a redact option for fields like cardNumber, userId, and email ensures compliance with data-minimisation expectations appropriate for a school-aged user base.

### Centralise PUBLIC_BASE_URL as validated startup config: 🔥 High Impact
Beyond fixing the host-injection issue, move all environment-dependent constants (base URL, allowed origins, DB URL) into a single validated config module that throws at startup if required vars are absent. This prevents silent misconfiguration in staging/production deployments and removes the need for inline fallbacks like `|| 'localhost'`.

### Add per-user AI rate limiting in addition to per-IP: 🔥 High Impact
Per-IP rate limiting can be bypassed by users on shared school networks (all students share one egress IP). Adding a per-authenticated-user-ID limiter (keyed on `req.user.id`) using a Redis store would give accurate per-student quotas, directly protecting OpenAI cost and fairness for all classroom users.

### Lazily import domain routers only when needed: ⚡ Medium Impact
All six domain routers are eagerly `Promise.all` imported at startup, increasing cold-start memory and parse time. For routes like emails or lessons that are infrequently hit, dynamic imports inside the route handler or a lazy-registration pattern would reduce baseline RSS and speed up the first meaningful response.

### Parallel OpenAI calls where extraction steps are independent: ⚡ Medium Impact
In handlers where multiple independent AI completions are made sequentially, wrapping them in Promise.all would cut wall-clock latency proportionally. Audit all multi-step AI flows (e.g. any report-generation handlers in the truncated section) for sequential awaits that have no data dependency between them.

### Centralised job payload schema validation with Zod: 🔥 High Impact
Job payloads arrive as untyped runtime data. Adding a Zod schema per job type and parsing at the top of each handler (before any DB or API call) would catch malformed payloads immediately, produce clear error messages in the job failure log, and eliminate the need for ad-hoc guards like the olderThanDays fix above.

### Object storage cleanup on terminal job failure: ⚡ Medium Impact
Once file uploads are moved to object storage pre-enqueue (fixing the payload bloat issue), orphaned objects will accumulate whenever a job exhausts retries. Add a finalisation hook in the terminal-attempt catch block of extract-paper (and any future upload jobs) to delete the stored object, preventing unbounded storage growth.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the currency default across at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, portfolioHoldings). Extract a DEFAULT_CURRENCY constant in a shared config file and reference it in each column definition. This prevents a future currency change requiring six schema edits and reduces the risk of a typo creating inconsistent defaults.

### Add a composite index to support savings goal progress queries: ⚡ Medium Impact
savingsGoals is likely queried by (userId, deadline) for dashboard deadline sorting and by (userId, currentAmount, targetAmount) for progress calculations. Adding index('idx_savings_goals_user_deadline').on(t.userId, t.deadline) will make dashboard loads that sort goals by upcoming deadline significantly faster as the goals table grows.

### Replace text() with varchar() for bounded string columns: 📌 Low Impact
Columns like currency (always 3 chars), ticker (typically 1-5 chars), lastFour (always 4 chars), and fileType (MIME type, bounded) are declared as unbounded text. Using varchar with explicit length limits (e.g. varchar('currency', {length: 3})) adds a DB-enforced data integrity check at zero performance cost and makes the domain constraints self-documenting in the schema.

### Centralise scheduler management into a SchedulerRegistry: 🔥 High Impact
Create a small `SchedulerRegistry` class that wraps `setInterval`, stores all handles, and exposes a single `shutdown()` method called from the graceful-shutdown handler. This removes the current manual bookkeeping risk and makes adding future scheduled jobs safe by default. Expected impact: eliminates the interval-leak class of bugs permanently and simplifies the shutdown path to one line.

### Add structured logging with log-level filtering and field redaction: 🔥 High Impact
Replace `console.log` with a structured logger (pino is already common in the Node ecosystem and near-zero overhead). Configure a `redact` list covering fields like `password`, `token`, `sessionId`, `email`, `studentName`. Set `LOG_LEVEL=info` in production and `debug` locally. This directly addresses the PII-in-logs finding and gives operators filterable JSON logs without code changes per route.

### Extract all magic numbers and intervals to validated env config: ⚡ Medium Impact
The scheduler intervals (10-minute weekly-digest poll, purge poll), the 180-day purge window, the 10mb JSON limit, and the digest hour (19) are all hardcoded. Move them to a single `config.ts` that reads, validates, and exports them at startup with clear error messages. This lets the ops team tune behaviour without code changes and makes the values visible in one place during incidents.

### Memoize filtered/sorted admin data lists: 🔥 High Impact
Search and filter operations on user, school, and leaderboard lists are almost certainly computed inline during render. Wrapping these with useMemo keyed on the raw data array and filter string would eliminate redundant O(n) passes on every keystroke-triggered re-render, noticeably improving responsiveness on lists with hundreds of students.

### Centralise hardcoded API base URL and pagination defaults: ⚡ Medium Impact
Values like page size (likely 10 or 20 per page), the /api/admin prefix, and any polling intervals appear inline throughout the file. Moving these to a single config/constants.ts with VITE_ env-var overrides eliminates divergence between staging and production and makes tuning straightforward without touching component code.

### Extract tab sections into lazy-loaded sub-components: 🔥 High Impact
The full AdminDashboard renders all tab UIs (charts, tables, forms) in one massive component tree. Splitting each tab into its own file loaded with React.lazy/Suspense would reduce the initial JS bundle parsed by the browser and isolate re-render scope, directly improving Time-to-Interactive for the most-visited Overview tab.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits several reliability and performance gaps including N+1 query patterns in budget and bill-reminder fetches, missing database transactions on multi-step writes, and sequential awaits where Promise.all would suffice. Security concerns include potential sensitive data exposure in logs and missing pagination guards on unbounded queries serving school-aged users at scale.

### API Routes (`server/routes.ts`)
The route file is generally well-structured with good patterns like parallel module imports and rate limiting on auth endpoints. However, the Supabase init chain runs sequentially when it could be parallelised, the AI/chat/image/audio routes lack any rate limiting before registration, and the robots.txt/sitemap host construction is vulnerable to host header injection which is a real risk for a school-aged user platform.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential question inserts that should be batched, and hardcoded model/token values that belong in environment config. The extract-paper handler is the highest-risk area combining all three concerns, and could cause significant DB bloat and slow retries in production.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable timestamps that should be NOT NULL, and the categories table lacking an index on userId which will cause full scans for per-user category lookups.

### Server Startup (`server/index.ts`)
The server startup has several reliability and security concerns: blocking synchronous operations before the HTTP server is ready, intervals that are never cleared on shutdown, and full JSON response bodies logged to console which can leak sensitive financial/PII data for school-aged users. The graceful shutdown path partially exists but does not cancel the scheduler intervals, creating zombie timers under process restarts.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx contains several reliability and security gaps including missing auth checks on sensitive admin routes, uncontrolled re-renders from inline functions in JSX, and a scroll ref callback that recreates a DOM function on every render. The chat feature lacks input sanitization and the help-chat endpoint may expose internal platform structure. Overall structure is reasonable but the issues compound in a school-aged user context where security and stability matter most.
