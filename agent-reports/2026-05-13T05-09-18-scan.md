## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: each PortfolioHolding record triggers a separate getStockById lookup inside a loop, meaning 10 holdings = 11 database round trips.

**Fix:** Rewrite as a single JOIN query using Drizzle's `.leftJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id))` so all holdings and their stock data are fetched in one query. Remove the per-row lookup entirely.

---

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** N+1 query pattern: student User records are fetched individually per enrollment row inside a loop rather than via a join or inArray batch fetch.

**Fix:** Replace the loop with a single query: join classEnrollments with users on userId, or collect all studentIds first then do one `db.select().from(users).where(inArray(users.id, studentIds))` and merge in-memory via a Map keyed by userId.

---

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (trade execution path)
**Issue:** Multi-step trade operations (update holding, insert portfolio transaction, update virtual balance) are executed as separate awaited statements with no wrapping database transaction. A failure mid-sequence leaves data permanently inconsistent — e.g., balance deducted but holding not updated.

**Fix:** Wrap all three operations in `await db.transaction(async (tx) => { ... })` using Drizzle's transaction API so all succeed or all roll back atomically.

---

### `server/routes.ts` — /api/auth/google handler
**Issue:** No rate limiting on the Google auth endpoint. An attacker can hammer this endpoint with arbitrary idTokens, forcing repeated calls to Google's token verification API and downstream DB writes (getClassByCode, getOrgEnvironmentByJoinCode, getOrganization). For a school-aged user base this is a trivial DoS vector and can exhaust external API quotas.

**Fix:** Add express-rate-limit before the handler: `import rateLimit from 'express-rate-limit'; const authLimiter = rateLimit({ windowMs: 15*60*1000, max: 20, keyGenerator: (req) => req.ip }); app.post('/api/auth/google', authLimiter, async (req, res) => {...})`. Apply a stricter limit (max: 5) if the endpoint is also used for account creation.

---

### `server/routes.ts` — AI/heavy endpoints (chat, image, audio routes registered via registerChatRoutes etc.)
**Issue:** registerChatRoutes, registerImageRoutes, and registerAudioRoutes are called without any wrapping rate limiter. The quota reservation logic (reserveQuota/finalizeUsage) guards per-org spend but does not prevent a single authenticated user from firing hundreds of concurrent requests that each successfully reserve quota before any finalises, bypassing the intended throttle.

**Fix:** Apply a per-user rate limiter at the Express layer before these route registrations: `const aiLimiter = rateLimit({ windowMs: 60*1000, max: 10, keyGenerator: (req) => req.user?.id ?? req.ip, message: { error: 'Too many AI requests' } }); app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` This acts as the first line of defence before quota logic runs.

---

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter / job.payload)
**Issue:** Raw base64 file content is stored directly in the job payload. For a typical exam PDF this can easily exceed several MB per job row, bloating the jobs table, causing slow queue polls, and potentially hitting DB row-size limits.

**Fix:** Upload the file to object storage before enqueuing, store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. This keeps job rows small and makes retries cheaper.

---

### `shared/schema.ts` — userId: varchar('user_id')
**Issue:** All userId columns use bare varchar() with no length argument, which defaults to 255 in Postgres — acceptable — but more critically, userId is not given a NOT NULL + index on every table used as a lookup key. Tables like budgets, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions have no index on userId alone, meaning every 'fetch all records for this user' query does a full table scan.

**Fix:** Add a single-column index on userId to each table that is queried by user: e.g., in budgets add `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })`. Repeat for savingsGoals, billReminders, linkedCards, documentUploads, portfolioHoldings, and portfolioTransactions. This is the highest-impact schema change given these are the primary access patterns.

---

### `client/src/pages/AdminDashboard.tsx` — useQuery queryFn callbacks throughout file
**Issue:** Every queryFn uses an inline arrow function that calls apiRequest and chains .then(r => r.json()) without any try/catch or error boundary. If the server returns a non-JSON body (e.g., an HTML error page on 502), r.json() throws an unhandled promise rejection that React Query catches but surfaces no actionable error to the user.

**Fix:** Create a typed helper: async function fetchJSON<T>(method: string, url: string): Promise<T> { const res = await apiRequest(method, url); if (!res.ok) throw new Error(`${res.status} ${res.statusText}`); return res.json(); } Use this in all queryFn definitions and ensure each useQuery destructures isError and renders an error state.

## 🟡 Warnings (17)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits are used to fetch total income, total expenses, transaction count, and budget data — each is an independent query but they are chained with await, adding full round-trip latency per query.
  → *Replace sequential awaits with `const [income, expenses, txCount, budgetData] = await Promise.all([...])` since these queries have no data dependency on each other. Expected latency reduction: 3x on a 4-query sequence assuming ~20ms per DB round trip.*
- **`server/storage.ts` (getTransactions (filters parameter)):** The `limit` filter inside getTransactions has no enforced maximum cap. A caller (or compromised client) can pass limit: 1000000, causing the DB to stream millions of rows into Node's heap — a straightforward memory exhaustion vector in a school app with shared infrastructure.
  → *Enforce `const safeLimit = Math.min(filters?.limit ?? 50, 200)` before applying the limit clause, and add a default of 50 so callers that omit limit don't accidentally fetch unbounded result sets.*
- **`server/routes.ts` (/api/auth/google handler — resolvedClass resolution block):** Three sequential awaits (getClassByCode → getOrgEnvironmentById → getOrganization) are chained serially inside the auth hot path. Each is an independent DB/network call; the total latency is additive and unnecessary. Under load this compounds with the lack of rate limiting to stall the Node event loop.
  → *After resolving resolvedClass, run the env and org lookups in parallel where the inputs are known: `const [classEnv, otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), ...])`. For the org lookup, chain only after classEnv resolves since it depends on classEnv.org_id, but eliminate any other independent sequential awaits in the same request lifecycle using Promise.all.*
- **`server/routes.ts` (objectStorage instantiation (top of file)):** `const objectStorage = new ObjectStorageService()` is constructed at module load time alongside `objectStorageClient` imported from the same integration. If ObjectStorageService holds a long-lived connection or polling interval internally and registerRoutes is ever re-invoked (e.g. in tests), a second instance is created without cleaning up the first, causing a resource/memory leak.
  → *Export a singleton from `./replit_integrations/object_storage` and import it directly rather than constructing a new instance inside routes.ts. If the class manages intervals or sockets, implement a `dispose()` method and call it in the server shutdown hook (`httpServer.on('close', () => objectStorage.dispose())`).*
- **`server/routes.ts` (catch blocks throughout async route handlers):** Multiple async handlers (visible in the truncated section and the Google auth block) pass `e?.message` directly into JSON error responses and, implicitly, into console/Sentry logs. For a children's platform, stack traces or internal error messages that include DB schema details, org IDs, or email addresses must not leak to clients or appear unredacted in logs.
  → *Create a central error-response helper: `function safeError(e: unknown, fallback = 'An unexpected error occurred') { console.error('[route-error]', e); return fallback; }` Return only the fallback string to the client. Log the full error server-side only. For Sentry, use `captureException(e)` without echoing e.message to the response body.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 40-question paper this is 40 serial round-trips to the DB, dramatically increasing total job time and holding a DB connection for longer than necessary.
  → *Collect all valid question objects into an array, then do a single batch insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the entire extract + insert block in a transaction so the idempotency delete and the new inserts are atomic.*
- **`server/jobHandlers.ts` (runExtractPaper — openai model and max_completion_tokens literals):** The OpenAI model name ('gpt-4o') and token limits (4096) are hardcoded in two separate completions.create calls. A model deprecation or cost-control change requires a code deploy instead of a config update, and the two calls can silently drift apart.
  → *Add OPENAI_EXTRACTION_MODEL (default 'gpt-4o') and OPENAI_EXTRACTION_MAX_TOKENS (default 4096) to environment config and reference them as process.env.OPENAI_EXTRACTION_MODEL and Number(process.env.OPENAI_EXTRACTION_MAX_TOKENS) in both calls.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call, and olderThanDays is used directly from job.payload without validation. A payload with olderThanDays: 0 or a negative value would delete all rows; a non-numeric value would produce an invalid Date silently passed to the query.
  → *Add input validation at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('Invalid olderThanDays'); Then wrap the db.delete in a try/catch that re-throws so the job worker can record the failure correctly.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** If the readable stream emits an error after headers have already been sent, the error handler silently swallows it with no logging. The client receives a truncated response with a 200 status and no indication of failure, making this class of error invisible in production.
  → *In the error handler, always log the error (e.g. console.error('[objectStorage] stream error', err)) regardless of whether headers were sent. Consider also calling res.destroy() after logging when headers are already sent so the client connection closes cleanly rather than hanging.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow()):** createdAt is defined without .notNull() on every table (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, learningModules). defaultNow() sets a server default but does not add a NOT NULL constraint, so a raw INSERT that explicitly passes NULL will succeed and leave a null audit timestamp.
  → *Change every occurrence to timestamp('created_at').defaultNow().notNull(). This is a one-line change per table and closes a silent data-integrity hole that would be painful to backfill later.*
- **`shared/schema.ts` (simulatedStocks table definition):** simulatedStocks has no unique index on ticker, yet ticker is the natural business key used in UI display and portfolio lookups. Duplicate tickers can be inserted, causing ambiguous joins in portfolioHoldings and portfolioTransactions without any DB-level protection.
  → *Replace the plain index (if any) with uniqueIndex('uq_simulated_stocks_ticker').on(t.ticker) inside the table's index callback. Also add index('idx_simulated_stocks_type').on(t.type) since filtering by stock vs bond is a common query pattern.*
- **`shared/schema.ts` (learningModules / userProgress tables):** The truncated portion of the file (learningModules, userProgress) almost certainly contains a composite unique constraint need: a user should not have duplicate progress rows for the same module. Without a unique constraint on (userId, moduleId), concurrent requests (e.g., rapid lesson completion taps on mobile) can insert duplicate rows that break completion-percentage calculations.
  → *Add uniqueIndex('uq_user_progress_user_module').on(t.userId, t.moduleId) to the userProgress table's index callback. Pair this with an ON CONFLICT DO UPDATE upsert in the application layer so retries are idempotent.*
- **`shared/schema.ts` (categories table — userId: varchar('user_id').references(() => users.id)):** userId on categories is nullable (no .notNull()), which is intentional for system categories, but there is no partial index to efficiently find a specific user's custom categories versus system ones. A query like WHERE user_id = $1 OR user_id IS NULL will never use a simple btree index and forces a full scan of all system + user categories on every category list request.
  → *Add two partial indexes: index('idx_categories_user').on(t.userId) for user-owned rows, and separately document that system-category queries should filter isDefault = true. Alternatively, split system categories into a separate seed/config table and make userId fully NOT NULL on the categories table, eliminating the OR IS NULL scan entirely.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside any memoization. Since it feeds directly into the queryKey array as a derived string, any parent re-render triggers a new params object and potentially a new query even when filter state has not changed.
  → *Wrap the params construction and the derived query string in a useMemo keyed on [actorType, action, from, to]: const paramString = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p.toString(); }, [actorType, action, from, to]); then use paramString in both the queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery in ObservabilityCard has no staleTime, refetchInterval, or error/loading state handling. The component silently renders empty/undefined data on error and will refetch aggressively on every focus event, producing repeated network hits with no user feedback.
  → *Add staleTime: 60_000, refetchOnWindowFocus: false to the query options. Destructure isLoading and isError and render a loading skeleton or error badge: if (isLoading) return <Skeleton />; if (isError) return <Badge className='bg-red-600'>Unavailable</Badge>;*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** fmtDate accepts 'any' type and silently swallows invalid dates by constructing a Date from an arbitrary value. An invalid date string returns 'Invalid Date' in the UI rather than the fallback '—', and the any type defeats TypeScript's safety entirely.
  → *Type the parameter as string | number | Date | null | undefined. Add an explicit validity check: const dt = new Date(d); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' });*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and tab icon components):** TABS is defined at module level which is correct, but icon components (LayoutDashboard, Globe, etc.) are stored as component references in the array and rendered inline as JSX in the sidebar map without useCallback or memoization on the map callback. Every top-level state change (e.g., activeTab) re-renders the entire sidebar and re-evaluates all icon instantiations.
  → *Extract the sidebar tab list into a memoized component — const Sidebar = React.memo(({ activeTab, onTabChange }) => ( <nav>{TABS.map(t => <TabButton key={t.id} tab={t} active={activeTab === t.id} onClick={onTabChange} />)}</nav> )) — and ensure onTabChange is wrapped in useCallback in the parent so the memo boundary is respected.*

## 💡 Opportunities

### Centralized pagination guard utility — 🔥 High Impact
Create a `safePaginate({ limit, offset, maxLimit = 200 })` helper used by every list-returning method (getTransactions, getLeaderboard, getAllQuestions, getGameSessions). This prevents unbounded queries site-wide and makes audit straightforward — one function to review instead of scattered ad-hoc checks.

### DataLoader-style batching for cross-entity lookups — ⚡ Medium Impact
Introduce a per-request Map-based batch loader for users and simulatedStocks lookups. Methods like getEnrollmentsByClass and getPortfolioHoldings both need related-entity data; batching via inArray and a shared cache eliminates redundant DB hits when these methods are called close together in a single request lifecycle.

### Move hardcoded seed/config values to environment config — ⚡ Medium Impact
seedMarketData and seedLearningModules likely contain hardcoded stock tickers, prices, and module content. Extract these to JSON config files loaded at startup and gated by NODE_ENV, preventing accidental data overwrites in production and making content updates deployable without code changes.

### Extract route handlers into domain-scoped sub-routers — 🔥 High Impact
routes.ts is a single file exceeding 195k characters. Splitting it into sub-routers (authRouter, lessonRouter, orgRouter, aiRouter) loaded via app.use() would make each domain independently testable, allow per-router middleware (e.g. rate limiters, auth guards) to be applied cleanly, and reduce the risk of security-relevant middleware being accidentally skipped for a subset of routes.

### Centralise and enforce isAuthenticated before all /api routes — 🔥 High Impact
Auth checks appear to be applied per-handler rather than as a router-level guard. A single `app.use('/api', isAuthenticated)` (with explicit opt-outs for public endpoints like /api/auth/*) would eliminate the entire class of accidentally unauthenticated endpoints, which is especially important given the student demographic and the presence of org/quota data.

### Move hardcoded file-size and MIME-type limits to environment config — ⚡ Medium Impact
videoUpload has a 500 MB limit and examUpload has a 10 MB limit defined as magic numbers. Externalising these to environment variables (e.g. VIDEO_UPLOAD_MAX_MB, EXAM_UPLOAD_MAX_MB) allows limits to be tightened in production without a code deploy and makes the values auditable in infrastructure config rather than buried in source.

### Atomic idempotency via DB transaction in extract-paper — 🔥 High Impact
Wrapping the delete-then-insert sequence in a single Drizzle transaction eliminates the race window where a retry could observe a partially deleted + partially inserted state. Combined with the batch insert fix, this reduces round-trips from O(n) to 3 (begin, delete+insert, commit) and makes retries fully safe.

### Centralised payload size guard on job enqueue — 🔥 High Impact
Add a guard in the enqueueJob function that rejects any payload exceeding a configurable byte threshold (e.g. JOB_MAX_PAYLOAD_BYTES defaulting to 64 KB). This catches accidental large-payload patterns at enqueue time across all job types, not just extract-paper, before they reach the DB.

### Structured logging for job handler lifecycle — ⚡ Medium Impact
Adding structured log entries (jobType, jobId, attempt, durationMs, outcome) at the start and end of each handler would enable alerting on retry spikes and p95 job latency without requiring additional instrumentation. Currently failures are only visible if the worker framework logs them.

### Centralise currency default as a schema-level enum — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). If the app expands to TTD or JMD, every column default must be hunted down manually. Define a pgEnum('currency', ['BSD','TTD','JMD']) and reference it across all currency columns. This also adds DB-level validation rejecting invalid currency codes on insert.

### Add executedAt / purchasedAt NOT NULL + index for portfolio time-series queries — 🔥 High Impact
portfolioTransactions.executedAt and portfolioHoldings.purchasedAt use defaultNow() without NOT NULL and have no index. Any chart showing portfolio value over time will sort or filter on executedAt. Adding NOT NULL and index('idx_portfolio_tx_user_executed').on(t.userId, t.executedAt) turns those time-series queries from full scans into efficient range scans, critical as transaction history grows.

### Extract repeated numeric precision into a shared column builder — 📌 Low Impact
numeric({ precision: 10, scale: 2 }) appears verbatim in at least 10 column definitions. A one-line helper const money = () => numeric('', { precision: 10, scale: 2 }) (overriding the name at call site) eliminates drift — e.g., a future change to precision: 12 for large investment amounts would require updating one place instead of ten, reducing the risk of inconsistent monetary precision across tables.

### Parallel data fetching on dashboard overview load — 🔥 High Impact
The overview tab likely fires multiple sequential useQuery calls (organizations, schools, students, etc.) that could be co-located under a single Promise.all or combined into one /api/admin/dashboard-summary endpoint. Parallelising these would cut initial load time by the sum of sequential round-trips, which on a Caribbean mobile connection (high latency) is material.

### Code-split heavy chart and table panels into lazy-loaded routes — 🔥 High Impact
Recharts, the DB Viewer, and the Audit Log panel are bundled into the initial AdminDashboard chunk. Wrapping each tab panel in React.lazy with a Suspense fallback would reduce the initial JS parse cost significantly — Recharts alone adds ~150 KB minified — which matters for school devices on limited bandwidth.

### Centralise admin auth guard into a single HOC or route wrapper — ⚡ Medium Impact
If useAdminAuth is called independently in multiple sub-components or panels, a network hiccup can leave individual panels unguarded while others correctly redirect. Move the auth check to a single AdminRoute wrapper at the router level so no panel is ever rendered before the auth state is confirmed, eliminating any window of unauthorised rendering.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic performance and reliability problems: N+1 query patterns in portfolio/enrollment lookups, missing database transactions on multi-step writes, and sequential awaits where Promise.all would reduce latency. For a school-aged financial app, missing auth guards on sensitive data fetches and likely hardcoded secrets compound the risk significantly.

### API Routes (`server/routes.ts`)
routes.ts is a large, monolithic route file with several reliability and security gaps. The most critical issues are missing rate limiting on AI/heavy endpoints (exposing the app to quota exhaustion and cost amplification), sequential awaits in hot paths that should be parallelised, and sensitive data potentially surfacing in logs. Overall structure is reasonable but the file's size makes auditing difficult and increases the risk of missed edge cases.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: the extract-paper job stores raw base64 file content in the job payload (large payload anti-pattern), question inserts run sequentially in a loop instead of in a batch, and hardcoded model strings and token limits should be environment config. Error boundaries and idempotency are partially addressed but incomplete in the CSV export path.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and basic composite indexes in place, but has several reliability and performance gaps. Key concerns include missing indexes on high-frequency filter columns, nullable createdAt timestamps that should be NOT NULL, and an undersized varchar for userId that could silently truncate Clerk/Auth UIDs.
