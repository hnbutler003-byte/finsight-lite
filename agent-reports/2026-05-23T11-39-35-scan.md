## 🔴 Critical (11)

### `server/storage.ts` — getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query is issued to sum transactions for that budget's category. With 10 budgets this is 11 queries; with 50 it's 51.

**Fix:** Replace the per-budget spent lookup with a single aggregated query using GROUP BY categoryId joined or subqueried against the budgets result set. Use Drizzle's sql`SELECT category_id, SUM(amount) FROM transactions WHERE user_id = ${userId} AND category_id IN ${categoryIds} GROUP BY category_id` and merge results in-memory with a Map keyed on categoryId.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (buy/sell flow caller)
**Issue:** The portfolio buy/sell flow calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three independent awaits with no wrapping database transaction. A process crash or DB error between any two steps leaves the ledger in an inconsistent state — e.g., balance debited but holding not recorded.

**Fix:** Wrap all three operations inside a single db.transaction(async (tx) => { ... }) block using Drizzle's transaction API. Pass the transaction client (tx) into each operation so they share the same atomic unit. Roll back automatically on any thrown error.

### `server/routes.ts` — registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 95-97)
**Issue:** No rate limiting is applied before registering AI-heavy routes (chat, image, audio). These endpoints likely call OpenAI APIs and have per-token cost implications. A school-aged user base is a prime target for abuse or accidental runaway usage, and there is no throttle protecting them.

**Fix:** Create a dedicated aiLimiter in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before calling the three register functions.

### `server/routes.ts` — app.get('/public-objects/:filePath(*)')
**Issue:** The filePath parameter is passed directly to objectStorage.searchPublicObject without sanitisation. A crafted request like `/public-objects/../../private/secrets` could traverse outside the intended public bucket prefix, exposing private objects depending on the ObjectStorageService implementation.

**Fix:** Normalise and validate the path before use: `const safe = path.normalize(filePath).replace(/^\.\.\//, ''); if (safe !== filePath || filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` — import Node's `path` module and apply this before calling objectStorage.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 — the full base64-encoded exam file — is stored directly in the job table payload. A single scanned PDF can easily be 5–20 MB encoded, meaning every enqueue, retry, and poll query carries this blob. This bloats the jobs table, slows queue polling, and risks hitting row-size limits.

**Fix:** Upload the file to object storage before enqueueing the job, store only the resulting storage path in the payload (e.g., { paperId, storagePath, ext, subject }), then download the buffer at the start of runExtractPaper using objectStorageClient. This keeps job rows small and makes retries cheap.

### `shared/schema.ts` — budgets table definition
**Issue:** No index on budgets(userId) or budgets(userId, categoryId). Every budget lookup by user — which is the primary query pattern — will do a full table scan as the table grows.

**Fix:** Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also enforces the business rule that a user can only have one budget per category per period.

### `shared/schema.ts` — billReminders, savingsGoals, linkedCards, portfolioHoldings, portfolioTransactions table definitions
**Issue:** No indexes on userId in billReminders, savingsGoals, linkedCards, portfolioHoldings, or portfolioTransactions. All of these tables will be queried by userId in every user-facing request, causing full table scans.

**Fix:** Add index definitions to each table's second argument, e.g. for billReminders: (t) => ({ userIdx: index('idx_bill_reminders_user').on(t.userId), dueDateIdx: index('idx_bill_reminders_due').on(t.nextDueDate) }). Repeat the pattern for each affected table with their respective userId columns.

### `server/index.ts` — seedDatabase / probeDatabase calls in startup IIFE
**Issue:** seedDatabase() and probeDatabase() are awaited sequentially during startup but have no individual timeouts. If either hangs — e.g. database unreachable — the process blocks indefinitely without ever binding the HTTP port, making health checks fail silently and leaving the process unkillable by orchestrators that expect a bound port within N seconds.

**Fix:** Wrap each call with Promise.race against a 10-second AbortSignal or manual timeout: `await Promise.race([probeDatabase(), rejectAfter(10_000, 'probeDatabase timed out')])`. Run probeDatabase and seedDatabase in parallel with Promise.all where order allows, and ensure the HTTP server binds before the heavy init so liveness probes can respond.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are assigned to variables but are never passed to clearInterval in the SIGTERM/SIGINT shutdown handler. If a new deployment sends SIGTERM while an interval callback is mid-await, Node will not exit cleanly; the process is force-killed, potentially leaving a half-written weeklyDigestRuns claim in the database with no rollback.

**Fix:** Capture both handles: `const digestTimer = setInterval(...)` and `const purgeTimer = setInterval(...)`. In the graceful shutdown handler, call `clearInterval(digestTimer)` and `clearInterval(purgeTimer)` as the very first step before awaiting server.close(), so no new scheduler invocations start during drain.

---

### `server/index.ts` — capturedJsonResponse logging — sensitive data exposure
**Issue:** The middleware logs the full JSON response body of every /api route, including routes that return user profile data, session tokens, or financial quiz results belonging to minors. For a COPPA/FERPA-adjacent app serving school-aged users in the Caribbean, logging PII to stdout (which flows to Replit's log aggregator) is a data-handling violation.

**Fix:** Add a response-body allowlist: only log bodies for routes that are explicitly marked safe (e.g. /api/health, /api/version). For all other /api routes log only method, path, status, and duration. If response inspection is needed for debugging, gate it behind a DEBUG_LOG_BODIES env flag that is off by default and never enabled in production.

### `client/src/pages/AdminDashboard.tsx` — TABS constant — tab id 'dbviewer'
**Issue:** A DB Viewer tab is exposed in the admin UI with icon Building2. If the underlying API endpoint does not enforce a secondary privilege check (e.g. super-admin role vs plain admin role), any admin account compromise gives direct database read access. The client-side tab definition also means the route is discoverable via source maps.

**Fix:** Gate the DB Viewer tab render conditionally on a `isSuperAdmin` flag returned from the auth hook: `TABS.filter(t => t.id !== 'dbviewer' || isSuperAdmin)`. Ensure the server-side /api/admin/db-viewer endpoint validates a `role === 'superadmin'` claim independently of the client filter.

## 🟡 Warnings (16)

- **`server/storage.ts` (getLeaderboard):** No pagination guard: getLeaderboard accepts an optional limit but if the caller omits it the query returns every user's aggregated score with no upper bound. On a school deployment with thousands of students this is an unbounded full-table scan returned entirely into memory.
  → *Apply a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 50, 200); and always append .limit(safeLimit) to the query. Document the cap in the interface signature with a JSDoc comment.*
- **`server/storage.ts` (getAllQuestions):** N+1 query pattern: getAllQuestions fetches extracted questions then, for each question, issues a separate query to fetch its parent ExamPaper to satisfy the { paper?: ExamPaper } return shape.
  → *Perform a single LEFT JOIN between extractedQuestions and examPapers in one Drizzle query: db.select().from(extractedQuestions).leftJoin(examPapers, eq(extractedQuestions.paperId, examPapers.id)). Map the flat result rows into the expected { ...question, paper } shape client-side.*
- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregations: total income, total expenses, transaction list, and savings goals are each awaited one after another despite having no data dependency on each other, adding cumulative latency on every dashboard load.
  → *Replace sequential awaits with const [income, expenses, txns, goals] = await Promise.all([queryIncome, queryExpenses, queryTxns, queryGoals]). Each query is already scoped to userId so they are safely parallelisable.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) (~line 101-109)):** The seven Supabase init functions are chained sequentially with .then(), meaning each waits for the previous before starting. If each takes ~200ms, startup side-effects take 1.4s+ unnecessarily. Additionally, the catch only logs; individual step failures silently skip all subsequent steps with no per-step error context.
  → *Run independent seed operations in parallel: `Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])` inside the chain where dependencies allow. At minimum split into two phases: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(...)`. Add per-step try/catch or `.catch` to log which step failed.*
- **`server/routes.ts` (app.get('/healthz') (~line 66)):** Dynamic import of `./db` and `drizzle-orm` inside the health check handler runs on every health probe. Railway and uptime monitors may call this every 10-30 seconds, causing repeated module resolution overhead and a new sql expression allocation each time.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The db module is a singleton and safe to import at module load time.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions extracted from the AI response are inserted one-by-one inside a sequential for loop. For a 60-question paper this is 60 serial round-trips to the database, each adding network latency and holding the job worker busy unnecessarily.
  → *Replace the for loop with a single bulk insert: filter valid questions into an array first, then call db.insert(extractedQuestions).values(validRows) once. If storage.createExtractedQuestion must be used for side-effects, at minimum use Promise.all on the mapped promises so inserts are concurrent.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls, model field):** The OpenAI model is hardcoded as the string "gpt-4o" in two separate completions calls. If the organisation needs to switch models (cost, availability, or a newer version), every call site must be found and edited manually, and there is no way to change this per environment without a redeploy.
  → *Add OPENAI_MODEL to environment config (e.g., process.env.OPENAI_MODEL ?? 'gpt-4o') and reference it from a single constant at the top of the file. Apply the same pattern to max_completion_tokens if those limits should be tunable.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage job has no try/catch around the db.delete call. An unexpected DB error (connection drop, lock timeout) will propagate as an unhandled rejection from the job handler. Depending on the job runner's error handling, this may crash the worker process rather than marking the job as failed and scheduling a retry.
  → *Wrap the db.delete call in try/catch, log the error with enough context (cutoff date, error message), and re-throw so the job framework can record the failure and retry according to its policy. Apply the same pattern to admin-csv-export and any other handlers lacking error boundaries.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — createdAt columns):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). Drizzle will type this as Date | null, so application code must null-check it everywhere. A row with a null createdAt is semantically invalid.
  → *Append .notNull() to every createdAt definition, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The defaultNow() ensures the DB always populates it, so notNull() is safe to add.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker column has no unique index. Duplicate tickers would silently corrupt portfolio calculations and investment simulation results, which is especially dangerous in a school-facing financial literacy context where data correctness teaches real habits.
  → *Add a unique index in the table's second argument: (t) => ({ tickerIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also consider adding .notNull() to annualReturnPct since a stock missing a return percentage will break any yield calculation without a null guard.*
- **`shared/schema.ts` (categories table — userId column):** userId on categories uses varchar without a length specifier. Depending on the auth provider (e.g. Clerk or Supabase UUID strings), this could silently truncate IDs on some Postgres configurations, causing orphaned or mismatched category lookups.
  → *Match the varchar length to the auth provider's ID format. If using UUIDs use varchar('user_id', { length: 36 }) or switch to the uuid() column type. Apply the same fix to all other varchar user_id columns for consistency.*
- **`server/index.ts` (res.json monkey-patch inside request logging middleware):** Every API response body is captured into capturedJsonResponse and then serialized again via JSON.stringify for the log line. For paginated list endpoints returning dozens of records this doubles serialization cost on every request and retains the entire response object in the closure until the 'finish' event fires. On a school-wide quiz submission spike this will cause GC pressure and inflated latency.
  → *Limit logging to non-200 responses or truncate: `logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 200)`. Better still, only capture the response body when NODE_ENV !== 'production' or when the status code is >= 400, controlled by an env flag like LOG_RESPONSE_BODIES=true.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes Sunday in the server's local timezone using `new Date()` and `setHours(19,0,0,0)`, which depends entirely on the TZ environment variable of whatever container the app runs in. For a Caribbean app (UTC-4/UTC-5) deployed on a UTC Replit VM, 7 PM local time becomes 11 PM UTC, so the digest fires at the wrong time and the weekStart ISO string sliced from toISOString() will often be Monday UTC, causing duplicate or skipped runs across DST boundaries.
  → *Pin the timezone explicitly: set `process.env.TZ = 'America/Port_of_Spain'` at the very top of the entry point (before any Date construction), or use the Temporal API / date-fns-tz to compute the target instant in a named timezone rather than relying on the host TZ setting.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams is rebuilt on every render outside of useMemo. Because the object reference changes each render, any child that receives it as a prop or any effect that depends on it will re-execute unnecessarily, and the queryKey array uses primitive spread while the actual URL string used in the fetch is derived separately — a mismatch that could cause stale data if filters change mid-flight.
  → *Wrap the params construction and the derived query string in useMemo: `const { paramStr } = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return { paramStr: p.toString() }; }, [actorType, action, from, to]);` then use `paramStr` in both the queryKey and the fetch URL.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no error state handled and no loading state rendered. If the /api/admin/observability endpoint fails or is slow, the component silently renders with all badges showing 'Disabled' — indistinguishable from a real 'all disabled' configuration state, which could mislead admins into thinking Sentry is not configured when it actually is.
  → *Destructure `isLoading` and `isError` from useQuery and render a skeleton or error message: `if (isLoading) return <Card>...loading spinner...</Card>; if (isError) return <Card>...error message with retry button...</Card>;`*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log page size limit of 200 is hardcoded inline. For a financial literacy platform with school-wide usage, this will silently truncate logs with no pagination or indication to the admin that records are missing, creating a compliance and auditability risk.
  → *Extract to a named constant at module scope: `const AUDIT_LOG_PAGE_LIMIT = 200;` and add a visible UI note when `rows.length === AUDIT_LOG_PAGE_LIMIT` such as 'Showing first 200 results — refine filters to narrow results.' Consider adding cursor-based pagination to the API.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (app.get('/api/health') and app.get('/healthz') (~lines 36, 66)):** There are two health check endpoints doing different things: `/api/health` returns a trivial JSON response with no DB check, and `/healthz` performs an actual DB liveness probe. This is duplicate/dead surface — Railway and uptime monitors should only need one canonical endpoint, and the shallow one gives a false positive if the DB is down.

The original fileB64 string from the payload is identical to this re-encoded value, making the Buffer.from/toString round-trip pure wasted CPU and memory allocation.
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper):** fmtDate accepts `any` and calls `new Date(d)` without validation. Passing a malformed string produces 'Invalid Date' rendered to the UI, and in a table with many rows this creates confusing output. The `any` type also defeats TypeScript's ability to catch misuse at call sites.

## 💡 Opportunities

### Centralise sensitive-field redaction before logging — 🔥 High Impact
Student PII (names, emails, virtual balances) and financial figures flow through a single storage layer. Adding a thin redactForLog() helper that strips sensitive keys before any console.log/error call prevents accidental PII leakage into log aggregators — critical for a COPPA/FERPA-adjacent school product.

### Extract magic numbers and default seeds to environment config — ⚡ Medium Impact
Values like default virtual balance amounts, leaderboard caps, and stock seed prices appear hardcoded in storage methods. Moving them to a validated config object (using zod or envalid at startup) makes tuning safe and auditable without code deploys.

### Add a DataLoader or request-scoped cache for repeated user/stock lookups — ⚡ Medium Impact
Several methods (getEnrollmentsByClass, getPortfolioHoldings) re-fetch the same User or SimulatedStock rows multiple times within a single request. A lightweight per-request Map cache or DataLoader would collapse these into single fetches and reduce DB round-trips by an estimated 40-60% on class-progress endpoints.

### Structured startup logging for Supabase init chain — ⚡ Medium Impact
Replace the single .catch console.error with per-step timing logs (console.info('[Supabase] seedStaticModules complete in Xms')). For a school app with frequent deploys this dramatically reduces debugging time when a seed step silently fails in production, with no runtime cost.

### Extract ObjectStorageService instantiation to a shared singleton module — 📌 Low Impact
objectStorage is instantiated at module load time inside routes.ts. If routes.ts is ever imported elsewhere or tested in isolation, a second instance is created. Move `new ObjectStorageService()` to its own `server/storage.ts` singleton export; this also makes it mockable in tests.

### Add request-scoped timeout middleware for AI routes — 🔥 High Impact
OpenAI streaming or image generation can hang indefinitely on network issues. Adding a per-request timeout (e.g. via `express-timeout-handler` or a simple setTimeout with res.destroy()) on the /api/chat, /api/image, /api/audio prefixes would prevent Node's event loop from accumulating stalled requests and exhausting connection pool slots — high value for a school environment with unpredictable network quality.

### Centralise OpenAI client configuration and model constants — ⚡ Medium Impact
The openai client is imported from a route file (replit_integrations/chat/routes) rather than a dedicated integration module, and model names are scattered as literals. Extracting a single ai.ts config module with typed model constants and the client instance would eliminate the cross-layer import, make model upgrades a one-line change, and allow per-environment overrides via env vars. Expected impact: reduced future regression risk and faster model migrations.

### Stream large CSV exports rather than buffering in memory — 🔥 High Impact
The admin-csv-export handler (visible in the truncated section) likely builds the full CSV string in memory before writing to object storage. For exports over thousands of rows this creates a large heap spike per job execution. Using a streaming CSV library (e.g., csv-stringify in streaming mode) piped directly to objectStorageClient's write stream would cap memory use to a fixed buffer size regardless of row count.

### Add idempotency key validation at job enqueue time for extract-paper — ⚡ Medium Impact
The current idempotency strategy deletes previously inserted questions at the start of each attempt, which is correct but destructive — a concurrent retry could race with an in-progress attempt and delete rows it did not insert. Generating a stable idempotency key (e.g., hash of paperId + file checksum) and checking it before enqueueing would prevent duplicate jobs from being created at all, removing the need for the delete-on-retry pattern entirely.

### Centralise currency default into a config constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a shared constant DEFAULT_CURRENCY = 'BSD' in a config file and reference it in each .default() call. This means a future currency change or multi-region expansion requires a one-line edit rather than hunting across the schema.

### Add partial index on documentUploads for pending processing jobs — ⚡ Medium Impact
Background jobs that poll for documents in 'processing' status will scan the entire documentUploads table. Adding a partial index — index('idx_document_uploads_processing').on(t.status).where(sql`status = 'processing'`) — limits the index to only unfinished rows, keeping it tiny and fast regardless of historical data volume.

### Replace text() with pg enum types for repeated enum columns — 📌 Low Impact
Columns like type, status, frequency, riskLevel, and period use Drizzle's text enum which stores raw strings in Postgres with no DB-level constraint enforcement beyond the application layer. Switching to pgEnum creates a true Postgres ENUM type, which is validated at the DB layer, uses less storage (4 bytes vs variable text), and documents the domain directly in the schema.

### Parallel startup initialization — 🔥 High Impact
registerRoutes(), serveStatic(), startJobWorker(), registerJobHandlers(), and the scheduler setup are all independent after probeDatabase resolves. Running them with Promise.all where safe would reduce cold-start time by roughly the duration of the slowest sequential await chain, which on a Replit cold boot with a remote DB can be 2-4 seconds.

### Externalize scheduler intervals to environment config — ⚡ Medium Impact
The digest and purge check intervals (currently hardcoded, visible in the truncated section) should be read from env vars like DIGEST_CHECK_INTERVAL_MS and PURGE_CHECK_INTERVAL_MS. This lets ops teams tune polling frequency without a redeploy and makes tests fast by setting them to 100ms.

### Replace child_process.spawn Vite dev call with in-process Vite middleware — ⚡ Medium Impact
Spawning a separate Vite process via spawn() in development adds a second Node process, doubles memory usage on a free Replit tier, and loses structured error propagation. Using Vite's createServer() API in-process gives the same HMR experience with shared memory, a single port, and errors that surface in the same log stream.

### Colocate per-tab data fetching into child components — 🔥 High Impact
If all tab queries are issued at the top-level AdminDashboard component, every tab switch triggers re-renders of all sibling query consumers. Moving each tab's useQuery calls into the tab's own child component means only the active tab fetches and re-renders, reducing network load and render cost significantly for a dashboard with 13 tabs.

### Replace inline JSX arrow functions on interactive elements with useCallback — ⚡ Medium Impact
Large inline functions in JSX (onClick handlers on Buttons inside table rows, for example) create new function references every render, causing React to re-render list rows that otherwise have stable props. Extracting these with useCallback and passing stable references to memoized row components would measurably reduce paint time on the Students and Schools tables which likely contain 50–500 rows.

### Add a staleTime and refetchInterval strategy per query sensitivity — ⚡ Medium Impact
Audit log and observability data does not need to refetch on every window focus, but job status and perf agent data probably does. Configuring `staleTime: 5 * 60 * 1000` on slow-changing queries (organizations, schools) and `refetchInterval: 10000` only on jobs/perf queries would cut redundant API calls by an estimated 60–70% during normal admin sessions.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits several critical reliability and security gaps including N+1 query patterns in budget/bill-reminder lookups, missing database transactions for multi-step writes, and sequential awaits that block unnecessarily. The file is large and complex enough that hardcoded fallback values and missing pagination guards compound into real production risk for a school-aged user base where data integrity matters most.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel module loading and rate limiting on auth endpoints. Key concerns are unprotected AI/chat/image/audio endpoints with no rate limiting, a swallowed Supabase init chain that runs sequentially instead of in parallel, and a path traversal risk on the public object storage endpoint.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and a hardcoded OpenAI model string that bypasses environment configuration. The extract-paper handler is the most critical path, carrying the full file payload in the job row and inserting questions one-by-one in a loop, both of which will degrade under load.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some composite indexes, but has several reliability and performance gaps. Key concerns include missing indexes on high-cardinality lookup columns, nullable createdAt timestamps that should be NOT NULL, and missing indexes on columns used in likely WHERE/ORDER BY clauses across multiple tables.

### Server Startup (`server/index.ts`)
The server startup sequence contains blocking synchronous-style async operations, interval handles that are never cleared on shutdown, and a graceful shutdown handler that is partially implemented. The logging middleware serializes every API response body to JSON on every request, which is a memory and CPU concern in production. Overall structure is reasonable but several reliability gaps need addressing before this handles real student data safely.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and performance concerns including missing error/loading states on queries, inline object allocations causing unnecessary re-renders, and a URLSearchParams construction that escapes useMemo and rebuilds on every render. The file is large and monolithic, making auth gate bypasses and duplicate query calls harder to audit. No critical security holes are visible in the truncated portion, but the patterns present risk at scale.
