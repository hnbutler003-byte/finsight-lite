## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 107-109)
**Issue:** No rate limiting is applied to AI chat, image, or audio routes before they are registered. These are the most expensive endpoints (OpenAI API calls) and have no per-IP or per-user throttle, making them trivially abusable for cost exhaustion attacks.

**Fix:** Add a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); immediately before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 43) and sitemap.xml handler (~line 56)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can inject an arbitrary host header to produce a sitemap or robots.txt pointing to a malicious domain, enabling cache-poisoning or SEO manipulation.

**Fix:** Resolve the base URL once at startup from a CANONICAL_URL environment variable (e.g. process.env.CANONICAL_URL || 'https://finsightlite.com') and use that constant in both handlers. Never trust host/x-forwarded-host for URL generation in outbound content.

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** ticker column has no unique index. In an investment simulation app, stock lookups by ticker (e.g. WHERE ticker = 'BISX') will perform sequential scans, and duplicate tickers can be silently inserted, corrupting portfolio calculations.

**Fix:** Add uniqueIndex to the simulatedStocks table definition: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This enforces uniqueness at the DB level and turns ticker lookups into O(log n) index scans.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are created with setInterval but their handles are never stored or cleared. The SIGTERM/SIGINT graceful-shutdown handler closes the HTTP server and job worker but does not clear these intervals, so the process can stay alive after shutdown is requested, and in test environments the open handles prevent process exit.

**Fix:** Capture handles: `const weeklyDigestInterval = setInterval(...)` and `const purgeInterval = setInterval(...)`. In the shutdown handler add `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before or alongside `httpServer.close()`.

---

### `server/index.ts`: res.json monkey-patch in request-logging middleware
**Issue:** Every API response body is captured into capturedJsonResponse and then serialized via JSON.stringify into the log line unconditionally. For endpoints that return user financial records, quiz scores, session tokens, or any PII this writes full sensitive payloads to stdout/console in production, violating data-minimisation principles and creating a compliance risk for a children's financial app.

**Fix:** Either drop response-body logging entirely in production (`if (process.env.NODE_ENV !== 'production' && capturedJsonResponse)`), or replace it with a safe summary (status code + top-level keys only, no values): `logLine += ` :: keys=[${Object.keys(capturedJsonResponse).join(',')}]`;`.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat.sendMessage
**Issue:** The full chat message history (`next`) is sent to /api/admin/help-chat on every message. As the conversation grows this sends an ever-increasing payload, and there is no cap or truncation. For a school-aged-user platform this also risks leaking prior conversation context (including any sensitive admin data mentioned) in every subsequent request body.

**Fix:** Slice the history before sending: `const contextWindow = next.slice(-10);` and send `{ messages: contextWindow }` instead of the full `next` array. Add a MAX_MESSAGES constant (e.g. 50) and trim `messages` state when it exceeds that limit to prevent unbounded memory growth.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level auth check
**Issue:** Admin route protection appears to rely entirely on the client-side `useAdminAuth` hook. If the hook only checks a client-stored token/session without the server enforcing admin role on every API call, a user who manually removes the redirect or replays a non-admin token can access all admin mutations (create/delete teachers, schools, etc.).

**Fix:** Ensure every `/api/admin/*` endpoint on the server validates the session token and asserts `role === 'admin'` server-side, returning 401/403 otherwise. The client hook should be treated as UX-only gating, never the security boundary. Add an integration test that hits admin endpoints with a student-role token and asserts 403.

## 🟡 Warnings (12)

- **`server/routes.ts` (Supabase init chain (~line 112-120)):** The chained .then() Supabase init sequence is non-atomic: if seedFinancialAcademyLesson() rejects, initStaticContentTables(), seedStaticModules(), seedRegionalContent(), and seedGameContent() are all silently skipped with only a single catch log. There is no way to distinguish which step failed or retry individual steps.
  → *Run independent seed steps in parallel where safe (Promise.allSettled) and log each rejection individually: Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] Seed step', i, r.reason))). Chain only steps with true sequential dependencies.*
- **`server/routes.ts` (/healthz handler (~line 76)):** The /healthz endpoint uses dynamic import('./db') and import('drizzle-orm') on every request. Dynamic imports are cached after first load but the repeated await import() calls add unnecessary microtask overhead on a hot path that uptime monitors hit every 30-60 seconds.
  → *Move db and sql imports to the top of the file as static imports alongside the other imports. The /healthz handler then uses the already-resolved module references with zero dynamic import overhead.*
- **`shared/schema.ts` (portfolioHoldings table definition):** No index on (userId, stockId). Every portfolio value calculation will require a full scan of portfolio_holdings filtered by userId. For a school-aged user base that may grow significantly, this becomes a latency bottleneck on the most frequently read table.
  → *Add a composite index and a unique constraint: (t) => ({ userStockIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). The unique constraint also prevents duplicate holding rows per user per stock, which would cause double-counting of positions.*
- **`shared/schema.ts` (budgets table definition):** No index on userId and no unique constraint on (userId, categoryId, period). Queries fetching budgets for a user scan the entire table, and a user can insert multiple budget rows for the same category+period, producing ambiguous budget lookups in application code.
  → *Add: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryPeriodUniqueIdx: uniqueIndex('idx_budgets_user_category_period').on(t.userId, t.categoryId, t.period) }). This enforces one budget per category per period per user and speeds user-scoped queries.*
- **`shared/schema.ts` (transactions, budgets, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — createdAt columns):** createdAt is defined as .defaultNow() without .notNull(). This allows NULL to be inserted if the default is bypassed (e.g. raw SQL inserts, migrations, or test fixtures), breaking any ORDER BY createdAt queries and audit trails silently.
  → *Chain .notNull() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same fix to executedAt in portfolioTransactions and purchasedAt in portfolioHoldings.*
- **`shared/schema.ts` (billReminders table definition — nextDueDate column; billReminders queries):** No index on nextDueDate or (userId, nextDueDate). The primary use-case for bill reminders is fetching upcoming bills (WHERE next_due_date <= now() ORDER BY next_due_date), which will full-scan the table without an index. At scale this is the worst-case query pattern for a reminder system.
  → *Add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due_date').on(t.userId, t.nextDueDate) }). This supports both user-scoped upcoming-bill queries and potential background jobs that sweep all reminders due within a date window.*
- **`server/index.ts` (probeDatabase / startup block):** probeDatabase() is awaited sequentially before seedDatabase() and startJobWorker(), but if probeDatabase itself does a synchronous DNS lookup or blocking I/O before returning a promise, it stalls the Node.js event loop during startup. Even if fully async, the three startup tasks (probe, seed, startJobWorker) run sequentially when probe and seed have no dependency on each other.
  → *Run independent startup tasks in parallel: `await Promise.all([probeDatabase(), seedDatabase()])` then `startJobWorker()`. Ensure probeDatabase() never calls any sync I/O (e.g., dns.lookupSync) internally.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — adding zero is a no-op and the comment is misleading. More importantly, the time is set to 19:00 in the server's local timezone, which is non-deterministic across deployments. For a Caribbean app targeting a specific timezone (e.g., AST UTC-4) this will fire at the wrong wall-clock time on any cloud host set to UTC.
  → *Replace the local-time calculation with an explicit UTC offset or use a timezone-aware library. At minimum document and pin the server TZ environment variable (`TZ=America/Puerto_Rico`) in deployment config, and rename the function to `getMostRecentSundayAt7pmAST` to make the assumption explicit.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest async scheduler callbacks):** Both functions are passed directly as the setInterval callback. If either throws an unhandled rejection (e.g., the inner catch re-throws or a code path lacks try/catch), Node.js will emit an unhandledRejection. The outer try/catch in each function catches db errors but a failure in enqueueJob's rollback path inside maybeRunWeeklyDigest is itself awaited inside a catch block with no surrounding try, meaning a rollback failure propagates uncaught.
  → *Wrap the rollback await in its own try/catch: `try { await db.delete(...) } catch (rollbackErr) { log('rollback failed: ' + rollbackErr.message); }`. Additionally wrap each setInterval callback at the call site: `setInterval(() => maybeRunWeeklyDigest().catch(e => captureError(e)), interval)`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat (scroll ref inline)):** The scroll-to-bottom element uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` directly in JSX. This creates a new function on every render and calls scrollIntoView on every re-render of the message list (including while typing in the input), not just when new messages arrive.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This scopes the scroll to only fire when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple top-level useQuery calls):** The dashboard appears to fire several independent useQuery calls (users, schools, lessons, analytics, etc.) sequentially at component mount. Queries that are not dependent on each other still individually contribute to waterfall loading and each missing an `enabled` guard means they fire even when the admin is on an unrelated tab.
  → *Add `enabled: activeTab === 'users'` (or whichever tab owns the data) to each query so only relevant data is fetched when that section is visible. For queries that are always needed, confirm they are not duplicated across sibling components which would cause redundant network calls.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline mutation/handler functions in JSX):** Button onClick handlers for delete, promote, and edit operations are defined as large inline arrow functions directly in JSX map callbacks (e.g. `onClick={() => { mutation.mutate(...); toast(...); }}`). These recreate new function references on every render of the list, which is particularly costly when mapping over large user/school arrays.
  → *Extract these into useCallback-memoized handlers outside the JSX: `const handleDeleteUser = useCallback((id: number) => { deleteMutation.mutate(id); }, [deleteMutation]);` and pass `onClick={() => handleDeleteUser(user.id)}` in the map. For list items, consider extracting a memoized row component with React.memo to prevent re-renders when unrelated state changes.*

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six registerX(app) calls after Promise.all are executed sequentially with await. If any route registrar does async work (DB calls, file reads) during registration, they block each other. Wrap them in a single Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) to reduce cold-start latency, which matters for Railway's spin-up time.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Replacing console.error in the Supabase catch with a structured logger (already used via Sentry) that includes step name, duration, and error code would make production incidents diagnosable without log-diving. Expected impact: faster MTTR when seeding fails silently on deploy.

### ObjectStorageService singleton extraction: 📌 Low Impact
objectStorage is instantiated at module load time as a module-level const but there is no connection pooling or error boundary if the storage service is unavailable at startup. Wrapping instantiation in a lazy singleton with a readiness check would prevent the entire route file from failing to load if object storage credentials are missing in a given environment.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract a const DEFAULT_CURRENCY = 'BSD' in a shared config file and reference it in each .default() call. If the app ever expands to TTD or JMD support, this is a single-line change instead of a risky find-and-replace across the schema.

### Add partial index for unprocessed document uploads: ⚡ Medium Impact
documentUploads rows with status='processing' are the ones actively polled by background jobs. Add a partial index: index('idx_document_uploads_processing').on(t.userId, t.createdAt).where(sql`status = 'processing'`). This dramatically reduces the index size and speeds up the polling query without touching the full-table index.

### Replace text type with varchar for bounded string columns: 📌 Low Impact
Columns like lastFour (always 4 chars), ticker (typically 2-5 chars), fileType (MIME type, ~30 chars max), and currency (3-char ISO code) are declared as unbounded text. Switching to varchar with explicit length limits (e.g. varchar('ticker', { length: 10 })) enforces data contracts at the DB layer, prevents runaway string storage from bad input, and signals intent to future engineers without adding application-layer validation.

### Structured log context instead of string interpolation: 🔥 High Impact
Replace the free-form log() string concatenation with a structured logger (pino or winston JSON mode). This lets log-aggregation tools (Datadog, Logtail) filter by userId, route, and statusCode without regex parsing, and makes it trivial to redact sensitive fields before they leave the process. Expected impact: faster incident diagnosis and enforced PII redaction policy.

### Centralise all magic interval durations in environment config: ⚡ Medium Impact
The weekly-digest check interval and purge check interval appear as numeric literals in setInterval calls. Move them to named env vars (WEEKLY_DIGEST_CHECK_INTERVAL_MS, AI_PURGE_CHECK_INTERVAL_MS) with validated defaults. This allows tuning in staging without code changes and makes the values auditable alongside SESSION_SECRET and AI_PURGE_OLDER_THAN_DAYS which are already env-driven.

### Add HTTP server timeout and keep-alive hardening: ⚡ Medium Impact
The raw http.createServer has no requestTimeout, headersTimeout, or keepAliveTimeout set. Under load or with slow clients this allows connections to hold the server open indefinitely. Set `httpServer.requestTimeout = 30000` and `httpServer.headersTimeout = 35000` before listen(). For a school app with unpredictable mobile network conditions this prevents resource exhaustion.

### Parallel data fetching with Promise.all on dashboard load: 🔥 High Impact
Any initialization logic that sequentially awaits multiple independent API calls (e.g. fetching user count, school count, and analytics separately in a single async function) should be converted to Promise.all([ fetchUsers(), fetchSchools(), fetchAnalytics() ]). This can cut initial dashboard load time by the sum of sequential latencies minus the slowest call.

### Memoize expensive chart data transformations: ⚡ Medium Impact
Recharts data arrays that are derived from raw query results (e.g. grouping users by registration date, aggregating lesson completions) should be wrapped in useMemo with the raw data as the dependency. Without this, every keystroke in a search input or any unrelated state update recomputes the full aggregation and re-renders the BarChart/LineChart trees.

### Move hardcoded API base paths and pagination limits to environment config: ⚡ Medium Impact
Values like page size (e.g. 10 or 20 items per page), the help-chat endpoint path, and any hardcoded domain strings should be sourced from `import.meta.env.VITE_*` variables or a central config module. This prevents silent breakage when endpoints change between environments and makes tuning pagination for lower-end Caribbean school devices (where DOM size matters) a single-line config change.

Overall structure is reasonable but these three issues directly affect reliability, performance, and security for a school-aged user base.

AdminDashboard.tsx has several reliability and security concerns including missing try/catch around async operations, unprotected admin routes that rely solely on client-side auth checks, and inline scroll refs created on every render causing unnecessary work.
