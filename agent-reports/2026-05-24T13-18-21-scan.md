## 🔴 Critical (4)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied beforehand. These are expensive, cost-incurring endpoints — a single unauthenticated user can trigger unbounded AI calls.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions.

---

### `server/routes.ts` — /public-objects/:filePath(*) handler, line ~78
**Issue:** The `filePath` wildcard param is passed directly to `objectStorage.searchPublicObject` without sanitisation. A request to `/public-objects/../../private/secrets` could traverse outside the intended public prefix depending on how the storage client resolves paths.

**Fix:** Sanitise before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, ''); if (!filePath || filePath.startsWith('/')) return res.status(400).json({ error: 'Invalid path' });` Then pass the cleaned value.

---

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full file contents are stored as base64 in the job payload. For a PDF or image this can easily exceed hundreds of KB to several MB per row, bloating the jobs table and causing slow queue polls as the worker deserializes large JSON payloads on every attempt.

**Fix:** Upload the file to object storage at enqueue time and store only the resulting storage path in the job payload. In runExtractPaper, download the file from object storage using objectStorageClient before processing. Remove fileB64 from JobPayloads['extract-paper'] entirely.

---

### `client/src/pages/AdminDashboard.tsx` — useMutation handlers (throughout)
**Issue:** Mutation onSuccess/onError callbacks and any async functions wired to form submissions lack try/catch blocks. If apiRequest throws a network error rather than returning a non-OK Response, the error propagates uncaught, React Query will surface it but any imperative side-effects (toast calls, state resets, navigation) after the await will silently not execute.

**Fix:** Wrap mutation handler bodies in try/catch: `try { await mutateAsync(payload); toast({ title: 'Success' }); } catch (err) { toast({ title: 'Error', description: (err as Error).message, variant: 'destructive' }); }`. Alternatively rely on mutation's onError callback and ensure it always fires a toast so no failure is silent.

## 🟡 Warnings (15)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain, line ~107):** The seven Supabase seed steps are chained sequentially with `.then()`. Steps like `seedStaticModules`, `seedRegionalContent`, and `seedGameContent` are likely independent and could run concurrently, adding unnecessary startup latency.
  → *After `initStaticContentTables()` completes (it may be a prerequisite), run the independent seeds in parallel: `await initStaticContentTables(); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]);` wrapped inside the existing `.then()` and `.catch()` block.*
- **`server/routes.ts` (/api/health and /healthz, lines ~36 and ~52):** There are two separate health check endpoints (`/api/health` and `/healthz`) with different behaviour — one checks DB liveness, the other does not. This is confusing for infrastructure tooling and the comment on `/api/health` says it is used by Railway, but it performs no real liveness check.
  → *Consolidate into a single `/healthz` endpoint that performs the DB ping. Add a lightweight `/api/health` that returns 200 immediately for load-balancer TCP checks only. Remove the duplicate and update Railway config to point to `/healthz`.*
- **`server/routes.ts` (sitemap handler — host/proto construction, line ~43):** `req.headers['x-forwarded-host']` and `x-forwarded-proto` are used without validation. On a misconfigured or publicly accessible server these headers can be spoofed, producing open-redirect-style sitemap entries that reference attacker-controlled domains.
  → *Validate against an allowlist: `const allowedHosts = (process.env.ALLOWED_HOSTS || '').split(','); const host = allowedHosts.includes(rawHost) ? rawHost : process.env.DEFAULT_HOST;` Define `ALLOWED_HOSTS` and `DEFAULT_HOST` in environment config.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a 40-question exam paper this means 40 serial round-trips to the database, multiplying latency and holding the job worker busy unnecessarily.
  → *Collect valid questions into an array first, then perform a single bulk insert: await db.insert(extractedQuestions).values(validRows) where validRows is built with a .filter/.map over the parsed questions array. This reduces N round-trips to 1.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The OpenAI client is imported from a routes module rather than a dedicated service or config module. This creates an undeclared dependency on Express route initialization side-effects, risks importing middleware and route registrations as a side-effect, and will silently break if the routes file is restructured. It also risks leaking the API key configuration into a module loaded in unexpected contexts.
  → *Move the OpenAI client instantiation to a dedicated singleton module (e.g., server/lib/openaiClient.ts) that reads OPENAI_API_KEY from process.env and exports the configured client. Import from there in both jobHandlers.ts and the routes file.*
- **`server/jobHandlers.ts` (registerJobHandler("purge-ai-usage") — olderThanDays arithmetic):** olderThanDays is taken directly from job.payload with no validation. A missing, zero, or negative value produces a cutoff date in the future, causing the handler to silently delete all AI usage events rather than old ones. There is no try/catch around the delete, so a DB error surfaces as an unhandled rejection with no recorded failure status.
  → *Validate at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); Wrap the db.delete call in a try/catch that rethrows with a descriptive message so the job framework records the failure correctly.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Any dashboard query fetching budgets per user (the primary read pattern) will do a full table scan as the table grows.
  → *Add a composite index in the table's index callback: index('idx_budgets_user_category').on(t.userId, t.categoryId). Also add index('idx_budgets_user').on(t.userId) if single-user lookups are common without a category filter.*
- **`shared/schema.ts` (billReminders, savingsGoals, linkedCards, portfolioHoldings, portfolioTransactions table definitions):** userId columns across all tables use varchar with no length argument, defaulting to unlimited length in Postgres. This is inconsistent with the referenced users.id and risks storing malformed IDs silently. If users.id is a fixed-length token (e.g. Clerk's 32-char ID), unbounded varchar provides no validation at the DB layer.
  → *Change all userId columns to varchar('user_id', { length: 32 }) (or whatever the auth provider's max ID length is) to match users.id exactly and catch oversized values at the constraint layer rather than at the application layer.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt):** All createdAt columns omit .notNull(). defaultNow() sets a server default but does not add a NOT NULL constraint in the generated DDL, meaning application code or a direct INSERT can store a NULL createdAt, breaking any ORDER BY createdAt queries silently.
  → *Append .notNull() to every createdAt column: timestamp('created_at').defaultNow().notNull(). Apply the same fix to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions.*
- **`shared/schema.ts` (simulatedStocks table definition — ticker column):** simulatedStocks.ticker has no unique index. Duplicate tickers can be inserted, which would break any lookup-by-ticker logic and corrupt portfolio display for school users comparing simulated assets.
  → *Replace the plain column definition with a uniqueIndex: add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table's index callback, or use .unique() directly on the column definition.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions — billReminders.nextDueDate index):** billReminders.nextDueDate is the primary filter column for upcoming-bill queries (WHERE next_due_date <= NOW()) but has no index. For a student user with many reminders this causes a sequential scan on every reminder check, and there is no composite index on (userId, nextDueDate) either.
  → *Add a composite index in the billReminders index callback: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). This covers both the per-user filter and the date range in a single index scan.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo. Because the params object reference changes each render, any downstream effect or callback that closes over it will re-run unnecessarily, and the queryKey array uses primitive values while the actual fetch uses the live params string — a subtle desync risk if params construction changes.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]);` and use `params.toString()` in the queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard and AuditLogPanel):** Neither query has isLoading or isError handling. If the /api/admin/observability or /api/admin/audit-log endpoints fail or are slow, the UI silently renders empty/undefined data with no skeleton, spinner, or error message — especially problematic for school-admin users who may misread missing badges as 'Disabled' rather than 'unavailable'.
  → *Destructure `isLoading` and `isError` from useQuery in both components. In ObservabilityCard, render a skeleton or 'Unavailable' state when isLoading or isError. In AuditLogPanel the isLoading is already destructured but isError is not checked — add: `{isError && <p className='text-red-400'>Failed to load audit log. <button onClick={refetch}>Retry</button></p>}`.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant (module level)):** TABS is defined at module scope as a plain array containing icon component references. This is fine for the constant itself, but if any tab-rendering map callback creates new handler functions inline (e.g. onClick={() => setTab(tab.id)}) directly in JSX without useCallback, every parent re-render recreates those handlers for all 13 tabs simultaneously, causing all tab buttons to re-render.
  → *Wrap the tab-click handler in useCallback: `const handleTabClick = useCallback((id: string) => setActiveTab(id), [])`. Pass it as a stable reference into the tab map. If tab items are rendered as a child component, add React.memo to that component.*
- **`client/src/pages/AdminDashboard.tsx` (params.set("limit", "200") in AuditLogPanel):** The audit log page size of 200 is hardcoded inline. For a financial-literacy app with audit compliance requirements, this limit controls how much data an admin can review at once and should be configurable. Additionally, fetching 200 rows unconditionally on every filter keystroke (no debounce visible on `action` field) causes excessive API calls.
  → *Move the limit to an environment-aware constant: `const AUDIT_LOG_PAGE_SIZE = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 200)`. Add debouncing to the action text input using a useDebounce hook (300ms) so the query only fires after the user stops typing, reducing redundant network requests.*

## 💡 Opportunities

### Centralise AI route auth guard — 🔥 High Impact
Currently auth enforcement for AI routes depends entirely on what each register function does internally. Adding a single `app.use(['/api/chat', '/api/image', '/api/audio'], requireAuth)` middleware before registration in routes.ts makes the security boundary explicit and prevents future routes from accidentally being added without auth.

### Lazy-load heavy domain routers on first request — ⚡ Medium Impact
The parallel `Promise.all` import block still executes at server startup, blocking the event loop during module evaluation. Using dynamic `import()` inside each route handler (with a module-level cache variable) would allow the HTTP server to start accepting connections faster, improving cold-start P99 on Railway.

### Add structured logging to Supabase init chain — 📌 Low Impact
The init chain only logs on error. Adding timed `console.info` (or Sentry breadcrumb) at each step completion would allow pinpointing which seed step is slow in production without attaching a debugger, reducing future incident investigation time significantly.

### Parallel OpenAI calls where extraction is independent of text-retrieval path — 📌 Low Impact
In the image branch, text extraction and the subsequent structured-extraction OpenAI call are fully sequential today. While the second call depends on the first, the paper metadata update (storage.updateExamPaper status to 'processing') and the idempotency delete could be issued in parallel with Promise.all before any AI call begins, saving one extra sequential DB round-trip per job invocation.

### Idempotency token on bulk question insert to prevent double-write on retry — ⚡ Medium Impact
The current idempotency strategy deletes all extracted questions at the start of each attempt. Under high concurrency or if two workers pick up the same job simultaneously, both could pass the delete check and then race to insert. Adding a unique constraint on (paperId, order) in the schema and using INSERT ... ON CONFLICT DO NOTHING would make the insert naturally idempotent without requiring the delete, eliminating the race window.

### Structured logging for all job handlers — ⚡ Medium Impact
Errors in several handlers (purge-ai-usage, perf-scan) propagate with no structured log entry before re-throw, making it hard to correlate failures with specific job IDs or payloads in production. Adding a thin logger wrapper (e.g., pino) that records jobId, handler name, attempt number, and error message at the catch site would dramatically reduce mean-time-to-diagnosis for production incidents without impacting throughput.

### Enforce currency as a validated enum instead of free-text — ⚡ Medium Impact
currency is defined as plain text with a string default of 'BSD' on six tables. A typo ('BDS', 'bsd') silently stores an invalid currency that breaks display formatting and aggregation. Switching to text('currency', { enum: ['BSD', 'TTD', 'JMD', 'BBD', 'USD'] }) enforces valid values at the DB level and removes the need for application-layer validation in every insert schema.

### Add a partial unique index on portfolioHoldings (userId, stockId) — 🔥 High Impact
Currently a user can have multiple rows for the same stock, requiring application code to merge quantities on every read. A unique constraint on (userId, stockId) would enforce a single holding row per asset, simplify update logic to a single UPSERT, and eliminate the risk of double-counting a position in portfolio value calculations.

### Extract repeated currency default into a shared column helper — 📌 Low Impact
The string 'BSD' and the identical text('currency').default('BSD').notNull() pattern appear in at least six table definitions. A single currencyColumn helper function would make a future default-currency change (e.g. adding TTD support) a one-line edit instead of a six-location find-and-replace that is easy to miss.

### Split AdminDashboard into per-tab route components — 🔥 High Impact
At 126k+ characters, this single file renders all 13 tabs in one component tree. Every state change (e.g. typing in audit log search) triggers reconciliation across Organizations, Schools, Students, etc. Splitting each tab into its own lazily-loaded route component with React.lazy/Suspense would reduce initial bundle size and scope re-renders to the active tab only. Expected impact: 40-60% reduction in unnecessary renders during tab-specific interactions.

### Parallelize independent admin data fetches with Promise.all — 🔥 High Impact
If the overview/dashboard tab fires multiple sequential useQuery calls (observability, stats, audit summary, job status), each waits for the previous to resolve. Consolidate them into a single /api/admin/dashboard-summary endpoint that runs its DB queries in parallel server-side with Promise.all, or use useQueries on the client to fire all requests simultaneously. Expected impact: dashboard load time reduced by (N-1) * avg_query_latency.

### Add request-scoped admin auth guard at component boundary — ⚡ Medium Impact
useAdminAuth is imported and presumably called, but if individual sub-components like ObservabilityCard and AuditLogPanel are ever extracted to separate files, they lose the parent's auth guard and will make unauthenticated API calls that rely solely on server-side enforcement. Add a shared AdminRoute wrapper component that checks auth state and redirects before rendering any child, ensuring auth is structurally enforced rather than by convention.
