## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget returned, a separate query is fired to calculate 'spent' amount per category. With 20 budgets this is 21 queries per dashboard load.

**Fix:** Replace the per-budget spent lookup with a single aggregated query using a GROUP BY on transactions joined to budgets: SELECT budget_id, SUM(amount) as spent FROM transactions WHERE user_id = $1 AND date BETWEEN ... GROUP BY budget_id. Merge results in JS with a Map keyed by budgetId before returning.

### `server/storage.ts`: enrollStudent / removeEnrollment
**Issue:** enrollStudent and any sibling writes that touch multiple tables (e.g. creating a classEnrollment and updating an enrollment count or firing a notification) are not wrapped in a database transaction. A mid-flight crash leaves referential state inconsistent.

**Fix:** Wrap all multi-table writes in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Pass the tx client to each insert/update inside the block so they share the same connection and atomicity guarantee.

### `server/storage.ts`: getClassProgressSummary
**Issue:** getClassProgressSummary accepts an opts.limit parameter but no guard enforces a maximum, allowing a caller to request unlimited rows and exhaust memory or saturate the DB connection pool.

**Fix:** Add: const safeLimit = Math.min(opts?.limit ?? 50, 200); and use safeLimit in the query. Return a { data, total, limit, offset } envelope so callers can paginate correctly.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 115-117)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. A single user or bot can exhaust API quota and incur unbounded cost. For a school-aged audience this is especially risky.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 46) and sitemap.xml handler (~line 60)
**Issue:** The base URL is constructed directly from x-forwarded-host and host request headers with no validation or allowlist. An attacker can send a crafted Host header and get a poisoned Sitemap directive injected into robots.txt or an arbitrary URL embedded in the sitemap, enabling cache-poisoning attacks.

**Fix:** Read the canonical base URL from a trusted environment variable (e.g. process.env.PUBLIC_BASE_URL) and fall back to it unconditionally: const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com'; Remove all header-derived base URL construction from both handlers.

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** portfolioHoldings has no index on (userId, stockId), yet every portfolio view will query by userId and every holding update will look up by both. A unique constraint is also missing, allowing duplicate holdings rows for the same user+stock pair, corrupting aggregated quantity calculations.

**Fix:** Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) in the table factory. This enforces one holding row per user per stock and provides the required query performance.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The return values of both setInterval calls are discarded. On SIGTERM/SIGINT the intervals are never cleared, so the process either hangs waiting for an in-flight DB query or the scheduler fires after the DB connection is closed, causing unhandled rejections.

**Fix:** Capture each handle: `const weeklyDigestInterval = setInterval(...); const purgeInterval = setInterval(...)`. In the graceful shutdown handler call `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before closing the HTTP server and DB pool.

---

### `server/index.ts`: capturedJsonResponse in the request-logging middleware
**Issue:** The logging middleware serializes the entire JSON response body with JSON.stringify and appends it to every /api log line. For endpoints that return lists of records (quiz history, usage stats) this logs potentially large PII-containing payloads to stdout in plaintext, and holds the full response body in memory until the finish event.

**Fix:** Either remove response-body logging entirely for production (check NODE_ENV), or whitelist a small set of safe scalar fields (e.g. `{ status, id }`) and truncate: `logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 200)}`;`. Never log full response bodies containing user data.

---

### `client/src/pages/AdminDashboard.tsx`: sendMessage – catch block
**Issue:** The catch block silently swallows the original error. For a school-facing admin tool any API failure (auth expiry, 500, network drop) is logged nowhere, making production debugging impossible and obscuring potential security events like 401s from an expired admin session.

**Fix:** Log the error and distinguish auth failures: `catch (err) { console.error('[AdminHelpChat] sendMessage failed', err); const isAuth = err instanceof Response && err.status === 401; setMessages(prev => [...prev, { id: crypto.randomUUID(), role: 'assistant', content: isAuth ? 'Session expired. Please log in again.' : 'Sorry, something went wrong.' }]); if (isAuth) redirectToLogin(); }`

## 🟡 Warnings (14)

- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices is called on an interval (visible in surrounding server bootstrap code) but there is no evidence the interval handle is stored or cleared on server shutdown, causing a memory/handle leak in long-running or test processes.
  → *Store the return value: const priceInterval = setInterval(...). Export a cleanup function or attach it to a shutdown hook: process.on('SIGTERM', () => clearInterval(priceInterval)). In test environments, call cleanup() in afterAll.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard joins to users to retrieve userName and avatar, then likely iterates results to shape the response — exposing whatever columns the user row contains to the mapping layer. If a future column (e.g. passwordHash, email) is added to users it will silently be available in the leaderboard object and could be accidentally serialised into API responses.
  → *Use an explicit column selection in the query: { userId: users.id, userName: users.name, avatar: users.avatar } rather than selecting the full user row. This creates a compile-time contract that breaks if sensitive columns are accidentally included.*
- **`server/routes.ts` (Supabase init chain (~line 120-128)):** The seven-step sequential Supabase init chain (.then().then()...) runs serially and a failure in any step skips all subsequent seeds with only a single catch log. Steps like seedStaticModules, seedRegionalContent, and seedGameContent are independent and the app may silently start with missing seed data.
  → *Run independent seed steps in parallel and capture per-step errors: await Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] seed step', i, r.reason))); Keep initSupabaseTables → trimOrgNamesInSupabase → seedFinancialAcademyLesson → initStaticContentTables sequential only where there is a real schema dependency.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 101)):** There is no authentication or authorization check before serving objects from object storage. Any unauthenticated request with a guessed filePath can download any stored file, including potentially sensitive org logos or user-uploaded content.
  → *Determine which objects are truly public (e.g. only paths under logos/ or videos/) and add an explicit allowlist check: if (!/^(logos|videos)\//.test(filePath)) return res.status(403).json({ error: 'Forbidden' }); For other paths require a valid session before calling objectStorage.searchPublicObject.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index, yet budget lookups are always filtered by userId and frequently joined to categories. Full table scans will degrade as user count grows.
  → *Add a composite index in the budgets table factory function: index('idx_budgets_user_category').on(t.userId, t.categoryId). Also add index('idx_budgets_user').on(t.userId) if single-column lookups are common.*
- **`shared/schema.ts` (userId varchar columns across all tables (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** varchar('user_id') is declared without a length parameter, which in PostgreSQL defaults to unlimited. This is fine for storage but inconsistent with the referenced users.id column. If users.id ever has a defined length, silent mismatches occur. More critically, there is no check that the varchar type aligns with the auth provider's ID format (UUID vs integer string), making cross-table joins fragile.
  → *Determine the auth provider's ID format (e.g., UUID = 36 chars). Change all userId columns to varchar('user_id', { length: 36 }) and ensure users.id uses the same declaration. Add a database-level CHECK constraint or use the uuid type from drizzle-orm/pg-core if IDs are UUIDs.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt):** All audit timestamp columns use .defaultNow() but omit .notNull(). Application code can explicitly insert NULL into these columns, breaking any reporting query that relies on createdAt being present, and violating the implicit contract that every record has a creation timestamp.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g., createdAt: timestamp('created_at').defaultNow().notNull(). This is a one-line change per column and prevents NULL audit timestamps at the database level.*
- **`shared/schema.ts` (simulatedStocks table definition — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text rather than timestamp, even though the table already has a priceUpdatedAt timestamp column serving an identical purpose. Storing a date as text prevents range queries, ordering by date, and index use on that column. The duplicate column also creates confusion about which field is authoritative.
  → *Remove lastPriceUpdateDate entirely and use priceUpdatedAt (timestamp) as the single source of truth for price recency. If a date-only value is needed for display, derive it in application code from priceUpdatedAt. Add a migration to drop the column and update any application references.*
- **`server/index.ts` (seedDatabase / probeDatabase / registerRoutes call-site in main()):** Startup awaits (probeDatabase, seedDatabase, registerRoutes, startJobWorker, registerJobHandlers) run sequentially before httpServer.listen(), blocking the event loop and delaying readiness. probeDatabase and seedDatabase are likely independent and could run in parallel.
  → *Use `await Promise.all([probeDatabase(), seedDatabase()])` for the independent initialization steps. Only then proceed with registerRoutes and startJobWorker. This cuts startup wall-time by the slower of the two operations instead of their sum.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The scheduler computes Sunday at 19:00 using `new Date()` which inherits the server's local timezone. On a UTC-based host (Replit containers run UTC) this fires at 19:00 UTC, not 19:00 Caribbean time (UTC-4/UTC-5). The comment says '7pm' implying local intent.
  → *Accept an explicit timezone offset or use a named timezone. Replace the bare `setHours(19,0,0,0)` with a calculation that accounts for the Caribbean offset, e.g. construct the target in UTC as `19 + 4 = 23:00 UTC` for AST, or add an `DIGEST_TIMEZONE_OFFSET_HOURS` env var defaulting to -4 and apply it before comparison.*
- **`server/index.ts` (setInterval for maybeRunWeeklyDigest and maybeRunAiUsagePurge):** Both scheduler callbacks are fire-and-forget: `setInterval(() => maybeRunWeeklyDigest(), ...)`. If maybeRunWeeklyDigest throws synchronously or rejects after its internal try/catch boundary, the rejection is silently swallowed and no telemetry is captured.
  → *Wrap each call: `setInterval(() => { maybeRunWeeklyDigest().catch(e => captureError(e)); }, interval)`. This ensures Sentry receives any unexpected scheduler failures that escape the internal catch block.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is recreated on every render and calls scrollIntoView unconditionally whenever the messages container re-renders, not just when a new message arrives. This causes jarring scroll jumps if the user manually scrolls up to read history while a typing indicator is shown.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length, loading]);` and render `<div ref={bottomRef} />`. This fires only when message count or loading state changes.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages.map key={i}):** Chat messages are keyed by array index. When the assistant reply is appended or when the loading placeholder is inserted/removed, React uses the same index for different content, causing mismatched reconciliation and potentially flickering or lost input focus.
  → *Assign a stable id when each message is created: `type ChatMessage = { id: string; role: 'user'|'assistant'; content: string }`. Use `crypto.randomUUID()` or a counter at push time and key on `m.id` in the map.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage, full messages array sent each call):** Every call to sendMessage POSTs the entire conversation history to /api/admin/help-chat. For a long session this payload grows unboundedly. On a school network this risks hitting server body-size limits and leaks the full conversation (including any accidentally typed sensitive data) on every single turn.
  → *Slice to a rolling window before sending: `const CONTEXT_LIMIT = 20; const payload = next.slice(-CONTEXT_LIMIT);` and send `payload` instead of `next`. Add a server-side body size limit guard as well (e.g. express `json({ limit: '32kb' })`).*

- **`server/routes.ts` (duplicate health check endpoints (~line 33 and ~line 79)):** Two health check routes exist: GET /api/health (simple timestamp) and GET /healthz (DB ping). Railway is documented as using /api/health but the more useful DB-level check is at /healthz. ## 💡 Opportunities

### Parallelise independent dashboard queries with Promise.all: 🔥 High Impact
getDashboardStats sequentially awaits total income, total expenses, savings goals, and budget summaries. These are independent reads and can be fired concurrently. Switching to Promise.all([incomeQuery, expenseQuery, goalsQuery, budgetQuery]) would cut dashboard latency by roughly 60-70% under normal DB load.

### Move virtual balance seed amount to environment config: ⚡ Medium Impact
The starting virtual balance for investment simulation (likely hardcoded as 10000 or similar) should be read from process.env.VIRTUAL_BALANCE_SEED_AMOUNT with a documented default. This allows per-environment tuning and sponsor-driven adjustments without code changes.

### Introduce a repository-layer error boundary with structured logging: 🔥 High Impact
No consistent try/catch or error-wrapping exists around async DB calls in storage methods. Adding a thin wrapper (e.g. a withStorage(fn) helper) that catches, logs with a correlation ID, and rethrows a typed StorageError prevents raw Drizzle/Postgres error messages — which may include table names or column data — from propagating to API responses visible to school-aged users.

### Parallel domain route registration: ⚡ Medium Impact
The six registerX(app) calls after the parallel import are still awaited sequentially. If any registration does async work (DB introspection, schema checks) this adds unnecessary serial latency at startup. Wrap them in Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) assuming routes don't depend on each other's side effects, which would reduce cold-start time proportionally to the slowest individual registration.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Currently init failures only log a single concatenated error. Adding per-step timing and structured JSON logs (step name, duration, success/failure) would make Railway log drain and Sentry much more useful for diagnosing partial degradation on deploy, with zero runtime cost.

### Extract header-trust logic into a shared utility: 📌 Low Impact
The proto/host resolution pattern is duplicated verbatim in both robots.txt and sitemap.xml handlers. Extracting a single getCanonicalBase(req) utility (or better, an env-var constant) removes the duplication and makes the future host-header fix a one-line change instead of two.

### Partial index on active bill reminders: ⚡ Medium Impact
billReminders is queried by nextDueDate to surface upcoming bills. Adding a partial index — index('idx_bill_reminders_upcoming').on(t.userId, t.nextDueDate) WHERE status IS NOT 'paid' (if a status column exists) or simply the composite index now — will make due-date dashboard queries O(log n) instead of sequential scans, which matters as reminder rows accumulate per user.

### Consolidate duplicate currency default 'BSD' into a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, portfolioHoldings). Extract DEFAULT_CURRENCY = 'BSD' into a shared config constant and reference it in each .default() call. This ensures a single change point if the app expands to multi-currency and prevents typo divergence across tables.

### Add index on documentUploads.status for processing queue queries: 🔥 High Impact
Background workers polling for documents in 'processing' status will perform a full table scan without an index on status. Because status has low cardinality, a partial index — index('idx_doc_uploads_processing').on(t.userId, t.createdAt) filtered to status = 'processing' — is more efficient than a full column index and directly speeds up the upload processing pipeline.

### Structured startup health-check endpoint: 🔥 High Impact
Add a /healthz route registered before registerRoutes() that returns 200 only after probeDatabase resolves. This lets a load balancer or Replit health check distinguish 'process started' from 'DB connected and ready', preventing traffic before the app is truly available.

### Centralise all magic interval/timing constants in environment config: ⚡ Medium Impact
The scheduler intervals (e.g. the weekly digest polling period, purge polling period) appear as inline numeric literals. Moving them to named env vars with documented defaults makes tuning in production safe without a redeploy and makes the intent visible in config rather than buried in code.

### Replace res.json monkey-patch with a response-interceptor approach: ⚡ Medium Impact
The current rawBody/capturedJsonResponse approach overrides res.json on every request, creating a per-request closure and a dangling reference until the finish event fires. Using a lightweight express middleware that reads Content-Length or uses a streaming transform would eliminate the per-request memory retention and the prototype mutation.

### Parallelize independent admin data fetches with Promise.all: 🔥 High Impact
The dashboard likely fires multiple sequential useQuery calls (users, schools, teachers, analytics) that are independent. Introducing a single aggregated /api/admin/dashboard-summary endpoint or using Promise.all in a custom hook would cut initial load time by the sum of all sequential round-trips, which on Caribbean mobile networks (high latency) is meaningful.

### Extract chat panel into a standalone lazy-loaded module: ⚡ Medium Impact
AdminHelpChat and its Recharts/Lucide imports are bundled into the main AdminDashboard chunk. Moving it to React.lazy + Suspense would shrink the initial parse cost for all admin page loads where the chat is never opened, improving Time-to-Interactive on low-end school devices.

### Add a rate-limit guard on the help-chat send button: ⚡ Medium Impact
Currently a fast user (or a script) can spam POST /api/admin/help-chat. Adding a client-side debounce (300 ms) and a per-session request counter that disables the input after N requests per minute protects both the AI API quota and the server from unintentional DoS from students who gain admin access.
