## 🔴 Critical (4)

### `server/routes.ts`: /public-objects/:filePath(*) handler (~line 93)
**Issue:** The filePath wildcard parameter is passed directly to objectStorage.searchPublicObject without any sanitisation. A request to /public-objects/../../private/secrets could traverse into non-public storage buckets depending on how searchPublicObject resolves paths, constituting a path traversal vulnerability.

**Fix:** Sanitise the path before use: const filePath = req.params.filePath.replace(/\.\./g, '').replace(/^\/+/, ''); then assert it matches a safe allowlist pattern such as /^[\w\-./]+$/ and return 400 if it does not. Also consider restricting to known prefixes like logos/ or videos/.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~lines 38-82)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without validation. An attacker can send a forged Host header, causing the Sitemap directive and all sitemap <loc> values to point to an attacker-controlled domain. If the sitemap is cached by a CDN or indexed by a crawler this constitutes host header injection.

**Fix:** Resolve the canonical base URL from a single environment variable at startup: const BASE_URL = process.env.CANONICAL_BASE_URL; and use that in both handlers. Reject or ignore the request-level host header entirely for URL construction. Add a startup assertion that CANONICAL_BASE_URL is set.

---

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded exam paper file is stored directly in the job payload (job table row). A single PDF or image can easily be 5–20 MB encoded, bloating the jobs table, slowing queue polling queries, and risking OOM when multiple jobs are dequeued concurrently.

**Fix:** Upload the file to object storage at enqueue time and store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / top-level admin route
**Issue:** The component imports and calls `useAdminAuth` but the visible code does not show the auth state being checked before rendering mutation-triggering UI (delete, create, edit actions). If `useAdminAuth` only redirects asynchronously, there is a render window where unauthenticated users can see and interact with admin controls before the redirect fires.

**Fix:** Immediately after calling `useAdminAuth`, destructure an `isAuthenticated` or `isLoading` flag and return null or a loading spinner early: `if (!isAuthenticated || isLoading) return <FullPageSpinner />;`. All admin queries and mutations must be inside this guard.

## 🟡 Warnings (14)

- **`server/routes.ts` (Supabase init chain (~lines 116-124)):** The sequential .then() chain means a failure in seedFinancialAcademyLesson halts all subsequent seeds silently (only the final .catch logs anything). Individual step failures are not distinguishable and the chain does not retry or report which step failed, making production debugging difficult.
  → *Replace the chain with an explicit async IIFE with per-step try/catch: (async () => { for (const [name, fn] of [["initTables", initSupabaseTables], ["trimOrgs", trimOrgNamesInSupabase], ...]) { try { await fn(); } catch (e) { console.error(`[Supabase] ${name} failed:`, e); } } })(). This isolates failures, logs which step failed, and continues the remaining seeds.*
- **`server/routes.ts` (registerChatRoutes, registerImageRoutes, registerAudioRoutes (~lines 112-114)):** The OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. For a school-aged user base this creates a risk of runaway API costs from a single user or scripted abuse, and the existing authLimiter/strictAuthLimiter is only applied to auth endpoints.
  → *Create a dedicated aiLimiter (e.g. 20 requests per user per minute using express-rate-limit with a keyGenerator based on req.user?.id ?? req.ip) and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the registerXRoutes calls.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with individual awaits. For a 60-question paper this is 60 serial round-trips to the database, adding seconds of unnecessary latency and holding the job worker busy far longer than needed.
  → *Replace the for loop with a single bulk insert: filter valid questions first into an array, then call db.insert(extractedQuestions).values(validRows) once. If storage.createExtractedQuestion must be used, collect promises and await Promise.all(promises), but prefer the single db.insert for atomicity.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (model and max_completion_tokens)):** Model name 'gpt-4o' and token limits (4096) are hardcoded in two separate places. Changing models for cost or capability reasons requires a code deploy, and the values are inconsistent between the vision call and extraction call, making behaviour hard to reason about.
  → *Add OPENAI_EXTRACTION_MODEL (default 'gpt-4o') and OPENAI_EXTRACTION_MAX_TOKENS (default 4096) to environment config and read them via process.env at call sites. Centralise into a single config object imported by jobHandlers.ts.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** The magic number 30000 is a hardcoded character truncation limit for PDF text with no comment explaining its origin (likely a rough token budget). It is silently applied, so a paper can be truncated without any warning logged or surfaced to the user, causing quietly incomplete question extraction.
  → *Move the limit to a named constant PDF_TEXT_CHAR_LIMIT = parseInt(process.env.PDF_TEXT_CHAR_LIMIT ?? '30000', 10). After slicing, if pdfData.text.length > PDF_TEXT_CHAR_LIMIT log a warning including paperId and original length so operators can detect truncation in production.*
- **`server/jobHandlers.ts` (admin-csv-export handler – let data (inferred from truncated section)):** The variable 'data' is declared with let and no type annotation before a conditional block that populates it, a pattern that typically means the entire result set is materialised in memory as a JS array before being serialised to CSV. For large tables (transactions, users) this is an unbounded memory allocation inside a job worker process.
  → *Stream query results using a cursor or paginated batches (e.g. drizzle's .limit/.offset loop or a raw pg cursor), writing each batch to an object storage upload stream rather than accumulating in memory. If the dataset is guaranteed small, add an explicit COUNT guard and throw if rows exceed a safe limit (e.g. 100,000) before fetching.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, yet every budget query will filter by userId. At scale this becomes a full table scan per user request.
  → *Add an index inside the table's index callback: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget rows per period.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate has no index. The most common query pattern for reminders is fetching upcoming bills by userId and nextDueDate range, which will be a sequential scan without it.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders table definition.*
- **`shared/schema.ts` (transactions table, createdAt column):** createdAt uses .defaultNow() but is missing .notNull(), so it accepts explicit NULL inserts. An audit/timestamp column should never be nullable; a NULL createdAt breaks sorting and audit trails.
  → *Change to createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same fix to budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (simulatedStocks table, ticker column):** ticker has no unique index. Duplicate tickers would cause data integrity issues in portfolio lookups and price-update jobs that key on ticker symbol.
  → *Replace the plain text definition with: ticker: text('ticker').notNull() and add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to the table's index callback.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` creates a brand-new function on every render. React calls the old ref with null then the new ref with the element on every re-render, causing a redundant scrollIntoView on every message list re-paint, not just when a new message is added.
  → *Replace with a useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and use `ref={bottomRef}` on the div. This fires only when messages or loading changes.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function inside AdminHelpChat):** apiRequest response is consumed with `res.json()` but there is no check for `res.ok` before parsing. A 4xx/5xx HTTP response will be parsed as JSON silently, and if the server returns an error body that lacks `data.reply`, the assistant message is simply never appended — swallowing the error without user feedback.
  → *After `const res = await apiRequest(...)` add `if (!res.ok) throw new Error(await res.text());` before `res.json()`. The existing catch block will then surface the error message to the user correctly.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple parallel useQuery calls for dashboard stats):** Dashboard data (users, schools, lessons, leaderboard, etc.) appears to be fetched in separate sequential useQuery hooks that each fire independently. While React Query runs them concurrently at the network level, any waterfall dependencies (e.g. fetching district list then school list based on result) will stall. More critically, there is no unified loading/error state shown to the user — if any single query fails, the dashboard renders partial data silently.
  → *For truly independent queries keep useQuery but add a combined error boundary: derive `const isAnyError = [q1, q2, q3].some(q => q.isError)` and render an error banner. For dependent queries use `enabled: !!parentData` and show a skeleton until all critical queries resolve. Consider a single `/api/admin/dashboard-summary` endpoint to reduce round-trips.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline JSX functions / missing useCallback on mutation handlers):** Mutation handler functions (delete user, save lesson, toggle publish, etc.) are defined as arrow functions directly inside JSX props such as `onClick={() => deleteMutation.mutate(id)}`. In a list of 50+ users/lessons this creates a new function reference per item per render, causing every list row to re-render even when unrelated state changes (e.g. search input keystroke).
  → *Extract list-row components into separate named components (e.g. `UserRow`, `LessonRow`) and wrap their handlers with `useCallback`. Pass stable mutation objects as props. This confines re-renders to the changed row only and removes the inline function allocation from the parent render cycle.*

## 💡 Opportunities

### Parallel Supabase seed execution where order permits: ⚡ Medium Impact
initSupabaseTables must run first, but seedStaticModules, seedRegionalContent, and seedGameContent appear independent. Running them with Promise.allSettled after initStaticContentTables would cut seed wall-clock time by roughly 60-70% on cold starts, improving server ready time on Railway deploys.

### Centralised CANONICAL_BASE_URL environment config: 🔥 High Impact
Both robots.txt and sitemap.xml independently recompute the base URL from headers. Extracting this to a single module-level constant derived from process.env.CANONICAL_BASE_URL at startup removes the host-injection surface and eliminates duplicated proto/host logic across two handlers.

### Object storage path allowlist via middleware: ⚡ Medium Impact
Introduce a small validateStoragePath middleware applied to /public-objects/:filePath(*) that enforces a regex allowlist and maps logical prefixes (logos, videos) to explicit storage prefixes. This makes it easy to add new asset types safely and keeps the route handler free of security logic, which is easier to audit and test.

### Idempotent bulk-upsert for question insertion: 🔥 High Impact
The current idempotency strategy deletes all questions for a paperId then re-inserts on every retry. Replace with INSERT ... ON CONFLICT (paperId, order) DO UPDATE so retries are safe without a destructive delete, eliminating a window where the paper has zero questions if the worker crashes mid-insert.

### Centralised OpenAI client with per-job timeout: ⚡ Medium Impact
Both openai.chat.completions.create calls have no timeout configured. A stalled OpenAI request can hold a job worker indefinitely. Pass an AbortSignal with a timeout (e.g. 90s) via the signal option on each call and catch AbortError to produce a retryable failure, preventing worker starvation under API degradation.

### Extract paper payload schema validation at enqueue boundary: ⚡ Medium Impact
There is no runtime validation of job.payload fields inside handlers; missing or malformed fields (e.g. undefined paperId) cause cryptic downstream errors. Add a zod schema per JobPayload variant and parse at the top of each handler, throwing a non-retryable error immediately if the schema fails so bad jobs do not exhaust their retry budget.

### Partial index on active savings goals: ⚡ Medium Impact
Add a partial index on savingsGoals(userId) WHERE deadline IS NOT NULL to speed up the common 'show me goals with deadlines' dashboard query without indexing rows for open-ended goals. Expected impact: reduced index size and faster filtered scans on the savings dashboard.

### Consolidate repeated currency default into a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract a DEFAULT_CURRENCY constant in a shared config file and reference it via sql template literal in each default. This prevents drift if the platform ever supports multi-currency and makes a single-point change for regional expansion.

### Add composite index on portfolioHoldings(userId, stockId): 🔥 High Impact
Portfolio value calculations join holdings to stocks filtered by userId. Without a composite index on (userId, stockId) every portfolio page load does a full holdings scan. A unique composite index here also enforces the business rule that a user holds one row per stock, replacing a potential application-layer dedup.

### Unified admin dashboard data endpoint: 🔥 High Impact
Replace 5-8 individual stat queries with a single GET /api/admin/summary endpoint that returns all counts and chart data in one response. This eliminates redundant query calls, reduces waterfall risk, and cuts dashboard load time by the number of saved round-trips (expected 300-600ms improvement on Caribbean mobile networks).

### Virtualise long admin lists: ⚡ Medium Impact
User, school, and lesson tables render all rows into the DOM. At 500+ records this causes jank and high memory use. Introduce @tanstack/react-virtual or react-window with fixed row heights to render only visible rows. Expected DOM node reduction: 80-95%, with measurable scroll frame-rate improvement.

### Move hardcoded API path strings to a shared constants module: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/admin/users', '/api/lessons' appear inline throughout the file. Extract them to client/src/lib/apiRoutes.ts so typos are caught at compile time, base URL changes require one edit, and future auth middleware can enumerate all admin routes programmatically.
