## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query is fired to compute `spent` (sum of transactions per category). With 20 budgets this is 21 queries per dashboard load.

**Fix:** Replace the per-budget loop with a single aggregated JOIN: `SELECT b.*, COALESCE(SUM(t.amount),0) AS spent FROM budgets b LEFT JOIN transactions t ON t.category_id = b.category_id AND t.user_id = b.user_id GROUP BY b.id` using Drizzle's `leftJoin` + `sql<number>` aggregate, then map the result set once.

### `server/storage.ts`: getClassProgressSummary
**Issue:** N+1 query inside enrollment loop: for each enrolled student the function issues separate queries for learning progress and game sessions, producing O(n) round-trips proportional to class size with no pagination guard on the outer enrollment fetch.

**Fix:** Fetch all student IDs first, then use `inArray(userLearningProgress.userId, studentIds)` and `inArray(gameSessions.userId, studentIds)` for two bulk queries. Apply the `limit`/`offset` opts to the enrollment query itself before fetching dependent data, not after.

### `server/storage.ts`: enrollStudent / createChallenge / completeModule
**Issue:** Multi-step writes (insert + XP update + badge check) execute without a database transaction. A process crash between steps leaves data inconsistent — e.g. XP credited but enrollment row missing.

**Fix:** Wrap each multi-step write in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass `tx` instead of `db` to every sub-query inside the block so all steps commit or roll back atomically.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 110-112)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. A school-aged user or bot can hammer these endpoints causing unbounded OpenAI spend and denial-of-service for other users.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

---

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap handler (~line 62)
**Issue:** The base URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header and inject an arbitrary URL into the Sitemap directive and all sitemap <loc> values, enabling cache-poisoning and open-redirect attacks.

**Fix:** Read the canonical base URL from an environment variable: `const base = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com';` and remove all header-derived base URL construction from both handlers.

---

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** simulatedStocks.ticker has no unique index. If two rows share the same ticker (e.g. 'BISX:CBL') portfolio join queries return duplicate rows, silently inflating portfolio valuations — a critical correctness bug for a financial app with school-aged users.

**Fix:** Add a unique index inside the table factory: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker). Also add a plain index on t.type since queries filtering by 'stock' vs 'bond' will be common.

---

### `server/index.ts`: seedDatabase / probeDatabase calls in startup (async IIFE or top-level await block)
**Issue:** probeDatabase and seedDatabase are awaited sequentially during startup before the HTTP server begins listening. If either call is slow or hangs (e.g. cold DB connection), the process holds the port closed and health checks fail, causing the process manager to restart the app in a crash loop.

**Fix:** Start listening immediately after registerRoutes(), then run probeDatabase and seedDatabase in the background with a hard timeout: wrap each in Promise.race([call(), new Promise((_,r)=>setTimeout(()=>r(new Error('timeout')),10_000))]). Log and exit(1) only on probe failure; allow seed failures to warn without killing the server.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by setInterval are never stored, so they cannot be cleared during graceful shutdown. When SIGTERM is received the intervals keep firing against a closing DB pool, causing unhandled promise rejections and potential duplicate job enqueues as the process drains.

**Fix:** Store both handles: const digestInterval = setInterval(...); const purgeInterval = setInterval(...); then in the SIGTERM/SIGINT handler call clearInterval(digestInterval) and clearInterval(purgeInterval) before closing the HTTP server and DB pool. Wrap the shutdown sequence in a single async handler guarded by a boolean 'shuttingDown' flag to prevent re-entry.

---

### `server/index.ts`: res.json monkey-patch inside the request-logging middleware
**Issue:** capturedJsonResponse stores the entire response body and JSON.stringify serializes it unconditionally into the log line for every /api route. For endpoints that return user financial records, quiz scores, or session tokens this writes sensitive data in plaintext to stdout/log aggregators, violating data-minimization expectations for a school-aged user base.

**Fix:** Remove body capture from the logging middleware entirely, or apply an allowlist: only log the body when a custom response header (e.g. X-Log-Body: 1) is set by the route, and strip any keys matching /token|secret|password|session/i before serialization. For audit purposes rely on structured Sentry breadcrumbs instead of raw log lines.

## 🟡 Warnings (10)

- **`server/storage.ts` (simulateMarketPrices):** Stock price simulation likely uses a hardcoded volatility constant and a fixed update interval (e.g. `setInterval(..., 30000)`) with no reference to an environment variable and no cleanup handle exported, causing a memory/timer leak when the module is re-imported in tests or hot-reload.
  → *Read interval and volatility from `process.env.MARKET_SIM_INTERVAL_MS` and `process.env.MARKET_SIM_VOLATILITY` with safe defaults. Store the return value of `setInterval` in a module-level variable and export a `stopMarketSim()` function that calls `clearInterval` so tests and graceful shutdown can clean it up.*
- **`server/storage.ts` (getLeaderboard):** No maximum cap on the `limit` parameter: a caller can pass `limit: 100000`, causing the DB to stream the entire users+gameSessions table into Node.js memory. On a school platform this can exhaust heap during peak exam periods.
  → *Add a guard at the top of the function: `const safeLimit = Math.min(opts?.limit ?? 20, 100);` and use `safeLimit` in the query. Mirror this pattern on every paginated method that accepts a caller-supplied limit.*
- **`server/routes.ts` (Supabase init chain (~line 115-122)):** The sequential `.then()` chain means a failure in any middle step (e.g. `seedFinancialAcademyLesson`) silently skips all subsequent steps because the single `.catch` only fires once. Seed failures leave the DB in a partial state with no per-step visibility.
  → *Run independent seeds in parallel and log per-step failures: `Promise.all([seedFinancialAcademyLesson(), initStaticContentTables(), seedStaticModules(), seedRegionalContent(), seedGameContent()].map(p => p.catch(e => console.error('[Supabase] Seed step failed:', e))));` after awaiting `initSupabaseTables()` and `trimOrgNamesInSupabase()` which must remain sequential.*
- **`server/routes.ts` (/healthz handler (~line 79)):** The /healthz endpoint dynamically imports `db` and `drizzle-orm` on every request. Dynamic imports are cached after first load, but the pattern is misleading and adds unnecessary overhead on the hot path for a health check polled every few seconds by Railway.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. The health check is not a circular dependency risk and the modules are used elsewhere in the server.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index despite being the primary filter columns for budget lookups. Every query fetching a user's budgets or budgets by category triggers a full table scan.
  → *Add a composite index in the budgets table factory: index('idx_budgets_user_category').on(t.userId, t.categoryId). Also add index('idx_budgets_user').on(t.userId) for single-user lookups.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** userId is declared as varchar with no length argument on every table. In PostgreSQL this is equivalent to text, but it silently diverges from the users.id column type defined in models/auth. If users.id is ever a fixed-length type (e.g. varchar(26) for ULID or uuid), referencing columns with unconstrained varchar will not enforce the same domain and can cause subtle join performance regressions.
  → *Decide on a canonical ID type (uuid or varchar(26) for ULID) and apply it consistently: userId: varchar('user_id', { length: 26 }).notNull().references(() => users.id) across all tables. This also enables the planner to use more efficient index scans.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** Audit/creation timestamps are declared without .notNull(). defaultNow() sets a server default but Drizzle's TypeScript type still marks these as nullable, meaning application code must handle null defensively everywhere, and a bug or raw SQL INSERT that omits the column produces a NULL audit timestamp with no database-level rejection.
  → *Add .notNull() after .defaultNow() on every audit timestamp: createdAt: timestamp('created_at').defaultNow().notNull(). The database default still fires automatically; NOT NULL simply prevents NULL from ever being stored.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on (userId, nextDueDate), which is the most natural query pattern for fetching upcoming bills for a user ordered by due date. Without it, the dashboard reminder widget scans the full table for every page load.
  → *Add an index in the table factory: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). Because this column drives ORDER BY in time-sorted queries, a composite index with userId first lets PostgreSQL satisfy both the WHERE and ORDER BY in a single index scan.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The day-of-week arithmetic uses (dow + 0) % 7 which is a no-op, making the function compute the most recent Sunday relative to UTC but execute the setHours(19,0,0,0) call in the server's local timezone. On a UTC host (Replit default) 7 PM local equals 7 PM UTC, which is fine, but the comment and intent imply Caribbean local time (UTC-4/-5). Digests will fire 4-5 hours early for the target audience.
  → *Replace the local Date arithmetic with an explicit UTC-offset calculation: compute the target timestamp as a fixed UTC hour (23:00 UTC = 7 PM AST UTC-4) and store that constant in an env var DIGEST_HOUR_UTC defaulting to 23. Remove the no-op (dow + 0) and document the chosen UTC anchor.*
- **`server/index.ts` (spawn call (Vite dev server child process)):** The child process spawned for the Vite dev server is not referenced in the graceful-shutdown handler. On SIGTERM the parent exits but the child Vite process becomes an orphan, holding the Vite HMR port open and preventing clean restarts on Replit.
  → *Store the return value: const viteProc = spawn(...); then in the shutdown handler call viteProc.kill('SIGTERM') before closing the HTTP server. Add a listener viteProc.on('error', ...) and viteProc.on('exit', ...) so crashes in the child are surfaced via captureError rather than silently dropped.*

## 💡 Opportunities

### Parallelize independent dashboard queries with Promise.all: 🔥 High Impact
getDashboardStats sequentially awaits total income, total expenses, savings goals, and budget summary — four independent queries. Wrapping them in `Promise.all([...])` would cut dashboard latency by roughly 60-70% on a typical 5 ms/query connection, which directly impacts the student home screen load time.

### Centralise pagination defaults and max-limit enforcement: ⚡ Medium Impact
At least six list methods (getTransactions, getLeaderboard, getClassProgressSummary, getPortfolioTransactions, getNotificationsByClass, getGameSessions) each handle limit/offset independently with no shared guard. Extract a `safePaginate(opts, maxLimit = 100)` utility returning `{ limit, offset }` and apply it uniformly to eliminate repeated boilerplate and close the unbounded-query risk in one place.

### Remove sensitive fields from storage return types before they reach route handlers: ⚡ Medium Impact
Teacher and user queries return full ORM rows including `passwordHash`. Defining a `sanitizeTeacher` / `sanitizeUser` projection (omitting hash fields) inside storage — rather than relying on each route to strip them — ensures credentials cannot leak through a forgotten route or a new endpoint added under time pressure.

### Parallel domain route registration: ⚡ Medium Impact
The six `await registerXRoutes(app)` calls after the parallel import block run sequentially. If any route registrar does async work (e.g. DB schema checks), this adds to cold-start time. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` — safe as long as route registrars don't depend on each other's side-effects, which they shouldn't.

### Structured logging for object storage errors: ⚡ Medium Impact
The `/public-objects/:filePath` error handler logs raw `err` objects via `console.error`. For a production app serving school users, replacing this with a structured logger (e.g. passing err to Sentry and logging `{ path: filePath, error: err.message }`) would make storage failures actionable in dashboards rather than buried in log streams.

### Extract base-URL resolution into a shared utility: 📌 Low Impact
The proto/host/base derivation logic is duplicated verbatim between robots.txt and sitemap.xml handlers. Once fixed to use an env var, centralise it as `getPublicBaseUrl()` in a `config.ts` module. This eliminates the risk of the two handlers diverging again and makes the config surface area explicit.

### Partition or archive transactions by date: 🔥 High Impact
transactions is the highest-volume table. Adding a PostgreSQL range partition on the date column (e.g. yearly) or a partial index on recent rows (WHERE date > NOW() - INTERVAL '90 days') would dramatically reduce scan costs for the dashboard summary queries that almost always filter to recent history.

### Replace magic currency string 'BSD' with a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions. If the app expands to support TTD or JMD, each definition must be found and changed manually with high risk of inconsistency. Extract DEFAULT_CURRENCY = 'BSD' into a shared constants file and reference it in all .default() calls.

### Add a unique constraint on portfolioHoldings (userId, stockId): 🔥 High Impact
Currently nothing prevents two rows with the same (userId, stockId) pair, which would cause double-counting of holdings. A uniqueIndex on (userId, stockId) enforces one holding row per user per stock and enables upsert-based buy logic (INSERT ... ON CONFLICT DO UPDATE) instead of fragile select-then-update patterns.

### Parallelize independent startup I/O with Promise.all: 🔥 High Impact
probeDatabase, seedDatabase, registerJobHandlers, and startJobWorker have no data dependency on each other. Running them concurrently with Promise.allSettled reduces cold-start latency by the sum of all sequential await times (potentially 2-5 s on a cold Replit container), making the server accept traffic faster.

### Extract scheduler configuration into validated env vars: ⚡ Medium Impact
The digest interval (hardcoded as a magic number in the setInterval call), purge retention days (AI_PURGE_OLDER_THAN_DAYS), and digest hour are scattered. Centralizing them in a single config module with zod validation at startup gives operators one place to tune scheduling without touching source, and produces a clear error message instead of silent fallback to defaults.

### Replace response-body monkey-patching with structured middleware: ⚡ Medium Impact
Monkey-patching res.json on every request adds a closure and a retained reference to the full response body for the lifetime of each request. Replacing it with a lightweight res.on('finish') that reads only res.statusCode, res.getHeader('content-length'), and an optional X-Request-ID header eliminates per-request heap allocation and removes the sensitive-data exposure surface in one change.
