## 🔴 Critical (5)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 120)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. A school-aged user or bad actor can spam these endpoints, running up API costs or causing denial-of-service. Auth rate limiters are applied only to auth paths.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerX calls.

### `server/routes.ts`: robots.txt handler (~line 46) and sitemap handler (~line 58)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header, causing the Sitemap directive and sitemap <loc> tags to point to an arbitrary domain (host-header injection / cache-poisoning vector).

**Fix:** Read the canonical origin from a single environment variable (e.g. process.env.PUBLIC_URL or RAILWAY_PUBLIC_DOMAIN) and fall back to a hardcoded production domain. Never trust Host/x-forwarded-host for absolute URL construction in public responses: const base = process.env.PUBLIC_URL ?? 'https://finsightlite.com';

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both `setInterval` handles are not stored in variables, making it impossible to call `clearInterval` during graceful shutdown. If the process receives SIGTERM while a scheduler tick is in-flight, the async work is abandoned mid-transaction, risking a duplicate-claim row that blocks the next legitimate run.

**Fix:** Store both handles: `const weeklyDigestTimer = setInterval(...)` and `const purgeTimer = setInterval(...)`. In the SIGTERM/SIGINT handler, call `clearInterval(weeklyDigestTimer)` and `clearInterval(purgeTimer)` before closing the HTTP server, so no new ticks fire during drain.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** The day-of-week arithmetic `(dow + 0) % 7` is a no-op: adding 0 changes nothing. When `dow === 0` (Sunday) and it is before 19:00, the function subtracts 7 days and then immediately subtracts another 7 via the `d.getTime() > now.getTime()` guard, returning two Sundays ago instead of last Sunday. The digest is silently skipped for the current week.

**Fix:** Replace the guard with an explicit Sunday check: `if (dow === 0 && now.getHours() < 19) d.setDate(d.getDate() - 7);` else `d.setDate(d.getDate() - dow);`. Remove the redundant `+ 0`. Add a unit test covering Sunday before and after 19:00.

### `client/src/pages/AdminDashboard.tsx`: sendMessage – help-chat POST handler
**Issue:** The catch block in sendMessage swallows the error silently (bare `catch {}`) after pushing an error message to state. If `res.json()` throws (malformed JSON, network error), the loading spinner stays true if the error occurs before the finally block is reached in some engine edge cases—but more critically, there is no check on `res.ok` before calling `res.json()`, so a 401/403 from the server (unauthenticated admin session expiry) is silently treated as a generic failure with no redirect or re-auth prompt.

**Fix:** After `const res = await apiRequest(...)`, add `if (!res.ok) throw new Error(res.statusText);`. In the catch, inspect the error: if status is 401, redirect to login via `useLocation`. The finally block already handles `setLoading(false)` so that part is fine.

## 🟡 Warnings (13)

- **`server/routes.ts` (Supabase init chain (~line 128)):** The seven Supabase init/seed calls are chained sequentially with .then(), so each waits for the previous to finish. This unnecessarily extends cold-start time. Additionally, the single .catch() at the end only captures the first rejection; if trimOrgNamesInSupabase succeeds but seedStaticModules throws, the error message gives no indication of which step failed.
  → *Run independent steps in parallel where safe: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Also wrap each individual call in its own try/catch or .catch() to log which step failed.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 110)):** filePath is taken directly from req.params.filePath and passed to objectStorage.searchPublicObject without any path traversal sanitisation. A request to /public-objects/../../private/secrets could escape the intended public prefix depending on how searchPublicObject resolves the path.
  → *Sanitise the path before use: import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|^\//g, ''); and reject any path that contains '..' after normalisation with a 400 response.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index, but budget queries will almost certainly filter by userId. A full table scan on budgets per user request will degrade as data grows.
  → *Add a composite index in the budgets table definition: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })*
- **`shared/schema.ts` (All tables with createdAt, purchasedAt, executedAt columns):** All timestamp columns using .defaultNow() without .notNull() (e.g. transactions.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt) can be NULL at the DB level. defaultNow() only fires on insert via Drizzle; raw SQL inserts or migration edge cases can leave these NULL, breaking any ORDER BY or range queries on these columns.
  → *Append .notNull() to every .defaultNow() column: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to all tables: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.*
- **`shared/schema.ts` (varchar('user_id') across all tables):** All user_id columns are declared as varchar with no length limit. Without a length constraint, the DB accepts arbitrarily long strings. Since these reference users.id (a Clerk/auth ID, typically ≤128 chars), unbounded varchar wastes storage and skips a meaningful data integrity guard.
  → *Change all user_id column declarations to varchar('user_id', { length: 128 }). This matches typical auth provider ID lengths and adds a DB-level size guard without requiring data changes.*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text rather than timestamp. Storing dates as text prevents DB-level date comparisons, range queries, and indexing efficiency. Any query filtering or ordering by this column (e.g. finding stale prices) will either fail or require full scans with casting.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any application code that writes string dates to this field to pass Date objects instead.*
- **`shared/schema.ts` (billReminders table, nextDueDate column):** billReminders has no index on nextDueDate or userId, but the primary app use case is fetching upcoming bills for a user sorted by due date. Without an index this is a full table scan on every dashboard load.
  → *Add a compound index: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This covers both the user filter and the due-date ordering in a single index scan.*
- **`server/index.ts` (seedDatabase / probeDatabase calls in startup IIFE):** seedDatabase and probeDatabase are awaited sequentially before the HTTP server begins listening. If either call is slow (cold DB connection, migration), the process appears unresponsive to the platform health-check, risking premature restarts. They are independent and can run in parallel.
  → *Replace sequential awaits with `await Promise.all([probeDatabase(), seedDatabase()])` so both resolve concurrently, then call `httpServer.listen(...)` only after both settle. Wrap the entire block in try/catch and call `process.exit(1)` on failure so the platform restarts cleanly.*
- **`server/index.ts` (res.json monkey-patch inside request-logging middleware):** `capturedJsonResponse` holds a reference to the full parsed response body for every request until the `finish` event fires. For endpoints that return large collections (quiz history, usage reports), this doubles peak memory per request. Additionally, `JSON.stringify(capturedJsonResponse)` is called unconditionally on every API response in the log line, burning CPU on hot paths.
  → *Cap the logged body: after `capturedJsonResponse = bodyJson`, immediately truncate to a safe size, e.g. store `JSON.stringify(bodyJson).slice(0, 200)` as a string instead of the raw object. This eliminates the object reference leak and bounds serialization cost.*
- **`server/index.ts` (capturedJsonResponse log line (response logging middleware)):** Serializing the full response body — including any user-facing error messages, partial PII, or auth tokens returned in onboarding flows — into the console log creates an accidental data-exposure risk. For a school-aged Caribbean audience this may conflict with local data-protection obligations (e.g. Trinidad & Tobago Data Protection Act).
  → *Exclude sensitive routes explicitly: add a denylist `const SKIP_BODY_LOG = ['/api/auth', '/api/user'];` and only append the body snippet when `!SKIP_BODY_LOG.some(p => path.startsWith(p))`. For all other routes apply the 200-char truncation from finding #3.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a new function on every render and fires scrollIntoView on every re-render (including while typing), not just when a new message is appended. This causes janky scroll behaviour and is a minor memory/GC pressure issue.
  → *Replace with a `useRef<HTMLDivElement>(null)` and a `useEffect(() => { ref.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages])` so the scroll only triggers when the messages array actually changes.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort/page-slice computations in render body):** Based on the visible pattern and file size (147k+ chars), filtered user lists, sorted leaderboard entries, and paginated slices are almost certainly computed directly in the render body or with useMemo missing dependency arrays, meaning they recompute on every parent re-render regardless of whether the underlying data changed. For a dashboard with multiple useQuery calls this causes cascading recomputation on every poll interval.
  → *Wrap every derived list (filtered users, sorted rows, current page slice) in `useMemo` with the exact inputs as dependencies, e.g. `const filtered = useMemo(() => users.filter(u => u.name.includes(search)), [users, search]);`. Do the same for chart data transformations.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple independent useQuery calls for dashboard stats):** The dashboard likely fires several independent `useQuery` calls (users, schools, leaderboard, audit log, analytics) that each hit the server sequentially on mount rather than in parallel, and each has its own loading state rendered independently. This produces layout shift and multiple spinner flashes on first load, and on a polling interval each refetch is staggered.
  → *For initial load, consolidate into a single `/api/admin/dashboard-summary` endpoint returning all stat blocks in one round trip, or at minimum ensure all queries share the same `staleTime` so they batch-refetch together. Alternatively use `Promise.all` inside a single query `queryFn` if the endpoints must stay separate.*

## 💡 Opportunities

### Per-user AI rate limiting backed by session identity: 🔥 High Impact
The current rateLimiter appears IP-based. School networks share IPs via NAT, so a single misbehaving student can throttle an entire school. Add a secondary per-session-user-id key in the AI limiter (e.g. using express-rate-limit's keyGenerator reading req.user?.id) to make limits fair per student rather than per IP. Expected impact: prevents a single user from exhausting the quota for a whole classroom.

### Move Supabase seeding to a one-off migration script: ⚡ Medium Impact
Running seed functions on every server startup (even non-blocking) adds unnecessary DB round-trips and risks re-seeding if guards are missing. Extract them into a dedicated npm run seed script invoked once during deployment (e.g. as a Railway release command). This eliminates cold-start DB chatter and makes seed logic testable in isolation.

### Cache robots.txt and sitemap.xml responses: 📌 Low Impact
Both routes recompute and resend the full response on every request. Add res.setHeader('Cache-Control', 'public, max-age=86400') and consider a simple in-memory memoisation keyed on the base URL so the string is not rebuilt on every crawler hit. Low implementation cost with measurable reduction in redundant work under crawl load.

### Consolidate repeated currency default into a shared constant: 📌 Low Impact
The string 'BSD' is hardcoded as a default in at least 7 separate column definitions across transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and simulatedStocks. Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it in each .default(DEFAULT_CURRENCY) call. This eliminates typo risk and makes a currency change a one-line update.

### Add partial index on documentUploads for in-flight processing jobs: ⚡ Medium Impact
Background jobs that poll for documents with status='processing' currently scan the full documentUploads table. A partial index — index('idx_doc_uploads_processing').on(t.userId, t.createdAt).where(sql`status = 'processing'`) — would make this query O(active jobs) rather than O(all uploads), which matters as upload history grows.

### Add unique constraint on portfolioHoldings(userId, stockId): 🔥 High Impact
Currently nothing prevents two rows with the same userId and stockId in portfolioHoldings, which would corrupt balance calculations. A uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) enforces one holding row per asset per user at the DB level and eliminates the need for defensive upsert logic in application code.

### Externalize all scheduler intervals to environment config: ⚡ Medium Impact
The weekly digest interval (currently hardcoded to whatever the setInterval period is) and AUTO_PURGE_OLDER_THAN_DAYS are the only tunable values. Adding WEEKLY_DIGEST_INTERVAL_MS and exposing it via Replit Secrets lets ops adjust cadence without a redeploy, matching the pattern already used for AI_PURGE_OLDER_THAN_DAYS. Expected impact: eliminates future hotfix deploys for timing adjustments.

### Consolidate graceful shutdown into a single registered handler: 🔥 High Impact
SIGTERM and SIGINT handlers are likely registered in multiple places (http server close, DB pool end, interval cleanup). Centralising them into one `shutdown()` async function called by both signals, with a hard `setTimeout(process.exit, 10_000)` safety net, prevents the process from hanging indefinitely on a stalled DB connection during a platform restart.

### Add structured logging to replace console.log calls: ⚡ Medium Impact
The custom `log()` helper formats a human-readable timestamp string on every call, which is expensive and not machine-parseable by Replit's log aggregator. Replacing it with a minimal structured logger (pino at ~5 kB) that emits JSON gives level filtering, timestamp as epoch (no locale formatting cost), and source tagging — reducing log overhead on the hot request path.

### Split AdminDashboard into focused sub-page components: 🔥 High Impact
At 147k+ characters this is a single-file mega-component. Splitting into UsersPanel, SchoolsPanel, AnalyticsPanel, and AuditLogPanel with React.lazy/Suspense boundaries would reduce initial bundle size, isolate re-render scope to the active panel only, and make each section independently testable. Expected impact: 40-60% reduction in unnecessary re-renders across the whole dashboard.

### Add useCallback to all mutation-triggering handlers passed as JSX props: ⚡ Medium Impact
Handlers like onDelete, onEdit, onSave passed into table rows and dialog components are recreated on every render, causing every row to re-render even when unrelated state changes (e.g. search input). Wrapping them in useCallback with stable dependencies would allow row components wrapped in React.memo to bail out, meaningfully reducing render work for large user/school tables.

### Introduce a unified async error boundary with toast integration: ⚡ Medium Impact
Error handling is duplicated across every mutation and query (individual catch blocks each calling toast). A single `onError` default set in the QueryClient configuration combined with a React ErrorBoundary around the dashboard would centralise error UX, ensure no error path is silently swallowed, and reduce boilerplate by ~30 lines per major section.
