## 🔴 Critical (3)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. Any authenticated or unauthenticated caller can hammer these endpoints, generating unbounded OpenAI API costs and potential DoS for a school-aged user base.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

---

### `server/jobHandlers.ts` — runExtractPaper (payload parameter fileB64)
**Issue:** The full base64-encoded file (potentially several MB for a PDF or image) is stored directly in the job table row via job.payload.fileB64. This bloats the jobs table, slows queue polling queries, and risks hitting row-size limits in Postgres.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download it inside the handler with objectStorageClient. This keeps job rows small and separates concerns cleanly.

---

### `shared/schema.ts` — simulatedStocks table, ticker column
**Issue:** ticker has no unique index. Duplicate tickers can be inserted silently, which would cause portfolio calculations to sum across duplicates and return wrong balances — a serious correctness bug for a financial literacy app.

**Fix:** Add (t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to the pgTable call. Also add a NOT NULL constraint to annualReturnPct or provide a sensible default (e.g., '0.00'), since downstream return calculations will produce NULL if this field is absent.

## 🟡 Warnings (11)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 98)):** The seven Supabase init steps are chained sequentially with .then(), meaning each step waits for the previous one to fully complete. If any step is slow (e.g. seedRegionalContent), the total cold-start delay is additive. Additionally, steps that are independent (e.g. seedStaticModules, seedRegionalContent, seedGameContent) gain nothing from sequential ordering.
  → *Run independent seed steps in parallel: `await initSupabaseTables(); await Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()]); await initStaticContentTables(); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]);` — wrap in an async IIFE with .catch() to preserve non-blocking behavior.*
- **`server/routes.ts` (registerRoutes (entire function body)):** The exported registerRoutes async function has no top-level try/catch. If any awaited call inside it throws (e.g. setupAuth, dynamic imports, domain route registration), the returned Promise rejects silently unless the caller handles it. In a school app startup, this could cause the server to appear to start while routes are missing.
  → *Wrap the function body in try/catch: `try { ...existing body... } catch (err) { console.error('[registerRoutes] Fatal startup error:', err); process.exit(1); }` — or ensure the call site in server/index.ts has .catch(err => { ... process.exit(1) }).*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 83)):** The filePath parameter is passed directly to objectStorage.searchPublicObject() without any sanitization or validation. A path like ../../private/config could traverse outside the intended public prefix depending on the ObjectStorageService implementation, exposing private objects.
  → *Sanitize before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` — import path from 'node:path' at the top of the file.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this serialises 40 round-trips to the DB. If any single insert throws, the paper is left partially inserted; the delete-at-top-of-retry logic recovers on the next attempt but the failed run leaves an inconsistent state during that window.
  → *Batch all valid questions into a single db.insert(extractedQuestions).values(allRows) call wrapped in a transaction together with the updateExamPaper call. This is one round-trip, atomic, and eliminates the partial-insert window entirely.*
- **`server/jobHandlers.ts` (runExtractPaper — model and token constants):** The OpenAI model name ('gpt-4o') and max_completion_tokens (4096) are hardcoded in two separate openai.chat.completions.create calls. Changing the model for cost or policy reasons (common in a school-budget app) requires a code deploy rather than a config change.
  → *Read from environment variables: const OPENAI_MODEL = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o' and const OPENAI_MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10). Reference these constants in both calls. Add validation at startup so a misconfigured value fails fast.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch. If the DB delete throws (e.g. a transient connection error), the unhandled rejection propagates to the job runner. Depending on how registerJobHandler surfaces errors, this may crash the worker process rather than recording a retryable failure.
  → *Wrap the db.delete call in try/catch, log the error with enough context (cutoffDate, job id), and rethrow so the job runner can mark the job as failed and schedule a retry. Apply the same pattern to any handler that currently omits error handling.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — fileName parameter):** The fileName value is interpolated directly into the Content-Disposition header without sanitisation: `attachment; filename="${fileName}"`. A crafted fileName containing a double-quote or newline can break the header and, in some clients, enable header injection.
  → *Sanitise the value before setting the header: const safeFileName = fileName.replace(/["\r\n]/g, '_'); res.setHeader('Content-Disposition', `attachment; filename="${safeFileName}"`).*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet every budget query will filter by userId. A full table scan will occur on every dashboard load for every user.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget rows for the same category and period.*
- **`shared/schema.ts` (categories table, userId column):** The categories.userId foreign key has no index. Any join or filter from transactions or budgets back to a user-owned category requires a sequential scan of the categories table.
  → *Add (t) => ({ userIdx: index('idx_categories_user').on(t.userId) }) to the pgTable call. Because userId is nullable (system categories have null), a partial index WHERE user_id IS NOT NULL would be even more selective.*
- **`shared/schema.ts` (varchar('user_id') — all tables):** Every userId column is declared as varchar with no length argument. In PostgreSQL this is an unbounded varchar, identical to text, but the intent almost certainly maps to a fixed-length auth provider UID (e.g., 36-char UUID or Supabase/Clerk ID). Without a length constraint, a misconfigured upstream can write arbitrarily long strings and foreign key joins become slower.
  → *Change all userId columns to varchar('user_id', { length: 36 }) (or 255 if using Clerk/Auth0 opaque IDs). This enforces a contract at the DB level and allows PostgreSQL to use a more compact storage strategy.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, linkedCards.createdAt):** All createdAt columns use .defaultNow() but are not marked .notNull(). Drizzle will type these as Date | null in TypeScript, forcing null-checks everywhere they are used, and any direct SQL INSERT that omits the column with a non-triggering driver can store NULL, breaking audit trails.
  → *Append .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The DB default still fires on INSERT so no application code needs to change.*

## 💡 Opportunities

### Parallel domain-router registration — ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six individual awaits. If any registerXRoutes function does async work internally (DB calls, dynamic imports), they block each other unnecessarily. Refactor each registerXRoutes to accept the app and perform their own async init internally without needing top-level await, reducing cold-start time by the sum of their individual init latencies.

### Sitemap route caching — 📌 Low Impact
The sitemap XML is regenerated on every request by string interpolation. Add a Cache-Control header (`res.setHeader('Cache-Control', 'public, max-age=86400')`) and optionally memoize the XML string at module load time since the URL list is static. For a school app with SEO goals, this prevents redundant work on every crawler visit.

### Structured startup logging for Supabase init — ⚡ Medium Impact
The Supabase init chain only logs on error. Adding per-step timing logs (e.g. console.info('[Supabase] seedGameContent done in 340ms')) would make Railway deployment logs and post-deploy diagnostics significantly faster to triage, especially when one seed step is slow or silently skipped due to already-seeded data.

### Parallelise independent async setup work per job handler — ⚡ Medium Impact
Several handlers that fetch multiple independent resources (e.g. user record + module record + progress record before building an email) await them sequentially. Wrapping independent fetches in Promise.all would reduce handler latency by the sum of all-but-the-slowest round-trips, which matters for time-sensitive jobs like badge award emails.

### Centralise job payload schema validation with Zod at handler entry — ⚡ Medium Impact
Handlers destructure job.payload fields (paperId, olderThanDays, type, etc.) without runtime validation. A malformed enqueue call silently produces NaN/undefined values that cause confusing downstream errors. Adding a per-handler Zod parse at the top of each handler gives early, descriptive failures and documents the expected contract for each job type.

### Cap and validate extractedText length before sending to GPT — 📌 Low Impact
extractedText is sliced to 30,000 characters for PDFs but images go through the vision model with no length cap on the returned content before it is passed to the extraction prompt. A very verbose OCR response could silently exceed the context window and produce a truncated or empty JSON response. Add a post-OCR trim mirroring the PDF path and log a warning when the cap is hit so ops can tune the limit.

### Composite unique constraint on portfolioHoldings — 🔥 High Impact
portfolioHoldings has no unique constraint on (userId, stockId). Without it, a race condition or retry can create two rows for the same holding, causing quantity and average-price calculations to double-count. Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) and merge logic in the application layer. This would prevent silent data corruption in the investment simulation feature.

### Extract default currency 'BSD' to a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). If the app needs to support TTD or JMD, every table must be manually updated. Extract to a DEFAULT_CURRENCY constant in a shared config file and reference it in each .default() call to ensure consistency and single-point changes.

### Add index on billReminders.nextDueDate for reminder scheduling — ⚡ Medium Impact
The bill reminder feature will almost certainly run a scheduled job querying WHERE next_due_date <= NOW() AND user_id = ? to surface upcoming bills. Neither column has an index beyond the implicit primary key. Add index('idx_bill_reminders_due').on(t.nextDueDate) — or a composite on (userId, nextDueDate) — so the scheduler query is an index range scan rather than a full table scan as reminder volume grows.
