## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 115-117)
**Issue:** AI-heavy endpoints (chat, image, audio) have no rate limiting applied before route registration. Any unauthenticated or authenticated user can hammer OpenAI APIs causing runaway cost and denial-of-service, especially critical for a school-aged user base.

**Fix:** Add dedicated rate limiters before these registrations: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter is e.g. 10 requests/minute per IP. Define aiLimiter in rateLimiter.ts alongside existing limiters.

### `server/routes.ts`: robots.txt handler (~line 45) and sitemap.xml handler (~line 59)
**Issue:** Both handlers construct a base URL directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header causing the Sitemap directive or canonical URLs to point to an arbitrary domain (host header injection).

**Fix:** Read the canonical base URL from a validated environment variable: `const base = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com';` and ignore request headers entirely for URL construction. Never trust Host/x-forwarded-host for generating absolute URLs in responses.

### `server/index.ts`: seedDatabase / probeDatabase calls in main()
**Issue:** seedDatabase() and probeDatabase() are awaited sequentially during startup and are fully blocking — if either hangs (e.g. DB unreachable), the HTTP server never binds and no health-check endpoint is reachable to signal the failure to an orchestrator.

**Fix:** Bind the HTTP server first (httpServer.listen()), then run probeDatabase() and seedDatabase() with a hard timeout via Promise.race([op(), rejectAfter(10_000)]) so the process fails fast with a clear error rather than hanging silently. Register a /healthz route before registerRoutes() that returns 503 until probes pass.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are stored in no variables and are never cleared. There is no SIGTERM/SIGINT handler that stops them before process exit, meaning in-flight async callbacks can fire after the DB connection is torn down, causing unhandled rejections or corrupted partial writes during rolling deploys.

**Fix:** Capture both interval handles: const weeklyHandle = setInterval(...). Register a single graceful shutdown handler: process.on('SIGTERM', async () => { clearInterval(weeklyHandle); clearInterval(purgeHandle); await httpServer.close(); await db.$client.end(); process.exit(0); }). Mirror for SIGINT.

### `client/src/pages/AdminDashboard.tsx`: sendMessage – apiRequest POST /api/admin/help-chat
**Issue:** The full growing `messages` array is sent to the backend on every turn with no length cap or sanitisation. For a school-aged-user platform this means unbounded payload growth and potential prompt-injection via user-supplied content that is forwarded verbatim to an LLM endpoint.

**Fix:** Cap history before sending: `const history = next.slice(-20);` and send `history` instead of `next`. Add a server-side system prompt that instructs the model to refuse off-topic requests, and strip or encode any HTML/script characters from `text` before appending to state.

---

### `client/src/pages/AdminDashboard.tsx`: useMutation handlers for delete/update operations
**Issue:** Mutation handlers (delete user, delete school, publish lesson, etc.) call the API without verifying that the current admin session is still valid client-side before firing. If the session cookie expires mid-session, the UI will show a success toast while the server returns 401, because the catch block only logs or shows a generic error without checking response status.

**Fix:** In each mutation's `onError` callback, check `if (error?.status === 401) { redirect('/admin/login'); }`. Additionally wrap every `apiRequest` call inside mutations with explicit status checking: `const res = await apiRequest(...); if (!res.ok) throw new Error(await res.text());` so non-2xx responses are not silently swallowed.

## 🟡 Warnings (8)

- **`server/routes.ts` (Supabase init chain (~line 120-128)):** The sequential `.then()` chain means if `trimOrgNamesInSupabase()` rejects, `seedFinancialAcademyLesson()` through `seedGameContent()` are silently skipped with only a single catch log. The app starts with potentially missing seed data, causing downstream runtime failures with no per-step visibility.
  → *Run independent seed operations in parallel with individual error isolation: `await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] Seed step ${i} failed:`, r.reason)));`. Run `initSupabaseTables` and `trimOrgNamesInSupabase` first sequentially if they are true prerequisites, otherwise include them too.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100)):** The filePath parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without path traversal sanitization. A request to `/public-objects/../../private/secrets` could potentially escape the intended storage prefix depending on ObjectStorageService implementation.
  → *Sanitize the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\./g, '').replace(/^\//, '');` or use Node's `path.normalize` and assert the result doesn't start with `..`. Add a check that filePath matches an allowlisted pattern like `/^[\w\-./]+$/`.*
- **`server/index.ts` (res.json monkey-patch in the request logger middleware):** capturedJsonResponse holds a reference to the full serialized response body for every API request. For endpoints that return lists of transactions or AI responses, this can be several kilobytes per request held in the closure until 'finish' fires. More critically, any response containing tokens, session data, or PII is written in full to stdout logs, violating data-minimisation expectations for a school-aged user platform.
  → *Remove the res.json override entirely. Log only method, path, status code, and duration. If body logging is needed for debugging, gate it behind a LOG_RESPONSE_BODIES=true env flag and add an explicit allowlist of safe, non-sensitive routes.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes Sunday using (dow + 0) % 7 which is a no-op — the + 0 is dead code left from a refactor. More critically, all time arithmetic uses the server's local timezone via new Date() and setHours(), so scheduled fire time drifts by UTC offset, causing the digest to run at the wrong wall-clock time when the server is not in the Caribbean timezone (UTC-4).
  → *Delete the dead (dow + 0). Perform all schedule math in UTC or use a fixed offset: replace setHours(19,0,0,0) with explicit UTC arithmetic, or store the target as UTC-4 by subtracting 4*3600*1000 from the UTC midnight boundary. Add a TZ=America/Port_of_Spain env var to the deployment config as a belt-and-suspenders measure.*
- **`server/index.ts` (maybeRunAiUsagePurge / AUTO_PURGE_OLDER_THAN_DAYS interval):** The purge scheduler has no mutex/lock between the DB check for an existing run record and the enqueue call, and the interval period is hardcoded at 6 hours inside setInterval rather than derived from config. In a multi-replica deployment both replicas can race past the 'already ran today' check within the same 6-hour window before either inserts the run record.
  → *Apply the same atomic insert-on-conflict pattern already used in maybeRunWeeklyDigest: insert the run record first with onConflictDoNothing and only enqueue if a row was returned. Expose the interval as PURGE_CHECK_INTERVAL_MS env var with a 6h default so it can be tuned without a redeploy.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and calls scrollIntoView unconditionally each time any state changes (e.g. input keystrokes), causing jank and unnecessary DOM thrashing. It is not a real memory leak but it fires far more often than intended.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages change.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats – useQuery calls for users, schools, lessons, etc.):** Multiple independent useQuery calls that fetch dashboard aggregate data (users, schools, teachers, lessons) are issued sequentially via separate queries with no staleTime set, causing redundant network requests on every tab focus and every parent re-render that remounts the component.
  → *Add `staleTime: 60_000` and `gcTime: 300_000` to each dashboard stat query, or consolidate them into a single `/api/admin/dashboard-stats` endpoint fetched with one useQuery, reducing round-trips from N to 1 and preventing re-fetch storms.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline JSX arrow functions in table/list renders):** Row-level action handlers such as `onClick={() => handleDeleteUser(user.id)}` and `onClick={() => setEditingLesson(lesson)}` are defined as new function instances inside `.map()` callbacks. With potentially hundreds of rows (users, lessons, schools), this creates O(n) function allocations on every render, and any state change in the parent (e.g. search input) re-renders and recreates all of them.
  → *Use a data attribute pattern to avoid per-row closures: add `data-id={user.id}` to the button and define a single `useCallback`-wrapped handler that reads `e.currentTarget.dataset.id`. Alternatively wrap the row in a memoised sub-component `const UserRow = React.memo(...)` accepting stable `onDelete` and `onEdit` callbacks defined with `useCallback` at the parent level.*

## 💡 Opportunities

### Validate PUBLIC_BASE_URL at startup: 🔥 High Impact
Add a startup assertion that `process.env.PUBLIC_BASE_URL` is set and is a valid HTTPS URL. This eliminates the host-injection risk permanently and catches misconfigured deployments at boot rather than in production traffic.

### Authenticate public-objects route for private assets: ⚡ Medium Impact
The `/public-objects/:filePath(*)` route serves all storage objects with no auth check. If any non-public files (e.g., org logos pending approval) land in the same bucket, they are world-readable. Add a middleware that checks a signed URL token or session for any paths outside an explicit `public/` prefix.

### Cache robots.txt and sitemap.xml responses: 📌 Low Impact
Both routes recompute their content on every request. Add `res.setHeader('Cache-Control', 'public, max-age=86400')` to both handlers. For a school-facing app with potential crawler and student traffic spikes, this eliminates redundant computation and reduces TTFB for crawlers.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase(), seedDatabase(), registerJobHandlers(), and startJobWorker() are currently awaited sequentially. If they are independent (no output of one feeds the next), wrapping them in Promise.all([probeDatabase(), seedDatabase()]) followed by Promise.all([startJobWorker(), registerJobHandlers()]) could cut startup latency by 40-60% on a cold Replit container where DB connection setup is the dominant cost.

### Structured logging with redaction middleware: 🔥 High Impact
Replacing console.log/log() with a structured logger (pino with pino-std-serializers) and adding a redact option for fields like token, password, sessionId, userId removes the manual effort of auditing every log call for PII. For a platform serving minors under Caribbean data-protection obligations this also simplifies compliance evidence collection.

### Extract scheduler logic into a dedicated module: ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, and their interval registrations into a schedulers.ts module that exports start() and stop() functions would make graceful shutdown trivial to test in isolation and prevent future interval handles from being orphaned in index.ts as the scheduler count grows.

### Parallelise independent admin data fetches with Promise.all: 🔥 High Impact
Any place where the component or an initialisation effect awaits multiple independent endpoints sequentially (e.g. fetching user count, then school count, then lesson count) should be replaced with `const [users, schools, lessons] = await Promise.all([fetchUsers(), fetchSchools(), fetchLessons()])`. This can cut initial dashboard load time by 50-70% on Caribbean mobile networks with high latency.

### Extract AdminHelpChat into its own lazy-loaded module: ⚡ Medium Impact
The chat component including its Recharts and Lucide icon imports adds to the main AdminDashboard bundle. Moving it to a separate file and importing via `const AdminHelpChat = React.lazy(() => import('./AdminHelpChat'))` with a Suspense boundary will reduce the initial parse cost, which matters on low-end school devices common in the Caribbean market.

### Move hardcoded API route strings to a central constants file: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/admin/users', '/api/admin/schools' appear inline throughout the component. Centralising them in `src/lib/apiRoutes.ts` as typed constants eliminates typo-driven bugs, makes base-URL changes (e.g. staging vs production) a one-line edit, and enables TypeScript to catch missing routes at compile time.
