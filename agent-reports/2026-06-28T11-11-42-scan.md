## 🔴 Critical (11)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches each budget's category and calculates 'spent' with a separate query per budget row inside a loop, causing O(n) round-trips to the database for n budgets per user.

**Fix:** Replace the per-budget loop with a single JOIN query using Drizzle's leftJoin(categories, eq(budgets.categoryId, categories.id)) and a grouped SUM subquery or a single inArray fetch of all relevant transaction totals keyed by categoryId, then assemble the result in JS.

---

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard on leaderboard query: the optional 'limit' filter defaults to an unbounded result set when not supplied, meaning a query against gameSessions could return every row and join against users in memory.

**Fix:** Add a hardcoded maximum cap (e.g., const safeLimit = Math.min(filters?.limit ?? 50, 200)) and always pass .limit(safeLimit) to the Drizzle query so callers cannot accidentally or maliciously request unlimited rows.

---

### `server/storage.ts`: upsertPortfolioHolding / updateVirtualBalance (buy/sell flow caller)
**Issue:** Missing database transaction wrapping multi-step portfolio writes: upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance are called sequentially without a db.transaction() wrapper. A failure between steps leaves balance and holdings in an inconsistent state—a serious correctness bug for a financial app.

**Fix:** Wrap the three operations in a single db.transaction(async (tx) => { ... }) block in the route handler or introduce a dedicated executeTrade(userId, ...) storage method that performs all three mutations inside one atomic transaction, rolling back on any error.

---

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 110-112)
**Issue:** AI-heavy routes (chat, image, audio) are registered with no rate limiting middleware. On a school-aged user platform these endpoints call external paid APIs and are fully open to abuse — a single user or bot could exhaust OpenAI quota or rack up costs.

**Fix:** Create an `aiLimiter` (e.g. 20 req/min per IP via express-rate-limit) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

---

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap.xml handler (~line 62)
**Issue:** The base URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header to inject an arbitrary domain into the Sitemap directive and the sitemap <loc> URLs, enabling cache-poisoning or phishing via manipulated crawlable content.

**Fix:** Add a validated base URL to environment config: `BASE_URL=https://finsightlite.com`. In both handlers replace the dynamic header logic with `const base = process.env.BASE_URL ?? 'https://finsightlite.com';` and remove the x-forwarded-host/host reads entirely.

---

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Full file content is stored as base64 in the job payload (`fileB64`). For a multi-page PDF or high-res image this can easily be several MB per job row, bloating the jobs table, slowing queue polls, and risking OOM when many jobs deserialize concurrently.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. Inside `runExtractPaper`, download the file from object storage, process it, then delete it. This keeps job rows small (<1 KB) and makes retries cheaper.

---

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** simulatedStocks.ticker has no unique index. Duplicate tickers would silently corrupt portfolio calculations (P&L, holdings aggregation) and make stock lookups by ticker non-deterministic.

**Fix:** Replace the plain column declaration with a uniqueIndex in the table's index factory: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker_unique').on(t.ticker) }). If tickers must be scoped per region/type, make the unique index composite on (ticker, type) instead.

---

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** The day-of-week rollback logic uses `(dow + 0) % 7` which is a no-op addition of zero, making Sunday (dow=0) produce daysSince=0 and never roll back, so the scheduler fires on Saturday night instead of Sunday.

**Fix:** Change `(dow + 0) % 7` to `dow === 0 ? 0 : 7 - (7 - dow) % 7` or more simply compute days since last Sunday as `dow` directly: `const daysSince = dow;`. Then verify with a unit test covering Sunday=0, Monday=1, Saturday=6.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by setInterval are never stored, so there is no way to call clearInterval() during graceful shutdown. Both schedulers will keep firing after the HTTP server closes, potentially dispatching jobs against a shutting-down database connection and preventing the Node process from exiting cleanly.

**Fix:** Store handles: `const weeklyHandle = setInterval(...)` and `const purgeHandle = setInterval(...)`. In the SIGTERM/SIGINT shutdown handler call `clearInterval(weeklyHandle); clearInterval(purgeHandle);` before closing the HTTP server and draining the DB pool.

---

### `server/index.ts`: res.json monkey-patch in the request logging middleware
**Issue:** Every API response body is serialised to a string and written to stdout via `JSON.stringify(capturedJsonResponse)`. For a financial literacy app with school-aged users this will log balance data, quiz answers, session tokens, or any PII returned by any endpoint, violating basic data-minimisation principles and creating a compliance risk.

**Fix:** Remove the `capturedJsonResponse` capture entirely, or replace it with a strict allowlist that logs only non-sensitive scalar fields (e.g. `{ status, count }`). At minimum add a check: `if (process.env.NODE_ENV !== 'production') { logLine += ... }` so production never emits response bodies.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / top-level component mount
**Issue:** The component renders its full JSX tree before `useAdminAuth` has resolved the session check. If the hook is async (e.g., fetches /api/admin/me), any admin-only queries fired by useQuery will execute with unauthenticated state for the duration of the check, potentially leaking query results or causing 401 errors that pollute the query cache.

**Fix:** Return a loading spinner immediately while auth is resolving: `if (authLoading) return <FullPageSpinner />;` before any query-dependent JSX. Ensure all admin useQuery calls have `enabled: !!isAdmin` so they are gated on confirmed auth state.

## 🟡 Warnings (17)

- **`server/storage.ts` (getBillReminders):** N+1 query pattern: getBillReminders fetches the optional category for each reminder inside a loop with individual db.query calls rather than a single joined query, producing O(n) database round-trips.
  → *Rewrite using a single leftJoin(categories, eq(billReminders.categoryId, categories.id)) so the category data is co-fetched in one query, then destructure into { reminder, category } shape to match the return type.*
- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregates: getDashboardStats issues income total, expense total, transaction list, and budget queries as sequential awaits even though none depend on each other's result, adding unnecessary latency on every dashboard load.
  → *Replace sequential awaits with const [income, expenses, txns, budgets] = await Promise.all([incomeQuery, expenseQuery, txnQuery, budgetQuery]) so all four queries execute concurrently. Expected latency reduction: ~60-75% of current wall-clock time for this function.*
- **`server/routes.ts` (Supabase init chain (~line 115-123)):** The seven Supabase init steps are chained sequentially with .then() and share a single .catch() at the end. If any intermediate step (e.g. seedFinancialAcademyLesson) throws, all subsequent steps are silently skipped with only a single generic error log — no indication of which step failed or that downstream seeds were not run.
  → *Run independent seeds in parallel where safe and wrap each critical step individually: `await Promise.all([initSupabaseTables(), initStaticContentTables()]).catch(e => console.error('[Supabase] Table init failed', e)); await seedFinancialAcademyLesson().catch(e => console.error('[Supabase] Academy seed failed', e));` etc. This isolates failures and logs precisely which step broke.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every invocation. Dynamic import resolution on each health check call adds unnecessary overhead and can interfere with uptime monitors that poll frequently (e.g. every 10s on Railway).
  → *Move `import { db } from './db'; import { sql } from 'drizzle-orm';` to the top of the file as static imports alongside the other imports. The /healthz handler then uses the already-resolved module references.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The filePath parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without any sanitisation. A path like `../../private/config` or URL-encoded traversal sequences could potentially reach unintended storage paths depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `import path from 'path'; const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); if (safePath !== filePath) return res.status(400).json({ error: 'Invalid path' });` Then pass `safePath` to the storage service.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential `await` inside a for-loop. For a 40-question paper this is 40 serial round-trips to the DB. A previous-attempt cleanup (`db.delete`) already runs before this, so idempotency is handled, but the insert strategy is needlessly slow and holds the job worker busy.
  → *Collect all valid question objects into an array, then do a single `await db.insert(extractedQuestions).values(validQuestions)` bulk insert after the loop. Add a `returning()` clause if you need inserted IDs. This reduces 40 round-trips to 1.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The OpenAI client is imported from a route file. Route modules commonly register middleware, initialise singletons with side-effects, and may import Express app state. Importing them from a background worker creates hidden coupling, risks double-initialisation, and could inadvertently pull in request-scoped state or secrets logged in that module.
  → *Extract the OpenAI client instantiation into its own module (e.g. `server/lib/openaiClient.ts`) and import from there in both the route file and jobHandlers. This eliminates the cross-layer dependency with no behavioural change.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The `olderThanDays` value comes directly from the job payload with no validation. A malformed or adversarially crafted job (e.g. `olderThanDays: 0` or a negative number) would delete all rows or produce an invalid date, causing silent data loss with no error surfaced.
  → *Add explicit validation at the top of the handler: `if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays value');`. Also assert the computed `cutoff` is a valid Date before executing the delete.*
- **`shared/schema.ts` (budgets table definition):** budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, and portfolioTransactions.executedAt all use .defaultNow() without .notNull(). This means application code can explicitly insert NULL into these audit columns, making time-based queries and audit trails unreliable.
  → *Chain .notNull() after .defaultNow() on every audit timestamp, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Also add .notNull() to transactions.createdAt for the same reason.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, even though every budget query will filter by userId. Similarly, billReminders, savingsGoals, linkedCards, documentUploads, portfolioHoldings, and portfolioTransactions all have FK columns to users.id with no covering index, causing full table scans as user count grows.
  → *Add a Drizzle index factory to each missing table, e.g. for budgets: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). For portfolioHoldings also add a composite index on (userId, stockId) since queries will filter on both. Apply the same pattern to all other user-owned tables missing a userId index.*
- **`shared/schema.ts` (categories, users FK columns (all tables)):** All userId columns are declared as varchar('user_id') with no length argument. In PostgreSQL this is equivalent to text, so no length enforcement occurs at the DB layer. If the upstream auth provider (e.g. Clerk/Supabase) changes its ID format or a bug passes an arbitrarily long string, there is no DB-level rejection.
  → *Audit the auth provider's actual ID format (e.g. Clerk uses 'user_XXXX' strings capped at ~32 chars). Set a concrete max length: varchar('user_id', { length: 64 }). Apply consistently to every userId column across all tables so FK types match exactly.*
- **`shared/schema.ts` (categories table definition):** categories.isDefault defaults to false but is nullable (no .notNull()). A NULL value in a boolean flag used in conditional logic (e.g. WHERE is_default = true) will silently exclude rows from both branches of the condition rather than matching false, producing incorrect 'default category' lookups.
  → *Add .notNull() to isDefault: isDefault: boolean('is_default').default(false).notNull(). Audit all other boolean columns (isAutoSynced, isAutoDetected) for the same omission and apply .notNull() to each.*
- **`server/index.ts` (seedDatabase and probeDatabase calls inside the startup IIFE):** If `seedDatabase` and `probeDatabase` are awaited sequentially and either one is slow (e.g. cold DB connection), the HTTP server is not yet listening, blocking all traffic. Additionally if the truncated startup block calls these before `httpServer.listen()` without a try/catch, an unhandled rejection will crash the process with no structured error log.
  → *Wrap the entire startup IIFE body in try/catch and log + process.exit(1) on failure. Run independent probes in parallel: `await Promise.all([probeDatabase(), seedDatabase()])`. Start `httpServer.listen()` immediately and defer non-critical seed work to a background task if seed latency is a concern.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and hardcoded scheduler intervals):** The weekly digest interval and the purge interval periods are hardcoded magic numbers embedded directly in setInterval calls (visible in the truncated portion). These cannot be tuned without a code change and redeploy, and differ from the already-configurable AUTO_PURGE_OLDER_THAN_DAYS pattern used for the purge threshold.
  → *Extract scheduler intervals to named environment-backed constants at the top of the file alongside AUTO_PURGE_OLDER_THAN_DAYS, e.g. `const WEEKLY_DIGEST_INTERVAL_MS = parseInt(process.env.WEEKLY_DIGEST_INTERVAL_MS ?? '3600000', 10)`. Validate with the same finite-positive guard already used for AUTO_PURGE_OLDER_THAN_DAYS.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The full messages array is sent to /api/admin/help-chat on every turn, growing unbounded. A 50-message conversation sends all 50 messages each time, and there is no server-side token or message count cap enforced client-side, making this a bandwidth and potential prompt-injection amplification risk.
  → *Slice to the last N messages before sending: `messages: next.slice(-10)`. Add a hard cap: if `next.length > 100` trim the oldest non-system messages. This keeps payloads small and limits prompt injection surface.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new function on every render of the messages list. In React, an inline ref callback that changes identity causes the element to be unmounted and remounted, triggering unnecessary scrollIntoView calls and degrading performance during fast typing or streaming responses.
  → *Replace with a stable useCallback ref: `const scrollRef = useCallback((el: HTMLDivElement | null) => { el?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and attach `ref={scrollRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple parallel useQuery calls for dashboard stats):** Several independent useQuery calls (users, schools, lessons, audit log, etc.) are declared sequentially but have no staleTime set, causing all of them to refetch simultaneously on every window focus event. For a dashboard with 8+ queries this creates a thundering-herd of requests every time the admin alt-tabs back to the browser.
  → *Add `staleTime: 60_000` (1 minute) to all read-only stat queries and set `refetchOnWindowFocus: false` for audit/log queries that do not need real-time freshness. Group related stats into a single `/api/admin/stats` endpoint to reduce round-trips.*

## 💡 Opportunities

### Centralise hardcoded virtual balance seed value: ⚡ Medium Impact
The initial virtual balance (visible as a numeric literal in getVirtualBalance/seedMarketData) should be sourced from an environment variable like VIRTUAL_BALANCE_SEED_TTD with a validated default. This allows admins to adjust the starting balance per school cohort without a code deploy.

### Add a shared query-result cache for market stock data: 🔥 High Impact
getMarketStocks is called on every portfolio page load and stock-browser render but the underlying simulated data changes infrequently. A simple in-process TTL cache (node-cache or a Map with a timestamp) with a 30-second expiry would eliminate repetitive full-table scans for the most-read entity in the app.

### Introduce repository-layer pagination contract for all list methods: 🔥 High Impact
Methods like getAllQuestions, getGameSessions, and getPortfolioTransactions accept no limit/offset parameters, making it impossible to paginate from the API layer without breaking changes later. Adding { limit?: number; offset?: number } to these signatures now—with safe defaults—prevents unbounded memory growth as the school user base scales.

### Parallel domain route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six consecutive awaits. If any registerXRoutes function does async work (DB calls, schema checks), moving them to Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) would reduce cold-start time proportionally to the slowest single initialiser rather than their sum.

### Duplicate health check endpoints: 📌 Low Impact
Both /api/health (line ~35) and /healthz (line ~80) serve overlapping purposes but /api/health has no DB check while /healthz does. This causes Railway and uptime monitors to potentially hit the wrong one. Consolidate to a single /healthz that includes the DB ping and remove /api/health, or clearly document and separate liveness vs readiness semantics.

### ObjectStorageService singleton instantiation: ⚡ Medium Impact
The ObjectStorageService is instantiated at module load time as a top-level `const objectStorage = new ObjectStorageService()`. If the constructor establishes connections or reads credentials, errors here will crash the module silently before registerRoutes is even called. Move instantiation inside registerRoutes with explicit error handling, or use a lazy-init pattern with a health check on first use.

### Parallel AI calls where extraction steps are independent: ⚡ Medium Impact
In handlers that make multiple sequential OpenAI calls where the second does not depend on the first (e.g. if classification and extraction were ever split), wrapping them in Promise.all would cut wall-clock job time proportionally. Audit any future multi-call additions with this pattern from the start.

### Centralised payload schema validation with Zod at handler entry: ⚡ Medium Impact
Each handler currently trusts payload shape implicitly via TypeScript types that are only checked at compile time. Adding a Zod parse at the top of each handler would catch type mismatches from DB-stored jobs (e.g. after a schema migration) at runtime and produce clear error messages instead of cryptic downstream failures.

### Stream large exports instead of buffering in memory: 🔥 High Impact
The admin-csv-export handler (truncated but inferable from the pattern) likely builds the full CSV string or buffer in memory before uploading. For large teacher/student datasets this risks heap pressure. Using a streaming CSV serialiser piped directly to the object storage upload stream would bound memory to a constant regardless of row count.

### Partial index for unprocessed document uploads: ⚡ Medium Impact
Add a partial index on documentUploads filtered to status = 'processing', e.g. index('idx_document_uploads_processing').on(t.userId).where(sql`status = 'processing'`). The background worker polling for pending uploads will hit this index instead of scanning all completed/failed rows, which will dominate over time.

### Consolidate repeated currency columns into a schema-level default: ⚡ Medium Impact
Eight tables independently hardcode .default('BSD') for currency. Extract a shared currencyCol helper (e.g. const bsdCurrency = () => text('currency').default('BSD').notNull()) and reuse it. This eliminates drift if the default currency changes and ensures .notNull() is applied uniformly, reducing future bug surface.

### Add composite index on portfolioTransactions for P&L history queries: 🔥 High Impact
P&L and trade-history screens will query portfolioTransactions by (userId, stockId, executedAt DESC). Add a composite index on all three columns in that order. Without it, filtering by user and stock requires a two-column scan followed by a sort, which becomes expensive as simulated trade volume grows per user.

### Structured JSON logging instead of console.log strings: 🔥 High Impact
The `log()` helper emits free-form locale-formatted strings. Replacing it with a structured logger (e.g. pino) that emits JSON with fields like `{ level, source, message, durationMs, method, path, statusCode }` would make logs parseable by Replit's log drain or any SIEM without regex scraping, and would make it trivial to filter out response-body data per finding #3.

### Centralised graceful shutdown coordinator: 🔥 High Impact
SIGTERM and SIGINT handling is currently absent or scattered. A single shutdown coordinator that sequentially closes intervals, drains the job worker, closes the HTTP server with a timeout, then closes the DB pool would prevent the in-flight job corruption risk identified in finding #2 and make Replit container restarts safe. Expected impact: zero dropped jobs on deploy.

### Idempotency guard for maybeRunAiUsagePurge matching the digest pattern: ⚡ Medium Impact
The weekly digest uses an atomic DB insert to prevent double-enqueue across processes, but the truncated purge scheduler likely lacks the same guard. Applying the same onConflictDoNothing pattern with an aiUsagePurgeRuns table row keyed on run-date would eliminate duplicate purge jobs if multiple dynos or a restart coincides with the interval firing.

### Consolidate admin stat endpoints into a single aggregated route: 🔥 High Impact
Replace 6-8 individual stat queries with one /api/admin/dashboard-stats endpoint that runs all DB aggregations in parallel server-side using Promise.all. This reduces client HTTP overhead from ~8 requests to 1 on every dashboard load and simplifies loading/error state management to a single query.

### Add React.memo to pure sub-panel components: ⚡ Medium Impact
Large sub-sections (user table, school table, lesson editor) rendered inside the dashboard re-render whenever any top-level state changes (e.g., active tab, search text in an unrelated tab). Wrapping each panel in React.memo with stable prop references will eliminate these wasted renders, which is particularly important on lower-powered Caribbean school devices.

### Move hardcoded API path strings to a central config constant: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/admin/users', '/api/admin/schools' are scattered throughout the file. Centralise them in a single API_ROUTES object (or auto-generate from the server router). This prevents typo-driven 404s, makes base-URL changes a one-line edit, and enables TypeScript to catch invalid route references at compile time.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts exhibits multiple N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step write operations, and sequential awaits that should run in parallel. The truncated file prevents full analysis but the visible patterns—combined with hardcoded fallback values and absent pagination guards on leaderboard/question queries—represent real reliability and performance risks for a school-aged user base.

### API Routes (`server/routes.ts`)
The route registration code is generally well-structured with good use of Promise.all for parallel imports and proper try/catch on async handlers. Key concerns are: unprotected AI/chat/image/audio routes registered without rate limiting, a host-spoofing vulnerability in robots.txt and sitemap generation, and a fire-and-forget Supabase init chain that silently swallows individual step failures.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads are stored in the job table (a classic OOM/DB bloat risk), sequential DB inserts in a loop should be batched, and the openai import from a route file couples concerns in a way that could expose chat state. Error handling is present but inconsistent across handlers.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with proper foreign keys and some indexes already defined, but has several reliability and performance gaps. Key concerns are missing NOT NULL constraints on audit timestamps, absent indexes on high-cardinality foreign keys used in likely WHERE clauses, and overly permissive varchar column widths that could accept unbounded input.

### Server Startup (`server/index.ts`)
server/index.ts has solid structural foundations with good environment validation and atomic job claiming, but suffers from blocking synchronous startup operations, unguarded async schedulers that swallow errors silently, and interval handles that are never cleared on shutdown causing resource leaks. The response-body logging middleware also risks exposing sensitive financial or PII data in plaintext logs for all API responses.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth checks on admin routes, unprotected async operations, and inline scroll refs creating new function instances on every render. The chat component leaks no resources but exposes the full conversation history in every API payload which could become a memory and bandwidth concern at scale. Overall structure is reasonable but needs hardening before production use with school-aged users.
