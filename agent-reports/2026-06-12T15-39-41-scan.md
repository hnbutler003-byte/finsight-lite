## 🔴 Critical (8)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app) (~line 112-114)
**Issue:** No rate limiting is applied before registering AI/image/audio routes. For a school-aged audience these are the most expensive endpoints (OpenAI tokens, compute), and any authenticated or unauthenticated user can hammer them without restriction, risking runaway costs and denial-of-service.

**Fix:** Add a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerX calls.

### `server/jobHandlers.ts` — runExtractPaper
**Issue:** fileB64 (full base64-encoded exam paper) is stored in the job payload row in the database. A single A4 PDF scan can easily be 2-5 MB encoded, meaning the jobs table accumulates many large BLOBs, bloating the DB, slowing job queue queries, and exposing raw file content to anything that reads job rows.

**Fix:** Upload the file to object storage before enqueuing, then store only the resulting object path in the payload. In runExtractPaper, download the buffer from object storage at the start of the handler and delete the staging object on completion or final failure. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts` — registerJobHandler('admin-csv-export') — implied by truncated payload usage
**Issue:** The admin-csv-export handler accepts a 'type' discriminator from job.payload without any authorization check inside the handler itself. If enqueueJob can be called by a non-admin API route (or if a job is replayed manually), arbitrary export types could be triggered, potentially exfiltrating full user or transaction datasets.

**Fix:** Record the requesting userId and role at enqueue time in the payload, and at the start of the handler assert that role === 'admin', throwing an unretryable error if not. Additionally, validate 'type' against an explicit allowlist of known export types before executing any DB query.

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique index. Duplicate tickers for simulated instruments would corrupt portfolio P&L calculations (e.g., two 'BISX:CBL' rows doubling price lookups), and there is no guard at the database level.

**Fix:** Replace the plain column declaration with a uniqueIndex in the table callback: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also consider adding a NOT NULL constraint to annualReturnPct or providing a sensible default (e.g., .default('0').notNull()) since downstream return calculations will produce NaN if this is null.

### `server/index.ts` — res.json monkey-patch in request logger middleware
**Issue:** `capturedJsonResponse` is unconditionally serialized with `JSON.stringify` and appended to every API log line. For a financial literacy app with school-aged users this will log full response bodies — including auth tokens, session data, user PII, grades, or balance information — to stdout/log aggregators with no redaction.

**Fix:** Either remove response-body logging entirely in production (`if (process.env.NODE_ENV !== 'production')`), or allowlist safe fields: only log `{ status, errorCode }` keys and explicitly strip any key matching /token|secret|password|email|dob|balance/i before serializing.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both `setInterval` return values are never stored, so the intervals cannot be cleared in the graceful shutdown handler. On SIGTERM the process closes the HTTP server and DB but the intervals fire one final time against a closing DB connection, causing unhandled promise rejections and potential zombie jobs.

**Fix:** Store both handles: `const weeklyHandle = setInterval(...)` and `const purgeHandle = setInterval(...)`. In the SIGTERM/SIGINT shutdown handler call `clearInterval(weeklyHandle)` and `clearInterval(purgeHandle)` before closing the HTTP server so no new scheduler invocations start during drain.

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth hook usage / top of AdminDashboard component
**Issue:** The admin dashboard renders its full JSX tree before the auth check resolves. If `useAdminAuth` returns a loading state, child components that fire useQuery calls for users, schools, and audit logs will execute immediately — making unauthenticated requests or leaking admin data in the query cache before a redirect occurs.

**Fix:** Destructure a loading/isAuthenticated flag from `useAdminAuth` and return `null` or a `<Loader2>` spinner early: `if (authLoading) return <Loader2 className='animate-spin' />;` before any query hooks would render child components that trigger their own queries.

### `client/src/pages/AdminDashboard.tsx` — sendMessage / apiRequest POST /api/admin/help-chat
**Issue:** The chat response is logged or rendered without any content filtering. For a platform used by school-aged users, an admin triggering the AI assistant could receive and inadvertently expose unsafe content. Additionally, `data.reply` is rendered directly with `whitespace-pre-wrap` with no XSS sanitization — if the API ever returns HTML, it renders raw.

**Fix:** On the server route, enforce a system prompt that restricts responses to platform-admin topics. On the client, since content is rendered as text nodes (not dangerouslySetInnerHTML), XSS is not an immediate risk, but add an explicit check: assert `typeof data.reply === 'string'` and truncate to 2000 chars before storing in state.

## 🟡 Warnings (16)

- **`server/routes.ts` (initSupabaseTables().then(...).catch(...) (~line 117-125)):** The Supabase init chain runs sequentially (7 chained .then() calls) when most steps are likely independent. More critically, only the final .catch() handles errors — if seedFinancialAcademyLesson() throws, initStaticContentTables() and all subsequent seeds are silently skipped with no per-step diagnostic, making startup failures very hard to debug.
  → *Run independent seeds in parallel with Promise.allSettled and log each rejection individually: Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error('[Supabase] seed error', i, r.reason))). Keep initSupabaseTables and trimOrgNamesInSupabase sequential if ordering is required.*
- **`server/routes.ts` (/api/health (~line 30) and /healthz (~line 79)):** Two health-check endpoints exist: /api/health returns a trivial JSON blob with no DB check, and /healthz performs a real DB probe. Railway/uptime monitors configured to the wrong one will report healthy even during a total DB outage. The duplicate is dead/confusing code.
  → *Remove /api/health entirely and point all monitors to /healthz, which already does a real 'select 1' probe with a 503 on failure. If a lightweight ping is needed for load-balancer TCP checks, keep /api/health but add a comment clearly stating it is not a deep health check.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 99-107)):** The filePath parameter is passed directly to objectStorage.searchPublicObject() with no sanitisation. A path like ../../private/keys could traverse into unintended storage prefixes depending on the ObjectStorageService implementation, and there is no authentication check — any anonymous user can probe arbitrary object paths.
  → *Sanitise the path: const safe = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); and reject requests where safe !== req.params.filePath. Additionally confirm whether these objects are truly public; if any are org-scoped, add a session/auth guard before serving.*
- **`server/routes.ts` (robots.txt and sitemap.xml handlers (~line 38-90)):** The host header is taken directly from req.headers.host / req.headers['x-forwarded-host'] and interpolated into response bodies without validation. A spoofed Host header causes the sitemap and robots.txt to contain attacker-controlled URLs, which can mislead crawlers or be used in cache-poisoning attacks.
  → *Resolve the canonical base URL once from an environment variable (e.g. process.env.PUBLIC_URL or RAILWAY_STATIC_URL) and fall back to the header only in local development. Add a guard: const allowedHosts = new Set([process.env.PUBLIC_HOST]); if (!allowedHosts.has(host)) use the env default instead of the header value.*
- **`server/jobHandlers.ts` (runExtractPaper — inner for loop over questions):** Each extracted question is inserted with a sequential await inside a for loop. For a 60-question paper this fires 60 individual round-trips to the database serially, making the handler artificially slow and holding the DB connection for an extended period.
  → *Replace the for loop with a single batch insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(allValidQuestions) once. If storage.createExtractedQuestion must be used, wrap all calls in Promise.all with the mapped array instead.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (text extraction + question extraction)):** The vision/PDF text-extraction step and the downstream question-extraction step are two sequential OpenAI calls that together can take 20-40 seconds. While the second call depends on the first, the paper status update and the question batch insert after parsing are also run sequentially when they could overlap with other independent work.
  → *This is a minor parallelism opportunity, but more importantly the two OpenAI calls should each have explicit timeout/abort signals via AbortController with a configurable deadline (e.g. OPENAI_TIMEOUT_MS env var, defaulting to 60000) to prevent the job from hanging indefinitely if the API stalls.*
- **`shared/schema.ts` (categories table definition):** categories.userId uses varchar without a length argument, which defaults to unlimited in Drizzle/Postgres and is inconsistent with other tables. More critically, there is no index on categories.userId, but every user-facing query will filter categories by userId to merge system (null) and personal rows.
  → *Add a dedicated index: index('idx_categories_user_id').on(t.userId) inside the pgTable callback. Also standardise the varchar length to match the auth provider's ID format (e.g., varchar('user_id', { length: 128 })) across all tables.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these tables have indexes on their userId columns. Every dashboard load will perform full sequential scans on these tables filtered by userId, which degrades linearly as user data grows — a serious problem for a school-aged audience that may have many low-value rows.
  → *Add a pgTable index callback to each table: e.g., for budgets: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). For billReminders also index nextDueDate since reminder queries will order/filter by it: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate).*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). The database default only fires on INSERT; application-level inserts that explicitly pass null will succeed, leaving audit timestamps as NULL and breaking any ORDER BY createdAt or time-range queries silently.
  → *Add .notNull() to every createdAt column: timestamp('created_at').defaultNow().notNull(). Same applies to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (portfolioHoldings — quantity column):** quantity is typed as integer, which cannot represent fractional share quantities common in bond simulations or drip investing scenarios the app may model. A sell of 0.5 units would be silently truncated to 0 at the database level, creating phantom holdings and incorrect P&L.
  → *Change quantity to numeric('quantity', { precision: 18, scale: 8 }).notNull() on both portfolioHoldings and portfolioTransactions. Update the corresponding Zod insert schemas to use z.number().positive() instead of z.number().int().*
- **`server/index.ts` (getMostRecentSundayAt7pm):** Day-of-week rollback logic is wrong: `(dow + 0) % 7` always returns 0 on Sunday (dow=0), so `daysSince` is 0, and the function returns today-at-19:00 rather than rolling back to the previous Sunday. The guard `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` only catches the future case but does not fix the structural error for Sunday itself.
  → *Replace `(dow + 0) % 7` with `dow` directly and remove the dead `+ 0`. Then verify the Sunday edge case: when `dow === 0`, `daysSince` should be 0 and the future-guard already rolls back 7 days, so the arithmetic is correct only if the intent is 'this Sunday'. Add a unit test covering Sunday before and after 19:00 UTC.*
- **`server/index.ts` (startup await sequence (probeDatabase → seedDatabase → registerRoutes → startJobWorker)):** Each startup step is awaited serially. `seedDatabase` and `probeDatabase` are independent of `startJobWorker` and `registerJobHandlers`. On a cold Replit instance with a slow DB this adds cumulative latency before the HTTP server is ready, increasing the window where health checks fail.
  → *Parallelize independent steps with `Promise.all`: `await Promise.all([probeDatabase(), seedDatabase()])` then `await registerRoutes(app, httpServer)` (which needs the DB ready), then `await Promise.all([startJobWorker(), registerJobHandlers()])`. Guard the whole block in a try/catch that calls `process.exit(1)` so startup failures are fatal.*
- **`server/index.ts` (maybeRunAiUsagePurge (and maybeRunWeeklyDigest)):** Both scheduler functions catch errors and log them, but only log `(e as Error).message`. If `e` is not an Error object (e.g. a Drizzle DB rejection that throws a plain object or string), `.message` is `undefined` and the log line is silent, making production incidents invisible.
  → *Replace `(e as Error).message` with a safe extractor: `e instanceof Error ? e.message : String(e)`. Also pipe these errors through `captureError(e)` so Sentry receives scheduler failures, not just HTTP-layer errors.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat / sendMessage):** The help-chat API call sends the full cumulative messages array on every turn with no length cap. A user can inflate this array to thousands of entries across a session, sending enormous payloads to /api/admin/help-chat and exhausting token limits or causing server timeouts.
  → *Slice the history before sending: `const history = next.slice(-20);` and pass `history` instead of `next` to the API. Add a guard: `if (next.length > 200) setMessages(next.slice(-100));` to bound in-memory growth.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat JSX — `<div ref={el => { if (el) el.scrollIntoView(...) }} />`):** The scroll anchor uses an inline ref callback with no useRef/useCallback, so a new function is created and a DOM scrollIntoView call is triggered on every single render of the messages list, including intermediate loading state renders. This causes janky scroll behavior and unnecessary layout thrashing.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null);` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);`. Render `<div ref={bottomRef} />` as a static element.*
- **`client/src/pages/AdminDashboard.tsx` (BarChart / LineChart data transforms in JSX render body):** Chart data derivations (aggregating users by school, computing weekly activity trends, etc.) are computed inline inside JSX or in the render body without useMemo. On every parent re-render triggered by tab switches or toast state changes, these O(n) array reduce/map operations re-run against the full dataset.
  → *Extract each data transform into a `useMemo` keyed on the source query data: `const chartData = useMemo(() => users.reduce(...), [users]);`. This avoids recomputation when unrelated state (e.g., `open` dialog state) changes.*

## 💡 Opportunities

### Centralise base-URL resolution — ⚡ Medium Impact
Both /robots.txt and /sitemap.xml independently reconstruct the base URL from headers using identical logic. Extract a getBaseUrl(req) helper (with the env-var guard from finding #5) to eliminate duplication and ensure consistent behaviour if the logic ever changes.

### Lazy-load heavy domain routers to reduce cold-start blocking — 📌 Low Impact
The six domain routers are imported in parallel with Promise.all, but their registration (registerAuthDomainRoutes, registerStudentRoutes, etc.) is still sequential with six awaited calls. If any registration is async and slow it blocks the rest. Wrap each registration in Promise.all as well — safe as long as routes don't depend on each other's middleware being registered first.

### Add structured startup logging for Supabase seed results — ⚡ Medium Impact
Currently seed failures are logged only as generic console.error strings. Emitting structured JSON logs (step name, duration, error code) would allow Railway log drains / Sentry to surface seed regressions automatically rather than requiring manual log grep after a broken deploy.

### Idempotent question deletion before re-extraction — ⚡ Medium Impact
The current idempotency guard deletes all extracted questions at the start of every retry, including successful partial batches from a previous attempt. Switching to an upsert strategy keyed on (paperId, order) would avoid re-doing work on retries and remove the delete-then-reinsert race window where another reader could see zero questions mid-retry.

### Stream PDF parsing instead of buffering full base64 in memory — 🔥 High Impact
Buffer.from(fileB64, 'base64') loads the entire decoded file into Node.js heap before pdf-parse sees it. Once the file is moved to object storage (fixing finding #1), the handler can pipe the object storage read stream directly into pdf-parse, reducing peak RSS significantly for large scanned papers and removing the double-memory cost of holding both the base64 string and the decoded buffer simultaneously.

### Centralise OpenAI client import to avoid cross-module coupling — ⚡ Medium Impact
Importing openai from './replit_integrations/chat/routes' couples jobHandlers to a route module, meaning any refactor of the chat routes can silently break job processing. Export the openai client from a dedicated './lib/openaiClient' module and import it directly in both places to decouple the dependency and make the client's configuration (API key, org, timeout defaults) a single source of truth.

### Centralise currency enum as a reusable Postgres enum type — ⚡ Medium Impact
The string literal 'BSD' default and currency text column are repeated across at least 6 tables with no enforcement of valid currency codes. Defining a pgEnum('currency', ['BSD','TTD','JMD','BBD','USD']) and reusing it would catch invalid currency insertions at the DB level, reduce storage marginally, and make adding a new Caribbean currency a single-line change.

### Add composite index on billReminders(userId, nextDueDate) for reminder scheduling queries — 🔥 High Impact
The reminder notification worker (implied by isAutoDetected and nextDueDate) will query WHERE userId = ? AND nextDueDate <= NOW() ORDER BY nextDueDate. Without a composite index this is a full table scan per user per cron tick. The index would make this query O(log n) and reduce cron execution time significantly as the user base grows.

### Replace text type with varchar(length) on high-cardinality FK columns — ⚡ Medium Impact
userId is varchar without length on categories but unbounded text on several join columns. Postgres stores short text and varchar identically, but explicit length limits prevent accidental insertion of malformed auth tokens (e.g., an entire JWT) into FK columns, which would pass FK checks but corrupt joins. Standardise to varchar('user_id', { length: 128 }) matching the Clerk/Auth0 ID ceiling across all tables.

### Structured startup health gate with readiness flag — 🔥 High Impact
Add a boolean `isReady` flag set to `true` only after all startup awaits complete. Register a `/healthz` route before `registerRoutes` that returns 503 while `isReady` is false. This prevents load balancers and Replit's health checker from routing traffic during the DB probe/seed window, eliminating spurious 500s on cold starts.

### Centralize interval management in a ShutdownRegistry — ⚡ Medium Impact
Create a lightweight `ShutdownRegistry` that accepts `{ handle, name }` entries via `.register()` and exposes a single `.shutdown()` method that clears all intervals and removes all event listeners in order. Replace all ad-hoc `clearInterval` calls in the SIGTERM handler with one `registry.shutdown()` call. This scales cleanly as more schedulers are added and prevents future leaks.

### Move AUTO_PURGE_OLDER_THAN_DAYS and scheduler intervals to validated config module — ⚡ Medium Impact
Both the purge-days value and the scheduler tick intervals (currently likely hardcoded numbers inside the truncated section) are runtime-tunable but require a redeploy to change. Extract them into a `config.ts` that reads, validates, and exports all env-derived constants. This also makes it trivial to shorten intervals in test environments via env vars, improving CI speed.

### Split AdminDashboard into lazy-loaded tab panels — 🔥 High Impact
The single file exceeds 130K characters, meaning all chart libraries, dialogs, and data tables are parsed and executed on initial load. Code-splitting each tab (Users, Schools, Analytics, Audit) with React.lazy and Suspense would cut initial JS parse time by an estimated 60–70% and isolate re-render scope to the active tab.

### Consolidate admin bootstrap queries with Promise.all — 🔥 High Impact
If the dashboard fires separate sequential useQuery calls for users, schools, teachers, and audit logs on mount, the waterfall adds hundreds of milliseconds on Caribbean mobile networks. Create a single `/api/admin/bootstrap` endpoint that returns all four collections in one round-trip, or use `Promise.all` inside a single custom hook to parallelize independent fetches.

### Move hardcoded pagination and page-size constants to config — ⚡ Medium Impact
Page size values (e.g., 10 or 25 rows per page) and API base paths hardcoded in the component should be extracted to a shared `config.ts` or environment variable. This avoids silent inconsistencies when the backend changes defaults and makes A/B testing page sizes trivial.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel module loading and rate limiting on auth endpoints. Key concerns are: unprotected AI/heavy routes (no rate limiting on registerChatRoutes/registerImageRoutes/registerAudioRoutes), unhandled promise rejections on the Supabase init chain, and a duplicate health-check endpoint that adds maintenance confusion.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and a hardcoded token limit that belongs in environment config. The extract-paper handler is the highest-risk area, combining all three problems plus a potential memory spike on large PDF/image files.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes already in place, but has several reliability and performance gaps. Key concerns are missing NOT NULL constraints on audit timestamps, absent indexes on high-traffic query columns, and an undersized varchar for user_id that could silently truncate values depending on the auth provider.

### Server Startup (`server/index.ts`)
The server startup sequence contains blocking synchronous-style serial awaits and unguarded async scheduler intervals that can silently fail or leak on shutdown. The most critical gaps are: intervals stored without cleanup in graceful shutdown handlers, response-body logging that can leak sensitive financial/PII data for all API responses, and sequential DB/seed/job-worker awaits that unnecessarily delay startup.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth guards on sensitive admin routes, inline scroll refs creating DOM nodes on every render, and chat messages sent without sanitization or rate limiting. The component is large and monolithic with expensive computations likely missing memoization, and async chat operations lack proper error boundary handling for school-aged user safety.
