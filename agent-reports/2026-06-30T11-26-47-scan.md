## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. A single user or bot can spam these endpoints, incurring unbounded API cost and latency for all users — especially risky for a school-aged audience.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP using express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 36 and ~line 52)
**Issue:** The `host` and `proto` values are taken directly from `x-forwarded-host` and `x-forwarded-proto` headers without any validation or allowlist. An attacker can send a crafted request that injects an arbitrary hostname into the Sitemap URL, which could be cached by crawlers or used for cache-poisoning attacks.

**Fix:** Read the canonical base URL from an environment variable: `const base = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com';` and ignore the forwarded headers entirely for these two routes. Remove the dynamic header-based construction.

---

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Raw base64 file content is stored directly in the jobs table payload. For a multi-page PDF or image this can easily exceed several MB per row, bloating the job queue table, slowing polling queries, and risking row size limits depending on the DB engine.

**Fix:** Upload the file to object storage before enqueuing and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of the handler. This keeps job rows small and makes payloads inspectable in the DB without encoding noise.

---

### `server/index.ts`: probeDatabase / seedDatabase calls in main()
**Issue:** probeDatabase() and seedDatabase() are awaited sequentially before the HTTP server listens, blocking startup. If either call hangs (e.g. DB unreachable), the process never becomes ready and no health-check endpoint is reachable to signal the failure to an orchestrator.

**Fix:** Start listening immediately after registerRoutes(), then run probeDatabase and seedDatabase in the background. Return a 503 from a /healthz route until the DB probe succeeds. Use Promise.race with a timeout (e.g. 10 s) around probeDatabase so a hung DB does not stall startup indefinitely.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are stored in local variables (digestInterval, purgeInterval) but the SIGTERM / SIGINT shutdown handler visible in the truncated code does not reference them. On process exit the intervals keep the event loop alive, preventing a clean shutdown and potentially firing a scheduler tick during teardown after the DB connection is closed.

**Fix:** Capture both handles at module scope. In the graceful shutdown handler call clearInterval(digestInterval) and clearInterval(purgeInterval) before closing the HTTP server and DB pool. Wrap the shutdown sequence in a Promise.race with a 15-second hard exit via process.exit(1) to guarantee termination.

---

### `server/index.ts`: res.json middleware / log(logLine) in the finish handler
**Issue:** Every API response body is JSON.stringify-ed and appended to the log line unconditionally. For endpoints that return user financial records, quiz scores, or session tokens this will print sensitive data to stdout/log aggregators in plain text, violating data-minimisation expectations for a school-aged user base.

**Fix:** Remove capturedJsonResponse logging entirely, or gate it on NODE_ENV === 'development'. If response-size metrics are needed in production, log only res.getHeader('content-length') and res.statusCode rather than the body.

## 🟡 Warnings (9)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 115)):** Seven Supabase init/seed functions are chained sequentially with `.then()`. If each takes even 200 ms, startup is delayed ~1.4 s before the chain resolves, and a failure in any step silently skips all subsequent steps with only a single catch at the end.
  → *Run independent seed operations in parallel where safe: `await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])`. Split the chain into: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await initStaticContentTables();` then `Promise.all` the seed steps. Wrap each in its own try/catch to isolate failures.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The `/healthz` endpoint dynamically imports `./db` and `drizzle-orm` on every call. Dynamic imports are cached after the first load, but the import() call overhead and the unconditional `SELECT 1` hit the database on every probe. Uptime monitors hitting this every 30 s means constant unnecessary DB queries.
  → *Move the db and sql imports to the top of the file as static imports. Optionally add a short-circuit cache: only re-query the DB if the last successful check was more than 10 s ago, returning the cached result otherwise.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 104)):** The `filePath` parameter from the URL is passed directly to `objectStorage.searchPublicObject(filePath)` with no sanitization. A path traversal payload like `../../private/keys` could resolve to unintended storage objects depending on how `searchPublicObject` is implemented.
  → *Sanitize the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Add `import path from 'path'` at the top.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop. For a 40-question paper this fires 40 serial round-trips to the database, adding substantial latency and leaving a wider window for partial failure between inserts.
  → *Collect all valid question objects into an array, then execute a single batch insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the entire operation in a transaction so partial inserts are impossible and the earlier defensive delete plus the new inserts are atomic.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage') handler body):** The purge-ai-usage handler has no try/catch. An unexpected DB error (connection drop, constraint violation) will surface as an unhandled promise rejection inside the job runner rather than being caught, logged with context, and rethrown cleanly. This makes debugging failures opaque.
  → *Wrap the delete call in try/catch, log the error with job context (olderThanDays, cutoff date), then rethrow so the job runner can record the failure and trigger retries: try { ... } catch (err) { logger.error({ job: 'purge-ai-usage', cutoff }, err); throw err; }*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** The magic number 30000 is hardcoded inline with no explanation or configurability. If the OpenAI model context window changes or the prompt overhead grows, this silent truncation can cause questions at the end of long papers to be silently dropped without any warning to the user or operator.
  → *Move to an environment variable with a documented default: const MAX_EXTRACT_CHARS = parseInt(process.env.EXTRACT_TEXT_MAX_CHARS ?? '30000', 10). Add a log warning when truncation occurs: if (pdfData.text.length > MAX_EXTRACT_CHARS) logger.warn({ paperId, originalLength: pdfData.text.length }, 'PDF text truncated before extraction').*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse – res.setHeader Content-Disposition):** The fileName parameter is interpolated directly into the Content-Disposition header without sanitization. A filename containing a double-quote or newline characters could break the header or enable header injection, which is a security concern even in a school-facing app.
  → *Sanitize the filename before use: const safeFileName = fileName.replace(/[\r\n"]/g, '_'). Then use res.setHeader('Content-Disposition', `attachment; filename="${safeFileName}"`). For broader compatibility consider the RFC 5987 encoded form for non-ASCII names.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The day-of-week arithmetic uses (dow + 0) % 7 which is a no-op, making the variable name daysSince misleading and the Sunday-detection logic incorrect. When now is Sunday (dow === 0), daysSince is 0 so d is unchanged, but the comment and intent imply rolling back to the most recent past Sunday. This means the digest can fire on any day of the week whenever dow happens to equal 0, and the guard d.getTime() > Date.now() is evaluated with the same Date object just mutated, making the comparison unreliable.
  → *Replace the arithmetic with: const daysSince = dow === 0 ? 0 : 7 - (7 - dow) % 7; — or more readably compute the offset as (dow === 0 ? 0 : dow) and subtract. Compute target before mutating d, or clone d after the setHours call to avoid comparing a mutated date against Date.now().*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / hardcoded scheduler intervals):** The digest and purge scheduler polling intervals (visible as numeric literals in the truncated section, typically 60000 ms) are hardcoded. In a school deployment the operator cannot tune polling frequency without a code change. Similarly the 10 mb JSON body limit is hardcoded rather than read from an env var, making it easy to miss when hardening for production.
  → *Add SCHEDULER_POLL_INTERVAL_MS and MAX_JSON_BODY_SIZE env vars with documented defaults (e.g. 60000 and '1mb' respectively). Read them with the same integer-parse + fallback pattern already used for AUTO_PURGE_OLDER_THAN_DAYS. This is especially important for the body limit given the app serves minors and a 10 mb limit is an easy DoS vector.*

## 💡 Opportunities

### Centralized AI rate limiting middleware module: 🔥 High Impact
Extract all rate limiter definitions into a single rateLimiter.ts that exports named limiters for auth, AI, and public endpoints. Apply them declaratively in routes.ts before each route group. This makes limits visible, auditable, and easy to tune — important for cost control on OpenAI calls from school users.

### Environment-driven base URL configuration: ⚡ Medium Impact
Replace all dynamic host-header-based URL construction with a single `PUBLIC_BASE_URL` env var read at startup. This eliminates the host-injection risk, simplifies robots.txt/sitemap logic, and makes the app trivially portable across Railway preview environments and production.

### Parallel Supabase seed execution with per-step error isolation: ⚡ Medium Impact
Refactor the sequential seed chain into a `Promise.allSettled` call for independent seeds, logging individual failures without aborting the others. Expected impact: ~60% reduction in startup initialization time and improved visibility into which specific seed step failed.

### Parallelize independent async lookups in badge/XP award handlers: 🔥 High Impact
Any handler that fetches user record, module details, and existing badges in sequence should use Promise.all for those independent reads. Based on the visible pattern (sequential awaits before conditional logic), converting to Promise.all([fetchUser, fetchModule, fetchBadges]) would cut handler latency by roughly 50-60% for those jobs with no correctness risk.

### Add idempotency guard to extract-paper beyond question deletion: ⚡ Medium Impact
The current idempotency strategy deletes questions on every retry, including retries triggered by transient network errors after a successful OpenAI call. Persisting the raw extracted JSON to object storage after the first successful OpenAI response and reusing it on retries would eliminate redundant paid API calls and speed up recovery.

### Centralize job error logging with structured context: ⚡ Medium Impact
Only the ai-health-check handler uses Sentry capture; other handlers throw bare errors. Wrapping registerJobHandler itself (or using a middleware pattern) to automatically capture unhandled errors with job name, attempt count, and payload keys (not values) would give uniform observability across all job types without per-handler boilerplate.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase(), seedDatabase(), registerJobHandlers(), and startJobWorker() have no data dependency on each other and can run concurrently. Awaiting them with Promise.all instead of sequentially could reduce startup time by the sum of the slowest-minus-fastest operation, meaningful on cold Replit container starts where DB round-trips can take 1-3 seconds each.

### Structured logging with redaction: 🔥 High Impact
Replacing console.log/log() with a structured logger (e.g. pino) and a redaction config for fields like token, password, sessionId, and userId would simultaneously fix the PII-in-logs finding and give log aggregators queryable JSON fields. Pino adds ~0.5 ms per log call overhead versus naive JSON.stringify, which is acceptable for an API server.

### Extract scheduler into a separate worker process: ⚡ Medium Impact
Running digest and purge schedulers inside the web server process couples availability of the HTTP tier to job execution. Moving them to a dedicated worker (a second Node process or a Replit Background Worker) means a scheduler crash or a slow DB query during job enqueue does not affect HTTP latency, and each process can be scaled and restarted independently.
