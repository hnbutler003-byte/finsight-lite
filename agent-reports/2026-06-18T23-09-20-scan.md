## 🔴 Critical (8)

### `server/routes.ts` — registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 113-115)
**Issue:** No rate limiting is applied to the OpenAI-backed chat, image, and audio routes before they are registered. Any authenticated (or unauthenticated, depending on those route handlers) user can hammer OpenAI endpoints, causing runaway API costs and degraded service for all users.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/min per IP via express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — adjust path prefixes to match what registerChatRoutes etc. actually mount.

### `server/routes.ts` — robots.txt handler (~line 47) and sitemap.xml handler (~line 60)
**Issue:** The `host` and `x-forwarded-host` headers are taken directly from the request and interpolated into output without validation. An attacker can send a crafted Host header to poison the Sitemap URL in robots.txt or inject arbitrary content into the sitemap, enabling cache-poisoning or open-redirect abuse.

**Fix:** Read the canonical base URL from a single env var set at deploy time: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and remove all dynamic header-derived base URL logic from both handlers. Throw or fall back loudly if the env var is missing at startup.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** The full base64-encoded file (fileB64) is stored directly in the job table payload. For a multi-page PDF or high-res image this can easily be 5–20 MB per job row, bloating the jobs table, slowing queue polls, and risking OOM if many jobs are loaded into memory simultaneously.

**Fix:** Upload the file to private object storage at enqueue time and store only the object path in the job payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient before processing, then delete the temp object on success or terminal failure.

### `shared/schema.ts` — budgets table definition
**Issue:** budgets.userId has no index. Any query fetching a user's budgets — which happens on every dashboard load — will do a full table scan as the table grows.

**Fix:** Add a composite index on (userId, categoryId) inside the budgets table's index callback: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This also replaces the need for a standalone userId index since the composite covers single-column userId lookups.

### `shared/schema.ts` — simulatedStocks table definition
**Issue:** ticker column has no unique index. Duplicate tickers can be inserted silently, and any lookup by ticker (e.g. resolving a trade) performs a full table scan instead of an index seek.

**Fix:** Replace the plain column definition with a table-level uniqueIndex: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This enforces uniqueness at the DB level and makes ticker-based lookups O(log n).

### `server/index.ts` — getMostRecentSundayAt7pm
**Issue:** The day-of-week arithmetic is wrong: `(dow + 0) % 7` always returns the current day index unchanged, so on any day other than Sunday the function walks back to the wrong date. On Saturday (dow=6) it backs up 6 days to Sunday, but on Monday (dow=1) it only backs up 1 day to Sunday of the same week — which may still be in the future, triggering the -7 correction — but on Wednesday (dow=3) it backs up 3 days to Monday, not Sunday. The intended expression is `(dow === 0 ? 0 : dow)` replaced by simply `dow` days back only when dow=0 means Sunday; the correct offset to reach the most recent Sunday is `dow` days (since Sunday=0 means today is already Sunday, offset=0).

**Fix:** Replace `const daysSince = (dow + 0) % 7;` with `const daysSince = dow;`. This makes Sunday (0) stay put, Monday (1) go back 1 day, Saturday (6) go back 6 days — all landing on the most recent Sunday.

### `server/index.ts` — setInterval calls in startup (maybeRunWeeklyDigest, maybeRunAiUsagePurge)
**Issue:** The return values of `setInterval` for both schedulers are discarded. There is no graceful shutdown handler (`SIGTERM`/`SIGINT`) that calls `clearInterval` on them or waits for in-flight jobs to finish before `process.exit`. Under Replit's container lifecycle or any orchestrator, the process receives SIGTERM before a hard kill; without cleanup the intervals keep firing during the drain window and the job worker may corrupt in-progress DB writes.

**Fix:** Capture each interval: `const weeklyTimer = setInterval(...)` and `const purgeTimer = setInterval(...)`. Add `process.on('SIGTERM', async () => { clearInterval(weeklyTimer); clearInterval(purgeTimer); await httpServer.close(); process.exit(0); });` and mirror for `SIGINT`. If `startJobWorker` returns a stop function, call it here too.

### `server/index.ts` — res.json monkey-patch in request logging middleware
**Issue:** `capturedJsonResponse` stores the full response body and `JSON.stringify(capturedJsonResponse)` is appended to every `/api` log line unconditionally. For endpoints that return user quiz results, lesson progress, or auth session data this will emit PII and potentially JWT/session tokens to stdout in plain text, violating data-minimisation requirements for a school-aged audience.

**Fix:** Add a denylist of sensitive path prefixes and strip or truncate the body for them: `if (path.startsWith('/api') && !path.startsWith('/api/auth') && !path.startsWith('/api/user')) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`.slice(0, 200); }`. Better still, log only `res.statusCode` and response size (`res.getHeader('content-length')`) rather than the body.

## 🟡 Warnings (12)

- **`server/routes.ts` (Supabase init chain (~line 118-126)):** The seven-step Supabase init chain runs fire-and-forget with only a terminal `.catch`. If any intermediate step fails (e.g. `seedFinancialAcademyLesson`), the subsequent seeds silently never run, leaving the DB in a partially initialised state with no structured error reporting per step.
  → *Run independent seeds in parallel where possible and wrap the whole block with per-step error visibility: `Promise.all([initSupabaseTables().then(() => trimOrgNamesInSupabase()).then(() => seedFinancialAcademyLesson()), initStaticContentTables().then(() => seedStaticModules()).then(() => seedRegionalContent()).then(() => seedGameContent())]).catch(e => { console.error('[Supabase] Init error:', e); captureException(e); });` Also verify which steps truly depend on each other before parallelising.*
- **`server/routes.ts` (/api/health (~line 30) and /healthz (~line 78)):** There are two separate health-check endpoints (`/api/health` and `/healthz`) with different behaviour: the first returns no DB check, the second does. Railway and uptime monitors may be pointed at the wrong one, giving a false-positive healthy signal even when the database is down.
  → *Consolidate to a single `/healthz` endpoint that performs the DB ping. Remove `/api/health` or have it redirect to `/healthz`. Document which endpoint Railway and each monitor should use.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject` without sanitisation. A path traversal payload like `../../private/keys` could reach unintended storage paths depending on how the ObjectStorageService resolves keys.
  → *Normalise and validate the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (!filePath || filePath.startsWith('..')) return res.status(400).json({ error: 'Invalid path' });` Also ensure ObjectStorageService itself never resolves outside its configured bucket prefix.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time in a sequential await loop. For a 40-question paper this is 40 serial round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy far longer than needed.
  → *Replace the for-loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(questionRows) once. If storage.createExtractedQuestion must be used, build the array and call Promise.all() — but a single bulk SQL insert is strongly preferred for atomicity and performance.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete before re-processing):** The idempotency guard deletes previously extracted questions unconditionally on every attempt, including the first. If a concurrent duplicate job starts between the delete and the final insert (e.g. due to at-least-once delivery), both jobs will race and the question table will end up with doubled or partial rows.
  → *Add a database-level unique constraint on (paperId, order) in extractedQuestions. Use INSERT ... ON CONFLICT DO NOTHING (drizzle: .onConflictDoNothing()) instead of a pre-delete wipe. Also set the exam paper status to 'processing' atomically only if it is currently 'pending', so a second concurrent job can detect it is already claimed and exit early.*
- **`server/jobHandlers.ts` (parsePrivatePath / uploadPrivateObject):** PRIVATE_OBJECT_DIR is checked at call time but there is no validation that the resulting relPath stays within the intended directory. A job payload with a crafted relPath containing '../' sequences could escape the configured root, writing objects to unintended bucket paths (path traversal).
  → *After constructing the full path, assert that it starts with the resolved root: if (!full.startsWith(resolvedRoot)) throw new Error('Path traversal detected'). Also normalize both paths with path.posix.resolve() before comparison rather than string concatenation.*
- **`server/jobHandlers.ts` (admin-csv-export handler (truncated section)):** Based on the visible pattern, the CSV export job likely fetches entire tables (users, transactions, etc.) into memory as JS arrays before serializing. For a growing Caribbean school deployment with tens of thousands of transaction rows this will cause heap spikes and potential OOM kills in the job worker process.
  → *Stream rows from the database using a cursor or paginated query (fetch 500 rows at a time), pipe them through a streaming CSV serializer (e.g. fast-csv in transform/stream mode), and write chunks incrementally to object storage using a resumable upload or a writable stream. This keeps memory usage O(chunk size) rather than O(table size).*
- **`shared/schema.ts` (transactions and budgets createdAt columns):** createdAt uses .defaultNow() but is missing .notNull(). If the default is ever bypassed (bulk insert, migration, raw SQL), NULL rows are accepted silently and will break any ORDER BY createdAt or date-range query.
  → *Change all createdAt definitions to: createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same fix to linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate has an index missing. The primary query for bill reminders is 'fetch all due before date X for user Y', but there is no index on (userId, nextDueDate), forcing a full scan of all reminders for every due-date check.
  → *Add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders table definition.*
- **`shared/schema.ts` (categories table, userId column):** categories.userId is a nullable FK used to distinguish system categories (null) from user categories (non-null), but there is no index on userId. Every query like 'fetch categories for this user OR system defaults' will scan all categories, including all system rows, as category count grows.
  → *Add a partial or full index: (t) => ({ userIdIdx: index('idx_categories_user_id').on(t.userId) }). For Postgres specifically, a partial index WHERE user_id IS NOT NULL via raw SQL migration will be even more efficient for user-scoped lookups.*
- **`server/index.ts` (initSentry() call before imports complete):** `initSentry()` is called synchronously at module evaluation time, before `registerRoutes`, `seedDatabase`, or `startJobWorker` are imported. If Sentry initialisation itself throws (e.g. invalid DSN env var), the error is uncaught at the module level and produces an unhelpful `Cannot find module` style crash rather than a clear startup failure message.
  → *Wrap the call: `try { initSentry(); } catch (e) { console.error('[startup] Sentry init failed:', e); }` and move it inside the async `main()` function after environment validation, so startup errors are handled uniformly alongside the SESSION_SECRET guard.*
- **`server/index.ts` (seedDatabase and probeDatabase in startup sequence):** Based on the visible code, `seedDatabase()` and `probeDatabase()` are likely awaited sequentially during startup. If both are independent (probe just checks connectivity, seed inserts reference data that does not depend on probe's return value), running them sequentially adds unnecessary latency to cold-start time — critical for a Replit environment that cold-starts frequently.
  → *Run them concurrently: `await Promise.all([probeDatabase(), seedDatabase()]);`. Ensure `seedDatabase` internally handles the case where the DB is not yet reachable by retrying, or move it after `probeDatabase` resolves if there is a hard dependency.*

## 💡 Opportunities

### Startup environment validation — 🔥 High Impact
Add a validate-env step at process start (e.g. with `zod` or `envalid`) that asserts APP_BASE_URL, DATABASE_URL, OPENAI_API_KEY and other critical vars are present before any route is registered. This eliminates an entire class of silent misconfiguration bugs in Railway deploys and makes the host-header fix straightforward.

### Parallel domain-route registration — ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six serial awaits. If any registration function is async and does I/O (e.g. DB schema checks), this adds unnecessary cold-start latency. Audit each register function; wrap those that are safe to parallelise in a second Promise.all.

### Structured startup logging for Supabase seeding — ⚡ Medium Impact
Replace console.error-only seed reporting with structured log entries (including step name, duration, and whether it was a no-op vs first-time seed). This makes it possible to diagnose partial-seed incidents in Railway log drains without a full redeploy, and pairs well with the Sentry integration already present.

### Parallel AI calls for text extraction and question parsing — ⚡ Medium Impact
In runExtractPaper the vision/PDF extraction step and the subsequent GPT-4o question-parsing step are necessarily sequential, but any per-subject metadata lookups or paper status pre-checks done before or after could be parallelised with Promise.all(). More impactfully, if the system ever processes multiple papers, the job queue concurrency limit should be tuned (currently unknown) to allow parallel workers rather than relying on a single worker processing one job at a time.

### Centralised job payload schema validation with Zod — ⚡ Medium Impact
Every handler destructures job.payload with no runtime type checking. Adding a per-handler Zod schema that validates payload on entry would catch malformed enqueue calls immediately (throwing a non-retryable error) rather than producing cryptic failures mid-handler. This also self-documents the expected shape of each job type and prevents the path-traversal class of issues at the boundary.

### Redact sensitive fields from job payload before persistence — 📌 Low Impact
Even after moving fileB64 to object storage, job payloads may contain user IDs, email addresses, or export filter parameters that are stored in plaintext in the jobs table and potentially surfaced in admin UIs or logs. Introduce a payload sanitiser at enqueue time that strips or encrypts PII fields, and ensure job error messages logged on failure do not echo the full payload.

### Consolidate repeated currency default into a shared constant — ⚡ Medium Impact
The string literal 'BSD' is hardcoded as a default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' in a config module and reference it in each .default() call. This prevents silent drift if the app expands to JMD or TTD and makes currency policy a single change point.

### Add a unique constraint on (userId, categoryId, period) for budgets — 🔥 High Impact
Currently a user can have multiple active budgets for the same category and period, which will produce double-counting bugs in spending-vs-budget calculations. A uniqueIndex on (userId, categoryId, period) enforces the business rule at the DB level and eliminates the need for application-layer deduplication guards.

### Replace serial PKs with UUIDs on user-facing tables — 🔥 High Impact
serial integer PKs on transactions, savingsGoals, and portfolioHoldings expose sequential IDs in URLs and API responses, allowing users to enumerate other users' records if any authorization check is ever missed. Switching to gen_random_uuid() (pgTable with uuid type) eliminates enumeration risk and aligns with the app's school-aged audience privacy requirements.

### Centralised async main() with unified error boundary — 🔥 High Impact
Wrapping all startup logic in a single `async function main()` with a top-level `.catch(e => { captureError(e); process.exit(1); })` eliminates the scattered try/catch blocks and ensures any unhandled rejection during boot (DB probe, seed, route registration) surfaces with a consistent log format and Sentry capture rather than a silent hang.

### Scheduler deduplication via DB lock instead of per-process interval — ⚡ Medium Impact
Running setInterval in every dyno/worker process means the DB conflict guard in `maybeRunWeeklyDigest` is the only thing preventing duplicate job enqueues. Replacing the interval with a single dedicated scheduler process (or a pg_cron / BullMQ delayed job) removes the race entirely, reduces DB write pressure, and eliminates the leaked-interval problem structurally rather than requiring per-deploy cleanup code.

### Response body logging via structured log levels with sampling — ⚡ Medium Impact
Replacing the monkey-patched `res.json` with a structured logger (e.g. pino) that emits JSON log records — and only samples full response bodies at `debug` level, disabled in production — would cut log volume significantly, remove PII risk, and make logs machine-parseable for Replit's log drain without changing any route code.

### API Routes (`server/routes.ts`)
The route registration code is generally well-structured with good use of Promise.all for parallel imports and appropriate rate limiting on auth endpoints. Key concerns are: unhandled promise rejections on the Supabase init chain running fire-and-forget, missing rate limiting on AI/chat/image/audio routes which is critical for a school-age user app with OpenAI cost exposure, and host header injection risk in robots.txt and sitemap generation.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance concerns: large base64 payloads are being stored in the job table (memory/DB pressure), sequential question inserts inside a loop should be batched, and the fileB64 field is logged or persisted in job payload without sanitization. Error boundaries are mostly present but idempotency is incomplete for the CSV export handler. Overall structure is reasonable but the data-in-job-payload anti-pattern is the most serious systemic issue.

### Database Schema (`shared/schema.ts`)
The schema is structurally sound with good use of foreign keys and enums, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency filter columns, nullable timestamps that should be NOT NULL, and an unindexed ticker column on simulated_stocks that will cause full table scans.

### Server Startup (`server/index.ts`)
The server startup mixes blocking synchronous initialization with unguarded async schedulers, and intervals created during startup are never stored or cleared on shutdown, causing memory leaks and preventing clean process termination. Response body logging unconditionally serializes every API response payload which risks leaking sensitive financial or PII data in production logs and adds non-trivial per-request overhead.
