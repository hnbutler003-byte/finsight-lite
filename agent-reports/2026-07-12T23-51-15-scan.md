## 🔴 Critical (6)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate SELECT to calculate 'spent' for each budget individually inside a loop, resulting in O(n) database round-trips per user request.

**Fix:** Replace the per-budget spent query with a single aggregated JOIN: SELECT budgets.*, COALESCE(SUM(transactions.amount),0) AS spent FROM budgets LEFT JOIN transactions ON transactions.category_id = budgets.category_id AND transactions.user_id = budgets.user_id AND transactions.type = 'expense' WHERE budgets.user_id = $userId GROUP BY budgets.id. Return the aggregated result set in one query.

### `server/storage.ts`: getPortfolioHoldings
**Issue:** N+1 query pattern: getPortfolioHoldings fetches holdings then calls getStockById(holding.stockId) inside a map/loop, issuing one SELECT per holding. For a class of 30 students each with 10 holdings this is 300 sequential queries.

**Fix:** After fetching holdings, collect unique stockIds into an array and issue a single query using inArray(simulatedStocks.id, stockIds), then build a Map<id, stock> for O(1) lookup when assembling the join result. Eliminate the per-holding getStockById call entirely.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI and media generation routes are registered with no rate limiting applied before or at registration. For a school-aged user app these are the most expensive endpoints (OpenAI API costs, latency) and are trivially abusable without per-IP or per-user throttling.

**Fix:** Add dedicated rate limiters before these registrations, e.g. `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` where aiLimiter is a stricter limiter (e.g. 10 req/min per IP) defined in rateLimiter.ts. Apply before calling registerChatRoutes etc.

### `server/routes.ts`: robots.txt handler (~line 38) and sitemap.xml handler (~line 55)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header to inject an arbitrary domain into the Sitemap directive or sitemap loc tags, enabling cache poisoning or phishing via SEO.

**Fix:** Resolve the base URL once at startup from a required environment variable: `const APP_BASE_URL = process.env.APP_BASE_URL;` (throw if missing). Use this constant in both handlers instead of deriving it from request headers. Remove all header-based URL construction.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by setInterval are not stored, so they cannot be cleared during graceful shutdown. When the process receives SIGTERM the intervals may fire mid-flight, causing partial DB writes or duplicate job enqueues, and the process will not exit cleanly if those async callbacks are pending.

**Fix:** Store both handles: `const weeklyDigestInterval = setInterval(...)` and `const purgeInterval = setInterval(...)`. In the SIGTERM/SIGINT handler, call `clearInterval(weeklyDigestInterval)` and `clearInterval(purgeInterval)` before closing the HTTP server and DB pool.

---

### `server/index.ts`: res.json middleware / logLine construction in app.use finish handler
**Issue:** Every API response body is captured via monkey-patched res.json and then serialized in full to console logs with `JSON.stringify(capturedJsonResponse)`. For a school-aged financial literacy app this will log quiz scores, user session data, AI responses, and any PII returned by the API in plaintext logs, violating basic data minimisation principles.

**Fix:** Either remove response body logging entirely in production (`if (process.env.NODE_ENV !== 'production')`), or whitelist only non-sensitive metadata fields (e.g., log only `{ status, recordCount }` where the route explicitly annotates the response). Also cap log line length to prevent log flooding from large payloads.

## 🟡 Warnings (12)

- **`server/storage.ts` (getClassProgressSummary):** Missing pagination guard: the opts.limit parameter has no enforced maximum, so a caller passing limit=100000 or omitting the parameter entirely will load every enrollment row with full learning progress into memory, which is dangerous for large school deployments.
  → *Add a hard cap: const safeLimit = Math.min(opts?.limit ?? 25, 100); const safeOffset = opts?.offset ?? 0; and reject or clamp values beyond the cap. Document the max in the IStorage interface signature.*
- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices updates each stock price with a separate UPDATE inside a loop and is called on an interval with no cleanup reference exported. This creates both N individual writes per tick and a potential memory/connection leak if the module is re-imported in tests or hot-reloaded.
  → *Batch the price updates into a single parameterized SQL CASE statement or use db.transaction() to wrap all UPDATEs in one round-trip. Export the interval handle (e.g. export const priceSimInterval = setInterval(...)) so callers can call clearInterval on shutdown or in test teardown.*
- **`server/storage.ts` (getDashboardStats):** getDashboardStats runs multiple independent aggregate queries sequentially (total income, total expenses, savings goals, recent transactions) with separate awaits, adding latency proportional to the number of queries times database round-trip time.
  → *Wrap all independent queries in Promise.all: const [incomeResult, expenseResult, goalsResult, recentTxResult] = await Promise.all([incomeQuery, expenseQuery, goalsQuery, recentTxQuery]); This parallelises the fetches and cuts wall-clock time to the slowest single query.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** The seven Supabase seed steps are chained sequentially with .then(). If any intermediate step (e.g. trimOrgNamesInSupabase) is slow or fails, all subsequent seeds are blocked or silently skipped. seedStaticModules, seedRegionalContent, and seedGameContent are independent and can run concurrently.
  → *Run independent seeds in parallel: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await seedFinancialAcademyLesson(); await initStaticContentTables(); await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()]);` — wrap in an async IIFE with a .catch so the non-blocking intent is preserved.*
- **`server/routes.ts` (/api/health (~line 29) and /healthz (~line 73)):** There are two health check endpoints (`/api/health` returning trivial JSON and `/healthz` doing a real DB probe) with no documentation distinguishing them. The shallow `/api/health` is likely what Railway's health check hits, meaning the app can report healthy while the DB is down. This is a reliability gap for a financial app.
  → *Consolidate to a single `/api/health` endpoint that performs the `SELECT 1` DB probe (as /healthz already does) and returns 503 on failure. Remove the trivial `/api/health` stub. Update Railway's health check path to use this single endpoint.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without sanitization. A path traversal payload like `../../private/secrets` could resolve to unintended storage keys depending on the ObjectStorageService implementation.
  → *Sanitize the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|^\//, '');` and reject any path containing `..` segments after normalization with a 400 response. Import `path` from Node core.*
- **`shared/schema.ts` (categories table definition):** categories.userId has no index, yet it is used as a FK reference and will be queried to filter user-specific vs system categories. Without an index, every category lookup by userId causes a full table scan.
  → *Add an index inside the pgTable call: `(t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) })`. Also add a partial index in a raw migration for WHERE user_id IS NOT NULL to keep system-category lookups cheap.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these tables define indexes on userId even though every real-world query will filter by userId. Missing indexes on high-cardinality FK columns will cause sequential scans as user data grows.
  → *Add `(t) => ({ userIdx: index('idx_<table>_user_id').on(t.userId) })` to each table. For billReminders also index nextDueDate since reminders are queried by due date. For portfolioHoldings add a unique composite index on (userId, stockId) to prevent duplicate holding rows.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt is defined with `.defaultNow()` but without `.notNull()`. Drizzle will infer the column as nullable, meaning application code or a bad insert can store NULL and break any ORDER BY createdAt or audit logic silently.
  → *Append `.notNull()` to every createdAt definition, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. The default still handles inserts automatically; notNull just prevents explicit NULL writes.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as `text` while priceUpdatedAt directly below it is a `timestamp`. Having two columns tracking price update time with different types is duplicate/dead data and the text column cannot be used for date arithmetic or range queries without casting, making it a correctness and performance hazard.
  → *Remove lastPriceUpdateDate entirely and rely solely on priceUpdatedAt (timestamp). Update any application code reading lastPriceUpdateDate to use priceUpdatedAt. If a formatted string is needed for display, format it in the application layer.*
- **`server/index.ts` (probeDatabase / seedDatabase calls in startup IIFE):** probeDatabase and seedDatabase are awaited sequentially during startup. If these are independent operations, startup is unnecessarily serialized. More critically, if probeDatabase does synchronous I/O or heavy computation before the first await, it blocks the Node.js event loop and delays the HTTP server from accepting connections.
  → *Run them concurrently with `await Promise.all([probeDatabase(), seedDatabase()])` if they have no dependency on each other. Confirm probeDatabase yields to the event loop immediately (first statement should be an await on a DB query, not sync work).*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince` as `(dow + 0) % 7`, where `+ 0` is a no-op. Sunday (dow=0) yields daysSince=0, so the function returns today when today is Sunday and it is before 7 pm — but then the `d.getTime() > now.getTime()` guard subtracts 7 days, making it return the previous Sunday. This means on Sunday before 7 pm the digest is correctly deferred, but the dead `+ 0` suggests a copy-paste error that may have masked an off-by-one for other target days (e.g., originally targeting a different weekday).
  → *Remove the `+ 0` dead operand and add a unit test covering: Sunday before 7 pm (should return previous Sunday 7 pm), Sunday after 7 pm (should return today 7 pm), and mid-week days. Confirm the intended target day is Sunday and document it with a named constant.*

## 🔵 Info / Tech Debt (2)

- **`shared/schema.ts` (userId varchar columns across all tables (categories, transactions, budgets, linkedCards, etc.)):** All userId columns are declared as `varchar('user_id')` with no length argument, which defaults to unlimited length in PostgreSQL. This is inconsistent with typical auth provider IDs (e.g. Clerk/Supabase UIDs are ≤255 chars) and prevents the DB from enforcing a reasonable bound, wasting storage and index efficiency.
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and hardcoded scheduler intervals):** The weekly digest and AI purge setInterval periods are hardcoded literals (e.g., every 10 or 60 minutes) buried in the source rather than being drawn from environment config. This forces a redeploy to tune scheduler frequency in production and makes the values invisible to ops.

## 💡 Opportunities

### Wrap multi-step writes in db.transaction(): 🔥 High Impact
Operations like enrollStudent (insert enrollment + update class count) and completeModule (upsert progress + award XP + conditionally insert badge) span multiple tables with no transaction boundary. A partial failure leaves data inconsistent. Wrapping each in db.transaction(async (tx) => { ... }) gives atomicity with minimal refactor effort and eliminates an entire class of data integrity bugs.

### Move hardcoded virtual balance seed amount to environment config: ⚡ Medium Impact
The initial virtual balance value (likely 10000 or similar) and stock price volatility multipliers appear to be hardcoded constants in the storage file. These should be read from process.env.VIRTUAL_BALANCE_SEED and process.env.STOCK_VOLATILITY_FACTOR so sponsors or school admins can configure them without a code deploy.

### Extract per-entity data access into focused repository modules: ⚡ Medium Impact
The single IStorage interface and DatabaseStorage class cover 60+ methods across users, budgets, investments, learning, gamification, and school admin. Splitting into InvestmentRepository, LearningRepository, ClassroomRepository etc. would reduce merge conflicts, make unit testing individual domains feasible without mocking the entire interface, and cut the file to a manageable size.

### Startup environment validation: 🔥 High Impact
Add a startup guard that validates all required environment variables (APP_BASE_URL, Supabase credentials, OpenAI key, session secret) before any route registration begins. This prevents silent misconfiguration in production where missing env vars cause runtime failures deep in request handling rather than at boot, which is especially important for a financially-sensitive app.

### Request ID middleware for distributed tracing: ⚡ Medium Impact
Add `express-request-id` or a small custom middleware early in the chain to attach a UUID to each request and include it in error logs and response headers (`X-Request-Id`). Currently console.error calls in route handlers (e.g. public-objects) log no correlation ID, making prod debugging across Sentry and Railway logs very slow.

### Typed domain route registration interface: 📌 Low Impact
The six domain routers are imported and called with no shared contract — each `registerXRoutes(app)` could silently accept wrong signatures. Define and export a `RouteRegistrar = (app: Express) => Promise<void>` type in a shared types file and enforce it at each call site. This prevents future route modules from breaking the parallel import pattern.

### Partial index for active bill reminders: ⚡ Medium Impact
billReminders is queried most often for upcoming/active reminders. A partial index `CREATE INDEX idx_bill_reminders_upcoming ON bill_reminders (user_id, next_due_date) WHERE next_due_date >= NOW()` would make dashboard reminder lookups O(log n) on only relevant rows instead of full table, reducing query time significantly as reminder history grows.

### Composite unique constraint on portfolioHoldings (userId, stockId): 🔥 High Impact
Without a unique constraint on (userId, stockId), concurrent buy operations can insert duplicate holding rows for the same user+stock, causing incorrect quantity and avgPurchasePrice calculations. Adding `uniqueIndex('uq_holdings_user_stock').on(t.userId, t.stockId)` enforces correctness at the DB level and eliminates the need for defensive upsert logic in the application layer.

### Consolidate currency default into a shared enum or config constant: 📌 Low Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table columns (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, etc.). If the app expands to support TTD or JMD as defaults, every column default must be changed individually. Extract a `DEFAULT_CURRENCY = 'BSD'` constant in a shared config and reference it, or use a PostgreSQL domain type, so currency policy is changed in one place.

### Structured JSON logging with redaction: 🔥 High Impact
Replace ad-hoc console.log string concatenation with a structured logger (e.g., pino) configured with a redact list covering fields like `password`, `token`, `sessionId`, `userId`. This eliminates the PII-in-logs risk systematically, gives machine-parseable logs for Sentry/Datadog ingestion, and adds log levels so debug verbosity can be toggled without code changes.

### Centralised graceful shutdown orchestrator: 🔥 High Impact
Introduce a single shutdown() function registered for SIGTERM and SIGINT that sequences: stop accepting new requests (server.close), clear all intervals, wait for in-flight job worker to drain, then close the DB pool. Currently shutdown logic is either absent or scattered. A clean shutdown prevents duplicate job enqueues and DB connection leaks during Replit container restarts, which are frequent in that hosting environment.

### Parallel startup with readiness gate: ⚡ Medium Impact
Move seedDatabase behind a `NODE_ENV !== 'production'` guard (seeding in production is a data-integrity risk) and run the remaining startup tasks — DB probe, route registration, job worker start — with coordinated Promise.all where dependencies allow. This can cut cold-start time noticeably and makes the dependency graph explicit, reducing the risk of future developers inadvertently adding more sequential awaits.
