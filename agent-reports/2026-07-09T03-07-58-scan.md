## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 106-108)
**Issue:** AI-backed routes (chat, image, audio) are registered with no rate limiting middleware. In a school-aged user app these endpoints call external paid APIs and are trivially abusable, causing both cost exposure and denial-of-service risk.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP using express-rate-limit) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 44) and sitemap handler (~line 57)
**Issue:** The `host` and `proto` values are taken directly from `x-forwarded-host` and `x-forwarded-proto` headers and interpolated into response bodies without validation. A spoofed Host header will cause the app to emit attacker-controlled URLs in robots.txt and the sitemap, enabling cache-poisoning or open-redirect phishing.

**Fix:** Add an allowlist check: define `const ALLOWED_HOST = process.env.PUBLIC_HOST` (e.g. 'finsightlite.com') and fall back to it when the header value does not match. Example: `const host = ALLOWED_HOSTS.includes(req.headers['x-forwarded-host']) ? req.headers['x-forwarded-host'] : process.env.PUBLIC_HOST;`

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** Five sequential awaits (getUserXp, txCount query, gameCount query, lessonRows query, badgeRows query) fire one after another for every user in the weekly-digest fan-out. With hundreds of students this serialises hundreds of independent DB round-trips per user before moving to the next user.

**Fix:** Replace the five sequential awaits inside studentWeekStats with a single Promise.all: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...txCount), db.select(...gameCount), db.select(...lessonRows), db.select(...badgeRows)]). This cuts per-user latency to the slowest single query instead of the sum of all five.

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** Inside bootstrapStudentContacts, for each student the code does a SELECT to check for an existing emailContacts row and then conditionally an INSERT, sequentially inside a for-loop with no transaction or ON CONFLICT guard. Concurrent weekly-digest jobs (or a retry after a partial failure) will race and insert duplicate contacts, and the loop will stall on every DB round-trip.

**Fix:** Use an INSERT ... ON CONFLICT (user_kind, user_id) DO NOTHING statement via Drizzle's onConflictDoNothing(), and batch the inserts outside the loop. This makes the bootstrap idempotent and reduces N sequential round-trips to a single bulk upsert.

### `shared/schema.ts`: transactions.createdAt, budgets.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioTransactions.executedAt
**Issue:** All audit timestamp columns use .defaultNow() but omit .notNull(). A missing NOT NULL means application code or a misconfigured INSERT can store NULL for createdAt, breaking any time-series query, audit log, or ORDER BY createdAt DESC that assumes the column is always populated.

**Fix:** Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Drizzle supports this. Apply consistently to createdAt on all tables and executedAt on portfolioTransactions.

### `server/index.ts`: maybeRunWeeklyDigest / maybeRunAiUsagePurge interval registration
**Issue:** setInterval handles for the weekly digest and AI usage purge schedulers are never captured, so they cannot be cleared during graceful shutdown. The intervals will fire against a half-torn-down process (closed DB pool, unregistered handlers), causing unhandled rejections and potential DB connection errors after SIGTERM.

**Fix:** Assign each setInterval call to a module-level variable (e.g. `let digestInterval: NodeJS.Timeout`), then call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` inside the SIGTERM/SIGINT shutdown handler before closing the HTTP server and DB pool.

---

### `server/index.ts`: capturedJsonResponse log line / res.json patch
**Issue:** Every API response body is logged to stdout via `JSON.stringify(capturedJsonResponse)` with no field redaction. For a financial literacy app with school-aged users, responses that include session tokens, user PII, or financial records will appear in plain text in any log aggregator that ingests stdout (e.g. Replit logs, Papertrail).

**Fix:** Either disable response-body logging entirely in production (`NODE_ENV === 'production'`), or introduce a denylist of top-level keys to redact (e.g. `token`, `password`, `email`, `dob`) before stringifying. Confirm with the team what the minimum necessary log data is and default to logging only status codes and durations.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage (POST /api/admin/help-chat)
**Issue:** The full conversation history (`messages: next`) is sent to the server on every message with no length cap. A user who keeps chatting will send an ever-growing payload, eventually hitting request size limits or causing runaway token costs on the LLM backend. There is also no auth token attached, so the endpoint is only as secure as the server-side session cookie—if that check is missing server-side, any user can hit the admin AI endpoint.

**Fix:** Slice the history before sending: `messages: next.slice(-20)` to cap context at 20 turns. Confirm server-side that `/api/admin/help-chat` runs the same `requireAdminAuth` middleware used by other admin routes. Add a comment asserting this so future reviewers don't remove it.

## 🟡 Warnings (17)

- **`server/routes.ts` (Supabase init chain (~line 111-118)):** The seven Supabase init/seed functions are chained with sequential `.then()` calls. Each waits for the previous to complete before starting, adding cumulative latency to server startup. Several of these (e.g. seedStaticModules, seedRegionalContent, seedGameContent) are likely independent and safe to run concurrently.
  → *Split into a sequential prerequisite group and a parallel seed group: `await initSupabaseTables(); await trimOrgNamesInSupabase(); await initStaticContentTables();` then `await Promise.all([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]);` wrapped in a single `.catch`.*
- **`server/routes.ts` (/healthz handler (~line 82)):** The `/healthz` endpoint dynamically imports `./db` and `drizzle-orm` on every request. Dynamic imports are cached after the first call but the import() call itself still incurs async overhead and obscures the dependency. More critically, this endpoint has no rate limiting, so it can be used to hammer the DB connection.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. Apply a loose rate limiter (e.g. 10 req/min) to `/healthz` and `/api/health` to prevent DB probing.*
- **`server/routes.ts` (/public-objects/:filePath handler (~line 95)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject()` without sanitisation. A path like `../../private/secret` could traverse into unintended storage buckets depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|\.\.\//g, '');` and reject requests where the normalised path differs significantly from the raw input, or explicitly prefix-constrain it to known buckets (logos/, videos/, etc.).*
- **`server/jobHandlers.ts` (admin-csv-export handler):** The entire export dataset (e.g. all students or transactions) is fetched into a single in-memory array, stringified into a CSV string, then converted to a Buffer before upload. For large schools this can exhaust the Node.js heap and block the event loop during string construction.
  → *Stream rows from the DB using a cursor or limit/offset pagination, pipe them through a Transform stream that writes CSV chunks, and use objectStorageClient's resumable streaming upload (set resumable: true and pipe the readable stream directly). Remove the Buffer.from(csv) materialisation entirely.*
- **`server/jobHandlers.ts` (registerJobHandler (all handlers)):** No handler wraps its body in a try/catch. If any handler throws synchronously or rejects (e.g. db connection drop, objectStorageClient error), the error propagates to the job runner uncaught. Depending on the job runner implementation this may crash the process or silently mark the job failed with no structured error payload returned.
  → *Wrap each handler body in try/catch and return a structured error object, e.g. { ok: false, error: err.message }, or confirm the job runner's registerJobHandler already guarantees catch semantics and documents that contract. At minimum add a top-level process.on('unhandledRejection') log with job context for observability.*
- **`server/jobHandlers.ts` (weekly-digest handler, bootstrapStudentContacts fallback):** When orgId is falsy, the code falls back to db.select().from(users) with no WHERE clause, no limit, and no column projection — fetching every column of every user row in the entire database into memory to extract only id, email, and username.
  → *Replace with db.select({ id: users.id, email: users.email, username: users.username }).from(users) and add a reasonable LIMIT or pagination guard. If a full-table scan is genuinely intended, add an explicit comment and a MAX_ROWS environment config guard that aborts with an error if exceeded.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index, yet budget lookups will always filter by userId and frequently join on categoryId. At scale this causes full table scans per user dashboard load.
  → *Add a composite index in the budgets table factory function: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This mirrors the pattern already used on transactions.*
- **`shared/schema.ts` (varchar('user_id') across all tables (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** All userId columns are declared as varchar() with no length argument. In PostgreSQL this is equivalent to text, which is fine for storage but signals no schema contract. More critically, the categories.userId column has no NOT NULL constraint, which is intentional for system categories, but there is no check constraint or partial unique index to prevent duplicate system categories per name/type pair.
  → *For user-owned tables, change to varchar('user_id', { length: 128 }).notNull() to match the actual auth provider ID format and enforce the constraint at the DB layer. For categories, add a uniqueIndex on (name, type) where userId IS NULL to prevent duplicate system categories: uniqueIndex('idx_categories_system_unique').on(t.name, t.type) with a partial condition.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text() instead of timestamp(). Storing a date as text prevents range queries, ORDER BY date correctness, and index efficiency. Any query filtering stocks by price freshness will either do string comparison (wrong) or require a CAST (index-defeating).
  → *Change to: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed data or insert schemas that currently pass a string. The drizzle-zod insert schema will then correctly coerce or reject non-date inputs.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** portfolioHoldings has no index on userId or stockId, and portfolioTransactions has no index on userId or executedAt. Every portfolio summary query (aggregate holdings, transaction history ordered by date) will do a full scan. These are the hottest read paths in an investment simulation feature.
  → *Add to portfolioHoldings: (t) => ({ userStockIdx: index('idx_holdings_user_stock').on(t.userId, t.stockId) }). Add to portfolioTransactions: (t) => ({ userDateIdx: index('idx_ptxn_user_date').on(t.userId, t.executedAt) }). Also consider a uniqueIndex on portfolioHoldings(userId, stockId) if the business rule is one holding row per user per stock.*
- **`server/index.ts` (seedDatabase / probeDatabase calls at startup (top-level await block)):** seedDatabase and probeDatabase are awaited sequentially before the HTTP server begins listening. If either call is slow (cold DB connection, migration backlog), the process appears unresponsive to the platform health-checker and risks being killed and restarted in a loop. They are also independent and can run concurrently.
  → *Run them in parallel with `await Promise.all([probeDatabase(), seedDatabase()])`. Additionally, start the HTTP server listening immediately after the express app is configured, and defer the probe/seed work so the port is bound before any long-running startup I/O.*
- **`server/index.ts` (res.json middleware / log line ~74):** The request-logging middleware monkey-patches `res.json` and stores the full parsed response body in `capturedJsonResponse`, then calls `JSON.stringify` on it again inside the `finish` handler for every API response. For large paginated responses or bulk data endpoints this doubles serialisation cost and holds the entire response object in closure memory until the request finishes.
  → *Cap the logged payload: after `JSON.stringify`, slice the string to a maximum length (e.g. 500 chars) and append `…` if truncated. For non-debug environments, gate the body capture behind `process.env.LOG_RESPONSE_BODIES === 'true'` so it is off by default in production.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — adding zero is a no-op, making the Sunday rollback logic incorrect. When `dow === 0` (Sunday) `daysSince` is 0, so `d` remains today. The subsequent guard `if (d.getTime() > now.getTime())` only subtracts 7 days if the computed 7 pm is in the future, which is correct for times after 7 pm but silently wrong for a Sunday before 7 pm where the intent is still to target the current day. More critically the dead `+ 0` suggests a copy-paste error that may have dropped an intended offset (e.g. `(dow + 1) % 7` for a Monday-anchored week).
  → *Clarify and test the intended anchor day with explicit unit tests. If Sunday is correct, remove the `+ 0` and document why `dow === 0` resolves to `daysSince = 0`. If a different anchor was intended, apply the correct offset. Add a Jest test covering Sunday before 7 pm, Sunday after 7 pm, and mid-week dates.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage):** The chat scroll anchor uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` directly in JSX. This creates a new function on every render, calls scrollIntoView during every re-render (including mid-typing keystrokes), and will fire even when the user has manually scrolled up to read history.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This scopes scrolling only to actual message list changes.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard data fetching (useQuery calls for users, schools, lessons, etc.)):** Multiple independent `useQuery` calls fire sequentially on mount and each triggers its own loading/error state, but the visible UI appears to only guard one of them. If secondary queries (e.g. schools, lessons) fail silently, the dashboard renders with empty data and no user-visible error, which for a financial literacy platform means admins may act on incomplete information.
  → *Add explicit `isError` checks for every query that feeds a rendered table or stat card. Show a `<Alert variant='destructive'>` when any critical query fails. Consider a shared error boundary wrapping the dashboard sections. At minimum: `if (usersError) return <ErrorState message='Failed to load users' />;`*
- **`client/src/pages/AdminDashboard.tsx` (User/school list filter and sort logic (inline in render body)):** Filter and sort operations over the full users and schools arrays are computed directly in the render body without `useMemo`. Because AdminDashboard holds many pieces of state (search text, pagination, dialog open states, active tab), any state update—including typing a single character in an unrelated field—re-runs the full filter/sort over potentially thousands of records.
  → *Wrap each derived list in `useMemo`: `const filteredUsers = useMemo(() => users?.filter(u => u.name.toLowerCase().includes(search.toLowerCase())).sort(...), [users, search, sortField, sortDir]);`. Do the same for schools and any other derived collections. This makes re-computation depend only on relevant inputs.*
- **`client/src/pages/AdminDashboard.tsx` (Mutation handlers and dialog submit callbacks defined inline in JSX):** Large async mutation handlers (create user, delete school, update lesson, etc.) are defined as inline arrow functions directly in JSX props like `onClick={async () => { ... }}`. These recreate on every render, defeat React's bailout optimizations for child components that receive them as props, and make the file extremely hard to test or audit individually.
  → *Extract each handler to a `useCallback` at the component scope level: `const handleCreateUser = useCallback(async () => { try { await createUserMutation.mutateAsync(form); } catch (e) { toast({ ... }); } }, [form, createUserMutation, toast]);`. Apply the same pattern to delete and update handlers. This also ensures try/catch is consistently present—several inline handlers in the truncated code are missing it.*

## 💡 Opportunities

### Centralise base-URL construction: ⚡ Medium Impact
The proto+host derivation logic is duplicated identically in both the robots.txt and sitemap handlers. Extract it into a `getBaseUrl(req)` helper in a shared utils module. This removes the duplication and means any future security fix (like the host-header allowlist) only needs to be applied once.

### Add authentication guard to object-storage endpoint: ⚡ Medium Impact
The `/public-objects/:filePath` route is intended for public assets but currently applies no content-type validation or path allowlisting. Adding an explicit allowlist of permitted subdirectories (logos, videos) and Content-Type mapping will prevent serving arbitrary stored files and reduce information-disclosure risk for a school-aged audience.

### Introduce startup health-gate before accepting traffic: 📌 Low Impact
Domain routers are registered with `await` but Supabase seeding is deliberately non-blocking. If seeding fails silently, endpoints that depend on seeded data will return errors for real users. Emitting a structured startup-complete log event with seed status (success/skipped/failed per step) would make incidents diagnosable without needing to grep raw error logs.

### Batch per-user digest enqueues into a single bulk insert: 🔥 High Impact
The weekly-digest fan-out calls enqueueJob inside a for-loop, producing one INSERT per user. Replacing this with a bulk enqueue API (or a single multi-row INSERT into the jobs table) would reduce DB round-trips from O(N) to O(1) and cut fan-out time for large cohorts by an order of magnitude.

### Extract hardcoded time constants to environment config: ⚡ Medium Impact
Values like 7 * 24 * 60 * 60 * 1000 (digest window) and the olderThanDays arithmetic in purge-ai-usage are scattered inline. Moving them to named constants or environment variables (DIGEST_WINDOW_DAYS, AI_USAGE_RETENTION_DAYS) makes behaviour configurable without a redeploy and self-documents intent.

### Add idempotency key to send-email job handler: ⚡ Medium Impact
The send-email handler has no deduplication guard. If the job runner retries after a transient failure post-send, users receive duplicate emails. Storing a hash of (to + subject + kind + weekStart) in emailContacts or a sent_emails log table and checking it before calling sendEmail would make the handler safe to retry.

### Centralise currency default via environment config: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract to a DEFAULT_CURRENCY constant sourced from process.env.DEFAULT_CURRENCY with a fallback, then reference it in each .default() call. This lets the app support other Caribbean currencies (TTD, JMD) without a schema migration.

### Add partial index for active bill reminders lookup: ⚡ Medium Impact
billReminders will be queried almost exclusively by (userId, nextDueDate) to surface upcoming bills. Add a composite index on (userId, nextDueDate) and consider a partial index excluding completed/archived reminders if that status is added later. This prevents a full scan every time the dashboard loads upcoming bills for a user.

### Replace serial primary keys with UUID for user-facing IDs: 🔥 High Impact
All tables use serial (auto-increment integer) PKs. For a school-aged user app, sequential integer IDs in URLs or API responses allow trivial enumeration of other users' records (e.g. /transactions/1, /transactions/2). Switching to gen_random_uuid() default UUIDs eliminates enumeration risk without requiring application-level obfuscation, and aligns with the UUID-style userId already coming from the auth provider.

### Structured JSON logging instead of console.log strings: 🔥 High Impact
Replacing the custom `log()` helper and all `console.log` calls with a structured logger (pino or winston with JSON transport) would make log lines machine-parseable, enable log-level filtering (suppressing debug output in production without code changes), and make Sentry breadcrumb correlation easier. Expected impact: faster incident diagnosis and smaller log storage costs.

### Centralise all scheduler intervals behind a SchedulerRegistry: ⚡ Medium Impact
Introduce a lightweight `SchedulerRegistry` class that wraps `setInterval`, stores every handle, and exposes a single `stopAll()` method called during shutdown. This prevents the recurring pattern of untracked intervals as new scheduled jobs are added, and makes it trivial to verify in tests that all timers are cleared.

### Move hardcoded digest audience list to environment/config: 📌 Low Impact
The array `['student', 'teacher', 'guardian']` is hardcoded in `maybeRunWeeklyDigest`. When a new role is added to the schema it requires a code change and redeploy rather than a config update. Extract it to a comma-separated env var (e.g. `DIGEST_AUDIENCES=student,teacher,guardian`) parsed and validated at startup alongside the other env checks.

### Split AdminDashboard into focused sub-page components: 🔥 High Impact
The 147k+ character single file combines Users, Schools, Lessons, Analytics, and Chat into one component tree. Splitting into `<UsersPanel>`, `<SchoolsPanel>`, `<LessonsPanel>`, and `<AnalyticsPanel>` with React.lazy/Suspense would mean tab switches only mount relevant queries and state, cutting initial JS parse time and eliminating cross-tab re-render cascades. Expected impact: 40-60% reduction in unnecessary renders when switching between admin tabs.

### Introduce a shared admin API hook with centralized error handling: ⚡ Medium Impact
Currently each query and mutation configures its own onError/onSuccess inline. A thin `useAdminQuery` wrapper around `useQuery` that injects the admin auth header check, standard toast-on-error, and a retry:1 policy would remove duplicated error handling across ~10+ query call sites and guarantee no silent failures. This also makes it trivial to add request logging for audit compliance later.

### Paginate or virtualize large data tables server-side: 🔥 High Impact
Fetching all users and schools in one query then filtering client-side does not scale. Passing `?page=` and `?search=` query params to the API and using TanStack Query's `keepPreviousData` option would reduce payload size, remove the need for client-side sort/filter memoization entirely, and prevent the browser from holding the full dataset in memory—important for low-end devices common among Caribbean school users.

### API Routes (`server/routes.ts`)
The routes file is generally well-structured with good patterns like parallel module loading and rate limiting on auth endpoints. Key concerns are unprotected AI/chat/image/audio routes with no rate limiting, a host-header injection risk in the robots.txt and sitemap handlers, and the Supabase init chain running sequentially when it could be parallelised.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including sequential DB queries inside per-user loops that should be parallelized, missing try/catch around async operations in job handlers leaving unhandled rejections, and large CSV payload construction happening in-memory with the full dataset before streaming to object storage. Security and idempotency gaps are also present in the weekly-digest fan-out bootstrap path.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes already in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency filter columns, nullable timestamps that should be NOT NULL for audit integrity, and an untyped varchar user_id that lacks a length constraint across all tables.

### Server Startup (`server/index.ts`)
The server startup logic contains several reliability and correctness gaps: blocking synchronous operations delay readiness, interval handles are never stored or cleared on shutdown, and the graceful shutdown handler is incomplete. The logging middleware unconditionally serialises every JSON response body to a string, which is a latency and memory concern on high-traffic API paths.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns: async operations lack consistent error handling, expensive filter/sort computations run on every render without memoization, and the chat scroll uses an inline ref callback that creates a new function instance every render. The file is extremely large (147k+ chars) suggesting it needs component decomposition, which also contributes to unnecessary re-renders across unrelated state changes.
