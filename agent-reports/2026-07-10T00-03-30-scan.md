## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes
**Issue:** No rate limiting applied to AI/OpenAI-backed routes before they are registered. These are the most expensive endpoints in the app and are fully open to abuse, which is a critical cost and availability risk for a school-aged user app.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); Place these lines immediately before the registerChatRoutes(app) call.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (lines ~43 and ~63)
**Issue:** The base URL is constructed directly from request headers x-forwarded-host and host without any validation. An attacker can send a crafted Host header to inject arbitrary URLs into the Sitemap directive and sitemap <loc> values, enabling cache poisoning or phishing via indexed sitemap entries.

**Fix:** Resolve the base URL once at startup from an environment variable: const BASE_URL = process.env.PUBLIC_BASE_URL || 'https://finsightlite.com'; Use BASE_URL in both handlers instead of reading headers at request time. Remove all header-based host resolution from these routes.

### `server/index.ts`: seedDatabase / probeDatabase calls in top-level async IIFE
**Issue:** seedDatabase and probeDatabase are awaited sequentially during startup but could run in parallel. More critically, if either rejects, the unhandled rejection may silently skip route registration and leave the HTTP server bound but non-functional, because the surrounding IIFE lacks a top-level catch that calls process.exit.

**Fix:** Wrap the startup IIFE in a .catch(err => { captureError(err); process.exit(1); }) chain. Replace sequential `await seedDatabase(); await probeDatabase();` with `await Promise.all([seedDatabase(), probeDatabase()]);` since neither depends on the other's result.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are discarded (not assigned to a variable). There is no SIGTERM/SIGINT handler that clears these intervals or waits for an in-progress scheduler tick to finish before the process exits. On a Replit restart or deploy, the process is killed mid-tick, which can leave a claimed weeklyDigestRuns row inserted but its job never enqueued, permanently blocking that digest for the week.

**Fix:** Store handles: `const digestInterval = setInterval(...);`. Register a graceful shutdown handler: `process.once('SIGTERM', async () => { clearInterval(digestInterval); clearInterval(purgeInterval); await httpServer.close(); await closeDatabasePool(); process.exit(0); });`. Mirror the same handler for SIGINT.

### `server/index.ts`: res.json monkey-patch in request logging middleware
**Issue:** The middleware unconditionally serializes and stores every JSON response body in `capturedJsonResponse`, then appends the full JSON string to the log line. For /api/auth or any endpoint returning session tokens, user PII, or quiz answers, this dumps sensitive data into stdout which flows to Replit's log aggregator and any third-party log drain configured via Sentry.

**Fix:** Add a blocklist check before logging the body: define `const SENSITIVE_PATHS = ['/api/auth', '/api/user', '/api/session'];` and in the finish handler replace the body append with `if (capturedJsonResponse && !SENSITIVE_PATHS.some(p => path.startsWith(p))) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`. Consider a max-length truncation as well.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage POST body
**Issue:** Every call to sendMessage sends the entire accumulated messages array to /api/admin/help-chat with no length cap. A user (or script) can accumulate thousands of messages and send megabyte-scale payloads. For a platform serving school-aged users this is a straightforward abuse / cost-amplification vector.

**Fix:** Slice the history before sending: `const context = next.slice(-20);` and send `{ messages: context }` instead of the full array. Add a hard cap that discards the oldest messages from local state too: `if (next.length > 100) next = next.slice(-100);` before setting state.

---

### `client/src/pages/AdminDashboard.tsx`: useMutation callbacks for delete/update operations
**Issue:** Admin mutations (delete user, promote teacher, reset password, etc.) call apiRequest directly inside onMutate or mutationFn without verifying the current admin session is still valid client-side before dispatching. If the JWT expires mid-session the mutation fires and the server returns 401, but the UI only shows a generic toast error with no redirect to re-authenticate, leaving the admin confused and potentially retrying sensitive operations.

**Fix:** In the shared mutation error handler (or a wrapper around apiRequest), check `if (res.status === 401) { setLocation('/admin/login'); return; }` before parsing the body. Centralise this in apiRequest so every admin call benefits automatically.

## 🟡 Warnings (13)

- **`server/routes.ts` (Supabase init chain (~line 118)):** The seven Supabase init/seed functions are chained sequentially with .then(), so each waits for the previous to complete. On a cold start this adds unnecessary latency. Additionally, if trimOrgNamesInSupabase or any mid-chain step fails, the subsequent seeds are silently skipped with no per-step error context.
  → *Group independent seeds into Promise.all: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Wrap each seed in its own try/catch internally so one failure does not abort the rest.*
- **`server/routes.ts` (registerAuthDomainRoutes through registerEmailRoutes (~lines 135-140)):** The six domain routers are registered sequentially with individual awaits after being imported in parallel. If any router's async registration depends only on app and not on a prior router's side-effects, this is unnecessary serialisation adding cold-start latency.
  → *Replace the sequential awaits with: await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), registerAiRoutes(app), registerOrgRoutes(app), registerLessonRoutes(app), registerEmailRoutes(app)]); Verify first that no router mutates shared state that another depends on.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 103)):** The filePath parameter is passed directly to objectStorage.searchPublicObject without path traversal sanitisation. A request to /public-objects/../../private/secret could potentially escape the intended public directory depending on the storage implementation.
  → *Sanitise the path before use: import path from 'path'; const safePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (safePath !== req.params.filePath || safePath.startsWith('..')) return res.status(400).json({ error: 'Invalid path' }); Then pass safePath to searchPublicObject.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet every budget query will filter by userId. With many users this becomes a full table scan on every dashboard load.
  → *Add a composite index in the table's index callback: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a single-column index on userId alone if you ever query budgets without a categoryId filter.*
- **`shared/schema.ts` (categories table, userId column):** userId in categories uses bare varchar() with no length argument and no index, yet it is used as a FK reference and likely appears in WHERE clauses to distinguish user-owned vs system categories. Missing index means every category lookup by user requires a full scan.
  → *Change to varchar('user_id', { length: 36 }) to match the users.id type exactly, and add an index: (t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) }). Apply the same length specifier to all other varchar user_id columns for type consistency.*
- **`shared/schema.ts` (transactions table, createdAt column; budgets.createdAt; linkedCards.createdAt; documentUploads.createdAt; savingsGoals.createdAt; billReminders.createdAt; portfolioHoldings.purchasedAt; portfolioTransactions.executedAt):** All audit/timestamp columns use .defaultNow() without .notNull(). Drizzle will not add a NOT NULL constraint, so the DB column is nullable. A bug or direct DB insert could leave these NULL, breaking any ORDER BY or range query that assumes a non-null timestamp.
  → *Chain .notNull() after .defaultNow() on every timestamp that tracks record creation or execution, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Generate and apply a migration after the change.*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text() instead of timestamp or date. Storing dates as text prevents efficient range queries, correct sorting, and DB-level date arithmetic needed for price-staleness checks in a financial app.
  → *Change to timestamp('last_price_update_date') or date('last_price_update_date') depending on whether time-of-day precision is needed. Update any application code that writes or reads this column. Add a migration.*
- **`shared/schema.ts` (billReminders table, nextDueDate column; portfolioHoldings and portfolioTransactions userId/stockId columns):** billReminders has no index on nextDueDate or userId, so a scheduled job querying 'due today' reminders across all users does a full table scan. portfolioHoldings and portfolioTransactions also lack a userId index, causing full scans on every portfolio dashboard request.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to billReminders. Add (t) => ({ userStockIdx: index('idx_portfolio_holdings_user').on(t.userId, t.stockId) }) to portfolioHoldings and (t) => ({ userIdx: index('idx_portfolio_tx_user').on(t.userId, t.executedAt) }) to portfolioTransactions.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince = (dow + 0) % 7`, where `+ 0` is a no-op dead code artifact. More importantly, `new Date()` and all date arithmetic use the server's local timezone, but Replit containers run in UTC. Sunday 7 PM UTC is not Sunday 7 PM in any Caribbean timezone (UTC-4/-5), so the digest fires roughly 4-5 hours early relative to the intended local schedule.
  → *Remove the `+ 0`. Introduce a `DIGEST_TIMEZONE` env var (e.g. `America/Port_of_Spain`). Use the Intl API or the `date-fns-tz` library already likely in the dependency tree to convert the current UTC instant to the target timezone before extracting day-of-week and hour, then perform the Sunday-7-PM comparison in that zone.*
- **`server/index.ts` (startJobWorker / registerJobHandlers call site):** startJobWorker and registerJobHandlers are called without await and without a try/catch. If the job worker fails to initialise (e.g. Redis/DB connection refused), the error is silently swallowed, the worker never starts, but the server continues accepting requests and enqueuing jobs that will never be processed.
  → *Await both calls inside the startup IIFE and wrap in the shared try/catch: `await registerJobHandlers(); await startJobWorker();`. If either throws, log the error via captureError and call process.exit(1) so the process restarts rather than running in a degraded state.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll ref callback inside JSX):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is a new function instance on every render and fires scrollIntoView unconditionally whenever the messages list re-renders, not just when a new message arrives. This causes jarring scroll-jacking if the user tries to scroll up to read earlier messages.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and attach `ref={bottomRef}` to the sentinel div. This fires only when a new message is appended.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats data-loading section – multiple independent useQuery calls):** The dashboard fires several independent useQuery calls (users, schools, lessons, audit log, analytics) sequentially defined at component mount. Because they share no coordination, a slow query stalls perceived load time and each failed query independently shows an error state, producing a fragmented broken UI rather than a single coherent loading/error experience.
  → *Wrap related seed queries in a single `useQuery` that calls a `/api/admin/dashboard-summary` endpoint returning all counts in one round trip, or use `useQueries` from TanStack and derive a unified `isLoading = results.some(r => r.isLoading)` and `isError` flag to gate rendering of the entire stats grid behind one consistent skeleton/error state.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort functions passed directly to .filter() and .sort() inside JSX render):** Filtering and sorting of users, lessons, and audit entries are computed inline during render (e.g., `users.filter(u => u.name.includes(searchTerm)).sort(...)`) without useMemo. On a list of hundreds of school users or audit rows this runs on every keystroke-triggered re-render of the parent and on every unrelated state change (e.g., toggling a modal).
  → *Extract each derived list into a useMemo: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(searchTerm.toLowerCase())).sort(...), [users, searchTerm, sortField, sortDir]);` This is especially important for the audit log which can be unbounded in length.*

## 💡 Opportunities

### Centralise base-URL resolution at startup: ⚡ Medium Impact
Move PUBLIC_BASE_URL to a validated config module loaded once at process start. This eliminates the duplicated proto/host header logic currently copy-pasted across robots.txt, sitemap.xml, and potentially other files, and makes the value available for canonical link headers and email templates without repeated header parsing.

### Add /healthz database latency metric: ⚡ Medium Impact
The /healthz endpoint currently returns only ok/error but not query latency. Recording the round-trip time (Date.now() before and after db.execute) and returning it in the response gives Railway and uptime monitors an early signal for DB degradation before it causes user-facing errors, with zero additional dependencies.

### Structured startup logging for Supabase init steps: 📌 Low Impact
Replace the single catch-all console.error with per-step console.info/error logging including step name and duration. For a school app with periodic content reseeding this makes it immediately visible in Railway logs which seed failed and how long each step took, reducing debugging time significantly.

### Centralise currency default to environment config: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table columns (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract it to a single exported constant DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' and reference it in every .default() call. This makes multi-currency expansion or environment-specific overrides a one-line change and eliminates drift risk.

### Add unique constraint on simulatedStocks ticker: 🔥 High Impact
The ticker column has no unique index, so duplicate tickers can be inserted silently, corrupting portfolio calculations that join on ticker. Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table index callback. This is especially important for a school-aged audience where incorrect gain/loss numbers would undermine financial literacy trust.

### Replace serial primary keys with uuid for user-facing tables: 🔥 High Impact
Using serial (auto-increment integer) PKs on transactions, budgets, savingsGoals and other user-facing tables exposes sequential IDs in API responses, allowing enumeration attacks (e.g. /api/transactions/1001 through /api/transactions/9999). Switch to uuid default gen_random_uuid() PKs for all tables whose IDs appear in client-facing routes. Impact is reduced even if auth checks exist, as defence-in-depth is critical for a minor-user app.

### Centralised startup orchestrator with health-check gate: 🔥 High Impact
Extract the startup sequence (probe DB, seed, register handlers, start worker, bind HTTP) into a single `bootstrap()` function with explicit ordered phases and a readiness flag. Expose a `GET /healthz` route that returns 503 until the flag is set. This prevents Replit's load balancer from routing traffic before the server is fully initialised, eliminating a class of race-condition 500 errors during cold starts.

### Replace monkey-patched res.json with a structured request-log middleware: ⚡ Medium Impact
The current res.json override is fragile (breaks if another library also patches it) and leaks response size into memory per-request until the finish event. Replacing it with a dedicated structured logger (e.g. pino-http) that records method, path, status, latency, and a safe subset of response metadata would improve observability, remove the PII risk, and cut per-request allocations.

### Move scheduler tick logic into the job queue itself: ⚡ Medium Impact
The setInterval-based scheduler duplicates scheduling concerns that belong in the job queue layer (e.g. BullMQ repeatable jobs or pg-boss cron schedules). Migrating weekly-digest and ai-usage-purge to queue-native cron expressions eliminates the dual-process double-firing risk, removes the manual DB claim/rollback pattern in maybeRunWeeklyDigest, and makes the schedule visible and manageable from a single place.

### Centralised admin API error boundary: 🔥 High Impact
Wrap the entire AdminDashboard in a React ErrorBoundary and a shared apiRequest interceptor that normalises 401/403/500 responses into typed errors. This eliminates the 15+ duplicated try/catch toast patterns spread across mutation callbacks and gives a single place to add logging or redirect logic, reducing bundle size and maintenance surface.

### Virtualise long admin tables: ⚡ Medium Impact
User, lesson, and audit-log tables render all rows into the DOM simultaneously. For a Caribbean school network with hundreds of students this will cause visible jank. Replacing the table body with TanStack Virtual (already likely in the dependency tree via TanStack Query) would cap DOM nodes at ~20 visible rows regardless of dataset size.

### Extract AdminHelpChat into its own lazy-loaded chunk: ⚡ Medium Impact
AdminHelpChat, its state, and its Lucide icon imports are bundled into the main AdminDashboard chunk even when the chat panel is never opened. Moving it to a separate file and importing via React.lazy/Suspense would shave the icons and component tree from the initial parse cost, improving Time-to-Interactive for the dashboard.
