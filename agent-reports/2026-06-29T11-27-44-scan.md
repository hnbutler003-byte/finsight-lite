## 🔴 Critical (5)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. For a school-aged user base this creates both abuse risk and runaway API cost exposure — a single user or bot can exhaust OpenAI quota.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 46 and ~line 63)
**Issue:** The `host` and `x-forwarded-host` headers are taken directly from the incoming request and interpolated into URLs without any validation. An attacker can send `Host: evil.com` to generate a sitemap or robots.txt pointing at an arbitrary domain — a classic host header injection leading to cache poisoning or phishing via search engine indexing.

**Fix:** Validate the resolved host against an allowlist from environment config: `const ALLOWED_HOSTS = new Set((process.env.ALLOWED_HOSTS || '').split(',')); if (!ALLOWED_HOSTS.has(host)) { host = process.env.DEFAULT_HOST || 'finsightlite.com'; }` Apply this to both handlers, and set `ALLOWED_HOSTS` in Railway env vars.

---

### `server/jobHandlers.ts`: runExtractPaper
**Issue:** fileB64 (full base64-encoded PDF or image) is stored in the job payload, which lives in the job table. A single A4 exam scan can easily exceed 5–10 MB encoded, bloating the DB and causing large row reads on every job poll.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download inside the handler with objectStorageClient. This keeps job rows small and makes payloads idempotent-safe to retry.

---

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId has no index. Any query fetching a user's budgets (the primary access pattern) will do a full table scan as the table grows. There is also no unique constraint preventing duplicate budgets for the same user+category+period combination, which would corrupt budget vs. spending comparisons.

**Fix:** Add index('idx_budgets_user_id').on(t.userId) and uniqueIndex('uq_budgets_user_category_period').on(t.userId, t.categoryId, t.period) in the table's index callback. Mirror the pattern already used on transactions.

---

### `shared/schema.ts`: portfolioHoldings and portfolioTransactions table definitions
**Issue:** portfolioHoldings has no index on userId or stockId, and portfolioTransactions has no index on userId or executedAt. Every portfolio dashboard load and trade-history query will full-scan these tables. portfolioHoldings also lacks a unique constraint on (userId, stockId), allowing duplicate holding rows for the same asset.

**Fix:** For portfolioHoldings add index('idx_portfolio_holdings_user').on(t.userId) and uniqueIndex('uq_portfolio_holdings_user_stock').on(t.userId, t.stockId). For portfolioTransactions add index('idx_portfolio_tx_user_executed').on(t.userId, t.executedAt).

## 🟡 Warnings (9)

- **`server/routes.ts` (Supabase init chain (~line 113)):** The seven Supabase init/seed functions are chained sequentially with `.then()`. Each step waits for the previous to complete even though most are independent (e.g. seedGameContent does not depend on trimOrgNamesInSupabase). This unnecessarily extends cold-start wall time.
  → *Run independent seeds in parallel: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` Validate actual dependencies first and collapse where there are none.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 101)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject()` with no sanitisation. A path like `../../private/secrets` could traverse into unintended storage namespaces depending on how the ObjectStorageService resolves paths.
  → *Sanitise before use: `import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Then pass the sanitised value to objectStorage methods.*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop):** Questions are inserted one-by-one with await inside a for loop. For a 60-question paper this means 60 sequential round-trips to the DB, adding hundreds of milliseconds of unnecessary latency and holding the DB connection for the entire loop.
  → *Collect valid questions into an array, then call a single db.insert(extractedQuestions).values(validQuestions) after the loop. Add the order field during the map step. Fall back to chunked batches (e.g. 25 at a time) if row limits apply.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The OpenAI client is imported from a route file rather than a dedicated singleton module. Route files often register middleware, initialise state, or have side-effects on import. This creates an implicit coupling and could cause express route registration to run inside a worker process.
  → *Move the openai client instantiation to a shared module (e.g. server/lib/openai.ts) that exports only the configured client. Both the route file and jobHandlers import from there.*
- **`server/jobHandlers.ts` (runExtractPaper – model: "gpt-4o" (appears multiple times)):** The model name is hardcoded as a string literal in two separate openai.chat.completions.create calls. Changing the model for cost or capability reasons requires editing multiple code sites and redeploying.
  → *Add OPENAI_MODEL to environment config (e.g. process.env.OPENAI_MODEL ?? 'gpt-4o') and reference a single constant at the top of the file. Also add max_completion_tokens to env config (OPENAI_MAX_TOKENS) for the same reason.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch. A DB error (e.g. connection timeout) will bubble as an unhandled rejection inside the job runner. Depending on the runner implementation, this may crash the worker process rather than marking the job as failed and retrying.
  → *Wrap the db.delete call in try/catch, log the error with enough context (olderThanDays, cutoff), and rethrow so the job framework can record the failure and apply its retry/backoff policy consistently with other handlers.*
- **`shared/schema.ts` (varchar('user_id') across all tables):** userId is declared as varchar with no length parameter, which in PostgreSQL defaults to unlimited length but provides no enforcement. If the auth provider (e.g. Clerk/Supabase) uses fixed-length IDs (e.g. 36-char UUIDs or 32-char Clerk IDs), the absence of a length constraint means a misconfigured upstream could insert arbitrarily long strings, silently bloating indexes and bypassing application-level assumptions.
  → *Declare all userId columns as varchar('user_id', { length: 64 }) (or the exact max length of your auth provider's IDs). This adds a cheap DB-level guard and improves index efficiency with a fixed-length column.*
- **`shared/schema.ts` (createdAt columns across transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** All createdAt columns use .defaultNow() but omit .notNull(). This means application code or a raw INSERT omitting the column will store NULL rather than the current timestamp, silently breaking any audit trail, ORDER BY createdAt queries, and time-based analytics that assume the column is always populated.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default ensures existing INSERTs continue to work without changes.*
- **`shared/schema.ts` (simulatedStocks table definition — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text rather than timestamp. Storing a date as a string makes range queries (e.g. 'stocks not updated in the last 24 hours') require CAST or application-side parsing, breaks index-assisted date comparisons, and silently accepts malformed date strings with no DB-level validation.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed or migration scripts that write string values to this column.*

## 🔵 Info / Tech Debt (1)

- **`server/routes.ts` (/api/health and /healthz (~line 30 and ~line 77)):** Two overlapping health check endpoints exist (`/api/health` and `/healthz`). `/api/health` returns no DB liveness check while `/healthz` does a real DB ping. Railway and uptime monitors should use only one canonical endpoint. The duplicate causes confusion about which check is authoritative and wastes a DB connection on every monitor poll from the deeper one.

## 💡 Opportunities

### Centralise base-URL resolution into a middleware helper: ⚡ Medium Impact
The proto+host resolution logic is duplicated verbatim in robots.txt and sitemap.xml handlers. Extract it into `getBaseUrl(req)` in a shared utils module. This removes two future divergence points and makes the host-validation fix from finding #2 apply in one place.

### Move Supabase seeding out of the HTTP server startup path: 🔥 High Impact
Running seed/migration logic inside `registerRoutes` means any seed failure can surface mid-startup and is only soft-caught. Move it to a dedicated `scripts/seed.ts` run as a Railway release command or a one-time job, keeping `registerRoutes` focused on HTTP concerns and making seeds idempotent and independently re-runnable.

### Add request-scoped logging with correlation IDs: ⚡ Medium Impact
Errors in the public-objects handler and healthz are logged with bare `console.error` and no request ID, making production debugging in Railway logs difficult. Attach a `x-request-id` header (or generate one) in a middleware and include it in all error logs. This is especially valuable for diagnosing object-storage failures for school users.

### Parallel OpenAI + storage operations where sequencing is not required: ⚡ Medium Impact
In runExtractPaper, the idempotency delete, buffer allocation, and any pre-flight storage checks are all sequential. The delete and buffer.from can run before the first await; structuring code to front-load sync work and use Promise.all for independent async steps (e.g. fetching file from storage while deleting stale questions) would reduce wall-clock time per job by 20–40% on retries.

### Structured logging instead of throw-only error propagation: ⚡ Medium Impact
Errors in handlers currently propagate with no structured metadata attached (paperId, jobId, attempt count). Adding a lightweight log call (e.g. logger.error({ paperId, attempt: job.attempts }, err)) before rethrowing would make Sentry/Datadog triage dramatically faster without changing retry semantics.

### Idempotency token on exam paper status updates: 🔥 High Impact
If the job succeeds but the worker crashes before acknowledging completion, the job reruns, re-deletes questions, re-calls OpenAI, and re-inserts. Adding a DB-level unique constraint on (paperId, status='completed') or checking paper status at handler entry and short-circuiting would eliminate redundant OpenAI spend on duplicate executions.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' appears as a hardcoded .default() in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Define a single DEFAULT_CURRENCY = 'BSD' constant in a shared config file and reference it in every table definition. When the app expands to TTD or JMD this change will be one line instead of six scattered edits.

### Add partial index for unprocessed document uploads: ⚡ Medium Impact
The documentUploads table will be polled or queried frequently for rows where status = 'processing'. A standard index on userId scans all statuses. Add a partial index: index('idx_doc_uploads_processing').on(t.userId).where(sql`status = 'processing'`). This keeps the active-job query fast even as the completed/failed history grows large.

### Enforce bill reminder uniqueness and add due-date index: ⚡ Medium Impact
billReminders has no index on nextDueDate, yet the primary scheduler query is 'find all reminders due on or before today'. Add index('idx_bill_reminders_due').on(t.nextDueDate) and consider a uniqueIndex on (userId, name) to prevent accidental duplicate reminders from re-import or auto-detection, which would otherwise send users duplicate notifications.
