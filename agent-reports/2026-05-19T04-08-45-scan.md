## 🔴 Critical (7)

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (buy/sell flow)
**Issue:** Virtual balance deduction and portfolio holding upsert are executed as separate statements with no wrapping database transaction. A crash or concurrent request between the two leaves the user with money deducted but no holding, or a holding added with no balance change — a critical data integrity gap for any financial feature.

**Fix:** Wrap both operations in a single Drizzle transaction: `await db.transaction(async (tx) => { await updateVirtualBalance(userId, newBalance, tx); await upsertPortfolioHolding(userId, stockId, quantity, avgPrice, tx); await createPortfolioTransaction({...}, tx); });`. Pass the transaction client (`tx`) as an optional parameter through each helper so they reuse the same connection.

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** getEnrollmentsByClass returns full `User` objects joined to each enrollment row. The User type almost certainly includes passwordHash, email, and other PII. For a teacher dashboard listing students, this leaks every student's sensitive fields to the client and into any logs that serialize the response.

**Fix:** Use a column-scoped select instead of a full join: `db.select({ enrollmentId: classEnrollments.id, studentId: users.id, displayName: users.displayName, avatarUrl: users.avatarUrl }).from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId))`. Define a `SafeStudent` type from these columns and update the return type signature accordingly.

### `server/routes.ts` — POST /api/auth/google and any AI tutor/chat endpoint
**Issue:** No rate limiting middleware on authentication or AI endpoints. The Google sign-in route and AI quota reservation endpoints can be hammered freely, enabling credential stuffing on auth and quota exhaustion/bypass on AI routes even if per-org quotas exist in DB logic.

**Fix:** Add express-rate-limit: `import rateLimit from 'express-rate-limit'; const authLimiter = rateLimit({ windowMs: 15*60*1000, max: 20, standardHeaders: true, legacyHeaders: false }); app.post('/api/auth/google', authLimiter, ...); const aiLimiter = rateLimit({ windowMs: 60*1000, max: 30 }); app.use('/api/ai', aiLimiter);` — place these before handler registration.

### `server/routes.ts` — multer configurations (logoUpload, videoUpload, upload, examUpload)
**Issue:** Four separate multer instances are constructed at module load time with hardcoded size limits (5 MB, 500 MB, 10 MB). The 500 MB videoUpload limit using memoryStorage will buffer entire video files in Node.js heap, causing OOM crashes under concurrent uploads from school networks.

**Fix:** Switch videoUpload to disk storage with a temp directory: `storage: multer.diskStorage({ destination: os.tmpdir(), filename: (_req, file, cb) => cb(null, `${crypto.randomUUID()}${path.extname(file.originalname)}`) })` and stream from disk to object storage in the route handler, deleting the temp file in a finally block. Move size limits to environment config: `limits: { fileSize: parseInt(process.env.MAX_VIDEO_BYTES ?? '524288000') }`.

### `server/jobHandlers.ts` — runExtractPaper (payload parameter fileB64)
**Issue:** Full base64-encoded file contents are stored in the job table payload. For a PDF or image this can easily be 1–10 MB per row, bloating the jobs table, slowing queue polling queries, and potentially hitting row-size limits depending on the DB driver.

**Fix:** Upload the file to object storage before enqueuing and store only the resulting object path in the payload. In runExtractPaper, download the file from object storage using objectStorageClient at the start of the handler, then delete or move it after processing. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `shared/schema.ts` — simulatedStocks pgTable definition
**Issue:** The ticker column on simulatedStocks has no unique constraint and no index. Duplicate tickers can be inserted silently, which will corrupt portfolio calculations (e.g., aggregating holdings by ticker returns double-counted rows). Query performance for ticker-based lookups is also O(n).

**Fix:** Replace the plain text('ticker').notNull() definition with a uniqueIndex: add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) as the third argument to pgTable. This enforces deduplication at the DB level and gives index-speed lookups.

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth hook usage / top-level admin route
**Issue:** The component imports useAdminAuth but the truncated code does not show a guard that short-circuits rendering before any data-fetching queries execute. If the auth check is asynchronous (e.g. validating a session token), all child useQuery calls fire in parallel before auth is confirmed, making unauthenticated requests to every /api/admin/* endpoint. Any 200 response received before the auth redirect resolves leaks admin data to the client.

**Fix:** Immediately after calling useAdminAuth, gate all rendering and query execution: const { isAdmin, isLoading: authLoading } = useAdminAuth(); if (authLoading) return <Spinner />; if (!isAdmin) { setLocation('/admin/login'); return null; }. Ensure this block appears before any useQuery hooks if React rules-of-hooks allow, otherwise set enabled: isAdmin on every admin query.

## 🟡 Warnings (17)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats fires multiple independent db queries sequentially (transactions sum, budgets, savings goals, bill reminders). Each query is individually awaited, adding full round-trip latency per query. On a remote Postgres instance this compounds to 4–6× the minimum possible latency.
  → *Replace sequential awaits with `const [txResult, budgetResult, goalsResult, remindersResult] = await Promise.all([db.select(...), db.select(...), db.select(...), db.select(...)]);`. All four queries are read-only and independent, so parallel execution is safe and will cut stat-load time by ~75% under normal network conditions.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard accepts an optional `limit` parameter but has no upper-bound guard. A caller (or attacker) passing `limit: 100000` will cause Postgres to return and Node to serialize an arbitrarily large result set, risking OOM and exposing every user's score data in a single response.
  → *Enforce a hard cap: `const safeLimit = Math.min(filters?.limit ?? 20, 100);` and use `safeLimit` in the query `.limit(safeLimit)`. Also ensure the query only returns public fields (userId, userName, avatar, totalScore, gamesPlayed) and never joins sensitive columns from the users table.*
- **`server/storage.ts` (seedMarketData / seedLearningModules):** Seed functions contain hardcoded stock prices, module content, and likely currency strings (e.g. 'TTD', 'BBD') directly in source. If Caribbean exchange rates or module metadata need updating, a full redeploy is required. Additionally these strings are not validated against the schema enum, so a typo silently inserts bad data.
  → *Move seed data to versioned JSON files under `server/seeds/` and load them with `fs.readFileSync`. Validate each record against the Zod schema (already defined in @shared/schema) before insert: `InsertSimulatedStockSchema.parse(record)`. Currency codes and initial virtual balance amounts (likely hardcoded as e.g. 10000) should come from environment variables: `VIRTUAL_BALANCE_START_TTD`, `SUPPORTED_CURRENCIES`.*
- **`server/routes.ts` (resolvedClass resolution block inside POST /api/auth/google (~line 230-260)):** Sequential awaits for class lookup, env lookup, and org lookup run one after another on every Google sign-in. Under load this adds 3 round-trip latencies to the auth critical path, and getOrganization/getOrgEnvironmentById results are not cached despite being called redundantly across the same request.
  → *After resolving resolvedClass, fetch env and org in parallel: `const [classEnv, resolvedOrgEnv] = await Promise.all([resolvedClass.envId ? getOrgEnvironmentById(resolvedClass.envId) : Promise.resolve(null), getOrgEnvironmentByJoinCode(code)]);` then derive classOrg from classEnv in a single subsequent call rather than nesting three sequential awaits.*
- **`server/routes.ts` (catch blocks throughout (e.g., healthz, POST /api/auth/google)):** Multiple catch blocks return `e?.message` or spread the raw error object directly into JSON responses (e.g., `res.status(503).json({ ok: false, error: e?.message })`). For internal errors this can leak stack traces, DB connection strings embedded in driver messages, or Supabase service role key references to end-users.
  → *Create a central error sanitizer: `function safeErrorMessage(e: unknown, fallback = 'An unexpected error occurred'): string { if (process.env.NODE_ENV !== 'production') return (e as any)?.message ?? fallback; return fallback; }` and replace all direct `e?.message` exposures in non-healthz routes with `safeErrorMessage(e)`.*
- **`server/routes.ts` (initSupabaseTables, initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent (called at startup inside registerRoutes)):** Multiple seed/init functions are awaited sequentially at server startup with no error isolation between them. If any seed call throws (e.g., Supabase is temporarily unavailable), the entire server fails to start and HTTP server is never returned, causing a hard crash rather than degraded startup.
  → *Wrap each init/seed call in its own try/catch and run independent ones in parallel: `await Promise.all([ initSupabaseTables().catch(e => console.error('supabase init failed', e)), initStaticContentTables().catch(e => console.error('static content init failed', e)) ]); await Promise.all([ seedStaticModules().catch(e => console.error('seed modules failed', e)), seedRegionalContent().catch(e => console.error('seed regional failed', e)), seedGameContent().catch(e => console.error('seed game failed', e)) ]);` — log and continue rather than throwing.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time in a sequential for-loop (await inside for). For a 40-question paper this serialises 40 round-trips to the DB. If any single insert fails mid-loop, the preceding rows are already committed but the paper status is never set to 'completed', leaving partial data that the earlier delete-for-idempotency does not clean up on the next retry because the retry deletes then re-inserts—but only up to the failure point again.
  → *Replace the loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(questionsArray) in one statement. This is atomic (all-or-nothing within the statement), eliminates N round-trips, and makes retries fully safe because the prior delete + single insert is truly idempotent.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation. A misconfigured or maliciously enqueued job with olderThanDays: 0 would delete the entire aiUsageEvents table.
  → *Add a guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 7) throw new Error('olderThanDays must be a number >= 7'); This keeps the purge safe and makes the constraint explicit and auditable.*
- **`server/jobHandlers.ts` (parsePrivatePath / uploadPrivateObject):** PRIVATE_OBJECT_DIR is read inside parsePrivatePath on every call rather than once at startup. If the env var is missing the error surfaces only when the first job runs, not at boot, making misconfiguration hard to catch in CI or during deployment smoke tests.
  → *Read and validate all required env vars (PRIVATE_OBJECT_DIR, and any others used in this file) in a single block at the top of registerJobHandlers(), throwing immediately if any are absent. Store them as local constants passed into the helpers.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — res.setHeader / createReadStream):** The Content-Disposition header interpolates fileName directly into the header value without sanitisation: filename="${fileName}". A fileName containing a double-quote or newline can break the header and in some clients allow header injection.
  → *Sanitise fileName before use: const safeName = fileName.replace(/["\r\n]/g, '_'); then use safeName in the header. For full RFC 6266 compliance, use the content-disposition npm package which handles encoding correctly.*
- **`shared/schema.ts` (userId: varchar('user_id')):** All userId columns use bare varchar() with no length argument. In PostgreSQL, bare varchar() is equivalent to text, so no truncation occurs, but this is semantically misleading and inconsistent — if a length is ever added (e.g., varchar(36) for UUIDs), auth provider IDs from Clerk/Auth0 that exceed that length will be silently truncated or throw at insert time, corrupting foreign key relationships across every table.
  → *Either change all userId columns to text('user_id') to be explicit that there is no length limit, or pin to varchar('user_id', { length: 255 }) after confirming the maximum ID length your auth provider emits. Apply the same type uniformly across categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow()):** createdAt is missing .notNull() on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions. defaultNow() sets a DB-level default, but without NOT NULL the column accepts explicit NULL inserts, making time-based queries (ORDER BY created_at, range filters) unreliable and forcing null-checks throughout application code.
  → *Add .notNull() to every createdAt column: e.g., createdAt: timestamp('created_at').defaultNow().notNull(). Similarly audit executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (budgets pgTable definition):** The budgets table has no index on userId even though budget lookups are always user-scoped. Every budget query will perform a full sequential scan of the budgets table filtered by userId. For a school-aged user base that grows, this will degrade linearly.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also add a composite index on (userId, categoryId) since the common query is 'find budget for this user and category': index('idx_budgets_user_category').on(t.userId, t.categoryId).*
- **`shared/schema.ts` (billReminders — nextDueDate, portfolioHoldings — userId/stockId composite):** billReminders has no index on (userId, nextDueDate), which is the primary query pattern for the reminders feature (fetch upcoming bills for a user ordered by due date). portfolioHoldings has no index on (userId, stockId), so calculating a user's current position in a stock requires a full table scan joined to simulatedStocks.
  → *For billReminders add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). For portfolioHoldings add: (t) => ({ userStockIdx: index('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). The holdings index also enables efficient upsert logic when updating quantity after a buy/sell.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render inside the component body, and the queryFn closes over it as a new reference each render. This causes the query to re-fire on unrelated parent re-renders even when filter state hasn't changed, because the queryKey array uses primitive values correctly but the queryFn captures a new params object reference.
  → *Move the URLSearchParams construction inside the queryFn itself, or memoize it with useMemo: const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return p; }, [actorType, action, from, to]). The queryKey already has the right deps, so this just prevents the closure from being stale or wasteful.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set("limit", "200")):** The audit log fetch limit is hardcoded to 200 as a magic string literal. For a school-aged user platform with potentially large school districts, this silently truncates audit records and offers no way to adjust without a code deploy. The number 200 also appears to be duplicated in at least one other query in the file.
  → *Extract to a named constant at module scope: const AUDIT_LOG_PAGE_LIMIT = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 200). Use this constant in the params.set call and in the queryKey so cache invalidation is correct if the env value changes.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper function):** fmtDate accepts type `any` and calls `new Date(d)` without validation. If d is an invalid date string or an unexpected type (e.g. a nested object accidentally passed from API response), this silently returns 'Invalid Date' in the UI rather than the fallback '—'. For a financial literacy app shown to students, displaying 'Invalid Date' is a UX and trust issue.
  → *Add a validity check: function fmtDate(d: unknown): string { if (!d) return '—'; const dt = new Date(d as string); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); }. Change the parameter type from any to unknown to catch misuse at compile time.*

## 💡 Opportunities

### Unified pagination helper for all list queries — 🔥 High Impact
getTransactions, getGameSessions, getPortfolioTransactions, and getAllQuestions all accept optional limit params with no shared enforcement. Extract a `paginate(query, { limit, offset, maxLimit = 200 })` utility that validates and clamps both values. This prevents unbounded queries across the app in one change and makes adding cursor-based pagination straightforward later.

### Repository-layer error wrapping with structured context — 🔥 High Impact
No storage methods wrap db calls in try/catch. A failed query propagates a raw Drizzle/Postgres error (including the SQL string in some drivers) up through the route handler and into the HTTP response. Wrapping each public method with a thin try/catch that throws a typed `StorageError({ operation, userId, cause })` lets route handlers return safe 500 messages and lets a future logger capture structured context without leaking SQL to clients.

### Class enrollment duplicate-check at DB level — ⚡ Medium Impact
enrollStudent likely does an application-level check for existing enrollment before inserting, but under concurrent requests two enrollments can race past the check. Adding a unique index on (classId, studentId) in the schema and using `onConflictDoNothing()` in the insert makes the operation idempotent at zero extra query cost and removes the need for the application-level pre-check entirely.

### Extract route groups into separate router files — 🔥 High Impact
The single routes.ts file exceeds 200K characters, making security audits, code review, and targeted testing impractical. Splitting into routers (authRouter, aiRouter, lessonRouter, orgRouter) mounted via app.use() would reduce per-file cognitive load, enable per-router middleware (e.g., isAuthenticated, aiLimiter) applied once cleanly, and cut CI lint times significantly.

### Add a DB query batcher for leaderboard and org lookups — ⚡ Medium Impact
getOrganization and getOrgEnvironmentById are called multiple times per request in auth and enrollment flows with the same org_id. A per-request memoization Map (or DataLoader pattern) would eliminate redundant Supabase round trips. Expected impact: 2-4 fewer DB calls per student login, meaningful latency reduction on class join flows used heavily at school start times.

### Replace hardcoded bcrypt salt rounds with environment config — ⚡ Medium Impact
bcrypt.hash calls throughout the file likely use a hardcoded rounds value (commonly 10). For a school-age user app running on shared Replit infrastructure, this should be `parseInt(process.env.BCRYPT_ROUNDS ?? '12')` to allow tuning without deploys, and 12 rounds is the current OWASP minimum recommendation for bcryptjs.

### Parallel AI calls in extract-paper — 📌 Low Impact
The text-extraction step (PDF parse or vision call) and the subsequent GPT-4o extraction call are strictly sequential, but the second call depends on the first—so they cannot be parallelised. However, the final storage.updateExamPaper and the bulk question insert are independent and could be run with Promise.all, saving one round-trip latency on every extraction job.

### Centralised job payload schema validation — 🔥 High Impact
No handler currently validates the shape of job.payload before using it. Adding a Zod schema per job type at the top of registerJobHandlers() and parsing the payload before the handler body runs would catch malformed jobs at dequeue time, produce clear error messages, and prevent subtle runtime failures like the olderThanDays:0 issue from propagating silently.

### Object storage streaming instead of buffer-in-memory for CSV export — ⚡ Medium Impact
The admin-csv-export handler (visible in the truncated section) likely builds the entire CSV in memory before uploading. For large datasets (all transactions, all users) this can spike Node.js heap. Replacing it with a streaming CSV writer (e.g. csv-stringify in streaming mode) piped directly to objectStorageClient's upload stream would keep memory flat regardless of row count.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioHoldings, portfolioTransactions). Extract a DEFAULT_CURRENCY constant and reference it everywhere. When the app expands to TTD or JMD, a single change propagates rather than requiring a grep-and-replace across the schema with risk of misses.

### Add a composite unique constraint to portfolioHoldings — 🔥 High Impact
A user should have exactly one holding row per stock (quantity is updated on buy/sell). Without a unique constraint on (userId, stockId), application bugs or race conditions can insert duplicate rows that silently double-count position size. Adding uniqueIndex('idx_portfolio_holdings_user_stock_unique').on(t.userId, t.stockId) enforces this invariant at the DB layer and enables safe INSERT ... ON CONFLICT DO UPDATE upsert semantics.

### Switch numeric monetary columns to integer cents storage — ⚡ Medium Impact
Using numeric(10,2) for amounts is correct for precision but slower than integer arithmetic in PostgreSQL for aggregations (SUM, AVG across thousands of transactions). Storing amounts as integer cents (e.g., 1099 = $10.99) with a smallint scale field eliminates floating-point edge cases entirely and significantly speeds up budget rollup and savings-goal progress queries that will run on every dashboard load.

### Split monolithic dashboard into lazy-loaded tab modules — 🔥 High Impact
The single file at 126k+ characters means the entire admin bundle — charts, audit log, DB viewer, job runner — loads on first admin page visit. Use React.lazy + Suspense per tab panel so only the active tab's code is parsed and executed. Expected impact: 40-60% reduction in initial admin JS parse time, directly improving TTI on low-end devices common in Caribbean schools.

### Centralise admin query configuration with a shared staleTime and retry policy — ⚡ Medium Impact
Each useQuery in the file independently omits staleTime and retry config, defaulting to React Query's aggressive refetch-on-window-focus behaviour. A shared adminQueryOptions base object (staleTime: 30_000, retry: 1, refetchOnWindowFocus: false) applied via queryOptions() or a custom useAdminQuery wrapper would eliminate redundant background fetches across all 13+ admin tabs without changing any business logic.

### Add error boundaries around each dashboard card — ⚡ Medium Impact
A runtime error in any single card (e.g. ObservabilityCard receiving unexpected API shape) currently crashes the entire AdminDashboard. Wrapping each Card or tab panel in a lightweight ErrorBoundary with a retry button contains failures to the affected widget and keeps the rest of the dashboard operational — critical for admin users who need to stay productive during partial outages.
