## 🔴 Critical (6)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Buy/sell stock operations update portfolio holdings, create a portfolio transaction record, and mutate virtual balance in three separate awaits with no wrapping database transaction. A failure between any two steps leaves the user's financial state permanently inconsistent — e.g., balance debited but holding never created.

**Fix:** Wrap all three operations in a single db.transaction(async (tx) => { ... }) call using Drizzle's transaction API. Pass the transaction context tx to each insert/update so they share the same atomic unit. Roll back automatically on any thrown error.

### `server/storage.ts` — getLeaderboard
**Issue:** getLeaderboard accepts a limit parameter but if the caller omits it there is no server-enforced cap, meaning the query can return every row in gameSessions joined with users — a full table scan that will degrade as student population grows. For a school platform this is a denial-of-service vector via a single unauthenticated (or low-privilege) request.

**Fix:** Apply a hard server-side maximum: const safeLimit = Math.min(filters?.limit ?? 50, 100); and use .limit(safeLimit) unconditionally in the query. Never trust the caller to supply a reasonable limit.

### `server/jobHandlers.ts` — runExtractPaper (job.payload.fileB64)
**Issue:** Full base64-encoded file content is stored in the job table payload column. For a PDF or image this can be hundreds of KB to several MB per row, bloating the jobs table, slowing queue polling queries, and risking the content appearing in structured logs or error dumps.

**Fix:** Before enqueuing, upload the file to private object storage and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of processing, then delete the object on terminal success or failure to avoid orphaned files.

### `server/jobHandlers.ts` — runExtractPaper — for loop calling storage.createExtractedQuestion
**Issue:** Extracted questions are inserted one-by-one inside a sequential for loop with individual awaits. For a 40-question paper this serialises 40 round-trips to the database, adding significant latency and leaving a long window during which a crash produces a partial insert that the retry wipe-and-redo strategy must clean up.

**Fix:** Replace the loop with a single bulk insert: filter valid questions first, then call db.insert(extractedQuestions).values(validQuestions.map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0,4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i + 1 }))); This reduces N round-trips to 1 and shrinks the partial-write window to zero.

### `server/jobHandlers.ts` — streamPrivateObjectToResponse
**Issue:** The function accepts an arbitrary objectPath string and constructs a bucket/object path by splitting on '/'. There is no validation that the resolved path stays within the expected private bucket prefix, making it possible for a caller to pass a path like '../../other-bucket/sensitive-file' and read objects outside the intended scope.

**Fix:** After splitting, assert that parts[0] matches the expected bucket name from PRIVATE_OBJECT_DIR (reuse parsePrivatePath logic): const { bucket, object } = parsePrivatePath(objectPath); then call objectStorageClient.bucket(bucket).file(object) directly. This ensures every access goes through the same boundary check used on writes.

### `shared/schema.ts` — budgets table definition (~line 30)
**Issue:** budgets.userId and savingsGoals.userId, billReminders.userId, linkedCards.userId, and portfolioHoldings.stockId have no indexes despite being the primary WHERE clause columns for every user-scoped query. Every dashboard load will perform a full-table scan on these tables.

**Fix:** Add index definitions in the table's second argument callback for each table, e.g. for budgets: `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })`. Do the same for savingsGoals, billReminders, linkedCards (userId), and portfolioHoldings (userId, stockId). Also add index on billReminders.nextDueDate since reminder queries will filter by date.

## 🟡 Warnings (13)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues at least three independent aggregate queries (total income, total expenses, budget summary) sequentially with individual awaits. Each round-trip adds latency unnecessarily; on a Caribbean deployment with higher DB latency this compounds noticeably for every dashboard load.
  → *Replace sequential awaits with Promise.all([ db.select(...income...), db.select(...expenses...), db.select(...budgets...) ]) and destructure the results. These queries have no data dependency on each other and are safe to run in parallel.*
- **`server/storage.ts` (getQuestionsByPaper / getAllQuestions):** getAllQuestions fetches extracted questions and then for each row that needs a paper, appears to do an additional lookup — a classic N+1 pattern. With exam papers containing dozens of questions per paper and multiple papers, this produces N+1 round-trips to the database on every call.
  → *Replace the per-row lookup with a single JOIN at the query level: db.select().from(extractedQuestions).leftJoin(examPapers, eq(extractedQuestions.paperId, examPapers.id)).where(...). Drizzle supports this natively and returns the joined shape in one query.*
- **`server/storage.ts` (getVirtualBalance):** getVirtualBalance inserts a default balance row with a hardcoded value (likely 10000 or similar) when no row exists for the user. Hardcoding the starting virtual balance in application logic means changing it requires a code deploy and leaves existing 'default' rows inconsistent with any future policy change.
  → *Move the default starting balance to an environment variable: const DEFAULT_VIRTUAL_BALANCE = parseInt(process.env.DEFAULT_VIRTUAL_BALANCE ?? '10000', 10); Reference this constant in the upsert. Document the variable in .env.example.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText truncation at 30000):** The magic number 30000 is hardcoded with no explanation or environment override. If the limit needs tuning for longer Caribbean CXC/CAPE papers or is tightened for cost control, it requires a code change and redeploy.
  → *Replace with a named environment variable: const MAX_EXTRACT_CHARS = parseInt(process.env.PDF_EXTRACT_MAX_CHARS ?? '30000', 10); then use it as pdfData.text.slice(0, MAX_EXTRACT_CHARS). Add a comment explaining the OpenAI context-window rationale so the value is not changed blindly.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch. If the DELETE fails (e.g., DB timeout), the unhandled rejection will propagate as an uncaught error in the worker process. While the job framework may catch it at a higher level, there is no guarantee, and there is no logging of the cutoff date or row count before the operation to aid post-mortem debugging.
  → *Wrap the db.delete call in try/catch, log the cutoff date at INFO level before executing, and re-throw after logging so the job framework can record the failure and schedule a retry: try { console.info('purge-ai-usage cutoff', cutoff.toISOString()); const result = await db.delete(...); ... } catch (err) { console.error('purge-ai-usage failed', err); throw err; }*
- **`shared/schema.ts` (varchar('user_id') — all tables):** All userId foreign key columns are declared as varchar with no length specified. PostgreSQL will accept any length, providing no guard against malformed IDs and wasting storage. For a UUID-backed auth system this should be varchar(36) or, better, uuid type; for Clerk/Auth.js string IDs it should be varchar(255) with an explicit max.
  → *Determine the exact ID format from the auth provider (e.g., Clerk uses 'user_xxxxxxxxxxxxxxxxxxxxxxxx', max ~32 chars). Replace `varchar('user_id')` with `varchar('user_id', { length: 255 })` across all tables as a safe upper bound, or switch to `uuid` type if IDs are UUIDs. Apply consistently in shared/models/auth.ts as well.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt (and all other createdAt columns)):** All createdAt timestamp columns use `.defaultNow()` but are not marked `.notNull()`. This means application code or a raw insert can store a NULL createdAt, breaking any audit trail or time-based ordering query silently.
  → *Chain `.notNull()` after `.defaultNow()` on every createdAt column, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. The database default ensures inserts without an explicit value still succeed.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** simulatedStocks.ticker has no unique index. Two rows with ticker 'BTC' would silently coexist, causing ambiguous portfolio lookups and double-display bugs in the UI.
  → *Add a unique index in the table callback: `(t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })`. Also consider adding a composite unique on (ticker, region) if the same ticker can exist across markets.*
- **`shared/schema.ts` (portfolioTransactions.executedAt, portfolioHoldings.purchasedAt):** executedAt and purchasedAt both use `.defaultNow()` without `.notNull()` and neither table has an index on userId+executedAt, which is the natural sort/filter key for trade history queries. Nullable timestamps on financial transaction records also create auditability gaps.
  → *Mark both columns `.defaultNow().notNull()`. Add a composite index on portfolioTransactions: `userExecIdx: index('idx_portfolio_tx_user_exec').on(t.userId, t.executedAt)`. This will make paginated trade-history queries use an index scan instead of a seq scan.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed and a new object reference is created on every render inside AuditLogPanel, but more critically the queryFn closes over `params` which is rebuilt each render. This is benign only because queryKey changes drive refetches, but the params object itself is never memoized, causing a new function closure on every render even when deps haven't changed.
  → *Wrap params construction and queryFn in useMemo: `const { params, queryFn } = useMemo(() => { const p = new URLSearchParams(); ... return { params: p, queryFn: () => apiRequest(...) }; }, [actorType, action, from, to]);` This prevents closure churn and makes the dependency contract explicit.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** ObservabilityCard has no loading or error state. When the `/api/admin/observability` query is in-flight or fails, the component silently renders with all `data` fields undefined, causing every Badge and anchor to render with fallback/broken values and no user feedback.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when `isLoading` is true, and an error banner with a retry button when `isError` is true: `if (isLoading) return <Skeleton />; if (isError) return <ErrorBanner onRetry={refetch} />;`*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** `fmtDate` accepts `any` and silently calls `new Date(d)` on arbitrary input. Passing a malformed string or numeric epoch from an API response produces an 'Invalid Date' string rendered directly to the UI, which is a poor UX and also a latent bug if callers assume a clean string is always returned.
  → *Add explicit type and validation: `function fmtDate(d: string | number | null | undefined): string { if (!d) return '—'; const dt = new Date(d); return isNaN(dt.getTime()) ? '—' : dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); }`*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel queryFn / apiRequest call):** The audit log endpoint is called with `limit` hardcoded to `'200'` as a string literal in the component body. This value controls how many audit rows are fetched and rendered into the DOM at once with no virtualization mentioned. 200 unvirtualized rows in a table will cause measurable render jank on low-end devices used by school administrators in the Caribbean target market.
  → *Move the limit to an environment-aware config constant (e.g., `AUDIT_LOG_PAGE_SIZE = 50`) and implement pagination controls already partially scaffolded by the ChevronLeft/ChevronRight imports. Use a `page` state variable and pass `offset = page * AUDIT_LOG_PAGE_SIZE` to the API to keep DOM node count bounded.*

## 💡 Opportunities

### Centralise ownership guard into a reusable helper — 🔥 High Impact
Patterns like and(eq(table.id, id), eq(table.userId, userId)) are repeated across deleteTransaction, deleteBudget, deleteSavingsGoal, deleteExamPaper, and others. Extract a typed assertOwnership(db, table, id, userId) helper that throws a typed NotFoundOrForbidden error if the row doesn't belong to the user. This eliminates the risk of a future method forgetting the userId clause — a significant auth bypass risk for financial records.

### Add query result caching for learningModules and marketStocks — ⚡ Medium Impact
getLearningModules and getMarketStocks return mostly static data that is read on every student dashboard load but written rarely. Adding a short-lived in-process cache (node-cache or a Map with a TTL of 60s) would eliminate repeated full-table reads. At 30 concurrent students this halves the read load on those tables immediately.

### Replace raw sql template expressions with parameterised Drizzle helpers — ⚡ Medium Impact
Several aggregate queries use sql`SUM(...)` raw template literals. While Drizzle does parameterise these, the pattern makes it easy for a future contributor to interpolate a variable directly into the template, introducing SQL injection. Migrating to Drizzle's sum(), count(), and avg() typed aggregation helpers enforces parameterisation structurally and improves type inference on result rows.

### Parallel OpenAI calls in extract-paper — ⚡ Medium Impact
The vision/OCR call and the follow-up extraction call are inherently sequential because one feeds the other, but the paper status update (updateExamPaper) and the bulk question insert are independent after the insert completes. More impactfully, if multiple papers are queued, the worker concurrency limit is the right lever — document it explicitly and set WORKER_CONCURRENCY in environment config so ops can tune throughput without code changes.

### Structured logging with redaction for job payloads — 🔥 High Impact
Job payloads containing fileB64 or any PII (student names, emails in email-contacts jobs) are likely serialised into log lines on job start/failure. Introduce a payload sanitiser that replaces known large or sensitive fields (fileB64, emailAddress, etc.) with '[REDACTED]' before logging. This prevents accidental data exposure in log aggregation tools used by school administrators.

### Idempotency token on admin-csv-export enqueue — ⚡ Medium Impact
If a teacher double-clicks the export button, two identical export jobs can be enqueued and both will write to object storage, wasting compute and potentially serving a stale file. Add a deterministic idempotency key (e.g., hash of type + teacherId + date-hour) at enqueue time and have the job handler skip work and return the existing object path if the file already exists, matching the pattern already used in extract-paper's question wipe.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, simulatedStocks, portfolioTransactions). Extract `export const DEFAULT_CURRENCY = 'BSD'` to a shared constants file and reference it via `.default(DEFAULT_CURRENCY)`. A future currency expansion (TTD, JMD) then requires one change instead of six, eliminating the risk of inconsistent defaults.

### Add a partial index on documentUploads.status for processing queue polling — ⚡ Medium Impact
Background workers likely poll `WHERE status = 'processing'` repeatedly. A partial index `CREATE INDEX idx_doc_uploads_processing ON document_uploads(created_at) WHERE status = 'processing'` would make this query O(processing_rows) rather than O(all_rows). As uploads accumulate this becomes a significant query time difference. Drizzle supports this via the `.where()` modifier on index definitions.

### Replace serial PKs with uuid on user-facing tables — 🔥 High Impact
Using serial (sequential integer) primary keys on transactions, savingsGoals, and documentUploads exposes row counts and insert rates via predictable IDs in API responses (e.g. /uploads/1, /uploads/2). Switch to `uuid('id').defaultRandom().primaryKey()` on these tables. This prevents enumeration attacks and is safer for a school-aged user base where privacy is a priority, with no meaningful performance downside at this scale.

### Parallel data fetching on dashboard overview load — 🔥 High Impact
The overview tab likely fires multiple sequential useQuery calls (organizations, schools, teachers, students, sponsors counts) that are independent. Ensuring all queries are declared at the same component level with no conditional chaining lets React Query fire them in parallel. If any are currently gated behind resolved data from a prior query, refactor to use `enabled` flags and `Promise.all` in a single combined queryFn to cut dashboard time-to-interactive by the number of sequential round-trips.

### Extract sub-panels into lazy-loaded route segments — 🔥 High Impact
With 13 tabs and a 126k+ character file, all panel code is bundled and parsed on initial load even for users who only visit the overview. Code-split each tab panel using React.lazy + Suspense. This reduces initial parse time — critical on the low-powered devices common in Caribbean school networks — and isolates re-render boundaries so a state change in AuditLogPanel cannot trigger re-renders in the Organizations panel.

### Centralize admin auth guard to prevent accidental exposure — ⚡ Medium Impact
Each panel independently relies on the caller to have already checked `useAdminAuth`. If a future developer adds a new panel and forgets the guard, sensitive data is exposed. Wrap the entire AdminDashboard in a single `<RequireAdminAuth>` component at the router level that redirects to login on auth failure, making the security boundary impossible to accidentally bypass at the panel level.
