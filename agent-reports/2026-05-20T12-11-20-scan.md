## 🔴 Critical (6)

### `server/routes.ts` — AI tutor/chat endpoints (registerChatRoutes, openai usage blocks)
**Issue:** No rate limiting middleware is applied to AI endpoints. Any authenticated student can hammer the OpenAI endpoints in a tight loop, burning quota and incurring unbounded cost. quotaErrorMessage and reserveQuota exist but are only called inside handler logic — there is no HTTP-level throttle to prevent burst abuse before the quota check even runs.

**Fix:** Add express-rate-limit before AI route registration: `import rateLimit from 'express-rate-limit'; const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip, message: { message: 'Too many requests, slow down.' } }); app.use('/api/chat', aiLimiter); app.use('/api/tutor', aiLimiter);` Tune max per your quota tier. This must sit before registerChatRoutes.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full base64-encoded file content is stored in the job payload (fileB64 field). For a multi-page PDF or image this can easily be 5–20 MB per job row, bloating the jobs table, slowing queue polling queries, and risking OOM during deserialization. This data is also included in any job logging, exposing raw document bytes.

**Fix:** Before enqueuing, upload the file buffer to private object storage and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the file from object storage at the start of processing. This keeps job rows small (<1 KB) and keeps file data in the purpose-built store.

### `server/jobHandlers.ts` — registerJobHandler('admin-csv-export') — type AdminRow and data fetch
**Issue:** The variable dat appears to be declared but its initialisation is cut off in the truncated code (let dat with no assignment shown), suggesting a potential undefined-use bug if the type branch falls through without assigning it. Combined with no auth check inside the job handler itself, a misconfigured enqueue call with an arbitrary 'type' value could trigger an export of any table with no validation of the caller's privileges.

**Fix:** Add an exhaustive type guard: if the type value is not in an explicit allowlist (e.g., ['users','transactions','sessions'] as const), throw immediately with a clear error before any DB access. Ensure dat is initialised to an empty array as a default to prevent a runtime reference error on the undefined branch.

### `shared/schema.ts` — userId: varchar('user_id') (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, documentUploads)
**Issue:** varchar('user_id') is declared without a length argument across all tables. In PostgreSQL, varchar without length is equivalent to text, so no truncation occurs, but the intent is unclear and inconsistent with explicit-length best practices. More critically, none of these userId columns have a supporting index, meaning any lookup filtering by userId on these tables (e.g. fetching a user's budgets, goals, or bill reminders) results in a full sequential scan.

**Fix:** Add a length to all user_id varchars matching the auth provider's ID length (e.g. varchar('user_id', { length: 128 })). Add single-column indexes: e.g. userIdx: index('idx_budgets_user').on(t.userId) inside the table's index callback for budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and documentUploads.

### `client/src/pages/AdminDashboard.tsx` — TABS constant (module level) and tab-panel components
**Issue:** The entire dashboard — organizations, schools, teachers, students, classes, sponsors, challenges, reports, jobs, perf, audit, DB viewer — appears to be in a single 126k+ character file. This means every tab's query hooks are registered and potentially fetched on mount, every mutation is bundled into one chunk, and a single compile error breaks the entire admin surface.

**Fix:** Split each tab into a lazily-loaded route or React.lazy component: `const OrgPanel = React.lazy(() => import('./admin/OrgPanel'))`. Wrap each in Suspense with a loading fallback. This eliminates unused query registrations for inactive tabs and reduces the initial JS parse cost significantly.

### `client/src/pages/AdminDashboard.tsx` — apiRequest calls throughout mutation onSuccess / useEffect blocks
**Issue:** No try/catch wrapping is visible around async operations in mutation callbacks and any useEffect async functions in the file. A failed API call in a mutation's onSuccess (e.g., a follow-up refetch or secondary POST) will produce an unhandled promise rejection that Sentry may not attribute correctly, and the UI will show no error state.

**Fix:** Wrap all async mutation callbacks and useEffect async functions: `onSuccess: async () => { try { await queryClient.invalidateQueries(...); } catch (err) { toast({ title: 'Sync error', description: String(err), variant: 'destructive' }); } }`. Also ensure every `useEffect` that sets up intervals or listeners returns a cleanup function to prevent memory leaks on unmount.

## 🟡 Warnings (13)

- **`server/routes.ts` (Google sign-in handler: resolvedClass resolution block (~line 220-240 visible)):** Three sequential awaits — getClassByCode, getOrgEnvironmentById, getOrganization — are chained one after another inside the Google auth flow on every sign-in. getOrgEnvironmentById and getOrganization are independent once classCode is resolved and could run in parallel, adding unnecessary latency to a user-facing auth request.
  → *After confirming resolvedClass.envId exists, fetch env and org concurrently: `const [classEnv, classOrg] = await Promise.all([ getOrgEnvironmentById(resolvedClass.envId), getOrganization(resolvedClass.org_id ?? '') ]); ` You will need to ensure org_id is available on the class record or derive it from classEnv after the first parallel fetch if org_id is not directly on the class.*
- **`server/routes.ts` (initSupabaseTables, initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent calls at startup):** Multiple independent initialization/seed functions are called sequentially at server startup inside registerRoutes. These are I/O-bound Supabase operations with no interdependency. Sequential execution unnecessarily delays server readiness.
  → *Group independent inits: `await Promise.all([ initSupabaseTables(), initStaticContentTables() ]); await Promise.all([ seedStaticModules(), seedRegionalContent(), seedGameContent(), seedFinancialAcademyLesson() ]);` Only sequence calls that have a true dependency (e.g., tables must exist before seeding).*
- **`server/routes.ts` (objectStorage instantiation and appBaseUrl/sendEmail usage):** appBaseUrl is imported from ./email and likely hardcodes a fallback string (e.g. 'http://localhost:3000') when APP_BASE_URL env var is missing. In a school deployment this silently sends verification emails with broken localhost links. Additionally, new ObjectStorageService() is instantiated at module load with no guard — if object storage credentials are absent the entire route registration fails with an opaque error.
  → *1) In email.ts ensure appBaseUrl throws or logs a loud warning at startup if APP_BASE_URL is not set in production (check NODE_ENV). 2) Wrap objectStorage instantiation: `let objectStorage: ObjectStorageService; try { objectStorage = new ObjectStorageService(); } catch(e) { console.error('Object storage unavailable:', e); }` and guard upload routes with a 503 if objectStorage is null.*
- **`server/routes.ts` (audit() calls throughout route handlers):** audit() is called with user-supplied strings (org names, lesson titles, file names) that are interpolated directly into audit log messages without sanitization. If audit log entries are ever rendered in an admin UI without escaping, this is a stored XSS vector — especially risky given the school-aged user base where a student could inject a script tag via a lesson title or class name.
  → *Sanitize all user-supplied values before passing to audit(): create a helper `function sanitizeForAudit(s: string) { return s.replace(/[<>"']/g, '') }` and apply it to any field sourced from req.body before interpolation in audit() call arguments. Long-term, treat audit log display as untrusted HTML and escape on render.*
- **`server/jobHandlers.ts` (runExtractPaper — question insert loop (for loop over questions array)):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this fires 40 serial round-trips to the DB. Combined with two OpenAI calls already in the critical path, this unnecessarily extends job duration and holds a DB connection open across many idle RTTs.
  → *Replace the for loop with a single batched insert: collect all valid question objects into an array first, then call db.insert(extractedQuestions).values(questionsToInsert) once. Add a guard for the empty-array case. This reduces N round-trips to 1 and makes the operation atomic, reinforcing the idempotency delete that precedes it.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage') and registerJobHandler('admin-csv-export')):** Neither handler wraps its DB/storage operations in a try/catch. If the delete or export query throws (e.g., connection timeout), the raw DB error propagates to the job runner unhandled. There is no status update or structured error return, so the job silently fails with an opaque stack trace and no actionable context in the job record.
  → *Wrap the body of each handler in try/catch. On catch, log a structured error (omitting sensitive data) and rethrow a typed Error with a human-readable message so the job runner can record it cleanly. For admin-csv-export, also ensure the object-storage upload is attempted even if row fetching partially fails, or explicitly mark the export as failed in its tracking record.*
- **`server/jobHandlers.ts` (runExtractPaper — OpenAI model name string literals ('gpt-4o')):** The model name 'gpt-4o' is hardcoded in two separate openai.chat.completions.create calls. If the deployment needs to switch models (cost control, regional availability, model deprecation), both call sites must be found and edited manually, with no ability to override per environment.
  → *Read the model name from environment config: const EXTRACTION_MODEL = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'; and reference EXTRACTION_MODEL at both call sites. This lets staging use a cheaper model (e.g., gpt-4o-mini) and lets production be updated via config without a code deploy.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow() — all tables):** createdAt is missing .notNull() on every table. defaultNow() only fires on INSERT from the ORM; a raw SQL insert or a bug in application code can write a NULL, making ORDER BY created_at and time-series reporting silently unreliable.
  → *Append .notNull() to every createdAt column: timestamp('created_at').defaultNow().notNull(). Apply the same fix to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table, ticker column):** ticker has no unique index. Two rows with the same ticker symbol (e.g. 'BTC') can be inserted, causing ambiguous joins with portfolioHoldings and portfolioTransactions and incorrect portfolio valuations for school-aged users learning from this data.
  → *Change the column definition to ticker: text('ticker').notNull().unique(), or add a named unique index in the table callback: uniqueTickerIdx: uniqueIndex('idx_stocks_ticker').on(t.ticker). Also confirm ticker should be .notNull() — it currently lacks that constraint.*
- **`shared/schema.ts` (portfolioTransactions table, executedAt column):** executedAt uses .defaultNow() but is not in any index.   → *Add a composite index in the portfolioTransactions index callback: userExecutedIdx: index('idx_portfolio_tx_user_executed').on(t.userId, t.executedAt). Mirror this for portfolioHoldings on (userId, purchasedAt) if historical holding snapshots are queried.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed and queryFn closure is recreated on every render without memoization. The params object is built outside useMemo, so even unrelated state changes in the parent cause a new object reference, potentially triggering spurious refetches depending on how the queryKey array is compared.
  → *Wrap the params construction and queryFn in useMemo: `const { params, queryFn } = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return { params: p, queryFn: () => apiRequest('GET', `/api/admin/audit-log?${p}`).then(r => r.json()) }; }, [actorType, action, from, to]);` and pass queryFn directly to useQuery.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no error state handled and no loading skeleton. If /api/admin/observability returns a 401 or 500, the component silently renders empty badges with no user feedback, which is especially risky in an admin context where misconfigured Sentry/alerting should be surfaced clearly.
  → *Destructure `isLoading` and `error` from useQuery. Render a loading skeleton (e.g., Loader2 spinner) when isLoading is true, and an error alert (using the existing AlertTriangle icon) when error is truthy. Add `retry: false` to the query options so auth failures don't retry three times before surfacing.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** `new Date(d)` called with an untyped `any` parameter with no validation. Passing an invalid date string (e.g., a malformed ISO string from the API) silently produces 'Invalid Date' in the UI. ## 💡 Opportunities

### Split monolithic routes.ts into domain-scoped routers — 🔥 High Impact
At ~200k+ characters, this single file is unreviable and unmaintainable. Extract auth, AI/tutor, lesson management, org admin, and student progress into separate Express Router files under server/routes/. This directly reduces the blast radius of a bug (a syntax error won't take down all routes) and makes targeted security review feasible — critical for an app handling minors' data.

### Centralise AI quota enforcement in middleware, not handler logic — 🔥 High Impact
reserveQuota/finalizeUsage/releaseReservation are called manually inside each AI handler, making it easy to forget in new endpoints (as seen with missing rate limiting). Extract into a reusable Express middleware factory: `aiQuotaMiddleware(orgId)` that attaches reservation to res.locals and registers a res.on('finish') hook to finalize/release. This guarantees consistent enforcement and eliminates duplicated try/finally quota cleanup blocks.

### Replace repeated per-request getOrganization DB calls with a request-scoped cache — ⚡ Medium Impact
getOrganization and getOrgQuotaSettings are called multiple times within the same request lifecycle across nested function calls. Add a lightweight Map keyed by orgId in a res.locals.orgCache populated once per request via middleware. For a school app with potentially hundreds of concurrent student sessions, this reduces redundant Supabase round-trips measurably and lowers costs.

### Parallelize independent async operations in extract-paper — ⚡ Medium Impact
The idempotency delete (db.delete extractedQuestions) and the file-download/parse step are independent. They can run concurrently with Promise.all, shaving one full DB round-trip off the critical path before the first OpenAI call. On a 200 ms DB latency this is a free 200 ms reduction per job.

### Introduce a dedicated file-staging table instead of object-path strings in payloads — ⚡ Medium Impact
Rather than passing object paths as raw strings through job payloads, a small file_uploads table (id, object_path, content_type, uploaded_by, expires_at) would enable TTL-based cleanup of orphaned uploads, audit trails, and referential integrity checks before processing starts. This prevents jobs from silently processing stale or deleted files.

### Add per-handler structured telemetry with duration and token usage — 🔥 High Impact
Both OpenAI calls in runExtractPaper return usage objects (prompt_tokens, completion_tokens). Capturing these alongside job duration into aiUsageEvents inside the handler (rather than relying on middleware) would give accurate per-paper cost attribution, enabling the purge-ai-usage job to make informed retention decisions and giving admins real cost-per-upload visibility.

### Centralise currency default as a shared enum or config constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 7 separate columns (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks, linkedCards implied). Extract a DEFAULT_CURRENCY = 'BSD' constant from an environment variable or shared config and reference it in all column defaults. This prevents silent regional mismatches if the app expands to Trinidad (TTD) or Jamaica (JMD) and makes currency policy change a one-line update.

### Replace repeated currency text columns with a shared pgEnum — ⚡ Medium Impact
Seven tables each define currency as a free-form text column with only an application-level default. Introduce a currencyEnum = pgEnum('currency', ['BSD', 'TTD', 'JMD', 'USD']) and use it across all tables. This enforces valid values at the database level, eliminates an entire class of silent bad-data bugs, and makes adding a new currency a single migration rather than seven.

### Add updated_at columns to mutable tables — 🔥 High Impact
budgets, savingsGoals, billReminders, and portfolioHoldings are user-editable but have no updatedAt timestamp. Without it, cache invalidation, audit logging, and conflict resolution (if offline sync is added) are impossible without expensive full-row compares. Add updatedAt: timestamp('updated_at').defaultNow().notNull() and a DB-level trigger or ORM hook to keep it current on every UPDATE.

### Parallel query fan-out for overview stats — 🔥 High Impact
If the Dashboard/overview tab fetches multiple independent endpoints (organizations count, students count, challenges, etc.) sequentially via chained awaits or waterfall query dependencies, replacing them with Promise.all or using independent parallel useQuery calls would cut initial load time proportionally to the number of sequential round trips — likely 3-5x faster on a Caribbean mobile connection with high latency.

### Memoize filtered/sorted table data in all list panels — ⚡ Medium Impact
Search filter and sort logic applied to rows arrays (students, teachers, schools, etc.) should be wrapped in useMemo with the raw data and filter state as dependencies. Without this, every keystroke in any unrelated input that triggers a parent re-render reprocesses potentially hundreds of rows. Expected impact: eliminates jank during typing in search fields.

### Extract hardcoded limit=200 in AuditLogPanel to config constant — 📌 Low Impact
The hardcoded `params.set('limit', '200')` in AuditLogPanel should be a named constant (AUDIT_LOG_PAGE_SIZE) or pulled from an environment-aware config object. At 200 rows the payload is manageable, but as audit volume grows this will silently truncate results with no pagination UI. Adding a named constant now makes it trivial to wire up proper pagination later.
