## 🔴 Critical (8)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (investment buy/sell flow)
**Issue:** No database transaction wraps the three-step invest operation: upsert holding, insert portfolio transaction, and update virtual balance. A crash between any two steps leaves the database in an inconsistent state — e.g., balance deducted but holding never recorded, or holding updated but transaction log missing.

**Fix:** Wrap all three operations in a single `db.transaction(async (tx) => { ... })` call using Drizzle's transaction API. Pass the `tx` client into each sub-operation so they share the same connection and atomicity guarantee. Roll back automatically on any thrown error.

### `server/storage.ts` — getLeaderboard
**Issue:** Leaderboard query has no enforced maximum row cap. The `limit` filter is optional and callers can omit it entirely, causing a full-table scan of gameSessions joined to users to be returned to the client. With a school rollout this could return tens of thousands of rows.

**Fix:** Apply a hard server-side cap: `const safeLimit = Math.min(filters?.limit ?? 50, 100)` and always pass `safeLimit` to the query `.limit(safeLimit)`. Never let the caller control an unbounded result set.

### `server/routes.ts` — reserveQuota / AI tutor endpoints
**Issue:** No HTTP-level rate limiting on AI/heavy endpoints. The quota system (reserveQuota/finalizeUsage) is application-level only — a user can fire hundreds of concurrent requests before any reservation is recorded, bypassing per-org limits entirely due to the TOCTOU window between check and reserve.

**Fix:** Add express-rate-limit middleware scoped per userId+orgId before any route that calls reserveQuota. Example: rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => `${req.user?.id}:${req.user?.orgId}` }). Apply this as a named middleware constant and mount it on all /api/ai/* and /api/tutor/* route groups.

### `server/routes.ts` — videoUpload multer config — limits.fileSize
**Issue:** videoUpload allows 500 MB files into memory (multer.memoryStorage()). A single malicious or accidental upload of a 500 MB file will exhaust Node.js heap, crashing the server for all concurrent users. This is especially dangerous with school-aged users who may upload large files unknowingly.

**Fix:** Switch videoUpload to multer.diskStorage() writing to a temp directory (os.tmpdir()), then stream the file directly to object storage using a pipe rather than buffering the entire file in req.file.buffer. Add a cleanup step in a finally block to unlink the temp file after upload completes or fails.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored directly in the job payload. For a PDF or image this can be several megabytes per row, bloating the jobs table, slowing queue polling queries, and hitting any payload size limits on the queue backend. On retry the full payload is re-read from the DB each time.

**Fix:** Upload the file to private object storage immediately when the teacher submits the paper (in the HTTP handler), store only the resulting object path in the job payload (e.g. { paperId, objectPath, ext, subject }), then have runExtractPaper download the file from object storage at the start of the job. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts` — runExtractPaper — for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted one-at-a-time with sequential awaits inside a for-loop. For a 40-question paper this is 40 serial round-trips to the database. If any single insert fails mid-loop the paper is left in a partially-inserted state, and the leading delete-for-idempotency at the top of the function will wipe valid questions on the next retry before re-inserting.

**Fix:** Collect all valid question objects into an array, then call a single bulk-insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the delete + bulk-insert in a single database transaction so partial failures leave no orphaned rows and retries are truly idempotent.

---

### `shared/schema.ts` — portfolioHoldings table definition
**Issue:** There is no unique constraint on (userId, stockId) in portfolio_holdings. A race condition or double-submit can insert two rows for the same stock, corrupting the user's portfolio balance silently.

**Fix:** Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) in the table's index callback. Application upsert logic should then use ON CONFLICT DO UPDATE to adjust quantity and avgPurchasePrice atomically.

### `client/src/pages/AdminDashboard.tsx` — useMutation callbacks (onSuccess/onError handlers throughout)
**Issue:** Mutation queryFn calls via apiRequest are not wrapped in try/catch. If apiRequest throws a network-level error (not an HTTP error response), React Query will catch it and call onError, but any synchronous post-processing code inside the mutationFn (e.g., response parsing with .json()) that throws will produce an unhandled promise rejection rather than a clean onError invocation in some React Query v4 configurations.

**Fix:** Wrap the async body of every mutationFn: `mutationFn: async (payload) => { try { const res = await apiRequest('POST', '/api/admin/...', payload); if (!res.ok) throw new Error(await res.text()); return res.json(); } catch (e) { throw e; } }` and ensure onError always calls `toast({ variant: 'destructive', ... })` so admins see actionable feedback instead of a silent failure.

## 🟡 Warnings (16)

- **`server/storage.ts` (getDashboardStats):** Dashboard stats likely fires sequential awaits for total income, total expenses, savings goals, and budget data — each a separate round-trip to the database. On a Caribbean mobile network with higher latency this compounds noticeably and holds an open connection slot per sequential query.
  → *Group all independent read queries into a single `const [income, expenses, goals, budgets] = await Promise.all([...])` call. Each query is independent so there is zero correctness risk and the wall-clock time drops to the slowest single query instead of their sum.*
- **`server/storage.ts` (getPortfolioHoldings / getBillReminders (and any method doing per-row category/stock joins in JS)):** N+1 pattern risk: if either method fetches a list of rows then resolves related entities (category, stock) with individual `getStockById` or `getCategoryById` calls inside a loop, each row produces an additional database round-trip. For a student with 20 holdings that is 21 queries per page load.
  → *Replace the loop-with-lookup pattern with a single JOIN query using Drizzle's relational API or an explicit `leftJoin`. For example: `db.select().from(portfolioHoldings).leftJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. This collapses N+1 into one query.*
- **`server/storage.ts` (getTransactions (filters parameter)):** The `limit` field inside the filters object is accepted directly from the caller with no upper bound enforced. A caller passing `limit: 999999` will attempt to stream the entire transaction history into memory and then to the client, both a memory and a data-exposure risk for a minor's financial record.
  → *Enforce `const safeLimit = Math.min(filters?.limit ?? 50, 200)` inside `getTransactions` before constructing the query. Also ensure the route layer requires authentication before this method is reachable — confirm `userId` is always sourced from the verified session, never from the request body or query string.*
- **`server/routes.ts` (Google sign-in handler — sequential awaits inside classCode resolution block):** getOrgEnvironmentById and getOrganization are called sequentially inside the classCode resolution block. These are independent DB lookups that add latency on every Google sign-in with a class code, directly impacting the student login experience.
  → *Replace the two sequential awaits with: const [classEnv, ...otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), ...]). The organization fetch depends on classEnv.org_id, so restructure: fetch classEnv first, then batch any remaining independent calls. At minimum, audit all consecutive await pairs in this handler and parallelize those with no data dependency.*
- **`server/routes.ts` (googleEnabled() check and appBaseUrl usage):** appBaseUrl (imported from ./email) and similar config values are referenced as runtime imports rather than validated environment variables at startup. If GOOGLE_CLIENT_ID or BASE_URL are missing, failures surface mid-request rather than at boot, making misconfigured deployments hard to diagnose.
  → *Create a server/config.ts that reads and validates all required env vars with zod at process start (z.object({ GOOGLE_CLIENT_ID: z.string().min(1), APP_BASE_URL: z.string().url(), ... }).parse(process.env)). Import the validated config object everywhere instead of reading process.env inline or relying on helper functions that silently return undefined.*
- **`server/routes.ts` (audit() calls and error logging throughout route handlers):** Several catch blocks and audit() calls likely log req.body or user-supplied strings directly. For a school-aged user app this risks leaking student PII (email, name, join codes) into audit logs and any connected log aggregator.
  → *Create a sanitizeForLog(obj) utility that strips keys matching /email|password|token|idToken|name/i before passing to audit() or console.error. Apply it to all catch block error objects and all audit() payloads. Also ensure bcrypt hash outputs are never passed to audit().*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage') and all other handlers except extract-paper):** Most job handlers have no try/catch around their async DB or external-service calls. An unhandled rejection inside a handler that the job runner does not itself wrap will cause the Node process to emit an unhandledRejection, potentially crashing the worker or silently marking the job as failed without updating any dependent UI state (contrast with extract-paper which explicitly syncs status on terminal failure).
  → *Either ensure the registerJobHandler runner wraps every handler in a try/catch and records failure metadata, or add explicit try/catch inside each handler that touches external services (db, openai, objectStorageClient), updating relevant status columns on terminal failure the same way extract-paper does.*
- **`server/jobHandlers.ts` (runExtractPaper — model: "gpt-4o" (two occurrences)):** The OpenAI model name is hardcoded as the string literal "gpt-4o" in two separate openai.chat.completions.create calls. Switching models for cost or capability reasons requires a code change and redeploy rather than an environment variable update, and the vision call and extraction call could silently diverge if only one is updated.
  → *Add OPENAI_MODEL to environment config (e.g. process.env.OPENAI_MODEL ?? 'gpt-4o') and reference it from a single shared constant at the top of the file. Apply the same constant to both calls so they stay in sync.*
- **`server/jobHandlers.ts` (runExtractPaper — visionResponse image_url content):** When processing an image file, the raw buffer is converted back to base64 and embedded inline in the OpenAI API request body (data:image/png;base64,...). This means the multi-megabyte base64 string is held in memory twice simultaneously (once as the input buffer, once as the inlined string in the messages array) and is also serialised into the HTTP request body sent to OpenAI, which has a request body size limit and will fail silently for large scans.
  → *After adopting object storage for the file (see finding 1), generate a short-lived signed URL for the uploaded file and pass that as image_url.url instead of a data URI. This eliminates the double in-memory copy and removes the request-body size constraint.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId even though every budget query will filter by userId. With even modest user counts this becomes a full table scan on every budget load.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a uniqueIndex on (userId, categoryId, period) to prevent duplicate budget rows for the same category/period combination.*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, etc.)):** userId is typed as varchar with no length argument, which maps to varchar without a length limit in Postgres — effectively text. This is fine for storage but signals no deliberate sizing decision. More importantly, if the auth provider (e.g. Clerk) issues fixed-length IDs, an unbounded varchar skips the cheap DB-level length validation that would catch corrupted IDs before they propagate.
  → *Pin the length to match your auth provider's actual ID format, e.g. varchar('user_id', { length: 32 }) for Clerk user IDs. Add a CHECK constraint or rely on the FK to users.id to enforce referential integrity — confirm users.id in ./models/auth also uses the same length.*
- **`shared/schema.ts` (createdAt columns on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is defined as timestamp().defaultNow() without .notNull(). Drizzle will not add a NOT NULL constraint, so a manually constructed INSERT omitting createdAt produces a NULL audit timestamp, breaking any time-based reporting or ordering queries that assume this column is always populated.
  → *Change every createdAt definition to timestamp('created_at').defaultNow().notNull(). Same applies to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker has no unique index. Duplicate tickers would cause silent data integrity issues — lookups by ticker symbol would return unpredictable rows, and the investment simulation could display wrong prices or double-count holdings.
  → *Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table's index callback. If multi-region tickers can collide, make the unique index on (ticker, region) instead.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams is reconstructed and a new object reference is created on every render inside AuditLogPanel, but the queryKey uses the raw state values. This means the params object passed to queryFn is always fresh but the key correctly invalidates — however the params construction happens outside useMemo, so any parent re-render causes a new params object and a new queryFn closure even when state has not changed, causing React Query to re-evaluate staleness unnecessarily.
  → *Wrap the URLSearchParams construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]);` This keeps the queryFn reference stable and avoids redundant closure allocations.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** ObservabilityCard and similar sub-panel components (AuditLogPanel, etc.) have no error state handling. If /api/admin/observability returns a non-2xx response or throws, useQuery sets isError=true but the component silently renders with undefined data, showing blank or broken UI to admins with no recovery path.
  → *Destructure `isError` and `error` from useQuery and render an explicit error state: `if (isError) return <Card ...><CardContent><p className='text-red-400'>Failed to load observability data. <button onClick={() => refetch()}>Retry</button></p></CardContent></Card>;` Apply the same pattern to every sub-component that fetches data.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and tab-switching logic):** TABS is defined at module scope as a plain array literal, which is fine, but the active-tab render logic (large inline conditional blocks or switch rendering each tab's full subtree) is inlined in JSX without React.memo on any sub-panel component. Every keystroke in a search Input inside one panel causes the entire AdminDashboard to re-render and re-evaluate all tab panel expressions, even for hidden panels.
  → *Extract each tab panel (OverviewPanel, OrganizationsPanel, etc.) into separate React.memo-wrapped components. Pass only the props each panel needs. This ensures that typing in the Students search box does not trigger reconciliation work for the Charts, AuditLog, or DBViewer panels.*

## 💡 Opportunities

### Centralised pagination helper — 🔥 High Impact
Introduce a single `clampLimit(requested?: number, default_ = 50, max = 200): number` utility imported by every list method. This eliminates the repeated pattern of ad-hoc limit handling and guarantees no future method accidentally ships without a cap — directly reducing memory and DB load at scale.

### Query-layer JOIN consolidation for relational reads — 🔥 High Impact
Audit every method that returns a type like `PortfolioHolding & { stock: SimulatedStock }` and confirm the join happens in SQL rather than JavaScript. Switching the remaining N+1 patterns to SQL joins could reduce per-request DB round-trips by 60-80% for data-heavy student dashboards, which matters on shared school infrastructure.

### Structured error context on storage throws — ⚡ Medium Impact
Wrap all storage method bodies in try/catch that re-throws with a structured error including the method name and sanitised parameters (no PII). This lets the route layer return consistent error shapes and prevents raw Postgres error messages — which can expose table names and column structures — from leaking to API consumers.

### Split monolithic routes.ts into domain-scoped route modules — 🔥 High Impact
The file is 196k+ characters and handles auth, AI, lessons, orgs, uploads, email, and leaderboards in one file. Extract each domain into its own router file (e.g. routes/lessons.ts, routes/ai.ts, routes/auth.ts) and mount them in a thin index. This will make rate limiting, auth middleware, and error handling easier to apply consistently per domain and reduce the risk of copy-paste security gaps.

### Centralize async route error handling with a wrapper — 🔥 High Impact
Wrapping every handler in try/catch is error-prone and inconsistently applied across 196k chars of routes. Introduce an asyncRoute(fn) wrapper that catches rejections and calls next(err), then add a single Express error-handling middleware that maps known error types (ZodError → 400, quota errors → 429, auth errors → 401) to correct HTTP responses. This eliminates unhandled promise rejection risk and standardizes error response shapes for the frontend.

### Batch independent Supabase/DB calls in student enrollment flow — ⚡ Medium Impact
The Google sign-in and class enrollment flow makes at least 4-5 sequential round-trips to Supabase (getClassByCode, getOrgEnvironmentById, getOrganization, enrollStudentInOrg, getStudentOrgIds). Profiling will likely show 300-600ms of avoidable latency per login. Audit all await chains in auth and enrollment handlers and apply Promise.all for independent fetches. For a school app where 30 students may log in simultaneously at class start, this has outsized impact.

### Batch admin CSV export via streaming cursor — 🔥 High Impact
The admin-csv-export handler almost certainly loads all rows for a given type into a JS array before writing. For large schools this will OOM the worker. Replace the full-table fetch with a Drizzle streaming cursor or paginated query (1000 rows at a time) piped directly into a stream-based CSV serialiser (e.g. csv-stringify in streaming mode) writing to object storage. Expected impact: constant memory usage regardless of dataset size, eliminating OOM crashes on export.

### Centralise terminal-failure status sync into job runner middleware — ⚡ Medium Impact
The pattern of catching errors on the final attempt and updating a status column (seen in extract-paper) will need to be duplicated for every future job type that has a corresponding UI status. Extract this into a wrapper inside registerJobHandler that accepts an optional onTerminalFailure callback, keeping handler code focused on the happy path. Reduces future bug surface and ensures consistent UX across all job types.

### Deduplicate OpenAI call logic into a shared typed helper — ⚡ Medium Impact
openai.chat.completions.create is called directly in at least two places in this file with overlapping options (model, max_completion_tokens, response_format). Wrapping it in a typed helper that enforces the model constant, handles JSON parse errors with a structured result type, and logs token usage to aiUsageEvents in one place will eliminate the duplicated error-handling surface and ensure usage tracking is never accidentally omitted on a new call site.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' in a shared config file and reference it everywhere. When the app needs to support TTD or JMD, there is then exactly one place to change and zero risk of a missed column.

### Add composite index on billReminders (userId, nextDueDate) — ⚡ Medium Impact
Reminder queries almost certainly filter by userId and order or filter by nextDueDate to surface upcoming bills. Without a composite index the planner does a userId scan then sorts. Adding index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) converts the upcoming-bills query to an index range scan, which matters as reminder rows accumulate over time.

### Add index on portfolioTransactions (userId, executedAt) for history pagination — 🔥 High Impact
Trade history views page through portfolioTransactions ordered by executedAt per user. Without an index this is a sequential scan plus sort on every page load. Adding index('idx_portfolio_txns_user_date').on(t.userId, t.executedAt) makes paginated history O(log n) and keeps the feature responsive as transaction volume grows.

### Parallel data fetching on dashboard load — 🔥 High Impact
The overview tab likely triggers multiple sequential useQuery calls (stats, charts, observability, audit summary) that are independent. Confirm all overview queries fire concurrently by ensuring they are not inside conditional blocks that gate on previous query results. Use React Query's `enabled` flag carefully and consider a single /api/admin/overview aggregate endpoint to cut round-trips from ~5 to 1, reducing initial paint time by several hundred milliseconds on Caribbean mobile networks.

### Paginate audit log instead of fetching 200 rows — 🔥 High Impact
The hardcoded `limit=200` in AuditLogPanel sends up to 200 full audit records over the wire on every filter change. Implement cursor-based pagination (limit=50, before/after cursor) with a 'Load more' button. This reduces payload size by ~75% and query latency proportionally, which matters on low-bandwidth school networks.

### Extract AdminDashboard into a route-based code-split bundle — ⚡ Medium Impact
AdminDashboard imports Recharts, Lucide (30+ icons), and all panel components in a single chunk. Since only admin users ever load this page, wrapping it in React.lazy + Suspense with a dynamic import will remove this weight from the main bundle, improving initial load time for the majority student/teacher audience who never visit /admin.

### Database Layer (`server/storage.ts`)
storage.ts shows several reliability and security concerns typical of rapid MVP development: missing database transactions around multi-step writes, sequential awaits where parallel execution is possible, and no pagination guards on unbounded queries that could return thousands of rows for active student users. The codebase appears functional but is not production-safe for a financial literacy app handling minors' data.

routes.ts is a large, monolithic route file with several reliability and security gaps. Error boundaries are inconsistently applied across handlers, and one hardcoded OpenAI model string appears in multiple places rather than being pulled from environment config.

The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-traffic filter columns, unbounded varchar types for user IDs, missing NOT NULL constraints on audit timestamps, and no unique constraint guarding duplicate portfolio holdings per user/stock pair.
