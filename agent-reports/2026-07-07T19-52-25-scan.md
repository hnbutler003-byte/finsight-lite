## 🔴 Critical (10)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate SELECT for each budget's category and a separate aggregate query for spent amount per budget, producing 2N+1 database round-trips for N budgets.

**Fix:** Replace the per-budget loops with a single JOIN query using Drizzle's leftJoin(categories, eq(budgets.categoryId, categories.id)) and a subquery or groupBy aggregate for spent amounts. Return all data in one round-trip: db.select({...budgetFields, categoryName: categories.name, spent: sql<number>`coalesce(sum(...))`}).from(budgets).leftJoin(categories,...).leftJoin(transactions,...).where(eq(budgets.userId, userId)).groupBy(budgets.id).

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional limit but defaults to no cap or a hardcoded value, and performs a full table scan across gameSessions joined to users with no maximum row limit enforced at the query level, risking unbounded memory consumption as game data grows.

**Fix:** Enforce a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 20, 100); and apply .limit(safeLimit) on the Drizzle query unconditionally before it reaches the database. Never trust the caller-supplied limit alone.

### `server/storage.ts`: enrollStudent / createClass / completeModule
**Issue:** Missing database transactions on multi-step writes: operations like enrollStudent (insert enrollment + update class count) and completeModule (insert progress + update XP + potentially award badge) perform multiple sequential INSERTs/UPDATEs outside a db.transaction() block, leaving the database in a partially updated state if any step throws.

**Fix:** Wrap all multi-step mutations in db.transaction(async (tx) => { await tx.insert(...); await tx.update(...); }). Drizzle's transaction callback automatically rolls back on any thrown error, ensuring atomicity.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 108-110)
**Issue:** OpenAI chat, image, and audio routes are registered with no rate limiting middleware. These are expensive external API calls billed per request and could be abused by school-aged users or bad actors to exhaust quotas or incur cost spikes.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registering these routes: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap.xml handler (~line 63)
**Issue:** Both handlers construct absolute URLs by reading `req.headers['x-forwarded-host']` and `req.headers.host` directly without validation. An attacker can send a crafted Host header to inject arbitrary content into the Sitemap directive or XML, enabling cache poisoning or phishing via search-engine-indexed sitemap output.

**Fix:** Replace the dynamic host resolution with a trusted value from environment config: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and remove all header-derived host/proto logic from both handlers. Set APP_BASE_URL in Railway environment variables.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Full file contents are stored as base64 in the job payload (fileB64 field). For a typical exam paper PDF this can be 1-5MB per job row, bloating the jobs table, slowing queue polling queries, and risking the payload appearing in structured logs or error reports sent to Sentry.

**Fix:** Upload the file to object storage before enqueuing, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `shared/schema.ts`: simulatedStocks table — ticker column
**Issue:** ticker has no unique index. Duplicate tickers can be inserted silently, which will cause incorrect portfolio value calculations when joining holdings to stocks by ticker symbol.

**Fix:** Add a unique index in the table's second argument: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }).

### `server/index.ts`: seedDatabase / probeDatabase calls in startup IIFE
**Issue:** Sequential awaits on probeDatabase, seedDatabase, registerRoutes, and startJobWorker run one-after-another at startup. probeDatabase and seedDatabase have no inter-dependency and could run in parallel, adding unnecessary latency to time-to-ready. More critically, if any of these async calls throw an uncaught rejection the process exits without running any cleanup, because the startup IIFE lacks a .catch() or top-level try/catch that calls httpServer.close().

**Fix:** Wrap the startup IIFE in try/catch with a finally that closes the httpServer. Parallelise independent steps: `const [_, seed] = await Promise.all([probeDatabase(), seedDatabase()]);`. Ensure registerRoutes and startJobWorker are only awaited after those resolve.

---

### `server/index.ts`: setInterval callbacks for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are created but the interval handles are never stored and never passed to clearInterval() in the graceful-shutdown handler. On SIGTERM/SIGINT the intervals remain active, preventing the Node event loop from draining, meaning httpServer.close() may never fire its callback and the process hangs or is force-killed, risking mid-flight job corruption.

**Fix:** Capture handles: `const weeklyDigestInterval = setInterval(...); const purgeInterval = setInterval(...);`. In the shutdown handler call `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);` before or alongside `httpServer.close()`.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth – route guard usage
**Issue:** If `useAdminAuth` returns a loading state before confirming admin status, any mutations (delete user, promote role, etc.) rendered in the same component tree are mounted and callable before auth is confirmed. There is no early-return guard that prevents rendering mutation-triggering UI while auth is still resolving, creating a window where unauthenticated users could invoke mutations if the auth check is async.

**Fix:** At the top of AdminDashboard add: `const { admin, isLoading } = useAdminAuth(); if (isLoading) return <Loader2 className='animate-spin' />; if (!admin) return null;` before any other hooks or JSX to ensure the entire dashboard, including all mutation buttons, only renders after auth is confirmed.

## 🟡 Warnings (17)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits that should be parallelised: getDashboardStats issues at least 4-5 independent SELECT queries (total income, total expenses, savings goals, budgets, recent transactions) with sequential await calls, adding their latencies in series rather than concurrently.
  → *Collect all independent queries into Promise.all: const [income, expenses, goals, budgets, recent] = await Promise.all([db.select(...), db.select(...), db.select(...), db.select(...), db.select(...)]); This reduces dashboard load time to the duration of the slowest single query instead of the sum of all queries.*
- **`server/storage.ts` (getTransactions / getExamPapers / getGameSessions):** Unbounded list queries with no server-enforced row limit: getTransactions accepts an optional limit filter but does not enforce a maximum, and getExamPapers / getGameSessions have no limit at all. A user with thousands of records can trigger full-table scans and large heap allocations per request.
  → *For getTransactions apply .limit(Math.min(filters?.limit ?? 50, 500)) unconditionally. For getExamPapers and getGameSessions add a default .limit(200) and expose a paginated interface (offset/cursor) in the IStorage interface. Also add .orderBy(desc(transactions.date)) before limit to ensure deterministic pagination.*
- **`server/routes.ts` (Supabase init chain (~line 113-121)):** Seven Supabase operations are chained sequentially with `.then()`. Each waits for the previous to complete before starting, making cold-start initialization unnecessarily slow. If `seedFinancialAcademyLesson` takes 2 s and `seedGameContent` takes 2 s, that's 14+ s of sequential I/O at startup for a school app with potentially many concurrent morning logins.
  → *Group independent seed operations into a single `Promise.all`: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` — keep initSupabaseTables first since seeds depend on tables existing.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96-105)):** The filePath parameter is passed directly to `objectStorage.searchPublicObject(filePath)` with no path traversal sanitization. A request to `/public-objects/../../private/keys` could resolve outside the intended public bucket prefix depending on the ObjectStorageService implementation, exposing private assets to unauthenticated users.
  → *Sanitize the path before use: `const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, '').replace(/^[\/]+/, '');` and return 400 if the result is empty. Also confirm ObjectStorageService itself never constructs filesystem paths from this value.*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop (for loop over questions)):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this serialises 40 round-trips to the database, dramatically increasing job duration and holding a DB connection for the full loop.
  → *Collect valid questions into an array, then do a single batch insert: await db.insert(extractedQuestions).values(validQuestions) where validQuestions is built with map/filter before the loop. Wrap the batch insert in try/catch so a single malformed row doesn't silently drop the rest.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The DB delete in the purge-ai-usage handler has no try/catch. An unhandled rejection here will crash the worker process rather than marking the job as failed and triggering a retry, because registerJobHandler likely does not wrap handler bodies in its own error boundary.
  → *Wrap the delete call in try/catch: catch the error, log it, then rethrow so the job runner can record failure and schedule a retry. Also validate that olderThanDays is a positive integer before computing the cutoff to prevent accidental full-table deletes if the payload is malformed.*
- **`server/jobHandlers.ts` (openai import – `import { openai } from './replit_integrations/chat/routes'`):** The OpenAI client is imported from a routes module rather than a dedicated singleton. This creates an implicit coupling to the HTTP layer, risks importing Express middleware side-effects during job worker startup, and makes the client impossible to mock in unit tests for job handlers.
  → *Extract the OpenAI client instantiation into its own module (e.g. server/openaiClient.ts) that reads OPENAI_API_KEY from process.env and exports a single instance. Import from there in both routes and jobHandlers.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId has no index. Any query fetching a user's budgets (which will happen on every dashboard load) requires a full table scan as the dataset grows.
  → *Add an index in the pgTable second argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also add a composite unique index on (userId, categoryId, period) to enforce one budget per category per period at the DB level.*
- **`shared/schema.ts` (savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, linkedCards, documentUploads table definitions):** None of these tables index userId, yet every application query will filter by it. billReminders also needs an index on nextDueDate for due-date polling queries.
  → *Add (t) => ({ userIdx: index('idx_<table>_user_id').on(t.userId) }) to each table. For billReminders add a second index: dueDateIdx: index('idx_bill_reminders_due_date').on(t.nextDueDate).*
- **`shared/schema.ts` (All tables — createdAt, purchasedAt, executedAt columns):** Timestamp audit columns use .defaultNow() without .notNull(), so they can be explicitly set to NULL by an insert, breaking audit trails and any ORDER BY createdAt query that assumes non-null values.
  → *Chain .notNull() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to all tables including portfolioHoldings.purchasedAt and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (userId: varchar('user_id') — all tables):** userId is declared as bare varchar with no length limit. Without a consistent length constraint matching the auth provider's ID format (e.g. Clerk uses 32-char strings), the DB accepts arbitrarily long values, wasting storage and risking mismatches in joins.
  → *Change to varchar('user_id', { length: 36 }) (or the exact max length of your auth provider's IDs, e.g. 32 for Clerk). Apply uniformly across all tables to guarantee join compatibility.*
- **`server/index.ts` (res.on('finish') inside the request-logging middleware):** The middleware monkey-patches res.json to capture and store the full parsed response body in capturedJsonResponse, then JSON.stringify's the entire body on every API response for logging. For endpoints returning lists of student records, quiz results, or financial data this (1) doubles serialisation cost, (2) logs potentially sensitive PII/financial data of minors to stdout in plain text, and (3) holds the full response object in closure memory until the finish event fires.
  → *Remove body capture entirely, or limit logging to non-sensitive fields: log only status code, duration, and content-length header. If response inspection is needed for debugging, gate it behind a `LOG_RESPONSE_BODIES=true` env flag and redact known sensitive fields before logging.*
- **`server/index.ts` (maybeRunWeeklyDigest and maybeRunAiUsagePurge async callbacks passed directly to setInterval):** Both async functions are passed directly to setInterval without a synchronous wrapper that catches rejections. If either async function throws after its first await (e.g. a DB connection drop), the rejection is unhandled. In Node 15+ this terminates the process without triggering the graceful-shutdown handler, leaving the DB and job worker in an inconsistent state.
  → *Wrap each scheduler: `setInterval(() => { maybeRunWeeklyDigest().catch(e => captureError(e)); }, interval)`. The existing try/catch inside each function handles expected errors; this wrapper catches unexpected throws that escape it.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince = (dow + 0) % 7`, where `+ 0` is a no-op left from what appears to be an incomplete day-offset calculation. Sunday is dow=0, so daysSince=0 always, meaning on any day other than Sunday the function subtracts 0 days and then conditionally subtracts 7 more — it never correctly identifies the most recent past Sunday on Mon–Sat without the conditional, but the conditional itself checks `d.getTime() > now.getTime()` after setting hours to 19:00, which means on a Monday at 08:00 the returned date is last Sunday at 19:00 but only because of the fallback, not the daysSince calculation. This is dead/wrong code masking a logic bug.
  → *Replace with: `const daysSince = (dow === 0 ? 0 : dow); d.setDate(d.getDate() - daysSince);` or more clearly use `dow` directly. Remove the `+ 0` entirely and add a unit test for each day of the week at times before and after 19:00.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scrollIntoView ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is a new function on every render, firing scrollIntoView on every message list re-render (including while the user is typing in the input), not just when a new message is appended. This causes jarring scroll resets mid-input.
  → *Replace with a useEffect that tracks message count: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` and attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – /api/admin/help-chat POST):** The help-chat endpoint receives the full message history (`messages: next`) on every send, meaning the entire conversation is serialized and sent with each request. For a school-aged platform this data includes potentially sensitive session content, and there is no size cap, so a long conversation will send an ever-growing payload with no truncation or limit enforced client-side.
  → *Truncate the history before sending: `const truncated = next.slice(-20);` (keep last 20 messages) and send `{ messages: truncated }`. Add a `MAX_MESSAGES` constant (e.g. 50) and trim `messages` state when it exceeds that limit to prevent unbounded memory growth.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple useQuery calls for dashboard stats/users/schools):** Multiple independent `useQuery` calls (stats, users, schools, lessons, etc.) each run with their own loading and error states, but the dashboard renders without a unified error boundary or loading state. If any query fails silently, the component renders partial data with no user-visible indication, which is especially problematic for admin accuracy.
  → *Collect the `isError` and `isLoading` flags from each query and add a top-level guard: `if (isLoadingStats || isLoadingUsers) return <DashboardSkeleton />;` and `if (isErrorStats) return <ErrorBanner message='Failed to load dashboard data' />;`. Create simple reusable skeleton and error banner components.*

## 💡 Opportunities

### Centralised query error wrapper: 🔥 High Impact
Most storage methods lack try/catch, so raw Drizzle/PostgreSQL errors (which may contain table names, column names, or constraint details) propagate directly to API route handlers and potentially to HTTP responses. Introduce a thin withDbError(fn) wrapper that catches, sanitises, and re-throws typed StorageError instances, preventing internal schema leakage in error messages.

### Extract hardcoded virtual balance seed amount to environment config: ⚡ Medium Impact
The initial virtual balance granted to new users (likely 10000 or similar) appears hardcoded in getVirtualBalance or seedMarketData. Moving it to INITIAL_VIRTUAL_BALANCE_CENTS in environment config allows educators and sponsors to configure starting conditions per deployment without a code change, which is directly relevant to the Caribbean school context.

### Replace simulateMarketPrices setInterval with a managed job: ⚡ Medium Impact
If simulateMarketPrices is called on a recurring interval registered at module load time, the interval handle is never stored or cleared, creating a memory/resource leak if the module is re-imported in tests or the server hot-reloads. Expose a stopMarketSimulation() function that calls clearInterval on the stored handle, and register it with process.on('SIGTERM').

### Structured startup logging for Supabase init: ⚡ Medium Impact
The catch block logs `[Supabase] Init error:` but swallows individual step failures silently. Wrapping each seed call with a named logger (e.g. Sentry breadcrumb or structured console.error with step name) would let you pinpoint which seed fails in production without redeploying with extra logs. Expected impact: faster incident diagnosis on school-day morning deploys.

### Cache-Control headers on /healthz: ⚡ Medium Impact
Railway and uptime monitors poll /healthz frequently. Without explicit `Cache-Control: no-store` headers, edge proxies or Railway's internal routing layer could cache a stale 200 OK after a DB goes down. Add `res.setHeader('Cache-Control', 'no-store')` to the /healthz handler to ensure every poll hits the DB check.

### Centralize domain route registration with error isolation: 🔥 High Impact
The six `await register*Routes(app)` calls are sequential and unguarded — if `registerAiRoutes` throws, `registerOrgRoutes` and `registerLessonRoutes` never mount, silently breaking auth and lesson delivery for all students. Wrap each in an individual try/catch that logs and rethrows, or use a helper that mounts routes and reports failures to Sentry before propagating, so partial registration failures are observable.

### Idempotency guard for XP and badge award jobs: 🔥 High Impact
Award-type jobs (XP grants, badge unlocks) likely have no idempotency check, so a retry after a partial failure can double-award. Add a unique constraint on (userId, eventSource, eventId) in the userXp table and use INSERT ... ON CONFLICT DO NOTHING, then return { alreadyAwarded: true } on conflict. This prevents duplicate awards without requiring distributed locks.

### Stream large file downloads instead of buffering in memory: ⚡ Medium Impact
streamPrivateObjectToResponse already pipes the read stream correctly, but any handler that calls Buffer.from(fileB64, 'base64') on a multi-MB payload allocates the full file in Node heap during job execution. After moving to object-path payloads, use file.createReadStream() piped through pdf-parse's stream API to avoid holding the full buffer in memory, reducing heap pressure under concurrent jobs.

### Consolidate AI call error handling into a shared wrapper: ⚡ Medium Impact
Every openai.chat.completions.create call is wrapped ad-hoc with inline JSON.parse and no validation of choices[0] being defined. Extract a typed callOpenAI(params) utility that validates the response shape, throws a structured AppError on empty choices, and logs token usage to aiUsageEvents in one place. This removes duplicated boilerplate across handlers and ensures usage tracking is never accidentally omitted.

### Consolidate repeated currency default into a shared constant: 📌 Low Impact
The string 'BSD' and the column definition .default('BSD').notNull() are duplicated across at least 7 tables. Extract a helper const defaultCurrency = (col: string) => text(col).default('BSD').notNull() in a shared file. Eliminates drift risk if the default currency ever changes and reduces schema line count by ~15 lines.

### Replace serial PKs with UUID on user-facing tables: ⚡ Medium Impact
serial integer PKs expose row counts and enable enumeration attacks (GET /transactions/1, /2, …). For a school-aged audience this is a meaningful privacy risk. Switching to gen_random_uuid() as the default for id on transactions, savingsGoals, and billReminders closes the enumeration vector without schema complexity.

### Add a composite unique constraint on portfolioHoldings(userId, stockId): 🔥 High Impact
Without this constraint, buy operations can create duplicate holding rows for the same stock, causing double-counted portfolio values. A unique constraint forces the application to upsert (update quantity/avgPurchasePrice) rather than insert, which is the correct semantics for a holdings ledger.

### Centralise shutdown signal handling: 🔥 High Impact
SIGTERM and SIGINT handlers are registered but the truncated code suggests they may not await job-worker drain before exiting. Add `await stopJobWorker()` inside the shutdown handler before `httpServer.close()`, and set a hard timeout (e.g. 10 s) via `setTimeout(process.exit, 10_000)` to guarantee the process exits even if a job hangs. This prevents zombie Replit dynos and lost job state.

### Externalise scheduler intervals to environment config: ⚡ Medium Impact
The digest and purge check intervals appear hardcoded. Exposing them as WEEKLY_DIGEST_INTERVAL_MS and AI_PURGE_CHECK_INTERVAL_MS env vars allows tuning without a redeploy and makes behaviour explicit in the Replit Secrets panel alongside SESSION_SECRET and AI_PURGE_OLDER_THAN_DAYS, which is already done correctly.

### Replace per-request res.json monkey-patch with structured audit logging: ⚡ Medium Impact
The current approach patches a core Express method on every request object, which breaks prototype chain assumptions and adds GC pressure. For a COPPA/FERPA-adjacent app serving minors, a separate audit-log sink (writing only route, userId, status, and duration to a structured log stream) achieves observability without capturing response payloads or mutating Express internals.

### Parallelise independent admin data fetches with Promise.all: 🔥 High Impact
Any place where multiple sequential `await apiRequest(...)` calls fetch independent resources (e.g. stats then users then schools in an export or bulk action handler) should be replaced with `const [stats, users, schools] = await Promise.all([...])`. For a dashboard loading 4-6 independent datasets this can cut perceived load time by 60-70% on Caribbean mobile connections.

### Memoize filtered and sorted user/school lists: ⚡ Medium Impact
Any inline `.filter().sort()` chains on large arrays (users list with search, school rankings) inside the render body should be wrapped in `useMemo` keyed on the source data and search/filter state. Without this, every parent re-render (e.g. typing in the help chat input updates parent state) re-executes the full filter/sort on hundreds of records.

### Extract AdminHelpChat into its own lazy-loaded module: ⚡ Medium Impact
AdminHelpChat with its Recharts imports and chat state is bundled into the main AdminDashboard chunk. Moving it to a separate file and loading it with `React.lazy` + `Suspense` will defer its JS parse cost until the user first opens the chat panel, reducing initial dashboard TTI for school network connections.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts is a large monolithic data layer with several reliability and security gaps. The most critical issues are N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that should run in parallel. The file also exposes sensitive data paths and lacks pagination guards on unbounded list queries.

### API Routes (`server/routes.ts`)
The route registration code is generally well-structured with good use of Promise.all for parallel imports and proper try/catch on most async operations. Key concerns are: unprotected AI/heavy routes lacking rate limiting, a host header injection vulnerability in the robots.txt and sitemap generators, and sequential Supabase seed calls that block each other unnecessarily.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps including large base64 payloads stored directly in job rows, sequential DB inserts that should be batched, and missing try/catch around the AI usage purge and paper extraction DB operations. Security exposure exists via base64 file content potentially appearing in logs and missing idempotency on XP/badge award jobs.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns include missing indexes on high-frequency query columns, nullable createdAt timestamps that should be NOT NULL, and an untyped varchar user_id that risks silent data corruption across all tables.

### Server Startup (`server/index.ts`)
The server startup sequence contains multiple blocking synchronous-style awaits that delay readiness, unguarded async scheduler callbacks that can silently crash, and intervals that are never cleared on shutdown creating memory leaks. The logging middleware unconditionally serializes every JSON response body to a string on every API request, which is a performance hazard and a data-exposure risk for a financial-literacy app serving minors.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security gaps including a scrollIntoView ref callback that creates a new function on every render causing unnecessary DOM operations, missing auth checks before sensitive mutations, and sequential awaits that could be parallelized. The chat component leaks no resources but has a subtle ref anti-pattern. Overall structure is reasonable but hardcoded API paths and missing error boundaries for the dashboard's query states are concerns.
