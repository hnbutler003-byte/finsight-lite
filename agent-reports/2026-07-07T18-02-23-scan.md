## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting. Any authenticated (or unauthenticated) user can hammer these endpoints, causing runaway API costs and denial-of-service for other users. This is critical for a school-aged audience where accounts may be shared or abused.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/min per IP using express-rate-limit) and apply it before registering these routes: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);`. Adjust window and max to match your OpenAI tier budget.

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 40-80)
**Issue:** The `base` URL is constructed directly from `x-forwarded-proto` and `x-forwarded-host` / `host` headers without any validation or allowlist. An attacker can send crafted headers to inject an arbitrary URL into the Sitemap directive or sitemap <loc> tags, enabling cache-poisoning or open-redirect abuse.

**Fix:** Use a single `APP_BASE_URL` environment variable (e.g. `process.env.APP_BASE_URL || 'https://finsightlite.com'`) as the canonical base instead of trusting request headers. Remove all header-derived base URL logic from these two handlers.

### `server/jobHandlers.ts`: extract-paper handler / job.payload.fileB64
**Issue:** The full base64-encoded exam paper file is stored in the jobs table payload. For a multi-page PDF or image this can be several MB per row, bloating the jobs table, slowing queue queries, and exposing file content to anything that reads job records (admin UIs, logs, DB backups).

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the buffer from object storage using the path, then delete the object after successful extraction. This keeps job rows small and file data in the appropriate store.

### `server/jobHandlers.ts`: runExtractPaper / for loop calling storage.createExtractedQuestion
**Issue:** Extracted questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 50-question paper this is 50 serial round-trips to the database, each blocked on the previous, creating unnecessary latency and holding the job worker busy far longer than needed.

**Fix:** Collect all valid question objects into an array first, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, replace the loop with await Promise.all(validQuestions.map((q, i) => storage.createExtractedQuestion({...}))) — but a single INSERT with multiple rows is strongly preferred.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** The day-rollback logic uses `(dow + 0) % 7` which maps Sunday (dow=0) to 0 days back, meaning on Sunday before 7pm the function returns today at 19:00, which is in the future and correctly bails—but on Sunday after 7pm it returns today, which is correct. However Monday through Saturday it subtracts 1–6 days, effectively targeting Monday–Saturday instead of the prior Sunday, producing wrong weekStart keys and duplicate or missed digest jobs.

**Fix:** Replace `(dow + 0) % 7` with `(dow === 0 ? 0 : dow)` and instead calculate days-since-Sunday as `dow` (0=Sun…6=Sat). The correct offset is just `dow` days back, which already handles Sunday=0. Verify with unit tests covering each day of week both before and after 19:00.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by setInterval are never stored or cleared. On graceful shutdown the process will stay alive because Node's event loop holds open interval references. There is also no SIGTERM/SIGINT handler shown, so the HTTP server and DB connections are never closed cleanly, risking in-flight job corruption.

**Fix:** Store handles: `const digestInterval = setInterval(maybeRunWeeklyDigest, ...)`. Add a shutdown handler: `async function shutdown() { clearInterval(digestInterval); clearInterval(purgeInterval); await new Promise(r => httpServer.close(r)); process.exit(0); }` and wire it to both `process.on('SIGTERM', shutdown)` and `process.on('SIGINT', shutdown)`.

---

### `server/index.ts`: res.json monkey-patch in request logging middleware
**Issue:** `capturedJsonResponse` captures the full JSON body of every API response and appends it to the log line via `JSON.stringify`. For a financial literacy app with school-aged users this will log session tokens, user PII, quiz scores, and any AI-generated content in plaintext to stdout/log aggregators, violating data minimisation principles.

**Fix:** Remove the `res.json` override entirely, or limit logging to a safe allowlist: log only `res.statusCode` and `duration`. If response-body debugging is needed, gate it behind `if (process.env.LOG_RESPONSE_BODIES === 'true')` and ensure that flag is never set in production.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / top-level dashboard render
**Issue:** If `useAdminAuth` returns a loading or unauthenticated state, the component must short-circuit before rendering any admin data or issuing any `useQuery` calls. Without seeing the hook internals, the pattern risk is that queries fire before auth is confirmed, returning data briefly visible before a redirect, or worse, the redirect never fires if the hook is async and the component tree mounts fully first.

**Fix:** Immediately after calling `useAdminAuth`, add: `if (authLoading) return <FullPageSpinner />; if (!isAdmin) { setLocation('/'); return null; }` — this ensures no queries mount and no UI renders until auth is resolved. Pass `enabled: isAdmin` to every `useQuery` call as a secondary guard.

## 🟡 Warnings (10)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 115-124)):** The seven Supabase init/seed functions are chained sequentially with `.then()`, so each waits for the previous to finish. This unnecessarily extends cold-start time. Additionally, individual step failures are swallowed by the single `.catch` at the end, making it impossible to identify which step failed without parsing the error message.
  → *Group independent operations with `Promise.all` where ordering permits, and add per-step error logging: `Promise.all([initSupabaseTables(), initStaticContentTables()]).then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));`. Adjust grouping based on actual data dependencies.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100-110)):** The `filePath` parameter from the URL is passed directly to `objectStorage.searchPublicObject()` without sanitisation. A path traversal payload like `../../private/secret` could potentially access unintended objects depending on how the storage service resolves paths.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and reject requests where the normalised path starts with `..` or `/`. Add `import path from 'path';` at the top of the file.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch. An unhandled rejection from the db.delete call will surface as an untracked worker crash with no structured error context, making it invisible in Sentry and leaving the job status ambiguous.
  → *Wrap the handler body in try/catch, call captureError in the catch block with job context (job name, olderThanDays, cutoff date), then rethrow so the job framework can mark the job as failed and schedule a retry.*
- **`server/jobHandlers.ts` (runExtractPaper / two sequential openai.chat.completions.create calls):** For the PDF path, text extraction completes and then a second OpenAI call is made for question extraction. These are necessarily sequential, but there is no timeout or deadline guard on either call. A hung OpenAI request will stall the worker thread indefinitely, blocking other jobs in the same process.
  → *Wrap each OpenAI call with a Promise.race against a timeout: const withTimeout = (p, ms) => Promise.race([p, new Promise((_, r) => setTimeout(() => r(new Error('OpenAI timeout')), ms))]). Use a configurable timeout via process.env.OPENAI_TIMEOUT_MS (default 60000). This ensures the job fails fast and retries rather than hanging.*
- **`server/index.ts` (probeDatabase call inside async IIFE at startup):** `probeDatabase` is awaited inside the top-level async IIFE before `registerRoutes` and `seedDatabase`, meaning any database latency (cold Replit container, network hiccup) blocks the entire startup sequence synchronously. If it throws after a long timeout, the process exits having never bound to a port, causing health-check failures and cold-start timeouts.
  → *Start the HTTP server and bind the port first so the process is alive for health checks, then probe and seed in the background with a retry loop. If the probe fails after N retries, call `shutdown()` gracefully. This decouples network readiness from process liveness.*
- **`server/index.ts` (seedDatabase and registerRoutes sequential awaits in startup IIFE):** `await probeDatabase()`, `await seedDatabase()`, and `await registerRoutes(app, httpServer)` are called sequentially. If seedDatabase and probeDatabase are independent of each other they serialise unnecessarily, adding hundreds of milliseconds to cold-start time for a Replit environment that already has slow container wake-up.
  → *If seedDatabase does not depend on the result of probeDatabase, run them concurrently: `await Promise.all([probeDatabase(), seedDatabase()])`. Confirm neither function has a hidden ordering dependency before combining, then measure startup time improvement.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage (AdminHelpChat)):** Every POST to /api/admin/help-chat sends the full accumulated messages array. For a school-aged platform this means the entire session conversation is transmitted on each turn, growing unboundedly and potentially leaking prior context. There is also no rate-limit guard on the client side, so rapid clicking or Enter-key spam fires parallel requests.
  → *Slice the context window before sending: `const context = next.slice(-10)` and send `context` instead of `next`. Add a ref-based debounce or disable the send button immediately on submit (already partially done with `loading` but the button check `!input.trim()` runs after state update race). Set `setLoading(true)` before pushing to `next` to close the race window.*
- **`client/src/pages/AdminDashboard.tsx` (inline ref callback in Messages scroll div):** `<div ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }} />` creates a new function object on every render. Because React treats a changed ref callback as unmount+remount, this fires scrollIntoView on every state change in the chat panel (including the loading spinner toggle), causing jarring scroll resets while the assistant is typing.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null)` declared at component top, attach `ref={bottomRef}`, and call `bottomRef.current?.scrollIntoView({ behavior: 'smooth' })` inside a `useEffect` that depends on `[messages, loading]`.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple top-level useQuery calls (stats, users, schools, teachers, etc.)):** Each admin data query is declared independently with no `enabled` flag tied to auth state (see finding above), and all fire simultaneously on mount. While parallel is desirable, without `enabled: !!isAdmin` each query will attempt to fetch even during the unauthenticated loading window, generating unnecessary 401 network requests and potential error-state flicker across all dashboard panels.
  → *Add `enabled: !!isAdmin` to every `useQuery` options object. For queries that depend on each other (e.g., school-specific user lists), also gate on the parent ID being defined: `enabled: !!isAdmin && !!selectedSchoolId`.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort functions inside JSX (user table, school table renders)):** Array filter and sort operations applied directly inside the JSX return (e.g., `users.filter(u => u.name.includes(search)).sort(...)`) run on every render triggered by any state change, including unrelated state like `open` dialog toggles. On lists of hundreds of users or schools this is a measurable CPU cost.
  → *Extract each filtered+sorted list into a `useMemo`: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(search.toLowerCase())).sort(...), [users, search, sortField, sortDir])`. This ensures the expensive computation only reruns when its actual inputs change.*

## 💡 Opportunities

### Centralise base-URL resolution as a startup-time constant: 🔥 High Impact
Rather than deriving the base URL on every request in robots.txt and sitemap handlers, resolve `APP_BASE_URL` once at module load time from an environment variable with a startup-time validation check. This eliminates per-request header parsing, removes the injection surface, and makes the value auditable in config.

### Add structured startup logging for route registration: ⚡ Medium Impact
Currently there is no logging when domain routers are registered or when Supabase init completes. Adding timestamped structured logs (e.g. `console.log('[routes] AI routes registered')`) would dramatically reduce debugging time during Railway deploys and cold-start investigations, which matters for a production app with school-hour traffic spikes.

### Guard `/public-objects` with response caching headers consistently: ⚡ Medium Impact
The `downloadObject` call passes a 30-day TTL but there is no `Cache-Control` header set at the Express layer as a fallback if the storage service omits it. Adding `res.setHeader('Cache-Control', 'public, max-age=2592000, immutable')` before the download call ensures CDN and browser caching regardless of storage service behaviour, reducing egress costs.

### Idempotency token on OpenAI calls: ⚡ Medium Impact
Passing a stable idempotency key (e.g. derived from paperId + attempt number) to OpenAI requests would prevent duplicate billing and duplicate responses if a network retry causes the same request to be sent twice. OpenAI supports this via the request header X-Request-Id. Expected impact: eliminates duplicate AI spend on retried jobs.

### Dead-letter queue for terminal job failures: ⚡ Medium Impact
Jobs that exhaust maxAttempts currently update paper status to 'failed' in a best-effort catch block but are otherwise silently dropped. Routing terminal failures to a dead-letter table or emitting a structured alert would allow ops staff to manually replay or investigate stuck papers without querying raw job rows.

### Stream PDF parsing instead of buffering full base64 in memory: 🔥 High Impact
Buffer.from(fileB64, 'base64') materialises the entire file in the Node.js heap before pdf-parse sees it. For large papers this doubles memory usage (base64 string + decoded buffer both live simultaneously). Storing the raw binary in object storage (per finding 1) and streaming it directly into pdf-parse reduces peak heap pressure and avoids the base64 decode overhead.

### Centralised scheduler registry with observable state: ⚡ Medium Impact
Replace ad-hoc setInterval calls with a lightweight scheduler registry that tracks each job's last-run time, success/failure count, and next-scheduled time, exposable via an internal `/healthz/schedulers` endpoint. For a financial app running digest and purge jobs, this gives operators visibility without grep-ing logs, and makes adding future scheduled jobs consistent and testable.

### Structured JSON logging to replace string concatenation: 🔥 High Impact
Replace `console.log` string interpolation with a structured logger (pino is zero-dependency and extremely fast). Each log entry would carry `{ level, source, method, path, statusCode, durationMs }` as discrete fields rather than a concatenated string. This enables log aggregators (Datadog, Logtail) to filter and alert on error rates without regex parsing, and eliminates the risk of accidentally embedding serialised objects containing PII.

### Extract scheduler timing constants to environment config: 📌 Low Impact
Interval durations for weekly digest and AI usage purge checks are hardcoded as magic numbers (e.g. every 10 minutes, every hour). Move them to `DIGEST_CHECK_INTERVAL_MS` and `PURGE_CHECK_INTERVAL_MS` environment variables with safe defaults. This allows tuning check frequency in staging vs production without a code deploy, which matters when debugging missed digest windows.

### Paginate admin queries server-side instead of client-side filtering: 🔥 High Impact
Currently all users/schools are fetched and filtered on the client. For a growing Caribbean multi-school deployment this payload will balloon. Adding `?page=&limit=&search=` params to the API and using React Query's `keepPreviousData` option would reduce initial load time and memory use significantly, with no UX regression.

### Extract sub-sections into memoized child components: 🔥 High Impact
The dashboard appears to render all tabs (Users, Schools, Teachers, Analytics) in a single massive component tree. Splitting each tab into its own memoized component (e.g., `UsersTab`, `SchoolsTab`) wrapped in `React.memo` would eliminate cross-tab re-renders when shared state like `search` or dialog toggles changes, reducing render work by an estimated 60-80% for inactive tabs.

### Centralise hardcoded API path strings into a typed route constant map: ⚡ Medium Impact
Strings like `/api/admin/help-chat`, `/api/admin/users`, `/api/admin/schools` appear inline throughout the file. A single `ADMIN_ROUTES` constant object would prevent typo-driven 404s, make endpoint auditing trivial, and allow base URL injection from environment config — important if the app runs against different API hosts per Caribbean territory.
