## 🔴 Critical (7)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction (investment trade flow)
**Issue:** Portfolio buy/sell operations update the holding, create a portfolio transaction, and update the virtual balance as three separate awaits with no wrapping database transaction. A failure between any two steps leaves the database in an inconsistent state — e.g., balance deducted but holding not updated.

**Fix:** Wrap all three operations in a single `db.transaction(async (tx) => { ... })` call using Drizzle's transaction API so the entire trade is atomic. Any thrown error will automatically roll back all three writes.

### `server/storage.ts` — getLeaderboard
**Issue:** The leaderboard query joins or fetches user display names and avatars for ranking, and this data is returned without filtering out any fields that may have been populated from OAuth profiles. If the user object contains email or OAuth tokens, they could be exposed to all leaderboard viewers — a significant privacy risk for school-aged users.

**Fix:** Explicitly select only `userId`, `userName`, and `avatar` columns in the query projection (Drizzle column selection) rather than spreading a full user object. Add a runtime type assertion or Zod parse on the return value before sending to callers to guarantee no extra fields leak.

### `server/routes.ts` — AI/tutor chat and explanation endpoints (registerChatRoutes and any route calling openai or reserveQuota)
**Issue:** No rate limiting middleware is applied to AI-backed endpoints before they reach quota reservation logic. A single authenticated student (or an unauthenticated caller if auth checks are missing) can hammer OpenAI calls; quotaErrorMessage only fires after quota is already consumed, not before the request is accepted.

**Fix:** Add express-rate-limit (or a Redis-backed limiter) as route-level middleware on all AI endpoints: `import rateLimit from 'express-rate-limit'; const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.use('/api/chat', aiLimiter);`. Apply before registerChatRoutes and any tutor/explain route.

### `server/routes.ts` — catch blocks throughout async route handlers
**Issue:** Several async route handlers (visible in the truncated sections referencing audit, enrollStudentInOrg, parseWithVeryfi, XLSX parsing) log the full error object — including e.message and e.stack — directly into the JSON response body sent to the client. For a school-aged audience this can expose internal file paths, DB schema hints, or API key fragments embedded in error messages.

**Fix:** Create a centralised error responder: `function safeError(res, e, fallback = 'An error occurred') { console.error(e); res.status(500).json({ message: fallback }); }`. Never forward e.message to the client. Log the full error server-side only, and route it through your Sentry integration so stack traces are captured without client exposure.

### `server/jobHandlers.ts` — runExtractPaper / extract-paper payload
**Issue:** The full base64-encoded file (fileB64) is stored directly in the job table payload. For a multi-page PDF or high-res image this can easily be 5–20 MB per row, bloating the jobs table, slowing queue polling queries, and potentially hitting row-size limits on Postgres.

**Fix:** Upload the file to private object storage first (in the HTTP handler that enqueues the job), store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer at the start of runExtractPaper using objectStorageClient. This keeps job rows small and makes retries safe since the file is already persisted.

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker is declared as text('ticker').notNull() with no unique constraint. Two rows with the same ticker symbol (e.g. 'BTC') would be silently allowed, causing portfolio calculations and UI lookups to return ambiguous or duplicated results.

**Fix:** Replace text('ticker').notNull() with text('ticker').notNull().unique(), and add the table-level factory: (t) => ({ tickerUnique: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to enforce uniqueness at the DB level and give the query planner an index for ticker-based lookups.

### `client/src/pages/AdminDashboard.tsx` — apiRequest calls inside queryFn lambdas
**Issue:** Every queryFn is written as `apiRequest(...).then(r => r.json())` with no error handling. If the server returns a non-2xx response, r.json() may still resolve (e.g. a JSON error body), silently populating the cache with error payloads. Downstream components then render garbage data rather than an error state, which is a reliability risk for an admin dashboard managing student data.

**Fix:** Add a response guard in each queryFn: `const r = await apiRequest('GET', url); if (!r.ok) throw new Error(await r.text()); return r.json();` React Query will then correctly populate isError and allow error UI to render. Alternatively, centralise this in apiRequest itself so all callers benefit.

## 🟡 Warnings (17)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues multiple independent database queries sequentially (total income, total expenses, savings goals, budgets, recent transactions) with individual awaits. Each query is independent and the cumulative latency compounds on every dashboard load.
  → *Replace sequential awaits with `const [income, expenses, goals, budgets, recent] = await Promise.all([query1, query2, query3, query4, query5])`. This parallelises all five round-trips and should cut dashboard load time by 60–80% under typical DB latency.*
- **`server/storage.ts` (getTransactions / getAllQuestions / getLeaderboard):** List endpoints accept an optional `limit` but do not enforce a maximum or default when none is supplied. A missing or maliciously large limit causes full-table scans returned to the client, risking OOM on the Node process and excessive DB load.
  → *Add a guard at the top of each function: `const safeLimit = Math.min(filters?.limit ?? 50, 200)` and always pass `safeLimit` to the Drizzle `.limit()` call. Never allow an unbounded query path.*
- **`server/storage.ts` (deleteTransaction / deleteBudget / deleteSavingsGoal / deleteExamPaper):** Delete operations filter by both `id` and `userId` in the WHERE clause, which is correct, but the result row count is never checked. If the record does not exist or belongs to another user, the function returns successfully with no error, silently masking ownership violations and making them undetectable in logs.
  → *After each delete, check the affected row count returned by Drizzle (e.g., `const result = await db.delete(...).where(...).returning()`) and throw a typed error such as `throw new NotFoundError('Record not found or access denied')` if `result.length === 0`. This surfaces auth bypass attempts.*
- **`server/routes.ts` (POST /api/auth/google handler (~line 170)):** Three sequential awaits — getClassByCode, getOrgEnvironmentById, getOrganization — each make independent DB/Supabase round-trips inside the request hot-path. None are data-dependent on each other at the point they are called, adding cumulative latency on every Google sign-in.
  → *After resolving the join code, fan out the independent lookups: `const [classEnv, classOrg] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), getOrganization(envId)])`. Apply the same pattern to any other sequential awaits in this handler that are not strictly data-dependent.*
- **`server/routes.ts` (googleEnabled(), appBaseUrl, and bcrypt salt rounds (scattered across file)):** Multiple values that differ between environments are hardcoded or inlined: bcrypt is called without a configurable salt-round constant, appBaseUrl is imported from email.ts but URL construction elsewhere uses raw string concatenation with req.headers.host without sanitisation, and googleEnabled() silently disables a feature rather than failing fast at startup.
  → *Define a central config object at module load time: `const BCRYPT_ROUNDS = parseInt(process.env.BCRYPT_ROUNDS ?? '12', 10);`. Validate required env vars (GOOGLE_CLIENT_ID, APP_BASE_URL, DATABASE_URL) at startup and throw if absent. For host-based URL construction, whitelist against APP_BASE_URL env var instead of trusting request headers.*
- **`server/routes.ts` (XLSX / PapaParse file processing (upload and examUpload handlers)):** XLSX.read() and Papa.parse() are called synchronously on the full in-memory file buffer inside async route handlers. For files up to the 10 MB limit this blocks the Node.js event loop for tens to hundreds of milliseconds, stalling all concurrent requests on the same process.
  → *Offload parsing to the existing job queue: enqueue a parse job with the buffer stored in object storage, return a 202 with the job ID, and let the client poll getJob(). Alternatively, run parsing in a worker_thread using `import { Worker } from 'worker_threads'` to avoid blocking the event loop. This is especially important given the student-facing load profile.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop over questions):** Questions are inserted one at a time inside a sequential for-loop with individual awaits on storage.createExtractedQuestion. For a 50-question paper this means 50 serial round-trips to Postgres before the paper status is updated, multiplying latency and holding the job worker busy unnecessarily.
  → *Replace the loop with a single bulk insert: filter the valid questions first, then call db.insert(extractedQuestions).values(validRows) once. If storage.createExtractedQuestion must be used, at minimum use Promise.all on the filtered array. Wrap the bulk insert in a transaction together with the updateExamPaper call so both succeed or both roll back atomically.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch. If the db.delete throws (e.g. connection timeout), the raw database error — which may include internal table names or connection strings in the stack — will propagate unhandled to the job runner and potentially get logged or surfaced without sanitisation.
  → *Wrap the handler body in try/catch. On catch, log a sanitised message (omit connection details) and rethrow a generic Error so the job runner can record the failure and retry according to its backoff policy. Apply the same pattern to admin-csv-export which is also missing an error boundary.*
- **`server/jobHandlers.ts` (runExtractPaper — model string 'gpt-4o'):** The OpenAI model name 'gpt-4o' is hardcoded in two separate openai.chat.completions.create calls. Changing the model for cost control or API availability requires a code deploy rather than a config change, and the two call sites could silently diverge.
  → *Read the model from an environment variable with a fallback: const AI_MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o'; and reference AI_MODEL in both calls. Validate at startup that the value is a non-empty string so misconfiguration fails fast rather than at job execution time.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete before re-extraction):** The idempotency guard deletes all previously extracted questions at the start of every retry attempt, then immediately begins expensive OpenAI API calls. If the paper was already successfully completed on a prior attempt but the job status write failed, this silently re-bills the OpenAI account and overwrites correct data.
  → *Before the delete, query the exam paper's current status. If status is already 'completed' and questionCount > 0, short-circuit and return the existing result immediately without calling OpenAI or deleting questions. Only proceed with the delete+re-extract path when status is 'processing' or 'failed'.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on (userId, categoryId) or (userId, period), but queries filtering a user's budgets by category or period will full-scan the table as row count grows.
  → *Add a composite index in the budgets table factory function: (t) => ({ userCategoryPeriodIdx: index('idx_budgets_user_category_period').on(t.userId, t.categoryId, t.period) }). Also add a uniqueIndex on (userId, categoryId, period) to prevent duplicate budget rows per user/category/period, which is almost certainly a business invariant.*
- **`shared/schema.ts` (varchar('user_id') on all tables (e.g. transactions, budgets, linkedCards)):** varchar('user_id') is declared without a length argument on every table. In PostgreSQL, unbounded varchar is functionally equivalent to text, but the missing length means no DB-level enforcement of the ID format (e.g. Clerk's 'user_xxx' IDs are always ≤32 chars). More critically, mismatched implicit types between tables can silently prevent FK index usage.
  → *Set an explicit max length matching your auth provider's ID format, e.g. varchar('user_id', { length: 64 }). Apply consistently across all tables so the FK join columns share an identical type, ensuring PostgreSQL uses index scans on FK lookups.*
- **`shared/schema.ts` (createdAt columns across transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is defined as .defaultNow() without .notNull(). In Drizzle-ORM this means the column is nullable in the TypeScript type and in the DB schema, so application code that inserts a row and then reads createdAt back must handle null, and any ORDER BY createdAt query may mis-sort rows where createdAt was explicitly inserted as NULL.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Also add .notNull() to executedAt and purchasedAt on the portfolio tables for the same reason.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on nextDueDate or on (userId, nextDueDate). Any scheduled job or dashboard query that fetches upcoming bills (WHERE next_due_date <= now() or WHERE user_id = ? ORDER BY next_due_date) will perform a sequential scan, which is the most common read pattern for this table.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due_date').on(t.userId, t.nextDueDate) }) to the billReminders table factory. If a global cron job scans all overdue reminders across users, also add a partial index: CREATE INDEX idx_bill_reminders_due_date ON bill_reminders (next_due_date) WHERE status != 'paid' (add a status column if not already present).*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object and query string are reconstructed on every render of AuditLogPanel, not just when filter state changes. Because this object is used directly in the queryFn closure, React Query sees a new function reference each render, causing unnecessary re-subscriptions even when params are identical.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... p.set('limit', '200'); return p; }, [actorType, action, from, to]);` — the queryKey array already has the right deps, so this just prevents the closure from being stale or rebuilt unnecessarily.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set("limit", "200")):** The audit log row limit is hardcoded to 200. On a busy production system this silently truncates results with no indication to the admin user, and the value cannot be tuned without a code deploy.
  → *Move the limit to an environment variable: `const AUDIT_LOG_LIMIT = import.meta.env.VITE_AUDIT_LOG_LIMIT ?? '200';` and replace the hardcoded string. Add a visible UI note such as `Showing up to {AUDIT_LOG_LIMIT} rows` so admins know results may be truncated.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery in ObservabilityCard has no error state and no loading state rendered. If the /api/admin/observability endpoint fails (network error, 401 after session expiry), the card silently renders all badges as 'Disabled' — indistinguishable from a real disabled state — misleading admins into thinking observability is off when the fetch simply failed.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner while loading, and render an explicit error badge (e.g. `<Badge className='bg-red-700'>Fetch failed</Badge>`) when isError is true, so admins can distinguish a real configuration state from a network failure.*

## 💡 Opportunities

### Centralised pagination middleware — 🔥 High Impact
Create a shared `parsePagination(filters: unknown): { limit: number; offset: number }` utility that clamps limit to a configurable MAX_PAGE_SIZE environment variable (default 100) and validates offset. Apply it at the start of every list method. This eliminates the repeated ad-hoc limit logic and makes the cap adjustable without touching storage code.

### Extract hardcoded virtual balance seed value to config — ⚡ Medium Impact
The initial virtual balance amount (likely 10000 or similar) is hardcoded in `getVirtualBalance` or `seedMarketData`. Move it to `process.env.INITIAL_VIRTUAL_BALANCE_CENTS` with a validated fallback. This allows curriculum designers to adjust starting balances per school or sponsor without a code deploy.

### Add a DataLoader layer for N+1 on enrollment and class queries — 🔥 High Impact
getEnrollmentsByClass fetches each student User record individually inside a loop (N+1 pattern). Introduce a simple in-request cache using a Map or the `dataloader` npm package keyed by userId so that bulk class progress summaries resolve all student records in a single `WHERE id = ANY(...)` query instead of one per row.

### Consolidate four multer instances into one configured factory — ⚡ Medium Impact
logoUpload, videoUpload, upload, and examUpload are defined separately with duplicated fileFilter logic. Extract a `makeUploader(allowedMimes, maxBytes)` factory that returns a configured multer instance. This removes ~60 lines of duplication and ensures fileFilter hardening (e.g., magic-byte validation beyond mimetype) is applied consistently across all upload surfaces.

### Split registerRoutes into domain-scoped sub-routers — 🔥 High Impact
The single registerRoutes function registers auth, AI, org management, student, leaderboard, and file routes in one 2000+ line file. Splitting into Express Router modules (authRouter, orgRouter, studentRouter, aiRouter) loaded via app.use() would make it feasible to apply middleware (rate limiting, auth checks, body-size limits) per domain, and would reduce the surface area for missed auth guards — a meaningful security improvement for a platform with minors.

### Cache Supabase org and environment lookups with a short TTL — 🔥 High Impact
getOrganization, getOrgEnvironmentById, and getOrgQuotaSettings are called on nearly every authenticated request with no caching layer. Adding a node-cache or in-process Map with a 30-second TTL for immutable-ish org config would cut Supabase read volume significantly under classroom load spikes (30+ students signing in simultaneously), reducing both latency and Supabase egress costs.

### Batch admin CSV export queries into a single transaction — ⚡ Medium Impact
If the admin-csv-export handler issues multiple sequential db queries to build report rows, wrapping them in db.transaction and selecting only required columns (avoiding SELECT *) will reduce both query round-trips and the risk of returning sensitive PII fields inadvertently. Expected impact: faster export generation and a smaller attack surface on exported data.

### Add a maximum payload size guard at job enqueue time — 🔥 High Impact
Enforce a size limit (e.g. 1 MB) on any job payload before it is written to the jobs table. A single utility function wrapping enqueueJob that JSON.stringifies and checks byteLength before inserting would prevent accidental large-payload writes across all job types, not just extract-paper, without requiring per-handler fixes.

### Centralise OpenAI call error handling with token usage logging — ⚡ Medium Impact
Both openai.chat.completions.create calls in runExtractPaper silently discard the usage field on the response. Recording prompt_tokens and completion_tokens to aiUsageEvents inside a shared wrapper function would give accurate cost attribution per job and feed the existing purge-ai-usage retention policy correctly.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least five separate table definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' in a shared config file and reference it in each .default() call. This prevents silent drift if the app ever needs to support TTD or JMD alongside BSD, and makes a currency change a one-line edit instead of a grep-and-replace exercise.

### Add a composite unique constraint to portfolioHoldings — 🔥 High Impact
portfolioHoldings has no uniqueIndex on (userId, stockId), so a race condition or retry in the buy-order flow could insert duplicate holding rows for the same user+stock, silently inflating portfolio value calculations. Adding uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) makes the DB the final consistency barrier and lets the ORM upsert (quantity += N) instead of insert on repeat purchases.

### Replace serial PKs with UUID or ULID for user-facing tables — ⚡ Medium Impact
Using serial (sequential integer) PKs on transactions, savingsGoals, and portfolioTransactions exposes internal row counts to API consumers (a student can infer how many total transactions exist in the system by observing their own IDs). For a school-aged audience this is also a minor COPPA/GDPR-adjacent data minimisation concern. Switching to gen_random_uuid() or a ULID default eliminates enumeration and makes IDs safe to expose in URLs and API responses without leaking business metrics.

### Parallel data fetching on Dashboard overview load — 🔥 High Impact
The overview tab likely triggers multiple independent useQuery calls sequentially as components mount. Prefetch all overview queries in parallel using `useQueries` from React Query or ensure query keys are primed via `queryClient.prefetchQuery` in a single effect. For a school dashboard that may have thousands of student records, this can cut initial load time by 40-60% on the overview screen.

### Centralised apiRequest error guard — 🔥 High Impact
Adding a single `if (!response.ok) throw new Error(...)` check inside the shared apiRequest utility in lib/queryClient.ts eliminates the need to fix each of the 10+ individual queryFn lambdas in this file. It also ensures any future queryFn authors get correct error propagation for free, which is critical given this app handles student PII.

### Memoize expensive chart data transformations — ⚡ Medium Impact
BarChart and LineChart data props in the overview section are likely derived from raw API responses via inline map/filter/reduce inside JSX. Wrapping these derivations in useMemo keyed on the raw query data prevents recomputation on every render tick, which matters when the parent component re-renders due to tab hover or tooltip interactions in Recharts.
