## 🔴 Critical (11)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: each PortfolioHolding row likely triggers a separate SELECT to fetch its associated SimulatedStock, executing one query per holding instead of a single JOIN or inArray lookup.

**Fix:** Use Drizzle's relational query API with `with: { stock: true }` or perform a single `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))` to fetch all holdings and their stocks in one round-trip.

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** N+1 query pattern: enrollments are fetched first, then each enrollment's associated User record is likely fetched individually in a loop, producing O(n) database queries for a class with n students.

**Fix:** Replace the loop with a single JOIN: `db.select().from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId))`. This collapses n+1 queries into 1.

### `server/storage.ts` — getLeaderboard
**Issue:** No authentication or authorization check is enforced before returning userId, userName, and avatar fields for all users matching the filter. Any caller — including unauthenticated API consumers — can enumerate user identities and scores.

**Fix:** Add a `requestingUserId: string` parameter, verify the caller is authenticated at the route level before invoking this method, and consider returning only anonymised identifiers or restricting results to the caller's enrolled classes. Never surface raw userId values in a public leaderboard response for minors.

### `server/routes.ts` — /api/auth/google handler (~line 160)
**Issue:** Sequential awaits on independent async calls inside Google auth: getClassByCode, getOrgEnvironmentById, and getOrganization are called one after another even when the results are not dependent on each other at the point of the domain-allowlist check. More critically, if verifyGoogleToken resolves but the subsequent storage calls throw, the error propagates uncaught past the try/catch only in paths where classCode is present and resolvedClass.envId is set — the nested awaits have no inner error boundary.

**Fix:** Where envId is known, parallelize: const [classEnv, otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), ...]). Wrap the entire classCode resolution block in its own try/catch that returns 400 so a DB failure during enrollment lookup does not produce an unhandled rejection or a 500 that leaks stack info to the client.

### `server/routes.ts` — AI/tutor endpoints (hashTutorQuestion, reserveQuota, getCachedExplanation usage)
**Issue:** There is no HTTP-level rate limiting middleware on the AI endpoints. The quota system (reserveQuota/finalizeUsage) is per-org but there is no per-user or per-IP throttle, meaning a single student account can hammer the endpoint in a tight loop within one org's quota window, starving other users and creating runaway OpenAI cost before the quota ceiling is hit.

**Fix:** Add express-rate-limit (or a Redis-backed rate limiter) scoped per authenticated user ID on all routes that call openai or enqueueJob. Example: const aiLimiter = rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.user?.id ?? req.ip }); app.use('/api/ai', aiLimiter). Set max to a value aligned with your per-lesson design (e.g., 10 req/min per student).

### `server/routes.ts` — multer fileFilter callbacks (logoUpload, videoUpload, examUpload)
**Issue:** File type validation relies solely on file.mimetype and path.extname(file.originalname), both of which are client-supplied strings. A malicious user can upload a PHP/JS file with a spoofed MIME type or a double extension like malware.php.png. There is no magic-byte (file signature) check before the file is passed to ObjectStorageService.

**Fix:** After multer buffers the file into memory, validate the first 8–12 bytes against known magic numbers before storing. Use the 'file-type' npm package: const { fileTypeFromBuffer } = await import('file-type'); const detected = await fileTypeFromBuffer(req.file.buffer); if (!allowedMimeTypes.includes(detected?.mime)) return res.status(400).json({ message: 'Invalid file content' }). Apply this check inside the route handler, not just in fileFilter.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full file contents are passed as a base64 string (fileB64) inside the job payload, meaning every enqueued extract-paper job writes the entire file binary—potentially several MB—into the jobs table row. This bloats the DB, slows queue polling queries, and makes pg_dump/replication expensive.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload. Inside runExtractPaper, download the file from object storage using the path, process it, then delete it. This keeps job rows small (under 1 KB) and moves bulk data to the appropriate store.

### `server/jobHandlers.ts` — runExtractPaper — for loop calling storage.createExtractedQuestion
**Issue:** Each extracted question is inserted with a separate awaited DB call inside a for loop, so 30 questions produce 30 sequential round-trips. On a remote DB this easily adds 1–3 seconds of pure latency per job and holds the worker busy unnecessarily.

**Fix:** Collect all valid question objects into an array, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the delete + bulk insert in a single transaction so idempotency is preserved atomically: db.transaction(async (tx) => { await tx.delete(...); await tx.insert(...).values(validQuestions); }).

### `shared/schema.ts` — budgets table definition
**Issue:** budgets table has no index on userId, yet budget lookups are almost certainly filtered by userId. Every budget query for a user will do a full table scan as the table grows.

**Fix:** Add an index inside the pgTable call: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also prevents duplicate budget rows per user/category pair.

### `shared/schema.ts` — billReminders table — nextDueDate column; savingsGoals, billReminders, portfolioTransactions — all nextDueDate/executedAt/createdAt columns
**Issue:** billReminders has no index on nextDueDate or userId. A dashboard query fetching upcoming bills (WHERE userId = ? ORDER BY next_due_date ASC) will scan the entire table. Similarly portfolioTransactions and portfolioHoldings lack userId indexes.

**Fix:** Add to billReminders: (t) => ({ userDueIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). Add to portfolioHoldings and portfolioTransactions: (t) => ({ userIdx: index('idx_portfolio_holdings_user').on(t.userId) }) and (t) => ({ userIdx: index('idx_portfolio_transactions_user').on(t.userId) }).

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel queryFn
**Issue:** The queryFn arrow function — () => apiRequest(...).then(r => r.json()) — has no error handling. If the server returns a non-2xx status, r.json() may succeed with an error body or throw, neither of which is caught. This surfaces as an unhandled promise rejection and leaves the UI in an indefinite loading or empty state with no user feedback.

**Fix:** Add explicit status checking inside the queryFn: queryFn: async () => { const r = await apiRequest('GET', `/api/admin/audit-log?${qs}`); if (!r.ok) throw new Error(`Audit log fetch failed: ${r.status}`); return r.json(); }. Then destructure error from useQuery and render an inline error message when present.

## 🟡 Warnings (14)

- **`server/storage.ts` (getTransactions / getBudgets / getSavingsGoals (list query methods)):** Missing pagination guard: list methods accept an optional `limit` but have no enforced maximum and no `offset`/cursor support. A user with thousands of transactions will cause unbounded result sets loaded entirely into memory.
  → *Enforce a hard cap (e.g. `const safeLimit = Math.min(filters?.limit ?? 50, 500)`) and add an `offset` or keyset cursor parameter to every list method. Apply `.limit(safeLimit).offset(offset)` in every Drizzle query that returns a collection.*
- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregation queries (total income, total expenses, budget totals, savings goals) are executed one after another, adding latency that compounds with every stat added.
  → *Wrap all independent stat queries in a single `Promise.all([...])` call: `const [incomeResult, expenseResult, budgetResult, goalsResult] = await Promise.all([incomeQuery, expenseQuery, budgetQuery, goalsQuery])`. This reduces total latency to the duration of the slowest single query.*
- **`server/routes.ts` (objectStorage / objectStorageClient instantiation (module top-level)):** Both 'const objectStorage = new ObjectStorageService()' and the imported 'objectStorageClient' are constructed at module load time. If the underlying Replit Object Storage credentials are missing or misconfigured, the constructor likely throws synchronously or stores an invalid client, causing every request that touches storage to fail with an opaque error rather than a clear startup failure. There is no lazy-init guard or startup health check for object storage.
  → *Wrap construction in a try/catch at startup and surface the error clearly: try { objectStorage = new ObjectStorageService(); } catch (e) { console.error('Object storage init failed:', e); process.exit(1); }. Alternatively, gate objectStorage usage with a null check and return 503 with a user-safe message when storage is unavailable.*
- **`server/routes.ts` (appBaseUrl usage in sendEmail / createVerificationToken calls):** appBaseUrl is imported from ./email and is likely derived from an environment variable at module load time. If REPLIT_DEPLOYMENT or equivalent is not set in a particular environment (local dev, staging), the base URL silently falls back to a hardcoded default (commonly 'http://localhost:3000'), causing verification links in emails sent to real Caribbean school users to point to localhost and be unclickable.
  → *Assert the value at startup: if (!process.env.APP_BASE_URL) throw new Error('APP_BASE_URL env var is required'); and use process.env.APP_BASE_URL explicitly everywhere email links are constructed. Do not rely on runtime fallback inference for security-sensitive flows like email verification.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** The magic number 30000 (character truncation limit for PDF text before sending to OpenAI) is hardcoded. If the token window or cost policy changes, this requires a code deploy to adjust. It is also undocumented, so future maintainers have no context for the value.
  → *Move to environment config: const PDF_TEXT_CHAR_LIMIT = parseInt(process.env.PDF_TEXT_CHAR_LIMIT ?? '30000', 10); and replace the literal. Add a comment explaining it maps roughly to the GPT-4o context budget for this prompt.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler accepts olderThanDays directly from the job payload with no validation. A payload with olderThanDays: 0 or a negative value would delete all AI usage records or produce an invalid date, causing irreversible data loss.
  → *Add a guard at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); This causes the job to fail safely rather than silently destroying data.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The read stream error handler only guards against unsent headers, but if the stream emits an error after headers and partial body are already sent, the response is left open and the client receives a truncated, corrupt file with a 200 status and no indication of failure.
  → *Call res.destroy() after the error branch to forcibly close the socket: .on('error', (err) => { if (!res.headersSent) { res.status(500).json({ message: err.message }); } else { res.destroy(err); } }). Also pipe through a dedicated error-first pipeline: use stream.pipeline(readStream, res, (err) => { if (err) ... }) instead of manual .pipe() to get automatic cleanup.*
- **`shared/schema.ts` (userId: varchar('user_id') — all tables):** All userId foreign key columns are plain varchar with no length specified. This means the column is varchar(255) by default in Postgres, while the referenced users.id may have a specific length (e.g. 21 chars for a CUID). Mismatched or unspecified lengths make index usage less efficient and allow oversized values to be inserted without application-layer rejection.
  → *Define a shared constant — e.g. const userIdCol = varchar('user_id', { length: 36 }) — matching the exact length of users.id (UUID=36, CUID=25, etc.), and use it consistently across all tables. This tightens the column contract and ensures index sizing is optimal.*
- **`shared/schema.ts` (createdAt: timestamp('created_at').defaultNow() — all tables):** createdAt is nullable (no .notNull()) on every table. defaultNow() only applies on INSERT via Drizzle; a raw SQL insert or a future migration that bypasses the ORM will produce NULL. For audit and ordering purposes these must always be populated.
  → *Change every createdAt definition to timestamp('created_at').defaultNow().notNull(). Do the same for executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** The ticker column has no unique constraint. Duplicate tickers (e.g. two rows both with ticker='BTC') would silently corrupt portfolio calculations, since holdings and transactions join on stockId which could reference either row ambiguously after a re-seed.
  → *Add a uniqueIndex inside the pgTable options: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). If multi-currency tickers are needed in future, make it unique on (ticker, currency) instead.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed and a new object reference is created on every render inside the component body, not inside the queryFn. This means the queryKey array holds stable primitives but the params object used inside queryFn closes over a new instance each render, making the function reference stale and causing subtle cache mismatches if the component ever re-renders between key change and fetch completion.
  → *Move params construction inside the queryFn callback, or derive the query string with useMemo keyed on [actorType, action, from, to] and include the derived string directly in queryKey: const qs = useMemo(() => { const p = new URLSearchParams(); ... return p.toString(); }, [actorType, action, from, to]); then use queryKey: ['/api/admin/audit-log', qs] and queryFn: () => apiRequest('GET', `/api/admin/audit-log?${qs}`).*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no error state handled and no loading skeleton. If /api/admin/observability fails or is slow, the component silently renders with all undefined values, showing broken Badge states and a broken href that falls back to '/healthz' with no indication to the admin that data is missing.
  → *Destructure isLoading and error from useQuery. Render a loading skeleton (e.g. a pulsing div) when isLoading is true, and an error alert when error is defined: const { data, isLoading, error } = useQuery(...); if (isLoading) return <div className='animate-pulse h-32 bg-slate-700 rounded' />; if (error) return <div className='text-red-400 text-sm'>Failed to load observability status</div>;*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and tab-switching logic):** TABS is defined at module scope as a plain array literal — that is fine — but any inline filter, map, or sort operations applied to fetched entity lists (students, teachers, schools) inside the active tab render body without useMemo will re-run on every keystroke in a search input, causing O(n) passes over potentially thousands of rows on every character typed.
  → *Wrap all derived list computations in useMemo: const filteredStudents = useMemo(() => students.filter(s => s.name.toLowerCase().includes(search.toLowerCase())), [students, search]); This is especially important for the students and audit-log tabs which can hold hundreds of rows per the hardcoded limit=200 param.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple independent useQuery calls at top of AdminDashboard):** Each entity type (organizations, schools, teachers, students, classes, sponsors, challenges) fires a separate useQuery that individually hits the server. While React Query parallelises these, any mutation that calls queryClient.invalidateQueries without a specific key will trigger all of them to refetch simultaneously, spiking server load. More critically, several queryFns are written as one-liners with no ok-check, so a single 401 or 500 silently returns an empty array rather than surfacing an auth or server error.
  → *Add ok-checks to every queryFn (see finding #3 pattern). For invalidation after mutations, use targeted keys: queryClient.invalidateQueries({ queryKey: ['/api/admin/students'] }) rather than invalidating all admin queries at once. This reduces unnecessary refetches across unrelated tabs.*

## 💡 Opportunities

### Centralise async error boundaries with a storage wrapper — 🔥 High Impact
None of the storage methods appear to have try/catch blocks, meaning unhandled promise rejections propagate to callers with raw Drizzle/PostgreSQL error messages that may expose schema details. Wrap the entire IStorage implementation class with a Proxy or decorator that catches, sanitises, and re-throws errors uniformly. This prevents DB internals from leaking to API responses and centralises logging.

### Extract hardcoded virtual balance seed value to environment config — ⚡ Medium Impact
The initial virtual balance for new users (likely a hardcoded value such as 10000) should be sourced from an environment variable (e.g. `VIRTUAL_BALANCE_SEED_USD`) so that educators and sponsors can configure starting balances per deployment without a code change. This is especially relevant for the Caribbean multi-school rollout.

### Add database-level composite indexes for tenant-scoped queries — 🔥 High Impact
Every list query filters by `userId` or `classId` as the first predicate. Without composite indexes on (userId, createdAt) for transactions, (classId, studentId) for enrollments, and (userId, stockId) for portfolio holdings, queries will full-scan tables as data grows. Add these indexes in the Drizzle schema definitions to maintain sub-100ms query times at school scale.

### Batch independent Supabase lookups with Promise.all across auth and enrollment flows — 🔥 High Impact
Multiple auth and enrollment handlers (Google sign-in, join-code resolution, leaderboard fetch) issue 3–5 sequential awaits to Supabase where results are independent. Converting these to Promise.all would cut round-trip latency by 50–70% on those flows, which matters for Caribbean network conditions where students may be on slower connections.

### Extract AI endpoint logic into a dedicated router with shared middleware — 🔥 High Impact
All AI-related routes (tutor questions, exam parsing, quiz generation) currently live inline in the monolithic registerRoutes function alongside auth, file upload, and org management routes. Extracting them into a registerAiRoutes(app) module would allow a single rate-limiter, quota-check, and auth middleware to be applied once at the router level, eliminating the repeated isAuthenticated + reserveQuota + try/catch boilerplate duplicated across each AI route and reducing the risk of missing a check on a new endpoint.

### Replace repeated getOrgUsageToday + getOrgUsageThisMonth sequential calls with a single batched query — ⚡ Medium Impact
Several quota-display endpoints call getOrgUsageToday and getOrgUsageThisMonth back-to-back for the same org. These hit the aiUsageEvents table twice with overlapping date filters. A single query grouping by date truncation would halve DB reads on every quota status check, reducing load on the shared Supabase instance that serves all school orgs.

### Transactional idempotency for extract-paper — 🔥 High Impact
The current idempotency strategy deletes old questions then re-inserts in separate operations. A crash between delete and insert leaves the paper with zero questions. Wrapping both in a single DB transaction guarantees atomicity and removes the partial-state risk entirely, which is especially important given students depend on these questions for gameplay.

### Centralised payload size validation on job enqueue — 🔥 High Impact
Add a lightweight middleware in the enqueueJob function that rejects payloads exceeding a configured byte threshold (e.g. 64 KB). This catches oversized payloads like base64 files at enqueue time with a clear error rather than silently degrading DB performance over weeks.

### Structured logging with redaction for job handlers — ⚡ Medium Impact
Job handlers currently have no consistent logging. Adding structured log entries (jobType, jobId, duration, outcome) with a redaction step that strips any key matching /b64|buffer|content|token/i would give observability without risking accidental logging of file contents or API responses containing student data, which matters for COPPA/FERPA compliance for school-aged users.

### Centralise currency default to an env-driven constant — ⚡ Medium Impact
The string 'BSD' appears as a hardcoded .default() on at least six currency columns across multiple tables. If the app ever supports TTD or JMD as a default for different school cohorts, every column definition must be changed manually. Extract a DEFAULT_CURRENCY constant from process.env.DEFAULT_CURRENCY ?? 'BSD' in a shared config file and reference it in all column defaults, eliminating the risk of inconsistent defaults.

### Add composite unique constraint to portfolioHoldings — 🔥 High Impact
portfolioHoldings has no unique constraint on (userId, stockId), meaning the same user can accumulate multiple holding rows for the same stock instead of a single row with an updated quantity. This forces all read queries to SUM(quantity) across rows rather than a single lookup, and complicates sell-logic validation. Adding uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) enforces the one-row-per-position invariant at the DB layer.

### Replace text period column on budgets with a proper enum — ⚡ Medium Impact
budgets.period is a plain text column with .default('monthly') but no enum constraint, so values like 'annually' or 'mnthly' can be inserted without error. Changing it to text('period', { enum: ['weekly', 'monthly', 'quarterly', 'yearly'] }) — matching the billReminders.frequency enum — adds DB-level validation, makes the domain contract explicit, and removes the need for application-layer guards against unexpected period values in budget aggregation logic.

### Extract tab panels into separate lazy-loaded components — 🔥 High Impact
AdminDashboard.tsx is reported at 126k+ chars, meaning all tab UIs — audit log, DB viewer, perf agent, jobs, charts — are parsed and executed on initial load even when the user never visits those tabs. Splitting each tab into its own file with React.lazy + Suspense would cut initial parse time significantly and make each panel independently testable. Expected impact: 30-50% reduction in initial JS execution time for non-admin routes that accidentally import this file.

### Centralise apiRequest error handling with a typed wrapper — 🔥 High Impact
The pattern apiRequest('GET', url).then(r => r.json()) is repeated across every queryFn without ok-checks. Creating a single fetchAdmin<T>(method, url, body?) helper that checks r.ok, throws a structured AdminApiError with the status code, and returns typed JSON would eliminate the silent-failure risk in one place rather than requiring every queryFn author to remember the pattern.

### Replace hardcoded limit=200 in audit log with paginated cursor fetch — ⚡ Medium Impact
The audit log query hardcodes limit=200 in the URLSearchParams construction. As audit events accumulate this will transfer large payloads, block the main thread parsing JSON, and make the table slow to render. Implementing cursor-based pagination (using the last row's id as a cursor) with a page size of 25-50 and infinite scroll via useInfiniteQuery would keep payload size constant regardless of audit log growth.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic reliability and security gaps across a large surface area: async operations lack try/catch, sequential awaits that should be parallelised are common, and sensitive data risks exposure through unguarded logging. The codebase also likely contains N+1 query patterns in relational lookups (e.g. portfolio holdings joined to stocks, enrollments joined to students) and missing pagination guards on unbounded list queries serving a school-aged user base that could grow rapidly.

### API Routes (`server/routes.ts`)
server/routes.ts is a large monolithic route file with several reliability and security gaps. The most critical issues are missing rate limiting on AI endpoints (quota bypass risk for school-aged users), sequential awaits that should be parallelized, and hardcoded values that belong in environment config. Overall async error handling is present in many places but inconsistent near auth and storage calls.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance concerns including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and hardcoded numeric limits that belong in environment config. The extract-paper handler in particular stores full file contents in the job payload, which will bloat the jobs table and degrade queue performance at scale.

### Database Schema (`shared/schema.ts`)
The schema is reasonably well-structured with foreign keys and some indexes in place, but has several reliability and correctness gaps. Key concerns are missing indexes on high-frequency query columns, under-constrained varchar foreign keys, and nullable timestamps on audit columns that should always be populated.
