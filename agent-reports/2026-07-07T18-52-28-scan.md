## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes have no rate limiting applied before registration. Any authenticated or unauthenticated caller can hammer these endpoints, incurring unbounded OpenAI API costs and enabling denial-of-wallet attacks. Auth endpoints directly above them DO have limiters, making this omission inconsistent and higher risk for a school-aged user base.

**Fix:** Add dedicated rate limiters before these registrations, e.g. `const aiLimiter = rateLimit({ windowMs: 60_000, max: 20 });` then `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before calling the register functions. Import the limiter from the existing rateLimiter module to keep policy centralised.

### `server/routes.ts`: robots.txt handler (~line 37) and sitemap.xml handler (~line 54)
**Issue:** The `host` and `proto` values are taken directly from `x-forwarded-host` and `x-forwarded-proto` headers without validation and interpolated into response bodies. A reverse-proxy misconfiguration or a spoofed request can inject arbitrary URLs into the Sitemap directive and sitemap XML, enabling cache-poisoning or open-redirect abuse.

**Fix:** Resolve the canonical base URL once at startup from an environment variable (`BASE_URL=https://finsight.app`) and use that constant in both handlers. Remove the runtime header-based construction entirely: `const BASE_URL = process.env.BASE_URL ?? 'https://finsight.app';`

### `server/jobHandlers.ts`: runExtractPaper
**Issue:** fileB64 (full base64-encoded PDF or image) is stored directly in the job payload, meaning every job row in the DB carries the entire file binary. For a 5 MB PDF this is ~6.7 MB of base64 per row, causing table bloat, slow job-queue queries, and high memory pressure when the worker deserialises the payload.

**Fix:** Before enqueuing, upload the file to object storage and store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at the start of execution and delete the object after successful extraction or terminal failure.

### `server/jobHandlers.ts`: parsePrivatePath / uploadPrivateObject
**Issue:** PRIVATE_OBJECT_DIR is checked for existence but its value is used directly in path construction without sanitising the relPath argument. A crafted relPath containing '../' sequences could escape the intended bucket prefix and write to or read from unintended object storage locations.

**Fix:** After constructing the full path, assert that it starts with the resolved root prefix: if (!full.startsWith(resolvedRoot)) throw new Error('Path traversal detected'); Use Node's path.posix.normalize on relPath before concatenation and reject any path whose normalised form starts with '..'.

### `server/index.ts`: res.json middleware / log(logLine)
**Issue:** Every API response body is captured into `capturedJsonResponse` and then unconditionally serialised into the log line with `JSON.stringify(capturedJsonResponse)`. For a financial literacy app serving school-aged users this will log quiz answers, scores, session tokens returned in response envelopes, or any PII the API surfaces — with no size cap beyond the already-applied 10 mb body limit, a large response can also cause a synchronous multi-MB string allocation on the hot logging path.

**Fix:** Add an allowlist of safe response shapes to log (e.g. only `{ ok, id, count }` scalar fields) or strip sensitive keys before logging. At minimum truncate: `JSON.stringify(capturedJsonResponse).slice(0, 200)` and skip logging entirely for auth-related paths (`/api/auth`, `/api/session`, `/api/user`).

### `server/index.ts`: startSchedulers / setInterval calls
**Issue:** The scheduler intervals (`setInterval` for weekly digest, AI purge, and job worker polling) are created at startup but their handles are never stored and never cleared in the graceful shutdown handler. On SIGTERM the process closes the HTTP server and DB but the intervals keep the event loop alive, preventing a clean exit and causing the process to hang until forcibly killed — or fire one final time against a half-torn-down DB connection.

**Fix:** Capture every `setInterval` return value: `const digestTimer = setInterval(...)`. In the `shutdown()` function call `clearInterval(digestTimer)` (and equivalents) before `httpServer.close()`. Wrap all timer handles in an array for a single `timers.forEach(clearInterval)` call.

---

### `client/src/pages/AdminDashboard.tsx`: sendMessage async function inside AdminHelpChat
**Issue:** apiRequest is awaited and then res.json() is awaited sequentially, but more critically there is no check on res.ok before calling res.json(). A non-2xx HTTP response (e.g. 401, 500) will attempt to parse an error body as a success payload, silently setting data.reply to undefined and showing no error to the user.

**Fix:** After `const res = await apiRequest(...)` add `if (!res.ok) throw new Error(await res.text());` before calling `res.json()`. The existing catch block will then surface the error message to the user correctly.

## 🟡 Warnings (12)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 118)):** Seven async Supabase init steps are chained sequentially with `.then()`. A single failure in any middle step (e.g. `seedFinancialAcademyLesson`) short-circuits all subsequent steps via the single terminal `.catch()`, so `seedStaticModules`, `seedRegionalContent`, and `seedGameContent` silently never run. Students could launch the app with missing game or regional content.
  → *Run independent seed steps in parallel where safe, and use `Promise.allSettled` so every step is attempted regardless of prior failures: `const results = await Promise.allSettled([seedFinancialAcademyLesson(), initStaticContentTables(), seedRegionalContent(), seedGameContent()]); results.filter(r => r.status === 'rejected').forEach(r => console.error('[Supabase] Seed error:', r.reason));`. Keep `initSupabaseTables` and `trimOrgNamesInSupabase` sequential only if they are true prerequisites.*
- **`server/routes.ts` (/healthz handler (~line 77)):** Dynamic `import('./db')` and `import('drizzle-orm')` inside a hot health-check endpoint causes Node's module loader to be invoked on every request until the module is cached. Under rapid polling from Railway or uptime monitors before the cache warms, this adds unnecessary overhead and obscures whether latency is from the DB or the loader.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports alongside the other imports. The health check handler then simply calls `await db.execute(sql\`select 1\`)` with no dynamic loading cost.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The db.delete() call inside the purge-ai-usage handler has no try/catch. An unexpected DB error will cause an unhandled promise rejection that the job runner may not surface cleanly, and the job will silently fail without a useful error recorded against it.
  → *Wrap the entire handler body in try/catch, log the error with context (olderThanDays, cutoffDate), and re-throw so the job runner can mark the job as failed and schedule a retry: try { ... } catch (err) { capture(err, { job: 'purge-ai-usage', cutoff: cutoff.toISOString() }); throw err; }*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop):** Questions are inserted sequentially inside a for loop with individual awaited storage.createExtractedQuestion() calls. For a 40-question paper this is 40 serial round-trips to the DB. There is also no transaction: if the process crashes mid-loop the partial set remains, and the idempotency delete at the top of the next retry wipes only previously committed rows, not any partially written ones from the current run.
  → *Collect all valid question objects into an array first, then insert in a single db.insert(extractedQuestions).values(validQuestions) call inside a transaction that also updates the paper status. This reduces DB round-trips from N to 1 and makes the operation atomic.*
- **`server/jobHandlers.ts` (runExtractPaper – idempotency on concurrent retries):** The idempotency guard (delete then re-insert) is not protected by a distributed lock or DB-level serialisation. If a retry is enqueued while a prior attempt is still running (e.g. slow OpenAI call), both workers execute the delete concurrently and then both insert, producing duplicates that the delete-before-insert pattern is meant to prevent.
  → *Add a DB-level advisory lock or use an optimistic status check: at the start of the handler, atomically set the paper status to 'processing' only if it is currently 'pending' or 'failed' (UPDATE exam_papers SET status='processing' WHERE id=? AND status IN ('pending','failed') RETURNING id). If no row is returned, a concurrent worker already owns the job—exit early with a no-op return.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** Day-of-week rollback logic is wrong. `const daysSince = (dow + 0) % 7` always returns 0 on Sunday (dow=0), so on Sunday before 7 pm the function returns today at 19:00 which is in the future, causing an early return — correct. But on any other day `daysSince` equals dow, meaning Monday (dow=1) subtracts 1 day and lands on Sunday, Wednesday (dow=3) subtracts 3 days and lands on Sunday — this part is actually correct only by accident because `% 7` with no offset produces the right arithmetic for non-Sunday days but the comment `(dow + 0)` suggests a placeholder offset was forgotten, and Sunday itself (dow=0) subtracts 0 days, so the guard `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` is the only thing saving it. The dead `+ 0` is a latent bug magnet if the offset is ever updated incorrectly.
  → *Replace `(dow + 0) % 7` with the explicit idiom `dow === 0 ? 0 : dow` or simply `dow`, and add a unit test covering Sunday before/after 7 pm and each weekday to pin the behaviour before any future edits.*
- **`server/index.ts` (startup sequence (probeDatabase, seedDatabase, registerRoutes)):** `probeDatabase()` and `seedDatabase()` are awaited sequentially before `registerRoutes()` is called and the HTTP server starts listening. If the DB probe takes several seconds (cold Replit container, connection pool warm-up) the process is unresponsive to health checks and load-balancer probes during that window, and a seed failure aborts startup entirely even in production where seed is a no-op.
  → *Start listening immediately after `registerRoutes()` returns, then run `probeDatabase()` and `seedDatabase()` in the background with their own error handling. Gate a `/_health` route on a `dbReady` flag that flips after the probe succeeds, so the load balancer defers traffic rather than the process refusing to bind.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest):** Both scheduler functions are called from `setInterval` callbacks with no concurrency guard. If a DB round-trip inside `maybeRunWeeklyDigest` takes longer than the polling interval (e.g. a slow Drizzle insert during a cold pool), the next tick fires a second concurrent invocation. The atomic `onConflictDoNothing` protects against double-enqueue for the digest, but `maybeRunAiUsagePurge` has no such guard and can issue overlapping DELETE statements.
  → *Add a module-level `let digestRunning = false` / `let purgeRunning = false` flag in each scheduler. At the top of each async function: `if (digestRunning) return; digestRunning = true;` and in a `finally` block: `digestRunning = false;`. This costs two boolean variables and eliminates the overlap risk entirely.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat - scroll div ref):** Inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and fires scrollIntoView on every re-render of the message list, not just when new messages arrive. This causes janky scroll behaviour whenever loading state toggles.
  → *Replace with a stable useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage - missing useCallback):** sendMessage and handleKey are plain functions redeclared on every render of AdminHelpChat. They are passed as event handlers to JSX elements, so every render produces new function references, preventing any downstream memoization from working and adding unnecessary allocation on every keystroke.
  → *Wrap both in useCallback: `const sendMessage = useCallback(async () => { ... }, [messages, input, loading]);` and `const handleKey = useCallback((e) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);`.*
- **`client/src/pages/AdminDashboard.tsx` (useQuery calls throughout dashboard sections):** Multiple sibling useQuery hooks for users, schools, lessons, analytics etc. are likely declared at the top level of the main AdminDashboard component without staleTime configuration, meaning every tab switch or parent re-render triggers background refetches for all data simultaneously, hammering the API unnecessarily for a school-aged app with predictable, slow-changing data.
  → *Add `staleTime: 5 * 60 * 1000` (5 minutes) to each useQuery options object for read-heavy admin lists (users, schools, lessons). Leave analytics queries at a shorter staleTime if near-real-time data is needed.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort expressions in JSX render body):** Filtering and sorting of users, lessons, and school lists (search input driven) are computed as inline expressions directly inside the JSX return, meaning they run on every render regardless of whether the underlying data or search term changed. With hundreds of school-aged users this is a measurable CPU cost on every keystroke.
  → *Extract each filtered/sorted list into a useMemo: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(searchTerm.toLowerCase())), [users, searchTerm]);` and reference filteredUsers in JSX instead of the inline expression.*

## 💡 Opportunities

### Centralise BASE_URL and app metadata as validated env config: 🔥 High Impact
Create a `config.ts` module that reads and validates all environment variables at startup (BASE_URL, NODE_ENV, SUPABASE_URL, etc.) using zod or a simple assertion helper. This eliminates the header-injection risk, removes magic fallbacks like `'localhost'`, and gives a single import for all routes that need the public URL.

### Add per-user AI request quotas backed by Supabase: 🔥 High Impact
IP-level rate limiting alone is insufficient for a school-aged audience sharing NAT IPs. Store per-user daily request counts in Supabase and enforce a configurable quota (e.g. 50 AI requests/day) in middleware before proxying to OpenAI. This bounds API costs per student and enables teachers/orgs to adjust limits.

### Structured startup logging for Supabase seed results: ⚡ Medium Impact
Replace bare `console.error` in the seed chain with structured log entries that include step name, duration, and success/failure status. This enables Railway log drain or Sentry to alert on partial seed failures at deploy time rather than discovering missing content through student bug reports.

### Parallelise independent async lookups in job handlers: ⚡ Medium Impact
Several handlers (visible in the truncated section) perform sequential awaits for independent DB reads such as fetching user data, module data, and badge state. Wrapping independent queries in Promise.all() would cut handler latency proportionally to the number of parallelisable calls, typically 2-4x for handlers with 3+ sequential lookups.

### Centralise job error capture into a handler wrapper: 🔥 High Impact
Error capture via Sentry is added ad-hoc only in ai-health-check but is absent from purge-ai-usage and extract-paper. Introduce a wrapHandler(name, fn) higher-order function that wraps every handler in a uniform try/catch with Sentry capture, structured logging, and re-throw. This eliminates repeated boilerplate and guarantees no handler silently swallows errors.

### Move OpenAI model identifiers to environment config: ⚡ Medium Impact
The string 'gpt-4o' is hardcoded in at least two openai.chat.completions.create calls and max_completion_tokens is hardcoded at 4096. Externalising these to OPENAI_EXTRACTION_MODEL and OPENAI_MAX_TOKENS env vars allows model upgrades or cost-reduction rollouts (e.g. switching to gpt-4o-mini for simpler tasks) without a code deploy.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase() and seedDatabase() are currently awaited sequentially. Running them with `await Promise.all([probeDatabase(), seedDatabase()])` after moving them to a background init function would cut startup latency by whichever of the two is faster (typically probe < 200 ms, seed variable). Low-risk change with measurable cold-start improvement on Replit.

### Structured logging to replace string concatenation: 🔥 High Impact
The `log()` helper and all scheduler log calls use `console.log` with interpolated strings. Replacing with a lightweight structured logger (pino is <5 kb, already common in the Node ecosystem) would emit JSON lines with level, timestamp, and context fields. This makes it trivial to filter out sensitive fields centrally, integrate with Sentry breadcrumbs, and add log-level gating so debug noise is suppressed in production without code changes.

### Extract scheduler logic into a dedicated module: ⚡ Medium Impact
Weekly digest scheduling, AI purge scheduling, and their DB-claim patterns currently live inline in server/index.ts alongside HTTP server setup. Moving them to server/schedulers.ts with exported start/stop functions would make the concurrency guards, interval handles, and shutdown cleanup testable in isolation and keep index.ts focused on app wiring — directly addressing the untested getMostRecentSundayAt7pm edge cases.

### Parallel data fetching with Promise.all on dashboard load: 🔥 High Impact
If any server-side or effect-based data fetching sequences multiple awaits (user count, school count, lesson count), replace with Promise.all([fetchUsers(), fetchSchools(), fetchLessons()]). For a Caribbean school deployment this could cut initial dashboard load time by 40-60% on high-latency connections.

### Extract tab sections into separate lazy-loaded components: 🔥 High Impact
AdminDashboard appears to render all tab content (users, schools, lessons, analytics, AI chat) in one giant component. Splitting each tab into its own component with React.lazy/Suspense would reduce initial JS parse time and isolate re-render boundaries, preventing analytics chart re-renders when the user table search term changes.

### Add role check guard at component top before any query fires: ⚡ Medium Impact
If useAdminAuth only redirects after the component mounts and queries begin, sensitive admin data (user lists, financials) is fetched before the auth redirect completes. Move a synchronous role assertion before the first useQuery declaration and return null early to prevent unauthorised data fetches from hitting the wire.
