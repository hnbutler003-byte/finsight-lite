## 🔴 Critical (9)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 113-115)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. For a school-aged user base, a single user or script can exhaust your OpenAI quota or cause runaway costs with no throttle in place.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/15 min per IP via express-rate-limit) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Mirror the pattern already used for auth routes.

### `server/routes.ts`: robots.txt handler (~line 45) and sitemap handler (~line 57)
**Issue:** Both handlers build the base URL directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header and have the app emit a sitemap/robots.txt pointing to an arbitrary domain, enabling cache-poisoning or SEO hijacking.

**Fix:** Add a validated base URL to environment config: `BASE_URL=https://finsightlite.com`. Read it with `process.env.BASE_URL` and fall back to the header-derived value only in development. Validate any header-derived host against a regex like `/^[a-zA-Z0-9.-]+(:\d+)?$/` before use.

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** Five sequential awaits (getUserXp, txCount query, gameCount query, lessonRows query, badgeRows query) run one-after-another per user. In a fan-out digest over hundreds of students this serialises all DB I/O and will saturate the event loop.

**Fix:** Rewrite studentWeekStats to use Promise.all for the four independent DB queries: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...txCount...), db.select(...gameCount...), db.select(...lessonRows...), db.select(...badgeRows...)]). This cuts per-user latency to roughly the slowest single query instead of the sum.

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** The function issues one SELECT then one INSERT/UPSERT per student row inside a for-loop with sequential awaits and no batching. For a school with 500 students this fires 1000+ individual DB round-trips synchronously, blocking the job worker thread and risking timeout.

**Fix:** Batch the existence check with a single inArray query on (userKind='student', userId IN [...studentIds]), build a Set of existing userIds, then bulk-insert only missing contacts with a single db.insert().values([...missingContacts]).onConflictDoNothing(). Cap batch size to 500 rows to stay within parameter limits.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Every budget lookup by user (the overwhelmingly common query path) will do a full table scan as the table grows.

**Fix:** Add a composite index in the budgets pgTable call: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This mirrors the pattern already used on transactions.

### `shared/schema.ts`: simulatedStocks table — lastPriceUpdateDate column
**Issue:** lastPriceUpdateDate is typed as text instead of timestamp. Storing dates as text prevents range queries, ordering, and index efficiency. For a price-feed column queried to detect staleness this is a correctness and performance bug.

**Fix:** Change the column to: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed/migration code that writes ISO strings to pass Date objects instead.

### `server/index.ts`: res.json monkey-patch in request logger middleware
**Issue:** Every API JSON response body is serialized via `JSON.stringify(capturedJsonResponse)` and appended to the log line unconditionally. For a financial literacy app handling student records, quiz results, and any auth/session payloads, this risks leaking PII, session tokens, or grade data into log aggregators (Sentry breadcrumbs, stdout scrapers, Replit logs). There is no truncation, allowlist, or redaction step.

**Fix:** Add a size/key guard before logging: only log the response body when `res.statusCode >= 400` (error debugging), or log only a fixed set of non-sensitive keys (e.g., `{ id, status, count }`). At minimum truncate: `logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 120)}`; and strip any key named `token`, `secret`, `password`, or `session`.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both `setInterval` handles are stored in local variables but are never passed to the graceful shutdown handler. On SIGTERM the process exits (or `httpServer.close()` fires) while intervals remain active, meaning an in-progress scheduler tick can attempt DB writes after the connection pool is being torn down, causing unhandled promise rejections and potential data corruption in the job queue.

**Fix:** Capture both interval IDs and clear them as the first step in the shutdown handler, before awaiting `httpServer.close()`: `clearInterval(weeklyDigestInterval); clearInterval(purgeInterval);`. Also guard each scheduler callback so it skips execution if a module-level `isShuttingDown` boolean is true, preventing a tick that fired just before SIGTERM from running to completion against a closing pool.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage POST /api/admin/help-chat
**Issue:** The full conversation history (`next`) is sent to /api/admin/help-chat on every message. There is no length cap, so a long session accumulates unbounded tokens/payload. For a school-aged app with potentially many concurrent users this creates a DoS vector and unpredictable LLM cost. Additionally, sensitive admin context embedded in earlier messages is replayed to the model on every turn.

**Fix:** Slice the context window before sending: `const context = next.slice(-10)` (keep last 10 turns) and send `{ messages: context }`. Add a server-side max-messages guard as well. Consider stripping any messages containing PII patterns before forwarding to the LLM.

## 🟡 Warnings (15)

- **`server/routes.ts` (Supabase init chain (~line 118-125)):** The seven sequential `.then()` seeding calls run in a single chain where each step must complete before the next begins. A failure in `seedFinancialAcademyLesson` silently skips `initStaticContentTables`, `seedStaticModules`, etc., with one catch swallowing the original error context and leaving the DB in a partially seeded state.
  → *Run independent seeds in parallel and capture per-step errors: `await Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`Seed step ${i} failed:`, r.reason)));`. Split steps that are truly dependent (initTables before seed) into sequential groups first, then parallelize within each group.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 100)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject` without path sanitization. A request to `/public-objects/../../private/keys` could traverse outside the intended public prefix depending on how the storage service resolves paths.
  → *Normalize and validate the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|^\//, ''); if (!filePath || filePath.startsWith('..')) return res.status(400).json({ error: 'Invalid path' });` Ensure the storage service also enforces a prefix allowlist server-side.*
- **`server/jobHandlers.ts` (registerJobHandlers (all handlers)):** No registerJobHandler callback wraps its body in try/catch. If any handler throws synchronously or an awaited promise rejects, the error propagates to the job runner uncaught. Depending on the runner implementation this can crash the worker process or silently mark jobs as succeeded.
  → *Wrap each handler body in try/catch and return a structured error result or rethrow with context: try { ... } catch(err) { throw new Error(`[${handlerName}] ${err instanceof Error ? err.message : String(err)}`); }. Alternatively add a top-level wrapper in registerJobHandler itself that enforces this contract for all handlers.*
- **`server/jobHandlers.ts` (admin-csv-export handler):** The job payload field 'type' is consumed directly to select which data to export (students, teachers, classes, schools, sponsors) with no server-side authorization check inside the handler. Any caller that can enqueue a job can export any dataset. Additionally the full dataset is materialized in memory as a JS array before CSV serialization, risking OOM for large tables.
  → *Add an explicit allowlist check (if (!['students','teachers','classes','schools','sponsors'].includes(type)) throw new Error('Invalid export type')) and validate that job.payload.requestedByRole === 'admin' against a DB lookup of the requesting user before fetching data. For large exports switch to a streaming cursor/query approach piped directly into the object storage upload stream rather than buffering the full result set.*
- **`server/jobHandlers.ts` (weekly-digest / bootstrapStudentContacts):** When orgId is falsy the code falls back to db.select().from(users) with no WHERE clause, returning every user in the system. This full-table scan payload is then iterated per-row. For a multi-tenant SaaS with many organisations this is an unbounded memory and CPU spike on a scheduled job.
  → *Remove the unguarded fallback full-table scan. Require orgId for digest fan-out or paginate with a LIMIT/OFFSET or keyset cursor (WHERE id > lastSeenId LIMIT 500) and enqueue child jobs per page rather than processing all users in one job execution.*
- **`shared/schema.ts` (transactions, budgets, documentUploads, savingsGoals, billReminders, linkedCards — createdAt columns):** createdAt uses .defaultNow() but no .notNull(). If a row is inserted with an explicit null the column accepts it, making audit trails unreliable and breaking ORDER BY createdAt queries that assume non-null.
  → *Append .notNull() to every createdAt definition, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Same applies to purchasedAt and executedAt in portfolio tables.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on (userId, nextDueDate), yet the primary product feature is surfacing upcoming bills sorted by due date per user. Without this index every reminder query scans the full table.
  → *Add to the billReminders pgTable definition: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, portfolioHoldings — userId varchar column):** userId is declared as varchar() with no length argument. In PostgreSQL this is equivalent to text, which is fine, but it is inconsistent with the referenced users.id column type and removes the self-documenting constraint that IDs have a known max length (e.g. 36 for UUID or 21 for NanoID). A mismatch in declared types can also cause Drizzle type inference warnings and subtle join plan differences.
  → *Decide on the ID format (UUID = 36 chars, NanoID = 21 chars) and declare consistently: userId: varchar('user_id', { length: 36 }).notNull(). Ensure users.id in models/auth uses the same length so FK join types align.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** getMostRecentSundayAt7pm contains a dead-code bug: `(dow + 0) % 7` always returns `dow` unchanged (the comment implies it was meant to offset to Sunday). For a Sunday (dow=0) this computes 0 days back, sets the date to today, then the `> now` guard subtracts 7 days—correct by accident. For any other day it returns the wrong date (e.g., Saturday dow=6 gives 6 days back, landing on Sunday, but Monday dow=1 gives 1 day back, landing on Sunday of the prior week rather than the most recent past Sunday). The `+ 0` is a residue of a deleted offset constant.
  → *Replace `(dow + 0) % 7` with `dow % 7` and add a unit test covering each day-of-week. If the intent is 'most recent Sunday at 19:00 local', verify whether `d.setHours` should use UTC to avoid DST drift in Caribbean timezones (UTC-4/UTC-5)—prefer `setUTCHours`.*
- **`server/index.ts` (startup sequence (probeDatabase, seedDatabase, registerRoutes, startJobWorker)):** The startup sequence awaits `probeDatabase()`, then `seedDatabase()`, then `registerRoutes()`, then `startJobWorker()` sequentially before `httpServer.listen()` is called. If any step takes longer than the platform's health-check timeout (common on Replit cold starts), the process is killed before it can serve traffic. Additionally, `seedDatabase` and `startJobWorker` are independent of each other and could run in parallel, but are not.
  → *Run `seedDatabase()` and `startJobWorker()` concurrently with `Promise.all([seedDatabase(), startJobWorker()])` after `registerRoutes()` completes. Wrap the entire startup block in a try/catch that calls `process.exit(1)` with a descriptive message on failure so the platform restarts cleanly rather than hanging. Consider moving `startJobWorker` to after `httpServer.listen()` so the port opens as early as possible.*
- **`server/index.ts` (maybeRunAiUsagePurge / AUTO_PURGE_OLDER_THAN_DAYS):** The AI usage purge deletes records older than `AUTO_PURGE_OLDER_THAN_DAYS` days using a date computed at scheduling time inside the job payload, but the deduplication check (`aiUsagePurgeRuns`) uses only the calendar date string as the key. If the server restarts mid-day the purge will be skipped for the rest of that UTC day even if the previous run's job never actually completed (e.g., worker crashed). There is no `status` column check against the `jobs` table to confirm prior completion.
  → *Before inserting the dedup row, query the `jobs` table for today's purge job and only skip if a job with `status = 'completed'` exists for that `runDate`. If the existing job is `failed` or missing, delete the stale `aiUsagePurgeRuns` row and re-enqueue. This mirrors the rollback pattern already used in `maybeRunWeeklyDigest`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a brand-new function on every render of the messages list. React treats a changed ref as unmount+remount, so scrollIntoView fires on every keystroke in the input (because `input` state change re-renders the parent), not only when a new message arrives.
  → *Extract the scroll target into a useRef<HTMLDivElement>(null) and call scrollIntoView inside a useEffect that depends on [messages.length]: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length]);` Then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages.map key={i}):** Chat messages are keyed by array index. When the assistant reply is prepended or any message is inserted, React reuses the wrong DOM nodes, causing flicker and potentially mismatched ARIA roles for screen-reader users (important for a school-aged audience).
  → *Assign a stable id when creating each message: `{ role, content, id: crypto.randomUUID() }` and use `key={m.id}` in the map. Update the ChatMessage type to include `id: string`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage and handleKey):** sendMessage and handleKey are recreated on every render because they are plain function declarations inside the component. The input element re-renders on every keystroke (input state changes), so both handlers are reallocated at high frequency. handleKey closes over sendMessage, making useCallback on sendMessage alone insufficient.
  → *Wrap sendMessage with useCallback([messages, input, loading]) and handleKey with useCallback([sendMessage]). This stabilises the onKeyDown reference passed to the input element and avoids unnecessary child reconciliation.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – catch block silent error swallow):** The catch block in sendMessage swallows the actual error and only appends a generic string to the chat. Network errors, 401 auth failures, and 500 server errors are all treated identically. An expired admin session will silently appear as 'something went wrong' with no redirect to login.
  → *Inspect the error before swallowing: `if (err instanceof Response && err.status === 401) { setLocation('/admin/login'); return; }`. For other errors log `console.error('[AdminHelpChat]', err)` (ensure no user PII is in the error object) and show a toast with the HTTP status when available so admins can self-diagnose.*

## 💡 Opportunities

### Centralised BASE_URL environment config: 🔥 High Impact
Replace all header-derived URL construction (robots.txt, sitemap, any redirect helpers) with a single `BASE_URL` env var read once at startup. This eliminates the host-injection surface, simplifies staging/prod environment management, and removes duplicated proto+host logic that currently appears in two separate handlers.

### Route-module registration parallelisation: ⚡ Medium Impact
The six domain routers are imported in parallel with Promise.all but then registered sequentially with six `await` calls. If any registration function performs async I/O (DB schema checks, etc.), these could also be parallelised with `Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])`, reducing cold-start time proportionally to the slowest initialiser.

### Structured startup logging for Supabase seed results: ⚡ Medium Impact
Replacing the single `.catch` on the seed chain with per-step `Promise.allSettled` results logged as structured JSON (step name, duration, error message) would make Railway/Sentry startup diagnostics actionable without needing to redeploy with added console logs. Expected impact: faster incident resolution when seeding fails in production.

### Idempotency key on weekly-digest child jobs: 🔥 High Impact
Child send-email jobs enqueued inside the digest fan-out have no deduplication key. If the parent job is retried after partial completion (e.g. worker crash at student 300 of 500), duplicate emails are sent. Add an idempotencyKey composed of weekStart + userId to each enqueueJob call so the job table can reject duplicates on re-enqueue.

### Extract hardcoded time constants to environment config: ⚡ Medium Impact
Values like 7 * 24 * 60 * 60 * 1000 (digest window) and the olderThanDays arithmetic in purge-ai-usage are scattered inline. Extract to named constants or process.env variables (DIGEST_WINDOW_DAYS, AI_USAGE_RETENTION_DAYS) so ops can tune retention and digest cadence without a code deploy.

### Stream CSV directly to object storage instead of buffering: ⚡ Medium Impact
admin-csv-export loads the entire result set into a JS array, serializes it to a string, then creates a Buffer before uploading. For exports exceeding ~50k rows this can allocate hundreds of MB. Replace with a Transform stream that writes CSV rows as they are fetched using a DB cursor, piped directly into the object storage write stream, reducing peak memory to a single chunk size.

### Partial index for active bill reminders: 🔥 High Impact
Add a partial index on billReminders filtered to rows where nextDueDate >= now() so the reminder-polling query (likely run on every dashboard load) hits only future-due rows rather than the full history. Expected impact: query time stays sub-millisecond even with years of historical reminder records.

### Centralise currency default into a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, portfolioHoldings). Extract to a const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it everywhere. When the app expands to TTD or JMD, one line changes instead of six with risk of missing one.

### Replace serial PKs with UUID or NanoID for user-facing tables: ⚡ Medium Impact
Sequential integer PKs on transactions, savingsGoals, and portfolioHoldings expose row counts and insertion rates to any authenticated user who inspects API responses — a minor but real information-disclosure risk for a financial app. Switching to gen_random_uuid() or a NanoID also eliminates enumeration attacks on document upload IDs. Migration cost is moderate but the security and privacy benefit is meaningful for a school-aged audience.

### Structured log context instead of string concatenation: 🔥 High Impact
Replace ad-hoc `log()` string building with a structured logger (e.g., `pino`) that emits JSON lines. This enables log-level filtering (suppress debug in prod), automatic redaction of configured key names (eliminating the PII-in-logs risk), and correlation IDs per request—critical for auditing in a school-data context. Expected impact: eliminates the response-body PII leak class of bugs and reduces mean-time-to-diagnose for production incidents.

### Centralise all magic numbers and schedule config into validated env schema: ⚡ Medium Impact
Interval durations (15 min for digest check, purge check frequency) are hardcoded literals scattered through the file. Add them to a Zod-parsed env config object at module load alongside `SESSION_SECRET` and `AI_PURGE_OLDER_THAN_DAYS`. A single parse-and-fail-fast block at startup with clear error messages is easier to audit and configure per environment (test: 1-min intervals, prod: 15-min) without code changes.

### Extract scheduler bootstrap into a dedicated module: ⚡ Medium Impact
Both weekly-digest and AI-purge scheduler setup (interval creation, dedup logic, enqueue calls) live inline in `server/index.ts`, making the file responsible for HTTP setup, middleware, shutdown, and job scheduling simultaneously. Moving schedulers to `server/scheduler.ts` that exports `startSchedulers(intervals: NodeJS.Timeout[])` and accepts a shared shutdown array would allow independent unit testing of scheduler logic and make the shutdown cleanup in `index.ts` a one-liner.

### Virtualise long admin lists with TanStack Virtual: 🔥 High Impact
If the dashboard renders paginated tables of users, schools, or audit logs as full DOM lists, switching to @tanstack/react-virtual would cap DOM nodes at ~20 rows regardless of dataset size. Expected impact: 60-80% reduction in layout time for lists >200 rows, which matters on low-end Caribbean school devices.

### Parallel-fetch independent admin queries with Promise.all: ⚡ Medium Impact
Multiple useQuery calls for stats, users, schools, and leaderboards likely fire sequentially if any shared suspense boundary or waterfall exists. Grouping independent fetches under a single initialising useQueries call (or Promise.all in a loader) would cut initial dashboard load by N-1 round trips where N is the number of independent endpoints.

### Move hardcoded API path strings to a central constants file: ⚡ Medium Impact
Strings like '/api/admin/help-chat', '/api/admin/users', etc. appear inline throughout the file. Centralising them in `src/lib/api-routes.ts` makes base-URL changes (e.g. staging vs production) a one-line edit and enables TypeScript to catch typos at compile time rather than at runtime.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good use of Promise.all for parallel imports and proper try/catch on async endpoints. Key concerns are: unprotected AI/chat/image/audio routes with no rate limiting, host header injection risk in robots.txt and sitemap, and sequential Supabase seed chain that blocks on failures without granular recovery.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including sequential awaits inside per-user loops that hammer the database, missing try/catch around async job handler bodies leaving unhandled rejections that may silently swallow failures, and a bootstrapStudentContacts function that does unbounded full-table scans and per-row DB writes in a loop with no batching. Security exposure is moderate: no auth/ownership check gates the admin-csv-export job payload, and student stats queries trust userId directly from job payload without validation.

### Database Schema (`shared/schema.ts`)
The schema is reasonably well-structured with foreign keys and some indexes in place, but has critical gaps in missing indexes on high-frequency query columns, unsafe nullable timestamps on audit fields, and an untyped varchar for user_id that should be constrained. Security and reliability risks are moderate but addressable.

### Server Startup (`server/index.ts`)
The server startup sequence performs multiple blocking synchronous-style async operations (DB probe, seed, job scheduling) before the HTTP server is ready to accept connections, and the interval-based schedulers have no cleanup path registered in the graceful shutdown handler, risking leaked timers and missed in-flight jobs on SIGTERM. Response body logging unconditionally serializes every JSON response payload—including potentially sensitive financial or PII data—into the process log stream.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security gaps including unprotected async operations, a scroll ref that creates a new function on every render causing memory pressure, and chat messages keyed by array index which breaks React reconciliation. The inline scroll ref callback and missing useCallback on sendMessage/handleKey are the most impactful render-path issues given the chat panel re-renders on every keystroke.
