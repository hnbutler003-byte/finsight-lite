## 🔴 Critical (8)

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 47 and 65)
**Issue:** The base URL is constructed directly from x-forwarded-proto and x-forwarded-host headers without any validation or allowlist. An attacker can spoof these headers to inject arbitrary URLs into the Sitemap directive and sitemap.xml <loc> tags, enabling cache-poisoning or open-redirect abuse.

**Fix:** Replace header-derived base URL with a single hardcoded environment variable: const base = process.env.APP_BASE_URL. If not set, fall back to a safe default and log a warning. Never trust x-forwarded-host for URL construction without validating it against an allowlist of known hostnames.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 118-120)
**Issue:** The OpenAI-backed chat, image, and audio routes have no rate limiting applied before they are registered. For a school-aged user base this creates direct exposure to cost amplification attacks and denial-of-service via AI endpoint abuse — the highest-cost operations in the system have weaker protection than the auth endpoints.

**Fix:** Apply a dedicated AI rate limiter middleware before these registrations: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); where aiLimiter uses a stricter window (e.g., 10 requests per user per minute) than the auth limiter. Also ensure auth middleware is verified inside each route handler.

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** Five sequential awaits (getUserXp, txCount query, gameCount query, lessonRows query, badgeRows query) run one-after-another per user. In the weekly-digest fan-out this function is called for every student, multiplying the latency by 5x per user and exhausting the DB connection pool under any real school population.

**Fix:** Wrap the four independent DB queries inside a single Promise.all: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...txCount), db.select(...gameCount), db.select(...lessonRows), db.select(...badgeRows)]). This reduces per-user wall-clock time from sum-of-latencies to max-of-latencies.

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** When orgId is falsy the function executes `db.select().from(users)` with no WHERE clause, returning every user row in the database into memory. For a platform with thousands of students this will cause an OOM in the job worker and expose all user records including any sensitive fields to the job processing context.

**Fix:** Never allow an unbounded full-table scan in a job handler. Add a hard guard: if (!orgId) throw new Error('orgId required for digest fan-out'); and require callers to always pass an orgId. If a cross-org digest is genuinely needed, paginate with LIMIT/OFFSET or use a cursor-based query, and stream enqueueJob calls in batches of e.g. 100 rather than loading all rows.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Queries filtering budgets by user (the dominant access pattern) will do full table scans as data grows.

**Fix:** Add a composite index in the table's index callback: index('idx_budgets_user_category').on(t.userId, t.categoryId). Mirror this pattern for savingsGoals, billReminders, linkedCards, and documentUploads which all query by userId but lack indexes.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Interval handles are never stored or cancelled. When the process receives SIGTERM the graceful shutdown handler closes the HTTP server and DB connection but leaves both intervals firing, which can cause in-flight DB queries to run against a closing pool and emit unhandled-rejection noise or corrupt the job table.

**Fix:** Store the return values: `const digestInterval = setInterval(...); const purgeInterval = setInterval(...);` then in the shutdown handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before awaiting httpServer.close() and db pool teardown.

### `server/index.ts`: res.on('finish') logging middleware (capturedJsonResponse block)
**Issue:** Every API response body is JSON.stringify'd and appended to the log line unconditionally. For a Caribbean school-aged user base this will routinely log quiz results, progress data, session tokens embedded in response payloads, and any future fields containing DOB or contact info — violating basic data-minimisation and COPPA/GDPR-equivalent obligations.

**Fix:** Add an allowlist of safe, non-sensitive paths or strip the body logging entirely: remove the `res.json` monkey-patch and the `capturedJsonResponse` branch from the log line. If response inspection is needed for debugging, gate it behind `process.env.LOG_RESPONSE_BODIES === 'true'` and ensure it is never enabled in production.

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage
**Issue:** The `/api/admin/help-chat` endpoint is called with the full message history on every send but there is no auth token or admin session validation attached to this client-side request. If `apiRequest` does not automatically attach admin credentials, any unauthenticated user who discovers the endpoint can query the AI assistant and potentially extract platform information.

**Fix:** Verify that `apiRequest` attaches the admin session cookie or Authorization header. If not, explicitly add `credentials: 'include'` and ensure the server-side `/api/admin/help-chat` route is protected by the same admin middleware used on other `/api/admin/*` routes. Add a server-side check that returns 401 before invoking the AI if the session is not an admin session.

## 🟡 Warnings (14)

- **`server/routes.ts` (Supabase init chain (~line 123-131)):** The seven Supabase init/seed functions are chained sequentially with .then(), meaning each waits for the previous to fully complete before starting. On cold start this serialises what could be independent operations, adding unnecessary latency before the server is fully ready.
  → *Group independent operations with Promise.all where ordering allows: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Verify which seeds have true data dependencies before parallelising.*
- **`server/routes.ts` (/healthz handler (~line 82)):** The /healthz endpoint uses dynamic import inside a hot request path (import('./db') and import('drizzle-orm') on every health check call). Dynamic imports are cached after first load but the async resolution overhead and module graph traversal add latency on every probe, and the pattern obscures the actual startup dependency.
  → *Move db and sql imports to the top of the file as static imports alongside the other imports. The /healthz handler then simply calls await db.execute(sql`select 1`) with no dynamic import overhead.*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export')):** The handler loads the entire dataset (students, teachers, classes etc.) into a single in-memory array before building the CSV string. For large schools this will exhaust worker heap. Additionally, the full CSV content is effectively stored in the job return value which is written back to the job table row, creating a large payload in the DB.
  → *Stream rows in batches (e.g. LIMIT 500 with cursor) and write chunks directly to the object storage upload stream rather than accumulating a string. Return only { objectPath, rowCount, fileName } from the handler — never the CSV content itself — so the jobs table row stays small.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage')):** The handler has no try/catch. A DB error (connection drop, lock timeout) will throw an unhandled rejection inside the job runner. Depending on the job framework's error handling this may silently mark the job failed with no structured error payload, or crash the worker process.
  → *Wrap the delete in try/catch: try { const result = await db.delete(...); return { deletedRows, cutoffDate }; } catch (err) { throw new Error(`purge-ai-usage failed: ${err instanceof Error ? err.message : String(err)}`); } so the job framework receives a typed error and can retry or alert correctly. Apply the same pattern to all other handlers missing error boundaries.*
- **`server/jobHandlers.ts` (bootstrapStudentContacts — existing contact upsert loop):** Inside bootstrapStudentContacts, for each student a SELECT is issued to check for an existing emailContacts row, followed conditionally by an INSERT — classic N+1 with no idempotency guard. If the weekly-digest job is retried or runs concurrently (e.g. two workers), this races and can create duplicate contact rows, causing duplicate digest emails to students.
  → *Replace the per-row SELECT+INSERT with a single bulk INSERT ... ON CONFLICT (user_kind, user_id) DO NOTHING using Drizzle's onConflictDoNothing(), batched in chunks of 100–500 rows. This is both idempotent and reduces N round-trips to a handful of batch inserts.*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text instead of timestamp. This forces application-layer date parsing on every price-staleness check, prevents database-level date arithmetic, and allows malformed strings to be stored silently.
  → *Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). Update any seed/insert code that currently writes ISO strings to pass Date objects instead.*
- **`shared/schema.ts` (categories table, userId column):** categories.userId is declared as varchar with no length argument. All other user_id columns are also bare varchar with no length. Drizzle maps this to VARCHAR with no limit in Postgres (effectively text), but it loses the alignment with whatever fixed-width ID your auth provider emits (e.g. Clerk uses 32-char IDs). A mismatch is a silent bug waiting to happen.
  → *Define a shared constant: const userIdCol = varchar('user_id', { length: 32 }) and use it everywhere a user_id column appears. This enforces the contract at the DB level and makes length mismatches immediately obvious during migrations.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt (and all other createdAt columns)):** All createdAt columns use .defaultNow() but are missing .notNull(). This means an explicit INSERT passing null for createdAt will succeed and store NULL, breaking any ORDER BY createdAt or audit logic that assumes the column is always populated.
  → *Append .notNull() to every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default handles the no-value case; notNull prevents an explicit null override.*
- **`shared/schema.ts` (simulatedStocks table, ticker column):** ticker has no unique index. If a seed script or admin tool inserts a duplicate ticker (e.g. 'BSD-GOV-10Y' twice), portfolioHoldings and portfolioTransactions will silently reference ambiguous stocks and aggregate queries will double-count positions.
  → *Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table's index callback. Also consider adding .notNull() to previousPrice and priceChangePct since downstream P&L calculations likely treat NULL as 0 anyway.*
- **`server/index.ts` (seedDatabase / probeDatabase / registerRoutes call-site in main()):** Blocking sequential awaits on startup: probeDatabase, seedDatabase, registerRoutes, and startJobWorker are awaited one-after-another. probeDatabase and seedDatabase have no dependency on each other and could run in parallel, adding unnecessary latency before the server is ready to accept connections.
  → *Wrap independent startup tasks with Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])` before calling registerRoutes, reducing cold-start time by the slower of the two operations minus the faster.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest async scheduler functions):** Both scheduler functions are passed directly to setInterval but any unhandled rejection inside them (e.g. a DB error after the outer try/catch boundary) will surface as an unhandledRejection on the process. The outer catch only logs; if the logging call itself throws, the interval callback exits with an uncaught exception.
  → *Wrap each setInterval callback in a top-level async IIFE that attaches a .catch() to the returned promise: `setInterval(() => { maybeRunWeeklyDigest().catch(err => captureError(err)); }, INTERVAL_MS)`. This ensures Sentry capture even if the internal try/catch is bypassed.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scrollIntoView ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a new function on every render of the messages list. This fires scrollIntoView not just when a new message is added, but on every re-render (e.g., when the input value changes), causing janky scroll behavior and unnecessary DOM thrashing.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage async function):** `res.json()` is called without checking `res.ok`. If the server returns a 4xx/5xx with a JSON error body, the code will silently use that error payload as if it were a valid reply, potentially rendering a raw server error object as a chat message or throwing an unhandled parse error.
  → *Add a response status check before parsing: `if (!res.ok) throw new Error(await res.text());` immediately after the `await apiRequest(...)` call, so the existing catch block handles server errors uniformly.*
- **`client/src/pages/AdminDashboard.tsx` (ChatMessage messages array – key={i}):** Using array index as React key (`key={i}`) on the messages list means React cannot correctly reconcile edits or deletions. If a message is prepended or removed, all subsequent message components will be needlessly re-mounted, and any internal state (e.g., copy-to-clipboard state) will be lost or misattributed.
  → *Assign a stable unique id when each message is created: `{ id: crypto.randomUUID(), role: 'user', content: text }` and use `key={m.id}` in the render loop. Update the `ChatMessage` type accordingly.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (registerRoutes function — duplicate health check endpoints (~line 37 and 82)):** Two health check endpoints exist: /api/health (returns JSON with no DB check) and /healthz (performs a live DB query). This is confusing for ops teams and uptime monitors — Railway and external monitors may hit the wrong one and get a false positive when the database is down.
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and hardcoded interval values):** The digest and purge scheduler intervals (e.g. 60_000 ms, 3_600_000 ms) are hardcoded magic numbers with no environment override. In a CI or staging environment this forces real timers to fire at production cadence, making test isolation difficult and wasting job-worker resources.
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – open state toggle and large parent file):** AdminHelpChat is defined inside the same ~150KB file as the full AdminDashboard. Because it is a module-level function declaration rather than a lazily imported component, the entire chat UI code (plus its closure over module scope) is parsed and executed on every admin page load, even when the chat is never opened. Additionally, the floating chat button causes the entire AdminDashboard tree to re-render on every keystroke in the chat input because state lives inside a component rendered inline with no memoization boundary.

## 💡 Opportunities

### Centralised auth middleware guard for all /api routes: 🔥 High Impact
Currently auth enforcement is delegated to individual route modules. Adding a single app.use('/api/', requireAuth) middleware (with explicit opt-outs for /api/health, /api/auth/*) before any route registration would prevent accidental unauthenticated exposure when new routes are added — critical for a platform serving minors.

### Structured startup logging with timing for init chain: ⚡ Medium Impact
Wrapping the Supabase init chain and the parallel module imports with console.time/timeEnd or a structured logger would give visibility into cold-start bottlenecks on Railway. This costs nothing and immediately surfaces which seed function is slow without needing a profiler.

### Extract sitemap URL list to config or CMS: 📌 Low Impact
The sitemap URLs are hardcoded in routes.ts. Moving them to a config file or generating them from route metadata would prevent the sitemap from silently going stale as new public pages are added, improving SEO reliability without manual maintenance.

### Centralised job error boundary wrapper: 🔥 High Impact
Create a wrapHandler(name, fn) higher-order function that wraps every registerJobHandler callback in a standardised try/catch, logs structured errors with job id and payload type (excluding PII), and re-throws. This eliminates the need to duplicate error handling in each of the 10+ handlers and ensures consistent observability across all job types.

### Digest fan-out concurrency limiter: ⚡ Medium Impact
The weekly-digest loop enqueues one job per contact sequentially. Adding a p-limit(10) concurrency wrapper around the enqueueJob calls would cut fan-out time by ~10x while preventing thundering-herd on the jobs table. Expected impact: fan-out for 1,000 students drops from ~10s to ~1s.

### Move PRIVATE_OBJECT_DIR validation to startup: ⚡ Medium Impact
parsePrivatePath throws at call-time if PRIVATE_OBJECT_DIR is missing, meaning CSV export jobs only fail at runtime mid-execution. Validate all required env vars (PRIVATE_OBJECT_DIR, email config, DB URL) in a single startup check that kills the process with a clear message before any requests or jobs are accepted, following the 12-factor fail-fast pattern.

### Centralise currency default as an env-driven constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract DEFAULT_CURRENCY from process.env.DEFAULT_CURRENCY ?? 'BSD' and reference it in each .default() call. When the app expands to TTD or JMD markets, one config change covers the entire schema instead of a risky multi-table migration.

### Add partial index on transactions for unsynced rows: ⚡ Medium Impact
Auto-sync and reconciliation jobs almost certainly query WHERE is_auto_synced = false repeatedly. A partial index—index('idx_transactions_unsynced').on(t.userId).where(sql`is_auto_synced = false`)—will make those scans O(unsynced rows) instead of O(all transactions), which matters most when the table is large and most rows are already synced.

### Replace serial PKs with UUIDs on user-facing tables: 🔥 High Impact
Sequential integer PKs on transactions, savingsGoals, and portfolioHoldings expose record counts and insertion order to clients (a compliance and enumeration risk in a financial app for minors). Switching to pgTable with uuid('id').defaultRandom().primaryKey() eliminates enumeration attacks and aligns with the varchar user_id already coming from the auth provider, at the cost of a one-time migration.

### Structured JSON logging instead of console.log string concatenation: 🔥 High Impact
Replace the log() helper and the middleware's string-concat log lines with a structured logger (pino is already common in the Node ecosystem). This eliminates the temptation to append raw response bodies, makes log ingestion in Caribbean-region cloud providers (e.g. Google Cloud Logging) trivially filterable, and gives you log-level control so debug verbosity can be disabled in production without code changes.

### Centralise graceful shutdown into a single registered handler with timeout: 🔥 High Impact
The current SIGTERM handler is incomplete and there is no SIGINT handler. Extract a single `shutdown(signal)` async function, register it for both SIGTERM and SIGINT, add a hard `setTimeout(() => process.exit(1), 10_000)` safety valve, and sequence: clearIntervals → httpServer.close() → db pool end. This prevents zombie processes in container restarts on Replit and ensures in-flight requests drain cleanly.

### Replace child_process.spawn for Vite dev server with a library or documented boundary: ⚡ Medium Impact
Spawning the Vite dev process from within the Express server startup couples two unrelated lifecycles and means a Vite crash silently detaches without restarting. Either run them as separate processes under a process manager, or at minimum attach stderr/stdout listeners and a robust 'close' handler that calls captureError so Sentry records Vite failures rather than swallowing them.

### Parallel data fetching with Promise.all: 🔥 High Impact
The dashboard almost certainly fires multiple sequential `useQuery` hooks that each wait for the previous network round-trip before starting. Identify independent queries (users, schools, analytics, leaderboard) and either batch them into a single `/api/admin/dashboard-summary` endpoint or ensure they are fired in parallel. This can cut initial load time by 2-4x on slow Caribbean mobile connections.

### Code-split and lazy-load dashboard tabs: 🔥 High Impact
At ~150KB of TSX, the entire dashboard is compiled into one chunk. Splitting each major tab (Users, Schools, Analytics, Content) into lazily loaded route segments or React.lazy components would reduce the initial JS parse time significantly on low-end devices, which is critical for a school-aged audience likely on budget Android hardware.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Any search/filter/sort logic applied to user or school lists should be wrapped in useMemo keyed on the raw data and filter state. Without this, every keystroke in a search input re-runs the full filter over potentially thousands of rows on the main thread, blocking input responsiveness.

### API Routes (`server/routes.ts`)
The route file is structurally sound with good practices like parallel module imports and rate limiting on auth endpoints. Key concerns are: the Supabase init chain runs sequentially instead of in parallel, host/proto headers are trusted without validation creating a potential open-redirect/SSRF vector in robots.txt and sitemap, and the AI/chat/image/audio routes registered via registerChatRoutes/registerImageRoutes/registerAudioRoutes have no visible rate limiting applied before registration.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including N+1 sequential awaits inside fan-out loops that should be parallelised, missing try/catch around async DB operations in job handlers that will silently fail or crash the worker, and the weekly-digest bootstrapStudentContacts function doing a full unbounded users table scan when no orgId is provided which is dangerous at scale for a school app. Security and idempotency gaps exist in the CSV export and digest fan-out paths.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency filter columns, nullable timestamps that should be NOT NULL, and an untyped varchar user_id that risks silent truncation across all tables.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability and security gaps: blocking synchronous-style sequential awaits delay startup, interval handles are never stored or cleared on shutdown, and the graceful shutdown handler is incomplete. The logging middleware unconditionally serializes full JSON response bodies to stdout, which risks leaking PII or sensitive financial data for every API call.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including missing auth guards on sensitive mutations, a ref callback creating a new function on every render causing excessive scroll operations, and async operations lacking proper error boundaries.
