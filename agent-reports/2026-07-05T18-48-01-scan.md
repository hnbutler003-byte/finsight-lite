## 🔴 Critical (8)

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 38 and ~line 56)
**Issue:** Host header injection: `req.headers['x-forwarded-host']` and `req.headers.host` are used verbatim to construct absolute URLs embedded in robots.txt and sitemap.xml. An attacker can send a crafted Host header to poison the Sitemap directive or inject arbitrary content into the XML, potentially poisoning caches or misdirecting crawlers.

**Fix:** Resolve the base URL from a single trusted environment variable (e.g. `process.env.PUBLIC_BASE_URL`) set at deploy time. Fall back to a hardcoded production domain, never to raw request headers. Remove all header-derived base URL logic from these two handlers.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 107-109)
**Issue:** AI and media generation endpoints (chat, image, audio) are registered with no rate limiting middleware. For a school-aged audience on a cost-sensitive platform these routes call external paid APIs (OpenAI etc.) and are trivially abusable, leading to runaway cost or denial-of-service for legitimate users.

**Fix:** Apply a dedicated AI rate limiter (e.g. 10 req/min per IP) before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` and export `aiLimiter` from rateLimiter.ts alongside the existing limiters.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** Full base64-encoded file content (fileB64) is stored in the job payload in the jobs table. A single PDF or image could easily be 5-20MB encoded, causing DB bloat, slow job queue queries, and potential OOM when multiple jobs are dequeued simultaneously.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object storage path in the payload. In runExtractPaper, download the file from object storage using the path instead of decoding a base64 string from the payload. This keeps job rows small (<1KB) and moves binary data to the appropriate store.

### `server/jobHandlers.ts`: runExtractPaper – question insert loop (for loop over questions)
**Issue:** Extracted questions are inserted one at a time in a sequential await loop. For a paper with 60 questions this fires 60 serial round-trips to the DB. If any single insert fails mid-loop, partial data is left and the idempotency delete at the top of a retry wipes valid prior work but the loop restarts from scratch with no bulk efficiency.

**Fix:** Collect all valid question objects into an array and use a single db.insert(extractedQuestions).values(allQuestions) call. Wrap it in a transaction with the preceding delete so the wipe and re-insert are atomic, eliminating partial-state risk entirely.

### `server/index.ts`: setInterval calls in startSchedulers (inferred from truncated section)
**Issue:** Scheduler intervals are started at boot but their handles are never stored or cleared. When the process receives SIGTERM (e.g. Replit container restart, deployment), the intervals keep the event loop alive, in-flight DB writes can be torn mid-transaction, and weekly-digest claim rows may be inserted without the corresponding job being enqueued — leaving orphaned claim rows that permanently block that week's digest from retrying.

**Fix:** Capture every setInterval return value: `const handles: NodeJS.Timeout[] = []; handles.push(setInterval(maybeRunWeeklyDigest, ...))`. In the SIGTERM/SIGINT handler call `handles.forEach(clearInterval)` before `httpServer.close()`. Also add a 10-second forced exit: `setTimeout(() => process.exit(1), 10_000).unref()` so the process never hangs if a DB call stalls during shutdown.

### `server/index.ts`: res.json middleware / capturedJsonResponse
**Issue:** The logging middleware captures and JSON.stringifies every API response body unconditionally. For endpoints returning student quiz results, guardian contact info, or AI-generated content this (1) logs PII/sensitive data to stdout which may be harvested from Replit logs, (2) holds the full response object in closure memory until the 'finish' event fires, and (3) can serialize multi-KB AI responses on every request adding measurable GC pressure.

**Fix:** Add an allowlist check before logging the body: only include it when `res.statusCode >= 400` (errors only) or when a non-sensitive debug flag is set. Truncate to 200 chars: `logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 200)`. For routes under /api/ai or /api/users explicitly skip body capture by checking `path` before monkey-patching `res.json`.

### `client/src/pages/AdminDashboard.tsx`: sendMessage (AdminHelpChat)
**Issue:** The full `messages` array (entire conversation history) is sent to `/api/admin/help-chat` on every turn with no length or token guard. A user could accumulate thousands of messages and send an unbounded payload, causing server-side cost abuse or denial of service against the AI provider.

**Fix:** Slice the history before sending: `const context = next.slice(-20);` (keep last 20 messages) and send `{ messages: context }`. Also add a max message count check: `if (messages.length > 100) { setMessages(prev => prev.slice(-50)); }` before building `next`.

---

### `client/src/pages/AdminDashboard.tsx`: apiRequest POST /api/admin/help-chat
**Issue:** There is no client-side auth check before sending messages to `/api/admin/help-chat`. If `useAdminAuth` is used elsewhere in the page but not inside `AdminHelpChat`, a session-expired or unauthenticated user who somehow has the DOM rendered could still fire requests to the AI endpoint, relying entirely on server-side enforcement with no UX feedback.

**Fix:** Import and call `useAdminAuth()` inside `AdminHelpChat` (or pass an `isAuthenticated` prop from the parent). Guard `sendMessage` with: `if (!isAuthenticated) { toast({ title: 'Session expired', description: 'Please log in again.' }); return; }` before making the API call.

## 🟡 Warnings (11)

- **`server/routes.ts` (Supabase init chain (~line 112-119)):** Seven async Supabase operations are chained sequentially with `.then()`. Each one must complete before the next starts, unnecessarily extending cold-start time. Some (e.g. `seedStaticModules`, `seedRegionalContent`, `seedGameContent`) are likely independent and could run concurrently.
  → *Group independent seed operations into `Promise.all`: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` — validate dependency ordering first.*
- **`server/routes.ts` (/healthz handler (~line 79)):** Dynamic `import('./db')` and `import('drizzle-orm')` are called on every health check request. Health checks are hit frequently by Railway and uptime monitors. Repeated dynamic imports after first resolution are cached by Node, but the repeated property destructuring and async overhead add unnecessary latency and obscure intent.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to static top-level imports alongside the other imports at the top of the file. The health check handler then simply calls `await db.execute(sql\`select 1\`)` directly.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch. A DB connection error or constraint violation will produce an unhandled rejection that may crash the worker process rather than being recorded as a failed job with retry semantics, depending on the job runner implementation.
  → *Wrap the db.delete call in try/catch, capture the error via the sentry captureError utility already used in other handlers, then rethrow so the job framework can apply its retry/backoff policy. Pattern matches the ai-health-check handler.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse – objectPath parameter):** The objectPath is split and used directly to construct bucket and object names with no validation or sanitization. A caller supplying a path like ../../sensitive-bucket/admin-keys could traverse to unintended buckets, leaking private objects.
  → *After splitting, validate that the resolved bucket matches an allowlist from environment config (e.g. ALLOWED_BUCKETS=exports,uploads) and that the object path contains no .. segments. Return 400 immediately if either check fails before calling objectStorageClient.*
- **`server/jobHandlers.ts` (openai import – `import { openai } from './replit_integrations/chat/routes'`):** The OpenAI client is imported from a route module rather than a dedicated singleton. This creates a hidden coupling where changing the route module's exports breaks job handlers, and it risks importing Express route side-effects (middleware registration, etc.) into the job worker process.
  → *Extract the OpenAI client instantiation into its own module (e.g. server/lib/openaiClient.ts) that reads OPENAI_API_KEY from process.env and exports the client. Import from there in both the route file and jobHandlers.ts.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** Day-of-week rollback logic is subtly wrong. `const daysSince = (dow + 0) % 7` with dow=0 (Sunday) yields 0, so `d` stays on today (Sunday). Then `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` only rolls back if the 7pm anchor is in the future, but if it is exactly Sunday before 7pm the function returns today's not-yet-reached 7pm window and `target.getTime() > Date.now()` guard exits early — correct. However on Sunday after 7pm dow=0 gives daysSince=0, d equals today at 19:00 which is in the past, so it does NOT subtract 7 days and correctly returns today. The real bug: on any other day, `(dow + 0) % 7` for Monday (dow=1) gives 1, rolling back 1 day to Sunday — correct. But the variable name and the `+ 0` suggest a timezone offset was intended and removed, leaving the logic correct only by accident. More critically the function uses `new Date()` wall-clock with no timezone pinning; for Caribbean users (UTC-4/UTC-5) a server running in UTC will fire the digest 4-5 hours early relative to local Sunday 7pm.
  → *Accept a timezone string parameter (default 'America/Jamaica' or read from env var DIGEST_TIMEZONE). Use the Temporal API or a library like date-fns-tz to compute 'most recent Sunday 19:00' in that local zone before converting to UTC for comparison. Add a unit test covering Sunday-before-7pm, Sunday-after-7pm, and Monday cases in both UTC and UTC-5.*
- **`server/index.ts` (seedDatabase and probeDatabase calls at startup):** Both `seedDatabase()` and `probeDatabase()` are awaited sequentially before the HTTP server starts listening. If the DB probe succeeds but seeding takes 3-10 seconds (inserting curriculum rows, etc.), the server is not yet accepting connections — load balancers and Replit's health checker will time out and mark the instance unhealthy, causing restart loops on cold starts.
  → *Start listening immediately after `probeDatabase()` succeeds, then run `seedDatabase()` in the background: `seedDatabase().catch(e => { captureError(e); log('seed failed: ' + e.message); })`. Guard all seeded-data-dependent routes with a readiness flag (`let isReady = false`) that returns HTTP 503 until seeding completes, rather than blocking the port.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest):** Both scheduler functions are missing outer async error boundaries for the DB claim insert itself. If `db.insert(...).onConflictDoNothing()` throws (connection dropped, schema mismatch), the error propagates out of the async setInterval callback as an unhandled promise rejection. In Node 18+ this crashes the process; in earlier versions it silently swallows the error and skips that run permanently.
  → *The inner try/catch in maybeRunWeeklyDigest only wraps enqueueJob, not the insert. Wrap the entire function body including the insert in a single top-level try/catch that calls `captureError(e)` and `log(...)`. Do the same for maybeRunAiUsagePurge. This is already partially done but the catch block must encompass the db.insert line.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat):** The scroll-into-view ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` creates a new function on every render, causing the scroll to fire on every state change in the component tree, not just when new messages arrive. This is also a subtle memory concern as React holds references to each discarded function until GC.
  → *Replace with a proper useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage / handleKey (AdminHelpChat)):** `sendMessage` and `handleKey` are recreated on every render because they are plain `async` functions and arrow functions defined inside the component body without `useCallback`. Since `AdminHelpChat` likely re-renders on `input` keystroke, these recreate on every character typed, and any child button receiving `onClick={sendMessage}` will also re-render unnecessarily.
  → *Wrap both in `useCallback`: `const sendMessage = useCallback(async () => { ... }, [input, messages, loading]);` and `const handleKey = useCallback((e: React.KeyboardEvent) => { ... }, [sendMessage]);`. Ensure the send button is a stable component or wrapped in `React.memo`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat (chat panel open state)):** When the chat panel is open and `loading` is true, the input element is disabled but there is no visual aria-live region or screen-reader announcement for the 'Thinking…' state. For a school-aged audience on an accessibility-sensitive platform, this creates an accessibility gap. Additionally `data-testid` attributes are present on interactive elements but the floating button has no `aria-label`, so screen readers announce only the icon's implicit text.
  → *Add `aria-label={open ? 'Close help chat' : 'Open help chat'}` to the floating button. Add `aria-live='polite'` and `aria-atomic='true'` to the messages container div so assistive technologies announce new messages automatically.*

## 💡 Opportunities

### Centralised base-URL config module: 🔥 High Impact
Create a `config.ts` that reads and validates all environment variables (PUBLIC_BASE_URL, DB_URL, OPENAI_KEY, etc.) at startup with a schema validator like zod. This eliminates the scattered `process.env` reads and header-derived URL construction across multiple handlers, and surfaces misconfiguration at boot rather than at request time.

### Domain-router registration parallelisation: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six individual awaited calls. If any registration function does async I/O, this is slower than necessary. Wrap the registration calls in Promise.all where the register functions are safe to run concurrently, reducing startup latency on cold boots.

### Structured startup logging with timing: ⚡ Medium Impact
Add timestamped structured log entries at the start and end of the Supabase init chain and route registration. Currently failures are swallowed into a single console.error. With timing data operators can detect slow seed operations degrading cold-start without adding APM tooling, which is especially valuable on Railway's ephemeral container model.

### Parallelize independent async setup calls in job handlers: ⚡ Medium Impact
Several handlers that do multiple independent awaits (e.g. fetching paper metadata and deleting old questions before extraction) could use Promise.all. Parallelising even two 50ms DB calls saves latency per job invocation and improves throughput under concurrent job load.

### Add a payload size guard at job enqueue time: 🔥 High Impact
Enforce a max payload size (e.g. 64KB) inside the enqueueJob function and throw before inserting oversized payloads. This acts as a systemic backstop preventing any future handler from accidentally storing large blobs in the job table, without relying on per-handler discipline.

### Centralise OpenAI call error handling with token usage logging: ⚡ Medium Impact
Both openai.chat.completions.create calls in runExtractPaper lack error handling and do not record token usage to aiUsageEvents. Wrapping them in a shared helper that catches API errors, logs token counts, and re-throws with structured context would improve cost visibility and debuggability across all AI job handlers.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase, any config fetches, and registerJobHandlers (if it does I/O) are currently sequential. Running them with Promise.all could cut cold-start time by 40-60% on Replit's shared DB tier where each round-trip is 50-150ms.

### Structured JSON logging instead of console.log strings: 🔥 High Impact
Replacing the custom log() function with a structured logger (pino is zero-dep and 5x faster than console.log) would let Replit/Sentry ingest log level, source, and request ID as filterable fields rather than parsing freeform strings — critical for auditing AI usage and digest runs for a regulated school platform.

### Extract scheduler state into DB-backed distributed lock: ⚡ Medium Impact
The current in-process interval + DB claim pattern breaks if Replit ever runs two instances (e.g. during a rolling deploy). Replacing setInterval with a lightweight pg_advisory_lock or the existing jobs table as a proper cron-lock ensures exactly-once execution without orphaned claim rows regardless of instance count.

### Memoize derived chart/table data computations: 🔥 High Impact
Any filtering, sorting, or aggregation of user/school data done inline in the render body (common in large admin dashboards) should be wrapped in `useMemo` keyed on the raw query data. For a dataset of hundreds of students this prevents O(n) recomputation on every keystroke in search fields, directly reducing input lag.

### Consolidate admin bootstrap queries with Promise.all: 🔥 High Impact
If the dashboard fires sequential `await apiRequest(...)` calls during initialization (e.g. fetching users, schools, stats separately in useEffects), replace with `const [users, schools, stats] = await Promise.all([fetchUsers(), fetchSchools(), fetchStats()])`. This cuts initial load time by the sum of individual round-trip latencies minus the longest one.

### Extract AdminHelpChat into its own file with React.memo: ⚡ Medium Impact
AdminHelpChat is a self-contained feature with its own state. Keeping it in the same file as the full dashboard means every hot-module reload or lint pass processes the entire 147k-char file. Moving it to `components/AdminHelpChat.tsx` and wrapping with `React.memo` prevents re-renders triggered by unrelated dashboard state changes (e.g. pagination, search input).
