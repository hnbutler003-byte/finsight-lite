## 🔴 Critical (6)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Buy/sell trade flow performs three separate async writes (upsert holding, insert portfolio transaction, update virtual balance) outside a database transaction. A crash between any two steps leaves the database in an inconsistent state — e.g., balance debited but holding not updated.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { ... }) using the Drizzle transaction API so that all writes commit atomically or roll back together. The calling route handler should propagate the thrown error to return a 500.

### `server/storage.ts` — getLeaderboard
**Issue:** Leaderboard query returns userId and userName for all users with no authentication or scoping check. A guest or unauthenticated caller can enumerate student identities and scores, which is a COPPA/privacy concern for school-aged users.

**Fix:** Add a caller-identity check at the route layer before invoking getLeaderboard, and restrict returned fields to a display alias rather than the raw userId. If cross-class leaderboards are not required, add a classId filter parameter and enforce it.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 is stored directly in the job payload, meaning multi-megabyte base64-encoded PDFs and images are persisted in the jobs table for every attempt. This bloats the DB, slows queue polling queries, and risks hitting row-size limits.

**Fix:** Upload the file to private object storage before enqueuing, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), and have runExtractPaper download the buffer via objectStorageClient at runtime. Delete the object after successful extraction or on terminal failure.

### `shared/schema.ts` — budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, documentUploads tables
**Issue:** No index on userId in budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and documentUploads. Every feature query filters by userId, so full sequential scans will occur on these tables as user data grows.

**Fix:** Add a single-column index on userId for each affected table using Drizzle's index() in the table's third argument callback, e.g. for budgets: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). For billReminders also add a composite index on (userId, nextDueDate) since due-date range queries are the primary access pattern.

---

### `shared/schema.ts` — categories (userId: varchar), transactions (userId: varchar), budgets (userId: varchar), and all other tables with userId
**Issue:** All varchar('user_id') columns are declared without a length argument, defaulting to unlimited length in PostgreSQL. If the referenced users.id in the auth model is a fixed-length type (e.g. varchar(21) for NanoID or uuid), mismatched effective types degrade index performance and risk silent data issues.

**Fix:** Inspect users.id in shared/models/auth.ts and match the length exactly on every referencing column, e.g. varchar('user_id', { length: 21 }). Enforce this via a shared constant: export const USER_ID_LENGTH = 21; then use varchar('user_id', { length: USER_ID_LENGTH }) everywhere.

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth hook usage / top of AdminDashboard component
**Issue:** The file imports useAdminAuth and useLocation but the truncated code does not show a redirect guard that runs before any data-fetching queries are initiated. If the auth check is asynchronous (e.g. awaits a session validation), all the admin queries — users, audit log, observability — will fire speculatively before auth is confirmed, leaking admin endpoint calls from unauthenticated sessions and potentially exposing data in React Query's in-memory cache.

**Fix:** After calling useAdminAuth(), gate every useQuery with enabled: !!isAuthenticated so no admin API requests are issued until auth state is resolved. Add an early return that renders a loading spinner or redirects to /admin/login when isAuthenticated is false or isLoading is true, before any query hooks with side effects can execute.

## 🟡 Warnings (13)

- **`server/storage.ts` (getPortfolioHoldings):** N+1 query pattern: the function fetches portfolio holdings then issues a separate stock lookup per holding inside a loop (or via map), resulting in 1+N database round trips for a user with N holdings.
  → *Replace the per-holding stock fetch with a single JOIN query: db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId)). This collapses N+1 into a single query.*
- **`server/storage.ts` (getEnrollmentsByClass):** Fetches all enrollments for a class then resolves each student record individually in a loop (or Promise.all over per-row selects), producing N+1 queries. For a class of 35 students this is 36 round trips.
  → *Use a single JOIN: db.select().from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId)). Map the joined rows to the { enrollment, student } shape expected by callers.*
- **`server/storage.ts` (getTransactions (limit filter)):** The limit parameter inside getTransactions filters is optional with no enforced maximum. A caller omitting the limit (or passing limit=1000000) will load an unbounded result set into memory, which is a denial-of-service and memory exhaustion risk.
  → *Apply a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 50, 200); and always append .limit(safeLimit) to the query. Expose a pagination offset parameter for legitimate large fetches.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for-loop. For a 40-question paper this means 40 round-trips to the DB, each waiting on the previous one, dramatically increasing job duration and holding a DB connection the entire time.
  → *Replace the loop with a single bulk insert: filter valid questions first, map them to insert rows, then call db.insert(extractedQuestions).values(rows) once. This reduces N round-trips to 1 and is already idempotent because the delete-before-insert guard is in place.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model and token values)):** The OpenAI model name 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two separate call sites. If the model is deprecated or token limits need tuning for cost control, every call site must be found and edited manually, with no ability to change them per environment.
  → *Add OPENAI_MODEL (default 'gpt-4o') and OPENAI_MAX_TOKENS (default '4096') to environment config and reference them as process.env.OPENAI_MODEL and parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10) at both call sites.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call. An unexpected DB error (connection drop, lock timeout) will propagate as an unhandled rejection inside the job runner. Depending on the runner implementation this may silently swallow the error or crash the worker process without recording a failed job status.
  → *Wrap the db.delete call in try/catch, log the error with enough context (olderThanDays, cutoff ISO string), and rethrow so the job framework can record the failure and trigger retry logic.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The ReadStream error handler only sends a 500 if headers have not been sent, but a streaming error mid-transfer leaves the response half-written with no indication to the client that the payload is truncated. There is also no cleanup — the pipe is not destroyed on error, risking a memory/handle leak if the client disconnects.
  → *Call res.destroy() (or stream.destroy()) inside the error handler after logging, and add a 'close' listener on res to call readStream.destroy() on premature client disconnect: const readStream = file.createReadStream(); res.on('close', () => readStream.destroy()); readStream.on('error', ...).pipe(res);*
- **`shared/schema.ts` (createdAt columns in transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, learningModules):** createdAt is defined as .defaultNow() without .notNull(), making it nullable in the TypeScript type and in the database column definition. A missing defaultNow value (e.g. from a raw SQL insert or ORM bug) will silently produce NULL, breaking any ORDER BY createdAt DESC queries and audit trails.
  → *Chain .notNull() after .defaultNow() on every createdAt column: createdAt: timestamp('created_at').defaultNow().notNull(). The default ensures inserts never need to supply the value while notNull() enforces the invariant at the DB level.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** The ticker column has no unique index. If seed data or an admin insert runs twice, duplicate tickers will exist, causing portfolio aggregation queries that join on ticker to return doubled holdings or incorrect prices. There is also no index on ticker despite it being a natural lookup key.
  → *Replace the plain column declaration with a uniqueIndex in the table callback: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This enforces uniqueness at the DB level and doubles as the lookup index.*
- **`shared/schema.ts` (categories table — userId, isDefault columns):** System-wide default categories (userId IS NULL, isDefault = true) have no unique constraint preventing duplicate system categories with the same name and type. Duplicate system categories will silently appear in every user's category picker and corrupt budget rollups.
  → *Add a partial unique index on (name, type) where userId IS NULL. In raw SQL migration: CREATE UNIQUE INDEX idx_categories_system_unique ON categories(name, type) WHERE user_id IS NULL; Document this in a Drizzle custom migration since Drizzle does not natively support partial indexes in the table builder.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render inside AuditLogPanel without memoization, and the hardcoded limit of 200 rows is baked into the render body. More critically, there is no error state rendered — if the audit log query fails, the panel silently shows an empty table with no user feedback.
  → *Wrap the params construction in useMemo keyed on [actorType, action, from, to]. Extract the limit to a named constant (AUDIT_LOG_LIMIT = 200) or an env-driven config value. Add an isError branch: if (isError) return <ErrorMessage /> before the table render.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** ObservabilityCard issues a useQuery with no staleTime, refetchInterval, or error handling. On every parent re-render the component may trigger unnecessary network activity, and if the /api/admin/observability endpoint returns a non-2xx response the card silently renders all badges as 'Disabled' with no indication that the data fetch failed — an admin could incorrectly conclude Sentry is unconfigured.
  → *Add staleTime: 60_000 and refetchInterval: 60_000 to the query options so the card refreshes on a sensible cadence rather than on every render cycle. Destructure isError from useQuery and render a visible error badge or tooltip when the request fails so admins can distinguish 'disabled' from 'fetch error'.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and tab-switching handler):** TABS is defined at module level (correct), but based on the visible pattern the active-tab change handler and any mutation callbacks passed as props to tab panel components are likely defined as inline arrow functions inside the main AdminDashboard render, recreating them on every render and forcing all child panels to re-render even when unrelated state changes.
  → *Wrap every handler passed as a prop (tab change, logout, search, export) in useCallback with appropriate dependency arrays. For mutation onSuccess/onError callbacks defined inline in useMutation calls, move them outside the component or into stable useCallback references to prevent stale-closure bugs and unnecessary child re-renders.*

## 💡 Opportunities

### Parallelize independent dashboard stat queries with Promise.all — 🔥 High Impact
getDashboardStats likely runs total-income, total-expense, category-breakdown, and recent-transactions queries sequentially with awaits. Running them via Promise.all([incomeQ, expenseQ, categoryQ, recentQ]) would cut dashboard load time by roughly (N-1)*avg_query_latency — typically 150-300ms on a cold connection pool.

### Extract hardcoded virtual balance seed amount to environment config — ⚡ Medium Impact
The initial virtual balance (likely a fixed value such as 10000) is hardcoded in getVirtualBalance or seedMarketData. Move it to an environment variable VIRTUAL_BALANCE_SEED_AMOUNT with a documented default so it can be adjusted per deployment or sponsor agreement without a code change.

### Add a shared query helper for ownership-scoped deletes to eliminate repeated auth patterns — ⚡ Medium Impact
deleteTransaction, deleteBudget, deleteSavingsGoal, deleteExamPaper, and deleteClass all repeat the same pattern of filtering by both id and userId/teacherId with no shared abstraction. A generic ownerDelete(table, id, ownerField, ownerValue) helper would remove ~40 lines of duplicate code and ensure the ownership check is never accidentally omitted.

### Batch object-storage existence check before streaming — ⚡ Medium Impact
file.exists() makes a remote API call on every download request. For high-traffic export endpoints, cache the existence result in memory (or a short-TTL Redis key keyed by objectPath) for 60 seconds. This eliminates redundant GCS/S3 metadata calls and reduces p99 latency on the export download route.

### Parallelize independent setup queries inside job handlers — 🔥 High Impact
Several handlers that were truncated likely run sequential awaits for independent data fetches (e.g. fetching user + module + badge in series). Audit each handler and wrap independent reads in Promise.all. For a handler with three 50ms queries this cuts wall-clock time from ~150ms to ~50ms per job.

### Move AI extraction prompt templates to versioned config — ⚡ Medium Impact
The system prompt inside runExtractPaper is a long inline string. Storing it in a versioned config file or DB table allows prompt iteration without redeployment, enables A/B testing extraction quality, and makes token-usage auditing easier by tagging events with a prompt version ID.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract export const DEFAULT_CURRENCY = 'BSD' into a shared config file and reference it in every .default() call. This eliminates the risk of a future currency change requiring a schema migration across six tables and reduces typo risk.

### Add composite index on billReminders (userId, nextDueDate) — 🔥 High Impact
The primary query pattern for bill reminders is 'fetch all upcoming bills for user X due within the next N days', which filters on both userId and nextDueDate. A composite index idx_bill_reminders_user_due on (user_id, next_due_date) will convert this from a userId scan plus filter into a single index range scan, which is critical for users with many recurring bills.

### Replace serial primary keys with UUID or NanoID for user-facing tables — 🔥 High Impact
Serial integer PKs on transactions, savingsGoals, portfolioTransactions, and documentUploads expose sequential IDs in API responses, allowing enumeration attacks (e.g. a student guessing /api/transactions/1001 to access another user's data). Switching to gen_random_uuid() or a NanoID default eliminates this class of IDOR vulnerability without requiring application-layer changes beyond the schema and any hardcoded integer assumptions in routes.

### Parallel admin bootstrap queries via Promise.all — 🔥 High Impact
The overview tab likely fires sequential useQuery calls for organizations, schools, teachers, students, and sponsors. If any of these are fetched inside a single bootstrap effect or loader, converting them to run in parallel with Promise.all (or by ensuring all useQuery hooks are enabled simultaneously rather than chained) would reduce initial dashboard load time by the sum of sequential round-trip latencies — potentially 300-800ms on Caribbean mobile networks.

### Virtualize large admin tables to prevent DOM bloat — ⚡ Medium Impact
The audit log fetches up to 200 rows and student/teacher tabs likely render full lists. Rendering hundreds of table rows at once causes layout thrashing and slow scrolling on low-end devices common in Caribbean schools. Replacing these with @tanstack/react-virtual or a windowed table would keep DOM node count under ~50 rows regardless of dataset size.

### Centralize admin query error boundary — ⚡ Medium Impact
Each panel (AuditLogPanel, ObservabilityCard, job panels) independently handles — or silently ignores — query errors. Wrapping the tab content area in a single React ErrorBoundary with a consistent fallback UI would catch uncaught render errors from bad API shapes, and adding a shared onError handler in the QueryClient config would give admins actionable toast messages without duplicating error handling logic across a dozen panel components.
