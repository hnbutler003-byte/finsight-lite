## 🔴 Critical (9)

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (called together in trade routes)
**Issue:** Multi-step trade operations (upsert holding + create portfolio transaction + update virtual balance) are executed as separate awaits with no wrapping database transaction. A crash between any two steps leaves the DB in a partially-updated, financially inconsistent state — e.g., balance debited but holding never recorded.

**Fix:** Wrap the three operations in a single `db.transaction(async (tx) => { ... })` block using Drizzle's transaction API. Pass the transaction client `tx` to each operation so all three succeed or all roll back atomically. Expose a `executeTrade(userId, stockId, quantity, price, type)` method on the storage interface that encapsulates this transaction.

### `server/storage.ts` — getPortfolioHoldings / getEnrollmentsByClass
**Issue:** Both methods perform an N+1 query pattern: they fetch a list of rows then issue one additional SELECT per row to resolve the joined entity (stock or user). With a class of 30 students or a portfolio of 20 stocks, this produces 31 and 21 round-trips respectively, which will cause measurable latency under concurrent school usage.

**Fix:** Replace the per-row lookups with a single JOIN query using Drizzle's `.leftJoin()` or `.innerJoin()`. For `getPortfolioHoldings`: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. For `getEnrollmentsByClass`: join `classEnrollments` with `users` in one query.

### `server/routes.ts` — Google auth handler — /api/auth/google
**Issue:** No rate limiting on the Google token verification endpoint. A single client can flood this endpoint with forged or replayed idTokens, causing repeated calls to verifyGoogleToken (which hits Google's tokeninfo API) and all downstream DB lookups, creating a DoS vector against both the app and its quota with Google.

**Fix:** Add express-rate-limit before this route: `const googleAuthLimiter = rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.ip }); app.post('/api/auth/google', googleAuthLimiter, async (req, res) => {...})`. Apply a similar limiter (max: 20-30/min) to any other unauthenticated AI or auth endpoints.

### `server/routes.ts` — AI/quota endpoints — reserveQuota / getCachedExplanation usage
**Issue:** No rate limiting middleware guards the AI tutor or explanation endpoints. Any authenticated student can send unlimited requests, burning org quota and triggering OpenAI spend with no per-user or per-org request-rate cap beyond the soft quota system. The quota system tracks usage after the fact but does not throttle concurrent bursts.

**Fix:** Add a per-user rate limiter using express-rate-limit with a Redis or memory store keyed on req.session.userId: `rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip })`. Apply it to all routes that call reserveQuota or openai.* before the quota check so burst protection happens at the HTTP layer.

### `server/jobHandlers.ts` — extract-paper handler / enqueueJob call site
**Issue:** Full file content is stored as base64 in the job payload (fileB64 field). For a multi-page PDF or image this can be several megabytes per row in the jobs table, causing table bloat, slow job polling queries, and high memory usage when the worker deserializes the payload.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object path in the payload. In runExtractPaper, download the file from object storage at the start of the handler using streamPrivateObjectToResponse or a direct bucket.file().download() call. This keeps job rows small (under 1 KB) and makes retries cheaper.

### `server/jobHandlers.ts` — runExtractPaper — for loop calling storage.createExtractedQuestion
**Issue:** Extracted questions are inserted one at a time inside a sequential for loop with individual awaits. For a 40-question paper this means 40 serial round-trips to the database, multiplying latency and leaving a long window for partial failure between inserts.

**Fix:** Replace the loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(allValidQuestions) once inside a transaction that also updates the exam paper status. Wrap both in db.transaction() so a crash mid-way leaves the paper in a clean state rather than partially inserted (the idempotency delete at the top already handles retries, so this is safe).

### `shared/schema.ts` — userId: varchar('user_id') — all tables
**Issue:** All userId foreign key columns are declared as plain varchar() with no length argument. PostgreSQL varchar without length is equivalent to text, so no truncation occurs, but the FK column type must exactly match the referenced users.id type. If users.id changes size or type (e.g., UUID), all FK columns silently diverge. More critically, there is no index on userId in budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, or portfolioTransactions — every per-user data fetch for these tables will result in a full sequential scan.

**Fix:** 1) Define a shared userId column type constant: `const userIdCol = (name: string) => varchar(name, { length: 36 }).notNull()` and use it everywhere. 2) Add explicit indexes: e.g., `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })` on budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions tables.

### `server/index.ts` — res.on('finish') logging block inside the request middleware
**Issue:** Every API response body is serialised with JSON.stringify and written to stdout unconditionally. For a school-aged-user app this will log session tokens, quiz answers, user PII, and any AI response content in plain text to wherever stdout is captured (CloudWatch, Papertrail, etc.).

**Fix:** Remove `capturedJsonResponse` logging entirely from production, or gate it behind a DEBUG flag: `if (process.env.LOG_RESPONSE_BODIES === 'true' && capturedJsonResponse) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`. Never log response bodies in a production financial/edu app.

---

### `server/index.ts` — startWeeklyDigestScheduler and the equivalent monthly AI purge scheduler functions
**Issue:** Both `setTimeout` and `setInterval` return handles that are never stored. There is no graceful shutdown path (no SIGTERM/SIGINT handler), so on deploy or crash the process exits mid-interval without draining the job queue, and the handles cannot be cleared even if shutdown logic is later added.

**Fix:** Store all handles: `const digestInterval = setInterval(...); const digestTimeout = setTimeout(...)`. Add a shutdown function: `async function shutdown() { clearTimeout(digestTimeout); clearInterval(digestInterval); clearInterval(purgeInterval); await httpServer.close(); await jobWorker.stop(); process.exit(0); }` and register `process.on('SIGTERM', shutdown); process.on('SIGINT', shutdown);` before the server starts listening.

## 🟡 Warnings (18)

- **`server/storage.ts` (getLeaderboard):** The leaderboard query fetches all matching game sessions with no enforced row cap before aggregating in application memory, and the `limit` parameter passed by callers is applied after the full result set is pulled into Node. On a multi-school deployment with thousands of sessions this will cause unbounded memory growth and slow response times.
  → *Push the LIMIT and aggregation into the SQL query itself: use `db.select({ userId: gameSessions.userId, totalScore: sql<number>`sum(${gameSessions.score})`.as('total_score'), gamesPlayed: sql<number>`count(*)`.as('games_played') }).from(gameSessions).groupBy(gameSessions.userId).orderBy(desc(sql`total_score`)).limit(filters?.limit ?? 50)`. Cap the maximum allowable limit at 100 via `Math.min(filters?.limit ?? 50, 100)` to prevent callers from bypassing the guard.*
- **`server/storage.ts` (getDashboardStats):** The method issues at least three sequential `await` calls (fetch transactions, fetch budgets, fetch savings goals) that are fully independent of each other. On a slow DB connection each adds its full round-trip latency in series, making the dashboard the slowest screen in the app.
  → *Replace sequential awaits with `const [txRows, budgetRows, goalRows] = await Promise.all([fetchTransactions(userId, filters), fetchBudgets(userId), fetchSavingsGoals(userId)])`. Each sub-query has no dependency on the others, so parallelising them reduces wall-clock time to the slowest single query rather than the sum of all three.*
- **`server/storage.ts` (seedMarketData / seedLearningModules):** Stock prices, module content, and XP thresholds are hardcoded as literals inside the seed functions rather than sourced from environment config or a seed file. This means changing a stock's starting price or a module's XP reward requires a code change and redeployment, and values cannot differ between staging and production without code branches.
  → *Extract seed data into JSON files under `server/seeds/` (e.g., `marketData.json`, `learningModules.json`) and load them with `require`/`import`. For values that legitimately differ per environment (e.g., starting virtual balance), read from `process.env.VIRTUAL_BALANCE_INITIAL ?? '10000'` with a validated default. This also makes the seed data testable without running the full storage layer.*
- **`server/routes.ts` (Google auth handler — resolvedClass branch, sequential awaits):** When a classCode is provided, getOrgEnvironmentById and getOrganization are called sequentially with await even though getOrganization depends on classEnv.org_id from the first call — but there are other places in the same handler where independent lookups (e.g. resolvedClass and resolvedOrgEnv checks) are also done sequentially, adding unnecessary latency on every student login.
  → *Where lookups are truly independent, wrap them in Promise.all. For the dependent pair (classEnv → classOrg) the sequential pattern is correct, but audit the full handler for any parallel-safe pairs. Example for any independent pair: `const [resolvedClass, resolvedOrgEnv] = await Promise.all([storage.getClassByCode(code), getOrgEnvironmentByJoinCode(code)]);`*
- **`server/routes.ts` (Multer file upload handlers — videoUpload, examUpload, upload):** File extension filtering in the examUpload and upload multer configs uses path.extname(file.originalname) which is fully attacker-controlled. A file named 'malicious.pdf.exe' has extname '.exe' and is blocked, but 'malicious.exe.pdf' passes. More critically, MIME type is not cross-checked against the extension, so a renamed executable with a .pdf extension is accepted.
  → *Add MIME type validation alongside extension checks for all upload configs, mirroring what logoUpload and videoUpload already do with regex on file.mimetype. For examUpload: check that mimetype is in ['application/pdf','image/jpeg','image/png'] AND the extension matches. Never rely solely on originalname for security decisions.*
- **`server/routes.ts` (sendEmail / audit calls throughout route handlers):** Multiple route handlers call sendEmail and audit() without awaiting them or wrapping in try/catch, meaning failures are silently swallowed and—if these functions return promises—create unhandled promise rejections that can crash the Node process in newer Node versions (unhandledRejection exits by default since Node 15).
  → *Either always await these calls inside the existing try/catch block, or wrap fire-and-forget calls explicitly: `sendEmail(...).catch(err => console.error('sendEmail failed:', err))`. Never call an async function without either awaiting it or attaching a .catch() handler.*
- **`server/jobHandlers.ts` (runExtractPaper — two sequential openai.chat.completions.create calls):** The image/vision path makes one OpenAI call to extract raw text and then a second call to parse questions from that text. These are necessarily sequential, but the PDF path also flows into the second call sequentially with no timeout or circuit-breaker. If either call hangs, the job worker thread is blocked indefinitely with no way to escape short of a process restart.
  → *Wrap each OpenAI call with a Promise.race against a configurable timeout (e.g., AbortSignal.timeout(process.env.OPENAI_TIMEOUT_MS ?? 60000)). Pass the signal via the openai client's signal option. This ensures the job fails fast and is rescheduled rather than hanging the worker pool.*
- **`server/jobHandlers.ts` (runExtractPaper — max_completion_tokens: 4096):** Token limits for both OpenAI calls are hardcoded to 4096. This makes it impossible to tune limits per environment (e.g., lower in staging to control cost, higher for dense papers) without a code change and redeployment.
  → *Replace with process.env.OPENAI_MAX_TOKENS ? parseInt(process.env.OPENAI_MAX_TOKENS, 10) : 4096 and document the variable in .env.example. Apply the same pattern to the model name 'gpt-4o' which is also hardcoded.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch and returns the raw Drizzle result cast through unknown. If the delete fails (e.g., DB connection drop), the unhandled rejection will surface as an untyped crash. Additionally, rowCount is only defined on the pg driver result and will silently return 0 on other drivers, making the return value unreliable for monitoring.
  → *Wrap the delete in try/catch and log the error before rethrowing so the job framework records a meaningful failure message. Use db.delete(...).returning() or a preceding COUNT query if you need an accurate deleted-row count that is driver-agnostic.*
- **`shared/schema.ts` (budgets table definition):** budgets.createdAt, linkedCards.createdAt, savingsGoals.createdAt, documentUploads.createdAt, billReminders.createdAt all use .defaultNow() without .notNull(). This means the column is nullable in the DB schema even though application code will never intentionally store NULL — a NULL createdAt breaks any ORDER BY createdAt DESC query silently and makes audit trails unreliable.
  → *Add .notNull() after .defaultNow() on every createdAt column: `createdAt: timestamp('created_at').defaultNow().notNull()`. Apply consistently to all eight tables including portfolioHoldings.purchasedAt and portfolioTransactions.executedAt.*
- **`shared/schema.ts` (simulatedStocks table definition):** simulatedStocks has no unique constraint on ticker. Two rows with ticker 'BISX01' can coexist, which corrupts portfolio calculations (joins on stockId are safe, but any lookup-by-ticker code returns multiple rows). Also, no index on ticker means every ticker-based lookup is a sequential scan across the stocks reference table.
  → *Add a uniqueIndex on ticker: `(t) => ({ tickerUdx: uniqueIndex('udx_simulated_stocks_ticker').on(t.ticker) })` in the table's index callback. This doubles as the needed index and enforces integrity.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on nextDueDate or (userId, nextDueDate). The primary app query for this table is 'fetch upcoming bills for user sorted by due date', which without a composite index requires a sequential scan of the entire table filtered in application memory — this degrades as user count grows.
  → *Add a composite index in the table definition: `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })`. This covers both the WHERE userId = $1 filter and the ORDER BY nextDueDate sort in one B-tree scan.*
- **`shared/schema.ts` (categories table — userId FK):** categories.userId is nullable (no .notNull()) to represent system categories, which is intentional, but there is no index on userId. Any query filtering user-specific categories (WHERE user_id = $1 OR user_id IS NULL) will full-scan the categories table. Additionally, categories.isDefault has no NOT NULL constraint on the underlying column — .default(false) sets the Drizzle default but without .notNull(), the DB column remains nullable, allowing NULL isDefault values that break boolean filter logic.
  → *1) Change isDefault to `boolean('is_default').default(false).notNull()`. 2) Add a partial index: `(t) => ({ userCatIdx: index('idx_categories_user').on(t.userId) })` — PostgreSQL will efficiently use this for the per-user lookup even though the column is nullable.*
- **`server/index.ts` (module-level initSentry() call and imports interleaved with side effects):** `initSentry()` is called before all imports are resolved (imports after it: registerRoutes, serveStatic, createServer, seedDatabase, etc.). In CommonJS this works by accident, but it means Sentry is not guaranteed to wrap the module loader errors for those subsequent requires, and any crash during those imports will be untracked.
  → *Move `initSentry()` to the very first executable line before any other import side-effects, or restructure as a top-of-file call with a comment. Better: consolidate all imports first, then call `initSentry()` as the first statement in an `async main()` function, followed by `await seedDatabase()`, route registration, and server listen — making startup order explicit and testable.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The target time is computed in the server's local timezone using `new Date()` and `setHours(19, 0, 0, 0)`, which uses the local system clock timezone. On a UTC-based cloud host this will schedule at 19:00 UTC, which may be correct, but the intent is undocumented and silently wrong if the host timezone ever changes.
  → *Use an explicit UTC calculation: replace `d.setHours(19, 0, 0, 0)` with UTC equivalents (`d.setUTCHours(19, 0, 0, 0)`) and add a comment stating the intended timezone. If the digest should fire at 7 PM in a Caribbean timezone (e.g. AST, UTC-4), compute the offset explicitly or use a library like `date-fns-tz`.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render outside of useMemo, and the queryKey array does not include all params (e.g. limit is hardcoded to 200 in the URL but absent from the key). This means cache invalidation is unreliable and a new params object is allocated every render even when nothing changed.
  → *Wrap the params construction in useMemo keyed on [actorType, action, from, to] and add 'limit' to the queryKey: `const params = useMemo(() => { const p = new URLSearchParams(); ... p.set('limit', AUDIT_PAGE_LIMIT.toString()); return p; }, [actorType, action, from, to]);` where AUDIT_PAGE_LIMIT is an exported constant (or env var).*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no error state handled and no isLoading guard. If /api/admin/observability returns a non-2xx response, r.json() will silently resolve to an error body or throw an unhandled rejection, leaving the card blank with no user feedback.
  → *Add explicit error handling in the queryFn: `queryFn: async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error(await r.text()); return r.json(); }` and render an error/loading state: `if (isLoading) return <Skeleton/>; if (error) return <ErrorBanner message={error.message}/>;`*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log limit is hardcoded to 200 inside the component. Fetching up to 200 rows unconditionally on every filter change is an unbounded payload that will degrade on mobile connections common in the Caribbean region this app targets, and makes the value untestable without editing source.
  → *Define `const AUDIT_LOG_LIMIT = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 50);` in a shared config file and use server-side pagination (page/offset params) instead of a single large fetch. Add a page state variable and pass `limit` + `offset` in both the URL and the queryKey.*

## 💡 Opportunities

### Introduce a repository-level query result cache for market stock data — 🔥 High Impact
SimulatedStock rows are read-heavy and write-rarely (only during seed or admin price updates). Wrapping `getMarketStocks` with a 60-second in-process cache (e.g., a Map with a TTL timestamp) would eliminate repeated identical SELECT calls across concurrent student sessions hitting the market screen simultaneously, reducing DB load proportionally to active user count.

### Add a shared pagination utility enforced at the storage interface level — 🔥 High Impact
Methods like `getTransactions`, `getGameSessions`, and `getAllQuestions` accept an optional `limit` but do not enforce a maximum or require an `offset`/`cursor`. Defining a `PaginationParams` type (`{ limit: number; offset: number }`) with a helper that clamps limit to 1–200 and applying it uniformly prevents any route from accidentally pulling unbounded rows, which is especially important as school data volume grows.

### Centralise auth ownership checks into a storage middleware layer — ⚡ Medium Impact
Methods like `deleteTransaction`, `deleteBudget`, `deleteExamPaper`, and `updateSavingsGoal` each manually re-query the row and compare `userId` before mutating. This pattern is duplicated 10+ times and will silently break if a future developer forgets it. Extracting a `assertOwnership(table, id, userId)` helper that throws a typed `ForbiddenError` on mismatch would reduce code duplication, make auditing ownership checks trivial, and produce consistent HTTP 403 responses across all routes.

### Split monolithic routes.ts into domain-scoped router modules — 🔥 High Impact
The file is over 196k characters and combines auth, AI, uploads, leaderboard, org management, and student flows in one file. Extract each domain into its own Express Router (e.g. authRouter, aiRouter, uploadRouter) and mount them in a thin index. This reduces merge conflicts, makes rate-limiter placement explicit per domain, and cuts the blast radius of any single bug.

### Centralise multer configuration and MIME validation into a shared factory — ⚡ Medium Impact
Four separate multer instances (upload, examUpload, logoUpload, videoUpload) duplicate limit and filter logic with inconsistent security posture. A single createUploader({ allowedMimes, maxSizeMb }) factory would enforce MIME+extension cross-checking uniformly and eliminate the current inconsistency where some configs check MIME and others only check extension.

### Move hardcoded file-size limits and quota thresholds to environment config — ⚡ Medium Impact
Values like 500MB video limit, 10MB exam limit, and 5MB logo limit are scattered as numeric literals. Expose these as UPLOAD_VIDEO_MAX_MB, UPLOAD_EXAM_MAX_MB etc. in environment config with safe defaults. This allows tuning limits per deployment environment without a code change and makes them auditable in one place.

### Idempotency key on OpenAI calls — ⚡ Medium Impact
Passing a stable idempotency key (e.g., `extract-paper-${paperId}-attempt-${job.attempts}`) to OpenAI requests via the idempotencyKey option would prevent duplicate charges and duplicate responses if the HTTP request is retried by the client or a proxy. This is especially important given the existing retry logic on the job handler.

### Structured logging instead of implicit throws — ⚡ Medium Impact
Replacing bare throw new Error(...) in extraction helpers with a structured logger (e.g., pino) that includes paperId, attempt count, and truncated extractedText length would make CloudWatch/Datadog triage significantly faster without exposing raw file content in logs. Current silent catch blocks (catch { /* ignore */ }) also swallow context that would be valuable during incident response.

### Extract OpenAI prompt strings to a versioned constants file — 📌 Low Impact
Both system prompts are long inline strings inside the handler function. Moving them to a dedicated prompts.ts file with named exports and a version comment would make A/B testing prompt changes trackable in git diffs, prevent accidental edits during refactors, and allow unit tests to assert on prompt content without importing the entire handler.

### Centralise currency default as an environment-driven constant — ⚡ Medium Impact
The string 'BSD' appears hardcoded as the currency default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract to a module-level constant `const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD'` and reference it via `.default(DEFAULT_CURRENCY)`. This allows regional deployments to switch currency without schema migrations and eliminates the risk of inconsistent defaults if one table is updated and others are missed.

### Add a composite index on portfolioHoldings (userId, stockId) with a unique constraint — 🔥 High Impact
portfolioHoldings likely expects at most one holding row per user per stock (quantity is updated on new purchases via avgPurchasePrice logic). Without a unique constraint on (userId, stockId), concurrent buy operations can insert duplicate rows, silently doubling reported quantity. Adding `uniqueIndex('udx_portfolio_holdings_user_stock').on(t.userId, t.stockId)` enforces this invariant at the DB level and provides the index needed for the portfolio summary query JOIN.

### Replace text() enum columns with pgEnum for type-safe DDL — ⚡ Medium Impact
Multiple columns use text() with an inline enum option (type, status, frequency, riskLevel, period). These generate CHECK constraints in Postgres but do not create reusable enum types, so renaming a value requires a multi-step migration with no type sharing. Replacing with pgEnum (e.g., `export const transactionType = pgEnum('transaction_type', ['income','expense'])`) gives a named DB type, enables Drizzle to generate cleaner migrations, and allows the TypeScript enum to be imported directly in API layer validation without duplicating the allowed values in Zod schemas.

### Centralise startup into an async main() function — 🔥 High Impact
All top-level `await` calls (seedDatabase, registerRoutes, server listen, scheduler starts) run as module-level side effects with no single error boundary. Wrapping them in `async function main() { ... } main().catch(err => { captureError(err); process.exit(1); })` gives one place to handle fatal startup failures, makes the order of operations explicit, and makes the file unit-testable.

### Parallelise independent startup I/O with Promise.all — ⚡ Medium Impact
If seedDatabase, registerJobHandlers, and any health-check pings to external services are currently awaited sequentially, they can likely run in parallel. For example `await Promise.all([seedDatabase(), registerJobHandlers()])` could cut cold-start time by the duration of the slowest sequential operation, which matters on container restart under load.

### Replace raw console logging with a structured logger — ⚡ Medium Impact
The `log()` helper emits human-readable strings with `toLocaleTimeString`. For a deployed app, structured JSON logs (e.g. via `pino`) enable log aggregators to filter by level, correlate by request ID, and suppress DEBUG lines in production without code changes. This also makes it straightforward to redact fields like userId or email before emission.

### Split monolith into feature panel modules — 🔥 High Impact
The 126K-char single file combines 13 tabs (Overview, Organizations, Schools, Teachers, Students, Classes, Sponsors, Challenges, Reports, Jobs, Perf, Audit, DB Viewer) in one module. Splitting each tab into its own lazy-loaded component (React.lazy + Suspense) would reduce initial parse/compile time by ~80% for any given admin session, improve test isolation, and make code review tractable.

### Centralise API error handling with a shared queryFn wrapper — 🔥 High Impact
Every useQuery and useMutation in the file duplicates the pattern `apiRequest(...).then(r => r.json())` without checking r.ok. A single `adminFetch` wrapper that checks response status and throws a typed AdminApiError would eliminate ~30 silent failure points, enable a global error boundary, and reduce boilerplate by roughly 60 lines.

### Add staleTime and refetchInterval config to dashboard queries — ⚡ Medium Impact
Overview/observability queries currently use TanStack Query defaults (staleTime=0), so every tab switch triggers a network request. Setting staleTime to 60_000 for slow-changing data (org counts, school lists) and using refetchInterval only for live data (jobs, perf agent) would cut admin dashboard network traffic by an estimated 70% during normal navigation.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts is a large monolithic data-access layer with several reliability and security gaps. The most critical issues are missing database transactions on multi-step writes, N+1 query patterns in portfolio and enrollment lookups, and sequential awaits that should run in parallel. Hardcoded seed values and missing pagination guards on unbounded queries compound the risk in a school-facing production context.

### API Routes (`server/routes.ts`)
server/routes.ts is a large monolithic route file with several reliability and security gaps. The most critical issues are missing rate limiting on AI/heavy endpoints, sequential awaits that should run in parallel, and sensitive data exposure risks in logging. Overall structure is reasonable but the file has grown too large to maintain safely.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts in a loop that should be batched, and a hardcoded token limit that should be environment-configurable. The extract-paper handler stores the full file as base64 in the job payload, which will bloat the jobs table and cause memory pressure at scale. Error boundaries are inconsistently applied across handlers.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with some good practices (composite indexes on transactions, FK constraints throughout), but has several reliability and security gaps. Key concerns are missing NOT NULL constraints on audit timestamps, untyped varchar FKs risking silent truncation, and absent indexes on high-frequency query columns across multiple tables.

### Server Startup (`server/index.ts`)
The server startup code has several reliability and security gaps: schedulers start without storing their handles so they can never be cleared on shutdown, the full JSON response body is logged on every API request which will leak PII and tokens in production logs, and there is no graceful shutdown handler to drain in-flight requests or stop the job worker before the process exits. Overall structure is reasonable but these gaps are significant for a student-facing financial app.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx is a large monolithic admin page with several reliability and performance gaps. Key concerns include missing error boundaries and try/catch on async mutations, unstabilized callbacks causing unnecessary re-renders, and hardcoded pagination/limit values that belong in environment config. The file's sheer size (~126K chars) suggests it should be split into feature modules, but within what is visible the issues below are the most impactful.
