## 🔴 Critical (7)

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** N+1 query pattern: the function fetches enrollments then issues a separate db query per enrollment to hydrate each student User record, scaling linearly with class size.

**Fix:** Replace the per-row lookup with a single JOIN or an inArray fetch: `db.select().from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId))`. Return the joined rows directly instead of looping and awaiting individual user queries.

---

### `server/storage.ts` — enrollStudent / createChallenge (multi-step writes)
**Issue:** Multi-step write operations (insert enrollment + update class count, insert challenge + notify students) are not wrapped in a database transaction, so a failure mid-way leaves the database in a partially-written state with no rollback.

**Fix:** Wrap all related inserts/updates in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the `tx` client to every statement inside the block so all writes commit or roll back atomically.

---

### `server/storage.ts` — getClassProgressSummary
**Issue:** Student usernames and progress data are logged or returned without stripping sensitive fields (email, passwordHash). For a school-aged user base subject to COPPA/local data protection rules, exposing these fields in API responses or server logs is a compliance and security risk.

**Fix:** Use an explicit select projection that omits `passwordHash`, `email`, and any PII not needed by the caller: `db.select({ id: users.id, displayName: users.displayName, xp: userXp.totalXp }).from(...)`. Never select `*` when joining the users table in a response-bound query.

---

### `server/jobHandlers.ts` — runExtractPaper / extract-paper handler
**Issue:** fileB64 is stored directly in the job payload. For a scanned exam paper this is easily 5-20 MB of base64 per job row, bloating the jobs table, loading the entire buffer into worker memory on every retry, and making the payload non-idempotent across attempts since the file content is re-processed each time.

**Fix:** Before enqueuing, upload the raw file to object storage and store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of the handler. This keeps job rows small, makes retries cheap, and gives you a single source of truth for the file.

---

### `shared/schema.ts` — budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Any dashboard query filtering budgets by user — the primary access pattern — will do a full table scan as the table grows.

**Fix:** Add an index inside the pgTable call: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The composite index on (userId, categoryId) covers both the list-all-budgets and the per-category budget-check queries.

---

### `shared/schema.ts` — billReminders table definition — nextDueDate column
**Issue:** bill_reminders has no index on nextDueDate or (userId, nextDueDate). Any cron job or dashboard query fetching upcoming due bills will scan the entire table. For a school-aged user base that grows, this is the most time-sensitive query in the app.

**Fix:** Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders pgTable call. This supports both 'bills due this week for user X' and admin-level 'all bills due today' queries efficiently.

---

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel — apiRequest queryFn
**Issue:** The queryFn calls `.then(r => r.json())` without checking `r.ok`. If the server returns a 401, 403, or 500, the JSON body (likely an error object) is silently treated as valid audit log data and assigned to `rows`, causing the table to render garbage or crash on property access.

**Fix:** Add a response guard in the queryFn: `queryFn: () => apiRequest('GET', '/api/admin/audit-log?' + paramStr).then(r => { if (!r.ok) throw new Error('Audit log fetch failed: ' + r.status); return r.json(); })` — TanStack Query will then populate `isError` correctly and prevent bad data from reaching the table.

## 🟡 Warnings (12)

- **`server/storage.ts` (getLeaderboard):** No pagination guard: the leaderboard query accepts an optional `limit` but applies no maximum cap, so a caller passing limit=999999 or omitting it entirely can trigger a full-table scan across gameSessions and users, returning a payload that can exhaust server memory.
  → *Enforce a server-side cap: `const safeLimit = Math.min(filters?.limit ?? 25, 100)` and always pass `safeLimit` to `.limit()`. Never trust the caller-supplied value directly.*
- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregates (total income, total expenses, category breakdown, savings goals) run one-after-another, adding latency proportional to the number of queries. Each query is independent and can run concurrently.
  → *Replace sequential `await` calls with `const [income, expenses, categoryTotals, goals] = await Promise.all([incomeQuery, expensesQuery, categoryQuery, goalsQuery])`. This reduces total latency to the slowest single query rather than the sum of all queries.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time with sequential awaits inside a for loop. For a 40-question paper this is 40 serial round-trips to the database, each waiting for the previous to complete, adding significant latency and holding the job open longer than necessary.
  → *Build the full array of valid question objects first, then insert with a single batched db.insert(extractedQuestions).values(questionsArray) call. This reduces N round-trips to 1 and makes the operation atomic — either all questions land or none do, which also strengthens idempotency.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (both vision and extraction)):** The model name 'gpt-4o' and max_completion_tokens value 4096 are hardcoded in two separate places. If the model is deprecated or token limits need tuning for Caribbean curriculum content, every call site must be manually updated and redeployed.
  → *Read from environment variables: const AI_MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o' and const AI_MAX_TOKENS = Number(process.env.OPENAI_MAX_TOKENS ?? 4096). Apply these constants to all openai.chat.completions.create calls in the file. Validate them at startup so misconfiguration fails fast.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler has no try/catch. If the db.delete call throws (connection timeout, constraint error), the uncaught exception will propagate undecorated to the job runner. More critically, olderThanDays is used directly from job.payload in arithmetic with no type guard — a null or NaN value produces an invalid Date that silently deletes zero rows or behaves unpredictably.
  → *Wrap the handler body in try/catch and return a structured error result on failure. Add a guard: const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days <= 0) throw new Error('Invalid olderThanDays value'); before computing the cutoff.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The stream's error handler only guards against sending headers twice, but does not destroy or unpipe the stream after emitting a 500 response. If the object storage read stream errors mid-transfer after headers are already sent, the response is left open and the client receives a partial, corrupt file with a 200 status and no indication of failure.
  → *Capture the read stream in a variable, and in the error handler call stream.destroy() after checking headersSent. For the post-header case, end the response with res.end() to close the connection cleanly: const readStream = file.createReadStream(); readStream.on('error', (err) => { readStream.destroy(); if (!res.headersSent) { res.status(500).json({ message: err.message }); } else { res.end(); } }).pipe(res);*
- **`shared/schema.ts` (All tables — createdAt columns):** Every createdAt column uses .defaultNow() but omits .notNull(). defaultNow() only sets the DB default; an explicit INSERT that passes null will succeed and store a null audit timestamp, breaking any ORDER BY createdAt or time-series reporting.
  → *Chain .notNull() after .defaultNow() on every createdAt: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Also apply to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions for the same reason.*
- **`shared/schema.ts` (userId columns across all tables — varchar with no length):** All userId foreign keys are declared as varchar('user_id') with no length limit. PostgreSQL will accept this but it silently maps to text, bypassing any upstream length contract from the auth provider (e.g. Clerk/Supabase UUIDs are 36 chars). A mismatch between auth token length and stored value causes hard-to-debug silent truncation.
  → *Declare a consistent length matching your auth provider: varchar('user_id', { length: 36 }) for UUID-based providers or { length: 255 } for opaque token IDs. Apply uniformly to every table so the type contract is enforced at the DB layer.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker is declared as text with no unique constraint. Nothing prevents two rows with ticker 'BTC' or 'BISX:FIN', which would corrupt portfolio P&L calculations that join on ticker or look up current price by ticker symbol.
  → *Add a uniqueIndex: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) in the pgTable call. If multi-region/multi-currency tickers must share a symbol, scope the unique index to (ticker, region) instead.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render and built outside useMemo, but the queryKey correctly lists individual filter values. However the queryFn closes over the params object which is a new reference each render — if React batching ever causes a stale closure, the wrong params string could be sent while the queryKey reflects the new values.
  → *Move params construction inside useMemo keyed on [actorType, action, from, to] and reference that memoized string in both queryKey and queryFn: `const paramStr = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p.toString(); }, [actorType, action, from, to]);` then use `queryKey: ['/api/admin/audit-log', paramStr]` and `queryFn: () => apiRequest('GET', '/api/admin/audit-log?' + paramStr).then(r => r.json())`.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery has no isLoading or isError handling. When the /api/admin/observability endpoint is slow or fails, the component renders with all data as undefined and silently shows 'Disabled'/'Not configured' badges — indistinguishable from a real 'disabled' state. This is a false signal to administrators.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when isLoading is true, and a visible error badge/message when isError is true: `if (isLoading) return <Card ...><CardContent><Loader2 className='animate-spin' /></CardContent></Card>; if (isError) return <Card ...><CardContent className='text-red-400'>Failed to load observability status</CardContent></Card>;`*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** fmtDate accepts `any` and calls `new Date(d)` without validation. Passing an already-invalid or malformed date string (e.g. from a bad API response) returns 'Invalid Date' rendered to the UI rather than the fallback '—'.
  → *Add an isNaN guard: `function fmtDate(d: unknown): string { if (!d) return '—'; const dt = new Date(d as string); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); }` Also type the parameter as `unknown` instead of `any`.*

## 🔵 Info / Tech Debt (1)

- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log page size is hardcoded to 200. This is a magic number embedded in the query string with no pagination, meaning if the audit log grows large this single request will return hundreds of rows, bloat memory, and slow render. There is also no environment-level config for this cap.

## 💡 Opportunities

### Centralise pagination defaults as named constants — ⚡ Medium Impact
DEFAULT_PAGE_SIZE and MAX_PAGE_SIZE are implicitly re-defined (or omitted) across getLeaderboard, getTransactions, and getAllQuestions. Extract them to a single config/constants.ts file and import everywhere. This prevents the cap from being forgotten in future query additions and makes tuning a one-line change.

### Add a DataLoader layer for user hydration — 🔥 High Impact
Multiple functions (getEnrollmentsByClass, getClassProgressSummary, getPortfolioHoldings) independently re-fetch user or stock rows per result row. Introducing a per-request DataLoader (npm: dataloader) would batch and deduplicate these lookups automatically, eliminating N+1 patterns globally rather than fixing them one function at a time.

### Move all magic numbers and seed values to environment config — ⚡ Medium Impact
Initial virtual balance amounts, XP thresholds, and badge criteria appear as hardcoded literals in seedMarketData and updateUserXp. Externalising these to environment variables or a config table allows balance adjustments for different school programmes without a code deploy, which is critical for a multi-school rollout.

### Parallel AI calls for vision + extraction pipeline — ⚡ Medium Impact
If a future version of the pipeline supports pre-extracted text alongside image input, the two openai.chat.completions.create calls (vision OCR and question extraction) could be restructured so extraction begins as soon as text is available. More immediately, any independent async lookups done before or after the AI calls (e.g. fetching paper metadata, updating status) should be audited and run in parallel with Promise.all where they do not depend on each other, reducing total job wall-clock time by the latency of those DB calls.

### Centralized payload schema validation at job handler entry — 🔥 High Impact
Each handler currently accesses job.payload fields with implicit trust. Adding a lightweight Zod schema parse at the top of each registerJobHandler callback would catch malformed payloads (wrong types, missing fields) immediately with a clear error, rather than allowing them to propagate into DB queries or OpenAI calls where failure messages are obscure. This is especially important for school-aged user data where corrupt jobs could silently affect XP, badges, or progress records.

### Dead base64 buffer retention during PDF parse — ⚡ Medium Impact
In runExtractPaper, Buffer.from(fileB64, 'base64') allocates the full file buffer, which is then held in scope for the entire duration of both OpenAI API calls. Once the pdf-parse or vision step completes, the buffer is no longer needed. Explicitly setting buffer = null after text extraction and allowing GC to reclaim it would reduce peak heap pressure during concurrent job processing, which matters when multiple teachers upload exam papers simultaneously.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as .default('BSD') in at least six separate column definitions across transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and simulatedStocks. Extract const DEFAULT_CURRENCY = 'BSD' (or read from an env var for multi-territory expansion) and reference it in all defaults. When FinSight Lite expands to JMD or TTD markets, this is a single-line change instead of a six-location grep.

### Add partial index on documentUploads for processing-status polling — ⚡ Medium Impact
Background jobs polling for documents in 'processing' status will scan the full document_uploads table. A partial index — index('idx_doc_uploads_processing').on(t.userId).where(sql`status = 'processing'`) — will be tiny (only in-flight rows) and make the polling query essentially free. Expected impact is reduced DB CPU during any bulk-upload event.

### Add portfolioHoldings unique constraint on (userId, stockId) — 🔥 High Impact
Currently nothing prevents two portfolio_holdings rows for the same user and stock, which would double-count positions in any net-holdings calculation. Add uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId). The application layer should then upsert and update quantity/avgPurchasePrice in place rather than inserting a second row on repeat buys.

### Split AdminDashboard into per-tab lazy-loaded route modules — 🔥 High Impact
At 126k+ characters this is a single mega-component. Each tab (AuditLogPanel, ObservabilityCard, DB Viewer, etc.) should be a separate file loaded with React.lazy + Suspense. This reduces initial bundle size, isolates re-renders to the active tab, and makes the codebase testable per feature. Expected impact: 40-60% reduction in initial parse time on low-end Caribbean school devices.

### Centralise apiRequest response guard into a shared wrapper — 🔥 High Impact
Multiple queryFns across the file call `.then(r => r.json())` without checking `r.ok`. Create a single `fetchJson` utility — `async function fetchJson(method, url) { const r = await apiRequest(method, url); if (!r.ok) throw new Error(r.status + ' ' + url); return r.json(); }` — and replace all raw `.then(r => r.json())` calls. This eliminates the entire class of silent-failure bugs in one change.

### Add refetchInterval only to volatile queries, not all admin queries — ⚡ Medium Impact
If any queries in this dashboard use a blanket refetchInterval (common in admin dashboards), polling should be scoped only to the Background Jobs and Perf Agent tabs where freshness matters. All other tabs (audit log, students, schools) should be refetch-on-demand only. Unnecessary polling multiplies server load and can hit rate limits when multiple admin sessions are open simultaneously.
