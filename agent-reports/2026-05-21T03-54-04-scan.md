## 🔴 Critical (4)

### `server/routes.ts` — registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 96-98)
**Issue:** AI-backed routes (chat, image, audio) are registered with no rate limiting. Each call proxies to OpenAI which is both costly and latency-sensitive. A single user or bot can exhaust API quota or budget.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP via express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions.

---

### `server/jobHandlers.ts` — runExtractPaper (job.payload: fileB64)
**Issue:** Full base64-encoded exam file is stored directly in the job table payload. A single PDF or image can easily be 1–5 MB base64-encoded, bloating the jobs table, slowing job queue queries, and risking row-size limits depending on the DB backend.

**Fix:** Upload the file to private object storage before enqueuing the job. Store only the storage path (e.g. `privatePath: 'papers/42.pdf'`) in the payload. In runExtractPaper, download the file from object storage using objectStorageClient at the start of the handler, then delete it after successful extraction. This keeps job rows small and the file lifecycle explicit.

---

### `shared/schema.ts` — simulatedStocks table definition — ticker column
**Issue:** The ticker column has no unique constraint. Inserting duplicate tickers (e.g., seeding the same stock twice) will silently create duplicate rows, causing portfolio calculations to double-count prices when joined.

**Fix:** Replace text('ticker').notNull() with text('ticker').notNull().unique(), or add uniqueIndex('idx_stocks_ticker').on(t.ticker) in the table's index callback. Also add .notNull() to annualReturnPct since a stock with an unknown return should be excluded from simulation, not treated as 0% via NULL.

---

### `client/src/pages/AdminDashboard.tsx` — ObservabilityCard
**Issue:** useQuery queryFn has no error handling. If the fetch fails or r.json() throws (e.g. non-JSON 500 response), the promise rejects silently — the component renders with undefined data and no user-visible error state, making failures invisible to admins.

**Fix:** Wrap the queryFn body in try/catch and surface errors: `queryFn: async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error(await r.text()); return r.json(); }`. Add `isError` / `error` destructuring from useQuery and render an error badge in the card when present.

## 🟡 Warnings (14)

- **`server/routes.ts` (initSupabaseTables chain (~line 101-109)):** The seven Supabase init/seed functions are chained sequentially with `.then()`. If any function takes 500 ms the total cold-start delay is 7x that. Additionally there is no per-step error granularity — the single `.catch` swallows which step failed.
  → *Group independent seeds with Promise.all and log per-step failures: `await Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])`. Keep `initSupabaseTables` first if others depend on it, then chain only truly dependent steps. Wrap each in its own try/catch and log the step name.*
- **`server/routes.ts` (healthz handler (~line 61-68)):** Dynamic `import('./db')` and `import('drizzle-orm')` inside the health-check handler re-evaluate the module path resolution on every call. Under load this adds latency and in some runtimes causes redundant module cache lookups.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. The health route then just calls `await db.execute(sql\`select 1\`)` with no dynamic overhead.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 82-91)):** The filePath parameter is taken directly from the URL and passed to `objectStorage.searchPublicObject` without sanitisation. A path like `../../secret-config` could traverse outside the intended bucket prefix depending on the storage implementation.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and reject any path containing `..` segments with a 400 response before calling the storage service.*
- **`server/routes.ts` (sitemap handler (~line 36-55)):** `x-forwarded-proto` and `x-forwarded-host` headers are used to construct absolute URLs without validation. A spoofed Host header will produce a sitemap pointing to an attacker-controlled domain, enabling cache-poisoning attacks.
  → *Drive the base URL from config instead of request headers: `const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com';`. Remove reliance on forwarded headers entirely for static content like sitemaps.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop over questions with sequential awaits):** Each extracted question is inserted with a separate awaited `storage.createExtractedQuestion()` call inside a for-loop. For a 60-question exam paper this is 60 sequential round-trips to the database, adding hundreds of milliseconds of latency and holding the job worker busy unnecessarily.
  → *Collect all valid question objects into an array first, then perform a single bulk insert: `await db.insert(extractedQuestions).values(validQuestions)`. If storage.createExtractedQuestion must be used for business logic, wrap all calls with `await Promise.all(validQuestions.map(q => storage.createExtractedQuestion(q)))` to run them concurrently.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** The 30,000 character truncation limit for PDF text and the max_completion_tokens value of 4096 are hardcoded magic numbers. If the AI model or token budget changes these will silently produce truncated or failed extractions with no visibility.
  → *Move these to environment variables: `PDF_TEXT_CHAR_LIMIT` (default 30000) and `OPENAI_MAX_COMPLETION_TOKENS` (default 4096), read via `parseInt(process.env.PDF_TEXT_CHAR_LIMIT ?? '30000', 10)`. Add a startup assertion that validates they are positive integers so misconfiguration fails fast rather than silently.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete before re-extraction):** The idempotency guard deletes all previously extracted questions unconditionally before re-running extraction. If the paper has already reached `status: 'completed'` from a prior successful run (e.g. a duplicate job was enqueued by mistake), this silently wipes valid data and re-bills an OpenAI API call.
  → *Add a pre-flight check at the top of runExtractPaper: fetch the current paper status via `storage.getExamPaper(paperId)`. If status is already `'completed'`, return early with the existing questionCount without deleting or calling OpenAI. Only delete extracted questions and proceed when status is `'processing'` or `'failed'`.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — stream error handler):** The createReadStream error handler only guards against sending a second response if headers are not yet sent, but does not call `res.destroy()` or end the response after writing the 500 JSON body. If headers were already sent mid-stream, the error is silently swallowed and the client receives a truncated file with a 200 status and no indication of failure.
  → *In the error handler, after the `if (!res.headersSent)` branch, add an `else { res.destroy(err); }` path to abort the connection when the response is already in flight. Log the error in both branches using a structured logger (not console.log) so partial-stream failures are observable in production.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets(userId) or budgets(userId, categoryId). Every budget lookup for a user—which happens on every dashboard load—results in a full table scan as user count grows.
  → *Add to the budgets pgTable third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also enforces one budget per user/category, preventing silent duplicate budget rows.*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, portfolioTransactions table definitions):** No indexes on userId for billReminders, savingsGoals, portfolioHoldings, or portfolioTransactions. All four tables are queried filtered by userId in every user session, making these full scans at scale.
  → *Add a third argument to each table with index('idx_<table>_user').on(t.userId). For portfolioHoldings also add a uniqueIndex on (userId, stockId) to prevent duplicate holding rows for the same stock.*
- **`shared/schema.ts` (userId varchar columns across all tables (categories, transactions, budgets, etc.)):** All userId foreign key columns are declared as varchar() with no explicit length. In PostgreSQL an unbounded varchar is stored identically to text, but omitting the length loses the self-documenting constraint and differs from whatever max length the auth provider enforces (e.g., Clerk IDs are 32 chars). A mismatch causes silent over-length inserts or confusing FK join performance.
  → *Change every userId column to varchar('user_id', { length: 64 }) (or match your auth provider's documented ID length). This makes the schema self-documenting and ensures the column width aligns with the referenced users.id definition in models/auth.*
- **`shared/schema.ts` (createdAt timestamp columns across all tables; executedAt in portfolioTransactions, purchasedAt in portfolioHoldings):** All audit timestamp columns use .defaultNow() but none add .notNull(). An INSERT that explicitly passes NULL for createdAt will succeed and store NULL, breaking any dashboard query that orders or filters by createdAt.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g., createdAt: timestamp('created_at').defaultNow().notNull(). Apply the same to executedAt and purchasedAt.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** `new URLSearchParams()` and its `.set()` calls execute on every render of AuditLogPanel, and the resulting params string is used directly inside the queryFn closure — not in the queryKey. This means React Query cannot diff cache keys correctly when params change, risking stale data being served from cache while fresh data is fetched in the background without invalidation.
  → *Build params inside a useMemo keyed on [actorType, action, from, to] and include the serialized string in queryKey: `queryKey: ['/api/admin/audit-log', paramsString]`. This ensures cache keys are stable and invalidation is precise.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — hardcoded limit):** The audit log fetch hardcodes `limit=200` as a magic string. For a school-aged financial literacy app with multiple Caribbean schools, 200 rows could grow to tens of thousands of records, making this a silent OOM / slow-render risk as data scales.
  → *Extract to a named constant `const AUDIT_LOG_PAGE_SIZE = 200` at module scope (or better, pull from an env-backed config object). Add pagination UI controls and pass `offset` into the query so the component scales without pulling unbounded rows.*

## 🔵 Info / Tech Debt (2)

- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper):** `fmtDate` constructs a new `Intl.DateTimeFormat`-backed `toLocaleDateString` call on every invocation with no memoization. Inside tables that map over hundreds of audit rows or student records this creates a formatter object per cell per render, which is measurably slow — Intl constructors are not free.
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and icon components):** TABS is defined at module scope which is correct, but each tab entry holds a direct reference to a Lucide icon component (e.g. `Loader2` for the 'jobs' tab). If TABS were ever moved inside the component (a common refactor), these would be recreated every render. More critically, `Loader2` is semantically misleading as the icon for a 'Background Jobs' tab — it implies a loading spinner, not a job queue, which will confuse maintainers and violates the principle of least surprise.

## 💡 Opportunities

### Centralise route-level auth middleware — 🔥 High Impact
Several domain routers (students, orgs, lessons) likely repeat session/auth guard logic internally. Applying a shared `requireAuth` middleware at the `/api/` prefix level in routes.ts before the domain routers register would eliminate duplication and prevent accidental unprotected endpoints in future routers.

### Parallel domain-router registration — ⚡ Medium Impact
The six `register*Routes` calls after the parallel import are awaited sequentially. If any of them performs async setup (e.g. DB schema checks), total registration time stacks. Wrapping them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` would cut startup time proportionally.

### Structured startup logging for Supabase init — 📌 Low Impact
The current `.catch(e => console.error(...))` gives no visibility into partial failures during seeding. Adding per-step timing logs (start/end timestamps and row counts) would make production incidents diagnosable without redeploying with extra debug flags.

### Batch admin CSV export DB queries with Promise.all — ⚡ Medium Impact
The admin-csv-export handler likely issues multiple sequential DB queries to join users, transactions, and progress data. Replacing sequential awaits with Promise.all for independent queries (e.g. fetching users and fetching transactions separately before joining in JS) could cut export job duration by 40–60% for large teacher cohorts.

### Add structured logging with job ID context to all handlers — ⚡ Medium Impact
None of the handlers appear to log with the job ID attached as a structured field. Adding `{ jobId: job.id, handler: 'extract-paper' }` context to every log line would make it possible to trace a single job's lifecycle across retries in production logs, which is critical when debugging stuck 'processing' papers for school users.

### Introduce a dead-letter mechanism for extract-paper final failures — 🔥 High Impact
Currently a final-attempt failure only sets paper status to 'failed' with no notification. Enqueuing a lightweight 'notify-teacher-extraction-failed' job on terminal failure (with just the paperId and teacherId) would allow teachers to be emailed automatically rather than discovering stuck papers manually, improving reliability perception for the school audience.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' appears as a hardcoded default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, simulatedStocks, portfolioTransactions). Extracting it to a const DEFAULT_CURRENCY = 'BSD' in a shared config file means a future multi-currency expansion requires one change instead of hunting six schema lines, eliminating the risk of inconsistent defaults.

### Add partial index on transactions for recent-date queries — 🔥 High Impact
Dashboard views almost exclusively query the last 30–90 days of transactions. Adding a partial index such as index('idx_transactions_recent').on(t.userId, t.date).where(sql`date > now() - interval '90 days'`) would make the most common queries significantly faster without the overhead of indexing the entire historical dataset.

### Replace text status/period/frequency enum columns with PostgreSQL native enums — ⚡ Medium Impact
Columns like transactions.type, budgets.period, billReminders.frequency, and documentUploads.status use Drizzle's text enum, which stores unchecked strings in Postgres. Migrating these to pgEnum creates a database-level constraint that rejects invalid values even from raw SQL or migration scripts, removing a class of data-integrity bugs that Zod validation alone cannot catch post-insert.

### Parallel data fetching with Promise.all in multi-query panels — 🔥 High Impact
Overview and Reports tabs likely fire multiple sequential useQuery calls that are independent. Consolidate into parallel fetches using Promise.all inside a single queryFn, or ensure all queries are defined at the same render level so React Query can issue them concurrently. This could cut initial dashboard load time by 40-60% on slower Caribbean mobile connections.

### Extract sub-panels into separate lazy-loaded route segments — 🔥 High Impact
At 126k+ characters, AdminDashboard.tsx is a monolithic file that ships the entire admin bundle to every admin page load. Split each TABS entry (AuditLogPanel, ObservabilityCard, DB Viewer, etc.) into its own file and use React.lazy + Suspense. This reduces initial parse/compile time and makes the codebase testable in isolation.

### Add useCallback to mutation handler props passed into child components — ⚡ Medium Impact
Inline mutation handler functions passed as props (e.g. onDelete, onEdit callbacks built inside JSX map calls) create new function references on every parent render, busting React.memo on any memoized child rows in student/teacher/school tables. Wrap with useCallback keyed on the row id to prevent full-table re-renders on unrelated state changes.
