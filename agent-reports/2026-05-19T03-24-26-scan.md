## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: each PortfolioHolding row triggers a separate getStockById lookup in a loop, meaning 50 holdings = 51 database round trips.

**Fix:** Replace the loop with a single JOIN query: db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId)). Drizzle supports this directly and returns typed joined rows.

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Buy/sell stock operations perform three separate DB writes (upsert holding, insert portfolio transaction, update virtual balance) with no wrapping database transaction. A crash between steps leaves balance and holdings permanently inconsistent — critical for a financial app.

**Fix:** Wrap all three operations in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Pass the tx client to each sub-operation so they share the same atomic unit. Roll back automatically on any thrown error.

### `server/storage.ts` — getLeaderboard
**Issue:** Leaderboard query fetches all game sessions with no enforced row cap before aggregating in application memory, then joins user display names via per-row user lookups. With a growing student user base this will cause OOM and timeout under load.

**Fix:** Push aggregation to SQL: SELECT userId, SUM(score) as totalScore, COUNT(*) as gamesPlayed FROM gameSessions GROUP BY userId ORDER BY totalScore DESC LIMIT ${filters?.limit ?? 50}. Join to users table in the same query. Remove the in-memory sort and slice entirely.

### `server/routes.ts` — AI/tutor/chat heavy endpoints (registerChatRoutes, any route calling openai)
**Issue:** No rate limiting middleware is applied to AI endpoints before or after the quota reservation logic. A single authenticated student or a credential-stuffed account can fire hundreds of concurrent requests; the reserveQuota function guards token budgets but does not throttle request rate, so it can be overwhelmed with parallel reservation attempts.

**Fix:** Add express-rate-limit (or the existing quota middleware extended with a sliding-window counter) directly on the AI route group: app.use('/api/ai', rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip })). Place this before registerChatRoutes so it cannot be bypassed by route ordering.

### `server/routes.ts` — catch blocks across async route handlers (e.g. POST /api/auth/google catch block)
**Issue:** Error objects are passed directly into JSON responses via e?.message, which can leak internal stack details, Supabase connection strings, or third-party API error payloads (e.g. Google token verification failures include raw HTTP response bodies) to the client. For a school-aged user base this is both a privacy and a recon risk.

**Fix:** Create a thin safeError(e) helper that returns a sanitised message: in production (NODE_ENV === 'production') always return a generic string like 'An unexpected error occurred' and log the full error server-side via your existing audit/Sentry infrastructure. Only surface e.message in development. Apply this consistently in every catch block that feeds res.json or res.status().json.

### `server/routes.ts` — multer fileFilter callbacks (logoUpload, videoUpload, examUpload, upload)
**Issue:** File type validation relies solely on file.mimetype (for logo/video) or path.extname(file.originalname) (for upload/examUpload). Mimetype is supplied by the client and trivially spoofed; extension-only checks on originalname are also bypassable. A student could upload a disguised executable or HTML file that gets stored in object storage and potentially served back.

**Fix:** After multer accepts the buffer, add a magic-byte check using the 'file-type' package (from Buffer) before any storage write: const { fileTypeFromBuffer } = await import('file-type'); const detected = await fileTypeFromBuffer(req.file.buffer); if (!allowedMimeTypes.includes(detected?.mime)) return res.status(400).json({ message: 'Invalid file content' }). This must be enforced server-side in the route handler, not just in the multer filter.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** The full file contents are stored as a base64 string in the job payload (fileB64). For a multi-page PDF or high-res image this can easily be several MB per job row, bloating the jobs table, slowing queue polls, and potentially exceeding row-size limits depending on the DB engine.

**Fix:** Upload the file to private object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the object by path, then delete it from storage after successful extraction (or on terminal failure). This keeps job rows small (<1 KB) and moves binary data to the appropriate store.

---

### `shared/schema.ts` — budgets, savingsGoals, billReminders, linkedCards, portfolioHoldings, portfolioTransactions table definitions
**Issue:** userId columns on 6 tables have no query indexes. These tables will always be queried by userId (e.g. 'WHERE user_id = ?') but only transactions has an explicit index. At scale, every budget/goal/reminder fetch will perform a full table scan.

**Fix:** Add index definitions to each table: e.g. for budgets add a third argument `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })`. Repeat for savingsGoals, billReminders, linkedCards, portfolioHoldings, portfolioTransactions. portfolioHoldings should also index stockId for join queries.

## 🟡 Warnings (16)

- **`server/storage.ts` (getDashboardStats):** Three independent aggregation queries (total income, total expenses, category breakdown) are awaited sequentially even though they have no data dependency on each other, adding unnecessary latency on every dashboard load.
  → *Replace sequential awaits with: const [income, expenses, breakdown] = await Promise.all([incomeQuery, expensesQuery, breakdownQuery]). Each query is already independently scoped to userId and date filters so parallelisation is safe.*
- **`server/storage.ts` (getTransactions (limit filter handling)):** The limit parameter in getTransactions filters is accepted from callers but has no enforced maximum cap. A caller passing limit=1000000 will stream the full transaction history into memory, and unauthenticated or misconfigured callers passing no limit get all rows with no guard at all.
  → *Apply a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 50, 200). Always append .limit(safeLimit) to the Drizzle query regardless of whether the caller supplied the parameter. Add offset support for proper pagination.*
- **`server/routes.ts` (POST /api/auth/google):** Three sequential awaits — getClassByCode, getOrgEnvironmentById, getOrganization — run one after another on every Google sign-in. Each is an independent DB/network call with no data dependency on the prior result at the point they are fetched.
  → *After resolving the join code to either a class or an org-env, batch the follow-up lookups: const [classEnv, classOrg] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), getOrganization(envId)]) rather than three sequential awaits. This cuts sign-in latency by ~2x on cold DB connections.*
- **`server/routes.ts` (GET /healthz):** The health check endpoint imports db and drizzle-orm dynamically on every request using await import() inside the handler. Dynamic imports are cached by Node after the first call, but the async overhead and potential for import() to throw an unhandled rejection if the module fails mid-import adds unnecessary fragility to an uptime-critical route.
  → *Move the db and sql imports to the top of the file as static imports alongside the other DB imports already present (emailDb, etc.). The health check handler then simply calls await db.execute(sql`select 1`) with no dynamic import overhead.*
- **`server/jobHandlers.ts` (runExtractPaper — the for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 40-question paper this fires 40 serial round-trips to the database before the paper status is updated, adding hundreds of milliseconds of unnecessary latency and holding the job slot for longer than needed.
  → *Replace the loop with a single bulk insert: collect valid questions into an array, then call db.insert(extractedQuestions).values(rows) once. Wrap it and the subsequent updateExamPaper call in a transaction so partial inserts cannot leave the paper in an inconsistent state if the status update fails.*
- **`server/jobHandlers.ts` (runExtractPaper — vision API branch, image_url content part):** The buffer is created from fileB64 with Buffer.from(fileB64, 'base64'), then immediately re-encoded to base64 via buffer.toString('base64') to build the data URL. This is a no-op round-trip that allocates a second copy of the binary data in memory with no benefit. For large images this doubles peak memory usage inside the handler.
  → *Use the original fileB64 string directly in the data URL: `data:${mimeType};base64,${fileB64}`. Remove the Buffer allocation entirely for the image branch (keep it only for the PDF branch where pdf-parse requires a Buffer).*
- **`server/jobHandlers.ts` (admin-csv-export handler):** There is no idempotency guard on the CSV export job. If the job fails after writing a partial file to object storage and is retried, a second export run will overwrite the in-progress file or produce a duplicate, and there is no way for the UI to distinguish a stale export from a fresh one. A retry after a successful upload-but-before-status-update will silently re-export and re-upload.
  → *At the start of the handler, check whether an object already exists at the target path for this job's idempotency key (e.g. jobId or a deterministic path like exports/{type}/{jobId}.csv). If it exists and the associated record is already marked complete, return early with the existing URL. This makes retries safe without duplicating work.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation before being used in arithmetic. A payload with olderThanDays: 0 or a negative value would delete all records or produce a future cutoff date, wiping data that should be retained. There is no minimum-value guard.
  → *Add a runtime guard at the top of the handler: `if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer');`. Pair this with a minimum safe value (e.g. 30) enforced at the job-enqueue call site, or read the minimum from an environment variable like PURGE_AI_USAGE_MIN_DAYS.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions ~line 35, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt is defined as `.defaultNow()` but without `.notNull()`. Drizzle will allow explicit NULL inserts, bypassing the default. Audit queries filtering or ordering by createdAt will silently misbehave on rows with NULL.
  → *Chain `.notNull()` after `.defaultNow()` on every createdAt column, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. Same fix applies to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (userId varchar columns across all tables (e.g. categories ~line 14, transactions ~line 21)):** userId is declared as bare `varchar('user_id')` with no length argument, while the referenced `users.id` column type and length are defined in the imported auth model. If users.id is a fixed-length string (e.g. UUID = 36 chars or CUID), the FK column type should match exactly. Mismatched types can silently prevent PostgreSQL from using index scans on FK joins.
  → *Inspect users.id definition in shared/models/auth.ts. If it is a UUID, change all userId declarations to `varchar('user_id', { length: 36 })`. If it is a CUID/nanoid, use the appropriate fixed length. Apply consistently to all 8+ tables.*
- **`shared/schema.ts` (simulatedStocks table, ticker column (~line 85)):** ticker has no unique index. If seed or sync logic runs more than once, duplicate tickers will be silently inserted. Portfolio join queries on ticker will then return multiple rows, corrupting holdings calculations.
  → *Add a unique index: `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` in the table's third argument. Also consider adding `.notNull()` to issuer and region if they are required for Caribbean-specific display logic.*
- **`shared/schema.ts` (billReminders nextDueDate and frequency columns (~line 68-70)):** nextDueDate has no index despite being the primary column for upcoming-bills queries ('WHERE next_due_date <= NOW() AND user_id = ?'). Without a composite index the reminder-fetch query will scan the full table on every scheduled check or dashboard load.
  → *Add `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })` as the table's third argument. This covers both the ownership filter and the date range scan in one composite index.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed and params are re-appended on every render inside the component body, outside useMemo. This causes the queryKey reference to differ from the actual URL on concurrent renders and makes the params object allocation occur even when none of the filter state changed.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]);` and use `params.toString()` in the queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set("limit", "200")):** The audit log page size is hardcoded to 200. For a school-aged user app that could grow to many schools/events this will either silently truncate results or cause oversized payloads. There is no pagination UI rendered when results hit the limit.
  → *Move the limit to an environment-aware constant: `const AUDIT_LOG_PAGE_SIZE = import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 100;`. Add a visible 'Load more' or page control tied to an `offset` state variable so admins are never silently missing records.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — useQuery queryFn):** The queryFn arrow function `() => apiRequest('GET', '/api/admin/observability').then(r => r.json())` is defined inline with no error handling. If the fetch rejects or the server returns a non-2xx, the promise rejects silently and `data` stays undefined, leaving the card rendering blank badges with no user-visible error state.
  → *Add a dedicated error UI: destructure `isError` from useQuery and render an error badge or retry button. Wrap the queryFn: `async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error(await r.text()); return r.json(); }` so React Query receives a real rejection and can surface it.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and icon components used inline in JSX):** TABS is defined at module level (correct) but each tab entry holds a direct component reference (e.g. `LayoutDashboard`) that is spread into JSX as `<tab.icon />` inside the render loop. When the parent AdminDashboard re-renders, any inline map over TABS creates new element instances every cycle. More critically, there is no useCallback guarding the tab-switch handler, so every child tab button receives a new function reference on every parent state change, breaking shallow equality for memoized children.
  → *Wrap the tab-click handler in useCallback: `const handleTabChange = useCallback((id: string) => setActiveTab(id), []);` and pass it down. If any per-tab derived data (filtered lists, chart series) is computed inside the render, move that into useMemo keyed on the relevant data dependencies.*

## 💡 Opportunities

### Centralise virtual balance mutation behind a single audited service method — 🔥 High Impact
updateVirtualBalance is called from at least three callsites with no pre/post balance logging. Wrapping it in a dedicated ledger method that records old_balance, new_balance, reason, and timestamp in an audit table would satisfy basic fintech traceability requirements and make debugging student balance discrepancies practical.

### Extract hardcoded seed data amounts and stock prices to a config or seed file — ⚡ Medium Impact
seedMarketData and seedLearningModules embed literal prices, XP values, and module metadata directly in storage.ts. Moving these to a dedicated seeds/ directory or environment-overridable config decouples business logic from infrastructure code and allows curriculum updates without touching the storage layer.

### Add a DataLoader or per-request cache for user and stock lookups — ⚡ Medium Impact
User and SimulatedStock rows are fetched repeatedly within single request lifecycles across leaderboard, enrollment, and portfolio methods. A lightweight per-request Map cache or a DataLoader pattern would collapse duplicate fetches to a single DB hit per entity per request with zero architectural change to callers.

### Consolidate four multer instances into a configurable factory — ⚡ Medium Impact
logoUpload, videoUpload, examUpload, and upload are four nearly identical multer configurations differing only in size limits and allowed types. Extract a createUploader({ maxSize, allowedMimeTypes, allowedExtensions }) factory. This eliminates ~60 lines of duplication and ensures future security fixes (like the magic-byte check) only need to be applied in one place.

### Replace repeated sequential org/env/class DB lookups with a single batched resolver — 🔥 High Impact
The Google auth flow and likely other enrollment routes each independently call getOrganization, getOrgEnvironmentById, and getClassByCode in sequence. A resolveJoinContext(code) helper that issues these as Promise.all and caches the result per request would reduce DB round-trips by 2-3x per auth event, which matters at school bell-time concurrency spikes.

### Extract route handlers into dedicated controller files — 🔥 High Impact
With 198K+ characters in a single routes.ts, individual handlers are untestable in isolation and the file is approaching the point where Node's V8 JIT will deprioritise optimisation of rarely-hit branches. Splitting into controllers/auth.ts, controllers/ai.ts, controllers/lessons.ts etc. and keeping routes.ts as a pure registration layer will improve cold-start parse time and make unit testing of quota/auth logic practical without spinning up a full Express server.

### Centralise OpenAI model name as environment config — ⚡ Medium Impact
The string 'gpt-4o' is hardcoded in at least two openai.chat.completions.create calls inside runExtractPaper. Promoting this to an env var (OPENAI_MODEL, defaulting to 'gpt-4o') lets you switch models for cost or capability reasons across all handlers with a single config change and no redeploy of code.

### Add structured logging with job context to all handlers — ⚡ Medium Impact
Currently errors are rethrown with no structured log entry at the handler boundary, so diagnosing failures requires correlating raw stack traces against job IDs in the queue table. Adding a thin logger.error({ jobId: job.id, handler: job.type, attempt: job.attempts }, err) call before rethrowing in each catch block would make on-call triage significantly faster without changing retry behaviour.

### Stream CSV export instead of building full string in memory — 🔥 High Impact
The admin-csv-export handler appears to accumulate all rows into a single in-memory string before uploading. For large datasets (all transactions, all users) this can exhaust the Node.js heap. Replacing the approach with a streaming CSV writer (e.g. fast-csv in object mode piped through a PassThrough stream) fed into objectStorageClient's resumable upload would cap memory usage to a single chunk regardless of export size.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract `const DEFAULT_CURRENCY = 'BSD'` at the top of schema.ts and reference it in each `.default()` call. This ensures a single change point if the app expands to TTD or JMD and eliminates silent inconsistency if one table is missed.

### Add partial index on documentUploads for processing-status polling — ⚡ Medium Impact
Background jobs likely poll `WHERE status = 'processing'` repeatedly. A partial index `CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing'` would make this O(active jobs) instead of O(all uploads). In Drizzle this can be expressed via `.where(sql`status = 'processing'`)` on the index definition once Drizzle supports partial indexes, or via a raw migration.

### Replace serial primary keys with UUID or CUID on user-facing tables — 🔥 High Impact
Sequential integer PKs on transactions, savingsGoals, and portfolioTransactions expose row counts and insertion order to API consumers — a privacy concern for school-aged users and an enumeration risk. Switching to `uuid('id').defaultRandom().primaryKey()` or a CUID column eliminates both risks with negligible schema migration cost at this stage of the project.

### Parallel data fetching on dashboard overview mount — 🔥 High Impact
The overview tab likely fires several independent useQuery calls sequentially as components mount in tree order. Prefetch all overview queries in parallel using `useQueries` or `Promise.all` inside a single suspense boundary. This can cut initial dashboard load time by 40-60% on slow Caribbean mobile connections where round-trip latency dominates.

### Extract tab panels into separate lazy-loaded components — 🔥 High Impact
With 13 tabs the entire AdminDashboard bundle is parsed and executed on first load even for tabs the admin never visits. Code-split each tab panel with React.lazy + Suspense. This reduces the initial JS parse cost, which matters on low-end Android devices common in the target demographic.

### Add a React Error Boundary around each tab panel — ⚡ Medium Impact
A runtime error in the DB Viewer or Perf Agent tab currently unmounts the entire AdminDashboard. Wrapping each tab in a lightweight ErrorBoundary limits blast radius to the individual panel and keeps audit log and student management accessible during partial failures.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns across portfolio, bill reminders, and leaderboard lookups, sequential awaits that should be parallelized, and missing pagination guards on unbounded queries. The file also contains hardcoded sensitive defaults and insufficient transaction wrapping around multi-step financial operations, which is particularly risky for a fintech app serving minors.

### API Routes (`server/routes.ts`)
server/routes.ts is a large, monolithic route file with several reliability and security gaps. The most pressing issues are missing rate limiting on AI endpoints (critical for a school app with quota systems already in place), sequential awaits that should be parallelised, and sensitive data exposure risks in error responses. The codebase shows thoughtful quota and audit infrastructure that is inconsistently applied across routes.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads are stored directly in the job table, sequential DB writes in extract-paper should be batched, and the base64 image payload is double-encoded before being sent to the vision API. Error handling is mostly present but idempotency for the CSV export job is absent.

### Database Schema (`shared/schema.ts`)
The schema is structurally sound with good use of foreign keys, enums, and composite indexes on transactions. Key reliability gaps include missing NOT NULL constraints on createdAt timestamps across all tables, missing indexes on high-frequency query columns (userId on budgets, savingsGoals, billReminders), and unbounded varchar on userId foreign keys that should match the referenced column type precisely.
