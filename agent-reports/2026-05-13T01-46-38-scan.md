## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings / upsertPortfolioHolding
**Issue:** Portfolio buy/sell operations span multiple DB writes (upsertPortfolioHolding + createPortfolioTransaction + updateVirtualBalance) with no wrapping database transaction. If any step fails mid-way, the user's virtual balance and holdings will be permanently inconsistent — e.g. balance deducted but holding never created.

**Fix:** Wrap the entire trade sequence in a single `db.transaction(async (tx) => { ... })` call using Drizzle's transaction API, passing the `tx` client to each sub-operation. Expose an internal `_tradeWithTx` helper that accepts the tx client so each step can be composed atomically.

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** N+1 query pattern: the method fetches all ClassEnrollment rows then likely issues a separate `db.select().from(users).where(eq(users.id, e.studentId))` per enrollment inside a loop to populate the `student` field. For a class of 30 students this is 31 sequential queries.

**Fix:** Replace the loop with a single JOIN: `db.select().from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId))`. Map the flat result into `ClassEnrollment & { student: User }` objects in application code.

### `server/routes.ts` — POST /api/auth/google and any AI/tutor endpoint
**Issue:** No rate limiting middleware on authentication or AI endpoints. A single student client can hammer Google token verification or AI quota reservation in a tight loop, causing upstream API cost blowout and potential DoS on a school-network NAT IP.

**Fix:** Add express-rate-limit before these route handlers. For auth: max 10 requests per 15 min per IP. For AI/tutor endpoints: max 20 requests per minute per authenticated user ID (use a custom keyGenerator: (req) => req.session?.userId ?? req.ip). Install once at the top of registerRoutes so it can't be accidentally skipped on new routes.

### `server/routes.ts` — POST /api/auth/google — resolvedClass resolution block (~line 220-240)
**Issue:** Three sequential awaits — getClassByCode, getOrgEnvironmentById, getOrganization — each depend on the previous result, but getOrgEnvironmentById and getOrganization are called inside an if-chain without a try/catch. If Supabase is slow, this entire auth flow stalls with no timeout and no error boundary, leaving the promise unhandled if the connection drops mid-chain.

**Fix:** Wrap the entire classCode resolution block in its own try/catch that returns a 500 with a safe generic message. Add a Promise.race with a 5-second timeout using AbortSignal or a manual setTimeout-reject wrapper so a slow Supabase call cannot hang the login response indefinitely.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full file contents are stored as base64 in the job payload. For a multi-page PDF or high-res image this can easily exceed 5-10 MB per row in the jobs table, bloating the DB, slowing queue polling queries, and hitting any payload column size limits.

**Fix:** Upload the file to private object storage at the API layer before enqueuing, then store only the resulting object path in the job payload. In runExtractPaper, download the buffer from object storage at the start of the job and delete it on terminal success or failure. This keeps job rows small (<1 KB) regardless of file size.

---

### `shared/schema.ts` — budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, documentUploads
**Issue:** userId columns on 6+ tables have no index. Every user-scoped query (the dominant query pattern in this app) will do a full sequential scan on these tables as data grows.

**Fix:** Add a single-column index on userId for each table missing one. Example: `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })` added as the second argument to pgTable for budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and documentUploads.

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** The ticker column has no unique index. Multiple rows with the same ticker symbol (e.g., 'BISX:FIN') would be inserted silently, causing incorrect portfolio valuation when looking up current price by ticker.

**Fix:** Add a unique index: `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` as the second argument to the simulatedStocks pgTable call.

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel — queryFn
**Issue:** apiRequest response is consumed with .json() directly inside the queryFn with no try/catch and no check on response.ok. A non-2xx HTTP response (e.g. 401 session expiry, 500 server error) will call .json() on an error body or throw an unhandled rejection that surfaces as an unformatted error in the UI rather than a user-readable message.

**Fix:** Add a response guard in the queryFn: `queryFn: async () => { const r = await apiRequest('GET', `/api/admin/audit-log?${queryString}`); if (!r.ok) throw new Error(`Audit log fetch failed: ${r.status}`); return r.json(); }` and destructure `error` from useQuery to render a visible error state.

## 🟡 Warnings (15)

- **`server/storage.ts` (getDashboardStats):** Dashboard stat aggregation issues two or more independent DB queries (total income, total expenses, budget totals, savings goals) with sequential `await` calls. Each query waits for the previous to complete despite having no data dependency on each other, adding 3-4x latency on every dashboard load.
  → *Collect all independent query promises and resolve them with `const [income, expenses, budgets, goals] = await Promise.all([incomeQuery, expensesQuery, budgetsQuery, goalsQuery])`. This reduces wall-clock time to the slowest single query.*
- **`server/storage.ts` (getLeaderboard):** The leaderboard query joins against the users table to fetch `userName` and `avatar` but these fields (especially avatar URLs or display names) are logged or returned without scrubbing. For a school-aged user base under COPPA/Caribbean data protection rules, returning any PII in an unauthenticated or broadly-scoped leaderboard endpoint is a compliance risk.
  → *Add an explicit auth check at the route layer before calling `getLeaderboard`. In the storage method, select only non-sensitive identifiers (e.g. a public display name, not email). Never log the returned rows at INFO level; guard any debug logging behind a `NODE_ENV === 'development'` check.*
- **`server/storage.ts` (getLeaderboard (filters.limit)):** The `limit` parameter passed by callers is applied directly to the DB query with no upper-bound guard. A caller can pass `limit: 100000`, causing the DB to return the full users table join, exhausting memory and response buffers.
  → *Clamp the value before use: `const safeLimit = Math.min(filters?.limit ?? 20, 100)`. Default to 20, never allow more than 100. Return a 400 from the route layer if the client requests more than the cap.*
- **`server/routes.ts` (getOrganization, getOrgEnvironmentById, getOrganization (repeated inside Google auth and likely other org-resolution paths)):** getOrganization is called at least twice in close succession for the same org_id (once to enforce domain allowlist for a class join, and again when resolving the org environment). Each call is a separate round-trip to Supabase with no in-request memoisation.
  → *Introduce a lightweight per-request org cache: const orgCache = new Map<string, Organization>(); then wrap getOrganization calls: const org = orgCache.get(orgId) ?? await getOrganization(orgId); orgCache.set(orgId, org);. This is request-scoped so there is no cross-user memory leak risk.*
- **`server/routes.ts` (XLSX and Papa.parse usage (import * as XLSX and Papa)):** Both xlsx sheet parsing and Papa.parse CSV parsing are synchronous and CPU-blocking. On a large uploaded file (up to 5 MB CSV or 10 MB exam file as permitted by the multer limits) these calls will block the Node.js event loop for hundreds of milliseconds, stalling all concurrent requests — including real-time lesson interactions from other students.
  → *Move XLSX.read and Papa.parse into a worker_threads Worker or enqueue them via the existing enqueueJob mechanism already imported in this file. Return a 202 Accepted with a jobId immediately and let the client poll via the existing getJob endpoint. This matches the pattern already used elsewhere in the file.*
- **`server/routes.ts` (Error handlers throughout — e.g. catch blocks returning res.status(500).json({ message: e.message })):** Raw Error.message values from Supabase, bcrypt, and OpenAI are returned directly to the client. These can leak internal table names, Supabase project URLs, model identifiers, or stack fragments — all visible to school-aged users and easily scraped. Supabase errors in particular embed the PostgREST query path.
  → *Create a central safeErrorMessage(e: unknown): string helper that returns e.message only when NODE_ENV === 'development', otherwise returns a fixed generic string per error category (auth failure, quota exceeded, server error). Pass all catch-block errors through this helper before sending to the client. Continue to log the full error server-side via your existing audit or Sentry integration.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time with sequential awaits inside a for loop. For a 50-question paper this serialises 50 round-trips to the DB, adding hundreds of milliseconds of unnecessary latency and holding the job open longer than needed.
  → *Replace the loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(allValidQuestions) once. Add a guard so the insert is skipped when the array is empty.*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export')):** The CSV export handler has no idempotency guard. If the job is retried after a crash mid-write, it will re-query, re-build, and overwrite the object storage file silently, which is acceptable, but it will also re-run potentially expensive full-table reads and re-upload without checking whether a completed export already exists for this job run.
  → *At the start of the handler, derive a deterministic object path from the job ID (e.g. exports/csv/{job.id}/{type}.csv). Check objectStorageClient.bucket(bucket).file(object).exists() before doing any DB work. If the file already exists and job.attempts > 1, skip straight to returning the path. This makes retries free after a successful upload.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create (vision call)):** The raw base64 image string is passed directly as the image_url data URI inside the OpenAI message payload. If extractedText or the buffer is logged anywhere in the call stack (e.g. a generic error logger that serialises the full job payload), student exam paper content is exposed in plain-text logs.
  → *Ensure the logging layer redacts keys named fileB64, image_url, and content when content contains a data: URI prefix. Add a payload sanitiser in the job runner's error handler: replace any string value longer than 500 characters with '<redacted:largeValue>'. Do not log job.payload for extract-paper jobs at DEBUG level.*
- **`shared/schema.ts` (categories (userId), transactions (categoryId, documentUploadId), billReminders (categoryId)):** Nullable foreign key columns have no index. Queries filtering by categoryId or documentUploadId (e.g., 'show all transactions in category X') will full-scan the transactions table, which is the highest-volume table in the schema.
  → *Add indexes: `categoryIdx: index('idx_transactions_category').on(t.categoryId)` on transactions (separate from the composite), and `categoryIdx: index('idx_bill_reminders_category').on(t.categoryId)` on billReminders. The existing userCategoryIdx composite already covers user+category together but a standalone categoryId index is needed for admin/cross-user queries.*
- **`shared/schema.ts` (userId varchar columns across all tables):** All userId columns are declared as `varchar('user_id')` with no length argument. PostgreSQL stores unbounded varchar identically to text, but the missing length means no database-level enforcement of the auth provider's ID format (e.g., Clerk uses 32-char IDs). A bug in the auth layer could write arbitrarily long strings.
  → *Change to `varchar('user_id', { length: 64 })` across all tables to match the maximum ID length of your auth provider and add a consistent constraint. If using Clerk, 32 is the current max; use 64 for headroom.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioTransactions.executedAt, portfolioHoldings.purchasedAt):** Audit/timestamp columns use `.defaultNow()` but are missing `.notNull()`. Drizzle will type these as `Date | null` in TypeScript, requiring null-checks everywhere they are consumed despite the fact that the DB always sets them.
  → *Chain `.notNull()` after `.defaultNow()` on all timestamp audit columns: `createdAt: timestamp('created_at').defaultNow().notNull()`. This makes the column non-nullable at the DB level and removes the null from the TypeScript inferred type.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is rebuilt on every render outside of useMemo, causing the queryKey array to reference a new params.toString() string only after the object is already discarded. More critically, the params object itself is reconstructed unconditionally, and if any parent re-renders AuditLogPanel the query will re-fire even when filter state hasn't changed.
  → *Wrap the params construction and toString() call in useMemo: `const queryString = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p.toString(); }, [actorType, action, from, to]);` then use queryString directly in queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery is called with no staleTime, no error handling, and no loading state. The component silently renders empty/undefined data if the fetch fails or is still in-flight. For an admin observability card showing Sentry and health-check status, a failed fetch renders all badges as 'Disabled'/'Not configured' which is actively misleading to an admin diagnosing a production issue.
  → *Destructure isLoading and isError from useQuery. Add staleTime (e.g. 30_000). Render a skeleton or spinner when isLoading, and an explicit error badge/message when isError: `if (isLoading) return <Skeleton />; if (isError) return <p className='text-red-400'>Failed to load observability status</p>;`*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** fmtDate accepts `any` and silently constructs `new Date(undefined)` which produces 'Invalid Date' rendered as a string in the UI. This will surface as visible 'Invalid Date' text in tables for students/teachers with null timestamps, a poor experience for a school-facing app.
  → *Tighten the type and validate before formatting: `function fmtDate(d: string | number | null | undefined): string { if (d == null) return '—'; const dt = new Date(d); return isNaN(dt.getTime()) ? '—' : dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); }`*

## 💡 Opportunities

### Extract hardcoded virtual balance seed amount to environment config — ⚡ Medium Impact
The initial virtual balance given to new users (likely a hardcoded number such as 10000) should be read from `process.env.VIRTUAL_BALANCE_SEED_AMOUNT` with a typed default. This lets the product team adjust onboarding economics per school sponsor without a code deploy.

### Add a DataLoader or batched-fetch layer for user lookups — 🔥 High Impact
Multiple methods (getEnrollmentsByClass, getClassProgressSummary, leaderboard) independently query the users table by ID. Introducing a per-request DataLoader (or a simple `inArray(users.id, ids)` batch helper) would eliminate redundant round-trips and is especially impactful as class sizes grow.

### Centralise pagination defaults to prevent unbounded result sets — 🔥 High Impact
getTransactions, getAllQuestions, and getGameSessions each have optional limit/offset params but no enforced maximum. A shared `paginate(query, { limit, offset, maxLimit })` utility applied at the storage interface boundary would consistently protect all list endpoints from memory exhaustion without repeating the clamping logic in every method.

### Parallel org-context resolution with Promise.all — 🔥 High Impact
Multiple independent Supabase lookups (e.g. getOrgEnvironments, getLeaderboard, getPublishedLessons for the same org) are awaited sequentially in several route handlers. Batching independent calls with Promise.all would cut median response time by 40-60% on dashboard-load routes where 3-4 round-trips are currently serialised.

### Extract AI quota and caching logic into a dedicated middleware — 🔥 High Impact
reserveQuota, getCachedExplanation, setCachedExplanation, finalizeUsage, and releaseReservation are copy-pasted across multiple AI route handlers. Extracting these into a single composable Express middleware (e.g. withAIQuota(handler)) would eliminate duplication, make it impossible to forget quota release in a catch branch, and give one place to add future rate-limit or abuse-detection logic.

### Move multer file-type validation to magic-byte inspection — ⚡ Medium Impact
All four multer instances validate file type by mimetype string and file extension, both of which are trivially spoofed by a client. For a platform used by school-aged users, an attacker could upload an executable renamed to .csv. Add a magic-byte check using the file-type npm package on the in-memory buffer after multer collects it, before any parsing occurs.

### Parallel AI calls in extract-paper — ⚡ Medium Impact
For PDF inputs, the text extraction (pdfParse) and any pre-validation could overlap with schema preparation. More impactfully, if the extraction response from OpenAI and the idempotency delete (db.delete extractedQuestions) are both needed before insertion, the delete could run concurrently with the first OpenAI call using Promise.all, saving ~200-400 ms per job on average.

### Job payload schema validation at enqueue time — 🔥 High Impact
Adding Zod schemas for each JobPayload variant and validating at enqueueJob call sites would catch missing or malformed fields (e.g. a missing paperId or olderThanDays) before the job is persisted, preventing stuck jobs that fail on every retry with a TypeError rather than a meaningful validation message.

### Streaming CSV generation for large exports — ⚡ Medium Impact
The admin-csv-export handler appears to load full table results into memory before serialising. For tables with tens of thousands of rows this creates a large in-process memory spike per concurrent export job. Replace with a cursor/streaming query piped through a CSV transform stream written directly to object storage using a resumable upload, capping peak memory to a fixed buffer size.

### Centralise currency default into a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 columns across transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, and linkedCards. Define `export const DEFAULT_CURRENCY = 'BSD'` in a shared config file and reference it via `.default(DEFAULT_CURRENCY)`. This prevents a silent data inconsistency if the app ever needs to support TTD or XCD as a second default.

### Add partial index for unprocessed document uploads — ⚡ Medium Impact
The document processing queue will frequently query `WHERE status = 'processing'`. A partial index `CREATE INDEX idx_document_uploads_processing ON document_uploads(created_at) WHERE status = 'processing'` would make polling queries near-instant and shrink as rows are processed, keeping the index tiny. Add via Drizzle's sql template in a migration.

### Replace serial PKs with uuid on user-facing tables — 🔥 High Impact
Serial integer PKs expose record counts and enable enumeration attacks (GET /transactions/1, /2, /3). For a school-aged audience app with sensitive spending data, switching to `uuid('id').defaultRandom().primaryKey()` on transactions, savingsGoals, and documentUploads eliminates this vector and aligns with the userId type coming from the auth provider.

### Consolidate overview stat queries with Promise.all — 🔥 High Impact
If the overview/dashboard tab fires multiple sequential useQuery calls for counts (organizations, schools, teachers, students, etc.), each with its own network round-trip, replace with a single /api/admin/stats endpoint that returns all counts in one response, or batch them with Promise.all in a single queryFn. Expected impact: reduces dashboard load time by N-1 round-trips and eliminates cascading loading spinners.

### Add role-check guard at component entry point — 🔥 High Impact
useAdminAuth is imported and presumably called, but if the auth check is only enforced via redirect after render, child components (ObservabilityCard, AuditLogPanel) will still fire their useQuery calls before the redirect completes, leaking admin API requests from unauthenticated sessions. Add an explicit early return: `if (!adminUser) return null;` before any tab rendering or query-triggering children mount.

### Paginate AuditLogPanel instead of hardcoded limit=200 — ⚡ Medium Impact
The hardcoded `limit=200` in AuditLogPanel loads up to 200 rows into memory and renders them in a single pass. For a school app that may accumulate thousands of audit events, this will cause slow renders and high memory use on low-end school devices. Implement cursor-based or offset pagination with a default page size of 25-50, using the existing ChevronLeft/ChevronRight icons already imported.

The most critical issues are missing database transactions around multi-step operations, sequential awaits that block unnecessarily, and hardcoded configuration values.
