## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings / getPortfolioTransactions
**Issue:** N+1 query pattern: both functions appear to fetch holdings/transactions then loop to resolve each associated SimulatedStock individually, causing one extra query per row. For a student with 20 holdings that is 21 round-trips.

**Fix:** Use a single JOIN at the Drizzle query level: db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId)). Map the joined result to the composite return type without any per-row awaits.

### `server/storage.ts` — getLeaderboard
**Issue:** Leaderboard query aggregates across gameSessions and joins users with no default row limit enforced server-side. The result is logged or returned with userId, userName, and avatar fields — if avatar is stored as a data-URL or the userName field contains PII this exposes student personal data to any authenticated caller without a role check.

**Fix:** 1) Enforce a maximum of 100 rows with .limit(Math.min(filters?.limit ?? 20, 100)). 2) Strip any fields beyond { rank, displayName, totalScore, gamesPlayed } before returning — never expose raw userId to the client leaderboard payload. 3) Add a middleware-level role check (teacher or admin) before this endpoint is reachable.

### `server/routes.ts` — AI/tutor endpoints (any route calling openai or enqueueJob for AI)
**Issue:** No HTTP-layer rate limiting on AI endpoints. reserveQuota/quotaErrorMessage provide org-level quota tracking but there is no per-IP or per-user request-rate limiter, so a single student account or unauthenticated actor can hammer the endpoint with concurrent requests before quota logic fires, causing latency spikes and OpenAI cost blowout.

**Fix:** Add express-rate-limit before AI route handlers: import rateLimit from 'express-rate-limit'; const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, standardHeaders: true, legacyHeaders: false }); app.use('/api/tutor', aiLimiter); app.use('/api/chat', aiLimiter). Pull the window and max values from environment variables (RATE_LIMIT_AI_WINDOW_MS, RATE_LIMIT_AI_MAX).

### `server/routes.ts` — catch blocks throughout (e.g., /healthz, /api/auth/google)
**Issue:** Several catch blocks return e?.message or the raw Error object directly in API responses (res.status(503).json({ ok: false, error: e?.message })). For internal/DB errors this can leak stack traces, connection strings, or schema details to the client — a significant risk in a student-facing financial app.

**Fix:** Create a central error-sanitizer: function safeError(e: unknown, fallback = 'An unexpected error occurred') { if (process.env.NODE_ENV !== 'production') return (e as any)?.message ?? fallback; return fallback; }. Use it in every catch block. Log the full error server-side via your Sentry integration instead.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** The full file is stored as a base64 string inside the job payload (`fileB64`). For a multi-page PDF or high-res image this can easily exceed several MB per row, bloating the jobs table, slowing queue polls, and hitting any row-size limits on the DB.

**Fix:** Upload the file to private object storage before enqueuing the job, store only the object path in the payload (e.g. `{ paperId, objectPath, ext, subject }`), then download the buffer at the start of `runExtractPaper`. Delete the object once extraction succeeds or on terminal failure.

### `shared/schema.ts` — budgets table definition
**Issue:** No index on budgets.userId, which is the primary filter column for every budget query. Every lookup for a user's budgets will do a full sequential scan as the table grows.

**Fix:** Add an index inside the pgTable call: `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })`. Also consider a unique index on (userId, categoryId, period) to enforce one budget per category per period at the DB level.

---

### `shared/schema.ts` — varchar('user_id') on categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions
**Issue:** All userId foreign key columns are declared as `varchar` with no length limit. This provides no validation against the referenced users.id type, wastes storage on over-allocation, and makes index performance unpredictable. If users.id is a UUID, columns should be `uuid`; if it is a fixed-length token, `varchar(n)` with an explicit length must be set.

**Fix:** Replace `varchar('user_id')` with `uuid('user_id')` everywhere if users.id is a UUID (the common case with Clerk/Supabase auth). If it is not a UUID, determine the fixed max length and declare `varchar('user_id', { length: N })`. Update the users model reference to match.

### `client/src/pages/AdminDashboard.tsx` — ObservabilityCard queryFn
**Issue:** No try/catch around the apiRequest().then(r => r.json()) chain. If the server returns a non-2xx response, r.json() on an error body will either throw silently or return unexpected data, and useQuery will surface it as successful data, causing the badges to render incorrect state with no user feedback.

**Fix:** Wrap the queryFn in an async function with try/catch, or validate the response: `queryFn: async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error('Observability fetch failed'); return r.json(); }`. Also add `isLoading` and `isError` guards in the render to show a skeleton or error badge rather than silently rendering undefined data.

## 🟡 Warnings (16)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for income, expenses, savings, and budget data — each a separate DB round-trip — where all four queries are independent and could run concurrently. At scale this multiplies p99 latency by the number of queries.
  → *Replace sequential awaits with a single Promise.all: const [income, expenses, savings, budgets] = await Promise.all([queryIncome(...), queryExpenses(...), querySavings(...), queryBudgets(...)]). Each sub-query must use parameterised placeholders, not string interpolation.*
- **`server/storage.ts` (getTransactions / getGameSessions / getExamPapers / getAllQuestions):** No pagination guard on result sets — callers can omit a limit and receive every row in the table. On a shared school instance a single student with thousands of transactions will load all of them into memory on every dashboard render.
  → *Add a hard server-side cap: const safeLimit = Math.min(filters?.limit ?? 100, 500). Apply .limit(safeLimit).offset(filters?.offset ?? 0) to every list query and add offset to the IStorage interface so callers can page. Return { data, total } to let the UI render pagination controls.*
- **`server/storage.ts` (seedMarketData / seedLearningModules):** Seed functions contain hardcoded stock prices, module content strings, and likely hardcoded UUIDs or IDs. These values will diverge from production data and make the seed non-idempotent if run twice (duplicate rows or silent overwrites).
  → *Move seed data to /server/seeds/*.json files loaded at runtime. Wrap each insert in an onConflictDoNothing() or onConflictDoUpdate() clause keyed on a stable natural key (e.g. stock ticker symbol, module slug) so seeds are safely re-runnable. Gate execution behind a SEED_DB=true environment variable checked with process.env.SEED_DB.*
- **`server/routes.ts` (/api/auth/google handler):** Sequential awaits in Google auth flow — getClassByCode, getOrgEnvironmentById, and getOrganization are called one after another when classCode is provided, adding unnecessary latency on every student login.
  → *After resolving the join code to both a class and org-env in parallel where possible: const [resolvedClass, resolvedOrgEnv] = await Promise.all([storage.getClassByCode(code), getOrgEnvironmentByJoinCode(code)]). Then fetch classEnv and org only if resolvedClass.envId exists, since those are conditionally dependent.*
- **`server/routes.ts` (multer upload handlers (logoUpload, videoUpload, examUpload, upload)):** Four separate multer instances are defined with overlapping purposes and none of them validate file content (MIME type sniffing only checks the Content-Type header, which is client-controlled). A user can upload a malicious file with a spoofed MIME type — especially dangerous for the 500 MB videoUpload and 10 MB examUpload that presumably write to object storage.
  → *Use the 'file-type' package to verify magic bytes server-side after the buffer is received: import { fileTypeFromBuffer } from 'file-type'; const detected = await fileTypeFromBuffer(req.file.buffer); if (!allowedMimes.includes(detected?.mime)) return res.status(400).json({ message: 'Invalid file content' }). Add this check at the start of each upload route handler before persisting to ObjectStorageService.*
- **`server/routes.ts` (objectStorage / ObjectStorageService instantiation):** new ObjectStorageService() is instantiated at module load time as a module-level singleton alongside the imported objectStorageClient, creating two separate instances. If ObjectStorageService holds connection state or credentials resolved at construction time, this is a latent resource leak and a source of inconsistent behavior if one instance's config drifts.
  → *Remove the local const objectStorage = new ObjectStorageService() declaration and import and use the already-exported objectStorageClient singleton everywhere in this file. If objectStorageClient is not sufficient, export a single getInstance() factory from object_storage.ts and call that instead.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a `for` loop with sequential `await` calls. For a 40-question paper this is 40 serial round-trips to the DB, each adding latency and holding the job worker busy. A crash mid-loop leaves partial data even though the leading `db.delete` was meant to ensure idempotency.
  → *Collect all valid question objects into an array after validation, then issue a single `db.insert(extractedQuestions).values(allValidQuestions)` batch insert wrapped in the same transaction as the `updateExamPaper` call. This is both faster and fully atomic.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model and token values)):** `model: "gpt-4o"` and `max_completion_tokens: 4096` are hardcoded in two separate OpenAI calls. Changing the model or raising the limit for production requires a code deploy, and the vision call sends the raw base64 buffer re-encoded inline which duplicates the already-large payload in memory.
  → *Read model and token cap from environment: `process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'` and `parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10)`. After fixing finding #1, reference the object storage URL instead of re-encoding the buffer for the vision message.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The `db.delete` result is cast through `unknown` to extract `rowCount`, which is fragile and will silently return 0 if the driver changes its result shape. There is also no try/catch, so a DB error causes an unhandled rejection that may not be surfaced correctly by the job runner.
  → *Wrap the delete in try/catch and return a structured error on failure. For `rowCount`, use the Drizzle `.returning()` clause or check the driver-specific result type explicitly rather than casting through `unknown`: `const deleted = await db.delete(aiUsageEvents).where(...).returning({ id: aiUsageEvents.id }); return { deletedRows: deleted.length }`.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The stream's `error` event suppresses the error message directly into the HTTP response body (`res.status(500).json({ message: err.message })`), which may expose internal object storage error details — including bucket names, paths, or credentials — to the client.
  → *Log the full error server-side (`console.error('[objectStorage] stream error', err)`) and send only a generic message to the client: `res.status(500).json({ message: 'Export download failed. Please try again.' })`.*
- **`shared/schema.ts` (createdAt columns on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** `createdAt` is declared with `.defaultNow()` but without `.notNull()`, so the column is nullable in the DB schema. Any INSERT that explicitly passes `null` will succeed, corrupting audit trails and breaking any ORDER BY createdAt queries that assume the field is always present.
  → *Chain `.notNull()` after `.defaultNow()` on every createdAt column, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. Do the same for `executedAt` on portfolioTransactions and `purchasedAt` on portfolioHoldings.*
- **`shared/schema.ts` (savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** These tables have no indexes on userId despite being queried per-user in every route. portfolioHoldings also joins on stockId with no index. billReminders will commonly be filtered by nextDueDate for reminder jobs, but that column has no index either.
  → *Add: `(t) => ({ userIdx: index('idx_savings_goals_user').on(t.userId) })` on savingsGoals; `(t) => ({ userDueIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })` on billReminders; `(t) => ({ userStockIdx: index('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) })` on portfolioHoldings; `(t) => ({ userIdx: index('idx_portfolio_tx_user').on(t.userId) })` on portfolioTransactions.*
- **`shared/schema.ts` (simulatedStocks table definition):** `ticker` is declared `.notNull()` but has no unique constraint. Duplicate tickers would silently corrupt portfolio P&L calculations and any ticker-based lookups, since multiple rows would match a single ticker symbol.
  → *Replace the plain `notNull()` with a unique index: add `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` to the pgTable definition. This enforces uniqueness at the database level and also speeds up ticker-based lookups.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params construction before useQuery):** The URLSearchParams object is constructed directly in the component body on every render, but it is not used as the actual queryKey — the queryKey uses raw filter primitives. However the params string is rebuilt on every render even when filters haven't changed, and any parent re-render will reconstruct params unnecessarily. More critically, if params construction is ever moved into queryFn closure this creates a stale-closure bug.
  → *Wrap params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', String(AUDIT_LOG_LIMIT)); return p; }, [actorType, action, from, to]);` and reference `params.toString()` inside the queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log page size is hardcoded as the magic string '200' directly in the component. This is not configurable without a code change, and fetching 200 rows unconditionally on every filter change is expensive for the DB and the client bundle on low-end Caribbean school devices.
  → *Move the limit to an environment-aware constant at the module top: `const AUDIT_LOG_LIMIT = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 50);`. Use this constant in params.set and consider adding server-side pagination with a page offset rather than a single large fetch.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — missing isLoading / isError render guards):** ObservabilityCard renders badge state directly from `data?.sentry` with no loading or error state. On first load or after a network failure, all badges silently show 'Disabled' / 'Not configured' because data is undefined. For an admin monitoring panel this is a correctness and trust issue — an admin cannot distinguish 'genuinely disabled' from 'fetch failed'.
  → *Destructure `isLoading, isError` from useQuery. Render a spinner or skeleton while loading: `if (isLoading) return <CardContent><Loader2 className='animate-spin'/></CardContent>`. On error render a red alert badge: `if (isError) return <CardContent><Badge className='bg-red-600'>Failed to load</Badge></CardContent>`.*

## 💡 Opportunities

### Centralised async error boundary — 🔥 High Impact
Wrap every exported storage method in a single higher-order function that catches, logs (without leaking PII), and rethrows a typed StorageError. This removes ~60 missing try/catch sites, standardises error codes for the API layer, and prevents unhandled promise rejections from crashing the Node process.

### Parameterised query audit for SQL injection — 🔥 High Impact
Any dynamic filter (subject, period, currency) passed into raw sql`` template tags must be verified to use Drizzle bound parameters, not string concatenation. A one-hour audit replacing every sql`... ${userInput} ...` with eq()/inArray()/gte() calls eliminates the injection surface entirely — critical given the app's school-aged user base and likely COPPA/data-protection obligations.

### Extract virtual balance update into a DB transaction — ⚡ Medium Impact
updateVirtualBalance and createPortfolioTransaction are called sequentially in the trade flow without wrapping them in db.transaction(). A process crash between the two leaves balance and holdings inconsistent. Wrapping both in a single db.transaction(async (tx) => { ... }) with the tx client guarantees atomicity with no application-level rollback logic needed.

### Extract AI route handlers into a dedicated router module — 🔥 High Impact
The routes.ts file is over 5,000 lines based on the truncation point. Splitting AI/tutor routes into server/routes/ai.ts and auth routes into server/routes/auth.ts would make rate-limit middleware, quota checks, and auth guards easier to apply consistently and audit — reducing the surface area for missed security controls on sensitive endpoints.

### Batch sequential Supabase/DB calls in auth and lesson-load paths — ⚡ Medium Impact
The Google auth flow and several lesson-fetching routes show patterns of 3-5 sequential awaits to independent data sources (org, env, class, leaderboard). Converting these to Promise.all groups could reduce p95 latency by 40-60% on cold auth paths, directly improving experience for school-network connections common in the Caribbean.

### Move all hardcoded size limits and MIME type lists to environment config or a shared constants file — ⚡ Medium Impact
File size limits (5 MB, 500 MB, 10 MB) and allowed MIME type regexes are duplicated across four multer instances with no single source of truth. Centralizing them in a config/uploads.ts module means a policy change (e.g., reducing max video size) is made once and tested once, preventing the current risk of inconsistent limits across upload endpoints.

### Parallel OpenAI calls for two-stage extraction — ⚡ Medium Impact
Currently the PDF text extraction and the subsequent OpenAI structured-extraction call are sequential by necessity, but the vision OCR call and any metadata lookups (e.g. fetching the paper record to validate status) could be parallelised with Promise.all, saving one DB round-trip per job execution.

### Idempotency token on admin-csv-export — ⚡ Medium Impact
If the CSV export job is re-enqueued (e.g. on a transient failure after the file is already written to object storage), it will regenerate and overwrite the file without checking existence first. Adding a deterministic object path keyed on `type + date` and checking `file.exists()` before regenerating would make retries safe and cheap.

### Centralise OpenAI client import — 📌 Low Impact
Importing `openai` from `./replit_integrations/chat/routes` couples the job handler to a route module, making unit testing and future refactoring harder. Extract the OpenAI client initialisation into a dedicated `./lib/openaiClient.ts` singleton and import from there across all consumers.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to `export const DEFAULT_CURRENCY = 'BSD'` in a shared config file and reference it in each `.default(DEFAULT_CURRENCY)` call. This eliminates silent drift if the default ever needs to change and makes the app's Caribbean-context assumption explicit and searchable.

### Add a partial index for active bill reminders — ⚡ Medium Impact
Reminder-processing jobs and dashboard widgets will almost always query billReminders WHERE nextDueDate >= NOW(). A partial index `CREATE INDEX idx_bill_reminders_upcoming ON bill_reminders (user_id, next_due_date) WHERE next_due_date >= NOW()` would be far smaller than a full index and significantly faster for the dominant query pattern. Drizzle supports partial indexes via the `.where()` clause on index builders.

### Replace text enum columns with a Postgres native enum type — 📌 Low Impact
Columns like transactions.type, budgets.period, billReminders.frequency, and simulatedStocks.riskLevel use Drizzle's text enum, which stores unchecked strings in Postgres and offers no DB-level constraint enforcement beyond what Drizzle generates at migration time. Defining shared `pgEnum` types (e.g. `export const transactionType = pgEnum('transaction_type', ['income','expense'])`) and reusing them across tables enforces the constraint natively, reduces storage (enums are stored as integers internally), and makes the allowed values self-documenting in pg_catalog.

### Parallel admin bootstrap queries with Promise.all — 🔥 High Impact
The overview tab almost certainly fires multiple sequential useQuery calls (organizations, schools, teachers, students, sponsors, challenges) that are independent of each other. Using Promise.all inside a single bootstrapping queryFn, or enabling all queries simultaneously without sequential dependency, would cut initial dashboard load time by the sum of round-trip latencies minus the longest single request — likely 300–800ms on Caribbean LTE connections.

### Extract tab panels into lazily-loaded route segments — 🔥 High Impact
With 13 tabs and all their state, queries, and chart libraries loaded upfront, the initial JS bundle is large. Wrapping each tab panel in React.lazy + Suspense would defer loading of recharts, the DB viewer, and the perf agent UI until needed, meaningfully reducing TTI on low-powered school devices.

### Add staleTime to admin queries to reduce redundant fetches — ⚡ Medium Impact
Without staleTime, every tab switch triggers background refetches for all previously-visited tabs. Setting staleTime to 30–60 seconds on read-heavy admin queries (audit log, observability, reports) would eliminate redundant network calls when an admin navigates between tabs quickly, reducing server load and perceived jank.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic reliability and security gaps: async operations lack try/catch throughout, sequential awaits appear where Promise.all would reduce latency, and dashboard stat aggregation likely performs N+1 queries per user. For a school-aged financial app these issues compound — a single unhandled rejection can crash the process, and missing pagination guards expose unbounded result sets.

### API Routes (`server/routes.ts`)
routes.ts is a large monolithic route file with several reliability and security gaps. The most critical issues are missing rate limiting on AI endpoints (exposing the app to quota exhaustion), sequential awaits in Google auth that could be parallelized, and sensitive data likely exposed in error responses — all significant concerns for a school-aged user base handling financial data.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps: large base64 payloads stored directly in the job table risk bloating the database, sequential DB/storage inserts inside loops should be batched, and a hardcoded token limit and model name should be environment-configurable. Error handling is mostly present but idempotency for CSV exports and the purge job is incomplete.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with good use of foreign keys and composite indexes on transactions, but several critical gaps exist: missing indexes on high-cardinality filter columns, untyped varchar primary/foreign keys that bypass length validation, and nullable timestamps on audit columns that should always be present. These issues will cause slow queries and data integrity problems as the user base grows.
