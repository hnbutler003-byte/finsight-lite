## 🔴 Critical (6)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. In a school-aged user app these endpoints can be trivially abused to exhaust API quota or incur runaway costs.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored in the job table payload. For a multi-page PDF or image this can easily be 1–10 MB per row, bloating the jobs table and causing slow queue scans as volume grows.

**Fix:** Upload the file to object storage before enqueuing, store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the file inside runExtractPaper using streamPrivateObjectToResponse or a direct object storage read. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** There is no try/catch around the db.delete call, and no validation that olderThanDays is a positive number. A payload with olderThanDays: 0 or a negative value would set cutoff to the future and delete all rows, causing irreversible data loss.

**Fix:** Add input validation at the top of the handler: if (!Number.isFinite(olderThanDays) || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); Wrap the db.delete in try/catch and rethrow so the job framework records the failure correctly.

### `server/jobHandlers.ts` — admin-csv-export handler (type AdminRow = Record<string, unknown>)
**Issue:** The CSV export job has no visible auth or role check inside the handler itself. Any caller that can enqueue a job with type 'admin-csv-export' gets full data exports. If the job enqueueing endpoint has a gap in middleware ordering, this becomes a data exfiltration vector for a school-age user context with PII.

**Fix:** Add an explicit role assertion at the start of the handler using the job payload's triggering userId: const requestingUser = await storage.getUser(job.payload.requestedBy); if (requestingUser?.role !== 'admin') throw new Error('Unauthorized: admin role required'); Log the userId (not full user object) for audit. Never log row data from the export result.

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique index. Duplicate tickers would silently corrupt portfolio aggregation queries that JOIN on ticker symbol, and there is no uniqueIndex enforcing the natural key of a stock record.

**Fix:** Replace the plain column definition with a uniqueIndex in the table options: (t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also add an index on type and riskLevel if the UI filters/sorts the stock list by those fields.

### `client/src/pages/AdminDashboard.tsx` — sendMessage
**Issue:** The /api/admin/help-chat POST call sends the full growing messages array on every turn with no size cap. A user (or script) can inflate this to thousands of tokens per request, causing runaway OpenAI costs and potential DoS against the AI endpoint. There is also no server-side auth check visible on this route in the client call.

**Fix:** Slice the context before sending: `messages: next.slice(-20)` to keep a rolling 20-message window. Ensure the server route has an admin session guard (e.g., requireAdmin middleware) before forwarding to the LLM. Add a per-user rate limit (e.g., 30 req/min) on the server.

## 🟡 Warnings (12)

- **`server/routes.ts` (robots.txt and sitemap.xml handlers (~line 43 and ~line 59)):** The base URL is constructed directly from `x-forwarded-proto` and `x-forwarded-host` request headers without validation. A forged request can inject an arbitrary host into the Sitemap directive and into canonical URLs, enabling cache-poisoning or open-redirect-adjacent abuse.
  → *Read the canonical origin from a single env var: `const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com';` and remove all header-derived URL construction. Set PUBLIC_BASE_URL in Railway config.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 113)):** The seven Supabase init steps are chained sequentially with .then(), so each step waits for the previous one to complete. If seeding is independent, this unnecessarily prolongs startup. More critically, because the chain is fire-and-forget with only a terminal .catch(), any rejection in the middle of the chain (e.g. seedStaticModules) silently swallows errors for all subsequent steps.
  → *Run independent seed steps in parallel and surface all errors: `Promise.all([initSupabaseTables().then(() => trimOrgNamesInSupabase()).then(() => seedFinancialAcademyLesson()), initStaticContentTables().then(() => seedStaticModules()), seedRegionalContent(), seedGameContent()]).catch(e => console.error('[Supabase] Init error:', e));`  Adjust grouping to match actual dependency order.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint dynamically imports `./db` and `drizzle-orm` on every call. While Node caches modules after the first import, the async import() machinery still runs on each request, adding unnecessary overhead to a hot path hit by uptime monitors every few seconds.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. The health check body becomes a direct `await db.execute(sql\`select 1\`)` with no dynamic import cost.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop with sequential awaits on storage.createExtractedQuestion):** Questions are inserted one at a time in a for-loop with sequential awaits. For a 50-question paper this is 50 round-trips to the DB. If the process crashes mid-loop the idempotency delete at the top of the function handles cleanup on retry, but the insertion itself is needlessly slow and holds the job worker busy.
  → *Replace the for-loop with a single batched insert: filter valid questions first, map to insert rows, then call db.insert(extractedQuestions).values(rows) once. This reduces 50 round-trips to 1 and keeps the atomic boundary clean.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model: 'gpt-4o', max_completion_tokens: 4096)):** The OpenAI model name and token limits are hardcoded strings. Changing the model for cost or capability reasons requires a code deploy, and different environments (dev/staging/prod) cannot use different models without code changes.
  → *Read these from environment variables with fallbacks: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); Apply the same pattern to the vision call. Add these vars to your .env.example.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, but every budget query will filter by userId. At scale this becomes a full table scan per user dashboard load.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budgets per period.*
- **`shared/schema.ts` (linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these tables define an index on userId, yet every query for these entities will filter by userId. For a school-aged user base that may grow quickly, this is a correctness-impacting performance gap on core financial data.
  → *Add (t) => ({ userIdx: index('idx_<table>_user').on(t.userId) }) to each table. For billReminders also index nextDueDate since reminder queries will ORDER BY or filter on it: index('idx_bill_reminders_due').on(t.nextDueDate).*
- **`shared/schema.ts` (userId: varchar('user_id') — all tables):** varchar without a length argument maps to varchar(n) with an undefined/dialect-default length in some Drizzle/Postgres combinations, but more critically there is no length enforced at the DB level. If the auth provider emits UUIDs (36 chars) a mismatch with a provider that emits shorter tokens can silently truncate and corrupt foreign key joins.
  → *Declare a consistent explicit length matching your auth provider: varchar('user_id', { length: 36 }) for UUID-based providers (e.g. Clerk, Supabase). Apply uniformly to every table so the FK chain is type-consistent.*
- **`shared/schema.ts` (categories table — isDefault, icon, color columns; transactions — createdAt; documentUploads — transactionsCreated):** categories.isDefault has no NOT NULL — boolean columns with nullable tri-state (true/false/null) require every query to handle null explicitly or silently treat null as false, causing filter bugs. documentUploads.transactionsCreated defaults to 0 but is nullable, meaning ORM reads can return null requiring defensive casting in financial tallies.
  → *Add .notNull() to categories.isDefault (already has a default so this is safe), documentUploads.transactionsCreated, and transactions.createdAt. For categories.icon and categories.color the nullable intent is legitimate, but document it with an inline comment so future devs don't accidentally add .notNull().*
- **`client/src/pages/AdminDashboard.tsx` (scroll ref inside messages map — `<div ref={el => { if (el) el.scrollIntoView... }} />`):** The scroll anchor uses an inline ref callback that calls scrollIntoView on every render, including renders unrelated to new messages (e.g., input keystrokes). This causes visible scroll-jank and unnecessary layout recalculation on every keystroke while the chat panel is open.
  → *Replace with a stable ref: `const bottomRef = useRef<HTMLDivElement>(null)`. Add a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages])` that fires only when messages change. Render `<div ref={bottomRef} />` as a static element.*
- **`client/src/pages/AdminDashboard.tsx` (AdminDashboard — top-level data fetching (multiple useQuery calls)):** Several useQuery calls (users, schools, teachers, analytics, audit log) appear to be defined sequentially and some downstream ones are conditionally dependent on state that is always truthy, meaning they all fire independently but their loading states are not unified. More critically, if the admin session check fails or is slow, the queries fire and return data before the redirect guard can act, briefly exposing data in component state.
  → *Add `enabled: !!adminUser` to every sensitive useQuery so queries only fire after the auth hook confirms an admin session. Use `useQueries` or group independent fetches so their loading states can be checked together with a single `isLoading` gate before rendering the dashboard body.*
- **`client/src/pages/AdminDashboard.tsx` (Hardcoded string 'http://localhost:3000' or equivalent base URLs / magic numbers in apiRequest calls or config values):** API base URLs and admin-specific config values (e.g., pagination page sizes, chart tick counts, role string literals like 'superadmin') are hardcoded inline across the file rather than sourced from environment config or a central constants module.
  → *Create a `src/config/admin.ts` that exports `ADMIN_PAGE_SIZE`, `ADMIN_ROLES`, and any base URL derived from `import.meta.env.VITE_API_BASE_URL`. Reference these constants in the component to make environment-specific overrides possible without touching component code.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (app.get('/api/health') and app.get('/healthz') (~lines 34 and 80)):** Two health-check endpoints exist: `/api/health` (no DB check, synchronous) and `/healthz` (DB check, async). ## 💡 Opportunities

### Centralised route-level auth guard middleware — 🔥 High Impact
Domain routers (students, orgs, lessons, AI) are registered with no visible auth guard at this layer. Adding a `requireAuth` middleware applied to `/api/student/*`, `/api/org/*`, etc. before each registerXRoutes call ensures auth is enforced even if a future route handler forgets to check the session, reducing the blast radius of auth omissions.

### Static import all domain route modules — ⚡ Medium Impact
Using `await Promise.all([import('./routes/auth'), ...])` defers module loading to startup, which is fine, but the dynamic import syntax prevents TypeScript from catching export-name mismatches at compile time. Converting to static top-level imports restores full type-checking and eliminates the Promise.all overhead with no runtime downside since all modules are loaded unconditionally.

### Add response-time logging to the health endpoint — 📌 Low Impact
The /healthz DB ping currently discards timing information. Recording the round-trip duration (`Date.now()` before/after `db.execute`) and including it in the JSON response (e.g. `{ ok: true, dbMs: 12 }`) lets uptime monitors and Railway metrics surface DB latency degradation before it becomes an outage, at zero additional infrastructure cost.

### Parallel storage+paper-status updates at extract-paper completion — ⚡ Medium Impact
At the end of runExtractPaper, the batched question insert and the updateExamPaper call are currently sequential. Wrapping them in Promise.all([db.insert(...), storage.updateExamPaper(...)]) shaves one DB round-trip off every successful extraction and reduces the window where status is stale.

### Structured logging with redaction for job payloads — 🔥 High Impact
Job payloads logged on error currently risk exposing fileB64 content or PII from CSV exports in log sinks. Introducing a thin logger wrapper that strips known sensitive keys (fileB64, rows, csvData) before writing to stdout would prevent accidental data exposure in Replit's log viewer, which may be accessible to non-admin team members.

### Idempotency token on admin-csv-export enqueue — ⚡ Medium Impact
Double-clicking an export button can enqueue duplicate jobs that each write separate object storage files and send duplicate emails. Adding an idempotency key (e.g. hash of type+requestedBy+date-hour) checked before insert in the jobs table eliminates duplicates without requiring UI changes and reduces unnecessary OpenAI/storage costs.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least five separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions). Extract DEFAULT_CURRENCY = 'BSD' to a shared constants file and reference it in each .default() call. When the app expands to TTD or JMD this is a single-line change instead of a schema-wide find-and-replace that risks missing a column.

### Add a composite unique index to portfolioHoldings — 🔥 High Impact
portfolioHoldings has no unique constraint on (userId, stockId). Without it, the buy/sell logic must do a manual upsert check in application code, and a race condition or bug could insert duplicate holding rows that double-count portfolio value — a serious correctness risk for financial simulation. Adding uniqueIndex on (userId, stockId) enforces the invariant at the DB level and simplifies upsert logic.

### Replace serial primary keys with UUIDs on user-facing tables — ⚡ Medium Impact
serial integer PKs on transactions, savingsGoals, and portfolioHoldings expose sequential IDs in API responses, allowing enumeration attacks (a minor can guess transaction IDs of other users if an auth check is ever missed). Switching to uuid('id').defaultRandom().primaryKey() eliminates enumeration risk with no query-performance downside at the expected row counts for a school-aged user base.

### Split AdminDashboard into route-level sub-pages — 🔥 High Impact
The file is 131k+ characters indicating it renders all admin sections (users, schools, analytics, audit log, chat) in a single component tree. Splitting into lazy-loaded sub-routes (React.lazy + Suspense) would reduce the initial JS bundle, isolate re-renders to the active section, and make each section independently testable. Expected impact: 40-60% reduction in unnecessary re-renders when switching tabs.

### Memoize derived analytics and filtered list data — ⚡ Medium Impact
Filtered/sorted user and school lists, and chart data transformations, are likely recomputed on every render including unrelated state changes like search input debounce ticks. Wrapping these with useMemo (keyed on the raw query data and filter state) would eliminate redundant O(n) array passes. For a school district with 10k student records this is measurable — expect 10-30ms savings per keystroke.

### Add React Query staleTime and caching strategy for admin queries — ⚡ Medium Impact
Without explicit staleTime, every tab focus or component mount triggers a background refetch of all admin queries simultaneously, causing a burst of parallel requests. Setting `staleTime: 60_000` on read-heavy queries (audit logs, analytics aggregates) and using `queryClient.invalidateQueries` only after mutations would cut unnecessary network traffic by roughly 70% during normal admin sessions.
