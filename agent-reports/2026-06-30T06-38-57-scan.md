## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: after fetching all budgets for a user, a separate SELECT is issued per budget row to calculate `spent` (sum of transactions for that category), resulting in 1+N database round-trips on every dashboard load.

**Fix:** Replace the per-budget loop with a single aggregating JOIN: `SELECT b.*, COALESCE(SUM(t.amount),0) AS spent FROM budgets b LEFT JOIN transactions t ON t.category_id = b.category_id AND t.user_id = b.user_id AND t.type = 'expense' GROUP BY b.id`. Return the result set in one query, eliminating all N child queries.

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional `limit` but if the caller omits it there is no server-side default cap, allowing a full table scan of gameSessions joined to users to be returned unbounded. Under classroom scale (hundreds of concurrent quiz sessions) this will cause memory spikes and slow responses.

**Fix:** Apply a hard default inside the function: `const safeLimit = Math.min(filters?.limit ?? 50, 200);` and always pass `safeLimit` to the Drizzle `.limit()` call so no path can return an uncapped result set.

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 104)
**Issue:** OpenAI-backed chat, image, and audio routes have no rate limiting applied before registration. Any authenticated or unauthenticated caller can hammer these endpoints, incurring unbounded OpenAI API costs and enabling DoS against a school-aged user base.

**Fix:** Create a dedicated `aiLimiter` (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three `register*Routes` calls.

### `server/routes.ts`: robots.txt handler (~line 42) and sitemap.xml handler (~line 57)
**Issue:** `req.headers['x-forwarded-host']` and `req.headers.host` are used verbatim to construct absolute URLs without validation. An attacker can send a crafted Host header to inject an arbitrary base URL into the Sitemap directive and sitemap <loc> tags, enabling cache poisoning or phishing via poisoned search-engine indexes.

**Fix:** Read the canonical origin from a single environment variable (`process.env.APP_ORIGIN`, e.g. `https://finsightlite.com`) set at deploy time and use that instead of deriving it from headers. Remove all `x-forwarded-host`/`host` header logic from these two handlers entirely.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter / extract-paper job enqueue site)
**Issue:** Full base64-encoded file content is stored in the jobs table payload. A 5 MB PDF becomes ~6.7 MB of JSON in the DB row, bloating the jobs table, slowing queue polls, and potentially hitting row-size limits. This is an idempotency gap too: if the file is large the job row itself may fail to insert.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient instead of decoding fileB64. This keeps job rows small and makes retries safe without re-uploading.

### `shared/schema.ts`: simulatedStocks table — lastPriceUpdateDate column
**Issue:** lastPriceUpdateDate is typed as text rather than timestamp. Storing dates as text means the DB cannot enforce valid dates, range queries will sort lexicographically rather than chronologically, and timezone handling is lost. This is a correctness bug for any 'prices updated since X' query.

**Fix:** Change the column type to timestamp: lastPriceUpdateDate: timestamp('last_price_update_date'). If timezone awareness is needed use timestamp with mode:'date' or timestamptz. Migrate existing string data during the schema change.

### `server/index.ts`: res.json monkey-patch in request logger middleware
**Issue:** Every API response body is captured and serialized into the log line unconditionally via `JSON.stringify(capturedJsonResponse)`. For a financial literacy app with minors, this risks logging session tokens, grades, personal data, or AI-generated content in plaintext stdout/Sentry breadcrumbs with no truncation or field redaction.

**Fix:** Add a size guard and allowlist: only log the response body for non-sensitive paths, or truncate to 200 chars and strip known sensitive keys (e.g. token, password, sessionId) before logging. Example: `logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 200)}`;` and skip entirely for paths matching `/api/auth` or `/api/user`.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both `setInterval` handles are never stored and never cleared. On SIGTERM/SIGINT the process will attempt graceful shutdown but these intervals keep the event loop alive and may fire mid-shutdown, racing against DB connection teardown and producing spurious errors or duplicate job enqueues.

**Fix:** Capture both interval handles: `const digestTimer = setInterval(...)` and `const purgeTimer = setInterval(...)`. In the graceful shutdown handler (SIGTERM/SIGINT) call `clearInterval(digestTimer)` and `clearInterval(purgeTimer)` before closing the HTTP server and DB pool. Register shutdown with `process.on('SIGTERM', shutdown)` and `process.on('SIGINT', shutdown)`.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / admin mutation handlers
**Issue:** Mutation handlers (delete, update, create) do not re-verify admin privilege client-side before firing. If useAdminAuth returns a stale or expired session, destructive mutations are dispatched without a freshness check. While the server should enforce this, the client provides no guard and will show success UI states if the server 403 is not explicitly caught.

**Fix:** In each mutation's onMutate or before calling mutate(), check `if (!isAdmin || !adminUser) { toast({ title: 'Session expired', variant: 'destructive' }); redirect('/admin/login'); return; }`. Additionally ensure every mutation's onError handler inspects the HTTP status and redirects on 401/403.

## 🟡 Warnings (20)

- **`server/storage.ts` (getBillReminders):** N+1 query pattern: bill reminders are fetched first, then a category lookup is performed individually for each reminder that has a categoryId, producing 1+N queries identical in structure to the budget issue.
  → *Fetch reminders with a LEFT JOIN to categories in a single query: `SELECT br.*, c.* FROM bill_reminders br LEFT JOIN categories c ON c.id = br.category_id WHERE br.user_id = ?`. Map the flat rows into `BillReminder & { category }` objects in application code.*
- **`server/storage.ts` (simulateMarketPrices):** Sequential awaits inside a loop: each stock price update is awaited individually inside a for-loop, serialising all UPDATE statements. With even 20 simulated stocks this multiplies latency linearly and blocks the event loop tick between each write.
  → *Collect all update promises into an array and resolve with `await Promise.all(updates)`. If atomicity is required, wrap in a single db.transaction() and use a batch UPDATE or a single `UPDATE … CASE WHEN` statement instead.*
- **`server/storage.ts` (completeModule / updateUserXp / addUserBadge (called together in route handlers)):** Missing database transaction across multi-step writes: completing a learning module involves inserting a progress record, updating XP, and potentially inserting a badge. These three writes are issued as independent statements with no wrapping transaction, so a mid-sequence failure leaves the user with partial state (e.g. XP updated but badge missing, or progress inserted but XP stale).
  → *Wrap all three operations in `db.transaction(async (tx) => { ... })` using the Drizzle transaction API so they commit atomically or roll back together. Expose a single `completeModuleWithRewards(userId, moduleId)` storage method that encapsulates the transaction.*
- **`server/routes.ts` (Supabase init chain (~line 108)):** The seven-step Supabase init chain uses sequential `.then()` calls, so a failure in `trimOrgNamesInSupabase` silently skips all subsequent seed steps (modules, regional content, game content) with only a single catch log. Individual step failures are indistinguishable in logs, making production seed failures hard to diagnose.
  → *Run independent seed operations in parallel and log each failure individually: `await Promise.allSettled([seedFinancialAcademyLesson(), initStaticContentTables(), ...]).then(results => results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] Step ${i} failed:`, r.reason)));`. Keep `initSupabaseTables` and `trimOrgNamesInSupabase` sequential only if they are true prerequisites.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 94)):** The `filePath` wildcard parameter is passed directly to `objectStorage.searchPublicObject(filePath)` without path sanitisation. A request to `/public-objects/../../private/config` could traverse outside the intended public prefix depending on how `searchPublicObject` resolves keys.
  → *Sanitise the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` Add `import path from 'path'` at the top of the file.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop. For a 40-question paper this is 40 round-trips to the DB executed serially, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Collect all valid question objects into an array, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, replace the loop with await Promise.all(validQuestions.map((q, i) => storage.createExtractedQuestion({...}))) after validating the questions array first.*
- **`server/jobHandlers.ts` (purge-ai-usage handler (~line 73)):** The purge-ai-usage job handler has no try/catch. An unexpected DB error (connection drop, constraint violation) will surface as an unhandled promise rejection inside the job runner rather than being caught and recorded as a job failure with a proper error message, making debugging harder.
  → *Wrap the handler body in try/catch: catch the error, optionally call captureError from sentry, then rethrow so the job framework can mark the job as failed and schedule a retry. Consistent with the pattern already used in ai-health-check.*
- **`server/jobHandlers.ts` (runExtractPaper – both openai.chat.completions.create calls):** max_completion_tokens: 4096 is hardcoded in two places. For a school-facing app with cost controls this should be an environment variable so it can be tuned without a redeploy. Similarly the model string 'gpt-4o' is hardcoded and used across multiple call sites.
  → *Add OPENAI_MAX_TOKENS (default 4096) and OPENAI_MODEL (default 'gpt-4o') to environment config and reference them via process.env. Define module-level constants at the top of the file: const AI_MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o'; const MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10);*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse – res.setHeader Content-Disposition):** fileName is interpolated directly into the Content-Disposition header without sanitization: filename="${fileName}". A fileName containing a double-quote or newline can break the header or enable header injection, which matters because this function serves files to school-aged users whose teacher-uploaded filenames are not validated.
  → *Sanitize the filename before setting the header: const safeFileName = fileName.replace(/[\\"/\r\n]/g, '_'); then use filename="${safeFileName}". For full RFC 5987 compliance also emit filename*=UTF-8''${encodeURIComponent(fileName)} as a second parameter.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Budget lookups will almost certainly filter by userId (e.g. 'show my budgets'), causing sequential scans as user data grows.
  → *Add a composite index in the budgets table options: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). A single-column index on userId alone is the minimum; composite is better if category filtering is common.*
- **`shared/schema.ts` (savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** No indexes exist on userId for savingsGoals, billReminders, portfolioHoldings, or portfolioTransactions. Every user-scoped query on these tables (which is virtually all of them) will perform a full table scan.
  → *Add index options to each table: (t) => ({ userIdx: index('idx_<table>_user_id').on(t.userId) }). For portfolioHoldings also index stockId since joins from simulatedStocks are likely. Example: idx_portfolio_holdings_user_stock on (userId, stockId).*
- **`shared/schema.ts` (userId: varchar('user_id') across all tables):** All userId columns are declared as varchar with no length limit. In PostgreSQL, unbounded varchar behaves like text, but the intent is unclear and differs from the users.id definition. If users.id ever has a defined max length, mismatched varchar could cause silent issues and prevents the DB from enforcing referential length consistency.
  → *If users.id is a fixed-format ID (e.g. Clerk/Auth0 UUID or prefixed ID), specify a max length: varchar('user_id', { length: 128 }). Apply consistently across all tables to match the source column definition in the auth model.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt (and other createdAt columns)):** All createdAt columns use .defaultNow() but are not marked .notNull(). This means application code or a future migration could insert a NULL createdAt, breaking any audit trail, time-based ordering, or reporting logic that assumes createdAt is always present.
  → *Chain .notNull() after .defaultNow() on every createdAt column: createdAt: timestamp('created_at').defaultNow().notNull(). Same applies to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions.*
- **`server/index.ts` (sequential awaits in startup: seedDatabase, probeDatabase, registerRoutes, startJobWorker, registerJobHandlers):** Startup awaits operations sequentially that are at least partially independent. `probeDatabase` and any config/env validation could overlap; more critically, `seedDatabase` runs a full DB seed synchronously on the main boot path with no timeout guard, meaning a slow DB at startup blocks the HTTP server from binding for an unbounded duration.
  → *Wrap `probeDatabase` with a timeout (e.g. `Promise.race([probeDatabase(), rejectAfter(5000)])`) so a hung DB fails fast. Where safe, run independent initializers with `Promise.all`, for example: `await Promise.all([probeDatabase(), registerJobHandlers()])`. Keep `seedDatabase` sequential only if it must precede route registration, but add a timeout wrapper.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `daysSince` as `(dow + 0) % 7` — the `+ 0` is dead/no-op code suggesting a copy-paste error. More critically, all datetime math uses the server's local timezone via `new Date()` and `setHours(19, 0, 0, 0)`, so 7 PM means 7 PM in whatever timezone the Replit dyno runs, not Caribbean time (UTC-4/AST). This will silently misfire digest scheduling for Caribbean users.
  → *Remove the `+ 0`. Fix timezone by computing the target in Atlantic Standard Time explicitly: use a library like `date-fns-tz` with `zonedTimeToUtc('Sunday 19:00', 'America/Barbados')`, or compute the UTC offset manually. Store the intended digest timezone as an env var `DIGEST_TIMEZONE=America/Barbados`.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and AI_PURGE_OLDER_THAN_DAYS env var):** The purge age is validated at module load but there is no minimum floor beyond `> 0`, meaning `AI_PURGE_OLDER_THAN_DAYS=1` would purge AI usage records after 1 day, silently destroying data needed for audit trails or compliance. For a financial app serving minors, data retention limits should be enforced as a lower bound, not just an upper one.
  → *Add a minimum retention floor, e.g. `const MIN_PURGE_DAYS = 30; return Math.max(MIN_PURGE_DAYS, Number.isFinite(v) && v > 0 ? v : 180);`. Log a warning if the configured value was below the floor: `console.warn('[startup] AI_PURGE_OLDER_THAN_DAYS below minimum, using 30')`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and fires scrollIntoView on every single re-render (including while the user is typing in the input), not just when new messages arrive. This causes jank and is effectively a memory-adjacent issue because React holds a new function reference each cycle.
  → *Replace with a useRef + useEffect that only fires when messages.length or loading changes: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length, loading]);` then render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage – apiRequest POST /api/admin/help-chat):** The catch block silently swallows the error with no logging and appends a generic message. More critically, there is no abort/cleanup if the component unmounts while the fetch is in-flight, causing a setState-on-unmounted-component warning and potential stale closure issues in long requests.
  → *Create an AbortController inside sendMessage, pass its signal to apiRequest, and in a useEffect cleanup (or on component unmount) call `controller.abort()`. Wrap the setState calls after the await with a mounted-ref guard: `if (!mountedRef.current) return;`. Also log the caught error to your telemetry layer before swallowing it.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats data fetching – multiple useQuery calls):** The dashboard appears to fire several independent useQuery calls (users, schools, lessons, analytics) sequentially in component scope. Each has its own loading state but there is no aggregate loading/error boundary, so the UI can partially render with some panels empty and no indication to the user that data is still loading.
  → *Wrap the stat-card section in a combined loading check: `const isLoading = usersQuery.isLoading || schoolsQuery.isLoading || lessonsQuery.isLoading;` and render a single skeleton/spinner when `isLoading` is true. For the requests themselves, if they are independent REST calls without React Query, use `Promise.all([fetchUsers(), fetchSchools(), fetchLessons()])` to run them in parallel.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort functions passed directly in JSX (user list, school list rendering)):** Filter and sort operations on user/school arrays (likely `.filter(...searchTerm...).sort(...)`) are written as inline expressions inside JSX or component body without useMemo. For admin dashboards with hundreds of users or schools, these re-run on every keystroke in any unrelated input field (e.g., the chat input), causing measurable frame drops.
  → *Extract to useMemo: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(search.toLowerCase())).sort(...), [users, search, sortField, sortDir]);` Do the same for schools and lessons lists. This ensures recomputation only when the actual dependencies change.*

## 💡 Opportunities

### Centralised pagination defaults via query helper: 🔥 High Impact
Extract a `safePaginate(limit?: number, offset?: number, max = 200)` utility used by every list method (getTransactions, getAllQuestions, getGameSessions, getLeaderboard). This eliminates the recurring omission of pagination guards and makes auditing trivial. Expected impact: prevents unbounded queries from reaching the DB under classroom load spikes.

### Parallelise independent dashboard stat queries with Promise.all: 🔥 High Impact
getDashboardStats currently awaits total income, total expenses, savings goals, and budget summary sequentially. All four are independent reads scoped to the same userId. Running them with `Promise.all` reduces dashboard latency by up to 3× on typical cloud DB RTT (10-30 ms each), directly improving perceived performance for school-aged users on mobile connections.

### Move hardcoded virtual balance seed amount to environment config: ⚡ Medium Impact
The initial virtual balance (likely 10000 or similar) is hardcoded in getVirtualBalance / updateVirtualBalance. Moving it to `process.env.VIRTUAL_BALANCE_SEED` allows sponsors or schools to configure starting balances without a code deploy, which is a named feature requirement for the sponsor dashboard already present in the schema.

### Parallel domain-router registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import are still sequential. Since each router only mutates the Express app instance and has no inter-dependency, wrapping them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` would reduce cold-start time proportionally to the slowest individual router rather than their sum.

### Structured startup logging for init chain: ⚡ Medium Impact
Replacing `console.error` in the Supabase init catch with a structured logger (e.g. Sentry breadcrumb + severity tag) would surface seed failures in the existing Sentry pipeline already imported below. Currently seed failures are log-only and produce no alert, which is high-risk for a production educational platform where missing game or lesson content directly affects users.

### Object-storage response caching headers: ⚡ Medium Impact
The `/public-objects/` handler passes a 30-day TTL to `downloadObject` but sets no `Cache-Control` header visible to intermediate CDN/proxies on the Express response itself. Adding `res.setHeader('Cache-Control', 'public, max-age=2592000, immutable')` before streaming would allow Railway's or Cloudflare's edge to cache logo and video assets, reducing origin bandwidth and latency for Caribbean school users on constrained connections.

### Batch idempotency delete + status update in one transaction: 🔥 High Impact
runExtractPaper currently does a delete of existing questions and later an updateExamPaper status as separate awaits with no transaction wrapper. If the process crashes between them the paper row and questions table are inconsistent. Wrapping both in a db.transaction() call guarantees atomicity and eliminates the inconsistent-state window, improving retry reliability.

### Stream PDF text extraction instead of slicing at 30 000 chars: ⚡ Medium Impact
extractedText = pdfData.text.slice(0, 30000) silently drops content from longer papers, causing missed questions with no warning logged. Chunking the text and either summarising or processing in segments, or at minimum emitting a structured warning into the job result when truncation occurs, would prevent silent data loss for Caribbean CXC/CSEC past papers which can be lengthy.

### Centralise OpenAI client import to avoid coupling to chat routes: ⚡ Medium Impact
openai is imported from ./replit_integrations/chat/routes, meaning a change to that module's export breaks every job handler. Export the openai client from a dedicated singleton module (e.g. ./lib/openaiClient.ts) and import from there. This also makes it straightforward to inject a mock client in tests.

### Unique index on simulatedStocks.ticker: 🔥 High Impact
ticker should be unique to prevent duplicate stock entries that would corrupt portfolio calculations. Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) in the table options. This also gives O(log n) lookup when seeding or updating prices by ticker symbol.

### Partial index for active bill reminders: ⚡ Medium Impact
Bill reminder queries almost certainly filter on nextDueDate for upcoming bills. A partial or composite index on (userId, nextDueDate) would eliminate full scans on what could be a large table. Since this is a school-aged user app with gamification, reminder queries likely run on every dashboard load.

### Centralise currency default into a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 7 separate column definitions across transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and simulatedStocks. Extract to a module-level constant DEFAULT_CURRENCY = 'BSD' and reference it in each .default() call. This prevents a future currency change requiring 7+ schema edits and a multi-table migration.

### Centralized graceful shutdown orchestrator: 🔥 High Impact
Introduce a single `shutdown(signal)` async function registered for SIGTERM, SIGINT, and uncaughtException that sequentially clears intervals, stops job workers, closes the HTTP server with `server.close()`, then closes the DB pool. Currently teardown is ad-hoc or absent, risking job duplication and Postgres connection exhaustion on dyno restarts.

### Structured JSON logging with log levels: ⚡ Medium Impact
Replace all `console.log`/`log()` calls with a structured logger (e.g. `pino`) that emits JSON with fields for level, source, requestId, and duration. This eliminates the ad-hoc string concatenation that currently makes redacting sensitive fields impossible, and makes log aggregation in Replit or an external sink (Datadog, Logtail) practical.

### Extract scheduler logic into a dedicated module: ⚡ Medium Impact
Move `maybeRunWeeklyDigest`, `maybeRunAiUsagePurge`, and their `setInterval` registrations into a `scheduler.ts` module that exports `startSchedulers()` and `stopSchedulers()`. This makes unit testing the deduplication logic straightforward and keeps `index.ts` as a thin composition root, reducing the risk of future copy-paste errors in timing logic.

### Centralised admin session expiry handler: 🔥 High Impact
Add a single Axios/fetch response interceptor (or a React Query global onError) that catches 401/403 responses from any /api/admin/* endpoint and redirects to /admin/login with a toast. This eliminates the need to handle auth errors in every mutation and query individually and prevents partial UI renders under expired sessions.

### Virtualize long admin lists with TanStack Virtual: ⚡ Medium Impact
User, school, and lesson tables that paginate client-side still mount all visible-page rows as real DOM nodes. Replacing with @tanstack/react-virtual row virtualization would cut initial paint time and memory use significantly for admins managing institutions with 500+ records.

### Extract AdminHelpChat to a lazy-loaded chunk: ⚡ Medium Impact
AdminHelpChat imports Bot, MessageCircle, Loader2 icons and holds its own state machine. Wrapping it in React.lazy + Suspense means the chat bundle is only fetched when the admin first clicks Help, reducing the initial AdminDashboard JS parse cost by the weight of the chat subtree.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that should be parallelised. For a school-aged financial literacy app, the absence of pagination guards on leaderboard and question queries is a reliability risk under classroom-scale load.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good patterns like parallel module imports and rate limiting on auth endpoints. Key concerns are unprotected AI/chat/audio routes with no rate limiting, trust of unvalidated host headers in SEO routes that could enable cache poisoning, and sequential Supabase seed chain that masks individual step failures and blocks startup diagnostics.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance gaps including large base64 payloads stored in the job table, sequential DB inserts that should be batched, missing try/catch around the purge-ai-usage handler, and a hardcoded token limit that belongs in config. The extract-paper handler is the highest-risk area due to payload size and per-row insert loops.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency query columns, nullable timestamps that should be NOT NULL, and unbounded varchar types on ID columns that could cause silent data truncation.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability and security gaps: async initializers run without Promise.all parallelism, schedulers are started unconditionally without graceful shutdown cleanup, and full JSON response bodies are logged on every API request which risks leaking PII or sensitive financial data for school-aged users. Core architecture is reasonable but these gaps could cause instability and compliance issues in a Caribbean ed-tech context.

### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)
AdminDashboard.tsx has several reliability and security concerns including unguarded async operations, a memory leak from an uncleared scroll ref callback on every render, and missing auth checks on sensitive admin actions. The chat component re-creates the scroll ref inline on every render cycle, and sequential awaits exist where Promise.all would improve throughput.
