## 🔴 Critical (6)

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance (investment trade flow)
**Issue:** The buy/sell flow calls upsertPortfolioHolding, createPortfolioTransaction, and updateVirtualBalance as three separate awaits with no wrapping database transaction. A process crash or network blip between any two steps leaves the user's balance, holdings, and transaction log permanently out of sync — real money (virtual here, but trust-critical for students) can be lost or duplicated.

**Fix:** Wrap all three operations in a single db.transaction(async (tx) => { ... }) call using Drizzle's transaction API. Pass the tx client into each operation so the entire trade is atomic. Roll back automatically on any thrown error.

### `server/storage.ts` — getLeaderboard
**Issue:** getLeaderboard accepts an optional limit parameter but the SQL query has no fallback cap. If the caller omits limit (or passes undefined), the query returns every row in gameSessions joined to users — unbounded on a school deployment with hundreds of students and thousands of sessions, causing memory bloat and slow responses.

**Fix:** Add a hard default and a maximum cap: const safeLimit = Math.min(filters?.limit ?? 50, 200); and always pass .limit(safeLimit) to the query. Never allow an uncapped leaderboard fetch.

### `server/routes.ts` — AI tutor/chat route handlers (registerChatRoutes and openai usage blocks)
**Issue:** No rate limiting on AI endpoints. The openai import is used directly in routes without any request throttling. In a school-aged user app, a single student or bad actor can exhaust OpenAI quota or spike costs with rapid repeated requests. quotaErrorMessage and reserveQuota exist but there is no HTTP-layer rate limiter (e.g. express-rate-limit) guarding these endpoints.

**Fix:** Add express-rate-limit middleware scoped per user/IP before AI route handlers: `import rateLimit from 'express-rate-limit'; const aiLimiter = rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.use('/api/ai', aiLimiter);`. Tune max to your quota budget.

### `server/routes.ts` — videoUpload multer config (~line 40)
**Issue:** 500 MB video uploads are stored entirely in process memory (multer.memoryStorage()). A single concurrent upload of a 500 MB file doubles the Node.js heap for the duration of the request. With multiple simultaneous teacher uploads this will OOM the process, which is especially dangerous on a shared Replit container.

**Fix:** Switch videoUpload to disk storage with a temp directory: `multer({ storage: multer.diskStorage({ destination: os.tmpdir(), filename: (_req, file, cb) => cb(null, crypto.randomUUID() + path.extname(file.originalname)) }) })`. Stream the temp file to object storage then `fs.unlink` it in a finally block. Cap concurrent uploads with a semaphore if needed.

### `server/jobHandlers.ts` — runExtractPaper / extract-paper handler
**Issue:** The full file contents are stored as a base64 string (fileB64) inside the job payload row in the database. For a PDF or image this can easily be 5–20 MB per row, bloating the jobs table, slowing queue polling queries, and potentially hitting row-size limits depending on the DB engine.

**Fix:** Upload the file to private object storage at enqueue time and store only the object path in the job payload (e.g. { paperId, objectPath: '/uploads/papers/123.pdf', ext, subject }). In runExtractPaper, download the buffer from object storage using objectStorageClient before processing. This keeps job rows small and makes retries cheaper.

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel queryFn
**Issue:** The queryFn arrow function inside AuditLogPanel calls apiRequest and chains .then(r => r.json()) with no try/catch or error boundary. If the network request fails or the response is not valid JSON, the error will surface as an unhandled promise rejection and React Query will retry indefinitely with no user-visible error state rendered in the component.

**Fix:** Wrap the queryFn body in try/catch and return a typed error, and add isError/error destructuring from useQuery to render an error message in the UI: `queryFn: async () => { try { const r = await apiRequest('GET', ...); return r.json(); } catch (e) { throw new Error('Failed to load audit log'); } }`. Also destructure `isError, error` and render a red alert card when isError is true.

## 🟡 Warnings (16)

- **`server/storage.ts` (getAllQuestions):** getAllQuestions fetches every ExtractedQuestion joined to ExamPaper with no limit or pagination parameters in the interface signature. As teachers upload exam papers (each potentially containing 50–100 questions), this becomes a full-table scan returned in a single response, with no way for callers to paginate.
  → *Add page: number and pageSize: number (max 100, default 50) to the filters parameter. Apply .limit(pageSize).offset(page * pageSize) in the query. Update the IStorage interface and all callers to pass pagination params.*
- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues multiple independent database queries sequentially (total income, total expenses, budget utilisation, recent transactions, savings goals) using sequential awaits. Each query is independent and blocking the event loop unnecessarily, adding cumulative latency on every dashboard load for every student.
  → *Collect all independent query promises into an array and resolve with Promise.all([incomeQuery, expenseQuery, budgetQuery, goalsQuery, recentTxQuery]). Destructure the results. This cuts dashboard load time to the latency of the single slowest query rather than the sum of all.*
- **`server/storage.ts` (getClassProgressSummary):** getClassProgressSummary retrieves all enrollments for a class and then fetches learning progress, XP, and badges for each student in a per-student loop — a textbook N+1 query pattern. For a class of 35 students this fires 105+ individual queries per request, which will visibly stall the teacher dashboard.
  → *Extract all studentIds from the enrollment result, then fetch all progress, XP, and badge rows in three bulk queries using inArray(userXp.userId, studentIds). Join the results in JavaScript by userId. This collapses 3N+1 queries to 4 queries regardless of class size.*
- **`server/routes.ts` (Google sign-in handler — resolvedClass resolution block (~line 220-240)):** Three sequential awaits — getClassByCode, getOrgEnvironmentById, getOrganization — run one after another when processing a join code during Google sign-in. Each is an independent DB/Supabase round-trip. Under load (e.g. an entire classroom logging in simultaneously) this triples latency and connection pressure unnecessarily.
  → *After fetching resolvedClass and confirming it has an envId, fetch the environment and organization in parallel: `const [classEnv, classOrg] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), getOrganization(resolvedClass.orgId)])`. Ensure getOrganization can accept the org id available on the class or env record to avoid the sequential dependency.*
- **`server/routes.ts` (healthz route handler (~line 155)):** Dynamic import of './db' and 'drizzle-orm' inside every health check request. Dynamic imports are cached after the first call but still incur promise overhead and obscure the module graph. More importantly, if the health check is called by a load balancer every 5 seconds this pattern is noisy and could mask real import errors at startup.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file with the other static imports. The health check handler becomes `await db.execute(sql\`select 1\`)` with no dynamic imports needed.*
- **`server/routes.ts` (Google sign-in handler — profile.email domain enforcement block):** The truncated code shows `profile.emai` — an apparent truncation mid-expression — but the pattern of logging or using profile fields (email, name) from the Google token payload in error responses or audit logs risks exposing PII. Additionally, if verifyGoogleToken throws (e.g. expired token), the outer try/catch will return the raw error message which may include the token or internal stack details.
  → *In the catch block return only a generic message: `res.status(401).json({ message: 'Authentication failed' })` and log the sanitized error server-side via your audit/sentry integration. Never reflect error.message from token verification libraries directly to the client. Ensure audit() calls only log userId/orgId, not raw email strings.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with an await on every iteration. For a 40-question paper this means 40 serial round-trips to the database before the paper status is updated, multiplying latency and holding the worker busy unnecessarily.
  → *Collect all valid question objects into an array first, then execute a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, batch with Promise.all on chunks of ~10 to avoid overwhelming the connection pool while still running in parallel.*
- **`server/jobHandlers.ts` (import { openai } from "./replit_integrations/chat/routes"):** The openai client is imported from a route module rather than a dedicated singleton. Route modules typically register Express handlers as a side effect of import, meaning this import silently executes route registration code every time jobHandlers.ts is loaded. It also creates a hidden coupling that makes the worker unrunnable without the full HTTP layer.
  → *Extract the openai client instantiation into its own module (e.g. server/lib/openaiClient.ts) that exports only the configured client. Both the route file and jobHandlers.ts should import from that shared singleton. This eliminates the side-effect risk and clarifies the dependency graph.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation. A payload with olderThanDays: 0 or a negative number would compute a future cutoff date and delete all rows in the table, causing irreversible data loss.
  → *Add an explicit guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); This turns bad payloads into a failed (retryable/alertable) job rather than a silent destructive operation.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId even though budget lookups will always be scoped by user. Every budget query will do a full table scan as the user base grows.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })*
- **`shared/schema.ts` (billReminders, savingsGoals, portfolioHoldings, portfolioTransactions, documentUploads — all createdAt columns):** createdAt uses .defaultNow() without .notNull(), meaning the column is nullable in the DB schema. A failed default or ORM bypass could insert a NULL createdAt, breaking any ORDER BY createdAt queries silently.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The transactions table already does this correctly — apply the same pattern everywhere.*
- **`shared/schema.ts` (varchar('user_id') across all tables):** userId is declared as varchar with no length argument, which maps to varchar without a limit in Postgres — effectively the same as text but with misleading intent. More critically, if the auth provider (e.g. Clerk/Supabase) issues IDs of a known max length (e.g. 36 chars for UUID, ~32 for Clerk), leaving the column unbounded removes a data-integrity guard and prevents the optimizer from using certain index strategies.
  → *Specify an explicit length matching your auth provider's ID format, e.g. varchar('user_id', { length: 36 }) for UUIDs or varchar('user_id', { length: 64 }) for Clerk. Apply consistently across every table.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker has no unique index and no NOT NULL length constraint. Duplicate tickers inserted via a seed race or admin bug would corrupt portfolio value calculations silently since portfolioHoldings.stockId references simulatedStocks.id, not the ticker, but display logic likely joins on ticker.
  → *Add uniqueIndex on ticker: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also consider varchar('ticker', { length: 10 }) to enforce realistic ticker lengths at the DB level.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params.set('limit', '200')):** The audit log query hardcodes a limit of 200 rows directly in the component. For a school-aged user base across the Caribbean this may be fine today, but as data grows this will cause slow queries and large payloads. It is also untestable without modifying source.
  → *Extract to a named constant at module scope or read from an environment variable: `const AUDIT_LOG_PAGE_SIZE = Number(import.meta.env.VITE_AUDIT_LOG_LIMIT ?? 200);`. Pass this constant into params.set so it is configurable per environment without code changes.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — useQuery queryFn):** ObservabilityCard defines its queryFn as an inline arrow function with no staleTime or gcTime configured. Because the component has no isLoading/isError guard, on the first render data is undefined and every property access (data?.sentry, data?.alertEmail) silently produces undefined, rendering all badges as 'Disabled'/'Not configured' even during loading — misleading for an admin.
  → *Add `isLoading` and `isError` from useQuery. Render a skeleton or spinner when isLoading is true. Add `staleTime: 60_000` to avoid refetching on every tab switch since observability config rarely changes.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params URLSearchParams construction):** The URLSearchParams object is reconstructed on every render of AuditLogPanel because it is defined in the component body outside of useMemo. This means a new params string is produced on every keystroke in any sibling state (e.g. the action input), causing the queryKey reference to change and triggering redundant network fetches even when the relevant filter values have not changed.
  → *Wrap the params construction in useMemo: `const queryString = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', String(AUDIT_LOG_PAGE_SIZE)); return p.toString(); }, [actorType, action, from, to]);` and use queryString in both queryKey and queryFn.*

## 💡 Opportunities

### Paginate getTransactions at the storage layer — 🔥 High Impact
getTransactions accepts an optional limit but no offset/cursor, making true pagination impossible. Add a cursor or offset parameter to the interface now, before the API surface is locked. Without this, a student with a year of daily transactions will transfer tens of thousands of rows on every load.

### Move hardcoded virtual balance seed amount to environment config — ⚡ Medium Impact
The initial virtual balance granted to new users (likely a hardcoded constant such as 10000) should be read from an environment variable (VIRTUAL_BALANCE_SEED_AMOUNT) so admins and sponsors can adjust it per school cohort without a code deployment.

### Add a centralised query error boundary with structured logging — ⚡ Medium Impact
Storage methods lack consistent try/catch; errors bubble as raw Drizzle/PostgreSQL exceptions that may expose table names or column details in API responses. Wrapping storage calls in a thin withDbError(op: string, fn) helper that catches, logs structured JSON (no PII), and rethrows a sanitised StorageError would harden all 50+ methods at once.

### Batch Supabase initialization calls at startup — ⚡ Medium Impact
initSupabaseTables, initStaticContentTables, seedStaticModules, seedRegionalContent, and seedGameContent are likely called sequentially at server start. Running them with Promise.all where there are no dependencies between them would cut cold-start time, which matters for Replit's container wake-up latency that students experience when first opening the app.

### Centralize multer error handling middleware — ⚡ Medium Impact
Four separate multer instances (upload, logoUpload, videoUpload, examUpload) each propagate errors differently depending on where they are used. Adding a single Express error-handling middleware `(err, req, res, next) => { if (err instanceof multer.MulterError || err.message) return res.status(400).json({ message: err.message }); next(err); }` registered after all routes eliminates repeated inline error checks and ensures consistent client-facing error shapes.

### Extract AI quota enforcement into reusable middleware — 🔥 High Impact
The reserveQuota/finalizeUsage/releaseReservation pattern appears to be duplicated across multiple route handlers. Extracting this into a named Express middleware factory `withQuota(orgId)` that wraps the handler and calls releaseReservation in a finally block would eliminate the risk of a code path forgetting to release a reservation, which currently causes quota to be permanently consumed even on failed requests.

### Idempotency token on extracted-question bulk insert — ⚡ Medium Impact
The current idempotency strategy deletes all previously inserted questions on retry before re-inserting. A safer pattern is INSERT ... ON CONFLICT DO NOTHING keyed on (paperId, order), which avoids a delete-then-reinsert window where a concurrent read could see zero questions. This also makes retries safe even if the delete itself fails mid-flight.

### Structured job result schema validation — 🔥 High Impact
Job handlers return plain untyped objects. Adding a Zod schema for each handler's return value and validating before return would catch cases where the OpenAI response JSON is malformed but JSON.parse succeeds with an empty object, silently marking the job complete with 0 questions. Catching this at the handler boundary enables automatic retry instead of silent data loss.

### Centralise OpenAI call retries and token logging — ⚡ Medium Impact
openai.chat.completions.create is called in at least two places in this file with no retry logic and no logging of prompt_tokens/completion_tokens from the response. Wrapping calls in a shared helper that records token usage to aiUsageEvents and applies exponential back-off on 429/500 responses would improve cost visibility and resilience without duplicating logic across handlers.

### Composite index on billReminders for due-date alerting queries — ⚡ Medium Impact
Bill reminder notifications will query WHERE user_id = $1 AND next_due_date <= $2. Without a composite index on (userId, nextDueDate), this becomes a full table scan per scheduled job run. Adding index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) would make the alerting job O(log n) per user instead of O(n).

### Enforce currency as an enum type rather than free text — ⚡ Medium Impact
currency is a plain text column with a hardcoded default of 'BSD' repeated across six tables. A mistyped currency string ('bsd', 'US', etc.) would pass silently and break any FX conversion logic. Defining a pgEnum for supported currencies (BSD, USD, JMD, TTD, etc.) and reusing it across tables adds DB-level validation and makes the supported currency list a single source of truth.

### Extract repeated numeric precision config into a shared constant — 📌 Low Impact
numeric({ precision: 10, scale: 2 }) appears at least 12 times across the schema. If the app needs to support amounts above $99,999,999.99 (plausible for investment simulations) every table needs a manual migration. Define a moneyColumn helper — const moneyColumn = (name: string) => numeric(name, { precision: 15, scale: 2 }) — and use it everywhere. Single-point change for precision adjustments and eliminates copy-paste drift.

### Parallel data fetching on dashboard mount — 🔥 High Impact
The overview tab likely fires multiple sequential useQuery calls (observability, stats, audit summary) that could each block on the previous. Consolidate into a single /api/admin/dashboard-summary endpoint that resolves all data server-side with Promise.all, reducing waterfall latency by the number of sequential round-trips — typically 2-4 saved network cycles on Caribbean mobile connections.

### Split AdminDashboard into lazy-loaded tab modules — 🔥 High Impact
At 126k+ characters this is a single bundle chunk. Splitting each tab panel (AuditLogPanel, ObservabilityCard, DB Viewer, Perf Agent) into React.lazy + Suspense dynamic imports would reduce initial JS parse time significantly, which matters for school-aged users on lower-end devices common in the Caribbean market.

### Add request debounce to filter inputs in AuditLogPanel — ⚡ Medium Impact
The action text input triggers a new query on every keystroke because the state change flows directly into the queryKey. Adding a 300ms debounce (via useDebounce hook or setTimeout/clearTimeout in useEffect) on the action and date inputs would eliminate redundant in-flight requests and reduce server load during active filtering sessions.
