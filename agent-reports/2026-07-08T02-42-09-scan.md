### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. A student or bot can hammer these endpoints to run up API costs or cause denial of service.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerX calls.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 40 and ~line 57)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation. An attacker can send a crafted Host header to inject an arbitrary domain into the Sitemap directive and <loc> tags, poisoning SEO caches or redirecting crawlers.

**Fix:** Add a whitelist check: define ALLOWED_HOSTS from an environment variable (e.g. process.env.CANONICAL_HOST). If the resolved host does not match, fall back to process.env.CANONICAL_HOST. Never trust x-forwarded-host blindly in sitemap/robots generation.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** Day-of-week arithmetic is wrong. `(dow + 0) % 7` always returns `dow` unchanged, so Sunday (0) maps to 0 days back (today) and Saturday (6) maps to 6 days back â€” but Monday (1) maps to only 1 day back instead of 2, meaning the function never correctly walks back to the most recent Sunday for weekdays Tuesdayâ€“Saturday.

**Fix:** Replace `(dow + 0) % 7` with `dow === 0 ? 0 : dow` is still wrong. Use `const daysSince = dow;` directly and add a guard: if the resulting Sunday-at-7pm is in the future (because it's still before 19:00 on Sunday), subtract 7 days. Simplest correct form: `const daysSince = d.getDay(); d.setDate(d.getDate() - daysSince); d.setHours(19,0,0,0); if (d > now) d.setDate(d.getDate() - 7);` â€” the current code already has the future-guard but the `(dow + 0) % 7` expression is a no-op leftover from a refactor and should just be `dow`.

### `server/index.ts`: setInterval (scheduler intervals in startup)
**Issue:** The `setInterval` handles for the weekly-digest scheduler and AI-purge scheduler are never stored and never cleared in the SIGTERM/SIGINT shutdown handlers. If the process receives a shutdown signal mid-tick, the in-flight DB writes and job enqueues have no cancellation path and the interval may fire one final time after the DB connection is closing.

**Fix:** Capture every interval: `const digestInterval = setInterval(maybeRunWeeklyDigest, ...)`. In the graceful shutdown handler call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` before awaiting `httpServer.close()`. Also gate each scheduler tick with a module-level `let shuttingDown = false` flag that the handlers check at entry and return early if true.

### `server/index.ts`: res.json monkey-patch in request logger middleware
**Issue:** Every JSON response body is captured into `capturedJsonResponse` and serialized with `JSON.stringify` unconditionally into the log line. For a financial literacy app with school-aged users this will log quiz answers, user profile data, session tokens returned in API responses, and any AI-generated content in plaintext to stdout/Sentry breadcrumbs.

**Fix:** Either remove response-body logging entirely for production, or add an allowlist of safe, non-sensitive paths (e.g. `/api/health`, `/api/version`) and only log bodies for those. At minimum add: `if (capturedJsonResponse && process.env.NODE_ENV !== 'production') { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`

### `client/src/pages/AdminDashboard.tsx`: sendMessage async function in AdminHelpChat
**Issue:** `res.json()` is called without checking `res.ok`. A non-2xx HTTP response (e.g., 401 Unauthorized, 500 Internal Server Error) will still attempt to parse the body and silently pass a server error object to `data.reply`, showing nothing to the user or surfacing misleading content.

**Fix:** After `await apiRequest(...)`, add `if (!res.ok) throw new Error(await res.text());` before calling `res.json()`. The existing catch block will then handle it correctly and show the fallback message.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook call / top-level component render
**Issue:** If `useAdminAuth` returns an unauthenticated state but does not hard-redirect, the rest of the dashboard (queries, mutations, sensitive data renders) continues to execute and fire network requests. This means admin data queries run client-side even for unauthenticated sessions, relying solely on server rejection rather than early client bail-out.

**Fix:** Immediately after calling `useAdminAuth`, add a guard: `if (!isAdmin) { setLocation('/admin/login'); return null; }` before any query hooks are invoked. This prevents all downstream queries and UI from mounting for unauthorized users.

A request like /public-objects/../../../etc/passwd could traverse outside the intended storage prefix depending on the ObjectStorageService implementation.
  â†’ *Sanitise the path before use: import path from 'path'; const safePath = path.normalize(filePath).replace(/^\.\.\/|\/\.\.\//g, ''); then assert safePath does not start with '..' before passing to objectStorage. Alternatively, strip leading slashes and reject any path containing '..'.*
- **`server/routes.ts` (/healthz handler (~line 80)):** The /healthz endpoint imports db and drizzle-orm dynamically on every request. Dynamic imports are cached by Node after the first call, but the pattern is fragile and adds overhead; more critically, the endpoint is unauthenticated and returns error details (e.message) that could leak internal DB error messages or connection strings to external callers.
  â†’ *Move the db import to the top of the module (static import). In the catch block, log the full error server-side but return only a generic message to the caller: res.status(503).json({ ok: false, error: 'db unavailable' }); Never echo e.message to unauthenticated clients.*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, startJobWorker calls)):** Database probe, seed, route registration, and job worker startup are awaited sequentially on the critical path before the HTTP server begins listening. `seedDatabase` and `probeDatabase` each open their own DB round-trips and if either hangs (e.g. DB not yet ready in a cold Replit spin-up) the entire startup blocks with no timeout, delaying the server indefinitely.
  â†’ *Wrap `probeDatabase` and `seedDatabase` in `Promise.race` with a timeout: `await Promise.race([probeDatabase(), new Promise((_,r) => setTimeout(() => r(new Error('DB probe timeout')), 10_000))])`. Where order does not matter (probe vs. seed are independent), run them with `await Promise.all([probeDatabase(), seedDatabase()])`. Add a top-level `process.on('uncaughtException')` handler that logs and exits so a hang-then-throw does not leave a zombie process.*
- **`server/index.ts` (maybeRunWeeklyDigest / maybeRunAiUsagePurge):** Both async scheduler callbacks are fired by `setInterval` but the returned Promises are not awaited and errors inside the outer try/catch only log a message â€” they do not call `captureError` / Sentry. A DB error that repeats every interval tick will silently spam logs but never alert on-call and the next tick starts before the previous one finishes if the DB is slow.
  â†’ *Wrap the interval callback to prevent overlapping executions and route errors to Sentry: `let digestRunning = false; setInterval(async () => { if (digestRunning) return; digestRunning = true; try { await maybeRunWeeklyDigest(); } catch(e) { captureError(e); } finally { digestRunning = false; } }, interval)`. Do the same for the purge scheduler.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat â€“ scroll div ref callback):** The inline `ref={el => { if (el) el.scrollIntoView(...) }}` callback is recreated on every render. Because it fires during every reconciliation pass, it triggers scrollIntoView on every state update (e.g., typing a character in the input), not just when a new message arrives. This causes jarring scroll resets mid-typing.
  â†’ *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then render `<div ref={bottomRef} />`. This scopes scrolling strictly to new messages.*
- **`client/src/pages/AdminDashboard.tsx` (Multiple top-level useQuery calls (dashboard stats, users, schools, etc.)):** Each admin section appears to issue its own independent `useQuery` with no shared `staleTime`, causing all queries to refetch simultaneously on window focus or component remount.   â†’ *Set a consistent `staleTime` (e.g., `staleTime: 60_000`) on all dashboard queries, or configure a global default in your `QueryClient` constructor: `defaultOptions: { queries: { staleTime: 60000 } }`. This halves unnecessary network traffic during normal admin navigation.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline handler functions passed as JSX props (onClick, onChange throughout table rows and form dialogs)):** Inline arrow functions defined directly in JSX (e.g., `onClick={() => handleDeleteUser(user.id)}` inside a `.map()`) are recreated for every item on every parent render. ## đź’ˇ Opportunities

### Centralise and document rate limiter coverage: đź”Ą High Impact
Rate limiters are applied ad-hoc before individual route prefixes. Create a single applyRateLimits(app) function in rateLimiter.ts that lists every protected prefix in one place. This prevents new routes being added without limit coverage and makes auditing trivial.

### Add a startup readiness gate before accepting traffic: âšˇ Medium Impact
Supabase init runs fire-and-forget. If seeds fail silently, the app serves requests against an uninitialised schema. Await a lightweight schema-check promise before returning httpServer, or expose a /readyz endpoint that returns 503 until the init chain resolves, so load balancers hold traffic.

### Replace dynamic imports inside /healthz with module-level imports: đź“Ś Low Impact
Moving db and sql to static top-level imports eliminates the repeated dynamic import overhead on every health probe and makes dependency graph explicit for bundlers and analysis tools. Health checks can be called every few seconds by Railway; dynamic imports inside them are unnecessary noise.

### Parallel startup I/O with Promise.all: đź”Ą High Impact
Running probeDatabase, seedDatabase, registerJobHandlers, and startJobWorker sequentially adds their latencies together on every cold start. Tasks that don't depend on each other (probe and seed are independent of job handler registration) can be grouped into Promise.all batches, potentially cutting startup time by 40â€“60% on a cold Replit container.

### Structured logging to replace string concatenation: âšˇ Medium Impact
All log calls use template-string concatenation into a single unstructured line. Replacing with a structured logger (pino is zero-dependency and 5Ă— faster than console.log) emitting JSON gives Sentry/Datadog parseable fields, prevents accidental PII interpolation, and eliminates the perf cost of JSON.stringify on every response body in the middleware.

### Externalize scheduler intervals to environment config: đź“Ś Low Impact
The digest and purge check intervals appear to be hardcoded magic numbers in the truncated portion of the file. Exposing them as DIGEST_CHECK_INTERVAL_MS and PURGE_CHECK_INTERVAL_MS env vars (with safe defaults) allows tuning in production without a redeploy and makes the values visible in the Replit Secrets panel alongside other operational config.

### Parallelize independent admin data fetches with Promise.all: đź”Ą High Impact
Any server-side or utility functions that sequentially await stats, user counts, and school counts should be refactored to `Promise.all([fetchStats(), fetchUsers(), fetchSchools()])`. Based on typical dashboard patterns this could reduce initial load time by 40â€“60% on the admin overview page.

### Memoize filtered and sorted table data: âšˇ Medium Impact
Search/filter state changes likely recompute filtered user and school arrays inline on every render. Wrapping these derivations in `useMemo` keyed on the raw data array and filter string prevents redundant iterations over potentially large datasets on unrelated state updates (e.g., dialog open/close).

### Move hardcoded API base paths and pagination limits to environment config: âšˇ Medium Impact
String literals like `/api/admin/help-chat` and any hardcoded page-size values (e.g., 10 or 20 rows per page) should be sourced from `import.meta.env.VITE_API_BASE` and a shared constants file. This prevents silent misconfiguration across staging vs. production Caribbean deployment environments.
