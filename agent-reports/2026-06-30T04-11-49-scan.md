## 🔴 Critical (4)

### `server/routes.ts`: /public-objects/:filePath(*) handler (~line 96)
**Issue:** The public-objects route has no authentication or path sanitization. Any unauthenticated caller can enumerate and download any file in object storage by guessing paths. The filePath parameter is passed directly to searchPublicObject with no validation, enabling potential path traversal (e.g., ../../secrets).

**Fix:** Add path sanitization using path.normalize and verify the result does not escape the intended prefix: `const safe = path.normalize(filePath).replace(/^\.\.\//, ''); if (safe !== filePath) return res.status(400).json({ error: 'Invalid path' });`. For non-public assets, add a session/auth check before serving.

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 118)
**Issue:** The AI/OpenAI integration routes (chat, image, audio) are registered with no rate limiting middleware applied before them, unlike the auth routes which have authLimiter/strictAuthLimiter. Heavy or costly AI endpoints are left unprotected from abuse, which is a financial risk (OpenAI billing) and a reliability risk for school-aged users on shared infrastructure.

**Fix:** Create a dedicated aiLimiter in rateLimiter.ts (e.g., 20 requests per minute per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls. Adjust the window and max to match expected classroom usage patterns.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Full file content stored as base64 in the job payload. A 5MB PDF becomes ~6.7MB of base64 sitting in the jobs table row for every attempt, bloating the DB, slowing queue polling queries, and leaking file content into any job-table log or export.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the file from object storage at the start and delete it on terminal success or failure. This keeps job rows small and file data out of the DB entirely.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth / admin route guards
**Issue:** If useAdminAuth returns a non-admin or unauthenticated state but the redirect via useLocation has not yet fired (e.g., during the auth query loading phase), admin-only mutations and queries are already mounted and will execute. There is no early return or null render while auth is resolving, so the dashboard briefly renders and fires queries for unauthenticated users.

**Fix:** After calling useAdminAuth, add `if (authLoading) return <FullPageSpinner />; if (!isAdmin) return null;` before rendering any child components or registering any useQuery/useMutation hooks that touch privileged endpoints.

## 🟡 Warnings (11)

- **`server/routes.ts` (/healthz handler (~line 83)):** The /healthz endpoint calls `await import('./db')` and `await import('drizzle-orm')` on every request. Dynamic imports are cached by Node after the first load, but the async resolution overhead and the unconditional `await db.execute(sql\`select 1\`)` against the real DB on every probe means a high-frequency uptime monitor (e.g., every 10s) will generate continuous DB load with no rate limiting.
  → *Hoist `db` and `sql` to module-level imports at the top of the file alongside other imports. Optionally add a short cache (e.g., 5s) on the health result to prevent DB hammering: cache the last check timestamp and return the cached status if called within the window.*
- **`server/routes.ts` (Supabase init chain (~line 122)):** The seven Supabase init/seed functions are chained with sequential .then() calls, meaning each waits for the previous to complete. On a cold start this blocks seeding completion unnecessarily; if seedFinancialAcademyLesson() is slow, seedGameContent() is delayed by the full sum of all previous durations.
  → *Group independent operations with Promise.all where order does not matter. At minimum, the seed functions that do not depend on each other can run concurrently: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()])).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));`*
- **`server/routes.ts` (robots.txt and sitemap.xml handlers (~lines 40-80)):** Both handlers trust `x-forwarded-proto` and `x-forwarded-host` headers directly from the request to construct absolute URLs without validation. A spoofed Host or X-Forwarded-Host header will cause the sitemap and robots.txt to emit attacker-controlled URLs, which search engine crawlers will follow.
  → *Read the canonical base URL from an environment variable: `const base = process.env.CANONICAL_URL || \`${proto}://${host}\`;`. Set CANONICAL_URL in Railway/Replit config to the production domain. This eliminates header-spoofing risk and also makes the value stable across environments.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one inside a for loop with sequential awaits. For a 60-question paper this serialises 60 round-trips to the DB, dramatically increasing total job duration and holding the worker slot.
  → *Replace the loop with a single bulk insert: filter valid questions first, map them to insert rows, then call db.insert(extractedQuestions).values(rows) once. This reduces N round-trips to 1 and makes the operation atomic, removing the need for the idempotency delete at the top of the function (or at least making it cheaper).*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (model and max_completion_tokens)):** Model name 'gpt-4o' and token limit 4096 are hardcoded in two separate OpenAI calls. Changing the model or budget for cost/compliance reasons requires a code deploy rather than a config change.
  → *Read these from environment variables: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o' and const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10). Apply to both calls. Add validation at startup to warn if the env vars are missing.*
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** The 30,000-character truncation is hardcoded and silently discards content beyond that limit with no log or warning. For a long exam paper this could cause legitimate questions to be silently dropped and the returned questionCount to be understated without any indication to the user or operator.
  → *Move the limit to an environment variable (EXTRACT_TEXT_MAX_CHARS). Before slicing, if extractedText.length > limit emit a structured warning log: console.warn({ paperId, originalLength: extractedText.length, truncatedTo: limit }, 'PDF text truncated'). Surface a truncated flag in the job result so callers can alert the teacher.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler deletes rows without a LIMIT, meaning a single job run could issue one unbounded DELETE against a large aiUsageEvents table, causing a long-running lock that blocks concurrent reads and writes on that table during the purge.
  → *Batch the delete: loop calling db.delete(aiUsageEvents).where(and(lt(aiUsageEvents.createdAt, cutoff), sql`ctid IN (SELECT ctid FROM ai_usage_events WHERE created_at < ${cutoff} LIMIT 1000)`)). Accumulate rowCount across iterations and break when the batch returns 0. Return total deletedRows. This keeps individual locks short and lets other queries proceed.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat / sendMessage):** The help-chat endpoint sends the full accumulated message history on every turn with no length cap. A user can craft an arbitrarily long conversation, bloating every subsequent request payload and potentially hitting API rate limits or token limits silently, with no error surfaced beyond a generic fallback.
  → *Slice messages to the last N turns before sending: `messages: next.slice(-20)`. Also validate `data.reply` is a non-empty string before calling setMessages to avoid rendering undefined content.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat JSX / scroll div ref):** The scroll-to-bottom sentinel uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` inside the render path. This creates a new function on every render and calls scrollIntoView every time any state changes in the parent, not just when messages change, causing unnecessary forced layout/reflow.
  → *Replace with a proper useRef + useEffect: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]); ... <div ref={bottomRef} />`*
- **`client/src/pages/AdminDashboard.tsx` (Multiple useQuery calls (users, schools, lessons, analytics)):** Each admin data query is declared independently and fires sequentially on mount with no dependency between them. This results in multiple waterfall round-trips at page load when all four datasets are independent and could be fetched in parallel.
  → *Either use `useQueries` from TanStack Query to fan out all independent queries simultaneously, or consolidate into a single `/api/admin/dashboard-summary` endpoint that returns all needed data in one round trip, reducing page load latency by 3x on slow Caribbean mobile connections.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline mutation handlers inside JSX (delete, promote, ban buttons)):** Mutation call-site handlers like `onClick={() => deleteMutation.mutate(user.id)}` are defined as new arrow functions inside the JSX map callback on every render of the user list. With hundreds of users this creates O(n) function allocations per render cycle, and these components will never bail out of re-rendering via React.memo because the prop reference always changes.
  → *Wrap each handler with useCallback outside the map, or extract a memoized UserRow component that receives stable callback props. Example: `const handleDelete = useCallback((id: number) => deleteMutation.mutate(id), [deleteMutation]);` then pass it as a prop so React.memo can do referential equality checks.*

## 💡 Opportunities

### Centralise CANONICAL_URL for SEO routes: ⚡ Medium Impact
Introduce a single CANONICAL_URL env var consumed by robots.txt, sitemap.xml, and any future OG/meta tag generation. This removes duplicated header-parsing logic (currently copied verbatim in two handlers) and makes environment promotion (staging → production) safe without code changes.

### Add /api/health response caching header: 📌 Low Impact
The /api/health endpoint currently returns no cache headers, so every uptime monitor call is a fresh round-trip. Adding `Cache-Control: no-store` is correct for /healthz (which does a real DB ping), but the lightweight /api/health could return `Cache-Control: max-age=5` to allow CDN/proxy coalescing and reduce noise in logs without impacting reliability monitoring.

### Parallel domain route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six individual awaits. If any registerXRoutes function performs async I/O (e.g., DB schema checks), this is unnecessarily serial. Wrapping them in a second Promise.all would halve cold-start registration time for routes with async setup.

### Parallel independent async operations in export handlers: ⚡ Medium Impact
Any admin-csv-export handler that fetches from multiple tables with independent queries (e.g. users + transactions + badges) likely does so sequentially. Wrapping independent fetches in Promise.all would cut export job wall-time proportionally to the number of parallel queries, reducing worker slot occupancy and improving perceived export speed.

### Structured logging instead of raw console calls: ⚡ Medium Impact
Replacing ad-hoc console.log/warn/error calls with a structured logger (pino or winston) that includes jobId, handlerName, paperId, and attempt number as fields would make Cloud log queries and alerting on failure rates dramatically faster without changing any business logic.

### Idempotency token on OpenAI calls: 📌 Low Impact
If the worker crashes after the first OpenAI call but before the second, a retry re-runs both calls paying double. Passing a deterministic idempotency key (e.g. `${paperId}-extract-v1`) to the OpenAI request header would allow the API to return the cached response on retry at no additional cost or latency.

### Memoize filtered and sorted user/lesson lists: 🔥 High Impact
Search filtering and sorting applied to user and lesson arrays are recomputed on every render including unrelated state changes like dialog open/close. Wrapping these with useMemo keyed on the data array and filter/sort state would eliminate redundant O(n) passes and prevent jank on large datasets common in school-wide deployments.

### Add loading and error boundary states to all admin queries: 🔥 High Impact
Several query results are used directly without checking isLoading or isError, meaning the UI renders with undefined data and crashes or shows empty charts. Adding consistent skeleton states and error messages with retry buttons would prevent blank-screen failures and improve trust with school administrators.

### Move hardcoded API path strings to a central config: ⚡ Medium Impact
Endpoint strings like '/api/admin/help-chat', '/api/admin/users' etc. are scattered as string literals throughout the file. Centralising them in a typed API_ROUTES constant object eliminates typo-driven 404s during refactors and makes endpoint auditing for security reviews trivial.
