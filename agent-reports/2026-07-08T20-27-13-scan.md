## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI-backed routes (chat, image, audio) are registered with no rate limiting middleware applied before them. In a school-aged user app these endpoints can be trivially abused to run up OpenAI costs or trigger inappropriate content generation at volume.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP using express-rate-limit) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three register calls.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 35 and ~line 53)
**Issue:** `x-forwarded-proto` and `x-forwarded-host` headers are used without validation to construct absolute URLs embedded in robots.txt and sitemap.xml. An attacker can send arbitrary Host/X-Forwarded-Host headers, causing the sitemap URL to point to an attacker-controlled domain (host-header injection).

**Fix:** Read the canonical base URL from a single trusted environment variable: `const base = process.env.PUBLIC_BASE_URL ?? 'https://finsightlite.com';` and remove all runtime header-based construction for these two routes. Set the env var at deploy time in Railway.

### `shared/schema.ts`: categories table definition
**Issue:** categories.userId is a bare varchar() with no length specified, matching the pattern used on all other tables. More critically, userId across every table uses varchar() without a length constraint, meaning Postgres defaults to varchar with no limit — functionally equivalent to text but without the self-documenting intent. If users.id is a UUID (36 chars) or CUID (25 chars), leaving this unconstrained allows insertion of arbitrarily long strings that will never match a real user, silently corrupting referential data that the FK won't catch at the type level.

**Fix:** Decide on a canonical user ID format (e.g., UUID or CUID) and enforce it: use varchar('user_id', { length: 36 }) for UUID or length: 25 for CUID on every userId column across all tables — categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions. Update the users.id column in models/auth to match so FK type alignment is exact.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets has no index on userId, yet every budget query for a user will be a full table scan. As users accumulate budgets over time this will degrade proportionally. Similarly billReminders, savingsGoals, linkedCards, documentUploads, and portfolioHoldings all lack a userId index despite being the primary lookup key in every application query.

**Fix:** Add explicit indexes: for budgets add (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Repeat for billReminders (also add nextDueDate composite: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate)), savingsGoals, linkedCards, documentUploads, portfolioHoldings (composite on userId + stockId), and portfolioTransactions. The transactions table already has userDateIdx so use that as the pattern.

### `server/index.ts`: seedDatabase / probeDatabase calls in top-level async IIFE
**Issue:** probeDatabase() and seedDatabase() are awaited sequentially during startup on the main thread. If either hangs (e.g. DB unreachable), the HTTP server never starts and no timeout kills the hang, causing silent indefinite startup failure.

**Fix:** Wrap both in Promise.race against a hard timeout (e.g. 15 s via a rejecting setTimeout promise). Run them with Promise.all where order allows, and call httpServer.listen() before the DB probe so health-check endpoints respond immediately with a 503 until the DB is ready.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are never stored and never cleared. On SIGTERM/SIGINT the intervals keep firing into a half-torn-down process, which can trigger DB writes after the pool is closed and prevent the Node process from exiting cleanly.

**Fix:** Store each handle: const weeklyDigestTimer = setInterval(...). In the shutdown handler call clearInterval(weeklyDigestTimer) and clearInterval(purgeTimer) before closing the DB pool and HTTP server.

### `client/src/pages/AdminDashboard.tsx`: sendMessage (POST /api/admin/help-chat)
**Issue:** The help-chat endpoint is called with the full accumulated messages array on every send. There is no auth token or admin-role header explicitly attached beyond whatever apiRequest adds globally. If apiRequest relies solely on a cookie and the endpoint does not enforce admin-role server-side, any authenticated school student who discovers the endpoint can query the admin AI assistant with full conversation history as context.

**Fix:** Confirm server route /api/admin/help-chat applies the same admin middleware guard used by all other /api/admin/* routes. On the client, do not expose this chat UI unless useAdminAuth() confirms the session; add a guard in AdminHelpChat: const { isAdmin } = useAdminAuth(); if (!isAdmin) return null;

## 🟡 Warnings (11)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 105)):** The seven-step Supabase init chain runs sequentially via chained `.then()` calls. A failure in any intermediate step (e.g. `seedStaticModules`) is caught only by a single `.catch` that logs a generic message with no indication of which step failed. Steps that are independent (e.g. seedRegionalContent and seedGameContent) block each other unnecessarily, increasing cold-start time.
  → *Group independent seeds into `Promise.all` batches and add per-step error context: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()])).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error at step:', e?.message, e));`*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 95)):** The `filePath` path parameter is passed directly to `objectStorage.searchPublicObject` without any sanitisation. A path like `../../private/keys` could traverse outside the intended public directory depending on how `searchPublicObject` resolves paths internally.
  → *Sanitise the path before use: `import { normalize } from 'path'; const safePath = normalize(filePath).replace(/^\.\.\/|^\//, ''); if (safePath !== filePath) return res.status(400).json({ error: 'Invalid path' });` Then pass `safePath` to the storage call.*
- **`shared/schema.ts` (simulatedStocks table — lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text() instead of timestamp(). Storing a date as text means Postgres cannot use date range operators, cannot sort chronologically without casting, and offers no validation — '2024-13-45' would be accepted. For a financial app tracking price staleness this is particularly dangerous: range queries to find stocks not updated in N hours will require a CAST on every row, preventing index use.
  → *Change lastPriceUpdateDate: text('last_price_update_date') to lastPriceUpdateDate: timestamp('last_price_update_date').notNull(). Add a partial index for the price update job: index('idx_stocks_price_updated').on(t.lastPriceUpdateDate). Update any seed or migration code that inserts string dates to use Date objects.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, and all other createdAt columns):** Every createdAt column uses .defaultNow() but omits .notNull(). This means application code can explicitly insert null for createdAt and Postgres will accept it, breaking any ORDER BY createdAt queries with nulls sorting unpredictably and causing audit trail gaps. For a financial literacy app with school-aged users where audit history matters for parent/teacher review, null audit timestamps are a compliance risk.
  → *Append .notNull() to every createdAt (and executedAt, purchasedAt) column: createdAt: timestamp('created_at').defaultNow().notNull(). This applies to transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, and portfolioTransactions. The default handles inserts so no application code changes are needed.*
- **`shared/schema.ts` (simulatedStocks — ticker column):** ticker has no unique index. In an investment simulation context, ticker is a natural business key used to look up stocks by symbol. Without a unique constraint, duplicate tickers ('BSD:FINCO' inserted twice) will produce silent data integrity violations — portfolio calculations will double-count holdings and price update jobs will update the wrong row depending on which duplicate Postgres returns first.
  → *Add a unique index to the simulatedStocks table definition: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_stocks_ticker_unique').on(t.ticker) }). If multi-currency or multi-exchange tickers are anticipated, make it a composite unique on (ticker, region) instead. Also add .notNull() to previousPrice and priceChangePct since downstream percent-change calculations will fail silently on null arithmetic.*
- **`server/index.ts` (res.json monkey-patch inside request-logging middleware):** capturedJsonResponse stores the entire JSON body of every API response in a closure for the lifetime of the request. For large paginated responses or file-like payloads this holds a full object copy in memory until the 'finish' event fires, and the reference is never explicitly released.
  → *Cap what is stored: only capture if the serialised length is under a threshold (e.g. 512 chars) and only in non-production environments. Set capturedJsonResponse = undefined inside the 'finish' handler after logging so the GC can collect it immediately.*
- **`server/index.ts` (logLine construction in res.on('finish') handler):** JSON.stringify(capturedJsonResponse) is called unconditionally on every API response body before logging. In production this serialises potentially sensitive financial or PII data (user balances, quiz answers, session info) into plaintext application logs with no redaction.
  → *Gate the body logging behind a NODE_ENV !== 'production' check, or introduce an allowlist of safe response shapes. At minimum strip any keys matching /token|secret|password|ssn|dob/ before stringifying.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The sendMessage async function is recreated on every render and is not wrapped in useCallback. It closes over `messages` and `input` state but has no stable reference, meaning any parent or sibling re-render invalidates the memoized identity, and the onKeyDown handler (`handleKey`) also recreates every render because it depends on sendMessage.
  → *Wrap sendMessage with useCallback and include [messages, input, loading] in the dependency array. Then wrap handleKey with useCallback depending on [sendMessage]. This also prevents the disabled button check from creating a new function reference on each keystroke.*
- **`client/src/pages/AdminDashboard.tsx` (<div ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }} />):** The scroll-to-bottom ref uses an inline arrow function that executes scrollIntoView on every render, not just when new messages arrive. This fires DOM scroll on every keystroke in the input field because `input` state changes trigger re-render of the entire AdminHelpChat component.
  → *Replace with a useRef + useEffect pattern: const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]); Then use <div ref={bottomRef} />. This scopes the scroll to only when messages change.*
- **`client/src/pages/AdminDashboard.tsx` (messages.map((m, i) => ... key={i})):** Chat messages are keyed by array index. When a message is prepended or the assistant error-recovery path splices in a message, React will re-render all subsequent bubbles unnecessarily and input focus/animation state will shift. For a school-aged user on a low-bandwidth Caribbean connection this causes visible layout jank.
  → *Assign a stable id to each message at creation time: type ChatMessage = { id: string; role: 'user'|'assistant'; content: string }; and generate id with crypto.randomUUID() or Date.now().toString(36) + Math.random().toString(36). Use key={m.id} in the map.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages state):** The full conversation history grows unboundedly in memory and is sent to the API on every message. For a long admin session this means the POST body and server-side token consumption grow linearly. There is no cleanup when the component unmounts (e.g., the panel is closed and reopened resets to empty which is fine, but a session could accumulate hundreds of turns).
  → *Cap the context window sent to the API: const context = messages.slice(-20); then POST { messages: context } instead of next. Optionally add a 'Clear chat' button that calls setMessages([]) to let admins reset without closing the panel.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (duplicate health check endpoints: /api/health (~line 27) and /healthz (~line 67)):** Two health check endpoints exist: `/api/health` returns a shallow status immediately, and `/healthz` performs a real DB ping. Railway/uptime monitors are likely configured to only one of these. The shallow endpoint gives a false positive when the DB is down, and the duplicates create maintenance confusion.
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes 'most recent Sunday' using (dow + 0) % 7, where the + 0 is a no-op remnant of an edit (likely was + 1 or similar). Sunday is dow === 0 so daysSince is always 0 on Sunday but the intent of targeting the most recent past Sunday is coincidentally correct only because the subsequent > now guard compensates. The dead expression obscures the logic and will mislead the next maintainer.

## 💡 Opportunities

### Parallel domain route registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six `await` calls. If any `register*` function does async I/O (DB migrations, etc.) these block each other. Audit each register function; those that only attach Express middleware can be made synchronous or fired concurrently, cutting cold-start time by the sum of any async init work inside them.

### Structured startup logging for Supabase seed chain: ⚡ Medium Impact
Replace `console.error` in the seed catch block with a structured logger call (e.g. Sentry captureException + a labelled console.error per step). Currently a seed failure is silent in production monitoring dashboards. Adding Sentry capture here would surface seed regressions immediately after deploys.

### Extract ObjectStorageService instantiation to a singleton module: 📌 Low Impact
`new ObjectStorageService()` is constructed at module load time inside routes.ts. If the service opens connections or allocates resources, multiple hot-reloads or test imports will leak instances. Move it to a dedicated `server/services/objectStorage.ts` singleton export so it is instantiated once and mockable in tests.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 columns (transactions.currency, savingsGoals.currency, billReminders.currency, portfolioTransactions.currency, simulatedStocks.currency, portfolioHoldings implicitly). Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it in each .default(DEFAULT_CURRENCY). When FinSight Lite expands to TTD or JMD this becomes a single-line change instead of a grep-and-replace across the schema and any seed files.

### Add composite index for bill reminders due-date polling: 🔥 High Impact
The bill reminder notification job will query WHERE userId = ? AND nextDueDate <= NOW() AND nextDueDate >= NOW() - interval. Without a composite index on (userId, nextDueDate) this scans all reminders for all users on every poll cycle. A single index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) converts this to an index range scan, which matters at scale when a cron job runs every hour against thousands of student accounts.

### Replace serial PKs with UUID or CUID on user-facing tables: ⚡ Medium Impact
Serial integer PKs on transactions, budgets, savingsGoals and similar tables expose row counts and insertion rates via sequential IDs in API responses — a minor but real information disclosure for a financial app. Using gen_random_uuid() or a CUID default also eliminates hot-spot contention on the PK index under concurrent inserts from multiple student users in a classroom setting. Drizzle supports this via uuid('id').defaultRandom().primaryKey().

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase, seedDatabase, registerRoutes, and startJobWorker are awaited one-by-one. probeDatabase and registerJobHandlers have no interdependency and could run concurrently. Using Promise.all for independent steps would cut median cold-start time by the duration of the slowest independent operation, likely 200-800 ms on a cold Replit container.

### Centralised graceful-shutdown registry: ⚡ Medium Impact
Shutdown logic is currently ad-hoc inline in the SIGTERM handler. Introduce a ShutdownRegistry that accepts cleanup callbacks (clearInterval, server.close, pool.end, worker.drain) registered at the point of resource creation. This eliminates the need to thread handles back to a central location and makes future resource additions self-documenting.

### Extract scheduler tick duration to environment config: 📌 Low Impact
The 10-minute (600 000 ms) scheduler polling interval is hardcoded. In test environments this forces real waits; in production operators cannot tune it without a code change. Expose it as SCHEDULER_POLL_MS with the same defensive parsing pattern already used for AI_PURGE_OLDER_THAN_DAYS.

### Split monolithic AdminDashboard into page-section components: 🔥 High Impact
At ~147k characters the file almost certainly renders all admin sections (users, schools, analytics, audit log) in one component tree. Any state change (e.g., search input) re-renders the entire dashboard including all Recharts instances. Extract each tab/section into its own memoized component with React.memo and move its query hooks inside it so only that section re-fetches and re-renders. Expected impact: 60-80% reduction in unnecessary renders on interactive actions.

### Parallelise independent admin data fetches with Promise.all: 🔥 High Impact
If the dashboard initiates sequential useQuery calls that are fired from a single useEffect or loader (e.g., fetching users then schools then stats), convert them to parallel fetches or ensure all useQuery hooks are declared at the top level so React Query fires them concurrently. On a 3G Caribbean mobile connection sequential 200ms requests add up to 600ms+ of blocked render time.

### Memoize chart data transformations: ⚡ Medium Impact
Recharts BarChart/LineChart data props likely derive from raw API responses via inline array maps or filter+reduce expressions passed directly as JSX props. Wrap each transformation in useMemo keyed to the raw query data. This prevents re-running O(n) dataset transforms on every render triggered by unrelated state (e.g., search field changes) and avoids Recharts re-animating bars unnecessarily.
