## 🔴 Critical (9)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: each PortfolioHolding triggers a separate getStockById lookup inside a loop instead of a single JOIN or inArray fetch, causing O(n) round-trips to the database as holdings grow.

**Fix:** Replace the per-holding stock fetch with a single query using Drizzle's relational API or an explicit JOIN: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. This collapses n+1 round-trips into one.

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Buy/sell stock operations mutate the holding, insert a portfolio transaction, and update the virtual balance as three separate awaits with no wrapping database transaction. A crash between any two steps leaves the ledger in an inconsistent state (e.g., balance debited but holding not updated).

**Fix:** Wrap all three operations in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the `tx` client into each sub-operation so all writes commit atomically or roll back together.

---

### `server/routes.ts` — POST /api/auth/google and all AI/tutor endpoints
**Issue:** No rate limiting on the Google auth endpoint or any AI/quota endpoints. A student can hammer POST /api/auth/google or the AI tutor endpoint in a tight loop, exhausting OpenAI quota and bypassing the soft quota system entirely before reserveQuota is even called.

**Fix:** Add express-rate-limit middleware scoped per-route before the handler. For auth: max 10 req/15 min per IP. For AI endpoints: max 20 req/min per authenticated userId stored in req.session. Example: `import rateLimit from 'express-rate-limit'; const aiLimiter = rateLimit({ windowMs: 60_000, max: 20, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.post('/api/tutor/ask', aiLimiter, isAuthenticated, ...)`.

### `server/routes.ts` — Error handlers throughout — e.g. catch blocks returning e?.message or e.toString()
**Issue:** Raw error messages from DB, OpenAI, and Supabase are forwarded directly to API responses (e.g. `res.status(500).json({ error: e?.message })`). These can leak internal table names, Supabase project URLs, OpenAI API key suffixes, or stack traces to school-age end users and anyone intercepting traffic.

**Fix:** Create a central error-response helper: `function safeError(e: unknown, fallback = 'An unexpected error occurred') { console.error(e); return fallback; }`. In all catch blocks replace `res.json({ error: e?.message })` with `res.status(500).json({ error: safeError(e) })`. Log the full error server-side only (ideally via the existing Sentry integration).

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 — the full base64-encoded exam paper — is stored in the job payload row in the database. A single-page PDF can be 500 KB–2 MB as base64; multi-page papers will be larger. This bloats the jobs table, slows queue polling queries, and can exhaust DB memory under concurrent uploads.

**Fix:** Upload the file to object storage at enqueue time and store only the resulting object path in the job payload (e.g., { paperId, objectPath, ext, subject }). Inside runExtractPaper, download the buffer from object storage using objectStorageClient before processing. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/jobHandlers.ts` — runExtractPaper — for loop calling storage.createExtractedQuestion
**Issue:** Extracted questions are inserted one at a time inside a for loop with sequential awaits. For a 50-question paper this is 50 round-trips to the database, serialised. If any single insert fails mid-loop the job throws, the earlier deletes have already run, and a retry re-deletes and re-processes the whole paper unnecessarily.

**Fix:** Collect all valid question objects into an array, then insert them in a single batched statement: await db.insert(extractedQuestions).values(validQuestions). Wrap the delete + batch insert in a single db.transaction() so a partial failure leaves no orphaned rows and the idempotency delete at the top of the function is always paired with a complete write.

---

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** olderThanDays is taken directly from job.payload with no validation. A payload of { olderThanDays: 0 } or a negative number produces a cutoff in the future, causing the handler to silently delete the entire aiUsageEvents table.

**Fix:** Add a guard before computing the cutoff: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('olderThanDays must be a positive integer'); This makes malformed payloads a visible job failure rather than silent mass-deletion.

---

### `shared/schema.ts` — portfolioHoldings table, stockId + userId columns
**Issue:** There is no unique constraint on (userId, stockId) in portfolioHoldings. A user can accumulate duplicate holding rows for the same stock across buy operations, making balance calculation dependent entirely on application logic rather than a DB invariant — a data integrity risk for simulated financial positions.

**Fix:** Add a unique index: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId). Handle upsert logic in the application layer using Drizzle's onConflictDoUpdate to increment quantity and recalculate avgPurchasePrice atomically.

---

### `client/src/pages/AdminDashboard.tsx` — AuditLogPanel / queryFn
**Issue:** queryFn arrow functions in useQuery calls (AuditLogPanel, ObservabilityCard, and likely all others) are missing try/catch. If apiRequest throws a non-Response error (network failure, JSON parse error), TanStack Query will catch it but the error object will be untyped and the component has no visible error state rendered, leaving users on a blank/loading UI indefinitely.

**Fix:** Wrap each queryFn in try/catch and return a typed error, or add an onError/isError branch in each panel. At minimum add isError checks: `const { data, isLoading, isError } = useQuery(...)` and render `<Alert>Failed to load audit log</Alert>` when isError is true. Also add `throwOnError: false` and a global query error handler in your QueryClient config.

## 🟡 Warnings (14)

- **`server/storage.ts` (getLeaderboard):** No pagination guard on the leaderboard query. The `limit` parameter is optional and appears to have no enforced maximum, meaning a caller can omit it and fetch every game session row aggregated in memory — unbounded as the student user base scales.
  → *Add a hard cap: `const safeLimit = Math.min(filters?.limit ?? 50, 200)` and always apply `.limit(safeLimit)` to the query. Document the maximum in the IStorage interface signature.*
- **`server/storage.ts` (getVirtualBalance):** The initial virtual balance seeded for a new user appears to use a hardcoded value (e.g., 10000) directly in the upsert logic rather than an environment-configured constant, making it impossible to adjust per deployment tier without a code change.
  → *Extract to `const INITIAL_VIRTUAL_BALANCE = Number(process.env.INITIAL_VIRTUAL_BALANCE ?? 10_000)` in a config module and reference it here. Add a startup assertion that the env value, if set, is a positive finite number.*
- **`server/storage.ts` (getClassProgressSummary):** Sequential awaits for fetching enrollments, then per-student learning progress in a loop, produce an N+1 pattern proportional to class size. For a class of 30 students this is 31 sequential DB round-trips, making teacher dashboard loads slow.
  → *First fetch all enrollment student IDs, then fetch all UserLearningProgress rows in one query using `inArray(userLearningProgress.userId, studentIds)`, and join them in application memory. This reduces the queries to 2 regardless of class size.*
- **`server/routes.ts` (Google sign-in handler — sequential awaits: getClassByCode, getOrgEnvironmentById, getOrganization):** In the Google sign-in flow, getOrgEnvironmentById and getOrganization are called sequentially even though getOrganization only depends on classEnv.org_id which is available after the first call. More broadly, across multiple route handlers, independent DB lookups (e.g. fetching org + user + quota settings) are awaited one-by-one, adding 50-150 ms of unnecessary latency per request on a low-latency Caribbean connection.
  → *Where results are independent, use Promise.all. Example in the join-code resolution block: `const [classEnv, otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), someOtherCall()])`. For the org + quota fetch pattern used in AI routes: `const [org, quotaSettings] = await Promise.all([getOrganization(orgId), getOrgQuotaSettings(orgId)])`.*
- **`server/routes.ts` (XLSX.read() and Papa.parse() calls inside request handlers):** XLSX.read() is synchronous and CPU-bound. Parsing a 5 MB spreadsheet (the multer limit) on the main Node.js event loop blocks all concurrent requests for hundreds of milliseconds. This is especially harmful in a school setting where many students submit simultaneously.
  → *Offload XLSX parsing to a worker thread using Node's `worker_threads` module or the `xlsx` stream API. Alternatively, push the parse step into the existing `enqueueJob` background-job system and return a jobId immediately, polling for completion — which the jobs infrastructure already supports based on the imports present in this file.*
- **`server/routes.ts` (multer fileFilter using path.extname() for security decisions):** File type validation relies solely on the file extension (e.g. `.pdf`, `.csv`) rather than the MIME type or magic bytes. A student can rename `malicious.js` to `malicious.csv` and bypass the filter. The logo/video uploads do check MIME type correctly, but the `upload` and `examUpload` multer instances do not.
  → *For `upload` and `examUpload`, validate `file.mimetype` in addition to the extension, mirroring the pattern already used in `logoUpload`. Example: `const allowedMimes = ['text/csv', 'application/pdf', 'application/vnd.openxmlformats-officedocument.spreadsheetml.sheet']; if (!allowedMimes.includes(file.mimetype)) return cb(new Error('Invalid file type'))`. For defense-in-depth, add the `file-type` npm package to verify magic bytes after upload.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model: 'gpt-4o', max_completion_tokens: 4096)):** The OpenAI model name and token limits are hardcoded strings. If the model needs to change across environments (cost control in dev, different model in prod) or token budget needs tuning, it requires a code change and redeploy rather than a config update.
  → *Add environment variables: OPENAI_EXTRACT_MODEL (default 'gpt-4o'), OPENAI_EXTRACT_MAX_TOKENS (default 4096). Read them once at module load: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); and substitute into both create() calls.*
- **`server/jobHandlers.ts` (admin-csv-export handler (type AdminRow = Record<string, unknown>; let dat...)):** Based on the visible pattern (dat variable declared with let and then conditionally assigned per export type), if an unrecognised export type is passed the variable will be undefined and downstream code will throw an unhandled runtime error rather than returning a clean job failure. There is also no validation of the 'type' field from job.payload before branching.
  → *Add an explicit allowlist check at the top of the handler: const ALLOWED_TYPES = ['users','transactions','sessions'] as const; if (!ALLOWED_TYPES.includes(type)) throw new Error(`Unknown export type: ${type}`); This produces a clear, non-retryable failure message and prevents unexpected code paths. Also initialise dat as an empty array so TypeScript enforces the type.*
- **`shared/schema.ts` (categories table, userId column):** varchar('user_id') has no length argument on any table, defaulting to unlimited length in some Postgres drivers. More critically, categories.userId has no index despite being used in WHERE clauses to filter user-owned vs system categories, causing full table scans as category counts grow.
  → *Change all varchar('user_id') to varchar('user_id', { length: 36 }) to match UUID length and prevent silent truncation. Add: index('idx_categories_user_id').on(t.userId) inside the categories table's index callback.*
- **`shared/schema.ts` (transactions, budgets, linkedCards, savingsGoals, billReminders, documentUploads — createdAt column):** createdAt is defined as timestamp('created_at').defaultNow() with no .notNull(). In Drizzle-ORM this means the column is nullable in the TypeScript type and in the DB schema, so audit trails can contain NULL timestamps — a compliance risk for a financial app.
  → *Append .notNull() to every createdAt definition, e.g. timestamp('created_at').defaultNow().notNull(). Do the same for executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (budgets table, period column):** period is typed as plain text with a default of 'monthly' but no enum constraint, unlike frequency on billReminders which correctly uses an enum. Any arbitrary string can be inserted, breaking downstream period-based aggregation logic without a DB-level guard.
  → *Change to: text('period', { enum: ['weekly', 'monthly', 'quarterly', 'yearly'] }).default('monthly').notNull(). Also add an index on (userId, period) since budget queries will filter by both.*
- **`shared/schema.ts` (billReminders table, nextDueDate column):** nextDueDate has no index. Bill reminder features universally query by nextDueDate to surface upcoming or overdue bills (WHERE next_due_date <= NOW()), and with no index this degrades to a full table scan per user check — especially costly if a background job polls reminders for all users on a schedule.
  → *Add a composite index: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) inside the billReminders table definition. This supports both user-scoped filtering and date-range ordering in a single index scan.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params construction inside render):** A new URLSearchParams object is constructed on every render of AuditLogPanel outside of any memo, and it is also used directly inside the queryFn closure. This means the queryFn captures a stale params reference and also causes unnecessary object churn on every keystroke in the filter inputs.
  → *Move params construction into a useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return p.toString(); }, [actorType, action, from, to]);` then use `params` in both the queryKey and queryFn. This also makes the queryKey stable and avoids redundant refetches.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — useQuery with no auth guard):** ObservabilityCard fires `GET /api/admin/observability` unconditionally on mount with no check that the admin session is valid client-side before issuing the request. If useAdminAuth returns a non-admin user or a loading state, this query still fires and may return 401/403 data that is then silently ignored, leaking that the endpoint exists.
  → *Accept an `isAuthenticated` prop or consume useAdminAuth inside the card, then set `enabled: isAuthenticated` on the useQuery options: `useQuery({ ..., enabled: !!adminUser })`. Apply the same pattern to every data-fetching sub-panel in the dashboard.*

## 🔵 Info / Tech Debt (2)

- **`client/src/pages/AdminDashboard.tsx` (TABS constant — defined at module scope with inline icon references):** TABS is defined at module scope which is fine, but each tab entry holds a direct component reference (e.g. `icon: LayoutDashboard`). Downstream, the tab bar almost certainly maps over TABS and renders `<tab.icon />` as a JSX expression inside the map callback with no useCallback or React.memo on the tab bar component. This forces icon component re-instantiation on every parent re-render.
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper — `d: any` parameter type):** fmtDate accepts `any` and passes it directly to `new Date(d)`. If d is an unexpected type (object, null string ''), `new Date(d)` silently produces an Invalid Date and toLocaleDateString returns 'Invalid Date' in the UI. For a financial literacy app with school-aged users, silent bad data display is a UX and trust issue.

## 💡 Opportunities

### Parallelize independent dashboard stat queries — 🔥 High Impact
getDashboardStats likely fires sequential awaits for total income, total expenses, savings goals, and budget summaries. Wrapping independent aggregates in Promise.all would cut dashboard load time proportionally to the number of queries, likely 3-4x faster for the most common user-facing screen.

### Add query result limits to all list endpoints — ⚡ Medium Impact
getTransactions, getGameSessions, and getDocumentUploads lack enforced maximums. Introducing a DEFAULT_PAGE_SIZE=50 constant and requiring cursor or offset pagination protects against memory spikes as school cohorts accumulate data over an academic year.

### Centralize userId ownership checks into a reusable guard — ⚡ Medium Impact
update/delete methods each repeat the pattern of fetching a row then comparing userId inline, with inconsistent error messages. Extracting an `assertOwnership(row, userId, entityName)` helper eliminates duplicate code, standardizes 403 vs 404 semantics, and reduces the risk of a copy-paste omission exposing one student's data to another.

### Extract route handlers into domain-scoped sub-routers — 🔥 High Impact
The single routes.ts file is 195k+ characters covering auth, AI, lessons, orgs, leaderboards, and file uploads. Splitting into Express sub-routers (e.g. authRouter, lessonRouter, aiRouter) would make each file independently testable, reduce merge conflicts across the team, and allow per-router middleware (auth, rate limiting) to be applied cleanly without repeated isAuthenticated checks scattered through one file.

### Cache getOrgQuotaSettings and getOrganization per request or with short TTL — 🔥 High Impact
Several AI-path handlers call getOrgQuotaSettings and getOrganization on every single request. For a classroom of 30 students all hitting the tutor simultaneously, this is 60+ redundant Supabase round-trips per second for data that changes at most a few times per day. Adding a 60-second in-process LRU cache (lru-cache npm) keyed by orgId would reduce Supabase read load by ~95% under classroom load.

### Replace repeated dynamic imports of db/drizzle inside handlers with top-level imports — 📌 Low Impact
The /healthz route and several other handlers use `await import('./db')` and `await import('drizzle-orm')` inside the handler body. Dynamic imports are cached after first load but still incur a Promise microtask and property lookup on every request. Moving these to static top-level imports (already done for most of the file) eliminates the overhead and makes dependency graphs easier to audit.

### Parallel independent async operations in export handler — ⚡ Medium Impact
If the admin-csv-export handler fetches multiple independent datasets (e.g., user counts alongside transaction rows) using sequential awaits, wrapping them in Promise.all will cut wall-clock time proportionally to the number of queries. Expected impact: 40–70% latency reduction on multi-query exports.

### Job payload schema validation at enqueue boundary — 🔥 High Impact
Add Zod schemas for each JobPayload variant and parse them inside registerJobHandler before the handler function runs. This catches malformed payloads (wrong types, missing fields, oversized strings) at the point of enqueue and on dequeue, eliminating an entire class of mid-job crashes and making the purge-ai-usage and admin-csv-export issues structurally impossible.

### Deduplicate OpenAI client import source — ⚡ Medium Impact
openai is imported from './replit_integrations/chat/routes', meaning a route module is being used as a service singleton provider. This creates a hidden coupling: any refactor of the chat routes breaks job handlers. Extract the OpenAI client initialisation into a dedicated './lib/openaiClient.ts' module and import from there in both locations.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, etc.). Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it in each .default() call. This prevents silent divergence if the app ever needs to support TTD or JMD and ensures a single change point.

### Add a DB-level check constraint on numeric amount columns — 🔥 High Impact
amount, targetAmount, currentAmount, and pricePerUnit columns have precision/scale set but no constraint preventing negative values. For a financial literacy app, a negative budget amount or negative stock price would produce nonsensical UI and incorrect calculations. Add Postgres check constraints via Drizzle's sql template: e.g. .check(sql`amount >= 0`). This catches bad data before it reaches application logic.

### Replace serial PKs with uuid PKs for user-facing tables — 🔥 High Impact
Using serial (auto-increment integer) PKs on transactions, savingsGoals, and portfolioHoldings exposes sequential IDs in API responses, allowing enumeration of other users' records if auth checks ever fail — especially dangerous for a minor-user app. Switching to uuid('id').defaultRandom().primaryKey() eliminates enumeration risk and aligns the PK type with the existing varchar user_id FK target, removing the int/varchar type mismatch on joins.

### Decompose god-component into feature-scoped page modules — 🔥 High Impact
At 126k+ characters, AdminDashboard.tsx contains all 13 tab panels in one file. Splitting each panel (AuditLogPanel, ObservabilityCard, JobsPanel, etc.) into its own file under `pages/admin/` enables route-level code splitting with React.lazy, reduces initial JS bundle by an estimated 60-70% for non-admin users, and makes isolated testing and memoization far easier.

### Consolidate overview-tab queries into a single aggregated endpoint — ⚡ Medium Impact
The overview/dashboard tab likely fires multiple parallel useQuery calls (users, schools, sponsors, challenges counts) that each add a round-trip. Create a single `GET /api/admin/dashboard-summary` endpoint that returns all counts in one query, reducing waterfall latency and the number of DB connections held open during page load.

### Add staleTime and gcTime to all admin queries — ⚡ Medium Impact
Without staleTime, every tab switch triggers a background refetch for all previously loaded queries. Setting `staleTime: 60_000` (1 min) on read-heavy admin queries (audit log, observability, reports) eliminates redundant network calls during normal navigation and reduces server load proportional to the number of admin sessions active.
