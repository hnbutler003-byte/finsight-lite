## 🔴 Critical (5)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI, image, and audio routes are registered with no rate limiting middleware. These routes call OpenAI APIs which are expensive and latency-sensitive. A single unauthenticated or authenticated user can spam them, causing runaway API costs and degraded service for all school-aged users.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerX calls.

### `server/routes.ts` — /public-objects/:filePath(*) handler (~line 82)
**Issue:** The filePath parameter is passed directly to objectStorage.searchPublicObject without any sanitisation. A path like ../../private/config could traverse into unintended storage prefixes depending on the ObjectStorageService implementation. Additionally, there is no auth check — any unauthenticated user can enumerate and download all public objects, which may include sensitive org logos or unreleased content.

**Fix:** Sanitise the path before use: const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); and reject requests where safePath !== filePath. If any objects are org-specific, add a session check or sign URLs with short TTLs instead of serving through this open proxy.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full file content stored as base64 in the job payload. A 5 MB PDF becomes ~6.7 MB of base64 sitting in the jobs table row for every retry attempt, bloating the DB and making job queue queries slow as the table grows.

**Fix:** Upload the file to private object storage at enqueue time and store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, stream/download the object by path before processing. This keeps job rows small (<1 KB) and lets object storage handle large blobs.

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** simulatedStocks.ticker has no unique index. Nothing prevents two rows with ticker 'BISX:BPL' being inserted, which would corrupt portfolio value calculations and confuse any lookup-by-ticker logic. For a simulated exchange this is a seed-data and runtime integrity risk.

**Fix:** Add a uniqueIndex to the table definition: (t) => ({ tickerUnique: uniqueIndex('idx_simulated_stocks_ticker_unique').on(t.ticker) }). If tickers can be scoped by region or exchange prefix, make the unique constraint composite on (ticker, region) instead.

### `client/src/pages/AdminDashboard.tsx` — apiRequest mutation calls (useMutation handlers)
**Issue:** Mutation onSuccess/onError callbacks and any direct apiRequest calls inside event handlers lack try/catch. An unhandled promise rejection in a mutation (e.g. network timeout during a delete operation) will surface as an unhandled rejection in the browser rather than a controlled toast, breaking the UX silently for students or teachers.

**Fix:** Wrap apiRequest calls in mutationFn with try/catch and always provide onError to useMutation: `onError: (err) => toast({ title: 'Error', description: err.message, variant: 'destructive' })`. For any imperative async handlers outside useMutation, add try/catch blocks explicitly.

## 🟡 Warnings (12)

- **`server/routes.ts` (initSupabaseTables().then(...).catch(...) chain (~line 103)):** The seven Supabase init steps are chained sequentially with .then(). If seedFinancialAcademyLesson() rejects, the remaining four steps (initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent) are silently skipped. The single .catch only logs the first failure with no indication of which step failed, making debugging very difficult in production.
  → *Run independent seed steps in parallel and capture individual failures: Promise.allSettled([initStaticContentTables(), seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => { if (r.status === 'rejected') console.error(`[Supabase] Step ${i} failed:`, r.reason); })); Run initSupabaseTables and trimOrgNamesInSupabase first if they are true prerequisites, with their own try/catch.*
- **`server/routes.ts` (/healthz handler (~line 68)):** The /healthz endpoint performs dynamic imports of db and drizzle-orm on every health check request. Uptime monitors typically poll every 30–60 seconds, causing repeated module resolution overhead and a new db.execute() call each time with no caching or connection reuse guard.
  → *Import db and sql statically at the top of routes.ts alongside the other imports. The modules are already used elsewhere in the app. This eliminates dynamic import overhead and lets the connection pool reuse existing connections properly.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for loop, meaning 40 questions = 40 round trips to the DB serialised one after another. This is slow and holds the job worker thread longer than necessary.
  → *Collect all valid questions into an array, then do a single batched insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the batch insert and the updateExamPaper call in a transaction so partial inserts on crash are atomic and the existing idempotency delete is sufficient.*
- **`server/jobHandlers.ts` (runExtractPaper — idempotency delete at top of function):** The DELETE on extractedQuestions runs unconditionally on every attempt, including attempt 1 where there is nothing to delete. On retry this is correct, but on the first attempt it adds an unnecessary DB round-trip and masks any upstream bug that pre-inserted questions before the job ran.
  → *Gate the delete on job.attempts > 1 (pass attempts into runExtractPaper). Alternatively, move question insertion into a transaction with the status update so a mid-flight crash leaves no partial rows, making the delete unnecessary entirely.*
- **`server/jobHandlers.ts` (openai import — `import { openai } from './replit_integrations/chat/routes'`):** The OpenAI client is imported from a routes module, tightly coupling job infrastructure to HTTP route implementation details. If that routes file is ever split or the client instantiation changes, job handlers silently break. Additionally, any route-level middleware or side-effects in that file execute at import time inside the job worker.
  → *Extract the OpenAI client into a dedicated shared module (e.g. server/lib/openaiClient.ts) that exports a single configured instance. Both routes and job handlers import from that module. This is a one-file change with no behavioural impact.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no composite or individual indexes. Queries like 'fetch all budgets for a user' or 'find budget by user+category' — which will be the dominant read pattern — will result in full table scans as data grows.
  → *Add an index in the table's third argument: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). If a user can only have one budget per category per period, add a uniqueIndex on (userId, categoryId, period) to also enforce that constraint at the DB level.*
- **`shared/schema.ts` (billReminders table — nextDueDate column; savingsGoals table — deadline column):** billReminders.nextDueDate is the primary polling column for due-date queries ('show bills due this week') but has no index. Similarly, savingsGoals.deadline has no index for deadline-based filtering. Both will cause sequential scans on user-scoped date-range queries.
  → *For billReminders add: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). For savingsGoals add: index('idx_savings_goals_user_deadline').on(t.userId, t.deadline). Include userId in both to match the expected WHERE userId = ? AND nextDueDate <= ? query shape.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions — userId varchar columns):** All userId columns are declared as varchar() with no length argument. In PostgreSQL, bare varchar() is equivalent to text, so the intended length constraint (e.g., Clerk's 32-char user IDs) is never enforced at the DB level. This allows arbitrarily long strings into a foreign-key column and wastes storage on any index built over it.
  → *Declare a consistent explicit length matching your auth provider's ID format, e.g., varchar('user_id', { length: 64 }) across all tables. This enforces the constraint at the DB layer and improves index efficiency. Confirm the exact max length against Clerk or your auth provider's documented ID format.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** All audit timestamp columns use .defaultNow() without .notNull(). In Drizzle ORM this means the column is nullable in the TypeScript type and in the DB schema — a row can be inserted with an explicit NULL createdAt, bypassing the default. For audit trails in a financial app this is a data integrity gap.
  → *Chain .notNull() after .defaultNow() on every audit timestamp, e.g.: createdAt: timestamp('created_at').defaultNow().notNull(). This ensures the DB rejects any explicit NULL and the TypeScript type reflects non-nullability.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render inside AuditLogPanel without useMemo, causing a new object reference each render and making the queryKey comparison unstable, which can trigger redundant network fetches.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return p; }, [actorType, action, from, to]);` and use params.toString() in both queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** useQuery in ObservabilityCard has no error state and no loading state rendered in the UI. If the /api/admin/observability endpoint fails, all Badge and anchor elements silently render with undefined data, producing broken UI with no user feedback.
  → *Destructure `isLoading, isError` from useQuery and render a skeleton or error message: `if (isLoading) return <Loader2 className='animate-spin' />; if (isError) return <p className='text-red-400'>Failed to load observability data</p>;`*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and repeated per-tab useQuery calls):** Each tab panel appears to independently declare its own useQuery for overlapping data (e.g. schools, teachers, students counts likely fetched both in 'overview' and their respective tabs). This creates redundant parallel network requests for the same data with different query keys, wasting bandwidth and causing inconsistent loading states across tabs.
  → *Consolidate shared data into a single query with a shared queryKey such as `['/api/admin/stats']` for overview counts, and ensure per-entity queries (e.g. `['/api/admin/schools']`) are reused via `useQuery` with the same key in both the overview card and the schools tab so React Query's cache deduplicates them.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (duplicate health check routes (~line 41 and ~line 68)):** There are two health check endpoints: GET /api/health (returns {status, ts}) and GET /healthz (does a real DB probe). This is confusing — Railway and uptime monitors may be configured to hit either one, and /api/health gives a false positive because it never checks the database. This is dead/duplicate code with a reliability risk.
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls, max_completion_tokens: 4096):** max_completion_tokens is hardcoded to 4096 in two separate OpenAI calls. If the model or quota changes this requires a code deploy, and there is no environment-level knob to tune it per deployment (e.g. lower for cost control in staging).
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel queryFn):** The queryFn arrow function `() => apiRequest(...).then(r => r.json())` is defined inline without useCallback, meaning a new function reference is created on every render. While React Query does not use referential equality for queryFn, this pattern combined with the unstable queryKey (see finding 1) compounds unnecessary refetch cycles.

## 💡 Opportunities

### Parallelise sequential domain-router registrations — ⚡ Medium Impact
The six await registerXRoutes(app) calls after the parallel import block run sequentially. If any of these registration functions perform async I/O (DB schema checks, cache warming), they add cumulative cold-start latency. Wrap them in Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) to cut cold-start time proportionally to the slowest individual registrar instead of their sum.

### Structured logging for Supabase init errors — ⚡ Medium Impact
All Supabase init errors are logged with console.error and a plain string prefix. Integrating with the existing Sentry setup (sentryRequestContext is already imported) and using captureException on init failures would surface seed errors in the Sentry dashboard with full stack traces and context, replacing silent log lines that are easy to miss in Railway log streams.

### Cache sitemap.xml response — 📌 Low Impact
The sitemap is rebuilt on every request including string concatenation and header assembly. Since the URL list is static at deploy time, add res.setHeader('Cache-Control', 'public, max-age=86400') and consider generating the XML once at startup into a module-level constant. This eliminates unnecessary per-request string allocation for a route that crawlers hit repeatedly.

### Parallel admin CSV data fetches — ⚡ Medium Impact
The admin-csv-export handler (visible in the truncated section) almost certainly fetches multiple tables sequentially. Replacing sequential awaits with Promise.all for independent queries (e.g. fetching users, transactions, and sessions simultaneously) would cut export job duration proportionally to the number of queries, likely 2-4x faster for typical export types.

### Job payload schema validation at enqueue boundary — 🔥 High Impact
There is no visible Zod or similar validation on job.payload before handlers destructure it. Adding a per-job-type payload schema validated at registerJobHandler time would catch malformed enqueue calls at the source rather than producing cryptic runtime errors mid-job, improving debuggability and preventing silent data corruption from missing fields like paperId.

### Extracted text truncation limit as named constant — 📌 Low Impact
extractedText is sliced to 30000 characters (hardcoded). Centralising this as EXTRACTED_TEXT_CHAR_LIMIT in a config or constants file makes it discoverable, tunable per subject type, and ensures the vision path (which currently has no truncation) can apply the same limit consistently — preventing unexpectedly large prompts from blowing token budgets.

### Currency as enum column instead of free-text default — ⚡ Medium Impact
At least six tables store currency as text().default('BSD'). There is no constraint preventing 'bsd', 'XCD', or garbage values from being written. Replacing with a pgEnum(['BSD','TTD','JMD','XCD','USD']) shared enum type would enforce valid Caribbean currency codes at the DB level, eliminate silent data inconsistencies in multi-currency reports, and make schema intent explicit — with zero application-code overhead.

### Extract shared period/frequency enums to avoid divergence — ⚡ Medium Impact
budgets.period is a plain text() with no enum constraint while billReminders.frequency uses a proper text enum. If a new period value is added to one table it will silently not apply to the other, breaking any shared reporting logic. Define a single pgEnum for recurring periods and reuse it across both tables.

### Add partial index on documentUploads.status for processing-queue queries — 📌 Low Impact
Background jobs polling for documents with status='processing' will scan the entire documentUploads table. A partial index — index('idx_doc_uploads_processing').on(t.userId, t.createdAt).where(sql`status = 'processing'`) — would make the polling query O(pending rows) instead of O(total rows), which matters as completed uploads accumulate over time.

### Parallel data fetching with Promise.all on dashboard load — 🔥 High Impact
The overview tab likely fires sequential or independent useQuery calls for organizations, schools, teachers, students, and sponsors counts. Consolidating these into a single `/api/admin/stats` endpoint that returns all counts in one round trip (or using Promise.all on the server side) would reduce dashboard initial load time by 4-5 round trips, critical on Caribbean mobile networks with higher latency.

### Virtualise long audit log and student/teacher lists — ⚡ Medium Impact
AuditLogPanel fetches up to 200 rows and renders them all to the DOM. For school-scale deployments with thousands of audit entries, replacing the list with a virtualised renderer (e.g. TanStack Virtual) would cut initial paint time and reduce memory pressure significantly, especially on low-end school devices.

### Extract sub-panels into separate lazy-loaded route components — 🔥 High Impact
All 13 tab panels appear to be defined in a single file exceeding 126k characters. Code-splitting each tab (Organizations, Students, Audit Log, etc.) with React.lazy and Suspense would reduce the initial JS bundle by an estimated 60-70%, improving time-to-interactive for school-aged users on slower connections.
