## 🔴 Critical (6)

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 110)
**Issue:** AI and image/audio routes are registered with no rate limiting. For a school-aged user base these endpoints call OpenAI which is expensive and abuse-prone. A single user or bot can exhaust API quota or trigger runaway costs.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the registerChat/Image/Audio calls.

### `server/routes.ts`: robots.txt handler (~line 47) and sitemap.xml handler (~line 62)
**Issue:** The base URL is constructed directly from `x-forwarded-host` and `host` request headers without any validation or allowlist. An attacker can send a crafted Host header to make the sitemap and robots.txt emit an arbitrary URL (e.g. `Sitemap: https://evil.com/sitemap.xml`), which some crawlers will follow.

**Fix:** Read the canonical origin from an environment variable: `const base = process.env.CANONICAL_ORIGIN || 'https://finsightlite.com';` and ignore the request headers entirely for URL construction in these two handlers.

---

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored directly in the job table payload. A single A4 exam image can easily be 2-5 MB encoded, and PDFs larger still. This bloats the jobs table, slows queue polling queries, and risks hitting row-size limits in Postgres.

**Fix:** Upload the file to object storage before enqueuing the job and store only the storage path in the payload. In runExtractPaper, download the file from object storage using the path instead of decoding fileB64. Remove fileB64 from JobPayloads['extract-paper'] entirely.

---

### `server/jobHandlers.ts`: runExtractPaper — for loop over questions with sequential awaits
**Issue:** Each extracted question is inserted with an individual await storage.createExtractedQuestion(...) inside a for loop. For a 60-question paper this means 60 sequential round-trips to the database, adding hundreds of milliseconds of unnecessary latency and holding the DB connection for the full duration.

**Fix:** Replace the for loop with a single bulk insert: await db.insert(extractedQuestions).values(validQuestions.map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0, 4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i + 1 }))); This also makes the write atomic so partial inserts cannot occur between the delete and the status update.

---

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** ticker has no unique constraint. Duplicate tickers in a financial simulation context will cause incorrect portfolio valuations when looking up current prices by ticker symbol, and nothing at the DB level prevents a seed script or admin insert from creating duplicates.

**Fix:** Replace the plain text definition with: ticker: text('ticker').notNull().unique(). Also add an index if price lookups by ticker are frequent: (t) => ({ tickerIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }).

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage
**Issue:** The full `messages` array is sent to /api/admin/help-chat on every request with no size cap. A user who chats long enough will send an ever-growing payload, and the server receives the entire conversation history, which may include sensitive admin data typed earlier in the session. There is also no rate-limit guard on the client side.

**Fix:** Slice the history to the last N turns before sending: `const context = next.slice(-10)` and send `context` instead of `next`. Add a debounce or a `loading` guard (already present) and document the server-side token limit. Consider stripping any message content that matches sensitive patterns before transmission.

## 🟡 Warnings (13)

- **`server/routes.ts` (Supabase init chain (~line 113)):** The seven sequential Supabase init/seed steps are chained with `.then()` and share a single `.catch()`. If `initSupabaseTables` succeeds but `seedStaticModules` fails, the error log says '[Supabase] Init error' with no indication of which step failed or what completed, making production debugging very hard. Steps that are independent also run sequentially when they could run in parallel.
  → *Wrap each step individually and group independent seeds with Promise.all: `initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()])).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));` and add the step name to each catch for diagnostics.*
- **`server/routes.ts` (app.get('/api/health') and app.get('/healthz') (~lines 34 and 79)):** There are two health-check endpoints doing different things. `/api/health` returns a shallow response with no DB check; `/healthz` does a real DB probe. The comment on `/api/health` says it is used by Railway and uptime monitors, but it gives a false positive when the DB is down. Having two endpoints also creates confusion about which one is authoritative.
  → *Remove `/api/health` and point Railway/monitors to `/healthz` exclusively, which already has a try/catch and returns 503 on DB failure. If a no-DB ping is needed for a CDN layer, name it `/api/ping` and document the distinction explicitly.*
- **`server/routes.ts` (app.get('/public-objects/:filePath(*)') (~line 98)):** There is no authentication or authorization check before serving objects from object storage. Any unauthenticated request can enumerate and download files by guessing paths. For a school-aged-user platform storing logos and videos this may be intentional for truly public assets, but there is no enforced boundary between public and private objects — both are served through the same unguarded route.
  → *Split into two routes: keep `/public-objects/` for explicitly public buckets/prefixes (logos, marketing videos) and add `/private-objects/` that requires a valid session middleware check before calling `objectStorage.downloadObject`. Add a prefix allowlist check in the public handler: `const ALLOWED_PUBLIC_PREFIXES = ['logos/', 'videos/']; if (!ALLOWED_PUBLIC_PREFIXES.some(p => filePath.startsWith(p))) return res.status(403).json({ error: 'Forbidden' });`*
- **`server/jobHandlers.ts` (runExtractPaper — model and token constants):** The OpenAI model name 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in two separate openai.chat.completions.create calls. Changing the model for cost or compliance reasons requires a code deploy rather than a config change, and the Caribbean school context may require a different default depending on budget.
  → *Add OPENAI_MODEL (default 'gpt-4o') and OPENAI_MAX_TOKENS (default '4096') to environment config and read them as const model = process.env.OPENAI_MODEL ?? 'gpt-4o' and const maxTokens = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10) at the top of the handler, then reference these in both API calls.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** PDF text is silently truncated to 30,000 characters with no log or warning. For a dense Caribbean CXC past-paper this can silently drop entire question sections, causing the job to succeed with missing questions and no indication to the teacher that the paper was partially processed.
  → *Before slicing, check if pdfData.text.length > 30000 and if so log a structured warning including paperId and the actual length. Also store a truncated: true flag on the exam paper record so the UI can surface a warning to the teacher. Consider raising the limit or chunking rather than truncating.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The purge-ai-usage handler has no try/catch and no validation that olderThanDays is a positive finite number. A malformed or missing payload value would produce a NaN cutoff date, and the resulting DELETE WHERE createdAt < 'Invalid Date' would silently delete zero rows or, depending on the driver, throw an unhandled rejection that crashes the worker process.
  → *Add input validation at the top of the handler: const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days <= 0) throw new Error('Invalid olderThanDays: ' + job.payload.olderThanDays); Then wrap the delete in a try/catch that re-throws after logging so the job framework can record the failure correctly.*
- **`shared/schema.ts` (categories table definition):** categories.userId has no index, yet every category lookup for a user (e.g. populating dropdowns, budget forms) will filter on this column. With school-aged users creating many custom categories, this becomes a full table scan.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_categories_user_id').on(t.userId) }). Since a user should never see duplicate category names, also consider a uniqueIndex on (userId, name) to enforce uniqueness at the DB level.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, yet every budget dashboard query filters by userId. Without an index, each budget page load scans the entire budgets table.
  → *Add indexes in a third argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId), userCategoryIdx: uniqueIndex('idx_budgets_user_category').on(t.userId, t.categoryId) }). The unique index also prevents duplicate budgets per category per user, which is a logical constraint that should be enforced.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, linkedCards.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** All audit/timestamp columns use .defaultNow() without .notNull(). This means a failed default or direct SQL insert can leave these NULL, making audit trails and ORDER BY createdAt unreliable for financial records in a regulated-adjacent context.
  → *Chain .notNull() after .defaultNow() on every audit timestamp, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for purchasedAt and executedAt in portfolio tables.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on nextDueDate or userId. A scheduled job or dashboard query that fetches all upcoming bills for a user (WHERE userId = ? AND nextDueDate <= ?) will perform a sequential scan, which is particularly painful if reminder checks run on a cron.
  → *Add (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) as the third argument to pgTable. This supports both per-user listing and date-range reminder queries efficiently.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages.map render, scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is recreated on every render and calls scrollIntoView unconditionally, firing a forced layout/scroll every time any state changes in the parent (e.g. typing in the input box triggers a scroll attempt).
  → *Replace the inline ref with a `useRef` + `useEffect` that only scrolls when `messages` or `loading` changes: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading]);` and render `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard data fetches (useQuery calls for users, schools, lessons, analytics)):** Multiple independent useQuery calls are defined sequentially. While React Query fires them concurrently by default, any derived computations that depend on all results use them in sequence without a combined loading/error guard, so partial data can reach rendering logic and cause runtime errors or misleading UI.
  → *Add a combined loading and error state: `const isLoading = usersQuery.isLoading || schoolsQuery.isLoading || lessonsQuery.isLoading; const isError = usersQuery.isError || schoolsQuery.isError || lessonsQuery.isError;` and render a single skeleton/error boundary before any derived JSX that consumes these results. Use `useQueries` from React Query for cleaner parallel orchestration.*
- **`client/src/pages/AdminDashboard.tsx` (Mutation handlers (deleteUser, deleteSchool, etc.) – useMutation onSuccess/onError):** Destructive mutations (delete user, delete school) do not perform any client-side confirmation of admin privileges before firing. If `useAdminAuth` returns a stale or unauthenticated state (e.g. token expired mid-session), the mutation fires and the server is the only guard. A failed server auth returns an error but the optimistic UI update may already have removed the row.
  → *Before calling any destructive mutate(), assert the auth state: `if (!adminUser) { toast({ title: 'Session expired', variant: 'destructive' }); return; }`. Also ensure mutations do not apply optimistic updates for delete operations — use `onSettled` to always refetch rather than manually removing items from cache before server confirmation.*

## 💡 Opportunities

### Parallel domain-router registration: ⚡ Medium Impact
The six domain routers are imported in parallel with Promise.all (good) but then awaited sequentially one by one. If any registration function does async I/O (DB schema checks, etc.) these could also be run in parallel with Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]). Verify there are no ordering dependencies, then collapse to a single await to cut cold-start time proportionally to the slowest individual registration.

### Structured logging for init errors: ⚡ Medium Impact
All Supabase init errors are logged with console.error and a plain string prefix. Replacing with a structured logger (e.g. pino) and emitting JSON with `{ step, error: e.message, stack: e.stack }` would make Railway/Sentry log aggregation searchable and would allow alerting on specific seed failures without parsing free-form strings.

### Cache-Control header on public object route: 📌 Low Impact
The `downloadObject` call passes a 30-day TTL but there is no `Cache-Control` header set at the Express layer before the stream begins. If `downloadObject` does not set it internally, CDNs and browsers get no caching directive. Adding `res.setHeader('Cache-Control', 'public, max-age=2592000, immutable')` before the download call ensures edge caches honour the intent and reduces origin load for static assets like logos served to every student on load.

### Parallel OpenAI calls for vision + extraction pipeline: ⚡ Medium Impact
The two openai.chat.completions.create calls in runExtractPaper are strictly sequential even though the second depends on the first. There is no parallelism opportunity there, but the updateExamPaper status write and any post-processing side-effects (badge award, XP grant visible in the truncated section) can be fanned out with Promise.all after the bulk insert completes, reducing end-to-end job time by the latency of whichever call finishes last.

### Idempotency token on bulk question insert: ⚡ Medium Impact
The current idempotency strategy (delete then re-insert) creates a window where a crash leaves zero questions in the table and the paper shows 'processing' until a retry. Replacing it with an INSERT ... ON CONFLICT DO NOTHING keyed on (paperId, order) eliminates the delete entirely and makes retries safe without a visibility gap.

### Stream PDF parse instead of loading full buffer into memory: 🔥 High Impact
Buffer.from(fileB64, 'base64') for a large PDF allocates the full decoded buffer in the Node.js heap synchronously. Once object storage paths replace the base64 payload (per finding 1), the file can be streamed directly from object storage into pdf-parse using its stream API, keeping heap pressure flat regardless of file size. This matters on the constrained Replit runtime.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards inferred). Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it in each .default() call. This prevents drift if the app ever needs to support TTD or JMD and makes a future currency migration a one-line change.

### Add composite index for portfolio performance queries: 🔥 High Impact
portfolioHoldings and portfolioTransactions both filter on (userId, stockId) for P&L calculations. Adding a composite index on each — (userId, stockId) for holdings and (userId, stockId, executedAt) for transactions — will make portfolio summary queries significantly faster as user portfolios grow, without requiring any application code changes.

### Enforce learningModules.slug uniqueness at DB level with an index: 📌 Low Impact
slug is marked .unique() which creates a constraint but not necessarily an index in all Drizzle/PG configurations. Explicitly declare a uniqueIndex('idx_learning_modules_slug').on(t.slug) in the table indexes to guarantee both the constraint and O(log n) lookups when resolving module routes by slug, which is the primary access pattern for the learning content pages.

### Virtualise long admin tables: 🔥 High Impact
User, school, and lesson tables render all rows into the DOM and rely on CSS overflow scroll. With hundreds of records this causes significant paint and memory cost. Replacing with TanStack Virtual (already a transitive dependency via TanStack Query) would keep DOM node count constant and cut initial render time by ~70% for large datasets.

### Extract tab panels into lazy-loaded route components: 🔥 High Impact
The entire AdminDashboard is one monolithic component loading Recharts, VideoField, and all admin sub-sections eagerly. Splitting each tab (Users, Schools, Lessons, Analytics, Chat) into React.lazy() chunks behind Suspense would cut the initial JS parse cost for the admin route and improve Time-to-Interactive for the most common landing tab.

### Centralise hardcoded API path strings into a config constant: ⚡ Medium Impact
API paths like '/api/admin/help-chat', '/api/admin/users', '/api/admin/schools' are scattered as string literals throughout the file. A single `API_ROUTES` constant object would prevent typo-driven 404s, make base-URL changes a one-line edit, and enable TypeScript autocompletion on route names.
