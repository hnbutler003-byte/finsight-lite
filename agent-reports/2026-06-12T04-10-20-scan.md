## 🔴 Critical (9)

### `server/routes.ts` — registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 112-114)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These call OpenAI APIs and are the most expensive endpoints in the app. A school-aged user (or an attacker) can hammer them without restriction, incurring unbounded API costs and latency spikes.

**Fix:** Add a dedicated AI limiter (e.g., 20 req/min per IP) and apply it before these registrations: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — define aiLimiter in rateLimiter.ts alongside the existing limiters.

### `server/routes.ts` — robots.txt handler (~line 48) and sitemap.xml handler (~line 62)
**Issue:** The absolute base URL is constructed directly from `x-forwarded-host` and `x-forwarded-proto` request headers without any validation or allowlist. An attacker can set these headers to an arbitrary value, causing the app to emit a Sitemap or canonical URL pointing to a malicious domain — a header injection / open-redirect vector.

**Fix:** Read the canonical origin from a single environment variable set at deploy time: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` — remove all header-derived URL construction from these two handlers.

### `server/jobHandlers.ts` — runExtractPaper / extract-paper handler
**Issue:** Full base64 file content (fileB64) is stored directly in the job payload, meaning every queued job writes potentially several MB into the jobs table. For a school-aged app with many uploads this will bloat the DB, slow queue polling queries, and risk hitting row-size limits.

**Fix:** Upload the file to object storage before enqueuing, then store only the resulting object path in the payload: `const objectPath = await uploadPrivateObject(relPath, buffer, contentType); enqueueJob('extract-paper', { paperId, objectPath, ext, subject })`. Inside runExtractPaper, download the buffer from object storage using that path instead of decoding fileB64.

### `server/jobHandlers.ts` — admin-csv-export handler
**Issue:** The job handler executes privileged bulk data exports (users, transactions, etc.) based solely on a `type` field from the job payload with no role or permission check inside the handler itself. If any code path can enqueue this job without proper upstream auth, it will silently export sensitive PII. Logs or error messages could also expose raw DB row content.

**Fix:** Add a `requestedByUserId` and `requestedByRole` field to the job payload at enqueue time (populated from the authenticated session), then verify at the top of the handler: `if (job.payload.requestedByRole !== 'admin') throw new Error('Unauthorized export attempt by user ' + job.payload.requestedByUserId);` This makes the auth check durable even if the job is replayed or enqueued via a background process.

### `shared/schema.ts` — categories table definition
**Issue:** userId in categories uses bare varchar() with no length, but more critically there is no index on userId even though row-level filtering by user is the primary access pattern for user-owned categories. Every query like WHERE user_id = $1 OR user_id IS NULL will do a full table scan.

**Fix:** Add a composite index: index('idx_categories_user_id').on(t.userId) in the table's index callback. Also explicitly size the varchar to match the auth users.id column (e.g. varchar('user_id', { length: 256 })) to prevent silent truncation if the auth provider uses long IDs (e.g. Clerk/Auth0 UUIDs or JWTs).

### `shared/schema.ts` — transactions, budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions tables
**Issue:** createdAt timestamp columns across all tables use .defaultNow() but are NOT marked .notNull(). This means the column is nullable in the Postgres schema, so any INSERT that explicitly passes null will succeed and leave audit rows with no timestamp, breaking any time-series reporting or compliance logging required for a financial app.

**Fix:** Chain .notNull() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Also apply the same fix to executedAt in portfolioTransactions and purchasedAt in portfolioHoldings since those record financial simulation events.

---

### `server/index.ts` — seedDatabase / probeDatabase call site in startup IIFE
**Issue:** seedDatabase() and probeDatabase() are awaited sequentially before the HTTP server starts listening, meaning any slow DB probe or seed operation blocks the process from accepting health-check requests. On platforms like Replit that probe /health immediately after spawn, this causes false-negative health failures and slow cold starts.

**Fix:** Start listening first, then run probeDatabase and seedDatabase in the background: httpServer.listen(port, host, () => log('listening')); probeDatabase().then(() => seedDatabase()).catch(err => { captureError(err); process.exit(1); }); Gate the /health route to return 503 until a `dbReady` flag is set true after probeDatabase resolves.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles are never stored or cleared. If the module is re-evaluated (e.g. hot reload, test teardown) new intervals stack on top of old ones causing duplicate job enqueues and a memory/handle leak. There is also no concurrency guard: if a scheduler invocation takes longer than the interval period, the next tick fires a second concurrent execution.

**Fix:** Store handles: const digestInterval = setInterval(...). Add a boolean semaphore per scheduler (let digestRunning = false; if (digestRunning) return; digestRunning = true; try { await ... } finally { digestRunning = false; }). Export a shutdown() function that calls clearInterval(digestInterval) and clearInterval(purgeInterval), then call it inside the SIGTERM/SIGINT handler before process.exit.

---

### `server/index.ts` — graceful shutdown handler (SIGTERM/SIGINT)
**Issue:** There is no SIGTERM or SIGINT handler. On container/Replit shutdown the process is killed hard, dropping in-flight HTTP requests, leaving the job worker mid-task, and skipping any DB connection pool drain. This risks data corruption for any job that writes transactionally.

**Fix:** Add: process.on('SIGTERM', shutdown); process.on('SIGINT', shutdown); async function shutdown() { log('shutting down'); clearInterval(digestInterval); clearInterval(purgeInterval); await stopJobWorker(); await new Promise(r => httpServer.close(r)); await db.$client.end(); process.exit(0); } Give it a hard timeout (setTimeout(() => process.exit(1), 10_000)) to avoid hanging forever.

## 🟡 Warnings (9)

- **`server/routes.ts` (Supabase init chain (~line 117-125)):** The seven-step Supabase init chain uses sequential `.then()` calls. If any step throws a non-Error value or a step silently resolves without completing work, subsequent seeds still run against a partially initialised schema. The single `.catch` at the end is the only safety net, and it only logs — the server continues serving with potentially broken DB state.
  → *Wrap the chain in an async IIFE with explicit per-step error handling, or at minimum await them sequentially inside a try/catch that sets a module-level `dbReady` flag checked by a middleware that returns 503 until init completes. Consider whether seedRegionalContent and seedGameContent are safe to run in parallel with Promise.all after the schema init steps finish.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop over questions):** Questions are inserted one at a time with sequential awaits inside a for-loop (`for (let i = 0; i < questions.length; i++) { await storage.createExtractedQuestion(...) }`). For a 60-question paper this issues 60 serial round-trips to the DB, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy.
  → *Replace the loop with a single batched insert: `const rows = questions.filter(q => q.question && ...).map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0,4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i+1 })); if (rows.length) await db.insert(extractedQuestions).values(rows);` This reduces N round-trips to 1.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays comes directly from job.payload with no validation before being used in arithmetic (`olderThanDays * 24 * 60 * 60 * 1000`). A malformed or malicious payload (NaN, negative, 0) could produce a cutoff date in the future and delete all AI usage records, or produce Invalid Date silently.
  → *Add an explicit guard at the top of the handler: `const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days <= 0) throw new Error('olderThanDays must be a positive number');` so the job fails fast and is logged rather than silently corrupting data.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The streaming helper attaches an error handler on the ReadStream but never calls `stream.destroy()` if the client disconnects mid-transfer. The underlying GCS read stream will continue reading and buffering data from the network even after the HTTP response is gone, causing a resource/memory leak under concurrent downloads.
  → *Capture the stream and listen for the response close event: `const stream = file.createReadStream(); res.on('close', () => stream.destroy()); stream.on('error', (err) => { if (!res.headersSent) res.status(500).json({ message: err.message }); }).pipe(res);`*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId even though every budget query will filter by user. With multiple budget periods per user and category, a full table scan on every dashboard load is guaranteed at even modest user counts.
  → *Add an index callback to the budgets table: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The composite index also supports the common query of fetching a specific category budget for a user.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker column is declared notNull and is logically a unique business key, but has no unique constraint and no index. Duplicate tickers can be inserted silently, which will corrupt portfolio calculations when joining portfolioHoldings to simulatedStocks by stockId after a double-insert seed or admin error.
  → *Replace the plain index with a uniqueIndex: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). This enforces uniqueness at the DB level and doubles as the lookup index when seeding or updating prices by ticker symbol.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate has no index. Reminder systems invariably query WHERE next_due_date <= NOW() AND user_id = $1 to find upcoming or overdue bills. Without an index this is a full scan on every scheduled job run and every dashboard render, and performance degrades linearly with total reminder rows across all users.
  → *Add an index callback: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). The composite preserves the user-scoping that prevents cross-user data leakage in the query planner and supports ORDER BY nextDueDate efficiently.*
- **`server/index.ts` (request logging middleware — capturedJsonResponse serialization):** The res.json monkey-patch captures and JSON.stringifies the entire response body into the log line unconditionally. For API responses that include user PII, session tokens, or AI-generated content this leaks sensitive data into stdout/log aggregators. In a school-aged-user context this is a COPPA/data-protection concern.
  → *Allowlist safe fields or truncate: logLine += ` :: ${JSON.stringify(capturedJsonResponse).slice(0, 120)}`; Better: only log non-sensitive keys (e.g. { status, count, error }) by having API handlers attach res.locals.logMeta with pre-sanitised data and reading that instead of the raw body.*
- **`server/index.ts` (getMostRecentSundayAt7pm — daysSince calculation):** const daysSince = (dow + 0) % 7 is a no-op (adding 0). When dow === 0 (Sunday) daysSince is 0, so d is set to today at 19:00. If it is currently Sunday before 19:00 the guard `if (d.getTime() > now.getTime())` correctly backs off 7 days, but the comment '+ 0' suggests a deleted offset (likely was dow + 1 for Monday-anchor logic). The result is that the scheduler targets Sunday correctly by accident, but the dead arithmetic will confuse future maintainers and may break if the target day changes.
  → *Remove the dead offset and document intent explicitly: const daysSinceSunday = dow; // getDay() returns 0 for Sunday. Add a unit test covering Sunday-before-7pm, Sunday-after-7pm, and mid-week cases so the boundary is regression-proof.*

## 💡 Opportunities

### Centralise environment-driven base URL config — 🔥 High Impact
Replace all header-derived URL construction (robots.txt, sitemap.xml, and any OAuth redirect URIs) with a single APP_BASE_URL env var validated at startup via zod or a manual check that throws if missing in production. This eliminates the header injection risk and makes environment promotion (staging → prod) explicit.

### Add a db-readiness middleware gate — ⚡ Medium Impact
Introduce a module-level `isDbReady` boolean set to true only after the Supabase init chain completes successfully. Register a middleware early in the stack that returns HTTP 503 with Retry-After for any /api/* request while the flag is false. This prevents serving requests against an uninitialised schema during cold start and gives load balancers a clean signal.

### Serve sitemap and robots.txt from static files or a CDN cache layer — 📌 Low Impact
Both documents are reconstructed on every request. Moving them to static files generated at build time (with APP_BASE_URL injected via envsubst) or caching the response with a Cache-Control: max-age=86400 header eliminates per-request string allocation and header parsing overhead, which matters under crawler bursts.

### Batch AI extraction with structured output validation — ⚡ Medium Impact
The two sequential openai.chat.completions.create calls (text extraction then structured extraction) could be collapsed into one call for image inputs since gpt-4o can return structured JSON directly from vision input. This halves AI latency and token cost per paper upload, which matters at scale for a school app with budget constraints.

### Job payload schema validation with Zod at handler entry — ⚡ Medium Impact
All job handlers trust job.payload shape implicitly. Adding a per-handler Zod schema parsed at the top of each handler (throwing on failure) would catch malformed payloads before they cause partial mutations or cryptic runtime errors, and would make retry behavior deterministic since schema failures would not be retried as transient errors.

### Stream CSV exports directly to object storage — 🔥 High Impact
The admin-csv-export handler appears to build full CSV strings in memory before uploading. For large schools with thousands of transaction rows this risks OOM on the worker process. Switching to a streaming CSV serializer (e.g. csv-stringify in streaming mode) piped directly into the object storage upload stream would cap memory usage to a fixed buffer regardless of row count.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract const DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it in each .default() call. When the app expands to TTD or JMD support, this is a single-line change instead of a grep-and-replace across the schema with risk of missing one.

### Add partial index for active document upload processing queue — ⚡ Medium Impact
documentUploads is queried by status = 'processing' on every polling cycle to find jobs to process. A standard index on (userId, status) stores all statuses but the hot query only needs the 'processing' subset. A partial index CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing' is smaller, stays in buffer cache longer, and speeds up the job queue scan. Drizzle supports this via .where() on the index definition.

### Replace serial primary keys with UUIDs for external-facing IDs — 🔥 High Impact
All tables use serial (auto-increment integer) PKs. For a financial literacy app with school-aged users, sequential integer IDs in URLs or API responses allow trivial enumeration of other users' transactions, documents, and savings goals (IDOR risk). Switching to uuid with gen_random_uuid() as the default eliminates enumeration while also making future horizontal sharding or multi-region replication straightforward. Drizzle-orm/pg-core exports uuid() for this purpose.

### Parallelize independent startup I/O with Promise.all — 🔥 High Impact
registerRoutes(), probeDatabase(), and registerJobHandlers() have no interdependency yet are awaited sequentially. Running them with Promise.all([probeDatabase(), registerRoutes(app), registerJobHandlers()]) would cut startup latency by the sum of the two slowest minus the slowest — typically saving 200-800 ms on cold start depending on DB round-trip time.

### Extract scheduler logic into a dedicated SchedulerService class — ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, and their intervals into a SchedulerService with start()/stop() methods makes unit testing trivial (inject a fake clock), eliminates the global interval leak risk, and gives a single shutdown hook to call. Expected impact: eliminates duplicate-run bugs in staging where nodemon restarts the module repeatedly.

### Replace hardcoded 10 mb JSON limit and interval periods with validated env config — ⚡ Medium Impact
express.json limit '10mb', the 60-minute digest interval, and the 24-hour purge interval are magic literals. Moving them to MAX_REQUEST_BODY_MB, DIGEST_INTERVAL_MINUTES, PURGE_INTERVAL_HOURS env vars (with the same IIFE validation pattern already used for AI_PURGE_OLDER_THAN_DAYS) allows ops tuning without code changes and makes the values visible in deployment config audits.
