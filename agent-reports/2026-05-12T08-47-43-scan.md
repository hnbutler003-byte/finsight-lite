## 🔴 Critical (8)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: each holding row triggers a separate getStockById lookup inside a loop, so a user with 20 holdings executes 21 queries instead of 1 JOIN.

**Fix:** Replace the loop with a single Drizzle query using .leftJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)) and return the joined result directly. This collapses N+1 into a single round-trip.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (buy/sell flow)
**Issue:** Virtual balance deduction and portfolio holding upsert are executed as separate awaits with no wrapping database transaction. A process crash between the two steps leaves the user's balance debited but the holding never created (or vice versa), corrupting financial state.

**Fix:** Wrap the entire trade sequence in db.transaction(async (tx) => { ... }) using the Drizzle transaction client so both writes either commit together or roll back atomically.

---

### `server/storage.ts` — getLeaderboard
**Issue:** Leaderboard query fetches game session aggregate scores then performs per-userId user lookups in a loop to resolve userName and avatar, producing an N+1 pattern that degrades linearly with participant count and has no limit guard when the caller omits filters.limit.

**Fix:** Join gameSessions with users in a single query: db.select({ userId: gameSessions.userId, userName: users.displayName, avatar: users.profileImageUrl, totalScore: sql<number>`sum(${gameSessions.score})`, gamesPlayed: sql<number>`count(*)` }).from(gameSessions).leftJoin(users, eq(gameSessions.userId, users.id)).groupBy(...).limit(filters?.limit ?? 50). Default limit of 50 prevents unbounded result sets.

### `server/routes.ts` — AI/tutor endpoints (reserveQuota, openai calls)
**Issue:** No rate limiting middleware is applied to AI-heavy endpoints that call reserveQuota and openai. A single student account (or unauthenticated caller if an auth check is missing) can fire unlimited concurrent requests, burning org quota instantly and causing cost spikes. This is especially risky for school-aged users where a single bad actor can exhaust a whole school's monthly allowance.

**Fix:** Add express-rate-limit (already a common dep in Node stacks) scoped per user and per org before any route that calls openai or reserveQuota. Example: `const aiLimiter = rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.use('/api/tutor', aiLimiter);`. Set a stricter org-level window (e.g. 60 req/min) as a secondary limiter keyed on orgId.

### `server/routes.ts` — Error handlers across multiple routes (e.g. Google auth catch block)
**Issue:** Catch blocks propagate raw error messages directly into JSON responses: `res.status(503).json({ ok: false, error: e?.message })`. For database or third-party SDK errors this can leak connection strings, table names, Supabase project refs, or OpenAI model identifiers to the client — all visible to school students in browser devtools.

**Fix:** Create a thin sanitiseError helper: map known error types to safe user-facing messages and log the full error server-side only. Example: `function sanitiseError(e: unknown) { console.error(e); return process.env.NODE_ENV === 'production' ? 'An unexpected error occurred' : (e instanceof Error ? e.message : String(e)); }`. Replace all `e?.message` in res.json error bodies with `sanitiseError(e)`.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 (full base64-encoded PDF or image) is stored directly in the job payload column. For a multi-page PDF this can easily exceed 5–20 MB per row, bloating the jobs table, slowing queue polling queries, and potentially hitting column size limits.

**Fix:** Upload the file to object storage before enqueuing, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper. This keeps job rows small (<1 KB) and makes the queue table scannable.

---

### `shared/schema.ts` — simulatedStocks table, ticker column
**Issue:** ticker is declared notNull but has no unique constraint. Duplicate tickers would silently corrupt portfolio value calculations (e.g. price lookups by ticker returning multiple rows) and break any JOIN or lookup keyed on ticker symbol.

**Fix:** Replace the plain notNull with a unique constraint: ticker: text('ticker').notNull().unique(), or add uniqueIndex('idx_stocks_ticker').on(t.ticker) in the table index config. This is a data-integrity requirement, not just a performance one.

---

### `shared/schema.ts` — portfolioHoldings table — no unique constraint on (userId, stockId)
**Issue:** Nothing prevents multiple holdings rows for the same user and stock. A race condition or duplicate buy operation creates two rows, causing double-counting of position size and incorrect P&L. The application would need error-prone deduplication logic instead of the DB enforcing the invariant.

**Fix:** Add a unique composite constraint: uniqueUserStockIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId). Buy logic should then use an upsert (INSERT ... ON CONFLICT DO UPDATE) to accumulate quantity and recalculate avgPurchasePrice atomically.

## 🟡 Warnings (14)

- **`server/storage.ts` (getBillReminders):** Category is resolved per reminder with an individual await inside a map/loop rather than a JOIN, creating N+1 queries. For a user with 30 reminders this is 31 database round-trips.
  → *Fetch all reminders in one query joined to categories: db.select().from(billReminders).leftJoin(categories, eq(billReminders.categoryId, categories.id)).where(eq(billReminders.userId, userId)). Map the flat rows to the { ...reminder, category } shape in application code.*
- **`server/storage.ts` (getDashboardStats):** Total income, total expenses, savings rate, and budget utilisation are fetched as four or more sequential awaited queries. Each depends only on userId/date filters, not on each other, making them fully parallelisable.
  → *Replace sequential awaits with const [income, expenses, budgetRows, goalRows] = await Promise.all([incomeQuery, expenseQuery, budgetQuery, goalQuery]) to cut wall-clock latency by roughly 3× on typical DB round-trip times.*
- **`server/routes.ts` (/api/auth/google handler):** Four sequential awaits in the Google auth flow — getClassByCode, getOrgEnvironmentById, getOrganization, and getUserByEmail — each block until the previous resolves. The class-lookup branch alone makes 3 round-trips before account creation begins, adding 150-400ms of latency per login on Caribbean network conditions.
  → *Where results are independent, use Promise.all. Specifically: after resolving the join code to either a class or org-env, fetch the org and the existing user record in parallel: `const [org, user] = await Promise.all([getOrganization(resolvedOrgEnv.org_id), authStorage.getUserByEmail(profile.email)])`. For the class branch, getOrgEnvironmentById and getUserByEmail can also be parallelised since neither depends on the other.*
- **`server/routes.ts` (multer fileFilter callbacks (logoUpload, videoUpload, examUpload, upload)):** File type validation relies solely on file.mimetype and path.extname(file.originalname), both of which are attacker-controlled HTTP header values. A malicious user can upload a PHP/JS file with mimetype 'image/png' and extension '.png'. For a children's platform storing files in object storage, this is a stored-content injection risk.
  → *After multer accepts the buffer, validate the actual magic bytes before processing. Use the 'file-type' npm package: `import { fileTypeFromBuffer } from 'file-type'; const detected = await fileTypeFromBuffer(req.file.buffer); if (!allowedMimeTypes.includes(detected?.mime)) return res.status(400).json({ message: 'Invalid file content' });`. Apply this check inside each route handler that receives uploads, not just in fileFilter.*
- **`server/routes.ts` (registerRoutes function — XLSX and Papa imports at module top level):** XLSX (SheetJS) and PapaParse are imported unconditionally at module load time. XLSX in particular is known to be a heavy synchronous parser that blocks the event loop when parsing large files. If a teacher uploads a large spreadsheet, the parse call runs synchronously on the main thread, stalling all concurrent student requests.
  → *Move spreadsheet parsing into the job queue already present in the codebase (enqueueJob/getJob). The route handler should accept the file, store it temporarily via objectStorage, enqueue a parse job, and return a jobId. The client polls getJob for completion. This keeps the event loop free and gives students consistent response times during bulk uploads.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one-at-a-time in a sequential for loop (await inside for). For a 40-question paper this means 40 sequential round-trips to the DB, multiplying latency and holding the job worker busy unnecessarily.
  → *Replace the loop with a single bulk insert: await db.insert(extractedQuestions).values(questions.filter(q => q.question && ...).map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0,4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i+1 }))); This reduces N round-trips to 1.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload with no validation. A corrupted or malicious job payload with olderThanDays: 0 would delete the entire aiUsageEvents table. There is no try/catch, so a DB error surfaces as an unhandled rejection rather than a clean job failure.
  → *Add input validation: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays value'); Wrap the delete in try/catch and rethrow with context so the job runner logs a structured failure.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The function constructs bucket and object path by splitting on '/' with no sanitisation. A caller passing objectPath = '../../sensitive-bucket/admin-keys' would traverse into unintended buckets. There is also no authentication check inside this utility — access control is entirely the caller's responsibility with no defence in depth.
  → *After splitting, assert that parts[0] strictly equals the expected bucket prefix from process.env.PRIVATE_OBJECT_DIR (i.e. validate parts[0] === allowedBucket). Throw a 403-mapped error if it does not match. Document that callers must verify auth before invoking.*
- **`shared/schema.ts` (categories table, userId column):** categories.userId is declared as varchar with no length, matching users.id which is also unbounded varchar. More critically, the FK column has no index. Every query joining categories to users (e.g. fetching user-specific + system categories with WHERE userId = ? OR userId IS NULL) will do a full table scan as the table grows.
  → *Add an explicit index: add a named index on (userId) in the categories table definition, e.g. userIdx: index('idx_categories_user_id').on(t.userId). Also standardise the varchar length to match users.id (e.g. varchar('user_id', { length: 36 }) if UUIDs are used) across all tables to prevent silent truncation mismatches.*
- **`shared/schema.ts` (transactions, budgets, savingsGoals, billReminders, linkedCards, documentUploads — createdAt columns):** createdAt uses .defaultNow() but no .notNull(). This means application code or a raw INSERT omitting createdAt silently stores NULL, breaking any ORDER BY createdAt or audit queries without an error.
  → *Add .notNull() to every createdAt (and executedAt, purchasedAt) column: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default still applies for normal inserts; notNull ensures the DB rejects pathological cases.*
- **`shared/schema.ts` (billReminders table, nextDueDate column):** billReminders has no index on (userId, nextDueDate), which is the primary query pattern for a reminders feature — 'fetch upcoming bills for user X due before date Y'. Without it, every due-date check scans all reminders for the user.
  → *Add a composite index in the billReminders table definition: userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate). This directly supports WHERE userId = ? AND nextDueDate <= ? queries used for notification scheduling.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render inside AuditLogPanel, and the queryKey array uses primitive state values but the queryFn closes over the `params` object built outside useMemo. If any parent re-renders AuditLogPanel, a new params string is silently built but the queryKey may not change if state hasn't changed, causing stale or mismatched fetches.
  → *Wrap params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); ... return p.toString(); }, [actorType, action, from, to]);` then use `params` directly in both queryKey and queryFn URL.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** ObservabilityCard has no loading or error state. When `data` is undefined (initial load or network failure), all property accesses like `data?.sentry`, `data?.alertEmail`, and `data?.healthz?.url` silently render fallback/empty UI with no user feedback, and the healthz link defaults to the hardcoded string '/healthz' which could be wrong in non-default deployments.
  → *Destructure `isLoading` and `isError` from useQuery and render a skeleton or error message: `if (isLoading) return <Spinner />; if (isError) return <ErrorBanner />;`. Move the '/healthz' fallback to an environment config constant like `VITE_HEALTHZ_URL` so it is not hardcoded.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — limit param):** The audit log query hardcodes `params.set('limit', '200')` as a magic string literal with no pagination. For schools with active usage this will return up to 200 rows on every poll/refetch, transferring unnecessary data and potentially causing slow renders on the main thread when mapping large arrays.
  → *Extract the limit as a named constant `const AUDIT_LOG_PAGE_SIZE = 200;` in module scope or pull from an env var. Add cursor-based or offset pagination state (`const [page, setPage] = useState(0)`) and include it in the queryKey so React Query caches pages independently.*

## 🔵 Info / Tech Debt (3)

- **`server/jobHandlers.ts` (runExtractPaper — vision branch):** When the file is an image, buffer.toString('base64') is called again after the buffer was already decoded from fileB64. This doubles peak memory usage (original base64 string still in scope + new base64 string) and wastes CPU re-encoding bytes that were already base64 in the payload.
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — useQuery):** The observability query has no `refetchInterval` or `staleTime` configured, so it uses the global default. For a status/health card that admins keep open, this means it either never auto-refreshes (stale data) or refetches on every window focus, generating unnecessary server load on an endpoint that likely calls external services like Sentry.
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper):** `fmtDate` accepts `any` and calls `new Date(d)` without validation. Passing an invalid string like 'N/A' or a corrupt DB value returns 'Invalid Date' rendered directly into the UI, and no try/catch exists to catch the RangeError that some JS engines throw for extreme date values.

## 💡 Opportunities

### Centralise pagination defaults as named constants — 🔥 High Impact
Multiple query sites have no limit applied or use ad-hoc magic numbers (e.g., leaderboard). Define DEFAULT_PAGE_LIMIT = 50 and MAX_PAGE_LIMIT = 200 in a shared config and enforce them at every list endpoint to prevent memory spikes and slow queries as data grows.

### Extract trade execution into a dedicated service with atomic DB transactions — 🔥 High Impact
Buy and sell paths touch virtual balance, portfolio holdings, and portfolio transactions across multiple storage methods. Consolidating them into a single transactional service method eliminates the consistency gap and makes the financial logic auditable in one place rather than spread across the route layer.

### Add a query-level result cache for market stock data — ⚡ Medium Impact
getMarketStocks and getStockById are called on every portfolio render and trade screen load but the underlying data changes only on a scheduled interval. A short TTL in-process cache (node-cache or a Map with setTimeout eviction keyed by currency) would eliminate redundant DB reads under concurrent user load.

### Split monolithic routes.ts into domain modules — 🔥 High Impact
At 194k+ characters, routes.ts is unmaintainable and makes targeted security review impossible. Split by domain: authRoutes, lessonRoutes, aiRoutes, orgRoutes, uploadRoutes. Each module gets its own middleware stack, making it straightforward to apply rate limiting, auth guards, and input validation consistently per domain. Expected impact: faster onboarding, fewer missed auth checks, easier audits.

### Centralise isAuthenticated guard with role assertions — 🔥 High Impact
Auth checks appear to be applied per-route manually throughout the file. Introduce a typed middleware factory — e.g. `requireRole('teacher' | 'student' | 'admin')` — that wraps isAuthenticated and checks req.session.role in one place. This prevents the common mistake of adding a new route and forgetting the guard, which is a real risk when the file is this large.

### Add structured request/response logging with correlation IDs — ⚡ Medium Impact
With AI quota tracking (reserveQuota, finalizeUsage, releaseReservation) spread across async flows, a failed request can leave orphaned reservations. Adding a correlation ID (uuid v4 injected by middleware into res.locals and all log lines) would let you trace exactly which requests failed to call finalizeUsage or releaseReservation, making quota leak debugging tractable without full APM.

### Parallel OpenAI calls for text extraction + question parsing — ⚡ Medium Impact
In the PDF branch, text extraction is CPU-bound (pdf-parse) and does not depend on the second OpenAI call, but both OpenAI calls are strictly sequential by design (second depends on first). However, paper metadata update (storage.updateExamPaper) and any post-processing steps that are independent could be parallelised with Promise.all. More impactfully, if multiple papers are queued, the worker concurrency setting should be reviewed to run extractions in parallel across workers rather than serially.

### Centralised payload schema validation with Zod at handler entry — ⚡ Medium Impact
Every handler reads job.payload fields with no runtime type checking. Adding a thin Zod parse at the top of each handler (reusing the same schemas already defined in JobPayloads) would catch malformed payloads immediately, produce structured error messages in the job failure log, and prevent undefined-access bugs mid-handler — all with negligible overhead.

### Idempotency token on bulk CSV export jobs — 📌 Low Impact
The admin-csv-export handler can be enqueued multiple times for the same export type within the same window. Without a deduplication key, concurrent workers will both query the DB, both generate CSVs, and the second upload will silently overwrite the first. Adding an idempotencyKey (e.g. `csv-export-${type}-${dayUtc}`) to enqueueJob and enforcing uniqueness in the jobs table would prevent redundant work and race-condition overwrites.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, simulatedStocks, portfolioTransactions). Extract it to a shared constant DEFAULT_CURRENCY = 'BSD' and reference it in each .default() call. A future currency change or multi-currency expansion then requires a single edit rather than hunting across the schema.

### Add index on documentUploads (userId, status) for processing queue queries — ⚡ Medium Impact
Document processing pipelines routinely query WHERE userId = ? AND status = 'processing'. Without an index this scans all uploads. A composite index on (userId, status) would make polling for in-progress uploads O(log n) and prevent slowdowns as upload history grows per user.

### Replace unbounded text columns with varchar for fixed-format fields — 📌 Low Impact
Fields like linkedCards.lastFour, currency codes, and icon/color values are stored as unbounded text. Using varchar with explicit lengths (e.g. varchar('last_four', {length:4}), varchar('currency', {length:3})) enforces data shape at the DB layer, reduces per-row storage overhead in Postgres TOAST, and surfaces bad data at insert time rather than at render time in the UI.

### Consolidate admin bootstrap queries with Promise.all — 🔥 High Impact
The dashboard likely fires multiple independent useQuery calls for organizations, schools, teachers, students, and sponsors sequentially on mount. Wrapping the initial data fetches in a single server-side endpoint or using `Promise.all` in a combined queryFn would cut initial load time by eliminating sequential round-trips, which matters on Caribbean mobile networks with higher latency.

### Memoize tab-panel list renders with useCallback on action handlers — 🔥 High Impact
Large inline arrow functions passed as onClick/onChange props inside mapped tab content (e.g., delete, edit, search handlers) are recreated on every parent render, causing every child row component to re-render even when data hasn't changed. Wrapping these with useCallback and stabilizing props will allow React.memo on row components to skip renders, reducing jank in the Students and Schools tables which can have hundreds of rows.

### Add React Query devtools and explicit error boundaries per panel — ⚡ Medium Impact
A single unhandled promise rejection in any panel (e.g., DBViewer, Perf Agent) will currently crash the entire AdminDashboard. Wrapping each tab panel in an ErrorBoundary component and adding QueryErrorResetBoundary will isolate failures, keeping the rest of the dashboard functional and giving admins a retry button per section rather than a full-page error.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns across portfolio holdings, bill reminders, and leaderboard lookups where per-row fetches replace proper JOINs. Missing database transactions on multi-step operations like buy/sell flows risk partial writes leaving balances and holdings inconsistent. Sequential awaits in several composite operations add unnecessary latency.

### API Routes (`server/routes.ts`)
routes.ts is a large, monolithic route file with several reliability and security gaps. jobHandlers.ts has several reliability and performance issues: large base64 payloads stored in the job table risk bloating the DB, sequential DB inserts in a loop should be batched, and the file is missing try/catch around several async operations. The schema is well-structured with foreign keys and composite indexes, but has several reliability and security gaps.
