## 🔴 Critical (5)

### `server/routes.ts`: robots.txt and sitemap.xml handlers (~line 43 and 61)
**Issue:** Host header injection: `req.headers['x-forwarded-host']` and `req.headers.host` are used verbatim to construct absolute URLs. An attacker can supply an arbitrary Host header, causing the Sitemap directive and sitemap <loc> tags to point to an attacker-controlled domain, which can be used for cache poisoning or SEO manipulation.

**Fix:** Read the canonical base URL from a single trusted environment variable (e.g. `process.env.PUBLIC_BASE_URL`) set at deploy time. Fall back to reconstructing from protocol+host only when running locally, and validate the host against an allowlist of known domains before using it.

### `server/routes.ts`: /public-objects/:filePath(*) handler (~line 99)
**Issue:** The wildcard `filePath` parameter is passed directly to `objectStorage.searchPublicObject()` without any path sanitisation. A request to `/public-objects/../../private/secret` could traverse outside the intended public prefix depending on how the storage service resolves paths, potentially exposing private objects.

**Fix:** Sanitise the path before use: `const safe = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); if (safe !== req.params.filePath) return res.status(400).end();` and additionally assert the resolved key starts with the expected `public/` prefix inside the storage service.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** fileB64 (full base64-encoded PDF or image) is stored directly in the job table payload. For a multi-page PDF this can easily be 5-20MB per row, bloating the jobs table, slowing job queue queries, and keeping sensitive exam content in a queryable table indefinitely.

**Fix:** Before enqueuing, upload the file to private object storage and store only the resulting object path in the payload (e.g. { paperId, objectPath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of the job and delete the object after successful extraction or on terminal failure.

---

### `shared/schema.ts`: documentUploads table, errorMessage column
**Issue:** errorMessage is plain text with no length cap. A runaway error string (e.g., a full stack trace logged from an upstream service) can store megabytes per row, bloating the table and leaking internal implementation details or file paths that expose server internals if ever surfaced to the client.

**Fix:** Change to varchar('error_message', { length: 500 }) to enforce a safe upper bound. In the application layer, truncate or sanitize error messages before persisting them, and ensure the API response layer never forwards raw errorMessage to school-aged end users without sanitization.

---

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook consumption at top of AdminDashboard
**Issue:** The file imports and calls useAdminAuth but the truncated code shows no guard that halts rendering or redirects if the admin session is invalid before child components (including data-fetching useQuery calls) execute. If useAdminAuth is async or token-based, queries will fire with no credentials during the authentication loading window, leaking the existence of admin endpoints to unauthenticated network observers.

**Fix:** Add an early return before any useQuery calls: `const { isAdmin, isLoading: authLoading } = useAdminAuth(); if (authLoading) return <Loader2 />; if (!isAdmin) { setLocation('/login'); return null; }` Ensure all useQuery hooks are defined after this guard or use the `enabled: isAdmin` option so they never fire without a confirmed session.

## 🟡 Warnings (15)

- **`server/routes.ts` (/healthz handler (~line 75)):** Dynamic `import('./db')` and `import('drizzle-orm')` inside the health-check route means the module loader is invoked on every health poll (Railway + uptime monitors can hit this every 10-30 s). This bypasses Node's module cache for the first call and adds unnecessary overhead; it also hides the real module from static analysis and bundlers.
  → *Move the db and sql imports to the top of the file as static imports alongside the other server imports. The health check then simply calls `await db.execute(sql\`select 1\`)` with zero import overhead.*
- **`server/routes.ts` (Supabase init chain (~line 115)):** The seven Supabase init/seed functions are chained with sequential `.then()` calls. If each averages 200 ms of async I/O, startup is blocked for ~1.4 s unnecessarily. Seeds that do not depend on each other (e.g. seedStaticModules, seedRegionalContent, seedGameContent) can run concurrently.
  → *After the mandatory sequential steps that have true ordering dependencies (initSupabaseTables → trimOrgNamesInSupabase → seedFinancialAcademyLesson → initStaticContentTables), fan out the remaining independent seeds with `Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])`. Wrap the whole block in a single `.catch()` as is already done.*
- **`server/routes.ts` (registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 112)):** The OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied in this file. The auth limiter covers auth paths only. Heavy AI endpoints left unthrottled on a school-aged consumer app risk runaway OpenAI cost from abusive clients or accidental loops.
  → *Create an `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP using express-rate-limit) and apply it with `app.use('/api/chat', aiLimiter)`, `app.use('/api/image', aiLimiter)`, `app.use('/api/audio', aiLimiter)` before the three register calls.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one-by-one in a sequential await loop. For a 50-question paper this means 50 round-trips to the database, adding significant latency and holding the job open unnecessarily long.
  → *Replace the for loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(questionsArray) once. Add a guard so the insert is skipped when the array is empty.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays comes directly from job.payload with no validation. A misconfigured or malicious enqueue call with olderThanDays = 0 or a negative value would delete all AI usage records, and a non-numeric value would produce an invalid Date silently deleting nothing or everything depending on the DB driver.
  → *Add explicit validation at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays value'); This keeps the check close to the data and lets the job fail visibly rather than silently corrupting data.*
- **`server/jobHandlers.ts` (runExtractPaper – two sequential openai.chat.completions.create calls for image path):** For image uploads the code calls the vision API to extract text, then immediately calls the extraction API on that text. These are unavoidably sequential, but the PDF path also awaits pdf-parse and then the extraction API sequentially with no timeout or circuit-breaker. If either OpenAI call hangs, the job holds a worker thread indefinitely with no abort signal.
  → *Wrap each OpenAI call with a Promise.race against a timeout: const withTimeout = (p, ms) => Promise.race([p, new Promise((_, r) => setTimeout(() => r(new Error('OpenAI timeout')), ms))]). Use a configurable timeout (OPENAI_JOB_TIMEOUT_MS env var, default 120000). This allows the job to fail cleanly and retry rather than blocking the worker pool.*
- **`server/jobHandlers.ts` (parsePrivatePath / uploadPrivateObject):** PRIVATE_OBJECT_DIR is checked at call time inside parsePrivatePath, meaning a misconfiguration is only discovered when a job runs, not at startup. Additionally, relPath is concatenated into the storage path with no sanitisation, so a payload containing ../../ could escape the intended directory.
  → *Validate PRIVATE_OBJECT_DIR at module load time (throw in registerJobHandlers if missing). Sanitise relPath by resolving it and asserting it stays within the expected prefix: const resolved = path.posix.normalize(relPath); if (resolved.startsWith('..')) throw new Error('Path traversal detected');*
- **`shared/schema.ts` (categories table definition):** userId on categories uses bare varchar() with no length, and the column has no index despite being used in WHERE clauses to filter user-owned vs system categories. Full table scans will occur as category count grows.
  → *Add varchar('user_id', { length: 36 }) to match the UUID format from users.id, and add an explicit index: index('idx_categories_user_id').on(t.userId) in the table's index callback. Apply the same length constraint to every other userId varchar column in the schema for consistency and to prevent silent truncation.*
- **`shared/schema.ts` (budgets, linkedCards, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these tables have indexes on userId, yet every realistic query against them will filter by userId. Without indexes, each user-scoped query does a full sequential scan. portfolioHoldings also JOINs on stockId with no index on that column.
  → *Add an index callback to each table. For budgets: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). For portfolioHoldings add a composite: index('idx_holdings_user_stock').on(t.userId, t.stockId). For billReminders also index nextDueDate since due-date queries are expected: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate).*
- **`shared/schema.ts` (createdAt columns across all tables):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull(). defaultNow() only sets a DB default; application-level inserts that explicitly pass null will succeed and store NULL, breaking any ORDER BY createdAt or audit queries silently.
  → *Chain .notNull() on every createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt in portfolioTransactions and purchasedAt in portfolioHoldings.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker has no unique index. Duplicate tickers can be inserted, causing ambiguous JOIN results when portfolioHoldings or portfolioTransactions reference a stock by ticker in application logic. annualReturnPct also has no NOT NULL, so return calculations will silently produce NULL instead of erroring.
  → *Add uniqueIndex('idx_stocks_ticker').on(t.ticker) in the table index callback. Add .notNull().default('0') to annualReturnPct so return calculations always have a numeric base value rather than propagating NULL through arithmetic.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage):** The /api/admin/help-chat POST request sends the full growing messages array on every turn. As conversation length grows this payload becomes unbounded, and there is no server-side truncation guard visible on the client. For a school-aged platform this also risks leaking prior conversation context including any PII typed by an admin.
  → *Slice the context window before sending: pass only the last N turns (e.g., 10) to the API: `messages: next.slice(-10)`. Add a MAX_MESSAGES constant (e.g., 50) and trim state when it exceeds that limit to prevent unbounded memory growth in the browser.*
- **`client/src/pages/AdminDashboard.tsx` (scroll div ref callback inside message list render):** The auto-scroll sentinel uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` directly in JSX. React calls this callback on every render, meaning scrollIntoView fires on every keystroke in the input field, not just when a new message arrives. This causes jarring scroll interruptions while the user is typing.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – handleKey):** handleKey is defined as a plain arrow function inside the component body with no useCallback. Because AdminHelpChat likely re-renders on every input change (controlled input), a new function reference is created and passed to the onKeyDown prop on every keystroke, causing the input element to detach and reattach its event handler repeatedly.
  → *Wrap with useCallback: `const handleKey = useCallback((e: React.KeyboardEvent) => { if (e.key === 'Enter' && !e.shiftKey) { e.preventDefault(); sendMessage(); } }, [sendMessage]);` and also wrap sendMessage with useCallback with deps [messages, input, loading].*
- **`client/src/pages/AdminDashboard.tsx` (apiRequest calls throughout dashboard data fetching):** The file hardcodes API path strings like '/api/admin/help-chat' as string literals scattered across the file with no central route constants. In a financial literacy app with role-based access, a typo in an admin route string silently falls through to a non-admin endpoint or returns a 404 with no type-level error, and there is no evidence of environment-based base URL configuration.
  → *Create a `src/lib/apiRoutes.ts` constants file: `export const API = { adminHelpChat: '/api/admin/help-chat', ... } as const`. Reference `API.adminHelpChat` everywhere. For the base URL, read `import.meta.env.VITE_API_BASE_URL` and prepend it in `apiRequest` so staging and production endpoints are not hardcoded.*

## 💡 Opportunities

### Parallel domain-router registration: ⚡ Medium Impact
The six `await register*Routes(app)` calls after the Promise.all import are sequential. If any of those functions perform async work (DB schema checks, cache warm-up), they add to cold-start time. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` to cut startup latency proportionally.

### Centralised base-URL config module: ⚡ Medium Impact
Both robots.txt and sitemap.xml independently duplicate the same proto+host extraction logic. Extract a `getBaseUrl(req)` helper that reads `process.env.PUBLIC_BASE_URL` with a validated request-header fallback. This eliminates duplication and makes the Host-injection fix a single-point change.

### Sitemap caching: 📌 Low Impact
The sitemap XML is regenerated on every crawler request. Since the URL list is static, generate the XML string once at startup and store it in a module-level constant, then serve it with a `Cache-Control: public, max-age=86400` header. This removes per-request string allocation and template processing.

### Centralise job payload validation with a schema layer: 🔥 High Impact
Add Zod schemas for every job payload type and parse them at the top of each handler before any async work. This catches misconfigured enqueue calls at job-start rather than producing silent data corruption or cryptic runtime errors, and doubles as living documentation of expected shapes.

### Add idempotency key tracking for admin-csv-export: ⚡ Medium Impact
If admin-csv-export is retried after the object upload succeeds but before the job status is written, a second export file will be written and the first orphaned. Store the target object path deterministically (e.g. derived from jobId) and check for existence before regenerating, skipping CSV generation and re-using the existing file on retry.

### Move OpenAI model names to environment config: ⚡ Medium Impact
gpt-4o is hardcoded in two places in runExtractPaper. Externalising this to OPENAI_EXTRACTION_MODEL and OPENAI_VISION_MODEL env vars allows model upgrades or cost-saving downgrades without a code deploy, which is especially important for a school-budget app where API cost control matters.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, etc.). Define const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' once and reference it in every .default() call. This lets regional deployments change currency without a schema migration and eliminates the risk of a typo creating mixed defaults.

### Replace serial PKs with UUID PKs for user-facing tables: 🔥 High Impact
Using serial (auto-increment integers) as primary keys on transactions, savingsGoals, and portfolioTransactions exposes sequential IDs to school-aged users in URLs and API responses, enabling enumeration attacks. Switching to uuid().defaultRandom().primaryKey() removes that surface entirely and aligns with the UUID already used for userId from the auth model.

### Add a partial index for active bill reminders: ⚡ Medium Impact
Queries for upcoming due bills will always filter by nextDueDate >= now() and implicitly exclude soft-deleted or inactive rows if that feature is added later. Adding a partial index now—index('idx_bill_reminders_upcoming').on(t.nextDueDate).where(sql`next_due_date >= now()`)—keeps the index small and fast as reminder volume grows across the student user base.

### Parallel admin dashboard data fetching: 🔥 High Impact
If AdminDashboard fires multiple useQuery calls for stats, users, schools, and audit logs sequentially (each waiting for the previous to resolve due to dependent state), refactor them as independent parallel queries with no cross-dependencies. This can cut initial dashboard load time by 2-4x depending on server latency, which matters on Caribbean mobile networks with higher RTT.

### Memoize filtered and sorted table data: ⚡ Medium Impact
Admin tables with search, filter, and sort state almost certainly recompute filtered arrays on every render. Wrap these derivations in useMemo with [rawData, searchTerm, filterValue, sortKey] deps. For a dataset of 500+ students or transactions this avoids O(n) filter passes on every keystroke and keeps the UI responsive on low-end devices common in the target demographic.

### Extract AdminHelpChat into its own file with React.memo: ⚡ Medium Impact
AdminHelpChat is a self-contained component embedded in a 1000+ line file. Moving it to `components/AdminHelpChat.tsx` and wrapping with React.memo prevents it from re-rendering whenever unrelated dashboard state (e.g., selected tab, search term) changes. This also makes the component independently testable and reduces the cognitive load of the main dashboard file.
