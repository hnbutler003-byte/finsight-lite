## 🔴 Critical (6)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: the function fetches all holdings for a user then issues a separate getStockById query per holding inside a loop, resulting in 1+N round-trips to the database for N holdings.

**Fix:** Replace the per-holding stock lookup with a single JOIN or inArray query: `db.select().from(portfolioHoldings).innerJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id)).where(eq(portfolioHoldings.userId, userId))`. Return the joined rows directly so only one query is issued regardless of portfolio size.

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** N+1 query pattern: enrollments are fetched first, then a separate user lookup is issued for each enrollment's studentId inside a map/loop, producing 1+N database calls for N students in a class.

**Fix:** Use a single JOIN: `db.select().from(classEnrollments).innerJoin(users, eq(classEnrollments.studentId, users.id)).where(eq(classEnrollments.classId, classId))`. This collapses all lookups into one query and is especially important as class sizes grow.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (trade execution path)
**Issue:** The buy/sell flow updates the portfolio holding and the virtual balance in two separate, non-transactional awaits. A process crash or timeout between these two writes leaves the user's balance and holdings in an inconsistent state (e.g. shares deducted but balance not updated).

**Fix:** Wrap both writes in a Drizzle transaction: `await db.transaction(async (tx) => { await tx.update(portfolioHoldings)...; await tx.update(userVirtualBalance)...; })`. If either write fails the entire operation is rolled back atomically.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full file contents stored as base64 in the job payload. A single PDF or image uploaded by a student can be several MB; base64 encoding inflates that by ~33%. Every retry re-reads this blob from the jobs table, causing repeated large row scans and high memory allocation when the worker deserializes the job.

**Fix:** Upload the raw file to private object storage immediately at upload time and store only the resulting object path in the job payload (e.g. { paperId, objectPath, ext, subject }). Inside runExtractPaper, stream or download the file from object storage using objectStorageClient before processing. This keeps job rows small and makes retries cheap.

---

### `shared/schema.ts` — simulatedStocks table definition
**Issue:** ticker column has no unique index. Two rows with the same ticker symbol (e.g. 'BTC') are allowed, which will cause ambiguous joins in portfolioHoldings and portfolioTransactions and incorrect portfolio value calculations for school-aged users relying on simulation accuracy.

**Fix:** Replace the plain column declaration with a uniqueIndex in the table config: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also add .notNull() to issuer and region if they are required for display, or document explicitly why they are nullable.

---

### `client/src/pages/AdminDashboard.tsx` — useAdminAuth (top of component)
**Issue:** useAdminAuth is imported and called but there is no guard that blocks rendering the dashboard — including all child queries — until auth is confirmed. If the hook resolves asynchronously, every child useQuery fires immediately and returns sensitive admin data to an unauthenticated render cycle before a redirect can occur.

**Fix:** After calling useAdminAuth, destructure an isAuthorized/isLoading flag and return null (or a spinner) early: `const { admin, isLoading } = useAdminAuth(); if (isLoading || !admin) return <Redirect to='/admin/login' />;`. This prevents all child queries from firing and prevents sensitive data from being passed to the DOM before auth is confirmed.

## 🟡 Warnings (13)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregation queries (total income, total expenses, transaction count, budget totals) are executed one after another, adding cumulative latency proportional to the number of queries.
  → *Wrap all independent stat queries in a single `Promise.all`: `const [incomeResult, expenseResult, txCountResult, budgetResult] = await Promise.all([incomeQuery, expenseQuery, txCountQuery, budgetQuery])`. This reduces wall-clock latency to the duration of the slowest single query.*
- **`server/storage.ts` (getTransactions / getBillReminders / getGameSessions / getAllQuestions):** Unbounded list queries with no default pagination limit. For a school-wide deployment with many students, these can return arbitrarily large result sets, exhausting Node.js heap and causing slow responses. The `limit` parameter is optional and defaults to no cap.
  → *Enforce a maximum page size constant (e.g. `const MAX_LIMIT = 200`) and apply it unconditionally: `query.limit(Math.min(filters?.limit ?? MAX_LIMIT, MAX_LIMIT))`. Add an `offset` parameter for cursor/page-based pagination. Expose `MAX_LIMIT` from an environment config variable so it can be tuned without a deploy.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a 50-question paper this is 50 serial round-trips to the database, multiplying latency and holding the worker busy far longer than necessary.
  → *Batch-insert all valid questions in a single statement: filter the questions array into valid rows first, then call db.insert(extractedQuestions).values(rows) once. If storage.createExtractedQuestion must be used, at minimum run Promise.all on all insert promises simultaneously, though a single bulk insert is strongly preferred for atomicity and performance.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** The 30,000-character truncation limit for PDF text is hardcoded. This magic number will silently drop content from longer exam papers without any warning to the user or log entry, causing incomplete question extraction with no observable signal.
  → *Move the limit to an environment variable, e.g. PDF_TEXT_CHAR_LIMIT defaulting to 30000. After slicing, log a warning with the paperId and original vs truncated length (e.g. logger.warn({ paperId, original: pdfData.text.length, truncated: 30000 }, 'PDF text truncated')) so operators can detect papers that are being cut short and tune the limit.*
- **`server/jobHandlers.ts` (runExtractPaper — second openai.chat.completions.create call (extraction)):** There is no try/catch around either OpenAI API call inside runExtractPaper. A transient 429 rate-limit or 5xx from OpenAI will propagate as an unhandled rejection. While the outer handler re-throws to let the worker retry, on the final attempt it sets status to 'failed' without recording why, making diagnosis hard.
  → *Wrap each OpenAI call in a try/catch. On catch, enrich the error with context before re-throwing: throw Object.assign(err, { paperId, stage: 'extraction' }). On the final attempt path, also persist the error message to exam_papers (e.g. a failureReason column) so the UI can surface a human-readable explanation rather than a generic failed state.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The olderThanDays value comes directly from the job payload with no validation before being multiplied into a cutoff timestamp. A malformed or malicious payload (e.g. olderThanDays: 0 or a negative number) would produce a cutoff of now or the future, deleting the entire aiUsageEvents table.
  → *Validate the payload before use: if (typeof olderThanDays !== 'number' || olderThanDays < 1 || olderThanDays > 3650) throw new Error('Invalid olderThanDays: ' + olderThanDays). Also add a dry-run count query before deletion in non-production environments so operators can verify scope before rows are removed.*
- **`shared/schema.ts` (budgets table definition):** budgets table has no index on userId, yet budget lookups will always filter by userId. Without an index, every budget query triggers a full table scan that worsens as users grow.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The composite index also enforces that duplicate category budgets per user are detectable at the query layer.*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, etc.)):** All userId columns are declared as varchar() with no length argument, which in PostgreSQL defaults to unlimited length — identical to text. This removes the database-level guard against oversized tokens being stored and wastes planner statistics accuracy since PostgreSQL optimises fixed-length varchars differently.
  → *Change every userId column to varchar('user_id', { length: 128 }) to match realistic Clerk/Auth user ID lengths. This adds a cheap constraint and improves pg_stats histogram accuracy for the planner.*
- **`shared/schema.ts` (createdAt columns on transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt uses .defaultNow() but no .notNull(), meaning application code or a bulk insert that explicitly passes null will store a null audit timestamp with no error. This silently corrupts audit trails critical for a financial app.
  → *Append .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). The default still fires for normal inserts; notNull prevents any explicit null bypass.*
- **`shared/schema.ts` (billReminders and savingsGoals — nextDueDate / deadline timestamp columns):** billReminders.nextDueDate is notNull (correct) but savingsGoals.deadline is nullable with no default and no index, despite being the primary sort/filter key for goal dashboards. Querying upcoming deadlines without an index causes full scans, and a null deadline with no business-rule validation lets goals exist in an indeterminate state.
  → *Add an index: (t) => ({ userDeadlineIdx: index('idx_savings_goals_user_deadline').on(t.userId, t.deadline) }). In the Zod insert schema, add .refine() to require deadline to be a future date when provided, making the null-means-no-deadline contract explicit at the validation layer.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel — params construction before useQuery):** URLSearchParams is reconstructed on every render of AuditLogPanel, and its toString() result is passed directly into the queryFn without sanitization. The `action` and `from`/`to` fields come from raw useState inputs. While React Query's queryKey deduplication helps, the params object itself is never memoized, causing unnecessary object churn, and string values are injected into a URL without encoding validation — a confused-deputy risk if the server uses those values in downstream queries without its own validation.
  → *Wrap params construction in useMemo keyed on [actorType, action, from, to]. On the server, ensure each accepted param is validated against an allowlist of field names and date formats before use in any query. Example: `const queryString = useMemo(() => { const p = new URLSearchParams(); ... return p.toString(); }, [actorType, action, from, to]);`*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard — useQuery without error or loading state):** ObservabilityCard renders Badge and anchor elements directly from `data?.sentry` with no isLoading or isError handling. On a slow network or server error, all badges silently render as 'Disabled'/'Not configured', which is indistinguishable from a genuine disabled state and will mislead operators into thinking Sentry is off when the fetch simply failed.
  → *Destructure `isLoading, isError` from useQuery. Render a skeleton or spinner when isLoading, and an explicit error callout (e.g., 'Could not load observability status') when isError, so operators can distinguish fetch failures from genuine configuration gaps.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper):** `fmtDate` accepts `any` and calls `new Date(d)` without a try/catch. Passing a malformed string (e.g., from an unexpected API schema change) causes Date to produce Invalid Date, and toLocaleDateString on an Invalid Date throws in some environments or silently returns 'Invalid Date' in others — either outcome corrupts table cells across every row that calls it.
  → *Add a guard: `function fmtDate(d: unknown): string { if (!d) return '—'; const dt = new Date(d as string); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); }`. Type the parameter as `unknown` to force callers to pass real values.*

## 💡 Opportunities

### Centralise hardcoded virtual balance seed amount — ⚡ Medium Impact
The initial virtual balance (likely a constant such as 10000) is hardcoded in getVirtualBalance's upsert fallback. Moving it to an environment variable (VIRTUAL_BALANCE_SEED_AMOUNT) allows educators and sponsors to configure starting capital per deployment without a code change.

### Add a query-level result cache for getMarketStocks — 🔥 High Impact
Simulated stock prices change on a schedule, not per-request. Caching the getMarketStocks result in-process for 30–60 seconds (using a simple Map with a timestamp) would eliminate redundant full-table scans on every portfolio page load across all concurrent users, which matters most during class sessions with 30+ simultaneous students.

### Extract repeated owner-check filter pattern into a typed helper — ⚡ Medium Impact
The pattern `and(eq(table.id, id), eq(table.userId, userId))` is duplicated across deleteTransaction, deleteBudget, deleteSavingsGoal, deleteExamPaper, and others. A small generic helper `ownerFilter(table, id, userId)` would eliminate this duplication and ensure auth ownership checks are never accidentally omitted on new delete/update methods.

### Pre-signed URL delivery instead of streaming proxy — 🔥 High Impact
streamPrivateObjectToResponse pipes object storage content through the Node process, consuming memory and an open HTTP connection for the full download duration. Replacing this with a short-lived pre-signed URL (valid 60–300 s) and redirecting the client eliminates the proxy overhead entirely. Expected impact: lower server memory and connection concurrency under concurrent export downloads.

### Idempotency key on extracted-question bulk insert — ⚡ Medium Impact
The current idempotency strategy deletes all questions for the paper then re-inserts on every retry. A retry that fails mid-insert leaves the paper with no questions and status still 'processing'. Wrapping the delete + bulk insert in a single DB transaction ensures the paper is never left in a half-written state, removing the need for the pre-delete strategy entirely.

### Structured payload schema validation at job registration — ⚡ Medium Impact
No payload schema validation exists at the registerJobHandler call sites. Adding a Zod schema parse at the top of each handler (parse once, fail fast) would catch misconfigured enqueueJob calls at enqueue time rather than mid-execution, and produce clear error messages instead of runtime TypeErrors deep in handler logic. This is especially important for school-facing features where silent failures confuse teachers.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table columns (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract it to a single exported const DEFAULT_CURRENCY = 'BSD' in a shared config file and reference it in each column definition. When the app expands to TTD or JMD markets, one change propagates everywhere instead of requiring six schema migrations.

### Add a partial index for unprocessed document uploads — ⚡ Medium Impact
documentUploads rows with status='processing' are the only rows polled by background jobs, but queries scan the full table. A partial index — CREATE INDEX idx_doc_uploads_processing ON document_uploads(user_id) WHERE status = 'processing' — would make the polling query O(active uploads) rather than O(all uploads). In Drizzle this can be expressed with .where(sql`status = 'processing'`) inside the index definition once Drizzle supports partial indexes, or via a raw migration.

### Replace learningModules.content text column with structured jsonb — 🔥 High Impact
Storing rich lesson content as a flat text column prevents the database from indexing specific content fields (e.g. module type, embedded quiz IDs) and forces full content deserialization in the application layer on every read. Migrating to jsonb with a GIN index on key paths would allow filtered queries like 'all modules containing a quiz' without fetching and parsing full content blobs, reducing memory pressure on Node.js for a read-heavy educational feature.

### Parallel query loading for Overview tab — 🔥 High Impact
The Overview/Dashboard tab likely fires multiple independent useQuery calls sequentially within the same render. Wrapping independent async fetches (e.g., stats, recent activity, job status) with Promise.all inside a single aggregating queryFn, or using React Query's useQueries, would cut initial dashboard load time by the number of sequential round-trips minus one — potentially 2–4 seconds on Caribbean mobile networks with high latency.

### Memoize TABS constant outside component tree — ⚡ Medium Impact
TABS is defined at module scope which is correct, but any derived structures (filtered tab lists, icon maps) computed inside components should be wrapped in useMemo. More impactfully, the tab-switching handler and any onClick passed into the sidebar should be useCallback-stabilized so the entire sidebar does not re-render on every parent state change (e.g., search input keystrokes in the active tab panel).

### Extract heavy tab panels into lazy-loaded route segments — ⚡ Medium Impact
With 13 tabs (organizations, schools, teachers, students, classes, sponsors, challenges, reports, jobs, perf, audit, dbviewer), all panel code is bundled and parsed upfront. Using React.lazy + Suspense per tab panel would reduce initial JS parse time and TTI, which is especially meaningful for school-network connections in the Caribbean where bandwidth is constrained.
