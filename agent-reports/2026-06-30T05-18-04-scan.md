## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget row returned, a separate query fetches the matching category and another aggregates spent transactions, producing 2N+1 database round-trips per call.

**Fix:** Replace the per-budget queries with a single JOIN or two batched queries: one `inArray(categories.id, budgetCategoryIds)` fetch and one grouped SUM query (`SELECT categoryId, SUM(amount) FROM transactions WHERE userId=? AND categoryId IN (?) GROUP BY categoryId`), then assemble results in-memory with a Map keyed by categoryId.

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional `limit` but if the caller omits it there is no default cap, allowing a full table scan of gameSessions and users to be returned to the client — a denial-of-service vector and memory risk.

**Fix:** Apply a hard default and maximum: `const safeLimit = Math.min(filters?.limit ?? 25, 100);` and pass safeLimit to the query's `.limit()` call unconditionally.

---

### `server/storage.ts`: enrollStudent / createClass
**Issue:** Multi-step writes (e.g., creating a class and inserting the teacher-class link, or enrolling a student and updating enrollment count) are executed as separate awaited statements with no wrapping database transaction, leaving the database in a partially-written state if any step throws.

**Fix:** Wrap each multi-step write in `await db.transaction(async (tx) => { ... })` using Drizzle's transaction API. All insert/update calls inside should use the `tx` parameter instead of `db` so they share the same connection and atomicity guarantee.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. A school-aged user or bot can hammer these endpoints freely, running up API costs or triggering OpenAI quota exhaustion.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP using express-rate-limit) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` then call the register functions. Verify the exact path prefixes those register functions use.

---

### `server/routes.ts`: robots.txt handler (line ~44) and sitemap.xml handler (line ~59)
**Issue:** The base URL is constructed directly from `x-forwarded-host` and `host` request headers with no validation or allowlist. An attacker can send a crafted Host header and inject an arbitrary domain into the Sitemap directive and all sitemap `<loc>` values, enabling cache poisoning or SEO manipulation.

**Fix:** Read the canonical base URL from an environment variable: `const base = process.env.CANONICAL_BASE_URL || 'https://finsightlite.com';` and ignore the request headers entirely for URL construction. Set this env var in Railway/Replit config.

---

### `server/jobHandlers.ts`: registerJobHandler("extract-paper")
**Issue:** fileB64 (a full base64-encoded PDF or image) is stored directly in the job payload, which lives in the jobs DB table. For a multi-page PDF this can easily be several MB per row, bloating the table, slowing queue polling queries, and potentially hitting row-size limits.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the buffer from object storage using objectStorageClient at the start of the handler, then delete the temporary object when extraction succeeds or finally fails. This keeps job rows small (< 1 KB).

---

### `server/index.ts`: maybeRunWeeklyDigest / maybeRunAiUsagePurge scheduler registration (bottom of file)
**Issue:** setInterval handles for the weekly digest and AI usage purge schedulers are never stored and never cleared. If the server receives SIGTERM and begins graceful shutdown, these intervals will continue firing, potentially enqueueing jobs against a half-torn-down DB pool and preventing the Node process from exiting cleanly.

**Fix:** Store each interval: `const weeklyDigestInterval = setInterval(maybeRunWeeklyDigest, ...)`. In the SIGTERM/SIGINT graceful-shutdown handler, call `clearInterval(weeklyDigestInterval)` and `clearInterval(purgeInterval)` before closing the HTTP server and DB pool. The shutdown handler itself must already exist to drain in-flight requests; add the interval teardown at the top of that handler.

---

### `server/index.ts`: request logging middleware (res.on finish handler, ~line 72)
**Issue:** Every API response body is serialized via `JSON.stringify(capturedJsonResponse)` and written to the console log unconditionally. For a school-aged-user financial app this will dump quiz scores, user profile data, session tokens embedded in response payloads, and any AI-generated content into stdout/log aggregators in plaintext with no truncation or field redaction.

**Fix:** Gate body logging behind a non-production flag: `if (process.env.NODE_ENV !== 'production' && capturedJsonResponse) { logLine += ' :: ' + JSON.stringify(capturedJsonResponse).slice(0, 200); }`. In production, log only method, path, status code, and duration. If response inspection is needed in production, use a structured logger with a field-level allowlist.

## 🟡 Warnings (16)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregates (total income, total expenses, category breakdown, recent transactions) each issue a separate database round-trip one after another, multiplying latency unnecessarily.
  → *Collect all independent queries into an array and execute with `const [income, expenses, breakdown, recent] = await Promise.all([incomeQuery, expensesQuery, breakdownQuery, recentQuery]);`. Queries that depend on a prior result (none do here) are the only valid reason to keep sequential awaits.*
- **`server/storage.ts` (getTransactions / getAllQuestions):** Neither function enforces a maximum result size. getTransactions accepts an optional `limit` that defaults to nothing, and getAllQuestions has no limit at all. On a growing school dataset these become unbounded SELECTs that can exhaust Node.js heap.
  → *For getTransactions add `const safeLimit = Math.min(filters?.limit ?? 50, 500);` and apply it to the query. For getAllQuestions add `.limit(200)` as a hard cap and document that callers needing more should paginate with an `offset` parameter added to the filter interface.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 113)):** Seven Supabase seed/init operations are chained sequentially with `.then()`. Each waits for the previous to complete, so total startup I/O time is the sum of all operations. Additionally, if any intermediate step throws an error that is not a rejection (e.g. resolves with an error value), subsequent seeds may run on a partially-initialised schema.
  → *Group independent operations with Promise.all where safe: `await Promise.all([initSupabaseTables(), initStaticContentTables()]).then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]))`. Wrap the whole block in a self-invoking async IIFE with try/catch rather than a terminal `.catch` so stack traces are preserved.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 96)):** The `filePath` parameter from the URL is passed directly to `objectStorage.searchPublicObject` without any sanitisation. A path traversal payload like `../../private/secrets` could reach unintended storage paths depending on the ObjectStorageService implementation.
  → *Sanitise the path before use: `import path from 'path'; const filePath = path.normalize(req.params.filePath).replace(/^\.\.\//, '');` and additionally validate it matches an expected pattern such as `/^[\w\-./]+$/`. Reject requests that fail validation with 400.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The entire handler body has no try/catch. An unexpected DB error (connection drop, constraint violation) will surface as an unhandled rejection at the worker level rather than being caught, logged, and allowing the job framework to apply its retry/backoff logic correctly.
  → *Wrap the db.delete call in try/catch: catch the error, call captureError from sentry with context { job: 'purge-ai-usage', olderThanDays }, then re-throw so the job runner records the failure and schedules a retry.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop over questions):** Each extracted question is inserted with a sequential await inside a for loop. For a 60-question exam paper this means 60 round-trips to the DB in series, adding hundreds of milliseconds of unnecessary latency and holding the job worker busy far longer than needed.
  → *Collect all valid question objects into an array and use a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Drizzle supports multi-row insert natively. Wrap it in the existing try/catch so a bulk failure still triggers the terminal-attempt status sync.*
- **`server/jobHandlers.ts` (runExtractPaper – idempotency on concurrent retries):** The delete-then-insert idempotency pattern is not safe if two workers pick up the same job simultaneously (e.g. job visibility timeout is too short and the first attempt is still running when the second starts). The second worker deletes questions the first just inserted, both then race to re-insert, producing duplicates.
  → *Add a database-level unique constraint on (paperId, order) in the extractedQuestions schema, then use INSERT ... ON CONFLICT (paperId, order) DO UPDATE in Drizzle (.onConflictDoUpdate). This makes each upsert idempotent regardless of concurrency and removes the need for the pre-delete step.*
- **`server/jobHandlers.ts` (runExtractPaper – visionResponse image_url content):** When the file is not a PDF, the raw base64 string is re-encoded from the already-decoded buffer (buffer.toString('base64')) and embedded inline in the OpenAI API request. If fileB64 is large (e.g. a high-res JPEG), this doubles memory usage for that string and the full data URI is included verbatim in any error logs or APM traces that capture request bodies, potentially exposing exam content.
  → *After moving to object-storage-backed payloads (see finding 1), generate a short-lived signed URL for the object and pass that as the image_url value instead of a data URI. This eliminates the memory spike and prevents raw image data from appearing in logs.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId and budgets.categoryId have no index. Queries filtering budgets by user (the primary access pattern for any budget dashboard) will do full table scans as data grows.
  → *Add a composite index in the budgets table factory function: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This covers both the common 'get all budgets for user' query and the 'get budget for user+category' check done before upserts.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** portfolioHoldings has no index on userId or stockId, and portfolioTransactions has no index on userId or executedAt. These tables back the investment simulation feature and will be queried by userId constantly; without indexes every portfolio load is a full scan.
  → *Add to portfolioHoldings: (t) => ({ userStockIdx: index('idx_holdings_user_stock').on(t.userId, t.stockId) }). Add to portfolioTransactions: (t) => ({ userDateIdx: index('idx_ptx_user_executed').on(t.userId, t.executedAt) }). The holdings index also enforces efficient uniqueness checks before averaging purchase price.*
- **`shared/schema.ts` (varchar('user_id') on every table referencing users.id):** userId is declared as plain varchar() with no length argument on all tables. If users.id in the auth model has a defined length (e.g., varchar(36) for UUID), mismatched lengths across FK columns can cause silent truncation on some Postgres drivers and will prevent efficient index use. If users.id is truly unbounded text, use text() instead for type consistency.
  → *Audit the users.id column type in shared/models/auth. If it is varchar(n), replace all bare varchar('user_id') with varchar('user_id', { length: n }) to match exactly. If it is a UUID, switch to uuid() type across all tables and use gen_random_uuid() as default on users.id. This also allows Postgres to store FKs more efficiently as fixed-width 16-byte values.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, savingsGoals.createdAt, documentUploads.createdAt (and other createdAt columns)):** All createdAt columns use .defaultNow() but are not marked .notNull(). This means an INSERT that explicitly passes null will succeed and store a null audit timestamp, silently breaking any time-based reporting or ordering that relies on createdAt.
  → *Chain .notNull() after .defaultNow() on every createdAt column: e.g., createdAt: timestamp('created_at').defaultNow().notNull(). The default handles normal inserts; notNull() guards against explicit null injection.*
- **`shared/schema.ts` (simulatedStocks.lastPriceUpdateDate column definition):** lastPriceUpdateDate is typed as text() instead of timestamp(). Storing a date as free-form text means Postgres cannot sort or range-filter it correctly, and any value like '2024-1-5' vs '2024-01-05' will produce wrong orderings. This column is likely used to determine staleness of price data.
  → *Change to lastPriceUpdateDate: timestamp('last_price_update_date') and migrate existing string data with: UPDATE simulated_stocks SET last_price_update_date = last_price_update_date::timestamp. Add .notNull() if a stock record should always have a known price date.*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, startJobWorker calls)):** The startup sequence awaits probeDatabase, seedDatabase, registerRoutes, and startJobWorker in strict series even though probeDatabase and seedDatabase are the only true dependency chain. registerRoutes and startJobWorker do not depend on each other and can run concurrently, adding unnecessary latency before the server begins accepting traffic.
  → *After `await seedDatabase()`, replace the sequential awaits with `await Promise.all([registerRoutes(app, httpServer), startJobWorker()])`. Ensure registerJobHandlers() is called before startJobWorker() as it already is, and confirm neither function has side-effects that race against the other (route registration and worker polling are independent).*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — the `+ 0` is dead arithmetic that does nothing. More critically, the 7pm schedule is hardcoded and uses the server's local timezone (whatever TZ the Replit dyno is set to), not a canonical Caribbean timezone (e.g. America/Port_of_Spain, UTC-4). A server restart in a different TZ or DST edge case will silently misfire the digest.
  → *Replace the bare `Date` arithmetic with a timezone-aware calculation: use the `Intl` API or a minimal library like `date-fns-tz` to anchor to `America/Port_of_Spain`. Expose the timezone as `process.env.DIGEST_TIMEZONE` defaulting to `'America/Port_of_Spain'`. Remove the `+ 0` no-op.*
- **`server/index.ts` (maybeRunAiUsagePurge (purge window query)):** The AI usage purge scheduler deletes records older than AUTO_PURGE_OLDER_THAN_DAYS but the delete operation itself has no try/catch around the actual DB delete call inside the atomic-claim block. If the delete throws (connection drop, constraint), the claim row remains but no job is enqueued and no rollback occurs, permanently blocking future purge runs for that window without any error surfaced beyond a silent swallow.
  → *Wrap the enqueueJob call and any subsequent DB delete inside a try/catch that mirrors the pattern already used in maybeRunWeeklyDigest: on failure, delete the claim row and log the error. Confirm captureError(e) is also called so Sentry captures purge failures, since silent data-retention failures carry compliance risk for a financial app.*

## 🔵 Info / Tech Debt (1)

- **`server/routes.ts` (/healthz handler (~line 79)):** The `/healthz` endpoint dynamically imports `./db` and `drizzle-orm` on every call. ## 💡 Opportunities

### Centralise hardcoded virtual-balance seed amount: ⚡ Medium Impact
The initial virtual balance (commonly $10,000 or similar) is likely hardcoded as a numeric literal inside getVirtualBalance or updateVirtualBalance. Move it to an environment variable `VIRTUAL_BALANCE_INITIAL_USD` with a validated default so it can be changed per deployment without a code release.

### Add a DataLoader layer for category lookups: 🔥 High Impact
Categories are fetched individually inside getBudgets and getBillReminders on every request. Introducing a per-request DataLoader (or a lightweight in-memory Map cached for the request lifetime) would collapse all category fetches into one batched query and benefit every code path that joins categories to another entity.

### Sanitise user-controlled filter fields before interpolation: 🔥 High Impact
String filter values such as `subject` in getAllQuestions and `period` in getLeaderboard are accepted from callers and may be interpolated into raw SQL fragments. Switch to Drizzle's parameterised `.where(eq(column, value))` for every filter field and remove any `sql` template tag usage that embeds caller-supplied strings directly, eliminating SQL injection risk.

### Duplicate health check endpoints: 📌 Low Impact
Both `/api/health` (line ~34) and `/healthz` (line ~79) serve as health checks. `/api/health` returns immediately without a DB probe while `/healthz` actually tests the database. Having two endpoints with different semantics will confuse uptime monitors and Railway config. Consolidate to a single `/healthz` that does the DB probe and remove `/api/health`, or clearly document and differentiate them with consistent naming.

### Structured startup logging for Supabase init: ⚡ Medium Impact
The Supabase init chain only logs on error. Adding per-step timing logs (or a single summary log on success with elapsed ms) would make Railway deployment logs actionable when cold-start times spike. Given this is a school app with potential burst traffic at class start times, knowing which seed step is slow has high operational value.

### Extract host-header trust logic into a shared utility: 📌 Low Impact
The proto/host extraction block (`x-forwarded-proto`, `x-forwarded-host`, fallbacks) is copy-pasted identically in both the robots.txt and sitemap.xml handlers. Extracting it into a `getRequestBaseUrl(req)` helper eliminates duplication and, more importantly, makes it a single place to fix when the canonical-URL env-var change (finding #2) is applied.

### Parallelise independent storage and DB writes in completion handlers: ⚡ Medium Impact
Several handlers (visible in the truncated section) likely do sequential awaits for operations that have no data dependency on each other (e.g. updating exam paper status and recording an AI usage event). Wrapping these in Promise.all would reduce handler wall-clock time proportionally to the number of independent awaits, typically 2-4x for 2-4 concurrent network calls.

### Centralise job payload validation with Zod schemas: ⚡ Medium Impact
Job handlers destructure payload fields (e.g. paperId, fileB64, olderThanDays) without runtime validation. Defining a Zod schema per job type and parsing at handler entry would surface misconfigured enqueueJob call-sites immediately with a clear error rather than a cryptic undefined-access failure mid-handler, reducing debugging time significantly.

### Extract OpenAI model names and token limits to environment config: 📌 Low Impact
The string 'gpt-4o' and max_completion_tokens values (4096) are hardcoded in at least two places in runExtractPaper. Moving these to OPENAI_EXTRACTION_MODEL and OPENAI_MAX_TOKENS env vars allows model upgrades or cost-control adjustments via config without a code deploy.

### Unique constraint on portfolio holdings (userId, stockId): 🔥 High Impact
Without a unique index on (userId, stockId) in portfolioHoldings, concurrent buy orders can insert duplicate rows for the same user+stock, requiring error-prone application-level deduplication. Adding uniqueIndex('idx_holdings_user_stock_unique').on(t.userId, t.stockId) pushes this invariant to the database and allows safe INSERT ... ON CONFLICT DO UPDATE for averaging purchase price atomically.

### Unique constraint on simulatedStocks.ticker: ⚡ Medium Impact
ticker has no unique constraint, so two rows with ticker 'BFSI' can exist, causing non-deterministic results when the app looks up a stock by ticker symbol. Add uniqueIndex('idx_stocks_ticker_unique').on(t.ticker). Expected impact: eliminates a class of data integrity bug with zero application code changes.

### Consolidate repeated currency default into a shared column config: 📌 Low Impact
The string 'BSD' is hardcoded as a default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). If the app ever needs to support TTD or JMD as a second default, all six must be updated in sync. Extract a currencyColumn helper: const currencyCol = () => text('currency').default('BSD').notNull() and reference it in each table. This also ensures the notNull() is never accidentally omitted on a future table.

### Structured startup health log with parallel probe results: ⚡ Medium Impact
Currently startup logs are scattered ad-hoc strings. Emit a single structured JSON startup-complete log line after the server binds, including DB probe latency, seed status, worker started, and Node version. This gives ops a single grep target and makes startup regressions immediately visible in log aggregators.

### Centralised graceful-shutdown coordinator: 🔥 High Impact
Shutdown logic is likely spread across SIGTERM handler, httpServer.close, and DB pool teardown. Introduce a single ShutdownCoordinator that registers all teardown steps (clear intervals, stop job worker, drain HTTP, close DB) with a configurable deadline (e.g. process.env.SHUTDOWN_TIMEOUT_MS defaulting to 10000). This prevents the process hanging indefinitely if any single teardown step stalls.

### Extract scheduler boot into a dedicated module: ⚡ Medium Impact
Moving maybeRunWeeklyDigest, maybeRunAiUsagePurge, and their setInterval registrations into a schedulers.ts module that exports start() and stop() functions would make the intervals testable in isolation, simplify index.ts, and make it trivial to add new scheduled jobs without touching server boot logic.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows several systemic reliability and performance issues: N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits where Promise.all would reduce latency. Given this is a school-aged financial literacy app, security gaps around data exposure and missing pagination guards also need immediate attention.

### API Routes (`server/routes.ts`)
The routing setup is largely well-structured with good practices like parallel module imports and rate limiting on auth routes. Key concerns are unprotected AI/heavy endpoints missing rate limiting, a trust-everything host header approach creating potential Host header injection in robots.txt and sitemap, and sequential Supabase seed operations that block each other unnecessarily.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and security gaps including large base64 payloads stored directly in the job table, sequential DB/storage writes that should be parallelised, and missing try/catch on the AI usage purge handler. The extract-paper handler's idempotency strategy is correct in intent but the sequential question insertion loop will be slow at scale and any mid-loop crash leaves a partial state that the delete-on-retry partially mitigates but does not fully protect against race conditions on concurrent retries.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with foreign keys and some indexes in place, but has several reliability and performance gaps: missing indexes on high-frequency query columns, nullable timestamps that will silently corrupt audit trails, and an unbounded varchar on user_id foreign keys that risks silent data truncation across all tables. The investment simulation tables are notably under-indexed for a feature likely to drive heavy reads.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability gaps: async schedulers (setInterval) lack cleanup handles for graceful shutdown, blocking synchronous-style sequential awaits exist where parallelism is safe, and full JSON response bodies are logged unconditionally which risks leaking sensitive student/financial data in production logs. The overall structure is reasonable but these gaps are unacceptable for a regulated educational finance app.
