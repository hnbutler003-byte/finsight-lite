## 🔴 Critical (5)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These delegate to OpenAI which has per-token cost and latency. A single user or bot can hammer these endpoints, causing runaway API spend and degraded service for all school-aged users.

**Fix:** Add a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); — then call the register functions. Confirm the path prefixes match what those helpers actually mount.

---

### `server/routes.ts` — sitemap.xml handler — const host = req.headers[...]
**Issue:** The Host and X-Forwarded-Host headers are attacker-controlled and are used unvalidated to construct absolute URLs embedded in the sitemap. A forged Host header produces sitemap entries pointing at an arbitrary domain, which can be used to poison caches or assist phishing.

**Fix:** Replace dynamic host derivation with a required environment variable: const base = process.env.PUBLIC_BASE_URL; if (!base) throw new Error('PUBLIC_BASE_URL not set'); — set this to 'https://finsightlite.com' in Railway config. Remove the header fallback entirely.

---

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** Full file base64 is stored in the job payload (fileB64 field). For a PDF or image this can easily be 1–10 MB per job row, bloating the jobs table, slowing queue polling queries, and potentially hitting row-size limits depending on the database.

**Fix:** Upload the file to object storage before enqueuing the job and store only the object path in the payload. In runExtractPaper, download the file by path at runtime: const buffer = await objectStorageClient.bucket(b).file(o).download(); Remove fileB64 from JobPayloads['extract-paper'] and replace with filePath: string.

---

### `shared/schema.ts` — budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no index. Budget queries virtually always filter by userId and frequently join on categoryId, but no index exists — only the FK constraint, which does not create an index in PostgreSQL.

**Fix:** Add an index in the budgets table definition: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) })

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker is declared as text with no unique constraint. Multiple rows with the same ticker symbol would silently corrupt portfolio value calculations and leaderboard lookups that join on ticker. There is no uniqueIndex applied here unlike the learningModules slug.

**Fix:** Add a unique index: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) and ensure application-level upserts use ON CONFLICT(ticker) DO UPDATE to safely handle re-seeding.

## 🟡 Warnings (9)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...)):** The seven-step sequential .then() chain means each seed step blocks the next, and any rejection in steps 2-7 is caught only at the final .catch() with no indication of which step failed. Additionally, steps that are independent (e.g. seedStaticModules, seedRegionalContent, seedGameContent) run sequentially when they could be parallelised.
  → *Split into two stages: await initSupabaseTables() then run independent seeds in parallel. Wrap each call individually so failures are attributable: Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r, i) => { if (r.status === 'rejected') console.error('[Supabase] seed step', i, 'failed:', r.reason); })); Keep this non-blocking by not awaiting the outer expression.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler):** filePath is taken directly from req.params.filePath and passed to objectStorage.searchPublicObject without any path sanitisation. A request to /public-objects/../../private/secrets could traverse outside the intended public prefix depending on how ObjectStorageService resolves paths.
  → *Sanitise the path before use: import path from 'path'; const safe = path.normalize(filePath).replace(/^\.\.\//, ''); if (safe.startsWith('..') || path.isAbsolute(safe)) return res.status(400).json({ error: 'Invalid path' }); then pass safe to objectStorage.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop over questions):** Extracted questions are inserted one-by-one with sequential awaits inside a for loop. For a 40-question paper this serialises 40 round-trips to the database, adding significant latency and leaving the paper in a partially-written state longer than necessary.
  → *Batch insert all valid questions in a single statement: const rows = questions.filter(q => q.question && Array.isArray(q.options) && q.options.length >= 2 && q.answer).map((q, i) => ({ paperId, questionText: q.question, options: q.options.slice(0,4), correctAnswer: q.answer, subject: detectedSubject, difficulty: q.difficulty ?? 'medium', order: i+1 })); if (rows.length) await db.insert(extractedQuestions).values(rows);*
- **`server/jobHandlers.ts` (runExtractPaper — image vision path):** When handling an image file, the raw buffer is re-encoded to base64 and embedded in the OpenAI API request body (buffer.toString('base64')). The buffer was already decoded from the job payload's fileB64, so this is a redundant round-trip through base64 and, combined with the payload-size finding above, means the full binary is held in memory twice.
  → *After fixing finding #1 (store path, not base64), download the buffer once at the top of runExtractPaper and pass it to both branches. This eliminates the double-encode and keeps a single Buffer in scope for the lifetime of the function.*
- **`server/jobHandlers.ts` (extractedText slice — ext === '.pdf' branch):** extractedText is hard-coded to slice at 30,000 characters while the subsequent OpenAI call uses max_completion_tokens: 4096 with no model-specific token budget enforced. The magic number 30000 is invisible to operators and will silently truncate long papers, causing missed questions with no warning in the job result.
  → *Move the limit to an environment variable: const PDF_TEXT_LIMIT = parseInt(process.env.PDF_TEXT_CHAR_LIMIT ?? '30000', 10); Log a warning if the text was truncated: if (pdfData.text.length > PDF_TEXT_LIMIT) logger.warn({ paperId, originalLength: pdfData.text.length }, 'PDF text truncated'). Surface truncated: true in the job return value so the UI can alert the teacher.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler deletes rows but returns rowCount from a cast through unknown. If the underlying driver returns undefined (e.g. after a connection retry), deletedRows silently becomes 0 and the caller has no way to distinguish a genuine zero-row delete from a failed count, masking potential data-retention compliance failures.
  → *Use a SELECT COUNT before deleting, or use Drizzle's returning() clause if supported: const deleted = await db.delete(aiUsageEvents).where(lt(aiUsageEvents.createdAt, cutoff)).returning({ id: aiUsageEvents.id }); return { deletedRows: deleted.length, cutoffDate: cutoff.toISOString() }; This gives an accurate count without casting through unknown.*
- **`shared/schema.ts` (varchar('user_id') columns across all tables):** All userId columns are declared as varchar with no length limit. In PostgreSQL, unbounded varchar is equivalent to text, but this is misleading and inconsistent — more importantly, if user IDs come from a system like Clerk or Auth0 (typical for this stack), the max length is known and should be enforced to prevent oversized data from FK-referenced auth tables.
  → *Declare as varchar('user_id', { length: 128 }) across all tables to match the max length of your auth provider's ID format (e.g., Clerk uses 'user_' + 24 chars = ~32, but 128 is a safe upper bound). This enforces a DB-level constraint and makes intent explicit.*
- **`shared/schema.ts` (createdAt columns across transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt is defined as .defaultNow() but without .notNull(). This means application code or raw SQL inserts can write NULL into createdAt, breaking any ORDER BY createdAt DESC queries silently and causing null-handling bugs in analytics aggregations.
  → *Change all createdAt definitions to: timestamp('created_at').defaultNow().notNull(). The default ensures existing insert paths require no changes.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on userId or nextDueDate. The primary app query for this table is 'find upcoming bills for a user sorted by nextDueDate', which will perform a full table scan as the dataset grows.
  → *Add: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }) to the billReminders table definition. This covers both the user filter and the date sort in a single composite index.*

## 💡 Opportunities

### Parallel domain-router registration — ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then awaited sequentially one-by-one. If any registerXRoutes function does async work (DB pings, schema checks), total registration time is their sum. Collect them in Promise.all as well: await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]). Verify no router depends on a previous router's side-effects first.

### Structured startup logging for Supabase init — ⚡ Medium Impact
Replace bare console.error in the Supabase init chain with a structured logger that includes a timestamp, step name, and error code. During an incident, knowing that seedRegionalContent failed at 09:14 vs a generic 'Init error' significantly reduces mean-time-to-diagnose for a team supporting Caribbean school deployments across multiple time zones.

### Object storage response caching headers audit — ⚡ Medium Impact
downloadObject is called with a 30-day TTL (60*60*24*30) hardcoded inline. If logos or videos are ever updated, users will be served stale content for up to 30 days. Extract this to OBJECT_CACHE_MAX_AGE_SECONDS env var and consider appending a content-hash or version query param to public-object URLs so assets can be cache-busted on update without waiting for TTL expiry.

### Decouple file storage from job enqueue with a pre-upload step — 🔥 High Impact
Introduce a two-phase enqueue: the HTTP route uploads the file to object storage and receives a path, then enqueues the job with only metadata. This keeps the jobs table narrow (<1 KB per row), enables efficient queue polling with indexed scans, and makes retry logic cheaper since the file is already durable. Expected impact: 10–100x reduction in job table row size and faster queue throughput under load.

### Add structured idempotency key logging on extract-paper retries — ⚡ Medium Impact
The current retry path deletes previously inserted questions silently. Adding a structured log entry (paperId, attempt number, deletedQuestions count) before the delete gives operators visibility into retry storms and helps detect upstream issues (e.g. OpenAI rate limits causing repeated failures). This costs one extra query on retry paths only.

### Centralise OpenAI model name and token limits in environment config — ⚡ Medium Impact
gpt-4o and max_completion_tokens: 4096 appear hardcoded in at least two call sites within the visible code. Extracting these to OPENAI_EXTRACTION_MODEL and OPENAI_MAX_TOKENS env vars allows cost/quality tuning without a deploy and prevents version drift when the model is updated, which is especially relevant for a budget-sensitive school app.

### Centralise currency default as a schema-level constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, portfolioHoldings). Extract const DEFAULT_CURRENCY = 'BSD' at the top of schema.ts and reference it in each .default(DEFAULT_CURRENCY) call. This prevents silent drift if the app expands to TTD or JMD and makes the currency policy immediately visible.

### Add a composite index on portfolioHoldings (userId, stockId) with a unique constraint — 🔥 High Impact
portfolioHoldings should logically have at most one row per (userId, stockId) — quantity is updated on each buy/sell rather than inserting new rows. Without a unique constraint, concurrent buy requests can insert duplicate holdings rows, corrupting quantity and avgPurchasePrice. Adding uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) enforces correctness at the DB level and doubles as the covering index for portfolio summary queries.

### Replace serial primary keys with UUIDs for user-facing tables — ⚡ Medium Impact
Using serial (sequential integer) PKs for transactions, savingsGoals, and documentUploads exposes row counts and insertion rates to any user who inspects API responses — a meaningful data leak for a fintech app. Switching to uuid().defaultRandom().primaryKey() eliminates enumeration attacks and aligns with the varchar user_id pattern already used for FK references. Impact: improves security posture with no query-performance downside given FK indexes are already present.
