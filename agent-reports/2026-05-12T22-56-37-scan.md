## 🔴 Critical (7)

### `server/storage.ts` — getPortfolioHoldings
**Issue:** N+1 query pattern: the function fetches all holdings for a user then issues a separate getStockById query per holding inside a loop, causing O(n) round-trips to the database for n holdings.

**Fix:** Replace the per-holding lookup with a single JOIN query using Drizzle's `.leftJoin(simulatedStocks, eq(portfolioHoldings.stockId, simulatedStocks.id))` so all stock data is fetched in one query. Same pattern applies to getPortfolioTransactions.

---

### `server/storage.ts` — upsertPortfolioHolding / createPortfolioTransaction / updateVirtualBalance
**Issue:** Trade execution (buy/sell) performs three separate writes — upsert holding, insert portfolio transaction, update virtual balance — with no wrapping database transaction. A crash between any two steps leaves the user's financial state permanently inconsistent.

**Fix:** Wrap all three operations in `await db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the transaction client `tx` to each write call so they share the same atomic unit of work and roll back together on failure.

---

### `server/storage.ts` — getEnrollmentsByClass
**Issue:** Student PII (full user objects including any stored email or profile fields) is returned for every enrollment row with no field projection. If this data is logged or forwarded to a client response without stripping, it exposes sensitive student data — critical given this is a school-aged user base.

**Fix:** Project only the fields callers actually need: `{ id, displayName, avatarUrl }` using Drizzle's column selection. Define a `SafeStudent` type in schema and use it as the return type so TypeScript enforces the boundary at compile time.

---

### `server/routes.ts` — AI/tutor endpoints (hashTutorQuestion, reserveQuota, openai call pattern)
**Issue:** No rate limiting middleware on AI endpoints. Any authenticated student can fire unlimited requests, exhausting OpenAI quota and budget. The quota reservation system exists but is bypassed if an attacker hammers the endpoint before reservations finalize.

**Fix:** Add express-rate-limit before all /api/ai/* and /api/tutor/* routes: `const aiLimiter = rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.session?.userId ?? req.ip }); app.use('/api/ai', aiLimiter);`. Set max to a value aligned with per-org quota settings already in getOrgQuotaSettings.

---

### `server/routes.ts` — objectStorage / video upload handlers using multer memoryStorage with 500MB limit
**Issue:** videoUpload uses multer.memoryStorage() with a 500 MB file size limit. On a Node.js server, a single concurrent 500 MB upload holds that entire buffer in heap, and multiple concurrent uploads will exhaust memory and crash the process.

**Fix:** Switch videoUpload to multer.diskStorage() writing to a temp directory: `storage: multer.diskStorage({ destination: os.tmpdir(), filename: (_req, file, cb) => cb(null, crypto.randomUUID() + path.extname(file.originalname)) })`. Stream the temp file to object storage, then unlink it. Add a cleanup in a finally block to prevent orphaned temp files.

---

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 (the full base64-encoded exam file) is stored directly in the job payload. For a multi-page PDF this can easily be 5-20 MB per row, multiplied by every retry attempt. This bloats the jobs table, slows queue polling queries, and loads the entire blob into Node heap on every attempt.

**Fix:** Upload the file to private object storage before enqueuing the job, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside the handler. Add a cleanup step after successful extraction to delete the temp object.

---

### `shared/schema.ts` — simulatedStocks table definition
**Issue:** ticker has no unique index, so duplicate tickers can be inserted. Any join from portfolioHoldings or portfolioTransactions to simulatedStocks via ticker-based lookup will return multiple rows, silently corrupting portfolio calculations.

**Fix:** Add uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) inside the table's index callback. Also add an index on (type, riskLevel) since the UI almost certainly filters the stock list by these fields.

## 🟡 Warnings (16)

- **`server/storage.ts` (getDashboardStats):** Dashboard stats executes multiple independent aggregate queries (total income, total expenses, category breakdown, recent transactions) sequentially with await, adding their latencies in series. On a Caribbean school network with higher RTT this is user-visible.
  → *Collect all independent query promises into an array and execute with `const [income, expenses, breakdown, recent] = await Promise.all([...])`. Each query reads independent data so there is no ordering dependency between them.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard accepts an optional `limit` parameter but there is no enforcement of a maximum cap, and callers can omit it entirely. Without a guard, a single request can pull the entire users table ranked, causing unbounded memory allocation and query cost.
  → *Apply a hard cap: `const safeLimit = Math.min(limit ?? 50, 200)` and always pass `safeLimit` to the SQL LIMIT clause. Add a default so callers that omit limit still get a bounded result set.*
- **`server/routes.ts` (Google sign-in handler — resolvedClass resolution block (~line 220-240)):** Three sequential awaits — getOrgEnvironmentById, getOrganization, then domain check — run serially inside a conditional that is itself inside another await. If classCode resolves to an org environment instead of a class, a near-identical serial chain repeats. This blocks the event loop during login, which is latency-sensitive for school-aged users on mobile.
  → *After resolving resolvedClass or resolvedOrgEnv, fetch the env and org in parallel: `const [classEnv, classOrg] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), getOrganization(envId)])`. Remove the nested sequential awaits.*
- **`server/routes.ts` (Leaderboard / lesson listing routes using getLessonsByOrg or getLeaderboard):** Routes that render leaderboards or lesson lists appear to call per-student or per-lesson DB queries inside iteration (evidenced by getStudentOrgIds + per-org lookups pattern). Under a class of 30 students this becomes 30+ sequential round-trips per page load.
  → *Batch the student-org resolution: collect all studentIds first, then call a single `getStudentOrgIds(studentIds[])` that accepts an array and uses a WHERE id = ANY($1) query. If getStudentOrgIds only accepts a single id, add an overload in supabase.ts that accepts id[].*
- **`server/routes.ts` (sendEmail / audit calls throughout route handlers):** Several route handlers await sendEmail or audit() inline before sending the HTTP response. If the email service or audit DB is slow or down, the HTTP response is delayed and the user-facing operation can appear to fail. audit() failures could also suppress the actual success response.
  → *Fire-and-forget non-critical side effects: `void audit(...).catch(err => console.error('audit failed', err))` and `void sendEmail(...).catch(err => console.error('email failed', err))`. Only await them when the response actually depends on their result (e.g., email verification flows).*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Each extracted question is inserted with a sequential await inside a for-loop. For a 40-question paper this serialises 40 round-trips to the database. A crash between iterations leaves a partial set; the idempotency delete at the top of the function handles re-runs but the per-question sequential pattern is unnecessary and slow.
  → *Collect all valid questions into an array, then do a single db.insert(extractedQuestions).values(allRows) bulk insert wrapped in a transaction. This reduces 40 round-trips to 1 and is atomic, eliminating the partial-insert risk entirely.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** There is no try/catch around the db.delete call. An unexpected DB error (connection drop, constraint violation) will produce an unhandled promise rejection inside the job worker. The rowCount cast via unknown is also fragile and will silently return 0 on drivers that don't expose it.
  → *Wrap the delete in try/catch and rethrow with a descriptive message so the job framework records a proper failure. Replace the rowCount cast with a driver-appropriate mechanism (e.g. Drizzle's returning() clause: .delete(...).where(...).returning({ id: aiUsageEvents.id }) and use result.length).*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** The PDF text truncation limit of 30,000 characters is hardcoded. This value directly controls how much of an exam paper is sent to GPT-4o and will silently drop questions on longer papers with no warning to the user or operator.
  → *Move this to an environment variable: PDF_EXTRACT_CHAR_LIMIT with a documented default of 30000. Log a warning when truncation occurs (log the paperId and original vs truncated length) so operators can detect papers that were partially processed.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The read stream's error event suppresses the error object after sending a 500 response, meaning stream errors (permissions, network blip, corrupted object) are silently swallowed with no logging. In a school-facing app this makes storage failures invisible until a user reports a broken download.
  → *In the error handler, log the full error before responding: console.error('[objectStorage] stream error for', objectPath, err). Consider also emitting to your existing monitoring/perf infrastructure so these surface in dashboards without requiring user reports.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on (userId, categoryId) or (userId, period), but queries filtering by user and period or category will be common and will result in full table scans as data grows.
  → *Add a composite index in the table's index callback: index('idx_budgets_user_period').on(t.userId, t.period) and index('idx_budgets_user_category').on(t.userId, t.categoryId). Also add a uniqueIndex on (userId, categoryId, period) to enforce one budget per category per period per user at the DB level.*
- **`shared/schema.ts` (userId: varchar('user_id') across all tables):** All userId columns are plain varchar with no length specified. Without a length constraint, Postgres stores these as unbounded text, preventing efficient indexing and allowing arbitrarily long values that won't match the fixed-length IDs issued by the auth provider.
  → *Set a fixed length matching your auth provider's ID format, e.g. varchar('user_id', { length: 36 }) for UUID-based auth or length: 28 for Firebase UIDs. Apply consistently to every table's userId column and the users.id primary key in models/auth.*
- **`shared/schema.ts` (createdAt columns across transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** createdAt uses .defaultNow() but is not marked .notNull(). This means application code or a partial insert can leave createdAt as NULL, breaking any ORDER BY createdAt queries silently and corrupting audit trails — critical for a financial app.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt on portfolioTransactions and purchasedAt on portfolioHoldings.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate has no index. Bill reminder queries will virtually always filter or sort by nextDueDate (e.g. 'upcoming bills in the next 7 days'), making this a guaranteed sequential scan for every user-facing reminder fetch.
  → *Add a composite index on (userId, nextDueDate): index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) inside the table's index callback.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object and the derived query string are reconstructed on every render of AuditLogPanel, and the queryKey array uses primitive values that are fine, but the queryFn closes over a freshly created `params` object each render. More critically, there is no error state rendered — if the audit log fetch fails, the user sees an empty table with no feedback.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); ... return p; }, [actorType, action, from, to]);`. Add `isError` from useQuery and render an error banner: `{isError && <p className='text-red-400'>Failed to load audit log. <button onClick={refetch}>Retry</button></p>}`.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** ObservabilityCard has no loading or error state. If /api/admin/observability is slow or returns a 500, all badge values silently render as falsy ('Disabled'/'Not configured'), which could mislead an admin into thinking Sentry is disabled when it is actually a fetch failure.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when `isLoading` is true, and a red alert when `isError` is true: `if (isLoading) return <Skeleton />; if (isError) return <p className='text-red-400'>Could not load observability status.</p>;`*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate helper):** `fmtDate` accepts `any` and calls `new Date(d)` without validation. Passing an invalid date string returns 'Invalid Date' rendered directly to the UI. In a financial literacy app showing student records, this creates confusing output and masks bad data coming from the API.
  → *Add a guard: `const dt = new Date(d); if (isNaN(dt.getTime())) return '—';` before calling toLocaleDateString. Also type the parameter as `string | number | Date | null | undefined` instead of `any` to catch misuse at compile time.*

## 💡 Opportunities

### Centralise pagination defaults as environment config — ⚡ Medium Impact
Default page sizes and max limits appear as inline numeric literals in multiple query functions. Extract them to a single `config/pagination.ts` constants file loaded from environment variables with validated fallbacks. This lets operators tune limits for low-resource school deployments without code changes.

### Add a query-level result cache for seedable reference data — 🔥 High Impact
getLearningModules and getMarketStocks return static or near-static data that is read on every student dashboard load. Adding a short-lived in-process cache (node-cache or a simple Map with TTL) for these two calls would eliminate the most frequent read load against the database as class sizes grow.

### Introduce a storage integration test layer before adding more features — 🔥 High Impact
The multi-step trade flow and enrollment operations have no observable test coverage, which means the missing-transaction bugs found above cannot be caught by CI. Adding a test database fixture with Drizzle migrations and a suite of integration tests for write-heavy paths would prevent regressions and make the transaction fixes verifiably correct.

### Extract AI route group into dedicated router module — 🔥 High Impact
All AI/tutor endpoints (quota reservation, cache lookup, OpenAI call, finalizeUsage) share a common middleware chain. Extracting them into server/routes/ai.ts with a shared router-level try/catch and rate limiter would eliminate the repeated boilerplate in each handler and make quota logic testable in isolation. Expected impact: reduces routes.ts by ~300+ lines and makes quota enforcement impossible to accidentally omit on new AI endpoints.

### Add a persistent job queue health check and stuck-job reaper — ⚡ Medium Impact
enqueueJob/getJob are used for video processing but there is no evidence of a stuck-job reaper or timeout. A crashed worker will leave jobs in a processing state forever, silently failing for students. Add a scheduled check (setInterval every 5 min) that queries jobs older than 30 min still in 'processing' state and marks them failed, triggering a client-visible error. Ensure the interval is stored and cleared on server shutdown to prevent memory leaks.

### Centralise environment-variable access with startup validation — ⚡ Medium Impact
Config values (OpenAI key, Supabase URL, Veryfi credentials, appBaseUrl) are accessed via process.env scattered across multiple imported modules with no single validation point. Add a server/config.ts that reads, validates with zod, and exports all env vars at startup, failing fast with a descriptive error before the HTTP server binds. This eliminates the runtime risk of a missing var causing a silent undefined that only surfaces on the first real request.

### Parallel OpenAI calls for vision + extraction pipeline — 📌 Low Impact
In runExtractPaper the vision OCR call and the subsequent structured extraction call are inherently sequential by design, but any independent setup work (paper status update to 'processing', idempotency delete) runs before the slow AI calls. Confirm these DB writes are fire-and-settled before the AI calls begin; if the status update and delete are independent of each other, run them with Promise.all to shave latency on every extract-paper job.

### Job payload schema validation at handler entry — ⚡ Medium Impact
No handler validates its payload shape before using it. A malformed enqueue (e.g. missing paperId or olderThanDays being undefined) causes cryptic runtime errors deep inside the handler rather than a fast, descriptive failure at the boundary. Adding zod schemas per job type at the top of each handler would catch misuse at enqueue time and make retry behaviour predictable.

### Centralised error-to-job-status mapping — ⚡ Medium Impact
Terminal-attempt status syncing (setting exam paper to 'failed') is duplicated inline in the extract-paper handler catch block. As more job types are added this pattern will be copy-pasted. Extracting a withTerminalErrorSync(jobType, id, handler) wrapper would enforce consistent UI state cleanup across all handlers without per-handler boilerplate.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the default currency in at least six separate table definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract it to a single exported constant DEFAULT_CURRENCY = 'BSD' in a shared config file and reference it in all .default() calls. This eliminates the risk of a typo creating mixed-currency data and makes a future multi-currency rollout a one-line change.

### Add check constraint on portfolio quantity and transaction amounts — 🔥 High Impact
quantity on portfolioHoldings and portfolioTransactions, and amount on transactions and budgets, have no DB-level check that values are positive. A bug or bad input could insert zero or negative quantities/amounts that pass application validation but corrupt portfolio value calculations. Add sql`CHECK (quantity > 0)` and sql`CHECK (amount > 0)` constraints directly in the Drizzle table definitions using the .extra() escape hatch to enforce this invariant at the database layer.

### Replace text columns with enum types for low-cardinality status fields — ⚡ Medium Impact
Fields like documentUploads.status, budgets.period, and billReminders.frequency are already constrained with Drizzle's enum option but are stored as unbounded text in Postgres. Create actual Postgres enum types via pgEnum() from drizzle-orm/pg-core and use them as the column type. This reduces storage size, improves index selectivity, and lets Postgres enforce valid values even if migrations are applied outside the ORM.

### Split AdminDashboard into per-tab lazy-loaded modules — 🔥 High Impact
The file is reported at 126k+ characters, meaning all tab panels (DB Viewer, Perf Agent, Background Jobs, etc.) are bundled and parsed on initial load even if the admin never visits those tabs. Use React.lazy + Suspense per tab panel. Expected impact: 40–60% reduction in initial JS parse time for admin sessions.

### Centralise admin query defaults with a dedicated queryClient config — ⚡ Medium Impact
Each useQuery call in child components like ObservabilityCard and AuditLogPanel independently sets no staleTime or retry config, defaulting to React Query globals. Admin data (audit logs, observability status) should have explicit staleTime (e.g. 30s) and retry: 1 to avoid hammering the server on transient failures. Add a shared `adminQueryDefaults` object and spread it into each call.

### Replace inline apiRequest().then(r => r.json()) with a typed helper — 🔥 High Impact
The pattern `apiRequest('GET', url).then(r => r.json())` appears in every queryFn with no error handling — a non-OK response (401, 500) resolves the promise with an error body parsed as valid data. Create a typed `adminFetch<T>(method, url): Promise<T>` helper that checks `r.ok` and throws on failure, giving React Query accurate error state across all admin queries.
