## 🔴 Critical (9)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: fetching each budget's category and calculating 'spent' with a separate query per budget row. For a class with 30 students each viewing a budget list of 10 items, this fires 20+ queries per user per page load.

**Fix:** Replace per-budget queries with a single JOIN: `db.select().from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).where(eq(budgets.userId, userId))`, then aggregate spent amounts in one `GROUP BY budgetId` subquery joined in the same statement. Eliminates N+1 entirely.

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard: getLeaderboard accepts an optional `limit` but if the caller omits it, the query returns all rows from gameSessions joined to users with no cap. On a busy school deployment this can scan the entire game_sessions table and OOM the Node process.

**Fix:** Add a hard default and maximum: `const safeLimit = Math.min(filters?.limit ?? 50, 200);` and always pass `.limit(safeLimit)` to the Drizzle query. Also add a covering index on `(subject, totalScore DESC)` if one does not already exist.

---

### `server/storage.ts`: completeModule
**Issue:** Multi-step write (insert userLearningProgress, then updateUserXp, then conditionally addUserBadge) is performed without a database transaction. A crash between steps leaves XP awarded without progress recorded, or a badge granted without XP, corrupting a student's learning record.

**Fix:** Wrap all three operations in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the `tx` client to each sub-operation so all succeed or all roll back atomically.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These call OpenAI APIs and are the most expensive endpoints in the app. A malicious or runaway client (including a school-aged user) can exhaust API quota or rack up cost with no throttle in place.

**Fix:** Create a dedicated `aiLimiter` (e.g. express-rate-limit: 20 req/min per IP) and apply it before registering those routes: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — or apply a single middleware prefix if those routers share a common path prefix.

### `server/routes.ts`: robots.txt handler (~line 43) and sitemap.xml handler (~line 57)
**Issue:** Both handlers construct an absolute base URL directly from `req.headers['x-forwarded-host']` and `req.headers.host` without validation. An attacker can send a crafted Host header to inject an arbitrary domain into the Sitemap directive and <loc> tags, enabling cache-poisoning or SEO manipulation.

**Fix:** Add an allowlist check: `const ALLOWED_HOST = process.env.PUBLIC_HOST || 'finsightlite.com'; const host = ALLOWED_HOST;` — or at minimum validate the header against a regex of expected domains before using it. Never trust raw Host/x-forwarded-host for URL construction in responses.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** The full base64-encoded file is stored in the job payload in the jobs table. For a multi-page PDF or high-res image this can easily be 5-20 MB per row, bloating the jobs table, slowing queue polling queries, and risking row-size limits depending on the DB.

**Fix:** Upload the file to object storage before enqueuing and store only the object path in the payload. In runExtractPaper, download the file from object storage at the start of the handler using the path. Remove fileB64 and ext from JobPayloads['extract-paper'] and replace with objectPath and mimeType.

---

### `server/index.ts`: res.json logging middleware (~line 68)
**Issue:** Every API response body is serialised with `JSON.stringify(capturedJsonResponse)` and written to stdout unconditionally. For a school-aged user base this will log student quiz answers, progress data, and any field the route returns, violating basic data-minimisation and potentially COPPA/local privacy obligations.

**Fix:** Gate the body log behind a `NODE_ENV !== 'production'` check, or replace full-body logging with a fixed-size summary: `JSON.stringify(capturedJsonResponse).slice(0, 120)` with an explicit truncation marker. Better still, log only `res.statusCode` and `duration` in production and let Sentry capture full payloads on errors only.

---

### `server/index.ts`: maybeRunWeeklyDigest / maybeRunAiUsagePurge setInterval calls
**Issue:** The interval handles returned by `setInterval` for the weekly digest and AI purge schedulers are not stored anywhere, so they cannot be cleared during graceful shutdown. If SIGTERM arrives mid-interval callback the DB operations continue against a closing pool, causing unhandled rejections and potentially corrupt job records.

**Fix:** Capture both handles: `const weeklyHandle = setInterval(...)` and `const purgeHandle = setInterval(...)`. In the SIGTERM/SIGINT handler (which should already exist for the HTTP server) call `clearInterval(weeklyHandle)` and `clearInterval(purgeHandle)` before draining the job worker and closing the DB pool.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage
**Issue:** The chat endpoint `/api/admin/help-chat` sends the full accumulated message history to the server with no length cap. A user (or attacker) can inflate the payload indefinitely across a session, causing runaway token costs on the LLM backend and potential denial-of-service via memory exhaustion on the server.

**Fix:** Slice the context before sending: `const context = next.slice(-20);` and pass `{ messages: context }`. Also enforce a max message count guard at the top of sendMessage: `if (messages.length > 100) { setMessages([]); }` to reset stale sessions.

## 🟡 Warnings (12)

- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregates (total income, total expenses, category breakdown, savings goals balance) execute one after another on the same connection. Each await blocks the next even though the queries share no data dependency.
  → *Replace sequential `const a = await q1; const b = await q2; ...` with `const [a, b, c, d] = await Promise.all([q1, q2, q3, q4])`. For a typical dashboard this reduces wall-clock latency by 60-75% on a pg connection with ~10ms RTT per query.*
- **`server/storage.ts` (getTransactions):** The `limit` filter is optional with no enforced maximum. A missing or maliciously large limit value (e.g. 1000000) causes a full table scan over the transactions table scoped only by userId, loading all rows into Node.js memory before serialising to JSON.
  → *Enforce `const safeLimit = Math.min(filters?.limit ?? 100, 500);` before building the query, and always apply `.limit(safeLimit)`. Add cursor-based or offset pagination parameters to the interface so callers can page through results rather than fetching unbounded sets.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** All seven Supabase seed/init functions are chained sequentially. If any step takes 2-3 seconds (network call to Supabase), startup diagnostic time compounds. More critically, if `seedFinancialAcademyLesson` fails for a transient reason, `initStaticContentTables`, `seedStaticModules`, `seedRegionalContent`, and `seedGameContent` are silently skipped with only a single catch log — leaving the app in a partially-seeded state with no per-step error attribution.
  → *Group independent steps with Promise.allSettled and log per-step failures: `await Promise.allSettled([seedFinancialAcademyLesson(), initStaticContentTables()]).then(results => results.forEach((r,i) => r.status==='rejected' && console.error(`[Supabase] step ${i} failed`, r.reason)));` Run trimOrgNamesInSupabase independently after initSupabaseTables only.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97)):** The filePath parameter is used directly in `objectStorage.searchPublicObject(filePath)` without path sanitization. A request to `/public-objects/../../private/secret` could traverse outside the intended public directory depending on how ObjectStorageService resolves paths.
  → *Sanitize the path before use: `import path from 'path'; const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); if (safePath !== filePath) return res.status(400).json({ error: 'Invalid path' });` — or use `path.basename` if only flat filenames are expected.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a 40-question paper this means 40 serial round-trips to the database before the handler completes, dramatically increasing total handler latency and time-to-failure window.
  → *Replace the loop with a single bulk insert: filter the valid questions first into an array, then call db.insert(extractedQuestions).values(validRows) once. If storage.createExtractedQuestion must be used, batch with Promise.all on chunks of 10 to stay within connection pool limits.*
- **`server/jobHandlers.ts` (purge-ai-usage handler (registerJobHandler call)):** The purge-ai-usage handler has no try/catch. An unexpected DB error (connection drop, constraint violation) will propagate as an unhandled rejection. Unlike the other handlers there is no Sentry capture, so the failure will be silent in observability tooling while the job runner may still mark it failed.
  → *Wrap the delete call in try/catch, capture the error via the shared captureError utility, and rethrow so the job runner can retry: try { const result = await db.delete(...); return {...}; } catch (err) { capture(err, { job: 'purge-ai-usage' }); throw err; }*
- **`server/index.ts` (getMostRecentSundayAt7pm):** Day-of-week calculation `(dow + 0) % 7` always returns Sunday as day 0, meaning on any non-Sunday the function walks back to the previous Sunday correctly, but the dead `+ 0` offset suggests a copy-paste stub where the intended offset was never filled in. More critically, if `now` is exactly Sunday before 7 pm, `d.getTime() > now.getTime()` subtracts 7 days, so the function silently skips the current week's digest until next Sunday.
  → *Replace `(dow + 0) % 7` with plain `dow`. Add a unit test asserting that calling the function at Sunday 18:59 returns the *previous* Sunday and at Sunday 19:01 returns *today*, then decide whether skipping the current Sunday is intentional and document it explicitly in a comment.*
- **`server/index.ts` (top-level startup sequence (probeDatabase, seedDatabase, registerRoutes)):** The startup awaits `probeDatabase()`, `seedDatabase()`, and `registerRoutes()` sequentially before the HTTP server listens. If `seedDatabase` is slow (first deploy with many fixtures) the process holds an open DB connection for tens of seconds with no timeout, and any uncaught rejection in the chain leaves the HTTP server never started with no log beyond the default Node unhandled-rejection warning.
  → *Wrap the entire startup block in a named `async function start()` with a top-level `.catch(err => { captureError(err); process.exit(1); })`. Add an `AbortSignal`-based timeout around `seedDatabase` (e.g. `Promise.race([seedDatabase(), timeout(30_000)])`). `probeDatabase` and any other independent init tasks that don't depend on each other can be parallelised with `Promise.all`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scrollIntoView ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` is recreated on every render and fires scrollIntoView on every re-render of the message list, not just when new messages arrive. This causes continuous scroll fighting when the component re-renders for unrelated reasons (e.g. input typing).
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and attach `ref={bottomRef}` to the sentinel div. This scopes the scroll to actual message changes.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage, apiRequest call):** The catch block swallows the error entirely (`catch { ... }`) with no logging. In a school-facing financial app this makes production debugging impossible and could hide auth failures (401/403) or server errors that indicate a real incident.
  → *Change the catch signature to `catch (err) { console.error('[AdminHelpChat] sendMessage failed:', err); }` and additionally check `res.ok` before calling `res.json()` to surface HTTP-level errors: `if (!res.ok) throw new Error(res.statusText);`*
- **`client/src/pages/AdminDashboard.tsx` (Top-level page component – useQuery calls):** Based on the file size and pattern, multiple independent useQuery calls for users, schools, teachers, analytics etc. are almost certainly fired sequentially at mount with no staleTime set, causing a waterfall of redundant network requests on every navigation to the dashboard and after every mutation invalidation.
  → *Add `staleTime: 60_000` to read-only queries (user lists, school lists, analytics) so React Query serves cached data between navigations. For mutation side-effects, use targeted `queryClient.invalidateQueries({ queryKey: ['users'] })` rather than invalidating all queries, which avoids re-fetching unrelated data.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline filter/sort functions in JSX (search/filter handlers)):** Given the file contains 147k+ characters with search, filter, and pagination logic, computed derived arrays (filtered users, sorted leaderboard rows, paginated slices) are almost certainly computed inline in the render body or as bare expressions, re-executing on every keystroke and every unrelated state change.
  → *Wrap all derived array computations in useMemo: `const filteredUsers = useMemo(() => users.filter(u => u.name.toLowerCase().includes(search.toLowerCase())), [users, search]);`. Similarly wrap sort and pagination slices. This is especially important because the admin dashboard serves as a single large component with many concurrent state slices.*

## 🔵 Info / Tech Debt (4)

- **`server/routes.ts` (/healthz handler (~line 79)):** The /healthz endpoint performs a dynamic `import('./db')` on every invocation. Dynamic imports are cached by Node after the first call, but the import() expression still incurs promise overhead and is semantically misleading — it implies the module may not be loaded, which could mask real module-load errors silently on the first real request.
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (max_completion_tokens: 4096)):** The token limit 4096 is hardcoded in two separate OpenAI calls. If the model or cost policy changes, both call sites must be found and edited manually, and there is no way to tune the limit per environment without a code deploy.
- **`server/jobHandlers.ts` (runExtractPaper – extractedText.slice(0, 30000)):** The 30,000-character truncation limit for PDF text is hardcoded. More importantly, there is no log or warning when truncation actually occurs, so silent data loss happens for long papers and the teacher has no indication that questions from later pages were never processed.
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / hardcoded scheduler intervals):** The scheduler poll intervals (the values passed to `setInterval` for both the digest and purge checks) appear to be hardcoded numeric literals in the truncated code. In a school environment digest timing is subject to term calendars; hardcoding intervals makes tuning require a redeploy rather than a config change.

## 💡 Opportunities

### Batch category hydration with a Map lookup: 🔥 High Impact
getBillReminders and getBudgets both join categories row-by-row. Extract a single `getCategoriesByIds(ids: number[])` helper that uses `inArray(categories.id, ids)` and returns a `Map<number, Category>`. Reuse this map across all hydration sites. Eliminates redundant round-trips and makes the pattern reusable across future features.

### Centralise pagination defaults as named constants: ⚡ Medium Impact
Default and maximum page sizes are currently inline magic numbers scattered across getTransactions, getLeaderboard, and potentially getGameSessions. Define `DEFAULT_PAGE_SIZE = 50` and `MAX_PAGE_SIZE = 500` in a shared config module imported by storage.ts. Makes future tuning a single-line change and prevents drift between endpoints.

### Add query-level result caching for static seed data: ⚡ Medium Impact
getLearningModules and getMarketStocks (between simulateMarketPrices ticks) return data that changes rarely or on a fixed schedule. Wrapping these with a short in-process TTL cache (node-lru-cache, 30s TTL) removes repeated DB hits for every student page load and is safe because the data is not user-specific.

### Centralized AI endpoint rate limiting middleware: 🔥 High Impact
Extract a single `app.use('/api/ai', aiLimiter)` guard that covers all AI sub-routes rather than per-route application. Pair it with per-user limits (keyed on session user ID, not just IP) since school networks share IPs. This would prevent a single student from blocking classmates' access to AI features.

### PUBLIC_HOST environment variable for URL construction: ⚡ Medium Impact
Replace all dynamic host-header-derived base URL construction with a single `process.env.PUBLIC_HOST` value set at deploy time on Railway. This eliminates the header-injection surface, simplifies the robots.txt and sitemap handlers, and makes the canonical domain explicit and auditable.

### Dedicated /healthz auth bypass with structured response: 📌 Low Impact
Ensure /healthz is explicitly excluded from any future auth middleware and returns a structured JSON body including db latency (`dbMs`) so Railway health checks and uptime monitors can detect slow-but-alive database connections, not just hard failures.

### Parallel OpenAI calls where extraction steps are independent: ⚡ Medium Impact
In handlers that make multiple independent OpenAI calls sequentially (vision extraction then JSON structuring are sequential by necessity, but health-check and perf-scan jobs imported on each invocation via dynamic import add cold-start overhead on every run), pre-import heavy modules at module load time rather than inside the handler to eliminate repeated dynamic-import resolution latency on hot paths.

### Idempotency token on extracted-questions delete-then-insert: 🔥 High Impact
The current idempotency strategy deletes all questions for a paperId at the start of each retry, then re-inserts. If the process crashes mid-insert on a retry the paper is left with partial questions and no record of how many were lost. Wrapping the delete + bulk insert in a single DB transaction guarantees atomicity: either all questions are replaced or none are, making retries fully safe without a pre-delete step.

### Structured payload size validation at enqueue time: ⚡ Medium Impact
Adding a payload size guard in enqueueJob (e.g. reject if JSON.stringify(payload).length > 64KB) would catch oversized payloads like base64 files before they reach the DB, giving developers an immediate error at the call site rather than silent table bloat discovered in production monitoring.

### Centralised graceful-shutdown coordinator: 🔥 High Impact
Create a single `shutdown(signal)` async function registered for both SIGTERM and SIGINT that sequentially clears intervals, stops accepting new HTTP connections (`httpServer.close()`), drains the job worker, then closes the DB pool with a 10-second hard timeout. Currently these concerns are scattered or absent; a coordinator prevents the race conditions found in findings 3 and 4 and satisfies container orchestration expectations (Kubernetes, Replit autoscale).

### Structured JSON logging with PII field redaction: 🔥 High Impact
Replace ad-hoc `console.log` string interpolation with a lightweight structured logger (pino is <5 KB and zero-dependency). Define a redact list (`['password','token','answer','sessionId']`) at initialisation. This directly fixes the PII logging finding, makes log ingestion by Sentry/Datadog trivial, and adds sub-millisecond overhead versus the current `toLocaleTimeString` call on every request.

### Parallel independent startup tasks via Promise.all: ⚡ Medium Impact
Group `probeDatabase()` and any config-validation checks into a `Promise.all` before the sequential `seedDatabase → registerRoutes → listen` chain. On a cold Replit container this shaves measurable wall-clock time off the startup path visible to the first student hitting the app after a sleep cycle, without requiring any architectural change.

### Split AdminDashboard into route-level lazy chunks: 🔥 High Impact
At 147k+ characters this file almost certainly contains Users, Schools, Analytics, Leaderboard, and Settings sections in one module. Splitting into separate React.lazy() route components reduces the initial JS parse cost by an estimated 60-70% for admins who only visit one section, and eliminates cross-section re-renders entirely.

### Parallelize independent admin data fetches with Promise.all: 🔥 High Impact
Any server-side or prefetch logic that sequentially awaits users, schools, and analytics should be converted to `const [users, schools, analytics] = await Promise.all([fetchUsers(), fetchSchools(), fetchAnalytics()])`. For a Caribbean deployment with higher latency, this can cut dashboard load time by 40-60% depending on the number of sequential calls.

### Add rate limiting and session token to help-chat endpoint: ⚡ Medium Impact
The `/api/admin/help-chat` endpoint is callable with any authenticated admin session and has no visible rate limit. Adding an express-rate-limit middleware (e.g. 20 requests per minute per admin user ID) and validating the admin JWT on every chat request prevents both cost abuse and prompt-injection attacks targeting the LLM system prompt.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget/bill-reminder lookups, missing database transactions around multi-step writes, and sequential awaits that should be parallelised. For a school-aged financial app, the absence of pagination guards on leaderboard and transaction queries is a reliability risk under load.

The route registration file is generally well-structured with good patterns like parallel module imports and non-blocking Supabase init. Key concerns are: missing rate limiting on AI/chat/image/audio endpoints which are expensive and abuse-prone, host header injection risk in robots.txt and sitemap generation, and the chained Supabase init running sequentially when it could partially parallelize.

jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, missing try/catch on the AI usage purge handler, and a hardcoded token limit that should be environment-configurable.
