## 🔴 Critical (6)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: for each budget returned, a separate query fetches the category and another aggregates spent transactions. With 10 budgets this fires 21 queries instead of 3.

**Fix:** Rewrite using a single Drizzle leftJoin on categories and a subquery or groupBy aggregation for spent amounts: `db.select({...budgetFields, categoryName: categories.name, spent: sql<number>`coalesce(sum(transactions.amount),0)`}).from(budgets).leftJoin(categories, eq(budgets.categoryId, categories.id)).leftJoin(transactions, and(eq(transactions.categoryId, budgets.categoryId), eq(transactions.userId, budgets.userId))).where(eq(budgets.userId, userId)).groupBy(budgets.id, categories.id)`

### `server/storage.ts`: getLeaderboard
**Issue:** No pagination guard: getLeaderboard accepts an optional limit but falls back to returning all rows when limit is undefined, causing full-table scans across gameSessions and users as the user base grows.

**Fix:** Enforce a hard cap: `const safeLimit = Math.min(filters?.limit ?? 25, 100);` and always append `.limit(safeLimit)` to the query. Reject or clamp any caller-supplied value above 100 at the route layer as well.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. In a school-aged user app, a single user or bot could exhaust OpenAI quota or run up unbounded cost with rapid requests.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g., 20 req/min per IP using express-rate-limit) and apply it before each registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three `register*Routes` calls.

### `server/routes.ts`: robots.txt handler (~line 37) and sitemap.xml handler (~line 52)
**Issue:** Both handlers reflect `x-forwarded-host` and `x-forwarded-proto` headers directly into response output without validation. A request with a crafted `x-forwarded-host: evil.com` header will embed that domain in the Sitemap URL and robots.txt, enabling cache-poisoning or SEO injection attacks.

**Fix:** Validate the resolved host against an allowlist from env config: `const ALLOWED_HOSTS = new Set((process.env.ALLOWED_HOSTS || '').split(',')); const host = ALLOWED_HOSTS.has(rawHost) ? rawHost : process.env.DEFAULT_HOST;` Use `process.env.APP_BASE_URL` as the canonical base if set, falling back to the validated header.

---

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler (fileB64 in job.payload)
**Issue:** The entire file is stored as a base64 string inside the job table row (job.payload.fileB64). For a multi-page PDF or high-res image this can easily be 5-20 MB per row, bloating the jobs table, increasing memory pressure on every worker poll, and causing slow JSON parse on each retry attempt.

**Fix:** Upload the file to private object storage immediately when the paper is submitted (before enqueuing), store only the storage path in the job payload (e.g. { paperId, storagePath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. This keeps job rows small and makes retries cheap.

---

### `client/src/pages/AdminDashboard.tsx`: AdminHelpChat – sendMessage
**Issue:** The help-chat endpoint is called with no admin auth token attached and no server-side auth check enforced at the call site. Any unauthenticated user who discovers the endpoint can query the AI assistant, potentially leaking platform-internal context embedded in the system prompt.

**Fix:** Ensure apiRequest forwards the session cookie or Authorization header and add an `isAdmin` guard at the top of sendMessage: `if (!adminUser) return;`. Verify the /api/admin/help-chat route has the requireAdmin middleware applied server-side.

## 🟡 Warnings (16)

- **`server/storage.ts` (getTransactions):** getTransactions has no server-side row cap when limit is not supplied by the caller, so a user with years of transaction history can trigger a full unbounded fetch into memory.
  → *Default limit to a safe value and enforce a maximum: `const rowLimit = Math.min(filters?.limit ?? 50, 500);` then append `.limit(rowLimit)` unconditionally. Add an `offset` parameter for cursor-based pagination on the same call signature.*
- **`server/storage.ts` (completeModule / updateUserXp / addUserBadge (called together in route handler)):** Multi-step write across completeModule, updateUserXp, and addUserBadge executes as three independent statements with no wrapping transaction. A failure after completeModule but before addUserBadge leaves the user in a partially-rewarded state with no rollback path.
  → *Wrap all three operations in a Drizzle transaction: `await db.transaction(async (tx) => { await tx.insert(userLearningProgress)...; await tx.update(userXp)...; await tx.insert(userBadges)...; });` Pass the transaction client (tx) into each sub-operation instead of the module-level db.*
- **`server/storage.ts` (getDashboardStats):** Dashboard stat aggregations (total income, total expenses, savings goals count, budget utilisation) are fired as sequential awaits, adding latency proportional to the number of independent queries when all are data-independent and can run in parallel.
  → *Replace sequential awaits with `const [income, expenses, goalsCount, budgetData] = await Promise.all([queryIncome, queryExpenses, queryGoals, queryBudgets]);` Each query must already carry its own userId/date filter so there is no correctness dependency between them.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 103)):** The seven Supabase seed/init functions are chained sequentially in a fire-and-forget promise. If any step throws, the `.catch` logs it but subsequent steps are silently skipped with no per-step visibility. Additionally, steps that are independent (e.g., seedRegionalContent, seedGameContent) waste startup time running serially.
  → *Group truly independent seeds with `Promise.all` and wrap each step individually: `try { await initSupabaseTables(); await trimOrgNamesInSupabase(); await Promise.all([seedRegionalContent(), seedGameContent()]); } catch(e) { console.error('[Supabase] Init failed at step:', e); }` Run this as a named async IIFE so each failure is attributable.*
- **`server/routes.ts` (await registerAuthDomainRoutes ... await registerEmailRoutes (~lines 117-122)):** The six domain routers are imported in parallel via Promise.all (good) but then registered with six sequential awaits. If any registration function does async work (DB calls, sub-imports), this adds cumulative latency to server startup unnecessarily.
  → *Replace the six sequential awaits with `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), registerAiRoutes(app), registerOrgRoutes(app), registerLessonRoutes(app), registerEmailRoutes(app)]);` — valid as long as route registrations don't depend on each other's side effects, which standard Express middleware registration does not.*
- **`server/jobHandlers.ts` (runExtractPaper – for loop over questions with sequential awaits):** Each extracted question is inserted with an individual awaited storage.createExtractedQuestion call inside a for-loop. For a 40-question paper this fires 40 sequential round-trips to the database, adding unnecessary latency and holding the DB connection open longer than needed.
  → *Replace the loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(questionRows) once. If storage.createExtractedQuestion must be used, at minimum use Promise.all on batches of 10.*
- **`server/jobHandlers.ts` (runExtractPaper – max_completion_tokens: 4096 (hardcoded)):** Token limits for both the vision extraction call and the JSON extraction call are hardcoded to 4096. If OpenAI raises limits or the product needs tuning, this requires a code change and redeploy rather than a config update.
  → *Add OPENAI_MAX_COMPLETION_TOKENS (or separate EXTRACT_TEXT_MAX_TOKENS / EXTRACT_JSON_MAX_TOKENS) to environment config with a default fallback: const maxTokens = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10). Reference this variable in both completions.create calls.*
- **`server/jobHandlers.ts` (runExtractPaper – vision branch: buffer.toString('base64') called again after already decoding from base64):** The job payload already contains fileB64 (the original base64 string). The code decodes it to a Buffer with Buffer.from(fileB64, 'base64') and then immediately re-encodes it with buffer.toString('base64') to embed in the OpenAI request URL. This is a wasteful round-trip and doubles peak memory for the image data.
  → *Pass fileB64 directly to the image_url data URI: url: `data:${mimeType};base64,${fileB64}`. Remove the Buffer.from decode step for the vision branch, or decode once and reuse the original string.*
- **`server/jobHandlers.ts` (purge-ai-usage handler – no try/catch, no row-count guard):** The purge-ai-usage handler has no error boundary. A DB connection failure will propagate as an unhandled rejection in the worker. Additionally, olderThanDays comes directly from job.payload with no validation, so a payload of { olderThanDays: 0 } would compute a cutoff of right now and delete the entire table.
  → *Wrap the delete in try/catch and return a structured error result instead of throwing, so the job is marked failed with a meaningful message. Add a minimum guard: if (olderThanDays < 1) throw new Error('olderThanDays must be >= 1') before computing the cutoff.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId has no index. Every query filtering budgets by user (the primary access pattern for this table) results in a full table scan. As user count grows this will degrade noticeably.
  → *Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget entries per period.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker column has no unique constraint and no index. Ticker symbols must be unique per asset type, and portfolio lookups joining on ticker will be doing full scans on a reference table every time.
  → *Replace the plain text column with: ticker: text('ticker').notNull(), then add in the table config: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also add .notNull() to issuer, region, and annualReturnPct or provide explicit defaults so application code doesn't have to null-guard everything.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** Neither portfolioHoldings nor portfolioTransactions has an index on userId or stockId despite being the core join path for portfolio summary queries. A user with many transactions will hit sequential scans on both tables on every dashboard load.
  → *Add to portfolioHoldings: (t) => ({ userStockIdx: index('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). Add to portfolioTransactions: (t) => ({ userIdx: index('idx_portfolio_txn_user').on(t.userId), executedAtIdx: index('idx_portfolio_txn_executed').on(t.executedAt) }).*
- **`shared/schema.ts` (All tables with createdAt columns (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt is defined as .defaultNow() without .notNull(). Drizzle will type this as Date | null even though a server default always populates it. This forces unnecessary null checks throughout application code and allows a NULL to be inserted if the default is ever bypassed (e.g., raw SQL migration or bulk insert).
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g.: createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for executedAt in portfolioTransactions and purchasedAt in portfolioHoldings.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` creates a new function on every render and fires scrollIntoView every time any state in AdminHelpChat changes (e.g. input keystroke), not only when a new message arrives. This causes janky scroll behaviour and wastes layout work.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (Dashboard stats data fetching (useQuery blocks for users, schools, lessons, activity)):** Multiple independent useQuery calls for dashboard statistics are defined sequentially, but if any underlying fetcher uses await chains internally they will block each other. More critically, if the same data is also fetched inside child tab components, the same endpoints are called redundantly on tab switch with no staleTime set, hammering the API on every re-render.
  → *Set a shared `staleTime: 60_000` on all dashboard stat queries to prevent redundant refetches on tab switch. For any truly independent parallel server calls inside a single async fetcher, replace sequential awaits with `Promise.all([fetchA(), fetchB(), fetchC()])` to cut aggregate latency.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline mutation handlers passed as JSX props (onSubmit, onClick handlers in table rows)):** Mutation callbacks and filter/sort handlers are defined as arrow functions directly inside JSX (e.g. `onClick={() => deleteMutation.mutate(row.id)}`). Because these are recreated on every parent render, every table row re-renders even when unrelated state (e.g. search input) changes, causing visible lag with large user/school lists.
  → *Wrap stable handlers with useCallback: `const handleDelete = useCallback((id: number) => deleteMutation.mutate(id), [deleteMutation]);` and pass `handleDelete` to row components. Extract table row into a memoized sub-component with React.memo so referential stability is preserved.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (/api/health (~line 27) and /healthz (~line 73)):** Two health check endpoints exist (`/api/health` and `/healthz`) with different behaviour — one is a trivial ping, one checks DB. This creates confusion for Railway and uptime monitors about which to use, and `/api/health` falls under the `Disallow: /api/` robots.txt rule, meaning monitors using it may be blocked by over-eager crawlers or WAF rules.
- **`shared/schema.ts` (categories table definition):** categories.userId uses varchar without a length specifier. PostgreSQL varchar without length is equivalent to text, so this is harmless at the DB level, but it is inconsistent with the FK target (users.id) and will mislead developers about the expected value size. More critically, there is no index on categories.userId, meaning filtering user-owned categories (the most common read) is a full scan.
- **`client/src/pages/AdminDashboard.tsx` (Hardcoded strings: API base paths and pagination page size):** Values such as the page size (e.g. `const PAGE_SIZE = 10`) and any hardcoded API origin strings appear inline in the component rather than being drawn from environment config. In a school deployment with varying dataset sizes or a staging vs production API split, this requires code changes instead of environment variable overrides.

## 💡 Opportunities

### Centralised pagination middleware: 🔥 High Impact
Extract a `parsePagination(query: unknown, maxLimit = 100)` utility that validates and clamps limit/offset for every list endpoint. Applying it consistently across getTransactions, getLeaderboard, getGameSessions, and getAllQuestions eliminates repeated ad-hoc guards and prevents future regressions as new list methods are added.

### Drizzle transaction wrapper for reward flows: 🔥 High Impact
Introduce a thin `withTx(fn)` helper that accepts a callback receiving a transaction client. All multi-step reward and enrollment writes (completeModule, enrollStudent, createGameSession+updateUserXp) should use it. This prevents partial state in the student-facing gamification layer, which is especially important for school users where trust in score integrity matters.

### Replace in-process seedMarketData/seedLearningModules with idempotent migration scripts: ⚡ Medium Impact
Seed functions called from application code run on every cold start and hold no guard against duplicate rows beyond a try/catch. Moving them to a dedicated seed script invoked once via `npm run seed` removes startup latency, eliminates duplicate-insert risk, and keeps the storage interface free of environment-setup concerns.

### Canonical APP_BASE_URL environment variable: 🔥 High Impact
Replace all `x-forwarded-proto` + `x-forwarded-host` header assembly in robots.txt and sitemap with a single `process.env.APP_BASE_URL` env var (e.g., `https://finsightlite.com`). This eliminates the injection surface, simplifies the code to one line each, and makes staging/production URL switching trivial via Railway env config.

### Structured startup logging for Supabase init steps: ⚡ Medium Impact
Replace the silent sequential `.then()` chain with named async steps that log entry and exit timestamps. This gives operators visibility into which seed step is slow or failing during Railway deploys, reducing mean-time-to-diagnose on startup regressions.

### ObjectStorageService singleton extraction: 📌 Low Impact
`new ObjectStorageService()` is instantiated at module load time in routes.ts. If ObjectStorageService opens connections or holds state, multiple hot-reloads in development leak instances. Move instantiation into a module-level singleton (e.g., `storage/index.ts`) exported and imported where needed, so it is constructed exactly once per process.

### Idempotency token on extracted-question bulk insert: 🔥 High Impact
The current idempotency approach deletes all questions for a paperId on each retry before re-inserting. A bulk upsert keyed on (paperId, order) would be safer and avoid a window where questions are missing if the process crashes between delete and re-insert. Use db.insert(...).values(...).onConflictDoUpdate() with a unique index on (paper_id, order).

### Stream PDF parsing instead of loading full buffer into memory: ⚡ Medium Impact
pdf-parse loads the entire PDF buffer into memory synchronously. For large exam papers this spikes RSS on the worker process. Switching to a streaming parser (e.g. pdf2json with stream input) or enforcing a hard file-size limit before enqueue would cap worst-case memory impact per job.

### Centralise OpenAI call retries with exponential back-off: ⚡ Medium Impact
Both openai.chat.completions.create calls in runExtractPaper rely solely on the job-level retry mechanism. A transient 429 or 503 from OpenAI will fail the entire job and consume an attempt. Wrapping OpenAI calls in a small local retry-with-backoff utility (max 3 inner retries, 1s/2s/4s delays) would reduce wasted job attempts and improve throughput under rate-limit conditions.

### Centralise currency default as a schema-level constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least six separate column definitions (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract const DEFAULT_CURRENCY = 'BSD' at the top of the file and reference it in each .default() call. When the app needs to support TTD or JMD this becomes a single-line change instead of a grep-and-replace across the schema.

### Add composite unique constraint on budgets to prevent duplicate periods: 🔥 High Impact
Nothing stops the application from inserting two monthly budgets for the same (userId, categoryId, period) combination. This is a data integrity gap that currently relies entirely on application-layer validation. A uniqueIndex on (userId, categoryId, period) in the budgets table config makes the constraint database-enforced, eliminates the need for a pre-insert existence check in route handlers, and prevents duplicates from raw migrations or future admin tooling.

### Replace repeated currency/period text columns with a shared Postgres enum: ⚡ Medium Impact
currency and period values are repeated across many tables as plain text with no DB-level enforcement of the allowed set. Defining a Postgres enum (pgEnum) for currency (BSD, TTD, JMD, USD) and for period (weekly, monthly, quarterly, yearly) enforces valid values at the DB layer, reduces per-row storage by roughly 8-16 bytes per column compared to unbounded text, and makes schema intent explicit. Drizzle-orm supports pgEnum natively.

### Virtualise large data tables: 🔥 High Impact
Rendering all users or schools into the DOM at once (even paginated client-side) forces the browser to layout hundreds of rows. Replacing the table body with @tanstack/react-virtual would cut initial paint time significantly for admins managing schools with large student rosters, which is a realistic Caribbean district scenario.

### Centralise query key factory: ⚡ Medium Impact
Query keys are currently string literals scattered across the file (e.g. ['admin-users'], ['admin-schools']). Introducing a typed queryKeys object (`export const adminKeys = { users: () => ['admin','users'] as const, ... }`) eliminates typo-driven cache misses and makes targeted invalidation after mutations reliable.

### Add Suspense and ErrorBoundary wrappers per dashboard section: ⚡ Medium Impact
Currently a single query error in any section (e.g. activity chart) has no isolated error UI, so the entire dashboard may show nothing. Wrapping each card section in an ErrorBoundary with a lightweight fallback lets the rest of the dashboard remain functional, which is important for admin users in low-connectivity Caribbean school environments.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill-reminder lookups, missing database transactions on multi-step writes, and sequential awaits that should run in parallel. For a school-aged financial app, the absence of pagination guards on unbounded list queries and missing try/catch on async DB calls are the most pressing reliability and safety concerns.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good separation of concerns and proper error handling in most async paths. Key concerns are: unprotected AI/chat/image/audio routes with no rate limiting, host header injection risk in robots.txt and sitemap, and sequential domain-router registration that adds unnecessary cold-start latency.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, and a hardcoded token limit that belongs in environment config. The extract-paper handler is the riskiest area: it stores potentially multi-MB file data in the job row, inserts questions one-by-one in a loop, and exposes raw base64 image data to the OpenAI vision API directly from the job payload on every retry.

### Database Schema (`shared/schema.ts`)
The schema is generally well-structured with proper foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns include missing indexes on high-frequency query columns, nullable timestamps that should be NOT NULL, and missing NOT NULL constraints on createdAt columns that could cause silent data integrity issues.
