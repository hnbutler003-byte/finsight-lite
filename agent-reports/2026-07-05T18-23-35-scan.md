## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These call OpenAI APIs and are the most expensive endpoints in the app. Any unauthenticated or authenticated user can spam them, causing runaway API costs and potential DoS.

**Fix:** Add a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three registerX calls.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 42 and ~line 57)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header (e.g. evil.com) and the response will contain Sitemap: https://evil.com/sitemap.xml or URLs pointing to an attacker-controlled domain, enabling cache-poisoning or phishing via poisoned sitemaps cached by crawlers.

**Fix:** Read the canonical base URL from an environment variable set at deploy time (e.g. process.env.APP_BASE_URL) and use that exclusively. Fall back to the header only in local dev when APP_BASE_URL is unset, and strip any characters outside [a-zA-Z0-9.:-] from the host before use.

### `server/jobHandlers.ts`: runExtractPaper / extract-paper handler
**Issue:** Full file contents are stored as base64 in the job payload (fileB64 field). For a typical exam PDF this can be 500KB–5MB of base64 in the jobs table row, multiplied by every retry attempt. This bloats the job table, slows queue queries, and can exhaust DB row size limits.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload. In runExtractPaper, download the file from object storage at the start using streamPrivateObjectToResponse or a direct buffer read. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `server/index.ts`: getMostRecentSundayAt7pm
**Issue:** Incorrect day-of-week math: `(dow + 0) % 7` is a no-op. Sunday (dow=0) produces daysSince=0, so d is never rolled back and the function returns the current day at 19:00 rather than the most recent Sunday, causing the digest to fire on every poll interval regardless of the day.

**Fix:** Change the offset to correctly target Sunday: `const daysSince = dow;` then verify the guard `if (d > now) d.setDate(d.getDate() - 7)` still handles the Sunday-before-7pm edge case. Add a unit test covering Monday, Sunday-before-19:00, and Sunday-after-19:00.

---

### `server/index.ts`: res.json logging middleware (~line 68)
**Issue:** Every API response body is serialised into the log line with `JSON.stringify(capturedJsonResponse)`. For FinSight Lite this will echo quiz answers, student progress, guardian contact details, and any JWT/session tokens that leak into response payloads — a COPPA/privacy violation for school-aged users.

**Fix:** Remove the response-body capture entirely, or introduce an allowlist: only log `res.statusCode` and `duration`. If debugging is needed, gate it behind `process.env.LOG_RESPONSE_BODIES === 'true'` and strip known sensitive keys (email, token, secret) before logging.

### `client/src/pages/AdminDashboard.tsx`: sendMessage – apiRequest call in AdminHelpChat
**Issue:** The full `messages` array (entire conversation history) is sent to /api/admin/help-chat on every turn with no length cap. For a long session this payload grows unbounded, increasing latency and potentially leaking prior sensitive admin context to the LLM provider.

**Fix:** Slice to the last N turns before sending: `const payload = next.slice(-10);` and pass `payload` instead of `next` to apiRequest. Add a server-side max-token guard as well.

---

### `client/src/pages/AdminDashboard.tsx`: AdminDashboard – useAdminAuth / route guard
**Issue:** If useAdminAuth returns a loading state before the session is confirmed, the dashboard briefly renders (and fires all useQuery network requests) before the redirect to login occurs. This leaks admin data fetches for unauthenticated users who land on the route momentarily.

**Fix:** Add an early loading guard before any query hooks execute: immediately after calling useAdminAuth, return a spinner if `authLoading` is true and return null/redirect if `!isAdmin`, before any useQuery declarations. Move all data-fetching hooks after the auth check resolves.

## 🟡 Warnings (12)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** The seven Supabase init steps are chained sequentially with .then(). A failure in any intermediate step (e.g. seedFinancialAcademyLesson) short-circuits all subsequent steps (seedStaticModules, seedRegionalContent, seedGameContent) and only a single catch at the end logs the error. Downstream steps that are independent are never executed, leaving the DB partially seeded with no per-step visibility.
  → *Run independent steps with Promise.allSettled and log each rejection individually: const steps = [trimOrgNamesInSupabase, seedFinancialAcademyLesson, initStaticContentTables, seedStaticModules, seedRegionalContent, seedGameContent]; await initSupabaseTables(); const results = await Promise.allSettled(steps.map(fn => fn())); results.forEach((r, i) => r.status === 'rejected' && console.error(`[Supabase] Step ${steps[i].name} failed:`, r.reason));*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** There is no authentication or authorization check before serving objects from object storage. Any unauthenticated request can enumerate and download any file by guessing or brute-forcing filePath. For a school-aged user app this could expose student-related assets.
  → *If these objects are truly public (logos only), enforce a path prefix allowlist: if (!/^logos\//.test(filePath)) return res.status(403).json({ error: 'Forbidden' }); For any non-public paths, require a valid session before proceeding.*
- **`server/jobHandlers.ts` (runExtractPaper – question insert loop (for loop over questions)):** Each extracted question is inserted with an individual awaited storage.createExtractedQuestion call inside a for loop. For a 60-question paper this means 60 sequential round-trips to the database, adding seconds of latency and holding the job worker busy unnecessarily.
  → *Replace the for loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(validQuestions) once. If storage.createExtractedQuestion must be used, batch with Promise.all on chunks of 10 to avoid overwhelming the pool while still parallelising.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The db.delete call has no try/catch. If the delete fails (DB timeout, connection drop), the job throws an unhandled rejection with no context logged, and the raw DB error including internal table names may propagate to the job runner's error log, leaking schema details.
  → *Wrap the db.delete in try/catch. On catch, log a sanitised message (e.g. 'ai-usage purge failed, cutoff=<date>') via your logger rather than rethrowing the raw DB error. Re-throw a generic Error so the job runner can retry without exposing schema internals.*
- **`server/jobHandlers.ts` (runExtractPaper – idempotency delete before re-extraction):** The idempotency guard deletes all previously extracted questions for the paperId unconditionally at the top of every attempt, including the first. If a concurrent request or a race between retry attempts runs two workers simultaneously, both will delete each other's partial inserts, resulting in a corrupt or empty question set that both workers then race to repopulate.
  → *Add a database-level advisory lock or a status check before deletion: only delete if exam_papers.status is still 'processing'. Additionally, set status to a 'retrying' sentinel before deletion and use a WHERE status != 'retrying' guard so a second concurrent worker exits early rather than proceeding.*
- **`server/jobHandlers.ts` (parsePrivatePath / uploadPrivateObject):** PRIVATE_OBJECT_DIR is checked at call time inside parsePrivatePath, but there is no startup validation. If the env var is missing, the error surfaces only when a job runs, potentially after a file has already been processed, causing partial state with no object stored. Additionally, relPath input is not sanitised against path traversal sequences (e.g. '../../secrets').
  → *Validate PRIVATE_OBJECT_DIR at server startup (alongside other required env vars) and throw a fatal error if absent. In parsePrivatePath, reject any relPath containing '..' segments: if (parts.some(p => p === '..')) throw new Error('Path traversal detected'). This is especially important for a school-aged user app where uploaded files come from teachers.*
- **`server/index.ts` (setInterval calls in startup (maybeRunWeeklyDigest / maybeRunAiUsagePurge)):** The intervals returned by setInterval are never stored or passed to the graceful-shutdown handler. On SIGTERM the process exits while the callbacks may still be mid-await, risking a torn DB transaction (e.g. the digest claim is inserted but the rollback on enqueue failure never runs).
  → *Store handles: `const digestInterval = setInterval(...); const purgeInterval = setInterval(...)`. In the SIGTERM/SIGINT handler call `clearInterval(digestInterval); clearInterval(purgeInterval);` before closing the HTTP server and DB pool. Use a boolean `shuttingDown` flag inside each callback to skip execution if shutdown has started.*
- **`server/index.ts` (startup sequence (seedDatabase, probeDatabase, registerRoutes, serveStatic)):** The startup awaits seedDatabase and probeDatabase sequentially before the HTTP server begins listening. If seedDatabase is slow (large fixture set on first deploy) the process is unkillable and health checks fail, causing the platform to restart the dyno in a loop.
  → *Separate liveness from readiness: call `httpServer.listen()` immediately after `registerRoutes()`, returning 503 from a `/healthz` route until a `ready` flag is set true. Run `probeDatabase` and `seedDatabase` concurrently with `Promise.all([probeDatabase(), seedDatabase()])` then set the flag. Add a startup timeout (e.g. 30 s) that calls `process.exit(1)` if readiness is never achieved.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest):** Neither scheduler awaits its own previous invocation before the next interval fires. If the DB is slow, two concurrent runs can race past the `onConflictDoNothing` guard within the same 60-second window, duplicating enqueued jobs.
  → *Wrap each interval body with a mutex flag: `let digestRunning = false; if (digestRunning) return; digestRunning = true; try { await maybeRunWeeklyDigest(); } finally { digestRunning = false; }`. This prevents overlapping executions without changing the schedule.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and fires scrollIntoView on every state update (including while typing), not just when a new message arrives. This causes janky scroll behavior and wastes layout cycles.
  → *Replace with a useRef + useEffect that only fires when messages.length or loading changes: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages.length, loading]);` then use `<div ref={bottomRef} />`.*
- **`client/src/pages/AdminDashboard.tsx` (AdminDashboard – top-level useQuery calls):** Multiple independent useQuery calls (users, schools, lessons, analytics, audit log, etc.) are declared sequentially and each defaults to its own staleTime/retry config. Any component that causes a parent re-render will re-evaluate all query selectors. Derived values computed inline from query data (filtering, sorting, pagination slices) are recomputed on every render without memoization.
  → *Wrap all per-query derived computations (filtered lists, paginated slices, stat aggregates) in useMemo with the relevant query data as dependencies. Example: `const filteredUsers = useMemo(() => users?.filter(...), [users, searchTerm, roleFilter]);`*
- **`client/src/pages/AdminDashboard.tsx` (AdminDashboard – mutation handlers (deleteUser, saveLesson, etc.)):** Large inline arrow functions are passed directly as onClick handlers (e.g. `onClick={() => deleteUser(row.id)}`) throughout JSX for mutations. These are recreated every render for every table row, preventing React from bailing out of child re-renders and adding GC pressure proportional to list size.
  → *Wrap each mutation handler with useCallback: `const handleDeleteUser = useCallback((id: number) => deleteUser(id), [deleteUser]);` and pass the stable reference to row components, or extract row components so they can memo-ize individually.*

## 💡 Opportunities

### Parallel domain-route registration: ⚡ Medium Impact
The six registerX(app) calls after the parallel import are still awaited sequentially. If any of these do async work at registration time (e.g. DB schema checks), they add to cold-start latency. Wrap them in Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) to parallelise registration and reduce startup time, provided the routes do not depend on each other's side effects.

### Structured startup logging for Supabase init: ⚡ Medium Impact
Currently init failures produce a single console.error line with no timing or step identity. Adding a simple wrapper that logs step name and duration (Date.now() delta) before and after each seed function would make it trivial to identify slow or failing seed steps in production logs without a debugger.

### ObjectStorageService singleton extraction: 📌 Low Impact
objectStorage is instantiated at module load time as a module-level const. If ObjectStorageService opens connections or allocates resources, multiple hot-reloads in development will leak instances. Move instantiation inside registerRoutes or into a dedicated singleton module with an explicit destroy/cleanup hook tied to server close events.

### Pre-flight env var validation at startup: 🔥 High Impact
Introduce a single validateEnv() function called at process start that asserts all required env vars (PRIVATE_OBJECT_DIR, OPENAI_API_KEY, DB_URL, etc.) are present and non-empty. This eliminates an entire class of runtime failures that currently surface mid-job and corrupt partial state.

### Structured job payload size enforcement: 🔥 High Impact
Add a payload size guard in the enqueueJob wrapper that throws if JSON.stringify(payload).length exceeds a configured limit (e.g. 64KB). This acts as a backstop against any future handler accidentally storing large blobs, and is trivially implementable as a one-line check before the DB insert.

### Centralise OpenAI call error handling with token usage logging: ⚡ Medium Impact
Wrap all openai.chat.completions.create calls in a shared helper that logs prompt/completion token counts to aiUsageEvents and normalises errors (rate limit vs. context length vs. network) into typed errors. Currently each call site has ad-hoc or no error handling, making cost attribution and quota management invisible.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase and seedDatabase are independent and can run concurrently. Using Promise.all cuts startup latency by whichever is slower minus the faster — typically saving 500 ms–2 s on cold starts against a remote Postgres instance, which matters for Replit's container spin-up SLA.

### Centralised scheduler registry for clean shutdown: 🔥 High Impact
Introduce a lightweight ShutdownManager that accepts interval handles, child processes, and open sockets, then drains them in dependency order on SIGTERM. This prevents the current ad-hoc signal handler from growing unbounded as new schedulers are added and ensures the job worker flushes its queue before exit.

### Structured logging to replace string concatenation: 🔥 High Impact
Replace the bespoke log() function and inline JSON.stringify with a structured logger (pino is already popular in the Node ecosystem and has negligible overhead). This enables log-level filtering so DEBUG response bodies never reach production, and gives Sentry/log aggregators parseable fields instead of free-text strings — directly reducing the PII-in-logs risk.

### Parallel data fetching with Promise.all on initial load: 🔥 High Impact
Bootstrap queries for users, schools, lessons, and analytics are all independent but sequential. Consolidate them into a single /api/admin/bootstrap endpoint that fetches all in parallel server-side via Promise.all, reducing round trips from 4+ to 1 and cutting initial dashboard TTI by the sum of the slowest sequential queries.

### Virtualise long admin tables: ⚡ Medium Impact
User and audit-log tables render all rows into the DOM and apply pagination purely in JS. Replace with TanStack Virtual (already likely in the dependency tree via TanStack Query) to render only visible rows, keeping DOM node count constant regardless of dataset size and eliminating the current O(n) re-render cost on filter changes.

### Extract tab panels into lazy-loaded route segments: ⚡ Medium Impact
All dashboard tabs (Users, Schools, Lessons, Analytics, Audit Log, AI Chat) are bundled and mounted together. Wrapping each tab panel in React.lazy + Suspense would cut initial JS parse time by ~60-70% for admins who only use one or two sections, and prevent heavy Recharts chart code from loading until the Analytics tab is actually opened.
