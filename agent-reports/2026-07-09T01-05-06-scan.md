## 🔴 Critical (7)

### `server/routes.ts`: /robots.txt and /sitemap.xml handlers (~line 39-78)
**Issue:** x-forwarded-host and x-forwarded-proto are used without validation to construct absolute URLs injected into response bodies. An attacker can set these headers to inject arbitrary URLs into the Sitemap directive and sitemap.xml <loc> values, and manipulate crawler behaviour or perform limited response injection.

**Fix:** Resolve the base URL once at startup from a trusted environment variable (e.g. BASE_URL=https://finsightlite.com). Fall back to req.protocol+req.hostname only if the app is behind a trusted reverse proxy with express's 'trust proxy' setting enabled. Never read x-forwarded-host directly in handlers.

### `server/routes.ts`: /public-objects/:filePath(*) handler (~line 103-112)
**Issue:** filePath is taken directly from req.params.filePath and passed to objectStorage.searchPublicObject without path sanitisation. A request to /public-objects/../private/secret could traverse outside the intended public directory depending on how ObjectStorageService resolves paths.

**Fix:** Sanitise the path before use: const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|^\//, ''); and reject any path that starts with '..' or is absolute. Add a check that the resolved key starts with the expected prefix (e.g. 'public/').

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** portfolioHoldings has no unique constraint on (userId, stockId). Multiple rows for the same user and stock can accumulate, causing double-counting of holdings when aggregating quantity or avgPurchasePrice, which is a financial correctness bug.

**Fix:** Add (t) => ({ userStockUnique: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) as the second argument to pgTable. Handle upsert logic in the application layer using Drizzle's onConflictDoUpdate to merge quantities and recalculate avgPurchasePrice.

### `server/index.ts`: probeDatabase / seedDatabase block in startup IIFE
**Issue:** startup awaits probeDatabase() and seedDatabase() sequentially before the HTTP server begins listening, meaning the process is completely unresponsive (no health-check endpoint, no readiness signal) for the entire duration of DB connection + seed. If either throws without a catch the process exits without cleanup.

**Fix:** Start listening immediately after registerRoutes(), then run probeDatabase() and seedDatabase() in the background. Wrap each in try/catch and surface failures via a readiness flag checked by a GET /healthz route. Example: httpServer.listen(port, () => { initBackground().catch(err => { captureError(err); process.exit(1); }); }); where initBackground does the probe/seed.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are never stored in variables, so they cannot be cleared during graceful shutdown. When SIGTERM arrives the process closes the HTTP server but both timers keep the event loop alive and can fire mid-shutdown, attempting DB writes against a closing connection pool.

**Fix:** Store both handles: const weeklyDigestTimer = setInterval(...); const purgTimer = setInterval(...); then in the SIGTERM/SIGINT handler call clearInterval(weeklyDigestTimer); clearInterval(purgeTimer); before closing the DB pool and the HTTP server.

### `server/index.ts`: res.json middleware / res.on('finish') logger
**Issue:** The logging middleware monkey-patches res.json to capture and then JSON.stringify every response body on every request. For API responses containing user financial data or PII this logs sensitive information to stdout. It also allocates and serializes the full body object on every response regardless of status, wasting CPU and memory under concurrent load.

**Fix:** Remove capturedJsonResponse capture entirely, or limit logging to non-200 error responses only (e.g. res.statusCode >= 400), and never log the body. Log only method, path, status, and duration: `${req.method} ${path} ${res.statusCode} in ${duration}ms`. If body logging is needed for debugging, gate it behind a DEBUG env flag and redact known sensitive fields.

### `client/src/pages/AdminDashboard.tsx`: useAdminAuth hook usage / top-level auth check
**Issue:** The component imports `useAdminAuth` but the truncated code pattern commonly seen in this structure renders child sections and fires `useQuery` data-fetch hooks before confirming the auth check resolves. If `useAdminAuth` is async (e.g., checking a session), queries for users, schools, and audit logs may execute and return data briefly before a redirect fires, leaking admin data into React Query cache accessible to non-admin users.

**Fix:** Gate all `useQuery` calls and JSX behind a confirmed-auth boolean: `const { isAdmin, isLoading } = useAdminAuth(); if (isLoading) return <Spinner />; if (!isAdmin) { redirect('/'); return null; }` — ensure no query hooks run before this guard using conditional rendering at the top of the component tree, not inside JSX.

## 🟡 Warnings (12)

- **`server/routes.ts` (/healthz handler (~line 82-90)):** Every call to /healthz triggers two dynamic import() calls ('./db' and 'drizzle-orm'). While Node caches modules after first load, the async resolution overhead and the pattern of doing I/O inside a health endpoint that may be polled every 10-30 seconds by Railway and uptime monitors is wasteful and obscures startup errors.
  → *Import db and sql at the top of the file statically alongside the other imports. The health endpoint should use the already-initialised db instance directly without dynamic imports.*
- **`server/routes.ts` (Supabase init chain (~line 120-128)):** The seven Supabase init/seed functions are chained sequentially with .then(), meaning each waits for the previous to complete before starting. Functions that are independent (e.g. seedRegionalContent and seedGameContent) add unnecessary startup latency for a school app where first-render speed matters.
  → *Group truly independent operations with Promise.all. For example: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson()])).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));*
- **`server/routes.ts` (registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 116-118)):** The OpenAI-backed chat, image, and audio routes are registered without any rate limiting middleware applied at the router level in this file. While child route files may add their own limiting, there is no defence-in-depth at the entry point, which is a concern for a public school-facing app that could be abused to exhaust AI API quotas.
  → *Apply a dedicated aiLimiter (e.g. 20 req/min per IP) via app.use('/api/chat', aiLimiter) and similarly for /api/image and /api/audio before the registerChat/Image/AudioRoutes calls, matching the pattern already used for auth endpoints.*
- **`shared/schema.ts` (budgets table definition):** budgets.userId has no index. Budget lookups are always scoped per user (WHERE user_id = ?), but there is no index on this column, causing full table scans as data grows.
  → *Add an index in the budgets table's second argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also add a composite uniqueIndex on (userId, categoryId, period) to enforce one budget per category per period per user.*
- **`shared/schema.ts` (categories, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions tables):** createdAt columns across all tables use .defaultNow() without .notNull(). This allows NULL to be inserted explicitly, breaking any ORDER BY createdAt or audit logic silently.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). This matches the intent and prevents NULL poisoning in time-based queries.*
- **`shared/schema.ts` (userId: varchar('user_id') across all tables):** All userId foreign key columns are declared as plain varchar() with no length constraint, while the referenced users.id type is unknown from this file. Without a consistent, explicit length (e.g. varchar(36) for UUID or varchar(21) for NanoID), the database cannot enforce storage efficiency and silently accepts any string length.
  → *Define a shared constant (e.g. USER_ID_LENGTH = 36) and use varchar('user_id', { length: USER_ID_LENGTH }) on every userId column including in the auth model. This enforces uniformity and improves index efficiency.*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text() instead of timestamp(). Storing dates as text prevents range queries (e.g. WHERE last_price_update_date > NOW() - INTERVAL '1 day') from using indexes and introduces format inconsistency risk.
  → *Change to lastPriceUpdateDate: timestamp('last_price_update_date'). Update any application code that writes or reads this field to use Date objects instead of strings.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function uses new Date() and local server time with no timezone specification. On a Replit container the system timezone may be UTC, but the intended business time is Caribbean (UTC-4/UTC-5). The day-of-week calculation `(dow + 0) % 7` is also a no-op (adding 0), meaning Sunday (dow=0) produces daysSince=0 and the date is not rolled back, returning the current day rather than the most recent Sunday.
  → *Replace with an explicit timezone-aware calculation using Intl or a lightweight library (date-fns-tz). Pin the timezone via a DIGEST_TIMEZONE env var defaulting to 'America/Port_of_Spain'. Fix the Sunday rollback: const daysSince = dow === 0 ? 0 : dow; and verify that when today is Sunday but before 7 pm the function correctly returns last Sunday.*
- **`server/index.ts` (startJobWorker / registerJobHandlers in startup):** registerJobHandlers() and startJobWorker() are called with no error handling and no reference stored to the worker, so there is no way to drain the job queue or stop the worker cleanly on shutdown. If the worker uses an internal interval or polling loop it will also keep the event loop alive after httpServer.close().
  → *Have startJobWorker() return a stop function or an object with a stop() method. Call it in the SIGTERM handler before process.exit: const stopWorker = startJobWorker(); process.on('SIGTERM', async () => { clearInterval(weeklyDigestTimer); clearInterval(purgeTimer); await stopWorker(); httpServer.close(() => process.exit(0)); });*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scrollIntoView ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...); }}` is recreated on every render, causing scrollIntoView to fire after every keystroke in the input field, not just when messages are added. This creates jank and wasted DOM work.
  → *Move scroll behavior into a useEffect that depends on `messages`: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]); ... <div ref={bottomRef} />`*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – sendMessage function):** `sendMessage` is defined as a plain async function inside the component body without useCallback. It captures `messages`, `input`, and `loading` in closure, and is passed to both the onKeyDown handler and the send button onClick. Every keystroke re-creates it, and the stale-closure risk means rapid calls could use stale `messages` state.
  → *Wrap in useCallback with deps `[messages, input, loading]`, or better: derive the next messages array inside a functional setState update to avoid stale closure on `messages`: `setMessages(prev => { const next = [...prev, { role: 'user', content: text }]; callApi(next); return next; })`*
- **`client/src/pages/AdminDashboard.tsx` (apiRequest POST /api/admin/help-chat):** The help-chat API call sends the full cumulative `messages` array on every turn with no size cap. For a school-aged platform with a potentially long session, this payload grows unboundedly, increasing latency and LLM token costs, and could expose prior sensitive admin conversation content in network logs.
  → *Slice the context window before sending: `const context = next.slice(-10);` then `apiRequest('POST', '/api/admin/help-chat', { messages: context })`. Add a server-side max-messages guard as well.*

## 💡 Opportunities

### Centralise base-URL resolution as a startup constant: 🔥 High Impact
Resolve and validate BASE_URL once at process start from environment config and export it. This eliminates the repeated per-request header-parsing logic in robots.txt and sitemap.xml, reduces latency on those routes, and removes the header-injection surface entirely.

### Add a dedicated AI endpoint rate-limiter module: 🔥 High Impact
Create a tiered aiLimiter (e.g. 20 req/min per IP, 200/day per authenticated user) and apply it to all OpenAI-backed routes at registration time in routes.ts. For a school app this prevents runaway costs from a single student or a scripted abuse attempt and provides a clear audit point for quota management.

### Replace sequential domain-route registration with parallelised setup: ⚡ Medium Impact
The six registerXxxRoutes calls after Promise.all imports are awaited sequentially. If any of them perform async DB schema checks or caching on startup, parallelising them with Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]) would reduce cold-start time, which matters on Railway where containers restart on deploy.

### Centralise currency default as an environment-driven constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as the currency default in at least six separate table definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract a DEFAULT_CURRENCY constant sourced from process.env.DEFAULT_CURRENCY with a fallback, then reference it in all .default() calls. This allows the app to support other Caribbean currencies (TTD, JMD) per deployment without schema changes.

### Add composite index on billReminders (userId, nextDueDate): 🔥 High Impact
Bill reminder dashboards and notification jobs will query WHERE user_id = ? ORDER BY next_due_date ASC. Without a composite index on these two columns the query requires a full scan plus sort. Adding index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) will make this O(log n) and is critical as reminder rows grow per user.

### Replace serial primary keys with UUID or NanoID for user-facing entities: 🔥 High Impact
Tables like transactions, savingsGoals, and documentUploads use serial (auto-increment integer) PKs. Sequential integer IDs are enumerable, enabling IDOR attacks if an auth check is ever missed in a route handler. Switching to gen_random_uuid() or a NanoID default eliminates enumeration risk without requiring application-layer ID generation, which is particularly important for a school-aged user base.

### Parallel startup I/O with Promise.all: ⚡ Medium Impact
probeDatabase(), seedDatabase(), and registerJobHandlers() are currently awaited sequentially. Running probeDatabase() and any other independent init tasks with Promise.all([probeDatabase(), ...]) would cut startup latency by the sum of all sequential waits minus the longest single task, meaningful on cold Replit container starts where DB connect alone can take 1-2 seconds.

### Centralise hardcoded scheduler intervals as env config: 📌 Low Impact
The weekly digest check interval and purge check interval are hardcoded as numeric literals in setInterval calls. Extracting them to DIGEST_CHECK_INTERVAL_MS and PURGE_CHECK_INTERVAL_MS env vars with documented defaults lets ops teams adjust polling frequency without a code deploy, and makes the values visible in infrastructure config.

### Add structured request-id correlation to all log lines: 🔥 High Impact
Currently each log line is a free-form string with no request or trace identifier. Adding a short uuid (or reusing the Sentry trace id already available via sentryRequestContext) to every log line emitted inside a request lifecycle would let operators correlate scheduler errors, job failures, and HTTP errors across a single user session, significantly reducing mean time to diagnose production incidents.

### Parallel data fetching with Promise.all: 🔥 High Impact
If the dashboard fires multiple independent useQuery calls sequentially (users, schools, analytics, audit log), any server-side prefetch or initialization logic using sequential awaits should be refactored to Promise.all. This can cut initial dashboard load time by the sum of all individual query waterfalls minus the slowest single query.

### Memoize filtered/sorted admin table data: ⚡ Medium Impact
Admin tables with search, filter, and pagination applied to potentially large user or school lists should wrap their derived arrays in useMemo keyed on the raw data and filter state. Without this, every parent re-render (e.g., from the chat panel toggling open) re-runs O(n) filter and sort operations on the full dataset.

### Extract AdminHelpChat into its own file with React.memo: ⚡ Medium Impact
AdminHelpChat is a fully self-contained stateful component but lives inline in a 1000+ line file. Moving it to its own module and wrapping with React.memo prevents it from re-rendering when unrelated admin dashboard state (selected tab, modal open state, query refetches) changes, and makes the file maintainable for a small team.
