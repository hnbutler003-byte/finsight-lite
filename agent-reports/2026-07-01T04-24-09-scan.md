## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then issues a separate SELECT per budget to calculate spent amount, resulting in N+1 database round-trips for a user with N budgets.

**Fix:** Replace the per-budget spent query with a single aggregated JOIN: SELECT budgets.*, COALESCE(SUM(transactions.amount),0) AS spent FROM budgets LEFT JOIN transactions ON transactions.category_id = budgets.category_id AND transactions.user_id = budgets.user_id AND transactions.type = 'expense' WHERE budgets.user_id = $1 GROUP BY budgets.id. Execute this as one Drizzle query using .leftJoin() and sql`sum(...)` aggregation.

### `server/storage.ts`: getTransactions / getDashboardStats
**Issue:** Missing pagination guard: getTransactions accepts an optional limit but has no enforced maximum, and getDashboardStats fetches all matching transactions with no limit at all. A user or attacker with thousands of transactions can trigger unbounded memory allocation on every dashboard load.

**Fix:** In getTransactions enforce a hard cap: const safeLimit = Math.min(filters?.limit ?? 100, 500). In getDashboardStats use a server-side date-bounded window (default last 90 days) and add .limit(1000) as an absolute safety ceiling, returning a warning flag in DashboardStats when the cap is hit.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI and media generation routes are registered with no rate limiting middleware. For a school-aged user app these endpoints call OpenAI APIs and are the most expensive both in cost and latency. A single user or bot can exhaust API quotas or cause runaway spend.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15 min per IP) in rateLimiter.ts and apply it before each registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Confirm the route prefixes match what each register function mounts.

### `server/routes.ts`: robots.txt handler (~line 38) and sitemap.xml handler (~line 52)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host with no validation or allowlist. An attacker can send a crafted Host header causing the Sitemap directive and sitemap <loc> tags to point to an arbitrary domain, enabling cache-poisoning or phishing via indexed sitemaps.

**Fix:** Add a CANONICAL_HOST environment variable (e.g. 'finsightlite.com'). At startup: const CANONICAL_BASE = process.env.CANONICAL_HOST ? `https://${process.env.CANONICAL_HOST}` : null; Then in both handlers: const base = CANONICAL_BASE ?? `${proto}://${host}`; This eliminates the injection surface for production while keeping localhost fallback for dev.

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler payload)
**Issue:** Full file contents (fileB64) are stored in the job table payload. A single exam paper image or PDF encoded as base64 can easily exceed several MB, bloating the jobs table, slowing job queue queries, and potentially hitting row size limits in Postgres.

**Fix:** Upload the file to object storage before enqueuing the job, store only the object storage path in the payload (e.g., { paperId, objectPath, ext, subject }), then download it at the start of runExtractPaper. This keeps job rows small and makes retries cheap.

### `server/index.ts`: seedDatabase / probeDatabase calls in startup IIFE
**Issue:** seedDatabase and probeDatabase are likely awaited sequentially during startup when they could run in parallel, and if either throws the error may crash the process without a clean shutdown signal to the HTTP server, leaving the port bound but the app non-functional.

**Fix:** Wrap both in Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])`. Surround the entire startup IIFE with a top-level try/catch that calls `httpServer.close()` and `process.exit(1)` on failure so the process exits cleanly and the supervisor can restart it.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The interval handles returned by setInterval are never stored and never cleared. On graceful shutdown these intervals keep the event loop alive and can fire after the database connection is closed, causing unhandled promise rejections and potential data corruption.

**Fix:** Store the return values: `const digestInterval = setInterval(...)` and `const purgeInterval = setInterval(...)`. In the SIGTERM/SIGINT shutdown handler call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` before closing the HTTP server and database pool.

### `client/src/pages/AdminDashboard.tsx`: sendMessage – apiRequest POST /api/admin/help-chat
**Issue:** The full conversation history (`messages: next`) is sent to the server on every turn with no length cap. For a school-aged-user platform this means unbounded token cost and a potential prompt-injection vector: a student or attacker who gains access to the admin chat can craft multi-turn messages to exfiltrate system-prompt content or exhaust API quota.

**Fix:** Enforce a client-side sliding window before sending: `const windowedMessages = next.slice(-10);` and send `{ messages: windowedMessages }`. On the server, independently validate message count and sanitise role values (reject anything other than 'user'/'assistant') before forwarding to the LLM provider.

## 🟡 Warnings (13)

- **`server/storage.ts` (simulateMarketPrices):** simulateMarketPrices is called on a setInterval (inferred from seedMarketData pattern) with no cleanup handle exported or stored, creating a memory/timer leak if the module is re-initialised during hot reload or tests.
  → *Store the interval reference: export const marketSimInterval = setInterval(simulateMarketPrices, 60000). In your server shutdown handler call clearInterval(marketSimInterval). In test setup call clearInterval after each suite to prevent leaked async work.*
- **`server/storage.ts` (enrollStudent / createGameSession):** Multi-step writes (e.g. enrollStudent inserts a ClassEnrollment then updates class enrollmentCount, createGameSession inserts session then calls updateUserXp) are performed as sequential independent queries with no database transaction, leaving data in a partially-written state if the second write fails.
  → *Wrap each multi-step write in db.transaction(async (tx) => { ... }) using Drizzle's transaction API. Pass the tx context to every sub-query inside the block so all steps commit or roll back atomically.*
- **`server/storage.ts` (getLeaderboard):** getLeaderboard logs or returns userId and userName fields that may be joined from the users table including email or other PII, with no field-level allowlist. Combined with the absence of a visible auth check on the leaderboard route, any authenticated user can enumerate other users' display names and scores without restriction.
  → *In the Drizzle select, explicitly allowlist only { userId: users.id, userName: users.displayName, avatar: users.avatarUrl, totalScore, gamesPlayed } and exclude email, phone, and any credential fields. Ensure the leaderboard route middleware validates the caller's session before executing this query.*
- **`server/routes.ts` (initSupabaseTables().then(() => trimOrgNamesInSupabase()).then(...)... (~line 103)):** All seven Supabase initialisation steps are chained sequentially. On a cold start this adds cumulative latency for each step before the app is fully ready. Steps like seedStaticModules, seedRegionalContent, and seedGameContent are independent and have no ordering dependency on each other.
  → *Split into two phases: await initSupabaseTables() first (schema must exist before seeding), then run independent seeds in parallel: Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()]).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Verify foreign-key dependencies before collapsing further.*
- **`server/routes.ts` (/healthz handler (~line 74)):** The /healthz endpoint dynamically imports 'db' and 'drizzle-orm' on every call. In a busy uptime-monitor scenario (e.g. 30-second polls) this causes repeated module resolution overhead and holds references that can confuse module cache diagnostics. The endpoint is also unauthenticated and reveals DB availability state externally.
  → *Move the db and sql imports to the top of the file as static imports alongside other server imports. Optionally restrict /healthz to internal networks or add a secret header check (req.headers['x-health-token'] === process.env.HEALTH_TOKEN) so external actors cannot probe DB status.*
- **`server/jobHandlers.ts` (runExtractPaper, for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one inside a sequential for loop. For a paper with 50 questions this is 50 serial round-trips to the database, dramatically increasing job runtime and holding the worker busy.
  → *Replace the loop with a single bulk insert: filter valid questions first, map them to insert rows, then call db.insert(extractedQuestions).values(rows) once. Add the idempotency delete before this single insert as already done.*
- **`server/jobHandlers.ts` (purge-ai-usage handler (registerJobHandler call)):** The purge-ai-usage handler has no try/catch. An unexpected DB error (connection drop, constraint violation) will produce an unhandled rejection that may crash the worker process rather than marking the job as failed and allowing retry logic to proceed cleanly.
  → *Wrap the handler body in try/catch, capture the error via Sentry (matching the pattern used in ai-health-check), then re-throw so the job runner can record the failure and schedule a retry.*
- **`server/jobHandlers.ts` (runExtractPaper, openai.chat.completions.create calls (max_completion_tokens: 4096)):** The token limit 4096 and model name 'gpt-4o' are hardcoded in two separate OpenAI calls. Changing the model or budget requires a code deploy rather than a config change, and different environments (dev vs prod) cannot be tuned independently.
  → *Read these from environment variables: const MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o' and const MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10). Apply to both completions calls in runExtractPaper.*
- **`server/index.ts` (res.json monkey-patch in request logging middleware):** The logging middleware unconditionally serializes and logs the full JSON response body for every /api route. For a financial literacy app with school-aged users this risks logging sensitive fields (scores, personal progress data, session tokens) to stdout which may be captured by log aggregators.
  → *Add a blocklist of sensitive routes or response keys: skip appending `capturedJsonResponse` for paths matching `/api/auth` or `/api/user`, or strip known sensitive keys (e.g. `password`, `token`, `secret`) before logging. At minimum, truncate the serialized body to 200 characters to limit accidental exposure.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The day-of-week arithmetic `(dow + 0) % 7` uses a no-op `+ 0` instead of the intended offset to find Sunday, and the result is that when `now` is exactly Sunday after 7 PM the function correctly returns today, but when called repeatedly via a 1-hour interval it may re-insert a claim for the same week on the following Sunday before the DB unique constraint fires, producing noisy rolled-back rows every hour.
  → *The `+ 0` is dead code; remove it. More importantly, the interval period should be validated: consider checking once per hour but adding a grace window (e.g. only attempt if within 30 minutes of 7 PM Sunday) to reduce unnecessary DB round-trips and conflict noise.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView({ behavior: 'smooth' }); }}` creates a brand-new function on every render of the messages list. This fires scrollIntoView on every re-render (including while typing in the input field), not just when a new message is appended, causing jarring scroll-jumps mid-keystroke.
  → *Replace with a useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` then use `ref={bottomRef}` on the sentinel div. This scopes the scroll to only fire when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (sendMessage async function inside AdminHelpChat):** There is no abort/cleanup mechanism if the AdminHelpChat panel is closed while a request is in-flight. The `setMessages` and `setLoading` calls will execute on an unmounted component, triggering a React state-update-on-unmounted-component warning and a potential memory leak if the component is toggled rapidly.
  → *Add an AbortController: `const abortRef = useRef<AbortController | null>(null);`. In sendMessage, set `abortRef.current = new AbortController()` and pass `{ signal: abortRef.current.signal }` to apiRequest. In a useEffect cleanup, call `abortRef.current?.abort()`. Guard state setters with an `isMounted` ref or check `signal.aborted` before calling setState.*
- **`client/src/pages/AdminDashboard.tsx` (AdminDashboard – overall file structure):** The file is a single monolithic component exceeding 147k characters containing charts, CRUD dialogs, pagination, role management, a chat widget, and likely multiple useQuery calls that all re-run whenever any top-level state changes (e.g. search input). Every keystroke in any search field will trigger re-evaluation of all sibling derived state and re-render every chart.
  → *Split into feature-level components (UsersPanel, SchoolsPanel, AnalyticsCharts, AdminHelpChat) each owning its own useQuery calls. Wrap expensive derived values (chart data transforms, filtered lists) in useMemo. This isolates render scope so a search input in UsersPanel does not cause AnalyticsCharts to re-render.*

## 🔵 Info / Tech Debt (4)

- **`server/routes.ts` (Duplicate health check endpoints: /api/health (~line 30) and /healthz (~line 74)):** Two health check endpoints exist with different response shapes ({status, ts} vs {ok, ts}) and different behaviour — /api/health never checks the DB while /healthz does. This creates confusion for operators and uptime monitors, and /api/health gives a false positive 200 even when the database is down.
- **`server/jobHandlers.ts` (runExtractPaper, extractedText slice (0, 30000)):** The raw PDF text is truncated to 30,000 characters before being sent to OpenAI, but the truncation point is hardcoded and there is no log or warning when truncation actually occurs. For long papers, questions near the end are silently dropped with no indication to the teacher or in job output.
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and AI_PURGE_OLDER_THAN_DAYS env var):** The purge threshold is read and frozen at module load time. If the env var is missing or mis-typed the fallback is silently applied (180 days) with no warning, making operational misconfiguration invisible. Additionally this value is used in a DB delete query; if it were ever sourced from user input the pattern would be injectable, and the silent fallback makes auditing harder.
- **`client/src/pages/AdminDashboard.tsx` (handleKey inside AdminHelpChat render):** `handleKey` is defined as a plain function expression inside the component body without useCallback. Because `sendMessage` closes over `messages`, `input`, and `loading`, both functions are recreated on every render and a new `onKeyDown` handler is attached to the input element on every keystroke, adding unnecessary GC pressure.

## 💡 Opportunities

### Parallel async fetches in getDashboardStats: 🔥 High Impact
getDashboardStats sequentially awaits total income, total expenses, savings goals, and budget summary queries. Replacing these with Promise.all([...]) would cut dashboard load latency by roughly 60-70% on a typical 20ms-per-query database, directly improving perceived performance for school-aged users on low-bandwidth Caribbean connections.

### Centralised pagination defaults via environment config: ⚡ Medium Impact
Default limits (100 transactions, 500 cap, leaderboard size) are hardcoded across multiple functions. Extract them into a single config object read from environment variables (DEFAULT_PAGE_SIZE, MAX_PAGE_SIZE, LEADERBOARD_LIMIT) so limits can be tightened in production without code changes.

### Consistent try/catch boundary at the storage layer: ⚡ Medium Impact
Several async storage methods lack try/catch, propagating raw Drizzle/PostgreSQL errors (which may contain table names, column names, or query fragments) up to route handlers and potentially into API responses. Adding a thin error-normalisation wrapper around each public method would prevent internal schema details from leaking to clients.

### Per-user AI rate limiting via session ID: 🔥 High Impact
Current rate limiting (once added) would be IP-based, which is ineffective behind school NAT where many students share one IP. Use express-rate-limit's keyGenerator to key on req.session.userId when present, falling back to IP. This prevents one student from exhausting the quota for an entire school.

### Structured startup logging for Supabase init phases: ⚡ Medium Impact
The catch-all console.error gives no visibility into which seed step failed. Wrap each phase with a labelled try/catch and emit structured JSON logs (step name, duration, success/failure). This dramatically reduces mean-time-to-diagnose on cold-start failures in production, which is especially important for a Railway-deployed app.

### Cache-Control headers on sitemap and robots.txt: 📌 Low Impact
Both /sitemap.xml and /robots.txt are generated dynamically on every request with no caching headers, so crawlers and CDNs will hammer the Node process. Add res.setHeader('Cache-Control', 'public, max-age=86400') to both handlers. Since the content is deterministic (driven by CANONICAL_HOST env var) this is safe and will eliminate redundant renders.

### Parallel object-storage existence check and header setup in streamPrivateObjectToResponse: ⚡ Medium Impact
Currently file.exists() is awaited before any headers are set, which is correct, but the bucket/object parsing and client lookup happen synchronously before the async call. No parallelism issue here, but the function lacks a try/catch around the createReadStream pipeline setup meaning an error thrown synchronously by createReadStream (bad path, client misconfiguration) will propagate as an unhandled exception rather than a 500 response. Wrapping the stream setup in try/catch would prevent silent worker crashes.

### Batch idempotency delete and bulk insert in a single transaction for extract-paper: 🔥 High Impact
Wrapping the delete of existing extractedQuestions and the bulk insert of new ones in a single db.transaction() call would make the idempotency guarantee atomic. Currently a crash between the delete and the insert leaves the paper with zero questions until a retry completes, which the UI could misinterpret as a successful but empty extraction.

### Centralise job error capture into a registerJobHandler wrapper: ⚡ Medium Impact
Sentry capture is only applied in ai-health-check; all other handlers silently swallow or rethrow errors without telemetry. Adding a thin wrapper around registerJobHandler that catches, captures to Sentry with job name and attempt count, then rethrows would give consistent observability across all job types with one change instead of updating every handler individually.

### Graceful shutdown handler: 🔥 High Impact
Add a single SIGTERM/SIGINT handler that sequentially stops accepting new connections (httpServer.close()), clears all intervals, waits for in-flight jobs to drain, then closes the DB pool. Without this, Replit container restarts mid-job can corrupt weekly digest claim rows or leave purge jobs half-executed, requiring manual DB cleanup.

### Structured startup health log: ⚡ Medium Impact
Replace the scattered individual `log()` calls during startup with a single structured summary emitted after all initialization steps complete, e.g. `{ dbReady, seeded, jobWorkerStarted, sentryEnabled }`. This gives operators a single line to grep for in production logs to confirm a clean start, which is especially valuable on Replit where restarts are frequent.

### Scheduler extraction into a dedicated module: ⚡ Medium Impact
Move maybeRunWeeklyDigest, maybeRunAiUsagePurge, getMostRecentSundayAt7pm, and their intervals into a `server/scheduler.ts` module that exports a `startSchedulers()` returning interval handles and a `stopSchedulers()`. This makes the lifecycle testable in isolation, eliminates the memory-leak risk from untracked handles, and keeps index.ts focused on wiring.

### Parallel data fetching with Promise.all: 🔥 High Impact
If the dashboard fires multiple sequential useQuery-backed fetches on mount (users, schools, analytics, audit log), convert any waterfall async initialisers to Promise.all. Combined with React Query's parallel queries option this can cut initial load time by 40-60% on slow Caribbean mobile connections.

### Virtualise long admin tables: ⚡ Medium Impact
Paginated tables rendering 20-50 rows of user/school records as full DOM nodes are manageable, but if page size is configurable or grows, switching to @tanstack/react-virtual will keep DOM node count constant and prevent jank on low-end Android devices common in the target market.

### Environment-based API URL configuration: ⚡ Medium Impact
Any hardcoded strings like '/api/admin/help-chat' or base URLs scattered through the file should be centralised in a typed config module fed by VITE_ environment variables. This prevents accidental prod/staging endpoint mix-ups and makes endpoint auditing straightforward for security reviews.
