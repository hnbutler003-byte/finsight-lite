## 🔴 Critical (3)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 106-108)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware. On a school-aged user platform these endpoints can be trivially abused to run up API costs or trigger inappropriate content generation at volume.

**Fix:** Create a dedicated `aiLimiter` in rateLimiter.ts (e.g. 20 req/min per IP) and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerXRoutes calls.

---

### `server/routes.ts`: robots.txt handler (~line 48) and sitemap handler (~line 61)
**Issue:** The `base` URL is constructed directly from `req.headers['x-forwarded-host']` and `req.headers.host` without any validation or allowlist. An attacker can send a crafted Host header and inject an arbitrary domain into the Sitemap directive and sitemap <loc> tags, enabling cache-poisoning or open-redirect abuse.

**Fix:** Read the canonical origin from a single env var: `const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com';` and remove all header-derived base URL logic. Set APP_BASE_URL in Railway/Replit secrets.

---

### `server/jobHandlers.ts`: runExtractPaper (extract-paper handler)
**Issue:** Full file content is stored as base64 in the job payload (fileB64 field). For a multi-page PDF or high-res image this can easily be 5-20 MB per job row, bloating the jobs table, slowing queue polls, and loading the entire payload into Node heap on every retry attempt.

**Fix:** Upload the file to object storage before enqueuing the job, then store only the object storage path in the payload. In runExtractPaper, download the file from object storage at the start of the handler and delete it after successful extraction. This keeps job rows small (<1 KB) and makes retries cheap.

## 🟡 Warnings (5)

- **`server/routes.ts` (Supabase init chain (~line 113-120)):** The seven sequential `.then()` seed calls are chained so that a failure in `seedFinancialAcademyLesson()` silently skips all subsequent seeds (seedStaticModules, seedRegionalContent, seedGameContent). The single `.catch` at the end only reports the first failure, leaving the DB in a partially seeded state with no visibility into which step failed.
  → *Run independent seeds in parallel and collect individual outcomes: `const results = await Promise.allSettled([seedFinancialAcademyLesson(), seedStaticModules(), seedRegionalContent(), seedGameContent()]); results.forEach((r,i) => r.status === 'rejected' && console.error('[Supabase] seed step', i, 'failed:', r.reason));` Keep initSupabaseTables and trimOrgNamesInSupabase sequential before the parallel block if order matters.*
- **`server/routes.ts` (/healthz handler (~line 79)):** The /healthz endpoint uses dynamic `await import('./db')` and `await import('drizzle-orm')` inside every request. While Node caches modules after first load, the async resolution path adds unnecessary overhead on every health-poll and masks import errors that should surface at startup.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. The health handler becomes a simple `await db.execute(sql\`select 1\`)` with no dynamic import cost.*
- **`server/jobHandlers.ts` (runExtractPaper – question insertion for-loop):** Extracted questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 40-question paper this is 40 serial round-trips to the database. Additionally there is no transaction wrapping these inserts, so a crash mid-loop leaves partial data that the idempotency delete at the top of the next retry correctly cleans up, but the lack of a transaction means a reader can observe a partial set between retries.
  → *Collect all valid question objects into an array and call a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the delete and bulk insert in a db.transaction() so readers never see a partial state and the operation is atomic on retry.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The entire handler body has no try/catch. A DB connection error or unexpected exception will surface as an unhandled promise rejection inside the job runner rather than a caught, logged, and retriable job failure. This is inconsistent with the ai-health-check handler which does wrap its body.
  → *Wrap the delete call in try/catch, log the error with context (olderThanDays, cutoff date), and rethrow so the job runner can mark the job as failed and schedule a retry. Alternatively apply a shared error-boundary wrapper at the registerJobHandler level.*
- **`server/jobHandlers.ts` (parsePrivatePath / uploadPrivateObject):** PRIVATE_OBJECT_DIR is checked at call time rather than at startup, so a missing env var surfaces as a runtime error inside a job handler mid-flight rather than a fast-fail at boot. The error message also echoes the raw resolved path which may expose internal directory structure in logs visible to non-admin users.
  → *Validate PRIVATE_OBJECT_DIR (and other required env vars) once at application startup and throw a descriptive startup error if absent. In the error thrown from parsePrivatePath, omit the full resolved path from the message; log it at debug level internally instead.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (duplicate health endpoints (~line 36 and ~line 79)):** Two health-check routes exist: `GET /api/health` (returns static ok) and `GET /healthz` (actually tests the DB). Railway and uptime monitors will only be pointed at one. The shallow `/api/health` could report healthy while the DB is down, giving false confidence.
- **`server/jobHandlers.ts` (runExtractPaper – vision path):** When the file is an image, the handler re-encodes the already-decoded buffer back to base64 to build the data URL: buffer.toString('base64'). The original fileB64 string from the payload is identical to this re-encoded value, so the round-trip decode→re-encode is wasteful and slightly error-prone if any transformation were ever added between them.

## 💡 Opportunities

### Parallel domain-router registration: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then awaited sequentially one-by-one. If any registerXRoutes function does async I/O at registration time (e.g. DB schema checks), running them with `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` would cut cold-start time proportionally. Estimated saving depends on each router's init work but could be 200-800ms on Railway cold starts.

### Centralised APP_BASE_URL config module: ⚡ Medium Impact
The proto/host derivation logic is duplicated identically in both the robots.txt and sitemap handlers. Extracting `getBaseUrl(req)` into a shared utility (or better, a single env-backed constant) eliminates the duplication and makes the host-injection fix a one-line change instead of two.

### Object-storage route authentication: 🔥 High Impact
The `/public-objects/:filePath(*)` route streams any file from object storage with no auth check. If private assets (org logos, student-uploaded content) share the same bucket as truly public files, an unauthenticated user can enumerate and download them by guessing paths. Add a prefix allowlist (e.g. only `logos/` and `videos/` are public) and require session auth for all other prefixes.

### Shared job error-boundary wrapper: 🔥 High Impact
Introduce a wrapHandler(name, fn) utility that wraps every registered handler in a try/catch, logs structured context (jobId, attempt, payload keys but not values), captures to Sentry, and rethrows. This removes the inconsistency where some handlers (ai-health-check) have error handling and others (purge-ai-usage, perf-scan) do not, and reduces boilerplate for every future handler.

### Batch DB writes for question insertion: 🔥 High Impact
Replacing the per-question await loop with a single bulk insert reduces database round-trips from O(n) to O(1) per extraction job. For a typical 40-question CSEC paper this cuts DB time from ~400 ms to ~15 ms and removes 39 opportunities for a mid-loop failure to leave partial data.

### Startup environment variable validation: ⚡ Medium Impact
Add a validateEnv() function called once at server startup that checks all required env vars (PRIVATE_OBJECT_DIR, OpenAI key, DB URL, etc.) and throws with a clear list of missing keys. This converts silent mid-job runtime failures into a fast, obvious startup crash that surfaces immediately in CI and deployment logs.
