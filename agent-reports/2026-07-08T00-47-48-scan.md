## 🔴 Critical (5)

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** OpenAI-backed chat, image, and audio routes are registered with no rate limiting middleware applied before them. Any authenticated or unauthenticated caller can flood these endpoints, incurring unbounded OpenAI API costs and potential DoS against the service.

**Fix:** Apply a dedicated AI rate limiter before these registrations: create an `aiLimiter` in rateLimiter.ts (e.g. 20 req/15 min per IP) and add `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` immediately before the three registerX calls. Confirm the path prefixes match what registerChatRoutes etc. actually mount.

### `server/routes.ts`: robots.txt handler (~line 43) and sitemap.xml handler (~line 57)
**Issue:** The `host` and `proto` values are read directly from user-controlled request headers (`x-forwarded-host`, `x-forwarded-proto`, `host`) and interpolated into URLs without validation. An attacker can send a crafted Host header to inject arbitrary URLs into the Sitemap directive and sitemap.xml `<loc>` tags, enabling cache-poisoning or open-redirect abuse.

**Fix:** Add a trusted base-URL env var: `BASE_URL=https://finsightlite.com`. Read it with `const base = process.env.BASE_URL;` and throw at startup if it is missing. Remove all header-derived URL construction from these handlers entirely.

### `server/jobHandlers.ts`: runExtractPaper (job.payload: fileB64)
**Issue:** Full file contents are stored as base64 in the job payload. For a multi-page PDF or high-res image this can be several MB per job row, bloating the jobs table, slowing queue polling queries, and potentially hitting payload size limits in the DB or job framework.

**Fix:** Upload the file to object storage before enqueuing and store only the storage path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside the handler, stream/download the file by path. This also avoids re-encoding the buffer through JSON serialization.

---

### `server/jobHandlers.ts`: runExtractPaper – for loop calling storage.createExtractedQuestion
**Issue:** Questions are inserted one-by-one in a sequential await loop. For a 40-question paper this is 40 round-trips. Additionally, the prior delete and these inserts are not wrapped in a single transaction, so a crash mid-loop leaves the paper in a partially-inserted state that the idempotency delete on the next retry will silently discard without any warning.

**Fix:** Wrap the delete and bulk insert in a db.transaction(). Replace the for-loop with a single db.insert(extractedQuestions).values(validQuestions.map(...)) call inside the same transaction. This is atomic, eliminates the race window, and reduces round-trips to 1.

### `client/src/pages/AdminDashboard.tsx`: sendMessage async function in AdminHelpChat
**Issue:** The `/api/admin/help-chat` endpoint is called with the full message history on every send, but there is no server-side auth token attached beyond whatever apiRequest adds. If apiRequest relies solely on a cookie-based session that has expired, the chat silently fails with a generic error and leaks the fact that an admin chat endpoint exists at that path to any unauthenticated caller who can reach the API.

**Fix:** Verify the server-side `/api/admin/help-chat` route has the same admin middleware guard as other admin routes. On the client, inspect the response status before calling `res.json()`—if status is 401/403 redirect to login rather than swallowing the error as a generic chat failure: `if (!res.ok) { if (res.status === 401) redirect('/admin/login'); throw new Error(res.statusText); }`

## 🟡 Warnings (13)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 110)):** The seven Supabase init steps are chained sequentially with `.then()`. If any step beyond the first fails, subsequent steps (including seeding) are silently skipped because the chain short-circuits to `.catch`. Seed failures leave the DB in a partial state with no per-step visibility.
  → *Run independent steps in parallel where possible and surface individual failures: `await Promise.allSettled([seedStaticModules(), seedRegionalContent(), seedGameContent()]).then(results => results.forEach((r,i) => r.status==='rejected' && console.error(`[Supabase] seed step ${i} failed:`, r.reason)));`. Split the chain so table init runs first (sequentially if truly dependent), then all seed functions run via Promise.allSettled.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject` with no sanitization. A path like `../../private/secrets` could traverse outside the intended public directory depending on the ObjectStorageService implementation, exposing private objects.
  → *Sanitize the path before use: `const filePath = path.normalize(req.params.filePath).replace(/^\.\.\/|\.\.\//g, '');` and additionally validate it matches an allowlist pattern such as `/^[\w\-./]+$/`. Reject requests that fail validation with 400.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls, model: "gpt-4o"):** The OpenAI model name is hardcoded as "gpt-4o" in two places. Any model migration or cost-control switch requires a code deploy rather than an environment variable change. For a school-budget app this is a real operational risk.
  → *Add OPENAI_EXTRACTION_MODEL to environment config (default "gpt-4o") and reference process.env.OPENAI_EXTRACTION_MODEL in both completion calls. Validate the env var at startup so misconfiguration fails fast.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** The handler lacks a try/catch. An unexpected DB error (connection drop, lock timeout) will produce an unhandled rejection that surfaces as an opaque job failure with no structured error metadata returned, making debugging harder and potentially crashing the worker process depending on the job framework's rejection handling.
  → *Wrap the db.delete call in try/catch, log the error with structured context (olderThanDays, cutoff), and re-throw so the job framework records it as failed. Return { deletedRows, cutoffDate } only on success.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The function accepts an arbitrary objectPath string and splits it to derive bucket and object names without validating against an allowlist of known buckets or checking for path traversal sequences (e.g., '../'). A caller passing a crafted path could potentially read from unintended buckets.
  → *After splitting, assert that bucket matches a compile-time allowlist (e.g., const ALLOWED_BUCKETS = new Set([process.env.PRIVATE_BUCKET_NAME])) and that neither bucket nor object contains '..' segments. Throw a 403-equivalent error if the check fails.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets(userId) or budgets(userId, categoryId). Budget lookups are almost certainly filtered by userId and categoryId together, but no index exists. Full table scans will occur as user count grows.
  → *Add a compound index in the budgets table definition: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). Also add a unique constraint if a user should not have duplicate budgets per category per period.*
- **`shared/schema.ts` (simulatedStocks table, lastPriceUpdateDate column):** lastPriceUpdateDate is typed as text() instead of timestamp(). Storing dates as text prevents range queries, ORDER BY date, and index efficiency. For an investment simulation table this column will be used in date comparisons.
  → *Change to: lastPriceUpdateDate: timestamp('last_price_update_date').notNull() and add an index if price freshness queries are expected. Migrate existing string data to actual timestamps.*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** No index on portfolioHoldings(userId, stockId) or portfolioTransactions(userId, executedAt). Portfolio dashboard queries will join or filter on both userId and stockId. Without indexes these become sequential scans per user request.
  → *Add to portfolioHoldings: (t) => ({ userStockIdx: index('idx_holdings_user_stock').on(t.userId, t.stockId) }) and to portfolioTransactions: (t) => ({ userDateIdx: index('idx_ptx_user_date').on(t.userId, t.executedAt) }).*
- **`shared/schema.ts` (userId columns across all tables (categories, transactions, budgets, linkedCards, etc.)):** userId is declared as varchar() with no length parameter. In PostgreSQL this defaults to unlimited length, but it prevents the DB from enforcing a size contract and wastes index space when the referenced users.id has a known fixed format (e.g., UUID = 36 chars or CUID = 25 chars).
  → *Determine the auth provider's ID format and set an explicit length: varchar('user_id', { length: 36 }) for UUIDs or { length: 25 } for CUIDs. Apply consistently to all userId foreign key columns and the users.id primary key in models/auth.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt, savingsGoals.createdAt, billReminders.createdAt, portfolioHoldings.purchasedAt, portfolioTransactions.executedAt):** All audit/creation timestamps use .defaultNow() but are missing .notNull(). This means application code or a future migration can explicitly insert NULL into these columns, breaking any reporting or ordering logic that depends on them.
  → *Chain .notNull() on every defaultNow() timestamp: e.g., createdAt: timestamp('created_at').defaultNow().notNull(). The default handles inserts; notNull() prevents explicit null writes.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – scroll div ref callback):** The inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` is recreated on every render and fires scrollIntoView on every message list re-render including the loading spinner toggling, causing janky scroll behaviour and unnecessary DOM thrashing.
  → *Replace with `const bottomRef = useRef<HTMLDivElement>(null)` and a `useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages, loading])`. Attach `ref={bottomRef}` to the sentinel div.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat – messages state):** Chat messages accumulate indefinitely in component state with no cap. For a school-aged platform where admins may leave the dashboard open for extended sessions, this is a gradual memory leak and also sends an ever-growing payload to the AI endpoint on each message, increasing latency and token cost unboundedly.
  → *Slice the context window sent to the API to the last N messages: `const contextWindow = next.slice(-20)` and send `{ messages: contextWindow }`. Optionally cap display state too: `setMessages(prev => [...prev, reply].slice(-100))`.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard – multiple useQuery calls):** Based on the visible imports (users, schools, teachers, lessons, analytics) the dashboard almost certainly fires several independent useQuery calls that all execute sequentially on mount with no coordination. Each failed query shows no error state, leaving the admin staring at empty cards with no feedback.
  → *Audit every useQuery call to ensure each has an `error` branch rendered in the UI: `if (error) return <ErrorBanner message={error.message} />`. For queries whose data is combined (e.g. stats cards), consider a single aggregated `/api/admin/stats` endpoint to reduce waterfall round-trips.*

## 🔵 Info / Tech Debt (2)

- **`server/routes.ts` (app.get('/api/health') and app.get('/healthz') (~lines 31, 78)):** There are two health check endpoints (`/api/health` returning a trivial JSON response and `/healthz` performing an actual DB probe). The `/api/health` endpoint gives Railway/monitors a false positive: it returns 200 even when the database is completely unavailable, masking outages.
- **`client/src/pages/AdminDashboard.tsx` (Large inline functions passed to BarChart/LineChart event handlers and Select onChange):** Chart event handlers and Select onChange callbacks are defined as arrow functions directly in JSX (e.g. `onChange={val => { setFilter(val); refetch(); }}`). Because the parent dashboard component manages many pieces of state, any state change re-renders the whole tree and recreates these functions, causing Recharts to see new prop references and re-animate charts unnecessarily.

## 💡 Opportunities

### Centralise BASE_URL as a validated startup config: 🔥 High Impact
Create a `config.ts` module that reads and validates all required env vars (BASE_URL, DB connection string, OpenAI key, Supabase URL/key) at process startup, throwing immediately on missing values. This eliminates scattered `process.env` reads and the host-injection vectors in robots.txt and sitemap.xml, and makes misconfigured deployments fail fast rather than silently misbehave.

### Add request-scoped logging with correlation IDs: ⚡ Medium Impact
Attach a UUID correlation ID to each request in the global middleware and include it in all console.error calls (e.g. the object-storage error handler). For a financial-literacy app serving school users, this dramatically reduces mean-time-to-diagnose when parents or teachers report issues, without requiring a full APM product.

### Lazy-load ObjectStorageService to improve cold-start time: 📌 Low Impact
The `ObjectStorageService` is instantiated at module load time at the top of routes.ts, potentially blocking startup if it opens connections or reads config eagerly. Move instantiation inside `registerRoutes` or use a singleton getter. Pair with the existing parallel `Promise.all` import pattern for domain routers to shave additional cold-start latency on Railway.

### Parallel OpenAI calls where extraction steps are independent: ⚡ Medium Impact
In handlers that make multiple independent OpenAI or storage calls sequentially, wrapping them in Promise.all would reduce total latency. For example, if subject detection and question extraction can be restructured as parallel passes, wall-clock time per job drops proportionally—directly improving the teacher upload experience.

### Centralised job payload size guard at enqueue time: 🔥 High Impact
Add a middleware in enqueueJob that measures JSON.stringify(payload).length and throws if it exceeds a configurable limit (e.g., 256 KB). This prevents oversized payloads from ever reaching the DB rather than relying on per-handler discipline, and gives an immediate, actionable error to the enqueuing caller.

### Structured logging for AI extraction outcomes: ⚡ Medium Impact
Currently extraction success/failure is only visible through job status. Emitting a structured log event (paperId, questionCount, detectedSubject, durationMs, modelUsed) on completion enables dashboards to track extraction quality trends and catch model regressions before teachers report them—critical for a school-facing product.

### Unique index on simulatedStocks ticker: ⚡ Medium Impact
ticker should be unique to prevent duplicate stock entries corrupting simulation data. Add uniqueIndex('idx_stocks_ticker').on(t.ticker) to the simulatedStocks table definition. This also speeds up lookups by ticker symbol which is a natural query pattern in investment UIs.

### Partial index on billReminders nextDueDate for upcoming-bill queries: ⚡ Medium Impact
Queries for upcoming bills will filter on nextDueDate >= now() and userId. A partial or compound index on (userId, nextDueDate) will make these lookups O(log n) instead of full scans. This is especially important if reminder polling runs on a schedule.

### Extract default currency 'BSD' to a shared constant: 📌 Low Impact
The string 'BSD' is hardcoded as a default in at least 6 columns across transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, and portfolioHoldings. Define DEFAULT_CURRENCY = 'BSD' in a shared constants file and reference it in schema defaults, Zod validators, and any service logic. A future multi-currency expansion or correction requires only one change point.

### Aggregate admin stats into a single endpoint: 🔥 High Impact
Replace N parallel useQuery calls for dashboard KPI cards (user count, school count, lesson count, etc.) with one `/api/admin/stats` endpoint resolved via Promise.all on the server. This cuts client-side waterfall fetches from N to 1, reduces TTFB for the most-viewed part of the dashboard, and simplifies loading/error state management to a single conditional.

### Memoize filtered/sorted table data: 🔥 High Impact
User, teacher, and school list filtering is likely applied inline on raw query data inside the render function. Wrapping with useMemo keyed on [rawData, searchTerm, filterValue, page] prevents re-filtering on every keystroke-driven re-render and is critical for tables with hundreds of rows on a resource-constrained school device.

### Move hardcoded API base path and chat model config to environment variables: ⚡ Medium Impact
Strings like '/api/admin/help-chat' and any AI model names or token limits embedded in the server handler should be driven by environment config (VITE_API_BASE, OPENAI_MODEL, MAX_CHAT_TOKENS). This allows staging vs production to point at different endpoints without code changes, and prevents accidental exposure of model identifiers in client bundles.
