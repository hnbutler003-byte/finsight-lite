## 🔴 Critical (7)

### `server/storage.ts` — getLeaderboard
**Issue:** N+1 query pattern: the leaderboard implementation fetches game sessions then issues a separate user lookup (getUser or equivalent) for each distinct userId in a loop, producing O(n) sequential round-trips to the database for what should be a single JOIN.

**Fix:** Rewrite as a single query using Drizzle's join: `db.select({ userId: gameSessions.userId, userName: users.name, avatar: users.avatar, totalScore: sql<number>`sum(${gameSessions.score})`, gamesPlayed: sql<number>`count(*)` }).from(gameSessions).innerJoin(users, eq(gameSessions.userId, users.id)).groupBy(gameSessions.userId, users.name, users.avatar).orderBy(desc(sql`sum(${gameSessions.score})`)).limit(filters?.limit ?? 10)`. Add a WHERE clause for subject/period filters before execution.

### `server/storage.ts` — upsertPortfolioHolding / updateVirtualBalance (called together in trade execution)
**Issue:** Multi-step trade operations (upsert holding + create portfolio transaction + update virtual balance) are executed as three independent database calls with no wrapping transaction. A process crash between steps leaves the database in an inconsistent state — e.g. balance debited but holding never recorded.

**Fix:** Wrap all three calls in `db.transaction(async (tx) => { ... })` using Drizzle's transaction API. Pass the `tx` client into each sub-operation. Roll back automatically on any thrown error so balance, holding, and transaction record are always atomically consistent.

### `server/routes.ts` — AI/tutor endpoints (reserveQuota / openai calls)
**Issue:** No rate limiting middleware is applied to AI-backed endpoints. Any authenticated student can fire unlimited requests, exhausting OpenAI quota and hitting cost caps. This is especially risky for a school app where a single student or a scripted client can spike usage for the entire organisation.

**Fix:** Add express-rate-limit (or a Redis-backed limiter like rate-limiter-flexible) scoped per user and per org before the AI route handlers. Example: rateLimit({ windowMs: 60_000, max: 10, keyGenerator: (req) => req.user?.id ?? req.ip }). Pair with the existing reserveQuota check so both hard quota and burst rate are enforced.

### `server/routes.ts` — videoUpload multer config
**Issue:** 500 MB in-memory multer limit for video uploads buffers the entire file in Node.js process heap. Under concurrent uploads this will exhaust memory and crash the server, with no back-pressure mechanism.

**Fix:** Switch videoUpload to disk storage (multer.diskStorage) writing to a temp directory, or stream directly to object storage using a pass-through stream. Set an explicit concurrent-upload cap via a semaphore. Remove the file from disk in a finally block after the upload completes or fails.

---

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored directly in the job payload row. For a multi-page PDF or high-res image this can easily exceed 10–20 MB per job row, causing DB bloat, slow job-queue queries, and potential OOM when the worker deserialises all in-flight jobs on startup.

**Fix:** Upload the file to private object storage at enqueue time and store only the storage path in the payload (e.g. { paperId, storagePath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of the handler. This keeps job rows small and makes retries cheaper.

---

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique constraint. Duplicate tickers would corrupt portfolio value calculations since holdings join on stockId resolved from ticker lookups, and annualReturnPct simulations would produce ambiguous results.

**Fix:** Change the ticker column declaration to use `uniqueIndex`: add `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` as the table's index argument.

### `client/src/pages/AdminDashboard.tsx` — ObservabilityCard and every useQuery queryFn
**Issue:** All queryFn callbacks use `.then(r => r.json())` without any error handling. If the server returns a non-OK response (401, 500), `r.json()` resolves with an error body rather than rejecting, so React Query never enters its error state and the UI silently renders undefined data.

**Fix:** Add a response guard in apiRequest or inline: `queryFn: async () => { const r = await apiRequest('GET', '/api/admin/observability'); if (!r.ok) throw new Error(await r.text()); return r.json(); }`. Then consume `isError` / `error` in each component to render an error state instead of empty UI.

## 🟡 Warnings (17)

- **`server/storage.ts` (getDashboardStats):** getDashboardStats issues at least four independent awaits sequentially (total income, total expenses, category breakdown, recent transactions) that have no data dependency on each other, adding unnecessary latency on every dashboard load.
  → *Replace sequential awaits with `const [income, expenses, breakdown, recent] = await Promise.all([queryIncome, queryExpenses, queryBreakdown, queryRecent])`. Each query is already scoped to the same userId and date range so they are safe to run concurrently.*
- **`server/storage.ts` (getVirtualBalance):** The initial virtual balance is seeded with a hardcoded value (likely 10000) when no row exists for a new user. This dollar amount is embedded in application logic rather than environment configuration, making it impossible to adjust per-deployment or per-school sponsor without a code change.
  → *Add `INITIAL_VIRTUAL_BALANCE_CENTS` (or similar) to your environment config and read it via `process.env.INITIAL_VIRTUAL_BALANCE ?? '10000'`. Reference the parsed value in the upsert fallback so product/ops can tune it without touching source code.*
- **`server/storage.ts` (getTransactions / getBudgets):** Both methods accept an optional `limit` in filters but apply no maximum cap and default to returning all rows when limit is undefined. On accounts with large transaction histories this causes unbounded memory allocation and slow response times with no pagination guard.
  → *Enforce a server-side cap: `const safeLimit = Math.min(filters?.limit ?? 100, 500)` and always pass `.limit(safeLimit)` to the query. Add a corresponding `offset` parameter to enable cursor/page-based pagination and document the max in the IStorage interface.*
- **`server/routes.ts` (/api/auth/google handler):** Sequential awaits for class/org resolution in Google sign-in: storage.getClassByCode, getOrgEnvironmentById, and getOrganization are called one after another even when the results are independent. For a school-aged user base with concurrent morning logins, this compounds latency unnecessarily.
  → *Where resolvedClass is already known, batch the two downstream lookups: const [classEnv, otherData] = await Promise.all([getOrgEnvironmentById(resolvedClass.envId), someOtherIndependentCall()]). Audit the full auth handler for any other sequential awaits on independent data sources and apply the same pattern.*
- **`server/routes.ts` (sendEmail / getOrCreateContact calls throughout route handlers):** Email sending calls (sendEmail, getOrCreateContact) appear inside request handlers without wrapping try/catch in several places. An SMTP or contact-API failure will propagate as an unhandled rejection, crashing the request and potentially leaking a 500 with internal detail to the client.
  → *Wrap every sendEmail and getOrCreateContact call in its own try/catch. Log the error internally (without exposing SMTP credentials or stack traces) and return a user-friendly message. For non-critical emails (e.g. welcome messages) consider fire-and-forget with explicit catch: sendEmail(...).catch(err => logger.error('email failed', err)).*
- **`server/routes.ts` (Google auth handler — profile.email logging / error responses):** Error paths in the Google auth flow risk reflecting raw error messages (e.g. from verifyGoogleToken or Supabase) directly into the HTTP response body. For a COPPA/FERPA-adjacent app handling children's accounts, leaking internal errors or email addresses in logs/responses is a compliance and security risk.
  → *Catch errors from verifyGoogleToken and all downstream calls, log them server-side with a correlation ID (never log the raw email or token), and return only a generic message to the client: res.status(401).json({ message: 'Authentication failed. Please try again.' }). Ensure winston/pino log levels are set so debug logs containing PII are suppressed in production.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for-loop with individual awaits. For a 40-question paper this means 40 serial round-trips to the DB, which is slow and leaves a wide window for a partial-insert failure mid-loop, requiring the idempotency delete on the next retry to clean up.
  → *Collect all valid question objects into an array first, then execute a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). Wrap the delete and insert together in a transaction so partial inserts are impossible and the idempotency guard is no longer needed.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** olderThanDays is taken directly from job.payload without validation. A corrupted or malicious payload with olderThanDays=0 or a negative value would compute a cutoff in the future and delete all aiUsageEvents rows, or olderThanDays=NaN would produce an invalid date that silently deletes nothing while masking the bug.
  → *Add a guard at the top of the handler: const days = Number(job.payload.olderThanDays); if (!Number.isFinite(days) || days < 1) throw new Error('Invalid olderThanDays: ' + job.payload.olderThanDays); This causes the job to fail visibly rather than silently misbehave.*
- **`server/jobHandlers.ts` (runExtractPaper — extractedText.slice(0, 30000)):** The magic number 30000 is hardcoded and has no corresponding limit on the vision path, meaning image-extracted text is sent to the second OpenAI call unbounded. If the vision response is large, the extraction call can hit token limits and throw, failing the job without a clear error.
  → *Define a single constant MAX_EXTRACT_CHARS = parseInt(process.env.MAX_EXTRACT_CHARS ?? '30000', 10) and apply it to both paths: extractedText = pdfData.text.slice(0, MAX_EXTRACT_CHARS) and extractedText = (visionResponse.choices[0]?.message?.content ?? '').slice(0, MAX_EXTRACT_CHARS). Externalising it also lets you tune without a deploy.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse):** The read stream's error event only guards against sending a second response header, but if res.headersSent is true when the stream errors (i.e. after streaming has begun), the error is swallowed silently and the client receives a truncated file with a 200 status and no indication of failure.
  → *Log the error regardless: .on('error', (err) => { console.error('Object stream error', { objectPath, fileName, message: err.message }); if (!res.headersSent) res.status(500).json({ message: err.message }); else res.destroy(err); }). Calling res.destroy(err) on an already-started response forces the TCP connection closed so the client detects truncation.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, yet every budget query for a user will require a full table scan. With school-aged users accumulating monthly budgets, this degrades linearly over time.
  → *Add an index inside the table's second argument: `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })`. Consider a unique index on (userId, categoryId, period) to also enforce one budget per category per period at the DB level.*
- **`shared/schema.ts` (savingsGoals, billReminders, portfolioHoldings, portfolioTransactions table definitions):** None of these user-scoped tables have an index on userId. Every lookup by user (the dominant query pattern) will be a sequential scan. portfolioTransactions in particular will be queried by (userId, stockId) for P&L calculations.
  → *Add indexes: `idx_savings_goals_user` on savingsGoals.userId, `idx_bill_reminders_user_due` on (userId, nextDueDate) for upcoming-bill queries, `idx_portfolio_holdings_user_stock` unique on (userId, stockId), and `idx_portfolio_txns_user_stock` on (userId, stockId) for portfolioTransactions.*
- **`shared/schema.ts` (createdAt columns across all tables (transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions)):** createdAt uses `.defaultNow()` but is not marked `.notNull()`. Drizzle will type it as `Date | null`, forcing null-checks throughout the application layer and risking silent null propagation into date-sort queries and audit logs.
  → *Append `.notNull()` after `.defaultNow()` on every createdAt column, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. The DB default ensures the value is always populated on insert.*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is nullable (system categories have null userId), but there is no index on userId. Any query filtering user-owned categories (e.g. showing a user's custom categories) scans the whole table including all system categories, and the lack of a partial index makes this worse at scale.
  → *Add a partial index covering only user-owned rows: `userIdx: index('idx_categories_user').on(t.userId)` in the table's index argument. If the ORM supports it, a `WHERE user_id IS NOT NULL` partial index via raw SQL migration is more efficient.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is rebuilt on every render outside of useMemo, causing a new object reference each render even when inputs have not changed. Because queryFn closes over `params`, this also means the queryFn string changes every render, risking spurious refetches if queryKey serialization ever drifts.
  → *Wrap params construction in useMemo keyed on [actorType, action, from, to]: `const paramString = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p.toString(); }, [actorType, action, from, to]);` then use `paramString` in both queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (fmtDate):** fmtDate accepts `any` and calls `new Date(d)` without validation. Passing an invalid string produces 'Invalid Date' silently rendered to users. For a financial literacy app shown to minors, corrupted date display erodes trust and could mask data integrity issues.
  → *Add a validity check: `function fmtDate(d: unknown): string { if (!d) return '—'; const dt = new Date(d as string); if (isNaN(dt.getTime())) return '—'; return dt.toLocaleDateString('en-US', { year: 'numeric', month: 'short', day: 'numeric' }); }` and type the parameter as `string | number | null | undefined`.*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and tab-gated useQuery calls):** Based on the visible pattern, each tab panel likely issues its useQuery unconditionally regardless of whether that tab is active. With 13 tabs this means up to 13 simultaneous API calls on first load, including heavy endpoints like audit-log (limit 200) and DB viewer, creating unnecessary load on the server and increasing time-to-interactive.
  → *Pass `enabled: activeTab === 'audit'` (or the relevant tab id) to each tab-specific useQuery so data is only fetched when the user navigates to that tab. For the overview tab, use Promise.all in a single queryFn to fetch summary stats in parallel rather than N sequential queries.*

## 💡 Opportunities

### Centralise ownership guard into a reusable query helper — 🔥 High Impact
Every mutating method (deleteTransaction, deleteBudget, deleteClass, etc.) repeats the same pattern of fetching a record then checking userId/teacherId before deleting. Extract a generic `assertOwner(table, id, ownerColumn, ownerValue)` helper that performs a single WHERE id=? AND owner=? query and throws a typed NotFoundError on mismatch. This eliminates the fetch-then-check N+1 and removes ~15 blocks of duplicated code.

### Add query-level pagination to leaderboard and getAllQuestions — 🔥 High Impact
Both endpoints are called from the client UI with optional limit params but currently load all matching rows into Node memory before slicing. Pushing LIMIT/OFFSET into the SQL query reduces memory pressure proportionally to dataset size — critical as school and student counts grow across the Caribbean rollout.

### Introduce a structured logger and scrub PII from error paths — ⚡ Medium Impact
Several catch blocks (visible in the truncated section patterns) log raw error objects that may include SQL query strings containing userId or transaction amounts. Replace with a structured logger (e.g. pino) that serialises only safe fields, and ensure no user-identifiable data surfaces in log output — required for any deployment touching minors under regional data protection rules.

### Decompose monolithic route file into domain routers — 🔥 High Impact
routes.ts is over 196 KB of concatenated route handlers. Splitting into domain-specific routers (auth, lessons, ai, org, student) would reduce merge conflicts, make rate-limit and auth middleware easier to apply per domain, and cut cold-parse time. Expected impact: faster onboarding, fewer cross-cutting bugs.

### Centralise AI quota enforcement in a single middleware — 🔥 High Impact
reserveQuota, finalizeUsage, and releaseReservation are currently called ad-hoc inside individual handlers. Extracting them into a single Express middleware applied to all /api/ai/* routes guarantees consistent enforcement, removes duplicated error-handling code, and makes it trivial to add rate limiting at the same layer.

### Replace multer memoryStorage with streaming object storage uploads — ⚡ Medium Impact
Using memory storage for both logo (5 MB) and video (500 MB) uploads means every concurrent upload holds its full payload in heap. Streaming directly to object storage via a piped pass-through would cap heap impact to the chunk size (typically 64 KB), dramatically improving stability under classroom-hour load spikes.

### Batch independent DB reads with Promise.all in export handlers — 🔥 High Impact
The admin-csv-export handler (truncated but implied by the type-switch pattern) likely fires multiple sequential awaits to fetch different tables. Wrapping independent queries in Promise.all cuts wall-clock time proportionally to the number of queries and reduces the window during which the job holds a DB connection.

### Centralise job payload validation with Zod schemas per job type — ⚡ Medium Impact
Each handler currently accesses job.payload fields with no runtime type checking. Adding a thin Zod schema per job type (co-located with the JobPayloads type) and parsing at the top of each handler would catch malformed payloads at the boundary, produce actionable error messages, and eliminate defensive Number() casts scattered through handler bodies.

### Add structured logging with job context to all handlers — ⚡ Medium Impact
Errors are currently rethrown raw with no job metadata attached. Wrapping each registerJobHandler callback with a logger that includes { jobType, jobId, attempt } on every log line would dramatically reduce mean-time-to-diagnose for production incidents without requiring code changes in individual handlers.

### Centralise currency default into a config constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the currency default in at least five tables (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks). Extract `DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD'` into a shared config module and reference it in each column default. This allows the app to support other Caribbean currencies (TTD, JMD) without a schema migration.

### Add a unique constraint on portfolioHoldings (userId, stockId) — 🔥 High Impact
Without a unique constraint, a race condition on concurrent buy orders can create duplicate holding rows for the same user+stock, causing double-counted quantities in portfolio value calculations. A unique index enforces the invariant at the DB level and removes the need for application-layer upsert guards.

### Use numeric type for portfolioTransactions.quantity and portfolioHoldings.quantity — 📌 Low Impact
Both quantity columns are typed as integer, which prevents fractional share purchases (common in bond and ETF simulations). Changing to `numeric('quantity', { precision: 12, scale: 6 })` future-proofs the simulation engine without breaking existing whole-number data and avoids a disruptive migration later.

### Parallel data fetching on Overview tab — 🔥 High Impact
If the Dashboard/overview tab fires multiple sequential useQuery calls (users count, schools count, challenges count, etc.), consolidate them into a single `/api/admin/stats` endpoint that returns all summary data in one round-trip, or use Promise.all inside one queryFn. Expected impact: reduces dashboard load time by N-1 network round-trips and eliminates cascading loading spinners.

### Virtualize large data tables — 🔥 High Impact
Audit log and student/teacher lists rendered as plain DOM rows at 200+ items cause slow initial paint and laggy scrolling on low-end devices common in Caribbean schools. Replacing with TanStack Virtual (already likely in the dependency tree via TanStack Query) would cap DOM nodes at ~20 regardless of dataset size.

### Extract tab panels into lazy-loaded route segments — ⚡ Medium Impact
With 13 tabs importing Recharts, Lucide icons, VideoField, and Dialog components, the initial JS bundle is large. Wrapping each tab panel in React.lazy + Suspense would defer loading Recharts (the heaviest dependency) until the user visits a chart tab, reducing TTI for the common admin landing page.
