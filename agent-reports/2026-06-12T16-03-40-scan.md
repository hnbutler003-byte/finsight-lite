## 🔴 Critical (8)

### `server/routes.ts` — registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 108)
**Issue:** No rate limiting is applied before registerChatRoutes, registerImageRoutes, or registerAudioRoutes. These call OpenAI APIs which are expensive and latency-sensitive. In a school-aged user app, a single misbehaving client or script could exhaust OpenAI quota or cause runaway billing.

**Fix:** Apply a dedicated AI rate limiter before each registration: e.g. app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the register calls. Create aiLimiter in rateLimiter.ts with a conservative window (e.g. 10 req/min per IP).

### `server/routes.ts` — robots.txt handler (~line 47) and sitemap.xml handler (~line 61)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header, causing the Sitemap directive or sitemap <loc> tags to point to an arbitrary domain — a classic Host Header Injection that can aid cache poisoning or phishing.

**Fix:** Read the canonical base URL from an environment variable: const base = process.env.PUBLIC_URL || 'https://finsightlite.com'; and ignore the request host entirely for these URL constructions. Remove the dynamic proto/host derivation from both handlers.

### `server/jobHandlers.ts` — runExtractPaper (fileB64 parameter)
**Issue:** Full base64-encoded file content is stored directly in the job payload row. A single 5 MB PDF becomes ~6.7 MB of base64 in the jobs table, multiplied by every retry attempt. This bloats the DB, slows job queue polling queries, and risks hitting row size limits.

**Fix:** Upload the file to object storage immediately on ingest and store only the resulting storage path in the job payload (e.g., { paperId, storagePath, ext, subject }). Inside runExtractPaper, download the buffer from object storage at execution time using objectStorageClient. Remove fileB64 from the payload type entirely.

### `shared/schema.ts` — categories table definition
**Issue:** userId in categories uses bare varchar() with no length argument, and the same pattern repeats across every table (transactions, budgets, linkedCards, etc.). Drizzle maps bare varchar() to varchar(255) in Postgres, but if the auth provider (e.g. Clerk, Supabase) issues IDs longer than 255 chars or you later switch providers, values will be silently truncated without an error in older Postgres versions, corrupting FK relationships.

**Fix:** Audit your auth provider's max ID length and pin every user_id column explicitly, e.g. varchar('user_id', { length: 36 }) for UUID-style IDs or varchar('user_id', { length: 128 }) for Clerk-style IDs. Apply consistently across all 10+ tables that reference users.id.

### `shared/schema.ts` — budgets table definition
**Issue:** The budgets table has no index on userId, even though every realistic query (fetch user's budgets, check budget vs. spend) will filter by userId. With no index, these queries will full-scan the table as data grows, which is especially problematic given this is a school-aged multi-tenant app where all users share the same table.

**Fix:** Add an index in the table's third argument: (t) => ({ userIdx: index('idx_budgets_user').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The composite index on (userId, categoryId) also enforces the logical uniqueness of one budget per user per category per period — consider a uniqueIndex there.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** The scheduler intervals are created but their IDs are never stored, so they cannot be cleared during graceful shutdown. On SIGTERM the process exits while intervals may have in-flight DB writes (the delete rollback in maybeRunWeeklyDigest), risking data corruption or duplicate job claims on restart.

**Fix:** Store interval IDs: `const weeklyInterval = setInterval(maybeRunWeeklyDigest, ...)`. In the SIGTERM/SIGINT handler call `clearInterval(weeklyInterval)` and `clearInterval(purgeInterval)` before closing the HTTP server and DB connection. Use a single `shutdown` async function registered on both signals to avoid duplication.

### `server/index.ts` — res.json monkey-patch in request logging middleware
**Issue:** Every API response body is captured into `capturedJsonResponse` and then serialized with JSON.stringify unconditionally into the log line. For endpoints returning auth tokens, session data, PII (student records), or AI completions this will leak sensitive data into stdout/log aggregators visible to ops staff.

**Fix:** Introduce a blocklist of sensitive path prefixes (e.g. `/api/auth`, `/api/user`, `/api/ai`) and skip body logging for those routes: `if (capturedJsonResponse && !SENSITIVE_PATHS.some(p => path.startsWith(p))) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`. For a school-age user base this is a COPPA/data-protection concern.

### `client/src/pages/AdminDashboard.tsx` — AdminHelpChat > sendMessage
**Issue:** The chat endpoint is called with the full accumulated message history on every send, but there is no server-side auth token attached and no rate limiting guard on the client. A non-admin user who discovers the /api/admin/help-chat endpoint can POST arbitrary message arrays directly. Combined with the fact that the catch block silently swallows all errors with a generic string, failed auth responses are indistinguishable from network errors.

**Fix:** Ensure apiRequest forwards the session/JWT cookie (verify withCredentials or Authorization header is set in the shared apiRequest helper). Add a client-side guard: if useAdminAuth() returns no valid admin session, do not render AdminHelpChat at all. Surface HTTP 401/403 specifically in the catch block so the UI can redirect to login rather than showing a generic error.

## 🟡 Warnings (15)

- **`server/routes.ts` (Supabase init chain (~line 114)):** The Supabase init chain runs sequentially (each .then() waits for the previous) when most of these seed operations are independent. Additionally, only a single .catch() covers the whole chain, so if seedFinancialAcademyLesson succeeds but seedStaticModules throws, earlier successful operations are already committed and the error message gives no indication of which step failed.
  → *Run independent seed operations in parallel: initSupabaseTables().then(() => Promise.all([trimOrgNamesInSupabase(), seedFinancialAcademyLesson(), initStaticContentTables()])).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e)); Also add per-operation labels in individual catches for observability.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 95)):** The filePath parameter is passed directly to objectStorage.searchPublicObject without any sanitization. A path traversal payload like ../../private/config could resolve to unintended storage locations depending on how ObjectStorageService implements path resolution.
  → *Sanitize the path before use: import path from 'path'; const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); and reject any path that contains '..' after normalization with a 400 response. Ensure ObjectStorageService also enforces a root prefix.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted sequentially in a for-await loop — 20 questions means 20 round-trips to the DB with no batching. This is slow and holds the job open longer than necessary, increasing retry collision risk.
  → *Replace the sequential loop with a single batch insert: collect all valid question objects into an array first, then call db.insert(extractedQuestions).values(validQuestions) once. Guard with if (validQuestions.length > 0) before inserting.*
- **`server/jobHandlers.ts` (runExtractPaper — openai.chat.completions.create calls (model: 'gpt-4o', max_completion_tokens: 4096)):** OpenAI model name and token limits are hardcoded strings. If the model needs to be changed for cost, compliance, or availability reasons (common in a school-budget app), every call site requires a code change and redeployment.
  → *Add OPENAI_EXTRACTION_MODEL (default 'gpt-4o') and OPENAI_MAX_EXTRACTION_TOKENS (default 4096) to environment config. Read them at handler registration time: const model = process.env.OPENAI_EXTRACTION_MODEL ?? 'gpt-4o'. Apply the same pattern to the vision call.*
- **`server/jobHandlers.ts` (purge-ai-usage handler):** No try/catch wraps the db.delete call. If the DB is temporarily unavailable the job worker will receive an unhandled rejection. Depending on the job runner's error handling, this may silently swallow the failure or crash the worker process rather than retrying cleanly.
  → *Wrap the delete in try/catch, log the error with enough context (olderThanDays, cutoff date) at error level, and re-throw so the job framework can schedule a retry. Also validate that olderThanDays is a positive finite number before computing the cutoff to prevent a runaway delete with a zero/negative value.*
- **`server/jobHandlers.ts` (streamPrivateObjectToResponse — file.createReadStream().pipe(res)):** The read stream is piped to the response but there is no 'end' or 'finish' event handler and no cleanup if the client disconnects mid-stream. If the client drops the connection, the GCS read stream may continue pulling data until it finishes, wasting bandwidth and leaving the stream open.
  → *Capture the read stream in a variable, listen for req 'close' event (pass req as a parameter), and call readStream.destroy() on client disconnect. Add a 'finish' handler on res to also call readStream.destroy() defensively: const rs = file.createReadStream(); req.on('close', () => rs.destroy()); rs.on('error', ...).pipe(res);*
- **`shared/schema.ts` (portfolioHoldings and portfolioTransactions table definitions):** portfolioHoldings has no index on userId or stockId, and portfolioTransactions has no index on userId or executedAt. Portfolio dashboard and trade history queries will both filter on userId and sort by executedAt, resulting in sequential scans on what could be the most-read tables in the investment simulation feature.
  → *Add to portfolioHoldings: (t) => ({ userIdx: index('idx_holdings_user').on(t.userId), userStockUniq: uniqueIndex('idx_holdings_user_stock').on(t.userId, t.stockId) }). Add to portfolioTransactions: (t) => ({ userDateIdx: index('idx_ptx_user_date').on(t.userId, t.executedAt) }). The unique index on holdings also prevents duplicate holding rows for the same user+stock, which is a data integrity bug waiting to happen.*
- **`shared/schema.ts` (transactions, documentUploads, budgets, linkedCards, savingsGoals, billReminders — createdAt columns):** createdAt is defined as timestamp('created_at').defaultNow() without .notNull() across all tables. Drizzle will infer the column as nullable in the TypeScript type, meaning application code that destructures createdAt must handle null even though the default guarantees a value at insert time. This creates unnecessary null-checks or, worse, suppressed nulls that cause runtime errors in date formatting.
  → *Append .notNull() to every createdAt: timestamp('created_at').defaultNow().notNull(). Do the same for purchasedAt in portfolioHoldings and executedAt in portfolioTransactions. This tightens the DB constraint and removes null from the inferred TS type.*
- **`shared/schema.ts` (simulatedStocks table — ticker column):** ticker has no unique index. Two rows with ticker 'AAPL' can be inserted, which would break any lookup by ticker symbol and produce duplicate results in portfolio displays. There is also no index on ticker despite it being the natural lookup key for the investment simulation UI.
  → *Change the column definition or add in the index block: (t) => ({ tickerUniq: uniqueIndex('idx_stocks_ticker').on(t.ticker) }). If multi-currency/multi-region tickers are intentionally non-unique, add a composite uniqueIndex on (ticker, region) instead and document that decision.*
- **`server/index.ts` (seedDatabase / probeDatabase / startJobWorker calls in startup):** DB probe, seed, and job worker initialization are likely awaited sequentially during startup. These are independent operations and each adds to cold-start latency. On a slow DB connection this can cause the HTTP server to accept no traffic for several seconds.
  → *Group independent startup tasks with Promise.all: `await Promise.all([probeDatabase(), seedDatabase()])` before `startJobWorker()` and `registerJobHandlers()` if those depend on DB readiness. Measure each phase with Date.now() deltas so regressions are visible in logs.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes `(dow + 0) % 7` — the `+ 0` is a no-op, and the result is that Sunday (dow=0) returns `daysSince=0`, setting the date to today. Then if that Sunday-at-7pm is in the past it is used correctly, but on any Sunday before 7pm the function returns last week's date, which is correct. However the real bug is that the digest fires relative to the server's local timezone (whatever the Node process TZ is), not a fixed Caribbean timezone, so digest timing drifts across environments.
  → *Replace `new Date()` construction with explicit UTC-to-AST conversion: set `process.env.TZ='America/Port_of_Spain'` at process start (before any Date construction), or use a library like `date-fns-tz` to compute the target in `America/Port_of_Spain`. Also remove the dead `+ 0` to avoid future confusion.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE and scheduler interval value):** The purge interval period (how often the scheduler ticks, e.g. every hour or every day) appears to be hardcoded in the truncated section. If it is hardcoded like the digest interval, there is no way to adjust scheduling frequency without a deploy. Additionally, AUTO_PURGE_OLDER_THAN_DAYS silently falls back to 180 with only a log warning absent — if an operator sets `AI_PURGE_OLDER_THAN_DAYS=abc` by mistake, data is silently purged on the default schedule with no alert.
  → *Log a warning when the env var is present but invalid: `if (process.env.AI_PURGE_OLDER_THAN_DAYS && !Number.isFinite(v)) { console.warn('[startup] AI_PURGE_OLDER_THAN_DAYS is invalid, defaulting to 180'); }`. Move the scheduler tick interval to an env var `DIGEST_INTERVAL_MS` with a documented default, validated the same way.*
- **`client/src/pages/AdminDashboard.tsx` (AdminHelpChat > messages.map render > inline ref callback):** The auto-scroll sentinel uses an inline ref callback `ref={el => { if (el) el.scrollIntoView(...) }}` rendered inside the message list. This callback is recreated on every render, and because it calls scrollIntoView unconditionally whenever the element mounts, it fires on every re-render of the parent (e.g., while the user is typing in the input field), hijacking scroll position unexpectedly.
  → *Replace with a stable useRef + useEffect pattern: `const bottomRef = useRef<HTMLDivElement>(null); useEffect(() => { bottomRef.current?.scrollIntoView({ behavior: 'smooth' }); }, [messages]);` and render `<div ref={bottomRef} />`. This fires only when messages actually change.*
- **`client/src/pages/AdminDashboard.tsx` (Top-level dashboard component — multiple useQuery calls):** Multiple independent useQuery calls (users, schools, lessons, analytics, audit log, etc.) are issued sequentially as separate hooks, but several are only needed for specific sub-tabs the admin has not opened yet. All queries fire on initial mount regardless of active tab, creating redundant network load and exposing full data payloads to the client before they are needed.
  → *Add `enabled: activeTab === 'users'` (or the relevant tab name) to each query's options so data is only fetched when the corresponding tab is active. For queries that are always needed (overview stats), keep them enabled. This reduces initial payload and avoids fetching sensitive user/audit data until an admin explicitly navigates to those sections.*
- **`client/src/pages/AdminDashboard.tsx` (Large inline mutation handlers inside JSX (Trash2/Pencil onClick handlers)):** Delete and edit mutation trigger functions are defined as anonymous arrow functions directly in JSX props (e.g., `onClick={() => deleteMutation.mutate(user.id)}`). Inside a list that maps over potentially hundreds of users/schools, this creates a new function reference for every item on every parent render, causing all list-item components to re-render even when unrelated state changes (e.g., search input keystroke).
  → *Wrap each handler with useCallback, or extract list items into a separate memoized component (e.g., `UserRow`) that accepts stable props and uses React.memo. The UserRow component's onClick handlers can then be defined once with useCallback inside it, keyed by item id, preventing unnecessary re-renders across the full list.*

## 💡 Opportunities

### Parallel domain route registration — ⚡ Medium Impact
The six registerXxxRoutes calls after the parallel import are still executed sequentially with individual awaits. If any route registrar does async work (e.g. DB schema checks), this adds latency. Wrap them in a single Promise.all: await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]); — safe as long as no registrar depends on another's side effects.

### Structured logging for Supabase init errors — ⚡ Medium Impact
The single .catch logs a raw error to console.error. Integrating with the already-imported Sentry context (sentryRequestContext is set up in the same file) by calling Sentry.captureException(e) in the catch block would surface seed failures in the error dashboard rather than silently burying them in logs, which is critical for a production financial literacy app.

### Object storage response caching headers — 📌 Low Impact
The /public-objects route passes a 30-day TTL to downloadObject but there is no Cache-Control header set on the Express response itself, meaning CDN and browser caching depends entirely on whatever ObjectStorageService sets. Explicitly set res.setHeader('Cache-Control', 'public, max-age=2592000, immutable') before calling downloadObject to guarantee edge caching for logos and videos, reducing latency for school-network users across the Caribbean.

### Parallel OpenAI calls for text extraction and parsing — ⚡ Medium Impact
For image inputs, the vision extraction and the subsequent GPT-4o JSON parsing call are strictly sequential. If the architecture ever supports pre-extracting text (e.g., cached OCR), the two calls could be pipelined. More immediately, the storage.updateExamPaper final status update and the last createExtractedQuestion insert are also sequential where the update could be deferred until after a confirmed batch insert completes — reducing total wall-clock job time by one DB round-trip.

### Idempotency token on extracted question batch insert — ⚡ Medium Impact
The current idempotency strategy deletes all existing questions for the paper before re-running. This creates a window where questions are absent if the job crashes between delete and re-insert. Using an upsert (INSERT ... ON CONFLICT DO UPDATE) keyed on (paperId, order) would eliminate the deletion window and make retries truly safe without data loss exposure.

### Centralise OpenAI client instantiation away from route imports — 📌 Low Impact
openai is imported from a chat routes file, coupling the job handler to a web-layer module. If that route file is refactored or its exports change, the job handler breaks silently at import time. Move the OpenAI client to a dedicated shared module (e.g., server/lib/openaiClient.ts) imported by both routes and job handlers, reducing coupling and making the dependency explicit.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as the column default in at least 6 tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract to a single exported constant DEFAULT_CURRENCY = 'BSD' in a config file and reference it in each .default(DEFAULT_CURRENCY). When the app expands to TTD or JMD markets, you change one line instead of hunting schema files.

### Add a partial index on documentUploads status for processing queue queries — ⚡ Medium Impact
Background jobs that poll for documents with status='processing' will scan the entire documentUploads table. A partial index — CREATE INDEX idx_uploads_processing ON document_uploads(created_at) WHERE status = 'processing' — reduces that scan to only in-flight rows. In Drizzle this can be expressed with .where(sql`status = 'processing'`) on the index definition once Drizzle supports partial indexes, or via a raw migration.

### Replace open-ended jsonb columns with typed sub-tables or typed columns — 🔥 High Impact
If the truncated portion of the schema contains jsonb columns (common for storing module content, quiz answers, or AI responses), these bypass Postgres type enforcement and make querying and indexing impossible without expression indexes. Identify each jsonb usage, define a concrete shape, and either normalise into a child table or at minimum add a Zod schema + DB check constraint to validate structure on insert.

### Unified graceful shutdown handler — 🔥 High Impact
Introduce a single `shutdown(signal)` async function that: clears all intervals, stops accepting new HTTP connections via `httpServer.close()`, waits for in-flight requests with a 10-second timeout, then closes the DB pool. Register it on SIGTERM and SIGINT. This prevents the current state where a Replit container restart mid-digest-enqueue leaves orphaned job claims that block the next run.

### Structured JSON logging with log levels — ⚡ Medium Impact
Replace `console.log` string concatenation with a structured logger (pino is zero-dependency and very fast). Each log entry would carry `level`, `source`, `durationMs`, and `path` as discrete fields rather than interpolated strings. This makes it trivial to filter out sensitive-path logs at the transport layer and enables log-level toggling without code changes — important for a production school app where verbose AI logs must be suppressible.

### Extract scheduler logic into a dedicated module — ⚡ Medium Impact
Moving `maybeRunWeeklyDigest`, `maybeRunAiUsagePurge`, `getMostRecentSundayAt7pm`, and their interval registrations into `server/scheduler.ts` that exports `startSchedulers(): { stop(): Promise<void> }` would make the intervals testable in isolation, ensure interval IDs are always paired with a cleanup method, and reduce server/index.ts to pure wiring. Current co-location of business scheduling logic with HTTP server setup is the root cause of the missing cleanup and timezone issues.

### Virtualize long admin data lists — 🔥 High Impact
User, school, and lesson tables likely render full DOM node sets for every paginated record. Replacing them with a windowed list (e.g., TanStack Virtual) would reduce DOM node count from O(n) to ~20 visible rows, meaningfully improving scroll performance and memory usage on low-end Caribbean school devices.

### Memoize derived stats and chart data — ⚡ Medium Impact
Analytics and chart data derived from raw query results (totals, grouped series) are recomputed on every render. Wrapping these derivations in useMemo keyed on the query data would eliminate redundant computation during unrelated state changes like tab switches or search input, which is especially noticeable when recharts data arrays are rebuilt unnecessarily.

### Split AdminDashboard into lazy-loaded tab modules — 🔥 High Impact
The single monolithic file (131k+ chars) bundles recharts, the help chat, all CRUD dialogs, and every admin table into one chunk. Splitting each major tab (Users, Schools, Lessons, Analytics, Audit) into React.lazy() dynamic imports would reduce initial JS parse time, benefiting users on slower mobile connections common in the Caribbean region.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. ### Admin Frontend (`client/src/pages/AdminDashboard.tsx`)

The component is large and monolithic, causing broad re-render surfaces, and several expensive operations lack memoization. Overall the code is functional but carries real risk for a school-aged financial app handling user data.
