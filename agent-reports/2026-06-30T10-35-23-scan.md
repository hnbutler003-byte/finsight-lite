## 🔴 Critical (8)

### `server/storage.ts`: getBudgets
**Issue:** N+1 query pattern: getBudgets fetches all budgets then computes 'spent' with a separate aggregation query per budget row inside a loop, resulting in 1+N database round trips.

**Fix:** Replace the per-budget loop with a single GROUP BY query joining transactions to budgets: SELECT budget_id, SUM(amount) as spent FROM transactions WHERE user_id = ? AND category_id IN (...) GROUP BY budget_id. Then merge results in memory using a Map keyed on budget_id before returning.

### `server/storage.ts`: getLeaderboard
**Issue:** Missing pagination guard on leaderboard query. The limit parameter is optional and likely defaults to no LIMIT clause, meaning a full table scan across all game_sessions and users is returned to the caller as the user base grows.

**Fix:** Enforce a hard cap: const safeLimit = Math.min(filters?.limit ?? 25, 100). Apply .limit(safeLimit) unconditionally on the Drizzle query. Add a corresponding guard in the route handler rejecting limit > 100 with a 400 response.

### `server/routes.ts`: registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** No rate limiting applied to AI/chat/image/audio routes before they are registered. These are the heaviest and most expensive endpoints (OpenAI API calls), yet any unauthenticated or authenticated client can hammer them without restriction, creating both cost exposure and DoS risk for a school-aged user base.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/15min per IP) in rateLimiter.ts and apply it before registering these routes: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions.

### `server/routes.ts`: robots.txt handler and sitemap.xml handler (~line 41, ~line 57)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header to make the sitemap and robots.txt emit arbitrary URLs (host header injection), which can be abused for cache poisoning or SEO manipulation.

**Fix:** Add a trusted host allowlist: const ALLOWED_HOSTS = (process.env.ALLOWED_HOSTS || '').split(','); then validate: const rawHost = req.headers['x-forwarded-host'] || req.headers.host || ''; const host = ALLOWED_HOSTS.includes(rawHost) ? rawHost : process.env.DEFAULT_HOST; Alternatively, set the canonical base URL once from a single env var (CANONICAL_URL) and ignore request headers entirely.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Large base64 file contents are stored directly in the job payload. For a PDF or image this can easily be several MB per row, bloating the jobs table, slowing queue polls, and potentially exceeding any payload column size limit.

**Fix:** Upload the file to object storage before enqueuing, store only the object path in the payload (e.g. { paperId, objectPath, ext, subject }), then download the buffer inside runExtractPaper using objectStorageClient. Remove fileB64 from JobPayloads['extract-paper'] entirely.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets table has no index on userId, yet it will always be queried by userId to render a user's budget dashboard. Full table scans will occur on every page load as the table grows.

**Fix:** Add an index in the budgets table definition: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId), userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). The composite index on (userId, categoryId) also supports the common 'fetch budget for a specific category' query.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are created but their handles are never stored or cleared. When SIGTERM/SIGINT fires the intervals keep the event loop alive, preventing clean shutdown and potentially firing mid-shutdown against a half-torn-down DB connection pool.

**Fix:** Store both handles: const digestInterval = setInterval(...) and const purgeInterval = setInterval(...). In the graceful shutdown handler call clearInterval(digestInterval) and clearInterval(purgeInterval) before closing the HTTP server and DB pool. Register shutdown via process.on('SIGTERM', shutdown) and process.on('SIGINT', shutdown).

---

### `server/index.ts`: res.on('finish') logging block inside the request logger middleware
**Issue:** capturedJsonResponse is serialized in full and appended to every API log line with no length cap, field allowlist, or redaction. For a financial literacy app with school-aged users this means tokens, grades, scores, or any sensitive fields returned by any endpoint are written to stdout/log sinks in plaintext on every request.

**Fix:** Replace unconditional full-body logging with a size-gated summary: if capturedJsonResponse is defined, log only its top-level keys and array lengths (e.g. '{keys:["data","total"], dataLen:42}') or truncate the serialized string to 120 chars. Never log fields like token, password, secret, or ssn — add an explicit blocklist strip before serialization.

## 🟡 Warnings (14)

- **`server/storage.ts` (getBillReminders):** N+1 query pattern: getBillReminders fetches reminders then resolves each reminder's category with a separate query per row, producing 1+N round trips.
  → *Collect all distinct categoryIds from the reminders array, fetch them in one query using inArray(categories.id, categoryIds), build a Map<id, Category>, then attach categories in-memory during the map over reminders. Zero extra queries.*
- **`server/storage.ts` (completeModule / updateUserXp):** Multi-step writes (insert progress row, then update XP, then conditionally award badge) are not wrapped in a database transaction. A crash between steps leaves the user with partial state: module marked complete but XP not awarded, or XP updated but badge missing.
  → *Wrap the full sequence in db.transaction(async (tx) => { ... }) using Drizzle's transaction API, passing tx as the query context to each sub-operation. Roll back automatically on any thrown error.*
- **`server/storage.ts` (getDashboardStats):** Sequential awaits for independent aggregations (total income, total expenses, savings balance, budget count) are executed one after another instead of in parallel, multiplying latency by the number of queries on every dashboard load.
  → *Replace sequential awaits with a single Promise.all: const [income, expenses, savings, budgetCount] = await Promise.all([fetchIncome(), fetchExpenses(), fetchSavings(), fetchBudgetCount()]). Each inner fetch is independent and has no data dependency on the others.*
- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 114)):** The seven Supabase init/seed functions are chained sequentially with .then(), meaning each step waits for the previous one even when there is no data dependency between them (e.g. seedRegionalContent and seedGameContent are independent). This unnecessarily extends cold-start time.
  → *After the mandatory prerequisite steps complete, run independent seeds in parallel: initSupabaseTables().then(() => trimOrgNamesInSupabase()).then(() => seedFinancialAcademyLesson()).then(() => initStaticContentTables()).then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] Init error:', e));*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 101)):** The filePath parameter from the URL is passed directly to objectStorage.searchPublicObject without any sanitization. A path like ../../secrets/config could traverse into unintended storage paths depending on the ObjectStorageService implementation.
  → *Sanitize the path before use: const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, ''); or use Node's path.normalize and assert the result does not escape a known prefix: if (!normalizedPath.startsWith('public/')) return res.status(400).json({ error: 'Invalid path' });*
- **`server/jobHandlers.ts` (runExtractPaper – for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one-by-one in a sequential await loop. For a 40-question exam paper this issues 40 serial round-trips to the database, adding significant latency and holding the job worker slot unnecessarily long.
  → *Collect all valid question objects into an array, then call a single db.insert(extractedQuestions).values(allQuestions) batch insert outside the loop. If storage.createExtractedQuestion must be used, wrap all calls in Promise.all() after building the array of promises.*
- **`server/jobHandlers.ts` (registerJobHandler('purge-ai-usage', ...)):** The purge-ai-usage handler has no try/catch. If the DELETE query throws (e.g. DB connection drop), the raw database error — which may contain schema details — propagates unhandled to the job runner with no structured logging or context attached.
  → *Wrap the db.delete call in try/catch, log the error with structured context ({ job: 'purge-ai-usage', olderThanDays, cutoff }), then re-throw so the job runner can retry. Mirror the pattern already used in the ai-health-check handler.*
- **`server/jobHandlers.ts` (runExtractPaper – openai.chat.completions.create calls (max_completion_tokens: 4096)):** Token limits and the model name 'gpt-4o' are hardcoded. Changing the model or raising limits for a specific environment requires a code deploy rather than a config change, and accidentally committing a wrong value affects all environments.
  → *Read these from environment variables with safe defaults: const EXTRACT_MODEL = process.env.OPENAI_EXTRACT_MODEL ?? 'gpt-4o'; const EXTRACT_MAX_TOKENS = parseInt(process.env.OPENAI_EXTRACT_MAX_TOKENS ?? '4096', 10); Apply consistently to both openai calls in runExtractPaper.*
- **`shared/schema.ts` (varchar user_id columns across all tables):** All userId columns are declared as varchar() with no length limit. Without a length constraint, Postgres stores it as unbounded text, bypassing storage optimisation, and there is no enforcement that IDs conform to an expected format (e.g. UUID = 36 chars). This also prevents the DB from catching truncation bugs at the application layer.
  → *Change all varchar('user_id') declarations to varchar('user_id', { length: 36 }) if using UUID-format IDs, or to the exact length your auth provider emits. Apply consistently to every table: categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, and any others in the truncated portion.*
- **`shared/schema.ts` (createdAt columns across all tables):** All createdAt timestamp columns use .defaultNow() but are not marked .notNull(). This means application code or a direct INSERT can explicitly pass NULL, creating rows with no creation timestamp, breaking audit trails and time-based queries silently.
  → *Chain .notNull() after .defaultNow() on every createdAt column, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.*
- **`shared/schema.ts` (simulatedStocks table definition):** ticker column has no unique index. Duplicate tickers can be inserted, causing ambiguous JOIN results when portfolioHoldings or portfolioTransactions reference the same real-world instrument via ticker lookup. For a financial literacy app showing live simulated prices, duplicate tickers will display conflicting prices to students.
  → *Add a unique index in the simulatedStocks table options: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also consider adding an index on (ticker) alone if lookups by ticker string are common in application queries.*
- **`shared/schema.ts` (billReminders table definition):** nextDueDate has no index. Bill reminder queries will almost certainly filter by userId and nextDueDate (e.g. 'show bills due in the next 7 days') but only userId-level filtering is indexable without this. As reminders accumulate this becomes a sequential scan of all a user's reminders.
  → *Add a composite index: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). This supports range queries like WHERE user_id = $1 AND next_due_date BETWEEN now() AND now() + interval '7 days' efficiently.*
- **`server/index.ts` (startJobWorker / registerJobHandlers / seedDatabase / probeDatabase calls in startup IIFE):** Startup awaits probeDatabase, seedDatabase, registerJobHandlers, and startJobWorker sequentially even though probeDatabase and seedDatabase have no dependency on the job worker. On a cold start this adds the full latency of each operation end-to-end before the HTTP server binds, delaying readiness unnecessarily.
  → *Split startup into two parallel phases: await Promise.all([probeDatabase(), seedDatabase()]) first, then await Promise.all([registerJobHandlers(), startJobWorker()]) second. This keeps the dependency ordering intact while eliminating unnecessary sequential stalling.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest scheduler functions):** Neither scheduler function awaits its own previous invocation before the next interval tick. If the DB or job queue is slow, two concurrent executions of the same scheduler can race, potentially double-inserting or double-deleting rows even though onConflictDoNothing provides partial protection for the insert path but not the delete/rollback path.
  → *Guard each scheduler with a module-level boolean flag: let digestRunning = false. At the top of maybeRunWeeklyDigest add 'if (digestRunning) return; digestRunning = true;' and in the finally block set 'digestRunning = false;'. Apply the same pattern to the purge scheduler.*

## 🔵 Info / Tech Debt (3)

- **`server/routes.ts` (duplicate health check routes: /api/health (~line 30) and /healthz (~line 74)):** Two health check endpoints exist with different behaviour: /api/health returns immediately without checking DB connectivity, while /healthz performs an actual DB liveness check. This is confusing for operators and Railway/uptime monitors may be pointed at the wrong one, masking DB outages.
- **`server/jobHandlers.ts` (runExtractPaper – image branch: buffer.toString('base64') on an already-decoded buffer):** The job payload arrives as fileB64 (a base64 string), is decoded to a Buffer via Buffer.from(fileB64, 'base64'), then immediately re-encoded to base64 with buffer.toString('base64') for the vision API call. This is a redundant decode/re-encode cycle that wastes CPU and doubles peak memory for large images.
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function computes 'most recent Sunday' using (dow + 0) % 7, where the '+ 0' is a no-op. Sunday (dow === 0) returns daysSince = 0, so on any Sunday before 7 pm the function returns today at 19:00 which is still in the future and is correctly skipped — but on any non-Sunday day it subtracts dow days, not the days-since-last-Sunday. For Monday (dow=1) it subtracts 1 day landing on Sunday, correct; for Saturday (dow=6) it subtracts 6 days, correct. The bug is the misleading dead '+ 0' and lack of comments, but there is also a real logic gap: the intended constant offset comment suggests a value other than 0 was planned, leaving intent unclear and the code unmaintainable.

## 💡 Opportunities

### Centralise pagination defaults as environment config: 🔥 High Impact
Default limits for leaderboard, transactions, and question queries are hardcoded at various call sites. Extract DEFAULT_PAGE_SIZE and MAX_PAGE_SIZE into environment-backed constants (e.g. process.env.DEFAULT_PAGE_SIZE ?? 25). This prevents accidental full-table scans when callers omit limits and makes tuning a config change rather than a code change.

### Add select field projection to user queries: 🔥 High Impact
Queries that return User objects for student rosters and leaderboards appear to select all columns including any hashed credentials or sensitive profile fields. Switch to explicit column projection (db.select({ id: users.id, name: users.name, avatar: users.avatar })). This eliminates accidental exposure of sensitive fields in API responses and reduces row transfer size.

### Extract simulateMarketPrices interval into a managed singleton: ⚡ Medium Impact
If simulateMarketPrices is called on a schedule via setInterval anywhere in the startup path, the interval handle is likely not stored and cleared on process shutdown or test teardown, creating a memory/handle leak. Expose a startMarketSimulation / stopMarketSimulation pair that stores the handle in module scope and clears it on SIGTERM, preventing leaked timers across test runs and graceful shutdown.

### Canonical base URL via environment variable: 🔥 High Impact
Replace all req.headers host-sniffing logic for base URL construction with a single CANONICAL_URL env var (e.g. https://finsightlite.com). This eliminates the host-injection surface, simplifies the code, and ensures consistent URLs in all generated XML/text even behind multiple proxies. Impact: removes a security class entirely and reduces code by ~10 lines.

### Structured startup logging for Supabase init: ⚡ Medium Impact
The current .catch only logs on total failure. Adding per-step timing logs (console.info('[Supabase] seedStaticModules completed in Xms')) would let the team identify which seed is slow during cold starts on Railway, directly reducing debugging time when deploys are sluggish.

### Domain router registration parallelisation: ⚡ Medium Impact
The six domain routers are imported in parallel via Promise.all (good) but then registered sequentially with six awaited calls. If any registerXRoutes function does async work (DB calls, further imports), they block each other. Audit each for async work and run independent ones concurrently with Promise.all to cut cold-start latency.

### Idempotency token on question insertion: 🔥 High Impact
The current idempotency strategy deletes all questions for a paperId before each attempt. If the job crashes mid-insert on retry, partial data is wiped repeatedly and any downstream foreign-key references break. Add a unique constraint on (paperId, order) and use INSERT ... ON CONFLICT DO NOTHING so retries are truly safe without destructive deletes.

### Structured error logging before re-throw in extract-paper: ⚡ Medium Impact
The outer catch in the extract-paper handler only calls storage.updateExamPaper on the final attempt and then re-throws with no log. Adding a structured log (paperId, attempt number, error message, stack) before re-throwing would dramatically reduce mean-time-to-diagnosis for extraction failures in production without any performance cost.

### Stream PDF parse instead of slicing to 30,000 chars: ⚡ Medium Impact
extractedText is hard-capped at 30,000 characters with a silent slice, meaning the bottom of long exam papers is silently dropped. Replacing the slice with a configurable env var (EXTRACT_TEXT_LIMIT) and logging a warning when the limit is hit would make truncation visible and tunable without a code change.

### Centralise default currency as a schema-level enum: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 separate column definitions (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Define a pgEnum('currency', ['BSD', 'USD', 'TTD']) once and reference it everywhere. This enforces valid values at the DB layer, eliminates the risk of a typo default, and makes adding a new supported currency a one-line change rather than a schema-wide find-and-replace.

### Add composite index on portfolioHoldings (userId, stockId): 🔥 High Impact
portfolioHoldings will be queried as 'get all holdings for this user' and 'check if this user already holds this stock before a buy order'. A composite uniqueIndex on (userId, stockId) satisfies both: it enforces one holding row per user per stock (preventing duplicate accumulation bugs) and serves as the lookup index. Without it, every trade execution does a sequential scan of holdings.

### Extract learnin module relation definitions into a separate file: ⚡ Medium Impact
The file is already truncated at 28,377+ characters. Keeping all table definitions, relations, and Zod insert schemas in a single file creates merge conflicts and makes it impossible to tree-shake. Split into schema/tables/, schema/relations/, and schema/validation/ subdirectories. This has no runtime performance impact but will materially reduce build times and onboarding friction for a student-facing project.

### Structured JSON logging: 🔥 High Impact
Replace console.log string concatenation in the log() helper and all middleware with a structured logger (pino is zero-dependency and <1ms overhead). Each log entry becomes a JSON object with level, timestamp, source, requestId, and an explicit safe-fields payload. This enables log aggregation, alerting on error rates, and guaranteed PII field exclusion via a serializer — critical before any regulatory review for a school-aged user product.

### Centralised graceful shutdown handler: 🔥 High Impact
Extract a single async shutdown(signal) function that sequentially closes: clearInterval for all schedulers, httpServer.close(), job worker drain, then db pool end — each with a per-step timeout and error catch. Register it for SIGTERM, SIGINT, and uncaughtException. Currently there is no shutdown path at all, so Kubernetes or Replit rolling restarts will SIGKILL the process leaving in-flight jobs and DB transactions abandoned.

### Environment config validation at boot: ⚡ Medium Impact
Add a single zod or plain-object schema that validates all required env vars (SESSION_SECRET, DATABASE_URL, AI_PURGE_OLDER_THAN_DAYS, SENDGRID_API_KEY, etc.) before any other module loads. Currently only SESSION_SECRET is checked; a missing DATABASE_URL surfaces only as a cryptic DB connection error mid-request. A startup schema check produces a clear error listing every missing var, reducing misconfiguration incidents in staging and production.

## Per-File Summaries

### Database Layer (`server/storage.ts`)
storage.ts shows systemic N+1 query patterns in budget and bill reminder lookups, missing database transactions around multi-step writes, and sequential awaits that should be parallelized. For a school-aged financial literacy app handling student data, the missing pagination guards on leaderboard and question queries are a reliability risk under load.

### API Routes (`server/routes.ts`)
The routes file is reasonably well structured with good practices like parallel module imports and rate limiting on auth endpoints. Key concerns are unprotected AI/chat/image/audio routes with no rate limiting, a sequential Supabase init chain that blocks each step unnecessarily, and host header injection risk in robots.txt and sitemap generation that could be exploited to produce malicious URLs.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential DB inserts that should be batched, missing try/catch on the AI usage purge handler, and a hardcoded token limit that should be environment-configurable. The extract-paper handler is the most critical path and has multiple addressable gaps.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes already in place, but has critical gaps: missing indexes on high-cardinality query columns, under-constrained varchar widths on user_id columns, and nullable createdAt timestamps that should be NOT NULL. These issues will cause silent data anomalies and query performance degradation at scale.

### Server Startup (`server/index.ts`)
The server startup sequence has several reliability and security concerns: blocking synchronous-style sequential awaits during startup inflate cold-start time, scheduler intervals are never cleared on shutdown causing the process to hang or leak, and full JSON response bodies are logged on every API call which risks exposing sensitive financial or PII data in plaintext logs. Core architecture is reasonable but graceful shutdown and log hygiene need immediate attention.
