## 🔴 Critical (8)

### `server/routes.ts`: registerChatRoutes / registerImageRoutes / registerAudioRoutes (~line 110)
**Issue:** AI and image/audio routes are registered with no rate limiting middleware. For a school-aged user base these endpoints call OpenAI which is expensive and abuse-prone. Any unauthenticated or authenticated user can hammer them freely.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); before the three register calls.

### `server/routes.ts`: robots.txt handler (~line 44) and sitemap handler (~line 56)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. An attacker can send a crafted Host header to inject an arbitrary domain into the Sitemap directive and loc URLs, enabling cache poisoning or phishing via poisoned sitemaps.

**Fix:** Read the canonical origin from a single env variable: const base = process.env.APP_BASE_URL ?? 'https://finsightlite.com'; and remove all header-derived base URL logic from both handlers. Never trust Host/x-forwarded-host for URL construction in generated content.

### `server/jobHandlers.ts`: studentWeekStats
**Issue:** Five sequential awaited DB queries per student inside studentWeekStats, called once per contact in the weekly-digest fan-out loop. For 500 students this is 2,500 serial round-trips before a single email is enqueued.

**Fix:** Replace the five sequential awaits inside studentWeekStats with a single Promise.all: const [xp, [txRow], [gameRow], lessonRows, badgeRows] = await Promise.all([storage.getUserXp(userId).catch(()=>null), db.select(...).from(transactions)..., db.select(...).from(gameSessions)..., db.select(...).from(userLearningProgress)..., db.select(...).from(userBadges)...]). Additionally, batch the outer student loop using chunked Promise.all (e.g. p-limit with concurrency 10) instead of sequential for-of.

---

### `server/jobHandlers.ts`: bootstrapStudentContacts
**Issue:** Inside bootstrapStudentContacts the fallback branch calls db.select().from(users) with no org filter, loading every user row in the database into memory, then iterates them with an awaited db.select per student to check emailContacts. This is an unbounded full-table scan plus N+1 queries and will OOM or time-out in production.

**Fix:** Add a mandatory orgId guard so the unfiltered users query is never reachable in production (throw if orgId is null). Replace the per-student emailContacts existence check with a single bulk query: fetch all existing (userKind='student', userId IN [...studentIds]) rows before the loop, build a Set of known userIds, and skip inserts for matches. Use batched INSERT ... ON CONFLICT DO NOTHING instead of individual inserts.

### `shared/schema.ts`: simulatedStocks table definition
**Issue:** ticker column has no unique index. Duplicate tickers will silently accumulate, causing incorrect portfolio calculations when joining on ticker symbol.

**Fix:** Add (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }) to the table definition. Also add an index on riskLevel if the UI filters by it.

---

### `shared/schema.ts`: portfolioHoldings table definition
**Issue:** No unique constraint on (userId, stockId). Multiple rows per user per stock will accumulate on repeated buys instead of updating quantity, making portfolio value calculations incorrect without extra deduplication logic.

**Fix:** Add (t) => ({ userStockUniqueIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }) and handle upsert logic (INSERT ... ON CONFLICT DO UPDATE) in the data layer.

---

### `server/index.ts`: res.json monkey-patch in request logger middleware
**Issue:** Every JSON response body is captured and serialized into the log line unconditionally. For a financial literacy app with school-aged users this risks logging tokens, session data, grades, or PII (e.g. user profile responses) to stdout/Sentry breadcrumbs with no allow-list or size guard.

**Fix:** Replace the blanket capture with an explicit allow-list of safe fields or drop body logging entirely for non-2xx responses. At minimum add: `if (capturedJsonResponse && JSON.stringify(capturedJsonResponse).length > 512) capturedJsonResponse = { _truncated: true };` and strip any keys matching /token|secret|password|email|dob/i before logging.

---

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both scheduler intervals are assigned to variables but are never passed to clearInterval() in any shutdown handler. If the process receives SIGTERM the intervals fire one last tick after the DB connection is already closing, causing unhandled rejections and potential duplicate job inserts.

**Fix:** Store both interval handles and clear them first in the SIGTERM/SIGINT handler before draining the HTTP server: `const digestTimer = setInterval(...); const purgeTimer = setInterval(...);` then in shutdown: `clearInterval(digestTimer); clearInterval(purgeTimer);` before `httpServer.close()`.

## 🟡 Warnings (11)

- **`server/routes.ts` (Supabase init chain (~line 116)):** The seven Supabase seed functions are chained sequentially with .then(), so a single failure in seedFinancialAcademyLesson() skips initStaticContentTables(), seedStaticModules(), seedRegionalContent(), and seedGameContent() silently. Only one catch at the end logs the error with no indication of which step failed.
  → *Run independent seeds in parallel and wrap each in its own try/catch: await Promise.all([seedFinancialAcademyLesson(), seedRegionalContent(), seedGameContent()].map(fn => fn().catch(e => console.error('[Supabase] seed error:', e))))); Keep initSupabaseTables → trimOrgNamesInSupabase → initStaticContentTables sequential only if they have true ordering dependencies, and give each its own labelled catch.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 101)):** filePath comes directly from the URL and is passed to objectStorage.searchPublicObject without any sanitisation. A path like ../../private/keys could traverse into unintended storage prefixes depending on the ObjectStorageService implementation.
  → *Sanitise before use: const safePath = path.normalize(req.params.filePath).replace(/^\.\.\//, ''); and reject any path that contains '..' after normalisation with a 400 response. Confirm ObjectStorageService also validates on its side.*
- **`server/jobHandlers.ts` (admin-csv-export handler):** The entire export dataset is fetched into a single in-memory array, serialised to a UTF-8 string, and then buffered again as Buffer.from(csv) before upload. There is no row-count or byte-size guard. A large school exporting all students could easily exhaust the Node.js heap.
  → *Add an early size guard: if data.length > MAX_EXPORT_ROWS (env var, default 50000) return an error result before building the CSV. For larger datasets use a streaming approach: pipe a Readable from a cursor-based DB query through a CSV transform stream directly to the object storage upload stream using objectStorageClient.bucket(bucket).file(object).createWriteStream(), avoiding full in-memory buffering.*
- **`server/jobHandlers.ts` (purge-ai-usage and admin-csv-export handlers):** Neither the purge-ai-usage nor admin-csv-export handlers have a try/catch. An unhandled rejection inside a registerJobHandler callback will propagate to the job runner; if the runner does not catch it the job will silently stall or crash the worker without updating job status to failed, making debugging very difficult.
  → *Wrap each handler body in try/catch and return a structured error result: try { ... } catch (err) { return { ok: false, error: err instanceof Error ? err.message : String(err) }; }. Ensure the job runner marks the job as failed when a handler returns ok:false or throws, and logs the jobId alongside the error — never log the full payload as it may contain PII.*
- **`server/jobHandlers.ts` (bootstrapStudentContacts / studentWeekStats):** Student email addresses and usernames containing '@' are passed through string concatenation into log-visible job payloads and return values with no redaction. For a Caribbean school-aged user base this is a COPPA/data-protection concern: PII must not appear in job result rows stored in the jobs table.
  → *Never include raw email addresses in job return values or log statements. Store only aggregate counts and the objectPath in job results. If per-user status is needed, write it to a separate audit table with appropriate access controls, not into the jobs table which is likely readable by any admin querying job history.*
- **`shared/schema.ts` (budgets table definition):** No index on budgets.userId even though every budget query will filter by user. With many users this becomes a full table scan on every dashboard load.
  → *Add an index in the table's second argument: (t) => ({ userIdx: index('idx_budgets_user_id').on(t.userId) }). Also consider a unique index on (userId, categoryId, period) to prevent duplicate budget rows per period.*
- **`shared/schema.ts` (categories table — userId column):** userId in categories is an unlengthed varchar with no index, yet every category lookup filters on userId OR userId IS NULL. Missing index means O(n) scans; missing length constraint on varchar risks inconsistent key sizes vs users.id.
  → *Mirror the exact type from users.id (e.g. varchar('user_id', { length: 255 })). Add index('idx_categories_user_id').on(t.userId) so the 'get all categories for user + system defaults' query stays fast.*
- **`shared/schema.ts` (documentUploads and transactions — createdAt columns):** createdAt columns across all tables use .defaultNow() but are not marked .notNull(). A row inserted with an explicit NULL createdAt will break any ORDER BY createdAt DESC query with silent bad ordering.
  → *Change every createdAt: timestamp('created_at').defaultNow() to timestamp('created_at').defaultNow().notNull(). Same applies to purchasedAt in portfolioHoldings and executedAt in portfolioTransactions.*
- **`server/index.ts` (startup await chain: seedDatabase, probeDatabase, registerRoutes, startJobWorker):** seedDatabase and probeDatabase are awaited sequentially before the HTTP server binds, blocking startup by the sum of both round-trips. If either throws without a wrapping try/catch the unhandled rejection crashes the process with no structured log.
  → *Wrap the entire startup block in try/catch that calls `captureError` then `process.exit(1)`. Run independent probes in parallel: `await Promise.all([probeDatabase(), seedDatabase()])`. Only operations with true ordering dependencies (registerRoutes depends on DB being ready) should remain sequential.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The calculation uses `new Date()` (local server timezone) without pinning to a specific timezone. On a Replit dyno the system timezone may be UTC, making '7pm Sunday' fire at 19:00 UTC rather than 19:00 Caribbean Standard Time (UTC-4), silently scheduling digests 4 hours early for all users.
  → *Replace the raw `setHours(19,0,0,0)` with an explicit UTC-4 offset calculation, or use `Intl.DateTimeFormat` with `timeZone: process.env.DIGEST_TIMEZONE ?? 'America/Port_of_Spain'` to resolve Sunday 19:00 in the correct zone before converting to a UTC timestamp for comparison.*
- **`server/index.ts` (maybeRunAiUsagePurge / AUTO_PURGE_OLDER_THAN_DAYS interval):** The purge scheduler fires on a fixed interval (visible in truncated code) and deletes rows older than AUTO_PURGE_OLDER_THAN_DAYS with no rate-limiting or row-count cap. A large aiUsage table will cause a long-running DELETE that locks rows, blocks the job worker queries on the same DB connection pool, and may time out.
  → *Add a LIMIT to the DELETE (using a subquery on primary key) and loop until affected rows < batch size: `DELETE FROM ai_usage WHERE id IN (SELECT id FROM ai_usage WHERE created_at < $cutoff LIMIT 500)`. Run in a job worker task rather than inline in the scheduler tick so it can be monitored and retried independently.*

## 🔵 Info / Tech Debt (1)

- **`server/routes.ts` (/api/health and /healthz (~lines 33 and 71)):** Two separate health-check endpoints exist (/api/health and /healthz) with different behaviour: /api/health returns immediately with no DB check while /healthz actually probes the DB. ## 💡 Opportunities

### Centralise APP_BASE_URL as single source of truth: 🔥 High Impact
Beyond fixing the host-spoofing issue, using one APP_BASE_URL env var for robots.txt, sitemap, Sentry DSN references, and any email links eliminates the risk of environment mismatch across handlers. Add a startup assertion that fails fast if the variable is missing in production.

### Lazy-load ObjectStorageService to cut cold-start time: ⚡ Medium Impact
new ObjectStorageService() is instantiated at module load time at the top of the file before any request arrives. If the service constructor opens connections or reads config, this adds to cold-start latency on Railway. Move instantiation inside registerRoutes() or use a singleton getter so it initialises on first use.

### Add request-scoped logging with correlation IDs: ⚡ Medium Impact
console.error is used bare in the public-objects handler and Supabase init catch. Adding a middleware that stamps each request with a UUID (stored in AsyncLocalStorage) and threading it into all log calls would make Railway log tailing dramatically faster for debugging student-facing errors without exposing sensitive data.

### Idempotency key on weekly-digest enqueue: 🔥 High Impact
weekly-digest enqueues one send-email job per contact with no idempotency guard. A retry or double-trigger will send duplicate emails to every student. Add a deterministic idempotency key (e.g. SHA256 of userId+weekStart) when calling enqueueJob for send-email so the jobs table unique constraint deduplicates retries automatically.

### Extract hardcoded time constants to environment config: ⚡ Medium Impact
Values like 7 * 24 * 60 * 60 * 1000 for the digest window and the olderThanDays computation appear in multiple places without validation. Centralise these as validated env vars (DIGEST_WINDOW_DAYS, AI_USAGE_RETENTION_DAYS) with zod/env-schema parsing at startup so misconfiguration fails fast rather than silently purging wrong date ranges.

### Stream CSV upload to object storage instead of double-buffering: ⚡ Medium Impact
Replacing Buffer.from(csv) + .save() with a pipeline from a Transform stream to objectStorageClient createWriteStream would reduce peak memory for exports from O(2×payload) to O(chunk-size), directly lowering the risk of OOM on the shared Replit worker during large school exports.

### Centralise currency default as a shared constant: ⚡ Medium Impact
The string 'BSD' is hardcoded as a default in at least 6 tables (transactions, savingsGoals, billReminders, simulatedStocks, portfolioTransactions, linkedCards). Extract const DEFAULT_CURRENCY = process.env.DEFAULT_CURRENCY ?? 'BSD' in a config file and reference it in each .default() call. This makes multi-currency support a one-line change and prevents drift if the app expands beyond the Bahamas.

### Add partial index for unread/active bill reminders: ⚡ Medium Impact
Bill reminder queries almost always filter on nextDueDate >= NOW(). A partial index — index('idx_bill_reminders_upcoming').on(t.userId, t.nextDueDate) — will significantly cut scan size compared to a full table index, especially as reminder rows accumulate over years per user.

### Replace text type with varchar length caps on free-text identity columns: ⚡ Medium Impact
Columns like bankName, fileName, name, and ticker use unbounded text. For a school-aged user app, unbounded input stored without length validation is both a storage risk and an injection surface if values ever reach logs or PDFs. Cap these at sensible lengths (e.g. varchar(100) for bankName, varchar(10) for ticker) and pair with Zod max() in the insert schemas to enforce the limit at the API layer too.

### Structured startup health-check endpoint: 🔥 High Impact
Expose GET /healthz that returns DB probe result and job worker status as JSON before the server accepts real traffic. Replit and any future load balancer can use this for readiness checks, preventing requests from hitting a half-initialised app during cold starts.

### Parallel scheduler claim inserts for weekly digest audiences: ⚡ Medium Impact
The three audience claims (student, teacher, guardian) are inserted sequentially inside a for-of loop. Replacing with Promise.all over the three insert+enqueue operations cuts digest scheduling latency by ~2x and removes a 3-serial-round-trip stall on the interval tick.

### Centralise hardcoded interval durations as named env config: 📌 Low Impact
Interval periods for the digest scheduler and purge runner appear as numeric literals in the truncated code. Extracting them to DIGEST_INTERVAL_MS and PURGE_INTERVAL_MS env vars with validated defaults allows tuning without a redeploy and makes the scheduler contract explicit in the README.

### API Routes (`server/routes.ts`)
The route file is generally well-structured with good patterns like parallel dynamic imports and rate limiting on auth endpoints. Graceful shutdown handling is incomplete, leaving the job worker and HTTP server potentially torn down in the wrong order.
