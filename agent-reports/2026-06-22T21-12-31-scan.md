## 🔴 Critical (7)

### `server/routes.ts`: registerChatRoutes, registerImageRoutes, registerAudioRoutes (~line 113-115)
**Issue:** No rate limiting is applied to the AI/image/audio routes before they are registered. These endpoints call OpenAI APIs and are the most expensive in the app. Any authenticated or unauthenticated client can hammer them, causing runaway API costs and service degradation for school-aged users on shared sessions.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter); then call the register functions. Confirm the path prefixes match what those helpers actually mount.

### `server/routes.ts`: robots.txt handler (~line 46) and sitemap.xml handler (~line 62)
**Issue:** The base URL is constructed directly from req.headers['x-forwarded-host'] and req.headers.host without any validation or allowlist. A crafted Host header will produce an arbitrary base URL in the sitemap and robots.txt, enabling host-header injection attacks (cache poisoning, phishing links in indexed sitemap).

**Fix:** Read the canonical origin from a single environment variable: const base = process.env.APP_ORIGIN ?? 'https://finsightlite.com'; Remove all req.headers host/proto logic from both handlers. Set APP_ORIGIN in Railway/Replit secrets.

### `server/jobHandlers.ts`: runExtractPaper (fileB64 parameter)
**Issue:** Full file contents stored as base64 in the job payload. For a PDF or image, this can be several megabytes per row in the jobs table, causing DB bloat, slow job queue polling, and potential OOM when many jobs are loaded into the worker process simultaneously.

**Fix:** Upload the file to object storage before enqueuing, then store only the object path in the job payload. In runExtractPaper, download the file from object storage using a stream at job execution time. Remove fileB64 from JobPayloads['extract-paper'] and replace with objectPath: string.

### `shared/schema.ts`: budgets table definition
**Issue:** budgets.userId and budgets.categoryId have no composite or individual index. Any query fetching budgets for a user (the most common read pattern) will do a full table scan as the dataset grows.

**Fix:** Add an index in the budgets table's index callback: (t) => ({ userCategoryIdx: index('idx_budgets_user_category').on(t.userId, t.categoryId) }). This covers both the common 'get all budgets for user' and 'get budget for user+category' queries.

---

### `shared/schema.ts`: simulatedStocks table — ticker column
**Issue:** ticker has no unique index. Duplicate tickers can be inserted silently, which will corrupt portfolio valuation calculations that join on ticker symbol and produce doubled or ambiguous results.

**Fix:** Replace the plain column declaration with a uniqueIndex in the table callback: (t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker_unique').on(t.ticker) }). Also add .notNull() to issuer and region if they are required for Caribbean regional filtering.

### `server/index.ts`: setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are never stored in a variable, so they cannot be cleared during graceful shutdown. On SIGTERM the process exits while intervals may be mid-flight, risking a partial DB write (e.g., inserting a weeklyDigestRuns claim row without enqueuing the job), and in test environments the open handles prevent the process from exiting cleanly.

**Fix:** Store handles: `const weeklyDigestTimer = setInterval(...)`. In the SIGTERM/SIGINT handler call `clearInterval(weeklyDigestTimer)` and `clearInterval(purgeTimer)` before closing the HTTP server and DB pool. Also await any in-flight scheduler Promise before exiting.

---

### `server/index.ts`: res.on('finish') logging middleware / capturedJsonResponse
**Issue:** The middleware monkey-patches res.json() and stores the entire response body in capturedJsonResponse, then serializes it again with JSON.stringify() into the log line for every API response. For endpoints returning large arrays (e.g., transaction lists, quiz results), this doubles peak memory for that response and writes potentially sensitive user data—balances, scores, session tokens—to stdout/log aggregators in plaintext.

**Fix:** Remove capturedJsonResponse capture entirely, or limit logging to status code and duration only. If response body logging is needed for debugging, truncate to a fixed byte limit (e.g., 200 chars) and explicitly redact keys like 'password', 'token', 'balance' before logging. Never log full response bodies in production.

## 🟡 Warnings (11)

- **`server/routes.ts` (Supabase init chain (~line 118-126)):** The sequential .then() chain runs seven async Supabase operations one after another. If seedFinancialAcademyLesson() fails, initStaticContentTables(), seedStaticModules(), seedRegionalContent(), and seedGameContent() are all silently skipped with no per-step error logged. The single .catch() at the end only reports the first failure, leaving the app in a partially seeded state with no visibility into which step failed.
  → *Run independent seed steps in parallel where safe and wrap each in its own try/catch: await initSupabaseTables(); await trimOrgNamesInSupabase(); await Promise.all([ seedFinancialAcademyLesson().catch(e => console.error('[Supabase] seedFinancialAcademyLesson failed:', e)), initStaticContentTables().then(() => Promise.all([seedStaticModules(), seedRegionalContent(), seedGameContent()])).catch(e => console.error('[Supabase] static content seed failed:', e)) ]); Keep non-blocking by not awaiting the outer call in registerRoutes.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 97-105)):** filePath is taken directly from req.params.filePath and passed to objectStorage.searchPublicObject without path traversal sanitisation. A request to /public-objects/../../private/secrets could resolve outside the intended public bucket prefix depending on the ObjectStorageService implementation.
  → *Sanitise before use: const filePath = req.params.filePath.replace(/\.\.\/|\.\.\\/g, ''); and additionally validate it matches a safe pattern: if (!/^[\w\-./]+$/.test(filePath)) return res.status(400).json({ error: 'Invalid path' });*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with an await on each iteration. For a 60-question exam paper this means 60 round-trips to the database, multiplying latency linearly and holding the job open far longer than necessary.
  → *Replace the for loop with a single bulk insert: collect all valid question objects into an array, then call db.insert(extractedQuestions).values(questionRows) once. If storage.createExtractedQuestion must be used, use Promise.all with a concurrency cap (e.g. p-limit at 5) rather than sequential awaits.*
- **`server/jobHandlers.ts` (runExtractPaper — image branch, buffer.toString('base64')):** The original fileB64 payload is decoded to a Buffer and then immediately re-encoded back to base64 to build the OpenAI vision URL. This wastes CPU and memory proportional to file size, and is redundant since fileB64 is already available in scope.
  → *Use the original fileB64 string directly in the image_url data URI: `data:${mimeType};base64,${fileB64}`. Remove the buffer re-encoding. If the fix from finding #1 is applied and buffer comes from object storage, encode once and cache in a local const.*
- **`server/jobHandlers.ts` (registerJobHandler('admin-csv-export') and all handlers — missing top-level try/catch):** The 'purge-ai-usage' and 'perf-scan' handlers have no try/catch. If the DB call or dynamic import throws an unhandled rejection, the job framework may not record a clean failure state, leaving the job stuck in 'processing' with no status update — the same UX problem the extract-paper handler explicitly guards against.
  → *Wrap each handler body in a try/catch that catches the error, updates any relevant status fields (e.g. export record to 'failed'), then re-throws so the job runner still marks the job as failed and can retry. Alternatively, add a global onJobError hook in the job runner that handles status sync for all job types.*
- **`shared/schema.ts` (All tables — createdAt, purchasedAt, executedAt columns):** createdAt (and analogous timestamp columns like purchasedAt, executedAt) use .defaultNow() but no .notNull(). Drizzle will accept explicit null inserts, leaving audit trail rows with no timestamp — a compliance risk for a financial app serving minors.
  → *Chain .notNull() after .defaultNow() on every audit timestamp: e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Apply to all tables: transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions.*
- **`shared/schema.ts` (billReminders table — nextDueDate column):** billReminders has no index on (userId, nextDueDate). The primary use case — fetching upcoming bills for a user sorted by due date — will scan the entire table. For school-aged users with many reminders this degrades quickly.
  → *Add index callback to billReminders: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due_date').on(t.userId, t.nextDueDate) }). This directly supports the WHERE userId = ? ORDER BY nextDueDate ASC query pattern.*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is nullable (system categories have null) but has no index. Any query filtering user-owned categories — e.g. WHERE userId = ? OR userId IS NULL — scans the full table. As the system category list grows this compounds.
  → *Add a partial or regular index: (t) => ({ userIdIdx: index('idx_categories_user_id').on(t.userId) }). If your Postgres version supports it, a partial index WHERE user_id IS NOT NULL via raw SQL migration is more efficient and avoids indexing system-category null rows.*
- **`server/index.ts` (seedDatabase / probeDatabase / startJobWorker calls in startup IIFE):** Startup awaits seedDatabase(), probeDatabase(), and startJobWorker() sequentially. If seedDatabase() takes several seconds (e.g., first deploy with data), the HTTP server is not yet listening and health checks fail. probeDatabase() could run in parallel with seedDatabase() since neither depends on the other's result.
  → *Lift the HTTP server listen() call before the async init work, or at minimum run independent operations in parallel: `await Promise.all([probeDatabase(), seedDatabase()])` then startJobWorker(). Guard each with its own try/catch so one failure does not abort the others.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The day-of-week offset calculation `(dow + 0) % 7` is a no-op (adding 0 changes nothing). Sunday is dow=0, so daysSince=0, meaning the function never subtracts days when today is Sunday and the 7pm window has not yet passed—it returns today at 19:00. The guard `if (d.getTime() > now.getTime()) d.setDate(d.getDate() - 7)` partially compensates, but the comment and variable name `daysSince` imply intentional arithmetic that is silently broken. On a Sunday before 7pm the function returns last Sunday, which is correct, but the dead `+0` suggests a missing constant (e.g., intended to be `(dow + 1) % 7` for a Monday anchor).
  → *Clarify intent: if the anchor is Sunday, replace `(dow + 0) % 7` with simply `dow`. Add a unit test covering Sunday-before-7pm, Sunday-after-7pm, and mid-week cases to lock in correct behavior.*
- **`server/index.ts` (AUTO_PURGE_OLDER_THAN_DAYS IIFE / hardcoded 180):** The purge day threshold falls back to a hardcoded 180 silently with no log warning when the env var is missing or invalid. For a school-aged-user app subject to Caribbean data-protection obligations (e.g., Trinidad & Tobago's Data Protection Act), operators must know what retention window is active. A silent fallback can mask misconfiguration in production.
  → *After the IIFE, add: `log(`[startup] AI usage purge retention: ${AUTO_PURGE_OLDER_THAN_DAYS} days (source: ${process.env.AI_PURGE_OLDER_THAN_DAYS ? 'env' : 'default'})`, 'config');` so every deploy emits the active value. Consider making an invalid/missing env var a hard error rather than a silent fallback given compliance stakes.*

## 💡 Opportunities

### Parallel domain-router registration: ⚡ Medium Impact
The six registerX(app) calls after the Promise.all import are awaited sequentially. If any of those registration functions do async work (DB schema checks, config loads), they could be parallelised with Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...]). This would reduce cold-start latency on Railway where every deploy restarts the process.

### Centralised APP_ORIGIN environment config: ⚡ Medium Impact
Beyond fixing the injection issue, moving all absolute URL construction to a single validated APP_ORIGIN constant (read once at module load, thrown if missing in production) eliminates the scattered proto/host header logic across robots.txt, sitemap, and any future handlers. This also makes local development predictable without needing forwarded headers.

### Object storage response caching headers: 🔥 High Impact
The /public-objects route passes a 30-day TTL to downloadObject but there is no Cache-Control header set on the Express response itself before streaming. Adding res.setHeader('Cache-Control', 'public, max-age=2592000, immutable') before the download call lets CDN/Railway edge and browsers cache logos and videos, reducing origin hits and latency for school-aged users on mobile connections.

### Idempotent bulk upsert for extracted questions: 🔥 High Impact
The current delete-then-reinsert idempotency strategy creates a window where a crash leaves zero questions in the table. Switching to an INSERT ... ON CONFLICT DO UPDATE (upsert) keyed on (paperId, order) eliminates that window, makes retries safe without deletion, and allows a single bulk statement — combining the reliability fix with the performance fix for sequential inserts.

### Stream-based PDF text extraction to cap memory use: ⚡ Medium Impact
pdf-parse loads the entire buffer into memory synchronously. For large exam papers this can spike RSS significantly inside the worker. Using a streaming PDF parser (e.g. pdf2json with stream input) and capping extracted text server-side before the OpenAI call would reduce peak memory per job and allow safer concurrent job execution on a single worker process.

### Centralised job error handler for status sync: ⚡ Medium Impact
Each job handler currently duplicates the pattern of catching errors, updating a status field, and re-throwing. Extracting this into a higher-order wrapHandler(type, fn) utility that accepts an optional onFinalFailure callback would eliminate the duplicated try/catch blocks, ensure consistent failure logging across all job types, and make adding new jobs less error-prone.

### Introduce currency enum type: 🔥 High Impact
Currency is stored as free-text with a default of 'BSD' across at least 6 tables. A typo ('bsd', 'BS D') will silently store invalid data and break currency-conversion logic. Define a Postgres enum or a shared Drizzle text enum constant (e.g. currencyEnum = ['BSD', 'USD', 'TTD']) and reuse it on every currency column. This also makes adding new Caribbean currencies a single-point change.

### Add composite unique constraint on portfolioHoldings: 🔥 High Impact
portfolioHoldings has no unique constraint on (userId, stockId). Multiple rows for the same user+stock are possible, forcing application code to aggregate quantities manually and risk double-counting in portfolio valuation. A uniqueIndex on (userId, stockId) enforces one row per position and lets the app do simple UPDATE ... SET quantity = quantity + ? upserts instead of read-then-write logic.

### Extract default currency to a shared constant: ⚡ Medium Impact
The string literal 'BSD' is hardcoded as a column default in at least 6 separate table definitions. If the app needs to support multi-currency schools (e.g. TTD for Trinidad users), every table default must be changed individually. Export a DEFAULT_CURRENCY constant from a config file and reference it in each .default() call to make regional configuration a one-line change.

### Structured JSON logging instead of formatted console.log strings: 🔥 High Impact
The current log() function emits human-readable strings with locale-formatted timestamps. Replacing it with structured JSON lines (e.g., using pino) would make log aggregation (Datadog, CloudWatch) queryable by userId, jobKind, or duration without regex parsing. Expected impact: dramatically faster incident triage and ability to build latency dashboards without code changes.

### Graceful shutdown draining for job worker and DB pool: 🔥 High Impact
There is no evidence of a SIGTERM handler that waits for the job worker to finish its current job before exiting. On container restarts (Replit, Railway), an in-flight AI job can be silently dropped. Adding a shutdown sequence—stopAcceptingJobs(), await currentJobDone(), httpServer.close(), pool.end()—eliminates orphaned AI API calls and duplicate job executions after restart.

### Consolidate scheduler interval into a single tick function: ⚡ Medium Impact
Two separate setInterval calls at different cadences (weekly digest vs. purge) each query the DB independently. Merging them into one schedulerTick() function on the shorter interval, with each sub-task doing its own recency check, reduces DB round-trips, makes the single clearInterval handle easier to manage, and ensures both tasks share the same error boundary and logging context.

### API Routes (`server/routes.ts`)
The route registration file is generally well-structured with good use of Promise.all for parallel imports and rate limiting on auth endpoints. Key concerns are unhandled promise rejections on the Supabase init chain that could silently corrupt startup state, missing rate limiting on AI/heavy endpoints registered via registerChatRoutes/registerImageRoutes/registerAudioRoutes, and a host header injection risk in the robots.txt and sitemap dynamic URL construction.

### Job Handlers (`server/jobHandlers.ts`)
jobHandlers.ts has several reliability and performance issues including large base64 payloads stored in the job table, sequential question inserts that should be batched, and a large binary payload being re-encoded to base64 mid-flight. The code also stores raw file content in job rows which will cause memory and storage problems at scale for a school-aged user base handling exam uploads.

### Database Schema (`shared/schema.ts`)
The schema is reasonably structured with foreign keys and some indexes in place, but has several reliability and performance gaps. Key concerns are missing indexes on high-frequency filter columns, nullable timestamps that should be NOT NULL, and missing NOT NULL on createdAt columns that will cause silent nulls in production data.

### Server Startup (`server/index.ts`)
The server startup sequence blocks on sequential async operations (seed, probe, job worker init) that could be parallelized, and the interval-based schedulers lack cleanup handles stored anywhere accessible for graceful shutdown. The logging middleware unconditionally serializes every API response body to JSON string on every request, which is both a memory and CPU concern, and risks leaking sensitive financial or PII data into logs.
