## 🔴 Critical (8)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI, image, and audio routes are registered with no rate limiting. For a school-aged user base these endpoints likely call OpenAI APIs; without a limiter a single user or bot can exhaust quotas or run up costs in seconds.

**Fix:** Create a dedicated aiLimiter (e.g. 20 req/min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` before the three registerX calls.

### `server/jobHandlers.ts` — runExtractPaper (extract-paper handler)
**Issue:** fileB64 (the full base64-encoded PDF or image) is stored directly in the job payload in the jobs table. A single scanned exam PDF can easily exceed 5-10 MB encoded in base64, bloating the jobs table, causing slow job queue polls, and risking row-size limits depending on the DB column type.

**Fix:** Upload the file to private object storage before enqueuing the job. Store only the object storage path in the payload (e.g., { paperId, objectPath, ext, subject }). In runExtractPaper, download the buffer from object storage at the start of execution. This keeps job rows small and makes the queue table scannable.

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** olderThanDays is taken directly from job.payload without any validation. A malformed or malicious job payload with olderThanDays: 0 or a negative value would compute a cutoff in the future and delete all aiUsageEvents rows, causing irreversible data loss.

**Fix:** Add an explicit guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('Invalid olderThanDays value'); This makes the job fail safely and retry rather than executing a destructive query with bad input.

### `shared/schema.ts` — simulatedStocks table — ticker column
**Issue:** ticker has no unique constraint and no index. Duplicate tickers would silently corrupt portfolio lookups and any ticker-based deduplication logic. It is also used as a natural lookup key but will always require a full scan.

**Fix:** Replace `text('ticker').notNull()` with a unique-indexed column: add `(t) => ({ tickerUniqueIdx: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) })` in the table definition, and ensure the application seed script relies on ON CONFLICT DO NOTHING against ticker.

---

### `shared/schema.ts` — portfolioHoldings table — userId + stockId columns
**Issue:** A user should hold at most one portfolioHoldings row per stock (quantity is aggregated). Without a unique constraint on (userId, stockId), concurrent buy operations can insert duplicate rows, causing double-counting of holdings — a critical correctness bug for any financial simulation.

**Fix:** Add `(t) => ({ userStockUniqueIdx: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) })` to portfolioHoldings. Update the buy logic to use an upsert (INSERT ... ON CONFLICT DO UPDATE SET quantity = quantity + excluded.quantity) instead of a plain insert.

### `server/index.ts` — setInterval calls for maybeRunWeeklyDigest and maybeRunAiUsagePurge
**Issue:** Both setInterval handles are not stored in variables and are never cleared. On SIGTERM/SIGINT the intervals keep the event loop alive and in-flight scheduler ticks can fire after the DB connection is closed, causing unhandled rejections or double-enqueuing jobs during rolling restarts.

**Fix:** Store each handle: `const digestInterval = setInterval(...)`. In the graceful-shutdown handler call `clearInterval(digestInterval)` and `clearInterval(purgeInterval)` before closing the HTTP server and DB pool, ensuring no new ticks fire after teardown begins.

---

### `server/index.ts` — res.json monkey-patch in request logging middleware
**Issue:** Every API response body is JSON-serialised into `capturedJsonResponse` and then concatenated into a log line unconditionally. For routes returning user financial records, quiz scores, or session tokens this will print full payloads to stdout/log aggregators, violating data-minimisation principles for school-aged users.

**Fix:** Gate body logging behind a dev-only flag: `if (process.env.LOG_RESPONSE_BODIES === 'true' && capturedJsonResponse) { logLine += ` :: ${JSON.stringify(capturedJsonResponse)}`; }`. In production keep the flag unset so only method, path, status, and duration are logged.

### `client/src/pages/AdminDashboard.tsx` — apiRequest calls throughout mutations
**Issue:** useMutation onMutate/mutationFn calls using apiRequest have no try/catch. If apiRequest throws a network error or non-2xx response that rejects the promise, TanStack Query will surface it to onError, but any fire-and-forget apiRequest calls outside mutations (e.g. direct awaits in event handlers or useEffect bodies) will produce unhandled promise rejections — crashing silently in production.

**Fix:** Audit every `await apiRequest(...)` call outside of useMutation. Wrap each in try/catch with toast notification on catch: `try { await apiRequest(...) } catch (err) { toast({ title: 'Error', description: (err as Error).message, variant: 'destructive' }); }`. For mutation chains, ensure onError is always defined.

## 🟡 Warnings (15)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...).catch(...)):** The Supabase init chain is a sequential .then() waterfall. If seedFinancialAcademyLesson() throws, initStaticContentTables, seedStaticModules, seedRegionalContent, and seedGameContent are all silently skipped — the single .catch only logs the first failure with no indication of which step failed or that subsequent steps were aborted.
  → *Wrap each step individually or use an async IIFE with try/catch per call so failures are isolated and logged with context: `async function initSupabase() { await initSupabaseTables(); await trimOrgNamesInSupabase(); ... }` with per-step try/catch and a named log prefix like '[Supabase:seedGameContent]'.*
- **`server/routes.ts` (/healthz handler (~line 57)):** `await import('./db')` and `await import('drizzle-orm')` are called on every health-check request. While Node caches modules after first load, the dynamic import machinery still runs on each call. Health checks are typically hit every 10–30 seconds by orchestrators.
  → *Move `import { db } from './db'` and `import { sql } from 'drizzle-orm'` to the top of the file as static imports. This also makes the dependency graph explicit and eliminates any risk of the health check racing module initialisation.*
- **`server/routes.ts` (/public-objects/:filePath(*) handler):** The filePath parameter is used directly in `objectStorage.searchPublicObject(filePath)` with no sanitisation. A path traversal payload like `../../private/secrets` could reach unintended storage keys depending on the ObjectStorageService implementation.
  → *Normalise and validate the path before use: `const safePath = path.normalize(filePath).replace(/^\.\.\//, ''); if (safePath !== filePath) return res.status(400).json({ error: 'Invalid path' });` and import Node's `path` module at the top of the file.*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Questions are inserted one at a time inside a sequential for loop with individual awaits. For a paper with 60 questions this is 60 serial round-trips to the database. If any single insert fails after partial completion, the delete-on-retry idempotency guard does clean up, but throughput is needlessly slow and the window for a mid-loop crash causing a stuck 'processing' status is maximised.
  → *Collect all valid question objects into an array and perform a single bulk insert using db.insert(extractedQuestions).values(allQuestions) after the loop. Wrap the bulk insert and the subsequent updateExamPaper call in a db.transaction() so both succeed or both roll back atomically, eliminating the partial-completion risk entirely.*
- **`server/jobHandlers.ts` (runExtractPaper and all openai.chat.completions.create calls):** The OpenAI model name 'gpt-4o' is hardcoded in two separate completions calls. If the organisation needs to switch models (cost control, compliance, regional availability), every call site must be found and edited manually, and there is no way to override at deploy time without a code change.
  → *Add OPENAI_MODEL to environment config (e.g., process.env.OPENAI_MODEL ?? 'gpt-4o') and reference it from a single config module. Apply the same pattern to max_completion_tokens limits if they need to vary by environment.*
- **`server/jobHandlers.ts` (admin-csv-export handler (visible in truncated section)):** Based on the handler signature and pattern, CSV row data for all users or transactions is likely assembled entirely in memory as a JavaScript array before being written. For a growing school-user base this can spike heap usage significantly during export jobs and cause the worker process to OOM-crash, silently failing the export.
  → *Stream rows from the database using a cursor or paginated query (e.g., chunks of 500 rows) and pipe each chunk through a csv-stringify Transform stream directly into the object storage upload stream. This caps memory usage to one chunk regardless of total row count.*
- **`shared/schema.ts` (budgets table definition):** budgets has no index on userId, yet every budget query will filter by userId. At scale this causes full table scans. Similarly, billReminders and savingsGoals have no index on userId or nextDueDate, and bill_reminders.nextDueDate is a common ORDER BY/WHERE target for due-date alerting.
  → *Add indexes in the table's third argument: `(t) => ({ userIdx: index('idx_budgets_user').on(t.userId) })` for budgets; `(t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) })` for billReminders; `(t) => ({ userIdx: index('idx_savings_goals_user').on(t.userId) })` for savingsGoals.*
- **`shared/schema.ts` (transactions.createdAt, budgets.createdAt, documentUploads.createdAt (and all other createdAt columns)):** All createdAt columns use `.defaultNow()` but omit `.notNull()`. Drizzle will type these as `Date | null`, forcing null-checks everywhere in application code and allowing explicit NULL inserts to corrupt audit trails — unacceptable for financial records.
  → *Chain `.notNull()` after `.defaultNow()` on every createdAt (and executedAt, purchasedAt) column, e.g. `createdAt: timestamp('created_at').defaultNow().notNull()`. Do the same for portfolioTransactions.executedAt and portfolioHoldings.purchasedAt.*
- **`shared/schema.ts` (categories table — userId column):** categories.userId is `varchar('user_id')` with no length cap and no index, yet every category fetch filters on `userId IS NULL OR userId = ?`. Without an index this is a sequential scan on every page load. The unbounded varchar also diverges from the fixed-length user ID format used elsewhere.
  → *Add an index: `(t) => ({ userIdx: index('idx_categories_user').on(t.userId) })`. If user IDs are UUIDs, change `varchar('user_id')` to `varchar('user_id', { length: 36 })` to match the auth model and allow the DB to store them efficiently.*
- **`server/index.ts` (seedDatabase / probeDatabase / startJobWorker calls in startup IIFE):** Startup runs seedDatabase, probeDatabase, registerRoutes, and startJobWorker sequentially with sequential awaits. probeDatabase and seedDatabase are independent and could overlap, adding hundreds of milliseconds to cold-start time before the HTTP server accepts connections.
  → *Use Promise.all for independent startup steps: `await Promise.all([probeDatabase(), seedDatabase()])` then proceed to registerRoutes and startJobWorker. This parallelises I/O-bound DB operations and reduces time-to-listen.*
- **`server/index.ts` (getMostRecentSundayAt7pm):** The function uses `new Date()` and `Date.now()` which resolve to the server's local system timezone. On a UTC-based host (Replit, Railway, Fly.io) '7 PM Sunday' becomes UTC, not the Caribbean local time (UTC-4 or UTC-5), so digests fire 4-5 hours early relative to users.
  → *Accept an explicit timezone string via env var (e.g. `DIGEST_TIMEZONE=America/Port_of_Spain`) and compute the target time using the Intl API or a lightweight library like `date-fns-tz` to get the correct Sunday 19:00 in that zone before converting to UTC for comparison.*
- **`server/index.ts` (maybeRunAiUsagePurge / maybeRunWeeklyDigest scheduler functions):** Neither scheduler guards against concurrent execution. If a tick takes longer than the interval period (e.g. slow DB on the insert/delete claim path), a second tick starts before the first completes, potentially issuing duplicate enqueueJob calls for the same weekStart+audience combination despite the onConflictDoNothing guard, because the rollback delete in the catch block races with the second tick's insert.
  → *Add a module-level boolean lock per scheduler: `let digestRunning = false`. At the top of each async scheduler function: `if (digestRunning) return; digestRunning = true;` and set it back to false in a finally block. This is simpler and safer than relying solely on the DB claim under concurrent ticks.*
- **`client/src/pages/AdminDashboard.tsx` (AuditLogPanel):** URLSearchParams object is reconstructed on every render without memoization. Since it feeds the queryKey indirectly through the query function closure, any parent re-render causes a new params object and potentially stale closure behavior, even when actorType/action/from/to have not changed.
  → *Wrap the params construction in useMemo: `const params = useMemo(() => { const p = new URLSearchParams(); if (actorType !== 'all') p.set('actorType', actorType); if (action) p.set('action', action); if (from) p.set('from', from); if (to) p.set('to', to); p.set('limit', '200'); return p; }, [actorType, action, from, to]);` and use params.toString() in both queryKey and queryFn.*
- **`client/src/pages/AdminDashboard.tsx` (ObservabilityCard):** The useQuery call has no error or loading state handled. If the /api/admin/observability endpoint fails or is slow, the component renders with all undefined data silently — badges show 'Disabled' and the healthz link falls back to '/healthz' with no indication to the admin that data failed to load.
  → *Destructure `isLoading` and `isError` from useQuery. Render a skeleton or spinner when isLoading is true, and an Alert/error badge when isError is true, e.g.: `if (isLoading) return <Skeleton className='h-40' />; if (isError) return <Alert variant='destructive'>Failed to load observability data</Alert>;`*
- **`client/src/pages/AdminDashboard.tsx` (TABS constant and large inline icon/label JSX in sidebar render):** TABS is defined at module scope which is correct, but the tab rendering almost certainly maps TABS and creates new inline arrow functions (onClick handlers) on every render. In a dashboard this large with many state variables, parent re-renders are frequent, causing all tab onClick handlers to be recreated every render cycle.
  → *Wrap the tab click handler in useCallback: `const handleTabClick = useCallback((id: string) => setActiveTab(id), []);` and pass it to each tab item. If tab items are their own component, wrap that component in React.memo to prevent re-renders when the active tab hasn't changed.*

## 💡 Opportunities

### Centralised route manifest with per-route rate limit config — 🔥 High Impact
Define routes, their handlers, and their rate limiters in a single manifest array and loop to register them. This prevents future AI/heavy endpoints from being added without a rate limit, which is the current critical gap — the omission is structural, not accidental.

### Supabase init health surfaced in /healthz — ⚡ Medium Impact
Track the outcome of the Supabase init chain in a module-level boolean and expose it in the /healthz response alongside the DB ping. Currently an orchestrator sees the server as healthy even if all seed data failed to load, masking silent startup failures for a school-aged audience that depends on lesson content.

### Extract sitemap URL list to config or CMS — 📌 Low Impact
The hardcoded `urls` array in the sitemap handler will drift from actual routes as the app grows. Move it to a config file or generate it from the registered Express router stack. This reduces the risk of the sitemap referencing removed or renamed routes, which affects SEO for the app's discoverability.

### Batch question validation before any DB writes — ⚡ Medium Impact
Currently, question validation (checking q.question, q.options.length, q.answer) happens inside the insert loop. Moving validation into a separate map/filter pass before any DB interaction produces a clean validated array, makes the logic unit-testable in isolation, and means the bulk insert receives only well-formed rows with no conditional branching mid-transaction.

### Centralise OpenAI client and model config in one module — 🔥 High Impact
openai is imported from a routes file (replit_integrations/chat/routes), coupling the job handler to a HTTP-layer module. Extracting the OpenAI client and its configuration (model, token limits, retry policy) into a dedicated lib/openai.ts reduces coupling, makes mocking in tests trivial, and provides one place to add rate-limit handling or cost logging for all AI calls across the app.

### Add structured job-level telemetry for AI cost tracking — ⚡ Medium Impact
Each openai.chat.completions.create response includes usage.prompt_tokens and usage.completion_tokens. Recording these in aiUsageEvents immediately after each call (inside the job handler, not just in HTTP routes) would give complete visibility into AI spend from background jobs, which is currently a blind spot for cost monitoring in a school-budget-constrained app.

### Centralise currency default via environment config — ⚡ Medium Impact
The string 'BSD' is hardcoded as the currency default in at least six tables (transactions, savingsGoals, billReminders, portfolioTransactions, simulatedStocks, linkedCards implicitly). Extract a DEFAULT_CURRENCY constant from process.env.DEFAULT_CURRENCY in a shared config module and reference it in all .default() calls. This lets the app support other Caribbean currencies (TTD, JMD) without schema migrations.

### Add a composite index on portfolioTransactions (userId, stockId, executedAt) — 🔥 High Impact
Portfolio P&L calculations join portfolioTransactions on userId and stockId then order by executedAt. A three-column composite index would eliminate the sort step and speed up the most read-heavy dashboard query. Expected impact: sub-10ms portfolio load vs current full-scan + sort for users with >50 transactions.

### Introduce a shared currencyEnum drizzle type — ⚡ Medium Impact
Currency is repeated as a free-text column across six tables with only a runtime default for safety. Define a pgEnum('currency', ['BSD','TTD','JMD','USD']) once and reuse it. This enforces valid values at the DB layer, eliminates an entire class of data-entry bugs, and makes future currency additions a single-file change.

### Structured JSON logging instead of interpolated strings — ⚡ Medium Impact
Replace the `log()` helper's console.log with a structured logger (pino is already common in this stack). Emit JSON objects with `level`, `source`, `durationMs`, and `path` as discrete fields. This makes log aggregation in Datadog/Logtail queryable without regex parsing and eliminates the risk of accidentally embedding object toString() noise.

### Formalise graceful shutdown into a single coordinated handler — 🔥 High Impact
Consolidate SIGTERM and SIGINT handling into one `shutdown()` async function that sequentially: stops accepting new connections (httpServer.close), clears all intervals, drains the job worker, then closes the DB pool. A single 10-second timeout should force-exit if any step hangs. This prevents the current risk of the DB pool closing before in-flight requests finish.

### Extract scheduler timing constants to environment config — 📌 Low Impact
The 60-second and 3600-second interval values for the digest and purge schedulers are hardcoded. Moving them to env vars (DIGEST_CHECK_INTERVAL_MS, PURGE_CHECK_INTERVAL_MS) allows tightening intervals in staging for faster testing and loosening them in production without code changes, reducing deploy risk.

### Split AdminDashboard into lazy-loaded route segments — 🔥 High Impact
At 126k+ characters, this single file almost certainly bundles all tab panels (Overview, Students, Reports, DB Viewer, etc.) into one chunk. Use React.lazy + Suspense per tab panel so that e.g. the DB Viewer and Perf Agent panels are not downloaded until accessed. Expected impact: 60-80% reduction in initial admin bundle size, faster TTI for admins.

### Consolidate overview stat queries into a single /api/admin/stats endpoint — 🔥 High Impact
Dashboard overview tabs commonly fire 5-8 separate useQuery calls for counts (users, schools, teachers, sponsors, etc.). Each is an independent HTTP round-trip. A single aggregated endpoint called once with Promise.all on the server side would reduce dashboard load from ~8 waterfalled queries to 1, cutting perceived load time by several seconds on Caribbean mobile connections.

### Add staleTime and gcTime to admin queries to prevent redundant refetches — ⚡ Medium Impact
Admin data like organization lists, school counts, and audit logs do not need to refetch on every window focus event. Without staleTime set, TanStack Query refetches on every tab switch and focus. Setting `staleTime: 60_000` on stable admin resource queries would eliminate redundant API calls when navigating between dashboard tabs, reducing server load and preventing UI flicker.

The extract-paper handler is the highest-risk area due to payload size, sequential per-question awaits inside a loop, and the fact that a crash between question inserts and the final status update leaves the paper in an inconsistent state despite the delete-on-retry guard.

### Database Schema (`shared/schema.ts`)
The schema is reasonably well-structured with FK constraints and some composite indexes in place, but has several reliability and security gaps: missing indexes on high-frequency filter columns, under-constrained column types, and nullable timestamps on audit-critical columns that should always be set. For a school-aged financial literacy app storing real transaction and portfolio data, these gaps pose data integrity and query-performance risks.

### Server Startup (`server/index.ts`)
The server startup sequence has multiple blocking async operations run sequentially before the HTTP server begins listening, creating unnecessary startup latency. Scheduler intervals are created without cleanup references, preventing graceful shutdown. Response body logging on every API response risks leaking sensitive financial or PII data to stdout.

AdminDashboard.tsx shows several meaningful reliability and performance gaps: async operations lack try/catch, query params are rebuilt on every render without memoization, and there are likely missing loading/error states in subcomponents.
