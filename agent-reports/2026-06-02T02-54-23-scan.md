## 🔴 Critical (6)

### `server/routes.ts` — registerChatRoutes(app) / registerImageRoutes(app) / registerAudioRoutes(app)
**Issue:** AI-heavy endpoints (chat, image, audio) are registered with no rate limiting middleware applied before them. In a school-aged user app these endpoints proxy to paid OpenAI APIs and are trivially abusable — a single student or bot could exhaust API quota or drive up cost.

**Fix:** Create a dedicated aiLimiter (e.g., 20 req/15 min per IP) in rateLimiter.ts and apply it before registration: `app.use('/api/chat', aiLimiter); app.use('/api/image', aiLimiter); app.use('/api/audio', aiLimiter);` — adjust path prefixes to match what registerChatRoutes etc. actually mount.

### `server/routes.ts` — robots.txt and sitemap.xml handlers (~line 44 and ~line 62)
**Issue:** Host and protocol values are read directly from `req.headers['x-forwarded-host']` and `req.headers['host']` without any validation or allowlist, then interpolated into rendered output. An attacker can send a crafted Host header to inject arbitrary URLs into the Sitemap directive or XML content (Host Header Injection / cache-poisoning vector).

**Fix:** Resolve the canonical base URL once at startup from an environment variable: `const BASE_URL = process.env.BASE_URL ?? 'https://finsightlite.com';` and use that constant in both handlers instead of reading request headers. Remove the header-based fallback entirely.

### `server/jobHandlers.ts` — extract-paper handler / job.payload.fileB64
**Issue:** Large base64-encoded file content (potentially MBs) is stored directly in the job payload row. This bloats the jobs table, slows queue polling queries, and can hit row-size limits in Postgres.

**Fix:** Upload the file to object storage before enqueuing the job and store only the resulting object path in the payload (e.g., { paperId, objectPath, ext, subject }). Inside the handler, stream or download the file from object storage using that path, then delete it after processing.

### `server/jobHandlers.ts` — purge-ai-usage handler
**Issue:** olderThanDays comes directly from job.payload with no validation before being used in arithmetic. A payload with olderThanDays: 0 or a negative value would delete all rows or produce a future cutoff date, causing an unintended full-table wipe.

**Fix:** Add a guard at the top of the handler: if (typeof olderThanDays !== 'number' || olderThanDays < 1) throw new Error('Invalid olderThanDays value: ' + olderThanDays); This causes the job to fail fast and visibly rather than silently destroying data.

### `shared/schema.ts` — simulatedStocks table definition
**Issue:** ticker is declared as text with no unique constraint. Duplicate tickers would silently corrupt portfolio valuation logic — any lookup by ticker symbol would match multiple rows, producing incorrect price or holding calculations.

**Fix:** Replace text('ticker').notNull() with text('ticker').notNull() and add a uniqueIndex in the table config: (t) => ({ tickerUniq: uniqueIndex('idx_simulated_stocks_ticker').on(t.ticker) }). Also add .notNull() to annualReturnPct or document explicitly that null means 'unknown return', since downstream arithmetic on null silently produces null in SQL.

---

### `shared/schema.ts` — portfolioHoldings table definition
**Issue:** There is no unique constraint on (userId, stockId) in portfolioHoldings. This allows multiple rows per user per stock, meaning buy/sell logic must manually deduplicate or aggregate holdings. A second concurrent buy request could insert a duplicate row, causing double-counted quantity and incorrect P&L.

**Fix:** Add a uniqueIndex on (userId, stockId): (t) => ({ userStockUniq: uniqueIndex('idx_portfolio_holdings_user_stock').on(t.userId, t.stockId) }). Update the buy logic to use an upsert (INSERT ... ON CONFLICT DO UPDATE) to atomically increment quantity and recalculate avgPurchasePrice.

## 🟡 Warnings (8)

- **`server/routes.ts` (initSupabaseTables().then(...).then(...) chain (~line 115)):** The seven Supabase seed/init calls are chained sequentially with .then(), so they execute one after the other. If any step throws, all subsequent steps are silently skipped (only a single top-level .catch logs). On a cold start this adds unnecessary latency to background init and makes failures hard to diagnose.
  → *Group independent operations with Promise.all and add per-step error logging: `Promise.all([ initSupabaseTables().catch(e => console.error('[Supabase] initTables:', e)), ... ]).then(() => Promise.all([seedFinancialAcademyLesson().catch(...), initStaticContentTables().catch(...)])).then(...)` — preserving ordering only where there is a real dependency (e.g., initStaticContentTables before seedStaticModules).*
- **`server/routes.ts` (/public-objects/:filePath(*) handler (~line 98)):** The `filePath` parameter is passed directly to `objectStorage.searchPublicObject()` without any sanitization. A path like `../../secrets/env` could traverse storage namespaces depending on the ObjectStorageService implementation, and there is no check constraining the path to safe characters or a known prefix.
  → *Sanitize before use: `const filePath = path.normalize(req.params.filePath).replace(/^\//, ''); if (filePath.includes('..')) return res.status(400).json({ error: 'Invalid path' });` and import Node's `path` module. Also consider an allowlist of accepted file extensions (e.g., jpg, png, mp4, pdf).*
- **`server/jobHandlers.ts` (runExtractPaper — for loop calling storage.createExtractedQuestion):** Extracted questions are inserted one at a time inside a sequential for-loop (await inside for). For a 40-question paper this is 40 round-trips, each blocking the next. A partial failure mid-loop leaves orphaned rows that the idempotency delete at the top of a retry will clean up, but the retry cost is high.
  → *Collect all valid question objects into an array, then perform a single bulk insert: await db.insert(extractedQuestions).values(validQuestions). If storage.createExtractedQuestion must be used, at minimum batch with Promise.all on chunks of 10.*
- **`server/jobHandlers.ts` (runExtractPaper — model and token constants):** The OpenAI model name ('gpt-4o') and max_completion_tokens (4096) are hardcoded in two separate openai.chat.completions.create calls. Changing models for cost/compliance reasons requires a code deploy.
  → *Read these from environment variables with safe defaults: const OPENAI_MODEL = process.env.OPENAI_MODEL ?? 'gpt-4o'; const MAX_TOKENS = parseInt(process.env.OPENAI_MAX_TOKENS ?? '4096', 10); Apply to both vision and extraction calls.*
- **`server/jobHandlers.ts` (runExtractPaper — sequential OpenAI calls for image path):** For image uploads the handler makes a vision call to extract text and then immediately a second GPT-4o call to parse questions. These are inherently sequential (second depends on first), which is correct, but there is no timeout or AbortSignal on either call. A hung OpenAI request will block the worker thread indefinitely, preventing it from processing other jobs.
  → *Wrap each openai call with a timeout using AbortSignal.timeout(ms): openai.chat.completions.create({ ..., signal: AbortSignal.timeout(60_000) }). Choose a timeout (e.g. 60 s) aligned with your job maxAttempts retry window.*
- **`shared/schema.ts` (categories, transactions, budgets, linkedCards, documentUploads, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions):** All createdAt columns use .defaultNow() without .notNull(), meaning application code or a migration bug can insert explicit NULL, silently breaking any audit, ordering, or analytics query that relies on createdAt being present.
  → *Add .notNull() to every createdAt definition, e.g. createdAt: timestamp('created_at').defaultNow().notNull(). Also apply to executedAt on portfolioTransactions and purchasedAt on portfolioHoldings for the same reason.*
- **`shared/schema.ts` (categories (userId), transactions (userId), budgets (userId), all tables with userId):** userId is declared as varchar('user_id') with no length argument. PostgreSQL will accept any length, but the referenced users.id column almost certainly has a defined length (e.g. 36 for UUID or 21 for CUID). A mismatch can prevent index usage on joins and FK lookups, and allows storing malformed IDs.
  → *Declare a consistent explicit length matching the auth provider's ID format, e.g. varchar('user_id', { length: 36 }) for UUID or { length: 21 } for CUID, on every userId column and on users.id itself in models/auth.*
- **`shared/schema.ts` (billReminders table definition):** billReminders has no index on (userId, nextDueDate), but the primary query pattern for this table — fetching upcoming bills for a user sorted by due date — requires scanning every row for that user without one. At scale this becomes a full sequential scan per dashboard load.
  → *Add an index in the table's third argument: (t) => ({ userDueDateIdx: index('idx_bill_reminders_user_due').on(t.userId, t.nextDueDate) }). If queries filter by userId alone first, a single-column index on userId is also acceptable but the composite is more useful.*

## 💡 Opportunities

### Parallel domain-route registration — ⚡ Medium Impact
The six `await register*Routes(app)` calls after the parallel import block are still executed sequentially. If any registration function performs async I/O (DB schema checks, cache warming) this adds cumulative cold-start latency. Wrap them in `await Promise.all([registerAuthDomainRoutes(app), registerStudentRoutes(app), ...])` — safe as long as they don't depend on each other's side-effects, which is the common case for Express route registration.

### Centralised BASE_URL environment config — 🔥 High Impact
Introducing a single `BASE_URL` env var (fixing the Host Injection finding above) also enables open-graph tags, email links, and any future absolute-URL generation to share one source of truth. Define it in a `config.ts` module validated at startup with zod or envalid so misconfiguration fails fast rather than silently producing wrong URLs in production.

### Structured logging for Supabase init errors — ⚡ Medium Impact
The single `.catch(e => console.error('[Supabase] Init error:', e))` loses which step failed and serializes Error objects inconsistently across Node versions. Replacing it with per-step structured log calls (including step name and error code) would make Railway log-drain queries and Sentry grouping far more actionable, especially during deploys where seed steps are most likely to fail.

### Idempotency key on extracted-question bulk insert — 🔥 High Impact
The current idempotency strategy (delete-then-reinsert) creates a window where a crash between delete and insert leaves zero questions. Replace with an upsert keyed on (paperId, order) using Postgres ON CONFLICT DO UPDATE, eliminating the gap and making retries truly safe without a full re-delete.

### Centralise OpenAI client configuration — ⚡ Medium Impact
openai is imported from a route file (replit_integrations/chat/routes), coupling job infrastructure to HTTP layer internals. Extract the configured OpenAI client into a shared singleton module (e.g., server/openaiClient.ts). This removes the cross-layer dependency and makes it trivial to inject mocks in tests.

### Stream CSV exports instead of buffering in memory — ⚡ Medium Impact
The admin-csv-export handler (truncated but referenced) almost certainly builds the full CSV string in memory before writing to object storage. For large datasets this risks OOM. Use a Transform stream pipeline piped directly into objectStorageClient's upload stream to keep memory flat regardless of row count.

### Centralise currency default as a shared constant — ⚡ Medium Impact
The string 'BSD' is hardcoded as a column default in at least six tables (transactions, savingsGoals, billReminders, portfolioHoldings, portfolioTransactions, simulatedStocks). Extract DEFAULT_CURRENCY = 'BSD' to a shared config or env variable and reference it in each column definition. This prevents drift if the app expands to TTD or JMD and ensures a single change point.

### Add index on documentUploads (userId, status) for processing queue queries — ⚡ Medium Impact
Document processing status polling — finding all 'processing' uploads for a user — has no supporting index. Adding a composite index on (userId, status) will convert this from a sequential scan to an index scan, which matters when users upload multiple statements and a background job polls for pending work.

### Replace text columns for enums with PostgreSQL native enums or add check constraints — ⚡ Medium Impact
Columns like transactions.type, budgets.period, and billReminders.frequency use Drizzle's text enum, which maps to plain text in Postgres with no DB-level constraint. If Drizzle is bypassed (raw SQL, migration scripts, admin tools), invalid values can be inserted silently. Define pgEnum types for each and reference them in column definitions to enforce constraints at the database layer.
